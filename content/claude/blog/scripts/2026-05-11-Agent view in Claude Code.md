## Shownotes

本期我们聊聊 Claude Code 在二〇二六年五月十一日上线的新功能 agent view,智能体总览视图。这是一个专门用来管理多个并行 Claude Code 会话的界面,目标是让你不再需要靠一堆终端标签和心智账本去追踪每个任务的状态。我们会拆解它的交互设计、几种早期用户摸索出的实战用法,以及上手之前需要注意的几个坑。

本期要点
- agent view 是什么:一个集中查看所有 Claude Code 会话状态的总览面板,按左箭头或运行 claude agents 即可打开
- 瞄一眼加快速回复:不离开列表就能看到会话最后一轮内容并直接回答卡住的决策,按回车可进入完整对话
- 后台化能力:用 slash bg 把现有会话转入后台,或用 claude --bg 直接以后台模式启动新任务
- 早期用户的四种实战玩法:批量并发跑任务、管理长循环智能体、在主线和旁支任务间穿梭、快速扫描哪些任务已交付
- 当前定位与使用门槛:研究预览版,覆盖 Pro、Max、Team、Enterprise 和 Claude API 用户,正常速率限制依然适用
- 主播个人观察:这代表 AI 编程工具从单会话聊天转向多智能体协同管理的趋势

大家好,欢迎收听 Claude FM,我是你的主播。今天我们要聊的这篇文章,英文标题是 Agent view in Claude Code,我把它翻译过来,大概的意思是 Claude Code 里的智能体总览视图。这篇文章发表于二〇二六年五月十一日,出自 Claude 官方博客,作者是 Anthropic 团队自己。距离今天二〇二六年六月二十日,刚刚过去一个月多一点,所以这是一个相当新鲜的功能更新,你完全可以把它当成最近一个月的产品动态来听。

为什么值得专门拿一期节目讲这个东西呢。因为如果你平时已经在用 Claude Code 写代码,你大概会有这样的经历,就是同时开着好几个任务,一个窗口里让它修一个 bug,另一个窗口里让它写测试,第三个窗口里又丢给它一个重构的活儿,结果你自己反而成了那个最忙的人,不停地在终端标签之间来回切换,生怕错过哪个任务卡住了等你确认。今天要讲的这个新功能,名字叫 agent view,中文我们可以理解成智能体视图或者智能体总览面板,它要解决的就是这个问题,让你不再需要自己当人工调度员。

先补充说明一下背景。Claude Code 是 Anthropic 推出的一款命令行编程助手,你在终端里跟它对话,它就能帮你读代码、改代码、跑测试、提交 pull request。这几年随着模型能力越来越强,很多开发者已经不满足于一次只跑一个任务了,大家开始尝试同时开多个 Claude Code 会话,让它们并行干活,这样整体效率会高很多。但问题也跟着来了,管理这些并行的会话本身就成了一项体力活。

我们可以想象一个很真实的场景。假设你早上九点坐到电脑前,打开了六个终端标签,第一个标签让它修一个线上反馈的 bug,第二个标签让它给某个模块补单元测试,第三个标签在做一次大规模的重命名重构,第四个标签是一个同事让你帮忙看看的小脚本,第五个标签在跑一次依赖升级,第六个标签纯粹是你顺手问的一个代码库相关的小问题。十分钟以后,你已经记不清哪个标签需要你回去确认一个命名方案,哪个标签其实早就跑完了在那儿干等着,哪个标签因为权限问题卡住了根本没在动。你只能一个一个标签点过去看,点到第三个的时候才发现它五分钟前就已经在等你拍板了,而你刚才一直在看第五个标签的日志刷屏,完全没注意到提示。这种状态积累一上午,你会发现自己花在找哪个该看了这件事上的时间,可能比你真正做决策的时间还长。

在 agent view 出现之前,常见的应对方式基本是两种。第一种就是刚才说的开一堆终端标签页,一个任务一个标签,自己在心里记一张表,记着哪个标签该回去看看了。第二种更硬核一点,用 tmux,也就是一种命令行下的终端多窗口管理工具,把屏幕切成网格,每个格子放一个会话,看起来很专业,但本质上还是靠你自己盯着屏幕,哪个格子闪了一下提示,你就手忙脚乱地点过去。文章里的原话很形象,说你得维护一个超载的心智账本,记着接下来要处理什么。这种状态,说白了就是人脑在做调度器该干的事,效率自然上不去,而且容易漏掉真正需要你拍板的那个会话。

好,那 agent view 具体怎么解决这个问题,我们一层一层拆开来看。

第一个核心点,是把所有会话集中到一个地方看。怎么打开呢,两种方式,一种是在任意一个 Claude Code 会话里按左箭头键,另一种是直接在终端里敲一条命令,叫 claude agents。打开之后你看到的是一个列表,列表里每一行对应一个会话,每一行会告诉你四件事,这是哪个会话,它现在是不是在等你做决定,它最后一条回复的内容预览,还有你上次跟它互动是什么时候。你可以把这个列表想象成机场的航班显示屏,每一行是一个航班,你一眼就能看出哪个已经起飞、哪个延误了在等地面指令、哪个已经落地。换成开发场景,你扫一眼这个列表,立刻知道哪几个智能体在闷头干活不用管,哪几个卡在那儿等你拍板,哪几个已经收工交付了。还是用刚才那个六个标签的例子来说,如果当时有这张列表,你根本不需要一个个标签去点,扫一眼就能看到第三个标签那一行被标成了等待输入,第五个标签那一行最后一条记录还在刷日志说明还没跑完,第四个标签那一行内容预览已经写着改动完成,你立刻就知道接下来三十秒该把注意力放在哪一行上。

这里我想多展开讲一下这个设计背后的思路。它其实是把原来分散在你脑子里、分散在十几个终端标签里的状态,统一收口到一张表上。这跟很多运维团队用的监控大盘是一个道理,你不需要去逐个服务器登录查看,而是有一块屏幕,把所有服务的健康状态、告警情况摆在一起,出问题的那个会自动标红。如果你写过后端服务,大概也用过类似 top 或者 htop 这种进程监控命令,一条命令把所有进程的状态摆在一张表里,谁占着 CPU,谁卡住了,一眼可见,不需要你挨个进程去问。agent view 本质上就是给你的编程智能体们装了这样一块监控大盘,只是监控的对象从服务器进程变成了一个个正在帮你写代码的 Claude Code 会话。

第二个核心点,叫做不用离开列表就能瞄一眼并且回复。具体的交互是这样的,你在列表里选中某一个会话,就能瞄一眼,也就是 peek,看到它最后一轮对话的内容。如果这个会话正卡在等你做决定,比如它问你这个文件要不要覆盖、这个依赖要不要升级,你可以直接在这个瞄一眼的界面里把答案打进去,会话拿到你的回复之后就会接着往下跑,你完全不需要正式切进去那个会话的完整界面。如果你确实想看完整的对话记录,把每一步推理过程都翻一遍,那就按回车键,直接接入这个会话,进入它的完整对话历史。

这个设计我觉得挺关键的,它区分了两种不同强度的关注。大多数时候,你只是需要确认一下方向对不对,回答一个简单的是或否,这种场景下你根本不需要把整个对话历史都翻出来,瞄一眼加快速回复就够了,成本很低,几秒钟就能处理完一个会话,然后继续看下一个。只有真正复杂、需要你仔细审视上下文的时候,你才需要按回车进入完整模式。这种分层设计能大幅降低你在多个任务之间切换的心理负担,因为大部分交互被压缩成了非常轻量的动作。打个比方,这有点像你在管理一个团队的时候,大部分汇报你站在工位旁边听十秒钟、说一句继续就行了,只有少数真正复杂的决策才需要拉进会议室坐下来认真过一遍,你不会对每一件事都用同样的重量级方式去处理,agent view 把这种分级处理的直觉,直接做进了交互设计里。

第三个核心点,是后台化的能力。这里有两条路径。第一条,如果你正在一个会话里干活,干到一半发现这个任务其实可以丢到后台慢慢跑,不需要你一直盯着,你可以用一条斜杠命令,叫 slash bg,把当前这个会话直接送进后台,它就会出现在 agent view 的列表里继续运行,但不再占着你的前台终端。第二条路径更直接,你可以完全跳过前台这一步,直接用一条命令,claude 加上 dash dash bg,后面跟上你要它做的任务,这样一来,一个全新的会话直接就在后台启动,你从头到尾都不需要为它单独开一个前台窗口。

我个人觉得这个设计很聪明的地方在于,它把任务的发起方式和任务的运行位置彻底解耦了。以前你要么从一开始就规划好这是个后台任务,要么就得一直占着一个前台窗口。现在你可以走着走着突然反应过来,这个其实不用我守着,一条命令丢出去,继续干别的事。这种灵活性对日常工作流的影响其实挺大的,因为很多时候你一开始并不知道一个任务会跑多久,需不需要你全程陪着,有了这个能力,你可以随时把判断错的前台任务降级成后台任务,而不需要重新开一个。你可以把这个体验类比成点外卖,你下单的时候不需要决定接下来这二十分钟要不要一直盯着配送进度条,你可以先去开个会,系统会在真正需要你签收的那一刻提醒你,中间这段等待的时间完全还给你自己。

接下来文章里还提到了几个早期用户已经摸索出来的实际用法,我觉得这部分特别值得展开讲,因为这才是真正告诉你这个功能能在日常工作里派上什么用场的地方。

第一种用法,叫做扩大并发会话的规模。简单说就是,你一次性发出去好几个不同的想法,每一个可选搭配一个 skill,也就是技能包,你可以理解成一组提前打包好的提示词和工具配置,专门用来处理某一类任务,比如专门写测试的技能包,或者专门做代码评审的技能包。发出去之后你就不用管了,等你回头再看 agent view 的时候,看到的是一份已经准备好等你审查的 pull request 列表,也就是一堆已经写好等你点头合并的代码改动请求。打个比方,这就像你早上出门前,同时给好几个不同的承包商打电话,分别交代了好几件装修的活儿,你出去转一圈回来,直接看哪几家已经把活儿干完了等你验收,而不是挨个打电话催进度。如果换成你来用这个模式,比较合理的做法可能是,把同一个问题拆成几个不同的解决思路同时发出去,比如修一个性能问题,你可以同时让一个会话尝试加缓存,另一个会话尝试改算法复杂度,第三个会话尝试调整数据结构,等结果出来以后你再挑一个最合适的方案合并,这种打法在以前一个会话一个会话排队跑的时候是不现实的,因为时间成本太高,现在并行铺开,反而是最自然的用法。

第二种用法,是管理那些长时间跑着不停的智能体。文章里举的例子包括 PR babysitter,你可以理解成专门盯着某个 pull request 的看护型智能体,只要这个 pull request 有新的评论或者持续集成跑出新结果,它就自动去处理一下,还有专门负责更新仪表盘的智能体,以及其他那种循环往复执行的任务。这类智能体的特点是它不是一次性跑完就结束,而是按某个周期反复醒来干一点活儿,再睡回去。agent view 很贴心的一点是,它会直接在列表里告诉你这个智能体下一次醒来执行是什么时候,你不需要猜,也不需要时刻盯着,只要在它该醒的时间点附近回来看一眼就行。这跟你设置一个定时任务、然后看日历提醒下次执行时间,本质上是一样的体验,只是现在所有这些定时任务都汇总到了同一张列表里。

第三种用法,是在多个独立会话之间灵活穿梭。场景是这样的,假设你正埋头在一个会话里处理一个比较复杂的任务,突然脑子里冒出一个相关但独立的小问题,或者你想顺手让另一个智能体去查一下代码库里的某个细节。这时候你按左箭头,从当前会话退到总览列表,开一个新的任务或者扔出一个关于代码库的快速提问,扔完之后按右箭头,直接弹回到你原来正在做的那个会话,继续刚才的思路。等那个新开的小任务有了结果,你只要再瞄一眼就能看到答案,完全不需要专门倒回去切换窗口确认。这种设计照顾到了一个很真实的开发场景,就是我们的思路经常会被一些小的旁支问题打断,但又不想真正放弃手头主线任务的上下文,左右箭头来回跳转刚好解决了这种轻量级的打断和恢复。

第四种用法,叫做一眼看出哪些任务真正交付了成果。每一行会话前面都有状态指示标记,配合瞄一眼界面里显示的标题,你可以很快扫一遍整个列表,看出哪些会话最终真的产出了一个 pull request,哪些只是聊了几句还没出活儿。这个用法听起来简单,但我觉得它解决的是一个很现实的管理问题,就是当你同时跑着十几个智能体的时候,你最关心的往往不是过程,而是结果,谁交了作业,谁还在磨。有了这种一眼可扫的状态标记和标题,你完全可以把它当成一个每天收工前的复盘清单来用,坐下来花两分钟扫一遍,今天到底交了几份作业,哪几份值得仔细看,哪几份还得再等等。

讲完这些用法,我们再说说这个功能目前处于什么阶段,方便大家判断要不要现在就上手。文章原文说得很清楚,agent view 目前是以研究预览版的形式开放的,也就是说它还在一个早期测试阶段,体验和细节未来可能还会调整。它面向的用户范围覆盖了 Pro、Max、Team、Enterprise 这几种订阅档位,以及 Claude API 的用户,基本上把 Claude Code 的主流用户群都覆盖到了。开启方式很简单,在终端里敲一条 claude agents 就能选择体验。需要注意的是,文章特别提了一句,正常的速率限制依然适用,也就是说这个新界面只是改变了你管理会话的方式,并不会额外给你解锁更多的用量配额,这一点对于已经经常顶着用量上限跑任务的同学来说,是个需要留意的细节。

好,讲完了文章里的内容,我们来说说这些信息对你的实际工作可能意味着什么,也就是你拿到这个功能之后,具体可以怎么用,以及要避开哪些坑。

如果你现在已经是 Claude Code 的日常用户,尤其是那种习惯一次开很多个任务并行处理的人,我建议你今天就可以去试一下,具体做法就是升级到支持这个功能的版本,然后在终端里敲 claude agents,先把列表界面打开看看长什么样,熟悉一下左右箭头和瞄一眼这套交互方式,通常十几分钟就能上手。

我们可以一起推演一下,如果你把这套东西用到一个完整的工作日里,大概会是什么样子。早上到工位,你先把昨天晚上挂在后台跑的那个依赖升级智能体打开看一眼,如果它已经跑完而且测试通过,你直接在瞄一眼界面里回一句可以合并,然后顺手把今天要做的三件正事一次性发出去,一件是修复昨天那个反馈的 bug,一件是给新模块补测试,一件是给某个性能瓶颈同时尝试两种优化方案。发完之后你不需要傻等,可以去处理邮件,处理完回到 agent view,扫一眼列表,看到修 bug 的那个会话标成了等待输入,你瞄一眼发现它是在问你要不要顺带修一个相关的小问题,你回一句顺带修了吧,它接着跑。中午之前你可能顺手又冒出一个小问题,想确认一下某个接口的调用方式,你按左箭头开一个新的快速提问,问完按右箭头弹回到刚才正在看的那个会话,继续原来的思路。下午你回头扫一遍整张列表,看看今天这几条线一共交了几份 pull request,挑出真正需要你坐下来认真看代码的那一份,剩下的简单走个流程合并掉。一整天下来,你花在调度和找进度上的时间,可能从原来的一两个小时,压缩到了十几分钟,省下来的时间全部还给了真正需要你动脑子的审查和设计工作。这只是我设想的一种典型用法,实际体验肯定因人而异,但大体的节奏应该是类似的。

具体的工作流上,我觉得有几个搭配值得你试试。第一个,把日常重复性的维护工作,比如依赖升级、文档同步、定期跑一遍代码扫描,都改成用 slash bg 或者 claude dash dash bg 直接丢到后台,让它们按自己的节奏跑,你只在 agent view 里偶尔扫一眼下次执行时间。第二个,如果你手头同时有几个独立的功能要开发,可以把每一个都配上一个合适的技能包,一次性发出去,然后把你自己的时间留给真正需要动脑子的设计决策,而不是盯着进度条。第三个,养成一个习惯,就是在你深度投入某个复杂任务的过程中,不要因为旁支小问题打断自己的思路,直接左箭头退出去丢一个新任务,右箭头弹回来接着干,把这种轻量打断变成你工作流里的常规动作而不是意外。

晚上下班前,你大概还会有最后一个动作,就是把一天里挂在后台的那些循环型智能体过一遍,看看明天早上它们各自下一次醒来执行的时间点,顺手把那些已经确认没问题的会话关掉,留下真正还在跑的几条线,这样第二天早上打开 agent view,看到的是一张干净的、只剩下真正需要关注的任务的列表,而不是堆积了一整周的陈年记录。这种收尾习惯,我觉得跟你下班前把桌面文件归归类是同一个道理,花两分钟整理,换来的是第二天打开电脑那一刻的清爽。

需要避开的坑,我觉得主要有三个。第一,这是研究预览版,意味着交互细节和命令行为未来可能会变,如果你的团队打算把整个工作流都搭建在这上面,要留一点心理预期,不要假设现在的命令和快捷键是永久不变的。第二,正常速率限制照样适用,这意味着你不能无限制地同时铺开十几个、二十几个并行会话,你需要根据自己的订阅档位和真实用量情况,把并发数量控制在一个合理范围,不然很容易在一天还没过完的时候就把额度用光了,反而耽误真正紧急的任务。第三,也是我觉得最容易被忽略的一点,agent view 解决的是你管理会话的效率问题,但它并不会替你审查代码的正确性,看到列表里有一堆已经准备好的 pull request,不代表你可以闭着眼睛直接合并,该有的代码评审、测试验证这些步骤一步都不能省,工具帮你解决的是调度和可见性的问题,质量把关这件事最终还是要落在你自己身上。我个人还想提一个原文没有直接讲到的隐患,就是当你真的把并发数量铺得很开,比如同时跑十几二十个会话的时候,这张总览列表本身也可能变得信息过载,一眼扫过去全是等待输入的红点,你照样会陷入选择困难。所以我建议给自己设一个心理上限,比如同时真正需要你盯着的任务不超过五六个,其他的尽量都用后台加定时巡检的方式去处理,而不是一股脑全部铺开。

我个人想多说两句自己的看法,跟原文的内容分开来讲。我觉得这次更新背后反映的是整个 AI 编程工具这个领域正在发生的一个更大的转变,就是从原来那种你坐在电脑前跟一个助手一对一聊天的模式,慢慢转向你像运营一支小型团队一样去管理一批智能体。这个转变其实跟我们之前经历过的很多技术变迁有点像,比如从单线程编程到多线程、再到分布式系统,核心难题永远不是怎么让每一个执行单元变强,而是怎么让人能够高效地协调一堆并行跑着的执行单元,知道什么时候该插手,什么时候可以放手。agent view 某种程度上就是在编程智能体这个场景下,给出了一个回答这个协调难题的具体方案。我自己用类似工具的体感是,这种总览视图加轻量瞄一眼回复的组合,确实能明显降低多任务并行时候的认知负担,但它也对使用者提出了新的要求,你得学会判断哪些任务真正值得并行去跑,而不是为了图新鲜把什么都一股脑丢出去,毕竟会话数量上去了,你审查质量把关的工作量也会跟着上去,工具解决的是调度问题,而不是帮你减少应该花的审查时间。

如果换成是你来设计这样一个功能,你会怎么想呢。我猜你大概也会面对同样的取舍,一方面想让用户尽可能少地操心调度细节,另一方面又不能把人完全排除在决策之外,毕竟智能体跑出来的结果终归需要人来拍板。agent view 选择的方案是把人放在一个轻量的旁观和快速决策的位置上,而不是完全自动化地放权,这种折中我个人觉得是比较稳妥的做法,既释放了并行带来的效率,又没有把控制权完全交出去。换个角度想,这其实也是在回答一个更根本的问题,就是随着智能体越来越能干,人在整个流程里应该往哪个方向退,退到完全不管显然不现实,因为产出质量没人把关会出问题,但继续事事亲力亲为又浪费了并行带来的效率红利,agent view 给出的答案是退到一个可以随时被拉回来、但默认不打扰你的位置上,这个分寸感我觉得是这次更新里最值得琢磨的地方。

最后我们简单总结一下这篇文章的内容。Claude Code 这次推出的 agent view,核心是给所有并行跑着的智能体会话提供了一个统一的总览界面,你可以通过左箭头或者命令 claude agents 打开,一眼看出谁在等你、谁还在跑、谁已经完工。可以在不离开列表的情况下瞄一眼最近的对话内容,直接回复卡住的决策点,也可以按回车进入完整对话。还可以用 slash bg 或者 claude dash dash bg 把任务随时丢进后台运行。早期用户已经摸索出几种很实用的玩法,包括批量并发跑多个想法、管理那些按周期循环执行的长任务、在主线任务和旁支任务之间灵活穿梭,以及快速扫一眼到底哪些任务真正交付了成果。这个功能现在以研究预览版的形式开放给 Pro、Max、Team、Enterprise 以及 Claude API 用户,使用正常的速率限制,你只需要在终端敲一条 claude agents 就能开始体验。

我个人的整体评价是,这是一次很实用、也很顺理成章的产品迭代,它没有改变模型本身的能力,而是改变了人和一群智能体打交道的方式,这种看起来不起眼的交互层改进,往往才是决定一个工具能不能真正嵌入日常工作流的关键。如果你平时已经习惯让 Claude Code 同时干好几件事,这次更新值得你抽十分钟去试一下,它解决的不是模型能力的问题,而是你作为使用者怎么和一群同时工作的智能体打交道的问题。好,这期就先聊到这里,我们下期再见。
