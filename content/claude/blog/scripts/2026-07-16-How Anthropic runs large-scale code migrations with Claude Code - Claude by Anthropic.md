## Script

今天我们要聊的文章，标题是 How Anthropic runs large-scale code migrations with Claude Code，中文可以翻译成，Anthropic 如何用 Claude Code 运行大规模代码迁移。这篇文章发表于二〇二六年七月十六日，也就是三天前，所以时效性非常新。它所处的背景，是 AI 编程已经从帮你补全函数、生成脚手架，进入到可以组织大量 agent，也就是智能体，长时间并行执行工程流程的阶段。

为什么这篇值得听？因为它讲的不是又一个炫技 demo，而是一个很多工程团队都拖了很久、也很怕碰的问题，代码迁移。比如从一种语言迁到另一种语言，从老框架迁到新框架，从动态类型迁到静态类型，或者把一套已经背着历史包袱跑了很多年的系统，重新搬到一个更安全、更快、更适合未来维护的技术栈上。过去这类项目常常意味着几年时间、双线维护、路线图冻结、团队士气下滑，以及一个很现实的职业风险。你把公司拉进一个迁移大坑，最后如果只做到百分之九十兼容，那不是胜利，而是灾难。

Anthropic 这篇文章给出的核心判断非常直接，大规模代码迁移的账变了。不是说它突然免费，也不是说人可以不管，而是最坏情况从以前的多年沉没成本，变成了你删掉一个分支，然后重跑一遍流程。文章里最重要的一句话是，你不修代码，你修产生代码的流程，也就是 loop，循环。这个说法很工程，也很值得我们展开。

先看两个案例。第一个是 Bun。Bun 的联合创始人 Jarred Sumner，现在也是 Anthropic 的技术人员，他用 Claude Code 把 Bun 从 Zig 迁移到了 Rust。文章说，这次迁移在不到两周里产出了一百万行代码，合并前 Bun 现有测试套件在持续集成里百分之百通过。合并之后发现了十九个回归问题，也都已经修复。这个 Rust 版本已经在六月被放进 Claude Code 里发布。

第二个案例来自 Anthropic Labs 的共同负责人 Mike Krieger。他把一个 Python 代码库迁成了十六万五千行 TypeScript，用了一个周末。这里面包括数百个 agents，八个阶段门禁，三轮对抗式评审，最后还做了一个行为等价检查，把每个命令的输出和原来的 Python 版本逐一对比。你可以想象，这已经不是一个人打开编辑器，叫模型帮忙改几个文件，而是把 AI 当成一组可以被调度、被审查、被重新训练规则的工程劳动力。

不过文章没有把这个故事包装成魔法。它明确说，迁移仍然要有商业理由。过去百万行迁移可能要在四年里消耗三百万到四百万美元的工程资源。现在成本下降了，但依然可能是数万到数十万美元。Bun 这次迁移消耗了五十九亿 uncached input tokens，也就是没有缓存的输入 token，还有六点九亿 output tokens，也就是输出 token。按 API 价格估算，大约十六万五千美元。Mike 那个迁移的主体部分是二千七百万 token。token 可以粗略理解成模型读写文本时使用的计费和处理单位。

所以这不是零成本神迹，而是经济边界改变。以前你需要一个近乎生死攸关的理由，才敢推动语言迁移。现在，也许一年里反复修内存 bug，或者一个长期编译瓶颈，就足够重新算账。Mike 的项目就是因为编译步骤太慢。他们的内部工具会以单个二进制发给用户，用 Python 工具链生成每个平台的二进制大概要八分钟，整个构建矩阵每次发布要等三十分钟。迁到 TypeScript 之后，同样的编译大约两秒，二进制启动快了六倍，还可以退掉一条单独的部署流水线。对工程师来说，这种变化不是抽象指标，而是每次发布少喝一杯冷掉的咖啡，少被构建队列打断一次上下文。

文章提到 Anthropic 使用的模型包括 Claude Fable 五、Claude Opus 四点八，以及动态工作流。Claude Code 是 Anthropic 面向开发者的编码代理工具，可以在代码库里读取文件、修改文件、运行命令、派生子 agent。这里的 agent 不是聊天窗口里的单轮问答，而是有目标、有上下文、能执行一段任务并把结果反馈给上层流程的智能体。Anthropic 说，Fable 和 Opus 四点八尤其擅长委派、指挥、验证并行工作流，能让很多子 agent 朝着一个目标找到多条路径。

为什么代码迁移特别适合高级模型和 agent？文章总结了几个原因，我们用更口语的话讲一遍。第一，它天然并行。一个大代码库有很多文件、包、crate、模块，只要依赖关系梳理好，就能把任务切成成百上千块。你不需要一个 agent 从第一行改到最后一行，可以让很多 agent 同时搬砖。第二，上下文很清楚。旧代码就是规格说明书。相比让模型从零猜业务逻辑，迁移有一个极大的优势，原系统已经在那里跑着，它既告诉你应该做什么，也告诉你边界行为是什么。第三，它有内置裁判。很多成熟代码库有测试套件，编译器、测试、diff 都可以变成客观裁判。模型最怕的是人类说这个写得不够优雅，那就很主观。但如果裁判是编译不过、输出不一致、测试失败，模型就可以连续几天围着一个明确目标打磨。第四，任务队列会自己生成。编译失败就是下一批修复项，测试失败就是下一批修复项，运行崩溃也是下一批修复项。第五，迁移需要一致性和边界处理，而这正好可以通过规则书来约束。每发现一种新坑，就把它变成规则，让后续 agent 都遵守。

这就引出全篇最关键的方法，不要一上来就让 AI 改代码。先建立裁判。文章说，开始迁移前必须有一个 strong judge，也就是强裁判，否则你不知道什么时候可以退出，也不知道成功是什么。这个裁判必须能公平评价原代码和目标代码。很多团队会说，我们有测试呀。但问题在于，原语言里的测试常常依赖内部函数、内部对象和内部结构，迁到新语言后这些东西不存在了。这样的测试不能直接当裁判。

所以第一步是把现有测试分类。哪些测试可以表达为外部调用，哪些依赖内部实现。然后把面向外部行为的测试改写成两套代码都能运行的断言。这里文章还建议用对抗式 agent 去检查，确保改写后的测试没有削弱原来的断言。最后要验证裁判本身。你要先让它跑原代码，确认应该通过。再让它跑一些故意弄坏的代码，确认应该失败。如果一个裁判抓不住破坏，那它就不是裁判。

Jarred 的情况比较幸运，Bun 有一套用第三种语言 TypeScript 写的大测试套件，所以可以比较独立地验证行为。但大多数项目未必有这么好的条件。Mike 的 Python 到 TypeScript 迁移，就做了一个 parity harness，可以理解成行为等价测试框架。他选了七个真实世界场景，只要新旧版本行为不同，就当作 bug 修。这个点对很多团队很重要。你不一定需要一开始就有完美测试套件，但你必须有某种客观判断机制。没有裁判，AI 迁移就会变成一堆看起来很像代码的文本堆积。

有了裁判以后，文章进入迁移基础建设。这里有三个东西，规则书、差异清单、依赖地图。听起来有点像工程管理文档，但在 AI 迁移里它们不是摆设，而是整个流程的操作系统。

规则书是什么？就是告诉 agent，遇到某种旧语言的类型、习惯用法、库调用、错误处理方式、内存管理方式，应该如何翻译到新语言。差异清单是什么？就是那些不能简单翻译、必须重构或重新设计的地方。依赖地图是什么？就是文件之间、模块之间谁依赖谁，帮助你决定哪些工作可以并行，哪些必须先做，哪些要放在同一批处理。

文章特别强调，顺序很重要，规则书要在差异清单之前。因为所谓差异，就是规则书默认规则覆盖不了的东西。如果你还没定义默认迁移方式，就不知道什么算例外。规则书和差异清单还要一起被审计，因为有时候你以为是例外，其实可以通过一条更好的通用规则解决。

规则书的形态取决于你一开始的架构选择。你是保留原结构迁移，还是趁机重新设计。如果是 Jarred 这种保留结构的迁移，规则书更像查表，把 Zig 的类型和习惯用法映射到 Rust，把难翻译的组件指向差异清单。如果是 Mike 这种带重新设计意味的迁移，规则书更像设计文档。文章说 Jarred 是通过和 Claude 对话来形成规则书的，对每个模糊区域制定策略。他还设计了八个子 agent，分别针对八类常见失败模式做评审，这些类别来自他自己的直觉和经验。

这里我个人认为非常关键。AI 迁移不是把人的判断拿掉，而是把人的判断前置，并放大。资深工程师最有价值的不是手改每个文件，而是在开始前说清楚，我们这个系统里最容易错的是什么，哪些风格必须统一，哪些边界不能碰，哪些历史包袱可以顺手消掉，哪些必须保持兼容。然后让 agent 在这个规则框架里大量执行。

依赖地图也很现实。有些语言和项目有清晰的 manifest，也就是依赖声明文件，解析起来比较容易。但很多老代码库，尤其 C、C++、Python 之类，依赖关系可能散落在 include、动态 import、运行时注册、约定式目录里。Anthropic 说，Claude Code 可以派 agent 去创建并运行一个确定性的脚本，生成依赖地图。注意这里是确定性的脚本，不是让模型凭感觉说哪个文件依赖哪个文件。模型可以写工具，但真正做裁判的仍然是脚本输出。

再说差异清单。文章用了两个例子。Zig 到 Rust 的核心差异之一是内存管理。Zig 和 C、C++ 类似，很多时候需要人工管理内存。一个函数分配了一块缓冲区，注释里可能写着调用者要释放。但如果调用者忘了释放，代码还是能编译，泄漏可能到运行时才暴露。Rust 的方式不同，所有权移动给调用者，生命周期和释放由语言机制约束。你释放两次，或者移动后再用，通常编译阶段就过不去。忘记释放这个动作本身也不存在，因为 drop 是自动的。

Python 到 TypeScript 的核心差异则是接口和契约。Python 里一个函数接收 handler，只要这个对象有 setup 方法和 run 方法就能工作。到底哪些对象会传进来，你可能要读完整个代码库才知道。TypeScript 要你写清楚接口。setup 返回什么，run 接收什么参数，返回什么 Promise，结果对象长什么样，都要声明。动态语言里隐含在使用方式和运行路径里的知识，到了静态类型语言里必须被写下来。这些隐含知识就是差异清单要捕获的内容。

文章提到，Jarred 是先做差异清单，Mike 则先翻译再通过审计生成差异清单。你可能需要两者都做。这也很符合真实项目。很多隐含契约，人在一开始也说不清，必须让模型试着迁一遍、编译一遍、跑一遍，错误才会浮上来。关键是错误浮上来后，不要只修一个文件，而是把它归纳成规则。

接下来是一个很有意思的环节，迷你迁移，也可以叫试航。文章说这一步不是为了推进进度，而是为了摇晃一下船，看哪里漏水。Jarred 的做法是，让一个 agent 按规则书翻译三个文件，让另一个 agent 像资深 Rust 工程师一样翻译同样三个文件，再让第三个 agent 比较差异，提炼新的翻译规则。就在这个阶段，他发现了两个关键问题。如果没有提前发现，一旦把工作铺开到全部一千四百四十八个文件，这两个问题会制造大量连锁错误。

这个方法只适合保留结构的迁移，因为两种翻译结果可以按文件、按行比较。如果你的迁移是重新设计式的，比如 Mike 的项目，那等价做法不是比较同一文件的两版翻译，而是让对抗式评审直接攻击设计文档，再用一次可丢弃的端到端迁移验证设计。文章强调，无论如何，这一步生成的翻译文件都要扔掉。目标是改进规则，不是积累代码进度。

这句话很反直觉，也很重要。我们工程师很容易有沉没成本心理，既然都翻了三个文件，能不能留着？文章的答案是，不要。因为早期试航的代码是在旧规则下产出的，留下它可能污染后续。真正有价值的是从试航里学到的规则。规则修好以后，让系统重新生成，比手动缝补更稳定。

然后进入主体循环，实施、评审、修复。这里就是大量 agent 开始工作的时候。文章说，可以把实施工作交给更小的模型，把评审工作留给更大的模型。比如 Mike 在主迁移时派出了十二个子 agent，用 Claude Sonnet 做大规模实施。较大的 Fable 或 Opus 用在更需要判断的位置，比如评审、写规则、规划流程。原因很简单，token 花费集中在循环里，如果所有高频任务都用最大模型，成本会爆炸。便宜模型负责高吞吐搬运，贵模型负责制定和审查规则，这才是工程化用法。

任务队列要机械化。文章给了一个非常朴素但强大的定义，完成就是输出文件存在于磁盘上。批处理脚本检查哪些目标文件已经存在，然后把未完成文件切成批次交给实施 agent。因为队列每次都从磁盘重建，所以天然可恢复。中间机器挂了、流程中断了、某些 agent 失败了，下次再扫一遍文件系统就知道从哪里继续。

这看起来简单，但我个人认为是很多 AI 工程流程成败的分界线。不要把状态藏在聊天记录里，不要靠一个超长上下文记得哪些做完了。状态要落盘，队列要可重建，完成条件要机械，失败要能重跑。这样你才是在做工程，不是在做一次性表演。

在实施阶段，agent 可能会过分谨慎，做得太少。文章说，解决方法有时候就是在 prompt，也就是提示词里更明确、更强硬地告诉它，尽量完成迁移，后面的编译器会抓错误。不能确信的地方，就打上待迁移标记并写明原因，留到后续步骤处理。从这时开始，待办事项会自己出现。编译器列出错误，冒烟测试找出崩溃，完整测试报告失败。

评审也不是一个 agent 点头就过。文章建议两个对抗式评审 agent 使用彼此独立的上下文来检查实施结果。如果两者意见不一致，就交给第三个 agent 裁决。更重要的是，当评审反复在多个文件里抓到同一类错误时，不要逐文件修。你要往规则书里加一句话，然后重新生成受影响的批次。文章说得很明确，规则书在这个阶段持续增长，代码不要被手工打补丁来对抗规则。

这里的理念可以类比编译器优化。如果你发现一千个输出文件都有同一个模式错误，你不会打开一千个文件逐个改，你会去改生成器。AI 迁移也是一样。代码是产物，规则和循环才是生产线。只修产物，会让系统越来越不一致。修生产线，后面所有产物都会一起变好。

还有一个很细但很实际的设计点，编译器放在哪里。Mike 的 TypeScript 编译很快，几秒钟就能检查一个单元，所以他把 TypeScript 编译器放进每一轮循环。Jarred 的 Rust 项目不一样，cargo 编译要几分钟，所以他把编译器从实施循环里拿掉，推迟到下一阶段统一处理。这个选择没有绝对答案，取决于反馈成本。如果反馈便宜，就尽早、频繁地用。如果反馈昂贵，就批量化、串行化，避免几十个 agent 同时触发重编译，把机器和预算打爆。

后面的几个阶段，文章说人工判断会越来越少，流程越来越像消耗队列。先是编译。Jarred 用一个编排脚本在整个工作区上运行编译器，然后让修复 agent 并行处理错误列表，再由对抗式评审核查修复。然后重新构建，如此反复。这里也会暴露系统性问题。比如 Jarred 遇到了成千上万的 Rust 模块错误，其中一部分来自循环 import。Zig 的惰性编译可以容忍某些结构，Rust 模块系统则要求更明确。他的做法不是让 agent 对每个模块错误现场发挥，而是把依赖该删除、该移动、还是该重构边界的分类逻辑写进循环。

然后是冒烟测试。冒烟测试不是完整验证，而是先看程序能不能启动、关键路径会不会崩。它的真相来源是崩溃。文章说这里的循环修复也是把问题按根因分组，再交给对抗式子 agent 评审。比如十几个命令都崩在配置加载，不要开十几个工单分别修，要识别是配置路径规则、初始化顺序，还是错误处理契约出了问题。

最后是行为对齐，也就是新旧程序在外部行为上的一致性。此时文件已经翻译，能编译，也通过了冒烟测试。接下来要把测试切片，运行最开始建立的裁判测试套件。失败测试交给修复 agent，修复 agent 要对照新旧两个代码库分析，不能只盯着新代码看。对抗式评审继续检查修复是否引入了偏差。

文章还提到一个 build daemon，也就是构建守护进程。它是唯一允许重建二进制的进程。修复 agent 只写 patch，守护进程把 patch 批处理后统一重建一次，再重新运行受影响测试，把结果反馈回去。这个设计是为了把最昂贵的操作串行化，而不是让多个 agent 各自触发重建。对大型项目来说，这很现实。AI agent 多了以后，你会遇到一个新瓶颈，不是没有人修，而是太多人同时修、同时跑命令、同时消耗资源。构建守护进程就像交通警察。

如果同一个失败在很多测试里反复出现，修复要上移。修改产生 bug 的规则，只重新生成被这条规则影响的文件。这也呼应开头那句话，你不修代码，你修循环。

Mike 的方法对没有完整测试套件的团队尤其有启发。他让 Claude 创建一个小脚本，把七个真实世界场景分别跑在新版本和原 Python 版本上，然后 diff，也就是比较输出差异。每个失败场景都有自己的修复 agent，循环直到七个都通过。然后他又往前走了一步，让 Claude 自己设计端到端测试套件，连续四个晚上自主运行，修复出错的地方再重跑。文章说，这抓到了场景列表预想不到的小毛刺。这里的经验是，缺少测试套件不会阻塞行为对齐。如果你不能继承一个裁判，就让 Claude 帮你建一个。无论如何，原代码库都是 ground truth，也就是真相来源。

听到这里，你可能会问，如果我在公司里真要做，怎么落地？我会把文章的方法翻译成几个实践建议。

第一，先选一个足够痛但边界清楚的项目。不要一上来迁全公司最核心、最混乱、没有测试、没有 owner 的巨石系统。可以从一个内部工具、一个服务里的独立模块、一个长期被编译速度或类型问题折磨的包开始。文章里的案例很大，但方法可以缩小。关键是原系统可运行，新系统目标明确，有外部行为可以对比。

第二，先问商业账，再问技术爽不爽。你不能只说 Rust 更安全，TypeScript 更可维护，所以要迁。你要算一年里这个代码库花了多少时间在内存问题、构建等待、部署流水线、线上回归、招聘维护难度上。以前这笔账可能不够支撑四年迁移，现在也许足够支撑一个几周到几个月的 AI 辅助迁移。文章的说法很克制，迁移理由不必再是生死存亡，但仍然必须合理。

第三，裁判要早于代码。你可以先让 Claude 帮你读测试，分类哪些是外部行为，哪些绑定内部实现。再让它生成跨语言可跑的测试脚本，或者挑选真实命令、真实输入、真实数据，做输出 diff。然后故意制造坏版本，确认裁判能抓住。这个步骤很无聊，但它决定你后面能不能安心让 agent 自主跑很久。

第四，规则书要比 prompt 更像工程资产。很多人使用 AI 编程时，把关键约束写在一次性 prompt 里，过几天就丢了。大规模迁移不能这样。你要有一个规则文件，里面写类型映射、错误处理、日志策略、并发模型、资源释放、命名风格、模块边界、例外情况。每次评审发现系统性错误，都改规则书，而不是只改代码。规则书应该能被所有 agent 读取，也应该能被人审查。

第五，任务队列必须可恢复。不要让一个聊天会话承担所有状态。用脚本扫描目标目录，存在就是完成，不存在就是待处理。失败记录落盘，错误列表落盘，修复 patch 落盘。这样即使某个 agent 输出很差，你也能丢掉它那一批，重新生成，而不是在混乱上下文里继续补救。

第六，评审要对抗，验证要机械。对抗式评审的意思不是让 agent 吵架，而是让它们从找茬角度独立检查。一个实施 agent 往往会默认自己的翻译合理，评审 agent 必须拿着规则书和原代码挑刺。但最终谁说了算，最好不是另一个主观意见，而是编译器、测试、diff、崩溃日志这些机械信号。文章反复强调，模型在有客观 ground truth 时表现最好。

第七，不要迷信最大模型。最大的模型适合做规划、写规则、发现模式、评审高风险改动。但如果你要翻译几千个相似文件，用小一些、便宜一些的模型做实施更合理。把预算花在循环设计上，而不是每个 token 都堆最高规格。

第八，人工时间要前置。文章说规则书和压力测试最耗人，后面主要是队列燃烧。这一点和传统迁移很不同。传统迁移里人一直在改代码，越到后面越疲惫。AI 迁移里，人更像流程架构师和质量负责人。前期要想清楚怎么判断成功、怎么切分任务、怎么处理例外、怎么把错误上升为规则。后面你的工作是观察模式，而不是追着每个失败跑。

当然，也要避坑。第一个坑是盲目照抄文章流程。Anthropic 自己也说，不要盲从，每个迁移都不同。语言不同、构建系统不同、测试质量不同、业务容忍度不同，都会改变设计。你应该把这篇当起点，先和 Claude 一起规划你自己的迁移方案，再投入资源。

第二个坑是把百分之九十通过当成功。迁移项目最危险的就是看起来大体能跑，但边缘行为悄悄变了。尤其是工具链、CLI、编译器、支付、权限、数据处理这类系统，小差异可能会变成用户灾难。所以行为对齐阶段不能省。哪怕没有完整测试，也至少要构造真实场景，对比输出。

第三个坑是手工热修太多。一旦人开始大量绕开规则手改文件，迁移产物就会变成一锅粥。短期看修得快，长期看不可重现、不可审计、不可再跑。文章反复强调，发现模式就改规则，再重新生成受影响批次。这个纪律可能比模型能力还重要。

第四个坑是没有资源调度。很多 agent 同时跑构建、跑测试、读写同一批文件，会造成互相覆盖和资源浪费。你需要像传统分布式构建一样设计锁、队列、批处理、守护进程。AI agent 不是越多越好，它们需要交通规则。

文章最后回到 Bun 的结果。Bun 的迁移已经进入生产，但每次迁移都有取舍。比如 Rust 代码里大约百分之四位于 unsafe 块里，主要是和 C、C++ 边界相关的单行指针操作。也就是说，迁到 Rust 不等于所有不安全瞬间消失，但它把不安全区域显式圈了出来，让团队可以看见、审查和控制。

新代码库也有可测量收益。团队工具能检测到的内存泄漏都已经修复。有一个两千次重复构建的基准，内存从六千七百四十五 MB 降到六百零九 MB。Linux 和 Windows 上二进制体积小了百分之十九。跨语言优化还让 HTTP 服务、next build、tsc 这类真实工作负载快了百分之二到百分之五。注意这些不是抽象的代码洁癖，而是性能、体积、稳定性和维护性的综合收益。

我个人认为，这篇文章最有价值的地方，不是告诉我们 Claude 已经能写一百万行代码。写一百万行本身没那么稀奇，危险的是写出一百万行你无法验证的代码。它真正展示的是，AI 编程正在从个人生产力工具，变成流程自动化工具。未来工程师的核心能力之一，可能不只是会不会写某门语言，而是会不会设计一个让智能体稳定产出、稳定评审、稳定收敛的工程循环。

如果你正在维护一个多年不敢动的代码库，这篇文章给你的问题不是，明天要不要直接全量迁移。更现实的问题是，你能不能先为它建立裁判，写出第一版规则书，跑一次可丢弃的迷你迁移，看一看这笔账是不是已经变了。以前我们忍受历史包袱，是因为迁移太贵。现在，至少值得重新计算一次。

总结一下，Anthropic 这篇文章的核心是，大规模代码迁移不再只是多年大项目，而可以被拆成裁判、规则、队列、评审、编译、测试和行为对齐组成的循环。AI agent 的价值不在于替你一次性写对所有代码，而在于在清晰规则和机械验证下反复执行。我的点评是，真正的门槛会从会不会用 Claude，转向你的团队有没有能力把隐性工程经验写成规则，把质量判断变成可运行的裁判。
