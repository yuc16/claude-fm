我们先说说背景。金融行业是一个很特殊的行业,特殊在两点。第一,它高度依赖文字和数字工作,研究报告、尽职调查、承保审核、理赔处理、模型评审、月末结账,这些岗位的核心工作内容,说到底就是阅读大量文档、提取关键信息、做计算、再生成报告,这恰好是大语言模型最擅长的事情。第二,它又是一个极度看重合规和风险控制的行业,任何新技术想要进来,都要先过监管、审计、数据安全这几道关,所以金融机构在拥抱AI这件事上,往往是动作很大但落地很慢,口号喊得响,真正铺开却很谨慎。

我们可以想象一个具体的场景,帮大家感受一下这种工作量。一个并购团队的初级分析师,接到一个尽职调查任务,对方公司给过来几百页的合同、财务报表、专利清单、诉讼记录,分析师要做的事情是把这些材料一份一份打开,找出里面的风险点,比如有没有没披露的诉讼、有没有异常的关联交易,然后把发现汇总成一份给合伙人看的备忘录。这种工作传统上要花几天甚至一两周,而且大部分时间不是花在分析判断上,而是花在翻文件、找信息、整理格式上。这个例子是我自己举的,用来说明这类岗位的工作形态,不是原文里的具体案例,但它能帮我们理解,为什么金融行业会被认为是大语言模型特别合适的落地场景。

正是在这样的背景下,Anthropic 面对的不是一个简单的问题,而是一个组合问题。金融机构想问的不是我们要不要用 Claude,而是我们有六七种不同的 Claude 产品,我们的研究团队、交易团队、合规团队、财务团队,各自应该用哪个,先从哪个开始,怎么循序渐进地推开。这篇文章想解决的就是这个问题,它给出的不是一个单点的功能介绍,而是一张产品地图加一份落地剧本。

文章开头就提到,现在金融行业里,机构已经在用 Claude 做研究、做交易撮合相关的工作、做承保、处理理赔、做模型评审,还有月末结账,而且关键的是,大部分机构不是只用一个 Claude 产品在做这些事,而是同时在用好几个产品,各自负责不同场景。

那么这几个产品具体是什么,各自负责什么,我们一个个来看,这是整篇文章最核心的一块内容。

第一个是 Claude 本身,也就是我们最熟悉的聊天和研究界面。在金融场景里,它的角色更像是一个随时待命的研究顾问。你把一份招股书丢给它,让它帮你提炼风险点,你把一段监管条文丢给它,让它帮你解读合规要求,这种一问一答、即问即答的场景,就是 Claude 聊天产品的主战场。

第二个是 Claude Cowork,这是一个项目级的协作工具,关键区别在于它不是单轮问答,而是能跨文件、跨应用地持续工作。打个比方,如果普通聊天像是你随时可以打电话问的一个顾问,那 Cowork 就更像是给你配了一个项目助理,这个助理可以同时打开你电脑里的多个文件夹,翻阅好几份合同,对照一份模型文档,再把结果汇总成一份报告,整个过程跨越好几个步骤、好几种文件格式,但你只需要交代一次任务方向,后面的执行它自己去串联。这对于尽职调查这种本身就要翻阅大量材料、横跨多个文档的工作特别合适。我们可以拿前面那个并购尽职调查的例子接着推演一下,如果团队用 Cowork 来做这件事,大概的工作方式会变成,分析师把几百页材料对应的文件夹指给Cowork,告诉它要重点排查诉讼风险和关联交易,Cowork 会自己去逐份打开文件,标记出可疑条款,再把这些条款连同出处整理成一份结构化的备忘录草稿,分析师拿到的不再是一堆原始文件,而是一份已经筛选过、带着引用出处的初步结论,他只需要去复核关键判断,而不用从头翻一遍几百页材料。这个推演是我自己根据Cowork的能力做的设想,不是原文里写出来的具体案例。

第三个是 Claude Code,这是给工程团队、量化团队用的命令行工具。在金融机构里,有大量的量化研究员和工程师本身就是写代码的人,他们需要的不是一个聊天框,而是一个能直接读写代码仓库、能跑脚本、能调试模型的助手。Claude Code 就是补上这一块,它面向的是那些每天泡在代码和数据管道里的人。

第四个是 Claude for Microsoft 三六五,这是嵌入到 Excel、PowerPoint、Word 和 Outlook 里的插件。这个产品看起来不起眼,但其实切中了金融行业最真实的工作场景,因为不管模型多先进,金融从业者每天打开的软件,还是这四个。分析师做财务模型在Excel里,投资经理做提案在PowerPoint里,律师和合规写文件在Word里,所有人每天处理邮件在Outlook里。把 Claude 直接做成这几个软件里的常驻助手,意味着不需要切换工具、不需要复制粘贴,大幅降低了使用门槛。你可以想象一下,一个财富管理顾问每天早上打开Outlook,邮箱里堆着几十封客户邮件,有的在问市场最近的波动,有的在要一份资产配置的更新建议,如果没有这类插件,顾问要先读邮件理解需求,再切换到别的工具去查数据、写分析,最后再切回邮件去回复,中间的切换本身就消耗大量时间。如果Claude直接嵌在Outlook里,顾问可以让它先帮忙把邮件分类,对需要数据支撑的问题直接在邮件草稿里生成一版回复,顾问再做修改确认就发出去,省掉的是那些来回切换工具的碎片时间。这个场景是我自己设想的,用来说明这个产品为什么重要,原文本身没有展开这么具体的画面。

第五和第六个,是 Claude Platform 和 Claude Managed Agents,这两个面向的是想要自己搭建定制化应用和智能体的团队。如果说前面几个产品是给你现成的工具去用,这两个就是给你一套工具箱和半成品,让你的工程团队按照自己机构的具体需求,搭建专属的智能体和应用,Managed Agents 还包含了托管运维的部分,意味着搭好之后机构不需要自己操心底层的运行维护。打个比方,如果一家保险公司觉得现成的理赔agent模板不够贴合自己的核保规则,想要一个完全按照自家产品条款、自家风控阈值定制的理赔审核agent,他们就可以用Claude Platform把这套定制逻辑搭出来,再交给Managed Agents去托管运行,公司自己的工程团队不需要再操心服务器扩容、模型版本升级这些底层运维细节。这个例子同样是我自己设想的应用场景,帮助理解这两个产品定位上的差异,不是原文给出的具体案例。

把这六个产品摆在一起看,你会发现这条产品线背后的逻辑其实很清楚,它不是想用一个万能产品去覆盖金融机构所有的需求,而是按照问答深度、协作复杂度、是否需要写代码、是否嵌入现有办公软件、是否需要定制化这几个维度,把产品线铺成了一张网,不同岗位的人,根据自己日常工作的形态,去对应不同的产品入口。这也是这篇文章里专门给出一张产品矩阵的原因,目的就是帮决策者照着自己机构里不同团队的工作方式,把人和产品对上号。

第二个值得深挖的点,是文章里提到的十个预制金融agent模板。这里的agent,你可以理解成一个能自主完成多步骤任务的智能体程序,它不是只能问一句答一句的聊天机器人,而是可以自己规划步骤、调用工具、持续推进一件事情直到完成。这些模板的意思是,Anthropic 不只是给金融机构一个通用模型,而是已经针对行业里几类高频、重复性强的工作,提前搭好了一批可以直接拿来用的agent,机构拿过去之后只需要按自己的业务再做一些微调,而不用从零开始设计。

原文具体点名了五类,我们一个一个拆开讲。

第一类叫提案构建agent,这是投行和私募股权团队会用到的场景。做并购或者融资项目的时候,团队经常需要快速做出一份给客户看的提案材料,里面包含行业分析、对标公司、估值区间、交易结构建议,这些内容传统上要分析师花上好几天时间手工拼,有了这个agent,大量的资料检索和初稿生成工作可以先交给它,人再去做判断和打磨。如果让我具体设想这个agent的工作步骤,大概会是,分析师先告诉它目标客户是谁、交易的大致方向是什么,agent去检索这个客户所在行业最近的财报和新闻,找出几家最相关的对标公司,再根据这些公司的估值倍数,推算出一个合理的估值区间,最后把行业背景、对标分析、估值区间、可能的交易结构,直接排版成一份PowerPoint草稿。分析师拿到的不是一张白纸,而是一份已经有骨架和数据支撑的提案,他要做的是核对数字、调整措辞、加上自己对客户关系的判断。这个步骤拆解是我自己的推测,原文只用一句话点了这个模板的名字,没有展开具体流程。

第二类是市场研究员agent,顾名思义,负责做行业和市场层面的研究,持续追踪某个赛道的新闻、财报、政策变化,帮研究团队把信息搜集和初步整理这部分工作做掉。

第三类是 KYC 审查agent,KYC 的全称是客户身份核查,这是银行和金融机构在接入新客户之前必须完成的合规流程,要核实客户身份、资金来源、是否存在洗钱或者制裁风险。这个流程本身高度依赖查阅各种文件和数据库,正好是agent可以大幅提速的环节。具体来说,一个新客户开户的时候,合规人员需要核对身份证件、查公司注册信息、比对制裁名单、查有没有负面新闻,这些信息分散在不同的系统和数据库里,人工查一遍往往要跨好几个系统反复登录、反复搜索。KYC审查agent可以做的事情,是把这些查询动作自动跑一遍,把每一项核查的结果和依据整理成一份清单,合规人员看到的是一份已经标好风险等级的报告,只需要对高风险或者有疑点的部分做人工复核,而不用每一项都从零查起。这个流程描述是我根据KYC业务的常规做法做的推测,原文没有展开具体步骤。

第四类是总账对账agent,这个对应的是财务部门最头疼的工作之一,把不同系统、不同账本里的数字一笔一笔核对清楚,确保账目一致,这种工作枯燥、重复,又特别容不得错,是agent非常适合介入的场景。

第五类是月结agent,对应的是每个月末财务团队最忙的那几天,需要把当月所有的交易、调整、报表汇总到一起,完成月度结账。这个场景的特点是流程固定、步骤明确,但是涉及的数据来源又特别多,agent可以负责把固定的流程跑通,人再去处理例外情况。月结这件事,通常涉及好几个系统的数据,销售系统、采购系统、银行流水,要把它们汇总对齐,确认每一笔收入和支出都记在正确的会计期间,还要核对有没有漏记或者重复记的条目。一旦哪个数字对不上,财务人员就要一笔一笔倒查,经常通宵加班。月结agent可以做的事情,是先把固定的汇总和核对步骤跑一遍,把所有对得上的部分确认完,只把对不上的那一小部分异常挑出来,交给财务人员去人工排查,这样财务人员的精力就能集中在真正需要判断的少数问题上,而不是把时间都耗在大量本身没有问题的核对工作上。这部分流程描述同样是我根据财务月结的常规做法做的推测。

我个人觉得,这种点名客户但不展开细节的写法,在企业软件的营销内容里很常见,目的往往不是教你怎么做,而是用真实的大公司名字给读者一个信号,告诉你这件事已经不是实验室里的概念,而是有保险公司、银行、交易平台、评级机构这几个不同类型的真实机构在用。至于具体怎么用、效果如何,通常要看完整指南或者后续的案例研究才会披露,这篇短文本身的作用更像是一个引子。

第三个值得展开的点,是文章里提到的三阶段落地剧本,分别是基础阶段、试点阶段和扩展阶段。原文提到每个阶段都配有具体的行动建议和一些小技巧,但这篇短文里没有把这些行动细节和技巧逐条列出来,具体内容应该是在完整指南里。不过这个三阶段的框架本身,其实是企业落地AI项目时一个很常见的思路,我们可以借着这个框架,结合行业里普遍的做法,讲讲每个阶段大概是在做什么,这部分更多是我个人根据行业经验做的展开,不完全是原文内容。

基础阶段,一般对应的是把地基打好,包括把数据和系统接通,确定哪些数据可以给AI用、哪些涉及敏感信息需要先做处理,同时明确内部的治理规则,比如谁有权限审批agent上线,出了问题责任怎么界定。这一步往往是最容易被忽略,但又最容易在后期出大问题的环节,尤其是金融行业,监管要求多,基础阶段如果偷懒,后面铺开的时候会处处受限。我们可以设想一家银行做这件事的样子,在基础阶段,IT部门要先梳理清楚客户数据、交易数据分别存在哪些系统里,哪些数据涉及个人隐私不能直接喂给模型,哪些可以经过脱敏处理后使用,同时合规部门要参与进来,明确agent输出的内容要不要人工复核、复核到什么程度才能对外使用。这一步做得扎实,后面试点的时候才不会因为数据没接通或者权限没理清而被迫停下来重新返工。这个银行的设想场景是我自己举例,不是原文内容。

试点阶段,一般是选一两个高价值又风险相对可控的场景先跑起来,比如前面说的KYC审查或者市场研究,用真实的业务数据小范围测试,收集一线员工的反馈,验证准确率和效率提升到底有多大,同时摸清楚哪些环节AI做得好,哪些环节还需要人工兜底。这一步的关键往往不是技术能不能跑通,而是能不能拿到让管理层和监管都认可的效果证据。

扩展阶段,就是把试点验证过的场景,推广到更多团队、更多分支机构,这个阶段考验的已经不是单点效果,而是组织能力,包括怎么培训大批员工去配合agent工作,怎么建立持续监控机制防止模型出错或者出现偏差,怎么把成功经验在不同部门之间复制。

这三个阶段背后的逻辑其实很朴素,先把风险控制住,再小范围验证价值,最后才考虑规模化,这也是为什么金融这种强监管行业推AI,往往比互联网行业要慢一拍,但一旦推开,因为前面打的地基扎实,后劲反而可能更足,这句关于后劲更足的判断是我个人的看法。

文章最后还提到,这份指南把用例按照三个板块分类,分别是投资银行和私募股权,财富管理和资产管理,以及零售和商业银行。

这个分类方式其实也呼应了我们前面讲到的背景,投资银行和私募股权这一块,核心工作是研究和交易撮合,对应的更可能是提案构建和市场研究员这类agent场景。财富管理和资产管理这一块,日常工作更偏向给客户做资产配置建议、做投资组合分析,需要持续追踪市场变化,这块的需求更接近研究和报告生成。零售和商业银行这一块,业务量最大、客户基数最广,核心痛点是大批量、重复性的客户服务和合规审查,前面说到的KYC审查、月结、总账对账这些,大概率是落在这个板块里。原文只是给出了这三个板块的分类,没有详细展开每个板块下具体配哪些用例,刚才这部分对应关系,也是我根据各板块业务特点做的合理推断。

讲完了文章的核心内容,我们来聊聊这对听众,也就是对我们这些做技术或者在金融机构里负责AI落地的人,到底意味着什么。

如果你是金融机构里负责AI战略的决策者,这篇文章最大的价值,其实是给了你一张可以照着对话的地图。下次开会讨论要不要上AI的时候,你可以把讨论拆解成更具体的问题,我们公司的研究团队应该先用哪个产品,我们的财务团队适合从哪个agent模板切入,我们现在处在基础阶段还是已经可以试点了。这种结构化的提问方式,比泛泛地讨论要不要拥抱AI要有效得多。

如果你是合规或者风控团队的负责人,我建议你换一个角度去看这篇文章,重点不是这些产品能做什么,而是基础阶段提到的数据治理和权限审批,这部分恰恰是你最该提前介入的地方。很多AI项目推不动,卡的不是技术,而是合规部门在试点快要上线的时候才被拉进来评审,结果发现一堆问题要返工。如果你能在基础阶段就参与进去,提前定好哪些场景的输出必须有人复核、复核标准是什么、出错了责任怎么界定,反而能让后面的试点和扩展跑得更快,因为大家不用在每个新场景上都重新吵一遍这些问题。这部分建议是我自己根据金融行业AI落地的常见经验给出的,不是原文内容。

如果你是工程师,具体负责把这些产品落地到自己公司,我会建议你重点关注两件事。第一,先搞清楚你们公司里不同岗位的人,日常工作形态到底是什么样的,是大量在Excel和PPT里干活,还是本身就在写代码,还是需要跨多个文档做综合判断,这决定了你应该先从哪个产品切入,而不是一上来就想着把所有产品都铺开。第二,优先选择那些规则相对明确、重复性强、出错代价可控的场景去做试点,比如总账对账或者月结这类工作,流程本身是固定的,即使agent一开始做得不够完美,也比较容易人工核查纠正,这种场景比一上来就啃KYC审查或者承保决策这种高风险场景,更容易快速看到正向反馈,也更容易说服管理层继续投入。

要避的坑,我觉得有三个比较明显。第一是不要指望一个产品打天下,文章里反复强调的是,真正在用Claude的机构,基本都是同时在用好几个产品,各自负责不同场景,如果你想着找一个万能入口解决所有团队的需求,大概率会卡在某个团队的具体工作流上。第二是别跳过基础阶段直接冲规模化,金融行业的监管和审计要求,决定了数据治理和权限管理这些看起来枯燥的基础工作,一旦在试点阶段就没做扎实,后面想要规模化推广的时候,大概率会被合规和风控部门挡下来,返工的成本比一开始就做扎实要高得多。第三是别选错试点场景,有些团队为了快速出成果,一上来就挑了承保决策或者大额理赔这种本身风险就很高、出错代价很大的场景去试点,结果一旦agent判断有偏差,整个项目的信任度会受到很大打击,反而比那些先从总账对账、月结这种容错性更高的场景切入的团队走得更慢。这三点判断,也是我自己结合文章内容和金融行业落地AI项目的常见经验给出的,不完全是原文逐句的内容。

另外值得一提的是,文章里专门强调这份指南既是给决定 Claude 放在公司里哪个位置的AI负责人看的,也是给真正动手落地的工程师看的,这个双重定位其实挺值得我们学习,任何技术导入,如果只有战略层认可而执行层没有清楚的操作手册,或者反过来只有工程师在推但拿不到管理层的资源支持,都很难真正落地,这篇指南想同时服务这两类读者,本身就是个值得参考的做法。

整体回顾一下,这篇文章虽然篇幅不长,但传递的信息很扎实,它告诉我们,金融行业现在用Claude,已经不是单点的聊天问答这么简单,而是一整套产品矩阵加上现成的行业模板,配合一份从地基到规模化的落地剧本。从AIG、澳大利亚联邦银行、IG Group、穆迪这几个客户名字也能看出,这件事已经不是停留在概念阶段,而是真的在保险、银行、交易、评级这几个不同的金融细分领域里跑起来了。

我个人的感受是,这篇文章最有价值的地方,不是某个具体的技术细节,而是它把一个原本听起来很模糊的问题,金融机构怎么用AI,拆解成了一张可以照着执行的清单。对于金融行业这种本身节奏偏保守、决策链条长的领域来说,有这样一份结构清楚的路线图,可能比单纯炫技更重要。如果你所在的机构也在考虑这件事,不妨就从文章里提到的产品矩阵和那五个已知的agent模板开始,找一个风险可控的场景先跑起来,这大概是眼下最稳妥的一条路。
