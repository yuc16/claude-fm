大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是 Using Claude Code, The unreasonable effectiveness of HTML,翻译过来就是,使用 Claude Code,HTML 出人意料的强大效果。这篇文章发表于二〇二六年五月二十日,作者是 Claude Code 团队的工程师 Thariq Shihipar,他在文章里特别说明,这篇文章表达的是他个人的观点,以及他对用 HTML 文件配合 Claude Code 工作这件事的偏爱。

先说说为什么这篇文章值得花时间听一听。这篇文章讨论的不是某个新模型、新功能,而是一个特别朴素却又特别实用的工作习惯问题,就是你和 Claude Code 之间,到底应该用什么样的文件格式来交流。我们大部分人,包括我自己在做技术写作和文档整理的时候,第一反应都是用 Markdown,因为它简单、轻量、纯文本、方便用 Git 做版本管理。但作者提出了一个挺颠覆的观点,他说随着 Claude 这样的智能体变得越来越强,Markdown 这种格式反而成了一个瓶颈,他现在几乎已经完全转向用 HTML 文件来跟 Claude Code 协作,生成计划、报告、设计稿、代码评审说明等等。这背后其实反映的是一个更大的趋势,就是当 AI 写代码、写文档的能力越来越强之后,人和 AI 之间的协作界面也需要跟着进化,而不是停留在过去那种 AI 帮我生成一段文字的简单模式上。

我们先讲讲背景。在过去几年里,Markdown 几乎成了智能体跟人类沟通的标准格式,这一点你应该很熟悉,不管是 Claude、还是其他大模型工具,生成的文档、回复、说明,十有八九都是 Markdown 格式,用井号表示标题,用星号表示加粗,用短横线列列表。这个格式确实有它的优点,简单、容易编辑、人和机器都看得懂,而且 Claude 在用 ASCII 字符画图这件事上,已经做得相当让人惊讶了,作者在文章里也提到了这一点,说 Claude 甚至能用文本字符画出还算清晰的示意图。

但问题是,随着 Claude Code 能处理的任务越来越复杂,它生成的文档篇幅也越来越长,内容也越来越丰富。作者举了一个挺扎心的例子,他说他自己已经很难读完一份超过一百行的 Markdown 文件了,更别说让团队里其他人去读。你可以想象一下那种场景,Claude 给你生成了一份两百行的实现方案,里面夹杂着标题、加粗文字、列表、代码块,你打开一看,密密麻麻的纯文本扑面而来,没有颜色区分,没有视觉层次,你的眼睛根本不知道该往哪儿看,于是你可能就扫一眼标题,大概浏览一下,然后心里想着,算了,先让它做着,有问题再说。这其实是一个挺危险的状态,因为你本来是想通过看计划来确认 Claude 的方向对不对,结果因为格式太枯燥,你反而放弃了仔细审查,这就跟在场感这件事直接相关了,这也是这篇文章后面会重点讲到的一个概念。

那作者给出的解决方案是什么呢,就是用 HTML 代替 Markdown。这听起来好像挺简单,无非就是换个文件后缀嘛,但作者想表达的核心,其实是 HTML 背后那种表达能力的质变。我们花点时间拆开讲一讲,我觉得文章里至少有三个核心论点特别值得展开,第一个是信息表达能力的差距,第二个是可读性和分享性的差距,第三个是 HTML 作为可交互画布这件事带来的全新工作方式。

先说第一点,信息表达能力。Markdown 本质上是一种排版语言,它能做的事情很有限,标题、加粗、斜体、列表、引用、简单表格、代码块,基本就这些。但 HTML 能做的事情完全是另一个维度的,你可以用表格标签做真正复杂的表格,可以用 CSS 来控制颜色、间距、布局,可以用 SVG 矢量图来画插图和流程图,可以用脚本标签嵌入代码片段甚至小工具,可以用 HTML 元素加上 JavaScript 和 CSS 做出真正可以点击、可以交互的界面,可以用绝对定位和画布元素来表达空间关系,还可以直接嵌入图片。作者说了一句话我觉得特别到位,他说在他看来,几乎不存在 Claude 能读懂的信息,是不能用 HTML 高效表达出来的。这句话听起来有点绝对,但你仔细想想,确实是这样,HTML 本质上就是现代浏览器渲染引擎的输入语言,而浏览器几乎可以呈现任何视觉形式的信息,文字、图片、矢量图形、动画、交互控件,样样都行。

反过来想,如果你只能用 Markdown,但又想表达一些复杂的信息,会发生什么呢。作者提到一个挺有意思的现象,他说 Claude 在 Markdown 的限制下,会做一些他形容为低效变通的事情,比方说用 ASCII 字符拼出示意图,或者更夸张的,用 Unicode 字符去估算和模拟颜色,你可以想象一下那个画面,Claude 想跟你表达这是一个从绿色渐变到红色的状态条,但因为 Markdown 没有颜色这个概念,它只能憋出一堆色块符号去拼凑视觉效果,这其实是一种被格式逼出来的奇技淫巧,本质上说明这个格式已经不太够用了。

第二个核心论点是可读性和分享性。这两者其实是一体的。可读性这块,作者强调的是,HTML 文档可以把内容按照视觉结构去组织,比如说用标签页来分隔不同模块,用插图辅助理解,用链接做跳转和交叉引用,甚至可以做成移动端自适应的样式,根据你是用手机看还是用电脑看,调整不同的呈现方式。这就跟一份纯文本文件完全不是一个量级的体验了。你可以类比一下,看一份学术论文的纯文本草稿,和看一份排版精美、配图清晰的期刊文章,信息量可能是一样的,但你愿意花时间去仔细读的概率天差地别。

分享性这一点,我觉得是很多人容易忽略,但其实特别实际的一个优势。Markdown 文件这东西,大部分浏览器是没办法直接友好渲染的,你发给同事一个 Markdown 文件,对方很可能要把它当附件下载下来,用某个编辑器或者插件打开才能看,体验是断裂的。但 HTML 文件不一样,只要你把它上传到某个地方,生成一个链接,你的同事点开链接,直接就能在浏览器里看到排版完整的内容,随时随地,不挑设备。作者讲了一句话我印象很深,他说,如果你的方案文档、报告、PR 说明是用 HTML 写的,那别人真正去读它的概率会大大提高。这句话听起来很朴素,但其实戳中了一个特别现实的问题,就是我们写了很多文档,但真正被认真读过的比例可能没有我们想象得那么高,而格式本身,其实是影响别人愿不愿意读的一个重要变量。

第三个核心论点,也是我觉得这篇文章里最有意思的一个延伸,就是 HTML 可以变成一个可交互的画布,而不只是一份静态文档。作者举了个例子,比如说你在调一个设计上的动画效果,你可以让 Claude 直接做一个带滑块、带旋钮的 HTML 页面,让你自己去拖动滑块调整参数,实时看到动画效果的变化,调到满意了,再让它给你一个复制按钮,把这次调出来的参数复制成文字,粘贴回 Claude Code 里,告诉它就用这组参数去改代码。这个工作流我觉得特别精妙,它解决了一个长期存在的沟通难题,就是有些东西真的很难用语言描述清楚,比如说一个缓动曲线该多陡,一个紫色该多深,一个动画该多快,这些东西你用文字去描述,要么很啰嗦,要么不准确,但如果你能亲手拖一拖滑块,效果立刻摆在眼前,这种反馈循环就比纯靠文字交流快得多,也准得多。

作者还提到了一个很关键的背景信息,就是为什么是用 Claude Code 来生成这些 HTML 文件,而不是用 Claude 网页版或者专门的设计工具。原因是上下文,Claude Code 可以直接读取你本地的代码文件夹,读取你的 Git 历史记录,甚至通过 MCP 这种协议连接到你的 Slack、Linear 这些工具,还可以借助 Claude 在 Chrome 里的浏览器插件去抓取网页信息。这意味着 Claude Code 生成的 HTML 文件,背后是有大量真实上下文支撑的,不是凭空生成的好看页面,而是真正基于你项目的实际情况整理出来的内容。作者自己写这篇文章的时候就用了这个方法,他让 Claude Code 去扫描他本地文件夹里所有生成过的 HTML 文件,把它们分类整理,然后再做一份带示意图的 HTML 文件,把每一类用法都画出来,文章里配的那些示意图,就是这么来的。这其实也是对文章本身的一个挺有意思的自我证明,作者是在用自己提倡的方法去写这篇提倡这个方法的文章。

讲到这里,我想强调一下,作者也特别提到一点,你不需要做什么复杂的准备工作才能让 Claude 生成这种 HTML 文件,你只需要在提示词里直接说,帮我做一个 HTML 文件,或者帮我做一个 HTML artifact,就是这么简单。真正的难点不在于怎么让 Claude 生成 HTML,而在于你自己要想清楚,你想让这份文件承担什么功能,你打算怎么用它。作者建议,如果你发现某个用法你反复在用,那时候再考虑把它沉淀成一个固定的技能或者模板,但一开始,从零开始去试,去摸索不同场景下怎么用,才是更好的学习方式。

接下来我想具体讲讲文章里列出的几个典型使用场景,这部分我觉得对大家实际工作特别有参考价值。

第一个场景是探索和规划。作者说,当他开始做一个新问题的时候,他现在已经不会简单地让 Claude 写一份 Markdown 计划了,而是会期待生成一整个由多份 HTML 文件组成的网络。具体流程大概是这样,先让 Claude 头脑风暴,生成几种不同的方案,做成探索性的页面,然后挑一个方向深入,做出界面的模拟稿或者示例,最后觉得方向定了,再让它写一份正式的实现计划。等这些文件都齐备了,他会开一个新的会话,把所有这些文件一起传进去,让 Claude 基于这些上下文去做真正的实现。等到要验证结果的时候,他还会让验证用的智能体也读一遍这些文件,这样验证的时候上下文会更完整,判断也会更准。

这里有个具体的提示词例子我觉得很实用,作者写道,他会跟 Claude 说,我还没想清楚引导页该往哪个方向做,帮我生成六种风格明显不同的方案,在布局、语气、信息密度上都要有差异,然后把它们排成一个网格放进一份 HTML 文件里,方便我并排对比,每个方案都标注一下它做了什么取舍。你可以想象一下,这种一次性看六个方案、还带着取舍标注的体验,跟你一条一条读六段文字描述,完全不是一个效率级别。

第二个场景是代码评审和 PR 讲解。这个场景我觉得特别贴近我们平时的工作。作者说,代码在 Markdown 文件里其实很难读,但用 HTML,你可以渲染出带行内注释的差异对比,可以画流程图来表示模块关系,可以用颜色去标注问题的严重程度。文章里给的提示词例子是这样的,帮我评审这个 PR,做一份 HTML 文件来描述它,我对其中流式处理和反压这部分逻辑不太熟,重点讲这部分,把实际的代码差异渲染出来,在旁边加注释,按严重程度给发现的问题上色,需要别的辅助手段也尽量用上。你想象一下,平时你评审一个复杂 PR,如果对方只甩给你一段纯文字描述,你大概很难建立起完整的心理模型,但如果有一份带颜色标注、带流程图的页面,你扫一眼颜色分布,大概就知道这个 PR 风险点集中在哪儿,这种视觉化的信息密度,对评审效率的提升是实实在在的。

第三个场景是设计原型和交互调参,这个跟 Claude Design 这个产品背后的逻辑是相通的。作者解释说,Claude Design 选择用 HTML 作为底层,正是因为 HTML 在表达设计这件事上特别强,即便你最终的目标平台不是网页,而是用 React、用 Swift 之类的原生技术,Claude 依然可以先在 HTML 里把设计构思勾勒出来,验证好了再翻译成目标语言。这里的提示词例子是,他想做一个新的结账按钮原型,点击之后先播放一段动画,然后迅速变成紫色,他让 Claude 做一份带很多滑块和选项的 HTML 文件,方便他试验不同的动画参数,试到满意了,再给一个复制按钮,把调好的参数复制出来。这跟我前面讲的那个滑块调参的例子是同一个思路,核心都是把不可言说的感觉,转化成可以拖动、可以试错、最后可以导出的具体数值。

第四个场景是综合报告和讲解类内容。Claude Code 一个很强的能力,是可以同时从多个数据源去整合信息,比如说同时去读你的 Slack 聊天记录、读你的代码库、读 Git 提交历史,甚至去搜索互联网上的资料,然后把这些信息汇总成一份容易读的报告。这种报告可以做成长篇的 HTML 文档,也可以做成交互式讲解页面,甚至可以做成幻灯片式的演示文稿。作者建议可以让 Claude 用 SVG 矢量图来画示意图,辅助理解。文章给的例子是,有人说他一直没搞懂团队里那个限流器具体是怎么运作的,他让 Claude 去读相关代码,产出一份单页的 HTML 讲解,里面要有一张表示令牌桶算法流程的示意图,挑出三到四段关键代码加上注释,最后再加一个常见坑的小节,整体是为了让人一遍读完就能搞懂,而不是需要反复回看。这个场景我觉得特别适合用在团队内部的技术分享、新人上手文档、还有事故复盘报告上面。

第五个场景,可能是这篇文章里最有创意的一个用法,作者把它叫做一次性的专用编辑器。意思是,有时候你心里想要的东西,真的很难用一段文字描述清楚,这时候作者会让 Claude 直接给他做一个临时的、专门针对眼前这一件事的小型编辑界面,这个界面不是要做成什么正式产品,也不是要长期维护的工具,就是为了解决这一件具体的事情而临时存在的。这里的关键技巧,是一定要让这个界面最后能导出结果,比如说一个复制为 JSON 或者复制为提示词的按钮,把你在界面上拖拽、勾选、调整的结果,转换成可以直接粘贴回 Claude Code,或者直接提交进文件的内容。这样你既能享受图形界面操作的直观,又不会脱离跟 Claude 协作的主线。

作者给了三个具体例子,我觉得都很有启发性。第一个是,他要重新给三十个 Linear 工单排优先级,他让 Claude 做一份 HTML 文件,把每个工单做成一张可以拖拽的卡片,分成现在、接下来、以后、取消这四个列,先按 Claude 自己的判断预排序好,然后加一个复制为 Markdown 的按钮,导出最终排序结果,每个分类再附一句话理由。第二个例子是关于功能开关配置的,他把功能开关的配置文件给 Claude,让它做一个基于表单的编辑器,按区域给开关分组,显示开关之间的依赖关系,如果他要打开一个开关,但它依赖的前置开关还没打开,就提醒他,最后加一个复制差异的按钮,只导出改动过的那部分配置。第三个例子是关于调试系统提示词的,他想做一个左右对比的编辑器,左边是可编辑的提示词,里面的变量插槎要高亮显示,右边放三个示例输入,实时根据左边的模板渲染出填充后的效果,还要加一个字符和 token 计数器,以及复制按钮。

这三个例子有个共同点,就是它们处理的都是那种结构清楚、但用纯文字操作很痛苦的任务,排序、分组、打勾、判断依赖关系、对比预览,这些事情如果靠你在聊天框里一句一句描述,效率会很低,而且很容易出错,但如果你有一个为这件事量身定做的小工具,哪怕这工具只用一次就扔掉,效率提升也是非常明显的。

作者还总结了这类用法适用的范围,大概包括给任何东西做排序、分类或者归堆,像工单、测试用例、反馈意见这些,编辑有结构、有约束的配置,像功能开关、环境变量、有格式要求的 JSON 或者 YAML,调试提示词、模板或者文案,并且要带实时预览,整理数据集,比如说挑选或者剔除某些样本,给样本打标签,再导出选中的结果,给文档、转录稿或者代码差异做批注,再导出批注内容,还有就是那些用文字表达起来特别费劲的取值,比如颜色、缓动曲线、裁剪区域、定时任务的时间表达式,或者正则表达式。

文章后半段,作者还回答了几个他经常被问到的问题,我觉得也值得讲一讲,因为这些问题可能也是你心里正在冒出来的疑问。

第一个问题是,这样做不是更没效率吗,毕竟 HTML 比 Markdown 要写更多字符,消耗更多 token。作者的回答是,虽然 Markdown 确实更省 token,但 HTML 带来的表达力提升,以及他自己真正去读这些内容的概率提升,带来的整体收益是更高的。而且他特别提到,现在 Opus 四点七模型已经支持一百万 token 的上下文窗口,在这么大的窗口下,HTML 多消耗的那点 token,基本感觉不出来。这其实也提醒我们,评估一个工作流是不是高效,不能只看 token 这个单一指标,还要看你最终有没有真正用上这些产出,如果生成了一堆 Markdown,但因为太难读你压根没仔细看,那省下来的 token 其实是被浪费掉的。

第二个问题是,那你现在还用 Markdown 吗。作者说他几乎已经完全不用了,他自己承认自己是偏向 HTML 极端派的那种人,大部分场景几乎都已经切换过去了。

第三个问题是,这种做法算不算是取代了写计划这件事。作者的回答挺有意思,他说与其说有一份计划,不如说他现在手头通常会有好几份针对不同阶段、不同部分的 HTML 文件,比如说一份实现计划,一份界面探索,还有一份汇总所有设计方案的列表组件,这些文件他会保留下来,作为以后查阅的参考资料,也会在做验证的时候重新拿出来用。

最后,作者说出了我觉得整篇文章里最关键的一句话,他说他真正改用 HTML 的原因,其实不是为了视觉效果多漂亮,而是因为这样能让他感觉自己跟 Claude 之间的协作更在场。他提到一个挺真实的体验,就是随着 Claude 能承担的工作越来越多,他发现自己反而越来越少认真读那些计划了,他想找一种方式,让自己重新参与到 Claude 做的每一个选择里,而不只是把任务甩出去,然后被动等结果。HTML 恰好满足了这个需求,他说他现在感觉比以前任何时候都更投入,更在场。

好,那我们现在来说说,这篇文章对我们普通听众,具体在实际工作里能怎么用。

首先,最简单能立刻上手的一件事,就是你下次让 Claude Code 写一份比较长的方案、报告或者评审说明的时候,直接在提示词里加一句,帮我做成一个 HTML 文件,而不是 Markdown。这个改动几乎零成本,Claude Code 本身就支持直接生成这样的内容,你不需要额外学什么新工具。如果你平时已经经常让 Claude 写需求文档、写技术方案,这个小习惯的改变,可能会明显提升你和团队成员真正去读这些文档的概率。

其次,如果你所在团队经常需要做代码评审,尤其是评审一些逻辑复杂、外人不容易看懂的模块,你可以试试作者那个 PR 讲解的用法,让 Claude 专门针对你不熟悉、或者你觉得评审者可能不熟悉的那部分逻辑,做一份带颜色标注、带流程图的讲解页面。我个人认为,这个用法特别适合那种跨团队评审的场景,比如说你的 PR 涉及到另一个团队负责的模块,对方对你的代码上下文了解有限,一份这样的讲解页面,可能比你口头解释半小时还管用。

第三,如果你的工作里经常涉及设计相关的微调,比如说动画效果、交互细节、视觉参数这些很难用语言精确描述的东西,可以试试让 Claude 给你做一个带滑块的临时调参界面。这个用法的好处是,你不需要去学专门的设计软件,也不需要跟设计师反复来回沟通细节,你自己动手调,调好了直接导出参数,继续推进开发。

第四,如果你经常要处理一堆杂乱的工单、反馈,或者配置项,需要排序、分类、打标签,你可以借鉴那个一次性编辑器的思路,让 Claude 给你做一个专门针对这批数据的临时界面,操作完之后导出结果。这个思路的精髓在于,你不是在打造一个长期工具,而是把一次性的脑力劳动,变成了一次性的图形界面操作,效率会高很多,而且做这个界面的成本几乎可以忽略,因为是 Claude 帮你现场生成的。

当然,这里也要提醒几个可能的坑。第一个坑是,如果你团队里有比较严格的文档管理规范,要求所有文档必须是纯文本、必须能被 Git 很好地做差异对比,那 HTML 文件在版本控制上的体验,会比 Markdown 差一些,因为 HTML 文件的差异对比,可读性通常不如纯文本。如果你的场景特别依赖频繁的多人协作编辑和版本追踪,可能还是要谨慎评估一下,要不要把核心的、需要长期维护的文档也换成 HTML。作者提到的场景,更多是阶段性的、一次性的产出物,比如说方案讨论、报告、临时编辑器,这些场景换成 HTML 风险比较小,但如果是那种需要长期维护、频繁修改的规范文档,可能还是要权衡一下。

第二个坑是,HTML 文件生成之后,你怎么分享、怎么存放,也是需要提前想清楚的问题。作者提到的方式是上传之后分享链接,但这背后默认了你有一个可以承载这些文件的地方,如果你所在公司对外部存储、外部分享有比较严格的安全合规要求,你需要先确认清楚,这些 HTML 文件该放在哪里、谁能访问,尤其是如果文件里包含了一些敏感的代码片段或者业务数据。

第三个坑,我个人认为是容易被忽略的一点,就是这种工作方式高度依赖你自己愿不愿意真的去打开这些 HTML 文件、认真看里面的内容。HTML 本身只是降低了读的门槛,但它没办法强迫你真的去读。如果你只是把生成 HTML 当成一个新鲜的形式,生成完了还是随手一扫就过,那这种方法带来的更在场的好处,其实就没有真正发挥出来。所以我觉得用好这套方法的关键,还是要回到作者最后强调的那句话上面,核心目的是让自己更投入地参与到 Claude 的判断过程里,而不是单纯换一个文件格式。

好,我们来稍微总结一下今天聊的这篇文章。这篇文章的核心观点其实很简单,就是随着 Claude 这样的智能体能力越来越强,人和智能体之间交流的格式,也应该跟着升级,Markdown 这种轻量级的纯文本格式,在表达复杂信息、保证可读性、方便分享这几件事上,正在显得力不从心,而 HTML 凭借它在排版、配色、矢量图、交互这些方面的丰富表达能力,正在成为越来越多 Claude Code 重度用户的新选择。文章里给出的几个典型场景,从方案探索、代码评审、设计原型,到综合报告、一次性编辑器,基本覆盖了我们日常工程工作里很常见的几类需求,而且每个场景都配了具体可以直接拿去用的提示词,实操性很强。

我个人认为,这篇文章最有价值的地方,不是在教你 HTML 比 Markdown 强这个表面结论,而是在提醒我们,随着 AI 能干的事情越来越多,我们自己怎么样才能保持在场,怎么样才能不沦为一个只会说好的继续的旁观者,这件事会变得越来越重要。选择一个更适合人类视觉和注意力习惯的输出格式,只是众多解法里的一种,但它背后那个问题,我觉得值得每个正在重度使用 Claude Code 的工程师认真想一想。好,这期就先聊到这里,我们下期再见。
