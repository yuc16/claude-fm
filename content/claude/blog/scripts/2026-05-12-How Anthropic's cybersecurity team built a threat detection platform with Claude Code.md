于是她的团队就坐下来问自己几个问题。我们每天的精力都耗在哪了?哪些工作让人觉得纯粹是体力活,重复又琢磨不出新东西?哪些事情拖住了我们,让我们没办法去做真正有价值的分析?答案也很清楚。光是判断一个警报是真是假,可能就要耗掉好几个小时;不同系统之间的数据要靠人工去对照、去关联,效率很低;再加上不停地在各种查询语法和界面之间切换,这种认知层面的消耗,一天下来是会累积的,人是会疲劳的。更麻烦的是,公司越长大,需要防护的范围也越大,攻击面跟着扩大,安全团队的工作量只会往上走,不会往下走。Jackie 说得很直接,如果不靠类似 Claude 这样的东西来辅助,团队是没办法跟上公司发展速度的。

那解决方案是什么呢?Jackie 的团队花了几个月时间,搭了一套叫 CLUE 的检测响应平台。CLUE 这四个字母,英文全称是 Claude Looks Up Evidence,直接翻译过来就是让 Claude 去查证据。这个名字其实已经把产品思路说明白了,它不是在原来一堆仪表盘之上再叠一个新的仪表盘,而是提供一个自然语言的交互入口,背后是 Claude,通过工具调用直接连到 Anthropic 内部的各个系统里去。

这里要解释一下工具调用是什么意思。简单说,就是给 Claude 配备一堆可以执行具体操作的接口,比如查数据库、读 Slack 消息、看代码仓库,Claude 根据当前任务自己判断该调用哪个工具、传什么参数,工具返回结果之后,Claude 再继续往下推理,这样一轮一轮下来,就能完成原来需要人工跳来跳去才能做完的事情。

特别值得一提的是这个平台的搭建速度。Jackie 的团队用 Claude Code 来开发,从立项到能跑起来的原型,只花了一天时间;设计文档、开发步骤、具体实现,一周之内就全部搞定。Jackie 自己的说法是,他们写代码的过程,很大一部分其实就是在跟 Claude Code 对话,Claude Code 既是设计上的搭档,也是实际动手的协作者。

文章里举了一个挺有意思的小例子。Jackie 当时让 Claude Code 给 CLUE 的界面加一个按钮,她心里已经做好了准备,觉得接下来要经历一段熬人的过程,各种前端框架的坑、样式调不对的折腾,她太熟悉这种感觉了。结果 Claude Code 几乎是立刻就把这个功能实现出来了,做得比她自己动手还要好。她说,就是在那一刻,她意识到自己不再被自己的技术能力边界所限制了,只要她能想到的东西,现在都可以试着做出来。我自己读到这一段的时候,觉得特别有共鸣,因为这种感觉,很多写过代码的人应该都体会过,以前你脑子里有想法,但实现起来要考虑你到底会不会某个框架、会不会某种语言,现在这道门槛被大幅拉低了,想法和实现之间的距离变短了。

CLUE 这套系统,具体拆开来看,有两个核心模块。第一个叫 CLUE Triage,负责做警报的初步分诊。

分诊这个词,大家如果熟悉医院的急诊流程就能秒懂,病人一来,先有个分诊台判断这个人是不是紧急情况,该往哪个科室送。CLUE Triage 干的就是类似的事情,警报进来之后,不是直接丢给人类分析师,而是先让 Claude 过一遍。Claude 会主动去调用工具,把这条警报和 Anthropic 内部各种各样的上下文信息拼到一起,包括 Slack 里的聊天记录、内部文档、代码仓库,还有数据仓库里的历史数据。拼完之后,Claude 会给这条警报打一个判断标签,是误报,是真实威胁,是恶意行为,还是属于正常的预期行为,同时还会附带一个置信度分数,告诉分析师这个判断有多大把握,该把注意力放在哪些警报上。

为什么这一步这么关键?因为任何做安全分析的人都知道,警报单独看的时候,往往就是一个孤立的信号。一次登录失败,一次不寻常的接口调用,一次配置变更,这些事情单独拎出来,什么都说明不了,顶多算是噪音。但是一旦你知道这个用户是谁,他当时在做什么工作,这个行为模式是不是和他所在团队平时的操作习惯一致,这些孤立的点马上就连成了线,信号也就清晰了。Jackie 说得很到位,那些藏在公司内部的上下文信息,恰恰是让警报真正贴合你所在环境的那块缺失的拼图。我自己设想了一下,如果换成是我来设计这套分诊逻辑,大概会先做一个规则引擎,把常见的误报模式一条条列出来,再人工调权重。但这种做法的天花板很低,新的误报模式一旦出现,规则就要重新写。CLUE 的做法不一样,它不是靠预先写好的规则去匹配,而是让 Claude 每一次都重新去查、重新去理解上下文,规则引擎会过时,但理解能力不会过时,这可能才是两者最本质的差别。

第二个模块叫 CLUE Investigate,负责深入调查。有了它之后,安全分析师可以直接用自然语言去查询所有跟安全相关的日志。比如你想知道,过去一天里,这台机器上所有登录失败的记录是什么,你就直接这样问 CLUE,背后 Claude 会自动把这句话翻译成需要执行的查询语句,跑出结果给你。Jackie 提到一个挺颠覆常规印象的观点,她说 Claude 写精确查询语句的能力,比人类工程师还要强。

这个模块跑起来是一个智能体式的循环,英文叫 agentic loop。具体来说,有一个总调度,英文叫 orchestrator,负责拆解任务、下达指令,下面挂着若干个子智能体,英文叫 sub agent,这些子智能体并行地去执行各自的查询任务,把各自查到的结果汇总回来,再由总调度整合成一份完整、连贯的调查报告。原来需要人工一项一项手动去关联、去比对的工作,现在跑一遍只需要三到四分钟。

文章给出了具体的数字。平均一次调查,CLUE 会发起大约二十五次工具调用,执行将近十一次查询,这个调用密度,已经远远超出了一个分析师在合理时间内手动能做到的范围,而且精确度还更高。你想想看,这二十五次工具调用,换成人工操作,意味着分析师要打开二十五次不同的控制台,或者在不同界面之间切换二十五次,光是这个动作本身的时间成本,就已经很可观了。

不过 Jackie 强调,真正让 CLUE 与外部安全产品拉开差距的,不是查询速度,而是内部上下文。市面上很多安全平台,再厉害,也接触不到一家公司内部真正的运转细节。CLUE 因为是直接连到 Anthropic 自己的系统里,Claude 能看到的东西,是外部产品永远拿不到的。警报响了,Claude 可以去 Slack 里查一下,团队是不是提前说过要做计划内的系统维护;可以去数据仓库里查一下,这个行为的历史基线长什么样,算不算异常;还可以去翻代码仓库,搞清楚这个服务到底是干什么用的,它平时应该有什么样的行为模式。这些信息,只有真正嵌入到公司内部的系统才拿得到。

文章里给了一个具体场景,来说明 CLUE Investigate 的实际效果,是一次数据治理方面的审查。团队想确认,三名外部承包商,在过去两个月里,有没有访问过他们本不该接触的文档。按 Jackie 的说法,这种调查,正常走人工流程,起码要花掉半天时间,要去查访问日志,要交叉核对权限设置,还要逐一审阅文档的分类级别,一项一项手动比对。换成 CLUE 来做,Claude 读懂这个请求之后,自己制定调查计划,生成一系列详细的查询语句,把背后复杂的技术细节都包装起来,分析师不需要懂查询语言,也不需要知道权限系统的底层逻辑。整个调查几分钟之内就跑完了,出来一份总结和建议,而且每一步执行过的查询都完全透明,可以回溯检查。半天的人工活,压缩到几分钟,这个对比是相当直观的。

光听上去快是不够的,Jackie 的团队也很认真地去量化了这套系统到底带来了什么变化,这部分我觉得是文章里最有说服力的地方。

第一个数字是误报率。在用上 CLUE Triage 之前,大概三分之一的警报,最后查下来都是误报,也就是说,分析师三分之一的精力都花在了根本不是问题的事情上。用上之后,这个比例降到了百分之七。这意味着分析师能把绝大部分注意力,放在真正值得关注的信号上。

第二点,Jackie 特别强调,可能比速度更重要的是覆盖范围的扩大。以前置信度比较低的信号,因为人力有限,很多时候根本来不及看,只能放在仪表盘里积灰。现在不一样了,CLUE Triage 会对每一条进来的警报都做一遍富化处理,再加上批量处理能力,过去那些只能堆在仪表盘里没人看的几千条低置信度信号,现在都能被实实在在地检查一遍。这等于是把团队的覆盖盲区给补上了,而不只是把原来能看的部分做得更快。

第三点是规模化之后的时间节省。文章给了一个三十天的统计窗口,这三十天里,CLUE 自动完成了大约一万二千次查询,两万七千次工具调用,团队估算,这些工作如果靠人工来做,大概需要一千八百七十个小时,换算成人天大约是二百三十四人天。也就是说,效率提升大概在五到十倍之间。这个数字背后的意义是,原来需要一整个团队连续干上大半年的活,现在压缩到了一个月里完成,而且是在分析师还要兼顾其他日常工作的情况下。

当然,文章也很坦诚地提到,有一块东西目前还很难量化,那就是准确性。速度好衡量,你可以掐表算时间,但 Claude 做出的判断到底准不准,这件事没那么容易下结论。团队目前的做法是,人工复核 CLUE 给出的判断,跟踪那些人和 Claude 意见不一致的案例,但是要真正搭建起一套完整的反馈闭环,搞清楚 Claude 到底在多大程度上能发现分析师会漏掉的东西,反过来分析师又在多大程度上能发现 Claude 漏掉的东西,这件事他们还在摸索。好在每一次调查的完整记录都会留存下来,可以随时回去审计,Claude 当时具体看了什么数据,是怎么一步步得出结论的,这种透明度,某种程度上弥补了准确性指标暂时缺失的问题。换个角度想,这其实也是一个挺诚实的态度。很多团队在对外宣传自己上线了 AI 系统的时候,往往只摆速度和数量上的提升,准确性这种不好量化、又容易暴露短板的指标,能避就避。Jackie 团队没有回避这个问题,而是直接承认,这是他们还在解决的事情,这种坦诚,我觉得反而比堆砌一堆漂亮数字更值得信任。

接下来是我觉得这篇文章里最值得细品的一部分。Jackie 提到了一个在 AI 研究领域很有名的概念,叫苦涩的教训,英文是 the bitter lesson。这个概念的核心意思是,如果你费尽心思把人类的具体推理方式、具体经验规则,一条一条硬编码进模型里,长期来看,效果往往比不上直接给模型通用的能力,然后放手让它自己去找办法解决问题。

这套逻辑用在安全运营上意味着什么?Jackie 说,团队在做 CLUE 的早期,内部争论过一个问题,到底要不要严格限制 Claude 的调查路径。她说那种来自传统安全编排、自动化与响应系统的思维习惯,英文缩写是 SOAR,本能反应就是要写好剧本,把每一步流程都定义清楚,让整个过程变得确定、可预测,跟流水线一样,这一步必须做完才能做下一步,绝不能跑偏。

但他们在实践中反复观察到一个现象。当他们给 Claude 更大的自由度,只给它一套可以用的工具,加上一个明确的目标,而不是一条事先写好的固定流程,Claude 经常会走出团队完全没有预设过的调查路径。而这些没人预设过的路径里,有些恰恰挖出了团队原本会错过的关键上下文。这就有点像你带一个非常聪明但没经验的新人去查案子,你如果把每一步都写成标准作业流程让他照做,他只能查到你想到的那些地方;但如果你只告诉他目标是什么,给他必要的工具和权限,放他自己去摸,他反而可能摸出一条你压根没想到的线索。

所以团队后来的做法是,给 Claude 划清边界,比如它能用哪些工具、能碰哪些数据,这个边界要明确,但具体怎么调查、走什么顺序,这个策略层面要尽量放开。这个思路,也决定了他们接下来要往哪个方向去发展 CLUE。

第一个方向,是从被动响应转向主动出击。现在的 CLUE,本质上还是反应式的,警报先响,Claude 才会动起来调查。但他们设想的架构,可以支持更进一步的玩法,就是持续不断的主动探索。不用等检测规则被触发,而是让 Claude 智能体一直在系统里巡逻,主动去找那些不符合任何已有规则的异常,找那些单独看每一项都很正常,但放在一起看就显得不太对劲的行为组合。这就好比传统安保是装个报警器,响了再出动,而新的思路是养一群一直在巡逻的安保人员,他们不需要等铃响,自己就能从细微的不对劲里嗅出问题。

第二个方向,是让 Claude 从自己过去的工作经验里学习。团队会把每一次调查的完整记录都存下来,这些记录正在变成一个知识库,Claude 可以反过来查询这个知识库,从过去无数次调查的轨迹里去找模式。久而久之,CLUE 会积累出一种没有任何单个人类分析师能维持得住的组织记忆,因为人会离职、会遗忘,但这套记录会一直留在系统里,越积累越厚。

第三个方向,是主动拥抱不确定性。传统安全工具的设计哲学是,同样的输入必须产出同样的结果,不一致就是缺陷,就要修。但 CLUE 反过来把这种不一致当成一个特点去利用。同一条警报,今天调查出来的路径,和明天调查出来的路径,完全可能不一样,团队觉得这没什么问题,有时候恰恰是第二条路径,找到了第一条路径没找到的东西。基于这个想法,团队正在尝试让多套调查策略并行跑,然后对比它们各自得出的结论,看看能不能从差异里挖出更多信息。

Jackie 最后总结说,安全运营领域的苦涩的教训就是,过去这些年,大家花了大量精力,把人类是怎么做调查的这件事一条一条编码进系统里;而下一代工具应该做的,是给模型真正的调查能力,然后放手让它们自己去找,可能会找到比人类事先设计的流程更好的办法。

说到这,我们来想想这篇文章对国内做安全运营、做 AI 应用的朋友们,具体意味着什么。

第一点,我觉得最直接的启发是,Claude Code 不只是一个写代码的工具,它同样适合用来搭建内部工具,尤其是那种需要快速验证想法的场景。Jackie 团队从立项到原型只花了一天,这个速度放在传统软件开发节奏里几乎是不可想象的。如果你所在的团队也有类似的痛点,比如散落在各个系统里的内部知识,没有一个统一的自然语言入口,你完全可以考虑,先拿一个小范围的场景,用 Claude Code 自己搭一个原型出来试试看,成本比想象中要低很多。

第二点,是关于人机协作模式的转变。CLUE 给我们的启发是,不要急着把每一步流程都用脚本和规则锁死。我个人认为,这一点对很多国内团队来说反而是更难接受的一步,因为我们做系统设计的时候,本能上更倾向于把流程标准化、确定化,觉得这样才稳。但 Jackie 团队的经验告诉我们,如果你给的是工具加目标,而不是固定的脚本,模型反而可能找到你没想到的路径。这不代表完全不要约束,边界还是要划清楚,哪些数据能碰,哪些操作可以做,这个层面的限制是必须的,安全场景尤其不能马虎。但具体怎么走,可以适当放手。

第三点,关于落地的坑,我想提醒大家两个地方。一个是内部上下文的接入,这其实是整篇文章里反复强调的关键差异点。CLUE 真正厉害的地方,不是查询速度快,而是它能连到 Slack、内部文档、代码仓库、数据仓库这些真正体现公司运作细节的系统里。如果你只是把模型接到一个孤立的日志系统上,效果会大打折扣,真正有价值的是把这些内部知识源一个一个打通。另一个坑是准确性的衡量,文章也坦诚承认这件事很难,如果你的场景涉及安全决策,千万别因为它跑得快、看起来很顺畅,就忽略了建立人工复核和审计追溯的机制,把每一步调查记录都留下来,方便随时回溯检查,这是必须配套做的事情,不是可选项。

第四点,如果你的工作场景跟安全运营离得比较远,也不要觉得这篇文章跟你没关系。CLUE 背后的核心思路,警报富化、给置信度打分、用智能体并行处理任务、把历史记录变成可查询的知识库,这些思路换个场景同样适用,比如客服工单分诊、运维故障排查、合规审查,本质上都是类似的信息过载问题,都值得借鉴这种自然语言入口加内部系统连接的做法。

第五点,是关于怎么起步的建议,这部分是我个人的想法,文章里没有细讲。我会建议不要一开始就照搬 CLUE 这种横跨好几个内部系统的大架构,可以先挑一个范围最小、风险最低的场景练手,比如先把某一个团队的内部文档和聊天记录接起来,做一个简单的问答机器人,看看 Claude 调用工具、组织上下文的效果到底怎么样,等你对它的行为有了直观判断,再逐步把更多系统接进来,把权限和审计机制一并补上。一口气追求大而全的方案,反而容易在前期就卡住,小步快跑去验证,效果通常更扎实。

最后还有一点想补充。文章末尾有个小注脚,说这些效果数据是用 Claude 的 Sonnet 和 Opus 模型跑出来的,不是某个单一的顶配模型才能做到这种效果。这其实也是个挺重要的信息,意味着这套思路对成本没有那么极端的要求,中等规模的模型搭配合理的工具调用架构,就能跑出实打实的效果,这对预算有限的团队来说,是个值得参考的细节。

好,这期就聊到这里,我们简单回顾一下。Anthropic 的检测平台工程团队,用 Claude Code 在几周时间内搭出了一套叫 CLUE 的威胁检测平台,核心是两个模块,一个负责警报初步分诊,一个负责深入调查,背后靠的是 Claude 通过工具调用直接连进公司内部各个系统,把原本散落各处的上下文信息拼到一起。效果上,误报率从三分之一降到了百分之七,三十天内自动完成的工作量,相当于人工二百三十四人天,效率提升五到十倍。更有意思的是,团队从这个过程里得出一个反直觉的经验,给 Claude 设定边界但放开具体策略,反而比写死每一步流程效果更好。

我个人的感受是,这篇文章最打动我的地方,不是那些漂亮的数字,而是 Jackie 那句话,她说她终于可以建造自己一直想要的工具了。这种从受限于自己技术能力,到几乎可以把任何想法做出来的转变,可能才是这一类工具真正在改变的东西,效率提升只是一个自然的结果。好,这就是今天的内容,我们下期再聊。
