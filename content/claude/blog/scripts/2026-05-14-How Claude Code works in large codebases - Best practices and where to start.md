欢迎收听 Claude FM。今天我们要聊的这篇文章,标题是 How Claude Code works in large codebases, Best practices and where to start,翻译过来就是,Claude Code 在大型代码库里是怎么工作的,最佳实践以及从哪里开始。这是 Anthropic 官方博客发的一篇文章,发表时间是二零二六年五月十四日。今天是二零二六年六月二十日,距离发表也就一个多月,所以这篇文章里讲的方法,基本可以认为是当下最新鲜的一线经验,不存在过时的问题,大家可以放心当作现在就能用的指南来听。

这篇文章其实是一个系列的开篇,这个系列叫 Claude Code at scale,直译过来是大规模场景下的 Claude Code,专门讲企业级工程团队怎么用好 Claude Code 这个最佳实践。如果你还不熟悉,Claude Code 是 Anthropic 推出的一款命令行智能编程助手,你可以理解成一个能直接在你的终端里读代码、改代码、跑命令的 agent,这里 agent 这个词,中文常翻译成智能体,指的是那种可以自主规划步骤、调用工具、一步步完成任务的程序,不是简单地回答一句话就结束。

文章关注的场景很具体,就是大型代码库。这里说的大,不是几万行代码那种大,而是真正意义上的巨兽级别。文章里举的例子包括几百万行代码的单体仓库,业内叫 monorepo,就是把很多项目的代码都放进同一个版本库里管理;还包括运行了几十年的遗留系统,代码分散在几十个微服务里,彼此之间甚至没有一个共同的根目录。文章特别提到,这些大代码库里很多用的还是 C、C 加加、Java、PHP 这些语言,过去大家普遍认为不太适合用 AI 编程工具,但文章说,随着模型能力提升,Claude Code 在这些语言上的表现比大多数团队预期的要好得多。

为什么要专门写这么一篇文章呢。因为大型代码库会遇到的麻烦,是小项目完全不会遇到的。比如说,每个子目录的构建命令可能都不一样,遗留代码分散在各个文件夹里,没有统一的入口和规范。这些问题不是靠模型变聪明就能自动解决的,而是需要在工程实践层面把环境和习惯搭对。这篇文章总结的,就是 Anthropic 的应用 AI 团队跟一大批企业客户合作之后,观察到的那些真正促成成功落地的共性模式。我个人觉得,这种经过大规模验证的经验总结,比单纯讲模型能力的文章更值得工程团队认真消化。

好,我们先从第一个核心问题说起,Claude Code 到底是怎么在一个巨大的代码库里找到它需要的东西的。

文章里说,Claude Code 探索代码库的方式,跟一个真正的软件工程师没什么本质区别。它会遍历文件系统,打开文件读内容,用 grep 命令去搜索关键词,grep 是 Unix 系统里一个经典的文本搜索工具,然后沿着代码里的引用关系一路跟踪下去。整个过程是在开发者自己的机器上本地运行的,不需要提前给整个代码库建一个索引,也不需要把代码上传到什么服务器上去处理。

这一点之所以重要,是因为它跟另一类主流的 AI 编程工具形成了鲜明对比。那一类工具用的是 RAG 技术,RAG 全称是检索增强生成,工作原理是先把整个代码库切成一段一段,转换成向量,这个过程叫嵌入,英文是 embedding,相当于给每段代码生成一个数学上的指纹,存到一个数据库里。等你提问的时候,系统再去这个数据库里检索跟你问题最相关的几段代码,喂给模型去生成回答。

听起来很合理,但问题出在规模上。文章里指出,在大规模场景下,这种嵌入流水线根本跟不上一个活跃的工程团队的提交速度。等开发者去查询这个索引的时候,索引反映的可能是几周前、几天前,甚至只是几小时前的代码库状态。于是检索结果就会出问题,它可能给你返回一个团队两周前就已经改名的函数,或者引用一个上个迭代周期里就已经被删除的模块,而且系统完全不会告诉你这些内容已经过时了。你拿着这些过时信息去写代码,后果可想而知。

而 Claude Code 用的这种基于 agent 的搜索方式,业内叫 agentic search,正好绕开了这个坑。因为它没有嵌入流水线,没有一个需要持续维护的中心化索引,所以哪怕几千名工程师同时在往代码库里提交新代码,每个开发者本地跑的 Claude Code 实例,看到的永远是当下这一刻活生生的代码库,不存在过时的问题。

但文章很诚实地指出,这种方式也有代价,那就是它非常依赖 Claude 一开始就拥有足够的背景信息,知道该往哪里看。如果你让它在一个十亿行规模的代码库里,去找某个模糊模式的所有出现位置,那大概率工作还没真正开始,它就先把自己的上下文窗口用完了。这里上下文窗口,英文是 context window,你可以理解成模型一次能记住、能处理的信息总量上限,就像人脑工作记忆是有限的一样。所以文章的结论是,Claude 导航代码库的质量,直接取决于代码库本身的组织方式好不好,有没有用 CLAUDE.md 文件和 skills 把背景信息一层一层地铺好。那些愿意在代码库整理这件事上提前投入的团队,后续用起来的效果会明显更好。这其实是个挺反直觉的提醒,很多团队第一反应是去换更强的模型,但这篇文章说,真正决定效果的,往往是你有没有把家里收拾好。

这就自然引出了文章里我认为最重要的一个观点,围绕模型搭建起来的那一整套生态系统,英文叫 harness,直译是挽具或者套具,你可以理解成给模型穿上的整套装备和支撑系统。很多团队对 Claude Code 有一个常见的误解,以为它的能力完全是由背后用的是哪个模型决定的,大家盯着模型的跑分和测试任务表现,但实际情况是,这个 harness 对最终表现的影响,往往比模型本身更大。

文章把这个 harness 拆解成五个扩展点,分别是 CLAUDE.md 文件、hooks、skills、plugins,还有 MCP 服务器,另外还有两个补充能力,语言服务器协议集成,简称 LSP,以及 subagent,也就是子智能体。文章特别强调,这几层东西搭建的顺序很重要,因为每一层都是建立在前一层基础之上的。我们一个一个来看。

第一层是 CLAUDE.md 文件,这是最基础也是最先要做的。这是一种上下文文件,Claude 在每次会话开始的时候会自动读取它。一般的做法是,仓库根目录放一个总览性质的文件,讲大局观的东西;每个子目录可以再放自己的 CLAUDE.md,讲这个局部模块特有的规范。这些文件给 Claude 提供了它做好任何事情都需要的代码库知识。但因为这些文件是每次会话都会被自动加载的,不管你具体要做什么任务,所以如果内容太杂太多,反而会变成拖累整体表现的噪音,这一点文章后面还会反复强调。

第二层是 hooks,中文经常翻译成钩子,简单说就是在 Claude Code 工作流程的某些固定节点上,自动触发执行的脚本。文章说,很多团队对 hooks 的理解还停留在用它去阻止 Claude 做错事,比如拦截一次危险的文件写入。但其实 hooks 更有价值的用法,是让整套配置具备自我改进的能力。举个例子,一个在会话结束时触发的 stop hook,可以让 Claude 在上下文还很新鲜的时候,回顾一下这次会话里发生了什么,然后主动提议应该往 CLAUDE.md 里补充哪些新内容。而一个在会话开始时触发的 start hook,可以动态地把团队特定的背景信息加载进来,这样每个开发者不用自己手动配置,就能拿到适合自己所在模块的正确设置。对于像代码格式化、代码规范检查这类自动化检查,用 hooks 来强制执行规则,会比单纯指望 Claude 记住一条指令要稳定可靠得多,因为前者是确定性的程序逻辑,后者依赖模型每次都记得住、不出错。

第三层是 skills,这个词在这里指的是打包好的、针对特定任务类型的专业知识和工作流程。文章解释说,在一个有几十种任务类型的大型代码库里,不是所有专业知识都需要常驻在每一次会话里。skills 解决这个问题的方式叫渐进式披露,意思是把那些原本会占用上下文空间、互相竞争资源的专业工作流和领域知识,挪到一个个独立的模块里,只有当任务真正需要的时候才加载进来。文章举的例子是,一个安全审查相关的 skill,只有当 Claude 在评估代码是否存在安全漏洞的时候才会加载;一个处理文档的 skill,只有当代码改动需要同步更新文档的时候才会加载。

skills 还有一个很实用的特性,就是可以绑定到代码库里特定的路径上,只在相关的那部分代码里激活。文章举了个例子,一个负责支付服务的团队,可以把他们的部署相关 skill 只绑定到支付服务所在的目录,这样当别的工程师在 monorepo 的其他地方工作的时候,这个 skill 根本不会被自动加载出来,不会去干扰跟它无关的工作。

第四层是 plugins,也就是插件。文章指出,大型代码库面临的一个现实挑战是,好的配置经验很容易变成只有少数人知道的部落知识,新人加入团队之后完全摸不到门道。plugins 把 skills、hooks 还有 MCP 配置一起打包成一个可安装的整体,这样新工程师第一天装上这个插件,就能立刻拥有跟那些已经用了很久 Claude Code 的老员工一样的背景信息和能力。插件的更新,还可以通过企业内部受管理的应用市场,统一分发给整个组织。

文章里举了一个挺生动的例子,一家大型零售企业,内部搭建了一个 skill,把 Claude 连接到他们内部的分析平台,这样业务分析师不需要离开自己原本的工作流程,就能直接拉取业务表现数据。在向全公司业务团队正式推广之前,他们就是用插件的形式先把这个能力分发出去的。

第五层是 MCP 服务器。MCP 这个缩写,全称是模型上下文协议,你可以理解成一套标准化的接口规范,让 Claude 能够连接到那些它原本完全接触不到的内部工具、数据源和 API。文章说,做得比较成熟的团队,会专门搭建 MCP 服务器,把结构化搜索能力包装成一个 Claude 可以直接调用的工具。还有一些团队用它把 Claude 连接到内部文档系统、工单系统,或者分析平台。

除了这五层之外,文章还补充了两个重要能力。第一个是语言服务器协议集成,简称 LSP。大型代码库常用的 IDE 里,通常都已经跑着一个语言服务器,支撑着我们平时用的跳转到定义、查找所有引用这些功能。把这种能力开放给 Claude 之后,它就能获得跟人类开发者在 IDE 里一样的符号级精确度,可以顺着一次函数调用直接跳到它真正的定义处,跨文件追踪引用关系,还能区分不同语言里两个同名但完全不同的函数。如果没有这一层,Claude 只能靠文本去做模式匹配,很容易匹配错符号。文章提到一家做企业软件的公司,在全面推广 Claude Code 之前,先在全公司范围部署了 LSP 集成,目的就是让 C 和 C 加加代码在大规模场景下的导航变得可靠。对于多语言混合的代码库来说,文章认为这是投入产出比最高的一项工作。

第二个补充能力是 subagent,子智能体。它是一个拥有自己独立上下文窗口的 Claude 实例,接到一个任务之后,自己把工作做完,只把最终结果返回给上一级的主智能体,中间过程不会污染主智能体的上下文。文章说,等整套 harness 搭建完成之后,一些团队会专门派出一个只读的 subagent,去把某个子系统的结构摸清楚,把发现的内容写进一个文件里,然后再让主智能体带着这份完整的认知去做真正的代码修改。这样就把探索和编辑两件事彻底分开了,互不干扰。

文章最后还用一个表格,总结了每个组件最容易被搞混的地方,我简单提一下方便大家记住。CLAUDE.md 最容易被错放进去本该属于 skill 的可复用知识;skills 最容易反过来被忽略,结果所有东西都堆进了 CLAUDE.md;plugins 最大的风险是好配置一直留在个人手里没被分发出去;LSP 容易被想当然地以为是自动具备的能力;MCP 服务器容易在基础设施还没打牢的时候就被提前搭建;subagent 最常见的误区,是把探索和编辑混在同一个会话里一起做。

讲完了 harness 的五层加两个补充能力,文章接下来讲了三种在成功落地案例里反复出现的配置模式,这部分我觉得是整篇文章里最具体、最能直接照着抄作业的部分,我们仔细展开讲一讲。

第一种模式,是把代码库改造得在大规模下依然可以被导航。文章的核心判断是,Claude 在大代码库里能帮上多大忙,上限取决于它找到正确上下文的能力。这里有个微妙的平衡,每次会话加载的上下文太多,会拖累表现,加载太少,Claude 就等于在盲找。所以最有效的那些落地案例,都是提前在让代码库对 Claude 变得清晰易读这件事上投入了功夫。

具体怎么做呢,文章给了好几条建议。第一条,保持 CLAUDE.md 文件精简,并且分层组织。Claude 在代码库里移动的时候,会一层一层叠加式地加载这些文件,根目录的文件负责讲大局,子目录的文件负责讲局部规范。根目录那个文件,应该只放路标性质的内容和真正关键的注意事项,其他什么都不放,因为放多了就会变成噪音,稀释掉真正重要的信息。

第二条建议,是在子目录里初始化会话,而不是总在仓库根目录启动。文章承认,这在 monorepo 场景下听起来有点反直觉,因为很多工具链默认假设你是从根目录操作的。但实际上,Claude 工作得最好的状态,是被限定在跟当前任务真正相关的那一小块代码库范围内。而且不用担心丢失全局信息,因为 Claude 会自动沿着目录树往上走,把它路过的每一个 CLAUDE.md 文件都加载进来,所以根目录的背景信息一点都不会丢。

第三条建议,是按子目录划定测试命令和代码检查命令。如果 Claude 只改了某一个服务的代码,却要触发跑全量测试套件,既会导致超时,又会把大量无关输出塞进上下文里。子目录级别的 CLAUDE.md 文件,就应该写清楚适用于这部分代码的具体命令。这种做法在服务化架构的代码库里效果最好,因为每个目录天然有自己独立的测试和构建命令;但在深度跨目录依赖的编译型语言 monorepo 里,按子目录划分命令就困难得多,可能还需要专门定制构建配置。

第四条建议,是用忽略文件去排除生成出来的文件、构建产物,以及第三方代码。文章建议把这些排除规则,通过 .claude 目录下的 settings.json 文件里的 permissions.deny 配置提交进版本库,这样整个团队的每个人都能享受到同样的噪音削减效果,不需要每个人自己再单独配置一遍。文章也提到一个细节,有些代码库里,生成出来的文件本身恰恰就是开发工作的对象,比如专门写代码生成器的开发者,他们可以在自己本地的配置里覆盖掉项目级别的排除规则,而不会影响团队里其他人。

第五条建议,是当目录结构本身没办法很好地承担说明作用的时候,就专门搭建一份代码库地图。可以在仓库根目录放一个轻量的 markdown 文件,列出每个顶层文件夹,配上一行说明,相当于给 Claude 一份目录索引,它可以先扫一眼,再决定要不要真正打开文件去看。如果顶层文件夹多达几百个,可以分层处理,根目录文件只描述最高层级,子目录的 CLAUDE.md 再提供下一层细节,按需加载。场景简单的话,直接在提示词里用 @ 符号指定具体该参考的文件和目录,效果也差不多。

第六条建议,是运行 LSP 服务器,让 Claude 按符号搜索,而不是按字符串搜索。如果你在一个大代码库里 grep 一个很常见的函数名,可能会返回几千条匹配,Claude 得一个一个打开文件去判断到底哪个才是真正相关的,白白消耗大量上下文。而 LSP 只会返回真正指向同一个符号的引用,相当于在 Claude 读取任何文件之前,过滤工作就已经提前做完了。文章也提了一句实操层面的话,要搭这个,需要给你用的语言安装对应的代码智能插件,以及对应的语言服务器二进制程序,具体可以参考 Claude Code 官方文档里关于可用插件和故障排查的部分。

文章也很坦诚地承认,这套基于层层 CLAUDE.md 的方案,在某些极端场景下依然会失效,比如那种有几十万个文件夹、几百万个文件的代码库,或者那些还在用非 Git 版本控制系统的遗留系统。这些更棘手的情况,文章说会在这个系列后续的文章里专门讨论,算是给了我们一个期待后续内容的预告。

第二种配置模式,讲的是要随着模型智能水平的进化,主动维护 CLAUDE.md 文件,而不是写完就丢在那里不管。这一点特别容易被团队忽略,但很关键。文章的洞察是,随着模型迭代,你为当前这一代模型写的指令,可能反而会束缚住未来更强的模型。打个比方,那些原本帮着 Claude 绕开它曾经不擅长应对的模式的 CLAUDE.md 规则,等新一代模型上线之后,这些规则可能就变得完全没必要,甚至会主动妨碍新模型发挥实力。

文章举了个很具体的例子,假设你写了一条规则,要求每次重构都必须拆解成只改单个文件的小改动,这条规则可能确实帮过去那一代模型守住了边界,不容易跑偏。但对于一个已经能够很好地处理跨文件协调式修改的新模型来说,这条规则反而成了一个紧箍咒,限制了它本可以更高效地完成的工作。

同样的逻辑也适用于那些专门为了弥补某个特定模型局限性而搭建的 skills 和 hooks,一旦这些局限性不存在了,这些补丁式配置就变成了纯粹的负担。文章给的例子是,曾经有团队写过一个 hook,专门拦截文件写入,强制走 Perforce 版本控制系统里的 p4 edit 流程,后来 Claude Code 原生支持了 Perforce 模式之后,这个 hook 就彻底变成了多余的东西。

所以文章给出的具体建议是,团队应该预期每隔三到六个月,就要做一次有实质内容的配置复查。而且不只是按固定周期来,每次有重大模型版本发布之后,如果你觉得 Claude Code 的表现好像遇到了瓶颈、卡在某个水平上不去了,那也值得专门做一次复查。我个人觉得这个建议特别接地气,因为很多团队的心态是配置一旦跑通了就再也不去碰,但这篇文章其实在提醒我们,配置本身是有保质期的,跟着模型一起升级换代,才能真正持续吃到模型进步带来的红利。

第三种配置模式,讲的是组织层面的事情,要给 Claude Code 的管理和推广明确分配责任人。文章的判断是,单靠技术配置本身,并不足以真正驱动全员采用,那些真正把这件事做对的组织,在组织架构这个层面也做了投入。

文章观察到,推广速度最快的组织,都是在大范围开放访问权限之前先做了专门的基础设施投入。可能只是一个很小的团队,甚至就一个人,提前把整套工具搭好,这样大部分开发者第一次接触 Claude Code 时,它已经天然契合大家的工作流程。文章举了两个例子,一家是两名工程师提前搭好了一整套插件和 MCP 连接;另一家是有专门团队在推广开始之前就把基础设施准备到位。这两种情况里,开发者的第一次体验都是顺畅高效的,推广也就从这种良好印象自然扩散开来。

文章还提到,目前真正在负责这件事的团队,通常归属在开发者体验或者开发者生产力这类职能下面,这通常也是负责新员工入职引导和搭建开发工具的部门。一个在好几家组织里都开始出现的新角色,叫做 agent manager,你可以理解成一种产品经理和工程师的混合岗位,专门负责管理整个 Claude Code 生态。对于那些还没有专门团队的组织,文章给出的最低限度方案,是设立一个 DRI,也就是直接负责人,这个人对 Claude Code 的整体配置拥有所有权,有权对设置、权限策略、插件市场、CLAUDE.md 的撰写规范做决策,同时也承担起持续维护这些东西始终保持最新的责任。

文章特别提醒了一个挺现实的风险,自下而上式的自发采用确实能激发热情,但如果没有人把真正有效的做法集中沉淀起来,这种热情很容易碎片化,变成各自为战。你需要有一个人或一个团队,专门去收集和推广好的使用习惯,比如标准化的 CLAUDE.md 层级结构,或一组经过筛选的 skills 和 plugins 组合,否则好的经验会一直停留在少数人脑子里,整体采用程度也会很快碰到天花板。

文章还谈到了治理层面的问题,尤其在受强监管的大型组织里会很早浮现。比如由谁决定哪些 skills 和 plugins 被允许使用,怎么避免几千名工程师重复造轮子,怎么保证 AI 生成的代码也经过跟人写代码一样严格的审查流程。建议是,一开始就圈定经过批准的 skills 集合,强制执行代码审查,先限制初期访问范围,等信心建立后再逐步扩大。落地最顺利的组织,往往很早就成立了跨职能工作小组,把工程、信息安全和治理合规的代表拉到一起,共同定义需求和路线图。

文章在收尾的地方也很坦诚地划出了适用边界。它说 Claude Code 是围绕传统软件工程环境设计的,假设工程师是代码库的主要贡献者,用 Git 做版本管理,目录结构遵循常规工程习惯。大多数大型代码库都符合这个模型,但如果场景比较特殊,比如游戏引擎里有大量二进制资源文件,或者用的是不常见的版本控制系统,又或者非工程师也会提交内容,就需要额外的配置工作,需要结合自己代码库和组织架构的具体情况去判断,Anthropic 也有一个应用 AI 团队专门跟企业直接合作落地。

听到这里,我们可以稍微停下来,想想这篇文章对我们普通听众,尤其是身处国内工程团队的朋友,到底有什么实际的参考价值。

如果你所在的团队也维护着一个规模不小的代码库,不管是单体仓库还是分布在多个仓库里的微服务集群,我觉得这篇文章给出的优先级顺序非常值得照着走一遍。先别急着去搭 MCP 服务器或者搞复杂的插件市场,先把最基础的 CLAUDE.md 文件写好,而且一定要克制住往里面塞东西的冲动,根目录只放真正大局观的内容和关键注意事项,局部规范放到对应的子目录里去。这一步看起来最朴素,但恰恰是决定 Claude 在你的代码库里好不好用的最大变量。

第二步,可以尝试在子目录里直接初始化会话,而不是习惯性地总在仓库根目录开工,如果是服务化架构,不妨把每个服务自己的测试和构建命令,写进对应目录的 CLAUDE.md 里。第三步,如果你的代码库涉及 C、C 加加、Java 这类有成熟语言服务器支持的语言,真的应该认真考虑投入时间把 LSP 集成搭起来,文章里那家企业软件公司专门为此在全公司提前部署 LSP,说明这件事的优先级可能比很多团队直觉上认为的要高。第四步,等基础打牢之后,再去考虑 skills 和 plugins,把已经验证过有效的工作流打包成可以分发给全公司的标准化模块,而不是任由它们继续停留在少数人脑子里。

最后,如果你是团队里的技术负责人,文章里关于组织层面的建议我觉得同样重要,甚至更容易被忽视。明确指定一个人,哪怕只是兼职角色,对 Claude Code 的配置和推广负责,这件事的投资回报可能比你想象的要高。我个人认为,这恰恰是国内很多团队最容易跳过的一步,大家把精力都花在技术选型上,却没人真正对工具的持续运营负责,结果用了一段时间就没人维护,效果也跟着衰减。

好,我们来给这一期做个总结。这篇文章的核心信息,可以归纳成一句话,在大型代码库里用好 Claude Code,关键不在于你用了哪个版本的模型,而在于你有没有花心思搭建那一整套围绕模型的工具生态,也就是文章反复强调的 harness。文章给出了具体的施工图,从最基础的 CLAUDE.md 文件,到 hooks、skills、plugins、MCP 服务器,再到 LSP 集成和 subagent,每一层都要按顺序搭建,并随着模型进步持续维护清理,不能一次搭完就置之不理。同时,纯技术配置只是一半,另一半是组织层面的责任分配,需要有人专门对生态系统的健康负责。我个人听完最大的感受是,这篇文章在提醒所有正在用 AI 编程工具的团队,不要把希望全部寄托在模型一次又一次的升级上,真正决定你能不能把这类工具用出生产力的,往往是那些看起来枯燥但很基础的工程习惯。好了,这一期就到这里,我们下期再见。
