大家好,欢迎收听 Claude FM。今天我们要聊的这篇文章,标题是 Onboarding Claude Code like a new developer,Lessons from 17 years of development,翻译过来就是,像带新人一样带 Claude Code,一个开发了十七年的项目教给我们的经验。这篇文章发表于二零二六年四月二十八日,发表到现在还不到两个月,所以它讲的基本就是当下这个阶段,大家在用 Claude Code 处理超大型老旧代码库时,最前沿的一线实践,时效性上完全不用担心。

这篇文章我个人很喜欢,因为它不是讲一个抽象的方法论,而是讲一个具体的人,具体的项目,具体的十七年。主角叫 Brendan MacLean,他是 Anthropic 的 Claude 开发者大使,他所在的实验室也是 Claude 开源项目支持计划的一部分。他维护的开源项目叫 Skyline,是一套用来做蛋白质分析的软件,帮研究人员在血浆和组织样本里检测和定量蛋白质,这件事对生物标志物的发现,对疾病研究,对新药研发都非常关键。Skyline 从二零零八年开始开发,一直做到现在,代码量超过七十万行 C Sharp 代码,十七年里一直维护着二十多万个自动化的夜间测试。顺便说一句,这篇文章里有个挺有意思的注脚,Anthropic 的联合创始人 Dario Amodei,以前就是这个实验室的一员,所以这个故事跟 Anthropic 自己的渊源还挺深的,某种程度上也可以理解为什么这家公司会特别认真地把这个案例写出来。

先说说这篇文章想解决什么问题,以及当时整个行业是个什么状态。我们都知道,这一两年 AI 编程工具突飞猛进,但凡用过这些工具的人都会发现,一旦项目大到一定程度,事情就开始变得别扭。你可以想象一下这个场景,你请了一位代课老师来上一堂课,这位老师很聪明,反应很快,但他从来没见过这个班的教学进度,不知道上节课讲到哪了,也不知道班里哪几个学生基础薄弱需要照顾。代课老师能临场把一道题讲明白,但他没办法接着上节课的脉络往下教,因为他对这门课没有持续的记忆。这其实就是早期很多人用通用 AI 编程工具碰到大型代码库时的真实写照。Brendan 一开始也是怀疑的,他觉得现代的 AI 编程工具,未必能像专门为 C Sharp 语言和这套环境量身定制的工具那样,真正理解他这几十万行代码。他最早是在浏览器里用 Claude 网页版试水,模式很简单,他描述一个问题,拿到一个回答,然后把整个 C Sharp 文件复制粘贴回项目里。这种方式只能处理那些跟项目代码没什么关联,可以单独描述清楚的小问题。他自己讲,一旦改动变得更碎片化,更渐进式,这个流程就变得非常吃力。每一次跟 Claude 网页版的对话,都像是从零开始,它不知道 Skyline 是什么,不知道各个模块之间怎么关联,更不知道十七年的开发历史里沉淀下来的那些约定和教训。这其实就是一个典型的老大难问题,大语言模型本身一次对话能装下的信息量,跟一个十七年量级的项目体量之间,存在巨大的落差。很多团队遇到这个问题,选择的办法是直接放弃,觉得这种工具只适合写写小脚本,碰不了真正的大项目,这也是当时不少老牌软件团队对 AI 编程工具普遍持保留态度的原因。

但 Brendan 没有止步于此,他做了一个很关键的联想。他突然意识到,自己面对 Claude 网页版的这种挫败感,跟他这十几年带新人,带本科生,带研究生,带博士后进实验室时的感觉,其实一模一样。一个刚进实验室的新人,你也不可能把七十万行代码一股脑甩给他,指望他第一天就能上手干活。你会先给他找一个范围有限,容易完成的小项目,带着他过一遍,等他理解力跟上来了,再慢慢扩大他能接触的范围,这几乎是所有带过团队的人都熟悉的直觉。Brendan 的思路转变就在这里,他说,我可以把 Claude 通过 Claude Code 引入到我这个大项目里,就像引入一个受训的新开发者一样,用足够的解释,帮它完成一个范围有限但能成功的小项目,然后把这次经验产出的成果,变成下一轮迭代更好的上下文。这句话其实是整篇文章的灵魂,我们接下来要讲的所有具体做法,都是这一个洞察的延伸。如果你换个角度想,这等于是把人类社会里磨了几千年的师徒制,平移到了人和智能体之间,新人需要时间理解全局,需要有人把零散的经验沉淀成可以传承的东西,智能体也一样需要,只是沉淀的载体从老师傅口头相传,变成了写进文件里的上下文。

这个洞察落地之后,第一件具体的事,就是把上下文这个东西,当成一个独立的,需要长期维护的项目资产来对待,而不是每次对话临时现场拼凑的东西。Brendan 的做法是,把所有给 AI 用的上下文,搬进了一个单独的代码仓库,叫 pwiz-ai,跟 Skyline 主代码库完全分开。这么做的理由很实际,因为这套上下文要适用于所有分支,所有时间点,它的演化速度跟代码本身是不一样的,如果硬塞在主代码库里,迟早会变得很受限。这就好比一家公司的新员工手册,你肯定不会把它塞进某一个项目的代码仓库里,因为这份手册要服务于公司所有的项目,所有的团队,它的更新节奏也跟某一行具体代码的提交节奏完全不同。这个仓库的根目录有一个 CLAUDE.md 文件,Claude Code 会自动读取这个文件来理解项目环境。但要注意,这个文件干的事,只是搭建环境,然后指引 Claude 去看真正相关的文档,用 Brendan 自己的话讲,这相当于给 Claude 一张地形图,告诉它哪里是哪里,而不是直接把专业知识灌输给它。这个区分我觉得特别重要,很多人一上来就想着把所有领域知识一股脑塞进一个配置文件里,结果文件越写越臃肿,Claude 反而抓不住重点,就像一本什么都往里塞的说明书,翻起来比不看还累。Brendan 的做法恰恰相反,地形图保持精简,真正的专业知识,放在另外一层,也就是技能里面。

这就引出第二个核心点,技能,英文是 skills,这是一种开放格式,用来给智能体灌输特定的能力和专业知识。技能可以手动触发,也可以让 Claude 自己判断什么时候该用。Brendan 对他最关键的几个技能,会写得非常明确,比方说他的调试技能,描述里直接写着,只要是在排查 bug,排查故障,排查任何意外行为,就必须加载这个技能。这个调试技能的作用,是把 Claude 从他称之为猜测加测试的模式里硬拉出来,逼着它先做根因分析,搞清楚问题到底出在哪里,然后才允许它去尝试修复。这个对比其实挺生活化的,有点像修车,一种修法是听到发动机有异响,就先换个火花塞试试,不行就换个皮带试试,纯靠试错碰运气,另一种修法是先接上诊断仪,把故障码读出来,顺着故障码去定位真正出问题的部件再动手。猜测加测试的方式,在小项目里也许能蒙对,但在一个十七年量级的代码库里是致命的,因为你根本不知道这次改动会不会在另一个看似无关的地方炸出新的问题,代码库越老,各个模块之间纠缠的隐性依赖就越多,一个看似无关的小补丁完全可能在三个月后的某个边界情况下引发新的崩溃。Brendan 另外还维护了一个面向项目本身开发流程的技能,负责定位 Claude 到这个项目和它的文档体系里,以及一个版本控制技能,把这个项目特有的提交规范和 PR 规范编码进去。这些技能有一个共同的设计原则,叫引用而不是嵌入,意思是每个技能本身尽量轻量,真正的内容放在一个中心化的文档知识库里,技能只是指过去的一个指针。这样做的好处是,技能本身好维护,不会因为内容重复而到处都要改,你只要更新一份文档,所有引用它的技能都会自动跟着同步,而不是散落在十个文件里,改了九个忘了第十个。Brendan 说,他原本最担心的事情是,Claude 真的能学会理解我这么大的一个项目吗。但随着这套机制跑起来,这个担心变得越来越清晰,答案其实是,上下文就是另一种需要持续维护和持续生长的项目资产而已,跟代码,跟文档,没有本质区别。

第三个核心点,是当 Claude 需要接触真实数据的时候,用 MCP 把它接进去。MCP 全称是模型上下文协议,简单说就是一种让 Claude 这类智能体,连接外部数据源和工具的标准协议,有点像给智能体配了一套通用插头,不管对面插座是什么系统,只要做了适配,智能体都能接上去拿数据,而不用每次都靠人工导出表格再贴回对话框。在这篇文章里,最让我印象深刻的一个例子,是 Skyline 教程截图的自动化重现。Skyline 的教程里有两千多张截图,过去这些截图靠人工对着软件一张张重新截,费时费力还容易出错,软件界面稍微改了个按钮位置,几十张教程图就全部过时,需要人工重新跑一遍流程再截一遍。现在这套流程已经做到几乎百分之百可重现的自动化,Claude Code 还把这套基础设施往前推了一步,加上了只看差异的视图,以及把像素级变化放大显示的功能,甚至 Claude 自己用 C Sharp 写了一个 MCP 服务器,专门让它自己有能力去看这些差异图。这个细节挺有意思的,相当于 Claude 给自己造了一双眼睛,去检查自己生成的东西对不对,而不是生成完就两眼一闭交给人去把关。另外一个例子是,Claude Code 现在每天早上会生成一份摘要,把 Skyline 夜间测试体系里的测试失败,异常报告,还有用户支持论坛里悬而未决的帖子,汇总好,直接送进 Brendan 的邮箱,等他坐下开始工作的时候,这份报告已经在等他了。支撑这份报告的,是 Claude 自己用 Python 写的另一个 MCP 服务器,这个服务器要同时打通三条完全不同的数据来源,一个是 LabKey Server 上的关系型数据,一个是团队邮件,一个是 GitHub 上带发布标签的代码记录。把三条互不相通的数据流揉到一份每日摘要里,这本身就是个不小的工程活,而且是 Claude 自己写出来的。这一点我觉得值得单独说一句,前面讲的上下文和技能,某种程度上还是人在主导设计,Claude 在执行,但到了 MCP 这一层,Claude 已经开始反过来给自己搭基础设施了,这跟单纯让 AI 补全几行代码,完全是两个量级的事情。

这三个核心机制,我自己更喜欢用一个统一的类比把它们串起来理解。你可以把整套体系想象成公司给新员工配的三件套,第一件是入职手册,对应的就是那个 CLAUDE.md 加上独立的上下文仓库,告诉新人公司大致是什么样,东西放在哪,该去哪查文档。第二件是岗位标准作业流程,也就是 SOP,对应的就是技能库,把具体岗位上反复出现的操作方法,踩坑经验,固化成一套套可以直接照着执行的流程。第三件是系统权限,对应的就是 MCP 集成,新人光看手册看流程还不够,真正干活的时候,他得有权限登录测试系统,登录工单系统,登录代码仓库,才能接触到第一手的真实信息。三件套缺一不可,光有手册没权限,新人什么也干不了,光有权限没流程,新人乱点一通容易捅出问题,光有流程没手册,新人不知道这套流程用在什么场合。Brendan 这一年多在 Skyline 上做的事情,本质上就是把这三件套,一件一件配给了 Claude Code。

讲了这么多机制,我们来看看这些机制到底换来了什么实际的产出,这也是文章里我最喜欢的部分,因为它特别具体,不是空对空地讲愿景。第一个例子,Skyline 里有一个文件视图面板,这是一个全新的界面,用来展示所有跟文档相关的文件,带文件系统监控,还能拖拽整理。这个项目原本做了快一年,后来负责的开发者离职了,就这么烂在那里没人接。Brendan 用 Claude Code 把它捡起来,两周之后就做完了,最后的提交记录里,Claude 是联合作者。Brendan 自己说,过去这种半成品项目,基本上就是永远被搁置的命运,因为学术实验室里人员流动特别快,研究生毕业了,博士后跳槽了,暑期实习生暑假结束就走了,任何一个项目只要负责人一换,后面的人很难无缝接上,因为半成品代码往往缺文档,缺注释,缺心智模型,接手的人光是搞懂前任想干什么就要耗掉大半精力。第二个例子更早,三年前,Brendan 因为失去了维护者,索性停掉了 Skyline 夜间测试管理模块的功能迭代,这个模块是用 Java 写的,跑在一个叫 LabKey Server 的科研数据门户上。最近,他找了一个熟悉 LabKey 的开发者,先用 Claude Code 生成了一套搭建文档,然后 Brendan 自己花了不到一天时间,就把这几年一直想加却没加的功能加上了,顺手用 CSS 把页面布局也改了,而这部分视觉设计的活,以前他都是花钱请专门的设计师来做的。第三个例子,是团队里一个原本对智能编程工具持怀疑态度的开发者,亲手做了一个新的绘图扩展,一个用来可视化离子迁移率数据的图谱面板,而且这个开发者主动把这个功劳记在了 Claude Code 头上,一个原本的怀疑者反过来成了案例的代言人,这种转变本身就很说明问题。Brendan 总结说,他现在看到的状态是,几乎每个人都开始愿意去尝试那些好玩的新功能了,这些功能以前大家可能因为手头工作太重,压根没空去碰。这几个例子放在一起看,你会发现一个很明显的趋势,就是 Brendan 自己讲的,他实验室里的开发者现在已经很少自己动手写代码了,更多时间是在指挥 Claude Code,让它自主生成自动化脚本,自主写 MCP 实现。这跟很多人对 AI 编程工具的想象不太一样,大家可能以为 AI 写代码就是帮你补全几行,但这里讲的其实是更高阶的事情,人退到了指挥和把关的位置上。我自己读到那个原本怀疑 AI 编程工具的开发者最后主动给 Claude 记功这一段时,感触挺深的,团队里推广任何新工具,最大的阻力往往不是技术本身,而是怀疑论者的心理防线,而这种防线一旦被一次实打实的成功案例攻破,后面的扩散速度会比管理层自上而下推动快得多,因为这是同事说服同事,而不是制度要求大家配合。

接下来我们聊聊这篇文章对我们普通听众,尤其是手头也维护着老旧代码库的工程师,到底有什么实际的参考价值,这部分我想多花点时间,尽量讲得具体一点,讲到你听完真的能动手试一下的程度。

第一条建议,文章里讲得特别直白,上下文是你最好的朋友。Claude 生成的待办清单和计划,是不会跨会话保留下来的,真正能留下来的,是你主动记录,主动维护的上下文。这一点恰恰是大多数人会跳过的环节,也是为什么很多人用 AI 编程工具用了一段时间,效果就卡在一个瓶颈上不去了。Brendan 的原话是,你要理解,如果你不主动把上下文记录下来,Claude 是没办法自己学会的,别指望什么魔法发生,要把搭建和维护这个上下文层,当成一项实实在在的投资,像对待项目里任何其他资产一样,给它做版本管理,让它持续生长,持续维护。如果你来做这件事,具体怎么落地呢,你可以学 Brendan 那样,单独开一个仓库专门放上下文,如果你的项目分支没那么复杂,放在同一个代码仓库里也完全可以,关键不在于放在哪里,关键在于它必须是被版本管理的,必须有人持续维护,而且在你真正需要的时候,它必须是可用的。具体到第一天能干什么,我建议你可以先花半小时,写一份属于你项目的根目录说明文件,不需要面面俱到,先把环境怎么搭,关键文档在哪里,几个核心模块大致负责什么,简单写清楚就够了,这就是你的第一版地形图。之后每隔一段时间,比如每次迭代复盘的时候,都顺手问自己一句,这次踩的坑,有没有什么是值得写进这份地形图,或者写进某个技能里的,长期坚持下来,这份文件就会像 Brendan 的 pwiz-ai 仓库一样,变成一份真正跟着项目一起生长的资产,而不是上线那天写完就再也没人打开过的摆设文档。

第二条建议,投资建设你自己的技能库。用技能把领域知识封装起来,让任何一次 Claude 的调用都能加载到这些知识。这里有个细节我觉得特别值得划重点,就是引用而不是嵌入这个原则。如果你打算给自己的项目搭一套技能体系,千万别把所有内容都写死在技能描述里,而是让技能去指向一个集中维护的文档库,这样你以后更新知识的时候,只需要改一个地方,而不是满世界去找哪个技能里也复制了一份过时的内容。具体可以先从哪几类技能下手,文章给出的范本是,一个负责给 Claude 介绍项目全貌和文档体系的技能,一个把你们团队提交规范和 PR 规范固化下来的版本控制技能,还有一个调试技能,专门用来防止 Claude 一遇到报错就开始瞎猜瞎改。这三类技能我个人觉得几乎是普适的,不管你做的是什么项目,新人进组要先弄懂项目全貌,提交代码要符合团队规范,排查故障要讲方法论,这三件事本身就是任何团队都要面对的。如果让我给一个具体的上手顺序,我会建议先做调试技能,因为它的投资回报最直接,你只要把团队里那些反复出现的踩坑经验,那些口口相传的排查思路写下来,立刻就能减少很多来回拉锯的低效对话。另外提一个小技巧,Brendan 给调试技能写的触发条件非常明确,直接写着只要是排查故障就必须加载,这种写法值得借鉴,如果你的技能描述写得太含糊,Claude 自己判断要不要加载的时候就容易漏判,反而起不到效果,所以触发条件宁可写得直白一点,甚至带点强制意味,也好过模棱两可。

第三条建议,在需要接触真实数据的场景里,该用 MCP 就大胆用 MCP。文章特别强调,要在 Claude 需要拿到真实数据的地方建立 MCP 集成,比如测试结果,异常报告,用户支持的对话记录。如果你来做这件事,可以想一想,你团队里有哪些信息,目前还分散在不同的系统里,需要人手动去几个地方拼凑才能看到全貌,这些往往就是最值得做成 MCP 集成的地方。前面提到的那个每日摘要,本质上就是把原本要人工巡查三套系统才能拿到的信息,变成了一份自动送到邮箱的报告,这种工作流程上的杠杆效应,可能比单纯让 AI 帮你多写几行代码,价值要大得多。换个角度想,如果你团队每周一早上开会前,总有个人要先去测试平台看看,再去邮箱翻翻,再去代码仓库看看最近发了什么版本,然后口头汇报给大家,那这件事本身就是一个现成的 MCP 候选项,值得你花时间把它自动化掉。

第四条建议,是专门写给做开源项目的听众的,但我觉得对任何团队都有启发。文章里说,对开源项目而言,搭建和维护这层上下文的分量格外重。开源项目通常没有专门的新人培养预算,除了写下来的东西,没有任何制度化的记忆,也没有任何保证说,某个贡献者明年还会留在项目里。而上下文这个东西,一旦建立起来,就对所有贡献者开放,而且会贯穿项目整个生命周期,这是人的脑子里的隐性知识永远做不到的事。文章原文用了一个挺感人的说法,Brendan 那个 pwiz-ai 仓库本身,也是一个开源的产物,这份上下文属于这个项目,不属于任何某一个具体的贡献者,它会比所有参与建造它的人都活得更久。我自己读到这一句的时候,会想到我们平时维护内部文档时常见的困境,文档写了没人更新,人走了知识就跟着走了,而这篇文章给出的思路,其实是把这套问题用工程化的方式重新框定了一遍,上下文不再是锦上添花的文档,而是项目能不能继续往前走的核心资产。哪怕你不是做开源项目,只是公司里一个普通团队,人员流动这件事一样存在,骨干跳槽,组织调整,这些事情发生的概率不比开源项目低,所以这条经验完全可以直接搬过去用。

当然我也要提一句,这篇文章讲的是一个特定场景下的成功案例,蛋白质分析软件,十七年的积累,一个对代码库理解极深的核心维护者。我个人认为,这套方法论能成功,很大程度上依赖于 Brendan 本人对这十七年代码库的理解深度,他知道该往 CLAUDE.md 里放什么,知道该把哪些经验抽成技能,这种判断力本身是需要长期积累的,不是随便拿一个新项目套这个模板就一定立刻见效。如果你的项目只有半年历史,可能你压根不需要一个独立的上下文仓库,先把第一版 CLAUDE.md 写好,先攒出第一个技能,就已经够用了,千万别一上来就照搬人家一套已经跑了一年多的完整体系,那样反而本末倒置。但这恰恰也是这篇文章最值得参考的地方,它给出的不是一个拿来即用的工具,而是一种思维方式的迁移,把你过去带新人的经验,直接套用到带 AI 智能体上面,这套思维方式是可以按你自己的节奏,从最小的一步开始逐步搭建起来的。

值得一提的是,这篇文章也从侧面印证了一个我个人一直比较认同的判断,智能体能不能在一个具体行业里真正落地,拼的往往不是模型本身聪明不聪明,而是周围这层工程化的脚手架搭得有多扎实。同样的 Claude Code,放在一个完全没有上下文沉淀的项目里,可能就是一个反应灵敏但什么都记不住的代课老师,放在 Brendan 这套体系里,就能变成一个跟得上十七年节奏的合作者。这中间的差距,不是模型升级带来的,而是这套上下文加技能加 MCP 的脚手架带来的,这也是为什么我觉得这篇文章对工程师比对纯粹的 AI 爱好者更有价值,因为它讲的本质上是一套工程方法,而不是一个模型评测。

我们最后总结一下。这篇文章讲的核心道理其实很朴素,你不会把一个七十万行代码的项目,一股脑甩给一个刚入职的新人,指望他第一天就出活,你会找一个有限的范围,带他走一遍,再随着他理解力的增长,慢慢放大他的活动半径。Brendan 用十七年时间在人身上验证过这套方法论,这一次,他把同一套方法论,用在了 Claude Code 身上,效果同样成立。只要你愿意像带新人一样,持续地给 Claude 喂上下文,搭技能库,接好它需要的真实数据,它就真的能在你的老旧代码库里,从一个什么都不懂的新人,一路成长成能独立挑大梁的伙伴。这期就先聊到这里,我们下期再见。
