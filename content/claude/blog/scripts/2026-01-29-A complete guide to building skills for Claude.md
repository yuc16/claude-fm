大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,英文标题是 A complete guide to building skills for Claude,翻译过来就是《为 Claude 构建技能的完整指南》。这篇文章发表于二〇二六年一月二十九日,是 Anthropic 官方博客上的一篇产品指南类文章。

先说一下时间线的问题。今天是二〇二六年六月二十一日,这篇文章发表到现在已经差不多五个月了。半年时间在 AI 这个行业里,说长不长,说短也不短。我们简单回顾一下背景,这样大家听起来会更有代入感。Anthropic 是在二〇二五年十月正式推出 Skills 这个功能的,也就是说,这篇指南是在 Skills 上线之后大概三个月发布的一份"使用说明书"加"最佳实践总结"。换句话说,这不是一个全新功能的发布公告,而是 Anthropic 在观察了大量开发者实际使用情况之后,沉淀出来的一份操作手册。所以它的价值不在于告诉你"我们又发布了什么新东西",而在于告诉你"这三个月里大家是怎么用这个功能的,踩了什么坑,什么打法是有效的"。这种来自一线反馈的总结性内容,即便过了五个月,核心的方法论部分依然是有参考价值的,只是如果你去实际操作,记得去官方文档确认一下界面和具体参数是否有更新,因为产品细节这块迭代通常会比方法论快很多。

那么我们先说说,Skills 这个东西到底是用来解决什么问题的。我们知道,Claude 作为一个大语言模型,也就是 LLM,它本身有非常强的通用能力,你可以让它写代码、改文案、做分析、回答问题。但是在实际的企业和个人使用场景里,大家很快就会发现一个痛点,就是很多工作并不是一次性的、临时的任务,而是重复性的、有固定套路的工作流程。比如说,你们公司每周都要做一份固定格式的周报,里面有固定的几个版块,固定的语气要求,甚至固定的图表样式。如果每次都要在对话里重新跟 Claude 解释一遍"我们公司的周报应该是什么样的",那效率提升其实是有限的,因为你把大量时间花在了"教"Claude 而不是"用"Claude 上面。

Skills 要解决的正是这个问题。它的核心思路,用我自己的话来总结,就是"教一次,用一直"。你把一个工作流程,比如说怎么写周报、怎么做特定格式的文档、怎么按照某种规范去审查代码,这些知识沉淀成一个可复用的技能包,之后每次需要执行类似任务的时候,Claude 就可以直接调用这个技能,而不需要你重新解释。这跟我们人类团队里"新人培训手册"的逻辑其实很像。一个新员工入职,你不会指望他凭直觉就懂公司的所有流程规范,你会给他一份操作手册,告诉他报销流程怎么走、代码提交规范是什么、客户沟通的语气要怎么把握。Skills 本质上就是给 Claude 写的这种操作手册,只不过它是结构化的、可以被模型识别和调用的。

文章里提到,过去这几个月,Anthropic 观察到了四类典型的用户群体在用 Skills。第一类是普通开发者,他们希望 Claude 能够稳定地遵循特定的工作流程,而不是每次表现得都有点随机。这一点我觉得特别值得展开讲一下。我们都知道,大语言模型有一个特点,就是同一个问题问两次,答案可能不完全一样,这种不确定性在创意写作场景里是优点,但是在工程流程里往往是缺点。比如说你希望 Claude 每次生成的 API 文档都遵循同一套格式规范,如果没有技能这种机制,你可能会发现这次生成的文档结构是 A ,下次又变成了结构 B,这对于需要规模化、标准化输出的团队来说是个麻烦事。技能相当于给模型上了一个"轨道",让它在执行这类任务的时候始终沿着同一条既定路径走,大大降低了输出的方差。

第二类用户是效率达人,也就是文章里说的 power users,他们在用 Skills 自动化一些重复性的任务,比如说文档创建、研究流程整理这一类工作。我自己的理解是,这类用户往往是单兵作战的知识工作者,他们没有团队去帮他们写自动化脚本,但是他们每天又要重复做大量结构相似的事情。比如说一个分析师,每天都要从不同来源整理资料,生成一份固定结构的研究简报。如果每次都手动操作,哪怕用 Claude 辅助,也还是要不断重复给指令。但如果把这套"从收集资料到整理结构再到产出简报"的完整流程沉淀成一个技能,效率达人就可以一次调用直接拿到接近最终成品的结果,省去了大量来回沟通的时间成本。

第三类是希望在整个组织内部统一 Claude 工作方式的团队。这一点对企业用户来说其实是非常关键的一个诉求。我们设想一个场景,一个公司里有几十个工程师都在用 Claude 辅助编程,如果每个人对 Claude 的"调教"方式都不一样,那么团队产出的代码风格、文档风格、甚至是代码审查的标准都会出现明显的不一致。这对于代码维护和团队协作来说是个隐患。Skills 提供的能力,就是让团队可以把"我们公司应该怎么用 Claude 做事"这件事固化下来,做成一套统一的技能库,所有团队成员调用的都是同一套标准。这其实跟企业内部推行统一的代码规范、统一的设计系统是同一个逻辑,只是这次规范的对象从人类员工变成了 AI 助手。

第四类用户是 MCP 连接器的开发者。这里需要解释一下 MCP 是什么。MCP 全称是 Model Context Protocol,模型上下文协议,这是 Anthropic 推出的一套开放协议,用来让大语言模型能够标准化地连接外部的数据源和工具,比如说连接到你公司的数据库、连接到某个第三方的 SaaS 平台、连接到本地的文件系统。你可以把 MCP 理解成给 Claude 装的"外接设备接口",有了这个接口,Claude 就能伸手去操作外部世界的各种系统,而不仅仅是在对话框里聊天。那么 MCP 连接器开发者面临的问题是什么呢,就是即便你把接口连好了,Claude 知道怎么调用这个接口,但它未必知道在什么场景下、用什么顺序、遵循什么业务逻辑去调用这些接口才是最优的。这就好像你给一个新员工配了一台电脑和各种系统账号,但是没有告诉他具体的工作流程,他知道这些系统存在,但不知道该怎么把它们串起来完成一个完整的业务动作。Skills 在这里起到的作用,就是把具体的工作流程知识叠加在 MCP 连接器之上,让 Claude 不仅有"手"能操作外部系统,还有"脑子"知道该怎么操作。这也是为什么这篇指南里特别留出了一个章节专门讲 MCP 加 Skills 的组合打法,因为这是当前生态里一个非常重要的演进方向。

讲到这里,我们可以稍微往深一层去看,标准化技能和增强型 MCP 集成,这两种技能形态到底有什么本质区别。标准化技能,我理解更多是依托 Claude 本身已经具备的内置能力,比如说生成文档、写代码、做分析,这些 Claude 本来就能做,技能只是把"怎么做得更符合你的偏好"这件事固化下来,属于是在已有能力基础上做"风格和流程"层面的定制。而 MCP 增强型的技能,则是把"能力的边界"往外推了一层,因为它涉及到调用外部系统、读取外部数据、触发外部动作,这已经超出了模型本身的能力范围,需要依赖 MCP 这层协议去打通。所以如果你是一个独立开发者或者个人用户,可能更多会从标准化技能入手,先把自己常用的几个工作流程固化下来。但如果你是企业级的应用开发者,尤其是已经在做 MCP 集成的团队,那么把技能叠加在 MCP 连接器之上,会是你接下来重点投入的方向,因为这才是真正把 Claude 从一个聊天助手变成一个能干活的智能体的关键一步。

接下来我们说说指南里提到的,构建技能的完整生命周期,这部分其实是这篇文章最核心的内容,也是对开发者最有实操价值的部分。文章里讲到,整个流程从规划和结构设计开始,一直到测试、迭代,最后到分发,是一个完整的闭环。

我们先说规划这一步。规划阶段最重要的事情,其实是想清楚你到底要把哪个工作流程固化下来。这一点听起来简单,但其实是很多人在实操的时候最容易翻车的地方。我自己的理解是,很多人一上来就想做一个"万能技能",试图把方方面面都覆盖进去,结果做出来的东西既不聚焦,又难以维护。一个更好的策略,应该是先挑出你日常工作里最高频、最标准化的两到三个workflow,先把这几个做扎实,再考虑扩展。这跟我们做产品的逻辑其实是一样的,先做最小可行版本,先验证核心场景跑通,再逐步迭代加功能,而不是一开始就贪大求全。

接着是结构设计这一步。技能需要有一个清晰的结构,告诉 Claude 这个技能是用来干什么的,什么时候应该被触发,具体的操作步骤是什么,有哪些边界条件需要注意。这有点像我们写一份标准作业程序,也就是常说的 SOP。一份好的 SOP 不会写成一篇散文,它会有清晰的触发条件,有明确的步骤拆解,有需要特别注意的例外情况说明。技能的设计逻辑也是类似的,你需要用清晰、结构化的语言告诉 Claude,在什么场景下应该调用这个技能,调用之后应该按照什么顺序去执行,遇到什么样的边界情况应该怎么处理。这一点其实对写作能力和逻辑思维能力都有一定要求,因为你本质上是在用自然语言写一份"给 AI 看的程序"。

然后是测试和迭代环节。这一步我觉得是最容易被忽视,但其实最重要的一环。很多人写完一个技能之后,觉得自己已经表达清楚了,就直接拿去用,结果发现实际效果跟预期有差距。这其实和我们写代码、写产品需求文档是一样的道理,你以为表达清楚了,但是执行方,无论是人还是 AI,理解出来的结果可能跟你想的完全不一样。所以测试这一步,本质上是要去验证 Claude 在实际触发这个技能的时候,是不是按照你预期的方式在执行,有没有遗漏的边界情况,有没有理解偏差的地方。指南里建议的做法,是用多种不同的真实场景去触发这个技能,观察输出结果,然后针对发现的问题去调整技能的描述和结构,这是一个典型的迭代循环,不是一次成型的事情。

最后一步是分发。如果你做的这个技能不仅仅是给自己用,而是要给团队甚至给更广泛的用户使用,那么就涉及到如何打包、如何说明使用方法、如何让别人在不了解你原始思路的情况下也能正确使用这个技能。这一点对团队协作场景特别重要,因为一份好的技能如果只有作者自己懂怎么用,那它的价值就会大打折扣。

文章里有一个数据点我觉得特别值得拿出来单独说一下,就是如果你已经明确知道自己想要自动化的前两到三个工作流程,使用官方提供的 skill-creator 这个工具,大概十五到三十分钟就可以构建并测试出第一个可用的技能。这个时间成本其实是相当低的。我们可以做个简单的对比,如果你要给一个新员工做培训,光是讲清楚一个工作流程的来龙去脉,可能都需要半天甚至一天的时间,还不算后续答疑和磨合的成本。而用 skill-creator 这个工具,半个小时之内就能跑出一个可用版本,这个效率提升是非常明显的。当然,半小时跑出来的是"可用"版本,不代表是"完美"版本,后续还是需要经过前面说的测试迭代环节去打磨。但作为一个起步门槛,十五到三十分钟这个数字,确实是降低了很多人去尝试构建技能的心理门槛。

讲完了核心内容,我们来聊聊这对大家实际工作意味着什么,也就是落地应用这块。

第一个建议,是从你自己工作中最痛的那个重复性环节下手,不要一开始就想着覆盖所有场景。如果你是程序员,可能你每次写完一个功能都要补单元测试,而且测试的风格、覆盖范围、命名规范都有自己的一套偏好,这种场景就特别适合做成一个技能。如果你是做内容运营的,可能你每周都要写一篇固定结构的行业观察,这种也是非常典型的可固化场景。关键是先挑一个真正高频、真正让你觉得"每次都要重新解释好烦"的场景下手,这样投入产出比是最高的。

第二个建议,是把技能的描述写得尽可能结构化和具体。我们前面提到,技能本质上是用自然语言写给 AI 看的操作手册,这意味着你写的时候要有意识地去想,如果一个完全不了解背景的人,只看你写的这份说明,他能不能准确地执行这个流程。如果答案是不能,那说明你的描述里还有隐含的、没有写出来的假设,这些隐含假设正是后续出问题的根源。我个人的经验是,但凡涉及到给 AI 系统下达指令,凡是你脑子里觉得"这个不用说,大家都懂"的地方,往往就是最需要明确写出来的地方,因为 AI 不会像人类同事一样根据上下文自动补全你没说出来的潜规则。

第三个建议,是不要指望一次成型,要把技能当成一个需要持续维护的资产,而不是一次性的配置工作。这一点跟我们维护代码库、维护产品文档其实是同一个道理。业务流程会变,你的偏好会变,公司的规范会变,如果技能写完之后就再也不去看了,那它很快就会和实际的工作方式产生偏差,反而会变成一种负担,因为按照过时技能产出的结果可能还需要你重新返工修改。所以比较健康的做法,是把技能的维护纳入到你常规的工作流程里,定期回顾,定期根据实际使用反馈做调整。

第四个建议,是如果你所在的团队已经在用 MCP 去连接各种内部系统,那么一定要认真考虑把技能叠加在这些连接器之上,而不要把技能和 MCP 集成当成两个割裂的事情来做。文章里特别强调了 MCP 连接器开发者这个群体,说明 Anthropic 自己也意识到,单独的连接能力和单独的技能能力,叠加在一起产生的价值会远大于各自独立存在的价值。如果你的团队已经投入资源去做了 MCP 集成,但是没有配套的技能去指导 Claude 怎么合理地使用这些连接,那其实是浪费了已经投入的工程资源。

第五个建议,是把技能当作团队协作和知识沉淀的载体,而不仅仅是个人效率工具。很多团队在引入新工具的时候,容易把它当成"个人生产力插件"来用,每个人各自摸索,各自形成自己的一套用法。但技能这个机制天然具备团队属性,因为它可以被打包、被分享、被统一管理。如果你是团队负责人,我个人会建议尽早建立一个内部的技能仓库,把团队里大家摸索出来的好用的技能集中管理起来,这样新人加入团队的时候,直接复用已有的技能库,就能很快达到团队里"老人"的工作效率水平,这其实是一种非常高效的知识传承方式,比传统的文档培训要直接得多。

我们再花一点时间聊聊潜在的坑,这部分文章原文没有展开讲太多细节,但是结合我自己对这类工具的理解,我想提醒大家几个需要小心的地方,这些纯粹是我个人的推测和经验性建议,大家可以批判性地参考。第一个坑,是过度设计。前面也提到过,有些人会希望一个技能尽可能通用,试图覆盖各种边缘情况,结果技能本身写得又长又复杂,Claude 在执行的时候反而容易在各种分支条件里出现理解偏差,执行效果还不如一个简单明确、只覆盖核心场景的版本。第二个坑,是技能之间的边界不清晰。如果你的技能库里同时存在多个相似的技能,Claude 在判断应该触发哪一个的时候可能会出现误判,所以技能的命名和触发条件需要尽量保持清晰的区隔,不要让多个技能的适用场景产生大量重叠。第三个坑,是缺乏版本管理意识。技能既然是会持续迭代的资产,那么团队协作的时候就需要有清晰的版本记录,知道当前生产环境用的是哪个版本的技能,改动之后有没有经过充分测试,这跟软件工程里的代码版本管理是同一套思路,不能因为技能是用自然语言写的,就放松了工程化管理的要求。

总结一下今天这期节目的内容。这篇指南的核心信息其实非常清晰,就是 Skills 这个机制解决的是大语言模型在重复性工作流程中输出不稳定、需要反复解释的问题,它的价值是"教一次,用一直"。文章面向四类典型用户,开发者、效率达人、希望统一团队工作方式的管理者,以及 MCP 连接器开发者,每一类用户都能在技能机制里找到适合自己的应用场景。构建技能是一个从规划、结构设计,到测试迭代,再到分发的完整生命周期,而且借助官方的 skill-creator 工具,上手门槛已经被压得很低,十五到三十分钟就能跑出第一个可用版本。

我个人的看法是,Skills 这个方向代表的是 AI 助手从"通用聊天工具"向"可定制的生产力基础设施"转变的一个重要趋势。我们这几年看到的趋势,是大模型的基础能力在快速提升,但是真正决定一个 AI 工具能不能在企业和个人工作流里产生持续价值的,往往不是模型本身聪明不聪明,而是这个模型有没有办法被精确地"调教"成符合你具体场景需求的样子,而且这种调教成果能不能被沉淀、被复用、被团队共享。Skills 机制恰恰是在回应这个需求。当然,五个月过去了,这个生态肯定还在快速演进,如果你对这个话题感兴趣,建议直接去看一下最新的官方文档,确认具体的工具界面和能力边界有没有更新。今天的节目就到这里,我们下期再见。
