我们先铺垫一下背景。二〇二五年三月前后,整个大模型应用开发的行业处在一个什么阶段。那个时候,绝大多数团队已经度过了最早期的玩具阶段,也就是单纯跟 Claude 聊天看看它能不能写诗、能不能写代码的阶段。大家开始真正认真地把大模型嵌入到生产系统里,做客服机器人,做代码助手,做内容审核,做各种垂直场景的智能体应用。但是这个时候,行业里普遍遇到一个很现实的麻烦,那就是提示词工程这件事,远远没有想象中那么简单。

你可能会问,提示词不就是写一段话告诉模型该干什么吗,能有多复杂。我们来看一下实际情况。一个真正要上生产的提示词,往往要经过反复打磨,要塞进具体的业务规则,要插入示例,要考虑边界情况,还要适配不同的工具调用。而且写好一版之后,你怎么知道这一版比上一版好,光靠人眼盯着输出结果去感觉,显然不靠谱,尤其是当你的应用要服务成千上万种不同的输入场景的时候。再加上,一个产品从想法到上线,往往不是一个人在写提示词,而是工程师、产品经理、业务专家、质量保证人员一起在磨。这时候问题就来了,这帮人怎么协作,提示词改了一版,该发到哪里去给别人看,別人提的意见怎么合并回去,版本对不上怎么办。

在这篇文章发表之前,很多团队的真实做法是什么样的呢,我们可以想象一下那个画面,工程师在自己的笔记里写了一版提示词,然后把这段文字复制粘贴到一个共享文档里,业务专家看了之后在文档里写评论,工程师再把改完的版本粘贴回代码,过几天产品经理又提了新需求,于是又复制一份发到聊天群里讨论。这个过程里,经常会出现的情况是,聊天记录里那一版其实是最新最好的,但代码库里跑的还是三天前那个旧版本,谁都不知道现在生产环境上线的到底是哪一版提示词。这就是文章开篇提到的真实痛点,也是这次 Anthropic Console 升级要解决的核心问题。

好,接下来我们正式进入文章的核心内容,我会挑出文章里最重要的几个点,带大家一层一层拆开讲。

第一个核心点,是 Anthropic Console 这次升级的整体定位变化。原来的 Console,更多是一个单点工具的集合,你可以在里面写提示词,可以在里面测试,但它更像是个人开发者的实验台。这次升级之后,Anthropic 明确把它重新定位成,一个让你和你的团队一起构建、测试、迭代 AI 部署的统一场所。这句话听起来简单,但背后的产品哲学转变是很关键的。它意味着 Anthropic 已经意识到,提示词工程不再是一个孤立的技术活儿,而是一个需要协作流程支撑的工程实践,跟写代码需要代码评审、需要版本控制是一个道理。

如果把这个类比往前推一步,我们可以想象一下软件工程的发展史。最早期写代码的时候,大家也是各自在自己电脑上写,写完了把文件打包发给同事。后来出现了版本控制系统,出现了代码评审流程,这才让团队协作写代码变得高效可靠。提示词工程现在正在经历类似的阶段,从个人手艺活,逐渐走向工程化、流程化、协作化。这次 Console 升级,可以理解成提示词工程领域第一次比较系统地把协作能力纳入产品设计的核心位置。

第二个核心点,是文章里详细介绍的提示词全生命周期工具链,一共分成四个环节,写提示词、自动生成提示词、评估模型响应、改进提示词。我们一个一个来看。

第一个环节是写提示词,文章里提到一个叫 Workbench 的工具,中文可以理解成工作台。你可以在这个交互式环境里搭建你的提示词结构,加入示例,接入外部工具,然后直接测试 API 调用的效果。这个其实是最基础的能力,任何一个提示词调试工具都得有,但 Anthropic 把它做成了一个所见即所得的环境,你改一句话,马上能看到模型怎么回应,这种即时反馈对提示词调试效率的提升是很直接的。

第二个环节,叫自动生成提示词,这个我觉得是这四个环节里最有意思的一个。你只需要用自然语言描述你想达成的目标,Claude 就会运用提示词工程的各种技巧,比如说思维链推理,自动帮你生成一段有效、精确、可靠的提示词。思维链推理简单解释一下,就是让模型在给出最终答案之前,先一步一步把中间的推理过程写出来,这样往往能让模型的回答更准确,因为它把复杂问题拆解成了一连串简单的小步骤。

这个功能背后其实有个很巧妙的设计思路,那就是用模型本身去优化提示词。你可以把它理解成,你要请一个写文案的人帮你写广告语,但这个写文案的人本身就是非常懂广告语写作技巧的专家,他不光会写,还知道什么样的结构和措辞更容易打动人。这里 Claude 既是最终执行任务的那个模型,某种程度上也扮演了提示词工程师的角色,帮你把一个模糊的目标描述,转化成一段结构清晰、包含合适示例和推理引导的专业提示词。

对于很多不是天天写提示词的产品经理或者业务专家来说,这个功能的价值非常实际。以前你想让模型帮你做某件事,得自己琢磨怎么措辞,要不要加示例,要不要强调输出格式,现在你只需要把你想要的结果用大白话描述清楚,剩下的技巧性工作交给 Claude 自己去补全。这其实也呼应了我们前面讲的协作问题,因为业务专家往往是最懂业务规则、但最不懂提示词技巧的那群人,有了自动生成功能,他们也可以直接参与到提示词的初稿创建里来,而不是完全依赖工程师做翻译。

第三个环节,是评估模型响应,这个环节我认为是整个工具链里最体现工程严谨性的部分。文章提到,你可以针对真实场景去评估你的提示词,系统支持自动生成测试用例,还支持把不同版本的输出结果并排比较。你可以直接运行测试集,给回答质量打分,然后基于数据去决定哪个版本的提示词应该被部署上线。

这一点其实戳中了提示词工程长期以来的一个软肋,那就是缺乏客观的评估标准。我们写代码的时候,有单元测试,代码改完了,测试一跑,通过还是不通过一清二楚。但提示词改完了怎么知道好不好呢,以前很多团队的做法就是工程师自己拿几个例子试一试,感觉答案顺眼就上线了,这种做法在小范围试验的时候问题不大,但一旦你的应用要服务成千上万种真实用户输入,这种凭感觉的判断方式就完全靠不住了。

举个例子,假设你在做一个客服机器人,处理用户退款请求。你改了一版提示词,想让模型在处理退款理由的时候更精准。如果没有系统化的评估手段,你可能就拿手头三五个常见的退款场景试一试,看着还行就上线了。但是真实世界里,用户提退款请求的方式千奇百怪,有人写得很详细,有人就甩一句话,有人夹杂着情绪化的抱怨,你那三五个测试场景根本覆盖不了这种多样性。万一新版本在某个边缘场景下表现变差了,你上线之后才会发现问题,这时候代价已经产生了。

而有了自动生成测试用例和并排比较的能力之后,你可以系统性地把各种真实场景都纳入测试集,新旧版本跑一遍,各项指标摆在一起对比,哪个更稳定、哪个在边缘场景下表现更好,一目了然。这种数据驱动的决策方式,本质上是把提示词工程从手艺活往工程学科那个方向再推了一步。

第四个环节,是改进提示词,这个环节是自动优化已有的提示词,运用更高级的提示词工程技巧来打磨。文章特别提到一个使用场景,就是当你的提示词原本是为别的模型写的,现在想迁移到 Claude 上来用,这个改进功能特别适合做适配性优化。这其实也反映了一个很现实的行业情况,很多团队在尝试不同模型的时候,并不是从零开始写提示词,而是把已有的提示词拿过来改改就用,但不同模型对提示词风格的偏好是不一样的,直接照搬效果往往打折扣,有这么一个自动改进的功能,可以省去很多人工调试的功夫。

而且这个改进功能不仅适用于模型迁移场景,对你自己手写的提示词也同样有用,相当于有一个经验丰富的提示词专家随时帮你做二次打磨。文章里提到,等你最终敲定了提示词,只需要点一下获取代码,就能拿到一段可以直接部署到生产环境的 API 调用代码。这一步看起来简单,但其实是把整个从构思到上线的链路打通了,你不需要再手动去拼接代码、调整参数格式,直接复制粘贴就能用,大大缩短了从想法到生产之间的距离,这也正好对应了文章标题里那句更快把你的 AI 应用送上生产环境。

第三个核心点,我们要重点讲讲共享提示词这个协作功能,这也是整篇文章里我个人觉得最值得展开聊的部分。

文章里说得很直接,开发者、领域专家、产品经理、质量保证专家,经常需要一起协作打磨提示词才能拿到最佳效果。但是以前,团队通常的做法是在文档之间或者聊天软件之间来回复制粘贴提示词,这就会导致版本控制问题和知识孤岛问题。

我们刚才已经描述过这种混乱的协作场景了,这里我想再深入聊聊知识孤岛这个说法,我觉得这个词用得挺精准的。什么叫知识孤岛,就是说团队里某个人,可能是某个资深工程师,他通过反复试验摸索出了一套很好用的提示词写法和调试技巧,但这些经验只存在他自己的脑子里和他自己电脑上的几个文件里,团队里其他人完全接触不到这些经验。等这个人休假了,或者换了项目,这些经验也就跟着消失了,团队里其他人遇到类似问题,又要重新摸索一遍,完全是在重复发明轮子。

共享提示词功能,本质上就是要打破这种孤岛状态。文章说,现在 Anthropic Console 提供了一个集中化的方式,让你在整个组织范围内开发、打磨、标准化提示词。团队成员可以访问并协作一个共享的提示词库,这样更容易建立最佳实践,确保所有用 Claude 构建的应用都能保持一致的质量水准。

这里我想用一个类比帮大家理解这个功能的价值。把提示词库想象成一个公司内部的代码仓库,在软件工程领域,我们早就习惯了把通用的、经过验证的代码逻辑沉淀成共享的函数库或者组件库,新来的工程师不需要重新发明一遍处理日期格式的代码,直接调用现成的、已经被无数人验证过的工具函数就行了。提示词共享库想要做的事情,其实是一样的逻辑,把组织里已经验证有效的提示词写法沉淀下来,变成可以复用、可以迭代、可以被所有人看到改进历史的资产,而不是散落在各个人的聊天记录和文档碎片里。

而且共享提示词还解决了另一个隐性问题,就是质量一致性。假设你的公司有五个不同的产品团队,都在各自调用 Claude 做客服场景的应用,如果每个团队各自摸索各自的提示词写法,很可能会出现五套风格完全不一样、质量参差不齐的提示词,有的团队摸索得比较透彻,处理得很周到,有的团队可能因为人手紧张,提示词写得比较粗糙。有了共享提示词库之后,质量好的写法可以被其他团队直接参考和复用,整个组织对外呈现出来的 AI 产品体验也就更容易保持一致。

第四个核心点,我们聊聊 Claude 三点七 Sonnet 和扩展思考预算控制这部分内容。文章里提到,Claude 三点七 Sonnet 是当时最新、最聪明的模型,它既可以给出近乎即时的回应,也可以进行扩展的、一步一步的思考过程,而且这个思考过程是可以展示给用户看的。这就是所谓的扩展思考能力,英文叫 extended thinking。

这里我简单解释一下扩展思考是什么意思。普通情况下,模型收到一个问题,直接给出答案,这个过程对用户来说是黑箱的。而扩展思考模式下,模型会先生成一段可见的思考过程,把自己是怎么一步步推理得到答案的过程展示出来,这对于处理复杂问题特别有用,比如数学推理、复杂的逻辑判断、需要多步骤分析的业务场景。这跟我们之前提到的思维链推理是同一个思路,只不过现在这个思考过程对用户是可见的,而且它的篇幅和深度是可以控制的。

文章里提到一个很关键的细节,就是 Anthropic Console 现在支持设置思考预算,也就是给思考过程设置一个最大令牌数,令牌可以简单理解成模型处理文字时的最小计量单位,有点像我们说话时的字数或者音节数。你可以通过设置这个预算上限,来控制模型在思考阶段最多能写多少内容。

这个功能为什么重要,我们可以从权衡的角度来理解。扩展思考能让模型在复杂任务上表现得更好,但思考得越久越细致,消耗的计算资源和时间也就越多,对应的成本和延迟也会上升。不是所有场景都需要模型想得那么深,有些场景你需要的是快速响应,比如简单的客服问答,而有些场景你确实需要模型仔细推敲,比如复杂的合同条款分析或者代码调试。思考预算这个功能,就是把这个权衡的控制权交到了开发者手里,让你可以根据具体场景去调节,既不浪费资源去做不必要的深度思考,也不会在真正需要深度推理的场景上偷工减料。

文章里还提到一点,就是在开启扩展思考之后,提示词的写法基本上还是延续原来的逻辑,但 Anthropic Console 让你可以方便地针对扩展思考场景去优化提示词,你只需要告诉系统这段提示词会配合扩展思考一起使用,Claude 就会针对这个使用场景去生成最合适的回应方式。这也算是前面讲到的自动生成提示词功能的一个延伸应用,把扩展思考这个新能力也纳入到了整个提示词工具链里。

好,核心内容我们讲完了,接下来我们聊聊这些东西对今天的听众,也就是正在做实际工作的工程师和 AI 从业者来说,到底意味着什么,能怎么用,要避开什么坑。

首先我想强调一点,虽然这篇文章发表已经有十五个月了,文章里提到的具体模型版本早就过时了,但它背后的产品理念是经得起时间考验的。如果你现在所在的团队,提示词管理还停留在我们前面描述的那种复制粘贴、版本混乱的状态,那么这篇文章里提到的几个思路,依然非常值得你去借鉴和落地。

第一条实践建议,是关于提示词的版本管理。如果你的团队现在还在用聊天群和文档协作提示词,我建议你认真考虑引入某种集中化的提示词管理方式,不管是用 Anthropic Console 这样的官方工具,还是自己搭建一套轻量级的内部系统,核心目标是要做到任何时候,任何人都能清楚知道生产环境上跑的到底是哪一版提示词,以及这一版提示词经过了哪些评审和测试。这一点说起来简单,但很多团队规模小的时候完全意识不到这个问题的严重性,等团队规模变大,产品线变多了,才发现提示词管理混乱已经成了一个真实的生产事故隐患。

第二条实践建议,是关于评估体系的建设。文章里反复强调用真实场景的测试用例去评估提示词,这一点我觉得是所有做提示词工程的团队都应该认真对待的事情。如果你现在判断一个提示词改得好不好,还停留在工程师自己拿几个例子试一试看着顺眼的阶段,我强烈建议你尽快往系统化评估这个方向走。具体怎么做呢,你可以先从收集真实的用户输入案例开始,把这些案例整理成一个测试集,覆盖尽可能多的边界情况和典型场景,然后每次改动提示词,都用这个测试集去跑一遍,新旧版本对比,看看具体哪些指标变好了,哪些变差了。哪怕你现在用的工具不是 Anthropic Console,这个评估的思路本身也是完全可以迁移到任何工具链上的。

第三条实践建议,是关于自动生成和自动优化提示词这两个能力的实际使用场景。我个人认为,这类自动化能力最适合用在两个地方,一个是项目初期,当你完全没有提示词写作经验,或者只是想快速验证一个想法是否可行的时候,用自然语言描述需求,让模型帮你生成第一版提示词,可以大幅缩短从想法到可测试原型之间的时间。另一个场景是模型迁移,如果你之前的提示词是针对别的模型写的,现在想切换到 Claude 上来,用自动改进功能做一次适配性优化,往往能比你自己手动调整更高效。但我要提醒一点,自动生成出来的提示词,只是一个起点,不是终点,真正面向生产环境的提示词,还是需要结合你自己业务的具体规则和边缘情况去人工打磨,不能指望完全交给自动化工具一次性搞定。

第四条实践建议,是关于团队协作流程的设计。如果你是团队负责人,或者负责 AI 产品落地的技术负责人,我建议你认真思考一下,在你的团队里,谁应该有权限修改生产环境的提示词,谁负责评审,提示词改动是不是应该跟代码改动一样,经过类似代码评审的流程才能合并上线。这听起来可能有点重,但提示词其实就是你 AI 应用的核心业务逻辑,它出错造成的影响,跟代码逻辑出错是一样严重的,甚至有时候更隐蔽,因为提示词层面的问题,往往不会报错,只会悄悄地让模型输出质量下降,而这种下降很容易被忽视,直到积累到一定程度才会被用户投诉发现。

第五条实践建议,是关于扩展思考和思考预算的取舍。如果你正在用支持扩展思考的模型构建应用,我建议你针对不同的业务场景做分级处理,不是所有请求都需要开启深度思考。你可以根据任务的复杂程度,动态决定要不要开启扩展思考,以及给多大的思考预算。比如简单的分类任务,可能完全不需要扩展思考,而复杂的多步骤推理任务,比如需要模型先理解一份长文档再做总结分析,这种场景就值得给更大的思考预算。这种分级处理的思路,其实跟我们做系统架构设计时考虑性能和成本权衡是一个道理,没有放之四海皆准的最优解,只有针对具体场景的合理取舍。

讲到这里,我也想从一个更宏观的角度,聊聊我个人对这次升级和这类工具演进方向的看法。我个人认为,这篇文章所代表的趋势,其实反映了整个大模型应用开发行业正在经历的一次成熟化过程。最早期的时候,大家觉得用大模型做应用,核心难点是模型能力够不够强,模型不够聪明,应用就做不好。但随着模型能力越来越强,大家慢慢发现,真正卡住生产落地速度的,往往不是模型能力本身,而是围绕模型能力建立起来的工程流程和协作机制,够不够成熟。提示词怎么写、怎么测试、怎么协作、怎么管理版本,这些看起来不那么炫酷的工程问题,反而成了决定一个 AI 产品能不能稳定、高质量、快速地走向生产环境的关键因素。

这次 Anthropic Console 的升级,某种程度上就是 Anthropic 在用产品的方式回应这个行业现实,把原来散落在各处的提示词工程能力,整合成一条完整的、支持团队协作的工作流。我觉得这个方向是对的,而且我估计,这类围绕提示词全生命周期管理的能力,在未来还会继续演进,可能会出现更细粒度的权限管理,更智能的自动评估指标,甚至跟你公司内部的其他研发工具链,比如代码仓库、持续集成系统,做更深度的打通。

当然,我们也要客观地看,这篇文章本身篇幅不长,更多是一个产品发布公告性质的内容,它没有深入讲技术实现细节,也没有给出非常具体的数据来证明这些新功能到底能带来多大的效率提升。所以我们听完这期节目,知道这次升级讲了什么道理就够了,具体这些功能在实际使用中效果如何,还是需要大家自己动手试一试,结合自己团队的实际场景去判断。

好,我们最后来做一个总结。这篇发表于二〇二五年三月的文章,讲的是 Anthropic Console 的一次重要升级,核心是把写提示词、自动生成提示词、评估提示词、优化提示词这四个环节,整合进了一条完整的工作流,同时引入了团队共享提示词的协作能力,解决了过去提示词管理混乱、知识孤岛、协作低效这些真实痛点,还顺带支持了当时最新的 Claude 三点七 Sonnet 模型和扩展思考预算控制功能。

放在今天二〇二六年年中这个时间点回看,具体的模型版本早就过时了,但这篇文章背后透露出的产品理念,把提示词工程当作需要工程化流程支撑的协作实践,而不是单纯靠个人手艺去解决的零散问题,我个人认为这个判断到今天依然成立,甚至随着大模型应用规模越来越大、团队越来越多,这个判断会变得更加重要。如果你的团队现在还在用复制粘贴提示词这种原始方式协作,这篇文章给你的最大启发,可能不是某个具体功能,而是提醒你,该认真考虑把提示词工程当成一项正经的工程实践来对待了。
