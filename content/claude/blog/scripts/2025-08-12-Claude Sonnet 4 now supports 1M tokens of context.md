我们先倒回去看看,这篇文章发布的那个时间点,行业是个什么状态。二〇二五年这一年,可以说是大模型圈子里"上下文窗口"这个指标被反复拿出来卷的一年。什么叫上下文窗口,简单说就是模型在一次对话或者一次请求里,能够"看到"和"记住"的内容总量,单位是令牌,也就是 token。Token 这个词大家可能已经很熟悉了,简单理解就是模型处理文本的最小单位,一个汉字大概算两三个 token,一个英文单词大概算一个到一个半 token。在 Claude Sonnet 四点零这次升级之前,它的上下文窗口是二十万令牌,这个数字其实已经不小了,大概能装下一整本中等厚度的书。但问题是,真实世界里的很多任务,二十万令牌还是不够用。

举几个例子你就明白了。比如说,你是一个软件工程师,手头有一个中型项目,代码仓库里有七万行、八万行代码,加上测试文件、配置文件、文档说明,轻轻松松就超过二十万令牌的容量。这时候你想让 AI 帮你做一次全局的代码审查,或者帮你重构一个跨越几十个文件的功能,你就没法把整个项目一次性喂给模型,只能东拆西凑,喂一部分代码、问一部分问题,模型看不到全貌,给出的建议自然也是局部的、可能还互相矛盾的。再比如说,你是做法律或者金融行业的,手头有几百份合同,你想让 AI 帮你做一次交叉比对,找出条款里的不一致之处,二十万令牌的窗口同样是装不下的。还有一种场景,是现在越来越火的 agent,也就是智能体,你让它自主地去执行一个任务,中间要调用很多工具、查很多资料、跑很多步骤,每一步的历史记录都要累积在上下文里,跑个几十步、上百步,二十万令牌很快就被吃光了,模型就开始"失忆",忘了前面做过什么、为什么要这么做。

所以,在这样一个背景下,Anthropic 把 Claude Sonnet 四点零的上下文窗口从二十万令牌一口气扩大到一百万令牌,放大了整整五倍。这不是一个简单的数字游戏,而是直接打开了一整批之前做不到的应用场景。我们接下来就分几个层次,把这次更新的核心内容拆开来讲清楚。

第一个核心点,我想先讲清楚一百万令牌到底是个什么概念,因为光说"五倍"这个数字,大家可能没有直观的感受。文章里给了一个具体的参照,一百万令牌大概可以装下超过七万五千行代码,或者是几十篇研究论文。我们用一个生活化的类比来理解一下,如果说二十万令牌的上下文窗口相当于你能一次性记住一本厚厚的小说,那一百万令牌就相当于你能一次性记住一整套丛书,五本厚厚的小说叠在一起还要更多。换到代码的场景里,七万五千行代码是个什么规模,很多中型甚至中大型的商业项目,核心代码量也就是这个量级,这意味着像 Anthropic 自己说的,你可以"加载整个代码仓库,包括源代码文件、测试文件和文档",让 Claude 一次性理解整个项目的架构。这跟之前那种"只能看局部、靠人工拼接上下文"的模式,完全是两种工作方式。

这里我想多展开讲讲为什么"看到全局"这件事对代码分析特别重要。我们写代码的时候经常遇到一种情况,叫跨文件依赖,就是说,你在 A 文件里改了一个函数的参数,结果 B 文件、C 文件、甚至 D 文件里有地方在调用这个函数,如果你不知道这些调用关系,改完之后系统就崩了。以前如果你让 AI 帮你做这种重构,你得自己先去搜索整个项目,把所有相关的文件手动找出来,再一个一个喂给模型,这个过程本身就很容易漏掉一些边缘情况。现在如果上下文窗口足够大,你可以直接把整个代码仓库喂进去,模型自己就能在内部建立起这些跨文件的依赖关系图,它能"看到"A 文件改了之后,B、C、D 文件分别会受到什么影响,然后给出一个真正考虑到"完整系统设计"的修改建议。这就是文章里强调的,Claude 能够"识别跨文件依赖,并提出考虑到完整系统设计的改进建议"。这种能力对做大型项目重构、做代码审查、做架构梳理的工程师来说,价值是非常直接的。

第二个核心点,我们来讲讲文档综合分析这个场景,英文叫 document synthesis。这个场景跟代码分析的逻辑很像,但应用的领域不一样。文章里举的例子是法律合同、研究论文、技术规格书。我们想象一个场景,你是一个企业的法务,你们公司刚刚完成了一轮并购,手头积压了几百份历史合同需要做尽职调查,你需要找出这些合同里哪些条款互相矛盾,哪些条款有潜在的法律风险,这些条款之间又是怎么互相关联的。如果上下文窗口只有二十万令牌,你可能一次只能塞进去十几份、二十份合同,模型看到的是局部图景,它没法告诉你"第三十七份合同里的违约条款,跟第一百二十份合同里的赔偿条款之间存在冲突",因为它压根没同时看到这两份合同。但如果上下文窗口扩大到一百万令牌,你理论上可以把几百份合同同时塞进去,让模型在一个统一的视野里去做交叉比对,这种"跨文档关系分析,同时保持完整上下文"的能力,是这次升级最直接的价值体现之一。

同样的逻辑放到学术研究领域也成立。假设你是一个做文献综述的研究者,你想让 AI 帮你梳理某个领域过去几年的几十篇论文,找出方法论上的演变趋势、找出互相引用和互相矛盾的结论。以前你可能得一篇一篇喂,模型记不住前面看过的论文细节,综述质量会打折扣。现在理论上可以把几十篇论文打包成一个请求,一次性分析完。这种"批量处理、整体理解"的能力,本质上是把 AI 从一个"单点助手"升级成了一个"全局分析师"。

第三个核心点,也是我觉得这次更新里最有意思、最面向未来的一个场景,叫做上下文感知的智能体,英文是 context-aware agents。这个场景跟前面两个不太一样,前面两个更像是"一次性"的大批量分析任务,而这个场景是"持续性"的、跨越很长时间和很多步骤的任务。我们先解释一下什么是 agent,智能体。简单理解,agent 就是让 AI 不只是回答一个问题,而是自主地去执行一个完整的任务,这个任务可能需要它自己去调用各种工具,比如搜索网页、读写文件、执行代码、调用 API,然后根据每一步的结果,自己决定下一步该做什么,一直循环执行,直到任务完成。

这种 agent 工作模式的一个核心痛点,就是"记忆"问题。你想象一下,一个 agent 要完成一个复杂的软件工程任务,可能需要连续执行几十次、上百次工具调用,每一次调用的输入、输出、中间的推理过程,都要累积保存在上下文里,这样 agent 才能记得自己之前做过什么、为什么这么做、下一步该往哪个方向走。如果上下文窗口太小,装不下这么多历史记录,系统就只能不断截断、丢弃早期的信息,agent 就会出现"失忆"的情况,可能会重复之前已经做过的工作,或者忘了之前定下来的一些约束条件,导致整个任务的连贯性被打断。

文章里特别强调,百万令牌的上下文窗口,可以让开发者构建出"能够在数百次工具调用和多步骤工作流中保持上下文"的 agent,而且可以"包含完整的 API 文档、工具定义和交互历史,而不丢失连贯性"。这句话信息量很大,我们拆开来看。第一,完整的 API 文档,这意味着你不需要再去精简、压缩你给 agent 看的工具说明文档,可以把所有工具的详细用法、参数说明、边界情况全部喂给它,降低 agent 误用工具的概率。第二,完整的工具定义,这个跟 API 文档类似,就是工具本身的接口定义不需要做裁剪。第三,完整的交互历史,这是最关键的一点,意味着 agent 在执行一个长任务的过程中,前面几十步、上百步做过的所有事情,都可以完整保留在记忆里,不需要做摘要压缩,这样它在做后续决策的时候,参考的信息是完整而准确的,而不是经过压缩失真之后的版本。

这里我想插一句我个人的看法。我觉得这个 agent 场景,其实是这次百万上下文升级里最具长期价值的一个方向。代码分析和文档综合,本质上是把"一次性能处理的数据量"做了放大,这个价值是线性的,数据量翻五倍,能处理的任务规模大致也翻五倍。但 agent 场景不一样,记忆容量的增加,带来的不只是"能记住更多",而是"能完成质的更复杂的任务"。因为很多复杂任务的失败,根源就是中间环节信息丢失导致的连贯性断裂,一旦这个瓶颈被打开,agent 能完成的任务复杂度,可能不是线性提升,而是会出现某种跃迁。这也是为什么我们后面会看到 iGent AI 这家公司用了"supercharged"这种很强烈的词来描述这次升级对他们产品的影响,这个我们后面客户案例部分再细说。

接下来我们讲第四个核心点,也是大家肯定最关心的一个问题,就是钱的问题,定价是怎么调整的。这次上下文窗口扩大,Anthropic 在定价策略上做了一个分层处理,核心逻辑是:超过二十万令牌的部分,因为计算开销更大,所以单价会更高。具体来说,如果你的提示词,也就是 prompt,总长度在二十万令牌以内,那价格维持原样,输入是每百万令牌三美元,输出是每百万令牌十五美元。但如果你的提示词超过了二十万令牌,价格就会跳到一个更高的档位,输入变成每百万令牌六美元,正好翻倍,输出变成每百万令牌二十二点五美元,涨了百分之五十。

这个定价设计背后的逻辑,其实跟模型推理的计算特性有关。我用一个不那么精确但比较直观的类比来解释一下,你可以把上下文窗口想象成一个房间,房间里坐的人越多,你想让某个人听清楚另一个人说的话,所需要的"协调成本"就会指数级上升,而不是线性上升。在 transformer 这种模型架构里,处理超长序列的计算复杂度,理论上是随着长度的平方在增长的,虽然 Anthropic 和其他厂商都在做各种优化来缓解这个问题,但超长上下文本身的计算成本依然显著高于短上下文。所以厂商在定价上做分层,本质上是把这部分额外的计算成本,合理地转嫁给真正用到超长上下文的那部分请求,而不是让所有用户都为这个能力买单。这种做法其实是比较厚道的,因为如果大多数请求都用不到二十万令牌以上,把价格统一拉高显然不公平,但如果完全不区分定价,那些动辄跑百万令牌请求的重度用户,又会让平台的整体计算资源被严重挤占。

讲到成本,文章里还提到两个非常重要的省钱工具,一个是 prompt caching,提示缓存,另一个是 batch processing,批处理。这两个工具结合百万上下文窗口一起用,价值会被进一步放大,我们值得详细讲讲。

先说提示缓存。它的核心逻辑是,如果你的请求里有一大段内容是反复使用、不会变化的,比如说你把整个代码仓库塞进了上下文,然后针对这个代码仓库,你想连续问十个不同的问题,那这个代码仓库本身这部分内容,系统可以把它缓存起来,后面九次请求就不需要重新计算和重新付费处理这一整段内容,只需要为新增的提问部分付费就行。这对于百万令牌级别的长上下文场景来说,价值非常明显,因为如果每次请求都要把这一百万令牌的内容重新处理一遍,无论是延迟还是费用都会很可怕。但如果你把这个大文档缓存住,之后做多轮提问,延迟会大幅降低,费用也会大幅降低,因为你不需要为同一段内容反复付费。

再说批处理。批处理适用的场景跟提示缓存不太一样,它更适合那种不要求实时返回结果、可以延后处理的大批量任务。比如说,你需要对公司过去十年积累的几千份文档做一次性的归档分析,这种任务不需要立刻给你答案,你可以把它们打包成一批,扔给批处理接口,系统会在后台慢慢处理,处理完了再把结果一次性返回给你。因为这种方式不需要占用实时响应的计算资源,所以官方给出的优惠是在原价基础上再省一半的费用,百分之五十的折扣。如果你结合百万上下文窗口来用批处理,意味着你可以把那种海量文档分析的任务,用相对很低的成本去完成,这对于预算有限但又有大量历史数据需要处理的团队来说,是一个非常实用的组合。

我们再来看一下文章里提到的两个客户案例,这两个案例其实能帮我们更具体地理解,百万上下文窗口在真实产品里到底带来了什么变化。

第一个案例是 Bolt.new。这家公司做的是一个浏览器端的网页开发平台,把 Claude 集成在里面,用户可以直接在浏览器里通过跟 AI 对话来开发网站和应用。他们的 CEO 兼联合创始人 Eric Simons 在文章里说,Claude Sonnet 四点零一直是他们代码生成工作流里的首选模型,在生产环境里持续地比其他主流模型表现更好。他特别提到,有了百万令牌的上下文窗口之后,开发者现在可以处理规模显著更大的项目,同时还能维持他们在真实编程场景里所需要的高准确度。这句话其实印证了我们前面讲的代码分析场景,Bolt.new 这种产品的用户,很多都是想快速搭建出一个完整的网站或应用,项目体量稍微大一点,以前的上下文窗口就会成为瓶颈,现在这个瓶颈被打开了,意味着 Bolt.new 平台上能支持的项目规模上限,直接被拉高了一个量级。

第二个案例是 iGent AI,这是一家总部在伦敦的公司,他们做的产品叫 Maestro,是一个能把对话转化成可执行代码的 AI 软件工程伙伴,定位上更接近一个自动化编程的智能体。他们的 CEO 兼联合创始人 Sean Ward 说了一句话,我觉得特别有意思,他说"曾经不可能的事情现在变成了现实",百万令牌上下文的 Claude Sonnet 四点零,极大地增强了 Maestro 的自主能力。他还特别提到一个词,叫"生产级规模的工程",指的是可以在真实世界的代码仓库上进行"多日的会话",这建立了智能体软件工程的一个新范式。

这句话信息量很大,我们展开讲讲。"多日会话"这个说法说明,iGent AI 的 Maestro 这种 agent,在用了百万令牌上下文窗口之后,已经不是那种几分钟、几十分钟就结束的单次任务了,而是可以持续运行几天,中间不断地积累上下文、调用工具、做出决策,而且整个过程能保持连贯,不会出现因为上下文丢失导致的"断片"。这正好印证了我们前面在讲 agent 场景时候提到的那个观点,记忆容量的提升,带来的不是简单的"处理量翻倍",而是任务复杂度和持续时间上的质变。一个能跑几分钟的 agent 和一个能跑好几天、还能保持思路连贯的 agent,这中间的差距,可能远远不是字面上"五倍上下文"能直接换算出来的。

好,核心内容我们讲完了,现在我们来聊聊,对于今天正在听这期节目的工程师朋友们,这件事到底意味着什么,实际工作中应该怎么用,又有哪些坑需要避开。

首先,最直接的应用场景,就是大型代码仓库的整体分析和重构。如果你手头的项目代码量在七万行到十万行这个区间,以前你可能习惯了把项目拆成一块一块喂给 AI,现在你完全可以考虑把整个仓库,包括源代码、测试文件、文档,一次性打包提交,让模型对整个项目建立起全局认知。这特别适合用在做代码审查、做架构梳理、找跨文件的潜在 bug 这些场景。但这里有个实际的坑要注意,就是成本。一旦你的请求超过二十万令牌,价格是翻倍甚至更高的,所以不是说"能塞进去就一定要塞进去",你得权衡一下,有些场景其实不需要全量代码,只需要相关模块加上接口定义就够了,盲目地把所有内容都塞进上下文,可能会带来不必要的费用支出,而且超长上下文本身也会带来更高的延迟,用户体验上未必是最优的。

第二个实际建议,是关于提示缓存的使用习惯。如果你的工作流程是,针对同一份大文档或者同一个代码仓库,要连续问很多个问题,那你一定要把提示缓存这个机制用起来,而不是每次都重新提交整个文档。这个习惯的改变,可能会给你的项目成本带来非常显著的差异,尤其是当你的应用是要给很多用户长期使用的产品形态的时候,缓存命中率的高低,直接决定了你的运营成本能不能控制住。

第三个实际建议,针对的是那些做海量文档批量处理的场景,比如说企业内部的知识库迁移、历史合同的批量审查、大批量的内容归档分类这些不要求实时响应的任务,优先考虑用批处理接口,而不是用普通的实时接口一条一条跑。批处理省下来的百分之五十的成本,在大批量场景下是非常可观的一笔钱,而且既然这些任务本身就不需要立刻拿到结果,用批处理在体验上也完全没有损失。

第四个实际建议,是关于构建 agent 这个方向的。如果你正在做或者打算做那种需要长时间运行、多步骤工具调用的智能体产品,百万令牌上下文窗口给你带来的,其实是一个架构设计上的新选择。以前因为上下文窗口有限,很多 agent 框架里都内置了一套复杂的"记忆管理"机制,比如说定期对历史对话做摘要压缩,或者用外部的向量数据库来存储长期记忆,需要的时候再检索回来。这套机制设计起来本身就很复杂,而且压缩和检索的过程中,信息难免会有损失。现在如果上下文窗口足够大,对于很多中等复杂度的任务,你完全可以考虑简化这套记忆管理系统,直接把完整的交互历史保留在上下文里,用更简单的架构去换取更高的信息保真度。当然,这不代表外部记忆系统就完全没用了,如果你的任务规模真的大到连一百万令牌都装不下,比如说要跨越几周甚至几个月的会话历史,那外部记忆系统依然是必要的,这里更准确的说法是,上下文窗口的扩大,把"需要用外部记忆系统"这个临界点,往后推了很多,对于大多数中等规模的任务,你现在有了一个更简单、更直接的选择。

第五个实际建议,关于实施层面的几个细节。文章里提到,这个百万令牌上下文的功能,在发布的时候是面向 Anthropic API 上 Tier 四等级以及自定义速率限制的客户开放的公开测试版,然后会在接下来几周逐步扩大覆盖范围。同时它也已经支持在 Amazon Bedrock 和谷歌云的 Vertex AI 这两个云平台上使用。考虑到现在已经是二〇二六年年中了,这些限制条件大概率已经发生了变化,如果你打算在自己的项目里用上百万上下文窗口这个功能,建议你直接去查一下当前 Anthropic 官方文档里关于这个功能的最新可用性说明,包括账户等级要求、具体支持的平台、以及当前的实际定价,不要直接照搬这篇文章里写的数字,因为差不多一年时间过去了,这些细节信息很可能已经更新过,尤其是定价这种东西,厂商经常会随着技术成熟、成本下降而做出调整。

第六个实际建议,我想提醒大家注意一个容易被忽略的风险点,叫做"上下文越大,不代表理解质量自动越高"。这一点文章里其实没有直接展开讲,但我觉得有必要在这里补充一下我自己的看法,这是我个人的判断,不是原文的内容。上下文窗口的扩大,解决的是"能不能装得下"这个容量问题,但模型在超长上下文里能不能始终保持对所有细节的精准注意力,是另外一个独立的问题,业内有个说法叫"大海捞针"测试,就是专门用来检验模型在超长文本里能不能准确找到某个关键细节,而不是被中间大量的无关信息淹没了注意力。所以即便你把整个代码仓库都塞进了一百万令牌的窗口,也不代表模型对仓库里每一行代码的关注度是均匀的,在实际使用时,如果是特别关键、特别需要精准定位的任务,建议还是要做一些验证,而不是盲目信任模型一定能在海量信息里准确抓到你真正需要的那个点。

最后,我想从整体上给这篇文章做一个总结和点评。这篇文章本质上讲的是一件听起来很简单,但实际影响很深远的事:Anthropic 把 Claude Sonnet 四点零的上下文窗口从二十万令牌扩大到了一百万令牌,五倍的提升,同时配套调整了定价策略,超过二十万令牌部分价格上涨,但结合提示缓存和批处理,依然可以把成本控制在合理范围内。这次升级直接打开了三类此前受限的应用场景,大规模代码仓库的整体分析、海量文档的交叉综合分析、以及能够持续运行很长时间还保持连贯性的智能体。从 Bolt.new 和 iGent AI 这两个客户的反馈来看,这次升级不只是一个数字上的提升,而是实实在在地改变了他们产品能力的天花板,尤其是 iGent AI 提到的"多日会话"和"生产级工程规模",这说明上下文窗口的扩大,对 agent 这个方向的影响,可能比对传统问答场景的影响更深远。

我个人认为,这篇文章发布的那个时间点,二〇二五年八月,正好是整个行业从"聊天机器人"范式向"自主智能体"范式加速转型的一个阶段,各家厂商都在比拼谁的模型能支持更长的自主运行时间、更复杂的多步骤任务,而上下文窗口的容量,正是支撑这种长程自主能力的一个基础设施层面的硬指标。所以这次百万令牌上下文的升级,与其说是一个孤立的技术参数提升,我更愿意把它理解成,是当时整个行业向 agent 化方向演进过程中,一块必须被补齐的基础拼图。今天我们站在二〇二六年年中回头看,如果你在实际工作中已经用上了支持百万级甚至更大上下文窗口的模型,理解这背后的逻辑,能帮你更清楚地知道,什么时候该一次性塞入全量信息,什么时候该做精简和检索,什么时候该用缓存和批处理去控制成本。
