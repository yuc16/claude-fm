## Script

今天要聊的这篇文章，英文原标题是"How one Anthropic seller rebuilt his team's workflows with Claude Code"，翻译成中文大意是："一位 Anthropic 销售员是如何用 Claude Code 重建他整个团队工作流的"。这篇文章发表于二〇二六年六月五日，距离我们录这期节目只有两周时间，是 Anthropic 官方博客"Anthropic 如何使用 Claude"系列的最新一篇。

这个系列是 Anthropic 自己出的，内容是分享 Anthropic 员工在内部是怎么用自家 AI 产品的。听起来有点像品牌宣传，但如果你仔细读这篇文章，会发现它背后有一些真正值得关注的东西：一个从来没有写过一行代码的销售员，用 Claude Code 构建了一个四千三百行的生产级应用，靠这个应用让自己从疲于奔命的客户主管变成了为整个销售团队造工具的产品经理。这件事的含义，已经远远超出了"公司内部案例"的范畴，所以我们今天专门来拆解它。

先来介绍主角。Jared Sires，现在是 Anthropic 的 GTM 产品经理。GTM 是 Go-To-Market 的缩写，中文可以理解为"市场进入"，这条线负责的是销售、商务拓展、客户成功这些方向。他加入 Anthropic 是在二〇二四年，在此之前，他一直在创业公司里做客户主管，就是跑客户、谈合同、维护客户关系那种角色，和技术完全没有交集。他自己说得非常直接："在加入 Anthropic 之前，我从来没有打开过终端。"更不用说写代码了。

进入 Anthropic 之后，Jared 很快就面临了一个非常具体的规模化困境。随着公司快速成长，他负责的客户账户增长到了六七百个。每天要打十到十五个客户电话，同时还要做出站销售，也就是主动联系潜在客户、开拓新账户。在这个基础上，邮件还在不停涌入。他的描述是："管理我的收件箱几乎是不可能完成的任务。还要在这之上做出站销售，你根本不知道该把精力放在哪里。"

最艰难的时候，他每天晚上要坐在桌子前直到九点、十点，才能把当天的客户邮件处理完。

而且这里面还有一个特别棘手的问题，是 Anthropic 这家公司的特殊性带来的。Anthropic 的产品更新速度极快，差不多每二十四到四十八小时就会有新的变化发布。而客户发来的问题，往往都集中在最近更新的那些细节上，比如批量 API 的服务等级协议是怎么算的，提示词缓存的折扣规则有没有变，各个模型的定价如何，SDK 某个接口的行为有什么调整。要准确地回答这些问题，就意味着 Jared 得反复在 Slack 频道、Google 文档、内部知识库、开发者文档之间来回搜索，然后第二天同样的过程再来一遍，而且这次有些事实可能又不一样了。

他把这种状态描述为："把技术文档中继给客户这件事非常难，尤其是在 Anthropic 这种产品迭代如此之快的地方。"

这种处境对很多在快速迭代公司里做技术销售、解决方案工程师、客户成功的人来说，应该特别有共鸣：信息来源分散，产品变化快，客户期望高，自己的精力是有限的。这是一个系统性问题，不是靠更努力工作能解决的。

在构建后来那个著名工具之前，Jared 先做了一个更小的实验，这个实验的逻辑链很有启发性。他用 Apps Script——这是 Google 提供的一个轻量级脚本平台，可以自动化 Google 工作区里的各种操作——结合 Claude，把内部系统里的产品使用数据拉出来，然后让 Claude 每天早上根据账户的增长速度，对他的六七百个客户账户做一个优先级排序，告诉他今天最应该把注意力放在哪些账户上。

他说："每天 Claude 都会根据客户的使用量给我一个简报，告诉我需要重点关注谁。"有了这个每日优先级清单，他不用再凭直觉猜测今天该打哪些电话，而是有了数据支撑的决策依据。

这个早期实验的意义不在于它本身，而在于它验证了一件关键的事：AI 可以帮他从海量的信息里做出筛选和判断，把他有限的注意力引导到最值得关注的地方。有了这个验证，他才有信心去构建更复杂的工具。

好，现在来说重头戏：CLAFTS。

CLAFTS，全称是 Claude Drafts，这个缩写设计得很有意思，把 Claude 的字母 L 嵌进了 Drafts 里，变成 C-L-A-F-T-S。它本质上是一个住在 Gmail 里的应用，通过调用 Claude 的 API，自动为客户邮件起草回复。

Jared 用 Claude Code 来构建这个工具。他坦承，光是看到 Claude Code 这个名字里有"code"这个词，就让他有点发憷，一开始不太敢动手。这种心理我相信很多非技术背景的人都能理解：感觉那是程序员的领地，自己闯进去会显得格格不入。但他说，"过了一段时间，我理解了它能够连接到我的电脑并回答关于电脑上文件的问题这种能力的威力所在。"

我想在这里停一下，说清楚 Claude Code 到底是什么，因为不是每个听众都用过它。Claude Code 是 Anthropic 推出的一个 AI 编程工具，它可以访问你的本地文件系统、运行终端命令、读取代码库，并直接帮你生成和修改代码。和普通的 AI 对话界面不同，Claude Code 更像一个可以直接动手操作的开发伙伴：你告诉它你想做什么，它不只是给你建议，而是直接在你的文件里做出改动、运行命令、看到结果、再做调整。这种"动手能力"是关键所在，它让 Jared 可以用它来构建一个真正运行在他电脑上的应用，而不只是得到一段可以参考但不知道怎么用的代码。

CLAFTS 最终大约有四千三百行代码，几乎全部由 Claude Code 写成。我想在这里特别强调这个数字。四千三百行，不是一个玩具级别的脚本，这是一个有完整功能的中等规模应用。做过软件开发的朋友知道，四千行代码大概是一个涵盖五到八个功能模块的系统，包含了处理逻辑、数据流、错误处理、外部 API 调用等各种细节。而这四千三百行，是一个从来没有写过代码的人，通过和 AI 持续对话完成的。这是一个很有力量的数据点，值得我们停下来认真想一想它意味着什么。

CLAFTS 的工作原理是这样的：当客户邮件进来，它会从一个共享的 Google Drive 文件夹里拉取相关的背景信息，也从其他第三方工具里获取上下文，同时通过网络搜索访问 Anthropic 最新的公开文档，然后综合这些内容，模拟 Jared 的写作风格，起草一封回复。到了一天结束，Jared 打开草稿箱，里面已经有一批等待他最终审阅的回复了。

你可以把这个过程想象成：有一个助手坐在 Jared 旁边，每次邮件进来，这个助手先查一遍最新的产品文档，再翻一遍这个客户的历史档案，然后用接近 Jared 平时语气的方式，把回复草稿放进草稿箱里。Jared 的工作从"写每一封邮件"变成了"审核并发送经过 AI 起草的邮件"，这是一个质的改变。

这里有一个设计细节非常值得关注：每当 Anthropic 发布产品更新，官方文档随之更新，而 CLAFTS 在下次生成草稿的时候，会通过网络搜索自动获取最新的文档内容。这意味着 Jared 不再需要追踪每一次产品变化并把它记在脑子里，这个知识维护的工作被外包给了工具本身。他说："Claude 能够使用网络搜索来理解我们网站上最新的文档，并在生成邮件时引用这些内容。我不需要把这些都记在脑子里。"

当然，CLAFTS 不是一上来就运行得很好的。Jared 花了大量时间来迭代系统提示词。系统提示词，就是你给 AI 的底层配置，告诉它以什么身份、什么风格、遵循什么规则来工作，类似于给一个新员工写的岗位手册，只是这本手册的读者是 AI。他说，他大概经历了"数百次迭代"来调整提示词，才让输出风格真正匹配他自己的写作习惯。

原始的 Claude 输出有两个明显的问题：一是篇幅太长；二是充斥着"可能"、"或许"、"建议您考虑"这类留余地的模糊表达，英文叫 hedging phrases。这种风格在 AI 安全的语境里可能是有意为之的，但放到客户邮件里，显得优柔寡断，不够利落。Jared 一轮一轮地调整，直到草稿开始有了他自己写作风格里的那种直接和简洁。

对所有想用 AI 辅助写作的人来说，这个数百次迭代的经历是一个非常重要的信号：不要期望 AI 一上来就输出你想要的风格。提示词的调优是一个需要持续投入的工作，而且这个工作没有捷径，你得亲自感受输出的哪里不对劲，然后有针对性地调整，再感受，再调整。这个循环的回报是很高的，因为调好之后，工具会变成真正贴合你工作方式的延伸，而不是一个你得将就着用的通用工具。

CLAFTS 最终每天为 Jared 节省了两到三个小时，每周节省十到十五小时。这是什么概念呢？假设你每天工作九个小时，其中有两到三个小时花在了机械性的邮件处理上，那这个工具相当于把你每天工作时间里将近三分之一的行政摩擦给消除了。被节省的不只是时间，还有准确性：客户收到的答案是基于最新文档的，而不是基于 Jared 上周记住的东西。他的原话是："在 CLAFTS 出现之前，我感觉我做的更多是行政工作而不是真正和客户在一起。CLAFTS 出现之后，我真正能够去做我想做的事情，也就是销售本身。"

CLAFTS 的第一个版本，除了 Jared 自己，还有一个关键的共建者：BDR John Albert。文章里提到，他常常工作到午夜还在给客户发邮件，是他和 Jared 一起把 CLAFTS 打磨出来的。CLAFTS 上线之后，他们商务开发团队的其他成员看到同事每天能拿回好几个小时，自然就来问怎么用了。这种口碑传播方式，往往比任何正式的内部推广都更有力量。

接下来是 CLAFTS 的一个功能迭代，我觉得特别有意思：CLAFTS Tones，也就是"音调"功能。

我们在日常工作中都有这种经验：给大客户写邮件、给平级同事发消息、给老朋友说事，用的语气和措辞完全不一样。给客户写邮件比较正式，措辞准确，边界清晰；给同事发消息更直接，可能还带点玩笑；给家人发消息又是另外一种随意。成熟的沟通者会在这些不同关系之间自然地切换，这种切换通常是无意识发生的。CLAFTS Tones 要做的事情，就是让 AI 识别这种差异，并相应地调整语气输出。

测试这个功能的方式非常有个人风格：Jared 用自己的私人邮件账户，给自己写了一系列语气越来越愤怒的邮件，来看 Claude 能不能识别并模仿那种情绪升级的风格。测试的结果是双重惊喜：Claude 确实捕捉到了语气的变化，开始模仿那种逐渐激烈的表达；但当邮件语气越过某个阈值之后，Claude 拒绝继续生成了，因为它不愿意为客户起草措辞激烈的邮件。

他说："Claude 开始模仿那个语气，但到了某个点，我开始遭遇拒绝，因为 Claude 不想为客户生成愤怒的邮件。就是在那个时刻，我知道 CLAFTS Tones 在正常工作了。"

这个测试结果背后有两件值得思考的事。第一是 Claude 的语气识别能力，已经精细到可以追踪情绪升级的程度；第二是 Claude 内置的价值观会在某些边界情况下自动介入，就像一个有自己判断力的助手，而不是无条件执行所有指令的机器。我个人认为，这是一个很重要的产品设计原则的体现：好的 AI 工具应该有它自己的底线，而不是一个完全被动的执行器。

接下来是 Jared 构建的第二套工具体系：每日简报和每日回顾，英文叫 Daily Brief 和 Daily Recap。

你可以把这两个技能想象成包裹在工作日两端的 AI 助手。每天早上，Daily Brief 会读取 Jared 当天的日历，对每一个要见面的人进行网络搜索，然后在第一个电话开始之前，生成专门针对这次会面的谈话要点和准备材料。这个技能通过 MCP 服务器连接到 Google Calendar 和 CRM 系统。这里的 MCP 是 Model Context Protocol 的缩写，就是让 AI 模型能够访问外部工具和数据源的标准协议，你可以把它理解为 AI 和各种应用之间的通用接口，就像 USB 接口一样，让不同的设备可以连接在一起。有了这些连接，Daily Brief 可以从多个数据源拉取每个客户的相关信息，生成真正有针对性的会前准备材料。

到了一天结束，Daily Recap 从 Google 文档和会议记录里拉取内容，起草后续跟进邮件。这个功能和 CLAFTS 类似，但专门处理会议结束后那类需要总结行动项、确认下一步的跟进邮件。

Jared 对这个组合的描述是："你把这两个放在一起，你就得到了 Claude 在管理你的日常任务，这本质上变成了一个 agent。"

agent，也就是 AI 智能体，现在是一个非常热门但也容易被过度包装的概念。很多人对它的理解停留在"一个能自主做复杂事情的神奇 AI"这个比较模糊的层面。Jared 这里给出了一个非常务实的 agent 构建示范：不是从零开始设计一个宏大的自主系统，而是从工作日的两个端点出发——开始和结束——分别用 AI 处理信息，把这两个点连接起来，就形成了一个覆盖整个工作日的 AI 辅助循环。当这个循环跑起来，在某种意义上，你的日常任务管理就有了一个 AI 层。这种"从边界点开始"的 agent 设计思路，是我认为很多人可以直接借鉴的方法论。

文章里还提到，Jared 正在进一步探索 Agent SDK，尝试把不同的工作流串联起来，让一个 Claude 任务的输出成为下一个任务的输入，形成真正意义上的自动化链条。这个方向还在实验阶段，但方向本身很清晰。

接下来是整个故事里让规模效应真正产生的部分：Claude Cowork 插件。

Jared 把他构建的所有技能和 MCP 连接器，通过 Claude Cowork 这个平台，打包成了一个团队插件，任何人都可以在几分钟内安装好。Claude Cowork 是一个支持团队共享 AI 技能和工具配置的协作环境，让一个人构建的工具可以快速在整个团队内复用，而不需要每个成员都从头摸索和配置。

结果是什么？在销售插件上线之后的几个月内，Anthropic 销售团队约百分之八十的成员开始使用它。剩下的百分之二十主要是刚入职的新人，而 Jared 认为帮助这些新人快速上手，正是他接下来要重点解决的挑战，因为这套工具本来就是专门为了让新人更快进入状态而设计的。

在有这个插件之前，每个新员工需要花数周时间自己摸索工作流程：搞清楚哪些工具在用、建立自己的信息来源、找到做客户研究的方法、理解各种内部流程。现在，新员工第一天就可以安装这个插件，立即获得二十多个已经连接好的技能，背后集成的工具包括 Salesforce、Intercom、Gong、Google Calendar、Gmail、Google Drive 和 BigQuery。这里的每一个名字你可能都不陌生：Salesforce 是客户关系管理系统，Intercom 是客户沟通平台，Gong 是销售电话录音和分析工具，BigQuery 是 Google 的大规模数据分析平台。以前这些工具都是独立运转的孤岛，销售代表得在它们之间不断切换，分别登录、分别查找、分别记录；现在，这些数据在一个统一的 AI 界面里汇聚，通过自然语言调用。

插件里有两个核心技能特别值得说清楚。

第一个叫 customer-context，也就是"客户上下文"。它可以在大约九十秒内，从所有这些数据源拉取一个客户的三百六十度全景视图。想象这样一个场景：开会前五分钟，你调用这个技能，九十秒后你就拿到了这个客户在 Salesforce 里的所有历史记录、他们在 Intercom 上提过的所有问题、他们与你们团队在 Gong 上的通话分析、以及他们的产品使用数据。这种质量的会前准备，以前可能需要一个非常有经验的销售员花半小时到一小时才能做到，而且未必做得这么系统和全面。

第二个核心技能叫 pipeline-management，也就是"销售管道管理"。它帮你识别哪些交易正在面临风险，提供预测性的指导，给出推进建议。你可以把它理解为：把一个有丰富经验的销售主管的业务直觉，打包成了一个任何销售代表都可以随时调用的判断工具。

这个插件还集成了 Cowork 的调度功能，让销售代表可以把某些技能设置为在特定时间自动运行，比如每天早上八点自动运行 Daily Brief，而不是每次都需要手动触发。

Jared 对这套系统的核心价值有一个非常精准的总结："我们的销售人员可以回归到有意义的对话，而不是去更新所有这些不同的应用程序。"

这句话触及了一个在现代工作中非常普遍但很少被正面讨论的问题：大量的工作时间被花在了"维护工具"而不是"做实际工作"上。每引入一个新工具，就有一层新的学习成本、维护成本、数据同步的摩擦。最后，工具本身变成了一种负担。Jared 的这套方案要做的事情，是把这层摩擦尽可能地去掉，让人回到真正有价值的工作：和客户建立关系，做有深度的判断，推进真正重要的事情。

随着这些工具在团队里的影响力越来越大，Jared 自己的角色也在发生变化。文章里提到，他现在以 GTM 架构师的身份，直接参与和产品工程师的设计讨论，帮助塑造 Anthropic 销售团队的新工具方向。他说："随着技术壁垒的消解，我几乎能够设计更多的产品，并且让高级工程师帮我完成最后一段实现。我能够扩展自己的能力，做更多的事情。"

好，说完了文章的核心内容，我们来聊一下这件事对你我这样的软件工程师或者 AI 从业者意味着什么，有几个实践层面的点我想认真展开。

第一点，也是最重要的：Claude Code 这类工具真正降低了"能够构建软件的人"这个问题的门槛，而这个变化的含义，比大多数人想象的要深远。Jared 的故事说明的不只是"AI 可以帮你写代码"，更深层的意思是：那些懂业务、懂流程、深刻理解用户痛点的人，现在也能把自己的理解转化成可以运行的工具。这不会让工程师变得不重要，但它确实在改变"谁能参与到工具构建中来"这个问题的答案。如果你现在是一个工程师，这是一个很好的时机去观察你身边的非技术同事开始尝试构建什么东西，那往往是离用户最近的需求，往往也是最真实的痛点。

第二点：从你最痛苦的那个具体点开始，不要试图设计一个大系统。Jared 的建议很简单：打开 Claude Code，找一个正在拖慢你的任务，然后问 Claude 怎么构建一个解决方案。他没有一开始就想"我要做一个全面的销售 AI 系统"，而是"我每天晚上要花两三个小时回邮件，我要解决这个问题"。从这样一个非常具体的问题出发，你会在很短的时间内看到真实的回报，而不是在设计文档和架构讨论里消耗精力，最后没有任何可以用的东西。把"最痛的那一点"作为起点，是一个经过验证的有效策略。

第三点：系统提示词的迭代是一项需要认真对待的技能，不是一次性的工作。Jared 花了数百次迭代才让 CLAFTS 的输出风格真正贴合他的需求。很多人用了 AI 工具之后觉得"效果不够好"就放弃了，但这恰恰是在最关键的时刻退出。正确的做法是把提示词当作一个需要持续打磨的产品来看待，每一次迭代，你都在让工具更贴合你的实际工作方式。这个过程需要耐心，但它的回报是指数级的：一旦工具调好了，它就会持续为你工作，而不需要你每次都从头解释。

第四点：agent 的构建可以从工作日的两个边界点开始。每日简报和每日回顾，两个技能本身都不复杂，但串联起来就形成了一个覆盖整个工作日的 AI 循环。如果你想在自己的工作中引入 agent 思维，可以先问自己两个问题：我每天开始工作的时候，如果有一个 AI 助手已经帮我准备好了什么，会最有帮助？一天结束的时候，如果有一个 AI 助手帮我把当天的信息整理成下一步行动，会是什么？把这两个问题的答案分别做出来，你就有了一个初步但真实可用的 agent 框架。从小处做，从边界点做，然后往中间扩展。

第五点：MCP 是让 AI 工具真正连接到现实系统的关键，值得深入研究。文章多次提到 MCP 服务器，这个协议允许 AI 模型连接到各种外部工具和数据源。如果你在考虑为自己的团队引入类似的工作流，MCP 生态现在已经相当成熟，Salesforce、Google Calendar、各类数据库都有现成的 MCP 连接器可以使用，很多情况下你不需要从头开发数据接入层，直接用已有的 MCP server 就可以了。能把 AI 和真实业务数据连接起来，才是让 AI 工具真正产生业务价值的关键一步。

还有一个心理预期的问题我想特别说一下：从"工具能用"到"工具好用"之间，有一段不短的调优过程。从 Jared 数百次迭代的经历来看，这段过程是真实存在的，而且通常不会被写进博客文章里，因为它不够光鲜。但它是必经的。如果你开始用 Claude Code 构建一个工具，初期感觉"差点意思"，那是正常的，这不是工具的问题，也不是你的问题，而是调优过程必经的阶段。坚持迭代，给自己一些时间，回报会来的。

好，让我们来做个收尾。

Jared Sires 的故事，是一条真实的从零代码到产品经理的职业路径，这条路是由 Claude Code 铺就的。他从一个每天晚上要坐到十点回邮件的客户主管，成长为为整个销售团队构建 AI 工具的产品经理和架构师。他的核心工具 CLAFTS，每周为他节省十到十五小时，现在已经被 Anthropic 约百分之八十的销售团队所使用。他自己在文章最后说："如果一年前告诉我，我会成为 Anthropic 的 GTM 产品经理，我会非常惊讶。我从来没有那种技术背景来参与这些对话。有了 Claude，我能够设计和构建不只是改善我自己日常工作流，也改善整个团队工作流的东西。"

我个人的点评是这样的：这篇文章表面上是"AI 让非程序员也能写代码"的故事，但它更深层的意义，是一种新的工作角色正在浮现。随着构建工具的技术门槛下降，那些深度理解业务和用户痛点的人，正在获得一种之前只有工程师才有的能力：把想法变成可以运行的东西。这不会让工程师变得不重要，但它确实在改变团队协作的形态。产品思维和工程能力之间的边界，正在以一种耐人寻味的方式模糊起来。

这种变化，我认为已经在发生。Jared 的故事，是其中一个早期的、清晰的、有据可查的例证。
