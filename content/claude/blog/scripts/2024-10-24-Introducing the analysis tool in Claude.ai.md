## Script

大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是 Introducing the analysis tool in Claude dot a i,翻译过来就是"在 Claude.ai 中引入分析工具"。这篇文章发表于二零二四年十月二十四日,也就是差不多两年前。

先说一句,这篇文章距离我们现在已经超过半年了,准确地说是一年八个月左右,所以我们有必要先校准一下时间感。二零二四年十月,是个什么样的节点呢。那个时候 Claude 三点五 Sonnet 刚刚发布不久,正是 Anthropic 在编程能力和数据处理能力上发力的阶段。整个行业当时的关注点,很大程度上还停留在"大模型能不能正确做数学题"这个朴素的问题上。我们都知道,大语言模型本质上是在做文本预测,它擅长的是语言和模式识别,但是一旦碰到精确的数字计算,比如说复杂的财务汇总、统计分析、多步骤的数据清洗,模型很容易出现似是而非的答案,看起来很有道理,但数字是编出来的,业内把这个问题统称为幻觉。所以在那个阶段,怎么让大模型给出可验证、可复现的精确结果,是一个非常现实的痛点。

这篇文章要解决的问题,说白了就是这一点。Anthropic 给 Claude.ai 加上了一个新功能,叫分析工具,英文是 analysis tool。这个功能的核心,是让 Claude 能够在对话过程中,直接编写并且运行 JavaScript 代码。这听起来好像没什么了不起,毕竟在这之前 Claude 早就能帮你写代码了,你把代码复制出去,自己跑一下,看看结果对不对。但问题就出在这里,以前 Claude 只能"写"代码,不能"跑"代码。它给你的答案,本质上还是基于语言模型自己脑子里的推理在猜测代码执行之后会发生什么,这就好比一个非常厉害的数学老师,口算能力超强,但是从来没有亲手在计算器上按过一遍,有时候算到第七步、第八步的时候,可能会有一些细微的偏差,毕竟人脑或者说模型的"脑子",在处理大量重复性、机械性的数值运算时,出错的概率会累积。而分析工具的出现,相当于给这位数学老师配了一台真正的计算器,而且这台计算器不是摆设,是会被实际拿来用的。

我们具体说说这个工具是怎么运作的。文章里用了一个很形象的比喻,把分析工具描述成一个内置的代码沙盒,英文叫 code sandbox。沙盒这个词在编程里很常见,它指的是一个隔离的、安全的运行环境,代码可以在里面执行,但是不会影响到外部系统,出了问题也不会波及其他东西。在这个沙盒里,Claude 可以做复杂的数学运算,可以分析数据,还可以在正式给你答案之前,先自己反复试验几种不同的思路,看看哪种做法跑出来的结果更靠谱,等确认没问题了,再把最终的答案呈现给你。这个过程其实特别像我们人类专家做事的方式。你想啊,一个真正的数据分析师拿到一份原始数据,他不会直接拍脑袋告诉你结论,他会先打开表格看一看数据长什么样子,有没有缺失值,有没有异常值,然后写一段代码或者公式去清洗数据,跑一遍看看结果是不是合理,如果不合理就回头检查哪一步出错了,反复调整,最后才敢把结论交给老板。分析工具就是把这一整套"动手验证"的流程,搬进了 Claude 和你的对话框里。

那么这件事情技术上是怎么做到的呢。我们刚才提到,这个底层依托的是 Claude 三点五 Sonnet,文章里特别强调了这个模型在编程和数据处理方面是当时最先进的水平,英文叫 state of the art。这就意味着 Anthropic 并不是单纯靠"加一个外部工具"来弥补模型的短板,而是因为模型本身的代码能力已经足够强,才有底气把执行权限直接交给它。这里有一个很关键的设计哲学,值得我们多说两句。一直以来,大语言模型有一个广为人知的弱点,就是数学计算不准。哪怕模型再聪明,你让它心算一个九位数乘以九位数的乘法,它出错的概率也不低。这不是因为模型不懂乘法的原理,而是因为它是按照语言生成的方式,一个字一个字往外蹦数字,这种生成方式天然就不适合做精确的、长链条的数值运算。那怎么解决呢,业内的共识答案就是,不要让模型自己心算,而是让模型去调用真正的计算工具,比如计算器、代码解释器,模型负责理解问题、设计解题思路、写出代码,而具体的数值运算,交给真正执行代码的计算机去做。这种思路在技术圈子里有一个说法,叫工具增强,也就是给语言模型配上外部工具,让它的能力边界从"语言理解和生成"扩展到"调用工具去解决具体问题"。分析工具正是这种思路在 Claude.ai 产品层面的一次具体落地。

我们再往深一层去看,文章里反复强调了一个词,叫精确,英文是 precise,还有一个词叫可验证、可复现,英文是 verifiable and reproducible。这两个词其实是这篇文章想要传达的核心价值主张。什么意思呢,就是说以前你问 Claude 一个涉及到数据分析的问题,它给你的答案,本质上是一种基于语言模式的"合理猜测",这个猜测可能对,也可能错,你很难验证。但是现在有了分析工具,Claude 给你的答案背后,是有一段真实跑过的代码做支撑的,这段代码做了什么,清洗了哪些数据,用了什么统计方法,跑出来的中间结果是什么,这一切都是可以摆在台面上让你检查的。这就好比,以前你问一个朋友某道数学题的答案,他凭直觉告诉你一个数字,你只能选择相信或者不相信。现在他不仅告诉你答案,还把整个演算过程的草稿纸给你看,你可以一步步核对,如果某一步觉得不对,你可以指出来,甚至可以让他重新跑一遍。这种"过程透明化"带来的信任感,是非常重要的,尤其是在企业场景里,数据分析的结果往往要拿去做决策,如果连过程都说不清楚,谁敢真的拿这个结果去做预算分配或者业务决策呢。

接下来我们说说文章里提到的具体应用场景,这部分其实挺实用的,我们可以多展开聊聊。文章列举了五类团队,分别是市场营销、销售、产品、工程和财务。我们一个一个来看。

先说市场营销团队。文章里说,市场营销人员可以上传客户在整个转化漏斗各个环节的交互数据,然后 Claude 会帮你发现可以提升转化率的机会点。我们可以想象一下具体的场景,比如说你是一个电商公司的营销负责人,你手上有用户从看到广告、点击落地页、加入购物车、到最后下单付款,每一步的数据。这中间的转化率可能是百分之三十、百分之十五、百分之五这样一路下降。传统做法是你自己拿这些数据去做透视表,或者找数据分析师帮你跑一遍,可能要等上一两天才能拿到结果。现在有了分析工具,你可以直接把数据丢给 Claude,让它帮你计算每一步的转化率,识别出流失最严重的那个环节,甚至可以进一步细分,看看是不是某个渠道来的用户特别容易在购物车环节流失。因为 Claude 是真的在跑代码做计算,而不是凭印象给你一个大概的判断,所以这个转化率的计算结果是精确的,你可以直接拿去用。

再说销售团队。文章举的例子是,销售团队可以上传全球范围内的销售数据,Claude 会帮你提供按国家划分的业绩分析。这个场景其实非常贴近很多跨国企业或者出海企业的真实需求。假设你管理着一个覆盖十几个国家的销售团队,每个月各个国家的销售报表格式可能还不太一样,有的是美元计价,有的是本币计价,数据量也很大。你想知道哪个国家这个季度的增长最快,哪个国家的客单价在下滑,传统流程可能需要财务先做汇率统一,再做数据整理,然后销售分析师再做透视和图表。现在你可以直接把原始数据扔给 Claude,让它在沙盒里写代码做汇率换算、按国家做汇总聚合,计算同比环比增长率,这一整套流程可以在一次对话里完成,而且因为是代码跑出来的,数字是准的,不会因为模型"脑补"汇率而出现误差。

然后是产品经理这个角色。文章说,产品经理可以上传用户参与度的数据,Claude 会帮助你为迭代规划和开发优先级排序提供参考。这个场景我觉得特别贴近做产品的同学的日常工作。你手上可能有几十个功能模块的使用数据,日活、留存、使用时长、报错率等等,你需要决定下一个迭代周期先做哪几个功能的优化。以前你可能要自己拉数据、做加权打分,现在你可以让 Claude 直接基于真实数据跑出一个优先级排序的建议,比如说哪个功能的使用频次高但是报错率也高,这种就应该优先修复,哪个功能使用频次低,大概率可以往后排。这个排序背后是有真实计算支撑的,不是 Claude 凭感觉给你列的清单。

接下来是工程团队。文章里提到,工程师可以上传来自不同服务器的性能日志,Claude 会帮你识别出可以更好利用资源的地方。这个对做基础设施和运维的同学来说也很有意思,服务器日志这种东西,数据量往往很大,而且格式比较琐碎,CPU 利用率、内存占用、请求延迟,各种指标交织在一起。人工去看日志找规律,效率很低,而且容易漏掉一些隐藏的模式。Claude 在分析工具的加持下,可以写代码去解析这些日志,做统计聚合,找出哪些时间段资源利用率特别低,哪些服务器配置可能存在浪费,这种基于真实计算的发现,比单纯靠模型"读"一遍日志然后给印象式的总结,要靠谱得多。

最后是财务团队。文章举的例子是,财务团队可以上传每月的财务数据,Claude 会帮你创建一个财务仪表盘,呈现关键趋势,辅助决策。财务这个场景对精确度的要求是最高的,毕竟涉及到真金白银的数字,如果分析结果有偏差,可能直接影响到预算决策。这也恰恰说明了为什么分析工具这个功能对企业用户来说价值很大,因为如果 Claude 还是停留在"凭语言模型推理"给财务数据下结论的阶段,没有哪个财务团队敢真的把这个结果拿去做汇报,但是一旦背后有真实代码执行做支撑,可验证、可复现,财务团队才有可能真正把这个工具用起来。

讲到这里,我想跟大家多聊几句关于这个功能本身的技术细节和限制。文章里提到,这个分析工具在当时是以功能预览的形式向所有 Claude.ai 用户开放的,英文叫 feature preview。这说明在二零二四年十月那个时间点,Anthropic 对这个功能还处于一个相对谨慎的推进节奏,先让用户试用,收集反馈,再决定怎么进一步打磨。开启方式也很简单,登录 Claude.ai,点击左下角的用户名,就能在功能预览的管理界面里把它打开。这种渐进式发布的方式,其实是当时整个行业在推新功能时比较常见的做法,毕竟代码执行这种能力,一旦放出去给所有用户用,涉及到安全性、稳定性、资源消耗等等一系列工程问题,需要在真实流量里慢慢验证。

另外要注意的是,文章原文用的编程语言是 JavaScript,而不是数据分析圈子更常用的 Python。这个选择我觉得也挺值得琢磨。JavaScript 本身是网页前端最主流的语言,在 Claude.ai 这种网页产品里,直接在浏览器端的沙盒环境跑 JavaScript,技术实现路径相对更顺畅,因为不需要额外搭建一套服务器端的 Python 执行环境,安全隔离的成本也更低。当然,JavaScript 处理数据分析这件事,生态上不如 Python 那么丰富,Python 有 NumPy、Pandas 这些非常成熟的数据科学库,JavaScript 相对就单薄一些。所以我个人猜测,这也是为什么这个功能后来会被新的代码执行能力所取代,因为单纯的 JavaScript 沙盒,长期来看在数据分析的深度和广度上是有天花板的。

说到这里,我要提一下这篇文章开头其实有一段更新说明,这是后来加上去的。更新说明写道,分析工具正在被更强大的代码执行能力所取代,这个更新的时间是二零二五年十一月五日。也就是说,从二零二四年十月发布,到二零二五年十一月被替代,这个分析工具的生命周期大概是一年多一点。更新说明里还提到,新的代码执行能力可以跑所有分析工具能跑的分析,而且还能额外生成可下载的文件,比如表格、CSV、报告,还能创建可视化图表,处理更复杂的多步骤工作流。这个信息量其实挺大的,它告诉我们几件事。第一,分析工具这个产品方向本身是被验证是对的,所以 Anthropic 没有放弃这条路,而是在它的基础上继续往前走,做得更强。第二,当年分析工具的一个明显短板就是只能做分析、给结论,不能直接产出一份你可以下载带走的文件,这意味着如果你想要一份正式的报告或者表格,你还是得自己手动整理 Claude 给你的文字结果。新的代码执行能力把这个短板补上了,直接能生成可下载的文件,这对企业用户来说体验会顺畅很多,毕竟分析完了之后,大家最终还是要一份能发邮件、能存档的成果物,而不是停留在聊天框里的一段文字。第三,可视化能力的加入,也呼应了我们刚才说的财务仪表盘、转化率漏斗这些场景,毕竟很多分析结果,用图表呈现远比用文字描述更直观。

那么我们今天回头看这篇文章,对我们普通听众,尤其是做技术或者做业务分析的朋友,有什么实际的参考价值呢。我想从几个角度来给大家梳理一下。

第一个角度,是理解大模型能力演进的一条主线。我们经常听到"agent"这个词,中文有时候翻译成智能体,它指的是大模型不再只是被动回答问题,而是能够主动采取行动,比如调用工具、执行代码、操作文件,去完成一个任务。分析工具其实就是 Claude 在这条路上迈出的一个具体步子,从"只能说"到"能做",这个转变看起来简单,但背后的意义很大。如果你在做企业内部的 AI 应用选型,或者在评估要不要把某个大模型集成到你的业务流程里,一个很关键的判断标准就是,这个模型能不能真的去执行代码、验证自己的答案,而不是只靠语言生成给你一个看起来很对但实际上没法验证的结果。这一点放到今天来看依然是成立的,甚至今天的代码执行能力比当年的分析工具要强得多,但是判断逻辑是一脈相承的。

第二个角度,是实际工作流程上的启发。如果你今天是产品经理、营销人员、销售管理者或者财务人员,你可以想一想,你手头有没有那种"数据量不算特别大,但是需要精确计算和归纳"的任务,比如月度业绩汇总、转化率分析、用户分群,这类任务过去你可能需要找数据分析师排期,或者自己手忙脚乱地在 Excel 里折腾。现在借助 Claude 现有的代码执行能力,你完全可以自己上传数据,直接让 Claude 帮你跑分析、生成图表甚至导出文件。这相当于把数据分析师这个角色的一部分工作,前置到了业务人员自己手里,大大缩短了从数据到决策的链路。当然这里要提一个坑,就是数据的隐私和合规问题,如果你的数据涉及到客户隐私或者公司机密,上传到任何云端 AI 工具之前,一定要先确认清楚公司的数据合规政策,不要图方便就把敏感数据随手丢上去。

第三个角度,是关于结果验证的习惯。哪怕今天的代码执行能力已经比当年强大很多,我个人认为,作为使用者,我们还是应该保留一个习惯,就是去看一看 Claude 实际跑的代码逻辑是什么,而不是闭着眼睛全盘接受最终结论。这篇文章反复强调的"精确、可验证、可复现",它的前提是你愿意花一点时间去核实这个验证的过程,如果你压根不看代码,那这个工具带来的可信度优势,你其实并没有真正享受到。所以我建议大家在用这类分析工具的时候,养成一个习惯,简单扫一眼它写的代码逻辑是否合理,数据清洗的逻辑是不是符合你的业务实际情况,这样才能真正把"工具增强带来的精确性"转化为你自己的判断力。
