## Script

大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是 Bringing memory to teams,我把它翻译成"把记忆能力带给团队"。这篇文章是 Anthropic 官方博客在二〇二五年九月十一日发布的,后面还有一条更新,是二〇二五年十月二十三日补充的,说这个记忆功能扩展到了 Pro 和 Max 这两个面向个人和重度使用者的付费计划。

先说说时间线的事儿。今天是二〇二六年六月二十一日,这篇文章发出来已经超过九个月了,中间还经历了一次功能扩展。所以咱们听这篇文章的时候,要把它当成一个历史节点来看,而不是当成最新资讯。在那个时间点,也就是二〇二五年九月,整个大模型行业正处在一个什么阶段呢?我们可以回想一下,当时 ChatGPT 已经有自己的记忆功能一段时间了,Claude 这边相对克制,迟迟没有大规模上线类似能力,主要原因就是 Anthropic 一直在强调安全和隐私的优先级。所以这篇文章的发布,某种程度上标志着 Claude 终于决定把记忆这个能力,以一种比较谨慎、比较克制的方式推给企业和团队用户。这也是为什么我觉得这篇文章值得拿出来讲一讲,因为它不只是一个产品功能公告,背后还藏着 Anthropic 对记忆这件事的产品哲学,这个哲学放到今天,二〇二六年年中,依然是有参考价值的。

我们先讲讲这篇文章要解决的问题是什么。如果你平时用 Claude 或者类似的对话式 AI 工具做工作,你一定有过这种体验:你今天跟它聊了一个项目的细节,聊得挺深入,第二天打开一个新对话,结果它什么都不记得,你又得从头把背景信息、项目背景、客户偏好、技术栈这些东西重新讲一遍。这种"每次对话都要重新自我介绍"的摩擦感,对个人用户来说可能还能忍,但对一个团队来说,这种摩擦是会被成倍放大的。想象一下,一个销售团队在跟进多个客户,每个客户的偏好、谈判进展、痛点都不一样,如果每次找 Claude 帮忙写邮件或者整理思路,都要把这些背景从头讲一遍,那效率提升根本没法兑现。再想象一个产品团队,在做一个跨多个迭代周期的项目,每个迭代都有新的规格变化、新的决策依据,如果 AI 助手没有记忆,那它能提供的帮助永远停留在单次对话的水平,没办法真正融入团队的工作节奏里。

所以文章一开头就点明了这个核心痛点,原文的说法是,不管你是在迭代一份战略提案,还是在调试一个问题,还是在管理多个项目,Claude 现在可以接着你上次的进度继续往下走,不用你重新解释上下文。这句话听起来很简单,但它其实点出了记忆功能最核心的价值主张,就是减少"重复解释"这个隐性成本。这个隐性成本,我们平时可能没有专门去计算,但它确实存在,而且累积起来是相当可观的。

讲完背景,我们来拆解这篇文章里最重要的几个核心点。我觉得可以分成四个层面来讲,第一个是记忆到底记什么、不记什么,第二个是项目级别的记忆隔离,第三个是用户对记忆的控制权,第四个是隐身聊天模式以及背后的安全考量。

先说第一个层面,记忆到底记什么。这一点其实是这篇文章里我觉得最值得细品的地方。Anthropic 没有把记忆功能设计成那种"记住你说过的每一句话"的全量记录器,而是专门强调,Claude 聚焦学习的是你的专业背景和工作模式,目的是让生产力最大化。具体来说,它记的是你团队的工作流程、客户需求、项目细节、优先级这些东西。文章里举了三个具体的场景,销售团队可以在不同的交易过程中保留客户背景信息,产品团队可以在不同的迭代周期之间维持规格的连续性,高管可以跟踪各种事项的进展,而不需要每次都重新搭建上下文。

这里我想用一个类比来帮大家理解这个设计思路。你可以把它想象成一个新入职的助理。一个不靠谱的助理,可能会把你说的每一句话都原封不动地记在小本子上,什么都记,但是记多了反而抓不住重点,关键信息被淹没在一堆琐碎细节里。而一个靠谱的助理,会学着抓住你工作中真正重要的模式,比如你跟哪个客户合作的时候特别看重响应速度,你做产品决策的时候习惯先看数据再看用户反馈,这些工作模式和偏好才是真正有价值的记忆内容,而不是流水账。Anthropic 在这篇文章里传达的意思,就是希望 Claude 扮演的是后一种助理的角色。

这个设计选择背后,其实也隐含了一个权衡。如果记忆记得太宽泛、太琐碎,一方面会增加隐私和安全风险,因为记的东西越多,潜在泄露或者被滥用的面就越大;另一方面也会降低记忆的实际效用,因为信息噪音太多,真正有用的工作上下文反而被稀释了。所以 Anthropic 选择把记忆功能的范围,有意识地收窄到工作场景、专业背景这个维度,这其实是一种产品上的克制,我觉得这个克制是值得肯定的,这是我个人的看法。

接下来讲第二个核心点,项目级别的记忆隔离。这一点我觉得是这篇文章里设计得最巧妙的地方。如果你用 Claude 的项目功能,也就是 Projects,那么 Claude 会为每一个项目单独创建一份独立的记忆。这意味着什么呢?意味着你做产品发布规划的那些对话内容,和你处理某个具体客户的对话内容,这两份记忆是完全分开的,不会互相串到一起。同样,涉及保密内容的讨论,和日常的一般性运营对话,也会被分割在不同的记忆空间里。

为什么这个设计很重要?我们可以想一个具体的场景。假设你是一个咨询公司的项目经理,你同时在服务三个客户,客户 A、客户 B、客户 C,每个客户都有自己的商业机密、报价策略、竞争对手分析。如果 Claude 的记忆是一个大水池,什么都往里面装,那风险就来了,万一某次你跟 Claude 聊客户 A 的项目的时候,Claude 不小心把客户 B 的某个细节带出来了,那这就是一个非常严重的信息泄露事故,在咨询行业、法律行业、金融行业,这种事故几乎是不可接受的。

通过把记忆按项目切分,Anthropic 实际上是在产品层面built进了一道安全护栏。文章原文里用的词是 safety guardrail,意思就是这种项目边界本身就是一种安全机制,而不仅仅是为了让信息更整洁。这跟很多企业软件设计里的"最小权限原则"或者"数据隔离原则"是一脉相承的思路。如果你自己在做企业级的 AI 应用,这个思路其实非常值得借鉴,就是不要把所有上下文一股脑塞进一个全局的记忆库里,而是要按照业务边界、按照客户边界、按照保密级别去做隔离,这样才能让 AI 工具在企业环境里真正落地,而不是变成一个潜在的合规雷区。

第三个核心点,是用户对记忆内容的控制权,这一点也呼应了文章一开头强调的"记忆是完全可选的"这个原则。Anthropic 设计了一个叫记忆摘要的东西,英文是 memory summary,它把 Claude 记住的所有内容汇总在一个地方,你可以在设置里直接看到 Claude 到底记了什么,并且可以随时编辑、修改。更进一步,你还可以直接通过跟 Claude 对话的方式,告诉它你想让它重点关注什么,或者告诉它请忽略某些内容,Claude 会根据你的指示动态调整它会引用的记忆。

这个设计其实回应了大家对记忆功能最常见的两个顾虑。第一个顾虑是黑盒,如果一个 AI 工具默默记住了一堆东西,但你完全不知道它记了什么,这种感觉是很让人不安的,就像你不知道一个同事在背后做了什么笔记一样。Anthropic 用一个透明的、可视化的摘要界面来解决这个问题,你随时可以打开看,这就是知情权的落地。第二个顾虑是失控,如果记忆内容是错的,或者记忆已经过时了,但你没有办法纠正它,那这个记忆功能反而会变成负担,甚至会持续地给出基于错误前提的建议。Anthropic 给出的解法是,你可以直接编辑摘要,也可以用自然语言告诉 Claude 该记什么不该记什么,这其实是把记忆管理这件事,变成了一种对话式的、低门槛的交互,而不是要你去后台数据库里手动删字段。

我觉得这个设计思路对所有想做长期记忆功能的开发者都有启发,就是控制权这个事情,不能只停留在"理论上用户可以关闭这个功能"这种粗粒度的开关上,而是要做到细粒度的可视、可编辑、可纠正。这是建立用户信任的关键一步,如果用户对记忆系统没有信任,那这个功能再强大,大家也不敢真正放心地用在工作场景里。

第四个核心点,是隐身聊天,英文是 Incognito chat。这个功能的设计动机其实很直接,就是有些时候你需要 Claude 帮你处理一些敏感的事情,但你不希望这次对话被记住,也不希望它出现在你的对话历史里。文章里举的例子包括敏感的头脑风暴、保密的战略讨论,或者你单纯就是想要一个全新的对话,不带任何之前对话的上下文干扰。

这个功能的好处在哪儿呢?它相当于给了用户一个"沙盒模式",在这个模式里,你正常的记忆和对话历史完全不受影响,这次对话产生的内容,既不会污染你的长期记忆,也不会留下历史记录。如果你是在 Team 或 Enterprise 计划下使用记忆功能,那么标准的数据留存规则在隐身模式下依然适用,也就是说隐身不等于完全没有任何记录留存,而是不进入记忆系统、不进入你常规可见的对话历史。

我觉得隐身聊天这个设计,其实补上了记忆功能上线之后必然会出现的一个缺口。一旦你有了长期记忆,用户心里难免会犯嘀咕,是不是我每一次对话都被悄悄记下来了。如果没有一个明确的逃生通道,用户在面对敏感话题的时候,可能会变得谨小慎微,甚至干脆不敢用这个工具去讨论真正棘手的问题,这样反而削弱了工具的实用性。隐身模式提供的是一种心理上的安全感,让用户知道,你随时可以选择不留痕,这种选择权本身就是产品体验的一部分。

讲到这里,我们还得提一下 Anthropic 在这篇文章里专门强调的安全测试这件事。原文提到,在把记忆功能推广到 Pro 和 Max 计划之前,他们做了大量的安全测试,专门针对一些跟用户福祉相关的敏感话题和边界情况。他们测试的具体问题包括三类,第一类是记忆会不会强化对话里已经出现的有害模式,第二类是记忆会不会导致 Claude 变得过度迎合用户,第三类是记忆会不会被用来尝试绕过 Claude 原本设置的安全护栏。通过这些测试,Anthropic 发现了一些需要打磨的地方,并针对记忆的工作方式做了有针对性的调整。

这一段我觉得特别值得拿出来单独说一说,因为它体现了 Anthropic 一贯的产品节奏,就是宁可慢一点上线,也要把潜在风险想清楚。我们可以拿一个具体的风险场景来理解,比如说,如果一个用户在跟 Claude 的对话中,反复表达一些负面的、消极的想法,而 Claude 因为有记忆,会把这种负面情绪当成这个用户的稳定特征记下来,那么在后续对话里,Claude 可能会不自觉地延续或者强化这种负面框架,这就是记忆可能带来的"强化有害模式"的风险。再比如,如果 Claude 因为记住了用户喜欢被顺着说话,慢慢变得越来越迎合用户、不敢提出不同意见,这种"过度迎合"也是记忆功能可能带来的副作用。还有一种风险是,有些用户可能会想办法利用记忆机制,比如先在多轮对话里逐步铺垫一些边界内容,试图让 Claude 在记忆的加持下,放松原本的安全限制,这就是绕过护栏的风险。

正是因为想清楚了这几类风险,Anthropic 才选择了一个分阶段的发布策略,先在 Team 和 Enterprise 这种企业场景里小范围验证,确认记忆功能在真实工作场景里跑得稳,然后才在一个多月后,扩展到面向个人重度用户的 Pro 和 Max 计划。这种节奏其实也是这篇文章一个隐藏的信息点,就是 Anthropic 把工作场景当成了记忆功能的第一个试验田,因为工作场景相对来说话题更聚焦、风险面更可控,等验证得差不多了,再逐步放开给更广泛的用户群体。

好,讲完了核心内容,我们来聊聊这些东西对大家实际工作意味着什么,尤其是如果你也在做 AI 相关的产品或者只是日常重度使用 Claude 这类工具的话,有哪些可以直接拿来用的启发。

第一个启发,是关于怎么用好这个记忆功能本身。如果你所在的团队已经在用 Claude 的 Team 或者 Enterprise 计划,那么我建议你按照项目去组织你的工作对话,而不是把所有事情都堆在一个对话窗口或者一个项目里。因为项目级别的记忆隔离,本质上是在帮你做信息治理,如果你自己不按项目分类,这个隔离机制的价值就发挥不出来。具体操作上,你可以为每个客户、每个产品线、每个保密程度不同的事项,单独建一个 Claude 的 Project,这样 Claude 给你的建议和记住的上下文,才会真正贴合那个具体场景,而不是被无关信息干扰。

第二个启发,是关于记忆摘要这个面板,我建议大家养成一个习惯,定期去检查一下 Claude 到底记住了什么。这个有点像我们平时会定期清理浏览器的缓存和历史记录一样。如果你发现某些记忆已经过时了,比如你换了岗位,或者某个项目已经结束了,那及时去编辑或者清理这部分记忆,可以让 Claude 给你的建议始终基于最新的、准确的工作背景,而不是被一些陈旧信息带偏。

第三个启发,是隐身聊天的使用时机。如果你要讨论的内容涉及到非常敏感的人事决策、还没有最终敲定的战略方向、或者你只是想要一个干净的、不受过往对话框架影响的头脑风暴空间,那这种时候就应该主动切换到隐身模式。这其实是一个很实用的工作习惯,就是把"需要被记住的工作上下文"和"不需要留痕的探索性思考"主动分开来处理,这样既能享受记忆带来的效率提升,又不会担心敏感内容被沉淀下来。

第四个启发,是给开发者和产品经理的。如果你自己也在设计带有长期记忆能力的 AI 产品或者 Agent 系统,这篇文章其实提供了一套现成的设计模板,可以直接参考。首先是记忆范围要收窄而不是贪多,聚焦在真正对用户有价值的工作模式和偏好上,而不是无差别地记录一切。其次是要有清晰的隔离边界,按照业务场景、客户、保密级别去切分记忆空间,把隔离机制当成安全护栏来设计,而不是事后补丁。第三是控制权要做到细粒度,让用户能看见、能编辑、能用自然语言纠正系统记的内容,而不是只给一个全局的开关。第四是要给用户留一个不留痕的逃生通道,类似隐身模式这样的设计,可以大大降低用户对记忆功能的心理负担。最后,在大规模铺开之前,一定要先做扎实的安全测试,尤其是要关注记忆可能放大的几类典型风险,比如强化负面模式、导致系统过度迎合、被用来绕过安全限制,这几个测试维度其实可以直接套用到任何带记忆功能的对话系统设计里。

第五个启发,我想多说几句,是关于节奏的问题。这篇文章发布在二〇二五年九月,扩展到 Pro 和 Max 计划是在十月,中间只隔了一个多月。这说明 Anthropic 内部对这个功能的信心是逐步建立起来的,先在风险相对可控的企业场景里验证,验证完了之后才敢推给更广泛的个人用户。如果你自己在做产品决策,面对一个有一定风险、但是潜在价值也很大的新功能,这种"先小范围验证、再逐步扩大"的节奏,其实是一个值得借鉴的稳妥做法,既不会因为过度保守而错失先发优势,也不会因为急于求成而把风险留给最广泛的用户群体去承担。

好,我们来做一个简单的总结。这篇文章讲的核心事情,是 Claude 在二〇二五年九月给 Team 和 Enterprise 用户带来了记忆功能,后来扩展到了 Pro 和 Max 计划。这个记忆功能聚焦在工作场景,记的是团队的流程、客户信息、项目细节,而不是泛泛地记一切;它通过项目级别的隔离,把不同业务场景的记忆分开,起到了安全护栏的作用;它给用户提供了透明可编辑的记忆摘要,让用户始终掌握主动权;它还提供了隐身聊天模式,给敏感对话留了一个不留痕的出口;在大规模铺开之前,Anthropic 也专门做了针对性的安全测试,去识别记忆可能带来的强化负面模式、过度迎合、绕过护栏这几类风险。

我个人的看法是,这篇文章虽然篇幅不长,但它其实展示了一种比较成熟的产品设计哲学,就是在推出一个强大但有风险的新能力的时候,克制比炫技更重要,边界感比功能堆砌更重要。放到今天二〇二六年年中这个时间点来看,记忆能力已经几乎成了主流对话式 AI 产品的标配,但这篇文章里强调的这几条原则,我觉得依然是检验一个记忆功能设计得好不好的有效标准。好了,这就是今天这期节目的全部内容,我们下期再见。
