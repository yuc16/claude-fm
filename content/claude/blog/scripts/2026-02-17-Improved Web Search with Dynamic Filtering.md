大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,英文标题是 Improved Web Search with Dynamic Filtering,我把它翻译过来就是用动态过滤提升网页搜索的效果。这篇文章发表于二〇二六年二月十七日,差不多是四个月之前。放在我们今天,也就是二〇二六年六月二十日这个时间点来看,这篇文章其实还挺新鲜的,因为它正好对应着 Claude Opus 四点六和 Sonnet 四点六这一代模型的发布节点,讨论的也是当下大家都在用的 agent 工具体系里最核心的一环,网页搜索。所以这期节目不算是回顾历史,更像是补一下最近这一波工具升级里大家可能没太注意的一个细节。

那这篇文章为什么值得花二十几分钟听一听呢。我自己的感觉是,这其实是 Anthropic 在解决一个所有做 agent 应用的人都会遇到的痛点,就是网页搜索这件事看起来简单,做起来特别烧钱也特别容易出错。如果你自己写过带网页搜索功能的 agent,你大概知道那种感受,模型搜一次,返回一堆链接,然后你得把这些网页内容一股脑塞进上下文窗口,模型再在这堆文字里大海捞针,找出真正有用的那几句话。这个过程效率很低,而且越往后聊上下文越臃肿,模型越容易被无关信息带偏。这篇文章讲的就是 Anthropic 怎么解决这个问题,办法说出来其实挺巧妙的,我们一点一点拆开讲。

先说说背景。在动态过滤这个功能出现之前,一个典型的网页搜索类 agent 是怎么工作的。模型先发起一次搜索请求,拿到一批搜索结果,这些结果通常是一些网页标题加摘要的列表。如果只看摘要不够,模型还要进一步去抓取某些网页的完整内容,这一步就是网页抓取,英文叫 web fetch。问题就出在这里,一旦把整页的 HTML 内容都拉进对话的上下文窗口,这里面会包含大量跟问题无关的东西,比如网站的导航栏文字,广告内容,版权声明,各种和正文无关的页面元素。这些东西全部要被模型当作上下文的一部分去阅读和理解,等于说模型要在一堆垃圾信息里去找那一两句真正有用的句子。

这样做有两个明显的坏处。第一个是效率问题,你想啊,网页里大部分内容跟用户的问题根本没关系,但模型还是得把这些内容当作输入 token 来处理,这意味着你在为大量无用信息付费,也意味着推理速度会被拖慢。第二个坏处更隐蔽,但其实更严重,就是质量问题。文章里原话提到,搜索拉进来的上下文经常是不相关的,这会拉低回答的质量。这个道理也好理解,模型的注意力是有限的,如果上下文里塞满了噪音,真正关键的信息反而容易被稀释掉,模型可能会被无关内容干扰,给出不准确甚至跑偏的答案。所以网页搜索这个场景,表面上是一个工具调用的问题,本质上是一个上下文管理的问题。

那 Anthropic 是怎么解决的呢,这就是这篇文章的核心,也就是动态过滤这个机制。简单说,这次随着 Claude Opus 四点六和 Sonnet 四点六一起发布的新版网页搜索和网页抓取工具,有了一个新本事,就是 Claude 现在可以在执行网页搜索的过程中,原生地编写并执行代码,去对搜索结果做一次后处理,把无关信息过滤掉,然后才把筛选后的结果加载进上下文窗口。

我们用一个类比来理解这件事。假设你让一个助理去图书馆帮你查资料,旧的方式是助理把整本整本的书都搬回来放你桌上,你自己一页一页去翻,去找你要的那段话。这种做法当然能找到答案,但你桌子上堆满了几十本书,你翻起来很费劲,而且很容易在翻书的过程中走神,被无关的章节带跑偏。动态过滤这个新方式,相当于助理现在变聪明了,他不再把整本书搬回来,而是先在图书馆里自己用一些工具,比如索引卡或者关键词检索,先把每本书里跟你的问题相关的那几页挑出来,复印好,只把这几页放到你桌上。你拿到的信息密度立刻就高了很多,你也不会被无关章节干扰,而且因为搬回来的纸少了,你的桌子也不会那么乱。这个挑选和复印的动作,对应到技术实现上,就是模型自己写代码去解析和过滤搜索结果。

这里有个细节值得多说一句,就是这个过滤动作不是靠模型在自然语言层面去判断哪句话有用哪句话没用,而是真的让模型去写代码,去执行,代码去做解析,过滤,交叉比对这些结构化的操作。这是一种更工程化的处理方式,代码在处理结构化或者半结构化的网页内容时,效率比纯靠语言模型的注意力机制去筛选要高得多,也更精确。这其实也呼应了 Anthropic 之前在别的 agent 场景里验证过的一个思路,就是让模型把一些原本要靠语言推理来完成的任务,转交给代码执行去完成,这样可以把中间过程中产生的大量噪音数据挡在上下文窗口之外,只把最终需要的结果带回到对话里。这篇文章里也提到,他们之前已经在别的 agent 工作流里验证过这个技巧是有效的,所以这次是把同样的技巧迁移应用到网页搜索和网页抓取这两个具体工具上。

讲到这里你可能会问,这个机制听起来不难理解,但它到底有多大的实际效果呢,这就要看接下来这部分的实验数据了。Anthropic 在两个基准测试上对比了开启和关闭动态过滤这两种情况下 Sonnet 四点六和 Opus 四点六的表现,而且测试的时候只启用了网页搜索这一个工具,没有叠加别的工具干扰结果。这两个基准测试分别叫 BrowseComp 和 DeepsearchQA,我们分别说说它们是测什么的,以及具体的数字。

先说 BrowseComp。这个基准测试的设计思路是,给模型一个问题,这个问题的答案就藏在网络上的某个地方,但是它故意设计得很难被搜到,你得在很多个网站之间跳转,层层缩小范围,最后才能锁定那一个具体的事实性答案。这种任务很考验一个 agent 有没有耐心去做多步骤的探索式搜索,而不是搜一次就草草作答。结果显示,开启动态过滤之后,Sonnet 四点六的准确率从百分之三十三点三提升到了百分之四十六点六,这个提升幅度相当可观,差不多是原来准确率的一点四倍。Opus 四点六这边提升更猛,从百分之四十五点三一路提升到百分之六十一点六。这意味着原本只有不到一半的概率能找对答案,现在变成了超过六成,这是一个质的飞跃。我个人认为,这个提升幅度其实超出了我一开始的预期,毕竟动态过滤听起来只是工程优化,但它带来的准确率提升,跟很多模型本身升级带来的提升相比,完全不逊色。

再说 DeepsearchQA。这个基准测试跟 BrowseComp 的思路不一样,它不是去找一个唯一的答案,而是给模型一个研究类的问题,这个问题正确答案有很多个,而且所有这些答案都必须靠网页搜索才能找全。它考验的是 agent 有没有能力系统性地规划好搜索步骤,一步一步地执行,而且不能漏掉任何一个该找到的答案。这种任务的评价方式用的是 F1 分数,这是一个同时兼顾精确率和召回率的综合指标,既要看你给出的答案对不对,也要看你是不是把所有该找到的答案都找全了。结果是,Sonnet 四点六的 F1 分数从百分之五十二点六提升到了百分之五十九点四,Opus 四点六从百分之六十九点八提升到了百分之七十七点三。两个模型在这个更考验系统性和完整性的任务上,都有大约七到八个百分点的提升,这个提升在搜索类任务里已经算是比较显著的了,毕竟越往后提升一个百分点往往越难。

那么平均下来,这两个基准测试加在一起,动态过滤给 Claude 带来了大约百分之十一的整体性能提升,同时还少用了百分之二十四的输入 token。这两个数字放在一起看其实挺有意思,一般我们做工程优化的时候,经常会遇到一个两难,就是要准确率往往要多花资源,要省资源往往要牺牲一些准确率。但动态过滤这个机制比较少见地做到了两边都讨好,既提升了准确率,又降低了输入 token 的消耗。原因也不难理解,因为它从根本上减少了进入上下文的垂圾信息量,垂圾信息少了,模型自然不容易被带偏,同时因为不需要处理那么多无关文本,输入 token 自然也省下来了。

不过这里我要提一个文章里特别强调的细节,大家不要被这个百分之十一和百分之二十四的漂亮数字带偏了节奏。文章里明确说了,token 成本的变化其实跟具体场景有关系,因为模型需要写多少代码来做过滤,这个量是不固定的,取决于搜索结果本身的复杂程度。具体来看,如果你用价格加权的方式去算总成本,会发现 Sonnet 四点六在两个基准测试上的成本都是下降的,这个和我们前面说的省 token 是一致的。但是 Opus 四点六这边反而出现了成本上升的情况。这个反差挺值得琢磨的,我自己理解,可能是因为 Opus 作为更强的模型,在面对复杂搜索任务的时候,愿意写更多更精细的过滤代码去争取更高的准确率,这部分多出来的代码执行本身也是要算 token 成本的,所以即便最终输入到对话历史里的 token 减少了,但整体算上代码执行这部分的开销,总价反而可能更高。所以文章给出的建议是,大家不要直接套用这两个基准测试上的数字去预估自己的成本,而是应该拿自己业务里有代表性的一批真实搜索查询,去实际跑一遍评测,这样才能摸清楚在自己的具体场景下,到底是省钱还是花钱,省多少或者花多少。这个建议我觉得非常实在,也是我们做工程落地的时候应该有的态度,基准测试的数字是参考,但真正决策还是要靠自己业务场景下的实测。

文章里还引用了一个真实的客户案例,来自 Poe by Quora 这个平台。可能有些听众不太熟悉这个产品,简单介绍一下,Poe 是 Quora 旗下的一个多模型 AI 平台,算是行业里规模比较大的同类产品之一,它让用户可以在一个界面里访问超过两百个不同的模型。Poe 团队的产品和研究负责人,在文章里分享了他们内部测评的结果,他说在他们自己的内部评测体系里,搭载了动态过滤的 Opus 四点六,跟其他前沿模型相比,拿到了最高的准确率。他还有一句话我觉得说得特别形象,他说这个模型表现得就像一个真正的研究员,会写 Python 代码去解析,过滤,交叉核对搜索结果,而不是直接在上下文里对着原始的 HTML 内容去做推理。这句话其实就是对动态过滤这个机制最生动的总结,模型从一个被动接收信息然后硬着头皮去理解的角色,变成了一个主动设计处理流程的角色,这种转变背后体现的是 agent 能力的一种进化方向,就是模型不光要会用工具,还要会用代码去管理和加工工具返回的结果。

说到这,我们再聊聊怎么在 API 里实际用上这个功能。好消息是,动态过滤这个能力,在你用新版网页搜索和网页抓取工具,搭配 Sonnet 四点六或者 Opus 四点六的时候,是默认开启的,你不需要单独去配置什么开关,只要你声明使用这两个新版工具,这个能力就自动生效了。文章里给了一个调用示例,大概的意思是,你在请求里指定使用 Opus 四点六这个模型,然后在工具列表里声明两个工具,一个是新版的网页搜索工具,一个是新版的网页抓取工具,这两个工具都带有一个版本号标识,代表的是二〇二六年二月九日发布的这一版接口。然后在用户消息里,你可以提一个具体的查询需求,比如说去搜索苹果公司股票代码 AAPL 和谷歌母公司股票代码 GOOGL 的当前价格,然后让模型去计算一下哪一个的市盈率更好。这个例子其实就很典型地展示了动态过滤能发挥作用的场景,因为这种查询需要模型去多个金融网站抓取实时数据,数据格式可能五花八门,有的是表格,有的是文字描述,模型需要先把这些原始页面里真正的数字提取出来,过滤掉页面上一堆和股价无关的内容,再做后续的计算和比较。这正是动态过滤大显身手的地方。

值得一提的是,文章里特别强调,对于复杂的网页搜索查询,比如说需要在大量技术文档里翻找特定信息,或者需要去核实某个引用或者某个事实的出处,这类场景下,你可以期待跟前面基准测试里展示的类似的性能提升。这给我们一个很明确的信号,就是动态过滤这个功能不是为了应付简单的一两句话搜索而设计的,它真正发挥威力的地方,是在那些需要多步骤,需要交叉验证,需要在大量信息源里去伪存真的复杂搜索任务上。

除了网页搜索和网页抓取的升级之外,这篇文章还提到了一批同期一起转入正式可用阶段的工具,我觉得这部分内容虽然篇幅不长,但信息量其实挺大的,值得我们花点时间说说,因为这几个工具放在一起看,能帮我们理解 Anthropic 现在对 agent 工具体系的整体设计思路。

第一个是代码执行工具,它提供一个沙箱环境,让 agent 在对话过程中可以运行代码,用来过滤上下文,分析数据,或者做一些计算。其实我们刚才聊的整个动态过滤机制,底层用的就是这个代码执行能力,所以这个工具进入正式可用阶段,某种程度上也是给动态过滤这个新功能提供了更稳固的基础设施支撑。

第二个是记忆功能,它让 agent 可以通过一个持久化的文件目录,在不同对话之间存储和读取信息,这样 agent 就不需要把所有东西都塞在上下文窗口里才能记住,而是可以把信息沉淀到外部存储里,需要的时候再取出来用。这跟动态过滤其实是同一个思路的不同体现,都是想办法把不必要的内容挡在上下文窗口之外,只在真正需要的时候再调取进来。

第三个是程序化工具调用,这个功能可以让复杂的多工具协作流程整体用代码来编排执行,这样中间产生的那些过程性结果就不需要进入到对话上下文里。你可以理解成,以前模型要调用好几个工具,每调用一次,中间结果都要回到对话里走一圈,模型再决定下一步干什么。现在有了程序化工具调用之后,模型可以直接写一段代码,把整个多步骤的工具调用流程串起来,一次性执行完,中间这些零碎的过程数据就不用进对话上下文了,只把最后真正有用的结果带回来。

第四个是工具搜索,这个工具背后的思路,是当你给 agent 配置了非常多可用工具的时候,不需要把所有工具的定义全部塞进系统提示词里,而是可以让模型按需去检索出当下最相关的那几个工具,这样可以大幅减少工具定义本身占用的上下文空间。其实我们这一期节目用到的工具搭载机制,本质上也是类似的设计理念。

第五个是工具使用示例,这个功能允许你在定义工具的时候,直接提供一些示例调用,用来演示这个工具应该怎么用,这样可以减少模型在调用工具时填错参数的概率。

把这五个工具和这次的动态过滤放在一起看,你会发现一个非常清晰的主线,就是 Anthropic 现在在持续地围绕着上下文窗口这个核心瓶颈做文章。不管是过滤搜索结果,还是把记忆挪到外部存储,还是把多工具调用的中间过程用代码包起来,还是按需检索工具定义,这些动作的本质都是一样的,就是尽量减少进入模型上下文窗口里的无关信息量,把真正有价值的信息留下来,把噪音挡在外面。这也呼应了我们前面讲网页搜索痛点时说的那个核心矛盾,就是上下文窗口是一个稀缺资源,你往里面塞的东西越精炼,模型的表现往往就越好,效率也越高。

那这些内容对我们这些听众,尤其是正在做或者打算做 agent 应用的工程师们,到底意味着什么呢,我们来聊聊实际应用的部分。

第一点,如果你现在的产品里已经用到了 Claude 的网页搜索功能,而且你用的是 API 直接调用方式,那这个升级基本上是一个不需要你额外做太多事情就能享受到的红利。因为只要你升级到新版的网页搜索和网页抓取工具,配合 Sonnet 四点六或者 Opus 四点六这两个模型,动态过滤默认就是开启的。所以从行动上来说,最直接的一步,就是检查一下你现在用的工具版本号是不是已经是新版的,如果还在用老版本的网页搜索工具,那应该尽快升级一下,这基本上是没什么成本就能拿到的性能提升。

第二点,如果你的业务场景里,网页搜索任务偏复杂,比如需要在大段技术文档里定位特定段落,或者需要核实多个信息源交叉验证某个事实,或者你做的是那种需要多步骤探索式搜索才能找到答案的场景,那这次升级对你来说价值会更明显。反过来说,如果你的搜索需求本身就很简单,比如就是查个天气,查个简单的事实性问题,那动态过滤带来的提升可能不会有基准测试里那么夸张,因为简单查询本身上下文里的噪音就不多,过滤的空间有限。

第三点,也是我觉得最值得大家记住的一点,就是关于成本评估这件事一定不要想当然。文章里反复强调了一句话,就是建议大家用自己业务里有代表性的查询去做实测,而不是直接拿基准测试的百分之十一和百分之二十四去套用。我们前面也提到了,Sonnet 四点六省了成本,但 Opus 四点六反而可能涨了成本,这个差异在你自己的业务里完全可能进一步放大或者反转。所以如果你打算把这个功能用到生产环境里,我建议的做法是先收集一批你产品里典型的搜索查询,做一个简单的对照评测,分别测一下开启和不开启动态过滤的时候,准确率有没有提升,价格加权后的总成本是涨了还是跌了,跑完这一轮你自己的数据,再决定是不是要全量上线。这种基于自己数据做决策,而不是直接信任官方基准的做法,我觉得在任何工程落地场景里都是值得坚持的好习惯。

第四点,我想多说一句,关于这次连带提到的那五个正式可用的工具。如果你现在的 agent 架构还比较朴素,工具调用都是一次一次地走对话回合,工具定义全部硬编码在系统提示词里,记忆全靠在对话历史里堆叠,那我建议你可以认真看看这五个工具,尤其是代码执行和程序化工具调用这两个,因为它们解决的问题跟网页搜索的问题是同源的,都是上下文膨胀的问题。如果你的 agent 经常要处理多步骤,多工具协作的复杂任务,这两个工具能帮你把很多中间过程的脏活累活挪到代码执行层面去做,而不是让模型在对话里一步一步地手动处理。

第五点,从产品设计的角度,我个人认为这次升级也释放了一个信号,就是未来 agent 工具的设计趋势,会越来越倾向于让模型自己具备一定的编程能力去管理工具调用的结果,而不是简单地把工具返回值原封不动地塞给模型去阅读。如果你在设计自己的工具接口,或者在做工具返回结果的格式设计,这次的动态过滤思路其实也提供了一个启发,就是可以考虑给模型提供一些可编程的接口或者沙箱环境,让它自己去做数据清洗和过滤,而不是事事都要你在工具层面提前帮它处理好。

好,我们来给今天这期节目做个总结。这篇文章核心讲的是一件听起来不复杂但效果很扎实的事情,就是让 Claude 在做网页搜索的时候,自己写代码去过滤搜索结果,只把有用的信息带入上下文,而不是把整页网页内容一股脑地塞给模型去阅读。这个改动带来的结果是,在两个权威基准测试上,准确率平均提升了百分之十一,输入 token 平均省了百分之二十四,而且这个功能在新版网页搜索和网页抓取工具上,配合 Sonnet 四点六和 Opus 四点六,是默认开启的。同时,这次发布还伴随着代码执行,记忆,程序化工具调用,工具搜索,工具使用示例这五个工具一起进入正式可用阶段,它们共同的主线都是想办法管理好上下文窗口这个稀缺资源。

我个人的评价是,这篇文章虽然篇幅不长,但是它很好地体现了 Anthropic 这一阵子在 agent 工具设计上的一个清晰思路,就是把越来越多原本依赖模型语言推理去完成的脏活累活,转交给代码执行去做,让模型从一个被动阅读者,变成一个主动设计数据处理流程的角色。这种思路我觉得是值得我们在自己做 agent 系统的时候认真借鉴的,不管你用不用 Claude 的这套工具,这个减少上下文噪音,把过滤工作交给代码而不是交给语言推理的设计理念,本身就是一个很有参考价值的工程经验。好了,今天的节目就到这里,我们下期再见。
