今天我们要聊的文章，英文标题是 Lessons from Anthropic on building effective human-agent teams，我把它翻译成中文，大概叫 Anthropic 关于构建高效人类与智能体团队的经验。这篇文章发表于二〇二六年六月二十四日，今天是二〇二六年六月二十八日，所以它是一篇非常新的文章，基本反映了 Anthropic 在当下这个阶段，对 agent，也就是智能体，进入真实团队协作之后的一线观察。

为什么这篇值得听？因为它讲的不是又一个更强模型，也不是某个炫技 demo，而是一个更接近我们日常工作的变化：过去我们使用 AI，通常是一个人打开一个聊天窗口，问问题、写代码、总结文档；现在 Anthropic 观察到，工作方式正在从单人游戏，变成多人游戏。这里的多人，不只是多人类，也包括多个 AI agent。人类负责设定方向、判断取舍、维持质量标准，Claude 这样的 agent 则进入 Slack、代码库、文档系统、会议纪要这些真实工作场所，参与团队目标的推进。

这篇文章的核心问题是：当 AI 不再只是个人副驾驶，而是团队成员的时候，我们到底该怎么组织它？它需要什么上下文？它应该有什么权限？谁给它分配工作？它能不能主动提建议？我们又怎样建立信任，避免一堆 agent 在旁边乱跑，制造重复劳动和信息噪音？这些问题，对今天的中国软件工程师、AI 应用开发者、技术管理者，其实都非常具体。

先看背景。Anthropic 在文章开头说，以前和 AI 一起工作，基本是一个人面对一个聊天窗口。后来模型越来越能处理复杂、长期运行的任务，比如写代码、做研究、做金融分析。工具入口也变多了，从终端、集成开发环境，到表格、演示文稿，AI 到处都能出现。但是，这种工作本质上仍然是单人体验。一个人带着一个 agent，完成自己的个人任务。

现在变化来了。随着 Claude Tag 这类工具发布，人类和 agent 可以在同一个工作空间里一起工作。文章说，这更像多人游戏。团队里的人类设定策略，Claude 执行具体工作。你可以想象一个 Slack 频道里，产品经理、工程师、设计师都在讨论，而 Claude 不是躲在某个人私聊窗口里的工具，而是直接在频道里看上下文、回应问题、接活、更新进展，甚至提出下一步要做什么。

Anthropic 把这类东西叫 multiplayer agents，也就是多人协作型智能体。它和普通 agent 一样，有自己的记忆，有自己的技能。但它又很不一样。它有自己的凭证，有自己的工具访问权限，而且它生活在真实工作发生的地方。Anthropic 举的场景是 Slack，也就是团队协作工具。agent 不是一个临时被唤起的问答框，而是一个可以和很多人同时协作的团队成员。

文章特别强调，想让 agent 真正在团队频道里有生产力，至少需要两个基础能力。一个是持久记忆。也就是说，它能记住团队目标，能随着时间不断调整自己的执行方式，而不是每次都像第一次见面。另一个是持续、广泛的信息访问。它要知道组织怎么运转，要能看到足够多的文档、代码、会议纪要和历史讨论，然后才能采取行动，服务团队目标。

但是，文章马上补了一句很关键的话：光有技术基础还不够。人类与 agent 团队要成功，还需要新的工作方式和共同规范。换句话说，agent 的效果不只是模型能力决定的，也取决于你的团队是不是会和它一起工作。这一点我个人认为非常重要。很多团队试 AI 的时候，会说这个模型不行，那个 agent 不稳定，但其实背后可能是团队的上下文不透明、角色不清楚、验收标准不存在。agent 被丢进一个混乱的团队，自然也只能放大混乱。

Anthropic 分享的第一条经验，是公开工作，并且给 agent 足够广泛的上下文。他们说，Anthropic 的团队会主动、开放地分享信息。当 agent 在团队里时，这一点尤其重要，因为 agent 对世界的理解，完全来自团队让它可搜索的文本，比如 Slack、代码、文档、会议记录。私聊、走廊里的口头讨论、受限文档，都不能给 agent 提供上下文。对 agent 来说，如果一件事没有写下来，而且它访问不到，那这件事就等于不存在。

这句话听起来简单，但放到真实工作里冲击很大。我们很多团队的关键决策，并不在文档里。可能是在一次临时会议里说了，可能是某个负责人在群里顺口提了一句，可能是两个资深工程师吃饭时达成了共识。人类还能靠记忆、关系、暗号、语气来补全信息，agent 不行。它只能读到可访问的文本。所以如果你希望 agent 帮你做靠谱的工作，第一步不是写一个更复杂的 prompt，也不是换一个更贵的模型，而是让团队的工作产物更加可见、可搜索、可复用。

文章里有一个很实际的做法：不要一份文档一个频道地纠结，到底这个能不能给 agent 看，那个频道是不是应该开放。Anthropic 倾向于使用清晰定义的安全边界，应用到整个 Slack 工作区、会议转录和文档库。边界之内，上下文流向每个队友，不管这个队友是人类还是 AI。这样做有两个好处。一方面，agent 和人都能获得更多信息；另一方面，它减少了日常工作的判断疲劳。人类和 agent 都很难处理那种软边界：这个频道该公开还是私有？这份文档能不能分享给那个人？这个 agent 能不能看到这个讨论串？如果每天都要做这种细碎判断，协作成本会非常高。

我个人觉得，这里其实是在讲一种组织层面的信息架构。很多公司以为权限管理越细越安全，但细到每个文档、每个频道、每个线程都要判断时，结果往往是大家不敢分享，agent 更是完全看不见关键上下文。Anthropic 的思路是，用少数几个清晰的工作区级别边界，替代无数个临时决策。比如公司内部可以有普通内部空间、敏感项目空间、严格保密空间。每个空间的规则很明确，只要在边界内，就默认信息可以被相关人和相关 agent 使用。这样既不是无边界开放，也不是碎片化封锁。

公开工作的回报是什么？文章给了几个例子。能读取团队会议决策的 agent，不会建议那些已经被降优先级的任务。能访问其他团队产品规格的 agent，可以推荐别的团队已经成功的模式。因为 agent 读大量文本的速度远远快于人类，它经常能把人类本来会错过的相关工作捞出来。Anthropic 说，他们很依赖 agent 来保持信息同步，在一个繁忙、快速变化的行业里维持协调。

你可以把这个想成一个永远不累的团队情报员。比如你负责一个支付系统，另一个团队上个月刚处理过类似的风控误报问题，但你不知道。人类不会每天去翻所有团队的规格文档和会议纪要，可 agent 可以。如果它有访问权限，也知道你的目标，它就可能提醒你：某某团队在类似场景里试过一种规则分层方案，结果不错，或许可以参考。这个价值不是来自神奇推理，而是来自上下文广度和检索能力。

当然，文章也没有说一切都必须公开。Anthropic 明确提到，有些交互是敏感的，需要在人类和 AI 之间保持私密。对于这些场景，可以给 Claude 发直接消息，或者使用已有的 Claude 点 ai 和 Claude Cowork 应用。这些工具通过个人 MCP 连接器访问私人信息，并且保持对话和共享内容的私密性。MCP 可以理解成 Model Context Protocol，也就是模型上下文协议，一种让模型连接外部工具和数据源的机制。这里的关键是，团队协作里的公开上下文和个人私密上下文要分开，不要混在一起。

第二条经验，是每个人类和每个 agent 都要有明确定义的角色，并且配备完成工作所需的工具。文章说，人类与 agent 团队共享一份成员名单、一组工作产物和一个工作空间。agent 有自己的凭证、技能和工具访问权限。不同 agent 也承担不同角色。比如一个 agent 负责项目的数据分析，另一个负责维护和执行设计标准，第三个负责研究综合。

这和我们平时随手叫一个 AI 做事差别很大。普通用法是，我打开 Claude，说帮我分析一下这个问题。多人 agent 团队里，问题变成了：团队里有哪些人类，哪些 agent？谁负责什么？哪个 agent 有什么工具？它能不能访问 BigQuery，也就是大规模数据分析平台？它能不能使用 Playwright MCP 来跑浏览器自动化和质量检查？它的职责范围到哪里结束？如果任务超出范围，它应该问谁？

文章说，当项目启动时，人类会和 agent 聊，决定要分配哪些角色，以及人类和 agent 怎么一起工作。一旦人类和 agent 的工作清楚了，一个 agent 甚至可能启动其他 agent，确保特定任务交给有合适记忆和合适访问权限的 agent 处理。这里的重点是，不同 agent 不是一锅粥，而是有分工、有边界、有工具的。

为什么角色这么重要？因为没有清晰角色时，人们会在旁边各自运行一堆个人 AI，重复劳动，团队上下文也被撕裂。文章举了一个很常见的例子：指标追踪。很多团队里，每个人都可能让自己的 AI 算一遍指标，口径不一致，数字不同，最后会议上吵半天。更好的方式是，一个多人协作 agent 统一负责这件事，做一次，让所有人看到同一组数字。这样不仅节省工作，更重要的是建立共同事实。

Anthropic 说，他们团队会为人类和 agent 约定任务集，大家明确谁做什么；人类和 agent 在同一个共享线程里工作，这样任何人都能接着别人做过的地方继续；每个人和每个 agent 都有完成职责所需的工具；还会写清楚 agent 的角色和范围。有一个工程团队甚至开始创建 roster，也就是团队花名册，用来编码人类和 agent 的角色。这样工作更容易推进，也更具体。

文章里还提到几个他们早期觉得很有用的点。具体角色能帮助人类轻松追踪责任归属，不管是单个任务，还是整个团队的责任集合。通过写 skill files，也就是技能文件，来定义特定 agent 的角色，可以让专业化更容易，也方便公司其他地方快速创建同类型 agent。当项目更复杂时，团队会添加新的 agent 聚焦新的领域。比如他们添加了一个 release manager agent，也就是发布经理智能体，专门处理新的软件发布。

这里我想多讲一下 skill files。原文没有展开技术细节，但从语义上看，它就是把一个 agent 的角色、能力、工作方式、约束、常用工具写成可复用的配置或说明。对工程团队来说，这很像把口口相传的岗位职责变成可版本化的团队资产。比如你有一个代码审查 agent，它的 skill file 里可以写清楚，它要关注安全风险、性能回退、测试覆盖、架构边界，不负责替产品做需求取舍。再比如一个文档 agent，它要遵守公司的写作风格指南，输出前要检查术语一致性和受众层级。这样新建同类 agent 时，不是从零开始调 prompt，而是复用已经验证过的角色定义。

这背后有一个更大的变化：团队的组织设计开始包括 AI 成员。过去我们做项目，会画人力分配，谁是后端负责人，谁是前端负责人，谁负责测试，谁负责发布。现在可能还要加上，哪个 agent 负责回归测试，哪个 agent 负责整理用户反馈，哪个 agent 负责把决策同步到文档，哪个 agent 负责每周生成风险报告。它们不一定替代人，但会改变人怎么分配注意力。

第三条经验，是设定 north star，也就是北极星目标，让 agent 更主动。文章说，Anthropic 有些 agent 只是完成被分配的任务，但最重要的一些 agent 会主动建议新项目和新工作流。这通常发生在一个团队已经给 agent 丰富上下文和清晰角色之后，又增加了另一个指引：北极星。

北极星是雄心比较大、覆盖范围比较广的目标，帮助团队判断哪些任务和工作流是正确的。Anthropic 强调，北极星始终由人类设定，而且要扎根在公司的使命和业务目标里。也就是说，agent 可以提出执行方案，可以建议工作流，但团队为什么要做这些事，长期方向是什么，这个判断还是人类负责。

当北极星被清楚地写下来之后，人类会把它分享给团队里的 agent。接着还有一个重要动作：人类要明确选择哪些 agent 可以主动建议新的工作流，来帮助实现这个长期目标。不是每个 agent 都应该主动提建议。文章说，不太可能团队里每个 agent 都具备主动成功建议工作的前提技能和信任水平。

这个限制很重要。很多人一听 agent 主动性，就会想到全自动、自主规划、自己发现问题自己解决。听起来很美，但在组织里很危险。一个没有足够上下文、没有足够角色边界、也没有建立信任的 agent，如果天天主动建议新项目，最后可能只是制造噪音。Anthropic 的做法更谨慎：先有人类设定北极星，再把北极星写下来，再指定少数适合的 agent 可以围绕它主动推荐工作流。

文章给了一个例子。一个内部工具团队的北极星是让产品 onboarding，也就是新用户或新员工上手流程，更有帮助。一个 agent 主动建议修改 onboarding 流程中错误消息的文案。这些改动在下一周可衡量地提高了 onboarding 成功率。这个例子很小，但非常说明问题。agent 不一定要上来就重构系统、制定战略，它可以在明确目标下发现一个具体可改的小点：错误提示不够清楚，用户卡住了，改文案可以提高完成率。

如果放到我们日常工作中，你可以想象一个平台团队的北极星是让内部开发者在十分钟内完成服务接入。那一个具备足够上下文的 agent 可能会主动发现，最近很多工单都卡在权限申请；或者文档里某一步过期了；或者错误日志提示太晦涩。它不是凭空想事做，而是把北极星和它能看到的数据、文档、工单、代码变化关联起来，然后提出高杠杆的小改动。

Anthropic 还提到，设定北极星之后，要保护高保真人类时间，也就是把人类最有价值的会议时间留给最重要的工作。当 agent 能处理更多信息整理、候选方案、执行细节时，会议就不应该继续被状态同步占满，而应该聚焦判断、取舍和方向。这个转变对很多团队很关键。AI 最先节省的，可能不是编码时间，而是信息搬运时间。如果团队会议仍然只是每个人读一遍进度，那 agent 的价值会被浪费。

第四条经验，是随着时间建立信任。文章说，Anthropic 的团队会按照 agent 已经证明的可靠性来授予自主权，然后有意识地扩大范围。他们的工程师已经成功派出团队里的 agent 独立处理五百个 bug 修复，但事情一开始肯定不是这样。

这段我觉得是全文最务实的部分。很多关于 agent 的讨论，要么很乐观，觉得马上能全自动干活；要么很悲观，觉得它总会出错不能用。Anthropic 的态度更像管理一个新同事。一个新人加入团队时，你不会第一天就让他独立改生产系统关键路径。你需要时间评估能力，形成协作习惯，多轮反馈，把那些隐性的知识外化出来。agent 也一样。用户必须尝试把许多不同任务交给 agent，才能知道它能做什么，怎样描述目标更清楚，它需要哪些 skill files，哪些 prompt 能引出想要的行为。

文章还提醒，随着模型变化和改进，任务要重新测试。以前有用的 prompt 可能需要改写，以前有帮助的护栏，可能会限制更聪明模型追求更有创造性的解决方案。这一点在今天非常现实。我们很多团队把某个 prompt 调好之后，就想长期不动。但模型版本一变，原来的措辞可能不再最优。更强模型可能不需要那么多手把手约束，过度约束反而让它绕不开问题。所以使用 agent 的团队，也需要像维护测试和依赖一样维护 prompt、技能文件和护栏。

Anthropic 发现，最好的长期运行 agent，通常有很多种方式在给人类看之前先验证自己的工作。代码当然有测试，但其他工作也可以验证。比如技术文档可以用评分规则和风格指南检查。当人类设定质量门槛，并确保交给 agent 的工作都能被审核时，质量就能保持在较高水平，不会偏离最初意图。另一个做法是，像人类团队一样，让一个 agent 负责做事，另一个 agent 负责检查第一个 agent 的工作。这通常叫 Doer Verifier agent harness，也就是执行者与验证者的智能体框架。

这个模式其实非常适合工程团队理解。我们从来不会因为某个工程师很强，就不做 code review、不跑测试、不看监控。agent 更应该如此。一个写代码 agent 生成补丁后，另一个验证 agent 可以跑测试、检查 diff、对照需求、找潜在破坏。一个写文档 agent 产出草稿后，另一个验证 agent 可以检查术语、事实、风格、缺失步骤。人类最后看的不是原始混乱输出，而是经过自检和互检的结果。

文章里列出 Anthropic 建立信任的方式，包括一开始人工审查 agent 工作，检查质量、提供反馈、设计任务验证清单；让 agent 使用 verifier，也就是验证者 agent，作为任务的一部分来检查工作；把反思纳入循环，让 agent 回顾自己的失误，从而随着时间改善；追踪每个 agent 已经在哪些任务类型上赢得自主权，并在多次成功后按任务类型扩大范围。

这里有一个很精彩的真实案例。Anthropic 的一位工程负责人接手了一个新团队，面对很大的 backlog，也就是待办事项积压。为了摸清情况，他邀请几个人类和几个 agent 一起帮助整理 backlog，并确定最重要的优先级。团队里一组 agent 阅读 backlog 里的所有条目，判断是否有人正在处理，并给没有负责人的条目打复杂度分。另一组 agent 从列表中读取条目，筛出中低复杂度项目，然后创建代码变更。

一开始，人类会审查 agent 做出的每一个决策，并标记哪些需要人类输入。然后人类教 agent 直接把这些决策抛给人类，确保那些有艰难取舍的决策始终有人类在环。每周，这位负责人和团队都会要求 agent 编写一份周报，其中包括 lessons and missteps，也就是经验和失误，让 agent 记录错误，未来避免重复。随着时间推移，这位负责人能够把越来越复杂的代码改动交给 agent，并减少日常指导 agent 的时间。

这个案例很值得拆开看。第一，它没有让 agent 一上来接管全部 backlog，而是先做分类、归属判断、复杂度评分这类比较适合大规模阅读和结构化整理的工作。第二，它把任务按复杂度过滤，让 agent 从中低复杂度代码变更开始。第三，它把人类审查放在早期，建立反馈回路。第四，它不是只看一次输出，而是每周让 agent 总结错误，形成组织记忆。第五，当 agent 更独立后，人类开始训练它如何更好地消耗人类注意力。

文章说，这位负责人后来会指导 agent 把人类注意力当作稀缺资源。具体做法包括：把问题批量整理，让人类一次性回答；重复关键上下文，让人类快速进入状态；限制每个人类一次看到的事项数量。这个点太真实了。很多 AI 工具看似省时间，但如果它每五分钟问你一个问题，每次都要你重新读一堆上下文，那它实际上是在碎片化你的注意力。一个好的团队 agent，不只是会做事，还要会汇报、会提问、会等待、会合并问题。

文章还提到，有些团队里专门有 agent 的角色，就是决定如何批量处理沟通，并只把最重要的信息提升给人类成员。另一些团队会设置护栏，限制 agent 每天做多少工作，这样人类能够有意义地参与审查。这样的护栏确保人类保持对自己重要的技能，也确保需要人类审查的事项数量可持续。

这点可能有些反直觉。我们总觉得 agent 越能干越好，最好一天做一千件事。但如果团队只有两个工程师能审查，而 agent 一天生成一百个代码变更，最后不是效率提升，而是审查崩溃。更糟的是，人类可能开始粗略通过，质量风险上升，或者完全放弃理解系统。Anthropic 这里提醒我们，agent 的产能必须和人类的判断带宽匹配。自动化不是无限加速，而是重新设计流量控制。

讲到这里，我们可以把全文的逻辑串起来。Anthropic 并不是说，只要有 Claude Tag，团队协作就自动变好。它说的是，要让多人 agent 有效，团队要做几件基础但很难坚持的事。信息要在清晰安全边界内尽量公开，让 agent 有上下文。人类和 agent 要有明确角色、责任、工具和共享工作空间。团队要有人类设定的北极星，让合适的 agent 可以主动围绕长期目标提出工作。信任要通过审查、验证、反馈、反思和逐步授权建立，而不是一次性授予。

文章最后给了一组问题，供团队在打基础时自查。虽然原文是问题形式，我在这里用口语串一下。你可以问问自己的团队：agent 和人类需要的信息与访问权限，是不是公开并且可广泛搜索？你能不能写下团队的成员名单，包括人类和 agent，并说明每个成员负责什么？每个人和每个 agent 是否都有完成工作所需的工具？关键工作产物有没有评分标准或测试，让人类和 agent 能够验证？团队有没有一个大家都能引用的清晰北极星？

这些问题看起来朴素，但如果认真回答，很多团队会发现自己还没准备好。比如文档散落在个人空间，关键决策只在私聊里，代码评审标准不一致，需求目标经常变化，团队没有明确的成功指标。在这种情况下，引入 agent 可能会让问题暴露得更快，而不是立刻解决问题。

接下来我们聊聊对听众的实践意义。假设你是一个中国互联网公司的后端工程师，或者 AI 平台团队成员，你现在想把这篇文章转化成可以上手的行动，我建议不要一开始就追求全自动 agent 团队。可以先从一个共享频道、一个明确任务、一个可验证产物开始。

比如你可以选一个最适合 agent 的场景：每周整理线上问题和 backlog。你先把相关工单、日志链接、代码仓库、历史事故复盘放到一个有清晰权限的空间里。然后定义一个 agent 的角色：它负责读取新增工单，聚类相似问题，判断是否已有负责人，给出初步复杂度建议，并生成每周摘要。它不负责最终优先级，不负责直接改生产代码，不负责跨团队拍板。这样角色边界就很清楚。

然后你设计验证方式。比如复杂度评分可以和人类标注对比，聚类结果可以抽样检查，摘要要包含原始链接，不能只给结论。前几周，人类逐条审查。发现 agent 常犯什么错，就把错误写进技能文件或检查清单。等连续几周质量稳定，再让它承担更多事情，比如为低复杂度问题起草修复方案，或者生成代码变更草稿。再往后，才考虑让它在测试充分的模块里创建 pull request。

如果你是技术管理者，可以从团队规范入手。不要只问，我们要不要买一个 agent 工具。要问，我们的工作是否足够公开？我们的决策是否落到文档和会议纪要里？我们有没有清晰的安全边界？我们能不能画出一张人类加 agent 的 roster？我们的北极星是什么？哪些 agent 可以主动建议工作，哪些只能被动执行？这些问题比工具选型更先行。

如果你是 AI 应用开发者，这篇文章也给了很强的产品启发。未来的企业 agent 产品，不应该只是一个更聪明的聊天框，而应该支持团队级别的权限模型、持久记忆、角色配置、工具凭证、共享线程、验证流程、审查队列、周报反思、沟通批处理。也就是说，agent 产品要从个人效率工具，变成组织协作系统。它需要懂组织边界，也需要懂人类注意力管理。

这里有几个坑我个人认为尤其要避开。第一个坑，是把所有信息都喂给 agent，却没有安全边界。公开工作不是无脑公开，而是在清楚的组织边界内公开。敏感信息、个人隐私、商业机密必须有明确隔离。第二个坑，是给 agent 太大的目标，但不给工具。比如让它提升转化率，却不给它访问数据、实验结果、用户反馈的权限，那它只能说空话。第三个坑，是让 agent 主动，却没有北极星。没有方向的主动性，就是噪音。第四个坑，是没有验证机制。只要工作无法测试、无法评分、无法审查，agent 的质量就很难稳定。第五个坑，是忽视人类审查带宽。agent 产出太多，人类看不过来，最后团队会进入一种假自动化状态，表面很忙，实际风险累积。

我还想强调一个对工程文化的影响。文章结尾说，这些模式对人类来说并不新。强北极星、清晰角色、强文档、共同质量标准、从错误中学习的空间，这些都是我们几十年来就知道的健康团队习惯。agent 只是让这些基础更不能被跳过。最能从 agent 获益的团队，是最有意识应用这些基本功的团队。

这句话其实有点扎心。AI 并不会让一个组织绕过管理和协作基本功。相反，它会放大这些基本功的差异。一个文档清楚、边界清楚、质量标准清楚的团队，引入 agent 后，agent 能迅速读懂上下文，承担重复工作，发现遗漏信息，推动执行。一个信息混乱、职责模糊、目标摇摆的团队，引入 agent 后，可能只是多了几个更快制造混乱的参与者。

所以我个人对这篇文章的总体评价是，它把 agent 的讨论从模型能力，拉回到了团队设计。过去我们问，模型能不能写代码？能不能总结文档？能不能做研究？现在更重要的问题变成，组织能不能给 agent 一个可工作的环境？能不能把隐性知识写下来？能不能为 AI 成员设置角色、权限、工具和质量标准？能不能让人类判断力用在最重要的地方？

文章最后提到，读者可以使用 Claude Code 里的 agent teams，或者使用 Claude Tag，开始构建多人 agent。这里我不展开具体产品，因为工具会持续变化。但这篇文章留下的方法论更稳定：让信息在安全边界内流动，让角色和责任可见，让目标足够清晰，让验证成为流程的一部分，让信任随着成功经验逐步增长。

总结一下，Anthropic 这篇文章告诉我们，AI 工作方式正在从单人聊天窗口，转向人类与多个 agent 在共享工作空间里协作。真正有效的多人 agent 团队，不靠神秘魔法，而靠公开上下文、清晰角色、北极星目标和持续验证。我个人认为，未来一年，很多工程团队的竞争力差异，不只在于谁用了更强模型，而在于谁能更早学会把 agent 当作团队系统的一部分来设计。
