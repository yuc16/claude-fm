文章里说，Claude 现在通过几个关键的构建模块，连接进了整个法律科技栈。第一个模块是 MCP 连接器。这里简单解释一下 MCP，它的全称是模型上下文协议，你可以把它理解成一种标准化的接口规范，让大模型可以用统一的方式去读取和操作外部系统里的数据，而不需要为每一个软件单独写一套定制对接代码。有了 MCP，Claude 就能把具体某个案子相关的文档、往来邮件、记录都拉进对话里，而不是停留在通用的法律常识问答层面。第二个模块是按执业领域打包的插件，把律师最常重复执行的那些任务封装成现成的工作流，开箱即用。第三个模块是开放协议带来的可定制性，因为 MCP 和插件体系都是开放的，律所和企业法务团队可以按照自己实际的执业习惯去定制 Claude，而不是被迫迁就一个一刀切的通用方案。这三块拼在一起，就是这次发布的整体框架。

我们先看 Claude 怎么嵌入到日常办公软件里。文章特别强调了一点，Claude 是直接在 Word、Outlook、Excel、PowerPoint 这四个应用内部工作的，而且关键是，它能在这四个应用之间保持上下文的连续性。这句话听起来平淡，但其实分量很重，我用一个具体场景给你拆一下。假设你在 Word 里完成了一份合同的红线修订，红线修订就是那种带着修改痕迹、双方可以看到具体改了哪里的版本。按照传统流程，你改完合同之后，还要专门花时间写一封邮件去解释你改了什么、为什么这么改，这部分解释往往是重复劳动，因为逻辑你脑子里已经想清楚了，只是要再翻译成邮件语言。但现在，因为 Claude 在做红线修订的时候本身就记录了每一处改动以及背后的理由，当这份合同变成 Outlook 里的一封附信的时候，Claude 可以直接把这些上下文带过去，不需要你重新讲一遍。同样的道理，这份合同对应的关键节点，比如签署日期、付款条款，可以自动变成 Excel 里交割清单上的一行，最后这个交易的摘要也可以直接变成 PowerPoint 里给董事会汇报用的几页纸。整个过程里，逻辑只解释一次，后面全是格式的搬运和复用，而不是每个软件里重新理解一遍。

具体到 Word 里，文章提到了一个叫 Claude skills 的概念，我把它翻译成可复用的指令集，本质上就是把一个团队的工作手册和标准，用结构化的方式写成 Claude 能直接调用的指令。这些 skills 负责起草合同、做红线修订、按照公司内部的合同审查标准做逐条比对，每一步改动都留痕，并且解释改动的理由。除了这些主线任务，Claude 还顺手把每一轮修改之间那些零碎的、机械性的活儿也接管了，比如发给对方律师之前，先把内部讨论留下的批注全部清理干净，避免把内部的犹豫和分歧暴露给对方；再比如在最终签署版本上做一次格式校验，确保字体、编号、页眉页脚都符合规范；又比如从公司批准过的合同模板库里，直接抓取某个条款的标准备选措辞。这些事情单个看都很小，但加起来是律师日常工作里相当大的一块时间消耗，而且基本不需要法律判断，纯粹是体力活，正好是交给 Claude 去做最划算的部分。

Outlook 这边，Claude 干的事情更偏向分流和调度。一个律所或者法务部门每天会收到大量跟具体案子相关的邮件，有人要求审一份合同，有人在问一个流程节点，还有人在催进度。Claude 在 Outlook 里负责把这些邮件做初步分类，识别出哪些是新的合同审核请求，草拟回复和附信，并且安排好后续的跟进时间点，避免事情堆积到没人记得的角落。这个场景其实非常贴近很多公司行政和运营岗位每天面对的邮件洪流问题，只不过法律场景里多了一层专业判断，需要识别出哪些请求涉及风险、需要升级处理。

到了 Claude Cowork 里，同样的连接器和插件被用在那种跨越很多文档的工作上，比如一次性分流一整批合同，给某个新产品上线做合规审查，或者给董事会起草一份监管动态的汇报。文章还提到可以设置定时任务，自动跑那些周期性重复的工作，比如每周做一次监管动态扫描，或者每天做一次新案子的分流登记。再加上 Projects 这个功能，案件团队也就是处理同一个案子的团队，可以拥有一个持续存在的工作空间，先例和之前的草稿在每一次对话里都能被保留和复用，不需要每次都重新上传、重新解释背景。

接下来我们看二十多个新 MCP 连接器具体接的是哪些系统，这部分信息量很大，我挑几个最有代表性的类别给你讲清楚，而不是逐个念名字。第一类是合同生命周期和起草工具，比如 Docusign 的连接器，能让 Claude 直接读取合同数据，快速定位续约日期、各方义务这些关键条款，并且能在起草、签署、签署后管理的整条链路上协调工作流。还有一个叫 Definely 的连接器，听起来比较专业，它给的是对合同结构的实时确定性访问，可以解析定义条款、校验交叉引用是否对得上、绘制条款之间的依赖关系，还能做结构化的差异比对，看一次修改会怎么沿着合同的其他部分传播下去。这个其实解决了一个律师审合同时最头疼的问题，就是改了第三条第二款，要去检查后面有没有别的条款引用了它，人工排查很容易漏掉，机器做这种结构化检索是它的强项。同一类别里还有 Ironclad 的连接器，让 Claude 直接访问企业自己的合同库和审批流程，用大白话提问就能找到答案，比如问某一类合同里通常怎么约定违约责任，而且每个人查到的结果都会按照他本来的权限自动收窄，不会出现越权看到别的部门合同的情况。

第二类是数据室和交易文档相关的连接器，比如 Box 这个云存储和内容协作平台，接进 Claude 之后可以检索和访问存放在里面的文件，对文档内容做提问，创建或更新内容，提取里面的元数据字段，并且全程遵守 Box 原有的安全和访问策略；再比如接入 Datasite 这个虚拟数据室平台，每年有大量并购交易是在这个平台上完成尽职调查的，Claude 接进去之后可以搭建文件夹结构、邀请用户、检索文档、跟踪买方提出的问题和回答，还能审计整个数据室的准备完整度，这种审计能力其实挺关键的，因为一笔并购交易里数据室资料准备得齐不齐全，直接影响交易能不能按时交割。第三类是文档管理平台，比如 iManage 和 NetDocuments，这两个是律所内部用来管理海量历史文档和制度知识的系统，连接进去之后，Claude 能在权限范围内访问案件历史和机构知识，不需要批量导出文件或者搭建定制接口，这一点对合规要求很高的律所特别重要，因为法律行业对谁能看什么文档管得很严。

这里我想插一句，文章里还专门提到了一类叫专家网络和技能市场的连接器，比如 Lawve AI，它提供的是一整套由执业律师、企业法务和法律科技专家亲手写的技能库，可以直接在 Claude 里搜索调用；还有 The L Suite 这个企业法务社群提供的两个连接器，一个叫 Lloyd，让会员能把 Claude 接进他们自己的会员交流平台，另一个叫 TopCounsel，帮企业法务根据社群积累的专有数据和评分算法，给某个具体案子匹配合适的外部律所。这一类连接器其实跳出了纯技术对接的范畴，本质上是把行业里原本靠人脉和经验积累的隐性知识库，变成了 Claude 可以检索的显性资源，这个思路我觉得在其他依赖圈子口碑的专业服务行业里也完全可以复制。

第四类是电子取证和审查工具，这是诉讼里非常核心的一环，案子涉及的邮件、文件动辄几十万份，全靠人工一份份看是不现实的。文章提到了 Everlaw 和 Relativity 这两个平台的连接器，Claude 可以用元数据、关键词、文档类型去检索和组织海量文档，并且直接给出审查链接，跳转到具体证据所在的位置。第五类是法律研究和案例法工具，比如 Legal Data Hunter 提供的是全球增长最快的法律语料库之一，超过三千一百万份文档，覆盖一百六十多个司法辖区；还有 Midpage 连接的案例法数据库，所有引用都带着指向原始来源的链接，方便律师核实。这一点我觉得设计得挺聪明的，大模型做法律研究最大的风险就是编造引用，这些连接器从架构上就把每条结论钉死在一个可验证的真实出处上，而不是靠模型自己背书。

第六类是已经做成完整产品的法律 AI 助手，文章特别提到了 Thomson Reuters 的 CoCounsel Legal，这是一个面向高风险法律工作的、号称信托级别的系统，覆盖起草、研究、审查、验证的全流程，底层依托 Westlaw 这个权威法律数据库、Practical Law 的实务指引，还有 KeyCite 的引证核查系统，并且强调输出是透明可验证的。这里有个很值得注意的细节，文章提到 Thomson Reuters 在今年二月份的一次发布会上，展示了 CoCounsel 是用 Claude Agent SDK 重新搭建的，而这次发布之后，这条集成变成了双向的，既是 CoCounsel 接进 Claude，也是 Claude 的能力底座撑着 CoCounsel。类似的情况在 Harvey 和 Solve Intelligence 这些法律科技公司身上也在发生。这其实说明了一个挺有意思的行业趋势，原本这些公司是站在 Claude 的下游，拿模型能力做应用，现在关系变得更复杂，他们既是 Claude 生态里的连接器供应方，自己的底层技术栈又是用 Claude 搭的，整个生态变成了一张互相嵌套的网。

最后一类，也是我觉得这篇文章里比较有温度的一部分，是面向公共服务的连接器。比如接入 Free Law Project 旗下的 CourtListener，那里收录了数百万份美国法院的判决意见、PACER 系统里的诉讼记录、法官档案、口头辩论录音和引证数据。还有专门给没有律师代理的普通诉讼当事人用的 Courtroom5，文章里提了一个数据，美国大约有百分之八十的民事诉讼当事人是没有律师、自己出庭应诉的，Courtroom5 提供的是分司法辖区的案件咨询、截止日期计算和下一步行动指引，覆盖全美五十个州。这件事我个人认为是这次发布里最值得多说两句的部分，因为大模型在法律领域最容易被想象成的应用，是帮大公司、大律所降本增效，但这篇文章特意把法律援助、普通人不知道自己有什么权利这件事摆出来讲，说明至少在叙事层面，Anthropic 想把这次发布定义成既服务头部客户、也下沉到普惠场景的两条腿走路。

讲完了二十多个连接器，我们再看十二个执业领域插件。这十二个插件分别对应不同的法律岗位，包括商业法务、公司法务、雇佣法务、隐私法务、产品法务、监管法务、人工智能治理法务、知识产权法务、诉讼法务、法学院学生、法律援助诊所，还有一个叫法律构建者中心的插件，专门用来从公开的技能注册库里发现和安装社区做的法律技能包，并且每次安装和更新都会自动跑一遍安全审查、许可证检查和新鲜度检查。这十二个插件有个共同的设计，就是每一个插件在你开始用之前，都会先做一次简短的设置访谈，了解你具体的工作手册、升级处理链条、风险偏好和团队的行文风格，这样 Claude 给出的答案才是贴合你团队实际打法的，而不是放之四海而皆准的通用建议。

我挑几个插件具体说说，你就能感受到这种设计的颗粒度有多细。商业法务插件负责按照你团队的审核标准去审供应商合同和保密协议，遇到需要升级处理的情况，会自动生成一份给业务部门看得懂的、用通俗语言写的摘要，而不是甩一堆法律术语过去。公司法务插件管的是并购交易里的一整套流程，包括数据室里的尽职调查、披露清单、董事会决议、交割前的检查清单，而且可以根据你具体是做董事会工作、上市公司治理，还是主体合规来分别配置。隐私法务插件会拿数据处理协议去对照你的审查标准，对隐私影响评估和数据保护影响评估做分流处理，在法定时限内准备好数据主体访问请求的回应，还会主动发现你写在纸面上的政策和团队实际操作之间存在的落差。诉讼法务插件覆盖的范围最杂，从案件受理和案件组合追踪，到法律保留通知、催款函、传票分流、时间线梳理、证词准备、特免权日志，一直到辩论状的起草都管。法学院学生插件就比较特别了，它面向的不是执业律师，而是法学院的学生，提供苏格拉底式的提问训练、案例摘要、用 IRAC 这种法律分析框架来批改作业，还有针对不同司法辖区差异的律师资格考试备考辅导。

另外两个我觉得特别值得一提，一个是雇佣法务插件，负责处理招聘、解雇、员工分类、休假截止日期和内部调查这些事，起草政策的时候会自动套用具体州的规则，这对在美国跨州经营的公司特别有用，因为各州劳动法差异很大，人工很难每次都记全。另一个是人工智能治理法务插件，专门用来按照公司自己设定的治理分级，给内部的 AI 应用案例做风险分流，跑算法影响评估，审查供应商提供的 AI 服务条款，还能帮你起草一份起步用的 AI 使用政策。这个插件我觉得对我们这些做 AI 产品的听众格外有现实意义，因为现在几乎每家公司都在被动或者主动地搭建自己的 AI 治理流程，而这个插件相当于把这件事打包成了一个可以直接调用的法务工具，如果你所在的公司还没有成型的 AI 治理框架，这其实是个很具体的参考起点。

这十二个插件每一个都可以在 Cowork 或者 Claude Code 里一键安装，而且其中商业法务、公司法务、产品法务、诉讼法务这几个还专门做成了可以部署成托管 agent 的方案，接入 Claude 平台之后能用程序化的方式批量调用，这对那种希望把法律审核能力嵌入到自己内部系统里的公司会很有用。文章里还强调了一句，说这些插件和技能体系都是建立在开放协议之上的，已经有 Box、Legal Quants、Lawve AI、Thomson Reuters 这些早期合作方提交了自己做的技能包、插件和文风规范，任何合作伙伴都可以通过目录提交自己的连接器和技能。这种开放生态的思路，其实跟我们熟悉的应用商店模式很像，Anthropic 自己提供地基和标准，内容的丰富度交给整个生态去填充。

文章末尾提到，这一整套更新是建立在 Claude Opus 四点七这个模型上的，文章称它是当时对外公开可用的、在法律推理和长文档处理上最强的模型。考虑到法律工作天然涉及大量长篇合同、判例和监管文件，模型本身的长文本理解能力，其实是这整套连接器和插件能不能跑顺的地基，如果模型读不懂、记不住一份几百页的合同，前面说的那些自动化流程都无从谈起。

说到这里，我们来聊聊这篇文章对我们这些做技术、做 AI 产品的听众，到底有什么实际的参考价值。我觉得最值得借鉴的不是法律这个具体行业，而是这次发布背后那套做垂直行业 AI 产品的方法论。第一点是分层解耦的思路，把数据连接、任务封装、定制化这三件事拆成三个独立但又能拼起来的模块。如果你正在给某个垂直行业做 AI 产品，不妨问自己一个问题，你的产品是不是也能拆成类似的三层，先解决怎么把行业里那些割裂的系统数据安全地接进来，再解决怎么把这个行业最常见的重复性任务打包成现成的工作流，最后再留一个开放的定制层，让客户按自己的实际情况去调整，而不是指望一套通用方案打天下。

第二点是权限和可验证性这两件事，在垂直行业的 AI 产品里几乎是生死线。你注意到没有，文章里提到几乎每一个连接器都在强调权限范围，结果自动限定在用户本来就有权限看到的内容，文档管理平台的连接器强调不需要批量导出、完全尊重原有的治理策略，法律研究类连接器强调每一条结论都挂着可验证的原始出处链接。这些细节如果你只是泛泛地读一遍很容易忽略，但如果你自己去设计一个面向企业的 AI 产品，这恰恰是最容易被客户的安全合规团队卡住的地方。换句话说，如果你来做一个类似的产品，光是模型能力强是不够的，你必须在架构层面就把权限收窄和结果可追溯这两件事做实，不能事后补丁。

第三点是那个设置访谈的细节，我觉得特别值得玩味。十二个插件每一个上手前都先做一轮访谈，把客户的工作手册、风险偏好、行文风格吃进去，这其实是在用对话的方式做轻量级的微调，不需要重新训练模型，只是用提示词和上下文工程把通用模型变成贴合具体团队习惯的专属助手。这个思路放到任何一个强调专业判断、强调行文规范的行业里都适用，医疗、金融、咨询都可以照搬这套打法，先用几轮对话把客户的隐性知识显性化，再让模型按照这份显性化的标准去执行后续任务。如果你来做一个类似的产品，我建议把这个设置访谈本身做成可复用的组件，而不是每个插件各写一套，因为不同执业领域要采集的信息结构其实是相通的，无非是工作手册、升级路径、风险红线、行文偏好这四类，把它抽象成一个通用的访谈框架，后面每加一个新领域的插件，成本会越摊越薄。

第四点是生态策略，这篇文章花了不少篇幅强调连接器和插件体系是建立在开放协议上的，已经有外部公司在提交自己的技能包和插件。这和单纯做一个封闭的官方功能列表是两种完全不同的打法，前者是把自己变成平台，让整个生态去帮你把场景覆盖完，后者是自己一个个把场景做全，后者注定做不完，因为任何一个行业内部的分支和惯例都比官方团队能想到的多得多。如果你来设计一个垂直行业的 AI 产品，值得提前想清楚自己是想做平台，还是想把每个场景都做透，这两条路径需要的组织能力和耐心是完全不一样的，半途从后者切换到前者往往要付出不小的重构成本。

如果你自己想上手体验，文章里提到这些连接器和插件都已经在 Claude Cowork 里开放，企业管理员可以在工作空间设置里直接启用，法律援助和公益类机构还可以通过 Claude for Nonprofits 项目申请折扣价格。当然如果你不是做法律相关工作的，这篇文章对你最大的价值，我觉得还是这套方法论本身，值得你套到自己所在的行业上想一遍，看看哪些系统是割裂的，哪些重复劳动是可以打包成插件的，哪些权限和可验证性的坑是你现在就该提前挖好护栏的。

最后简单总结一下。这篇文章讲的核心事情，是 Claude 通过二十多个 MCP 连接器和十二个执业领域插件，把整个法律科技生态系统性地接了进来，从合同起草、尽职调查、电子取证，到法律研究和公益法律援助，几乎覆盖了法律工作的每一个角落，而且始终在 Word、Outlook、Excel、PowerPoint 这些律师每天都在用的软件里原地工作，不需要额外学一套新系统。我个人认为这篇文章最大的亮点，不是连接器数量堆得多，而是背后那套分层解耦加上权限优先的产品方法论，这套打法对任何想做严肃垂直行业 AI 产品的团队都有参考价值。好了，这就是今天的内容，我们下期再聊。
