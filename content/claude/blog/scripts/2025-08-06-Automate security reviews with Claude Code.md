大家好,欢迎收听 Claude FM,这是一期通勤路上适合听的科技播客。今天我们要聊的这篇文章,标题是 Automate security reviews with Claude Code,翻译过来就是"用 Claude Code 自动化安全审查"。这篇文章发表于二〇二五年八月六日,也就是去年夏天,距离我们今天录这期节目,已经过去差不多十个半月了。

先说一句提醒。十个月在普通行业里不算长,但是在 AI 辅助编程这个领域里,十个月几乎可以算是一个时代。这篇文章发布的时候,Claude Code 这个产品本身还相对年轻,业界对"让 AI 自己审查代码安全性"这件事,还处在一个比较新鲜、很多团队在观望、还没有完全形成共识的阶段。如果你现在去看 Claude Code 的官方文档,你会发现这套安全审查的能力可能已经迭代了好几个版本,细节可能和文章里写的有所不同。所以我们今天聊的,更多是这个产品思路和设计理念,至于具体的命令参数、配置方式,建议大家以官方最新文档为准,我们权当是回顾一下这个功能最初是怎么诞生的,以及它解决了什么问题。

那我们先说说背景。这篇文章一开篇就点出了一个矛盾,这个矛盾我相信很多写代码的朋友都会有共鸣。随着开发者越来越依赖 AI 来加快开发速度、构建更复杂的系统,代码的安全性反而变得更加重要,也更容易被忽视。这句话听起来有点绕,我们拆开来讲。

以前我们写代码,速度是有上限的,这个上限就是人脑思考和打字的速度。一个工程师一天能写多少行代码、能改多少个文件,是有生理极限的。但是现在有了 AI 编程助手,这个速度被极大地提升了。你可以让 Claude Code 在几分钟之内生成一个完整的功能模块,改动几十个文件,这在以前可能是一个工程师好几天的工作量。

速度提升当然是好事,但是这里有个隐患。以前我们写得慢,但是每一行代码基本上都是自己亲手敲出来的,你对自己写的代码,多少是有一种"肌肉记忆"式的理解的。但是当 AI 帮你快速生成大量代码之后,这种细粒度的理解就被稀释了。你可能只是大概扫一眼,觉得逻辑没问题,就提交了。而安全漏洞往往就藏在这些"扫一眼觉得没问题"的细节里,比如一个 SQL 查询语句里少做了一次参数化处理,比如一个跨站脚本攻击的输入校验少写了一个过滤条件,比如一个鉴权逻辑里有一个边界情况没有覆盖到。这些问题,在传统的人工代码审查里,本来就容易被漏掉,现在生成代码的速度变快了,审查的工作量却没有相应地增加,这就形成了一个安全风险的"剪刀差"。

所以 Anthropic 提出的解法,听起来其实很自然,那就是用 AI 来追平这个剪刀差。既然 AI 能帮你快速写代码,那也应该用 AI 来帮你快速审查代码的安全性,这样两边的速度才能匹配起来,而不是开发速度一路狂奔、安全审查却还停留在原来的人工节奏上。

这篇文章里介绍了两个具体的产品功能,我们一个一个来讲。

第一个功能,叫 security-review 命令,这是一个在终端里直接调用的斜杠命令。它的使用场景是这样的,你在本地写代码,写完一个功能,准备提交之前,你可以在 Claude Code 里直接敲一个命令,让 Claude 去扫描你当前的代码库,找出潜在的安全隐患,并且给出详细的解释。

这里有个细节值得展开讲讲,这个命令背后用的是一个专门针对安全审查场景设计的提示词,也就是 prompt。我们平时跟 Claude 聊天用的是通用的对话模式,但是做安全审查这件事,如果你只是泛泛地问"这段代码有没有问题",得到的回答可能会比较表面。而 Anthropic 在设计这个命令的时候,专门针对几类最常见的漏洞模式做了优化,文章里列出来的包括五大类。

第一类是 SQL 注入风险,这是个非常经典的漏洞类型,简单说就是攻击者通过在输入框里输入精心构造的字符串,让数据库执行了本不应该执行的查询命令,比如直接绕过登录验证,或者把整个数据库的数据都导出来。

第二类是跨站脚本攻击,英文简称 XSS,这种漏洞是攻击者在网页里注入恶意的脚本代码,当其他用户访问这个页面的时候,恶意脚本就会在他们的浏览器里执行,可能导致用户的登录凭证被窃取。

第三类是身份验证和授权方面的缺陷,这个范围其实很广,包括登录逻辑写得不严谨、权限校验有遗漏,比如一个普通用户通过修改请求参数,本来不该看到别人的数据,结果却看到了,这种叫越权访问,是非常常见也非常隐蔽的一类问题。

第四类是不安全的数据处理,比如敏感信息没有加密就存到数据库里,或者日志里把用户的密码、身份证号这些敏感字段直接打印出来了。

第五类是依赖项的漏洞,也就是你的项目引用的第三方库本身存在已知的安全问题,如果你没有及时更新版本,这个漏洞就会被继承到你自己的项目里。

这五类问题,基本上覆盖了应用安全里最常见、出现频率最高的那些坑。而 Claude 在执行这个命令的时候,会针对性地去检查这几个方向,而不是漫无目的地通读代码。这就好比你请了一个有经验的安全工程师,他脑子里有一份核对清单,知道该重点看哪些地方,而不是像一个新手一样把每一行代码都平等地看一遍,效率和准确率自然会高很多。

而且这个命令用完之后,如果发现了问题,你还可以直接让 Claude 把这些问题修复掉。也就是说,这不是一个"只报告问题、不解决问题"的工具,它是闭环的。你可以在写代码的过程中,随时插入这样一次安全检查,发现问题立刻修,这就把安全审查嵌入到了开发者的内循环里。这个"内循环"这个词,文章里也专门提到了,意思是说,这个检查不是等代码都写完、提交了、进入了 CI 流程之后才做,而是在你自己电脑上写代码的当下就能做。这个时间点非常关键,因为修复一个漏洞的成本,是随着它被发现的时间点往后推移而指数级增长的。你刚写完这段代码的时候,上下文还在你脑子里,改起来又快又准。等代码进了测试环境,甚至已经上线了,这时候再发现问题,你得重新理解这段代码的上下文,可能还要协调多个团队,成本完全不是一个量级的。

好,这是第一个功能,本地终端的即时安全检查。接下来我们讲第二个功能,这个功能的覆盖范围更广,叫做 GitHub Actions 集成。

如果你不熟悉 GitHub Actions,我简单解释一下,这是 GitHub 平台提供的一种自动化工作流机制,你可以设定一些触发条件,比如"每次有人提交一个 Pull Request",然后自动执行一系列预先设定好的任务,比如跑测试、检查代码格式、部署到测试环境等等。Pull Request,简称 PR,就是开发者把自己改的代码提交给团队,请求合并到主分支的那个操作,在国内的语境里,你可以理解成提交一次代码变更申请,等着同事或者自动化工具来审查。

Anthropic 这次推出的功能,就是把 Claude Code 的安全审查能力,做成了一个可以挂载在 GitHub Actions 上的自动化动作。一旦配置好,这个动作会做四件事。

第一,它会在每一次新的 PR 被创建的时候自动触发,你不需要手动去点什么按钮,系统自己就会去跑这次检查。第二,它会审查这次代码变更里涉及的安全漏洞,注意这里强调的是"这次变更",也就是说它不是每次都把整个代码库从头到尾扫一遍,而是聚焦在这次 PR 改动的那部分代码上,这样既能保证审查的针对性,也能控制审查所需要的时间和资源。第三,它支持自定义规则,可以用来过滤掉一些误报,或者过滤掉团队已经知道、暂时不打算处理的已知问题。这一点其实非常实用,因为任何自动化检查工具,如果完全不能调整、不能定制,很容易在实际使用中产生大量的噪音,团队的工程师看到一堆误报之后,慢慢就会对这个工具失去信任,甚至直接忽略掉它的提醒。所以能够自定义规则、排除已知的误报,是让这个工具能在团队里长期、可持续使用下去的关键设计。第四,它会把发现的问题,以行内评论的形式直接贴在 PR 页面上,而且不只是指出问题在哪,还会给出具体的修复建议。

这四点加起来,其实就构成了一个相对完整的自动化安全把关流程。我们可以想象一下没有这个工具之前,团队的安全审查通常是怎么做的。很多团队的做法是,定期,比如每个月或者每个季度,请专门的安全团队或者外部顾问,对代码库做一次集中的安全审计。这种做法的问题在于,审计的周期太长,等审计结果出来,可能这段时间里已经有几十甚至上百个 PR 被合并、上线了,如果审计发现了问题,这时候去修复,影响的范围已经很大了。还有一些团队依赖工程师在 Code Review,也就是代码评审环节里,凭借个人经验去发现安全问题,但是这非常依赖评审者本人的安全知识储备,如果团队里没有专门的安全专家,很多隐蔽的漏洞类型,比如刚才提到的 SSRF,普通的业务工程师可能压根没听说过,自然也不会去重点检查。

而有了这个 GitHub Action 之后,相当于给团队的每一次代码提交,都配了一个一直在线、不知疲倦、还掌握着安全知识的审查员。这个审查员不会因为今天加班累了就漏看一个细节,也不会因为这周事情多就把审查往后拖一拖。它的反馈速度也很快,基本上是 PR 一提交,很快就能看到评论。这就形成了一种"一致性的安全审查流程",文章里原文用的词是 consistent,我把它翻译成"一致性",意思就是不管是谁提交的代码,不管这次提交是大改动还是小改动,都会经过同样标准的安全把关,不会出现"这次没人注意到就漏过去了"的情况。

讲到这里,我想多花一点时间聊聊这个功能背后体现出的一种思路转变,这也是我个人觉得这篇文章最值得琢磨的地方。

传统意义上,我们做软件工程的"质量门",通常分为好几层,有单元测试,有集成测试,有代码风格检查,有人工的代码评审,安全审计往往是单独拎出来的一层,而且经常是比较靠后、比较重量级的一层,需要专门的安全团队介入,流程也比较慢。但是 Anthropic 这次的做法,是把安全审查直接做成了像跑测试、像跑代码风格检查一样的、轻量级的、可以嵌入到日常开发流程里的一个环节。它把安全审查这件事的门槛,从"需要专门团队才能做"降低到了"任何一个普通工程师在写代码的当下,或者提交 PR 的时候,都能自动获得"的程度。

这个转变背后的假设是什么呢,就是 AI 模型本身已经具备了足够好的安全知识储备,可以在很大程度上替代专门的安全专家,去完成第一轮、最基础的安全审查工作。当然,这不代表 AI 能完全替代专业的安全团队,文章里也没有说要取代谁,它更像是在说,把这套能力当成一个"前置过滤器",把那些常见、容易识别的安全问题,在最早的阶段就拦截下来,这样专业的安全团队就可以把精力,投入到那些更复杂、更需要人类专业判断的高级威胁分析上,而不是把时间都耗费在反复检查 SQL 注入、XSS 这些已经有成熟检测方法的基础问题上。

接下来文章里给出了一个挺有意思的部分,就是 Anthropic 自己内部使用这套工具的真实案例。我觉得这部分特别值得展开讲讲,因为案例往往比抽象的功能描述更能让人理解这个工具到底有多大的实际价值。

文章说,Anthropic 团队自己也在用这两个功能,来保护他们自己往生产环境推送的代码,包括 Claude Code 这个产品本身的代码。换句话说,这是一个"自己造的工具,自己先用起来"的典型案例,这种自我验证的方式,我觉得是挺有说服力的,因为如果连开发者自己都不信任、不愿意用自己造的工具,那这个工具的可信度自然会大打折扣。

文章里举了两个具体的例子。

第一个例子,是上线前一周,Anthropic 内部团队为一个内部工具构建了一个新功能,这个功能依赖于启动一个本地的 HTTP 服务器,这个服务器原本的设计意图是只接受本地的连接请求。听起来好像没什么问题,本地服务嘛,应该挺安全的。但是这个 GitHub Action 识别出了一个远程代码执行的漏洞,英文简称 RCE,这是安全领域里最严重的一类漏洞之一,因为一旦被利用,攻击者就可以在目标机器上执行任意代码,危害极大。而这个漏洞的利用方式,是通过一种叫 DNS 重绑定的技术。

我稍微解释一下 DNS 重绑定是什么意思,方便大家理解这个案例的严重性。我们平时访问一个网站,浏览器会先去查询这个域名对应的 IP 地址,这个查询过程叫 DNS 解析。正常情况下,一个域名解析出来的 IP 地址应该是固定的,或者至少在短时间内是稳定的。但是攻击者可以控制一个域名的 DNS 服务器,先让这个域名解析到一个正常的、远程的 IP 地址,等用户的浏览器通过了一些安全校验之后,攻击者迅速把这个域名重新解析到本机地址,也就是俗称的 localhost,这样一来,原本设计成"只接受本地连接"的服务,就会被骗,以为这次请求确实来自本地,实际上请求的发起方是远程的攻击者,通过这种"调包"的手法,绕过了本地连接的限制。这是一种相当经典、也相当隐蔽的攻击手法,如果不是专门做过安全研究,普通的业务工程师确实很容易在设计本地服务的时候忽略这个风险点,毕竟直觉上会觉得"反正只有本地能连,应该没事"。

而这个漏洞,在文章描述里,是在 PR 被合并之前就被这个 GitHub Action 发现并修复的。这个时间点非常关键,我们前面也讲过,越早发现问题,修复成本越低,如果这个漏洞是在代码合并之后,甚至是产品已经上线之后才被外部安全研究人员发现,那不仅修复的工程成本更高,可能还会涉及到公开的安全披露流程,对公司声誉也会有影响。这次能在 PR 阶段就拦下来,可以说是把损失降到了最低。

第二个例子,是一名工程师构建了一个代理系统,proxy system,这个系统的设计目的是为了安全地管理内部的凭证信息,也就是各种密钥、令牌之类的敏感信息。结果这个 GitHub Action 自动标记出来,这个代理系统存在 SSRF 漏洞的风险。SSRF 的全称是服务器端请求伪造,简单理解就是攻击者诱导服务器去发起一个本不该发起的网络请求,通常目标是访问一些内部网络资源,比如云服务商的元数据接口,这种接口里往往存放着非常敏感的身份凭证信息。如果一个代理系统本身的职责就是管理凭证,结果这个代理系统自己又存在 SSRF 漏洞,这就形成了一种很讽刺但也很现实的局面,本来是用来保护凭证安全的工具,结果反而成了凭证泄露的突破口。好在团队及时修复了这个问题。

这两个案例放在一起看,我觉得有一个共同点很值得注意,那就是这两个漏洞都不是那种特别"低级"的错误,不是少写了一个分号、变量名拼错了之类的问题。这两个漏洞都涉及到相对进阶的安全知识,DNS 重绑定和 SSRF,都是需要对网络协议、攻击链路有一定理解才能识别出来的问题类型。这恰恰说明了,这套工具确实能够在普通工程师的知识盲区里,补上一道防线,而不只是检查一些显而易见的语法错误。

第二条路径,是针对团队和企业的,如果你想用 GitHub Action 这个功能,需要按照官方文档的步骤去做安装和配置,这通常涉及到在你的 GitHub 仓库里添加一个工作流配置文件,配置触发条件、权限范围等等。

那么,如果你现在打算把这套东西用到自己的实际工作里,我有几点个人的建议,这些是我自己的看法,跟原文不完全是一回事,大家可以参考。

第一点,我建议先从本地的 security-review 命令开始用起来,而不是一上来就铺开整个团队的自动化流程。原因很简单,任何新工具引入团队流程之前,最好先自己摸清楚它的边界在哪里,它对你具体的代码库,擅长发现哪类问题,又有哪些盲区。你自己先用几周,积累一些经验,知道这个工具的"性格",再去决定要不要把它接入到团队的 CI 流程里,这样会更稳妥。

第二点,关于团队层面接入 GitHub Action,我觉得有个很容易被忽视的坑,就是误报问题。任何自动化安全扫描工具,只要规则设计得足够保守,一定会有误报,这是没办法完全避免的。如果一开始接入的时候没有花时间去调整过滤规则,团队成员很快就会被一堆"看起来吓人但其实没什么用"的提醒搞得疲劳,进而对这个工具产生抵触情绪,甚至开始习惯性地忽略它的所有提醒,这样一来,真正重要的提醒反而也会被一起忽略掉,这就是所谓的"警报疲劳"问题,在安全运营领域是个很经典的现象。所以我建议,接入初期一定要安排专人,花一段时间去观察这个工具实际给出的反馈,把那些反复出现的误报类型,通过文章里提到的自定义规则功能筛出去,把信噪比调整到一个团队成员愿意认真看的水平,这个磨合过程急不来,但是值得投入。

第三点,我想强调一下,这个工具不应该被当成安全工作的终点,而应该被当成一个"前置过滤层"。也就是说,即便你把这套自动化审查接入了团队流程,也不代表你可以完全不需要安全专业知识了,更不代表你可以裁掉安全团队或者不再做专项的安全审计。一个比较合理的心态是,把这套自动化工具当成第一道关卡,专门处理那些常见、有成熟检测方法的问题类型,比如 SQL 注入、XSS 这种,而对于一些更复杂的、涉及业务逻辑的安全问题,比如一个复杂的权限模型里某个边界条件被绕过、或者多个系统组合起来产生的攻击链路,这些往往还是需要有经验的安全专家,结合对你具体业务的理解,才能识别出来。自动化工具和人的专业判断,应该是互补关系,不是替代关系。

第四点,我想从一个稍微宏观一点的角度,聊聊这件事背后反映出的趋势。我们现在看到,AI 写代码的能力在快速提升,这件事大家应该都有体感了。但是 AI 审查代码安全性的能力,其实是和写代码的能力,几乎是同一套底层模型能力支撑起来的。这意味着,未来开发流程里,"写代码的 AI"和"审查代码的 AI",很可能会变成标准搭配,就像现在大家写代码默认会配一个代码风格检查工具一样自然。如果你现在还没有开始在自己的工作流里引入类似的安全检查习惯,我觉得现在是一个比较好的时间点去开始尝试,因为越早把这种习惯建立起来,等到团队规模扩大、代码量级上升之后,你会发现这种自动化的安全把关机制,带来的收益是会持续累积的,而不像是临时补救措施那样,一次性解决了问题就完事了。

我个人对这件事还有一点感触,想多说两句。这种"用 AI 来审查 AI 自己生成的代码"的模式,听起来有点像是在自我监督,初听可能会觉得有点奇怪,既然漏洞可能就是 AI 自己写代码的时候引入的,那再让 AI 自己去检查,会不会有种"自己骗自己"的风险。但是仔细想想,这个逻辑其实是成立的,因为生成代码和审查代码,本质上是两种不同的认知任务,即便是同一个底层模型,在"生成"模式下和在"审查"模式下,关注的焦点和思考路径是不一样的。生成代码的时候,模型的主要目标是实现功能,把业务逻辑跑通,这个过程里,注意力天然会更多地分配在功能正确性上,而不是安全边界上。但是当你专门给它一个"审查安全性"的任务,用一个专门设计的提示词去引导它,让它换一个视角,专门去找漏洞模式,这时候模型关注的重点就完全不一样了,它会更像一个专业的安全审查员那样去思考。这就跟人类写代码和审查代码也是两种不同的脑力活动是一个道理,一个人自己审查自己刚写的代码,确实容易有思维盲区,因为脑子里还停留在刚才"怎么实现功能"的思路里,这也是为什么传统软件工程里强调要有"第二个人"来做代码评审。AI 这里相当于通过切换提示词和任务设定,模拟出了"换一个视角"的效果,所以这个思路在逻辑上是讲得通的,只是大家在心理上需要适应一下,接受让同一类模型,在不同的任务设定下,扮演两种不同的角色。

当然,我也要提醒一句,这种自动化安全审查工具,终究是基于模型已有的知识和训练数据来工作的,它能够识别的,是那些已经被广泛认知、有成熟攻击模式的漏洞类型。对于一些非常新颗、刚刚被发现的攻击手法,或者是高度依赖特定业务上下文才能识别出来的复杂漏洞,自动化工具的覆盖能力肯定是有限的。所以我建议大家保持一个理性的预期,把它当成一个能大幅提高基础安全审查覆盖率和及时性的好帮手,而不是一个"装上就能一劳永逸解决所有安全问题"的万能盾牌。

好,我们来总结一下今天聊的这篇文章。Anthropic 在去年八月发布了两个新功能,一个是终端里的 security-review 命令,方便开发者在写代码的当下随时做安全自检,并且可以直接让 Claude 帮忙修复发现的问题。另一个是接入 GitHub Actions 的自动化动作,可以在每次提交 PR 的时候自动做安全审查,把发现的问题以行内评论的形式直接反馈给开发者,还支持自定义规则来过滤误报。Anthropic 自己内部就在用这套工具保护他们自己的产品代码,文章里举了两个真实案例,一个是 DNS 重绑定导致的远程代码执行漏洞,另一个是凭证管理代理系统里的 SSRF 漏洞,两个都是在代码合并之前就被自动拦截下来的。

我个人对这件事的总体看法是,这是一个挺务实、挺接地气的产品方向,它没有去描绘什么宏大的愿景,而是踏踏实实地解决了一个具体的工程痛点,就是开发速度和安全审查速度之间的落差。如果你或者你的团队还没有尝试过类似的工具,我觉得不妨找一个小项目先试用一下这个命令,亲自感受一下它能发现哪些问题,再决定要不要进一步接入到团队的日常流程里。
