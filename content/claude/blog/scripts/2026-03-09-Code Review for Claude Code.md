## Shownotes

## Script

大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是 Bringing Code Review to Claude Code,我把它翻译过来,大概就是给 Claude Code 带来代码评审能力。这篇文章发表于二零二六年三月九日,也就是说距离我们现在录这期节目,差不多过去了三个多月。三个多月在 AI 这个行业里,说长不长,说短也不短,基本上还在一个可以拿来直接参考的时间窗口内,所以今天讲的这些内容,大家可以放心地当作当下还在生效的产品信息来听,但具体的定价、覆盖范围这些细节,如果你真的要上手用,还是建议去官方文档确认一下最新状态。

那我们先说说这篇文章到底想解决什么问题。其实开头那句话就把背景交代得很清楚了,Anthropic 自己内部统计,过去一年里,每个工程师的代码产出量增长了百分之二百,也就是涨了两倍。你可以想象一下这个场景,以前一个工程师一周可能提交十个 PR,也就是 Pull Request,中文里我们一般叫拉取请求,就是工程师写完代码,提交给团队评审、准备合并到主分支的那个东西。现在因为有了 Claude Code 这类编程助手,同一个工程师一周可能要提交二十个甚至三十个 PR。代码产出的速度被 AI 大幅拉高了,但审查代码的人,还是那几个人,人的精力和时间并没有跟着涨两倍。

这就导致了一个很现实的矛盾,文章里直接点名说,代码评审已经变成了瓶颈,客户也每周都在跟他们反馈同样的问题。工程师们忙得团团转,很多 PR 收到的不是认真细读,而是走马观花地扫一眼。我们都知道,代码评审这件事,如果只是扫一眼,基本等于没审。很多隐藏很深的 Bug,比如一个边界条件没处理好,或者一个类型不匹配导致的数据悄悄损坏,这些东西在快速浏览的时候根本看不出来,只有真正去推演这段代码在各种输入下会发生什么,才能发现问题。可现实是,当评审者自己也在用 AI 拼命产出代码的时候,他哪有时间去给别人的代码做这种深度推演呢。所以你可以理解,这其实是 AI 提效带来的一个连锁反应,生产端提速了,但质量把控端没有跟上,于是就出现了一个新的缺口,这个缺口正是 Code Review 这个产品想要填补的。

那 Anthropic 给出的方案是什么呢,文章里说得很直白,这是他们自己几乎用在每一个 PR 上的评审系统,现在拿出来,以研究预览的形式开放给 Team 和 Enterprise 计划的客户。注意这里有个细节,它不是说要取代人工评审,文章里专门强调了一句,它不会去批准 PR,合并代码这个最终决定权还是留给人类。它的角色定位很清楚,是在人工评审之前,先做一遍深度扫描,把那些容易被忽略、容易被看漏的问题先挖出来,这样真正的人工评审者就可以把精力放在那些机器筛出来的高优先级问题上,而不是从头到尾去啃一份几百行的 diff。

我们再来看看这套系统具体是怎么运作的,这也是我觉得整篇文章里最值得深挖的部分。当一个 PR 被打开的时候,Code Review 不是简单地丢给一个大模型去通读一遍代码就完事了,而是会派出一整个智能体团队,英文叫 agent team。这些智能体会并行工作,各自去找 Bug。你可以把这个过程想象成请了好几位风格完全不同的审稿人,同时拿到同一份稿子,有的人专门盯逻辑漏洞,有的人专门盯安全隐患,有的人专门盯边界情况,大家各自看完之后,再把各自的发现拿出来对一遍。

找到疑似的 Bug 之后,系统并不会马上就把它扔给你,而是会进入一个验证环节,英文叫 verify,目的是过滤掉误报,也就是我们常说的 false positive。这一步其实特别关键,因为我们都知道,如果一个自动化工具老是报假警报,人很快就会对它失去信任,然后干脆把所有报警都当成噪音直接忽略,这样这个工具就彻底失去价值了。所以 Anthropic 在设计的时候,特地加了这道验证的工序,让另一组或者说同一组智能体反过来去质疑、去复核之前找到的问题,确认它是不是真的会导致问题,而不是似是而非的猜测。

验证完之后,还有一步是按严重程度排序,英文是 rank bugs by severity。这一步也很贴心,因为评审者最怕的不是没有发现,而是发现一大堆,但分不清哪个是致命的、哪个是无关紧要的小毛病。如果系统一股脑把二十条意见全部堆给你,效果跟没有筛选基本没区别,人还是要花大量时间去自己甄别优先级。Code Review 把这一步也自动化了,最终呈现给你的,是一条高信号的总览评论,英文叫 high-signal overview comment,加上针对具体代码行的内嵌评论,英文是 in-line comments。这种呈现方式我觉得设计得挺聪明的,既给你一个宏观的总结,让你三十秒之内就知道这个 PR 整体风险如何,又给你具体定位到行,方便你点进去直接看上下文,不需要自己再去满世界找问题出在哪一行。

文章里还提到一个细节,我觉得也很实用,就是评审的深度是会跟着 PR 的规模和复杂度动态调整的。如果你提交的是一个改动量很大、逻辑很复杂的 PR,系统会派出更多的智能体,做更深的读取和推演。如果你提交的只是一个很小的、改动很简单的 PR,它就会用一个轻量级的流程快速过一遍,不会小题大做。这其实也是在控制成本和效率的平衡,毕竟不是每一个 PR 都值得动用一整个智能体团队去深挖。根据他们的测试,平均一次评审大概要花二十分钟左右。

讲到这里,可能有听众会问,这套东西到底有没有用,光听描述感觉挺像那种听起来很美但实际效果一般的工具。所以接下来我们看看 Anthropic 自己内部跑了几个月之后给出的数据,这部分我觉得是这篇文章最有说服力的地方。

第一个数据是,在引入 Code Review 之前,只有百分之十六的 PR 会收到实质性的评审评论,英文是 substantive review comments,也就是说剩下八成多的 PR,基本上是被人扫一眼就放过去了。引入之后,这个比例涨到了百分之五十四,也就是说现在超过一半的 PR,都能收到有实际价值的评审意见。这个数字的变化我觉得特别能说明问题,它不是说原来工程师们偷懒不认真,而是这反映出原来人工评审的产能确实跟不上代码产出的速度,机器补上了这个缺口之后,真正的审查覆盖率一下子从不到两成涨到了过半。

第二个数据,是按 PR 规模拆分的发现率。对于改动超过一千行的大型 PR,有百分之八十四会被找出问题,平均每个能找到七点五个问题。这个数字听上去挺吓人的,大型 PR 几乎是必然会有问题被揪出来,而且不是一两个,是七八个。换个角度想,这也说明大型改动里隐藏的风险确实更多,毕竟改动越大,涉及的代码路径、边界情况、模块交互就越复杂,人脑很难在有限时间里把所有分支都想清楚。而对于改动小于五十行的小型 PR,发现率降到百分之三十一,平均能找到零点五个问题。这个对比其实也验证了我们的直觉,小改动确实风险更低,但即便是小改动,依然有接近三分之一的概率藏着值得注意的问题,这也提醒我们,千万不要因为改动小就掉以轻心。

第三个数据,也是我觉得最关键的一个,误报率。文章里说,工程师标记为不正确的发现,占比不到百分之一。这个数字非常重要,因为它直接决定了这个工具能不能被工程师真正信任和依赖。如果误报率高,比如百分之二十、百分之三十,那工程师很快就会产生评审疲劳,看到机器的提示就直接划走,根本不会认真核实。但不到百分之一的误报率,意味着机器给出的每一条意见,基本上都值得你认真看一眼,这种信任感是这个工具能不能真正融入工作流程的核心。

文章里还举了两个具体的案例,我觉得特别值得跟大家分享一下,因为光看数字有点抽象,案例能让你更直观地感受到这东西到底厉不厉害。

第一个案例发生在 Anthropic 内部。有一次,有个工程师提交了一个针对生产服务的一行代码改动,看起来非常普通,普通到正常情况下基本上就是快速扫一眼然后批准合并的那种改动。但是 Code Review 把这个改动标记为了严重级别,critical。结果一查,这个改动如果合并上去,会直接破坏这个服务的身份验证功能,也就是 authentication。文章里特别强调了一句话,我觉得说得特别到位,这种失败模式,在读 diff 的时候很容易被忽略掉,但一旦被指出来,又显得非常明显。这其实是代码审查里一个经典的认知陷阱,叫做我们大脑对简单改动的天然信任。一行代码的改动,我们的直觉会告诉自己这肯定没什么风险,所以注意力反而会放松,但恰恰是这种放松,才让真正危险的逻辑漏洞溜了过去。这名工程师后来也承认,如果靠自己,他大概是发现不了这个问题的。这个案例其实给我们提了个醒,代码改动的风险程度,跟改动的行数从来都不是线性关系,有时候一行代码就能捅出大问题。

第二个案例来自一个早期使用客户,叫 TrueNAS,这是一个做存储系统的开源项目。他们当时在做一个 ZFS 加密功能的重构,ZFS 是一种文件系统,在存储行业里挺有名的。Code Review 在审查这个重构 PR 的时候,发现了一个早就存在的、跟这次改动相邻代码里的 Bug,具体来说是一个类型不匹配的问题,会导致加密密钥缓存在每次同步的时候被悄悄清空。这个问题特别有意思,因为它不是这次 PR 引入的新问题,而是早就潜伏在代码里的一个隐患,只是因为这次改动碰巧touch到了那块代码,系统就顺手把这个潜伏的坑也挖出来了。文章里说得很到位,这种东西,人工评审者在扫描整个改动集的时候,通常不会主动去翻查这种问题,因为他的注意力天然会集中在这次改动本身相关的代码上,而不是去审视那些看起来跟这次改动没直接关系的旧代码。这其实揭示了 Code Review 一个挺有价值的副作用,它不光是审查这次的改动有没有问题,有时候还能顺带把周边代码里埋了很久的雷给排掉。

讲完了效果,我们再来说说这东西到底要花多少钱,毕竟这是大家上手之前肯定会关心的问题。文章里说得很坦白,Code Review 走的是深度优先的路线,所以它比那些更轻量级的方案要贵,比如他们现有的 Claude Code GitHub Action,这个开源工具依然保留并且可以继续用。Code Review 是按 token 使用量来计费的,平均每次评审大概在十五到二十五美元之间,具体费用会随着 PR 的规模和复杂程度浮动。这个价位说实话不算便宜,如果你团队一天提交几十个 PR,全部跑深度评审,一个月下来的账单可能还挺可观的。

但是文章也提到,Anthropic 给管理员提供了几种控制开支的手段,我觉得这部分对于真正要落地使用的团队管理者来说特别实用。第一个是月度组织上限,你可以给整个组织设定一个每月评审总花费的封顶线,超过这个数字就不再触发新的评审,这样就不会出现账单失控的情况。第二个是仓库级别的控制,你不需要让公司所有的代码仓库都跑这个评审,可以挑选你认为最重要、最值得深度审查的仓库去启用,比如核心的生产服务代码,而一些边缘的小工具仓库可以先不开。第三个是分析仪表盘,可以追踪有多少 PR 被评审过、采纳率是多少,以及总的评审花费,这样团队负责人就能用数据去判断这个工具到底值不值这个价钱,要不要调整使用范围。

那接下来咱们聊聊实际应用的部分,也就是说,如果你是听这期节目的工程师或者团队负责人,这东西对你意味着什么,你应该怎么去用它,又有哪些坑需要提前知道。

首先一个很现实的问题是,这东西目前只针对 Team 和 Enterprise 计划开放,而且是研究预览阶段,也就是说还在打磨阶段,不是一个完全成熟稳定的正式功能。如果你是个人开发者,或者你的团队用的是更基础的计划,可能暂时还用不上这个功能,得先去看看自己的订阅计划是不是符合条件。

其次,如果你是团队里的管理员角色,启用流程文章里说得挺简单的,大概分三步,先在 Claude Code 的设置里把 Code Review 功能打开,然后安装对应的 GitHub App,这是因为这个评审系统需要接入到你的 GitHub 仓库才能监听 PR 的开启动作,最后再选择你想让它跑评审的仓库范围。考虑到前面说的成本问题,我个人建议,刚开始上手的时候,不要一上来就给所有仓库都打开,可以先挑一两个团队最关心、出问题代价最高的核心仓库试用一段时间,观察一下发现率和采纳率,再决定要不要扩大范围。

对于普通的开发者来说,文章里说一旦管理员启用了这个功能,后面的事情其实就跟你没什么关系了,评审会在你提交新 PR 之后自动运行,不需要你做任何额外的配置。这种自动化、零配置的体验我觉得是这类工具能不能被广泛采用的关键,如果每次提交 PR 都要手动去触发评审,或者要写一堆配置文件,大概率很多人嫌麻烦就不会用了。

那具体使用的时候,有哪些坑是需要注意的呢。第一个我觉得是心态上的坑,千万不要把这个工具当成万能的安全网,然后就完全放松对自己代码的把关。文章里反复强调了,它不会批准 PR,这意味着最终责任还是在人。机器找出来的问题,即便误报率很低,也依然需要人去判断这个问题在当下的业务场景里是不是真的值得修,有没有更紧急的优先级排序问题。把它当成一个非常靠谱的助手,而不是当成一个可以替你做决定的裁判,这个心态上的边界要拿捏清楚。

第二个坑是关于成本控制的,前面说了平均每次评审十五到二十五美元,这个数字看起来不大,但乘以你团队一个月提交的 PR 总数,可能就不是一个小数目了。所以我建议团队在启用之初就先设置好月度上限,而不是等账单出来之后再后悔,这是一个很基础但很容易被忽略的运营细节。

第三个坑,是要警惕评审疲劳的反向问题。虽然文章里说误报率不到百分之一,听起来已经很低了,但如果团队体量很大,PR 数量很多,即便是百分之一的误报,绝对数量也可能不少。所以建议团队内部要建立一个简单的反馈机制,比如发现误报之后及时标记,这样不仅能帮助团队自己更好地判断什么时候该信任机器的判断,某种程度上这种反馈本身,大概也是 Anthropic 用来持续打磨这套系统的重要数据来源。

第四个坑,关于大小 PR 的策略差异。前面数据显示,大型 PR 的问题发现率特别高,百分之八十四,小型 PR 只有百分之三十一。这其实提示我们,如果团队资源有限,需要在评审策略上做取舍,可以优先把这个工具用在那些动辄上千行的大型重构、大型功能改动上,这种地方机器审查的边际价值最高,性价比也最高。而对于那种几行代码的小修小补,虽然也有价值,但相对来说边际收益会低一些,如果预算紧张,可以适当降低这部分的覆盖优先级。

我个人觉得,这篇文章背后反映出的趋势,其实比 Code Review 这一个具体功能更值得我们关注。当 AI 编程助手让代码产出效率提升了两倍,整个软件工程的流程链条里,原本由人力支撑的下游环节,比如评审、测试、部署这些步骤,如果不跟着提效,迟早会变成新的瓶颈。Code Review 本质上是 Anthropic 在用 AI 去解决 AI 自己带来的问题,先用 Claude Code 让大家写代码更快,再用另一套基于 Claude 的智能体系统,去填补因为写代码变快而被拉大的评审缺口。这种自我闭环的产品逻辑,我觉得未来在软件工程的其他环节上,可能还会继续重复出现,比如测试覆盖、文档同步、依赖升级,这些原本依赖人工但又跟不上 AI 提效速度的环节,很可能都会陆续出现类似的智能体团队解决方案。

好,那我们今天就简单总结一下。这篇文章讲的是 Anthropic 给 Claude Code 加上了一套基于多智能体团队的代码评审系统,叫 Code Review,目前以研究预览的形式向 Team 和 Enterprise 客户开放。它的核心逻辑是用一组智能体并行查找 Bug,互相验证过滤误报,再按严重程度排序,最终把高信号的总览和具体的行内评论呈现给人类评审者,但最终是否合并代码,决定权依然在人。从 Anthropic 自己内部的数据来看,这个工具确实把评审覆盖率从一成多提升到了过半,而且误报率控制得相当低。我个人的看法是,这是一个值得团队认真评估、小范围先试用起来的工具,尤其是用在那些大型、高风险的代码改动上,但千万别把它当成替代人工判断的终极方案,人在回路里的角色,短期之内还是不会消失的。好了,今天的节目就到这里,我们下期再聊,再见。
