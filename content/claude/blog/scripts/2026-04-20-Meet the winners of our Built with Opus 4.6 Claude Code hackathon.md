## Script

大家好,欢迎收听 Claude FM。今天我们要聊的这篇文章,标题是 Meet the winners of our Built with Opus 4.6 Claude Code hackathon,翻译过来就是认识一下我们这次使用 Opus 四点六开发的 Claude Code 黑客松的获奖者们。这篇文章发表于二零二六年四月二十日,距离今天差不多正好两个月,所以内容还是相当新鲜的,不存在什么过时的问题。

为什么值得专门拿出一期来讲这件事呢?因为这篇文章不是在讲什么新模型发布的参数表,也不是在讲什么基准测试又刷新了多少分,它讲的是一群活生生的人,用 Claude Code 这个工具,在短短一周时间里,从零做出来的真实产品。而且更有意思的是,这五位获奖者里面,有四位根本不是专业程序员。有律师,有医生,有政府公务员,还有一名电子音乐人。专业软件工程师只占了五分之一。这件事本身就很说明问题。我们今天就来仔细拆解一下,他们到底做了什么,他们是怎么做出来的,以及这件事对我们这些天天写代码、或者天天琢磨怎么用 AI 的听众来说,到底意味着什么。

先说说这场黑客松的背景。Anthropic 之前办过一次叫做 Built with Opus 四点六的虚拟黑客松,邀请社区成员用刚发布的 Opus 四点六模型,结合 Claude Code 这个命令行智能体编程工具,在一周之内做出一个项目。这里插一句,Claude Code 大家应该已经比较熟悉了,它是 Anthropic 推出的一个跑在终端里的编程助手,你可以用自然语言跟它对话,告诉它你想要什么,它帮你写代码、跑测试、改 bug、读文档,它甚至可以自己把一个大任务拆解成很多个小任务,再调用多个子智能体并行去完成,有点像一个项目经理带着一群分工明确的下属同时干活,而不是一个人埋头从头写到尾。

这次黑客松,Anthropic 找了一个叫 Cerebral Valley 的社区合作伙伴,一起筛选出五百名参与者,每个人发五百美元的 API 信用额度,限时一周去构建一个项目。一周时间说长不长,说短不短,刚好卡在一个很微妙的点上,既逼着你做减法、只保留最核心的功能,又给了足够的时间让一个想法落地成一个能跑起来的东西。最后由 Claude 团队内部的评委,从所有提交的项目里挑出六个获奖项目,瓜分总共十万美元的 API 信用额度奖金,这笔钱不是发到手就完事,而是用来支持获奖者继续把项目往后开发。文章里详细介绍了五个获奖项目背后的故事,正好覆盖了住房、教育、医疗、基础设施、音乐这五个完全不搭边的领域。

我觉得这篇文章真正想传达的核心信息,不是某个具体项目有多酷,而是 Claude Code 已经把软件开发这件事的门槛大幅拉低了,低到一个完全不会写代码、甚至从没读过代码的人,只要他对某个领域的具体问题足够了解,就可以在一周之内,把脑子里想了很久的解决方案,变成一个真正能跑的产品。这跟我们过去几年对编程黑客松的想象完全不一样。过去办黑客松,比的是谁的代码写得快、写得巧,谁能在四十八小时里熬夜堆出最多的功能。现在这场黑客松比的反而是谁对一个具体问题的理解最深、谁攥着的那个真实痛点最痛。代码能力,从决定性因素,降级成了一个相对容易补齐的环节。这个变化背后的含义其实挺大的,我们一个一个项目仔细看一下,看完你大概就能感受到这种变化有多具体。

顺便说一句,这也不是 Anthropic 第一次办这种黑客松,文章末尾提到,他们紧接着又宣布了下一场,叫做 Built with Opus 四点七的黑客松,规则基本延续了上一场的格式,五百名参与者、五百美元额度、一周时间。也就是说这种用真实问题换真实产品的模式,已经被验证可行,正在变成一个常态化的活动,而不是一次性的营销噱头。如果你对这类活动感兴趣,完全可以关注后续的报名信息,说不定下一个出现在获奖名单里的就是你。

第一个项目叫 CrossBeam,做的人是迈克·布朗,他的真实职业是人身伤害律师,完完全全不是程序员,平时打的官司跟代码毫无关系。他要解决的问题,是加州的住房许可证危机。文章里有个数据特别扎心,加州的住房许可申请,第一次提交的拒绝率超过百分之九十,平均要拖延六个月,这六个月的拖延给业主带来的额外成本大概是三万美元。迈克有句话说得特别精辟,他说大家都以为加州有住房危机,其实加州没有住房危机,加州有的是许可证危机。你想盖一套新房子,办许可证花的时间,可能比真正动手盖房子还长。这就好比你想开一家小餐馆,装修和买设备只用了两个月,但是办各种证件却拖了一年,真正卡住你的从来不是施工本身,而是文件和流程。

那为什么拒绝率会这么高呢,原因其实很官僚、很琐碎,完全不是什么高深的技术问题,比如某个地方少签了一个名字,引用的建筑规范编号写错了一位数字,表格里某一栏忘了填。这些问题单独看一个一个都很小,随便一个有经验的审图员一眼就能看出来,但是当几十万份申请同时堆在系统里的时候,这些零散的小错误叠加起来,就变成了一个巨大的系统性瓶颈,把整个城市的住房供应都给堵住了。

CrossBeam 是怎么解决这个问题的呢。建筑商把自己的图纸和市政府退回来的整改信,直接拖拽进这个工具里,后台会启动多个并行的子智能体去解析这些文档,先建立一个空间索引,也就是把图纸上的每一个具体位置,和整改信里提到的每一条要求,一一对应起来,搞清楚这条整改意见到底是冲着图纸上的哪个角落、哪根梁、哪扇窗户说的。对应清楚之后,再给每一条具体的整改意见,单独分配一个专门的智能体去钻研处理。大概二十分钟之后,建筑商手里就能拿到一份非常精确的行动计划,清清楚楚地告诉他该怎么改,改完就能顺利通过审批,不用再像以前那样,自己对着几十页的整改信和图纸大眼瞪小眼,猜审批人到底是什么意思。

这个工具还有另外一面,是专门给政府部门用的。市政府那边可以用 CrossBeam 批量处理收到的海量许可证申请,自动生成整改信的草稿,大幅减轻审批人员的工作量。你可以把整个系统想象成一座桥,桥的一端是申请许可证的建筑商,另一端是负责审批的市政府工作人员。过去这两端之间全靠人工传递纸面文件,中间隔着邮寄、排队、人工核对,效率极低,稍微有点积压就会拖上几个月。现在 CrossBeam 把桥的两端都装上了智能体,信息在中间高速流转,原来需要几个月才能走完的一个来回,现在压缩到了二十分钟。

南加州有一个叫布埃纳帕克的城市,到二零二九年需要审批超过八千九百套住房,但二零二四年实际只批了大概一百二十套,这个缺口大到几乎离谱,按这个速度根本不可能完成目标。这个城市现在正考虑采用 CrossBeam,来同时加速申请人和审批人这两端的效率。迈克自己说,如果能同时解决加州住房许可证危机的这两端问题,也许就真的能解决加州的住房危机。

最让我意外的一点是,迈克说他构建 CrossBeam 的整个工作方式,就是不断地给 Claude Code 提需求,描述他想要什么效果,然后让 Claude 自己去写测试、自己去验证。他原话是说,赢了这个比赛对他来说非常离谱,因为他没写过一行代码,甚至没读过一行代码。这句话我听完是有点震撼的,这意味着他完全是站在产品逻辑和业务规则的层面去驱动整个开发过程,代码层面的细节,从头到尾都交给了 Claude 去处理。如果换作是你,手里攥着一个类似的、跟法律或者行政流程相关的痛点,你大概也能想象,自己完全不需要先去学一门编程语言,而是直接把问题描述清楚,让 AI 把实现细节填满。

第二个项目叫 Elisa,这个项目的来源特别温暖。开发者乔恩·麦克比本人是一名专业软件工程师,他十二岁的女儿要参加七年级的科学展览,项目内容需要给微控制器编程,也就是那种嵌在硬件设备里、负责控制传感器和电机的小芯片。乔恩希望女儿能用上他自己每天工作都在用的工具,也就是 Claude Code,但问题是,一个黑底白字的终端命令行界面,对一个初中生来说实在是太陌生太难上手了,所以他决定干脆自己做一个专门给孩子用的版本出来。

Elisa 是一个基于积木块的可视化集成开发环境,你可以理解成是少儿编程工具 Scratch 的进阶版,表面上看是拖拖拽拽搭积木,但是积木背后真正驱动代码生成的是 AI。用户在界面上把一些基础模块拼在一起,这些模块分别代表目标、需求、智能体、技能、规则、入口、部署这些抽象概念,拼好之后,系统里有一个元规划器,会把这套可视化的规格说明,自动拆解成一张完整的任务图,再交给具体的智能体去逐项执行,真正的代码是在用户完全看不到的背后自动生成出来的。它还内置了一个教学引擎,会针对当前正在用到的编程概念,实时给出适合孩子理解程度的解释,这样一来,每一次搭积木的过程,同时也是一次编程概念的学习过程,孩子在不知不觉中就把抽象概念学会了。乔恩的女儿就是用这个工具,把代码刷写进微控制器里,完成了她的科学展览项目,从头到尾她没有写过一行代码,但她确实理解了自己的项目是怎么运转的。

乔恩用 Claude Code,在三十个小时里就把 Elisa 整个做出来了,一共提交了七十六次代码,写了超过三万九千行代码,还包含一千五百多个测试用例。他说自己懂系统架构,懂硬件集成,懂怎么定义和测试软件,这些都是他二十多年职业生涯积累下来的硬功夫,但是 Claude Code 帮他把这些散落的知识,在短短六天之内,变成了一个真正能交付给别人用的产品。这句话其实挺关键的,他不是从零学会了什么新技能,而是把已经会的东西,通过 Claude Code 这个放大器,迅速转化成了产品形态,这跟前面迈克·布朗那种完全靠领域知识驱动的方式还不太一样,是两种不同起点、却殊途同归的用法。

Elisa 拿到了第二名,获得了三万美元的 API 信用额度奖金,乔恩打算用这笔钱去推动 Elisa 真正走进课堂,目前已经有教育工作者主动联系他,希望在课堂上使用这个工具。乔恩还提出了一个挺有意思的判断,他认为软件开发者很快会完全停止手写源代码,转而去写定义清晰的测试和规格说明,剩下具体怎么实现的事情,统统交给 AI 去处理。Elisa 这个项目,其实就是把这个判断,用一个适合孩子理解的可视化界面包装了出来,某种意义上是在给下一代提前演练这种新的工作方式。他给这个项目起名叫 Elisa,用的就是他女儿的名字,他说,因为她正是这个项目存在的意义。

第三个项目叫 PostVisit,做的人是米哈乌·内多希茨科,他是常驻布鲁塞尔的一名心脏科医生,同时过去二十年里一直在业余时间开发医疗软件,医生和程序员这两个身份在他身上并行了二十年。他之前已经做过一个产品叫 Previsit dot AI,是一个 AI 患者预问诊系统,已经在比利时、希腊、波兰三个国家落地使用,帮助患者在见到医生之前先把基本情况整理清楚。但他真正想做、已经在脑子里构想了两年的,是这个产品的另一半,也就是病人离开诊室之后会发生什么。他说自己做过几千例心脏导管手术,手术台上的事情他熟悉得不能再熟悉,但真正的挣扎,往往是在病人走出诊室那一刻才刚刚开始,病人回到家,看着一堆专业术语的诊断报告,根本不知道接下来该怎么办。

PostVisit 是一套工具组合,用通俗易懂的语言给病人解释诊断结果是什么意思,分析就诊记录和 AI 转录的医患对话内容,并且从科学文献和病人完整的健康记录里,找出真正相关的临床证据,整个分析过程始终有医生在旁边监督把关,不是放任 AI 自己下结论。这样一来,病人能更清楚地了解自己的病情到底意味着什么、接下来该注意什么,医生也能在两次复诊之间的空白期里,看到病人的真实状况变化,而不是每次复诊都从零开始重新了解情况。整个系统在设计上特别强调隐私、安全和临床最佳实践,毕竟处理的是最敏感的健康数据。

米哈乌做这个项目的方式也特别有意思,他是在开车从布鲁塞尔一路开到旧金山参加黑客松活动的途中完成的,他自己说,在路上开车的时候,反而是他灵感最活跃的时刻,公路上大段大段不被打扰的时间,刚好用来理清思路。一周时间,几千英里的车程之后,他把构想了整整两年的产品真正做出来了。他有句话我觉得说得特别好,他说医学本来就是建立在证据之上的一门学科,现在通过把健康记录、科学证据和就诊数据三者结合起来,病人对自己就诊之后会发生什么,第一次拥有了完整的掌控权和理解,而不是被动地等着下一次复诊才能知道结果。

第四个项目叫 TARA,背景发生在乌干达。基耶雲·卡兹贝韦当时在乌干达的工程与交通部工作,他每天面对的问题是,乌干达的道路基础设施需求,远远超过财政预算能覆盖的范围,可是传统的可行性研究在这种局面下根本帮不上太多忙,因为做一次正经的可行性研究要花一百万到四百万美元,耗时九到十四个月,而且研究的重点往往只放在经济预测上,根本没有把这条路真正要服务的那些沿线社区的实际情况考虑进去。你可以想象一下,一个预算紧张的地方政府,光是为了搞清楚该不该修一条路,就要先掏出几百万美元去请咨询公司,等上一年,这个门槛本身就足够劝退绝大多数本来更需要这条路的偏远地区。

TARA 要做的事情,是把行车记录仪一路拍摄下来的道路视频,直接转化成一份完整的投资评估报告。它利用 Opus 四点六的视觉理解能力,逐帧分析路面状况,识别路面的破损模式,比如哪里有坑洼、哪里路面开裂,同时还能识别路边的活动情况,包括行人、骑自行车的人,还有路边摆摊的集市。系统会把整条道路按状况自动分段,给每一段自动估算所需的干预成本,然后生成一份完整的经济评估报告,里面包含净现值、现金流预测、敏感性分析这些专业的财务和工程内容。它还会额外生成一个公平性评分,专门评估这条路真正惠及的是谁,这个评分会结合道路周边设施的分布情况,以及行车过程中识别出的那些高关注区域,确保修路的钱真的花在了最需要的地方,而不是只服务于经济数据上最好看的路段。

在黑客松期间,基耶雲实际上传了乌干达基拉到马图加公路的真实行车记录仪视频,这条路当时正在施工中。他说,只需要点一次按钮,就能生成一份完整的 PDF 报告,里面包括路况评估、经济分析、公平性发现、敏感性解读,所有内容都汇总在一份文档里。他说这个流程过去要花上几周时间,TARA 现在只需要五个小时就能跑完。

这个项目我觉得特别值得多展开讲一下,因为它体现了 Opus 四点六视觉能力在一个非常具体的场景里的落地。你可以想象一下,如果你是某个发展中国家地方政府的工作人员,过去你要评估一条路该不该修、该怎么修,只能依赖专业的咨询公司,花大价钱、等上大半年,等报告出来的时候,可能资金窗口都已经错过了。现在你只需要找个司机,把行车记录仪装上车,把这条路从头到尾开一遍,剩下的事情统统交给系统去做。这种转变,本质上是把过去只有大机构、大政府才能负担得起的专业分析能力,下放给了资源极其有限的地方政府和社区组织,这对发展中国家的基础设施决策来说,意义可能比表面上看起来的还要大得多。如果这类工具进一步普及,未来甚至有可能反过来推动可行性研究这个行业本身的定价和效率发生变化。

第五个项目叫 Conductr,做的人是阿塞普·巴格贾·普里安达纳,一名电子音乐人,他完全是从音乐创作的需求出发,而不是从技术炫技的角度出发去做这个项目的。他做的是一个能把 Claude 变成现场虚拟乐队伙伴的工具,本质上是一个跑在浏览器里的 MIDI 乐器,这里说的 MIDI 是一种电子乐器之间用来传输演奏信息的标准协议。你在 MIDI 控制器上弹和弦,它会实时聆听你的演奏,分析你弹的内容,然后实时生成四条音轨,分别是鼓、贝斯、旋律和和声,跟着你的演奏同步演奏出来。你甚至可以直接打字告诉它,让节奏更放克一点,或者把情绪往一个高潮推一推,整个编曲会立刻跟着你的指令实时变化,就像旁边真的站着一个反应飞快的乐队成员。

阿塞普说,这个项目最大的技术难点,是怎么让音乐保持完全不中断。他用 C 语言写了一个底层引擎,编译成 WebAssembly,这是一种能让浏览器以接近原生程序的速度去运行代码的技术,这个引擎每隔十五毫秒就要生成一次音符,这样一来,AI 做出的编曲决策,才能够在不打断演奏节奏的前提下,悄无声息地重新塑造整个编曲。他形容这种延迟是音乐上完全不可察觉的,人耳根本反应不过来。整个 Conductr 项目代码量,算下来大概只有四千八百行 JavaScript 和 WebAssembly 代码,对于一个能听、能思考、还能实时跟你一起即兴演奏的乐器来说,这个代码量其实相当精简,完全不是什么庞然大物。

讲完这五个项目,我们来说说这件事对我们这些听众,尤其是软件工程师和 AI 从业者,到底意味着什么,我们应该从中拿走哪些能直接用得上的东西。

第一点,我觉得最值得反复琢磨的信号是,这五个获奖项目里有四个出自非专业开发者。律师、医生、政府公务员、音乐人,他们共同的特点不是代码能力强,而是对自己所在领域的具体问题理解得极其透彻,知道哪里痒、哪里痛、过去的解决方案具体卡在哪一步。如果你也在某个垂直领域工作,手里攥着一个长期困扰这个行业、但一直没有专门的程序员愿意花时间去解决的具体问题,现在你完全可以认真考虑,直接用 Claude Code 这样的工具,自己把第一版原型做出来,而不是先去想办法说服一个程序员相信这个想法值得他投入时间。换句话说,过去你需要先找到一个会写代码的合伙人才能起步,现在你自己就可以是那个起点,会写代码这件事的优先级,确实比几年前下降了一大截。

第二点,从这几个项目的具体实现方式里,我们能总结出几个比较通用的模式,值得你直接搬到自己的工作场景里试一试。一个是并行子智能体处理结构化文档,就像 CrossBeam 那样,把一份复杂文档拆解成多个相对独立的部分,分配给不同的智能体并行处理,最后再把结果汇总起来,这个模式适用于任何需要处理大量重复性文书工作的场景,比如合同审查、报销单据核对、合规检查、招标文件比对,只要文档结构相对固定、错误模式可以归类,这套思路都能搬过去用。另一个是规格驱动开发,也就是先把需求和验收标准写清楚,定义好测试,让 AI 去填充具体的实现细节,这正是 Elisa 这个项目背后的核心理念,也是乔恩自己预言的软件开发未来形态,如果你现在还习惯于先想清楚每一行代码再下手,不妨试着反过来,先把你想要的结果和判断标准写清楚,看看 AI 能不能直接帮你填满中间的实现过程。还有一个模式,是多模态输入加专业领域知识的组合,TARA 把视觉理解能力和工程经济学的专业知识结合起来,这提示我们,如果你的工作里有大量依赖人工目测、人工判断的环节,比如质量检测、设备巡检、图纸审核,都非常值得认真想一想,能不能用类似的方式把这部分工作自动化掉,把人力释放出来去做更需要判断力的事情。

第三点是要提前避开的坑。这些项目都是在一周时间里赶出来的原型,文章里也特别提到了像 PostVisit 这种涉及医疗数据的项目,反复强调了隐私、安全和医生监督这几个环节。如果你听完这一期,也想动手做一个类似的项目,我个人认为有一条经验非常值得提前刻在脑子里,凡是涉及到真实用户数据,尤其是医疗、法律、财务这类高度敏感的领域,黑客松级别的原型确实可以快速验证一个想法是否成立,但是从原型走到一个真正能上线、能对用户结果负责的产品,中间还有很长的一段路要走,数据安全、合规审查、责任归属这些问题,不会因为代码是由 AI 写出来的就自动消失或者自动解决,反而因为整体开发速度变快了,你更需要在项目刚起步的最早期,就把这些环节提前规划进去,而不是等产品做大了、用户多了,再回头手忙脚乱地去补这些课。

如果你是一名专业软件工程师,听到这里可能会有点不是滋味,觉得自己辛苦多年学会的技能,好像一下子变得不那么稀缺了。但我倒觉得这五个项目恰恰说明了另一件事,真正稀缺的能力没有消失,只是换了一个位置。乔恩·麦克比能在三十小时里做出一千五百个测试和三万九千行代码背后扎实的系统,靠的还是他二十年积累下来的系统架构和硬件集成经验,Claude Code 放大的是这份经验,而不是替代了它。换句话说,如果你本来就是工程师,这件事对你来说不是威胁,而是把你的产出速度和能覆盖的问题范围,一下子拉宽了好几倍。

总的来说,这篇文章表面上是在介绍一场黑客松的获奖结果,逐一报一遍获奖者的名字和项目,但我觉得它真正想说的,是 Claude Code 正在把开发者这个身份的边界,重新画一遍。过去你得先学会写代码,才能把脑子里的想法变成一个产品,代码能力是入场券,没有它你连起跑线都摸不到。现在这个顺序很可能要反过来了,你得先对一个具体问题有足够深的理解和足够强的执念,代码反而成了相对容易补齐的那一部分,真正稀缺的反而是问题本身。我个人认为,这五个项目里最打动我的不是任何一处技术细节,而是迈克·布朗那句话,他说自己赢了这场比赛,却没写过一行代码,甚至没读过一行代码。这句话放在两年前几乎是不可想象的,但现在它正在变成一种新的常态,而且这种常态扩散的速度,可能比我们大多数人预期的都要快。好,这一期就先聊到这里,我们下期再见。
