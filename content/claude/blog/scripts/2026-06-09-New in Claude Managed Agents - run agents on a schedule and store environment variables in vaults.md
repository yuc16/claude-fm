二〇二六年，AI agent，也就是 AI 智能体，已经从纯粹的概念验证阶段，开始向规模化的生产部署迈进。大多数有技术实力的团队，已经做过 demo，跑过原型，甚至完成了第一批内部测试。但是真正把 agent 推进生产环境，让它七乘二十四小时稳定运行，让它每周自动跑任务，让它可以安全地访问公司内部的 API 和数据库，让它在出错的时候能够优雅地处理——这一步，对很多团队来说依然是一道坎。

这道坎不是因为模型不够聪明，也不是因为技术路线不成熟，而是因为工程基础设施还没有到位。你需要一套调度系统来定期触发 agent，你需要一套凭证管理方案来让 agent 安全地访问外部服务，你需要考虑 agent 出错了怎么办、怎么监控它的运行状态、出了问题怎么排查。这些问题每一个单独拿出来都不算难，但是叠加在一起，就会让一个原本应该专注于 AI 产品逻辑的团队，陷入大量基础设施工程工作的泥潭。你想做的事情是"让 AI 帮我每天分析一遍日志"，但结果你在搭调度服务和密钥管理这些事情上花掉了好几个工程师好几周的时间。

Claude Managed Agents，也就是 Claude 托管智能体，正是在这个背景下推出的一套托管服务。它的核心主张是：让你专注于 agent 的任务逻辑，底层的运行环境、工具沙盒、基础设施，全部交给平台来管。而这次发布的两个功能，定时部署和 Vault 环境变量，就是在进一步补上这个主张的最后几块重要拼图。

我们现在来逐一深入这两个功能，先说定时部署，也就是让 agent 按照计划自动运行这个能力。

定时部署的工作方式说起来其实很朴素：你给一个 agent 部署配置一个 cron 计划。cron 是一种非常经典的定时任务语法，在 Unix 系统里已经用了几十年了，它用几个数字来表示"几点几分、每周几、每月几号"这种时间模式。比如"每天凌晨一点执行"，或者"每周一早上八点执行"，或者"每月第一天零点执行"。这个语法非常简洁，但表达能力很强，几乎所有常见的定时需求都可以用它来描述。

你配置好 cron 计划之后，每次计划时间触发，系统就自动为这个 agent 启动一个全新的会话，让它去完成它的任务，完成之后会话结束。整个过程完全不需要你手动干预，也不需要你自己维护一个后台服务来做这件事。

我想让你在这里停一下，想象一下在没有这个功能之前，你要实现同样的效果需要干什么。

你大概要这样：先租一台服务器，或者使用一个云函数；然后在上面配置 cron job，或者用云平台的定时触发器；这个定时器触发之后，要去调用你的 agent 的运行逻辑，也许是一个 Python 脚本，也许是一个 API 接口；你还要管理这整套服务的生命周期，要监控它有没有挂掉，要处理失败重试的情况，要看日志排查问题，要在它崩掉的时候能及时发现并恢复。

对于一个成熟的后端工程师来说，这些事情当然不是不会做，但这不是你来做 AI agent 的初衷。你想做的是"让 AI 帮我每天分析一遍日志"，而不是"我来维护一套稳定运行的定时任务调度系统"。平台帮你把这个基础设施层直接收走了，这才是定时部署这个功能真正的价值所在——它让你只需要关心"agent 要做什么"，而不需要关心"agent 怎么被触发运行"。

文章里列举了三个真实使用这个功能的团队，我来仔细说说每一个，因为这些案例非常有代入感，能帮助我们建立具体的认知。

第一个是乐天，Rakuten，大家应该都熟悉，日本的大型电商平台，在国内也有不少工程师朋友用过它的技术产品。乐天的团队用定时部署来分析电子表格数据，然后按照每周或者每月的周期，自动生成业务报告和幻灯片。除此之外，他们还让 agent 监控生产系统的日志和各种运行指标，这样产品经理可以直接看到应用的健康状况，而不需要去专门搭一套监控大盘或者可视化系统。

我觉得乐天这个案例揭示了一类非常普遍的企业需求：数据明明就在那里，但是要把数据变成业务人员能直接看懂的报告，中间有一道鸿沟。以前这道鸿沟要么需要一个会写 SQL 和做数据可视化的工程师来填，要么就得上一套 BI 系统。而现在 agent 可以承担这个中间层的角色，而且它是用自然语言来理解数据、生成报告的，比传统的 BI 工具更灵活，配置门槛也更低，不需要学习复杂的报表配置语言。

第二个案例是 Actively AI，这是一家做 AI 驱动销售工具的初创公司。他们的核心业务是帮助销售团队做跨账户的智能搜索和研究。他们之前自己搭建了一套调度基础设施，用来定期刷新搜索结果的缓存。但是迁移到 Claude 的定时部署之后，直接把自己之前搭的那套调度系统给拆掉了，因为平台已经帮他们做了同样的事情，而且做得更可靠、不需要他们自己维护。

这个案例给我的启发是：你自己维护的任何基础设施都是一个工程负担，哪怕它功能上完全够用。一旦有可靠的托管方案出现，迁移过去是值得的，因为你可以把那部分工程精力从维护基础设施上解放出来，放到对业务更有实际价值的地方去。这种技术栈的简化，随着时间累积，效果会越来越显著。

第三个案例是 Ando，这是一家帮助招聘团队和销售团队提升工作效率的工具。他们的 agent 做的是这样一件事：自动监控特定的沟通频道，比如 Slack 频道或者邮件，关注有没有出现需要跟进的待办事项，到期的时候自动发出跟进通知，还会发会议提醒。

这个用例的特别之处在于，它不是"定时跑一个批处理任务"这种直接模式，而是"持续在那儿守着，每隔一段时间检查一遍，一旦发现需要行动的信号就行动"。这种"守夜人"型的工作，以前只能靠人来做，或者自己搭一套复杂的监控和通知系统。现在配合定时调度，agent 可以定期检查频道状态，发现需要跟进的事情就主动触发相应的动作。对于招聘和销售这种"响应速度很重要"的场景，这个能力非常实用，因为错过一个跟进时机，可能就错失了一个候选人或者一个客户。

总结一下定时部署这个功能：它的本质是消除了"调度层"这个需要开发者自己解决的工程问题，让 agent 可以无人值守地定期运行。适合所有具有周期性的任务，包括每天的数据处理和质量检查，每周的业务分析报告，每月的合规扫描，以及持续的监控和告警。这些场景加在一起，覆盖了企业里相当大一部分重复性、规律性的工作。

文章还提到了一些操作层面的细节：你可以随时暂停、恢复或者归档一个定时部署，也可以手动触发额外的临时运行。这些能力让定时部署在实际使用中更灵活——你不会被计划死死绑住，遇到需要紧急手动干预的情况，或者某个外部依赖临时不可用不想让 agent 空跑的情况，你依然有完整的控制权。

现在我们来聊第二个功能，也是我个人认为技术含量更高、更值得深入理解的一个：Vault 环境变量，也就是在安全库里存储环境变量，让 CLI 工具和其他需要身份验证的外部工具可以被 agent 安全调用。

先帮你建立一个背景认知。agent 要做有实际价值的事情，几乎不可避免地需要访问外部系统：调用一个 API 来拉取数据，用命令行工具来操作某个服务，登录一个平台执行某个操作。而所有这些，都需要某种形式的身份凭证。凭证的管理，在 AI 系统里是一个特别敏感、也特别容易出问题的领域。

为什么敏感？因为 agent 的核心是语言模型，而语言模型有一个特性：它把自己的上下文，也就是到目前为止对话里的所有内容，都当成可以处理和响应的信息。如果你把 API 密钥放在 prompt 里告诉 agent"用这个密钥去调用那个接口"，这个密钥就存在于模型的上下文里了。一旦发生所谓的 prompt 注入攻击，也就是 agent 在处理外部内容时，那些内容里藏了恶意指令，诱导 agent 把上下文里的某些信息发送到攻击者的服务器——你的密钥就可能就这么泄露出去了。

这不是什么理论上的假设风险，这是实际上在一些 agent 应用里已经出现过的安全事件。对于企业级的使用场景，这种风险是完全不可接受的。

所以工程上需要解决的问题是这样一个看起来矛盾的要求：agent 要能使用凭证来调用外部服务，但是 agent 本身不应该"知道"这个凭证的实际值。这两件事听起来矛盾，但是在工程上是可以解决的，关键是把凭证的传递放到模型层之外。

Anthropic 的解法是在网络层做凭证注入。具体来说，你在 Vault，也就是安全库里，注册一个 API 密钥，给它一个名字，这个名字就是对应的环境变量名，同时指定这个密钥被允许访问哪些域名。然后，在 agent 的沙盒运行环境里，这个环境变量是存在的，CLI 工具可以读取到它，但是它的值是一个占位符，不是真实的密钥值。

真正的密钥在哪里？它在网络边界。当 agent 的沙盒里运行的某个 CLI 工具发起一个 HTTP 请求的时候，这个请求在离开沙盒、到达网络边界的时候，系统会检查这个请求的目标域名是否在你预先配置的允许列表里。如果是，就把真实的密钥附加到请求的认证头里，然后让请求发出去。如果不是，密钥不被附加，请求虽然可以发出去，但是没有有效凭证，自然会被外部服务拒绝。

你理解这个设计的妙处了吗？我来用一个比喻帮你更直观地感受一下。

你可以把这个机制想象成这样一个场景：你是一家公司的管理者，你雇了一个员工去帮你处理一些需要用到公司信用卡的事务。但你不想直接把信用卡号码告诉这个员工，因为你没有办法保证他不会记下来，或者在某种情况下误用它。于是你想到了一个办法：你让员工拿着一张只有你名字的空白授权单，去找公司的专属结算服务。结算服务那边知道这张授权单对应哪张信用卡，但员工永远看不到实际的卡号。员工每次需要付款，就拿着授权单去找结算服务，结算服务确认这笔付款的商家是你预先审批过的，然后直接替你和商家完成结算，整个过程员工全程都不知道卡号是多少。如果员工想去一个你没有授权的商家消费，结算服务就直接拒绝，不帮他刷卡。

在这个比喻里，员工是 agent，信用卡号是 API 密钥，空白授权单是沙盒里的环境变量占位符，专属结算服务是网络边界的密钥注入组件，你预先审批的商家列表就是你配置的允许域名白名单。整个流程里，员工从来不知道卡号是什么，但业务还是能正常进行。

这个设计有几个非常漂亮的工程特性。

第一个是密钥从始至终都没有进入模型的上下文。模型运行的沙盒里只有占位符，真实密钥只存在于网络边界的某个安全组件里。这彻底断绝了通过 prompt 泄露密钥的可能性，不管发生什么样的 prompt 注入，模型也不知道真实密钥是什么，因为它从来就没有看到过。

第二个是密钥更新零停机。如果你需要轮换 API 密钥，也许是因为安全政策要求定期更换，也许是因为原来的密钥可能已经泄露，你只需要在 Vault 里更新密钥的值。正在运行的 agent 会在它下一次发起调用的时候，自动使用新密钥，整个过程对 agent 完全透明。不需要重启服务，不需要重新部署，业务完全不中断。

第三个是最小权限原则的自动执行。每个凭证只能被用于访问你明确允许的那几个域名，agent 没有办法拿着这个凭证去访问你没有授权的服务。这把 agent 在凭证使用层面可能造成的损害范围，限制在了你事先圈定的范围之内。

这个功能目前适用于大多数通过 HTTP 请求发送密钥的命令行工具，也就是 CLI 工具。CLI 工具，也就是命令行界面工具，是一种非常常见的轻量级集成方式——很多 SaaS 服务都会提供一个可以直接在终端里运行的命令行程序，你在终端里敲一个命令，就可以操作对应的服务。agent 可以直接在沙盒里调用这些命令行程序，就像一个人类工程师在终端里操作一样，这是一条搭建成本非常低的集成路径。

文章里提到了几个目前已经支持的 CLI：Browserbase、KERNEL、Notion、Ramp 和 Sentry 的 CLI。这几个工具覆盖了浏览器自动化、数据库管理、知识协作、财务费用和工程错误监控等几个常见的工作场景。

有一个细节值得我们单独停下来多说两句：Browserbase 和 KERNEL 这两个工具，给 Claude Managed Agents 带来了浏览器操作能力，而且文章特别说明这是 Managed Agents 第一次拥有浏览器能力。

这件事的意义比看起来要大很多。想想互联网上有多少系统根本就没有开放公共 API，你与它们交互的唯一入口就是网页界面——你需要打开浏览器、登录进去、点击按钮、填写表单、提交数据、看到结果。以前要让 agent 操作这类系统，要么你得自己写 Selenium 或者 Playwright 这类浏览器自动化脚本，但这类脚本对网页结构的变化非常脆弱，一旦目标网站改了 CSS 类名或者调整了布局，脚本就可能全部失效；要么使用一些专门的浏览器自动化服务，但那些服务和 agent 框架的整合往往比较复杂，需要额外的工程工作。现在通过 Vault 安全认证的 Browserbase CLI，这条路被打通了，而且是在 Managed Agents 的整体框架里做的，与其他工具和能力自然地整合在一起，不需要额外的架构工作。

好，我们来看看文章里关于 Vault 环境变量功能的几个落地案例，我觉得每一个都各有侧重，值得仔细说说。

第一个是 Notion。Notion 是大家都非常熟悉的笔记和协作工具，在国内工程师和产品经理群体里用户不少。他们在推出自己的 CLI 工具的同时，通过 Vault 环境变量来完成身份验证，让 agent 获得了文件上传能力。他们在案例介绍里特别强调的一点是：整个过程里 API 令牌从来没有被传给模型。

我觉得这个强调非常有意思。它不只是一个技术细节，它更是一个对企业客户的信任证明。企业在采购任何涉及 AI 处理的工具时，必然会问的问题之一就是"我的 API 凭证在哪里、谁能看到它"。Notion 能够明确回答"令牌永远不会进入 AI 模型的上下文"，这就直接消除了一类重要的安全顾虑，让企业客户在决策时少了一个障碍。这对推广 agent 工具的团队来说，是非常实用的一张安全牌。

第二个案例是 Browserbase。他们维护着一个公开的浏览器技能目录，这个目录里收录了大量可以复用的浏览器自动化技能。为了确保目录里的技能始终是有效的，他们用一个定时部署的 agent，定期跑一遍目录里的所有技能，验证每一个技能是否还能正常工作。一旦发现某个技能失效，可以及时修复或者从目录里移除。

这个案例很有意思的地方在于：它把定时部署和 Vault 环境变量这两个新功能都用上了，是一个很典型的组合应用场景。同时它代表的是"自动化质量保障"这个场景——维护一个技能库不只是要把技能写进去，还要持续验证它们还能不能用。用人工来做这件事非常枯燥，而且很容易遗漏；用一个定时跑的 agent 来做，效率高，覆盖率也更有保证。

第三个案例是 KERNEL，一家做 AI 基础设施的公司。他们用 Vault 环境变量让 agent 安全连接到自己的数据库——这些数据库里存储的是用量数据和客户对话记录，是非常敏感的业务数据。agent 的任务是持续监控用量的突然飙升，一旦检测到异常，立刻发出报警，让团队去跟客户确认这个用量是否是预期的行为。

这个场景的敏感性在于：数据库访问凭证是所有凭证里风险最高的一类，一旦泄露，攻击者可以直接读取甚至修改数据库里的所有数据。以前面对这种级别的敏感凭证，很多团队的选择是干脆不让 agent 直接访问数据库，而是通过中间层来隔离。但 Vault 机制提供了一个既能让 agent 使用这类凭证、又不会让凭证进入模型的方案，让"高权限凭证和 AI agent"这个组合在工程上变得可控和可接受。

第四个案例是 Milana，一家做 AI 产品工程师的公司，他们的 agent 帮助客户自动寻找并修复代码库里的 bug，同时做大规模的数据分析。他们通过 Vault 让 agent 安全连接到客户的代码库。文章里提到了一个细节：大规模数据分析的速度"比以前更快了"。这个说法很有意思，它暗示着 agent 不只是在替代人工做原来人工能做的事，而是利用了并行化和自动化带来的速度优势，做到了人工很难达到的处理速度。

这四个案例放在一起，给我们描绘了一幅清晰的图景：Vault 环境变量不是一个小的安全补丁，它是一个能力放大器。它让那些因为安全顾虑而不敢给 agent 开放权限的场景，变得在工程上可行，也让企业在向客户证明"我们的 AI 工具是安全合规的"这件事上，多了一个有力的论据。

关于定时调度的实践建议，我来说几个最重要的点。

第一，识别你团队里的周期性工作。如果你的团队现在有任何"定期要做一次"的事情——每天要跑一遍数据清洗和质量检查、每周要生成一份分析报告、每月要做一次合规扫描，或者每隔几小时要同步一次外部数据源——这些场景都是定时部署的天然落地场景。你现在就可以去 Claude Console 探索这个功能，不需要另外搭任何基础设施。

第二，设计好跨次状态的存储方案。这个点非常重要，是初次使用定时部署时很容易忽略的陷阱。定时部署每次触发都会启动一个全新的会话，agent 的对话历史在新会话里是不存在的，所以如果你的任务在不同次运行之间需要保持某些状态——比如"上次处理到哪条记录了""上次运行的结果是什么"——你必须显式地把这些状态写到外部存储里，比如数据库、云存储，或者一个结构化的文件。不要指望依赖 agent 的对话历史来传递跨次状态，因为那个历史每次都是空的。

第三，在 prompt 里写清楚容错逻辑。生产环境里的定时任务总会遇到外部依赖偶尔不可用的情况，比如你的数据源今天更新晚了，或者你要写入的目标系统短暂有故障。你需要在 agent 的 prompt 或者工具的逻辑里，明确告诉它遇到这类情况应该怎么处理：是等待重试，如果是，要重试几次、间隔多久；还是跳过这次任务并记录日志；或者是发一条告警消息到某个频道。不要期待 agent 自己想清楚容错策略，给它明确的指令会让整个系统更可靠。

关于 Vault 环境变量的实践建议，也有几个要点要说。

第一，认真评估是否把凭证管理迁到 Vault。如果你现在正在开发一个需要调用外部服务的 agent，而你的凭证还放在配置文件里或者代码里，请认真考虑迁移到 Vault 管理。即使你团队规模不大，当前的安全要求也没有那么高，Vault 有一个非常实用的日常运维好处：密钥轮换零停机。以前你换一个 API 密钥，需要改配置、重新部署服务；用了 Vault 之后，你在后台更新密钥值，正在跑的 agent 下次调用时自动使用新密钥，完全透明，不影响业务。

第二，认真对待域名白名单的配置。在你注册凭证的时候，系统要求你指定这个凭证可以访问哪些域名。这个配置要做得合理：白名单太宽松，比如允许所有域名，就失去了最小权限保护的意义；太严格，可能导致 agent 在访问某些合法的 API 端点时密钥不生效，产生难以排查的错误。我的建议是：在配置上线前，实际跑一遍你的 agent 的完整工作流，把它会访问的所有域名都记录下来，再按照这个完整列表来配置白名单，做到既不遗漏也不多余。

第三，最小权限原则要落到实处。不要把所有凭证都塞进一个大的 Vault 配置里。应该按照不同的 agent 部署，分别管理对应的凭证集合。一个做数据报告的 agent，不应该持有访问代码库的凭证；一个做客服的 agent，不应该持有访问财务系统的凭证。权限范围控制得越小，出了问题的影响面就越小，事后排查也更容易。

第四，关于 CLI 的版本管理。CLI 作为集成方式有一个在 API 集成里相对少见的问题：版本升级可能带来破坏性变化。API 通常有明确的版本号和向后兼容性保证，但 CLI 版本升级有时候会改变命令的格式或者输出的内容结构。而 agent 往往依赖解析 CLI 的输出来获取信息，一旦输出格式悄悄变了，agent 可能就会出错，而且这类错误往往不容易第一时间发现。所以如果你在生产环境里依赖某个 CLI，要固定 CLI 的版本，不要让它随意跟着系统更新，等你确认新版本没有破坏性变化之后再主动升级。

第五，关于浏览器能力，如果你有需要操作没有开放 API 的网页系统的场景，Browserbase 和 KERNEL 带来的浏览器能力值得认真探索。agent 来做浏览器操作比传统基于 CSS 选择器的爬虫脚本有更强的弹性，因为 agent 可以通过语义理解来适应一定程度的 UI 变化，它不是靠死记硬背"第三个按钮在这里"来操作，而是能理解"这个按钮的意思是提交"。但这也不是银弹，对于 UI 结构变化非常剧烈的场景，还是需要有备用方案或者人工介入机制。
