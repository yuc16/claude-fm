大家好,欢迎收听Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是Introducing Agent Skills,翻译过来就是"Agent技能系统正式发布"。这篇文章是Anthropic官方博客发的,发表时间是二〇二五年十月十六日。

算一下日子,今天是二〇二六年六月二十一日,距离这篇文章发表已经过去了大半年,差不多八个月时间。所以在正式展开之前,我要先提醒大家一句,这篇文章里描述的功能形态,可能已经在这八个月里经历了不少迭代。实际上文章本身就有一个更新说明,提到在二〇二五年十二月十八日的时候,Anthropic已经给Skills加上了组织级管理功能,还推出了一个收录合作伙伴制作的技能的目录,甚至把Agent Skills发布成了一个跨平台的开放标准。这意味着Skills这个东西从一个新功能,已经快速演变成了一个行业里大家都在用的通用协议。所以大家在听这期节目的时候,要把它当作了解这个功能诞生之初的设计理念和动机来听,如果你想知道现在最新的玩法,还是要去看官方最新的文档。

那我们先来说说背景。在二〇二五年那个阶段,大语言模型已经发展到了一个很关键的节点,就是模型本身的通用能力已经相当强了,但是大家逐渐发现,通用能力强不代表在具体场景里好用。打个比方,你请了一个名校毕业、什么都懂一点的应届生来公司,他确实聪明,但是他不知道你们公司财务报表用的是哪种格式,不知道你们品牌手册里logo要留多少边距,也不知道你们内部那套Excel模板该怎么填。这种"通用智能"和"具体场景专业能力"之间的鸿沟,在Claude这样的Agent产品身上体现得特别明显。

之前大家解决这个问题,主要靠两种办法。第一种是把所有的背景知识都塞进系统提示词里,也就是prompt里,这样做的问题是,提示词会变得又长又乱,而且不管你当前任务用不用得到这些知识,它都会一直占着上下文窗口,也就是model能看到的那部分输入空间,既浪费token,也就是模型处理文本的最小单位的额度,又可能让模型注意力分散。第二种办法是开发者自己写代码,在外部判断任务类型,然后调用不同的工具或者拼接不同的提示词,这种办法灵活,但是开发成本高,而且很难在不同产品之间复用,你在API上写的逻辑,搬到Claude Code里基本就得重新写一遍。

正是在这样的背景下,Anthropic推出了Agent Skills这个概念,中文我们可以理解成"代理技能"或者"智能体技能"。核心思路其实挺朴素的,就是把那些针对特定任务的说明、脚本和资源,打包成一个个独立的文件夹,而不是一股脑塞进提示词里。Claude在处理任务的时候,会先扫一眼有哪些技能包可用,如果发现某个技能包跟当前任务相关,它才会去加载这个技能包里的具体内容。如果不相关,那这个技能包就完全不会占用上下文,相当于是一个躺在书架上的工具书,你需要的时候才翻开看,不需要的时候它就安安静静放在那儿,不会影响你正在做的事情。

这个设计听起来简单,但我觉得它解决问题的方式特别巧妙,值得我们细细拆解一下。我们今天主要挑三个核心点来讲深一点,第一个是Skills的渐进式加载机制,第二个是Skills的三大特性,也就是可组合、可移植和高效,第三个是Skills在三大产品线里分别是怎么落地的,也就是Claude应用、API开发者平台和Claude Code。

我们先来说第一个核心点,渐进式加载机制。文章里有一句话特别关键,它说Claude在处理任务的时候,会扫描所有可用的技能包来寻找相关的匹配项,一旦匹配上了,它只会加载完成任务所必需的最少信息和文件,这样既能保持Claude的运行速度,又能让它获得专业领域的知识。

这句话背后其实藏着一个很重要的工程思路,就是按需加载,而不是全量预加载。我们可以打个比方,假设你是一个图书馆的馆长,你的图书馆里有成千上万本专业书籍,涵盖法律、医学、烹饪、编程等等各个领域。如果每次有读者来问问题,你都把这成千上万本书全部搬到他面前,那读者肯定看花眼,你自己整理起来也累得不行。但如果你训练出一种本能,读者一开口说"我想知道怎么做红烧肉",你立刻就知道该去厨艺区的哪个书架,把那本最相关的菜谱书递给他,这样效率就高多了。

Skills的渐进式加载,本质上就是让Claude具备了这种"快速定位、精准取材"的本能。具体来说,每个Skills文件夹里通常会有一个叫SKILL.md的核心文件,这个文件会写明这个技能包是用来做什么的,适用于什么场景。Claude在浏览所有可用技能包的时候,只需要先看一眼这些简短的描述,这个开销是很小的,基本不会拖慢速度。等它判断出某个技能包跟当前任务确实相关,它才会进一步加载这个技能包里更详细的内容,比如具体的操作步骤、参考资料,甚至是可以直接执行的脚本代码。

这里我要强调一下,文章里特别提到,Skills不只是静态的文字说明,它还可以包含可执行的代码,用在那些传统编程方式比让模型直接生成token更可靠的场景里。这一点我觉得特别值得展开讲讲。我们知道大语言模型生成文字,本质上是一个概率过程,每一次生成下一个token,都是基于概率分布去采样的。对于很多创造性的任务,比如写一段总结,这种概率性是优点,它能带来多样性和灵活性。但是对于另外一些任务,比如说你要往一个Excel表格里填入精确的数字,或者要按照严格的格式生成一个文件,概率性反而是缺点,因为哪怕模型只有百分之一的概率出错,在批量处理的场景下也会积累出不少错误。

这时候,如果技能包里直接打包了一段成熟的脚本代码,比如说一个专门用来读写Excel文件、处理公式的Python脚本,那么Claude遇到需要操作Excel的任务时,就不需要自己去现场生成代码,然后还要担心代码逻辑有没有写对,它可以直接调用技能包里已经写好、已经测试过的代码,这样准确率会大大提升。这种"模型决策加传统代码执行"相结合的模式,我个人认为是这篇文章里最有工程智慧的一个设计,它没有迷信"模型生成一切",而是清醒地认识到,有些任务该交给确定性的代码去做,有些任务才适合交给模型的生成能力去发挥。

好,我们再来看第二个核心点,就是Skills的三大特性,可组合、可移植和高效。文章里专门用了几个关键词来形容Skills,我们一个一个拆开说。

第一个特性是可组合。原文的说法是,技能包可以互相叠加使用,Claude会自动识别出当前任务需要哪些技能包,并且协调它们之间的配合。这个特性其实很好理解,我们可以想象一个真实场景,假如你让Claude帮你做一份年度总结的PPT,这里面其实涉及到多个专业领域的知识,一个是怎么去分析和整理你提供的数据,另一个是怎么按照PowerPoint的格式标准生成一份美观的演示文稿,还有可能涉及到你公司的品牌规范,比如logo要放在哪里,字体颜色要用什么。如果这些专业知识分别打包成不同的技能包,那么Claude在执行这个任务的时候,完全可以同时调用多个技能包,数据分析的技能包帮它理清思路,PowerPoint生成的技能包帮它把内容转换成符合格式标准的文件,品牌规范的技能包帮它确保视觉呈现符合公司要求。这几个技能包之间不需要互相打架,Claude会自己判断什么时候用哪个,这种自动协调的能力,我觉得正是Agent这个概念里"agent"也就是代理人或者智能体这个词所强调的自主性的体现。

第二个特性是可移植。Skills在所有产品里用的是同一种格式,你只需要构建一次,就可以在Claude应用、Claude Code和API这三个不同的产品形态里复用。这一点对开发者来说意义非常大。我们可以回想一下之前提到的那个痛点,以前你如果想让Claude在API场景下学会某种专业能力,你往往得专门写一套提示词工程的方案,但是这套方案搬到Claude Code里可能就不适用了,因为产品形态不一样,交互方式也不一样。现在有了Skills这种统一格式,你写好一个技能包,不管是分析财务数据,还是按照特定的代码规范来写程序,这个技能包都可以原封不动地搬到三个不同的产品场景里使用。这种"一次构建,到处运行"的思路,其实跟我们软件工程里很多经典的设计理念是相通的,比如说我们写一个公共的工具库,目的就是让不同的项目都能复用,而不需要每个项目都重新发明轮子。

第三个特性是高效,这一点其实跟我们前面说的渐进式加载机制是相呼应的,只在需要的时候加载需要的内容,这样能保证Claude在大部分时候依然保持轻快的反应速度,不会因为挂载了一堆技能包就变得又慢又笨重。

文章里还有一个很形象的比喻,它说你可以把Skills想象成定制的"入职培训材料",这个比喻我特别喜欢,因为它精准地抓住了Skills的本质。我们想一下,任何一个新员工入职一家公司,公司都会给他一套入职培训资料,告诉他公司的规章制度、业务流程、常用工具怎么操作。这些资料的作用,就是让一个本来什么都不懂这家公司具体情况的新人,迅速变成一个对这家公司业务了如指掌的专业人士。Skills对Claude起的作用其实是一样的,通过打包专业知识,你相当于给Claude做了一次有针对性的入职培训,让它从一个通用的智能体,变成在你关心的领域里的专家。

接下来我们说第三个核心点,就是Skills具体是怎么在不同产品里落地使用的。文章里分了三大块来讲,Claude应用、Claude开发者平台也就是API,还有Claude Code。

我们先说Claude应用这一块。这个功能对Pro、Max、Team和Enterprise这几类付费用户是开放的。Anthropic自己提供了一批面向常见任务的技能包,比如文档创建相关的技能,还提供了一些示例技能包供用户参考和自定义,当然用户也可以自己创建专属的技能包。这里有一个细节我觉得特别人性化,就是Claude会根据你的任务自动调用相关的技能包,完全不需要你手动去选择该用哪个技能,而且你甚至可以在Claude展示思考过程的时候,看到它正在调用哪些技能包。这种透明度对用户体验来说是很重要的,因为它让原本黑箱式的模型决策过程,变得可以被观察和理解。

更有意思的是,文章提到创建技能包这件事本身,也变得非常简单,因为Anthropic做了一个叫skill-creator的技能,专门用来交互式地引导你创建新的技能包。它会主动问你的工作流程是什么样的,然后帮你生成对应的文件夹结构,自动把SKILL.md这个核心说明文件的格式排好,还会帮你把需要的资源文件打包进去,整个过程完全不需要你手动去编辑文件。这一点我觉得体现了一种很典型的产品思路,就是用Claude自己的能力去降低使用Claude新功能的门槛,有点像是"用魔法打败魔法"的感觉。如果你是个普通用户,想给Claude定制一个技能,但是又不太懂技术细节,你完全可以直接跟它对话,让它一步步引导你完成整个创建过程。

启用这个功能的方式也很简单,在设置里就能打开。如果你是团队版或者企业版用户,管理员需要先在组织层面把Skills功能整体启用,然后团队成员才能使用。

接下来我们说API开发者平台这一块。文章提到,Agent Skills现在可以加到Messages API的请求里,而且专门开了一个叫v1斜杠skills的接口端点,给开发者提供了编程方式去管理自定义技能的版本和发布。这里有一个前提条件需要注意,Skills功能需要依赖代码执行工具的beta测试功能,因为运行技能包里那些可执行代码,需要一个安全的隔离环境,这个代码执行工具就是用来提供这个安全沙箱的。

在这一块里,Anthropic自己也提供了一批官方制作的技能包,可以让Claude读取和生成专业的Excel电子表格,里面甚至包含公式计算,还有PowerPoint演示文稿,Word文档,以及可以填写的PDF表单。这些都是企业日常办公里非常高频的需求。对于开发者来说,你也完全可以创建自己的定制技能包,去扩展Claude在你具体业务场景下的能力。

文章还提到,开发者可以通过Claude Console这个控制台界面,很方便地创建、查看和升级技能包的版本。这一点对企业级的应用来说很重要,因为技能包不是一成不变的,随着业务需求的变化,你可能需要不断迭代和优化你的技能包内容,有了版本管理功能,你就可以像管理代码一样去管理这些技能包,保证每次更新都是可追溯、可回滚的。

最后我们说Claude Code这一块。Claude Code是Anthropic面向程序员的命令行编程助手产品,Skills在这里的作用是用来扩展团队的专业知识和工作流程。文章里提到,你可以通过插件市场,具体来说是anthropics斜杠skills这个仓库,去安装技能包,Claude会在判断相关的时候自动加载它们。更重要的是,这些技能包是可以通过版本控制系统在团队内共享的,这意味着你们团队可以把自己的最佳实践,比如代码审查的标准、特定框架的使用规范,打包成技能包,然后通过Git这样的版本控制工具分发给团队里的每一个成员,这样大家在用Claude Code的时候,都能享受到统一的、符合团队规范的辅助效果。

另外,你也可以手动安装技能包,只需要把它们放到用户主目录下的点claude斜杠skills这个文件夹里就行。文章最后还提到,Claude Agent SDK,也就是给开发者用来构建自定义Agent的软件开发工具包,也提供了同样的Agent Skills支持,这意味着如果你想基于Claude自己搭建一个定制化的智能体产品,你同样可以复用Skills这套机制,而不需要从零造轮子。

好,讲完了这三个核心点,我们来聊聊这个功能对我们普通听众,尤其是软件工程师和AI从业者来说,在实际工作中意味着什么。

如果你是一个企业内部的开发者或者技术负责人,我觉得Skills这个机制最直接的价值,是给了你一个标准化的方式去沉淀和复用团队的专业知识。我们都知道,很多团队的经验和最佳实践,往往散落在某个老员工的脑子里,或者藏在某份很少有人看的内部文档里,新人接手的时候很难快速上手。如果你能把这些经验提炼出来,打包成一个个Skills,那么不管是新员工还是Claude本身,都可以通过这个标准化的渐进式加载机制,快速获取到这些专业知识,而且这个知识包可以在你们公司用的所有Claude产品形态里复用,不管是日常聊天用的Claude应用,还是写代码用的Claude Code,还是你们自己业务系统里调用的API。

如果你是个人开发者,我觉得最值得你去试一试的,是先用skill-creator这个交互式工具,体验一下创建技能包的流程,哪怕只是给自己日常写作或者整理资料的习惯做一个小小的技能包,你也能直观感受到Skills和普通提示词工程的区别在哪里。普通的提示词,你每次新开一个对话,都得重新讲一遍背景,而技能包则是持久化的,一旦创建好,以后每次相关任务它都会被自动调用。

不过这里也有一些坑需要提前避开。文章在最后特别提了一句安全方面的提醒,我觉得这一点不能忽视。原文的意思是,这个功能让Claude具备了执行代码的能力,这个能力很强大,但也意味着你需要谨慎选择使用哪些技能包,尽量只用来自可信来源的技能包,这样才能保护好你的数据安全。

我个人认为这一点提醒得特别及时,因为我们都知道,任何"可执行代码加自动加载"的机制,本质上都存在被滥用的风险。想象一下,如果有人故意制作一个看起来很正常、描述写得很诱人的技能包,但是里面藏着恶意代码,一旦你不小心安装了它,而Claude又恰好判断这个任务跟它相关,把它加载执行了,那这个恶意代码就有可能在你的环境里运行,造成数据泄露或者其他安全问题。所以我建议大家在使用第三方技能包的时候,一定要对来源做基本的审查,优先选择官方提供的、或者你信任的开发者和团队公开维护的技能包,如果是来路不明的技能包,最好先打开看一下里面具体的代码逻辑,再决定要不要用。

另外我也想提一下,这篇文章里提到的"什么是下一步"那一段,Anthropic说他们正在朝着简化技能包创建流程和企业级部署能力的方向努力,目标是让组织更容易在团队间分发技能包。结合我们前面提到的,八个月之后这个功能已经新增了组织级管理、合作伙伴技能目录,还把整个协议开放成了跨平台标准,可以看出Anthropic确实是按照这个方向在快速推进的。这也提示我们,如果你现在打算认真投入精力去构建一套属于自己团队的技能包体系,值得去看看最新的官方文档,因为很多当时还在beta阶段或者还没做的功能,现在可能已经成熟落地了。

好,我们最后来给整篇文章做一个总结。这篇文章核心讲的是Anthropic推出的Agent Skills功能,它通过把专业知识打包成可以按需加载的文件夹,解决了之前提示词工程里"知识塞太多变臃肿,知识塞太少又不够专业"这个两难问题。它的设计亮点在于渐进式加载机制带来的高效性,模型生成和传统代码执行相结合带来的可靠性,以及统一格式带来的跨平台可移植性。从Claude应用到API开发者平台,再到Claude Code,这一套机制贯穿了Anthropic几乎所有的产品线,体现出一种"一次设计、处处复用"的工程哲学。

我个人认为,Skills这个设计思路其实代表了Agent产品发展的一个重要方向,就是从单纯追求模型本身有多聪明,逐渐转向追求怎么把模型的智能和外部确定性的工具、知识有机结合起来,形成一个真正可靠、可定制、可复用的系统。这种思路不仅仅适用于Claude,我相信也会给其他做Agent产品的团队带来启发。如果你也在做相关的产品或者业务,我建议你不妨花点时间去研究一下Skills这套机制的具体实现方式,说不定能给你自己的项目带来一些新的灵感。
