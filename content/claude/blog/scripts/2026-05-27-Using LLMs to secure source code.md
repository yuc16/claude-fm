## Script

今天要跟大家聊的这篇文章，标题是 "Using LLMs to secure source code"，我把它翻译成"用大语言模型保护源代码安全"。这篇文章发表于二〇二六年五月二十七日，距今大约三周，作者是 Anthropic 的 Eugene Yan 和 Henna Dattani，还有好几位安全工程师共同贡献了实际经验。

为什么值得今天花二十五分钟来聊？因为它不是那种"AI 也许可以帮你做安全"的泛泛展望，而是一篇有真实数字、有案例、有踩坑经历的实战总结。Anthropic 的团队一直在配合多个安全团队，帮助他们扫描自己的代码库和开源软件，这篇文章是那些经验的浓缩。如果你的工作涉及代码安全，或者你对 AI agent 怎么真正落地到复杂工程问题感兴趣，这篇文章的信息密度非常高。

先来说说背景。软件安全漏洞不是 AI 时代才有的问题，几十年来安全工程师都在用各种方法找代码里的漏洞：静态分析工具按规则扫描可疑模式，模糊测试随机生成输入试图让程序崩溃，人工代码审计靠有经验的工程师一行行地读代码。但这件事一直有一个核心矛盾：代码的总量在爆炸式增长，而能做安全审计的工程师数量增长很慢。你想象一下，一个大公司的核心服务可能有几百万行代码，涉及几十个团队维护的几百个模块，靠人工审计根本不可能覆盖所有角落。早期的自动化工具，比如静态应用安全测试，也有一个硬伤：它们按提前写好的规则找漏洞，对规则里没有的新型漏洞视而不见。

大语言模型的出现让这个局面开始了根本性的改变。模型不需要提前写规则，它可以像一个有安全经验的工程师一样读代码，理解代码的意图，推断出哪里可能有问题。而且，你可以同时启动几十个甚至几百个 AI agent，去扫描代码库的不同部分。发现漏洞这件事，已经可以大规模并行化了。

但 Anthropic 团队在实际工作中发现了一个至关重要的现象：真正的瓶颈不在发现，而在发现之后。他们一直在扫描开源软件，截至二〇二六年五月二十二日，已经向相关项目披露了一千五百九十六个漏洞。

你知道这里面有多少个被修了吗？只有九十七个。

一千五百九十六个发现，九十七个被修，比例不到百分之七。我在看到这个数字的时候停顿了一下，因为它非常说明问题。这不是说那些漏洞不重要，而是因为从"发现一个漏洞"到"验证它真的能被利用"，再到"有人分配优先级"，再到"工程师真正去修它"，这中间有巨大的摩擦。工具找漏洞的速度，远超过人们消化和处理漏洞的速度。

这篇文章要解决的核心问题正是：怎么把这个比例提高？怎么设计一套流程，让发现和修复之间的鸿沟变小？

Anthropic 团队把实战经验总结成了六个步骤：威胁建模、沙箱搭建、发现、验证、分级筛选、修复。前两步是一次性的准备工作，通常在每个代码库上做一次，代码发生重大变化时重新审视。后四步是你要循环运行的主流程，第一次跑的时候要多跑几遍，直到新发现的漏洞数量降下来为止。

我们一步一步来拆解。

第一步，威胁建模。

威胁建模，英文叫 threat model，说白了就是在开始扫描之前，先搞清楚一个问题：对于这个系统来说，什么叫做漏洞？

你可能觉得这个问题很显然，但文章里有一个案例说明它有多关键。一个团队扫了一个大型项目，误报率高达百分之四十。注意，这不是模型在瞎猜，这些发现都可以复现，PoC，也就是概念验证代码，都能跑通，可以从技术上证明漏洞是可以被利用的。但是负责这段代码的开发团队说这些都是误报。为什么？因为这些漏洞不符合这个项目的威胁模型。这个项目的首席信息安全官说了一句非常精辟的话："模型对代码的理解很好，但对我们的理解不够好。"

这就是威胁建模的核心价值。你需要告诉模型：在我们这个系统里，配置文件里的内容是可信的；认证用户发来的请求是可信的；某一类漏洞在我们的业务场景下不构成威胁，不需要报。这些假设如果不明确写出来，模型只能从代码里猜，而代码里不一定有这些信息。最常见的误报原因正是：模型对你系统的信任边界有偏差，要么把可信的输入当成潜在攻击来源，要么把内部接口误认为是暴露在外网的服务。

怎么做威胁建模？文章给出了两步。

第一步是从代码、文档和历史漏洞里自动生成一个初稿。把你给新来安全工程师看的东西都喂给模型：架构文档、wiki、代码入口点、git 提交历史、过去披露的漏洞记录。让模型生成一个包含系统上下文、核心资产、入口点、信任边界的威胁模型，再分析历史漏洞找出这个系统最容易出现哪类问题。

文章里有一个团队的做法特别值得学习：他们回顾了几百个历史 CVE，也就是公开的安全漏洞披露，还有过去的安全修复提交，把它们提炼成"漏洞形状"的线索，然后让模型回答两个问题：这次修复完整吗？同样的问题在代码库其他地方也存在吗？就这样，他们在一个小时内找到了三个可利用的真实漏洞。他们总结说，"过去有什么漏洞被利用过"，有时候是比"帮我在代码库里找漏洞"更有效的作弊码。这个思路背后的逻辑是：历史漏洞是这个代码库在真实世界已经被验证过的薄弱点，沿着这条线挖往往事半功倍。

第二步是让模型去采访一个了解系统的人。文章引用了安全领域经典的肖斯塔克四问：我们在构建什么？什么可能出错？我们正在怎么应对？我们做得好吗？有了第一步的草稿，受访者不需要从零开始，而是在一个已有框架上补充和修正，效率高很多。这个采访步骤是可选的，但它能提供模型从代码里拿不到的背景信息，比如团队内部的设计决策、已知的运营约束、有意为之的权衡。

做好的威胁模型要存在代码库里，作为一个专门的文件。这个文件在后续的发现阶段用来划定扫描范围，在分级阶段用来校准漏洞的实际严重程度，是整套流程的地基。

第二步，沙箱搭建。

沙箱，sandbox，就是一个隔离的环境，让你可以安全地运行可能有风险的操作。在这套流程里，沙箱有两个目的。

第一个目的是保护你自己的系统不受 agent 意外行为的影响。你要让 agent 自主地读代码、写代码、执行代码，如果没有隔离，agent 可能会做你没有预期的事。文章里有两个真实案例：一个团队告诉模型说它没有网络访问权限，但实际上有，模型发现了它可以从 GitHub 拉代码。另一个团队的 agent 在扫描过程中自发地回复了一个 GitHub 上的 issue。这两个行为都不是恶意的，但都说明了一点：光靠提示词说"你没有网络权限"是不够的，你需要用真正的代码和基础设施配置来强制执行约束。隔离级别要匹配你的威胁模型：容器对只读代码的发现 agent 够用，但如果要运行目标系统和 PoC，最好用微虚拟机或者完整虚拟机，并且锁死出站流量，不要让任何东西能触达你的生产系统。

沙箱的第二个目的是用来证明漏洞真的可以被利用。在纯粹读代码的静态分析阶段，模型是在假设和推断，它没法真正知道某条代码路径是不是可达的，或者上游有没有别的防护措施拦截了攻击。有了沙箱，agent 可以编译代码、跑测试、真正运行一个 PoC，不可利用的假阳性就会大幅减少。文章里有一个进攻型安全团队，他们给 agent 搭了一个测试床，设定了一个简单的规则：只有 agent 能写出 PoC 并在测试床上成功跑通，才算是真正的漏洞。六周之后他们的评价是："给模型提供测试床、运行中的真实系统和让它运行 PoC 的能力，是提升发现效果最大的杠杆。"

搭沙箱有几个关键实践。首先，锁定所有依赖的版本，包括 Docker 镜像的标签、代码提交的具体哈希值、所有依赖包的版本，这样每次运行都在完全相同的环境里。文章里有一个教训：一次扫描发现了一个漏洞，但后来发现 agent 拉取的是一个老版本的库，而不是实际部署的版本，这个"漏洞"在真实生产环境里根本不存在。他们现在用 Docker 容器把依赖锁定到与生产环境完全一致的版本，确保发现 agent 和验证 agent 用的是攻击者实际会面对的同一份东西。

其次，严格控制沙箱的网络权限。搭建阶段拉好所有依赖、部署目标服务、跑通现有测试之后，给整个环境做一个快照，然后断掉网络。扫描期间只允许流量通过本地代理访问模型 API。每次扫描开始时加载快照，确保每次都从同一个干净状态出发。

当然，有些复杂系统有很多云依赖，搭一个完整的沙箱不切实际。文章说在这种情况下可以先跳过沙箱，直接进入发现阶段，前沿模型光靠分析源代码就能找到大量真实漏洞，代价是验证阶段要多花人工投入，因为没有 PoC 来自动证明可利用性。等发现量证明投入沙箱是值得的，再回来建。

第三步，发现。

现在来到让模型去代码库里找漏洞的阶段。

文章在这里有一个非常反直觉的发现，我觉得值得单独强调：对于当前的前沿模型，提示词越简单，发现效果越好。更详细的提示词，比如一个长长的检查清单，反而让模型发现的漏洞变少，新奇漏洞更少。

这个反直觉的背后有一个清晰的逻辑：当你给模型一个二十条的检查清单，模型就会按这二十条来检查，不会去想清单之外的东西。但安全漏洞的有趣之处恰恰在于它们往往出现在你没有预见到的地方。如果你只给目标和背景，把"怎么找"这件事留给模型自己决定，前沿模型有足够的安全知识，会去尝试它认为有效的方法，反而能找到更多你没想到的漏洞。

实践建议是：告诉模型为什么要扫描、什么样的发现是有价值的、扫的是什么系统，然后把方法论交给模型。如果想专门针对某类漏洞，就描述这个漏洞类型是什么、危害是什么、通常出现在代码的哪个位置，让模型去识别它在这个代码库里的具体表现形式。

在工具方面，给模型提供搜索和读取代码库的基础工具，比如 grep 和 glob，另外也可以给它你们团队用的安全专用工具。前沿模型现在越来越擅长在需要时自己写工具，可以允许它这么做。文章里有一个渗透测试团队，他们给发现 agent 提供了发送请求、检查响应、查询流量日志的工具，结果 agent 不再需要猜某条路径是否可达，可以直接对每个候选漏洞做真实测试，把真阳性率提升到了接近百分之百。

在规模化方面，文章给出的正确做法是三层架构：先让一个 agent 对整个系统做分区，比如按攻击面、端点或模块来划分；然后把这些分区分配给并行的发现 agent 去分头扫描；最后再做一个系统级的综合，把各分区的发现作为上下文。

这里有一个很重要的教训：简单地增加 agent 数量来暴力扩展，会很快遇到收益递减。一个团队说，他们一开始不停地加 agent 数量，结果发现的大量漏洞都是彼此的重复，并没有覆盖更多的代码。另一个团队增加了关注区域和并行 agent 数量，发现了很多问题，但大多数都是重复的。正确的做法是先做好分区，让不同 agent 关注不同方向，而不是让所有 agent 都扎在同一段代码里。

第四步，验证。

发现阶段优化的是召回率，就是尽量把所有可能的漏洞都找出来，哪怕有些是假的。验证阶段优化的是精确率，就是把真正可利用的漏洞从假阳性里筛出来。

文章说他们通过痛苦的教训学到了一件事：如果你让同一个 agent 既做发现又做验证，它会自我审查，过滤掉一些其实可以被利用的真阳性。这个现象可以理解，同一个 agent 验证自己的发现，很难做到客观，它找到的漏洞，它会倾向于为自己辩护，同时也会对不确定的发现过度怀疑。所以发现和验证必须分开，用两个独立的 agent 来做。

验证 agent 的核心设计要求是独立性。要在一个全新的容器里运行，不能跟发现 agent 共享文件系统，也不能共享对话历史。如果验证 agent 看到了发现 agent 的推理过程，它可能会直接同意，而不是做真正的独立验证。验证 agent 只应该拿到两样东西：发现 agent 给出的漏洞描述或 PoC，以及代码库本身，让它自己去找发现 agent 可能遗漏的缓解措施，比如上游的输入验证、认证检查、类型约束、或者根本不可达的代码分支。

提示策略方面，让验证 agent 预设每个发现都是假阳性，去找能证明这个假设的理由。这种对抗性验证让验证更严格，更难被原始发现的框架所锚定。如果一个发现通过了这种对抗性验证，才说明它确实值得重视。

从实际效果来看，加一个对抗性验证 agent，大约能把发现阶段的非可利用发现率减半。再进一步要求验证 agent 写出可运行的 PoC，可以把假阳性率降到接近零。两个措施加在一起，大幅降低了后续分级筛选和修复所需的工作量。

需要注意的是，PoC 没跑通，不代表漏洞不存在，只代表暂时无法证明可利用。这不能作为判断假阳性的充分条件。

第五步，分级筛选。

分级筛选这个词来自医疗急救，意思是按紧迫程度排优先级，决定先处理谁。在这里，就是对验证过的漏洞做优先级排序。

文章说，在以前，找漏洞需要很多人工精力，找到漏洞的工程师会同时做分级，因为样本量不大。现在，模型可以在一个上午找到一百个候选，分级就成了真正的瓶颈。而且分级做不好有一个很具体的后果：警报疲劳。如果你给产品工程师发了一堆重复的、严重程度虚高的报告，他们会停止认真看，包括那些真正需要紧急修复的。对于开源项目的维护者来说这个问题更严重，因为他们同时接收来自很多不同用户的报告。多个团队分享了同样的教训：如果我们给工程师发的报告里大多数是假阳性或者夸大的，他们就会失去对报告的信任，然后不再认真对待任何报告。

去重分两步。第一步用便宜的确定性规则快速过滤：同一文件、同一漏洞类别、行号在十行以内的，直接标记为重复。第二步让模型处理剩下的，做定性判断。同一根因用不同措辞描述，算重复；同一个漏洞在多个调用点各报了一次，算重复；一个全局缺失的防护措施被每个端点各报了一次，算重复。但是，同一文件里两个不同类型的漏洞，算独立发现；不同变量流向不同接收点，算独立发现；相似的缺失检查出现在两个端点但各自需要独立的修复，也算独立发现。

去重之后，评估每个漏洞的严重程度。文章给出了一个评估框架，包括六个维度：可达性，攻击者能不能从真实入口到达这段代码；攻击者的控制程度，不可信的输入能不能完整到达漏洞触发点；前置条件，触发漏洞需要什么特定条件；认证要求，未认证的用户也能触发吗还是需要登录用户甚至管理员；读写影响，攻击者只能读数据还是也能修改；影响范围，如果 PoC 成功，是影响一个用户还是所有用户，是一个租户还是整个平台。

文章给出了一个实用的出发点：零前置条件、可被未认证用户远程触发，高危或严重；有一两个前置条件或者需要认证，中危；三个以上前置条件或者只能本地触发，低危。然后根据自己系统的特点调整阈值。

模型容易虚报严重程度，主要原因是上下文不足：它不知道攻击者实际能控制哪些输入，或者看不到代码之外的防护措施，比如外层的访问控制网关或者 WAF。解决方法还是威胁模型，在分级阶段把威胁模型提供给分级 agent，明确告诉它哪类漏洞在你的系统里重要、哪类不重要，这样可以有效避免基于漏洞类别机械打分的问题，比如一看到 SQL 注入就打严重，但实际上这个注入点只有内部管理员能触达。

第六步，修复。

终于来到最后一步，修复是整个流程的最终目标，是把发现转化成真实安全提升的关键。

修复之前，文章强调要先写一个会失败的测试，然后实现修复，确认测试现在通过了，而且原有的测试套件没有被破坏。就是测试驱动开发的思路。不加测试，修复可能会悄悄退化，也很难事后证明这个漏洞曾经是真实存在的。

模型生成的补丁有几种可预测的失败模式。

第一种是只修症状不修根因。模型只在发现漏洞的具体调用点做了修改，但产生这个漏洞的根本原因还在，代码库其他地方的同类漏洞依然存在。文章说，在提示词里直接告诉模型"识别并修复根因"往往就能解决这个问题。然后让模型在两个层面查找变体：相同模式，就是代码库里其他地方有没有同样的问题；相同类别，就是一个有 SQL 注入的代码库，往往还有更多 SQL 注入，要主动去找。

第二种常见失败是补丁太激进。文章里有一个团队说，他们最常遇到的补丁问题是：推荐的补丁倾向于尽可能地限制，限制到会破坏与其他服务的连接。它解决了安全问题，但破坏了让服务正常运行的依赖关系。解决方法是在提示词里明确要求最小化补丁，专注于修复根因，不要顺手重构，不要清理代码风格，不要重新格式化，就做最小的改动。

补丁提交前还要做一个对抗性检查：让一个全新的发现 agent 以攻击者的视角去测试这个补丁，确认它真的有效，没有明显的绕过方式。

文章给出了一个验证补丁的阶梯，从便宜到贵依次是：代码能编译、新测试通过；原来的 PoC 不再有效，这能发现无效补丁；原有测试套件全部通过，这能发现过于激进的破坏性补丁；对抗性 agent 找不到新的攻击路径，这能发现不完整的补丁。

最后一点非常重要：虽然模型可以写补丁，但人还是需要对这个补丁负责。模型可能不了解团队即将有什么代码变更，不知道某个设计决策背后的原因，也可能不熟悉代码风格偏好。整个流程的目标是尽可能多地自动验证，让人工审查尽量聚焦在模型无法替代的判断上，而不是完全绕开人工审查。

好，六步全部讲完了。来说说对你实际工作意味着什么。

最重要的一条建议：在你扩大扫描规模之前，先把扫描之后的流程建好。很多团队犯的错误是不停地运行发现，结果发现了很多，都堆在那里没有被验证、分级和修复。文章里明确说：在你为更多扫描做预算之前，先为扫描之后的流水线做预算。

具体的上手路径是：选一个你负责的服务或包，从代码和文档里引导出威胁模型，有条件的话找一个了解系统的人做一次采访。然后投入时间搭建一个沙箱。接着运行发现。用独立的 agent 做验证，高危以上的全部人工审查。修复，并为每个修复加测试。然后定期重新扫描，或者每次代码有重大变化时扫描。

要避开的坑：威胁模型不清晰导致大量误报，让工程师失去对报告的信任；发现和验证用同一个 agent 导致真阳性被自我过滤；去重做得不好，同一个漏洞报了五六次，淹没了真正重要的发现；补丁太激进，解决了安全问题但破坏了系统功能；没有为每个修复写测试，导致修复悄悄退化。

我个人认为，这篇文章最值得借鉴的架构思想是把召回率优先和精确率优先这两个目标，分开给不同的 agent 来实现。这个设计原则不只适用于安全扫描，对于任何需要先大范围发现再精细筛选的 AI 工作流，都是一个非常实用的模式。不要让一个 agent 同时做目标相互矛盾的两件事，这会让它陷入一种内在的自我妥协。

还有一个让我印象深刻的数字：一千五百九十六个发现，九十七个修复。这不是 AI 能力的问题，这是工程流程的问题。模型已经可以帮我们找到大量漏洞，但把漏洞转化成修复，需要人和流程，这就是这篇文章试图填补的那个鸿沟。

来做个总结。这篇文章的核心信息是：用大模型发现代码漏洞已经不是瓶颈，瓶颈在验证、分级和修复。解决这个问题的方法是六步流程：威胁建模、沙箱搭建、发现、验证、分级筛选、修复。其中最反直觉的两点是：给发现 agent 的提示词越简单越好，详细检查清单会适得其反；发现和验证必须分开，用独立的 agent，才能避免自我审查过滤掉真阳性。

我个人觉得，这篇文章标志着 AI 辅助安全工程进入了一个新阶段。问题不再是"AI 能不能帮你找漏洞"，而是"你怎么设计一套流程，把 AI 的发现能力转化成真正被修复的安全漏洞"。对所有做安全工作、乃至更广泛地做 AI agent 落地的工程师来说，这个问题都值得认真思考。
