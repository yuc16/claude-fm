## Script

欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是 1M context is now generally available for Opus 4.6 and Sonnet 4.6,翻译过来就是,百万级上下文窗口现在已经在 Opus 四点六和 Sonnet 四点六上全面可用了。这篇文章发表于二〇二六年三月十三日,距离我们今天录这期节目,也就是二〇二六年六月二十日,已经过去了大概三个多月。三个多月不算特别久,但是在 AI 这个领域里,三个月已经足够发生很多变化了,所以我们今天聊的时候,会更多地聚焦在这个能力本身的设计逻辑和工程价值上,而不是纠结于具体某个版本号是不是还在最新。如果你现在去查文档,具体的限速数字可能已经有调整,但是这次更新背后体现出来的产品思路,我觉得是非常值得拆开来好好讲一讲的。

先说说为什么这件事值得我们花二十多分钟来聊。如果你平时用大模型写代码,或者做一些需要喂很多资料进去的任务,你肯定遇到过这种情况,就是聊着聊着,模型开始说胡话了,或者它明明前面已经告诉过它的东西,它又问你一遍,甚至开始编造一些根本不存在的函数名或者变量名。这种现象背后,很大一部分原因就是上下文窗口不够用,模型记不住那么长的对话,或者为了塞进更多东西,系统不得不把早期的对话内容做压缩,甚至直接砍掉。这篇文章讲的就是,Anthropic 把百万级别的上下文窗口,从一个需要额外付费、需要发测试请求头才能用的实验性功能,变成了一个人人都能用、而且价格跟正常用法完全一样的标准能力。这听起来是个挺技术性的更新,但是它解决的其实是一个所有重度使用大模型的人都会撞上的痛点,就是模型的记忆力问题。

我们先讲讲背景,也就是说在这次更新之前,行业里大家面对长上下文是个什么状态。其实从二〇二三年开始,长上下文就一直是各家模型厂商比拼的一个重要维度。一开始大家比的是四千 token,然后是三万二千,然后是十万,后来逐渐有模型号称支持到几十万甚至上百万 token。但是这里有一个很关键的问题,就是支持长上下文和真正能用好长上下文,完全是两件事。很多模型理论上能接收很长的输入,但是一旦你真的塞进去几十万 token 的内容,它就开始出现我们行话里说的那种"大海捞针"失败的情况,也就是说你藏在文档中间某个角落的一句关键信息,模型压根没注意到,或者注意到了但是没能正确地跟其他信息关联起来。所以即便窗口标的是一百万,实际能用的有效容量,可能远远小于这个数字。

另外一个行业里普遍存在的状态,就是长上下文往往伴随着额外收费,或者叫长上下文溢价。逻辑也不难理解,因为处理更长的输入,对模型的计算资源消耗确实更大,所以厂商往往会设计一个阶梯式的定价,比方说请求超过二十万 token 之后,单价就会跳一个档次,变得更贵。这种设计从商业角度讲是合理的,但是从开发者体验角度讲,其实挺糟糕的,因为它意味着你在写代码调用 API 的时候,还得自己额外计算一下,我这次请求会不会不小心超过那个临界点,导致账单突然变贵。再加上速率限制这件事,很多厂商对长上下文请求会单独设置更严格的限流,这就导致开发者在真正想要发挥长上下文能力的时候,反而处处受限,体验非常割裂。

在这样的背景下,Anthropic 这篇文章里讲的几个变化,其实每一条都是在针对性地解决上面这些痛点。我们一条一条来拆解。

第一个核心点,就是定价方式的彻底简化。文章里说得很直接,百万 token 的整个窗口,统一按标准价格计费,没有任何倍数加价。具体数字是,Opus 四点六的价格是每百万 token 输入五美元,输出二十五美元。Sonnet 四点六的价格是每百万 token 输入三美元,输出十五美元。这里最关键的一句话是,一个九十万 token 的请求,和一个九千 token 的请求,按照完全相同的单价计费。这句话听起来很简单,但是如果你站在工程师的角度想一想,这个意义其实挺大的。打个比方,这就像你去坐地铁,以前的规则是,坐十站以内是一个价格,但是一旦超过十站,后面每一站都要按双倍价格计费,你心里就会一直惦记着这个临界点,生怕一不小心坐过站多花钱。现在的规则变成了,无论你坐多远,每一站的单价都完全一样,你就可以彻底放心地规划你的路线,不用再为价格表去做心理建设。对应到写代码的场景里,就是你不用再在业务逻辑里专门写一段判断逻辑,去防止请求不小心越过那个长上下文的价格门槛,你可以直接把这件事完全交给自然的业务需求去决定,而不用迁就计费规则。

第二个核心点,是速率限制的统一。文章里提到,你的标准账户吞吐量,在整个上下文长度区间内都是适用的。这句话翻译一下就是说,以前可能存在的情况是,你账户额度允许你每分钟处理多少 token,但是一旦你的单次请求超过某个长度,系统会单独给你设置一个更紧的限流,导致你即便账户额度还有富余,也没法真正发挥出来。现在这个区别被取消了,不管你的单次请求是短的还是长的,都按照你账户本身的额度去算,不会因为你用了长上下文就被额外限制。这一点对于做批量处理或者高并发场景的团队来说,意义是非常实际的。举个例子,假如你是一家做法律科技的公司,你们的产品需要把整本几百页的合同喂给模型去做条款审查,如果长上下文请求要被单独限流,那意味着你的吞吐量天花板会被这个最耗资源的场景死死卡住,你可能得专门为长上下文请求单独申请配额,或者排队处理。现在这个限制被拿掉之后,你完全可以按照正常的业务并发量去设计系统架构,不需要为了适配一个特殊的限流规则而专门去做额外的排队和退避逻辑。

第三个核心点,是媒体处理能力的提升。原文提到,单次请求能处理的图片或者 PDF 页面数量,从一百提升到了六百,整整六倍。这个数字听起来挺技术化,但是我们换个场景来理解会更直观。假设你现在要做一个企业知识库审计的项目,客户给你一份招股说明书,或者一份完整的尽职调查报告,这种文件动辄就是几百页,里面还夹杂着大量的扫描图片、表格截图、手写签名页。以前一百页的上限,基本上意味着你拿到一份三百页的 PDF,得先自己写脚本把它拆成三段,分三次请求分别处理,然后再想办法把三次的结果拼接起来,还得操心拆分的边界刚好把一个关键条款切成两半导致语义断裂的问题。现在六百页的上限,意味着绝大多数常见的合同、年报、技术手册,你可以一次性整份扔进去,模型自己去理解整篇文档的上下文关联,不需要你再做这种繁琐的人工拆分和拼接。这种体验上的提升,看起来只是数字翻了六倍,但其实它直接消除了一大类原本必须靠工程手段去绕过的麻烦事。

第四个核心点,是不再需要发送测试版请求头。这是一个很小的技术细节,但是它反映出一种很重要的态度变化。在这次更新之前,如果你想用超过二十万 token 的长上下文,你得在 API 请求里专门加一个标记自己是测试版功能的请求头,相当于你得明确地告诉系统,我知道这是个实验性功能,我愿意承担相应的风险。现在这道门槛被去掉了,超过二十万 token 的请求自动就能生效,完全不需要任何特殊标记。而且文章特别提到,如果你之前已经在代码里加了这个测试版请求头,现在它会被直接忽略,不会报错,所以你完全不需要去改动现有的代码。这种处理方式我觉得挺贴心的,它考虑到了已经在生产环境跑着的真实业务代码,不会因为底层能力升级了,就强迫所有开发者去做一遍代码迁移。这其实是一个很值得我们学习的工程态度,就是当你把一个实验性功能转正的时候,尽量做到对已有用户无感升级,而不是让大家都得重新适配一遍新规则。

第五个核心点,也是我觉得最值得展开讲的一个,就是这次更新里专门提到的一个评测数字,叫做 MRCR 二代,英文全称是 Multi Round Co reference Resolution,中文大致可以理解成多轮指代消解测试。这个测试到底在考什么呢,我们用一个生活化的场景来类比一下。想象你在一个微信群里聊了特别长的一段对话,群里好几个人轮流发言,中间夹杂着各种话题切换,有人开玩笑用代词指代前面提到的某个人或者某件事,比如说有人说"那他后来怎么处理的",这个"他"到底指的是谁,取决于你有没有真的认真读完前面所有的发言,并且在脑子里把这些指代关系全部理清楚。MRCR 这个测试,本质上就是在很长的文本里塞进大量类似的指代关系和细节,然后看模型能不能在读完几十万甚至上百万 token 之后,依然准确地把这些指代关系对应回正确的内容。这个测试之所以重要,是因为它直接对应了我们前面提到的那个"大海捞针"问题的更难版本,不是简单地找一个孤立的关键词,而是要在大量相似干扰信息中,准确地把多处提及的同一个事物关联起来。文章里说,Opus 四点六在这个测试上的成绩是百分之七十八点三,在同等上下文长度下是所有前沿模型里最高的。这个数字直接回答了我们前面提到的那个核心疑问,就是窗口大不代表真的能用好,Opus 四点六给出的答案是,它不仅窗口大,而且在这个窗口里真的能保持比较高的准确召回能力。

讲完这五个核心点,我们再往深一层去想一想,这些变化加在一起,到底改变了什么。文章里有一句话我觉得特别精炼,它说,以前长上下文工作必须依赖的那些工程手段,比如有损的摘要压缩,以及主动清空上下文,现在都不再需要了,完整的对话可以原样保留。这句话其实揭示了一个很重要的工程范式转变。我们简单回顾一下,以前为了让模型在有限的窗口里坚持工作更久,工程师们想了很多办法,其中一个常见做法就是摘要压缩,也就是定期让模型自己把前面的对话内容总结成一段更短的摘要,然后用这段摘要代替原始的完整对话,继续往后塞新的内容。这种做法听起来挺聪明,但是它本质上是有损的,意思就是压缩的过程中,一定会丢掉一些细节,而你永远不知道丢掉的那部分细节,会不会恰好是后面某个关键决策需要用到的信息。另外一种做法叫上下文清空,简单说就是定期把历史对话直接砍掉,只保留最近的一部分,类似于我们手机里聊天软件的内存清理,旧的不去新的不来,但代价就是早期讨论过的内容彻底没了踪影。

如果你曾经用 agent 也就是智能代理去跑过比较长的自动化任务,你应该深有体会,这种压缩和清空带来的副作用有多麻烦。举个例子,假如你让一个 agent 去帮你完成一个比较复杂的代码迁移任务,这个任务需要先扫描整个代码库,理解项目结构,然后逐个文件去做改造,改造过程中可能会调用各种工具,比如执行测试,读取报错信息,再根据报错去调整代码,这样反反复复循环很多轮。整个过程里,会产生大量的工具调用记录,运行结果,还有 agent 自己的中间推理过程。如果上下文窗口不够大,系统就得不断地把早期的这些记录做摘要或者直接丢弃,结果就是 agent 可能在第五十轮的时候,已经完全忘记了它在第十轮里发现的某个重要约束条件,于是它开始重复犯同样的错误,或者做出跟前面逻辑矛盾的决策。这种问题在长链路自动化任务里其实是非常常见的失败模式,很多团队为了缓解这个问题,专门去设计各种外部记忆系统,把重要信息存到数据库里,再在合适的时机重新喂给模型,这其实就是工程师在用人工的方式,去补偿模型本身记忆容量不够的缺陷。现在如果整个任务的全部历史轨迹,包括工具调用,观察结果,中间推理,都能完整地放进一个百万级的窗口里,而且不会因为窗口大就被多收钱或者被限流卡住,这意味着这种外部记忆系统的必要性会大大降低,至少对很多中等复杂度的任务来说,你完全可以依赖模型原生的长上下文能力去解决问题,而不需要再自己搭一套额外的基础设施。

我们再聊聊文章里专门提到的 Claude Code 这个产品场景。Claude Code 是 Anthropic 推出的一个面向开发者的编程助手工具,这次更新提到,Max、Team 还有 Enterprise 等级的用户,在使用 Opus 四点六的时候,百万上下文会默认自动开启,而且这个能力以前是需要额外消耗用量配额才能用的,现在变成标配了。文章里特别提到一个词,叫做 fewer compactions,意思是更少的压缩次数。这里的压缩指的就是我们前面讲的那种,当对话太长的时候,系统会自动把早期内容总结精简,以便腾出空间继续对话。如果你平时用过编程助手类工具做比较长时间的会话式开发,你应该遇到过这种情况,就是聊到一半,工具突然提示说要做一次上下文压缩,压缩完之后你明显感觉到它好像忘记了前面提到的某个命名约定,或者忘记了你早些时候特别强调过的某个限制条件,你得重新跟它解释一遍。百万上下文默认开启之后,这种压缩发生的频率会大幅降低,意味着你跟编程助手之间那种长时间、连续性的协作,会变得更顺畸,中间不会频繁出现那种"它好像突然失忆了"的尴尬时刻。

那么这些变化到底对我们听众的实际工作意味着什么,我们花一点时间聊聊实践应用这部分。

第一个层面,是关于成本控制思路的调整。以前如果你的团队在设计一个需要处理长文档的产品功能,你可能得专门做成本预估的时候,把长上下文溢价这个因素考虑进去,甚至要在产品设计上做一些妥协,比方说限制用户单次上传文档的页数,以避免触发更贵的计费档次。现在既然长上下文不再额外加价,你完全可以把这个限制重新评估一遍,看看是不是可以放宽,让产品体验更自然。当然我要提醒一句,虽然单价没有变化,但是处理更长的输入,消耗的 token 总量肯定还是会更多,所以总的费用还是会随着输入长度线性增长,这不是说长上下文变成免费了,而是说计费方式变得更简单更可预测了,你不用再去担心那种因为跨过某个临界点而突然跳档的情况。

第二个层面,是关于产品架构设计的简化机会。如果你的团队之前为了应对长上下文限制,专门搭建了一套文档分片处理的流水线,比如把长文档切块,分别处理,再合并结果,现在可能值得重新评估一下这套流水线是不是还有必要继续维护。当然这里我要提一个我个人的看法,我觉得即便现在能处理六百页的 PDF,分片处理这件事在某些场景下依然有价值,比方说当你的文档实在特别长,超过了百万 token 的总量上限,或者当你的业务逻辑本身就需要对文档的不同部分做完全独立的处理逻辑,这时候分片依然是合理的设计。但是如果你之前做分片仅仅是因为受限于长度上限,而不是出于业务逻辑本身的需要,那么现在确实是个好时机,去重新审视一下,是不是可以用更简单的架构,直接整体处理,从而减少系统的复杂度,减少出错的环节。

第三个层面,是关于长链路 agent 任务的设计思路。如果你正在做或者计划做一些需要模型自主执行多轮操作的自动化系统,比方说自动化的代码审查机器人,自动化的客服处理流程,或者自动化的数据分析管道,我建议你可以重新思考一下,是不是真的需要那么复杂的外部记忆管理机制。我个人的建议是,可以先尝试用最简单的方式,把完整的执行轨迹直接放进上下文里跑一遍,观察模型在长轨迹下的表现是不是已经足够好,如果效果已经达标,那就没有必要再额外引入一套外部记忆系统增加维护成本。但如果你的任务轨迹特别长,长到即便有百万级窗口也装不下,或者任务对早期信息的引用频率特别低,那么适当地保留一些精简记忆的设计,依然是合理的工程选择,这是一个需要根据实际场景去权衡的问题,不是说有了大窗口就一定不再需要任何记忆管理。

第四个层面,我想提一下关于评测这件事给我们的启发。文章里专门强调了 MRCR 二代这个评测成绩,我觉得这其实是在提醒所有做相关产品的人,窗口大小这个参数,本身并不是一个可以直接拿来做能力承诺的指标。如果你在评估要不要把某个长上下文能力用到你的生产环境里,我建议不要只看官方宣传的窗口数字,而是要去找类似 MRCR 这样专门测试长文本理解准确度的评测结果,甚至最好是拿你自己业务场景里真实的长文档,去做一些有针对性的小规模测试,看看模型在你的具体场景下,是不是真的能够准确地抓住分散在长文档各处的关键信息。这种针对性的验证,我觉得比单纯相信窗口数字本身要靠谱得多。

第五个层面,是关于跨平台部署的一点提示。文章末尾提到,这次百万上下文能力,不仅在 Anthropic 自家平台上可用,在亚马逊云的 Bedrock,谷歌云的 Vertex AI,还有微软的 Foundry 上也都同步支持。这对于那些已经在某个特定云平台上构建了业务的团队来说是个好消息,意味着你不需要为了用上这个新能力,被迫迁移到另一个云厂商的生态里,你可以继续留在你熟悉的基础设施上,直接享受到这次更新带来的好处。

我们花了不少时间把这篇文章里的几个核心变化都拆解了一遍,现在做个简单的总结。这篇文章的核心信息其实很清晰,就是百万级上下文窗口,从一个限定场景、额外收费、需要特殊标记才能用的实验性功能,变成了 Opus 四点六和 Sonnet 四点六的标准能力,定价统一,限流统一,媒体处理能力大幅提升,而且在 Claude Code 里也默认开启,同时还有 MRCR 二代评测成绩作为背书,证明这个大窗口不只是数字好看,而是真的能在长文本里保持比较高的信息提取准确度。

我个人的看法是,这次更新最有意思的地方,不在于窗口本身的大小,因为长上下文这件事行业里已经卷了挺久了,真正的亮点在于 Anthropic 这次选择把它从一个特殊功能,彻底变成了默认的基础能力,而且配套去掉了价格和限流上的种种附加门槛。这种"把高级功能下放成标配"的产品策略,我觉得反映出他们对自己模型在长上下文场景下真实表现的信心,毕竟如果模型本身在长文本理解上的准确度不够高,贸然把这个功能无门槛地推给所有用户,反而容易暴露出更多翻车的场景。所以这次更新背后,其实也是对 MRCR 这类评测成绩的一种间接呼应。当然我也要提醒大家,具体的定价数字和限速规则,可能随着时间推移会有调整,大家如果真的要在生产环境里依赖这个能力,还是要去看最新的官方文档,以确保自己拿到的是当下最准确的信息。
