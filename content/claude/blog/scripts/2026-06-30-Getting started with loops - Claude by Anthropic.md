嗨，各位朋友，欢迎收听 Claude FM。我是你们的主播。今天这期节目要聊的内容，跟每一位正在日常工作中使用编程助手、或者正在琢磨怎么让自己团队的开发流程更自动化的朋友都有关系。

我们可能都已经习惯这样一种工作方式：打开 Claude Code，敲一段提示词，让它帮我修个 bug、加个功能，它啪地一声给出代码，我们检查，可能再来一轮。这其实就是最基本的“一轮问答”。但如果任务复杂一点，比如要改完代码还要跑测试，还要看浏览器里到底对不对，还要处理 CI 的失败，那光靠我们手动一句一句地戳，效率就上不去了，而且我们自己就成了那个瓶颈。

所以今天这期节目，我会根据 Anthropic 刚发布的一篇文章来做一次深度的解读和展开。这篇文章的标题是 Getting started with loops，我把它翻译成“循环入门指南”。它发表于二〇二六年六月三十日。也就是说，就在刚刚过去的那一周，这还是热乎乎的新内容，所以非常适合当下马上去理解和使用。

整篇文章的核心就一个词：循环，英文叫 loop。但它讲的不只是一个简单的重复执行，而是 Claude Code 团队对于“如何让一个编码智能体在一个明确的任务边界里持续地、自我修正地工作”这一整套设计思路。我们接下来会看到，从你平时发出的那一条 prompt 开始，其实已经进入了一个最基本的循环，只不过这个循环是手动驱动的。而通过逐步叠加目标设定、时间触发、事件触发，以及多个智能体的协同，你最终可以搭建出一个几乎不用人管的自动工作流。

说实话，对于正在收听节目的你，如果你是一名软件工程师或者 AI 相关的从业者，这篇文章可能就是你从“偶尔用用编码助手”跨越到“把助手变成自动化的同事”的一个关键拐点。所以今天我会用非常通俗的方式，把原文里提到的四种循环模式一个个讲清楚，并且结合大量你可以马上上手的例子，同时也讲一讲使用的时候要注意哪些坑，怎么控制成本，怎么保证代码质量。

好，那我们先从一个最基本的问题开始：到底什么是循环？这文章里的定义特别清晰。它说，循环就是智能体重复执行工作周期，直到某个停止条件被满足。英文原话是 agents repeating cycles of work until a stop condition is met。你可能会觉得，这不就是死循环嘛？不一样，因为 stop condition 是我们设计的。而且这个循环并不是简单地重复一个动作，它是“感知、行动、检查、再决定继续还是停止”这样一个闭环。这其实非常接近我们程序员平常自己工作的方式：写了一段代码，编译或者跑测试，如果失败就再看、再改，直到通过。

但放到 Claude Code 里，这件事需要被显式地建模出来，因为模型本身在执行一个指令时，它很容易在认为“差不多行了”的时候就停下来，或者不知道该不该继续。而循环机制就是给它划定了边界和规则，让它能在一个可控的范围内反复迭代。

文章把循环根据触发方式和停止方式分成了几种类型，我们就一个一个来看。

第一个类型，也是每个人都在用、但可能没意识到的，叫基于回合的手动循环。它触发的条件就是你在 Claude Code 里发出一条提示词，也就是一个 prompt。停止的条件是 Claude 判断它已经完成了你交给它的任务，或者它觉得需要你提供更多上下文了。文章里把这个叫作 agentic loop，也就是最基础的智能体循环。

我给你一个很具体的场景。假设你对 Claude 说：“帮我给我的网站加一个点赞按钮。”这时候，Claude 会自己去读你现有的代码，找到相关的前端组件，创建一个按钮，加上状态管理的逻辑，可能顺手写个测试，然后告诉你“我做好了”。在这个过程中，它其实已经内部进行了一个小循环：收集上下文、采取行动、检查自己的工作、如果需要就再改一下，最后交给你。但这个循环只有一个回合的样子，一旦它把结果交回来，它就停了。剩下的验证工作得靠你——你手动打开浏览器，点点按钮，看看控制台有没有报错，确认样式没崩。

发现没有，这里有个很大的问题：Claude 认为自己完成了，但其实它并没有真正验证自己的工作成果。它只是基于代码编辑成功了就停止了。所以，文章里给出了一个非常实用的改进方法，就是把你的验证步骤也编码进去。这就引出了一个关键工具，叫做 SKILL 文件，技能文件。

你可以在项目里创建一个叫做 SKILL.md 的文件，里面详细描述怎么验证一件事情是真的做好了。而且这个技能文件不仅仅是文字说明，它可以引用外部工具或者连接器，让 Claude 能够真的去看、去测量、去交互。

文章举了一个例子非常生动。它创建了一个名为 verify-frontend-change 的技能，就是验证前端修改。里面的内容大致是这样的：永远不要仅仅因为代码编辑成功就报告 UI 修改完成。你要像人肉审核一样去验证。第一步，启动开发服务器，在浏览器里打开修改过的页面。第二步，直接跟这次改动交互，比如对于新的按钮，你要真的点击它，确认状态转换正确，并且截取前后对比的截图。第三步，检查浏览器控制台，确保没有新的错误或者警告。第四步，用 Chrome DevTools 的 MCP 协议运行性能追踪，审计核心网页指标。如果任何一步失败，就修复问题，然后从第一步重新来——不要提交一份没有经过完全验证的工作。

你想象一下，如果你每次都写这样的技能文件，再配合上手动的 prompt，这样 Claude 在每次交回结果之前，自己就能跑一个完整的自我验证循环。它就不仅仅是一个“代码改动器”，而是一个真正能确认改动是否有效的协作者。对于咱们这些长期被验证步骤折磨的工程师来说，这相当于把最烦人的那部分重复性体力活外包了出去。

文章管这种方式叫改进验证步骤，通过编码手动检查步骤，让 Claude 能在端到端确认自己的工作。而且它特别强调了一句话：检查越是量化的，Claude 就越容易完成自我验证。这一点我完全认同。你自己想想，如果你只是说“检查一下有没有问题”，Claude 很难定义什么叫“没问题”。但如果你说“运行测试，必须全部通过；浏览器控制台不能有任何报错；Lighthouse 性能评分必须高于九十”，这些就是硬性的通过与否，模型很容易判断。

所以，手动循环虽然是最简单的，但它引入技能文件之后，能力会上一个大台阶。你可以把团队积累的那些人工检查经验全部沉淀成技能，然后每次发 prompt 的时候引用一下，就能让单次循环的质量大大提高。

好，接下来我们进入第二个类型的循环，也是这篇文章花了较大篇幅讲的，叫做基于目标的循环。触发方式依然是手动的、实时的 prompt，但停止条件不再是 Claude 自己去判断“我觉得行了”，而是你明确定义了一个目标必须达成，或者达到了你设定的最大尝试次数。

这个功能是通过一个叫做 /goal 的命令来实现的。什么时候该用这种循环呢？当你发现一个任务是有明确的可验证的退出标准的，比如说“把所有单元测试通过率提高到百分之百”、“把首页加载时间降到两秒以内”、“让代码重复率降到百分之五以下”。这些任务光靠一轮指令往往不够，因为 Claude 可能改了一次，测试还是失败，它就得重新分析、重新修改。而如果你只是普通地问它，它可能稍微改一下就觉得足够了，提前结束。但如果你告诉它“不达目标不罢休，最多尝试五次”，它就会进入一个自我驱动的迭代过程。

这里的技术实现其实挺巧妙。当 Claude 认为自己完成了、想要停止循环时，有一个额外的评估模型会跳出来检查你设定的那个目标到底达成了没有。如果没有，它会直接把 Claude 拽回去继续干活，一直到目标真的满足，或者重试次数用光了为止。这就解决了模型自我判断“足够好”不够严格的问题。

文章举了一个非常具体的例子：/goal get the homepage Lighthouse score to 90 or above, stop after 5 tries. 翻译过来就是“目标：让首页的灯塔评分达到九十或以上，最多试五次”。你看，目标非常明确，是确定性标准。Claude 可能第一次改动，分数从七十涨到八十，它觉得不错了想提交，评估模型一看，九十没到，给我回去继续。于是它再优化图片、调整代码分割、去掉阻塞渲染的资源，第二次八十五，还是不够，再试第三次，直到九十。如果五次还没到，就停下来，把当前最好的结果交给你，并告诉你没能达到目标。

这种确定性标准为什么这么有效？因为数字不会骗人。它不像“让代码更优雅”这种模糊的指令，评估模型可以很简单地通过运行测试或者调用 API 来获取分数，然后判断。这就把循环的退出条件从主观变成了客观。

我个人认为，基于目标的循环是这四种循环里最能直接提高产出的一个，因为它非常适合我们日常开发中大量存在的“有指标可衡量”的任务。你想想，你平时的 Jira 工单里有多少是可以被量化的？代码覆盖率、性能指标、可访问性评分、安全漏洞数量、构建时间等等。以前你可能要自己一遍遍改，一遍遍测。现在你用 /goal 设定好，让它在后台自己去冲，你在旁边喝着咖啡等着看结果，或者说等它最后一次失败的时候再来介入，这会大大降低你的心理负担和任务切换成本。

但是，这里也要注意文章里提到的使用管理技巧。你要设置明确的最大尝试次数，比如“试五次就停”。因为如果不设上限，它可能一直在一个不可能解决的问题上死磕，白白烧掉大量 token。同时，设定完成标准时要确保它是真的可以自动检测的，不要用模糊的描述。

接下来我们聊聊基于时间的循环。这类循环的触发条件不再是你的主动 prompt，而是你事先指定的一个时间间隔。停止条件要么是你手动取消，要么是它所处理的工作自然完成，比如 PR 已经合并了，或者一个消息队列已经清空了。

这个功能在 Claude Code 里对应的是 /loop 命令。用法很简单，比如 /loop 5m check my PR, address review comments, and fix failing CI。翻译过来就是“每五分钟检查我的 PR，处理审查评论，修复失败的 CI”。你看，这其实解决了一个非常经典的痛点：你提交了一个 PR，然后你去做别的事了。过了一会儿，同事留了评论，或者 CI 挂了，你如果不定时去刷一下，可能过了好几个小时才发现。现在，你可以开一个基于时间的循环，让它每五分钟去拉一下 PR 的状态，如果有人留言，它就去分析并尝试修改代码；如果 CI 失败，它就去读日志、改代码、再推送。直到 PR 合并，这个循环的任务完成了，你就可以把它关掉。

文章还特意提醒了，/loop 是运行在你的电脑上的，如果你的电脑休眠了或者关机了，它也就停了。如果你希望它在云端一直跑，可以使用 /schedule 命令来创建一个例行任务。这就引出了我们今天要讲的第四种，也是最强大的一种循环，叫做主动循环，或者说事件驱动的循环。

主动循环的触发条件是一个事件或者是一个日程安排，而且在它运行的整个过程里，不需要有真人在实时盯着。每一个子任务当它的目标达成后会自动退出，但整个例行任务本身会一直运行，直到你手动把它关闭。这种循环最适合处理一类问题，就是“持续到来的一系列明确工作项”，比如 bug 报告、问题的分类和派发、代码迁移、依赖升级等等。这些东西的共同特征就是，工作会源源不断地流入，而处理每一件的模式又是相似的。

文章用了一个组合的例子来说明这种主动循环到底能有多强。它把前面提到的那些原语，再结合上 Claude Code 的其他一些功能，比如 auto mode 自动模式、dynamic workflows 动态工作流，这些组合在一起，就能变成一个长时间运行的自动处理管线。

具体怎么用呢？文章给了一段非常直观的 prompt。它是这么写的：/schedule every hour: check #project-feedback for bug reports. /goal: don't stop until every report found this run is triaged, actioned, and responded to. When fixing a bug, use a workflow to explore three solutions in parallel worktrees and have a judge adversarially review them. 我来慢慢解释一下这段话。

首先，/schedule every hour 创建了一个每小时运行一次的例行任务。它去检查某个频道里有没有新的 bug 反馈。接下来 /goal 定义了这次的停止条件：这次运行中发现的所有报告必须被分类、被处理、被回复了，才算是完成。注意，这是一个整体目标，而不是针对某一个 bug，因为新的报告可能发现三个，那这三个都得处理完。后面又加了一句：在修 bug 的时候，用动态工作流探索三个并行的解决方案，再让一个评判模型像对手一样去审查它们。

这个设计思路非常了不起。因为单个 agent 有时候会陷入思维定势，它想出来的第一个方案可能就直接用了，但这个方案未必是最好的。但现在它可以同时在三个独立的 worktree 里尝试三种不同的修复方式，然后由另一个 agent 扮演挑剔的审查员，专门找每份方案的缺点，选出最好的一个。这基本上就是模拟了我们人类团队里的代码评审和方案对比过程，但是全自动的，而且速度极快。

你可能会意识到，这种组合拳一旦用好了，它就不只是提高效率的问题，它可能会改变你组织开发任务的方式。以前可能是“我写代码，助手给建议”，现在变成“我定义任务流和验收标准，一群智能体协作完成，我只在它们搞不定或者需要决策的时候出现”。当然，文章里也提到了，像 dynamic workflows 和 /schedule 这类功能现在还处于研究预览阶段，但它已经给我们展示了这种范式的可行性。

到这里，我们已经把文章的核心内容——四种循环——都梳理清楚了。从手动回合循环，到目标驱动的循环，到时间驱动的循环，再到事件驱动的主动循环，复杂性是递增的，但它们并不是互相替代的关系，而是可以层层叠加、组合使用的。

接下来，我想花一些时间聊聊这篇文章对我们实际工作的启发。也就是说，听完这些概念，你到底该怎么迈出第一步。

文章给的建议非常接地气。它说，要开始用循环，先看看你现在已经在做的那些工作。挑一个你是瓶颈的任务，然后问自己几个问题：我能不能把这个任务里的验证检查写成一个明确的步骤？这个任务的目标是不是足够清晰，清晰到可以量化为停止条件？这项工作是会按照某个时间规律不断出现的吗？如果是，那你就可以尝试把它交给循环。

我打个比方，也许你每天上班第一件事是打开邮箱，查看昨晚的 CI 运行报告，把失败的 build 分配给对应的同事，或者在简单的失败上自己顺手修一下。这个过程其实就是一个可以被循环化的任务。你完全可以用 /schedule 写一个每天早晨帮你检查 CI 的例行程序，它能读报告，用固定规则分类，简单的错误自己修，复杂的情况自动创建一个 issue 并 at 相关的人。你只需要看一下总结和它处理不了的少数情况就可以了。这样，你早上宝贵的深度工作时间就不用被打断了。

但是，我们在设计这些循环的时候，必须要考虑到文章后面特别强调的关于代码质量和 token 使用的问题。因为这些循环一旦跑起来，如果设计得不好，不但可能给你搞出一堆烂代码，还会吃掉大量的 token 配额。

文章花了很大的篇幅来讲怎么“管理”这些循环，我把它归纳成两个方面：一是怎么保证产出的质量，二是怎么控制成本。

质量方面，有几个核心建议。第一，保持代码库本身的整洁。因为 Claude 会延续你项目里已有的模式和惯例，如果项目本身一团乱麻，那循环产生的代码也很可能是混乱的。这就意味着，引入自动循环之前，最好先花一些精力把项目的基本架构、命名规范、测试模式都梳理清楚。第二，一定要给 Claude 自我验证的途径。也就是我们前面说的技能文件，把你的标准、你的测试用例、你的检查清单都编码进去。只有让它在每次提交前都能严格自检，你才可以放心的让它自动运行。第三，让文档容易触及。如果你的项目用了很多框架和库，确保最新的文档能被索引到，这样 Claude 在修改时就能参考正确的用法，而不是自己瞎猜。第四，用第二个智能体去做代码审查，也就是 peer review。因为主要干活的那个 agent 往往会在自己的推理里形成偏见，越看自己的代码越顺眼。安排一个独立的审查员，用全新的视角去看，这样能抓出很多隐藏的问题。Claude Code 里可以用内置的 /code-review 技能，或者是整合 GitHub 上的代码审查。

文章还有一句我非常喜欢的总结：当某个单独的结果不符合标准时，不要只修复这个单独的问题，试着把它编码成规则，从而改进整个系统，让未来所有的循环迭代都能因此受益。这其实就是把智能体循环当成一个可进化的系统来对待。

成本控制方面，文章更是给出了足足七条建议，非常实操。我挑几条最重要的和你分享。

第一条，为任务选择合适的原语和模型。不是所有任务都需要多个 agent 或者循环。有些非常简单的工作，比如把一个字符串替换一下，没必要启动一个复杂的目标循环，直接一句话就够了。而且针对一些简单的任务，你可以使用更小更快更便宜的模型，避免“杀鸡用牛刀”。

第二条，定义清晰的完成标准和停止条件。这其实是一句双保险。完成标准让循环更早走上正确的解法路径，停止条件防止它在一个无解问题上无限空转。两者搭配得好，token 消耗会非常可控。

第三条，大规模运行之前，一定要先做小规模试点。文章专门提到，动态工作流可能会一次性启动上百个 agent。如果你没有在少量数据上测试过，很可能一跑起来就把你的配额吃光或者造成混乱。所以，先用一小部分任务样本试一下，看看流程跑不跑得通，每次消耗多少 token，然后再扩大。

第四条，对确定性工作使用脚本。意思是，如果一个任务是高度确定的，比如把某个表单填充好然后导出 PDF，与其让模型每次都重新推理一遍怎么填，不如写一个填表脚本，通过技能文件让 Claude 遇到这个任务时直接运行脚本。因为执行脚本便宜多了，模型推理可是按 token 收费的。文章举的例子就是一个 PDF 技能，它包含一个填表脚本，Claude 每次就直接调脚本，而不是重新生成代码。

第五条，不要比实际需要更频繁地运行循环。回到那个 PR 检查的例子，如果同事的审查评论通常一两个小时才来一次，你没必要每两分钟就去查一次。匹配好你的轮询频率和外部变化的频率。

第六条，善用监控命令。Claude Code 提供了 /usage 命令，可以分解出近期各种技能、子智能体和 MCP 连接分别用了多少 token。/goal 不带参数可以显示当前任务已经进行了多少轮尝试以及 token 用量。/workflows 则能展示每个正在运行的 agent 各自用了多少 token，而且你随时可以手动叫停某个 agent。这些监控手段就像是仪表盘，让你在烧钱的时候心里有数。

听到这里，你可能已经感觉这就像在管理一个微型团队了，有人负责执行，有人负责审查，还有成本控制和监控仪表盘。没错，这就是 agentic 工作流向前发展之后必然的形态。

最后，我想用两三句话来总结这篇文章，同时也给出一点我个人的点评。

这篇文章所讲的，其实是 Claude Code 从“工具”走向“协作者”乃至“自动化团队”的一张进阶路线图。它告诉我们，循环不是一种炫技，而是一种分工思维：定义好触发、执行和停止的规则，然后让系统自己去跑。你不再是唯一的驱动者，而是变成了目标的设定者、规则的制定者和异常的处理者。

我个人认为，这种基于循环的智能体使用模型，会很快成为 AI 编程的主流方式。但它的风险也同样明显：如果一个循环设定得过于宽泛或者停止条件不严谨，它可能会像一只不受控制的机器，疯狂地消耗资源，甚至在代码库里做出你不想看到的大面积修改。所以 papi 的渐进式原则特别重要：从最简单的手动循环开始，逐步加 skill、加目标、加时间调度，每一步都验证，再迈下一步。同时，永远不要绕过人工审查那些会影响核心业务逻辑或数据安全的修改，即便循环能自己审查，人也必须保持最终的把控。

好，那今天关于 Claude Code 循环机制的解读就到这里了。如果你正在日常工作中使用这类工具，强烈建议你今天就挑一个小任务，试试看把它变成一个基于目标的循环，说不定你会发现，原来那些繁琐的重复性工作，真的可以这么轻松地甩出去。感谢收听 Claude FM，我们下期节目再见。
