先说一点背景。Anthropic 在二〇二五年发布 Claude Code 的时候，他们原本可能预期主要用户是开发者。Claude Code 是面向终端，也就是命令行环境的 agent 工具。这里的 agent 可以理解成能持续执行任务、调用工具、查看文件、修改内容的智能助手。可是他们发现，很多非技术用户也开始尝试 Claude Code。有些人以前从来没打开过终端，却开始用它创建小 agent 来整理文件夹、去重文件、写电子表格公式。

这其实很说明问题。终端虽然对工程师来说很自然，但对很多知识工作者来说，它是一只黑盒。你打开一个黑底白字的窗口，光标在那里闪，很多人会不知道下一步该做什么。命令行要求你有一种很特定的操作习惯，也要求你相信自己不会把什么东西搞坏。所以，Anthropic 做 Claude Cowork 的动机，就是把 Claude Code 里的 agentic capabilities，也就是 agent 式能力，扩展到大家已经熟悉的聊天界面里。简单说，以前你可能要在终端里告诉 Claude 去看文件、改文件、执行任务，现在你可以在一个像聊天一样的界面里，把这些协作型任务交给它。

从文章来看，Claude Cowork 是在二〇二六年一月推出的。到五月份，Anthropic 对它进行了一次使用情况分析。他们抽样了一百二十万次匿名且聚合的 Claude Cowork 会话，时间范围是二〇二六年五月十一日到五月三十一日，也就是三周左右。这些会话来自超过六十万个组织。Anthropic 用一个自动化系统，把这些会话按照二十种工作类别进行分类。这里要注意，他们强调没有人类分析师去阅读单个用户会话，分析工具是保护隐私的，最后只处理聚合之后的分类统计。

这篇文章的核心发现，第一点就是 Claude Cowork 最大的使用类别，是 business process and operations，中文可以理解为业务流程和运营，比例是百分之三十三点四。这个类别包括什么呢？原文举了几个例子，比如把散落在不同地方的项目更新整理成一份报告，创建新员工入职清单，对账或者核对电子表格。你听到这里可能会觉得，这不就是办公室里最常见的杂活吗？对，恰恰如此。它们不是某个单一岗位独有的工作，而是很多岗位都会碰到的共同工作。财务会碰到，人力资源会碰到，行政会碰到，项目经理会碰到，工程团队的负责人也会碰到。

我们可以把它想象成一家公司里的管道系统。真正的业务成果，像发布一个产品、签下一个客户、完成一次招聘，当然很重要。但这些成果背后，需要大量的信息从一个人流向另一个人，从一个表格流向一份报告，从一次会议流向一个项目看板。业务流程和运营类任务，就是这些信息管道里的阀门、接头和过滤器。平时没人觉得它们高大上，但它们一旦堵住，整个组织就开始卡顿。Claude Cowork 在这个类别里占到三分之一，说明很多用户不是把它当作一个炫技工具，而是把它当作信息整理和流程推进的助手。

第二大类别是 content creation and copywriting，内容创作和文案写作，占百分之十六点四。文章里说，这类任务包括需要大量综合能力的商业沟通工作，比如起草文稿、制作 slide deck，也就是演示文稿，写帖子，写 proposal，也就是方案或提案。Anthropic 特别提到，面对空白页面经常是开始工作的第一道门槛。Claude Cowork 的价值就在于，它可以把你的想法和资料串起来，变成一份粗稿。

这个观察对所有知识工作者都很真实。很多时候，我们并不是没有想法，而是不知道怎么开始。比如你要给老板写一份项目状态更新，你脑子里知道后端接口完成了百分之多少，知道某个依赖延期了，知道下周要做什么，也知道风险在哪里。但这些信息散在聊天记录、任务系统、邮件和你的脑子里。真正难的不是找到一个华丽的表达，而是把这些碎片拉到同一个上下文里，变成一段别人能读懂、能做决策的文字。Claude Cowork 在这里扮演的不是作家，而更像一个耐心的编辑和整理员。你给它材料，它先帮你把结构搭起来，再让你把专业判断补进去。

这两个类别加起来，大约就是全部 Claude Cowork 使用的一半。Anthropic 对此的解释是，这些使用都具有很强的 connective nature，也就是连接性质。电子表格把分散的数据点放进一个可以阅读、比较和追踪的上下文里。演示文稿把一个想法或决策传达给更广泛的受众，而且这些受众的背景知识可能不同。入职清单帮助新员工接入组织内部的知识。换句话说，Claude Cowork 最先吃下的，不是某个职业的最核心技能，而是组织协作中最容易被忽视的连接层。

这一点很重要。过去两年大家讨论 AI 对工作的影响，经常会问，AI 会不会替代程序员？会不会替代设计师？会不会替代律师？但从这篇文章的数据看，至少 Claude Cowork 这个产品的早期使用形态，更多是替代或者增强岗位周边的连接性工作。比如原文举了律师的例子。律师可能会用 Claude Cowork 处理文档格式和归档，这样就有更多时间把法律判断用在困难案件上。招聘经理可能会用它安排会议，综合面试反馈，从而有更多时间和候选人交流，评估作品样本。团队负责人可能会用它制作解释困难决策的演示文稿，从而把精力留给真正的艰难判断。

你会发现，文章刻意没有说 AI 替你做法律判断，替你决定录用谁，替你做组织中的困难决策。它说的是，AI 帮你处理那些让你能更好使用专业能力的周边任务。我个人认为，这是现在企业里部署 AI 更靠谱的一条路径。很多公司一上来就想让 AI 做最高价值、最高风险、最不可逆的事情，比如自动审批合同、自动决定候选人、自动设计架构。这样的想法很诱人，但落地成本、信任成本和风险都很高。相比之下，让 AI 先进入汇总、整理、起草、格式化、追踪这些环节，更容易获得用户信任，也更容易产生可见收益。

接下来我们看其他类别。软件开发占 Claude Cowork 使用的百分之八点七。DevOps and infrastructure，也就是运维和基础设施，占百分之七。research and intelligence，研究和情报，占百分之六点四。data analysis and business intelligence，数据分析和商业智能，占百分之五点八。document processing and extraction，文档处理和信息抽取，占百分之四点一。sales and revenue operations，销售和收入运营，占百分之四。其他类别都低于百分之四，包括 personal assistance，个人助理，占百分之三点八，education，教育，占百分之二点四，meeting intelligence，会议智能，占百分之一点八。

这里最容易让工程师注意的是，软件开发只有百分之八点七。听起来好像有点低，毕竟 AI 编程是最近几年最受关注的方向之一。但文章解释说，这其实很合理。Claude Code 更常被软件开发者用于他们角色中的核心部分，也就是构建、调试和发布代码。开发者如果要写代码，更可能去用 Claude Code，而不是 Claude Cowork。那他们在 Claude Cowork 里做什么呢？更可能做那些围绕软件工程的沟通和连接工作，比如整理状态、解释决策、写文档、汇总反馈。

这给我们一个非常好的分工视角。Claude Code 像是工程师工作台上的电动工具，直接接触代码仓库、命令行、测试和构建流程。Claude Cowork 更像办公室里的协作助手，接触报告、文档、表格、演示和跨团队沟通。两者都叫 Claude，都有 agent 能力，但入口不同，用户心智不同，适合的任务也不同。对技术团队来说，这意味着你不应该只问，AI 能不能帮我们写代码。你还应该问，AI 能不能减少工程师写状态同步、写复盘、写设计解释、整理会议结论的时间。

我们不妨做一个通勤路上的小推演。假设你是一个后端团队的 tech lead，这周你要推进一个数据库迁移项目。真正的技术难题可能是迁移策略、回滚方案、数据一致性验证。但你的一周并不会只花在这些事情上。你要把当前进度整理给产品和运营，要把风险解释给业务方，要把迁移窗口同步给 SRE，要把行动项拆给团队成员，还要把上一轮压测结果整理成一页大家能看懂的材料。这里面有多少是你的核心技术判断？有。可是有多少是信息搬运、格式整理、语气调整、结构化表达？也非常多。Claude Cowork 的数据说明，很多人已经在把后一类任务交给 AI。

再举一个产品经理的例子。原文没有专门讲产品经理，但它说这些类别横跨很多角色。产品经理可能会把用户反馈、销售输入、客户成功记录和数据指标放进 Cowork，让它先帮忙整理成一份需求背景。然后产品经理自己再判断什么是真问题，什么是噪音，什么要进路线图。这里 AI 并没有替代产品判断，而是减少了从散乱输入到可讨论材料之间的摩擦。类似地，市场同事可以让它根据已有资料起草发布文案，业务发展同事可以让它打 proposal 的底稿，项目经理可以让它把项目状态从多个来源汇总成更新邮件。

不过我们也要认真看这篇文章的数据方法和局限。Anthropic 在文末补充了调查方法。首先，分类是按工作内容来分，不是按用户职位来分。也就是说，如果一个市场同事让 Claude Cowork 帮忙整理活动执行清单，这个会话可能被归入业务流程和运营，而不是市场。文章还特别说明，他们的分类体系里没有单独的 marketing，finance，或者 HR 类别，也就是市场、财务、人力资源没有作为独立大类出现。这些职能的很多任务都被业务流程和运营类别吸收了，这可能也是为什么业务流程和运营占到三分之一的原因之一。

这点很关键。如果你看到百分之三十三点四，就直接说全世界三分之一的 Cowork 使用都是运营人员在用，那是不准确的。更准确的说法是，在他们的样本中，三分之一会话被自动分类为业务流程和运营类型的工作，而这些工作可能来自很多不同职位。一个工程经理写项目更新，可能归进去。一个财务人员核对表格，可能归进去。一个 HR 做入职清单，也可能归进去。分类关注的是任务，而不是人。

第二个局限是样本不是固定比例抽样，而是 capped rate，也就是按固定的每小时最大会话数采集。它不是说总流量里抽百分之一，而是每小时最多收多少个样本。因此报告里的数字代表的是采样会话里的占比，不代表 Claude Cowork 的绝对使用量，也不能用来判断增长速度。文章还提醒，因为这种方式，白天更繁忙时段的使用可能相对于较安静时段被低估。对数据同学来说，这一点很熟悉。采样方法会影响你能解释什么，也会影响你不能解释什么。

第三个局限是时间窗口。他们选择的是五月十一日到五月三十一日这三个完整周，而不是更长时间。原因是五月十一日前后类别占比发生了变化，而这种变化更像是标签管线，也就是 labeling pipeline 的变化，而不是用户行为真实改变。于是他们只报告变化之后的数据。这个细节反而让我觉得文章比较谨慎。它没有假装数据天然干净，而是承认自动分类系统本身会影响结果。

第四个局限是工作和个人使用混在一起。样本覆盖的是外部组织的使用，不是个人用户，但其中仍然包含一些个人、非工作用途。文章说，个人助理、爱好和陪伴式对话加起来大约占百分之五。所以这个样本不能被理解成纯粹的 workplace activity，也就是纯工作场景。第五个局限是自动分类可能出错。标签由自动系统应用，不是人工审核。一个会话如果可以同时归入多个类别，最终结果就取决于分类体系的定义。

这些局限告诉我们，这篇文章不能当作一份严肃的劳动经济学研究，也不能直接推导某个行业的生产率提升。但它可以作为一个很有用的参考点。尤其是对正在思考如何把 AI 工具引入团队的人来说，它告诉你用户自然会把 AI 拿去处理哪些痛点。很多时候，产品路线不是从厂商宣传里看出来的，而是从用户真实使用里长出来的。Claude Cowork 的数据很像是在说，用户最先把 AI 放进了组织的信息缝隙里。

这就引出了一个更大的行业背景。过去几年，AI 应用的公众讨论很大程度上被编程占据。原因很简单，代码是文本，反馈相对明确，测试可以自动化，工程师也愿意尝试新工具。所以 Claude Code、Cursor、GitHub Copilot 这一类工具很容易成为焦点。但在真实公司里，软件开发只是知识工作的一部分。大量人每天的核心产出其实是信息的创建、整理、传递和对齐。状态报告、周报、会议纪要、路线图、销售提案、项目看板、客户摘要、数据解读、培训材料，这些东西不是代码，却决定了组织能不能协作。

我个人认为，Claude Cowork 这篇文章非常像是在给 AI 办公软件画一个坐标系。横轴可以理解为任务是否核心，纵轴可以理解为任务是否结构化。写代码对开发者来说很核心，而且某些环节结构化程度高，所以 Claude Code 这样的专用工具会很强。写周报、整理表格、做 slide，对很多人来说不是最核心，但频率高、耗时长、结构相对清晰，所以 Claude Cowork 这样的聊天式 agent 很适合。至于那些既核心又高度依赖判断的任务，比如最终法律意见、录用决策、战略取舍，AI 可以辅助，但不应该轻易替代人的责任。

那对我们的实际工作有什么启发？如果你是软件工程师，第一件事不是把 AI 只当成写代码工具。你可以盘点一下自己每周的工作，把任务分成两类。一类是你真正需要亲自判断的，比如系统设计取舍、复杂 bug 根因分析、性能瓶颈定位、团队技术方向。另一类是为了让这些判断被别人理解和执行而产生的周边工作，比如把技术方案改写成非技术同事能看懂的版本，把事故复盘整理成时间线和行动项，把需求变更影响整理成表格，把多个 issue 的状态合并成一封更新邮件。后一类就是 Claude Cowork 数据里最典型的场景。

如果你是团队负责人，可以从固定重复的协作节点开始。比如每周项目更新，可以让 AI 根据任务系统、会议记录和你提供的要点，先生成一个草稿。你要做的是检查事实、补充判断、调整语气。比如新成员入职，可以让 AI 根据已有文档整理 onboarding checklist，列出第一周、第二周、第三周应该读什么、做什么、找谁对齐。比如跨团队技术决策，可以让 AI 帮你把背景、选项、利弊、风险、下一步行动组织成一份 deck 的大纲。这里的关键不是让 AI 做决定，而是让 AI 降低表达和整理成本。

如果你是 AI 产品从业者，这篇文章也给了一个产品设计提示。很多人不喜欢终端，并不代表他们不需要 agent 能力。他们可能只是需要更熟悉的入口、更明确的安全边界和更低的心理门槛。Claude Code 的能力很强，但终端天然筛掉了一部分用户。Claude Cowork 把类似的能力放到聊天界面里，就进入了另一类工作流。这里的差异不是简单的 UI 换皮，而是任务生态的变化。在终端里，用户更容易想到文件、命令、仓库、测试。在聊天协作界面里，用户更容易想到文档、表格、报告、会议、沟通。

当然，上手用这类工具也有坑。第一个坑是把整理当成判断。AI 可以帮你把采访反馈综合成摘要，但是否录用候选人仍然需要人来负责。AI 可以帮你把合同材料格式化，但法律意见不能因为摘要看起来顺就直接通过。AI 可以帮你把项目风险写成漂亮的幻灯片，但风险本身有没有被解决，不会因为 deck 漂亮而消失。Claude Cowork 的价值在于帮你 assemble and structure information，也就是组装和结构化信息，让你更好地使用自己的专业能力。这个边界一定要记住。

第二个坑是输入材料不清楚，却期待输出准确。很多人用 AI 写报告，给一句话就希望它生成能直接发给老板的版本，结果自然容易空泛。更好的方式是把上下文给足。比如告诉它目标读者是谁，他们已经知道什么，不知道什么，你希望他们看完后做什么决定，哪些事实必须保留，哪些语气要避免。你也可以把散乱要点丢进去，让它先问你缺什么，而不是直接生成终稿。把 AI 当作协作者，而不是许愿机，效果会完全不同。

第三个坑是忽视隐私和权限。文章里 Anthropic 强调他们的数据分析是匿名和聚合的，没有人类分析师读单个会话。但这不等于企业内部使用时就可以随便把任何敏感资料丢进去。每家公司都应该根据自己的合规要求和工具配置，明确哪些文档可以进入 AI，哪些需要脱敏，哪些场景必须有人复核。尤其是销售数据、候选人信息、客户合同、法律文件、源代码和安全配置，这些都需要更谨慎的流程。

第四个坑是没有把 AI 输出接回现有工作流。比如 AI 帮你生成了一个状态报告，但它没有进入项目管理系统，也没有触发行动项，也没有被相关人确认，那它只是一份漂亮文本。Claude Cowork 这类工具真正有用，是因为它能帮助跨团队追踪和沟通信息。所以你最好在团队里约定清楚，哪些报告由 AI 起草，谁负责审核，更新到哪里，多久一次，如何标注不确定信息。否则 AI 可能只是制造更多文档，而不是减少协作摩擦。

第五个坑是用 AI 放大坏流程。如果一个组织本来就有太多没必要的周报、太多没人看的 slide、太多重复填报的表格，AI 可以让这些东西生成得更快，但不一定让组织更健康。我的建议是，先用 AI 暴露流程里的重复劳动，再问这些劳动是否真的必要。比如如果每周有三份不同格式的项目更新，AI 可以帮你生成，但更好的问题也许是，能不能统一成一份？如果同一组数据要手工粘贴到多个表格，AI 可以帮你搬运，但更好的问题也许是，数据源能不能统一？AI 是放大器，好的流程会被放大，坏的流程也会被放大。

文章最后说，Claude Cowork 仍然很新，用法变化很快。这次分析只是一个 single snapshot，也就是单个时间点的快照。Anthropic 的目标是给那些正在思考如何把 AI 产品整合进日常工作的人提供参考，并展示使用最集中的地方。他们还计划随着使用增长和变化，继续发布数据，报告哪些东西会随时间改变。文章也提到 Claude Cowork 面向所有 Claude 用户开放，并提供入门课程。

我个人认为，这篇文章的最大价值，是把 AI 应用的讨论从单点能力拉回到组织协作。我们总是喜欢问模型能不能通过某个考试，能不能写出某段代码，能不能一次生成完美方案。但企业里的真实效率，往往损耗在信息不完整、上下文不一致、表达不清楚、交接不顺畅这些地方。Claude Cowork 的数据说明，用户已经在用 AI 去补这些缝。它不像模型发布那样轰动，但可能更接近 AI 在办公室里真正改变工作的方式。

所以如果用两三句话总结今天这篇文章，就是 Claude Cowork 的早期使用中，最集中的不是某个单一职业的核心技能，而是业务流程、运营、文档、文案、汇报、表格、演示这些连接性工作。Anthropic 的样本显示，业务流程和运营占百分之三十三点四，内容创作和文案占百分之十六点四，两者合计接近一半。对我们来说，这提醒我们别只盯着 AI 写代码，也要认真看看自己每天为了让工作流动起来而做的那些周边任务，因为那里很可能正是 AI 最先稳定落地的地方。
