今天我们聊的文章标题是 Base44 uses Claude Fable 5 for its hardest engineering work，中文可以译作，Base44 用 Claude Fable 五处理最困难的工程工作。文章页面里的完整标题还有一句 Working at the frontier，意思是在前沿工作，为什么 Base44 信任 Claude Fable 五来处理他们最具挑战的工程任务。这篇文章发表于二〇二六年七月十五日。今天是二〇二六年七月十九日，也就是说它只发布了四天，非常新，不存在明显的时效性折扣。它反映的是当前 AI 编程工具进入深水区之后，一个很典型的问题：我们到底能不能把真正复杂、会牵一发动全身、以前只敢交给资深工程师的工作，交给模型来做。

这期值得听的原因，不只是 Anthropic 又发了一篇客户案例。客户案例大家都见多了，常见套路是某某团队用了模型，效率提升，成本下降，听起来很热闹，但工程师听完往往会问一句，具体难在哪儿，怎么评估的，出了错谁兜底。Base44 这个例子有意思，是因为它讲到的不是写一个按钮，不是补一个表单，也不是生成一个 demo，而是系统 prompt 的重建、应用内 agent 的底层 harness 修改、原生移动端构建能力这些平台核心部件。对于做工程的人来说，这些工作都有一个共同点：它们不是单点功能，而是改了以后可能影响很多用户、很多路径、很多成本模型。

先补一下背景。Base44 是一个所谓 vibe coding 平台。vibe coding 这个词这两年很流行，直译有点怪，大概意思是用户用自然语言描述一个感觉、一个目标、一个产品想法，然后 AI 帮你把应用搭出来。Base44 面向的人群很宽，不要求用户有技术能力。文章里说，它可以让任何人构建全栈应用和网站。它的客户既包括没有开发者的小企业，也包括用它来做完整 SaaS 产品的公司。换句话说，它不是只做玩具 demo，而是希望真的让用户把一个业务应用跑起来。

文章采访的是 Base44 的产品负责人 Yoav Orlev。他是 Base44 的第一位员工，从很早期就加入了这个平台。他提到，自己工作里最有满足感的一部分，是看到小企业通过 Base44 做出原本没时间、没预算、没知识去做的东西。比如一个数字店面，或者给餐厅员工排班用的应用。你可以想象，一个小餐馆老板，如果去找外包做一套排班系统，需求沟通、报价、上线、维护，整个过程对他来说都很重。但如果他能直接描述，我需要员工换班、请假、老板审批、手机通知，平台就把应用做出来，这就是 Base44 想做的事情。

但这里有一个产品悖论。你越想让普通人用起来简单，平台背后就越复杂。用户不需要理解数据库、权限、前端、后端、部署、移动端适配、登录鉴权、缓存、账单这些东西，可平台必须替他理解。Base44 的使命，用文章里的说法，是不断扩大产品能力，同时保持对所有人都可用。对工程团队来说，这句话背后的压力很大。因为你一边要让功能越来越强，一边不能把体验变成工程师专用工具。

文章说，Base44 的产品和工程团队一直移动很快，尤其是发布小范围和中等范围功能的时候。但一旦改到平台核心，触碰多个相互依赖的部分，这些任务以前只能交给最资深的工程师。这里我们可以停一下。很多团队都有类似分层：写一个页面，很多人能做；加一个 API，也许中级工程师能做；但如果要改一套代码生成系统的核心 prompt，要改移动端基础设施，要改 agent 执行链路，那就不是“会不会写代码”的问题了，而是“你知不知道这块东西在整个系统里会影响谁”。

文章举了两个瓶颈。第一个是 Base44 的 system prompt，也就是系统提示词。对于大模型应用来说，system prompt 可以理解成模型行为的底层说明书。它告诉模型，你是谁，你要遵守什么规则，如何理解用户请求，怎么输出，什么时候追问，什么时候生成代码，哪些边界不能碰。Base44 的 system prompt 不是一段简单文本，而是有几百种排列组合。为什么会有这么多？因为用户状态不同，prompt 就要不同。比如这是用户第一次做应用，还是第五次做应用；用户是免费用户，还是订阅用户；他正在做的应用属于什么类别，要有哪些功能。这些因素组合起来，prompt 就会变成一个复杂系统。

第二个瓶颈是原生移动端基础设施。这个也很容易理解。一个平台如果只生成网页，复杂度已经不低；如果还要生成原生移动应用，牵涉到的技术栈、构建环境、设备差异、调试方式都会变多。文章说，这类修改只有具备移动端专业知识的工程师才能做。也就是说，它不是普通产品经理或者普通前端工程师可以随便动的区域。

那 Claude Fable 五 出现之前，为什么早期模型不能做这些事？文章说，Base44 从二〇二五年初上线开始，应用生成引擎就由 Claude 模型驱动，他们几乎用过从 Sonnet 四以来的每一代 Claude 模型。但 Yoav Orlev 认为，早期模型还不能被信任去做最核心的工程工作。一个很典型的问题是，当模型遇到错误卡住时，它会一直在眼前这个点上打转，而不是意识到，类似问题很可能已经在代码库的其他地方被解决过，所以应该去搜索、去对照、去借鉴。

这点对工程师特别有共鸣。一个初级工程师刚接手大代码库时，经常会在局部死磕。他看到某个测试失败，就盯着这几行代码改来改去。但资深工程师的反应通常不一样。他会想，这个系统里一定已经有一个类似模块，或者以前有人处理过同样的边界情况。与其凭空发明，不如先 grep 一下，看看已有约定是什么。比如你要加一个新的支付渠道，资深工程师不会从零写一套抽象。他会先看支付宝、微信、Stripe 的实现，看它们如何处理回调、幂等、重试、日志和告警。软件工程里很多正确答案，不在模型的通用知识里，而在你这家公司已有代码库的惯例里。

Yoav Orlev 在文章里说，“下一步做什么”这个决策非常关键，而早期模型大多数时候会采取一种天真的方法。这个“天真”不是说模型不会写语法，而是它不会像熟悉工程系统的人一样判断路径。它可能看到错误就修错误，看到类型不匹配就改类型，看到测试失败就让测试通过。但它不一定知道，真正的问题可能是你违反了某个内部框架约定，或者某个缓存键被改了，或者某个构建 harness 有一套隐藏规则。

Claude Fable 五 让 Base44 团队觉得不同的地方，正是在这里。Yoav Orlev 说，这是他们测试过的第一个能像理解软件如何构建那样进行推理的模型。注意，这句话很强，但它是受访者的判断，不是论文里的严格结论。我们要忠于原文，同时也要保持工程上的清醒。文章不是说 Fable 五 具有人类一样的真正理解，而是说在 Base44 的真实任务里，它表现得像一个更懂软件系统的人。

Base44 怎么测试新模型？文章提到，他们会把每个新的 Claude 模型跑过一组 evals，也就是评测，覆盖不同类型的应用，衡量延迟、成本和构建错误。延迟很好理解，用户说我要一个库存管理系统，如果等十分钟才有反应，体验会很差。成本也很关键，因为模型调用是有钱的，特别是平台有大量用户的时候。构建错误则直接决定生成的应用能不能跑起来。除此之外，他们还会做一些特殊测试，比如让模型构建一个 Minecraft 克隆，用来看它怎么处理游戏物理和机制。

这个 Minecraft 克隆测试很有意思。因为它不是一个普通 CRUD 应用。CRUD 就是增删改查，很多业务系统本质上都是表单、列表、详情页、权限和数据库操作。但游戏会考验状态更新、碰撞、物理、渲染循环、用户输入和规则一致性。让模型做一个 Minecraft 克隆，不一定是为了上线一个游戏，而是用它当压力测试。就像你考一个后端工程师，不只问他会不会写接口，也会问他高并发、缓存穿透、数据一致性怎么处理。游戏类任务能暴露模型在复杂交互、边界条件和系统性设计上的能力。

在这些评测里，Claude Fable 五 有两个突出的表现。第一，它完成任务需要的轮次少得多。这里的轮次可以理解成用户和模型来回沟通的次数。轮次少，不只是省时间，也意味着模型一开始就抓住了问题，减少返工。第二，它从第一个 prompt，也就是第一次提示，就能构建更完整的应用，包括早期模型经常跳过的边界情况。

边界情况这四个字，工程师都知道有多要命。一个应用 demo 看起来能跑，和一个产品真的可用，中间差的往往就是边界情况。比如一个排班系统，正常情况是老板创建班次，员工确认。但边界情况是，两个员工同时申请同一个班，某个人已经离职但还在历史排班里，跨时区的营业时间怎么算，免费用户能创建多少个员工，网络断了以后操作要不要重试。早期模型容易生成 happy path，也就是最顺利路径。Fable 五 被文章强调的点，是它在首版里就更能照顾这些容易被跳过的东西。

接下来是这篇文章最核心的案例：Base44 把 Fable 五 指向了一个以前只留给最资深工程师的任务，重建 Base44 的 system prompt。我们刚才说了，这个 prompt 有几百种排列组合，跟用户阶段、订阅状态、应用类别和功能相关。你可以把它想成一套巨大的交通规则系统。不是改一句“请更友好”这么简单，而是你改一个规则，可能让新手引导变好，却让高级用户的某些功能失效；你让免费用户路径更顺，却可能误开放订阅功能；你让某类应用生成更完整，却可能增加 token 消耗和延迟。

文章说，Fable 五 先和团队进行了大约一个小时的来回问答。这个细节很重要。它不是一上来就闭门造车。像资深工程师接一个复杂任务，也会先澄清目标、约束、历史包袱、成功标准、风险点。然后 Claude Fable 五 自己运行了四个小时，最后返回了团队所需内容的百分之九十到百分之九十五。这里也要注意，不是百分之百自动上线。团队仍然使用自己的 A B 测试基础设施来测量效果，并在当天下午发布这些变更。

我个人认为，这个流程比“AI 自动写完所有代码”更值得工程团队借鉴。它是一个人机协作闭环：先给模型足够上下文，让它做主要探索和改造；然后团队用已有测试、评测、A B 实验来验证；最后由人来批准上线。模型承担的是高强度脑力和执行工作，人承担的是目标设定、风险控制和最终责任。对严肃工程来说，这种分工比完全自动化更现实。

更有意思的是，在做这个 system prompt 工作时，Claude Fable 五 还指出了 Base44 自己 evals 里的一个缺口。团队没有测试 cache hits，也就是缓存命中。为什么这个重要？因为 prompt 变化可能破坏缓存。在大模型应用里，缓存常常是控制成本和延迟的重要手段。如果相同或相似请求可以命中缓存，就能少调用模型、少花钱、响应更快。但如果你改了 prompt，使原本可以复用的内容不能复用，缓存命中率下降，在百万级用户规模下，成本会显著上升。文章说，模型发现了这个盲点，并修正了它。

这件事背后其实有一个高级工程判断。很多人评估模型，只看功能能不能跑。但生产系统里，能跑只是起点。你还要看成本、延迟、可观测性、缓存、回滚、灰度、测试覆盖。Fable 五 指出 cache hit 缺失，说明它不只是围绕“生成结果对不对”思考，还把平台运行成本纳入了考虑。对一个有百万用户规模的平台来说，这种意识非常关键。一个 prompt 改动，如果让每次生成多消耗很多 token，单个用户看不出来，但账单会看出来。

文章的第二个重要案例，是 Base44 应用内 agent 背后的 harness 修改。这里先解释一下，agent 通常翻译成智能体，也就是不仅回答问题，还能围绕目标连续执行步骤的软件代理。harness 可以理解成运行框架或测试支架，是把 agent 放到应用环境里执行、观察、控制的一套底层机制。文章说，当 Claude Fable 五 在修改这个 harness 时卡住了，它没有继续在原地乱试，而是推理说，同样的问题可能已经在代码库其他地方被解决过，于是去调查那一部分，最后带着修复方案回来。

这就是前面说的“像资深工程师”的具体表现。我们可以想象一个场景：你在一个大仓库里改一个任务调度器，遇到某个权限校验总是失败。初级做法是盯着当前文件，不断加 if，不断改参数。资深做法是搜索一下这个权限校验在其他调度器里怎么用，有没有 helper，有没有统一封装，有没有历史迁移。因为在成熟代码库里，最好的答案往往不是新写一套，而是遵循已有模式。模型能主动想到“这可能已经在别处解决过”，对大规模软件工程来说，是能力跃迁。

Yoav Orlev 直接把使用 Fable 五 的体验类比为和资深工程师合作。他说，初级工程师需要你把每一步都说清楚，还要持续检查；但对资深工程师，你只需要说明目标和为什么要做。这里的“为什么”非常重要。你如果只告诉模型“改这个文件”，它可能完成一个局部任务。你如果告诉它“我们要降低新用户生成应用时的失败率，同时不能增加订阅用户的延迟，也不能破坏缓存”，它就有机会做更全局的权衡。

这对我们怎么使用 AI 编程工具有直接启发。很多人还停留在把模型当自动补全，给它一段代码，让它补几行。这样当然有价值，但不是文章想表达的重点。Base44 的用法更像把模型当一个可协作的工程成员。给它任务背景、业务目标、约束条件、已有评测、代码库上下文，然后让它提出方案、执行、发现问题。你越把它当纯粹的打字机器，它越只能省键盘时间。你越给它资深工程师需要的信息，它才越可能表现出资深工程师式的判断。

文章还讲了一个跨出工程团队的例子。有一位产品经理想把原生移动应用构建能力带进 Base44。过去，这类工作要等有移动端经验的专家或顶级工程师有空。现在，他把 Claude Fable 五 指向这个任务，大约两个半小时后，得到了一个可工作的环境，完成度大约是团队进入生产所需的百分之九十。注意，文章没有说产品经理直接把它上线，也没有说不需要工程师。它说的是，过去需要专家空出时间才能启动的事情，现在可以先由模型执行到一个很接近可用的状态，再由团队审查、测试、批准。

这一点对组织效率的影响很大。很多公司真正慢的地方，不是每个任务本身都很难，而是关键人太少。所有高风险改动都排在几位资深工程师队列里，产品经理等，设计师等，业务方等。资深工程师每天像一个人肉瓶颈，既要写核心代码，又要 review，又要救火，还要开会。如果模型能承担一部分复杂执行工作，资深工程师的角色就会更偏向定义架构、审查关键路径、设计评测、做最后把关。这样不是削弱资深工程师，而是把他们从重复性的深水区劳动里解放出来。

当然，我们也不能把这篇文章读成“模型可以无监督改核心系统”。原文里非常明确地说，现在模型执行任务，而 Base44 团队审查、测试并批准代码，然后再发布。这几个动词一个都不能少。审查，是看设计和代码有没有明显问题；测试，是验证功能和回归；批准，是人类团队承担上线责任。特别是在 system prompt 这种位置，模型改得再好，也要通过实验和监控。因为 prompt 的风险很隐蔽，它可能不报错，但会让模型行为慢慢偏掉。

接下来文章谈到未来。随着 Claude 模型能力提升，Base44 团队对平台的目标也在提升。他们希望把 Base44 从一个“构建应用的工具”，变成一个也能帮助人们管理和增长已构建应用的平台。这里提到 Base44 Superagents，现在已经公开。Superagents 可以围绕这些应用运行 workflows，也就是工作流。原文没有展开技术细节，所以我们不能替它编造太多。但可以理解为，Base44 不只想帮你生成一个应用，还想在应用运行之后，帮你做运营、自动化、管理和增长。

这其实是 AI 应用平台的自然演进。第一阶段是生成，用户说我要什么，平台给你搭出来。第二阶段是维护，应用上线后出现 bug、需求变化、数据增长，平台能帮你改。第三阶段是运营，应用不是一个静态资产，而是业务的一部分。比如你做了一个小店铺系统，后面还需要促销、库存提醒、客户分层、员工排班优化。Superagents 如果围绕应用工作，就可能把“建应用”扩展成“经营应用”。不过这部分在文章里只是方向，我们点到为止。

因为 Fable 五 能处理复杂任务，Yoav Orlev 现在鼓励产品经理和设计师去构建平台里一些他们以前不敢碰的部分。以前他们担心弄坏东西，现在更有信心尝试。文章最后引用他说，Fable 给了他们做更大胆商业动作的信心，把产品带到一个全新的领域和可能性，而以前他们会害怕去做。

讲到这里，我们可以把文章的核心观点压缩成一句话：Claude Fable 五 在 Base44 的真实工程场景里，不只是更会写代码，而是更会做软件工程判断。它减少沟通轮次，首版覆盖更多边界情况，能在代码库里寻找已有模式，能发现评测盲点，还能把过去只敢交给资深工程师的任务推进到接近生产可用。

那这对正在通勤路上听节目的你，意味着什么？如果你是软件工程师、技术负责人，或者 AI 产品的工程同学，我觉得有几条非常实际的启发。

第一，不要只用玩具任务评估编程模型。Base44 的做法是跑不同应用类型，测延迟、成本和构建错误，还用 Minecraft 克隆这种任务测复杂机制。你自己的团队也可以建立类似 evals。不要只问模型会不会写一个 Todo List。你应该拿真实业务里的难点来测，比如权限系统、报表生成、异步任务、支付回调、数据迁移、缓存策略、移动端构建、监控告警。评测越贴近生产，模型能力的判断才越有意义。

第二，评测指标要包括成本和运行特征，而不是只看代码通过不通过。文章里 cache hit 的例子非常值得警惕。很多团队在引入大模型时，早期只看效果。等用户量上来，才发现成本曲线不对，延迟不稳定，缓存命中率低，prompt 改动牵一发动全身。所以如果你们做的是 AI 产品，建议从一开始就把 token 消耗、缓存命中、延迟分位数、失败率、重试率、人工介入率纳入仪表盘。模型帮你发现盲点当然很好，但更可靠的是团队自己也有这套意识。

第三，给模型任务时，要像给资深工程师 brief，而不是像给脚本传参数。什么叫 brief？就是告诉它目标、背景、为什么做、成功标准、约束、不能破坏的东西、已有实现在哪里、评测怎么跑。比如你不要只说“重构这个模块”。你可以说，我们要降低新用户创建应用时的失败率，保持订阅用户功能不变，不能增加平均延迟，注意 prompt cache，改完需要通过这些测试，并总结潜在风险。这样的任务描述，才可能激发模型做系统性推理。

第四，把模型放进已有工程流程，而不是绕开工程流程。Base44 没有说模型写完就直接上线。它有 A B 测试基础设施，有 review，有测试，有批准。对企业团队来说，这条红线很重要。AI 编程工具最危险的用法，是让它在没有保护网的情况下直接改生产。更好的做法，是让它开分支、写说明、跑测试、生成迁移计划、列风险清单，然后由人和自动化系统共同把关。模型越强，流程越不能松，因为它能改的范围也越大。

第五，重新思考资深工程师的杠杆。过去资深工程师的稀缺性，体现在他们亲自做最难的活。未来可能更多体现在他们定义什么是难、什么是对、什么是安全，搭建评测和守护边界。一个好的资深工程师，可以把复杂任务拆成模型能执行的 brief，再设计验证方法，最后审查结果。这有点像从“最强工匠”变成“工程系统的设计者”。这不是降级，反而是更高层次的杠杆。

第六，如果你是产品经理或设计师，这篇文章也和你有关。它说明未来一些以前你不敢碰的技术区域，可能可以通过模型先探索。但前提是你要理解风险边界。你可以让模型搭一个移动端构建环境原型，可以让它探索某个功能可行性，可以让它生成方案和初版实现。但进入生产前，仍然要让工程团队审查。模型降低了进入门槛，不代表取消专业责任。

我们也要说说可能的坑。第一个坑，是把“百分之九十完成”误解成“可以上线”。工程里最后百分之十往往最贵，里面包括异常处理、权限、可观测性、兼容性、灰度、回滚、文档和长期维护。文章里 Base44 得到百分之九十到百分之九十五所需内容，或者两个半小时得到百分之九十生产所需环境，都很厉害，但它们后面都有团队验证。不要在自己公司里只复制前半段，忽略后半段。

第二个坑，是过度相信模型的全局理解。Fable 五 在文中表现出会查找代码库其他地方的能力，但这不代表它永远知道你的业务规则。很多业务约束没有写在代码里，可能在销售承诺里，在法务要求里，在某个客户的特殊配置里，在历史事故复盘里。人类团队仍然要把这些上下文显式化。否则模型可能做出技术上漂亮、业务上错误的改动。

第三个坑，是忽视 prompt 本身已经成为软件资产。Base44 的 system prompt 有几百种排列组合，这说明在 AI 产品里，prompt 不是一次性文案，而是核心代码的一部分。它需要版本管理、评测、灰度、回滚、缓存策略和变更记录。如果你的团队还把 prompt 存在一个文档里，谁想改就改，那么随着产品复杂度上升，很快会进入不可维护状态。我的建议是，把 prompt 当配置代码来管理，把不同用户状态、不同功能路径的 prompt 组合纳入测试。

第四个坑，是没有给模型足够时间和权限去探索。文章里 Fable 五 先问了一个小时，又独立运行四小时。这和很多人使用模型的方式不一样。很多人给模型三十秒，如果第一版不对，就说不行。但复杂工程任务本来就需要探索。关键是探索要在安全边界内进行，比如只读代码库、沙盒环境、分支开发、可回滚。你要给它足够上下文，也要给它足够护栏。

如果把这篇文章放到二〇二六年七月这个时间点看，它代表的趋势是 AI 编程从“生成代码片段”走向“参与工程流程”。早期我们惊讶于模型能写函数，后来惊讶于它能生成完整页面，再后来它能跑测试、改 bug、做 agent。现在 Base44 的例子显示，模型开始进入更高风险的区域：系统 prompt 重构、基础设施修改、跨团队能力建设。行业的核心问题也从“模型会不会写”变成“模型能不能被信任到什么程度，以及我们怎样设计信任机制”。

我个人认为，这篇文章最有价值的不是证明 Claude Fable 五 多强，而是给出了一个更成熟的采用姿势。Base44 没有把信任建立在感觉上，而是建立在 evals、真实任务、A B 测试、成本监控和人类审批上。它也没有把模型只用于边角料，而是敢于把高价值问题交给模型，但保留工程纪律。这种组合，可能是接下来很多团队采用 AI 工程工具的标准范式。

最后收个尾。Anthropic 这篇文章讲的是 Base44 为什么愿意把最难的工程工作交给 Claude Fable 五。它的答案不是“模型更聪明”这么简单，而是模型在真实代码库里表现出更接近资深工程师的推理方式：少走弯路，找已有解法，覆盖边界情况，甚至提醒团队评测盲点。对我们来说，真正该学的，是如何把模型放进严肃的软件工程系统里，让它扩大团队能力，而不是替代工程责任。我的点评是，未来最强的团队，可能不是单纯拥有最强模型的团队，而是最会把模型、评测、流程和资深工程判断结合起来的团队。
