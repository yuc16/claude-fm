## Shownotes

这一期我们聊聊 Anthropic 在二〇二五年七月推出的一项功能：让 Claude 直接在对话里生成可以调用 AI 能力的互动应用，并且一键分享给别人用,而你自己不用掏一分钱 API 费用。我们会拆解这个机制是怎么运作的,为什么它解决了开发者一直头疼的"做了 demo 没钱养"的问题,以及普通工程师现在可以拿它做点什么。

- Claude 的 artifacts 升级成可以反向调用 Claude API 的"活"应用,不再是静态代码片段
- 使用者用自己的 Claude 账号鉴权,消耗算在使用者自己的订阅额度上,开发者零成本
- 不需要管理 API key,不需要部署服务器,做完直接发链接就能分享
- 已经出现的真实案例:会记忆的 NPC 游戏、自适应难度的学习工具、上传 CSV 就能问答的数据分析小应用
- 目前的限制也很关键:不能调外部 API、没有持久化存储、只能用文本补全接口
- 我个人的看法:这是 Claude 在"轻量级应用分发"赛道上的一次重要试探,也是理解后续 Claude 生态演化的一个关键节点

## Script

大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是 Build and share AI dash powered apps with Claude,翻译过来就是,用 Claude 构建和分享由 AI 驱动的应用。这篇文章发表于二〇二五年七月二十五日,后面还有两次更新,一次是七月十七号补充的,可以往 artifact 里上传 PDF、图片等文件,另一次是七月三十一号补充的,这个功能正式开放给 Team 和 Enterprise 这种企业级方案的用户。

先说一句关于时效性的提醒。今天是二〇二六年六月二十一号,这篇文章发表已经快一年了,在 AI 这个行业里,一年时间足够发生很多变化,所以文章里提到的"目前的限制",到我们录这期节目的时候,很可能已经被突破或者调整过了。我们今天聊的是这个功能刚推出时候的设计思路和产品逻辑,这个逻辑本身的价值是长期的,但具体的功能边界,大家如果想实际动手做,一定要去 Claude 的官方文档里确认最新状态。我们把这篇文章放回到当时的背景里看,二〇二五年年中,正是各家大模型厂商疯狂卷"agent"这个概念,也就是能自主执行多步骤任务的智能体的时候。OpenAI、Google,还有各种创业公司都在讨论怎么让普通人,而不只是程序员,能够用自然语言直接"生成"一个可用的软件。这篇文章,其实就是 Anthropic 在这个大背景下,给自己的 Claude 产品交出的一份答卷。

那这篇文章具体要解决什么问题呢?我们先从一个很多做过开发的朋友都会有共鸣的痛点说起。假如你是一个独立开发者,或者是公司里一个喜欢周末搞点小项目的工程师,你用 Claude 写了一个挺酷的小应用,比如说一个能根据你上传的简历自动生成面试模拟对话的小工具。这个工具背后调用了 Claude 的 API,也就是说每次有人用这个工具问一句话,系统背后都要向 Anthropic 的服务器发一次请求,这次请求是要花钱的,钱从你的账户里出。

这时候问题就来了。如果这个小工具只有你自己用,那没什么,花的钱不多。但如果这个工具做得好,你想分享给朋友用,甚至发到社交媒体上,让更多人体验,那麻烦就来了。访问量越大,你需要支付的 API 调用费用就越高。很多独立开发者都遇到过这种情况,辛辛苦苦做了一个很受欢迎的小工具,结果因为用户太多,自己扛不住后端的调用成本,要么把功能阉割,要么干脆把项目关掉。这其实是这几年很多"AI 套壳应用"创业者最大的痛点,你的产品越受欢迎,你越亏钱,这是一个很反直觉、也很让人沮丧的经济模型。

这篇文章里 Anthropic 给出的方案,核心思路其实特别简单,但很巧妙。他们说,让用户用自己的 Claude 账号去验证身份,也就是英文里说的 authenticate,这样一来,当这个用户在用你做的这个应用,并且这个应用背后调用了 Claude 的能力时,产生的费用是算在这个用户自己的 Claude 订阅额度里,而不是算在你这个开发者头上。换句话说,谁用,谁付费,跟你这个做工具的人没关系。

我们来打个比方,这就好像你开了一家很受欢迎的咖啡馆,但咖啡馆里的咖啡豆,不是你来囤货采购的,而是每个客人自带咖啡豆来,你只负责帮他们冲泡和提供桌椅,提供体验和服务。客人越多,你的"成本"反而没有上升,因为原料是客人自己带的。这种模式直接把开发者和分享者从"按用户数量付费"的恐惧里解放出来了。

这个机制具体是怎么实现的呢?这就要说到这篇文章的另一个核心概念,叫 artifacts,中文我们可以理解成"产出物"或者"作品"。如果你用过 Claude,应该知道 artifacts 这个功能已经存在一段时间了,它本质上是 Claude 在对话框旁边单独开一个窗口,把你让它生成的代码、文档、网页、图表之类的内容,渲染展示出来,方便你直接看效果,而不用切换到别的工具里去运行代码。

但在这篇文章之前,artifacts 基本上是"静态"的。也就是说,Claude 帮你写了一段代码,生成了一个网页或者一个小工具,这个东西本身是死的,它不会再主动去调用 Claude 的智能来处理新的输入。比如说你让 Claude 做一个计算器,这个计算器artifact 里面的逻辑就是普通的加减乘除代码,跟 AI 没有任何进一步的关系,它只是 Claude 帮你"写"出来的一个产品,而不是一个还在"用"AI 的产品。

而这次更新最关键的变化,就是让 artifacts 里面跑的代码,可以反过来再去调用 Claude 的 API。这听起来有点像俄罗斯套娃,我们梳理一下这个链条,你在 Claude 对话里,让 Claude 帮你写一个应用,Claude 写出来的这个应用,本身内部嵌入了访问 Claude API 的逻辑,然后你把这个应用分享出去,任何人打开这个链接,在他们自己的浏览器里跑这个应用,这个应用在运行过程中,如果需要用到 AI 的智能,比如说要理解用户输入的一段自然语言,要生成一段对话回复,它会实时再去调用一次 Claude,而这次调用是用打开这个链接的人的身份和额度去付费的。

这个设计带来的体验上的转变,其实非常微妙但很关键。以前的 artifacts,你可以理解成"作品展示",它展示的是 Claude 一次性写代码的能力。而现在的 artifacts,变成了"活的应用",它展示的是 Claude 持续提供智能服务的能力。这是一个从静态到动态、从展示到运行的本质转变。

我们具体看看文章里举的几个案例,能帮我们更直观地理解这个能力到底能干什么。第一个案例是 AI 驱动的游戏,里面的 NPC,也就是游戏里的非玩家角色,能够记住跟玩家之前的对话内容,并且根据玩家做出的选择动态调整自己的反应。这在以前是很难想象的,因为传统的网页小游戏,逻辑都是写死的剧情树,你选 A 走向结局一,你选 B 走向结局二,顶多复杂一点变成更多分支,但终归是有限的、写死的脚本。而现在,因为这个 NPC 背后是真的在调用 Claude 进行实时的语言理解和生成,它可以表现出某种程度的"涌现性",也就是说,设计者并没有把每一句对话都提前写好,而是让 AI 自己根据上下文去生成,这就让小游戏里的角色第一次有了一点"活人感"。

第二个案例是学习工具,能根据每个学习者的水平自动调整难度,提供个性化的辅导。我们可以想象一个学英语单词的小应用,以前的做法可能是固定的题库,所有人做的都是同一套题。但如果背后接入了 Claude 的实时生成能力,这个应用就可以观察你做题的正确率和反应速度,实时生成难度更匹配你水平的题目,甚至用你刚才答错的那个知识点,变出新的例句来加深你的理解。这种"因材施教"的体验,在没有大模型之前,是只有真人家教才能做到的事情,而现在变成了一个普通开发者用自然语言就能描述出来、让 Claude 帮你实现的小工具。

第三个案例,数据分析应用,用户可以上传 CSV 表格,然后用自然语言提问。这个场景其实是我们工作中特别常见的需求。假设你是公司里负责整理销售数据的同事,你每个月都要导出一份 CSV,然后筛选、汇总、画图,给老板看一份简报。这个流程本身工作量不算大,但很琐碎,而且每次问题稍微变一下,比如老板突然问"那华南区上个季度的环比增长率呢",你又要重新筛选重新算。如果有一个小应用,你把 CSV 拖进去,然后直接用大白话问它问题,它在背后调用 Claude 去理解你的表格结构和你的问题,直接给你答案甚至图表,这种体验会比你自己写一套 Excel 公式或者一套 Python 脚本要顺畅很多,而且这个应用还可以分享给同部门的同事一起用,因为消耗的是各自的账号额度,不存在谁要补贴谁的问题。

第四个案例,写作助手,从写剧本到写技术文档都能帮上忙。这个我们比较好理解,因为很多人已经在用 Claude 本身做写作辅助了,这里的区别在于,你可以把这种写作辅助"产品化",变成一个针对特定场景定制的小工具,而不是每次都要重新打开 Claude 对话框,重新写一遍很长的提示词去说明你的写作要求。

第五个案例是 agent 工作流,也就是那种需要协调多次 Claude 调用才能完成的复杂任务。这其实是最有想象空间的一类应用。我们可以设想一个场景,你想做一个自动整理会议记录的工具,这个工具背后可能需要分几步走,第一步先把录音转成文字,第二步用 Claude 提炼出关键决策点,第三步再用 Claude 把这些决策点按照负责人分类整理成待办事项列表。这种多步骤、有编排逻辑的复杂流程,以前需要你自己写后端服务、管理状态、处理各种异常情况,现在可以直接让 Claude 在 artifact 里面把这整套编排逻辑写出来,变成一个可以分享、可以复用的小应用。

讲到这里,我们要说说这篇文章里特别强调的一点,就是 Claude 写的是"真实的代码",这句话听起来平平无奇,但其实背后有一个很重要的产品哲学。

很多做"无代码"或者"低代码"工具的产品,本质上是把功能藏在一个黑盒子里,你拖拖拽拽配置一些参数,系统帮你跑起来,但你看不到、也改不了底层到底是怎么实现的。Anthropic 这次的设计理念跟这个不一样,他们强调你能看到 Claude 写出来的真实代码,你可以阅读它,可以修改它,可以自由地分享、复制、二次开发,这就是文章里提到的可以 fork,也就是分叉复制别人的 artifact,在别人做的基础上改成你自己想要的样子。

这一点对工程师听众来说其实特别有意义。因为这意味着,这不是一个让你彻底"放弃理解底层逻辑"的傻瓜工具,而是一个降低了"从想法到可运行原型"这个过程门槛的工具,但门槛降低之后,如果你想深入,你依然有路径可以深入,你可以去读 Claude 写出来的 React 代码,理解它的状态管理逻辑,理解它怎么组织对 API 的调用,然后根据自己的判断去调整、去优化。这种"既能让小白上手,又不剥夺专业人士掌控权"的设计,我个人认为是这类工具能不能真正被工程师群体接受的关键。

接下来我们说说具体能做什么,以及目前还做不到什么,这部分对大家实际动手尝试会很有帮助。

文章里明确列出了三件你可以做的事。第一,在你的 artifact 里使用 Claude 的 API,这是核心能力,也是我们前面讲了很多的那个机制。第二,可以处理文件,并且用 React 这个前端框架构建丰富的用户界面,这意味着你做出来的不只是一个简陋的文本框,而是可以有按钮、有图表、有交互动画的真正意义上的网页应用。第三,可以查看、分叉、自定义任何别人做的 artifact,这就形成了一种类似开源社区的协作生态,好的想法可以被别人看到、学习、改造,变成更好的版本。

而当时的三条限制,也值得我们仔细说一下,因为这直接决定了你能拿这个功能干什么、不能干什么。

第一条限制是不能调用外部 API。也就是说,你做的这个 artifact,只能跟 Claude 自己的服务对话,不能再去连接比如说天气查询接口、股票行情接口,或者你公司内部的数据库接口。这个限制其实砍掉了很多更复杂的应用场景,比如说你想做一个旅游规划助手,理想状态下它应该能实时查机票价格、查酒店空房情况,但在这个限制下,它只能基于 Claude 已经知道的知识和你给它的信息来做规划,没法去抓取实时数据。

第二条限制是没有持久化存储。意思是,你这个应用里产生的数据,用户输入的内容,对话的历史记录,一旦页面关掉或者刷新,可能就没了,没有数据库帮你把这些东西保存下来供下次访问的时候继续用。这对于很多想做"长期陪伴型"应用的人来说是个不小的约束,比如说前面提到的那个会记住玩家选择的游戏 NPC,如果没有持久化存储,这种"记忆"很可能只能停留在当前这一次会话里,你关掉浏览器重新打开,它大概率就不记得你之前做过什么选择了。

第三条限制是只能用基于文本的补全接口,也就是说,你不能调用 Claude 更高级的一些功能模块,比如说工具调用,也叫 tool use,或者其他更复杂的 agent 编排能力,只能是最基础的"给一段文字输入,拿到一段文字输出"这种最朴素的交互方式。

了解了这三条限制之后,我们大概能判断出来,这个功能在刚推出的时候,定位其实非常清晰,它是给"轻量级、实验性、社交分享型"的小工具用的,而不是给那种需要长期运营、需要复杂数据持久化、需要对接很多外部系统的正式产品用的。这个定位本身没什么不好,因为大部分人想做的,其实正是那种"灵光一闯,想试试看效果,做完想分享给朋友体验一下"的小东西,而不是一个要正式上线运营的商业产品。

那这篇文章对我们普通听众,也就是软件工程师和 AI 从业者来说,到底意味着什么,我们应该怎么去用它,接下来这部分我想多花点时间,因为这是整期节目里最实用的部分。

第一个层面,这个功能极大降低了"做一个能展示给别人的 AI 应用"的门槛。以前你如果想做一个调用大模型 API 的小应用并且分享出去,你大概要经历这么几步,先去申请一个 API key,然后写一个后端服务来转发请求,因为直接把 key 暴露在前端代码里是很危险的,这是一个基本的安全常识,API key 绝对不能写死在客户端代码里,否则别人打开浏览器的开发者工具,几秒钟就能把你的 key 偷走拿去用,你不仅要承担费用,还有可能违反服务条款。接着你还要把这个后端服务部署到某个云平台上,处理好域名、证书这些杂事,最后才能把链接发给朋友。这一整套流程,对纯粹的产品经理或者设计师来说几乎是不可逾越的门槛,即便对工程师来说,也是一个不小的"周末项目税"。

而现在,这整套流程被压缩成了,在 Claude 对话框里描述你想要什么,Claude 帮你写代码,你在旁边的 artifact 窗口里直接看到效果,觉得不满意就继续跟 Claude 聊,让它改,改好了直接点分享,生成一个链接发出去。中间所有跟密钥管理、后端部署、服务器运维相关的环节,全部被这个机制悄悄吸收掉了,因为运行的载体是 Claude 自己的基础设施,鉴权机制也是 Claude 自带的账号体系。这种"开发体验"上的简化,我觉得是这篇文章最值得工程师朋友们关注的地方,它本质上是在做一种"AI 应用分发"的基础设施创新,而不只是一个聊天机器人功能的小升级。

第二个层面,这个机制对独立开发者和小型团队特别友好,因为它从根本上改变了一个产品能不能"病毒式传播"的经济考量。以前你做一个小工具,如果担心传播开了之后自己扛不住成本,你可能会主动限制它的曝光,或者一开始就收费来对冲成本压力。现在因为成本被分摊到了每个使用者自己的账户上,你可以更大胆地把东西分享出去,不用担心"火了反而亏钱"这种尴尬局面。这对鼓励大家做实验、做创意尝试,我觉得是一个挺正向的激励机制。

我们可以想象一下,如果你是一个喜欢搞副业项目的工程师,以前你做了个东西放在 GitHub 上,顶多收获一些星标,想让普通人直接体验,你得自己掏钱搭一个 demo 站点,流量大了你还得考虑限流,甚至关站。现在你可以直接把它做成一个 Claude 里的 artifact,分享链接出去,不管多少人来用,你都不会收到一笔意外的账单。这种心理负担上的解放,其实比技术能力上的提升更重要,因为很多人放弃做产品,不是因为做不出来,而是因为担心做出来之后维护和成本的麻烦事。

第三个层面,我想提一下关于"信任"和"账号体系"的设计巧思。让使用者用自己的账号鉴权,这件事看起来只是一个技术细节,但它其实重新定义了开发者和用户之间的关系。以前你做一个 AI 应用给别人用,你是要承担"服务提供方"角色的,你要为对方的每一次调用负责,这其中隐含着一种信任和责任的转移,如果对方滥用你的工具狂刷接口,吃亏的是你。而现在这种模式下,用户是用自己的身份和自己的资源在使用这个工具,他们和 Claude 之间是直接的关系,你这个"开发者"角色,更像是一个创意提供方和代码编排者,而不是资源提供方。这种角色的重新划分,我觉得对于鼓励更多人大胆分享创意是有帮助的,因为责任边界更清晰了。

第四个层面,我想提醒大家注意这个功能的"实验"属性,以及它对我们理解 Claude 整体产品方向的意义。这篇文章发布的时候,这个能力还是以 beta 测试版的身份,开放给 Free、Pro 和 Max 这几档订阅计划的用户,后来在七月三十一号才扩展到企业级的 Team 和 Enterprise 用户。这种逐步放开的节奏,其实是大模型厂商一种很常见的产品策略,先在普通用户和创作者群体里小范围验证这个机制好不好用、会不会有滥用风险,等机制稳定了再推广到对安全性和稳定性要求更高的企业客户那里。

那么从这个节奏里,我们能读出什么呢?我个人认为,这其实反映了 Anthropic 在二零二五年中期那个阶段的一个战略判断,就是 Claude 不应该仅仅是一个"问答助手",而应该逐渐演变成一个"应用构建和分发平台"。如果你把 artifacts 这个功能放到更长的时间线里去看,从最早的静态代码展示,到现在能反向调用 API 形成"活"的应用,这其实是一条很清晰的演进路径,朝着让普通用户用自然语言就能创造出可分发、可使用的软件产品这个方向走。这条路径跟我们之前讨论过的很多大模型公司的"agent 化"战略,本质上是同一件事情的不同侧面,都是在探索,大模型除了陪你聊天之外,到底能不能成为一个真正意义上的生产力工具,能不能让"人人都是开发者"这句话从一句营销口号,变成一个实际可以兑现的能力。

说到这里,我也想从工程师的角度,给大家一些更具体的、关于"如果你想上手试试这个功能,应该注意什么"的建议,虽然这些建议是基于文章发布时的设计,大家实际操作前一定记得去核实最新的产品状态有没有变化。

第一个建议是,从小处着手,先做一些不依赖外部数据、不需要长期记忆的小工具去练手。比如说前面提到的那种数据分析助手,你上传一份 CSV,问几个问题,拿到答案,这种"一次性"交互的场景,正好避开了那两条关于没有持久化存储和不能调外部 API 的限制,完全可以在现有框架下实现得很好。

第二个建议,如果你确实需要某种"记忆"效果,可以考虑一些变通的做法,比如说让用户自己在对话过程中,把关键信息重新输入进去,或者设计成单次会话比较长、但是不依赖跨会话记忆的体验,这样可以一定程度上绕开持久化存储缺失的问题,当然这只是一个权宜之计,不是真正意义上的数据持久化方案。

第三个建议,对于想做更复杂应用的朋友,这篇文章其实也在给我们一个信号,就是这类轻量级、嵌入式的 AI 应用平台,未来很有可能会逐步补齐外部 API 调用、持久化存储这些能力。所以如果你今天在这个框架里搭建的小工具,设计的时候稍微注意一下模块化,把"调用 Claude 做智能处理"这部分逻辑和"展示界面、收集用户输入"这部分逻辑分得清楚一点,那么等以后平台功能扩展了,你的工具升级起来也会更顺畅,不用推倒重来。

第四个建议,是关于分享和传播心态的。因为这套机制把成本压力转移给了使用者自己,所以你做一个小工具分享出去的时候,可以更专注在创意本身上,不用过度焦虑"这玩意儿会不会火,火了我会不会破产"这种事情。这种心理上的轻装上阵,我觉得对鼓励大家多去尝试一些天马行空的小点子,是很有好处的。

最后,我想总结一下这篇文章想传达的核心信息,然后说说我自己的一点感想。

这篇文章的核心,其实就是讲了一件事,Claude 现在能让你做出真正会"动"的 AI 应用,而不只是静态的代码或者文档展示,这些应用可以直接调用 Claude 自己的智能能力,并且因为采用了"使用者自己账号付费"的巧妙机制,开发者完全不用为别人使用你的工具而承担费用,这就解决了过去很多独立开发者最头疼的"分享越广、亏得越多"的尴尬局面。文章举的那些例子,从游戏 NPC 到学习工具,到数据分析,到写作助手,再到复杂的 agent 工作流,本质上都是想告诉我们,这个能力的应用场景其实非常广,只受限于你的想象力,而不是受限于技术门槛。当然,在那个时间点上,这个功能还处于早期阶段,不能调外部接口、没有持久化存储,这些限制意味着它更适合做轻量、实验性的小工具,而不是正式的商业产品。

我个人的看法是,这篇文章看似只是一个产品功能更新,但放在更大的视角里看,它代表着大模型厂商正在认真探索"软件分发的下一种形态"。以前我们说软件分发,要么是应用商店,要么是网页托管,现在出现了第三种可能,就是直接寄生在大模型对话界面里的轻量应用,创作和分发的门槛被压缩到了极致。这种模式会不会成为未来的主流,我现在还不敢断言,毕竟它目前的限制还不少,但这个方向本身,我认为是值得每一个做 AI 相关工作的工程师持续关注的。如果你是那种喜欢周末折腾点小项目的工程师,我建议你真的去试一试这个功能,亲手做一个小工具,体会一下这种"想法到可分享产品"之间的距离被压缩到什么程度,这种亲身体验,比听我在这里讲十分钟都更有说服力。
