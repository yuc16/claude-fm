大家好，欢迎来到 Claude FM，我是你们的主播。今天我们一起来聊聊 Anthropic 在 2026 年 6 月 29 日刚刚发布的一篇文章。就在上周，这篇文章正式上线，标题是 Introducing the Claude apps gateway for Amazon Bedrock and Google Cloud，我给它的中文翻译是：为 Amazon Bedrock 和 Google Cloud 推出的 Claude 应用网关。如果你所在的公司正在用或者正打算大规模用 Claude Code 来辅助开发，但苦于安全、成本、策略这些管理问题，那这期节目很可能就是你一直等待的答案。这篇文章几乎重新定义了企业该如何把 AI 编码工具安全地、透明地、可控制地交到每一个开发者手上，所以今天我们会非常细致地，从宏观背景到每一个技术细节，再到你该怎么真正落地，把它讲透。

先花一点时间，把背景铺开。在过去的两年里，Claude Code 从一个早期内测工具，迅速成长为许多工程师终端里的常驻助手。用它写代码、改 bug、重构模块、甚至生成整个测试套件的体验，已经让大量团队尝到了甜头。但尝到甜头的同时，工程管理者、平台工程师和基础设施负责人，却开始频繁地在内部群里吐槽同样的三个痛点。

第一个是凭证把人逼疯。你要用 Amazon Bedrock 上的 Claude 模型，以前的标准做法是给每个开发者分配一个 AWS 的访问密钥，或者创建单独的 IAM 用户。同样，如果在 Google Cloud 上用，每人得有自己的服务账号密钥。这些长期有效的密钥分布到几十台、几百台笔记本上，离职交接时你得祈祷不要遗漏任何一处；而密钥一旦在 Slack 上误传或者被不小心提交到 GitHub，整个云账户就可能面临巨额盗刷。这根本不是开发者的错，而是这种模式本身就是脆弱的。

第二个痛点是成本看不见。每个开发者都在消费 token，但月底云账单只会合并显示一个总费用。你根本不知道这八千美元到底是谁花的，是核心算法组用了大量 Sonnet 来优化复杂逻辑，还是某个后端实习生开着自动化脚本批量跑了三天测试忘记关掉。工程总监想要引入 Claude Code 但又害怕预算失控，往往只能按兵不动。

第三个痛点是策略管不住。你想说，咱们团队能不能只用 Haiku，不要碰更贵的模型？或者能不能设置每次会话的上下文长度，避免 token 浪费？在没有网关的情况下，这些只能靠口口相传或者自己写一套复杂的代理层。而且如果有人绕过规则，你甚至没法发现，更别说阻止。

正是在这样一种集体焦虑中， Anthropic 看到了企业级落地的真正瓶颈不在于模型能力，而在于工程化治理。于是就有了今天我们要聊的这个核心方案：Claude apps gateway，一个完全自托管的控制平面。如果用一句话来概括它解决什么问题，那就是：把 Claude Code 在企业内的使用，从无政府状态变成井井有条的现代化服务。

好，我们现在进入文章最核心的部分。整个网关的设计哲学，我觉得可以用一个特别容易理解的类比来展开：把它想象成一座智能大楼的门禁系统。在没有门禁的旧时代，每个员工要进不同的会议室或者机房，都需要各自带着一大串物理钥匙。这些钥匙既容易丢失，又无法追踪谁在什么时间进了哪个房间。这也正对应了之前每个开发者都在本地配置云凭证的状态。而网关，就好比在大楼入口处设立了一个统一的前台，所有员工都只需要刷自己的工卡——也就是用公司统一的 SSO 账号登录一次。前台背后接入了整个建筑的权限控制系统，它知道你被允许进哪几层、用哪些设施，并且在每一次开门动作后都会记录下来。不同的是，这个门禁系统不是现成的硬件，而是一个你可以部署在自己基础设施上的无状态容器，外加一个 PostgreSQL 数据库。

文章详细介绍了网关是如何做到这一点的。它被设计为运行在 Linux 环境下的单个无状态容器，非常轻量。数据库只需一个标准的 PostgreSQL 实例。容器的职责就是扮演一个集中的凭证持有者和策略执行者。它手里有唯一一份上游凭证，这个凭证就是通往 Amazon Bedrock 或者 Google Cloud 的正式钥匙。而开发者机器上不再需要任何长期云密钥。网关直接跟你们公司的身份提供者交互，比如大家常用的 Google Workspace、微软 Entra ID 或者 Okta，验证每一个来访的开发者的身份。你如果想给一个新同事开通 Claude Code 权限，步骤简化到只需要在身份系统里把他的账号加到对应组或应用中就行，就跟给他开通邮箱一样简单。反过来，一个人离职，把他的账号从身份系统中移除的那一瞬间，网关里关于他的短期会话就会自然过期，所有访问立即中断，再也不用去一台一台电脑上刮掉残留的凭证。

这个身份体系在技术上的实现，是网关充当了 OpenID Connect 依赖方。这个词听起来有点抽象，我把它拆开说。当开发者打开 Claude Code 客户端时，如果网关模式已激活，客户端会主动拉起浏览器跳转到网关的登录地址。网关再把请求重定向到你公司的登录页面，开发者在那个页面输入自己的工号和密码，完成多因素认证。身份提供者验证成功后，向网关返回一个确认。网关就根据这个信息临时签发一个短期的会话令牌，交给客户端。整个过程开发者机器上从来没有、也不会存储任何密码或者长期凭证。所有的秘密只停留在网关和身份提供者之间。而且因为网关容器在你自己的服务器上运行，你可以完全控制它的安全加固、日志审计和访问控制。Anthropic 只把软件和协议交给你，凭证和数据全部留在你的网络内部，他们看不到。

再来说策略能力，这是让管理从被动变为主动的关键。在没有网关时，要想统一推送一个配置，比如规定大家只能用 Claude 三点五 Haiku，你可能得写一封全员邮件，或者更高级一点，把配置写死在共享脚本里。但这都太容易被绕过。而网关的策略执行是双重的。首先，你作为管理员在网关上一次性定义好一套管理设置，这些设置会以托管策略的形式存在。当开发者通过网关登录后，客户端自动下载并应用这些策略。你可以规定允许使用的模型列表，禁止 Opus 或者只允许 Sonnet 和 Haiku；可以设定默认的采样温度，防止开发者随意调高导致质量不稳定；还可以设置最大上下文长度，避免某个提示词一次性塞进二十万 token 导致费用飙升。这些配置会安静地加载到开发者的 Claude Code 环境中，他几乎感受不到限制，只是当他尝试选择一个被禁用的模型时，才会收到拒绝提示。

更厉害的是，网关不仅在登录时分发策略，它还在每一个请求上做强制执行。即使开发者通过各种手段在本地修改了配置，想越权使用更贵的模型，当请求到达网关时，会被立刻拦截并拒绝。这种服务端二次校验的设计，杜绝了客户端篡改的可能性。你可以把它理解成，大楼的前台不仅给你发了一张标有权限区域的门禁卡，而且每一扇门内还有一个独立的刷卡器在二次确认。即便你在卡上私自刻了更多的区域编码，实际开门时依然会被拒绝。

遥测，这是很多工程负责人最关心的一点。过去，AI 编码工具的花费像是一个黑洞，月底才看到汇总账单，而网关把这一切透明化了。文章提到，Claude Code 客户端会在每一个请求上自动打上用量标记，包括消耗的 token 数量、属于哪个用户、调用了哪个模型、请求时间等元数据。然后客户端通过网关把这些信息以 OTLP 格式发送出去。OTLP 是 OpenTelemetry 的标准协议，已经广泛支持 Prometheus、Grafana、Splunk、Datadog 等几乎所有主流的可观测性平台。所以你可以直接把网关的遥测数据接入到现有的监控链路里，不需要再单独搭建一套新系统。这意味着，从网关上线的那一刻开始，你就可以在仪表盘上看到实时的、按开发人员维度细分的使用量和预估花费。谁今天已经用掉了三十美元，谁这个月一直压在五十美元以内，都一目了然。

这里特别值得强调的是，这些使用数据与实际的推理流量一样，全部流经你自己的网络，按照你自己的数据保留策略来处理。除非你主动在配置里把路由指向 Anthropic 的 Claude API，否则他们永远看不到你的数据和用量。这个设计给了注重数据主权的企业非常大的安全感。

聊完遥测，再来看路由能力。这个功能乍一看似乎只是简单的请求转发，但实际上它蕴含着极大的灵活性。网关作为唯一的推理出口，持有你在多个云服务商那里的上游凭证。你可以配置让请求优先发往 Amazon Bedrock，备用 Google Cloud。为什么需要这样？想象这样一个场景：你的 CI 流水线里集成了 Claude Code 来做自动化的代码审查和修复，如果 AWS 某个区域突然发生故障，Bedrock 不可用了，整个流水线就会报错中断，阻塞几十个开发者的提交。而有了网关的自动故障转移，它几乎可以在几秒内把请求无缝切换到 Google Cloud 上的 Claude 实例，同样模型、同样参数，开发者甚至都不知道背后发生过什么。当然，你也可以配置始终只使用一个提供商，完全由你决定。

路由还有一个隐藏的好处，就是减轻与云平台的耦合。如果你的公司未来要从主要用 AWS 转向多云的策略，或者需要在不同区域满足数据驻留要求，只需调整网关里的路由规则，而不用把几百台开发者机器的配置全部改一遍。所有的复杂性都被收敛到了网关这一个点上。

接着是花费上限。这个功能的出现几乎是必然的。当 AI 编码成为日常，预算失控的风险就不再是假设。文章里明确说，你可以设置每天、每周、每月的花费上限，而且这个上限可以作用在不同层级：整个组织、一个用户组，或者某个具体的用户。举个例子，你可以给所有暑期实习生分配到一个组，给这个组设置每天五美元的上限，确保他们在学习和实验时不会意外刷爆公司的 API 预算。而给核心编码组每人每月三百美元的额限，因为他可能每天有大量重构和生成代码的需求。这里的上限一旦接近或触及，网关会直接拒绝新的请求，直到进入下一个计费周期。和遥测结合起来，你还能够设置预警，比如花费达到额限的百分之八十时，由监控系统通过 Slack 或邮件自动通知本人和他的主管。这套组合拳，能让财务和工程主管都睡得安稳。

好，五个核心能力都讲完了，我们再来看一个贯穿全文的设计亮点，那就是网关与客户端同源构建。Anthropic 直接把网关功能打包到了同一个 claude 二进制文件里。也就是说，你下载的 Claude Code 命令行工具，本身就内置了网关模式。部署的时候，你只需要指定运行参数，就可以用同一个二进制文件启动一个网关容器。因为客户端和网关是在一个代码库里构建和测试的，它们之间的登录流程、策略同步、遥测上报全都是天然集成、原生兼容的。你不会遇到因为客户端版本太新、网关版本太旧而导致功能缺失的兼容性问题。再加上 Anthropic 还公开了网关使用的这一整套协议，这意味着，如果你所在的公司已经有一个基于 Envoy 或者 Kong 构建的统一 API 网关，希望把 Claude Code 的登录和策略控制也纳入到同一个管理平面下，你可以根据官方协议自己开发一个插件来实现，不必非得运行 Anthropic 提供的容器。这是一个非常开放的生态信号。

现在，让我们进入最落地的部分，也就是如果你被指派去在公司推这套网关，具体应该怎么做，有哪些坑必须提前踩掉。我会按照实际的部署路径，带你一步一步走一遍。

首先，准备基础设施。你需要一台 Linux 服务器，或者更现代的，在你的 Kubernetes 集群里分配一个 namespace。正如文章所说，网关是单容器无状态设计，这意味着你完全可以把它托管在一个现有的容器编排平台上，用 Deployment 来保证实例数量。数据库是唯一有状态的部分，文章推荐 PostgreSQL。你可能需要联系 DBA 申请一个专用的数据库实例，或者如果你有 RDS 或者 Cloud SQL 之类的托管数据库，直接申请就行。注意，这个数据库存储的是会话信息、策略快照和用量缓冲，不是海量数据，所以一开始不需要特别高的规格，但要启用自动备份。

第二步，准备上游凭证。你打算主用哪个云服务商？假设是 Amazon Bedrock，你需要一个具有调用 Bedrock 权限的 IAM 用户，并为它生成访问密钥。但绝不能把明文的密钥直接写在 gateway.yaml 文件里面。最安全的做法是使用你公司内部的密钥管理服务，比如 AWS Secrets Manager 或者 HashiCorp Vault，在容器启动时以环境变量或者挂载文件的形式注入密钥。同理，如果用 Google Cloud，用服务账号密钥，也采用类似的方式。这个上游凭证是整个网关安全的核心，一旦泄露，攻击者就可以以你的名义调用模型，所以务必谨慎。

第三步，对接身份提供者。登录到你公司的 Google Workspace 或者 Entra ID 管理员后台，注册一个新的 OIDC 应用。在注册过程中，你需要填写回调地址，这个地址就是你的网关对外暴露的域名加上 /login 路径，比如 https://claude-gateway.yourcompany.com/login。注册完成后，你会得到一个客户端 ID 和客户端密钥。把它们也写入网关的配置里。然后你在身份系统里配置哪些用户或组有权限访问这个应用。这一步做完，权限的最终控制就完全掌握在你的身份提供者手中了。

第四步，部署网关容器。把 gateway.yaml 配置文件准备好，里面包含 OIDC 提供者的信息、上游凭证的引用路径以及数据库连接字符串。如果你用 Kubernetes，就创建一个 ConfigMap 和一个 Secret，分别存放配置和凭证，然后定义 Deployment。启动后检查日志，确保网关能成功连接到数据库和身份提供者，并且能够完成一次测试登录。你可以先用自己账号在浏览器里访问网关地址，看是否能正常跳转登录。

第五步，也是让开发者无感知的关键一步，就是向客户端推送 managed setting。文章里提到两个核心参数，forceLoginMethod 和 forceLoginGatewayUrl。你需要准备一个 managed-settings.json 文件，将 forceLoginMethod 设置为 sso，将 forceLoginGatewayUrl 设置为你的网关地址。这个文件该如何分发到每个开发者的机器上呢？如果你的公司已经使用设备管理工具，比如 Jamf 或 AirWatch 管理 Mac，或者通过 Chef、Ansible 管理 Linux，那就可以把文件下发到 Claude Code 的配置目录。对于没有这类工具的团队，你也可以在内部 Wiki 上提供下载链接，让开发者在终端里跑一句 curl 命令把文件放到指定位置。当开发者下一次启动 Claude Code 时，客户端检测到这个 JSON 里的强制登录策略，会立刻拉起浏览器引导完成 SSO 认证，之后每一次敲命令，都自动经过网关。

到这里，你可能已经在设想实施过程中会遇到的几个现实问题，我来结合一些已知的实践提前给你提醒。

第一个是网络架构。网关必须能让所有开发者访问到。如果你的开发者分布在全国甚至全球各地，并且公司的网络并没有全互联，那就需要考虑把网关部署在公网可达的位置，但必须在前面挂一个反向代理，比如 Nginx 或者云负载均衡器，配置严格 TLS，并且最好配合 IP 白名单或者零信任网络访问方案，只允许来自公司设备或公司 VPN 的流量进入。还有一个细节，OIDC 登录流程需要开发者的浏览器能够同时访问身份提供者和网关，所以公网可达性必须同时考虑这几个组件。

第二个是高可用。如果你全公司一百多个开发者都依赖这个网关，一旦网关容器挂掉，所有人的 Claude Code 终端都会报错中断，这可能会瞬间引爆工作群的抱怨。因此从一开始就要设计多实例部署。由于网关无状态，你完全可以同时跑三个副本，放在负载均衡器后面。数据库本身应当做主备或者使用托管的高可用版本。另外，你放在负载均衡器后的网关实例，最好配置健康检查，监控能否正常访问数据库和完成模拟登录测试。

第三个是策略的渐进式推行。我建议不要一上来就把所有限制都拉满，比如直接禁用 Sonnet，只留 Haiku。可以先只设定花费上限，然后观察一周的遥测数据，了解各个团队的实际使用模式。你可能会发现，有些组日常使用量很小，但偶尔需要 Sonnet 来做复杂的算法优化，这种偶发需求如果被完全禁掉，反而阻碍效率。你可以针对不同组制定不同的模型白名单，让策略既有底线又有弹性。策略的修改在网关上直接生效，下一次开发者发起请求就会被应用，不需要他们重启客户端。

第四是成本归因的闭环落地。遥测数据进入到你的监控系统后，你可以建立按团队归类的仪表盘。更进一步，可以导出数据做月度结算。比如你们公司内部采用 chargeback 机制，IT 成本要精确分摊给每个业务单元。以前没有细粒度数据，只能按人头分摊，现在有了每个用户的 token 消费和对应的价格，可以真正做到用多少付多少。财务同事会因此对 AI 编码的推广投下赞成票，因为他们看到了可预测的支出。

另外，文章提到网关不会向 Anthropic 发送推理流量或使用数据，除非你配置它使用 Claude API。这是一个在数据安全层面非常重要的区分。如果你想享受完全私有化的部署，数据不出企业网络，那么可以选择只配置 Bedrock 或者 Google Cloud 的路由，甚至走 VPC 内的私有端点。当你的合规部门问起数据会不会被外部模型厂商看到时，你可以笃定地回答：推理流量和用量记录都闭环在我们的基础设施上，Anthropic 不收走任何数据。这有助于打消很多数据敏感行业，比如金融、医疗的最后一点顾虑。

最后，关于开放协议产生的生态影响，我想多聊两句。Anthropic 公开了网关与客户端交互的协议，这意味着第三方网关开发者可以基于同一个协议实现与官方客户端兼容的 SSO 和策略强制功能。这对已经有成熟内部 API 网关体系的组织意义重大。假如你们公司已经有一个全公司共用的大一统 API 网关，所有外部 API 调用都必须经过它，那么你可以招募一个平台工程师，参考 Anthropic 的协议去编写一个插件，让这个已有的网关也能够完成 Claude Code 的认证和策略控制。这样一来，你就不需要额外维护 Anthropic 网关容器这一整套部署，直接复用现有的安全、监控和审计设施，减少了架构的异构性。当然，对于没有这种内部平台团队的多数公司而言，直接跑官方容器是最快出效果的。
