大家好，欢迎收听 Claude FM，我是你的主播。今天这期节目，我们来聊一篇发布在 Anthropic 官方网站上的文章，英文标题是 Best practices for getting started with Claude Cowork，我把它翻译成：使用 Claude Cowork 的最佳实践，从零开始的上手指南。这篇文章发表于二零二六年六月三日，距今大概半个月，内容非常新鲜。

这篇文章有一个很特别的地方，它不是 Anthropic 的产品经理写的功能介绍，也不是工程师写的技术文档，而是 Anthropic 增长营销团队的负责人 Austin Lau 亲自写的。他用自己的亲身经历告诉你，一个不懂写代码的知识工作者，是怎么用 Claude Cowork 把日常工作里最烦人、最枯燥的部分彻底托管出去的。

我觉得这个视角非常宝贵。关于 AI 工具的文章，大多数要么是技术人员写给技术人员的，充满了 API 接口、系统架构这类内容；要么是 AI 公司写的产品宣传文章，充满了"革命性"和"颠覆性"这种词，但具体怎么用，看完之后还是不知道。而这篇文章是一个真实在用这个工具的人，用自己的工作案例，诚实地告诉你哪些活可以交给 Claude Cowork、哪些不能、怎么才能上手。这种一手的使用经验，对我们每一个在思考如何用好 AI 工具的人来说，都有很高的参考价值。

好，我们先花一点时间建立一些背景，这样后面讲到具体内容的时候大家会有更好的感觉。

我们先回溯一下 AI 工具这几年的演进轨迹。文章开头 Austin 就做了一个很好的时间线梳理。二零二四年，我们用 AI 的方式基本上是这样的：打开一个聊天窗口，把问题或者文字粘贴进去，得到一个回答，然后你得自己把这个回答转化成有用的东西。你可能有体感——我同时开着一个 AI 聊天窗口，旁边还开着 Word、Excel、邮件客户端、各种文档，然后在这些标签之间不停地来回切换、复制、粘贴。AI 给了你一个答案，但是从"答案"变成"可以交付给别人的成果"，这段路还是你自己走的，AI 只是一个聪明的参谋，执行还是你。

二零二五年，Claude Code 出现了，主要面向软件工程师。它让开发者可以把整个软件开发流程——写代码、跑测试、修 bug、调整结构——都委托给 AI 来做，工程师们的生产力出现了质的飞跃。Austin 说，旁边不做代码工作的同事，看着开发团队用 Claude Code 以前所未有的速度交付，多少有点羡慕。

Austin 说，他去年就开始用 Claude Code 来处理一些长的、多步骤的任务，尽管他本人是做营销的，对写代码并不在行。他说他在一周之内就从"不知道终端是什么"变成了"可以用 Claude Code 在三十秒内完成原本需要三十分钟的任务"。但那时候 Claude Cowork 还不存在，他是在用一个本来是给开发者设计的工具，凑合着来处理非技术的工作。

Claude Cowork 就是在这个背景下出现的。它是 Anthropic 专门为非技术知识工作者设计的 AI 工作助手，集成在 Claude 桌面应用里。你不需要知道什么是代码，不需要知道什么是智能体（也就是 agent），也不需要知道怎么构建自动化流程。你只需要能描述清楚你想要什么结果。Austin 说，现在他九成的工作都在 Claude Cowork 里完成。这就是今天这期节目的主角。

好，现在我们来看文章的第一个核心框架：chat、Claude Cowork 和 Claude Code，这三者有什么区别，分别适合什么情况？

文章说，这三者在技术底层是一样的，同样的 Claude 模型在驱动这三个产品。但它们是为不同类型的工作而设计的独立工作空间。你可以这样来理解：同一套大脑，但根据不同的工作场景，给它配备了不同的工作台和不同的工具。

Chat，也就是我们最熟悉的聊天模式，基本逻辑是"你把工作带给 Claude"。你复制一段文字过来，上传一个文件，然后让 Claude 帮你分析或者回答。Chat 最适合的场景是：你需要一个答案、一个想法、一个解释，或者只是想把脑子里一个模糊的问题快速理清楚。你问一个问题，Claude 给你一个回答，交互是即时的、来回的。

Claude Cowork 是反过来的逻辑——"你把 Claude 带到你的工作里"。你告诉它你的 Google Drive 文件夹在哪、你的 Slack 频道有哪些、你用什么工具来管理内容，然后给它下任务。它去读你的文件，连接你的应用，跨多个平台把活干完，最后交给你一个成品。不是一个答案，是一个文件、一个报告、一个表格——一个可以直接发出去的东西。

Claude Code 是专门给开发工程师设计的，你的工作在代码里，就用这个。今天我们不展开这块，专注在 Claude Cowork 上。

我来用一个更具体的类比帮大家理解 chat 和 Claude Cowork 之间的本质差别。

Chat 相当于你在请教一个很聪明的顾问朋友。你说"我这周有个业务评审会议，你觉得我应该讲哪些内容？"，他给你出主意，告诉你应该覆盖哪几个模块、从什么角度切入，然后你自己去写 PPT，自己去准备数据，自己去组织语言。他给了你思路，但活还是你的。

Claude Cowork 相当于你在委托一个专业的助理。你说"帮我去公司的共享文件夹里读一下过去三个月的会议记录，然后用我们公司的 PPT 模板，帮我做一个季度业务回顾的演示文稿，重点突出三个项目的进展和下季度的计划"，然后你去处理其他的事情，一段时间之后回来，演示文稿做好了放在那里等你，你只需要检查一遍，把不满意的地方改一改。

这个区别不只是使用体验上的差异，背后反映的是一种根本性的工作模式转变：从"AI 辅助你做事"到"你委托 AI 做事"。前者你是主角，AI 是出主意的参谋；后者你是决策者，AI 是执行者。

我个人认为，这种转变对很多知识工作者来说需要一个心理上的适应过程。因为它要求你真正"放手"——你要相信 AI 能把活干完，而不是时刻盯着它的每一步、每一个细节。这个信任的建立需要一些时间，但一旦建立起来，你会发现整个工作方式都不一样了。你的时间开始大量集中在真正需要你判断力的地方，而不是消耗在那些机械性的中间步骤上。

好，说完了三者的区别，我们来看一个更实用的问题：到底什么情况下应该用 Claude Cowork，什么情况下用 chat 就够了？

Austin 给了一个非常干净的判断法则：如果你想要的是一个想法、一个解释、一次头脑风暴或者一个快速的确认，用 chat；如果你想要的是一个可以交付给别人的成果，用 Claude Cowork。

文章里有一组非常好的对比，让我用口头的方式给大家讲一遍，你在脑子里对比一下。

第一组：你问"我们的季度业务评审会议应该覆盖哪些内容？"——这是 chat 的活，一个思路性的问题，chat 几秒钟给你答案。但如果你说"帮我去 Google Drive 里这个文件夹里读一下过去三个月的会议记录，然后用我们公司的模板给我做一个季度业务回顾的演示文稿"——这是 Claude Cowork 的活，有具体的数据来源，有明确的输出格式，是一个可以直接用的成品。

第二组：你问"Excel 里 VLOOKUP 函数怎么用？"——chat 三秒钟给你解释完。但如果你说"帮我把这几个 Excel 表格里所有的 VLOOKUP 都替换成 INDEX MATCH"——这是 Claude Cowork 的活，它需要实际操作多个文件，完成真实的修改。

第三组：你说"帮我给这个网页想一个更好的标题和页面描述"——chat 来做，给你几个选项参考。但如果你说"用这个表格里的新标题和描述，把这三十个页面全都更新掉，通过我的内容管理系统接口来写入"——Claude Cowork，涉及多个页面、外部系统、实际的数据写入。

你有没有感觉出来这里的规律？关键区分不是任务难不难，而是任务的输入和输出是什么形态，以及谁来执行中间的过程。Chat 是你给一个问题，得到一个想法，然后你去执行。Claude Cowork 是你给一个任务，AI 去执行，你得到一个成品。

文章里 Austin 也提到了两种最常见的错误。第一种：什么事情都用 chat，永远感受不到 Claude Cowork 能带来的那种质变。这是最普遍的错误，很多人发现 chat 挺好用的，就一直停在那里，没有迈出去试一试 Cowork 的步子，错过了真正大幅提效的机会。第二种是相反的错误：把什么事情都往 Claude Cowork 里扔，结果用它来处理一些 chat 五秒钟就能回答的简单问题，在那里等啊等，浪费时间。两种错误都是对工具特性认识不清晰的表现，找准什么事用什么工具，才能真正提效。

下面我们进入文章里我认为最有价值的一个部分——Austin 给出的一个框架，用来判断一个任务是不是适合交给 Claude Cowork 的。他叫它"五个要素"，你不需要全部满足，但满足得越多，这个任务就越适合放进 Claude Cowork 里来做。

第一个要素：有多个输入。如果你要处理的只是一个文件或者一段文字，chat 基本上就够用了。但如果你需要整合多个文件、一整个文件夹、或者来自不同应用的数据——比如同时读取 Gmail 和 Slack，或者同时拉取来自 Google Ads 和 Meta Ads 的广告数据——这就是 Claude Cowork 的强项。因为它可以同时连接多个数据源，在不同的应用之间自由穿梭，这是聊天窗口根本做不到的事情。

第二个要素：输出是一个文件。你的最终成果是不是一个你可以发邮件、分享链接、打印出来、或者放进会议里让别人看的文件？是报告、是演示文稿、是数据表格、是 CSV——如果是这类有形的、可以交付出去的成品，Claude Cowork 更合适。

第三个要素：这件事你会反复做。一次性的任务当然也可以放进 Claude Cowork，但真正体现其价值的是重复性任务的自动化。你可以给它设置定时运行，让它在你还没到办公室的时候就把活干完。Austin 的例子是每天早上六点自动运行的日报任务，你在睡觉，它在工作。这种"躺着提效"的感觉，是 chat 完全给不了你的。而且一旦你把一个重复任务设置好，它就会一直跑下去，你几乎不需要再花额外的时间。

第四个要素：你知道好的输出长什么样。这个标准非常关键，但经常被人忽略。为什么"你必须知道好的输出长什么样"？因为 Claude Cowork 很快就能给你生成一个看起来很像样的东西，但如果你自己对这件事没有判断力，你就没办法在十五秒内判断它是对的、是需要调整的、还是完全跑偏了。

这里有一个值得深想的点。AI 工具能放大你已有的能力，但放大的前提是你在这个方向上已经有一定的认知和判断力。如果你对某个领域完全陌生，Claude Cowork 给你生成了一份报告，你看不出来对错，你也就无法迭代和改进。这种情况下，更好的路径是先用 chat 建立基础认知，等你能判断什么是好的、什么是差的，再用 Claude Cowork 来提效。这个顺序不能颠倒。

第五个要素，也是 Austin 说的最精彩的一个：中间过程是无聊的。他原文里有一句话，翻译过来大概是：思考发生在开始和结束，中间那段才是你应该交出去的。这话说得非常透彻，我来帮大家理解一下这是什么意思。

想想任何一个知识工作里的典型任务。在开始的时候，你在做什么？你在思考：我要什么，这个任务的目标是什么，什么样的结果算成功，有什么边界条件需要注意。在结束的时候，你在做什么？你在判断：这个结果对不对，哪里需要调整，有没有遗漏什么重要的信息，能不能直接发出去。这两端，是真正需要你的经验、你的判断力、你的业务理解的地方，没有人能替代你。

那中间那段是什么？提取数据、整合来源、格式转换、重新排列、对比两组数字、把三个文件的内容合并成一个、把一百行数据整理成一个摘要——这些机械性的操作，它们没有什么技术难度，也不需要什么特别的创造力，但它们枯燥，它们消耗你大量的时间，它们让你觉得自己在做搬砖的工作。这才是你应该让 AI 来做的部分。

你可以用这五个要素回想一下你自己的日常工作，看看哪些任务可以放进 Claude Cowork。邮件汇总：多个来源进来，报告文件出去，每天都要做，你知道好的汇总长什么样，中间的整理工作是纯机械的——五个要素全满足，完全是 Claude Cowork 的活。每周的项目周报：从各个地方提取项目状态，整合成一份文档，每周重复做，你知道什么叫合格的周报，中间的整理很无聊——同样是满分。月度数据汇总、竞品分析整理、客户沟通记录归档……这些都是。

但你的同事发来一条消息问你"这个数据为什么会有这么大的波动"，让你快速解释一下背后的原因——这种事 chat 更快，直接在聊天里说清楚就行，没有必要绕到 Claude Cowork 去做一份正式报告。

好，理论框架说完了，我们来看 Austin 在 Anthropic 内部实际用 Claude Cowork 做什么。他举了三个非常具体的例子，都是他自己日常工作里真实的案例，我一个一个给大家拆解，并且帮大家想想在自己的工作里可以怎么平移。

第一个例子：每日简报。Austin 做增长营销，每天要处理的 Slack 消息和邮件数量是"压倒性的"，他自己这么形容的。他设置了一个每日简报任务，每天早上六点自动运行。Claude Cowork 连接到他的 Slack 和 Gmail，读取他的未读邮件和他关注的那几个频道，然后把信息分类整理，生成一份简短的报告。

这份报告包含什么内容？哪些事情需要他重点关注的摘要，按类型分组的邮件标记，各个频道的内容概览，以及任何可能影响营销计划的产品相关事件或者线上问题。

他早上醒来，这份报告已经在那里等他了。他不需要自己去刷邮件，不需要一个个频道点进去，不需要花二十分钟来了解昨晚发生了什么。所有他需要知道的信息已经被整理好了，他可以在五分钟内完成信息消化，然后直接决定今天先处理什么。

这个例子命中了五个要素里的哪几个？多个输入——Slack 和 Gmail 两个不同的平台，还有不同的频道和收件人；有文件输出——一份结构化的日报；重复性——每天自动运行；他知道好的简报长什么样——他自己设计的格式；中间是无聊的——把几百条消息归类汇总，这种活谁做谁烦。五个要素全中。

我来帮大家做一个平移，看看这个"早晨简报"的模式在不同的工作角色里可以怎么用。做项目管理的同学，可以让 Claude Cowork 每天早上汇总各个项目的最新状态，把需要你注意的风险点和阻塞项整理出来，让你上班第一件事就是做判断，而不是花时间收集信息。做销售的同学，可以让它汇总 CRM 系统里昨天的客户互动记录，提醒你今天需要跟进哪几个潜在客户、上次的承诺有没有落实。做研发或者技术管理的同学，可以让它每天汇总项目里的代码提交动态，把需要你关注的变动提炼出来。这个模式几乎适用于所有需要多线程处理信息的工作角色，方向对了，直接套就行。

第二个例子：广告预算追踪。Austin 说，广告预算的日常追踪是他最不想做的工作之一，原因很简单——枯燥、重复、而且全是手工操作，没有任何智识上的挑战性。传统的做法有两种：要么是每天手动从各个广告平台导出数据，然后复制粘贴进 Excel 表格，手动计算执行进度；要么是花钱买一个第三方工具来帮你自动化这个提取、转换、加载数据的过程，但这要额外付费，而且配置起来也麻烦。

他的解决方案是：让 Claude Cowork 直接连接到 Google Ads 和 Meta Ads，然后生成一个实时的仪表盘，自动拉取每天的广告花费数据，计算预算执行进度。仪表盘是以 HTML 的格式生成的，在桌面应用里就能看。更重要的是，他可以用自然语言告诉 Claude 怎么过滤和分析数据——比如"只看这几个国家的广告活动"、"只看这个产品线"、"把花费异常波动的活动标记出来"——不需要写任何公式、不需要用任何查询语言，用说话的方式来调整分析的维度。而且广告平台已经通过接口连接好了，他随时想更新数据，直接刷新就行。

这个例子我觉得特别有启发性，因为它打破了一个很常见的思维定式：很多人觉得"自动化数据报表"是技术团队的事情，需要工程师来写代码、配接口、搭数据管道，普通的业务人员接触不到这件事。但 Claude Cowork 让非技术背景的人也能直接驱动这件事，只要你能把你想要什么说清楚，执行可以全部交给 AI。

这背后是一个更大的趋势：AI 工具正在消除知识工作者和工程实现之间的那道鸿沟。以前，你有想法，需要排期、找工程师来实现；现在，相当一部分以前需要工程实现的事，直接用自然语言驱动就能做到了。这对非技术背景的人来说，是一个非常重要的能力解锁。

第三个例子：搜索排名报告。这个例子相对简单一些，但很能说明 Claude Cowork 在消除重复劳动上的实际价值。Austin 把 Claude Cowork 连接到 Google Search Console，让它自动拉取他关心的搜索词数据、来自不同国家的流量分布、具体页面的表现数据，然后整合进一个统一的表格里。

这里有一个细节值得注意。他还会给 Claude 具体的分析指令，比如"重点对比最近七天和前七天的变化"、"只看特定几个国家的数据"、"把变化幅度显著的项目单独标记出来"、"用这个报告模板来写最终的结论和分析"。这些都是有判断力的指令，他在开始的时候就把这些标准和上下文告诉了 Claude，让 Claude 在执行的时候按照他的思路来走。之后他还可以继续问 Claude 追加的问题，或者让它调整某些部分。

他说这个报告以前每周要花他三十分钟，现在只用五分钟。更重要的是，这五分钟全都花在真正需要他判断力的地方：填充 AI 不知道的背景信息，审核和打磨最终输出的结论和表述。那二十五分钟的机械性整理工作，消失了。不是压缩了，是彻底消失了。

这三个例子背后有一个共同的结构，我来帮大家总结一下，这也是你构建自己的 Claude Cowork 工作流时可以直接套用的模式。你需要做三件事：第一，告诉 Claude 数据从哪里来，让它连接到相关的数据源或者文件；第二，告诉 Claude 你想要什么格式的最终输出，给它一个模板参考更好；第三，告诉 Claude 在分析时应该关注什么角度、以什么标准来判断好坏。把这三件事说清楚之后，让它去干，你去做别的。这就是 Claude Cowork 核心的工作方式。

好，我们现在来看怎么上手。文章给出了一个"十分钟上手"的步骤，我来一步步给大家讲，结合一些我自己的理解。

第一步，打开 Claude 的桌面应用，切换到 Claude Cowork 的标签页。

第二步，给 Claude 一些东西来处理。可以是几个本地文件，可以是一个文件夹，也可以连接一个你日常用的应用，比如 Slack、Gmail、Notion，或者你们公司的客户关系管理系统。这个"连接"的操作，Claude Cowork 通过接口插件来实现，大部分主流应用都有支持。

这里有一个非常重要的认知点，Austin 专门强调了，我觉得值得在这里多说几句：Claude Cowork 的输出好不好，关键不在于你的 prompt 写得有多精准有多完美，而在于你有没有给 Claude 足够丰富的上下文和原材料。这个认知和 chat 的使用思路是不一样的，需要切换一下脑子。

很多人在用 chat 的时候，花了大量时间研究"怎么写 prompt 才能让 AI 给出更好的回答"，甚至去找各种 prompt 技巧和模板。但在 Claude Cowork 里，你要做的是"给 Claude 更多的原材料"——你的文件、你的数据、你的历史记录、你公司内部的模板、你的个人偏好——这些上下文才是决定输出质量的关键。

这个道理其实很好理解，我来打个比方。你让一个刚加入公司的助理帮你整理月度销售报告，如果你只告诉他"帮我整理一下上个月的销售数据"，他不知道格式应该是什么，不知道你最关注哪个维度，不知道你的老板最在意什么，他出来的东西可能跟你想要的差了十万八千里。但如果你给他看一份上上个月的报告作为格式参考，告诉他数据在哪个系统里可以找到、老板最关注的是哪个地区的增长趋势，他就能做出更接近你期望的东西。Claude Cowork 也是完全一样的道理：原材料越丰富，上下文越充分，输出质量越高。这是很多人刚开始用 Claude Cowork 的时候容易忽略的一个关键点。

第三步，告诉 Claude 你想要的结果。描述你最终要拿到的是什么东西，以及必要的背景信息。

第四步，从一个你自己熟悉的真实任务开始。为什么强调要从你熟悉的任务开始？因为如果你做的是你自己每周都在做的事情，你立刻就能看出来 AI 在哪里做得好、哪里需要你补充信息，而且你已经知道"好"长什么样，可以快速判断和迭代。如果你用一个你自己都不太熟悉的任务来测试，你不知道该怎么评价输出的质量，也看不出来问题出在哪里，就算输出有偏差你也发现不了，更别提改进了。

然后文章里有一个我认为是全文最有价值的实操技巧，我要单独拿出来讲，因为它真的非常好用，而且很多人想不到：让 Claude 在开始之前先主动问你问题。

Austin 说，这是他养成的最有用的习惯之一。他在每个 Claude Cowork 任务的 prompt 里都会加上这样一段话：在我们开始之前，先把我的需求重复给我听，确认我们理解一致，然后尽量多问我问题。

这两句话为什么这么有价值？它做到了一件非常关键的事：把你自己没有想到要说清楚的那些隐含的假设和遗漏的信息，强制暴露出来。比如说，时间范围是哪一段？这里的"好"具体意味着什么？有哪些特殊情况是你知道但 Claude 不可能知道的？你有没有格式上的要求没有说到？对结论的表述方式有没有限制？

Austin 说，事先回答五个问题，可能就花你三十秒。但如果你不问，任务完成之后才发现这些遗漏——你得重新跑一遍任务，时间浪费了，AI 的计算消耗也浪费了，而且返工起来非常麻烦。事前三十秒，省去后面可能的三十分钟，这个投入产出比是非常值的。

我个人对这个技巧特别有共鸣，因为它本质上是一个优秀的项目管理思维在 AI 使用上的迁移：在执行之前先对齐期望，比执行之后返工要高效得多。这不只是 AI 使用的技巧，也是任何委托任务时的最佳实践。你委托一个同事做一件事的时候，好的做法也是先花几分钟确认双方的理解完全一致，而不是直接说"你去做吧"，然后等对方做完了再来纠正偏差。

让 Claude 主动发问，是一个非常聪明的强制对齐机制。它把原本你可能因为觉得"应该不需要解释"或者"它应该能猜到"而省略掉的信息，通过提问的方式逼出来了。而且有时候 Claude 问的问题，你自己在开始之前都没有想清楚——这也是一个帮助你理清需求的过程。

文章还提到一个附加的技巧：Claude 有记忆功能，可以搜索你之前的历史对话。所以如果你刚开始用 Claude Cowork，不知道从哪里开始，不知道哪些任务值得放进来，可以直接问 Claude 自己——"看看我之前的对话，你觉得我经常做哪些任务，其中哪些可以试着用 Claude Cowork 来处理？"这是一个很聪明的自我盘点方式，让 AI 帮你找到最值得自动化的工作，而不是你自己去猜。

当然，Austin 也说了，他并没有完全放弃 chat。他还是大量用 chat 来讨论营销定位的问题、在确定一个方向之前进行推敲和压力测试，或者问各种随机的问题。他的总结非常简洁：chat 适用于你的输出是你脑子里的一个想法的时候；Claude Cowork 适用于你的输出是一个要交给别人的东西的时候。

这个区分我觉得画得非常准。AI 聊天模式不会消失，但它的角色会越来越清晰——它是你的思考伙伴，帮你想清楚方向。而 Claude Cowork 这类 agent 工具会越来越多地承担实际的执行者角色，帮你把方向落地为可交付的成果。两者不是竞争关系，而是分工关系：你脑子里的决策用 chat 来磨，确定了要做什么之后，用 Claude Cowork 来执行。

好，今天这期节目，我们把 Austin Lau 这篇关于 Claude Cowork 最佳实践的文章，从背景到框架到案例到上手步骤，完整地拆解了一遍。

我来做一个最终的总结，帮大家把核心点串起来。第一，搞清楚 chat 和 Claude Cowork 的分工：chat 是拿答案，Cowork 是拿成品，不要混用，混用会降低效率。第二，判断一个任务是不是 Claude Cowork 的活，用那五个要素来过滤——有没有多个输入、要不要文件输出、会不会重复做、你知不知道好坏标准、中间过程是不是无聊，满足越多越值得放进去。第三，给 Claude 足够的上下文和原材料，不要只优化 prompt 措辞，而是把相关的文件、数据源、模板、标准都提供给它。第四，让 Claude 在开始之前先问你问题，强制对齐期望，避免做完之后才发现遗漏。

我个人认为，这篇文章最有价值的地方，不是它介绍了 Claude Cowork 有哪些功能，而是它提供了一个可以迁移、可以复用的判断框架——关于怎么识别哪些工作值得托管给 AI 来做。这个框架不只适用于 Claude Cowork，它几乎适用于你面对任何 AI 工具时的决策。当你下次面对一个任务，犹豫要不要用 AI 来处理的时候，可以在心里过一遍那五个要素：输入多不多、要不要文件输出、会不会重复、我有没有判断力、中间是不是无聊——想通了，决定就很容易了。

最后，Austin 在文章结尾说：找一个你每周都要做的重复任务，试着用 Claude Cowork 来做，看看结果怎么样。前几次可能会觉得有点不顺手，但很快你就会从"这个怎么用"变成"下一个交给它什么"。我觉得这是最好的入门建议——不是去研究工具的所有功能，而是拿一个真实的活来试，边做边学。
