大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,英文标题是 Token-saving updates on the Anthropic API,翻译过来就是"Anthropic API 上的省 token 更新"。这篇文章发表于二〇二五年三月十三日,也就是去年的春天。今天是二〇二六年六月二十一日,算下来已经过去一年多了,在 AI 这个日新月异的行业里,一年多的时间跨度其实不算短。所以听这期节目的时候,大家要有一个心理预期,这篇文章讲的是 Claude 三点七 Sonnet 时代的更新,当时 Claude 四系列和后面更强的模型还没有出现。但是我要提前说一句,这篇文章里讲的几个机制,比如 prompt caching 也就是提示词缓存,比如工具调用的 token 优化思路,这些底层的设计理念到今天依然适用,甚至可以说是现在所有 Claude API 使用者都应该懂的基础知识。所以这期节目虽然讲的是一篇"老"文章,但内容并不过时,反而更像是给大家补一节内功课。

我们先说说当时的背景。二〇二五年初,大模型这个行业已经从"能不能用"进入到"怎么用得更便宜、更快"的阶段。Claude 三点七 Sonnet 刚刚发布,带来了更强的推理能力,也是 Anthropic 第一次把所谓的"混合推理"模型,也就是可以选择直接回答还是先深度思考再回答的模型推向市场。但是模型能力强了之后,马上就会遇到一个老问题,就是成本和速率限制。如果你做过 AI 应用开发,你肯定遇到过这种情况,你的应用需要把大量的背景知识,比如说一整个代码库,或者一份很长的产品文档,塞到每一次请求里面,这样模型才能给出有质量的回答。但是问题来了,每次都把这些内容重新发一遍,既费 token 又费时间,而且很容易撞到速率限制的天花板。这就是这篇文章要解决的核心问题,Anthropic 想办法让开发者能够在不牺牲质量的前提下,把这些重复性的开销降到最低。

好,那我们具体来看文章里讲的几个更新,我挑了三个最重要的点,带大家一层一层拆开讲。

第一个点,也是我认为整篇文章里最有意思的一个改动,叫做缓存感知的速率限制,英文是 cache-aware rate limits。要理解这个东西,我们得先搞清楚什么是 prompt caching,也就是提示词缓存。简单说,如果你的应用每次请求都要带上一大段固定不变的内容,比如一份产品手册,或者一个代码库的某个固定部分,那么传统做法是每次请求都把这段内容原封不动地发给模型,模型每次都要重新"读"一遍,这个读的过程是要花 token 也要花时间的。Prompt caching 的思路就是,既然这段内容每次都一样,那干嘛不让 Claude 把它先存起来,后面的请求只需要说一声"还是用刚才那段",模型直接从缓存里读取,不用重新处理整段文本。这样做的好处是显著的,文章里提到,对于长提示词,缓存可以把成本降低最多九成,把延迟降低最多八成五。这个数字听起来有点夸张,但其实很好理解,你想象一下,如果你每次跟同事开会,都要先把上次会议的所有记录从头念一遍才能开始讨论新内容,那效率得多低。有了缓存,就相当于同事已经记住了上次讲的内容,你只需要说"接着上次讲",直接进入正题。

但是这次更新更进一步的地方在哪里呢。原来即便你用了缓存,缓存读取所消耗的那部分输入 token,依然会被计入你的输入 token 速率限制,英文叫 Input Tokens Per Minute,简称 ITPM。这个限制是 Anthropic 用来控制每个账号每分钟能往 API 里塞多少输入内容的一个天花板。如果缓存读取也占用这个天花板的额度,那就意味着,即便你通过缓存省了钱,你的吞吐量上限其实没怎么提升,因为速率限制这道关卡依然卡在那里。这次更新的核心变化就是,对于 Claude 三点七 Sonnet,缓存读取的 token 不再计入 ITPM 限制了。这是什么意思呢,打个比方,如果你把 ITPM 想象成一条高速公路的车道数量,原来无论你的车是新车还是老车,占的车道数都一样,现在相当于规则改了,只要你的车是"回头客",也就是缓存里已经有的内容,它可以走一条专用的快速通道,不占用主车道的名额。这样一来,你主车道上能跑的全新内容就更多了,整体吞吐量自然就上去了。需要注意的是,这里说的只是输入端的限制放宽了,输出端的速率限制,也就是 Output Tokens Per Minute 简称 OTPM,这个限制没有变化,所以这个优化主要惠及的是那种输入内容很长、但输出相对没那么长的场景。

文章里特意举了三类应用场景来说明这个更新有多有用。第一类是文档分析平台,这种应用经常需要把一个庞大的知识库一直挂在上下文里,让模型随时能查阅。第二类是编程助手,需要参考一个很大的代码库才能给出准确的建议,这个场景我相信很多做开发工具的听众都深有体会,代码库越大,助手懂得越多,但是每次都重新发一遍代码库显然不现实。第三类是客服系统,需要依赖详细的产品文档才能准确回答用户问题。这三类场景有一个共同特点,就是它们的输入内容是相对固定、体积庞大,但是会被反复使用的。缓存感知的速率限制对这类场景来说简直是量身定做。我自己理解,这其实反映了 Anthropic 在做产品设计的时候,一直在观察真实用户的使用模式,然后针对这些模式去优化底层的资源调度逻辑,而不是简单粗暴地"提高限制"了事。

第二个点,我们来聊聊提示词缓存本身的一个简化,这个改动看起来不起眼,但其实大大降低了开发者的心智负担。在这次更新之前,如果你要用缓存功能,你需要自己去设置所谓的缓存断点,也就是告诉系统"从这里到这里的内容我要缓存",然后在后续请求里,你还得自己记着、自己去指定具体要复用哪一段缓存内容。这个过程听起来简单,但实际操作起来挺繁琐的,尤其是当你的应用里同时维护着好几种不同版本的缓存内容的时候,开发者很容易搞混,或者干脆因为管理成本太高而放弃使用缓存功能,白白损失了省钱省时间的机会。

这次更新之后,逻辑变成了,你只需要设置一个缓存断点,Claude 会自动去找你之前缓存过的、最长的那一段匹配前缀,然后直接复用。这里"前缀"这个词稍微解释一下,在大模型的语境里,你发给模型的内容是有顺序的,前缀就是从开头算起的一段连续内容。系统自动寻找最长匹配前缀,意思就是它会智能地判断,在你之前缓存的所有内容里,哪一段跟你这次的请求重合度最高、能复用的部分最多,然后自动用上,不需要你手动指定。这就好比你去图书馆借书,原来的流程是你需要先去查目录卡片,自己找到书架编号,自己去对应的书架取书。现在变成了,你只需要告诉管理员书名,管理员会自动帮你判断有没有现成的、最接近你需求的版本可以直接借给你,你完全不需要操心具体的存放逻辑。这种自动化带来的好处,文章里说得很直接,一是减轻了开发者的工作量,二是因为系统能更精准地复用缓存,实际上也释放出了更多可以省下来的 token。这个改动同时在 Anthropic 自己的 API 上线,也在谷歌云的 Vertex AI 平台上线了,说明这是一个跨平台统一推进的标准化能力,而不是某个特定渠道的独家功能。

第三个点,也是我觉得对做工具调用、做 agent 应用的开发者来说最实用的一个更新,叫做 token 高效的工具调用,英文是 token-efficient tool use。这里先简单解释一下什么是工具调用。Claude 本身除了能直接对话回答问题,还可以调用外部的工具或者函数来完成任务,比如说从一段杂乱的文本里提取出结构化的数据,或者通过调用一个 API 去自动完成某个简单的操作。这种能力是现在所有 agent 应用,也就是能自主完成多步骤任务的智能体应用,最核心的底层机制之一。问题是,工具调用这个过程本身是需要消耗输出 token 的,因为模型在决定调用哪个工具、传什么参数的时候,需要生成一段结构化的描述,这段描述本身也算输出内容,要计入费用和速率限制。

这次更新,Claude 三点七 Sonnet 支持了一种更省 token 的工具调用方式,文章里给出的数字是,输出 token 消耗最多能降低百分之七十,而早期用户平均观察到的降幅是百分之十四。这两个数字差得有点大,我自己的理解是,降幅七成应该是在某些特定场景下,比如工具调用特别频繁、每次调用的结构化描述占比很高的场景里能达到的上限,而平均百分之十四则是把各种各样真实业务场景混合起来之后的一个综合表现,毕竟不是所有应用的工具调用都密集到能把这种优化的潜力完全发挥出来。但即便是平均百分之十四的节省,如果你的应用本身工具调用量很大,这个数字累积下来也是相当可观的成本节约。

那这个功能怎么用呢,文章里说得很简单,你只需要在发起工具调用请求的时候,加上一个叫做 beta 头部的标记,具体内容是 token-efficient-tools-2025-02-19,这串字符你可以理解成一个开关的钥匙,告诉 API 说"请用省 token 的方式来处理这次工具调用"。如果你是用 SDK 也就是软件开发包来调用的,需要确保你用的是带 beta 也就是测试版功能的 SDK 路径,文章里提到的具体写法是用 anthropic.beta.messages 这个调用方式。这里要提醒一下,既然这是一个 beta 功能,意味着它当时还在测试阶段,接入的时候建议先在小流量上验证效果,确认没有副作用之后再全量切换,这是我自己根据经验给的建议,文章原文并没有特别强调这一点。这个功能当时是在 Anthropic 自家的 API、亚马逊云的 Bedrock,以及谷歌云的 Vertex AI 上同步开放测试的,说明这又是一个 Anthropic 想要在所有主流云平台上统一推进的能力。

除了工具调用本身变得更省 token,这次更新还引入了一个全新的工具,叫做 text editor 工具,中文可以理解为文本编辑器工具。这个工具是专门为那种用户需要跟 Claude 协作编辑文档的场景设计的。我们设想一个场景,你正在用 Claude 帮你改一份很长的研究报告,或者帮你改一段源代码。如果没有这个专门的编辑工具,Claude 想要修改文档里的某一处内容,最直接的办法就是把整篇内容重新生成一遍,哪怕你只是想改一个错别字,或者调整一段话的措辞。这样做有几个明显的坏处,一是浪费输出 token,因为大部分没改动的内容也要重新生成一遍,二是延迟变长,因为生成的内容量变大了,三是更容易出错,因为重新生成整篇内容的过程中,有可能不小心改动了原本不该改的地方,或者引入新的不一致。

有了 text editor 工具之后,Claude 可以只针对文本中的特定部分做有针对性的修改,而不需要把整篇内容重新吐出来。这就好比你在用文字处理软件改文档的时候,你是用"查找替换"或者"定位到某一行直接编辑"的方式去改,而不是把整篇文档删掉重新打一遍。这种方式不仅省 token、降低延迟,文章里还特别提到,这样做的准确性也会更高,因为修改的范围更精确,出错的概率自然就小了。对于开发者来说,接入这个工具也不复杂,只需要在 API 请求里把这个工具带上,然后处理好 Claude 返回的工具调用响应就行了。这个工具同样是在 Anthropic API、亚马逊 Bedrock 和谷歌 Vertex AI 三个平台同步上线的。

讲到这里,我想稍微跳出文章本身,做一个我个人的解读。我个人认为,这三个更新放在一起看,其实反映了 Anthropic 在那个阶段产品策略上的一个明显转向,就是从单纯卷模型能力,逐渐转向卷"模型能力之外的工程化体验"。三点七 Sonnet 这个模型本身已经足够强了,但是如果开发者用起来成本高、速率限制紧、操作繁琐,那么这个模型的商业价值就发挥不出来。所以这次的更新,本质上都是在优化"使用 Claude 这件事"的整体效率,而不是优化模型的智商。这种思路其实是很多成熟科技公司在产品发展到一定阶段都会经历的,先把核心能力做出来,然后把使用门槛和使用成本一点点磨平。

文章里还引用了一个真实案例,来自 Cognition 这家公司。Cognition 是一家应用 AI 实验室,他们最出名的产品叫 Devin,定位是一个能跟工程师协作的 AI 队友,帮助有野心的工程团队完成更多事情。Cognition 的联合创始人兼 CEO Scott Wu 在文章里说了这样一段话,他说提示词缓存让他们能够提供更多关于代码库的上下文,从而获得更高质量的结果,同时降低成本和延迟。他还特别提到,有了缓存感知的输入速率限制之后,他们能够进一步优化提示词缓存的使用方式,提升吞吐量,把现有的速率限制用得更充分。这段话其实印证了我们前面讲的内容,Devin 作为一个需要持续理解大型代码库的编程助手,正好踩在文章里提到的那三类典型应用场景之一,也就是"需要参考大量代码库的编程助手"这一类。Cognition 用真实的产品反馈证明了这次更新确实解决了真实的痛点,而不是 Anthropic 自己想象出来的需求。

好,讲完了这三个核心更新,我们来聊聊对大家实际工作有什么意义,也就是这期节目的实践应用部分。

如果你现在正在用 Claude API 做开发,尤其是用到了三点七 Sonnet 或者更新的模型,我建议你可以从这几个方面入手去检查自己的应用。第一,看看你的应用是不是有大段重复使用的上下文内容,比如固定的系统提示词、固定的知识库片段、固定的代码库内容。如果有,那你应该确认自己已经启用了 prompt caching,如果还没用,这是性价比最高的优化,几乎是零成本就能拿到的收益。第二,如果你已经在用缓存了,那么你可以重新评估一下你的速率限制策略,因为缓存读取不再占用输入速率限制这件事,意味着你原来为了规避限速而做的一些妥协,比如人为缩短上下文、人为拆分请求,现在可能已经没有必要了,你可以把这部分预算重新用来塞更多有效的全新内容进去,从而提升单次请求的质量。第三,如果你的应用大量依赖工具调用,也就是那种需要 Claude 频繁去调用外部函数、查数据库、调 API 的 agent 类应用,我建议你去尝试一下加上那个 beta 头部标记,看看实际能省多少 token,这个改动的接入成本非常低,基本上就是加一行配置,风险也可控,值得花十分钟去试一下。第四,如果你的产品形态是那种用户和 Claude 一起协作改文档、改代码的场景,比如一个写作助手或者一个代码审查工具,那么 text editor 工具非常值得你认真评估一下,因为它解决的是一个很具体的体验问题,就是用户不希望每次小修改都看到整篇内容被重写一遍,局部编辑的体验会更接近用户对一个真正的协作工具的预期。

这里我也想提一个要避的坑。因为这些功能里有不少当时是以 beta 测试版的形式发布的,如果你现在已经在生产环境里跑了一年多的应用,我建议你重新去查一下最新的官方文档,确认这些功能是不是已经从 beta 状态转为正式稳定状态,因为接口的具体调用方式、参数名称,在从测试版转正的过程中是有可能发生变化的。这一点文章本身没有讲,完全是我根据软件行业的常见规律给大家的提醒,毕竟这篇文章已经是一年多前的内容,具体的实现细节,大家在动手之前最好以当下的官方文档为准。

另外我还想提一句,文章最后给出的三个行动建议其实很朴实,就是去用提示词缓存配合三点七 Sonnet,去加上那个 beta 头部用上省 token 的工具调用,以及去试试 text editor 工具。这三件事的共同特点是,它们都不需要你对应用架构做大改动,都是在现有调用方式上做一些参数层面的调整就能拿到收益。这其实也是技术更新里我比较喜欢的一种类型,投入产出比很高,不需要伤筋动骨。

好,我们来给今天这期节目做个总结。这篇文章核心讲的是三件事,一是缓存读取的 token 不再占用输入速率限制,帮你在现有限速天花板下跑出更高的吞吐量,二是提示词缓存的使用方式被简化,系统会自动帮你匹配最长的缓存前缀,省去手动管理的麻烦,三是工具调用变得更省 token,同时还多了一个专门用于局部编辑文档的新工具。这三件事放在一起,本质上都是 Anthropic 在帮开发者把"用 Claude 的成本"降下来,把"用 Claude 的体验"做得更顺手。我个人的点评是,虽然这篇文章已经是一年多前的旧闻,模型也早就从三点七 Sonnet 迭代到了更新的版本,但是文章里体现出的工程思路,比如缓存机制要跟速率限制联动设计,比如工具调用要考虑 token 经济性,这些思路放到今天依然是评估任何大模型 API 好不好用的重要标准。如果你是一个正在搭建 AI 应用的工程师,我觉得花时间去理解这些底层机制,比单纯追逐最新的模型版本号,可能更有长期价值。
