为什么我专门拿这篇文章来做一期节目？因为它描述的，不只是一次普通的 SDK 接入。Anthropic 发布了一个新的 Swift 包，让苹果开发者可以在苹果自己的 Foundation Models 框架里，无缝地调用 Claude 这个云端大模型来处理复杂任务。这件事背后有一个很重要的架构意义：在同一个用户流程里，让设备端的小模型和云端的大模型各司其职，对用户来说是一次流畅的体验，对开发者来说是一套清晰的分层架构。这个思路，我认为会深刻影响未来几年苹果平台 AI 应用的设计方式，值得每一个在做苹果生态产品的工程师和产品经理认真了解。

我们先来建立背景，因为不了解行业的来龙去脉，就很难体会这次发布的真实意义。

在过去两三年里，如果你在做移动端 AI 功能，你一定感受过一种两难的拉扯。设备端 AI 模型有很多令人心动的优点：响应速度极快，不需要走网络，用户数据不会离开设备，隐私有保障，就算在地铁里信号为零也能正常工作。但它的能力上限很低，因为手机和电脑上能跑的模型，参数量有限，计算资源有限，它能胜任的任务，基本上集中在相对简单的文本处理上：提取关键词、写简短摘要、做分类、做格式转换。你要让它做跨多个步骤的逻辑推理、生成复杂的代码、实时查询互联网上的最新信息，它就力不从心了。

云端大模型恰好是另一个极端。Claude 这类模型，参数量可以比设备端模型大几十倍乃至上百倍，推理能力、长上下文理解、代码生成、复杂规划，都强得多。还可以配备联网搜索、代码执行等工具，几乎没有能力上的天花板。但它的代价是：每次调用需要走网络，有延迟，有 API 费用，而且用户的输入内容要被发送到云端服务器。这在处理日记、健康数据、财务记录等敏感内容的时候，会让很多用户有顾虑，也会给产品的隐私合规带来相当大的压力。

所以，做苹果平台 AI 应用的开发者，这几年一直在这两条路之间痛苦地权衡。选全端侧，能力受限，用户体验天花板低；选全云端，延迟高、成本高、隐私压力大，而且完全依赖网络，断网场景下用不了。很多团队反复摇摆，找不到一个令人满意的平衡点。

苹果自己也在思考这个问题。他们推出了 Foundation Models 框架——这是一套让 Swift 开发者可以在代码里直接访问苹果自家设备端模型的官方接口。苹果的风格一贯是把开发者体验做到极度简洁，Foundation Models 框架也不例外。原文里提到，最少三行代码就能调用端侧模型，并且得到的不是原始字符串，而是有类型的 Swift 值。也就是说，你定义了什么形状的数据结构，模型就返回什么形状的数据，可以直接被你的 Swift 代码使用，不需要额外的解析步骤。

这里有一个技术机制非常关键，值得单独解释，叫做「guided generation」，引导式生成，配合一个叫做 @Generable 的 Swift 宏标注来实现。这对不太熟悉苹果开发的朋友来说可能有点陌生，我用一个具体的例子来把它讲清楚。

在传统的大模型调用里，你发一段提示词给模型，模型返回一段自由文本，然后你写代码去解析这段文本，提取你需要的信息。这个过程说实话很脆弱，因为模型的输出本质上是不确定的。它可能按你期望的格式回答，也可能在开头加一段解释性的客套话，也可能用略微不同的措辞，你的解析代码随时可能出错，维护成本很高，线上环境里出问题的概率也不低。更烦人的是，每次模型版本更新，输出格式可能又变了，你的解析代码又要跟着改。

苹果的 @Generable 机制从根本上解决了这个问题。你先用 Swift 定义好你需要的数据结构，比如一个叫做 MoodSummary 的结构体，里面有今日情绪得分、主要情绪关键词列表、影响情绪的关键事件这三个字段。你在这个结构体上加一个 @Generable 标注，Foundation Models 框架就会根据这个结构体的定义，引导设备端模型生成符合这个形状的输出，然后直接给你一个 MoodSummary 对象，可以立刻在 Swift 代码里使用，不需要任何额外的字符串处理和格式校验。这把过去需要靠精心调教提示词才能实现的格式控制，变成了 Swift 类型系统自然保证的事情。代码更简洁，更可靠，出问题的可能性更低，就算后来换了一个版本的模型，类型约束也还在那里。

好，现在苹果有了这个框架，Anthropic 在这之上做了什么？他们发布了一个新的 Swift 包，让开发者可以在 Foundation Models 框架的调用链里，把任务交棒给 Claude。这个 Swift 包封装了和 Claude API 通信的全部逻辑，包括流式传输、工具调用、以及把结构化响应返回到 SwiftUI 视图。开发者不需要自己写网络请求代码，不需要自己处理流式传输的缓冲逻辑，只需要描述想让 Claude 做什么，然后把结果接住就好。

我来具体描述一次完整的调用流程，让你感受一下这套架构有多自然。假设你在做一个日记应用，用户写了三个月的日记，某天点了一个"帮我梳理最近三个月的情绪变化趋势"的按钮。

这个请求到来之后，首先，Foundation Models 框架在设备端启动。它利用你用 @Generable 定义好的数据结构，从用户过去三个月的日记文本里提取结构化信息：每篇日记的写作时间、情绪关键词、用户提到的重要事件。这一步在设备端完成，速度很快，用户的原始日记内容全程不离开手机，端侧模型处理的速度可能也就零点几秒。

然后，这些已经整理好的结构化数据，以 Swift 类型的形式，被传入 Anthropic 的 Swift 包，作为发给 Claude 的输入。这里有一个细节很关键：Claude 收到的不是用户三个月原始日记的全文——那些文本可能语气散漫、夹着很多情绪化的措辞、信息混杂——而是经过端侧模型整理过的、类型化的结构体，信息已经提取好了，格式也干净了。这让 Claude 的工作更精准，也在一定程度上减少了上传到云端的数据量，从隐私角度来说也更合理一些。

接下来 Claude 开始做它擅长的事：跨多个时间点进行综合推理，识别情绪变化的趋势和规律，找出某些重复出现的情绪模式，把这段时间用户提到的外部事件和情绪波动联系起来，生成一段有深度的分析报告。这是一个需要处理大量上下文、进行多步骤综合推理的任务，设备端模型做不了，但 Claude 擅长。

Claude 的分析结果以流的方式一段一段地传回来，SwiftUI 视图实时更新，用户看到的是文字逐渐出现在屏幕上，就像模型在实时思考和打字，等待感大大降低。

这整个流程，从用户视角看，就是点了一个按钮，看到了分析结果。他完全感受不到背后有两个模型在协作，感受不到数据在端侧和云端之间的流转。这正是原文那句话的含义：对用户来说是一次体验，但背后是为每个步骤选了最合适的模型。

这个设计思路让我想到移动互联网时代"本地缓存 + 云同步"这个架构范式的出现。那时候，做移动应用的团队都在思考一个核心问题：什么数据存本地、什么数据存云端、什么时候同步、冲突怎么处理。把这个问题想清楚的团队，做出的应用响应极快，就算断网也能用，数据还能跨设备同步，用户体验远好于竞争对手。当时的最佳实践是"本地数据优先、后台异步同步"，后来这成了移动应用数据层的标准范式，被写进了无数开发者的肌肉记忆里，现在几乎每个移动应用都在用这套思路，大家已经不需要再特意讨论它了。

今天，AI 应用开发者面临一个类似的基础架构问题：哪些推理放端侧、哪些放云端、两者之间怎么交接。Foundation Models 框架加上这个 Swift 包，给出了一套开箱即用的答案：苹果的框架负责端侧部分，Swift 包负责云端部分，@Generable 的类型化数据作为两者之间的接口标准。开发者不需要自己设计交接协议，不需要自己定义数据格式，可以专注在业务逻辑上，把精力放在"这个功能对用户有什么价值"上，而不是"怎么让两个模型说上话"上。

我个人认为，"分层 AI 架构"这个思维，会在未来几年成为移动端 AI 应用开发的标准范式，就像"本地缓存 + 云同步"成为了移动数据层的标准范式一样。现在大家还在讨论怎么做，过几年等最佳实践成型之后，就会变成所有开发者默认就会想到的事情。先把这套架构想清楚、用熟悉的团队，会在产品体验和运营成本两个维度上都领先对手，而且这个领先不是短暂的功能差距，而是架构思维上的积累差距，拉开了之后很难被快速追上。

我们再来细聊这次集成具体解锁了哪些能力。原文提到了几个方向，我逐一展开，把每个方向的实际意义讲清楚。

第一个是联网搜索。Claude 可以实时访问网络获取当前信息。这是端侧模型完全做不到的事情，因为设备端模型是离线运行的，它的知识有截止日期，它对上周发生的事情一无所知。比如一个金融资讯类应用，用户问某家公司最近一个季度的业绩亮点，设备端模型无法回答，因为这是最新的动态信息，不在任何模型的训练数据里。但 Claude 可以联网去搜，整合多个来源的信息，用清晰的语言总结给用户。再比如一个学术辅助应用，用户想了解某个研究领域最近的新进展，端侧模型的知识截止于某个日期，联网的 Claude 才能给出真正最新的答案。这个联网能力，在新闻聚合、投资研究、学术辅助、市场分析这类对实时信息有强烈需求的应用场景里，是端侧和云端能力差距最直观的体现。

第二个是代码执行。Claude 不只能生成代码，还能运行代码，拿到实际的计算结果，再把结果用自然语言解释给用户。这对数据分析类应用来说极为实用。想象一个个人财务管理应用，用户说"帮我分析一下我这个季度的支出，哪个类别最超预算？"这个任务需要几个步骤：理解问题的语义，生成相应的数据查询和统计逻辑，在用户的账单数据上运行这个逻辑，然后把运行结果转换成用户看得懂的分析报告。Claude 配上代码执行能力，可以把整个流程一次性跑通，从用户的问题到最终的分析结论，中间不需要用户参与任何额外的操作，也不需要开发者自己写复杂的数据处理逻辑。

第三个是工具调用，英文叫 tool calls。这是指大模型在回答过程中，可以主动决定调用某个外部工具——比如搜索引擎、数据库、日历、天气服务——来获取额外信息，然后把工具返回的结果整合进最终回答里。工具调用把模型从"只会说"变成了"既会说又会做"。一个支持工具调用的模型，遇到超出自身知识范围的问题，不会胡说，而是去调工具查。这个 Swift 包封装了工具调用的完整循环逻辑，开发者只需要定义好工具的名称和功能描述，不需要自己写"模型发起调用→执行工具→把结果还给模型"这个循环的处理代码，这块过去是一个相对繁琐的工程细节，现在包帮你处理好了。

第四个是流式响应，英文叫 streaming。Claude 的回答会一段一段地实时传回，SwiftUI 视图实时更新，用户看到文字逐渐出现在屏幕上。这对用户体验的影响非常大，可以说是影响用户对产品感知的关键细节之一。一个需要十五秒才能完成的复杂推理任务，如果是流式返回，用户感受是"模型在实时回答我"，体验像在和一个思维速度很快的人聊天；如果是等十五秒一次性返回，用户感受是"应用卡死了，我要不要重试"。这不是技术噱头，而是真实影响用户留存的体验细节。好的移动应用在任何异步操作上都要给用户及时的视觉反馈，流式响应在这里是最自然的解决方案，而且苹果的 SwiftUI 对这种增量更新本来就有很好的支持。

第五个是结构化响应。对应端侧的 @Generable，Claude 这边也支持返回结构化的数据，而不只是自由文本。你定义好数据结构，Claude 直接返回符合这个结构的内容，可以立刻被 SwiftUI 的数据绑定机制使用，不需要额外的解析步骤。整个从端侧到云端再回到视图层的数据流，都是类型安全的，没有脆弱的字符串拼接和手动解析环节。这对代码的长期可维护性来说是很大的好处，也让端侧和云端两个模型之间的数据交换变成了一个标准化的过程，而不是每个开发者自己摸索的黑盒。

这五个能力放在一起，意味着什么？意味着你现在可以用纯 Swift，在苹果官方认可的框架里，写出一个完整的 AI agent 闭环：用户输入进来，端侧预处理，Claude 做推理（可以调工具、可以联网、可以执行代码），结果流式返回到视图层。整条链路在一个统一的框架里，没有需要自己从头搭建的胶水代码，没有七拼八凑的第三方库，开发者可以把精力集中在业务逻辑上。

说完了能力，我们来看原文里举的两个具体应用场景。第一个是日记应用。端侧模型每天在设备上根据用户的写作习惯生成个性化的日记提示，帮用户开始写作，这是一个简单的、隐私敏感的、响应速度要求高的任务，完全适合在端侧完成。当用户说"帮我找一找我过去几个月日记里反复出现的主题或者情绪模式"，这个任务涉及跨时间的综合分析，需要处理大量的上下文，超出了端侧模型的能力边界，就交给 Claude。一次应用内的用户请求，由两个模型接力完成，用户不感知任何切换，体验是流畅连贯的。

第二个是学习辅助应用。端侧模型在设备上给一个概念做简单的定义或者解释，这个任务快速、标准化，完全可以离线完成，对于学生来说能立刻拿到答案。当学生追问"这个知识点对我们整门课的知识框架有什么深层意义，它和其他几个概念是怎么关联的"，这是一个需要跨多个知识点进行关联推理、理解知识结构的问题，端侧搞不定，就把这个问题连同当前的对话上下文一起传给 Claude，由 Claude 来做更深入、更有洞察力的阐释。

这两个例子都体现了同一个设计原则：端侧模型做快速、标准化、数据不出设备的任务；Claude 做深度推理、需要更多上下文、或者需要外部信息的任务。这个分工不是非此即彼的，而是在同一个用户对话流程里动态流动的，根据用户在这一刻的需求来决定谁上。

好，现在来聊最实际的部分：作为开发者，你怎么上手，有哪些坑要提前知道。

首先说上手路径。这个 Swift 包从二〇二六年六月九日开始可以使用，支持 iOS 二十七、iPadOS 二十七、macOS 二十七、visionOS 二十七以及 watchOS 二十七，也就是苹果今年的全系新版操作系统。watchOS 也支持这一点值得特别注意，苹果手表上的应用如果能借助这个框架调用 Claude，意味着可穿戴设备上的 AI 辅助场景也打开了，比如运动数据的深度分析、健康趋势的长期跟踪，都有了新的可能性。

把包加到你的 Xcode 项目里，用你的 Anthropic API key 做认证，然后利用 @Generable 的类型化输出作为给 Claude 的输入，把 Claude 的响应流回你的 SwiftUI 视图。基础集成按原文描述，流程是相当直接的，门槛不高。

要想用好这套框架，有一个核心的设计判断要做清楚：什么任务放端侧，什么任务给 Claude。我给你一个粗略的判断框架。如果一个任务可以在零点几秒内完成、不需要实时网络信息、数据量在设备端模型的上下文窗口范围内、而且数据敏感性要求不出设备，就用端侧。如果一个任务需要联网、需要跨大量数据进行综合推理、需要生成代码或调用工具、或者上下文很长、或者答案的质量要求很高，就给 Claude。这个判断不是一次性做完就固定了的，随着产品迭代，你对哪类任务该分到哪一层会越来越有感觉，初期可以先保守一点，用端侧能搞定的坚决不调 Claude。

有一个很实用的技巧：在触发 Claude 之前，先用端侧模型做一次意图识别，判断用户的这一次请求到底需不需要云端能力。需要才调，不需要就直接用端侧回答。这一步额外的分类，可以大幅减少不必要的 API 调用次数，对成本控制的效果非常显著。

说到成本控制，这是开发者最容易低估、但代价可能最高的问题之一。Claude API 是按处理的 token 计费的，token 可以理解为模型处理文本的基本单元，大概来说，一千个中文汉字大约消耗一千五到两千个 token 左右。如果你的应用面向大量普通用户，每次用户交互都触发一次 Claude 调用，月底的账单可能会让你相当震惊。你需要从产品设计阶段就把调用触发逻辑想清楚，确保只有真正需要云端能力的任务才调 Claude，其他的都走端侧。另外，传给 Claude 的上下文要精炼，不要把大量无关内容一起发过去，这既浪费 token，又可能干扰模型的判断。这个成本意识要从一开始就建立，不要等上线之后发现费用爆了再来做优化，那时候改起来代价更大。

隐私边界是第二个必须在产品设计阶段想清楚的问题，而且不只是技术问题，也是产品价值观的问题。苹果端侧模型的核心卖点之一是数据不出设备，这对用户来说是一个很重要的信任承诺。但一旦你把数据发给 Claude，这部分数据就上云了，进入了 Anthropic 的服务器，和苹果的承诺是不同的语境。你的产品需要在设计上和用户协议上明确告知用户：什么情况下数据会被发送到 Anthropic 的服务器，发送的是哪些数据，数据如何被处理和保存。特别是日记应用、健康类应用、财务管理应用这类涉及敏感个人信息的场景，绝对不能让数据在用户不知情的情况下上传。一个好的设计模式是，把"调用云端分析"做成用户主动触发的明确操作，在触发前展示清晰的提示，让用户知道这次操作会把哪些信息发到云端，并且提供拒绝的选项。这不只是法律合规的要求，也是建立用户长期信任的基础。

API key 的安全管理是第三个容易被忽视、但代价可以非常惨重的问题。你的 Anthropic API key 是你账户的凭证，谁拿到它，谁就能以你的名义产生费用，而且这个费用你是逃不掉的。这个 key 绝对不能硬编码在客户端代码里。iOS 的应用包可以被反编译，硬编码的字符串很容易被提取，网上有大量工具可以做这件事，会的人不在少数。正确的做法有两种选择：如果你在做面向开发者的工具类应用，可以让用户输入自己的 API key，然后把它存在 iOS 系统的 Keychain 里，Keychain 有系统级别的安全保护，比存在 UserDefaults 里安全得多；如果你在做面向普通消费者的应用，正确的做法是在你自己的服务端管理 key，客户端向你的服务端发请求，由你的服务端代为调用 Claude，这样 key 对任何客户端用户都是不可见的。这个安全架构要在开发初期就设计好，后期改起来不只是代码改动，还要涉及用户迁移，代价很大。

断网降级是第四个要提前考虑的工程问题。Claude 需要网络，但用户的使用场景不总是有好网络——地铁、飞机、偏远地区、网络信号差的会议室——这些都是真实存在的使用情境。你需要设计清楚：当网络不可用时，应用能否退化到只使用端侧模型的模式，仍然给用户提供基础功能？如果你的应用某些核心功能完全依赖 Claude，断网就整个无法使用，用户体验会非常糟糕，差评和卸载很可能随之而来。这个降级逻辑要在开发阶段专门测试，模拟断网场景走一遍完整的用户流程，确保降级模式下 app 仍然能给用户提供价值。不要等用户投诉了才发现断网体验一塌糊涂，那时候的修复成本要高得多。

延迟体验管理是第五个需要在 UI 设计层面认真对待的问题。Claude 是云端调用，哪怕支持流式传输，第一个字出现之前也有一段等待时间，可能是一到三秒，在网络条件差的情况下可能更长。你的 UI 在这段等待期间不能什么都不显示，否则用户会认为应用卡死了，可能直接退出 app。一个推荐的做法是两阶段更新：先展示端侧模型的初步结果——一个简短的摘要，或者一个占位性的初步回答——让用户知道应用在正常工作；然后在 Claude 完成深度分析之后，把界面更新成 Claude 的完整结果。这种两阶段更新的模式，可以同时给用户"快"和"深度"两种感受，是在 Claude 的网络延迟和用户的等待体验之间取得平衡的一个很好的设计策略。

说完了实践细节，我们来看这件事更大的行业意义。

苹果在 Foundation Models 框架里留出了正式的接口，允许 Claude 以苹果认可的方式接入，这背后有一个很重要的信号。苹果一贯喜欢掌控整个技术栈，从芯片到操作系统到开发框架，这种垂直整合是苹果生态的核心竞争力，也是苹果一贯的战略偏好。这次他们主动为外部云端模型开了一扇门，而且是以官方框架集成的方式，说明他们在苹果生态的 AI 布局上做出了一个选择：对于设备端模型无法满足的高阶能力，选择开放合作，而不是封闭等待自己追上来。这个选择本身就值得关注。

对 Anthropic 来说，成为第一个以这种深度方式集成进苹果 Foundation Models 框架的外部 AI 提供商，是非常有分量的生态位占领。苹果开发者生态庞大，文档、教程、社区经验、开源示例，都会先在"端侧 + Claude"这个方向积累。开发者学会了一套模式，用顺手了，迁移成本就高了，这种先发优势在开发者生态里通常是黏性很强的。

从 visionOS 的角度来说也值得单独提一句。visionOS 是苹果空间计算平台 Vision Pro 所在的系统，原文提到 visionOS 二十七同样支持这次集成。空间计算是一个对 AI 能力要求密度更高的场景，在三维空间里给用户提供上下文感知的智能辅助，比手机上的二维应用对推理能力的要求高得多。Claude 在 visionOS 上的就位，为这个平台的开发者打开了一个很有想象力的可能性空间。

好，我们来做个总结。

Anthropic 在二〇二六年六月八日发布了一个新的 Swift 包，让苹果开发者可以在苹果官方的 Foundation Models 框架里调用 Claude。核心价值是定义了一种混合 AI 架构：设备端的苹果模型负责快速、隐私、简单的任务，Claude 负责复杂推理、联网搜索、代码执行这类超出端侧能力边界的任务。两个模型之间通过 @Generable 类型化数据来交接，这个包整合了流式传输、工具调用、结构化响应等工程能力。支持 iOS 二十七以上的全系苹果平台，包括 watchOS，开发者从六月九日起就可以开始试用。

我个人的看法是：这是苹果平台 AI 应用架构的一个重要里程碑，它让"端侧 + 云端协作"的混合 AI 架构，从一个需要团队自己摸索的复杂工程问题，变成了有官方工具支撑、有清晰范式可以遵循的成熟选项。对于正在做苹果平台应用的工程师和产品经理来说，现在是个很好的时机，认真梳理一下你的应用里有哪些 AI 功能适合这种混合架构、哪些放端侧、哪些交给 Claude。把这个判断做清楚，然后用这套工具把它们接起来，用户会得到一个既快又聪明的体验，你会得到一个成本和体验都可控的工程方案。
