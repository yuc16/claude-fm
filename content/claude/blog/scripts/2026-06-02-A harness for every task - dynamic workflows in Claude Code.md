好，我们开始。

今天要聊的这篇文章，英文标题是 A harness for every task: dynamic workflows in Claude Code，我把它翻译成中文是"为每个任务量身定制执行框架：Claude Code 中的动态工作流"。这篇文章发表于二〇二六年六月二日，距离今天只有十七天，可以说是非常新鲜的内容。作者是 Anthropic 的两位技术员工——Thariq Shihipar 和 Sid Bidasaria，他们专门做 Claude Code 方向的研究和开发。

在我们进入正题之前，我想先告诉你为什么这篇文章值得用二十五分钟来认真聊。如果你平时用 Claude Code 或者类似的 AI 编程助手处理工作，你很可能有过这样的体验：对于简单的、单一的任务，AI 表现得非常好；但一旦任务复杂起来、步骤多起来、需要长时间运行，效果就开始下滑，甚至让你怀疑是不是自己的 prompt 写得不够好。这篇文章要解决的，恰恰就是这个问题。它不是在教你怎么写更好的提示词，而是在告诉你，有一种全新的方式让 AI 来组织自己的工作——让 Claude 给自己写执行方案，自己设计流程，自己调度资源。这个转变比表面上看起来要深刻得多。

我们先来讲背景，聊一聊单一上下文窗口到底有什么局限。

当你给 Claude Code 一个任务，它在一个单一的上下文窗口里同时做两件事：规划，以及执行。对于大多数日常的编程任务，这个方式非常有效。你让它重构一个函数，它看一眼代码，给你改好，整个过程在一个来回里完成，上下文窗口够用，逻辑也清晰。

但是，当任务变成这样的规模：你要做一次完整的代码安全审查，涉及五十个检查点；或者你想把项目里所有使用了废弃 API 的地方全部替换掉，涉及数百个文件；又或者你想让 AI 持续搜集、对比、综合大量信息，产出一份研究报告——这些任务可能需要几十分钟甚至几个小时，需要维持大量的中间状态，还需要对结果进行独立的验证。这时候，单一上下文窗口的架构就开始出现问题。

文章里用了三个术语来描述这三类失败模式。我觉得这三个概念非常精准，值得仔细展开，因为理解了它们，才能真正理解动态工作流为什么有价值。

第一个叫做 agentic laziness，我把它翻译成"代理懈怠"。指的是 Claude 在处理一个特别复杂、步骤特别多的任务时，做到一半就停下来，宣布"任务完成"了。文章里举了一个例子：你让它做一个安全审查，里面有五十个检查项，它做了三十五个之后就开始给你总结交差。

表面上看，这像是 AI 在偷懒，但背后有更深的原因。你可以把上下文窗口想象成工作记忆——就像人在做特别复杂的任务时，随着工作记忆被各种中间信息占满，维持"我还没有完成"这个全局认知的能力会下降。AI 有类似的现象：上下文越长，它维持全局任务状态的能力越弱，就越容易在某个局部完成的节点上，误以为整体已经做完了。

第二个叫做 self-preferential bias，翻译成"自我偏好偏差"。意思是，如果你让 Claude 验证或评价自己刚刚生成的内容，它有强烈的倾向去肯定自己的结果。

这背后是一个认知心理学的基本规律。你有没有注意到，当你花了很多时间写完一篇文章，自己来检查错误，往往会漏掉很多明显的问题？这是因为你的大脑会自动补全、自动修正，按照你的意图来"读"文字，而不是按照文字实际写的内容来读。AI 也有类似现象——当它刚刚在当前上下文里生成了某段代码，再让它来验证这段代码，它很难用真正独立的眼光来审视。就像让一个作者给自己的稿子做最终审校，结果必然不如请一个独立的审校员来看。

第三个叫做 goal drift，翻译成"目标漂移"。这是在超长对话里最容易出现、也最难察觉的一类失败。

Claude Code 有一个叫做"压缩"的机制。当上下文窗口快满了，它会把早期的对话内容压缩成摘要，腾出空间给新的内容。问题是，每一次压缩都是有损的——就像把高清照片压缩成低分辨率的小图，细节必然丢失。你一开始说的"注意，这个 API 接口的签名不能改变"，或者"边缘情况要特别处理，不能改动这里的行为"，经过几次压缩之后，这些约束就可能悄悄从上下文里消失了。等你发现的时候，AI 已经在错误的方向上走了很远。

这三个问题合在一起，就解释了为什么对于"长时间运行、大规模并行、高度结构化、需要对抗性验证"的任务，单一上下文的 Claude Code 会遇到天花板。而动态工作流，正是 Anthropic 为这类任务设计的解法。

好，现在我们来讲动态工作流的本质是什么。

动态工作流的核心思路可以用一句话来描述：让 Claude 为当前的具体任务，即时生成一个 JavaScript 脚本，这个脚本定义了整个任务怎么被拆分、子任务分别交给哪些子代理来执行、以及结果怎么汇总。

我想用一个比喻来帮你理解这个转变的意义。

过去我们用 Claude Code，就像是雇了一个全栈工程师，所有的事情他一个人来做。这个模式在任务规模小的时候效率极高，但当项目规模上升，单打独斗的工程师就会开始出现各种问题——精力分散、顾此失彼、细节遗漏。

现在有了动态工作流，这个全栈工程师先转变角色，变成项目经理。他先分析项目的需求，制定拆解方案，然后招募一批各有专长的分包商，每个人负责一个清晰的、范围明确的子任务，最后统筹汇总所有结果。

关键的区别在于：这个项目经理不是你来指定的，而是 Claude 自己判断这件事需要拆解，自己设计拆解方案，自己分配任务，自己统筹执行。你只需要描述目标，Claude 来决定如何达到目标。

实现上，每个子代理都有自己独立的上下文窗口，这就从根本上消灭了目标漂移的问题——子代理的上下文从任务开始就是干净的，不会受到漫长历史对话的污染，也不会积累越来越重的包袱。因为不同子代理相互独立，你还可以让一个子代理的工作成为另一个子代理的验证对象，从而打破自我偏好偏差的循环。

还有一个技术细节值得注意：Claude 可以为每个子代理选择不同的模型，也可以决定子代理是否运行在独立隔离的工作区里。这意味着 Claude 在执行工作流时，会动态地做资源分配——需要深度推理的任务用 Opus 四点八，简单的文本分类可以用 Haiku 四点五，需要修改代码的任务放在独立工作区里跑，防止不同子代理的改动互相干扰。如果工作流在运行中被中断，比如你关掉了终端，下次恢复会话时可以从中断的地方继续，不需要从头开始，这对长时间运行的任务来说非常重要。

接下来是文章最有价值的部分：五个核心编排模式。这五个模式不只对 Claude Code 有用，对任何想设计多代理系统的人来说都是很好的参考框架。

第一个模式叫分类路由，英文是 Classifier 加 Router。思路很直接：先用一个分类代理来判断任务属于哪种类型，然后根据类型把任务路由给不同的专门代理处理。文章里举了一个让我印象很深的例子：你问 Claude "解释一下 auth 模块是怎么工作的"，但这个问题的实际难度完全取决于 auth 模块有多少个文件、代码库的整体结构是什么样的。如果 auth 模块只有两个文件，Sonnet 就够了；如果涉及几十个文件和复杂的依赖关系，就需要 Opus 来做深度推理。所以先派一个分类代理去探索代码库，评估复杂度，然后再决定路由到哪个模型和哪种处理方式。这背后有一个很实用的工程哲学：把计算资源花在真正需要的地方，而不是对所有任务一视同仁地分配同等规格的模型。

第二个模式叫扇出合并，英文是 Fan-out 加 Synthesize。把一个大任务拆成很多个并行的小任务，每个小任务派一个子代理去执行，最后再用一个合并代理把所有结果整合成最终输出。合并这一步是一个"同步点"——它等待所有扇出的子代理都完成了，再把它们的结构化输出合并在一起。这个模式特别适合两种情况：一是子任务数量很多，需要并行处理来节省时间；二是每个子任务都能从干净的上下文窗口中受益，不希望不同子任务之间互相污染。比如你要对代码库里的一百个模块分别做安全分析，如果顺序做，前面的分析结果会占用后续模块的上下文空间，做到后面 AI 可能已经开始混淆不同模块的情况。用扇出模式，每个模块都有自己独立的代理，相互之间完全隔离，汇总出来的分析结果也不会有交叉污染。

第三个模式叫对抗验证，英文是 Adversarial Verification。这个模式的设计非常优雅：每个做任务的子代理，都配一个专门的验证代理来质疑它的输出。做代码的代理和审查代码的代理，是两个完全独立的实体，审查代理从来没有参与过代码的创作过程，它的上下文窗口里只有被验证的内容和验证标准，没有任何关于"这段代码是怎么被写出来的"的历史信息。这就从系统架构层面保证了验证的独立性，彻底打破自我偏好偏差的循环。你可以想象成：做实验的科学家和负责独立复现实验的科学家如果是同一个人，可信度会大打折扣；但如果是两个独立的团队，结论的可信度就会显著提高。

第四个模式叫锦标赛，英文是 Tournament。这个模式不是把任务拆分给不同代理分别做不同的部分，而是让多个代理同时竞争完成同一个任务，然后用裁判代理做两两比较，通过淘汰赛的方式选出最好的结果。文章里特别指出一个有意思的细节：两两比较的判断比直接给每个方案打分要可靠得多。这实际上是认知科学里的一个经典发现——无论是人类还是 AI，在做"A 和 B 相比哪个更好"这样的相对判断时，比在做"给 A 打一个一到十分的分数"这样的绝对评分时，结论要更加一致和准确。锦标赛模式特别适合"没有标准答案、依赖品味"的任务，比如给命令行工具起名字、从多个设计方案里选一个、评选最优雅的代码实现方式。

第五个模式叫循环直到完成，英文是 Loop Until Done。对于工作量事先未知的任务，不设定固定的循环次数，而是持续生成子代理去工作，直到某个停止条件被触发——比如"已经没有新发现了"，或者"日志里已经不再出现错误了"。这个模式的价值在于它适应了任务本身的不确定性。如果你让 AI 去找代码库里所有潜在的安全漏洞，你一开始并不知道有多少个，所以不能设定"找五遍"这样的固定循环；你需要它一直找，直到再也找不到新的漏洞为止。这个"直到结果稳定为止"的循环逻辑，是处理开放式探索任务非常自然的框架。

好，现在我们来看具体的应用场景，我会挑七个有代表性的来讲，让你把这五个模式映射到自己的实际工作里。

第一个场景是大规模代码重构。文章里提到，Bun 这个 JavaScript 运行时，把整个底层从 Zig 语言重写成 Rust，就是用动态工作流完成的。你可以想象这个任务的规模：几十万行代码，成千上万个调用点需要修改，不同模块之间有复杂的依赖关系，不可能靠一个 AI 实例在单一上下文里搞定。工作流的思路是：先把整个重构任务分解成一系列独立的修改单元，为每个修改单元在独立的工作区里派一个子代理来实施修改，再配一个对抗性的代理来审查每个修改的正确性，通过审查才合并进主干。文章还提到一个工程细节：要告诉各个代理不要使用资源密集型的命令，这样才能让尽量多的代理同时并行运行，而不会把本地机器的资源耗尽。这个细节很重要——并行是动态工作流的核心优势，但并行越多，资源消耗越大，需要在两者之间做好平衡。

第二个场景是深度研究。Anthropic 在 Claude Code 里内置了一个叫 deep research 的技能，也就是 slash deep-research 命令，它本身就是用动态工作流实现的。具体运作方式是：先扇出多个并行的网络搜索，每个搜索代理拉取不同的信息来源，然后用对抗性代理来验证各个来源的内容是否自洽、是否可信，最后用一个合成代理把所有验证过的信息整合成一份带引用的报告。这个模式不只适用于网络搜索。文章举了几个类似的例子：你可以让 Claude 去 Slack 频道里整理过去六个月的故障事件，归纳反复出现的根本原因；你可以让它深入一个你不熟悉的代码库，探索某个功能的完整工作原理，产出一份研究报告。这类任务的共同特点是：信息分散、来源多样、需要综合验证，不能靠单一的上下文来完成。

第三个场景是事实核查。如果你有一篇技术博客或技术报告，想在发布之前确认每一个技术声明都是准确的，可以用这样的工作流：第一个代理通读全文，识别出所有的事实性声明；然后为每一个声明派一个独立的子代理去代码库或文档里核实；还可以再加一层，用一个验证代理来评估每个核实来源的质量——是官方文档还是过时的博客，是主干代码还是某个已废弃的分支。文章里的原话是"我不想发布任何有错的内容"，这类需求在技术写作里非常普遍，但手动逐一核查既费时又容易遗漏，自动化工作流是一个很自然的解法。你可以把这个场景想象成一个编辑部里的查核流程，只不过现在这个流程可以完全自动化运行。

第四个场景是大规模排序筛选。文章里举的例子是你有一千张客户支持票，想按问题严重程度排序，把最严重的优先处理。把一千行数据塞进一个上下文窗口，不只是可能超出长度限制，即使能塞进去，评判质量也会大幅下降，因为在同一个上下文里同时比较这么多项目，AI 很难保持一致的评判标准。工作流的解法是锦标赛模式：把所有支持票两两配对，每对由一个独立的代理做比较判断，逐轮淘汰，最终选出最严重的若干条。每次比较只在两条支持票之间做，上下文窗口非常干净，判断质量也因此更高。同样的逻辑适用于简历筛选、功能优先级排序、设计方案选择。文章开头举了一个具体的例子，说"用工作流对一个命令行工具开一个锦标赛选出最好的三个名字"，现在你明白这背后的机制了——它就是在用锦标赛模式来处理一个依赖品味、没有标准答案的选择题。

第五个场景是挖掘 CLAUDE.md 规则。CLAUDE.md 是 Claude Code 的配置文件，你可以在里面写各种使用约束和偏好规则，比如"不要使用这个废弃的 API"、"提交信息必须遵循这个格式"，Claude Code 会在每次任务开始时读取这些规则。但问题是，你往往不知道应该往 CLAUDE.md 里面写什么。你需要先长期使用 Claude Code，观察它在哪些地方反复犯同类型的错误，然后才能总结出规则。这个过程本来需要人工去观察和归纳。现在可以这样做：建立一个工作流，让它扫描你最近几十个会话的历史记录，用并行代理来聚类你反复做的同类型修正，再用对抗性代理来验证每一条候选规则——"这条规则真的能预防一个实际发生过的错误吗？还是只是一个偶发的修正？"——最后把通过验证的高质量规则写入 CLAUDE.md。这是一个用 AI 来优化 AI 使用习惯的元循环，我个人认为这是整篇文章里最有创意的场景之一。

第六个场景是结构化调试。调试复杂 Bug 的最佳实践是先提出几个独立的假设，然后分别验证，看看哪个假设能解释所有观察到的现象。但如果在单一上下文窗口里做这件事，Claude 会有强烈的倾向去强化自己最初提出的那个假设，而不是平等地对待所有可能性——因为它已经在同一个上下文里"见过"自己生成这个假设的全过程，自我偏好偏差在这里非常明显。工作流可以结构性地解决这个问题：为不同的证据来源——日志、源代码、数据库记录——分别派一个独立的代理，每个代理从自己的视角出发，基于自己看到的证据独立生成假设。这些假设再交给一个"陪审团"代理来综合评估，判断哪个假设最能解释所有证据。这个结构让不同假设从产生之初就是相互独立的，没有任何一个假设在"辩论"中处于有利的起点。文章还特别说了，这个框架不只适用于代码调试，同样适用于分析销售数据为什么在某个月下滑、排查数据管道的故障，或者任何需要做根因分析的场景。

第七个场景是持续分类处理工作流，文章里叫 triage workflow。每个团队都有某种持续流入的队列——用户反馈、Bug 报告、客服请求——这些东西的量远超人工能处理的速度。分类工作流可以自动对每个新进入的条目做分类、与已有记录去重、然后决定是直接处理还是上升给人工。

这个场景里有一个安全设计细节值得特别提一下，文章里叫做"隔离"模式。意思是：读取不可信外部内容的代理，和执行高权限操作的代理，必须严格分开。比如，读取用户提交的 Bug 报告的代理，不能直接触发"关闭某个 GitHub Issue"或者"修改某个配置文件"这样的操作——它只能把解析结果传递给一个专门的行动代理，由行动代理来做实际的高权限操作。这样即使有人通过精心构造的用户输入来尝试做提示词注入攻击，也无法绕过这个隔离层来执行恶意操作。这是在多代理系统设计里非常重要的安全原则，值得记住。把分类工作流配合 loop 命令使用，就可以让它持续自动运行，实现真正意义上的异步自动化处理。

好，现在来到最实际的部分：作为一个工程师，你怎么开始用动态工作流？

最简单的入口是直接告诉 Claude Code 你想用工作流来做这件事，或者使用触发词 ultracode——这会确保 Claude Code 进入工作流创建模式，而不是直接开始执行任务。ultracode 这个触发词有点意思，它等于是在给 AI 发一个信号说"这件事需要你先设计怎么做，再去做，而不是直接上手"。

关于什么时候值得用工作流，文章给了一个很好的自测问题：这件事真的需要更多的计算量吗？大多数普通的编程任务，比如修一个 Bug、加一个功能，单代理模式已经足够，不需要开一个五人审查员的面板。动态工作流最适合的场景是：任务执行时间跨度长、可以大规模并行处理、有明显的独立验证需求，或者有大量同质化的子任务需要批量处理。

实际使用时有几个技巧你需要知道。

关于提示词，详细描述产生最好的结果。当你要求建立工作流时，如果在提示词里提到具体的模式——比如"用对抗性验证来检查结果"、"用锦标赛来选择最好的方案"——Claude 更容易理解你的意图，设计出更符合预期的工作流结构。

关于成本控制，你可以设置 token 预算。在提示词里说"用一万个 token 来完成这个任务"，Claude 会据此来控制工作流的规模和并发程度，避免超出预期的成本。文章里特别提醒，动态工作流通常会使用显著更多的 token，最适合复杂的、高价值的任务，不建议对所有任务都开工作流。

如果任务规模不大，你可以要求一个"快速工作流"。比如"快速地对这个假设做一个对抗性审查"，不需要开一个大规模的扇出，只是一个简洁的两方辩论就够了。

工作流可以保存和复用。在工作流菜单里按下 s 键可以把当前工作流保存下来，存放在用户目录下的 workflows 文件夹里。你也可以把 JavaScript 工作流文件打包成技能，分享给你的团队使用。如果你提供的工作流是以模板形式存在的，可以提示 Claude 把它当成模板而不是需要照单全收的固定脚本，允许它根据具体任务做适当的调整，这样灵活性会更好。

对于需要定期重复的工作流，比如持续的分类处理，或者每周一次的研究总结，可以配合 loop 命令设置成定时自动运行。结合 goal 命令可以设置明确的完成条件，让工作流在达到目标之后自动停止，而不是无限循环下去。

好，我们来做最后的总结。

这篇文章描述的，表面上是 Claude Code 的一个新功能，但更深层来看，它代表了一个思维方式的重要转变：从"你来设计 AI 的执行流程"，到"你描述目标，AI 来设计自己的执行流程"。动态工作流通过让每个子代理拥有独立的上下文窗口，从结构上解决了代理懈怠、自我偏好偏差和目标漂移这三个根本性的缺陷；同时，它把过去需要专业工程师手写的多代理编排代码，变成了 AI 可以按需即时生成的东西，大大降低了使用门槛。

我个人认为，有两点需要注意。第一，动态工作流的质量高度依赖于 Claude 对任务的理解。如果你的任务描述本身不清晰，或者 Claude 对任务的分析就已经偏了，那工作流的设计可能从根本上就存在问题，而这种"系统性设计缺陷"可能比单代理执行的错误更难察觉，因为流程看起来很规整。我建议在执行前要求 Claude 先解释一下它的工作流设计思路，确认逻辑合理再让它正式跑起来。第二，关于 Bun 重写的案例——工作流提高了并行度和过程的结构性，但不能凭空保证代码的正确性，正确性验证的责任仍然在测试套件上。没有完善的测试覆盖，工作流的大规模修改同样会引入难以发现的错误。

总体来说，如果你在用 AI 编程助手的过程中，有那种"这件事 AI 应该能帮我做，但每次效果都差强人意"的感受，不妨对照一下今天讲的三个失败模式，想想你遇到的情况是不是代理懈怠、自我偏好偏差还是目标漂移——说不定动态工作流正好是那把缺失的钥匙。
