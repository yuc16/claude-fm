好，我们开始今天的节目。

今天要聊的这篇文章，英文标题是 Lessons from building Claude Code: How we use skills，我把它翻译成中文叫"构建 Claude Code 的经验教训：我们如何使用技能插件"。这篇文章发表于二〇二六年六月三日，也就是半个月前，作者是 Anthropic 技术团队成员 Thariq Shihipar，他是 Claude Code 的核心开发者之一。

为什么这篇文章值得花二十五分钟认真听？因为它的性质和普通产品文档完全不同。这不是官方功能介绍，不是发布公告，而是 Anthropic 内部团队在真实工作中积累了几百个 skills 之后，把踩过的坑和总结出的经验，毫无保留地写出来的工程实践总结。这类文章之所以珍贵，是因为它讲的是那种你自己摸索可能要花好几个月才能悟到的东西。如果你在用 Claude Code，这篇文章能帮你大幅压缩试错成本；如果你还没在用，它能帮你提前想清楚这个工具对工程工作流到底意味着什么。

我们先来说说背景。

Claude Code 是 Anthropic 出品的一个 AI 辅助编程工具，你可以把它理解成一个能在你终端里工作的 AI 助手，它不只是生成代码，还能读文件、运行命令、管理 git、执行测试，真正做实际的工程操作。Skills 则是 Claude Code 里面的扩展机制，你可以把它理解成给这个 AI 装插件。当你输入一个斜杠命令，比如斜杠 deploy，或者斜杠 review，Claude Code 就会找对应的 skill，按照 skill 里面的指令来执行操作。Skills 让你可以把团队的最佳实践、内部工具的使用规范、各种自动化流程，打包成可以共享和复用的模块。

文章在一开始就纠正了一个非常普遍的误解。很多人以为 skills 就是 markdown 文件，就是一段写给 AI 看的说明书，类似于提示词工程里的系统提示，改改措辞就行了。但文章明确指出，skills 实际上是一个文件夹，里面可以有脚本、资产文件、数据文件，各种类型的内容，agent 可以在运行时去发现这些文件、读取它们、操作它们。而且在 Claude Code 里面，skills 还支持注册动态的钩子函数，也就是 hooks，让你在特定事件发生的时候触发特定操作。

这个区别听起来可能很微妙，但实际上影响极大。我们来想一个类比：你雇了一个实习生，如果你只给他一张纸，上面写着"去帮我处理用户投诉"，他只能按字面意思猜着走。但如果你给他一个工具包，里面有投诉分类的决策树、不同类型投诉对应的标准处理流程、常用的回复模板、升级给谁处理的联系人列表，以及一个记录每次处理结果的日志，他就能自主完成复杂得多的任务，遇到新情况也知道该从哪里入手。Skills 就是这个工具包，而不只是那张纸。

好，背景铺垫完了，我们来看文章最核心的内容。

Anthropic 的团队把内部所有在用的 skills 做了一个系统梳理，发现这些 skills 大致可以分成九个类别。他们指出，写得好的 skill 通常清晰地归属于其中一个类别；那些效果不好的 skill，往往是想同时覆盖太多功能，结果让 agent 搞不清楚该在什么情况下用它、用了之后该重点关注哪个方面。这九个类别形成了一张可以用来审视自己工具库的地图，帮你看出哪里有空白、哪里最值得继续投入。

第一类叫做使用指南类，定位是告诉 Claude 怎么正确使用某个特定的库、命令行工具或者 SDK。这里的关键词是"正确"，重点不是把官方文档搬进来，而是专注在 Claude 特别容易出错的地方。文章给的例子非常接地气：你内部有个计费库，有一堆边界情况和容易踩的坑，那就写一个 billing-lib 的 skill，把这些坑专门列出来。你内部有个平台命令行工具，每个子命令的适用场景和用法，也专门写一个 skill 来说清楚。再比如，你们公司的出口网关有特殊配置，能连哪些外部主机、遇到连接被拒绝时该怎么排查、怎么申请添加白名单，这些实际操作细节都可以写进去。这类 skill 的核心价值在于，它补充的是 Claude 通过训练数据无法学到的那部分知识，也就是你们团队内部特有的情况和规范。

第二类叫做验证类，作用是描述如何测试和验证你的代码在真实场景下能正确工作。文章在这里用了很重的笔墨，他们说这类 skill 在实际测试中对 Claude 输出质量的提升是所有类别里最明显的，并且专门建议，值得让一个工程师花一整周时间来打磨你的验证类 skills，这个投入非常划算。

为什么验证类技能这么重要？你来想想这个场景：Claude 写完代码之后，如果没有一个可靠的验证机制，它只能用"看代码"的方式来判断对不对，而这种判断的准确率是非常有限的。但如果你给它一个可以真正运行起来的验证框架，它就能得到实际的反馈信号，知道代码是不是真的按预期工作了。这是从"自我评估"到"外部验证"的本质跨越。

具体的做法可以很有创意：让 Claude 录制测试执行的视频，这样你事后可以回放，看它到底测了什么步骤、是怎么操作的；或者在测试流程的每一步都加入程序化的状态断言，而不只是看最终的返回值对不对。文章给的例子包括：注册流程驱动器，用无头浏览器走一遍完整的注册流程，包括邮件验证和新用户引导，在每一步都有钩子验证当前状态；结账验证器，用 Stripe 的测试卡驱动结账界面，验证发票是否真的正确落地在预期状态里；还有给需要真实终端环境的交互式命令行测试专用的 tmux 驱动器。

第三类是数据和监控类，把 Claude 和你的数据栈、监控系统连接起来，给它提供精确的数据访问路径。如果你做过数据分析就知道，在真实的生产环境里，查哪张表、join 哪些字段、用什么过滤条件，这些细节极其重要，搞错了分析结论就会完全跑偏，而且你不一定马上发现。这类 skill 里面会包含获取数据的库、凭据配置、仪表盘 ID，以及常见的数据查询工作流。漏斗查询技能会告诉 Claude，要看用户从注册到激活再到付费的转化，应该 join 哪些表，真正的用户唯一标识符在哪张表里。群组比较技能可以对比两个用户群的留存率或者转化率，自动标记统计显著的差异，链接到对应的群体定义。Grafana 技能里存着数据源的唯一标识符、集群名称，以及遇到什么类型的问题去看哪个仪表盘的查找表。Datadog 技能里有字段命名规范、服务列表、指标前缀约定。

第四类是工作流自动化类，把重复性的操作整合成一条命令。这类 skill 的指令本身往往不复杂，但可能依赖其他 skills 或者外部工具来完成。文章特别提到一个很实用的技巧：把每次执行的结果保存到日志文件里，下次运行时 Claude 可以回顾历史记录，保持前后一致，或者只报告新增的变化，避免重复信息。每日站会发布器可以聚合你的工单系统、GitHub 活动和以前的 Slack 内容，自动生成格式化的站会更新；工单创建器强制执行字段格式校验和枚举值验证，创建后自动通知审核人并在 Slack 发链接；每周回顾把合并的 PR、关闭的工单、部署记录汇总成格式化的帖子。

第五类是脚手架类，为代码库里特定类型的功能生成框架代码。这类 skill 特别适合那些包含自然语言层面规则的脚手架需求，因为这些规则很难通过纯代码工具来强制执行。比如，新建一个 service 时需要带上你们公司特定的注解格式、日志配置和部署配置；创建数据库迁移文件时要套用模板并提示常见的踩坑点；从零初始化一个新的内部应用时，带着认证模块、日志系统、部署配置全部预先接好。

第六类是代码质量类，强制执行代码规范、辅助代码审查。文章里有一个特别有意思的例子叫对抗性审查：这个 skill 会创建一个全新的子 agent，让它用全新的视角批评你的代码，根据批评结果实施修复，不断迭代，直到所有发现的问题都降级到可以接受的小问题为止。这个模式的有趣之处在于，它利用了不同 agent 实例之间的视角差异来提升审查质量，而不是让同一个 Claude 自己审查自己写的东西。还有代码风格技能，专门处理 Claude 默认风格不符合你们团队规范的地方；测试实践技能，明确告诉 Claude 怎么写测试、什么情况下测什么。

第七类是部署类，涵盖拉取、推送、部署代码的全流程。PR 托管技能可以持续监控一个 PR，自动重试不稳定的 CI，解决合并冲突，条件成熟时开启自动合并；服务部署技能从构建到冒烟测试、到逐步流量切换、到错误率实时对比、再到出现回归时自动回滚，全流程覆盖；热修复的 cherry-pick 工作流在隔离的工作树里操作，安全地把修复摘到生产分支，并附带标准化的 PR 模板。

第八类是故障调查类，接收一个故障症状，比如 Slack 里的告警消息或者特定的错误特征，走一遍多工具的自动调查流程，最终产出结构化的报告。想象一下，凌晨三点告警响了，你刚从睡梦中醒来，如果有一个 skill 帮你自动检查最常见的嫌疑点、关联相关日志、给出初步调查报告，那会省去多少时间和脑力。服务调试技能把症状映射到调试工具，再映射到具体的查询模式，针对你流量最高的服务量身定制；值班处理技能拉取告警内容，按照预设的调查流程逐步检查，格式化发现的问题；日志关联技能给定一个请求 ID，从所有可能处理过这个请求的系统里拉取对应日志，把分散在各处的信息整合起来。

第九类是运维操作类，执行例行维护和那些涉及破坏性操作的流程，需要额外的护栏来保证安全。孤儿资源清理会先找出孤立的 pod 或者存储卷，发到 Slack 告知团队，等一个浸泡期，确认无误后才执行级联删除，这套流程既自动化又安全；依赖管理按照你们组织的审批流程推进升级；成本调查技能专门帮你定位"为什么我们的存储或者出口账单突然涨了"这类问题，带着具体的数据桶和查询模式。

好，九个类别讲完了。如果让我用一句话概括这套分类框架的价值，我会说：它给了你一张可以用来做自我审计的地图。你把手头现有的 skills 对应到这九个格子里，空着的格子就是下一步最值得投入的方向。不需要凭感觉，对着地图看就行了。

接下来，文章进入了我认为价值最浓缩的部分：如何写好一个 skill。文章给出了九条核心原则，我把最重要的几条展开来说。

第一个核心原则：不要在 skill 里重复 Claude 已经知道的东西。Claude 本身就会写代码，本身就能读你的代码库。如果你的 skill 里面大部分都是 Claude 默认就会做的事，你只是换个措辞重新说了一遍，这个 skill 只是在占用上下文 token，没有增加任何实质价值。那什么是真正有价值的内容呢？文章说，是那些能把 Claude 推出它默认思维方式的信息。

文章举的最好例子是前端设计技能。这个技能是 Anthropic 的一个工程师跟用户反复迭代出来的，专门改善 Claude 的设计品味。Claude 在默认情况下有非常明显的视觉风格倾向：总是选 Inter 字体，总是用紫色渐变，这种风格一眼就能认出来是 AI 生成的，缺乏辨识度和个性。设计技能专门针对这些默认倾向给出了替代方向，告诉 Claude 什么样的设计选择才是有品味的、独特的。这是典型的"需要用 skill 来纠正 Claude 默认行为"的场景，也是这类知识最值钱的地方。

第二个核心原则：Gotchas 段落是整个 skill 里信噪比最高的内容。Gotchas 就是踩坑记录，记录 Claude 在使用这个 skill 时特别容易犯的错误。文章建议你持续积累这个段落，每次 Claude 踩了新的坑，就把它追加进去。随着时间推移，这个段落会成为 skill 里最有价值的那部分。

文章给了三个非常具体的 Gotchas 例子，我觉得值得一字一句地重复一遍，因为它们完美展示了这类内容该怎么写、写到什么颗粒度。

第一个例子是这样的：subscriptions 表是追加写入的，你想要的那一行是 version 字段值最大的那一行，不是 created_at 最新的那一行。你看，这句话信息密度极高，它不只是说"注意版本号"这种模糊提示，而是精确描述了错误的做法，给出了正确的替代方案，还给出了具体的字段名。把这句话放进 skill 里，Claude 就不会再犯这个错。

第二个例子：这个字段在 API 网关里叫 @request_id，在计费服务里叫 trace_id，但它们是同一个值。这种跨系统字段命名不一致的问题，文档里不会特别说明，Claude 也完全猜不到，只有在两个系统之间来回查过数据、真实踩过坑的人才会知道。这类知识一旦写进 Gotchas，就永远不会再因为这个原因出错了。

第三个例子：Staging 环境就算 Stripe Webhook 没有真正处理成功，也会返回两百这个 HTTP 状态码，你必须去检查 payment_events 表才能知道支付的真实状态。这种行为从接口层面看完全正常，但实际上隐藏着状态不一致的风险，非常容易让人以为测试通过了，但其实支付根本没有完成。

这三个例子告诉我们，好的 Gotchas 不是泛泛的"注意这里"，而是精确描述那些"看起来可以这么做，但实际上是错的"的情况，同时给出正确的做法和具体的细节。粒度越细，价值越大。

第三个核心原则：把整个文件夹结构当成一种上下文工程来设计。文章说，你要告诉 Claude 你的 skill 文件夹里有哪些文件，它会在合适的时机主动去读。这个思路叫做渐进式披露，意思是按需展示信息，而不是一次性把所有内容都塞进主文件里给模型看。

最简单的实现方式是把详细的技术参考文档单独拆成文件。你可以把详细的函数签名和使用示例放在 references 目录下的 api.md 里，主 skill 文件只告诉 Claude 这个参考文件在哪、什么情况下去读。如果 skill 最终要产出一个特定格式的报告，你可以在 assets 目录下放一个模板文件，让 Claude 复制这个模板然后填充内容，格式就不会跑偏。你还可以有 scripts 文件夹存可以直接执行的脚本、examples 文件夹存参考示例，这些都是 Claude 可以在需要时取用的资源。

这个思路和软件工程里的关注点分离非常像。你不会把所有业务逻辑都堆在一个函数里，同样地，你也不应该把所有 skill 知识都堆在一个 markdown 文件里。分层组织信息，不仅让 skill 更容易维护，也让 Claude 更容易在对的时机找到它需要的东西。

第四个核心原则：给 Claude 足够的灵活性，不要把指令写得太机械。文章提醒，Claude 在执行时会尽量严格遵循你的指令，而 skill 是设计来被反复复用的，覆盖各种不同的情况，所以你要小心不要把每一步都规定死。好的 skill 指令是原则性的，告诉 Claude 目标是什么、约束是什么，而不是程序性的，逐步告诉它该怎么走。给它信息和判断依据，然后让它自己决定怎么应对具体情况。如果你把所有细节都写死，Claude 遇到你没有预料到的情况时就会卡住，或者机械地套用不适合的步骤，反而出问题。

第五个核心原则：skill 的描述字段是触发条件，不是功能摘要。这是很多人会忽视的细节，但文章说它的影响非常关键。Claude Code 启动一个会话时，会构建一个所有可用 skills 的索引，每条包含该 skill 的描述字段。Claude 扫描这个索引来判断，当前的用户请求有没有合适的 skill 可以处理。所以，描述字段的内容直接决定了 Claude 在什么情况下会想到调用你的 skill。

如果你的描述写的是"用于代码审查和质量检查"，Claude 只有在用户明确说到这几个词时才可能触发。但如果你写的是"当用户准备提交代码、发起合并请求、或者要求检查实现质量时调用"，覆盖的场景就宽泛得多。写描述的时候，你要从"Claude 在什么时候应该想起来用这个工具"的角度出发，而不是从"这个工具做了什么"的角度出发。这个认知转变，对 skill 的实际触发率影响非常大。

第六个核心原则：可以给 skill 添加记忆能力，让它在多次运行之间保持状态。这是文章里我个人觉得最有创意的技巧。你可以在 skill 的数据目录里保存数据，最简单的形式是一个追加写入的文本日志文件，复杂一点可以用 JSON 文件，甚至 SQLite 数据库。

举一个具体的例子：一个每日站会发布 skill 可以把每次生成的站会内容保存在日志文件里。下次你运行这个 skill 的时候，Claude 先读取自己的历史记录，对比当前状态和上次汇报之间的差异，然后只生成包含新增内容的站会更新，而不是从零开始重新汇报所有事情。这解决了一个很实际的问题：没有记忆的 skill 每次运行都是全新开始，生成的内容可能前后不一致，或者重复报告已经汇报过的事情。

第七个核心原则：给 Claude 提供可以直接调用的脚本和工具库，让它专注于组合和决策，而不是重建基础代码。你给 Claude 提供现成的函数，它就可以把推理能力花在判断"下一步做什么"上，而不是反复写同样的样板代码。

文章举了数据科学技能的例子：skill 里面包含一组辅助函数，有从你的事件数据源拉取数据的函数、过滤特定用户群的函数、聚合各种指标的函数。Claude 可以即时生成脚本来组合这些函数，回答"上周二到底发生了什么"这类更复杂的分析问题。没有这些辅助函数的话，Claude 每次都得重新写一遍数据连接逻辑，不仅浪费 token，还容易在细节上出错，而且每次写的方式可能都不一样，结果难以复现。

另外，文章还提到了一个相对小但值得知道的功能：可以在 skill 里注册只在当前会话内有效的钩子。这些钩子在你调用这个 skill 的会话期间激活，关掉会话就自动失效，不会影响其他会话。这对于那些平时不想一直开着、但在特定场景下非常有用的功能来说很合适。文章给的例子是一个叫 careful freeze 的操作模式，一种更保守谨慎的操作姿态，只在你明确请求这种保护时才会开启，用完就关掉。

好，核心原则部分讲完了，我们来看文章的最后一块：如何在团队里分享和分发 skills。

有两种主要的分发方式。第一种是直接把 skill 检入到代码库，放在 .claude/skills 目录下，所有使用这个 repo 的人都能直接用到，简单粗暴，对小团队来说已经足够。第二种是把 skill 做成插件，上传到 Claude Code 的插件市场，让团队成员按需安装，每个人的安装列表可以不一样。

文章给出了一个重要的权衡点：每个被检入 repo 的 skill 都会给模型增加一点上下文开销。团队规模小、skills 数量少的时候无所谓。但随着 skills 越来越多、团队规模扩大，这个开销就变得可观了。插件市场模式的好处是让每个人只带着自己真正会用到的工具，保持上下文的干净。

Anthropic 自己管理内部插件市场的方式很有启发性：他们没有一个中央团队来审核和决定哪些 skill 能进市场，而是完全有机生长。想分享 skill 的人把它上传到 GitHub 的一个沙盒文件夹，然后在 Slack 或者内部论坛上发消息告诉大家。如果一个 skill 获得了足够的使用和关注，它的作者就提一个 PR，把它正式迁移进插件市场。这个方式用使用量来投票，没有官僚审批流程，完全去中心化。潜在的问题是可能有一些很有价值的 skill 因为知名度不够而被埋没，但这也是他们愿意接受的代价。

关于 skills 之间的依赖，文章也提了一下目前的现状。比如，你可能有一个文件上传 skill，然后有一个 CSV 生成 skill，后者需要调用前者来上传最终结果。这种依赖关系目前还没有原生的依赖管理机制，但你可以在 skill 的指令里直接引用另一个 skill 的名字，模型运行时如果发现那个 skill 已经安装，就会去调用它。这个机制目前比较原始，但能解决基本的组合需求。

最后，如何追踪 skill 的使用效果？Anthropic 内部用一个工具调用前钩子来记录每次 skill 的调用情况，这样他们就能看到哪些 skill 调用频率很高，哪些 skill 的触发率低于预期，需要优化描述字段或者加大推广力度。

好，所有内容都讲完了，我们来谈谈这些对你实际工作的意义。

如果你现在还没有认真使用 Claude Code 的 skills，这篇文章给了你一个非常具体的起点。文章最后说得很清楚，他们内部最好的 skills 大多数都是从几行指令和一两个 Gotcha 开始的，随着 Claude 反复踩坑，不断往里面加内容，慢慢变好的。所以最好的方式是：从你最频繁遇到的一个痛点开始，写一个最简单的 skill 版本，用一两周时间观察它在哪里出错，然后把这些错误加进 Gotchas 段落，迭代改进。不要等到想清楚了整套体系再开始，因为你想不出来，只有用了才知道。

如果你已经在用 skills，这篇文章的九类框架是做审计的好工具。把手头的 skills 对应到九个类别上，看看哪些格子空着。我个人认为，最容易被忽视的是验证类和故障调查类，因为这两类前期投入比较大，但文章明确说验证类的效果最显著，故障调查类在真正遇到突发情况时价值巨大。先把这两个空的格子填上，可能比把现有 skills 打磨得更精细收益更大。

对于带团队的工程师或者负责工程效率的人来说，Anthropic 的有机生长模式很有参考价值。你不需要自上而下设计一套完整的 skill 体系，而是建立一个低摩擦的分享机制，让愿意贡献的人方便地贡献，用实际使用量来自然筛选出最有价值的工具。然后用监控钩子来追踪哪些 skill 触发率低，定期去看一眼那些 skills 的描述字段有没有写成触发条件的形式，这往往是触发率低的最直接原因。

我个人认为，这篇文章最大的价值不在于任何一个具体技巧，而在于它背后的认知框架：skills 不是你给 AI 写的说明书，而是你和 AI 之间协作界面的设计。设计这个界面，意味着你要考虑 Claude 在什么时候该想起用这个 skill、在 skill 执行过程中如何获取它需要的信息、如何处理它默认会犯的错误、如何在多次使用之间积累经验。这套思路，和你设计一个给人用的工具的思路，其实高度相似。你设计的不只是一段提示词，而是一个能和 Claude 协同工作的、持续进化的工具系统。
