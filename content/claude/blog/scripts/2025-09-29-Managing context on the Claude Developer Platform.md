大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,英文标题是 Managing context on the Claude Developer Platform,翻译过来就是"在 Claude 开发者平台上管理上下文"。这篇文章是 Anthropic 官方博客发表的,发表时间是二〇二五年九月二十九日。

先说一下时间感。今天是二〇二六年六月二十一日,这篇文章发表到现在已经差不多过去八九个月了,放在 AI 这个行业里,这个时间跨度其实已经不算短了。所以我要提醒大家,听这期节目的时候,要把它当成一个历史节点来理解,而不是当成最新动态。这篇文章发表的那个阶段,正好是 Claude Sonnet 四点五这个模型刚刚推出的时间点,当时整个行业里关于智能体,也就是英文里说的 agent,这个概念已经从一个新鲜的实验性话题,逐渐变成了大家真正拿去做生产级应用的东西。换句话说,二〇二五年下半年是各家大模型公司开始拼"智能体到底能跑多久、跑多稳"的一个关键阶段,而这篇文章正是在这个背景下出现的。

那为什么这篇文章值得我们花二十多分钟来讲呢?因为它讲的不是一个花哨的新功能,而是触及了智能体开发里一个非常本质、非常痛的问题,就是上下文窗口的管理问题。如果你自己写过哪怕一个稍微复杂一点的智能体程序,你大概都遇到过这种情况,智能体跑着跑着,突然就报错说上下文超限了,或者表现越跑越差,问它前面做过的事情它说不知道。这篇文章就是专门来解决这类问题的,所以我觉得它的实用价值非常高,值得我们仔细拆一拆。

我们先讲讲背景,为什么会有这个问题。我们知道,大语言模型每次处理请求,都是在一个叫上下文窗口的有限空间里工作的,这个窗口你可以理解成模型的"工作台",台子有多大,决定了它一次能同时看多少东西。早期我们用 Claude 聊天,基本上就是一问一答,上下文消耗得很慢。但是智能体不一样,智能体要不断地调用工具,比如读文件、搜索网页、跑测试、查数据库,每一次工具调用产生的结果,都会原封不动地塞进上下文窗口里。

你可以想象一下,一个智能体在帮你处理一个比较复杂的编程任务,它可能要读几十个文件,跑几十次测试,搜索几十次资料。这些工具调用的结果,有很多其实只在当时那一步有用,后面就没用了,但是它们依然占着上下文窗口里的位置,越积越多。这就好比你在一个会议室里开一场马拉松式的会议,每个人发言的时候都把自己带的资料摊在桌子上,从来不收走,开到后面整张桌子已经被堆满了,根本找不到真正有用的那几份文件在哪里。这就是文章里说的,智能体在处理越来越复杂的任务、产生越来越多工具调用结果的时候,很容易把有效的上下文窗口给耗尽。

那开发者在这种情况下通常会怎么办呢?文章里点出了一个很真实的困境,就是开发者往往被迫在两个都不太好的选择之间二选一。第一个选择是直接截断,把早期的对话记录硬生生地砍掉,这样虽然能省出空间,但是很可能把关键信息也一起砍没了,智能体就会"失忆",忘了之前已经做过什么决定、踩过什么坑。第二个选择是不截断,但是任由上下文堆满垂直信息,模型的表现就会因为信息过载而下降,因为这些堆积的旧工具结果会稀释掉真正重要的信号,让模型抓不住重点。这两个选择,说白了就是"丢内容"和"丢性能"之间二选一,听上去都不太美好。

Anthropic 这次给出的方案,就是同时推出两个互相配合的能力,一个叫上下文编辑,一个叫记忆工具。这两个东西解决的是两个不同维度的问题,我们一个一个来拆。

先说上下文编辑。它的核心思路其实挺直观的,就是当对话快要接近 token 上限的时候,系统会自动清理掉那些已经过期、不再有用的工具调用和工具结果,但是会保留对话的整体脉络和逻辑连贯性。这里的关键词是"自动",也就是说开发者不需要自己写一套复杂的逻辑去判断哪些内容该删哪些该留,这个判断和执行的过程是系统帮你完成的。

我用一个比较接地气的类比来说明这件事。你可以把它想象成你电脑上的内存管理机制。你打开了很多个浏览器标签页,电脑内存快不够用的时候,操作系统会自动把那些你很久没用的标签页"挂起",释放出内存给你正在用的那个标签页,但是这些标签页的存在感,比如它们的标题、它们在浏览历史里的位置,其实还在,你需要的时候可以再打开。上下文编辑做的事情有点类似,它不是粗暴地把整段对话历史一刀切掉,而是有针对性地把那些已经"用过的"工具调用细节清掉,但是保留对话往前推进的那条主线,让 Claude 依然知道自己是在做什么任务、走到了哪一步。

这样做带来两个好处。第一个好处很直接,就是延长了智能体能够持续运行的时间,因为腾出来的空间可以用来容纳后面新产生的工具调用结果,这就相当于给智能体的"工作台"做了定期清理,让它能一直干活而不被自己堆的杂物绊住脚。第二个好处其实更有意思,文章里提到这样做还能提升模型本身的表现,因为当上下文里只剩下真正相关的信息的时候,Claude 在做推理判断的时候,注意力会更集中,不会被那些已经过时的、没用的工具结果干扰。这一点我觉得特别值得强调,因为很多人一开始会觉得上下文管理只是为了"省空间",但实际上它对模型推理质量本身也是有直接帮助的,信息噪音少了,信号自然就更清晰。

接下来我们讲记忆工具,英文叫 memory tool。这个功能解决的是另外一个维度的问题,就是跨会话的信息保留问题。上下文编辑解决的是"当前这一场对话里东西太多怎么办",而记忆工具解决的是"这场对话结束了,下次再开一场新对话,之前积累的经验怎么留下来"。

记忆工具的实现方式,简单说就是让 Claude 能够通过一套基于文件的系统,去创建、读取、更新、删除存储在一个专门的记忆目录里的文件,这个目录是存放在你自己的基础设施上的,也就是开发者自己管理这个存储后端,数据存在哪里、怎么持久化,完全由开发者自己掌控。这一点其实和我们这次对话里提到的我自己的记忆系统设计思路是相通的,就是把信息变成文件,存在一个持久化的地方,下次需要的时候再去读取,而不是每次都从零开始。

文章里特别强调了一点,记忆工具是完全在客户端通过工具调用的方式来运作的。这句话的意思是,Anthropic 并没有在他们自己的服务器上偷偷帮你存一份数据,而是通过标准的工具调用机制,让 Claude 发出"我要写一个文件"或者"我要读一个文件"这样的指令,具体怎么落地存储,是你自己的代码去实现的。这种设计对开发者来说其实是很友好的,因为数据主权完全在你自己手里,你可以选择存在本地磁盘,可以存在云端数据库,也可以接入你公司已有的知识库系统,灵活度很高,同时也意味着安全合规方面的责任也落在了开发者自己身上,这一点需要注意。

那记忆工具具体能带来什么实际效果呢?文章举了几个例子,我觉得很形象。比如智能体可以随着时间推移,逐步建立起一个知识库,因为每次会话学到的东西都能存下来,不会因为会话结束就清零。再比如,智能体可以在多次会话之间维持项目状态,你今天让它帮你处理一个项目,做了一半,明天接着让它做,它依然知道昨天做到哪一步、遇到了什么问题。还有就是,智能体可以引用之前学到的经验,而不需要把所有东西都塞进当前这次对话的上下文里,这就避免了每次都要重新"复述"一遍背景信息,既省了 token,也减少了出错的机会。

讲到这里,我想用一个大家比较容易理解的场景来串一串这两个功能怎么配合工作。假设你让一个 Claude 智能体去做一个比较大的编程任务,比如帮你重构一个有点年头的代码库。这个任务可能要持续好几个小时,中间要读很多文件,跑很多次测试,还可能要查阅外部文档。上下文编辑在这个过程里做的事情,是不断地把那些已经读过、已经跑完、已经确认没问题的旧文件内容和旧测试结果清理掉,腾出空间;而记忆工具做的事情,是把这个过程里产生的关键洞见,比如说"这个模块的接口设计有一个隐藏的坑,要小心"或者"这个架构决策是为了兼容老版本的某个依赖",写到记忆文件里存起来。这样一来,即便这次会话因为某种原因中断了,或者任务量实在太大需要分多次会话来完成,智能体在下一次会话开始的时候,依然能够通过读取记忆文件,迅速找回之前积累的关键判断,而不需要重新把整个代码库读一遍。

文章里同时强调,这两个功能在 Claude Sonnet 四点五这个模型上得到了进一步的增强,因为这个模型本身内置了上下文感知能力,也就是说模型自己能够在对话进行的过程中,持续追踪还剩多少可用的 token 空间,从而更主动、更聪明地去管理上下文,而不是被动地等系统外部去触发清理动作。这一点其实挺关键的,因为这意味着模型本身已经具备了一定的"自我感知"能力,知道自己的工作台还剩多少空间,这样它在决定要不要主动总结、要不要主动存档信息的时候,会更有分寸感。

接下来我们看看官方给出的实测数据,这部分我觉得是这篇文章里最有说服力的内容。Anthropic 在一个内部的智能体搜索评测集上做了测试,专门考察上下文管理能力对复杂多步骤任务的影响。结果显示,把记忆工具和上下文编辑结合起来使用,相比于没有这些功能的基线,性能提升了百分之三十九。如果只单独用上下文编辑,不配合记忆工具,提升幅度是百分之二十九。这两个数字放在一起看,你能感受到,单独的上下文编辑已经能带来不小的提升,但是再叠加上记忆工具之后,提升幅度还能进一步拉高十个百分点左右,说明这两个功能确实是互补的,不是简单的叠加重复。

另外还有一个非常直观的实验,是在一个一百轮的网页搜索评测里做的。这里"一百轮"指的是智能体要连续完成一百次搜索相关的交互步骤,这种任务如果没有上下文管理,基本上跑到中途就会因为上下文耗尽而失败,根本撑不到一百轮。而引入了上下文编辑之后,这种原本会失败的工作流程能够顺利跑完,同时还有一个让人印象很深的数字,就是 token 消耗降低了百分之八十四。这个数字其实很有意思,因为它不光是说"任务能跑完了",还说明跑完这个任务所消耗的成本大幅下降了。换算成实际意义就是,同样的预算,你能让智能体多跑接近六倍的类似任务,这对于成本敏感的生产环境来说,吸引力是非常大的。

文章里还列举了几个比较典型的应用场景,我觉得值得我们一个个展开说一说,因为这些场景基本上覆盖了大部分智能体开发者会遇到的真实需求。

第一个场景是编程类的智能体。在写代码、调试代码这类任务里,上下文编辑负责清理掉那些旧的文件读取记录和测试结果,因为这些东西一旦确认过了,留着也没用,而记忆工具则负责保存调试过程里得到的关键洞见和架构决策。这样组合起来,智能体就能够在一个很大的代码库上持续工作,而不会因为读了太多文件、跑了太多测试而把进度搞丢。

第二个场景是研究类的智能体。做研究这种任务,往往需要做大量的搜索,搜索结果一波接一波地涌进来,大部分搜索结果在被消化吸收之后其实就没用了,这时候上下文编辑就负责把这些旧的搜索结果清掉,而记忆工具负责把真正提炼出来的关键发现存下来。这样随着时间推移,智能体积累的是越来越精炼的知识库,而不是越来越臃肿的搜索记录堆积,整体表现会随着时间推移变得越来越好。

第三个场景是数据处理类的智能体。这类任务通常会涉及大批量的原始数据处理,智能体一边处理一边把中间结果存进记忆里,同时让上下文编辑把那些已经处理完的原始数据从上下文里清理出去。这样就能处理那些原本规模会超出 token 上限的工作流程,因为真正占空间的是原始数据而不是中间结论,把原始数据腾出去之后,智能体就能在有限的上下文里持续吞吐更多的数据量。

讲完这些场景,我们来聊聊这对我们这些实际做开发的人来说,到底意味着什么,我们应该怎么把它用起来,又有哪些坑需要提前避开。

首先一个很现实的问题是,这两个功能目前的状态是公开测试版,也就是英文里说的 public beta。截止到文章发表的时候,它们已经可以在 Claude 开发者平台上原生使用,同时也支持在亚马逊的 Bedrock 平台和谷歌云的 Vertex AI 平台上使用。这意味着不管你现在用的是哪一家云服务来调用 Claude,基本上都有机会接入这两个能力,不需要被绑定在某一个特定的云平台上,这对企业级用户来说是个比较友好的设计。当然因为是测试版,具体的接口细节、参数命名、限制条件,随着时间推移肯定会有调整,如果你打算在生产环境里用,我建议你直接去翻官方最新的文档,而不要完全照搬这篇文章里描述的细节,毕竟过去这八九个月,接口层面应该已经有不少迭代了。

第二点,我觉得开发者在动手接入这两个功能之前,首先要想清楚自己的智能体到底属于哪一类场景。如果你的智能体任务相对简单,一次会话几分钟就结束了,工具调用次数不多,那其实你可能根本用不上这套机制,强行接入反而增加了系统复杂度,没有必要。但是如果你的智能体要做的是那种动辄运行几十分钟甚至几个小时的长任务,中间要调用大量工具,比如说自动化测试智能体、代码重构智能体、长周期的市场调研智能体,这种场景下,上下文编辑几乎是必备的能力,不接入的话,大概率你迟早会撞到上下文上限那道墙。

第三点是关于记忆工具的落地细节,我想多说几句,因为我觉得这是这篇文章里最值得开发者认真消化的部分。记忆工具的本质是把"记忆"这件事从模型内部剥离出来,变成一个外部的、可控的文件系统。这意味着你作为开发者,需要自己去设计这个记忆系统的存储方案,包括存在什么地方、怎么组织文件结构、什么时候清理过期的记忆、怎么防止记忆文件无限膨胀。这其实给了开发者很大的自由度,但同时也是一份责任。如果你完全不做任何治理,任由 Claude 自己往记忆目录里堆文件,时间长了你的记忆系统本身也可能会变得臃肿、混乱,甚至出现信息过时但没有更新的问题,这一点跟我们人类自己的笔记系统会遇到的问题其实是一样的,记笔记容易,定期整理笔记才是难的部分。

第四点是安全和隐私方面的考量,这一点文章里没有展开讲,但是我觉得作为开发者必须自己想到。因为记忆工具存储的数据是在开发者自己的基础设施上,如果你的智能体处理的是涉及敏感信息的任务,比如说客户数据、内部财务信息,那么这些信息一旦被写进记忆文件,就需要按照你公司原本的数据安全规范来管理,包括访问权限控制、加密存储、定期审计等等。不能因为这是"Claude 自己存的数据"就放松了对这部分数据的管控,本质上它跟你系统里其他的持久化数据没有区别,该有的安全措施一样都不能少。

第五点,关于上下文编辑这个功能,我想提醒大家注意一个细节,就是它清理的是"过期的工具调用和结果",但是怎么界定"过期",这背后是有一套判断逻辑的,系统会试图保留对话的连贯性。这意味着在大多数情况下,你不需要太担心关键信息被误删,但是如果你的任务有一些特别依赖早期工具调用细节的逻辑,比如说某个早期搜索的具体原始返回内容后面还要反复引用,那么这种情况下,你可能需要主动通过记忆工具把这些关键细节提前固化下来,而不是完全依赖上下文编辑自动帮你保留,因为它的设计初衷是清理冗余,不是无差别地保留所有细节。

我自己看完这篇文章,有几个个人的感受想跟大家分享一下。我个人认为,这次发布最有价值的地方,不是单独的某一个功能,而是把"清理"和"留存"这两件事拆开来,分别用两个互相独立又能配合的机制去解决,这个设计思路本身是值得我们学习的。很多时候我们在做系统设计的时候,容易把"删除旧数据"和"保留重要数据"这两件事混在一起处理,结果两头都做不好,要么删多了丢失了关键信息,要么为了不丢信息什么都不删,系统越跑越臃肿。Anthropic 这次的做法是把这两件事彻底解耦,上下文编辑专心负责清理短期的、会话内的冗余,记忆工具专心负责长期的、跨会话的知识沉淀,各管一段,职责清晰,这种思路其实不止适用于 Claude 智能体开发,放到我们自己平时设计任何带状态的系统里,都是一个值得借鉴的模式。

另外我个人还有一个观察,就是这篇文章发表的时间点,正好踩在智能体从"能跑"走向"能稳定长时间跑"的这个转折阶段。如果你回顾一下二〇二四年到二〇二五年上半年这段时间行业里讨论智能体的内容,大家更多关注的是智能体能不能完成任务、推理能力够不够强,但是到了二〇二五年下半年,讨论的重心明显往"工程化"和"可靠性"上偏移了,大家开始更关心智能体能不能稳定地跑很长时间、能不能控制住成本、能不能在多次会话之间保持连贯。这篇文章正好是这个趋势下的一个典型产物,所以即便现在已经过去了八九个月,它所代表的这个行业转折点,我觉得依然是有参考价值的,值得我们回过头来理解清楚。

好,我们来总结一下今天这期节目的内容。这篇文章讲的是 Anthropic 在二〇二五年九月底,给 Claude 开发者平台引入的两个上下文管理新能力,一个是上下文编辑,负责自动清理会话内已经过期的工具调用和结果,延长智能体的有效运行时间并提升推理质量;另一个是记忆工具,通过一套基于文件的系统,让 Claude 能够跨会话地保存和读取关键信息,把知识沉淀下来。这两个功能配合使用,在官方的内部评测里带来了百分之三十九的性能提升,在长达一百轮的网页搜索测试里,还把 token 消耗降低了百分之八十四。这两项能力当时已经在 Claude 原生平台和亚马逊 Bedrock、谷歌云 Vertex AI 上以公开测试版的形式开放。我个人认为,对于正在做或者准备做长周期智能体应用的开发者来说,这篇文章提供的不仅是两个具体的工具,更重要的是一种思路,就是把"清理冗余"和"留存知识"当成两件独立的事情分别去设计,这个思路放到任何带状态的系统设计里都是适用的。好了,今天的节目就到这里,我们下期再见。
