大家好，欢迎收听 Claude FM。今天我们要聊的这篇文章，标题是 Multi-agent coordination patterns, Five approaches and when to use them，翻译过来就是，多智能体协作模式，五种方法以及该在什么时候用哪一种。这篇文章发表于二〇二六年四月十日，距离我们现在录这期节目已经过去了大概两个多月，时间不算长，行业里的判断和架构思路基本还是新鲜的，你可以放心地把它当作当下这个阶段最实用的一份参考手册来听。

先说说这篇文章为什么会出现。Anthropic 在更早一篇文章里讨论过一个问题，就是什么时候应该用多智能体系统，什么时候单个 agent 反而是更好的选择。这里我先简单解释一下 agent 这个词，它指的是能够自主调用工具、自主决策、完成多步骤任务的人工智能程序，中文社区里常翻译成智能体，我们这期节目里会直接说 agent，听起来更顺。那篇更早的文章解决的是要不要用多智能体的问题，而我们今天要讲的这篇，是写给那些已经做出决定，确定要用多智能体系统的团队看的，它要回答的是下一个问题，也是更难的问题，就是该选哪一种协作模式。

作者在文章开头就点出了一个挺扎心的现象，他们说自己见过很多团队选协作模式，标准不是这个模式适不适合手头的问题，而是这个模式听起来够不够高级，够不够炫。这其实是个很容易踩的坑，因为多智能体系统的设计空间很大，媒体和社区里经常会冒出各种听起来很厉害的架构名词，团队很容易被名词吸引，先选了一个复杂的架构，再去委屈地适配自己的业务，而不是反过来。所以作者给出的建议非常朴素，先从能解决问题的最简单模式开始用，观察这个模式在哪里开始吃力，再针对吃力的地方往更复杂的模式演化。这句话听起来简单，但我觉得它其实是这篇文章最重要的一条暗线，后面讲完五种模式之后我们还会回到这句话上。

文章总共讲了五种协作模式，我们一个一个来拆。

第一种叫生成者验证者模式，英文是 generator dash verifier。这是最简单的一种，也是目前落地最多的一种。它的结构是这样的，一个生成者智能体接到任务，产出一个初版结果，然后把这个结果交给一个验证者智能体去检查。验证者会判断这个结果是不是满足要求，如果满足就直接通过，如果不满足，就把具体哪里出了问题的反馈传回给生成者，生成者拿着这个反馈再生成一版。这个循环会一直进行，直到验证者点头通过，或者到达了预先设定的最大尝试次数。

文章举的例子是一个客服系统，用来自动生成回复客户工单的邮件。生成者会参考产品文档和工单上下文，先写出一版回复。验证者要做三件事，第一是去知识库里核对回复里的事实是否准确，第二是检查语气是否符合品牌调性，第三是确认回复有没有把工单里提到的每一个问题都接住了。如果某一项没通过，验证者给出的反馈一定是非常具体的，比如指出某个功能被错误地归到了某个定价档位，或者工单里提到的某个问题被漏答了，而不是笼统地说这条回复不够好。

这个模式适合输出质量要求很高，同时评判标准能被写清楚的场景。典型的例子包括代码生成，一个智能体写代码，另一个智能体写测试并运行测试，还包括事实核查、按照评分细则打分、合规性检查，总之就是那种一旦输出错了，代价会比多跑一轮生成要大得多的场景。

这个模式的风险点很直接，验证者的好坏完全取决于它拿到的评判标准。如果只告诉验证者去检查这个结果好不好，没有给出更具体的标准，验证者大概率会直接盖章通过，文章里用的词是橡皮图章，意思是它只是走个形式，没有真正起到质量把关的作用。作者说团队最常见的失败方式，就是搭好了生成者验证者这个循环的骨架，却没有真正定义清楚验证到底要验证什么，结果造成了一种质量把关已经存在的错觉，但其实内核是空的。

另外这个模式还假设生成和评判是两种可以分开的能力。如果评判一个创意方案的难度，跟生成这个方案本身的难度差不多高，验证者就未必能可靠地抓出问题，因为它本质上也没有比生成者更懂这件事。最后一个风险是循环可能卡住，如果生成者始终无法针对验证者的反馈做出改进，系统就会在原地震荡，反复生成又反复被打回，却始终没有收敛。解决办法是设置一个最大迭代次数，并且配一个兜底策略，比如到达上限就转人工处理，或者把目前最好的一版结果连同说明一起返回，而不是让它无限循环下去。

第二种模式叫协调者子智能体模式，英文是 orchestrator dash subagent。这个模式的特点是有明确的等级关系，一个主智能体充当团队负责人的角色，负责规划任务、把工作分发出去、再把各路结果汇总起来。子智能体负责具体的某一块职责，完成之后把结果汇报回去。

主智能体接到任务以后，会先想清楚整体该怎么拆，它可能自己直接处理一部分子任务，同时把另一些子任务派发给子智能体去做。子智能体做完工作之后把结果返回，主智能体再把这些结果合成出最终输出。

文章特别提到，Claude Code 本身用的就是这种模式。主智能体会自己写代码、编辑文件、运行命令，但当它需要去搜索一个很大的代码库，或者需要调查几个互相独立的问题时，它会在后台派出子智能体去做这些事情，这样主线工作不会被打断，结果会陆续流回来。每个子智能体都在自己独立的上下文窗口里工作，只把提炼出来的结论带回来，这样主智能体的上下文就能一直聚焦在主线任务上，探索性的工作则在并行的子空间里进行，不会污染主线的注意力。这里我多解释一句上下文窗口是什么，它指的是模型在一次对话里能看到、能记住的信息容量上限，把探索性的工作放到独立的子智能体里，相当于给主线任务腾出更多干净的脑容量。

文章给的另一个例子是一个自动化代码评审系统。一个 pull request，也就是一次代码提交请求送进来以后，系统需要分别检查安全漏洞、测试覆盖率、代码风格，还要评估架构上是否一致。这四件事彼此独立，需要的上下文也不一样，而且每一项都有清晰的输出。主智能体把每一项检查分别派给专门的子智能体，收集回来的结果之后，再合成出一份统一的评审报告。

这个模式适合任务拆分本身就很清楚，而且子任务之间互相依赖很少的场景。主智能体能始终保持对整体目标的把握，子智能体则可以专注在各自负责的那一小块上。

但是这个模式有一个结构性的弱点，就是主智能体会变成信息的瓶颈。如果一个子智能体发现了某个跟另一个子智能体的工作相关的信息，这条信息必须先绕回主智能体，再由主智能体判断要不要转发出去。比如安全检查子智能体发现的身份验证漏洞，恰好影响架构评估子智能体的判断，主智能体就必须意识到这两件事有关联，主动把信息路由过去。可一旦经过几次这样的转手，关键的细节常常会在转述和摘要的过程中被悄悄丢掉。另外子智能体大多是顺序执行的，除非你专门做了并行化处理，否则系统会承担多智能体带来的全部 token 成本，却拿不到并行带来的速度好处，这里 token 是模型处理文本时按片段计费的计量单位，用得越多花费越高。

第三种模式叫智能体团队，英文是 agent teams。这个模式里，一个协调者会把多个工作者智能体当作独立的进程派出去，这些工作者从一个共享的任务队列里领取任务，自主地、跨越多个步骤地完成工作，完成以后发出信号表示自己做完了。

这个模式和上面那个协调者子智能体模式最大的区别，在于工作者的持续性。在协调者子智能体模式里，主智能体针对一个有边界的子任务派出一个子智能体，这个子智能体做完就结束了，下一次需要又会重新派一个全新的、没有记忆的子智能体出去。而智能体团队模式里的工作者是持续在线的，会在很多次任务分配之间一直存活，随着接的任务越来越多，它会积累出对自己这块领域的上下文理解和专业积累，表现也会随之变好。协调者只负责分配任务和收集结果，不会在两次任务之间把工作者重置归零。

文章举的例子是把一个大型代码库，从一个技术框架迁移到另一个技术框架。每一个服务都可以由一个工作者独立完成迁移，各自有自己的依赖关系、测试套件和部署配置。协调者把每个服务分配给一个工作者，每个工作者自主地走完整个迁移流程，包括更新依赖、修改代码、修复测试、做验证，中间不需要协调者介入细节。协调者最后收集所有迁移完成的服务，再跑一次全系统的集成测试。

这个模式适合子任务彼此独立，同时又能从持续的多步骤投入里获益的场景。每个工作者会逐渐建立起对自己那部分领域的理解，而不是每次被派活都从零开始。

它的风险点也很明确，独立性是这个模式能成立的前提条件。协调者子智能体模式里主智能体还能在中间斡旋，帮忙传递信息，但智能体团队里的工作者是真正自主运作的，彼此没办法轻易共享中间发现。如果某一个工作者的工作其实会影响到另一个工作者，而两边都不知道这件事，它们各自产出的结果就可能互相冲突。完成情况的判断也更难处理，因为工作者是自主工作、耗时不固定的，可能有的工作者两分钟就做完了，有的要花二十分钟，协调者必须能处理这种参差不齐的完成节奏。如果多个工作者还共享同一份资源，比如同一个代码库、同一个数据库或者同一套文件系统，问题会被放大，两个工作者很可能去改同一个文件，或者做出互相不兼容的改动。所以这个模式要求你在任务拆分和冲突处理机制上下足功夫，不能想当然地认为分配出去就万事大吉。

第四种模式叫消息总线，英文是 message bus。当智能体数量变多，彼此之间的交互方式也变复杂之后，靠人工去设计每一对智能体之间该怎么直接对话，会变得越来越难管理。消息总线引入了一层共享的通信层，智能体通过发布和订阅这两个动作来互动。

智能体会订阅自己关心的主题，有一个路由器负责把匹配的消息送到订阅了对应主题的智能体那里去。这个设计的好处是，当你有了新的能力、想加一个新的智能体进来时，它只要订阅相关主题就能立刻开始接收相关工作，完全不需要去改动已有智能体之间的连接关系。

文章用一个安全运营自动化系统来说明这个模式的优势。安全告警从多个来源涌进来，分类智能体判断每条告警的严重程度和类型，把高严重度的网络类告警转给网络调查智能体，把身份凭证相关的告警转给身份分析智能体。调查智能体可能发布信息补充请求，由上下文收集智能体去满足。最终的调查发现流向负责协调响应措施的智能体，由它决定该采取什么行动。

这个流水线很适合用消息总线来做，因为事件是一站接一站往下流动的，团队可以随着威胁类型的演变，不断加入新的智能体类型，而且不同的智能体团队可以各自独立开发和部署自己负责的那一段，互相不需要紧耦合。这个模式适合那种工作流程本身是由事件驱动、而不是按照固定顺序走的场景，尤其是当你预期这个智能体生态会持续长大的时候。

它的代价主要在可追溯性上。事件驱动带来灵活性的同时也让追踪变难，一条告警触发横跨五个智能体的一连串事件，想搞清楚发生了什么，就需要细致的日志记录和事件关联，排查难度比顺着协调者从头到尾的决策链路走要高得多。路由的准确性也是关键，如果路由器把某条消息分错了类，或者干脆丢掉了，系统会表现成一种很隐蔽的失败，它不会报错也不会崩溃，只是悄无声息地什么都没处理。如果用大语言模型来做这个路由判断，会获得语义层面的灵活性，但也会带来它自己的失败模式，比如理解错了意图导致路由判断出错。

第五种模式叫共享状态，英文是 shared state。前面几种模式，无论是主智能体、团队负责人，还是消息路由器，本质上都是有一个中心节点在统一管理信息怎么流动。共享状态这个模式则是把这个中间人去掉，让智能体直接通过一个所有人都能读也能写的持久化存储来互相协作。

在这个模式下，智能体是完全自主运作的，直接读写一个共享的数据库、文件系统或者文档。没有中心协调者存在。每个智能体会去这个存储里查看有没有跟自己相关的信息，根据看到的内容采取行动，再把自己的发现写回去。工作的起点通常是一个初始化步骤，往存储里放进一个问题或者一份数据集，结束的条件可能是到了时间上限，也可能是某种收敛阈值被触发，或者是某个专门被指定的智能体判断存储里已经积累出了足够好的答案。

文章举的例子是一个研究综合分析系统，多个智能体分别去调查一个复杂问题的不同侧面，一个去查学术文献，一个去分析行业报告，一个去查专利申请记录，还有一个去追踪新闻报道。每个智能体的发现都可能会启发其他智能体接下来的调查方向，文章里给的例子是，做学术文献调查的智能体可能会发现某个关键研究者，而这个研究者背后的公司，恰好是做行业分析的智能体应该重点去查的对象。

在共享状态模式下，所有发现是直接进入共享存储的，做行业分析的智能体可以立刻看到学术文献智能体的发现，不需要等一个协调者把这条信息路由过来。智能体在彼此的工作基础上持续叠加，共享存储本身会变成一个持续演化的知识库。共享状态还有一个额外的好处，就是去掉了协调者这个单点故障。如果任意一个智能体停掉了，其他智能体照样可以继续读写存储，系统不会被卡死。而在协调者模式或者消息总线模式里，一旦协调者或者路由器本身出问题，整个系统就会陷入停滞。

这个模式的风险也很现实。因为没有显式的协调，智能体之间很可能重复做同一件事，或者各自朝着互相矛盾的方向去推进。两个智能体完全可能各自独立地去调查同一条线索，谁也不知道对方也在做。系统呈现出来的行为是众多智能体互动之后涌现出来的结果，而不是自上而下设计出来的，这就让整体结果变得更难预测。

文章特别强调了一种更难处理的失败模式，叫反应性循环。举个例子，智能体甲写下一个发现，智能体乙读到之后写了一条后续回应，智能体甲看到这条回应之后又做出反应，如此反复。系统会不停消耗 token，却没有真正朝着收敛的方向前进。作者说，重复劳动和并发写入冲突这两个问题，工程上有现成解法，比如加锁、版本管理、任务分区。但反应性循环是行为层面的问题，需要从设计之初就内建终止条件，可以是一个时间预算，可以是一个收敛阈值，比如连续若干轮都没有新发现就停下来，也可以指定某一个智能体专门负责判断存储里的内容是不是已经足够给出答案了。那些把终止条件当成事后才补的系统，往往会陷入两种结局，一种是无限循环下去，另一种是某个智能体的上下文窗口被撑满了就草草收场，这两种结局都不是真正意义上的完成。

讲完五种模式各自的样子，文章接下来回答更实际的问题，到底该怎么在它们之间做选择。作者提出一个总原则，他们在更早那篇关于多智能体的文章里提倡过一种以上下文为中心的任务拆分思路，意思是拆任务的时候，应该按照每个智能体需要看到什么样的上下文信息来拆，而不是按照工作的类型来拆。这条原则在选协作模式的时候同样适用，因为这五种模式的本质区别，其实就是它们各自用什么方式去管理上下文边界、管理信息怎么流动。

文章给了几组两两对比，我觉得这是整篇文章里最值得记下来的部分。

第一组对比是协调者子智能体模式和智能体团队模式，这两种都是中心节点把工作派发给别人去做，区别在于工作者需要维持上下文多长时间。子任务短、聚焦、产出明确，选协调者子智能体模式，代码评审系统就是这样，每项检查在一次有边界的调用里跑完分析就返回，不需要跨多轮保留状态。子任务需要持续多步骤的投入才能做好，选智能体团队模式，代码库迁移就是这样，每个工作者会逐渐熟悉自己负责的服务，这种积累出来的熟悉程度，是一次性派活完全做不到的。

第二组对比是协调者子智能体模式和消息总线模式，区别在于工作流程本身有多大的可预测性。步骤顺序提前就能确定，选协调者子智能体模式，代码评审系统走的是收到提交、跑检查、汇总结果这条固定流水线。工作流程是随事件涌现出来的，可能因为发现了什么而变化，选消息总线，安全运营系统就是这样，没人能提前预知会来什么类型的告警，也没人能预知一条调查路径会牵出什么后续动作。文章里有句话我印象很深，他们说当协调者里累积的条件判断逐渐多到要应付各种各样的情况时，消息总线能把这种路由逻辑明确地表达出来，而且更容易扩展。

第三组对比是智能体团队模式和共享状态模式，区别在于智能体之间是不是需要看到彼此的发现。各自在互不相关的分区里工作、彼此不需要互动，选智能体团队，代码库迁移就是这样，每个工作者管自己的服务，协调者只在最后把结果合并起来。工作本质上是协作性的，发现应该实时流动起来，选共享状态，前面说的研究综合系统更适合这种安排，学术调查智能体一旦发现关键研究者，这个信息立刻就对行业分析智能体有用，不需要等。一旦工作者之间需要的不只是交个结果，而是彼此沟通，共享状态会让这种沟通更自然。

第四组对比是消息总线模式和共享状态模式，区别在于工作究竟是以离散的事件形式流动，还是不断累积成一个共享的知识库。在一个流水线里对事件做出反应、一个事件触发下一个事件，选消息总线，安全运营系统就是按阶段处理告警的，优势是能把事件精准路由给有能力处理的智能体。在长期积累的发现基础上持续工作，选共享状态，研究综合系统就是不断往知识库里添砖加瓦。还有一点值得注意，消息总线终究还有一个路由器存在，意味着还是有中心组件在决定事件该去哪，而共享状态是真正去中心化的，如果你最在意消除单点故障，共享状态做得更彻底。文章里还补了一句很实用的判断标准，如果你发现消息总线里的智能体发布消息，目的其实是分享发现而不是触发某个动作，那很可能说明共享状态才是更合适的模式。

文章最后提醒，实际生产系统里经常是好几种模式混在一起用的，常见的组合是用协调者子智能体模式来管理整体工作流，再在某个需要密集协作的子任务上嵌入共享状态。另一种常见组合是用消息总线来做事件路由，同时让负责处理每种事件类型的工作者用智能体团队的方式持续运作。作者的原话是，这些模式是搭积木用的基本单元，不是互相排斥、只能选一个的选项。

最后作者给了一条总建议，对绝大多数场景，他们推荐从协调者子智能体模式开始入手，理由是这个模式能覆盖最广的问题类型，同时协调成本最低。等用起来之后，观察它具体在哪里开始吃力，再有针对性地往其他模式上演化。

接下来我想结合咱们国内做 agent 产品和应用的工程师实际会遇到的场景，说说这套框架具体能怎么用，这部分是我个人的一些延伸思考，跟原文的观点要分开看。

如果你正在做的是内部工具，比如代码评审机器人或者客服助手，我个人认为默认起点就该是协调者子智能体模式，先别急着上更复杂的架构。大多数内部工具的任务拆分天然就是清楚的，无非是几个互相独立的检查项或者信息来源，协调成本最低，出问题也最容易排查，先把这个跑顺，比一上来就堆叠多层架构划算得多。

如果场景是输出错一次代价就很大，比如生成对外发布的合规文档，或者生成要直接合并进主干的代码，我会建议在协调者子智能体模式之外，再叠加一层生成者验证者模式作为质量关卡。这里最容易被忽略、却是成败关键的一步，是把验证标准写清楚，而不是简单地让验证者去判断这个东西好不好。我自己的经验是，验证标准能拆成一条一条具体可检查的规则，这个组合就会工作得很好，标准还停留在感觉层面，基本就会退化成摆设。

如果你面对的是一次性的大规模迁移类工作，比如把整套微服务体系从旧框架搬到新框架，或者把大批历史数据迁移到新系统，我会建议考虑智能体团队模式，但一定要提前想清楚共享资源的冲突怎么处理，有没有现成的加锁或者版本控制机制能兜底，这件事不提前规划，后期排查冲突会非常痛苦。

如果你做的是告警类型会不断变化、不断长出新种类的场景，比如安全运营、风控、运维监控，我会建议优先考虑消息总线模式，但要把日志和事件关联能力当成头等大事来投入，因为这套架构最大的隐性成本不在搭建阶段，而在出问题之后排查的那一刻。

如果你做的是需要多个专业方向的智能体互相启发、共同把一个复杂问题搞清楚的场景，比如行业研究、竞品分析、尽职调查，我会建议尝试共享状态模式，但千万不要把终止条件当成事后补丁，一定要在设计阶段就想清楚什么时候算完成，是设一个时间预算，还是设一个连续多少轮没有新发现就收尾的阈值，还是专门指定一个智能体来做最终判断。

总的来说，我觉得这篇文章最有价值的地方，不是介绍了五个听起来很专业的名词，而是给出了一套可以拿来直接用的提问清单，帮你判断自己的场景到底落在哪一种模式上。归根结底就是几个问题，子智能体需要不需要跨多轮保留状态，工作流程能不能提前预知，智能体之间需不需要看到彼此的发现，工作是离散事件还是持续累积的知识。把这几个问题想清楚，选择自然就出来了，而不是先挑一个听起来厉害的架构，再去说服自己业务需要它。

这篇文章就讲到这里，核心就是一句话，先用最简单的协调者子智能体模式把事情跑起来，观察它在哪里开始吃力，再针对吃力的地方，有选择地引入生成者验证者、智能体团队、消息总线或者共享状态。我个人认为，这套从简单到复杂、按需演化的思路，比起一上来就追求架构完备，更适合大多数团队真实的资源和时间约束。我们下期节目再见。
