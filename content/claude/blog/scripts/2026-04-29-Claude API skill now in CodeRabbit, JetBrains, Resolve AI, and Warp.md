## Shownotes

Anthropic 把那些老司机才懂的 Claude API 使用经验，打包成了一个叫 claude-api 的技能，从三月份的 Claude Code 一路铺到了 CodeRabbit、JetBrains、Resolve AI 和 Warp。无论你在哪个工具里写调用 Claude 的代码，这份关于智能体设计、模型参数和提示缓存的专业知识都能随时被唤起，而且会跟着官方 SDK 自动保持更新。这一期我们聊聊这个技能具体装了什么、怎么用，对工程师和做开发者工具的团队分别意味着什么。

- claude-api 技能三月先在 Claude Code 上线，四月底扩展到 CodeRabbit、JetBrains、Junie、Resolve AI 和 Warp
- 技能内置三类专业知识：agent 设计套路、模型代际参数差异、提示缓存使用规则
- 跟着 Anthropic SDK 自动更新，新模型新功能发布即同步，不会像文档一样过时
- 四句可以直接对 Claude 说的话：提升缓存命中率、加上下文压缩、升级到新模型、搭建深度研究 agent
- Claude Code 里已经封装成斜杠命令：claude-api migrate 和 claude-api managed-agents-onboard
- 技能本身开源在 anthropics 的 skills 仓库，任何编程类智能体产品都可以接入

我自己设想过这样一个场景，如果你是一个后端工程师，老板突然丢给你一个需求，说咱们这个产品也要加一个 AI 客服功能，接 Claude 的 API，一周之内上线。你可能很快就能跑通一个最基础的版本，调用接口，传一段提示词，拿到回复，看起来没问题。但是真正上线之后，账单可能会比你预想的高出不少，因为系统提示词每次都在重复付费，没有用上缓存。或者过了两个月，团队决定升级到新模型，你才发现当初写的代码里，有些参数名字、有些用法，已经是上一代的写法了，改起来又要重新翻一遍文档。这些细节，单靠一次性学习是很难一次到位的，而这恰恰就是这篇文章想要解决的问题。

我们先说说背景。今天我们听到 Claude 的技能这个词，可能会觉得有点陌生，这里简单解释一下。Claude 的 Skills，你可以把它想象成给 Claude 配的一个专业工具包。这个工具包里装的不是代码，而是一份份说明文档，还有一些辅助脚本，专门针对某一类任务，比如说怎么做表格，怎么操作幻灯片，或者像今天这期讲的，怎么写好调用 Claude API 的代码。Claude 在判断当前任务跟某个技能匹配的时候，会自动把这份说明书读进来，按照里面写的最佳实践去执行，而不需要你每次都在对话里，把所有的注意事项重新讲一遍。这个机制如果用一个我们工程师更熟悉的东西来类比，有点像团队里那种写得特别细致的代码规范文档，加上一个会自动检查、自动按规范改代码的 linter，只不过它检查的不是缩进和命名，而是你调用大模型接口的方式对不对。

那么这次的主角，claude-api 这个技能，它最早是在三月份，也就是二〇二六年三月，首先出现在 Claude Code 里的。Claude Code 是 Anthropic 自己的命令行编程工具，很多重度用户已经习惯了在终端里直接跟 Claude 对话写代码。claude-api 技能上线之后，凡是你在 Claude Code 里写涉及调用 Claude API 的代码，Claude 都会自动调用这份技能，帮你避开一些常见的坑。而到了这篇文章发表的四月底，这个技能的覆盖范围，从 Claude Code 一家扩大到了四个新的开发者工具，CodeRabbit、JetBrains、Resolve AI，还有 Warp。

这里为什么是个值得讲的事情。我们退一步想，在没有这个技能之前，一个开发者想把 Claude API 用好，靠的是什么。无非是三条路，一是啃官方文档，文档写得再好，信息也是分散的，你得自己拼起来。二是踩坑，上线了发现缓存没生效，账单超预算了才回头查资料。三是问社区，问同事，但每个人的经验都是零散的，而且过时得很快，模型一升级，旧经验可能就不准了。claude-api 技能想要解决的，就是把这条经验积累的曲线，从工程师个人身上，挪到 Claude 自己身上，而且让这份经验本身能跟着 Anthropic 的 SDK 一起更新，不会过时。

我们具体看看这个技能里装的是什么。文章里提到了三类核心知识。第一类，是 agent 模式的选择，英文是 agent pattern，中文我们可以理解成智能体的设计套路。简单说，你想做一个能自主完成多步骤任务的 Claude 应用，实现方式有很多种，有的场景适合简单的一问一答循环，有的场景需要工具调用加反思，有的场景需要并行跑多个子任务再汇总。举个例子，假如你要做一个回答产品常见问题的客服机器人，大概率一个简单的单轮或者最多带几轮记忆的循环就够用了，没必要上多智能体编排那么复杂的架构，因为问题本身不需要拆解成多个子任务。但如果你要做的是一个能自动读代码库、定位 bug、写补丁、再跑测试验证的编程智能体，那就必须要有工具调用、阶段性反思、出错重试这些机制，简单的一问一答循环根本扛不住这种多步骤的任务。选错了套路，代码写起来又绕又难维护，跑起来效果也打折扣。claude-api 技能里就沉淀了什么样的任务该用什么样的套路这部分判断力，Claude 在帮你写代码之前，会先判断你的场景属于哪一类，再给你对应的实现方式，而不是无脑套用一个万能模板。

第二类知识，是模型代际之间参数的变化。这一点工程师应该最有体感。Claude 模型从三点五，到四，到四点五，再到现在的四点六，每一代之间，接口的参数都会有细微调整，有的参数改名了，有的参数加了新的取值范围，有的老参数被新的机制取代了，比如说控制模型思考强度的参数，在不同代际模型上的取值范围和默认行为可能就不一样。如果你的代码是照着旧版本文档写的，直接套到新模型上，轻则报错，重则悄悄地效果变差，你都不一定能立刻发现，因为接口没报错，只是输出质量打了折扣，这种坑反而是最隐蔽的。claude-api 技能记住了这些代际差异，相当于一个一直跟着 Anthropic 内部同步的版本说明书，你不需要自己去翻每一代的更新日志，逐条对比。

第三类知识，是什么时候该用提示缓存，英文叫 prompt caching。这个我们多展开说几句，因为这是我个人觉得这篇文章里最实在的一个点。提示缓存，简单理解就是，如果你每次调用 Claude API 的时候，前面有一大段内容是重复的，比如说一份很长的系统提示词，或者一份你反复引用的产品文档，Claude 这边可以把这部分内容缓存下来，下一次你再传同样的内容过来，就不用重新计费、重新计算，直接复用缓存结果，这样既省钱又省时间。这个机制其实有点像你点外卖的时候，平台记住了你的常用地址，你不用每次都把小区名字、楼号、门牌号重新打一遍，只要地址没变，直接复用就行，一旦你搬家了，地址变了，那这份缓存自然就失效了，要重新填一遍。听起来很简单，但是真要把缓存命中率做高，里面有不少讲究，比如说内容要放在请求的固定位置，内容的顺序不能随便变，缓存是有时效性的，过期了就要重新计算。这些规则，原文说很多开发者其实并不清楚，或者知道规则但是写代码的时候没有照着做，结果就是花了大价钱却没拿到该有的折扣。claude-api 技能里，就把这些容易被忽略的缓存规则给沉淀下来了，所以你只要在带着这个技能的工具里问一句，帮我提升一下缓存命中率，Claude 就会按照这些规则去检查你的代码，该挪位置的挪位置，该固定顺序的固定顺序，把能改的地方改掉。

第二个值得讲透的点，是这个技能自己会更新这件事。原文里有一句话我觉得说得挺到位，大意是，只要 Anthropic 的 SDK 在变，这个技能就会跟着变，新模型发布了，接口加了新功能，Claude 自己就已经知道了。这句话听起来简单，但它背后解决的是一个长期存在的痛点，文档和代码示例的保鲜期问题。

我们设想一下，如果没有这种自动更新机制，会是什么样子。比如说网上有很多教程，博客文章，问答帖子，是针对某个旧版本模型写的。新模型出来之后，这些内容大概率没人会去更新，于是它们就变成了知识垃圾，还在被搜索引擎索引，还在被新人看到，新人不知道这是过时的内容，直接照搬，结果就是踩坑。这种知识衰退的问题，在大模型这个发展速度极快的行业里特别明显，因为模型迭代周期短，可能几个月就有新版本，每次升级，接口、参数、推荐用法都可能跟着变。这跟我们以前学一门编程语言不太一样，一门语言的语法可能十年都不会大改，但是大模型接口这两年几乎是隔几个月就有一次值得关注的变化。

claude-api 这个技能的解法，是把这部分应该怎么用的知识，跟 Anthropic 自己的 SDK 发布流程绑在一起，而不是依赖外部的文档作者或者教程作者去手动维护。这有点像订阅一份会自动更新的活页资料，而不是买一本印好了就不会再变的书，书一旦印出来，内容就定型了，而活页资料可以随时抽掉过时的那一页，换上最新的一页。这意味着什么呢，意味着你不需要去判断你看到的某篇教程是不是过时的，你只需要在支持这个技能的工具里，直接问 Claude，而 Claude 给你的答案，默认就是跟当前最新的 SDK 版本对齐的。这种知识跟着源头自动更新的设计思路，我个人认为是这篇文章里最值得工程师注意的一点，因为它换了一种解决文档过时问题的思路，不是靠人去维护文档，而是靠机制去保证知识的新鲜度。

第三个我们要讲的点，是文章里给出的四个具体的对话示例，这部分非常实用，我建议你如果在用 Claude Code，或者刚好在用 JetBrains、Warp 这些工具，可以直接拿这几句话去试。

第一句，提升我的缓存命中率。前面已经讲过，这一句会触发 Claude 去检查你的代码里，提示缓存的应用是不是符合规则，该调整位置的调整位置，该补全的配置补全。

第二句，给我的 agent 加上上下文压缩，英文是 context compaction。这里稍微解释一下背景，如果你的 Claude 应用是那种需要长时间运行、多轮对话的智能体，比如说一个需要反复调用工具、反复读取文件的编程助手，随着对话轮数增加，上下文会越堆越长，长到一定程度，一是费用上升，二是模型处理起来效率下降，甚至可能影响输出质量。举个具体场景，假设你的智能体正在帮你做一次跨五十个文件的大重构，跑到第三十个文件的时候，前面二十几个文件改了什么、为什么这么改的细节，可能早就把上下文塞满了。上下文压缩，就是在合适的时机，把历史对话压缩、提炼成更精简的摘要，保留关键决策和关键信息，丢掉冗余细节，比如说把前面二十个文件的具体改动过程，压缩成一句这些文件已经按照某个规则改完，具体原因是什么，从而让长时间运行的智能体能够持续工作下去，而不会被越堆越长的上下文拖垂。原文说，这个技能会带你过一遍 Anthropic 文档里关于压缩原语和 agent 设计模式的内容，相当于把原本你需要自己去啃文档才能搞懂的这部分，变成了对话式的引导。

第三句，帮我升级到最新的 Claude 模型。这一句解决的是模型迁移的问题。文章里举了个例子，如果未来出了一个新模型，比如说 Opus 四点七，Claude 会帮你审查现有代码，带你过一遍需要改的地方，模型名字怎么改，提示词哪里需要调整，还有一个叫努力程度的参数设置，英文是 effort，你可以理解成你让模型在多大程度上深思熟虑，这个参数在不同代际模型之间往往也需要重新校准。值得一提的是，如果你是在 Claude Code 里用这个技能，这个升级流程已经做成了一条命令，你直接打斜杠加 claude-api 加 migrate，就可以直接跑这个迁移流程，不需要你一句一句去问。

这里我要补充一个后续进展。文章发表的时候是四月底，举的例子里 Opus 四点七还是一个未来时的假设场景。但是从我们现在，六月二十号往回看，Opus 四点七实际上已经发布了，而且紧接着 Opus 四点八也已经上线，我们节目之前也提到过不少团队拿这两个新模型办的黑客松。也就是说，这篇文章里讲的模型升级靠技能辅助这个场景，在过去两个月里已经是实际发生过的事情了，而不只是一个预告。这也算是给这篇文章的一个小小的时效性补充。

第四句，帮我针对我所在的行业，搭建一个深度研究 agent。这一句背后对应的是 Claude Managed Agents，可以理解成 Anthropic 提供的一种托管式智能体服务，你不需要自己从零搭建一整套长时间运行、自主调研的系统架构，而是通过配置的方式，把长时间运行的深度研究这个能力直接用起来。原文的原话是，这样配置下来，长时间运行的研究工作，变成了几句对话就能搞定的事情，而不是一个需要专门立项的工程项目。同样，在 Claude Code 里，这个流程也封装成了一条命令，斜杠加 claude-api 加 managed-agents-onboard。这里也插一句后续的进展，Claude Managed Agents 这条产品线本身这两个月也在持续往前走，后来还加上了按计划定时跑任务，以及把环境变量安全地存放在专门的密钥库里这些能力，如果你对这条产品线感兴趣，可以留意一下我们之前那期专门讲 Managed Agents 新功能的节目。

第四个点，我们说说这个技能是怎么长到四个新工具里的，这其实是这篇文章标题里最直接的信息，但背后的逻辑值得说一说。

CodeRabbit，是一个专门做 AI 代码评审的工具，很多团队会用它在 Pull Request 阶段自动做代码审查，如果你的 PR 里改了调用 Claude API 的代码，它现在就能用上这份技能里的专业知识来给你提建议。JetBrains，大家应该比较熟，是做 IntelliJ 系列 IDE 的公司，这次连同他们自己的 AI 编程助手 Junie 一起，把这个技能也用了起来，这意味着你在 IDE 里写代码的时候，补全和建议也会带上这部分专业知识。Resolve AI，做的是面向运维和事故响应场景的智能体产品，这类产品本身经常要调用大模型去分析日志、定位故障，用上这个技能，等于是把自己的核心能力打得更扎实。Warp，是一个主打 AI 能力的终端工具，很多开发者已经把它当成日常的命令行替代品，在终端里直接写代码、跑命令的时候，也能享受到这份专业知识的加持。我们可以想象一个具体场景，如果你在 CodeRabbit 上提交了一个 PR，里面新加了一段调用 Claude API 写日志摘要的代码，但是系统提示词每次都用不同的顺序拼接，导致缓存完全用不上，CodeRabbit 现在就可以直接在 PR 评论里指出来，告诉你应该把固定不变的部分放在前面，把每次都变的部分放在后面，这样才能命中缓存，而不是等你自己上线之后看账单才发现问题。这四家公司虽然产品形态完全不一样，一个做评审，一个做 IDE，一个做运维，一个做终端，但是它们有一个共同点，它们的产品里都有一个环节，需要生成或者建议怎么调用 Claude API 的代码。

这里有个问题值得我们多想一层，这份专业知识为什么不是直接训练进模型本身，而是要单独打包成一个技能。我个人的理解是，模型本身学到的是语言能力和推理能力，相对稳定，但是 API 怎么调用，参数怎么配，这些是接口层面的工程细节，会随着版本发布频繁变化，如果把这些细节也固化进模型权重里，那每次接口调整，都要重新训练或者微调模型，成本太高，周期也太长。把这部分知识做成一个可以独立更新的技能，相当于把容易变化的部分，从相对稳定的模型本体里剥离出来，单独维护，这样接口一变，只需要更新技能这一层就行，不需要去动模型本身，这种分层的设计思路，我觉得是这篇文章背后没有直接说，但是细想一下挺合理的一个工程决策。

文章里专门用了一段，讲给做编程类智能体产品的开发者看的话，大意是，任何一个编程智能体，都可以把 claude-api 这个技能打包进自己的产品里，让自己的用户在写 Claude API 代码的时候，自动获得这部分专业知识。如果你正好在做这样的工具，这个技能本身是开源的，代码放在 anthropics 的 skills 仓库里。而且 Anthropic 提供了一份打包指南，原文说大概二十行 CI 配置就能搞定接入，接入之后，这个技能会自动跟着保持更新，你不需要自己再去维护这部分内容。

这个细节我觉得值得多说一句。这说明 Anthropic 在做这件事的时候，目标不只是把这个技能塞进自己的产品线，而是把它做成一种可以被整个生态复用的基础设施。换句话说，他们想要的效果是，不管你是通过哪个工具在写 Claude API 代码，CodeRabbit、JetBrains、Warp，或者你自己公司内部的某个工具，只要接了这个技能，你拿到的最佳实践都是一致的，而且都会自动保持新鲜。这跟单纯发一份官方文档，效果是完全不一样的，文档是被动的，你得自己去看，技能是主动的，它会在你写代码的当下，直接介入提供建议。如果我们把视野放得更远一点，这种把厂商自己的专业知识打包成可分发的标准件，而不是停留在文档层面的做法，我个人猜测以后可能会被更多模型厂商效仿，变成一种行业里默认的做法，当然这只是我自己的推测，文章本身并没有提到其他厂商会不会跟进。

说了这么多，我们落到实际工作上，这件事对你意味着什么，你具体能做什么。

第一种情况，如果你已经在用 Claude Code、CodeRabbit、JetBrains 或者 Junie、Resolve AI、Warp 这几个工具里的任何一个，好消息是，这个技能已经在里面了，你不需要单独安装，也不需要任何额外配置，直接用就行。下次你在写涉及 Claude API 调用的代码的时候，可以试着直接问一句，帮我看看这段代码的缓存命中率怎么样，或者，这里要不要加上下文压缩，Claude 会自动调用这份技能里的知识来回答你，而不是给你一个泛泛的、通用的回答。

举个更具体的例子，假设你问的是，帮我看看这段代码的缓存命中率怎么样，Claude 可能会告诉你，你的系统提示词里混了一段每次请求都会变化的时间戳，这段时间戳正好卡在缓存边界的前面，导致后面所有本该能复用的内容全部失效，只要把这段时间戳挪到请求的最后面，缓存命中率就能明显提升。这种具体到某一行代码、某一个位置的建议，如果不是有这份专业知识在背后撑着，光靠泛泛地说一句你应该用好缓存，是给不出来的。

第二种情况，如果你手头正好有一个用了旧版本 Claude 模型的项目，趁着这个机会，可以试试看在 Claude Code 里直接跑一下斜杠 claude-api migrate 这个命令，让 Claude 帮你梳理一下，从模型名字、参数设置，到提示词本身，有哪些地方需要跟着新模型调整。比起自己一行一行翻文档对照，这种方式效率要高得多，而且不容易漏掉细节。

第三种情况，如果你所在的团队，正好有需要长时间运行、自主调研性质的需求，比如说市场调研、竞品分析、行业报告这类工作，以前可能要专门拉一个项目，搭一套智能体架构来做，现在可以考虑先用 Claude Managed Agents 这条路试一试水温，看看是不是真的能把原本几周的项目周期，压缩成几句对话就能跑起来的轻量级方案。

第四种情况，如果你自己正好是在做编程类的智能体产品，无论是代码评审工具，还是 IDE 插件，还是其他形式的开发者工具，这篇文章里提到的开源技能和打包指南，值得你去看一看。把这个技能接进你自己的产品里，相当于借用 Anthropic 自己维护的专业知识，而不需要你自己摸索一套关于 Claude API 最佳实践的内容，还能省下后续持续维护这部分内容的成本。

我还想到一种情况，如果你是团队里负责带新人的工程师，以前新人接手一个调用 Claude API 的项目，往往要花不少时间消化这些零散的最佳实践，你可能要专门写一份内部文档，或者一遍一遍口头讲解。现在如果团队里用的工具本身就带着这个技能，新人遇到不确定的地方，可以直接问 Claude，省下的不只是新人自己摸索的时间，也省下了你作为带教人重复讲解的精力，这部分隐性的协作成本，我觉得是这篇文章没有直接讲，但是顺着逻辑可以推出来的一个好处。

这里我也想提一个需要注意的地方。这个技能解决的是怎么把 Claude API 用对这个层面的问题，它不会替你做架构决策，比如说你的产品到底要不要用智能体架构，要不要引入长时间运行的任务，这些更上层的产品判断，依然需要工程师和团队自己来做。技能更像是一个经验丰富的同事，在你已经决定了大方向之后，帮你把执行层面的细节做对，而不是帮你做决策本身。这一点我觉得是听众在实际用的时候要心里有数的，别指望它能替代架构设计这部分的思考。

我们简单总结一下今天这期的内容。这篇文章讲的是 Anthropic 把 claude-api 这个技能，从最早三月份在 Claude Code 里上线，扩展到了 CodeRabbit、JetBrains、Junie、Resolve AI 和 Warp 这几个开发者常用的工具里。这个技能里装的是关于智能体设计套路、模型代际参数差异、提示缓存规则这些容易被忽略，但是又实实在在影响代码质量和成本的专业知识，而且它会跟着 Anthropic 自己的 SDK 更新保持新鲜，不会过时。文章还给出了几句可以直接拿来用的对话，提升缓存命中率，加上下文压缩，升级到新模型，搭建深度研究智能体，如果你正好在用这几个工具，不妨现在就去试一试。

我个人的看法是，这篇文章本身篇幅不长，信息量却不小，它反映出的是 Anthropic 在做开发者生态这件事上的一种思路，不是单纯堆功能，而是把怎么把工具用好这件事，做成一个可以被整个生态系统复用、还能自动保鲜的能力。这种思路，我觉得值得我们这些做工程的人多留意，因为它预示着，以后判断一个开发者工具好不好用，可能不只是看它本身的功能强不强，还要看它有没有这种持续学习、持续同步上游变化的能力。另外多说一句我自己的观察，对于我们国内做大模型应用的团队来说，这种厂商把自己接口最佳实践打包成可以直接对话获取的技能，其实也提供了一个值得参考的产品思路，不管你接的是 Claude 还是其他模型，把容易踩坑、容易遗忘的接口使用细节，沉淀成一份可以被自己内部工具调用的知识包，而不是散落在各个人的脑子里或者某个不常更新的内部 wiki 里，长期来看应该都是值得投入的方向。这一点是我自己的延伸思考，文章原文并没有专门讨论国内团队的情况。
