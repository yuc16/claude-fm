## Script

今天我们要聊的这篇文章，发表于二〇二六年四月八日，来自 Anthropic 官方博客，英文标题是 Claude Managed Agents: get to production 10x faster，我把它翻译成中文，大致是"Claude 托管式 Agent：快十倍上线"。距今大概两个半月，在 AI 这个领域，两个半月已经足够发生非常多的变化了。但今天选这篇文章，是因为它说的不是某个具体模型的能力提升，而是一套针对 Agent 开发的基础设施能力。这类东西的生命周期相对更长，对实际工程工作的影响也更持久，所以今天花一期节目的时间好好讲清楚，非常值得。

我们先来建立一下感性认知，说说为什么这篇文章值得你认真听。如果你在工作里试图构建过任何一个正经的 AI Agent 系统，你大概率有过这样的体验：模型本身其实挺好使，核心的 prompt 设计也没什么大问题，但你花了三分之二甚至更多的时间，是在处理和 AI 本身几乎没什么关系的工程问题。安全沙箱怎么搭？Agent 跑到一半网络断了怎么恢复？工具调用的凭证怎么安全存储和轮换？每次模型升级了，整个 agent loop 要改多少地方？多个 Agent 之间怎么协调？权限控制怎么做才能满足合规要求？

这些问题每一个单独拿出来都不算特别难，但加在一起，就是实实在在的几个月的工程工作。而且这些工作做完之后，你还没给用户提供任何新价值——你只是把 Agent 搬到了生产环境而已。

Claude Managed Agents 说的，就是：这些活，我们帮你做了，你来专注用户体验就好了。

我们先来建立背景，理解一下在这篇文章出来之前，AI Agent 这个方向的行业状态是什么。

如果你回溯一下过去两三年 Agent 这个方向的演进轨迹，能看到一个很清晰的三段论。第一阶段，大概是二〇二三年到二〇二四年上半年，是概念探索期。大家都在聊"AI 能不能自主完成任务"，有各种令人兴奋的演示，GPT-4 写代码、Claude 操作浏览器、各种 Agent 框架涌现出来——LangChain、AutoGen、CrewAI，都是这个阶段的产物。这时候大家的重心是证明这个方向可行。

第二阶段，大概是二〇二四年下半年到二〇二五年，是工程摸索期。早期的概念验证已经做出来了，一批先行者开始试图把 Agent 推向生产环境，然后就撞上了那堵墙：基础设施。这个时候业界开始形成共识：Agent 开发最难的部分不是模型，是工程。这一阶段涌现了很多专注于 Agent 编排和基础设施的工具，但基本上都是开源社区自己拼凑的方案，缺乏统一的企业级支持。

第三阶段，也就是二〇二六年，大型 AI 公司开始直接下场提供托管基础设施。Claude Managed Agents 就是在这个节点出现的。这个节点很关键，它意味着 AI 公司已经不仅仅在提供模型 API，而是往更高的平台层走：我不只给你一个聪明的大脑，我给你一套完整的运行环境，你直接在上面构建应用。

理解了这个背景，我们再看文章的核心内容就会清晰很多。

文章把 Managed Agents 定义为"一套可组合的 API，用于大规模构建和部署云端托管 Agent"。核心的承诺是：你来定义 Agent 要做什么、能用什么工具、遵守什么规则，剩下的运行、调度、状态管理，Anthropic 来负责。内置的编排层会自动决定什么时候调用工具、如何管理上下文、以及如何从错误中恢复。

文章里列出了四个核心模块，我一个一个来拆开讲。

第一个模块叫 production-grade agents，生产级 Agent 运行时。这一块提供的是最基础的运行环境：安全的沙箱、身份认证机制、工具执行引擎，全部帮你处理好。听起来不是很性感，但我想特别展开说一下沙箱这件事，因为很多工程师会低估它的复杂性。

当你的 Agent 需要执行代码的时候，你首先要回答自己几个问题。第一，谁的代码？是你事先写好的固定代码，还是 Agent 自己生成的动态代码？如果是后者，那代码里可能包含任何内容，你的执行环境必须能安全地运行一段你完全不知道内容的代码，同时不让它影响到你的其他系统。第二，代码运行的网络权限是什么？可以访问互联网吗？可以访问你的内网吗？如果可以访问内网，范围限制是什么？第三，代码可以读写文件系统的哪些部分？第四，如果代码跑了太长时间，超时机制是什么？第五，出现执行错误时怎么清理执行上下文？

每一个问题背后都是实实在在的工程实现，而且还要考虑这些机制之间的相互影响。这不是一件很难的事，但它是一件很烦的事，而且做不好会留下安全隐患。文章里说的"安全沙箱执行"，就是把这些烦事全部帮你处理掉，你不需要自己去搭容器隔离、配网络策略、写执行超时逻辑。

第二个模块叫 long-running sessions，长会话支持。这个特性我认为是整个 Managed Agents 里最直接影响你能做什么的部分，因为任务运行时长直接决定了你能处理的任务类型。

我们来做个对比。如果一个 Agent 只能跑几十秒，那它能完成的任务范围是非常有限的：回答一个问题、总结一段文字、做一个简单的检索。这类任务直接调 API 就够了，不需要什么 Agent 框架。

但真正有商业价值的 Agent 任务，通常需要更长的时间。让 Agent 读一个有几万行代码的代码库、理解架构、制定修复方案然后提交 PR——这个任务跑完可能需要几十分钟甚至更长。让 Agent 处理一份包含几百页的法律合同、提取关键条款并生成合规报告——同样需要相当长的时间。让 Agent 做市场竞品分析，需要搜索、阅读、整理多个来源的信息——这种任务也很难在几分钟内完成。

Managed Agents 说的 long-running sessions 能"自主运行数小时"，而且进度和输出在整个过程中持续保存，即使网络连接中断也不影响。用户可以把任务交出去，关掉页面，过几个小时回来看结果。这个体验，如果你自己实现，需要搭的是：任务队列系统、进度追踪服务、结果存储层、以及推送通知机制，每一块都要自己搭。Managed Agents 把这些全部内置进去了。

你可能会问：长会话对网络断开真的有容错吗？文章里明确说了"persist even through disconnections"，也就是即使连接中断，进度和输出依然保存。这背后依赖的是检查点机制——Checkpointing。每隔一段时间或者在关键步骤完成时，系统自动保存当前状态，如果出现中断，任务可以从最近的检查点恢复继续跑，而不是从头重来。这个机制对于跑几个小时的任务来说非常重要，自己实现起来也相当有工程量。

第三个模块叫 multi-agent coordination，多 Agent 协作编排。这个功能目前还在 research preview 阶段，需要申请才能访问，但它代表的方向非常值得深入理解，因为这才是 Agent 能力真正爆发的地方。

简单说，就是你的主 Agent 可以动态地启动其他 Agent，让它们并行处理不同的子任务，然后把结果汇总。我给你一个具体的业务例子来建立体感。

假设你要构建一个尽职调查 Agent，投资机构用它来评估目标公司。输入是目标公司名称，输出是一份全面的评估报告，涵盖财务状况、市场竞争格局、法律合规风险、技术栈评估、创始团队背景。

如果你只有一个 Agent，它要串行完成所有这些工作：先查财务数据，再做市场分析，再看法律文件，再评估技术，再研究团队……整个过程可能需要两个小时。

但如果你有 multi-agent coordination，主 Agent 拿到任务后，立刻启动五个专项子 Agent：财务分析 Agent、市场研究 Agent、法律合规 Agent、技术评估 Agent、团队背景 Agent，五个并行跑。每个子 Agent 专注自己的领域，各自完成后把结果汇报给主 Agent，主 Agent 综合所有输入生成最终报告。整个时间可能从两个小时压缩到二十分钟。

这个并行化的思路在分布式系统里不是新鲜事，但放到 AI Agent 里，协调层的实现是个真实的难点。主 Agent 怎么知道子 Agent 完成了？怎么处理某个子 Agent 失败的情况？子 Agent 之间的输出格式怎么标准化以便主 Agent 能处理？这些协调逻辑，Managed Agents 说由内置的编排层来负责，你不需要自己写一套调度系统。

第四个模块叫 trusted governance，可信治理。这个模块在文章里说得比较简短，但对于企业用户来说，这往往是决定 Agent 能不能在生产环境里真正落地的关键因素。

具体包含三个子能力。第一是细粒度的权限管控。你的 Agent 调用外部系统时，你可以精确定义它有哪些权限：比如它可以读取特定数据库的特定几张表，但不能写入；它可以发送 Slack 消息到某些频道，但不能删除消息；它可以读取共享文档，但不能把文件分享给外部人员。这种最小权限原则，在合规敏感的行业——金融、法律、医疗——是必须的，自己实现需要对接每个外部系统的权限接口，还要维护一套统一的权限模型，工程量不小。

第二是身份管理。你的 Agent 以谁的身份在外部系统里操作？是用系统服务账号还是模拟特定用户的身份？如果 Agent 代替某个员工去发邮件、提交工单，这个员工的身份凭证怎么安全存储和使用？这些处理不好，既有安全风险，也有合规风险。

第三是执行链路追踪。文章里特别提到，在 Claude Console 里可以看到 Agent 的每一个工具调用、每一个决策点、每一个失败模式的完整记录。这对于实际工程工作非常有价值。调试 Agent 行为是出了名的困难——你很难像调试普通程序那样单步追踪一个大语言模型的推理过程。但如果你有完整的执行记录，你可以看到它做了什么、调用了哪个工具、工具返回了什么、它拿着这个结果做了什么决定，然后从这条链路倒推它为什么会产生某个不符合预期的输出。有了这个记录，排查问题的效率会提升非常多。

好，四个核心模块讲完，我们来重点讲一个在文章里有点低调，但我认为非常值得单独拿出来说的能力：自评估和迭代。

文章里说，Managed Agents 支持让 Claude 自己定义成功标准，然后自我评估输出质量、循环迭代，直到达到目标。这个功能目前也在 research preview 阶段。

我来解释一下这个能力的本质，以及它为什么重要。我们平时用 AI 的交互模式是这样的：你给一个 prompt，模型给一个输出，你评估输出的质量，如果不满意就改 prompt 再试。"评估"这个环节是在你这一侧的，是你在做判断，是人在做的。

自评估说的是，把这个判断环节也交给模型自己来做。你告诉它"我要一份按季度分类的财务摘要，要涵盖所有关键财务指标，格式要符合这个模板，准确率要达到九十五分以上"，Claude 生成一个版本，然后它自己对照你给的成功标准检查：格式对了吗？关键指标都涵盖了吗？有没有数字错误或者遗漏项？如果有问题，它会自己修正，再生成一个版本，再评估，直到满足标准为止。

文章里给出了一个实际测试数据：在结构化文件生成的任务上，这个自评估迭代模式和传统的单次 prompting 相比，任务成功率最高提升了十个百分点。而且文章特别强调，提升最大的是"最难的那些问题"。

这个"最难的问题提升最大"的规律，值得仔细想一想。我的理解是：对于简单任务，第一次生成通常就能满足要求，自评估带来的额外收益有限。但对于复杂任务，第一次生成往往有多处需要改进的地方，自评估能够系统性地找出这些问题并修正，每一轮迭代都带来可见的提升。这个效应在任务越复杂的时候越显著，也就是说你最需要它的时候，它效果最好。

从工程角度说，自评估循环本质上是在 agent loop 里加了一个"验证再迭代"的环节。你自己也可以实现类似的逻辑：生成之后调用另一个 prompt 来评估质量，如果评估不通过就重新生成。但难点在于评估的 prompt 怎么设计，如何让评估结果能有效指导下一次生成，迭代多少次是合适的，以及循环终止条件是什么。这些细节做好了才有效，Managed Agents 内置的实现是经过 Anthropic 内部充分调优的版本。

好，理论部分聊完，我们来看真实世界里已经在用 Managed Agents 的团队，以及他们具体是怎么用的。文章里提到了四个具体案例，每一个都值得细看，因为它们代表了不同的使用模式。

第一个是 Notion。Notion 让用户可以在自己的工作区里直接把任务委托给 Claude Agent，这个功能叫做 Notion Custom Agents，现在在 private alpha 阶段，也就是还没有对所有用户开放。

有意思的细节是，它的使用场景是分人群的：工程师用它来写代码上线，知识工作者用它来生成网站和演示文稿。多个任务可以并行运行，团队成员可以在 Agent 的输出上一起协作。

这个案例代表的是一种非常有价值的产品形态：AI Agent 不是作为一个单独的 AI 助手工具存在，而是直接嵌入在用户已经在用的工作空间里，以团队成员的方式参与协作流程。你不需要打开另一个工具、切换另一个界面，Agent 就在你本来工作的地方。这个方向，我个人认为会是未来企业 AI 应用里非常主流的一种形态——不是"我去打开 AI 助手来帮我做这件事"，而是"我的工作流本身就有 AI Agent 参与其中"。

第二个案例是 Rakuten，日本的大型电商和互联网集团，在中国可能相对不那么熟悉，但在日本是非常重要的科技公司。他们在产品、销售、市场和财务四个部门分别部署了专项 Agent，这些 Agent 接入了 Slack 和 Microsoft Teams，员工可以直接在即时通讯工具里把任务分配给 Agent，Agent 会完成工作后交付产出物——电子表格、演示文稿或者应用程序。

这里有一个数字特别值得关注：每一个专项 Agent，在一周内完成了部署。文章开头说自己搭基础设施需要几个月，这里的实际数字是一周。而且 Rakuten 不是只部署了一个 Agent，是横跨四个部门的多个专项 Agent，如果每个都要几个月，整个项目周期会拉得极长。

Rakuten 这个案例还有一个值得注意的产品决策：他们把 Agent 接入 Slack 和 Teams 这种员工已经在用的即时通讯工具，而不是做一个单独的 AI 界面。这是很聪明的选择，因为员工不需要改变工作习惯，就能多了一个 AI 同事。降低使用门槛，是让企业 AI 工具真正被用起来而不是闲置的关键因素。

第三个案例是 Asana 旗下的 Vibecode。Vibecode 是一个帮用户把自然语言描述直接变成部署好的应用程序的工具，他们把 Managed Agents 作为默认的技术基础集成进来，让用户启动同等能力的基础设施比以前快了至少十倍。

这个案例有点特别，因为它不是一个端用户在用 Managed Agents，而是一个 B 端产品把 Managed Agents 作为自己的底层基础设施，再给 C 端用户提供服务。这说明 Managed Agents 的使用场景是分层的：你可以直接用它来构建面向用户的 Agent 应用，也可以把它作为你自己 AI 产品的基础设施层，把构建和运行 Agent 的能力再开放给你的用户。对于做 AI 工具产品的团队来说，这个路径让你可以专注于自己的产品差异化，而不是把精力耗在底层运行时的搭建上。

第四个案例是 Sentry。如果你做过工程，对 Sentry 应该很熟悉，它是一个广泛使用的错误监控和性能追踪工具。Sentry 有一个叫 Seer 的 AI 调试 Agent，他们把 Seer 和一个 Claude 驱动的 Agent 配对，这个新 Agent 的职责是写修复补丁并提交 pull request。

整个工作流是这样的：Sentry 系统检测到一个 bug，Seer 负责诊断问题的根因，然后 Claude Agent 根据诊断写出修复代码，创建一个 PR。开发者收到通知，审核一下 PR 的内容，确认没问题就合并。从系统发现 bug 到开发者面前出现一个可以审核的修复方案，整个流程完全自动化，开发者只需要做最后的把关。

文章说这个集成在几周内完成，而不是原本预期的几个月。这对工程师来说有很强的体感：一个从 bug 检测到自动生成修复 PR 的完整 Agent pipeline，几周就能上线，背后正是因为 Managed Agents 处理了大量基础设施的复杂性。

把这四个案例放在一起看，你会发现一个规律：它们覆盖了非常不同的行业和业务场景，但都有一个共同的特征——Agent 在真实业务流程里完成具体任务，产出的是有实际业务价值的东西：代码、文档、报告、应用程序。这不是 AI 做简单的问答，而是 AI 在承担真正的工作负载。这正是 Managed Agents 所针对的场景，也是它能带来价值的地方。

好，我们来聊最实际的部分：如果你是中国的 AI 工程师，或者在考虑帮公司落地 Agent 项目，你应该怎么思考这件事，怎么判断什么场景适合用，要避什么坑。

首先说哪些场景最适合用 Managed Agents。我认为可以从三个维度来判断。

第一个维度是任务时长。如果你的 Agent 任务需要超过几分钟才能完成，Managed Agents 的长会话支持和检查点机制就非常有价值。如果你的任务本来就是毫秒级或者秒级的实时问答，那普通的 API 调用可能更直接，引入托管运行时反而增加了额外的层次。判断标准很简单：你的任务如果可以在一个 HTTP 请求的生命周期内完成，不需要 Managed Agents；如果不能，那就要认真考虑它了。

第二个维度是工具集成复杂度。你的 Agent 需要调用几种外部工具？每种工具的认证方式是否不同？如果 Agent 只是做文本生成，一个简单的 API 调用就够了。但如果它需要同时接入数据库、调用外部 API、执行代码、读写文件系统，那工具调用的安全性和可靠性就是个复杂问题，Managed Agents 帮你省掉这部分能节省大量时间。

第三个维度是治理要求。你们公司对 AI 系统的权限管控和审计有多高的要求？如果有合规需求，自己实现一个合格的治理层成本相当高。Managed Agents 内置的细粒度权限控制和执行追踪，可以帮你直接满足这些要求，不需要从头搭。

那有哪些场景现阶段可能不是最佳选择？我觉得有两个要注意。

第一个是对延迟极度敏感的实时场景。Managed Agents 是云端托管服务，中间有额外的调度和管理层，相比直接调 Claude API，会有一些额外的延迟开销。如果你的场景是用户在等待实时响应，每一毫秒都非常重要，那需要在性能和便利性之间权衡。

第二个是对数据主权要求很高的场景。如果你们公司的数据合规要求非常严格，数据不能出境、不能经过第三方托管服务，那使用任何云端托管服务都需要认真评估，Managed Agents 也不例外。这个问题不是 Managed Agents 特有的，而是所有云端 AI 服务都会遇到的问题，需要根据你们的具体合规要求来判断。

关于如何开始。文章里提到了两种接入方式：一种是通过 Claude Console 和文档直接接入；另一种是通过最新版的 Claude Code CLI。如果你现在在用 Claude Code，文章说可以直接告诉它"帮我开始 Managed Agents 的接入流程"，它会引导你完成整个 onboarding。对于想快速试用的工程师来说，这个入口非常方便。

关于定价。文章说的是 Claude Platform 的标准 token 费率，加上每个 session 小时零点零八美元的主动运行时费用。如果你的任务运行一个小时，就要多付大约八分美元。乍一看这个数字很小，但如果你有大量并发的长时间任务，这个运行时费用会积累起来。比如你同时跑一百个任务，每个任务跑两个小时，那就是一百乘以二乘以零点零八，十六美元的运行时费用，再加上 token 费用。在系统设计阶段就把这个成本结构算清楚，避免上线后出现意外的账单。

还有一点需要特别提醒：multi-agent coordination 和自评估迭代这两个功能目前还在 research preview 阶段，需要申请访问权限，不是所有人都能立刻用上。如果这两个功能对你的场景很关键，现在就去申请，等待审批的时间可以先把其他部分搭起来、做好准备工作。

有几条实践建议想给到你。第一，不要一上来就把最复杂、最关键的系统迁移过去。找一个相对独立、风险可控的场景先试水，感受一下工作流和调试体验，再根据实际效果决定下一步。第二，好好利用 Claude Console 里的执行追踪功能。养成习惯去看每一个工具调用和决策点的记录，你会对 Agent 的实际行为有更清晰的认识，排查问题的时候也会快很多。第三，在设计你的 Agent 的时候，尽量把任务拆解得清晰一些，给 Agent 明确的成功标准，而不是模糊的目标——这样不管是你自己评估输出质量，还是让 Claude 自评估，都会更容易判断。

最后来做个总结，说说我个人的一些看法。

Claude Managed Agents 解决的问题非常清晰：把 Agent 从实验室搬到生产环境的那段最痛苦的基础设施旅程，Anthropic 帮你走完了。从理论上把几个月的工程工作压缩到几天，这是它的核心价值主张，而 Rakuten 一周部署、Sentry 几周上线这些真实数据，也在支撑这个主张。

我个人觉得，这件事的意义不仅仅是"开发快十倍"。更深层的意义在于，它改变了 Agent 开发的准入门槛。以前，只有有足够工程资源来搞定基础设施的公司，才能把 Agent 真正跑在生产环境里。现在，一个小团队甚至一个独立开发者，也有机会快速把一个严肃的 Agent 系统上线，去验证它在真实用户面前是否有价值。这会加速整个 Agent 生态的发展速度，让更多有创意、有实际价值的应用出现。

当然，客观地说，把基础设施外包给任何供应商，都意味着一定程度的依赖和对底层控制权的让渡，在服务出现问题时你的排查手段也会受到限制。如何在开发速度和系统掌控权之间找到平衡，是每个团队需要根据自己的技术能力、业务需求和风险偏好来判断的问题，没有放之四海而皆准的答案。
