## Script

今天这期节目，我们来聊一篇来自 Anthropic 官方博客的文章，英文标题是 Running an AI-native engineering org，我把它翻译成"如何运营一个 AI 原生的工程团队"。这篇文章发表于二〇二六年六月三日，距今大概只过了两个多星期，是非常新鲜的第一手资料。

文章的作者是 Fiona Fung，她在 Anthropic 担任 Claude Code 和 Claude Cowork 这两款产品的工程总监。文章的内容，来自她在"Code w/ Claude SF 2026"大会上的演讲。我选这篇文章，是因为它非常难得——它不是技术论文，不是产品发布公告，而是一位真正在 AI 工程团队前线工作的领导者，讲她自己的团队在把 agentic coding 也就是"智能体辅助编程"当成默认工作方式之后，到底发生了哪些真实的变化。

大家知道，现在讲 AI 改变工程工作的文章和演讲，满天都是，但绝大多数停留在"AI 让我们更高效"这种层面。Fiona 这篇文章不一样，她讲的是组织结构、流程规范、招聘思路、团队治理这些更深层的东西。而且她讲的是自己团队的真实状态，不是假设，不是展望，是"我们现在就是这么做的"。这种第一手的转型经验，在中文内容里非常稀缺，所以今天我们要认认真真地把它拆开来讲。

我相信听节目的朋友，很多都在通勤路上，可能你自己在用 Cursor、Claude Code、Copilot 这类工具，也可能你在团队里推广 AI 辅助开发。不管哪种情况，Fiona 这篇分享里有很多东西可以直接拿来参考和对比。我们这就开始。

先说一个大背景，理解这个背景非常重要，因为 Fiona 整篇文章的逻辑都建立在这个基础之上。

她说，在过去很多年里，工程师的时间和精力，是开发软件最贵的那部分成本。围绕这个核心假设——"写代码很贵"——我们建立了整整两代软件工程方法论。

第一代是瀑布式开发。逻辑很简单：既然编码阶段很贵，那就在编码之前把所有需求想清楚，设计全部定好，然后一批一批交付，尽量减少返工。这套方法在制造业时代非常合理，因为它借鉴的就是制造业的生产管理思路。

第二代是敏捷开发。它的出发点也是一样的：写代码有成本，所以要把项目切成小迭代，快速验证方向是否正确，避免在错误路径上耗费太多工程资源。

Fiona 自己说，她职业生涯的起点是二〇〇〇年代初期在微软做 Visual Studio。那个时代软件是刻在光盘上卖的，有硬性的出厂截止日期，整个工程流程是围绕制造业节奏设计的。后来互联网出现了，软件可以在线发布，于是行业进化到持续集成、持续交付。每次这种进化，驱动力都是相同的：降低分发和发布的边际成本，然后调整流程来适应新的成本结构。

而现在，这个行业正在经历第三次范式转移。这一次改变的不是发布渠道，而是"写代码"这件事本身的成本结构。

Fiona 的原话非常直接：写代码、写测试、做重构，这些事情在他们团队里已经几乎不构成瓶颈了。但是——她特别强调——瓶颈并没有消失，而是移位了。现在卡住他们的，是验证代码的正确性、做代码审查，以及处理安全相关的问题。

这个观察，我觉得是整篇文章最核心的一个洞察，值得我们多想一想。用一个类比来理解：假设工厂里安装了一台超高速机器，原来一小时生产一百个零件，现在一小时生产一万个。但质检线还是原来那条，一小时只能检查两百个。于是库存开始积压，整个系统的吞吐量不是由生产速度决定的，而是由质检线决定的。瓶颈从生产转移到了质检。软件工程现在面临的，就是这样的结构性问题。

Fiona 说，她经常被其他工程领导者问到一个问题："你们怎么在代码审查上跟得上？"这个问题背后反映的，就是大家普遍感受到的那种张力：AI 让我们能非常快地生成大量代码，但验证这些代码是否正确、是否安全、是否符合预期的机制，并没有同步提升。

理解了这个背景，Fiona 后面分享的所有具体变化，才能看清楚它们背后的逻辑——它们都是在回应这个新的瓶颈。

我们来看第一个重要变化：规划节奏的根本性转变。

Fiona 用了一个很聪明的类比，叫做 JIT planning，也就是"即时规划"，借鉴的是编程里的 JIT compiling 概念，即"即时编译"。即时编译是指程序不在运行前把所有代码全部预编译好，而是在运行到哪里、需要的时候再编译。Fiona 把同样的逻辑引入了团队规划：不要提前把六个月的路线图全部规划清楚，而是在需要的时候，做恰好足够的规划。

这背后的理由很直接。她分享了一个亲身经历：她刚加入 Claude Code 团队的时候，团队写了一份质量不错的六个月路线图。但是，正因为他们在用 Claude Code 工作，工程速度大幅提升，产品方向随着用户反馈不断调整，结果到了第三个月，这份路线图就已经过时了。三个月，一半的时间，路线图就作废了。

你可能会问：提前规划不是应该更稳吗？Fiona 的回答是：在工程速度慢的年代，规划是用来"对冲执行成本"的。因为写代码很贵，所以你要先把问题想清楚再动手，避免走弯路。但现在执行成本大幅下降了，做一个原型的成本几乎可以忽略不计。在这种情况下，不确定性下的重度预规划，本身就成了一种风险。规划做得越细、越远，就越有可能在环境变化之后变得没有意义，而你在这份规划上投入的时间和精力，全部浪费掉了。

具体到日常工作里，Fiona 的团队把规划重心从设计文档，移到了 PR 里的讨论和实际的原型上。他们的流程是：先快速做出一个可以用的原型，让内部大量用户试用，然后基于真实的反馈来决定下一步。他们不做大型的产品评审会，因为那意味着大量时间花在了还不确定能不能成的东西上。

这个思路的转变，需要一整套文化支撑。团队要能容忍方向随时调整，领导层要接受路线图不是承诺而是猜测，而且不能把"偏离规划"当作失败的信号。这对很多大公司的团队来说，可能比技术上的挑战更难。

对于在听节目的朋友，我觉得这个洞察很实用。如果你的团队现在还在花很多时间写详细的技术方案文档，不妨问自己一个问题：这份文档的主要价值，是在动手之前澄清问题和对齐认知，还是主要就是一个存档？如果是前者，可能确实有必要。如果主要是后者，那也许直接做个原型更高效。当然，不同的团队情况不同，这里没有万能答案，但至少值得有意识地想一想。

我们来看第二个重要变化：代码溯源方式的重构。

以前，当你看到一段代码有问题，第一反应是用 git blame 查一查，找到提交这段代码的工程师，然后去问他。这是非常自然的流程：代码是人写的，出了问题就问写代码的人。

但现在，Claude Code 团队里几乎所有的 PR 都有 AI 的参与。这种情况下，"谁写了这段代码"这个问题变得没那么有意义了，因为答案可能是 AI，或者是某个工程师和 AI 协作完成的，而这个工程师自己未必能说清楚每一行逻辑背后的细节。

Fiona 说，他们的新规范是往深一层问：你真正需要知道的是什么？

她给了几个非常具体的场景。第一个场景：你发现了一个回归问题，也就是本来好用的功能突然出问题了。你需要知道的不是"谁提交了这段代码"，而是"这段逻辑在什么时候、因为什么原因发生了变化，以及当时的决策依据是什么"。这个问题你可以直接问 Claude，让它帮你分析 git history，追踪逻辑变化的来龙去脉。第二个场景：有个客户问了一个技术问题，你需要找人来回答。你需要的不是"找到最早写这个模块的人"，而是"找到现在对这个问题了解最深的人，或者找到最快能给出准确答案的信息来源"。第三个场景：你需要理解某个架构决策背后的理由。传统做法是找到当初做这个决策的人问，但如果那个人已经离职了呢？如果文档写得很简略呢？现在 Claude 可以帮你梳理 PR 的讨论记录、issue 的历史、commit message，重建决策背景，这往往比找到某一个人问更全面。

Fiona 更进一步说，他们团队的习惯是，对于任何这类问题，都要额外多问一句：这件事有没有可能自动化？

她举了一个她自己的例子。她以前每天早上拿着咖啡，手动浏览客户反馈渠道，把用户的反馈梳理汇总成一份摘要，然后再分享给团队。这件事本身很有价值，因为理解用户声音对产品决策非常重要。但它作为一个人工操作的流程，效率不高，而且它的价值在于"汇总之后的内容"，不在于"汇总这个动作"本身。现在，这个任务已经被自动化了。Claude 每天早上自动去抓取相关渠道的内容，整理出摘要，她只需要看结果就好了。人的时间，从"做汇总"释放出来，可以用于"判断这些反馈意味着什么、应该怎么应对"。

这背后有一个值得记住的思维框架：把"例行工作"和"判断工作"明确分开。浏览、汇总、分类、归档，这些是例行工作，结构明确，可以被自动化。判断某条用户反馈是否代表一个系统性问题，决定这个问题在当前阶段是否值得优先处理，这才是判断工作，需要人来做。在 AI 时代最有效率的工程师和管理者，往往是那些能清楚区分这两类工作、并且毫不留情地把例行工作交给 AI 的人。

我们来看第三个核心点：人工审查的重新定位。

Fiona 的团队在代码审查上做了非常细致的层次划分，把不同类型的审查工作分给了 AI 和人，两者各有侧重，但分工逻辑清晰。

AI 负责的部分包括：代码风格和 lint 检查，根据 PR 要求给出反馈，在代码完整提交之前发现并修复 bug，以及自动补充测试用例。这些事情，以前需要工程师花大量时间去做，现在 Claude 可以全部处理。对于大多数团队来说，这些工作在代码审查总时间里占了相当大的比例。把它们交给 AI，意味着人工审查可以聚焦在更高价值的地方。

那么人工审查集中在哪些地方？Fiona 给出了三类，每一类都有具体的理由。

第一类是法律合规相关的决策。涉及版权、隐私协议、许可证条款、风险敞口这类问题，Fiona 说她一定要有法律合作伙伴参与进来。AI 可以帮你起草合同条款，可以帮你解释某个监管要求，但它不知道你公司当前的法律暴露程度，不知道某个商业决策在你们所在市场的监管环境里意味着什么风险，不知道你的法律伙伴上周刚刚谈妥了什么新的合规要求。这些都需要真正懂法律、了解公司实际状况的人来判断。

第二类是涉及信任边界和安全的代码。权限管理、数据隔离、加密实现、身份验证这些模块，Fiona 要求领域专家必须参与审查，没有任何弹性空间。这里有个微妙但很重要的技术原因：AI 在生成安全相关的代码时，会非常"自信"地给你写出看起来正确的代码。但如果它的训练数据里存在某个常见的错误模式，它很可能会重复这个错误，而且写得非常流畅，看起来没有任何问题。安全漏洞的特点是平时不显现，出了事才知道，而且代价极高。所以在这个领域，宁可多花人工成本，也不能轻易降低审查标准。

第三类是产品直觉和设计品味。这个有点出乎意料，但我觉得 Fiona 说得非常准确。她说产品经理和设计师需要深度参与审查，因为这类判断涉及的是"产品感"和"审美品味"，这是 AI 目前很难可靠模拟的东西。AI 可以帮你生成十种界面布局，可以分析哪种布局在过去的设计案例里更常见，但它不知道你的目标用户群体的具体使用习惯，不知道你的品牌调性要求什么感觉，不知道这个功能在当前的产品阶段是应该做还是不该做。这些判断，依赖的是对用户、对市场、对公司策略的深度理解，这种理解目前还是人类特有的优势。

Fiona 还补充了一个非常重要的观点：这条分界线不是静止的。随着模型能力的持续提升，今天需要人工的事情，明天可能就可以被安全地自动化了。所以工程领导者需要定期重新评估这条边界，而不是把某个规则固化下来，几年不动。这个观点我非常认同，而且我个人认为，这种动态调整的能力，会成为未来工程团队一个非常重要的竞争力。

第四个核心点，是关于团队角色的演变和招聘思路的重构。

Fiona 说，AI 重塑了团队里几乎每个人的角色。她举了一个她自己觉得"很有意思"的现象：他们的产品经理们现在写了很多代码。

这句话背后的含义值得仔细想想。产品经理写代码，不是指他们在替代工程师，也不是指他们在做什么复杂的后端系统。而是说，借助 Claude Code 这类工具，他们能够把自己脑子里的产品直觉，快速转化成一个可以看到、可以点击、可以让真实用户试用的原型。

以前这条路是什么样的？产品经理有了一个想法，先要把它写成一份 PRD，也就是产品需求文档，然后拿给工程师评估，估工时，等排期，等资源，等工程师做完了其他手上的事情。等到真正能看到一个可以用的原型，可能已经是好几周、甚至好几个月以后了。在这段时间里，很多早期的假设可能已经错了，市场可能已经变了，但你还没有任何真实的验证数据。

现在，产品经理可以用一个下午的时间，自己搭出一个粗糙但能用的原型，放给几个真实用户试一试。核心假设对不对，两天内就能有感觉。如果假设错了，损失的是几个小时的时间，不是几个月的工程资源。这对产品迭代速度的影响是根本性的。

反过来，工程师们也在做一些传统上属于其他职能的工作。Fiona 提到，工程师们开始涉足内容创作和设计工作。这不是说他们变成了专业的设计师或者文案，而是说借助 AI 工具，他们能够在这些领域做出足够好的初稿，然后再拿给专业人员打磨和确认。

这种角色边界的模糊化，对招聘思路有深远的影响。Fiona 说她在招人的时候，主要看两种人。

第一种，她称之为"有产品感的创意型建设者"。原文里她用的词是 dreamers who are deeply curious and passionate about shipping products that solve problems，翻译过来就是：充满好奇心、对解决真实问题充满热情的梦想家。这类人的核心价值，不是他们掌握几门编程语言，也不是他们的代码写得有多快，而是他们能不能清楚地看到用户真正需要什么，能不能把这种洞察转化成有价值的产品。

第二种，是有深度系统背景的工程师。Fiona 举了一个非常具体的例子：当他们在做 Claude Code 的 Web 版本的时候，她发现团队里缺少有系统底层背景的专家。这里的"系统背景"指的是操作系统、网络、运行时这类底层知识。要让 Claude 在所有环境里稳定可靠地运行，需要深入理解底层的约束和行为，这类知识目前 AI 还无法自我提供，需要真正懂这些东西的人类专家来把关。

而 Fiona 明确说她不太看重的，是"原始产出速度"。这在以前是衡量工程师价值的核心指标之一：你一天能写多少代码，能多快交付功能。但现在，这个能力已经被 AI 大量替代了。更重要的问题，是在哪些地方你仍然需要人类的专业判断，在那些地方才是需要重点投入人才的地方。

对于正在听节目的工程师朋友，这一点我觉得非常值得正视。如果你的核心竞争力主要是"写代码快"，那确实需要想一想怎么升级自己的价值点。但如果你有某个领域的深厚知识积累，有敏锐的产品直觉，有能够理解复杂系统全局的能力，或者有非常强的跨职能协作能力，这些在 AI 时代反而变得更稀缺、更值钱了。

第五个核心点，是他们的团队治理方式，这部分我觉得对工程管理者特别有参考价值。

Fiona 分享的治理框架，可以用一句话概括：有少数几条核心原则是全团队不可妥协的"必须遵守"，在这几条原则之内，各个小组——他们叫做 pod——有很大的自主权，可以自己决定怎么运作。

先说那几条必须遵守的核心原则，一共三条。

第一条是：疯狂地自用自己的产品，英文叫 dogfooding，字面意思是"吃自己的狗粮"。Claude Code 团队的所有成员，不仅是工程师，还包括产品经理、设计师、法务这些交叉职能合作伙伴，都必须在日常工作中使用 Claude Code 和 Claude Cowork。注意，这里用的词不是"鼓励"也不是"推荐"，而是"所有人都用"，是一个强制性的规范。背后的逻辑是：如果你自己不真正使用这个产品，你就不可能真正感受到用户的痛点，也不可能在日常工作中随时注意到产品的问题。这条原则还有一个副作用：它让新人的上手速度非常快，因为入职第一天就被迫用这个工具做真实的工作，而不是先看一堆文档、等几个月后再开始接触产品。

第二条是：保持尽可能扁平的组织结构。Fiona 说她加入 Claude Code 团队之后，希望每一位管理者在成为管理者之前，先以独立贡献者的身份在团队里工作一段时间，亲身体验工程师的日常，弄清楚"在 Anthropic 做工程师是什么感觉"。团队有一个统一的整体使命，各个 pod 的负责人支持具体方向的工作，但人员可以根据工作需要灵活流动，不被固定在某个小组里。扁平结构的好处在 AI 时代尤其突出：信息传递快、决策快，而且管理层不容易脱离一线工作，不会出现"领导不知道工程师每天在用什么工具、面对什么问题"的情况。

第三条，也是 Fiona 在文章里着墨最多的一条：毫不留情地杀掉不再有用的流程。她说，团队成员有明确的授权，去质疑和终止那些已经没有意义的旧流程。这条原则听起来简单，但实际执行起来非常难，原因是大多数组织都有一种保守惯性——人们倾向于维持现状，因为提出改变需要承担风险，而不提出改变不会有任何代价。Fiona 的做法是把这种"挑战旧流程"的行为，从一种潜在的风险行为，变成一种被团队文化明确支持的行为。这两者之间的差别很大——一个是你可以去做但需要鼓起勇气，另一个是你应该去做、这是你的职责。

在这三条核心原则之内，各个 pod 的自主权非常大。他们可以自己决定怎么用 Claude 来处理问题分类，怎么组织规划会议和站会，哪些工作流程先进行"Claude 化"改造，哪些暂时保持不变。

我个人非常欣赏这种"少数核心原则加大量局部自主权"的治理方式。因为 AI 工具的具体用法，不同职能、不同项目、不同团队文化之间差异非常大。如果用一套统一的详细规范强制所有小组按同一个方式操作，反而会扼杀每个小组根据自己的实际情况找到最优用法的可能性。而少数几条明确的核心原则，保证了整体方向一致和底线质量，同时留出了足够的空间让实验发生。

好，我们刚才把 Fiona 文章里的五个核心点都讲完了。现在我们来聊聊，这些东西对我们的实际工作意味着什么，以及有哪些具体的行动可以参考。

Fiona 在文章里给出了三个她建议每个工程领导者现在就开始追踪的数字，我觉得这部分特别实用，值得详细说一下。

第一个数字是新人上手时间。一个新加入的工程师、设计师、或者产品经理，需要多长时间才能开始真正发挥作用，开始提交有实际价值的工作？Fiona 说她的团队现在这个时间比一年前大幅缩短了，工程师在入职第一周就能提交真实的代码，不是演示代码，不是练手代码，而是进入生产代码库的真实代码。

对很多团队来说，新人上手时间还相当长，要两到三个月才能让新人开始独立工作，中间需要大量的带教和环境配置和流程摸索。有 AI 辅助之后，很多这类摩擦可以大幅减少：新人可以直接问 Claude 来理解代码库结构，可以让 Claude 帮他解释某段陌生逻辑，可以让 Claude 帮他写第一个 PR 的测试用例。所以如果你的团队还在走老路，值得有意识地想一想，中间哪些步骤可以借助 AI 提速。

第二个数字是 PR 周期时间。从提交一个 PR 到这个 PR 被合并，平均需要多长时间。Fiona 特别提醒，这个指标背后可能藏着你流水线里的新瓶颈。她的原话值得引用一下：随着我们生成的代码量大幅增加，构建系统和 CI——也就是持续集成系统——有时候可能跟不上了。

这个提醒非常及时。很多团队在引入 AI 辅助编程之后，发现代码的产出确实增加了，但最终的交付速度并没有同比提升，甚至有时候反而因为 CI 排队时间变长而下降。这个时候，你的瓶颈就从"写代码"变成了"等测试跑完"。优化 CI 的并行化程度、提升测试执行速度、减少不必要的测试范围，在这个阶段就变得比以前重要得多了。

第三个数字是 Claude 辅助的提交比例。Fiona 说在他们团队里，默认情况下每一个 commit 都有 Claude 的参与。她说过去四个月里，她基本上没见过没有 AI 辅助的 commit 了，这是一个接近百分之百的比例。但她立刻补充了一个非常重要的提醒：不要把产出速度和成功等同起来。她的原话是：产出是一个指标，但真正的指标应该是你在解决什么问题。在方向正确的前提下，更高的产出速度可以帮助你更快地解决问题。但如果方向错了，更快的产出只会更快地把你带到沟里。

这个警告我觉得非常必要。AI 辅助开发有一个潜在的陷阱：你会感觉效率极高，PR 一个接一个地出，功能一个接一个地上，团队士气高昂。但如果没有持续地检验方向是否正确，这种高产出可能只是在一个错误的方向上跑得更快而已。

关于实际上手，Fiona 在文章结尾给出了一个非常具体的建议：找你最"嘈杂"的工作流程。

嘈杂可以是不同含义的：可能是你们团队成本最高的流程，可能是大家最头疼的例行工作，也可能是团队成员最不愿意参与的那种会议或者审查。找到这个流程之后，问两个问题：第一，这个流程现在还在服务它当初存在的目的吗？第二，如果它还有价值，能不能把它自动化？

她讲了一个故事来说明这一点。她曾经在一个团队里，每周有一个大型进展汇报会，很多人坐在同一个会议室里。但会议进行的时候，所有人都开着笔记本电脑在做自己的事情。只有轮到你汇报的时候，你才会抬起头来，说一下自己的进展，然后再低头继续。Fiona 就问了一个问题："我们为什么还要开这个会？这看起来是一种很贵的时间使用方式。"就这一个问题，让所有人一下子意识到这个会议已经没有必要了。于是这个会议就被取消了。

这个故事让我印象非常深。因为它揭示了一个普遍的组织现象：很多已经过时的流程，之所以还在运行，不是因为它们有价值，而是因为没有人去问"我们为什么还在做这件事"。有时候，一个简单的问题，就能打破这种惯性。

对于在大公司工作的朋友，这一点可能比在小团队里更难实践，因为很多流程是跨团队协调的结果，不是一个人能单方面改变的。但至少可以从自己控制的范围开始。如果你管一个小组，你的小组里有哪些例行的会议、汇报、审批流程，已经演变成了一种惯性而不是真正有价值的活动？先从这里开始，比等待整个组织变革要现实得多。

还有一个关于工具使用的实际建议。不要试图一开始就划定一条完整的"AI 做什么、人做什么"的清单，然后全面推行。可以从一到两个最具体的任务开始试验：比如先把代码 lint 检查完全交给 Claude，观察一个月，看看它漏掉了什么类型的问题，节省了多少人工时间，积累真实数据。然后再扩大范围，把更复杂的审查任务逐步纳入。这种渐进的方式，比一次性大规模推广风险更低，也更容易建立团队信任。

对于安全相关的代码，我个人的建议是：无论 AI 的能力进化到什么程度，在接下来相当长的时间里，都应该保持最高级别的人工审查。不是因为 AI 一定会出错，而是因为安全漏洞的代价是不对称的——你可能有几百次都没问题，但只要出一次，损失可能是灾难性的。在这种风险结构下，宁可多付出人工成本，也不能因为 AI "通常是对的"而放松警惕。

好，今天我们聊了 Anthropic 工程总监 Fiona Fung 在"Code w/ Claude SF 2026"大会上分享的这篇文章。核心脉络是：当 AI 接管了写代码这件事，工程团队的瓶颈移位到了验证、审查和安全，而应对这个新现实需要在规划节奏、代码溯源、审查分工、团队角色、组织结构和招聘思路上做出系统性的调整。

我个人觉得，这篇文章最有价值的地方，不是它给出了哪几条具体的操作规范，而是它背后的一种根本性思维方式：所有的流程都是为了解决某个问题而存在的。当那个问题不再存在，或者有了更好的解决办法，流程就应该被更新或者取消。这听起来是常识，但在真实的组织里，能真正做到的团队并不多见。流程天然有惯性，而质疑流程天然需要勇气和权限。Fiona 团队的做法，是把"质疑旧流程"这个行为变成一种被明确支持的文化，而不是让它停留在一种理论上可行但实践中无人敢做的状态。

Fiona 的团队有一个特殊优势，他们是 Claude Code 的开发者，自然是最早、也是最深度使用这套工具的团队。我们大多数人不会处于这样的位置。但这不影响这些经验的参考价值。因为 AI 辅助开发的大方向是确定的，这些问题——规划节奏的调整、人机分工的界定、组织结构的重塑、招聘标准的更新——迟早是每个工程团队都会面对的。

从这个角度来看，Fiona 的这篇分享，可以看作是一份来自稍微靠前位置的参考报告。现在就开始思考这些问题，比等问题已经摆在面前再应对，要从容得多。
