那我们先来说说背景。如果你平时用 AI 编程助手做开发,你大概会遇到一个特别现实的问题。你跟 Claude 或者其他agent聊一个新任务的时候,你得先告诉它,这个项目是用什么框架写的,数据库怎么连,代码风格是什么样,测试怎么跑,常用的命令是什么。每次开一个新的对话,这些东西就得重新说一遍。如果你的项目结构复杂,模块之间的关系绕来绕去,团队里又有一堆约定俗成但没写在文档里的规矩,那你每次跟 AI 沟通的成本就会越来越高。这其实是个挺普遍的痛点,你不是在写代码,你是在不停地给一个聪明但没有记忆的助理补课。

文章里提到,这个问题会随着代码库的增长而加剧。复杂的模块关系、特定领域的模式、团队的约定,这些东西不会自动浮现出来。你只能在每次对话开始的时候,把同样的架构决策、测试要求、代码风格偏好,重新解释一遍。这就是 CLAUDE 点 MD 文件要解决的问题。

那 CLAUDE 点 MD 到底是什么东西。简单说,它就是一个放在你代码仓库里的特殊配置文件,给 Claude 提供项目相关的持续性context,也就是上下文信息。你可以把它放在仓库根目录,这样整个团队都能共享;如果你是 monorepo,也就是一个仓库里管理多个项目的那种结构,你可以放在父级目录里;你甚至可以放在自己的 home 目录下,这样它就会应用到你所有的项目里。

这个机制最妙的地方在哪里呢。文章里说,CLAUDE 点 MD 文件会变成 Claude 系统提示词的一部分。也就是说,每次对话开始的时候,这些context已经被加载进去了,你不需要每次都重新解释一遍基本的项目信息。我觉得这个设计思路特别像我们人类团队里的入职文档。一个新同事入职,你不会指望他什么都靠口头交接,你会给他一份 readme,告诉他这个项目用什么技术栈,代码在哪个目录,测试怎么跑。CLAUDE 点 MD 本质上就是给 AI助手准备的那份入职文档,而且这份文档是每次对话都会自动递给它看的。

文章给了一个示例,我们简单还原一下里面的内容,方便大家理解它的结构。这个示例是一个用 FastAPI 写的用户认证和用户资料的 REST API项目。CLAUDE 点 MD 里写了几块内容,第一块是项目上下文,告诉 Claude 在这个代码库里工作的时候,优先考虑代码的可读性而不是炫技,做架构改动之前要先提出澄清性问题。第二块是关于这个项目的简介,说明它用 SQLAlchemy 做数据库操作,用 Pydantic 做数据验证。第三块是关键目录的说明,比如 app 斜杠 models 是数据库模型,app 斜杠 api 是路由处理,app 斜杠 core 是配置和工具函数。第四块是代码规范,要求所有函数都要写类型提示,测试用 pytest,fixtures放在 tests 斜杠 conftest 点 py 里,代码风格遵循 PEP 8,每行一百个字符。第五块是常用命令,比如怎么启动开发服务器,怎么跑测试。最后还有一些备注,比如所有路由都用斜杠 api 斜杠 v1 这个前缀,JWT token二十四小时过期。

你看,这份文件其实不复杂,但信息密度很高。文章特别强调,这个文件没有固定格式要求,建议是保持简洁、人类可读,把它当成那种人和 AI都需要快速理解的文档来写。

那这里就出现一个很实际的问题了,如果是一个全新的、或者你不太熟悉的代码库,要从零开始写这份文件,听起来挺麻烦的。文章里提到了一个解决方案,就是斜杠 init 这个命令。你在任何 Claude Code 的会话里,进入你的项目目录,启动 claude,然后输入斜杠 init,Claude 就会自动去检查你的代码库,读取包管理文件、已有的文档、配置文件、代码结构,然后生成一份量身定制的 CLAUDE 点 MD。生成的文件通常会包含它检测到的构建命令、测试指令、关键目录,以及代码规范。

但是这里有个特别重要的提醒,文章原话的意思是,把斜杠 init 当作一个起点,而不是成品。生成的 CLAUDE 点 MD 能捕捉到一些明显的模式,但很可能漏掉你工作流里那些细微的、特定的东西。所以你需要去审阅 Claude 生成的内容,根据团队实际的实践去打磨它。

我觉得这一点特别值得展开讲讲,因为这是这篇文章里我认为最实用的一个建议。很多人用工具的时候有个误区,觉得自动生成的东西就是终点,生成完了就完事了。但实际上,斜杠 init 生成的文件,它只能基于代码本身能推断出来的信息,比如你用的框架、目录结构、依赖关系。但有些东西是代码里看不出来的,比如你们团队的分支命名规范,部署流程是怎么走的,代码评审有什么具体要求。这些隐性知识,只能靠人去补充。

文章给出了几个具体的后续步骤,跑完斜杠 init 之后应该做什么。第一,审查生成内容的准确性。第二,补充 Claude 推断不出来的工作流说明,比如分支命名约定、部署流程、代码评审要求。第三,删除那些不适用于你这个项目的通用性指导。第四,把这份文件提交到版本控制系统里,这样整个团队都能受益。

另外文章还提到一个很巧妙的小技巧,就是在跑斜杠 init 的过程之外,平时用 Claude Code 工作的时候,如果你发现自己反复在重复某条指令,你可以用井号键来添加这条指令,这些添加的内容会逐渐积累,变成一份真正反映团队实际工作方式的 CLAUDE 点 MD。这个细节我觉得特别值得拿出来说一说,因为它揭示了一种持续迭代的思路。CLAUDE 点 MD不是写一次就结束的静态文档,而是应该随着你日常使用的摩擦点不断打磨的活文档。每次你发现自己又跟 Claude 重复解释了一遍同样的东西,那就是一个信号,告诉你这条信息该写进 CLAUDE 点 MD 里了。

好,接下来我们进入文章里我认为最核心的三个方法论部分,分别是怎么帮 Claude 导航复杂架构,怎么让它了解你的自定义工具,以及怎么通过标准工作流防止返工。

第一个,关于复杂架构的导航问题。文章说,解释你的项目架构、关键库、代码风格,如果每次新任务都要做一遍,这个事情会变得很枯燥。你需要 Claude 对你的代码库结构维持一个一致的认知,而不需要你每次都手动去强化它。

解决方案是,在 CLAUDE 点 MD 里加入项目摘要和高层级的目录结构。文章里给了一个例子,用一个简单的树形结构展示关键目录,比如 main 点 py 下面有 logs 目录放日志,modules 目录下面有 cli 点 py、logging_utils 点 py、media_handler 点 py、player 点 py 这些文件。这种树形结构能让 Claude 立刻明白不同组件分别放在哪里。

文章还建议,要包含主要依赖、架构模式,以及任何非标准的组织方式的信息。如果你用的是domain driven design,也就是领域驱动设计,或者微服务架构,或者某些特定框架,都应该写进去。Claude 会用这份地图来更好地判断去哪里找代码,以及在哪里做修改。

这里我想多展开讲一下,为什么这个看起来很简单的目录树会有这么大的作用。你想象一下,如果你接手一个新项目,没有任何文档,你打开文件夹,看到几十个子目录,你第一反应是什么。你大概会先大致扫一遍,猜测哪个目录是核心业务逻辑,哪个是工具类,哪个是配置。这个过程对人类来说可能要花上几十分钟甚至几个小时。对 Claude 来说,虽然它检索代码的速度比人快得多,但如果没有一个明确的地图,它同样要靠猜测和试探去摸清楚结构,这个过程会消耗大量的token,也就是它处理文本的基本单位,而且容易出错。给它一份现成的目录树和架构说明,相当于直接把这张地图递到它手里,它就不需要再自己花力气去重新画一遍地图了。

第二个核心方法论,是关于自定义工具的集成。文章说,Claude 继承了你完整的开发环境,但它需要指引才知道该用哪些自定义的工具和脚本。你的团队很可能有专门用于部署、测试或者代码生成的工具,Claude 应该知道这些工具的存在。

解决方案是,在 CLAUDE 点 MD 里记录你的自定义工具,带上使用示例。包括工具名称、基本用法模式,以及什么时候该调用它。如果你的工具支持 dash dash help这种帮助文档的查看方式,也要提一下,这样 Claude就知道可以去查。对于复杂的工具,可以加上你团队经常用的调用示例。

这里文章还专门提到了 MCP,也就是 Model Context Protocol,模型上下文协议。Claude 可以作为 MCP 的客户端,连接到扩展它能力的 MCP 服务器。这些服务器可以通过项目设置、全局配置,或者签入版本控制的点 mcp 点 json 文件来配置。如果工具没有按预期出现,可以用 dash dash mcp-debug 这个参数来排查连接问题。

文章举了一个例子,假如你们公司配置了一个 Slack 的 MCP 服务器,你想让 Claude 知道怎么用它,你可以在 CLAUDE 点 MD 里写,这个 Slack MCP 只能往 dev-notifications 这个频道发消息,用于部署通知和构建失败的提醒,不要用来发单个 PR 的更新,因为那些走 GitHub 的 webhook通知,而且这个工具每小时限制十条消息。你看,这种细节级别的说明,其实就是把团队里那些只存在于老员工脑子里的潜规则,写成了 AI 也能读懂的指令。这种潜规则一旦没写下来,新人会撞墙,AI 助手同样会撞墙,它要么乱用工具,要么压根不知道这个工具存在。

第三个核心方法论,可能是我认为最重要的一个,就是标准工作流。文章里说,如果让 Claude 直接跳进代码改动而不先做规划,会造成返工。Claude 可能实现一个遗漏了需求的方案,选错了架构思路,或者做出破坏现有功能的改动。

所以你需要 Claude 在行动之前先思考。文章建议在 CLAUDE 点 MD 里定义标准工作流,针对不同类型的任务该遵循什么流程。一个靠谱的默认工作流,在做改动之前要先回答四个问题。第一,这是不是一个关于当前状态的问题,需要先去调研。第二,这个改动是不是需要一份详细的计划才能开始实施。第三,还缺少哪些额外的信息。第四,怎么去测试这个改动的有效性。

具体的工作流可以包括,针对新功能的探索-规划-编码-提交这种四步法,针对算法类工作的测试驱动开发,针对界面改动的视觉迭代法。文章还建议把测试要求、提交信息的格式、任何需要审批的步骤都写清楚。当 Claude 提前知道你的工作流,它就会按照团队实际的流程去组织工作,而不是凭自己猜测。

文章给了一个示例性的工作流指令,大概是这样写的,在修改某几个特定位置的代码之前,要考虑这个改动会不会影响到其他相关模块,要先构建一份实施计划,再制定一份测试计划,用来验证相关函数的正确性。

我觉得这个返工的问题特别值得我们多想一层。你回想一下你自己带新人的经历,如果一个新同事刚入职,你直接给他派一个复杂任务,让他自己摸索着改代码,大概率会出问题,因为他不了解全局,容易改了 A 忘了 B,或者用了一个跟现有架构格格不入的方案。但如果你先让他过一遍需求,先出一份方案,你审一遍,确认没问题再让他动手,返工率就会大幅下降。Claude 也是一样的道理,它能力很强,但它对你的项目没有那种浸润式的、长期积累的直觉,所以把这种思考前置的习惯写进 CLAUDE 点 MD,本质上是在帮它建立一种类似人类工程师的审慎习惯。

讲完这三个核心的方法论,文章接下来介绍了三个额外的技巧,这三个技巧不属于 CLAUDE 点 MD 文件本身,但是跟它配合使用,效果会更好。

第一个技巧,是用斜杠 clear来清空上下文。文章说,长时间使用 Claude Code,会积累一堆不相关的context。早期任务里的文件内容,已经无关紧要的命令输出,还有跑偏的对话内容,都会堆积在 Claude 的上下文窗口里。随着信号和噪音的比例下降,Claude 就很难继续专注在当前任务上。

解决方法很简单,在不同任务之间用斜杠 clear来重置上下文窗口。这会清除积累的历史记录,但会保留你的 CLAUDE 点 MD 配置,Claude 依然能用全新的context去处理新问题。你可以把这个动作想象成关闭一个工作会话,打开另一个新的会话。

文章举的例子是,当你调完认证流程的 bug,要开始实现一个新的 API 端点的时候,就该清空上下文了。认证相关的细节已经不重要了,留着反而会干扰新的工作。

第二个技巧,是用子代理来隔离上下文。这个跟斜杠 clear解决的是类似的问题,但场景不太一样。文章给的例子是,假设你刚刚调完一个复杂的认证流程,现在你需要对同一段代码做安全审查。调试过程里积累的context会给 Claude 的安全分析染上颜色,可能导致它忽略某些问题,或者过度关注已经解决过的问题。

解决方法是告诉 Claude,针对工作里不同的阶段去使用子代理。子代理会维持一个独立隔离的context,防止早期任务的信息干扰新的分析。比如你刚实现完一个支付处理器,你可以让 Claude 用一个子代理对这段代码做安全审查,而不是在同一个对话里继续往下聊。

文章特别提到,子代理最适合用在多步骤工作流里,每个阶段需要不同的视角。实现阶段需要架构context和功能需求,安全审查阶段需要带着一双只盯漏洞的新眼睛去看。把context分开,能让两边的分析都保持锐利。

我自己琢磨这个点的时候,会想到一个类比。如果你请同一个人,先去写一份代码,然后马上让他给自己的代码做安全审查,他很容易带着一种自我辩护的心态去看,觉得这里写成这样是有道理的,反而看不出真正的漏洞。但如果换一个完全没参与开发、只盯着安全问题来看的人,他反而更容易发现问题。子代理这个机制,其实就是在模拟这种换人审查的效果,只不过这里是给同一个 AI 换一个干净的视角,而不是真的换一个人。

第三个技巧,是自定义斜杠命令。文章说,重复性的提示词会浪费时间。你可能发现自己反反复复地输入审查这段代码有没有安全问题,或者分析一下这段代码有没有性能问题,每次还得回忆怎么措辞才能拿到好的结果。

解决方案是,把这些提示词存成 markdown文件,放在点 claude 斜杠 commands 目录下。比如创建一个叫 performance-optimization 点 md 的文件,里面写好你偏好的性能优化提示词,它就会以斜杠 performance-optimization 这个命令的形式,在任何对话里直接可用。这些命令还支持通过 ARGUMENTS,或者带编号的占位符比如美元符号一、美元符号二,来传入具体的参数,你可以借此传入特定的文件或者参数。

文章给了一个非常详细的性能优化命令的范例。这份命令文件要求 Claude 分析提供的代码的性能瓶颈和优化机会,覆盖几个大类。数据库和数据访问方面,要看有没有 N加一查询的问题,有没有缺失数据库索引,有没有低效的联表或者子查询,有没有缺失分页,有没有查询结果缓存,连接池有没有问题。算法效率方面,要看时间复杂度是不是 O n方或者更差,有没有可以优化的嵌套循环,有没有冗余计算,数据结构选择是否合理,有没有缺失记忆化或者动态规划的机会。内存管理方面,要看有没有内存泄漏,有没有该流式处理却整体加载的情况,循环里有没有过度的对象实例化。异步和并发方面,要看有没有该异步却阻塞的 I O操作,有没有可以并行却串行执行的操作,有没有缺失 Promise 点 all这种并发执行模式。网络和 I O方面,要看 API 调用是不是太频繁,有没有响应缓存策略,大负载有没有压缩,有没有用 CDN。前端性能方面,要看有没有阻塞渲染的 JavaScript或 CSS,有没有代码分割和懒加载,图片资源有没有优化,长列表有没有虚拟化。最后是缓存方面,要看 HTTP缓存头有没有缺失,应用层有没有缓存层。

这份命令还规定了输出格式,要求针对每个发现的问题,说明问题描述、具体的文件和行号位置、严重程度的评级、当前的时间和空间复杂度、具体的优化建议、优化后的代码示例,以及可量化的预期改进幅度。如果代码已经优化得很好,也要确认这一点,列出已经正确实施的最佳实践。

这份范例其实信息量很大,我觉得它最大的价值不在于这份清单本身,清单你网上随便都能找到类似的,它的价值在于,把这份清单固化成一个可以反复调用的命令之后,你每次想做性能审查,不需要再重新打字描述要求,直接敲一个斜杠命令,Claude 就会按照这套结构化的标准去执行。这其实是把你脑子里那套审查标准,变成了一个可复用的工具。

而且文章还提到一个挺贴心的细节,你不需要自己手动去写这些命令文件,你可以直接让 Claude 帮你创建。比如你跟它说,创建一个叫斜杠 performance-optimization的自定义斜杠命令,用来分析代码里的数据库查询问题、算法效率、内存管理和缓存机会,Claude 就会把这份 markdown 文件写到点 claude 斜杠 commands 斜杠 performance-optimization 点 md 里,这个命令立刻就能用了。这种自我生成工具的能力,我觉得是这篇文章里一个挺巧妙的闭环设计。

讲完了具体的方法,文章最后给出了几条最佳实践的提醒,我觉得这部分对实际动手的听众特别有用。

第一条提醒,不要一上来就想写一份大而全的 CLAUDE 点 MD。文章原话的意思是,这种冲动要克制住。因为 CLAUDE 点 MD每次都会被加入到 Claude Code 的context里,所以从context engineering,也就是上下文工程,和 prompt engineering,也就是提示词工程的角度出发,应该保持简洁。一个可行的做法是,把信息拆分成多个独立的 markdown文件,然后在 CLAUDE 点 MD 里引用这些文件。

这一点我觉得特别值得强调,因为很多人第一反应会觉得,信息越详细越好,把所有能想到的东西都塞进去。但这篇文章提醒我们,CLAUDE 点 MD是要在每次对话里被重新读取的,这意味着它占用的不是一次性的成本,而是持续性的成本。每多写一行没用的内容,就是在每一次对话里都白白消耗一点token,而且还可能稀释掉真正重要信息的权重。所以更合理的策略,是保持主文件的精炼,把详细的、不常用的信息拆出去,按需引用。

第二条提醒,也是我认为最关键的一条安全建议,不要在 CLAUDE 点 MD里包含敏感信息,比如 API 密钥、凭证、数据库连接字符串,或者详细的安全漏洞信息,尤其是当你把这份文件提交到版本控制系统的时候。因为 CLAUDE 点 MD会变成 Claude 系统提示词的一部分,你应该把它当成一份可能会被公开分享的文档来对待。

这一条我觉得怎么强调都不为过。我们平时在写代码注释、写文档的时候,可能会图省事,顺手把一个测试用的密钥或者内部的服务器地址写进去,觉得反正是内部文档。但 CLAUDE 点 MD 这个文件有它的特殊性,它会被系统提示词吸收,也经常会被提交到 Git仓库里跟团队共享,如果仓库本身是公开的,或者哪天权限设置出了问题,这些敏感信息就会暴露出去。所以这条建议其实是在提醒我们,把 CLAUDE 点 MD的安全等级,提升到跟对外公开文档一样的标准去对待。

文章结尾总结说,CLAUDE 点 MD 文件把 Claude Code从一个通用助手,变成了一个专门针对你代码库配置过的工具。建议从简单的项目结构和基础文档开始,然后根据工作流里真实遇到的摩擦点去扩展。最有效的 CLAUDE 点 MD文件,解决的都是真实的问题,它们记录你反复输入的命令,捕捉那些原本需要花十分钟才能解释清楚的架构context,建立能够防止返工的工作流。这份文件应该反映团队实际的开发方式,而不是听起来很美好但跟现实不符的理论上的最佳实践。最后文章强调,把这种定制工作当成一项持续性的实践,而不是一次性的设置任务。项目会变化,团队会学到更好的模式,新工具会进入工作流,一份维护得好的 CLAUDE 点 MD会随着代码库一起演进,持续降低跟 AI 协作处理复杂软件时的摩擦。

好,那我们花点时间想想,这篇文章对我们实际工作意味着什么,具体能怎么上手。

如果你现在还没有给自己的项目配置 CLAUDE 点 MD,我的建议是,今天就去试一下斜杠 init这个命令,花十分钟看看它给你生成了什么。看完之后,别急着满意,挑出那些它没猜对、或者完全没提到的地方,比如你们团队的代码评审流程,部署需要走哪些审批,这些信息补进去。

如果你已经有了一份 CLAUDE 点 MD,但是很久没更新过,这是一个很好的检查时机。你可以回想一下,最近这段时间,你有没有反复跟 Claude 解释过同一件事,如果有,那就是一个信号,这条信息该写进 CLAUDE 点 MD 里了。文章里提到的用井号键添加指令这个小技巧,我建议大家养成习惯,看到自己又在重复输入同样的指引时,顺手就记下来。

另外,如果你的项目里有自定义的脚本和工具,而你之前从来没有专门跟 Claude 介绍过它们,这也是一个值得马上动手的点。哪怕只是简单写一句这个工具是干什么的,什么时候该用,也比什么都没写要好得多。

至于子代理和斜杠 clear这两个技巧,我个人认为它们特别适合那种任务切换频繁、上下文容易互相干扰的工作场景。如果你发现 Claude 在你切换任务之后,还在用前一个任务的思路来理解新问题,这通常就是该清空上下文,或者该派一个子代理去处理新任务的信号。

至于自定义斜杠命令,我建议从你最常重复的那一两句提示词开始做。不需要一开始就写得像文章里那份性能优化范例那么详尽,先把你最常用的那句话存下来,跑通之后再慢慢往里面加细节。

要避的坑,我前面也提到了,一个是别想着一次性写出完美的大而全文件,先简单开始,慢慢迭代;另一个是绝对不要往里面塞任何密钥、凭证或者敏感的安全信息,尤其是这份文件大概率会被提交进版本库,被整个团队甚至外部协作者看到。

好,我们今天聊的这篇文章,核心就是一句话,CLAUDE 点 MD不是什么花哨的功能,它是一种让你和 AI 协作时减少重复劳动的基础设施。它的价值不在于一次性写完,而在于持续地、根据真实摩擦点去打磨。我个人认为,这篇文章最值得带走的,不是文中那些具体的示例代码,而是它背后那种思维方式,把你每次跟 AI 助手解释的内容,看作是一种应该被沉淀、应该被复用的知识资产,而不是每次对话都要重新交付的一次性劳动。如果你之前从来没认真写过这份文件,不妨今天就花十分钟试试斜杠 init,看看会有什么惊喜。
