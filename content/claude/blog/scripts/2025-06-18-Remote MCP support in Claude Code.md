大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是 Remote MCP support in Claude Code,翻译过来就是"Claude Code 当中的远程 MCP 支持"。这篇文章发表于二〇二五年六月十八日,也就是去年的六月十八号。

先说一句提醒,这篇文章距离我们录这期节目,已经过去整整一年多了。今天是二〇二六年六月二十一号,在 AI 这个领域里,一年的时间跨度其实相当长,差不多可以让一整代模型完成迭代,也足够让一个早期的实验性功能,变成行业里大家习以为常的标准配置。所以我们今天聊这篇文章,既要还原它当时的历史背景,讲清楚它解决了什么问题,也要带着今天的视角,去看这个功能后来走向了哪里,有没有过时,有没有被超越。这是我们听播客很重要的一种姿态,不是把旧文章当成最新动态,而是把它当成理解技术演进路径的一个坐标点。

我们先把时间拨回到二〇二五年年中。那个阶段,整个行业正处在从"模型能力竞赛"逐渐转向"模型怎么真正落地到工作流里"的过渡期。大家发现,光是模型本身聪明已经不够用了,真正卡住生产力的,往往是模型和外部世界之间的连接问题。你的代码助手再聪明,它也看不到你公司内部 Sentry 上报的报错日志,看不到 Linear 里那条正在跟进的任务卡片,看不到你们团队的知识库。这些信息散落在十几个不同的工具里,工程师每天要做的事情,很大一部分就是在这些工具之间来回切换标签页,复制粘贴上下文,然后再回到代码编辑器里继续干活。这种割裂感,在当时被反复提起,也催生了一个叫做 MCP 的协议,英文全称是 Model Context Protocol,模型上下文协议。

MCP 这个协议你可以把它理解成一种通用接口规范,有点像电脑外设的 USB 接口。在 USB 出现之前,每个外设都有自己专属的接口和驱动逻辑,鼠标一个样,打印机又是另一个样,厂商各自为政,用户和开发者都很痛苦。USB 出现之后,只要外设和电脑都遵循同一套协议,插上就能用,不需要为每一种组合单独写适配代码。MCP 想解决的就是 AI 应用和外部数据源、外部工具之间的这种适配难题。一个 MCP 服务器,可以把某个具体的工具或者数据源,比如说 Sentry 的报错系统,包装成一套标准化的"资源"和"工具"接口,暴露给任何支持 MCP 协议的 AI 客户端。只要 Claude Code 这边也支持 MCP,理论上它就可以连接任何遵循这套协议的服务器,不用单独为每一个工具写定制化的集成代码。这是这篇文章背后真正的协议基础,虽然原文没有从头解释 MCP 是什么,但作为听众,理解这一层背景,你才能明白接下来要讲的"远程"这个词,到底改变了什么。

好,那我们进入这篇文章真正的核心内容,也就是 Claude Code 这次更新具体带来了什么。文章一开头就点出了这次发布的关键词,叫做远程 MCP 服务器支持。在这之前,MCP 服务器主要是以本地方式运行的,也就是说,你需要在自己的电脑上,或者公司的某台机器上,把这个 MCP 服务器跑起来,变成一个本地进程,然后 Claude Code 才能通过这个本地进程去访问对应的工具或者数据。这种本地部署模式听起来好像也没什么大问题,但其实背后藏着不小的运维负担。你需要自己安装依赖,自己处理版本更新,自己保证这个进程一直健康运行,如果换了一台电脑,还得重新搭建一遍环境。对于个人开发者来说这已经够烦的了,对于一个团队、一个公司来说,这种负担会被成倍放大,因为每个工程师的本地环境都可能长得不一样,出了问题排查起来也麻烦。

远程 MCP 服务器要解决的正是这个痛点。它的核心思路是,把 MCP 服务器这一层,从你自己的电脑上挪到云端,由提供这个服务的厂商去托管和运维。换句话说,如果你想用 Sentry 的 MCP 服务器,你不需要自己跑一个 Sentry MCP 的本地进程,你只需要在 Claude Code 里填入 Sentry 官方提供的一个远程服务器地址,也就是一个 URL,剩下的事情,比如说服务的更新迭代、扩容、可用性保证,全部都由 Sentry 这边负责。文章里原话讲得很直接,远程 MCP 服务器提供了一种维护成本更低的替代方案,你只需要把厂商的网址加到 Claude Code 里,不需要手动搭建,厂商负责处理更新、扩容和可用性,这样你就可以专注在写代码上,而不是去维护服务器基础设施。

我觉得这个设计思路,其实跟整个云计算行业过去十几年走过的路非常像。我们可以做一个类比,早期大家都是自己买服务器、自己装机房,后来云服务商出现了,大家把基础设施这一层外包出去,自己只需要关心业务逻辑。MCP 服务器的本地到远程的迁移,本质上也是同一种"基础设施外包"的逻辑,只不过这次外包的对象,从计算资源变成了"AI 工具的连接层"。如果你来设计这样一个系统,你大概也会得出类似的结论,因为让每一个最终用户都去自己维护一套服务进程,从规模化的角度来说是不划算的,谁更靠近这个工具的源头,谁就更适合去负责维护这层接口,厂商显然比终端用户更适合干这件事。

接下来文章提到了第二个很关键的技术细节,就是身份验证的问题。这一点其实非常重要,因为只要涉及到把你的账号、你的数据连接给一个外部服务,安全性永远是第一位要考虑的事情。过去如果你想让一个工具访问你的第三方账号,常见的做法是去申请一个 API key,也就是一个长期有效的密钥字符串,然后把这个密钥粘贴到配置文件里。这种方式的问题在于,密钥一旦泄露,后果可能很严重,而且很多人会把密钥直接写在代码仓库里,或者存在不太安全的地方,管理起来很麻烦。Claude Code 这次给远程 MCP 服务器配上了原生的 OAuth 支持。OAuth 你可以理解成一种"授权而不是给密码"的机制,你点一下登录按钮,跳转到对应服务比如说 Linear 或者 Sentry 的官方登录页面,在那个页面上完成身份验证,验证通过之后,系统会给 Claude Code 发一个有时效性、有权限范围限制的授权凭证,而不是把你的账号密码直接交出去。文章里写得很清楚,只需要对你的服务器认证一次,Claude Code 就会处理剩下的事情,不需要管理 API key,也不需要存储任何凭证。

这个设计对工程师来说其实解决了一个长期存在的心理负担。我们很多人在配置开发工具的时候,最不愿意做的事情之一,就是去申请、保存、轮换各种密钥。一旦你手上有十几个工具都要配密钥,管理成本会变得很高,而且密钥过期了要重新生成,密钥泄露了要紧急撤销,这些都是隐性的维护成本。OAuth 这种"登录一次,后续自动续期"的模式,把这部分负担转移给了标准化的身份验证流程,用户体验上会顺畅很多,安全性上也更有保障,因为厂商可以随时撤销或者限定这次授权的权限范围,而不需要让用户自己去想办法处理密钥泄露的善后工作。

讲完了架构和安全这两层,我们再来看文章里给出的两个具体案例,这两个案例其实是帮我们理解"远程 MCP 到底能干什么"的最好切入点。

第一个案例是 Sentry。Sentry 是一个很多团队都在用的错误监控和性能追踪平台,你的线上服务出了异常,报错堆栈、触发频率、影响范围这些信息,都会汇总到 Sentry 里。文章里提到,通过把 Claude Code 跟 Sentry 的 MCP 服务器对接起来,你可以直接在 Claude Code 里访问 Sentry 上的错误和问题信息,然后基于这些问题的上下文去做调试,而完全不需要离开你的终端窗口。我们可以想象一下这个场景,你平时排查一个线上 bug,可能要先打开浏览器登录 Sentry 后台,找到对应的报错记录,看堆栈信息,再切回 IDE 去定位代码,中间可能还要把堆栈信息复制粘贴到某个地方做记录。现在如果 Claude Code 直接连了 Sentry 的远程 MCP 服务器,你可以直接在终端里跟它说,帮我看一下昨天那个支付接口报的那个异常是怎么回事,Claude Code 就能去 Sentry 那边把对应的报错详情拉过来,结合你本地的代码一起分析,定位问题出在哪一行,甚至给你一个修复建议。这个过程省掉的不是什么惊天动地的大功能,但省掉的是大量琐碎的上下文切换,而这些切换恰恰是工程师每天精力消耗最大的地方之一。

第二个案例是 Linear。Linear 是一个项目管理和任务跟踪工具,很多技术团队用它来管理需求、跟踪 bug、安排迭代周期。文章里引用了 Linear 工程负责人 Tom Moor 的一段话,他说 Linear 的 MCP 集成把 Linear 的项目和任务直接带进了 Claude Code,有了来自 Linear 的结构化、实时的上下文,Claude Code 就能把任务详情和项目状态拉进来,工程师在做规划、写代码、管理任务这几件事之间切换的时候,能够保持在状态里不被打断,标签页更少了,复制粘贴也更少了,软件做得更好,速度也更快。这段话其实点出了一个很实际的工作场景,我们平时写代码之前,往往需要先去看一下这个任务卡片到底写了什么需求,验收标准是什么,有没有相关的讨论记录。如果这一步要手动打开 Linear 网页去查,效率上多少都会打个折扣。一旦 Claude Code 能直接读到 Linear 里的任务上下文,你可以直接在终端里说,帮我看看当前这个任务的需求细节,然后让它结合需求直接开始写代码或者写测试,整个流程会顺畅很多。

讲到这里,我想多花一点时间,从更宏观的角度,谈谈这次更新背后真正的意义是什么。我个人认为,这次发布最值得关注的地方,不是某一个具体的功能点,而是它代表的一种产品哲学的转变。早期的 AI 编程助手,大部分能力都局限在它自己的训练数据和上下文窗口里,你给它什么文本,它就只能基于这些文本去生成答案。后来出现了工具调用的能力,AI 可以执行一些预先定义好的函数,但这些函数往往还是需要开发者自己手写集成代码,每接入一个新工具,都要重新写一遍适配逻辑。MCP 协议加上远程服务器这套组合,实际上是把"接入新工具"这件事,从一次性的、定制化的开发工作,变成了一次标准化的配置操作。这意味着整个生态系统可以以更快的速度扩张,因为每多一个厂商发布自己的远程 MCP 服务器,所有支持 MCP 协议的 AI 客户端都能立刻享受到这个新的连接能力,不需要每个客户端都单独跟每个厂商谈合作、写适配代码。文章里也提到了这一点,叫做"不断壮大的服务器生态系统意味着新的能力在持续上线"。这种协议层的标准化,往往是一个生态系统能不能真正起飞的关键,我们在互联网历史上见过很多次类似的故事,比如说 HTTP 协议让网页可以互相链接,RSS 协议让内容可以被订阅聚合,这一类标准化协议出现之后,往往会催生出一轮生态爆发。MCP 在 AI 应用和外部工具之间扮演的角色,跟这些协议的历史作用是相似的。

现在我们站在二〇二六年年中往回看,这篇文章发表的时候,其实正处在 MCP 协议刚刚开始被行业广泛接受的早期阶段。那时候 MCP 协议本身才推出没多久,远程 MCP 服务器这个能力,可以说是这个协议从"概念验证"走向"生产可用"的一个重要节点。在那之后,我们看到了越来越多的工具厂商开始提供官方的远程 MCP 服务器,MCP 目录这种东西也逐渐成型,变成了一个大家可以浏览、挑选合适服务器的地方。这条演进路径,跟我们前面讲的"基础设施外包"逻辑是一致的,先有协议标准,然后有早期的本地实现做验证,接着头部厂商开始提供官方托管的远程服务,最后形成一个生态目录,方便用户发现和选择。如果你今天去看 Claude Code 或者其他支持 MCP 的工具,你会发现远程 MCP 服务器已经变成了一个相当常见的选项,这说明这篇文章里描述的方向,基本上得到了验证,而且发展得比较顺利。

接下来我们聊聊实践层面的东西,也就是说,如果你是一个正在用 Claude Code 或者类似工具的工程师,这个功能对你今天的日常工作意味着什么,你应该怎么上手,有哪些坑需要提前避开。

第一步,你要先搞清楚你团队日常依赖的工具里,哪些已经提供了官方的远程 MCP 服务器。常见的候选名单包括项目管理工具,比如说 Linear 或者 Jira 这一类,错误监控工具比如说 Sentry,知识库和文档系统,以及一些云服务厂商自己的控制台。一般来说,这些厂商如果支持远程 MCP,会在自己的官方文档里给出一个具体的服务器地址,你要做的就是把这个地址按照 Claude Code 的配置方式填进去。

第二步,完成首次授权。因为采用的是 OAuth 机制,你点击连接之后,会跳转到对应服务的登录页面,正常登录授权就行,这一步不需要你手动去申请什么密钥,也不需要把任何敏感信息粘贴到配置文件里。授权完成之后,Claude Code 会记住这次授权,后续使用的时候不需要你重复操作。这里有一个小提醒,授权的时候你应该留意一下这个服务请求的权限范围到底有多大,比如说它是只读访问你的任务列表,还是同时还能修改或者删除内容,尽量遵循最小权限原则,只授予真正需要用到的权限。这不是文章里直接写出来的内容,但是从安全实践的角度,我觉得这是任何使用 OAuth 授权的场景都应该养成的习惯。

第三步,在实际对话里,学会主动告诉 Claude Code 你想让它去查什么。远程 MCP 服务器接入之后,并不代表 Claude Code 会自动猜到你想要哪条信息,你仍然需要用自然语言明确表达你的意图,比如说让它去查某个具体的任务编号,或者某个时间段内出现的某类报错。把这种习惯培养起来,你会发现原本需要打开好几个标签页才能拼凑出来的上下文,现在可以在一次对话里就拿到。

还有几个需要留意的坑。第一个是,不同厂商提供的远程 MCP 服务器,功能完备程度可能差别很大,有些厂商可能只开放了部分资源的读取权限,具体能做到什么程度,最好先去看一下对应厂商的文档说明,不要凭空假设它什么都能做。第二个是,如果你的公司对数据安全有比较严格的合规要求,在接入任何远程 MCP 服务器之前,最好先跟你们的安全团队确认一下,这种连接会不会涉及敏感数据出公司内网的问题,毕竟远程服务器意味着数据要经过厂商的云端,这跟纯本地部署在数据流向上是有本质区别的。第三个是,因为这套机制依赖于厂商持续维护他们的远程服务器,如果厂商那边服务出现故障或者下线,你这边的集成也会跟着受影响,这是把基础设施外包出去必然伴随的一种依赖关系,你需要对这种依赖有清醒的认识,而不是把它当成绝对可靠、永远在线的东西。

好,我们来给今天这期节目做一个收尾。这篇文章核心讲的是,Claude Code 在二〇二五年六月加入了对远程 MCP 服务器的支持,本质上是把原本需要本地部署维护的工具连接层,变成了一种只需要填地址、做一次 OAuth 授权就能用的标准化能力,背后依托的是 MCP 这套协议,目的是让 AI 编程助手能够更顺畅地连接到 Sentry、Linear 这类外部工具和数据源,减少工程师在不同应用之间反复切换、复制粘贴上下文的负担。我个人认为,这次更新真正的价值,不在于某一个具体集成做得多么花哨,而在于它把"接入新工具"这件事变成了一种规模化、标准化的过程,这种协议层的统一,往往才是一个生态系统能不能真正繁荣起来的关键所在。今天我们站在一年后再回看这篇文章,可以说它描述的方向已经基本成为了行业的常态,这也提醒我们,很多看似不起眼的基础设施层面的改动,反而可能是后续一连串生态爆发的起点。好,这一期就到这里,我们下期再聊,再见。
