## Script

今天我们来聊一篇来自 Anthropic 官方博客的文章，英文标题是 New in Claude Managed Agents: self-hosted sandboxes and MCP tunnels，我把它翻译成中文大概是"Claude 托管智能体新功能：自托管沙箱与 MCP 隧道"。这篇文章发表于二零二六年五月十九日，到今天正好过了大约一个月。对于 AI 这个领域来说，一个月前的内容仍然是相当新鲜的，这两个功能可以说是刚刚落地、墨迹未干。

今天这期节目，我特别想推荐给正在为公司做 AI 工程化落地的同学们听，或者你在一家 AI 创业公司里负责给企业客户做产品，这期节目几乎是为你量身定制的。因为 Anthropic 这次发布的两个功能，直接打穿了长期以来阻碍企业用 AI 智能体的那道墙——数据安全边界的问题。

好，我们从背景开始说起。

过去一年多时间里，AI 智能体这个概念几乎成了所有 AI 从业者嘴里最热的词。agent、agentic system、multi-agent，这些词你在技术社区里到处都能听到。智能体的核心理念其实并不复杂：让 AI 不只是回答你的问题，而是真正去执行任务。它能自己规划步骤，自己调用工具，自己写代码然后执行，自己处理错误，一直到把你交代的目标完成为止。

Claude 的 Managed Agents，也就是我们今天要讲的托管智能体，是 Anthropic 在这个方向上的一套完整产品。你给它一个高层次的目标，它帮你把整个智能体的执行循环都管好——任务编排、上下文管理、工具调用、错误恢复，整个流程。这听起来很美好，对吧？但问题来了。

当企业开始认真考虑把 AI 智能体用于真正的业务场景时，马上就遇到了一道绕不过去的坎：数据安全。你想一想，如果你要让一个 AI 智能体帮你的公司做工作，它需要访问什么？它需要访问你们的代码库、内部文档、客户数据库、业务系统。这些东西，没有一个在公网上，全都在你公司防火墙后面的私有网络里。而且即便不考虑网络访问权限的问题，你能把客户数据发到 Anthropic 的服务器上去跑吗？在金融、医疗、政府这些行业，这在合规层面几乎是不可能的。你可能面临监管处罚，可能违反和客户签的数据处理协议，严重的话会有法律风险。

这就形成了一个很尴尬的局面：AI 智能体的能力让所有人眼红，但真正的企业数据和真正的企业业务，就是没办法接进去。技术演示场景很漂亮，但一到生产落地，就卡在数据安全这道关。技术能力的前沿和企业实际可用之间，有一道很深的鸿沟。

Anthropic 这次发布的两个功能，就是专门来填这道鸿沟的。一个是自托管沙箱，解决执行环境的问题；另一个是 MCP 隧道，解决私有网络访问的问题。我们一个一个来讲，讲深讲透。

先讲第一个功能：自托管沙箱，英文叫 self-hosted sandboxes。

要讲清楚这个功能，我们先得搞清楚沙箱在 AI 智能体里面扮演什么角色。当一个 AI 智能体说"我要执行这段代码来完成你的任务"，这段代码不能直接在宿主服务器的根环境里跑，必须跑在一个隔离的执行容器里，也就是沙箱。沙箱确保了智能体能完成任务，但做不了沙箱边界之外的事情，不会影响到宿主系统，也不会把什么东西带出去或带进来。你可以把沙箱理解成一个密封的实验室：科学家可以在里面做任何实验，但实验的过程和产物被严格限定在这个房间里。

在这次更新之前，Claude Managed Agents 只有一种沙箱：Anthropic 自己的沙箱。也就是说，所有的工具执行都发生在 Anthropic 的基础设施上。这对个人开发者来说没什么问题，但对企业来说，麻烦很多，而且麻烦是结构性的。

我来帮你把这些问题梳理清楚，这样你才能真正理解这个新功能解决的是什么。

第一个问题是数据主权。当你的智能体在 Anthropic 的沙箱里跑，它操作的文件、读取的代码、执行的命令，这些都在 Anthropic 的服务器上发生。对于很多企业，尤其是有严格数据主权要求的组织，数据不能离开自己掌控的环境，这是一条不能触碰的红线，不是技术问题，是合规和法律问题。

第二个问题是网络隔离。企业最有价值的数据和系统，那些内网数据库、内部代码仓库、私有 API 服务，都在防火墙后面，只能从内网访问。Anthropic 的云端沙箱根本看不到这些服务，所以智能体就没办法真正深入企业的核心资产，它只能处理那些已经暴露出来的东西，价值大打折扣。

第三个问题是资源控制。如果你的智能体任务是计算密集型的，比如要编译一个几十万行代码的大型项目，或者要做大量图像生成，你需要精确控制分配给它的 CPU 和内存，甚至可能需要 GPU 加速。共享的云端资源在这些场景下性能是不可预测的，你拿到多少算力、什么时候被限速，完全不在你的掌控范围内。

自托管沙箱就是为了同时解决这三个问题而来的。它的核心架构思路是一个很精妙的关注点分离：把智能体的"大脑"和"手脚"分开放在不同的地方。Anthropic 这边继续负责"大脑"部分，也就是任务理解、步骤规划、上下文管理、错误恢复，这套编排逻辑在 Anthropic 的基础设施上跑。但是"手脚"，也就是实际的工具执行、代码运行、文件操作，移到你自己配置的环境里来。

我觉得有个类比能帮你快速建立直觉。你有没有用过远程工作的模式？想象一下，你公司雇了一个远程工程师，他每天通过网络来协作。他的思考、规划、和你的沟通，这些是他自己脑子里的事，在他那边发生。但他执行任务用到的工具、代码库、开发环境，全都是你公司配置的企业设备，受你公司的 IT 管理，你的安全策略在他的工作环境上起作用。Anthropic 的智能体就像这个远程工程师——大脑在 Anthropic 那，编排逻辑也在那，但干活用的工具和执行环境在你这里，你掌控。

这个架构变化带来的好处非常直接。你的网络策略、审计日志、安全工具全都已经就位，智能体做任何操作都在你的监控下。你的文件、代码、私有依赖包，根本不会离开你的网络边界。你完全控制沙箱的计算资源，CPU 多少、内存多大、用什么运行时镜像，全都是你来定，而不是用公共的共享资源，为计算密集型任务保驾护航。

好，架构讲清楚了，我们来看文章里提到的几家支持的沙箱提供商。Anthropic 在这次更新里，并没有试图自己解决所有事情，而是采用了开放合作伙伴的策略，和四家专门做沙箱基础设施的平台合作，让企业可以根据自己的需求来选择。这四家分别是 Cloudflare、Daytona、Modal 和 Vercel。

我们一家一家来说。

第一家是 Cloudflare。说到 Cloudflare，大家第一反应可能是 CDN 和 DDoS 防护，但 Cloudflare 这几年在开发者平台这个方向上做了大量投入。Workers、Pages 这些你可能都知道，沙箱是这个布局里相对新的一块。

Cloudflare 的沙箱是用微虚拟机和更轻量的隔离机制来跑的，这种架构的好处是启动快、可以大规模运行，适合需要快速启动大量沙箱实例的场景。

但 Cloudflare 最让我觉得有意思的，其实是它在网络控制这块的能力，这部分是它的独特优势。第一个能力是零信任密钥注入，你可以把 API 密钥、数据库凭证等敏感信息安全地注入到沙箱里，而不是写死在配置文件里或者通过明文环境变量传，这在安全设计上是更规范的做法。第二个能力是可定制的出站代理，你可以配置代理来审计所有出站的网络请求，或者把请求重路由到不同的目标，甚至修改请求内容，这对合规要求严格的企业来说是非常有价值的功能，你能精确知道智能体在网络层面做了什么，什么时候访问了哪里。第三个能力是通过 Cloudflare 自己的网络来连接内部服务，这让沙箱可以在受控的方式下触及私有网络资源。

文章里提到了一个真实的用户案例：有家叫做 Amplitude 的公司，他们在用 Managed Agents 加 Cloudflare 构建一个叫做 Design Agent 的内部工具，用来生成符合品牌规范的生产级 UI 界面和营销设计图。他们选这套方案的理由，就是更好的可观测性和控制力。这很典型——做内部工具的团队，需要能清楚地看到智能体在做什么，出了问题能追查，合规要求要能留记录，Cloudflare 在这几点上都给了很好的答案。

第二家是 Daytona。Daytona 的定位和其他沙箱提供商有一个很明显的不同：它特别强调有状态的、长时间运行的沙箱。大多数沙箱环境是无状态的，一次任务结束就销毁环境，下一次是全新的。但 Daytona 的沙箱可以持续运行好几个小时，状态一直保留，你可以把它想象成一台云上的虚拟开发机，而不是一次性的执行容器。

这有什么实际意义呢？我来帮你想一个具体的场景。假设你让 AI 智能体去做一个复杂的软件开发任务：先克隆代码库，然后安装项目依赖，接着理解现有代码结构，然后做功能修改，然后跑测试，最后提交代码。这个完整的过程可能需要跑几个小时。如果沙箱是无状态的，每次工具调用之间的状态都消失了，上一步装好的依赖包、改好的文件、跑出来的中间结果，全都没了，智能体几乎没办法执行这种跨越很长时间的连贯任务。但 Daytona 的有状态沙箱就能支撑这种场景。

Daytona 还有两个细节值得特别提一下。第一个是访问方式的灵活性：你可以通过 SSH 连接进正在运行的沙箱，或者通过认证后的预览 URL 来访问沙箱里运行的 Web 服务，这在你想监控智能体的运行状态或者调试的时候非常实用。第二个是暂停和恢复的能力：如果你的任务工作流里有需要等待的节点，比如需要人工审核才能继续的步骤，你可以把沙箱暂停，状态完整保留下来，等条件具备了再恢复，不需要从头重建环境。这对于有人工干预节点的智能体工作流来说是非常实用的特性。

第三家是 Modal。Modal 这几年在 AI 开发者社区里知名度越来越高，它的定位非常清晰：专门为 AI 工作负载构建的云平台。这和那些做通用云计算的平台是有区别的，它的底层设计是围绕 AI 场景来优化的。

Modal 的沙箱和它的函数计算、存储、网络基础原语共用同一套底层。这意味着如果你在 Modal 上构建系统，沙箱不是一个孤立的组件，它和 Modal 的其他基础设施是原生集成的，你可以无缝调用 Modal 的对象存储、托管数据库、消息队列等等，整套东西可以在一个统一的平台上构建出来。

Modal 有三个技术亮点，都很有说服力。第一是亚秒级冷启动，这来自他们自研的容器运行时，可以在任何镜像上实现不到一秒的启动时间。在 AI 智能体的场景里，如果每次工具调用都要等待几秒钟的容器启动，整个任务的执行体验会很差，而且时间浪费在等待上非常可惜。亚秒级启动让这个问题基本消失。第二是大规模并发能力，可以扩展到几十万个并发沙箱同时运行。对于需要大量并行智能体的场景，比如同时批量处理大量用户任务，这个能力的意义就非常重大了，你不需要自己去管扩缩容。第三是按需的 CPU 和 GPU 资源，你可以在任务需要时直接申请 GPU，对于智能体需要做模型推理、图像生成、视频处理等计算密集型操作的场景，这是其他沙箱提供商很难直接提供的能力。

第四家是 Vercel。Vercel 在前端开发者里知名度非常高，Next.js 的生态让它积累了一大批用户。文章里对 Vercel 沙箱的技术细节没有展开太多，但提到了一个很有代表性的用户案例：Rogo，一家面向机构投资者的 AI 平台。Rogo 在用 Managed Agents 加 Vercel Sandbox 来构建一个分析师智能体，处理他们的专有金融数据，安全性是选型的核心考量。机构金融数据的敏感程度属于最高级别，监管要求非常严格，能在私有沙箱环境里处理而数据不离开安全边界，对他们来说几乎是这个产品能不能合规上市的前提条件。

好，沙箱讲完了，我们来讲第二个重量级功能：MCP 隧道，英文叫 MCP tunnels。

在讲 MCP 隧道之前，我先简单介绍一下 MCP 这个协议是什么，因为这是理解隧道功能的基础。MCP 全称是 Model Context Protocol，中文叫模型上下文协议。这是 Anthropic 在二零二四年底推出的一个开放协议，设计目标是让 AI 模型连接外部工具和数据源的方式标准化。

你可以把 MCP 想象成 AI 领域的 USB 协议。在 USB 出现之前，每个外设厂商都有自己的接口规范，键盘是键盘的线，打印机是打印机的口，鼠标是鼠标的协议，搭配起来很麻烦。USB 出来之后，所有外设都用同一种接口，你只需要实现这一个标准，就能和任何支持 USB 的设备通信。MCP 做的是同样的事情：不管你要把什么工具或数据源接给 AI 用，数据库也好、搜索引擎也好、代码仓库也好、日历系统也好，只要你把它包装成一个 MCP 服务器，就能被任何支持 MCP 的 AI 模型调用，不需要为每个 AI 系统单独适配。

所以你现在能看到，整个生态里涌现出了大量的 MCP 服务器，把各种各样的服务包装成 AI 可以直接使用的工具。这个生态现在非常活跃。

但 MCP 服务器面临一个和自托管沙箱完全对称的问题：企业里最有价值的那些 MCP 服务器，往往运行在私有网络里，从外部根本够不到。你想把公司的内部知识库、内网数据库、工单系统、CRM、ERP 这些核心业务系统包装成 MCP 服务给 AI 智能体用，但这些系统全都在防火墙后面，只有内网才能访问。云端的 AI 智能体从外面根本看不到这些服务。

按照传统做法，要解决这个问题，你有几种选择，但每种都有明显的问题。

第一种是把内网服务直接暴露到公网上。这是最直接的，但也是最危险的。你等于给本来只对内网开放的系统开了一个公网入口，攻击面直接扩大，安全风险大幅上升，对大多数企业来说这是不可接受的。

第二种是在 Anthropic 的服务和你的内网之间建 VPN 连接。在技术上是可行的，但运维复杂度很高，你需要配置和维护 VPN 设施，需要双方的网络都有 VPN 基础设施，而且一旦 VPN 出问题，所有依赖内网资源的智能体功能就全挂了。可靠性和维护成本都是问题。

第三种是把内网数据同步一份到公网环境。比如把数据库内容定期同步到云端。这有数据一致性的问题，同步是有延迟的，智能体拿到的可能是旧数据。而且数据还是离开了内网边界，合规问题依然存在。

MCP 隧道给出了第四种选择，而且是最优雅的一种。它的工作原理是这样的：你在自己的私有网络里部署一个轻量级的网关组件，这个网关从内向外主动发起一个出站连接，和 Anthropic 的平台建立起一个加密的隧道。然后 Anthropic 这侧的智能体，就可以通过这个已经建立好的连接，访问你内网里的 MCP 服务器，进而访问它们背后的各种内部系统。

这里有一个关键的技术细节，是这个方案优雅的地方：连接是从内向外发起的。不是 Anthropic 的服务主动往你这里连，而是你内网的网关主动连出去。这意味着什么呢？你完全不需要在防火墙上开任何入站规则，不需要暴露任何公开的网络端口。从外部看，你的内网依然是完全封闭的。但通过这个已经建立的出站连接，Anthropic 侧可以把请求转发进来，访问你内网里的服务。

另外，整个传输过程是端到端加密的。从你内网的网关，到 Anthropic 的服务，整条隧道里的流量都是加密的，中间没有任何节点能看到明文内容。

我来打一个更接地气的比方帮你理解这个机制。你用过 ngrok 或者 frp 这类内网穿透工具吗？比如你本地跑了一个 Web 服务，想让外部的人访问，你可以用 ngrok 跑一个客户端，它连到 ngrok 的公网服务器，ngrok 服务器再把公网的请求通过这个连接转发进来，你不需要真正开放本地防火墙。MCP 隧道的原理和这个很像，只是 Anthropic 把这个机制直接集成到了 Claude 平台里，不需要借助第三方工具，安全性更高，稳定性和功能都由 Anthropic 来保障，还有管理界面来统一配置。

有了 MCP 隧道，你的 AI 智能体可以访问的范围就大大扩展了。内网数据库、私有知识库、内部 REST API、工单系统、代码仓库管理系统，只要你能把它们包装成 MCP 服务器，通过隧道就可以让云端的智能体调用，而这些服务本身完全不需要做任何网络开放的改动。

从运维管理的角度来看，MCP 隧道的配置是在 Claude Console 里由组织管理员来完成的，这有一定的治理层次，不是任何开发者都能随便配置的，有权限管控。另外，MCP 隧道同时支持 Managed Agents 和 Messages API，也就是说无论你是用 Anthropic 的托管智能体方案，还是自己直接调用 API 来构建智能体应用，都可以享受到这个功能。

讲完两个功能的原理，我想帮你把它们串联起来，想象一个完整的企业应用场景，这样你能更直观地感受到这次更新的价值。

假设你是一家中型互联网公司的 AI 工程团队，你们想构建一个 AI 编程助手，能真正深度集成到开发流程里。不是那种只会回答技术问题的聊天机器人，而是一个真的能干活的助手：它能读你们的内部代码仓库，查 Jira 里的工单，访问 Jenkins 的构建日志，然后根据这些真实的业务上下文，帮工程师分析问题、生成代码改动、甚至直接提交 pull request。

在这次更新之前，你面临的困境是：代码仓库在内网，Jira 在内网，Jenkins 在内网，这些最有价值的上下文根本没办法让 Claude 的智能体看到，你只能用一些公开的、脱敏的数据来演示，而不是真正地落地。

现在有了这两个功能，你的技术方案就可以是这样的：在内网部署几个 MCP 服务器，分别把代码仓库访问、Jira 工单查询、Jenkins 日志读取这几个功能包装成 MCP 工具。然后部署 MCP 隧道网关，让这些 MCP 服务器可以被 Claude 的智能体通过隧道调用到。同时，配置自托管沙箱，让智能体生成的代码片段在你们自己的基础设施里执行验证，而不是跑在 Anthropic 的云端。整个智能体的工具执行和数据访问，都在你们公司的网络边界内完成，代码没有外泄，数据主权完整保留，但智能体又能真正深度访问你们的核心业务系统。这才叫真正落地，而不是演示。

好，现在我们来聊实践应用的部分，这两个功能对你的实际工作意味着什么，你应该怎么上手，要注意哪些坑。

首先，谁应该最关注这次更新？

如果你是在企业内部做 AI 工具的工程师，这次更新意义相当重大。以前数据安全的顾虑让你根本没办法把核心业务数据接入 AI 智能体，你只能在边缘场景里玩，核心业务动不了。现在这个技术障碍在很大程度上被移除了。你可以认真规划一下，公司里哪些内部系统可以通过 MCP 服务器来接入，哪个沙箱提供商更符合你们的技术栈和安全要求，然后做一个小范围的试点。

如果你在 AI 创业公司里，给企业客户做 AI 产品，这两个功能是你销售过程中的一把新钥匙。企业客户最常见的顾虑就是数据安全，以前这是你绕不开的痛点，只能用各种方式解释或者规避。现在你可以直接告诉客户：数据处理全在你们自己的基础设施里，工具执行不会离开你们的网络边界，私有系统通过隧道连接不需要开公网口。这会大大降低客户的顾虑，加速销售周期。

如果你是独立开发者或者小团队，自托管沙箱对你可能没有太迫切的需求，因为 Anthropic 自己的沙箱对你来说已经够用了。但 MCP 隧道还是有一定价值的。如果你在本地跑了一些 MCP 服务，比如你本地的文件系统、你自己的数据库实例，想把它们接给 Claude 使用，官方支持的隧道方案比自己折腾内网穿透工具要可靠得多，而且不需要维护额外的第三方基础设施。

如果你是平台工程师或者 DevOps，这两个功能会给你带来一些新的工作项，同时也是新的技术机会：帮业务团队选型和搭建沙箱环境，设计和维护 MCP 隧道网关，规划相应的监控体系和告警机制，制定沙箱资源的计费和配额策略。这些新需求会在未来几个月里陆续落到你们平台团队身上，提前了解是很值得的。

在选择沙箱提供商的时候，技术选型怎么做？我来帮你梳理几个关键维度。

如果你的任务是短暂无状态的，需要极低延迟启动和大规模并发，而且有 GPU 需求，Modal 是最强的选择。亚秒级冷启动和几十万并发沙箱的能力，在这两个维度上是目前最领先的，专门为 AI 工作负载设计的平台对这类需求有天然优势。

如果你的任务需要长期运行，状态需要在工具调用之间保留，任务可能有暂停和恢复的需求，那 Daytona 的有状态沙箱更合适。它更像一台远程开发机，而不只是一次性执行容器。复杂的多步骤开发任务，或者有人工审核节点的工作流，用 Daytona 的体验会更接近熟悉的工作方式。

如果你对网络流量有细粒度控制的强需求，比如你的合规要求需要审计所有出站请求，或者你需要把智能体的网络访问严格限定在特定范围内，Cloudflare 的出站代理和零信任密钥注入是独特的优势，其他提供商不一定能提供同等粒度的控制。如果你本来就在用 Cloudflare 的生态，集成起来会非常自然。

如果你的整体技术栈是以 Vercel 为核心，前端生态是重心，Vercel 的沙箱和你现有的部署流程整合起来会减少很多运维复杂度，不需要跨平台管理。

还有一些需要提醒你注意的坑，我来一一说清楚。

第一个坑：自托管沙箱只是把执行环境移过来了，安全不会自动到位。很多人以为用了自托管沙箱，安全问题就解决了，实际上只是把一部分控制权拿回来了。你还需要自己设计和实施沙箱内部的权限隔离，限制沙箱能访问的网络范围，配置完整的审计日志，以及对沙箱行为做监控和告警。Anthropic 的编排层不会帮你管这些，这是你自己的责任范围。

第二个坑：成本估算要提前做好，不要等账单到了才发现超预期。自托管沙箱意味着要付沙箱提供商的费用，或者自己搭的话要付基础设施的钱。在规模化使用之前，一定要对不同场景的成本有清晰的预估。特别是 GPU 资源，在 Modal 这样的平台上，GPU 的按需使用费率是相当可观的，如果你的智能体任务频繁调用 GPU，成本会快速积累。

第三个坑：网络延迟是个新变量，而且可能比你预想的影响更大。当工具执行从 Anthropic 的基础设施移到你自己的环境，Anthropic 的编排层和你的沙箱之间的往返延迟会直接叠加在每次工具调用上。如果你的智能体任务需要几十次甚至上百次工具调用，每次工具调用多出几十毫秒，累计起来可能让整个任务的执行时间显著增加。在选择沙箱部署区域的时候，要考虑到和 Anthropic 服务节点的网络距离，尽量部署在延迟低的区域。

第四个坑：MCP 隧道网关是个新的单点故障，需要特别对待。MCP 隧道依赖你内网里部署的网关组件，这个网关如果不可用，所有依赖内网 MCP 服务的智能体功能就全部失效了。这是一个你引入的新的关键依赖，你需要给它做好监控，考虑高可用部署，并且设计好当网关不可用时系统如何优雅降级，而不是直接崩溃。

第五个坑：MCP 隧道目前还在研究预览阶段，意味着功能和接口可能还会变化。如果你现在就打算基于它构建生产系统，要做好接口变更的准备，保持跟踪 Anthropic 官方文档的更新。研究预览阶段的产品，功能稳定性和文档完整性都不如正式版，这是需要接受的现实。

最后说一个从发布节奏上能读出来的信号。自托管沙箱已经进入公开测试版，说明 Anthropic 认为它相对成熟了，基础功能是稳定的，可以放开给所有人用。MCP 隧道还在研究预览阶段，需要申请访问，说明它还在探索和打磨的早期阶段。如果你对 MCP 隧道有强烈的需求，研究预览阶段反而是个好时机去申请——参与的用户少，你能得到更多来自 Anthropic 团队的直接关注，你提的反馈也更容易影响到产品的最终形态，这是早期参与的隐形价值。

好，今天这期节目我们做个收尾。

这篇发表于二零二六年五月十九日的文章，宣布了 Claude Managed Agents 的两个关键更新。自托管沙箱把工具执行从 Anthropic 的基础设施移到了用户自己控制的环境里，同时解决了数据主权、私有网络访问、计算资源控制三个核心企业痛点，并且通过和 Cloudflare、Daytona、Modal、Vercel 四家伙伴的合作，提供了一个灵活的生态选择空间。MCP 隧道则通过一个从内向外发起的加密通道，让内网里的私有服务可以被云端 AI 智能体安全访问，不需要开放任何防火墙入站端口，端到端加密，同时支持托管智能体和 Messages API 两种使用方式。

我个人认为，这两个功能加在一起，是 AI 智能体真正进入企业生产环境的一个重要里程碑，意义比很多人第一眼看到时意识到的要深。Anthropic 在这里表现出了一个很成熟的产品策略：它没有试图自己包揽所有的基础设施问题，而是做了非常清晰的关注点分离，编排和智能的部分在 Anthropic 这里，执行环境和数据边界的控制权回到用户手里。这个设计不是在妥协，而是在理解企业需求之后做出的正确判断——你没办法要求企业把控制权交出来，你只能把控制权还给他们。同时通过开放的合作伙伴生态，给不同规模、不同技术背景、不同场景需求的企业都提供了可以匹配的选项。这种架构上的谦逊和生态上的开放，反而可能让这套方案在企业里走得更稳、更远。
