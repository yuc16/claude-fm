## Script

今天这期节目，我们来聊一篇 Anthropic 数据团队刚刚发布的干货文章。

文章英文标题是 How Anthropic enables self-service data analytics with Claude，中文可以翻译成「Anthropic 如何用 Claude 实现自助数据分析」。这篇文章发表于二〇二六年六月三日，是 Anthropic 内部数据科学和数据工程团队五位工程师联名写的实战复盘。

为什么值得专门做一期节目聊这篇文章？原因有三。第一，它讲的不是理论，而是 Anthropic 自己内部真实跑起来的系统，有具体数字支撑。第二，它非常诚实，把走过的弯路、失败的实验都公开说了，这在科技公司的对外内容里并不多见。第三，它解决的问题是几乎每个数据团队迟早都要面对的：能不能让非技术同事用自然语言问 Claude 数据问题，直接从数据仓库里拿到可靠的答案？

结论先说：Anthropic 内部目前有 95% 的业务分析查询都由 Claude 自动处理，综合准确率约为 95%，在某些特定领域能达到 99% 左右。但在他们引入精心设计的 skill 体系之前，Claude 的准确率只有 21%。从 21% 到 95%，这中间发生了什么，这才是这篇文章最核心的价值所在。

好，我们从背景开始讲起。

所谓自助数据分析，就是让业务侧的同事，比如产品经理、市场同学、销售运营，不需要找数据分析师帮忙，自己能查数据、自己能得出结论。这是每个数据团队的终极目标，也是每个数据团队长期以来的头疼问题。

传统的解法主要有两条路，但两条都走不通。

第一条路是把数据模型做宽做平。把复杂的数据库表结构，通过宽表和去范式化的方式，简化成非技术人员也能理解的格式，让大家直接拖拽工具查询。这条路在早期往往很有效，但随着业务规模扩大，问题就来了：各个部门开始各自维护自己的宽表，定义开始出现分歧。市场部的「活跃用户」和产品部的「活跃用户」用的是不同口径，两个人对着同一个数字开会，谈的根本不是同一件事。最后的结果是指标泛滥，没有两个人看的是同一份数据，数据反而变成了争议的来源而不是决策的依据。

第二条路是做精细化权限控制，给不同用户群提供不同的数据入口，每个部门有专属仪表盘。这条路的问题是覆盖不了长尾。每次有人问一个没有现成仪表盘的问题，还是得找数据分析师，还是得排期。结果是仪表盘越来越多，但数据孤岛也越来越严重，每个团队维护自己的一套，相互之间还不打通。

大语言模型的崛起看起来提供了第三条路：让 agent 直接连接数据仓库，用户用自然语言提问，Claude 帮你写 SQL、跑查询、返回答案。听起来完美，对不对？Anthropic 的团队一开始也是这么想的。

但他们很快发现了一个严重的问题。简单地把 Claude 接到数据仓库，然后让 agent 自由执行，会产生一种「虚假的精确感」。用他们原文的话来说，一开始那种终于不用处理临时查询的解放感，很快变成了恐惧。因为这种方式把用户和底层的数据基础设施、文档、专业知识全都隔开了。而正是这些文档和专业知识，过去会引导用户去用正确的数据集。把这层隔开，结果就是 agent 可以流畅地生成 SQL 并返回一个看起来很像答案的数字，但没有人知道那个数字是不是真的对。

所以问题的核心，并不在于 Claude 会不会写 SQL。Claude 写 SQL 的能力是完全够的。问题在于，在茫茫数百张甚至数千张表、数百万个字段里，Claude 能不能找到那个「正确的、唯一的、最新的」数据来源，并且用正确的方式使用它。

这就引出了文章最核心的一个观点：数据不是软件。

这个对比非常值得细细展开讲。我们想想代码生成这件事，它天然有很多纠错机制。你让 Claude 帮你写一个函数，写错了会跑不过测试，或者直接报错，模型可以看到错误信息，然后迭代修改。代码有类型签名、有 README、有文档字符串，给模型提供了大量可以参考的上下文。而且对于同一个编程问题，往往有很多种正确的写法，模型的创造性在这里是优势。

但数据分析完全不同。一个业务问题，往往只有一个正确答案，用的是一个正确的数据源。没有单元测试告诉你用错表了，也没有报错信息说你的过滤条件不对。更要命的是，用了一张相似但不对的表、用了一个过期的指标定义，得出来的答案看起来很像真的，格式对、数量级对，没有专业知识的人根本发现不了问题。这种错误才是最危险的，因为它会被当成正确的答案往上传递。

所以 Anthropic 把数据分析的核心挑战归结为一句话：能不能把用户的问题，映射到数据模型里具体的、最新的实体，并且知道如何正确使用它们。如果能做到这件事，后面生成 SQL 和执行查询其实都是相对简单的事。

基于这个洞察，他们识别出了造成绝大多数错误的三种失败模式。

第一种，叫概念与实体的歧义。你可能觉得「活跃用户数」是个简单问题，但在真实数据库里，这个问题可能对应几十种不同的计算方式。什么样的行为算活跃？登录算不算？完成某个核心操作才算？欺诈账号要不要排除？时间窗口是七天还是三十天？还是自然月？一个数据模型里可能有几百张表、数百万个字段，agent 很容易在一堆相似的候选项里选错那个看起来最接近但其实不是「唯一正确」的那一个。

第二种，叫数据陈旧。业务在变，表结构在变，指标定义在变。你今天给 agent 的文档和知识，明天可能就过期了。最可怕的是，过期的知识不会产生明显的报错，它会返回「看起来差不多但其实不对」的答案。Anthropic 提到他们曾经观察到系统上线一个月后准确率从 95% 下滑到了 65%，原因就是没有重视 skill 文档的持续维护。

第三种，叫检索失败。有时候正确的信息确实存在于数据模型里，也有合理的标注，但面对海量的搜索空间，agent 就是找不到它。你可以想象一下，让一个人在没有目录索引的巨大图书馆里找一本书，找到了是运气，找不到才是常态。

这三种失败模式，是理解 Anthropic 整个解决方案架构的关键。他们搭建的智能分析技术栈，每一层都是针对这些失败模式来设计的。整个栈分四层：数据基础、真相来源、skill 技能、验证。我们逐层来讲。

第一层叫数据基础，主要针对的是概念与实体的歧义问题，也兼顾数据陈旧问题。

Anthropic 的核心逻辑是：如果一个概念在你的数据仓库里只能解析到一张权威表，而不是四十张看起来差不多的表，那歧义问题在 agent 开始搜索之前就已经消失了。所以他们做的第一件事是建立规范数据集，也就是 canonical datasets。

他们的描述是这样的：最常见的失败，是 agent 无法把「产品 X 的收入」这样一个概念，映射到唯一正确的表、列和指标定义，原因通常是存在多个看起来差不多但实现上有细微差别的候选项。解决方法是建少量、高度治理的逻辑模型：精心筛选一小组规范的单一数据源数据集，明确所有者，确保可被发现，然后把那些几乎重复的表积极地下线、废弃掉。物理层面的汇总表和缓存表仍然可以存在，但它们应该机械地从这些规范模型派生出来，而不是作为独立的替代选项并排存在。目标是：当 agent 搜索一个概念的时候，只找到一个受治理的答案。

第二件事是强制执行标准。Anthropic 发现，数据治理只有在三个层面同时执行才能持续下去：工具层面，让 agent 在结构上必须优先使用规范数据源；持续集成层面，绕过规范数据源的代码变更在代码审查时直接被拦截；还有一个组织授权层面，下游团队要么基于治理层来构建，要么解释为什么不这样做。治理如果缺乏执行，很快就会退化回「多个候选项」的老问题。

第三件事是把所有数据相关的产出物放在同一个代码仓库里，也就是产出物的同存。建模代码、语义层、参考文档、规范仪表盘定义，全部放在一个仓库，持续集成检查跨层完整性。如果一个建模变更会破坏下游仪表盘或者让已文档化的指标失效，持续集成会报出来，修复在同一个 pull request 里完成。这是他们对抗数据陈旧问题的第一道防线，因为改动数据模型的那个人，同时也要更新描述它的文档。

第四件事是把元数据当成一等公民。他们用了一个很好的类比：代码仓库之所以对 AI 模型友好，是因为有 README、类型签名、文档字符串，代码本身是可读的。你的数据仓库同样可以做到这样，但前提是列描述、表描述、规范指标定义、数据粒度文档、有效值范围、数据血缘、所有权信息，这些要和代码转换逻辑本身用同样的严格度去维护。

第二层叫真相来源，也就是 sources of truth，主要针对概念与实体的歧义问题，同时部分解决检索失败的问题。这一层是 agent 用来在数据模型里导航的参考界面，按信任度从高到低有四个子层。

最高信任度的是语义层，也就是编译好的指标和维度定义。如果一个问题能直接映射到语义层里已定义的指标，agent 调用一个函数就能拿到一个数字，而且这个数字和公司里所有其他地方产出的是同一个数字。Anthropic 在 skill 指令里强制要求 agent 每次都优先使用语义层，而不是直接写原始 SQL。

这里有一个他们尝试了但失败的做法值得专门说一下。他们试过用大语言模型自动从原始数据表和历史查询日志里生成指标定义，来快速启动语义层。结果呢？产生了一大堆看起来很合理的定义，但这些定义把他们正努力消除的那些歧义原封不动地编码了进去，跟精心人工筛选的小型语义层相比，在评测集上的表现反而更差。所以他们的结论是：可以让 Claude 起草文档、生成描述，但指标的定义这件事必须由人来把关和审核。

第二个子层是数据血缘和转换图谱。当语义层覆盖不到某个问题的时候，数据血缘和基于引用次数的表排名，可以让 agent 推断出哪些上游模型跟某个概念相关、哪些已经废弃、哪些共享相同的数据粒度。

第三个子层是查询语料库，也就是历史 SQL，来自仪表盘、工作区和历史分析项目。这里有一个文章里非常反直觉的发现，我要重点讲。直觉上，给 agent 访问历史查询的权限应该很有价值，因为那里面记录了每一个已经被正确回答过的问题。但他们实际测试的结果是：给 agent 直接访问数千个历史查询文件的权限，准确率的变化不到一个百分点。

他们很认真地追查了原因。首先，他们验证了 agent 确实在回答问题之前读了那些查询，而不是跳过了。其次，他们检查了那些仍然答错的问题，发现大约 80% 的情况下，正确的答案确实在查询语料库里。但即便如此，agent 看了那些 SQL，还是没有用上。

这个结论告诉他们：瓶颈不在于「有没有访问到历史信息」，而在于「有没有结构」，也就是说，agent 无法把一个新的问题映射到正确的历史查询上。结果，大量的非结构化 SQL 就算摆在面前，也是无效的噪音。这个单一实验改变了他们好几个月的产品路线图方向，他们开始把历史查询当成原材料，从中提炼出结构化的、按领域分类的参考文档，而不是让 agent 直接去翻那堆原始记录。

第四个子层是业务背景，也是大多数团队最容易忽略的一层。一个不了解你业务的 agent，会按照字面意思回答问题，而不是回答用户真正想知道的事情。它不知道「Q2 发布」指的是哪个具体产品，不知道两个团队对同一个术语有不同定义，也不知道这个问题被问是因为周四要开董事会。Anthropic 把公司知识图谱，包括内部文档索引、产品路线图、决策记录和组织架构，都导入了系统，让 agent 能够解析这些模糊的引用，并提出更好的澄清问题。

好，现在我们到了整个技术栈里最关键、价值最高的一层，也就是 skill。数字是最有说服力的：没有 skill 的时候，Claude 在他们的评测集上准确率只有 21%。加上设计合理的 skill 之后，综合准确率稳定超过 95%，在某些特定领域能达到 99%。从 21% 到 95%，这就是 skill 这一层带来的价值。

那什么是 skill？在 Claude Code 里，一个 skill 就是 agent 按需读取的一个 markdown 文件夹。如果真相来源层是 agent 的「陈述性知识」，也就是一个指标意味着什么，那 skill 就是它的「程序性知识」：先查哪个数据源、按什么顺序查、遇到歧义数据该怎么处理、一个完整的分析成果应该长什么样。

他们分享了几个重要的 skill 设计原则。

第一个是成对创建 skill。他们把 skill 分成两种类型，配合使用。一种叫知识 skill，扮演的是一个顶层路由器的角色，它告诉 agent：先试语义层，如果语义层没有覆盖，这里有大约三十个领域参考文件，描述了这个领域里相关的表、字段、连接方式和注意事项。知识 skill 是他们对抗检索失败问题的主要武器：与其让 agent 在数百万个字段的仓库里自由搜索，不如用路由器先把搜索空间缩小到几十个精心筛选的文件，再开始写查询。另一种叫分析 skill，它编码的是一个资深分析师的完整工作流程：先澄清问题，然后找数据源，然后跑查询，最后把结果送过一个对抗性审查子 agent 进行检验。分析 skill 里还打包了十几个可复用的分析模式，比如留存曲线分析、费率拆解、漏斗分析，这样常见的请求就不需要每次从头发明。

第二个是创建合适的参考文档。这些参考文档是专门为大语言模型检索而写的，不是给人读的。他们的参考文档包含：表的数据粒度、适用范围和排除条件；各种坑点的具体描述，比如「排除已知的免费邮件域名，但保留 anthropic.com 这样的自定义企业域名」；以及明确的路由触发条件，比如「如果问题涉及实验效果提升，不要用于原始事件计数」。文档描述的是原则和边界，而不是会随着业务变化而过期的处方式步骤。

第三个原则，也是他们踩了大坑才认识到的：把 skill 维护当成工程流程的一等公民，而不是事后加的文档工作。他们曾经亲眼看着系统上线一个月后准确率从 95% 滑到 65%，原因就是 skill 文档没有跟上数据模型的变化，而且当时没有机制能发现这个问题。

他们的解决方案是把 skill 的 markdown 文件和数据转换模型放在同一个仓库里。这样，改数据模型和更新对应的 skill 文档，就自然变成了同一个 pull request 里的事情。他们还加了一个代码审查钩子：任何涉及报表模型的代码改动，如果没有同时修改对应的 skill 文件，就会被钩子标记出来，提示审查者注意。最终结果是大约 90% 的数据模型 pull request 都包含了同步的 skill 修改。

第四个原则是跨所有界面提供一致体验。同一个 skill 必须在 Slack 里、在 IDE 里、在仪表盘工具里、在独立的 agent 会话里，给出一致的答案。他们的实现方式是保证只有一个规范的数据源，skill 变更合并之后自动同步：同步到 IDE 用的插件市场，同步到托管应用读取的云存储，以及通过 MCP 协议以资源形式提供服务。他们从一开始就设计了可移植性，没有硬编码仓库路径，没有特定界面的命名空间。

最后一层是验证，分成离线评测和在线验证两部分。

离线评测就是问答对。给 agent 一个问题，让它回答，跟标准答案比对。Anthropic 有两类：一类是基于仪表盘的评测，由 Claude 自动生成然后人工验证，覆盖最常见的业务问题；另一类是长尾评测，把业务背景喂给 Claude，让它生成这个领域里可能出现的各种问题，覆盖边缘情况。他们还有一个持续运行的机制：每次利益相关方在对话里纠正了 agent，那条纠正记录就是一个候选评测用例。

他们有几个评测的最佳实践值得记住。评测的基准数据要锚定到一个快照时间，防止底层数据变动导致评测本身过期。评测结果要像遥测数据一样存储，记录 skill 版本、代码提交哈希、模型 ID、每条断言的通过失败情况、token 数量和耗时，这样「这次改动到底有没有帮助」就变成了一个数据库查询，而不是靠感觉。每个领域有准确率门槛，负责人要等自己那个领域的评测集过关了，才能向业务侧宣布 agent 上线，这把问题的修复逼到了用户看到之前。离线评测的目标准确率应该接近 100%，不是说系统永远不会出错，而是说没有已知的覆盖盲点。

在线验证方面，他们有几个关键机制。一是对抗性审查，让另一个 Claude skill 来激进地质疑最终答案的所有底层假设。他们发现这让评测集准确率提升了 6 个百分点，但代价是 token 使用量增加了 32%，延迟增加了 72%。这个权衡没有标准答案，要根据你自己的场景判断。二是来源说明页脚，每个回答都附带一个说明，告诉用户数据来自哪个层次，是语义层、精心整理的参考表，还是原始探索性查询，底层数据有多新鲜，谁是这个数据模型的所有者。它不让答案更准确，但让用户能判断这个答案值多少信任。如果页脚显示「原始表，新鲜度未知」，那在转发给领导之前应该先验证一下。三是主动错误收集，一个定时运行的 agent 每隔几小时扫描利益相关方的 Slack 频道，找到那些「那个表用错了」「你漏了欺诈过滤器」这样的纠错语言，自动起草对相关参考文档的一行修改，开一个 pull request 发给领域负责人。修复路径故意设计得极其简单：编辑 markdown 文件，合并，自动同步到所有地方。这样领域负责人不需要花太多时间，修复的摩擦力被降到最低。

他们也坦诚地承认，有一种失败模式是他们目前没有完整解决方案的：沉默的错误。答案是错的，但看起来合理，没有人质疑就被用掉了。他们的缓解措施是来源页脚、对所有发给管理层的内容要求人工确认，以及每天对核心 KPI 做一次自动核查，跟已知正确的仪表盘数字比对。但这条战线他们没有宣称已经打赢了。

好，现在讲对你的实际工作意味着什么，你能上手用什么，要避开什么坑。

首先，Anthropic 自己建议从零开始的团队不要一上来就把所有东西都上。他们给的建议是先做三件事：建立几个规范数据集，准备几十个离线评测问答对，写一个薄的知识 skill。这三件事能拿到大部分的价值，其他的东西都是在这个基础上按需叠加的。

第二点，文档维护是关键，而且是需要工程化的关键。如果你只记住这篇文章里的一件事，我建议是这个：把 skill 文档和数据模型代码放在同一个仓库，让它们在同一个 pull request 里变更。这不是在说要写更好的文档，而是说要让文档的更新变成和代码更新一样不可分割的工程习惯。如果文档更新的摩擦力太高，它就不会发生，然后准确率就会悄悄下滑，就像 Anthropic 经历过的那样。

第三点，不要把历史 SQL 直接喂给 agent，而要把它当成原材料。这是文章里最反直觉、也最有价值的工程洞察。你可能已经有很多历史查询记录，直觉告诉你这是宝贵的数据，应该让 agent 参考。但他们的实验证明，非结构化的历史 SQL 对准确率几乎没有帮助。把精力花在从历史查询里提炼出结构化的领域参考文档，回报率要高得多。

第四点，离线评测要从一开始就建。很多团队在搭系统的时候完全没有评测，完全靠用户反馈来发现问题。但等到用户开始抱怨，已经是问题累积之后的事了。离线评测让你有一个客观的质量标准，每次改动都能看到数字的变化，也能在上线前发现覆盖盲点。

第五点，有几个问题应该先想清楚再决定做多复杂的系统。你的业务数据量有多大，数据模型有多复杂？如果你的数据量不大、消费者不多，很多 Anthropic 这篇文章里讲的机制可能是过度设计，反而会让系统难以维护。你的受众有多技术化？如果是给数据工程师用的，他们能识别答案是否合理，错误容忍度可以高一些；如果受众对底层数据完全没有了解，你就需要更严格的验证。你愿意为准确率花多少额外的计算成本？对抗性审查是个好东西，但 token 成本增加三成、延迟增加七成，对于实时场景来说可能是不可接受的代价。

最后，他们还提出了一个很有价值的关于时机的判断框架。AI 模型在快速迭代，今天需要复杂工程基础设施才能解决的问题，可能下一代模型直接就解决了。所以在构建某个复杂机制之前，值得问一下：这是在补一个真实的业务需求缺口，还是在弥补当前模型能力的一个暂时性局限？如果是后者，等模型进步来填补的成本，可能远低于自己搭一套系统的成本。

好，来做个收尾。

Anthropic 的数据团队用这篇文章分享了一套完整的方法论：从数据基础的治理，到语义层和血缘图谱的导航，到 skill 的结构化知识管理，再到离线评测和在线监控的闭环验证。这套东西让他们把 Claude 处理数据分析问题的准确率从 21% 提升到了 95%，并且让数据科学团队从大量重复性的临时查询里解放出来，可以专注于更有价值的因果建模、预测和机器学习工作。

我个人认为，这篇文章最值得珍惜的地方，不是那些成功的实践，而是那些「我们试了但没用」的实验记录。直接给 agent 喂历史 SQL 语料没有效果，让大语言模型自动生成指标定义反而适得其反，文档叠加到一定数量之后继续加反而变差……这些负面结果的记录，是非常宝贵的工程经验，让后来者不需要重复这些代价不菲的实验。这种诚实，是这篇文章最难得的地方。
