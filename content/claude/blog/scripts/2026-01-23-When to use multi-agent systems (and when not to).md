## Shownotes

本期我们聊一聊 Anthropic 官方博客的最新文章,什么时候该用多智能体系统,什么时候不该用。文章的核心态度挺出乎意料:大多数场景下,一个设计得当的单智能体远比想象中强大,多智能体系统只在三种特定情况下才真正划得来。我们会拆解这三种场景、一个反直觉的任务拆分原则,以及一个性价比很高的验证子智能体模式。

- 单智能体被严重低估:配置得当的单智能体能完成的任务,远超大多数开发者的预期
- 多智能体的隐藏成本:同等任务下通常要多花三到十倍的 token,协调开销可能反超执行本身
- 真正该用多智能体的三种场景:上下文污染需要隔离、任务可以并行探索、不同任务需要不同工具或专业知识
- 反直觉的拆分原则:按上下文边界分工,而不是按角色或工作类型分工,否则容易陷入信息层层失真的电话游戏
- 验证子智能体:用一个独立智能体专门做黑盒测试和校验,成本低、效果好,但要把验证标准写得足够死板具体
- 几个实用信号:工具数量超过十五到二十个时,先试试动态工具检索,而不是急着拆多智能体

## Script

大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是 When to use multi-agent systems and when not to,翻译过来就是,什么时候该用多智能体系统,什么时候不该用。这篇文章是 Anthropic 官方博客在二〇二六年一月二十三日发布的,作者是 Anthropic 的 Cara Phillips,还有几位同事参与了贡献,而且文章末尾特别说明,这是他们关于多智能体系统这个系列的第一篇,后面还会继续展开。算下来距离今天,也就是二〇二六年六月二十日,差不多过去了五个月。这个时间点其实挺微妙的,因为这半年正是整个行业从单智能体疯狂往多智能体架构涌的阶段,几乎所有在做 agent 产品的团队,都在讨论要不要上多智能体。这篇文章算是 Anthropic 内部踩过坑之后,给大家的一份冷静剂。

为什么值得听这一篇?因为它不是在鼓励你去构建复杂的多智能体系统,反而是在劝退。Anthropic 自己说,他们看到不少团队花了好几个月时间搭建复杂的多智能体架构,结果发现只要把单个智能体的提示词写得更好一点,效果就能打平,甚至更好。这跟市面上铺天盖地宣传多智能体多厉害的论调完全相反,所以我觉得特别值得拆开来好好讲讲。

我们先说说这篇文章想解决的问题,以及当时整个行业的背景。过去这一两年,随着 Claude 这类大模型的能力越来越强,工具调用也越来越成熟,很多团队开始尝试搭建所谓的 agent,也就是能够自主使用工具、自主决策、自主完成多步骤任务的智能体。再往前想一步,业内很自然地就会问,如果一个智能体已经很强了,那把好几个智能体拼起来,让它们分工协作,是不是能做得更强?于是多智能体系统就成了这段时间 agent 领域里最热的话题之一。

但 Anthropic 在文章里坦白说,他们作为这个领域走得比较靠前的公司,亲眼见证了大量团队走偏。具体的偏差是什么呢?就是把多智能体系统当成了万能药,觉得只要把任务拆成好几个角色,比如一个负责规划,一个负责写代码,一个负责测试,一个负责审查,效果自然就会更好。但实际跑下来,很多团队发现,每一次智能体之间交接任务,都会丢失一部分上下文信息,也就是丢失了之前积累的背景知识。结果就是,协调这几个智能体之间消息传递所消耗的算力和 token,也就是模型处理文字的最小单位,反而超过了真正用来干活的那部分。这篇文章就是在这种行业普遍踩坑的背景下,试图给出一个相对克制、相对理性的判断框架。

那我们先把基本概念理清楚。文章里说,多智能体系统指的是这样一种架构,多个大语言模型的实例,各自拥有独立的对话上下文,然后通过代码把它们协调起来一起工作。协调的模式有很多种,比如智能体蜂群模式,基于能力分工的系统,还有消息总线架构,但这篇文章重点讲的是一种叫做编排者和子智能体模式的结构。这个模式说白了就是有一个主智能体,我们可以叫它编排者,它负责生成并管理若干个专门干特定子任务的子智能体,是一种层级式的结构。Anthropic 说,这种模式协调逻辑最简单,特别适合刚开始尝试多智能体系统的团队,所以文章主要围绕它展开,其他更复杂的协调模式留给后续文章再讲。

接下来是这篇文章最核心的一个判断,可以理解为整篇文章的基调。文章直接说,今天行业里很多场景,其实单个智能体表现得会更好,但大家却习惯性地往多智能体方向去设计。Anthropic 特别强调,一个设计得当、配备了合适工具的单智能体,能完成的任务远远超出大多数开发者的预期。换句话说,很多人低估了单智能体的潜力,过早地引入了复杂度。

为什么多智能体会带来额外代价?文章给出了一个挺直观的解释。每多加一个智能体,就多了一个可能出错的环节,多了一套需要维护的提示词,也多了一个可能出现意外行为的来源。Anthropic 提到一个具体的实验现象,他们见过团队搭建了规划、执行、审查、迭代四个独立的智能体,结果发现每一次交接都会丢失上下文,协调所花的 token 比真正执行任务花的还多。根据他们的实测数据,完成同样的任务,多智能体方案通常会消耗单智能体方案三到十倍的 token。这个开销主要来自三个地方,一是不同智能体之间会重复存储相同的上下文信息,二是智能体之间需要互相发送协调消息,三是在交接结果的时候需要做总结归纳,这些总结归纳本身也要消耗算力。

听到这里你可能会想,那是不是说多智能体系统就是个坑,根本不该用?文章给出的答案更细致一些,它说多智能体架构只有在能解决单智能体真正解决不了的具体限制时,才有价值。换句话说,你不能因为多智能体听起来很酷、很前沿就去用,而是要先确认,你遇到的问题是不是单智能体确实搞不定的那种。

文章接下来给出了它的重头戏,也就是三种经过验证、多智能体系统确实能跑赢单智能体的场景,我们一个一个拆开讲。

第一种场景叫上下文污染。这里先解释一下上下文这个概念,你可以把它理解成大模型在对话过程中记住的所有信息,包括之前说过的话、查询到的数据、调用工具返回的结果等等。每个大模型都有一个上下文窗口的容量上限,而且即便没到上限,随着上下文里塞进去的信息越多越杂,模型的回答质量也会跟着下降,这就是注意力被稀释的效果。

文章举的例子特别接地气,是一个客服场景。假设你在做一个客服智能体,它既要查用户的订单历史,又要帮用户诊断技术问题。如果每次查订单,系统都把整段订单详情,可能两千个 token 以上的内容塞进对话上下文里,那这个智能体在尝试推理技术问题的时候,脑子里就会一直背着这一堆和当前问题无关的订单细节,推理质量自然会打折扣。这就好比你正在专心修一台电脑,旁边有人一直在你耳边念购物小票,你说能不分心吗?

文章给出的解法是,单独搭建一个订单查询子智能体,它拥有自己独立的上下文窗口,专门负责去查订单系统,处理那一大堆原始数据,然后只把提炼出来的关键信息,比如订单状态和购买日期,这种几十到一百个 token 的精华内容,传递给主智能体。主智能体的上下文因此能一直保持干净,只装着和当前任务真正相关的信息。

文章特别给出了适用上下文隔离的三个判断标准。第一,子任务会产生大量上下文,通常超过一千个 token,但这些信息对主任务大部分都是无关的。第二,子任务本身边界清晰,有明确的标准知道该提取哪些信息。第三,这种场景特别适合查找和检索类的操作,因为查出来的原始数据往往需要先过滤筛选,再交给主流程使用。

第二种场景是并行化。这个场景的核心逻辑是,让多个智能体同时跑,可以探索比单个智能体大得多的搜索空间。文章举的例子是 Anthropic 自家的 Research 功能,也就是一个面向研究类任务的产品功能。它的做法是,一个主智能体先分析用户的查询,把它拆解成好几个不同的研究方向,然后并行地派生出多个子智能体,每个子智能体独立去搜索自己负责的那个方向,搜完之后把提炼出的发现汇报回来。Anthropic 提到,这种多智能体搜索相比单智能体方式,在准确率上有显著提升,原因就是它能够覆盖更大的信息空间。

这里有个特别值得注意的细节,文章很诚实地指出,这种并行带来的覆盖面提升是有代价的。同样是消耗三到十倍的 token,而且更反直觉的一点是,虽然并行执行确实能比把所有工作串行做完节省总耗时,但因为总的计算量大幅增加了,多智能体系统整体跑下来,经常反而比单智能体系统花的时间更长。这一点我觉得特别值得重复一遍,因为很多人对多智能体有个误解,以为并行就等于更快,但文章很明确地说,并行化带来的核心收益是全面性,不是速度。如果你的场景是需要在一个很大的信息空间里搜索,或者从很多角度调研一个复杂问题,并行智能体确实能比受限于自己上下文窗口的单智能体覆盖更多内容,但你要拿更高的 token 消耗,以及往往更长的总耗时,去换这个更全面的结果。这是一个实打实的取舍,不是免费的午餐。

第三种场景是专业化。这个场景背后的道理是,不同的任务有时候真的需要不同的工具集合、不同的系统提示词,或者不同领域的专业知识。与其给一个智能体塞几十个工具,不如把工具按职责拆分给几个专门的智能体,各自只拿和自己任务相关的那一部分工具,这样可靠性会更高。

文章给了三个信号,告诉你什么时候该考虑工具专业化拆分。第一个信号是数量,当一个智能体拥有的工具超过二十个的时候,它在选择该用哪个工具上就会开始吃力。第二个信号是领域混淆,如果这些工具横跨好几个互不相关的领域,比如数据库操作、外部 API 调用、文件系统操作混在一起,智能体很容易搞不清楚当前任务到底属于哪个领域。第三个信号是性能退化,如果你往一个智能体里新增工具,结果导致它在原有任务上的表现反而变差了,这说明它已经达到了工具管理的容量上限。

专业化不只体现在工具数量上,还体现在角色设定的冲突上。文章举的例子是,客服智能体需要表现得有耐心、有同理心,而代码审查智能体则需要精确、挑剔。一个合规检查智能体需要严格按规则办事,而一个头脑风暴智能体则需要保持创造性和灵活性。当一个智能体被迫在这些互相矛盾的行为模式之间来回切换的时候,表现往往会变得不稳定。这时候把它们拆成几个各自有专属系统提示词的智能体,反而能让每个智能体的行为更一致。

另外还有一种专业化体现在领域知识的深度上。有些任务需要非常深厚的专业背景知识,如果硬塞进一个通用智能体的上下文里,反而会把它给淹没。比如一个法律分析智能体可能需要大量案例法和监管框架的背景知识,一个医学研究智能体可能需要临床试验方法论方面的专业知识。与其把所有领域知识都塞进一个万事通智能体,不如让每个专门的智能体只携带和自己职责相关的那部分专业背景。

文章给了一个挺实际的多平台集成的例子。假设你要做一个系统,需要同时操作客户关系管理系统,也就是 CRM,市场营销自动化平台,以及消息通讯平台,每个平台大概有十到十五个相关的接口需要调用。如果让一个智能体同时管理四十多个工具,它经常会选错,把不同平台上长得很像的操作搞混。但拆成几个专门的智能体,每个智能体只配备自己平台相关的工具集合,再配上量身定制的提示词,这种选择错误的问题就基本解决了。这个模式其实很像现实世界里专业团队的协作方式,各个领域的专家用各自最熟悉的工具协同工作,效果通常比一个万事通试图同时精通所有领域要好。不过文章也提醒,专业化会引入路由复杂度,需要一个编排者准确判断该把请求分配给哪个专家,一旦分类错误,结果就会很差,而且维护多个专门智能体也会增加提示词维护的工作量。专业化最适合的场景,是各个领域本身边界清晰、路由判断不存在歧义的情况。

文章除了给出三大场景的框架之外,还补充了几个更具体、更容易上手判断的信号。第一个信号是接近上下文窗口的上限,如果你发现智能体经常占用大量上下文,而且性能确实在下降,那上下文压力可能就是真正的瓶颈。但文章特别提了一句很值得记住的话,最近上下文管理方面的技术进展,比如压缩技术,正在不断减弱这个限制,让单智能体也能在更长的时间跨度里维持有效的记忆。这句话其实在暗示,很多团队今天觉得自己非上多智能体不可的理由,可能过一两个版本的模型迭代就自动消失了。

第二个信号是工具数量太多。当一个智能体管理的工具达到十五到二十个甚至更多的时候,模型要花费大量的注意力和上下文去理解自己手头有哪些选项。但文章在这里给了一个特别实用的建议,在你直接跳到多智能体架构之前,先考虑用一种叫做 Tool Search Tool 的工具搜索能力,它能让 Claude 按需动态发现工具,而不是一开始就把所有工具的定义都塞进上下文里。文章给出的数据是,这种方式能把 token 消耗降低多达百分之八十五,同时还能提升工具选择的准确率。这一点对国内做 agent 产品的团队特别有参考价值,因为很多人遇到工具太多选不准的问题,第一反应就是拆多智能体,但其实可能先优化一下工具的检索和加载机制就解决了大半。

第三个信号是任务天然可以拆成互相独立的部分,比如要去好几个不同的信息源做调研,或者要给好几个独立的模块分别写测试,这种场景下并行的子智能体确实能带来明显的速度提升。文章最后还补了一句提醒,这些阈值会随着模型能力的提升而不断变化,今天总结出来的这些经验数字,只是当下阶段性的实用指南,不是什么永恒不变的硬约束。

如果你已经判断出自己确实需要上多智能体了,文章认为接下来最重要的设计决策,就是怎么在不同的智能体之间分配工作。这一段我个人感觉是整篇文章里信息密度最高、也最反直觉的部分。

文章提出一个关键洞察,在拆解工作的时候,应该采用以上下文为中心的视角,而不是以问题类型为中心的视角。这句话听起来有点抽象,我们用文章里的对比来说明。

以问题类型为中心的拆分,往往是反效果的。比如你按照工作种类来分工,一个智能体专门写功能代码,另一个专门写测试,第三个专门做代码审查。这种分法听起来很像现实世界里软件团队按角色分工,显得很合理。但文章说,这种分法会造成持续不断的协调开销,每一次交接都会丢失一部分上下文。写测试的智能体根本不知道当时为什么要做出某些实现上的决策,代码审查的智能体也缺乏整个探索和迭代过程中的背景信息。

相比之下,以上下文为中心的拆分通常才是真正有效的。意思是说,应该按照上下文的边界来划分工作,而不是按照工作类型来划分。举个例子,负责实现某个功能的智能体,也应该顺手负责给这个功能写测试,因为它已经拥有了写测试所需要的全部背景信息。只有当上下文真的可以被彻底隔离开的时候,才值得把工作拆给不同的智能体。

文章提到了一个真实的实验现象,挺有意思的。他们按照软件开发里常见的角色去拆分智能体,设了规划者、实现者、测试者、审查者四个角色,结果发现这些子智能体花在互相协调沟通上的 token,居然比花在真正干活上的还多。这个现象被文章形象地称为电话游戏,就是那种排成一队互相传话,传到最后信息已经严重失真的游戏。每一次交接,就是一次信息保真度的损失,传得越多次,失真越严重。

那到底什么样的拆分边界是有效的呢?文章给了三个例子。第一种是独立的研究路径,比如调研亚洲市场趋势和调研欧洲市场趋势,这两件事完全可以并行进行,彼此不需要共享上下文。第二种是接口清晰的独立模块,比如前端和后端,只要有清晰定义好的接口契约,两边就可以各自并行开发。第三种是黑盒式的验证,一个只需要跑测试、汇报结果的验证者,根本不需要知道被测系统具体是怎么实现的。

反过来,文章也给出了三种容易踩坑的拆分边界。第一种是同一项工作里前后相继的几个阶段,比如同一个功能的规划、实现、测试,这几个阶段之间共享的上下文太多了,硬拆开反而吃亏。第二种是耦合很紧的几个组件,如果几个模块之间需要频繁地来回沟通确认细节,那它们其实就该放在同一个智能体里处理。第三种是需要共享状态的工作,如果几个智能体需要频繁地同步彼此的理解,那也说明它们本来就不该被拆开。

文章接下来重点介绍了一种在各个领域都表现得相当稳定可靠的多智能体模式,叫做验证子智能体。这是一种专门负责测试或者校验主智能体工作成果的独立智能体。

文章里有一句话我觉得特别值得拿出来说,它提到,随着编排者模型本身能力越来越强,比如用 Claude Opus 四点五来做编排者,这类更强的模型已经越来越能够直接评估子智能体的工作成果,不一定非要单独再搭一层验证步骤。但即便如此,验证子智能体在三种情况下依然很有价值,一种是当你用的编排模型能力没那么强的时候,一种是当验证本身需要用到专门工具的时候,还有一种是当你想在工作流程里强制设置明确的验证检查点的时候。

为什么验证子智能体这种模式特别好用?因为它天然就绕开了前面说的电话游戏问题。验证这件事本身,按性质来说只需要很少量的上下文传递,一个验证者完全可以用黑盒的方式去测试一个系统,根本不需要了解这个系统当初是怎么一步步被构建出来的。

具体的工作流程是,主智能体完成一个工作单元之后,在继续往下走之前,先派生出一个验证子智能体,把需要验证的成果交给它,同时给它明确的成功标准,以及完成验证所需要的工具。这个验证者不需要理解这个东西当初为什么是这样构建的,它只需要判断这个东西是不是满足了给定的标准。

文章列出了验证子智能体特别适合的几类场景。第一类是质量保证,比如跑测试套件、做代码检查、对照格式规范校验输出。第二类是合规性检查,比如校验文档是否符合公司政策,或者检查输出是否符合各种规则要求。第三类是输出校验,在最终交付之前确认生成的内容是不是符合规格。第四类是事实核查,让一个独立的智能体专门去核实生成内容里的论断或者引用是否准确。

但文章也很坦诚地指出了验证子智能体最容易出问题的失败模式,就是验证者偷懒,只跑了一两个测试,看到通过了就草草宣布验证成功,而不是做全面彻底的测试。这种偷懒行为在实际部署里相当常见,因为模型本身也有一种倾向,会找捷径来快速完成任务。

针对这个问题,文章给出了几条具体的应对策略。第一是把验证标准定得足够具体,比如直接要求跑完整的测试套件并汇报所有失败项,而不是笼统地说确保它能正常工作。第二是要求验证者做全面的检查,覆盖多种场景和边界情况,而不只是验证最常见的那一种情况。第三是设计反向测试,也就是故意构造一些理应失败的输入,让验证者去确认系统确实会正确地拒绝或者报错。第四是把指令写得非常明确,文章原文里举的例子是,在提示词里直接写上,你必须在标记为通过之前跑完整的测试套件,这种明确到近乎严苛的指令是必不可少的,因为如果不对全面验证提出明确要求,验证智能体就会抄近路。

讲完了这些核心内容,我们来聊聊这篇文章对我们这些实际在做产品、写代码的人,到底意味着什么,能怎么落地。

第一点建议是,在你准备动手搭建多智能体系统之前,先老老实实问自己三个问题。第一个问题是,我面临的限制是不是真实存在的,具体来说就是是不是真的撞到了上下文窗口的瓶颈,是不是真的有可以并行处理的任务,或者是不是真的需要专业化分工。如果答案含糊不清,那大概率你还不需要多智能体,先把单智能体的提示词、工具集、上下文管理打磨好,可能就已经够用了。我自己的感觉是,这一点在国内的 agent 创业团队里特别容易被忽视,很多团队看见同行在讲多智能体故事,自己也急着搭一套,结果工程复杂度上去了,效果却没什么提升,这其实是在交智商税。

第二点建议,如果你确实需要拆,拆分的依据一定是上下文边界,不是工作类型。这一点我建议大家做产品设计的时候直接把它当成一条硬性检查项。每次你想按角色拆智能体,比如想拆一个规划智能体加一个执行智能体,先停下来想一想,这两个角色之间需要传递多少背景信息,如果传递的信息量很大、而且很难压缩成简洁的摘要,那这次拆分大概率会变成一次电话游戏,得不偿失。反过来,如果你发现某两块工作之间确实几乎不需要共享上下文,比如调研两个完全不同的市场,那这就是一个值得拆开、并行处理的好候选。

第三点建议是,如果你的瓶颈是工具太多导致选择不准,别急着上多智能体,先试试看类似 Tool Search Tool 这种动态加载工具定义的机制。这其实给了我们一个很重要的提醒,工程上很多看起来需要架构级解决方案的问题,其实可能用一个更轻量的优化就能解决,比如按需检索工具定义,而不是把四十个工具的说明书一次性全部塞进上下文。这是一个性价比极高的优化方向,值得在拆多智能体之前先试一遍。

第四点建议,如果你决定要做验证这件事,验证子智能体是一个性价比很高的起点,因为它的上下文传递成本最低,最不容易掉进电话游戏的坑里。但一定要把验证标准写得足够死板和具体,不要用那种模糊的语言,比如确保功能正常这种话基本等于没说。最好的做法是把测试套件的覆盖范围、边界情况、甚至应该失败的反向用例,都写进给验证智能体的提示词里,并且明确要求它必须全部跑完才能下结论。

第五点建议,如果你已经在生产环境里跑着一套多智能体系统,不妨回头审计一下,看看自己的 token 消耗有没有落在文章说的三到十倍这个区间内,如果远远超出这个区间,说明你的协调机制可能存在低效的地方,值得专门花时间去优化交接环节,减少重复上下文和冗余的协调消息。

最后我想多说一句我自己的看法。我个人认为,这篇文章最大的价值,不是教会大家怎么搭多智能体系统,而是在给整个行业降温。过去这一年多,多智能体几乎成了一种政治正确的技术叙事,好像谁不上多智能体,谁的产品就显得不够先进。但 Anthropic 作为这个赛道里最有发言权的公司之一,反而站出来说,大多数场景下,单智能体配合好的提示词和好的工具设计就足够了,这种诚实在我看来是挺难得的。文章末尾也提到,如果你想继续深挖单智能体怎么做好,可以去看他们另外两篇文章,一篇讲怎么构建有效的单智能体,一篇讲怎么做好上下文工程,而他们的多智能体研究系统具体是怎么搭出来的,也专门写了一篇文章详细拆解,有兴趣的同学可以找来看看。

我们简单总结一下今天这期节目。这篇文章的核心结论是,多智能体系统不是默认更好的选择,它只在三种场景下才值得投入,上下文污染需要隔离的时候,任务可以真正并行处理的时候,以及不同任务需要不同工具和专业知识的时候。除此之外,协调的开销往往会超过收益。如果你真的要上多智能体,记住要按上下文边界拆分工作,而不是按角色或者工作类型拆分,同时不要忽视验证子智能体这种低成本、高回报的模式,并且一定要把验证标准写得足够具体和严苛。

我个人的点评是,这篇文章最打动我的一句话是,先从最简单的方案开始,只有当证据支持的时候,才去增加复杂度。这句话不只适用于多智能体系统,几乎适用于所有工程决策。好,今天的 Claude FM 就到这里,我们下期再聊,再见。
