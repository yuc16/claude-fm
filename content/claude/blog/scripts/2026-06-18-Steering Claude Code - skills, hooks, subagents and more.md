## Script

今天要跟大家聊的这篇文章，英文标题是 "Steering Claude Code: skills, hooks, subagents and more"，我把它翻译成「掌控 Claude Code：技能、钩子、子智能体及更多」。这篇文章发表于二零二六年六月十八日，也就是昨天刚刚在 Anthropic 的官方博客上发布，是一篇关于如何系统化定制 Claude Code 行为的技术指南。

昨天刚出炉，时效性极强，大家完全可以今天就对照文章去优化自己的配置。

那为什么要专门做一期节目来聊这篇文章？因为它解决的，正是每一个认真使用 Claude Code 做工程项目的人都会遇到的真实困惑：我的项目规模越来越大，给 Claude 的指令越来越多，但它好像越来越"记性不好"——该遵守的规范忘了，该走的流程跳过了，精心写好的约束在长对话里消失得无影无踪。到底该怎么组织这些指令，才能让 Claude 在整个项目生命周期里持续、可靠地按我的意图工作？

这个问题触及了 AI 编程助手在真实工程落地时最核心的挑战，而这篇文章给出了一套系统性的答案。

我们先来理解一下背景。

大多数人刚开始用 Claude Code 的时候，会接触到一个叫 CLAUDE.md 的文件。把它放在项目根目录，Claude 每次开始对话都会读这个文件。一开始这个文件很清爽，大概十几行，写几条编码规范、一两个构建命令，效果很好。但随着项目生长，各个团队不断往里加内容：前端加了组件命名规范，后端加了数据库访问约定，安全组加了几条"绝对禁止"的规则，DevOps 把发布流程也写了进去，PR 描述模板、测试命名规范、注释风格……久而久之，这个文件膨胀到了几百行。

然后问题就暴露出来了。每次对话开始，这几百行全都要加载进上下文，不管你这次只是在改一个 README 文件还是在重构核心服务，这些内容都占着位置，消耗着 token 预算。更要命的是，Claude 是一个语言模型，在对话进行到一定长度后，它对上下文早期内容的注意力会下降，你在第一轮对话开始时写的那些规则，可能到第十轮时它就不那么严格遵守了。

还有一个更隐蔽的问题：CLAUDE.md 越长，里面有效指令的密度越低，Claude 对每条指令的遵守程度也会随之下降。不是 Claude 变差了，而是大量不相关的内容在稀释它的注意力。

Anthropic 的这篇文章，正是在这个背景下提出了一套更精细化的控制体系。他们把控制 Claude Code 行为的所有方式整理成了七种方法，并且每种方法在三个维度上做了区分：第一，这个指令什么时候会被加载进上下文；第二，在长对话中它能不能保持存在，也就是压缩之后的持久性行为；第三，它的权威程度有多高，Claude 对它的遵守程度处于什么量级。

理解了这三个维度，你才能理解为什么七种方法各有各的适用场景，而不是随便选一种往里塞就行。

好，我们来一个一个看。

第一种，也是最基础的，CLAUDE.md 文件。

说是"最基础"，其实 Anthropic 对它的定位也做了更精确的梳理。CLAUDE.md 有两个变体。第一种是"始终加载型"：放在项目根目录的 CLAUDE.md，每次对话开始就自动读入，而且在整个会话期间持续存在。即使对话很长，Claude 需要压缩之前的上下文，它也会重新把这个文件读进来，确保里面的内容始终在场。第二种是"按需加载型"：放在子目录里的 CLAUDE.md，比如 app/api/CLAUDE.md，只有当 Claude 访问那个目录下的文件时才会加载进来，其他时候它不占上下文空间。

这两种变体共享的一个问题，文章说得很直接：CLAUDE.md 很容易变成一个无序扩张的文件，就像没有人真正负责的公共配置文件一样，每个团队不断往里加，没有人去删。这是有代价的，因为每一行都会加载进每一个工程师的每一次会话，不管那行内容跟他们当前的任务有没有关系。

所以文章给出了一个非常具体的建议：CLAUDE.md 要保持在两百行以内，要像对待核心代码一样有专人负责、代码审查变更。它应该给 Claude 提供对代码库的整体概览，或者作为一个索引，指向其他可以查找更多信息的文件，而不是把所有规则都堆在里面。

在大型 monorepo 里，每个团队的目录应该有自己的子目录 CLAUDE.md，各自管各自的规范，不要全部堆在根目录那一个文件里。文章还提到了一个企业场景：如果有一些必须适用于整个组织的安全策略或合规要求，可以把这些内容放进一个集中管理的 CLAUDE.md，通过 MDM 或配置管理工具下发到开发者的机器上，而且个人设置无法排除这类文件。

第二种方法，规则，英文叫 rules。

规则是放在 .claude/rules/ 目录下的 markdown 文件。没有设定路径范围的规则，行为跟根目录 CLAUDE.md 一样，会话开始就加载，压缩时也会重新注入，永远占着上下文空间。

但规则有一个 CLAUDE.md 没有的能力：路径范围控制。你可以在规则文件头部加一个 paths 字段，指定这条规则只对哪些目录或文件类型生效。举个具体例子，"所有 API 处理器必须先用 Zod 做输入校验"这条规则，可以配置成只对 src/api/ 目录下的文件和所有 .handler.ts 文件有效。当你在改一个完全无关的前端组件文件时，这条规则根本不会出现在上下文里，既节省了 token，又避免了无关指令干扰 Claude 的注意力。

文章里的建议是：如果是某种跨越多个文件但又不是全项目都适用的约束，比如"数据库迁移文件只能追加写入，绝不能修改已有内容"，用路径范围的规则比用嵌套的 CLAUDE.md 子文件更合适。因为这类约束往往跨越多个目录，用子目录 CLAUDE.md 来承载反而管不全，而一条带路径范围的规则可以同时覆盖多个位置。

第三种方法，也是这篇文章特别强调的新机制之一——技能，英文叫 skills。

技能这个概念非常有意思，它的核心设计思想是"延迟加载"。技能文件放在 .claude/skills/ 目录下，每个技能是一个文件夹，里面包含详细的操作指南、脚本，甚至可以带相关资源文件。每个技能有一个 SKILL.md 文件，包含名称、描述和具体正文内容。

关键点在这里：会话开始时，只有技能的名称和描述加载进上下文，可能就是一两行文字，描述这个技能叫什么、用来做什么。当 Claude 判断某个任务需要用到这个技能，或者你通过斜线命令，比如 /code-review，主动调用它，技能的完整内容才会动态加载进来。

这个设计的价值是什么？你可以积累很多专业技能，但每次对话只消耗那些真正被用到的技能的上下文空间。就像一个工匠的工具箱，工具都在里面，但你只拿出当前这个任务需要的那几件。

我来举一个具体例子帮你感受一下。假设你们团队有一套发布流程，需要按顺序做代码冻结、运行完整测试套件、更新变更日志、打版本标签、发布到预发环境、做健康检查、再发到生产环境……整个流程大概有二十几个步骤。如果把这些写进 CLAUDE.md，它就永远占着上下文，不管你当下是不是在做发布操作，不管你只是在改一行注释还是在修一个小 bug，这二十几步全都在那里占着空间。但如果把它写成一个叫 deploy 的技能，平时上下文里只有"deploy：执行完整发布流程，包含测试、打标签、分阶段部署和健康检查"这一行描述，真正执行 /deploy 命令时，Claude 才会把完整的发布手册读进来，一步一步带你走完整个流程。

Claude Code 自带了一些内置技能，/code-review 就是其中之一，它会按照固定的步骤审查你的代码改动，每次调用结果可预期，不会因为你的表达方式不同而走出不同的流程。你也可以写自己的自定义技能，文章里提到 Anthropic 有完整的技能开发指南。

技能还有一个值得注意的压缩行为：在一个很长的对话里，如果你调用了多个技能，当上下文需要压缩时，Claude Code 会按照一个总预算来重新注入曾被调用的技能，最早被调用的技能会最先被挤出去。这在超长会话里要留意，如果你需要在整个会话里保持某个技能的上下文，要考虑这一点。

第四种方法，子智能体，英文叫 subagents。

如果说技能是给主对话增加了可按需展开的操作手册，那子智能体做的是完全不同的事：它是把一个任务"分包"出去，让一个独立的智能体在自己完全隔离的上下文窗口里完成，然后只把最终结果汇报回主对话。

子智能体文件放在 .claude/agents/ 目录下，每个文件的头部用 YAML 格式定义名称、描述，以及可选项：这个子智能体用什么模型、能访问哪些工具。文件正文就是这个子智能体的系统提示词，定义了它的角色、能力和行为方式。

跟技能一样，会话开始时只有名称和描述加载进主上下文。当主 Claude 需要用到这个子智能体，它通过一个叫 Agent 工具的接口来调用，传入一段任务描述。子智能体就在一个全新的、干净的上下文窗口里启动，独立完成工作，然后把最终消息返回给主对话。整个中间过程，所有的推理步骤、工具调用、临时数据，全部留在子智能体自己的世界里，完全不会出现在你的主对话中。

你可以把这个想象成把一个项目分包给外部团队。你的主对话是公司的主线工作，子智能体是你外包出去的专门团队。你把一个明确的任务交给他们，他们去做，做完了给你一份结果报告，他们工作时用的办公室、开的会议、写的草稿，你都不用管，也不需要知道。这让主对话保持整洁，始终聚焦在主要任务上。

文章里给出了一个很好的使用场景：当你需要对大量日志做分析，这个分析过程会产生成百上千行的中间数据，但你只关心最终的异常摘要。如果让这些中间数据都出现在主对话里，主上下文会迅速被填满，影响后续工作的质量。用一个专门的日志分析子智能体，把分析任务包出去，只拿回一份摘要报告，主对话干净清爽，Claude 的注意力也没有被耗散掉。

子智能体还可以嵌套，最多支持五层。这意味着你可以构建相当复杂的智能体层次结构：主智能体协调几个中层智能体，每个中层智能体再调度多个执行层子智能体。文章提到，在动态工作流里，可以同时调度几十甚至上百个后台子智能体并行工作，而它们的中间结果存在脚本变量里，不进入任何 Claude 的上下文窗口，这使得大规模并行任务成为可能，而不会出现上下文爆炸的问题。

文章里关于技能和子智能体的选择，给出了一条很清晰的判断原则：如果你需要看到过程、在过程中可能需要干预和调整方向，用技能，让流程在主对话里展开，你可以随时插手；如果过程是黑盒、你只关心最终结果，而且中间过程会产生大量你不会再引用的中间内容，用子智能体，让隔离帮你保持主线的清洁。

第五种方法，也是我个人认为工程价值最高的一种——钩子，英文叫 hooks。

钩子是用户自定义的命令、HTTP 端点，或者 LLM 提示词，它们在 Claude 生命周期中的特定事件发生时自动触发。比如工具被调用之前，文件被写入之后，会话开始时，会话压缩之前。你在 settings.json 里注册钩子，告诉它监听什么事件、触发时执行什么操作。

文章把钩子分成了几种类型：命令类型，执行一个 shell 命令；HTTP 类型，调用一个接口；MCP 工具类型，调用一个 MCP 工具；提示词类型，让一个语言模型来判断应该做什么；智能体类型，调用一个子智能体来处理。前三种是完全确定性的，后两种引入了模型判断。

钩子最核心的价值在于一个词：确定性。文章里有一句我觉得非常值得单独拿出来说的话：模型选择去运行一个格式化工具，和格式化工具自动运行，这是性质完全不同的两件事。

这句话值得我们仔细体会。你可以在 CLAUDE.md 里写"每次修改代码后运行 Prettier"，Claude 在大多数情况下会遵守这条指令。但在一个很长的对话里、在一个复杂的多步骤任务里、在 Claude 分心去处理更紧迫的子任务时、甚至在遭遇文件里精心设计的提示词注入攻击时，Claude 可能会忘记这条指令，可能判断这次不需要运行，可能被其他意图覆盖了这个行为。这种"大多数时候会做"跟"每次都一定会做"之间有一道很深的鸿沟。

用钩子来做这件事就不同了。你注册一个 PostToolUse 钩子，在文件写入事件发生后触发 Prettier，这是框架层面的代码在执行，不依赖于 Claude 是否记得，不依赖于 Claude 当前的注意力状态，它就是会发生，而且每次都会发生。

安全场景里钩子的价值就更加突出了。文章特别指出：你不能只靠在 CLAUDE.md 里写"永远不要执行 rm -rf"来保证安全。因为"永远不要"这样的绝对禁止指令，在足够复杂的环境下，在受到对抗性输入时，模型是有可能被绕过的。真正的护栏必须是确定性的。PreToolUse 这类钩子可以拦截任何工具调用，检查工具的参数，用退出码二来拒绝执行，并通过标准错误输出告诉 Claude 这次调用被拒绝了以及原因。这才是真正意义上的防护线，不是靠 Claude 的遵守意愿，而是靠代码层面的强制执行。

文章还提到了一个更高级的场景：托管设置，也就是企业管理员部署的配置。这种配置无法被用户本地的个人设置覆盖，是目前唯一能实现整个组织层面确定性约束的方式。对有合规需求的企业来说，这是把 AI 工具纳入组织治理体系的关键能力。

钩子还有一个很好的特性：上下文成本很低。因为钩子的配置住在 settings.json 里，而不是写在 Claude 的上下文指令里，Claude 不需要"记住"要执行这个钩子，是底层框架直接运行的，所以它不消耗主上下文的 token 预算。偶尔会有例外，比如一个阻断型钩子的标准错误输出会被写入上下文，让 Claude 知道这次调用被拒绝了、原因是什么——但这是有意为之的最小化信息传递。

第六种方法，输出风格，英文叫 output styles。

输出风格文件放在 .claude/output-styles/ 目录下，它的内容会被注入进系统提示词。因为坐在系统提示词这个位置，它的权重是所有方法里最高的，Claude 对系统提示词级别的指令遵守度确实最高。听起来很好，但文章对它提了一个非常重要的警告，我觉得每个想用它的人都必须先了解。

当你自定义输出风格时，它会替换掉 Claude Code 默认的系统提示词，而不是追加在它上面。除非你在风格文件里明确声明保留编码指令，否则这个替换会把很多重要的行为默认抹掉。Claude Code 默认系统提示词里包含大量对软件工程实践至关重要的行为指南：怎么限定代码改动的范围，什么时候应不应该加注释，遇到安全漏洞怎么处理，做完工作要不要先跑测试再宣布完成……这些是让 Claude Code 作为专业编程助手运行的核心指令。一旦你用自定义输出风格替换了默认风格，这一切全部消失，Claude 就变成了一个通用的、不懂软件工程规范的助手。

文章的建议是：Anthropic 内置了三种风格，先试试这三种，不要轻易自己写。Proactive 主动式，让 Claude 更多地自主判断和执行，减少确认步骤；Explanatory 解释式，Claude 会一步一步解释它做了什么，更适合教学和学习场景；Learning 学习式，更适合协作开发，Claude 会更多地问你的意图再行动。这三种覆盖了大多数常见需求。

第七种方法，追加系统提示词，英文叫 append-system-prompt。

这个方法比修改输出风格要温和得多。它不是替换，而是在原有系统提示词的基础上追加内容，不改变 Claude 的基本角色定位，只是添加一些额外的指令。而且它是在调用时传入的，不保存在文件里，只对当次调用有效，会话结束了这个追加的内容就没了。

适合用这个方法的场景：你想临时加一些特定的编码风格要求、输出格式规范，或者某个特定项目的领域专有知识，但不想改变 Claude 的整体行为模式，也不想在会话之间持续存在这些指令。

不过文章也提到了一个重要的限制：追加的内容越多、越复杂，Claude 对其中每条指令的遵守程度越低。当追加的内容之间出现矛盾时，这个问题会更明显。所以这个方法适合精准、简短的补充指令，不是用来塞一大堆规则的地方，那样做反而适得其反。

聊完了七种方法，我们来看文章最后总结的几个决策建议。这部分非常实用，我觉得是整篇文章最有直接操作价值的内容。

第一个建议：如果你在 CLAUDE.md 里写了"每次做 X 就要 Y"这样的句子，停下来想一想，这个"要 Y"应该由 Claude 来判断并执行，还是应该无论如何都直接发生？如果是后者，用钩子。典型例子：编辑代码后运行 linter，提交后发送 Slack 通知，这些都是钩子场景，而不是写在上下文指令里的场景。

第二个建议：如果你在 CLAUDE.md 里写了"永远不要做某事"，要意识到这条指令可能在关键时刻失效。真正不能发生的事情，要用钩子来硬性拦截，企业场景要用托管设置来强制执行。写在上下文指令里的"永远不要"，只是概率上降低了发生的可能性，不是工程意义上的真正禁止。

第三个建议：如果你有一段三十行的操作流程写在 CLAUDE.md 里，把它搬到 .claude/skills/ 里去，变成一个技能。CLAUDE.md 是 Claude 随时需要知道的背景事实，技能是按需加载的操作手册。把操作流程放在 CLAUDE.md 里，是在用有限的全局上下文空间承载本来可以按需加载的内容，是一种结构性的浪费。

第四个建议：如果你有一条只适用于某个特定目录或文件类型的规则，一定要给它加路径范围限制。不加路径限制的规则在行为上等同于把它写进 CLAUDE.md，永远加载，永远消耗 token，不管这次的任务跟它有没有任何关系。

第五个建议：个人偏好，比如你习惯用语义化的提交信息，这类只属于你自己的偏好，应该放在用户级别的配置文件里，而不是放在项目 CLAUDE.md 里。项目级别的配置是全团队共享的，你的个人工作习惯不应该强加给所有团队成员。

好，我们来谈谈这些内容对实际工作的影响。

如果你现在已经在用 Claude Code，而且有一个已经很臃肿的 CLAUDE.md，这篇文章给了你一张非常清晰的瘦身路线图。第一步，找出所有流程性的、多步骤的操作清单，把它们移到 .claude/skills/ 里，变成技能。第二步，找出那些只适用于特定目录的约束，把它们改造成带路径范围的规则文件。第三步，把所有"每次都必须发生"的自动化行为，改造成对应事件的钩子。第四步，审视剩下的内容，删掉那些你其实已经不再关注、只是"以防万一"留着的指令。做完这四步，你的 CLAUDE.md 应该能从几百行瘦身到几十行，而且 Claude 对里面剩余内容的遵守程度会明显提升，因为信噪比高了。

如果你在构建更复杂的 AI 工作流，子智能体的架构值得认真研究。五层嵌套、几十到上百个并行子智能体，这已经是相当复杂的智能体系统了。文章用了一个词来描述这种架构下中间状态的存放位置——脚本变量，而不是 Claude 的上下文窗口。这是一种很重要的架构原则：上下文窗口是用来承载当前需要推理和决策的信息的，而不是用来存储所有历史数据的。把大量中间结果卸载到外部变量和存储里，主智能体的上下文保持干净，才能在整个长任务里持续保持高质量的判断力。

对于关注安全合规的团队，这篇文章里最重要的信息可能是这一点：用钩子和托管设置来实现硬性约束，不要依赖上下文指令来做安全护栏。这不只是 Claude Code 的最佳实践，也是任何 AI 系统在工程化落地时都应该遵循的基本原则：确定性的保障不能依赖于概率性的模型行为。把安全关键的约束放进代码和配置里强制执行，把灵活判断的空间留给模型去发挥，这是两件事，不能混淆。

文章最后还提到了一个值得关注的功能：插件。一旦你有了几个运转良好的技能、子智能体、钩子、输出风格，可以把它们打包成一个插件，在团队成员或不同项目之间共享一套完整的工作方式。这是从个人提效到团队规模化复制的关键一步。

我个人认为，这篇文章最大的价值不是在于介绍了七种工具，而是背后的一个更深层的工程哲学：上下文是一种有限且昂贵的资源，每一条指令都应该出现在它最合适的地方，以最合适的形式承载，在最需要它的时刻加载进来，在不需要它的时候彻底消失。这种精细化管理的意识，不只是关于 Claude Code 的配置问题，它反映的是工程师对工具底层运作机制的真正理解，以及把这种理解转化为系统设计决策的能力。把正确的内容放在正确的位置，这件看起来简单的事，背后其实是对整个系统运行方式的深刻掌控。

好，我们收个尾。

今天这期，我们聊了 Anthropic 昨天刚刚发布的 Claude Code 定制化系统指南。文章提出了七种控制 Claude 行为的方法：始终在场的 CLAUDE.md 文件、可以按路径范围控制的规则、按需加载的技能、在独立上下文中隔离运行的子智能体、能够确定性执行的钩子、权重最高但需谨慎的输出风格，以及只对单次调用有效的追加系统提示词。每种方法有自己的加载时机、持久性和权威程度，对应不同的指令类型和使用场景。

如果你只记住一件事，就记住这个：让必须可靠发生的行为走钩子，让流程性操作走技能，让 CLAUDE.md 保持小而精，只放 Claude 随时需要知道的基本事实。这三件事做到，你会发现 Claude Code 变成了一个真正可信赖的工程伙伴，而不是一个"大多数时候还不错"的助手。
