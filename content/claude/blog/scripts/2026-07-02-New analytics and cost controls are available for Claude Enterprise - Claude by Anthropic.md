我为什么要选这篇文章来聊呢？因为最近这半年，我身边越来越多的技术团队和 AI 从业者，都开始在组织里正式部署那种能做复杂代理工作的 AI 助手，让它去自动写代码、操作文件、调用各种插件。但是大家普遍遇到的一个头疼问题就是：钱花得跟流水一样，但谁也说不清楚到底花在了哪儿，更不知道怎么合理地控制它。而这个痛点，恰恰就是 Anthropic 这次更新瞄准的核心。

你可能会说，成本控制不就是设置一个预算上限，超过了自动停掉不就完了吗？事情真没那么简单。当 AI 从一个偶尔陪你聊天的工具，变成一个天天在后台替你干活、执行一连串复杂任务的数字员工时，它的使用模式和成本形态就完全变了。传统的按 token 计费加上一个全组织的总预算帽，那种管理方式实在是太粗糙了，它解决不了透明性问题，也解决不了价值观层面的问题——你到底是希望限制它用贵的模型做低价值重复劳动，还是希望它在该发挥威力的时候因为预算一刀切被中断？所以，我们非常需要用一种更精细、更聪明的方式来管理 AI 的支出。Anthropic 这一整套新工具，其实就是在回答这个问题：如何在把 AI 能力真正嵌入到组织血脉的同时，让每一分钱都花得有道理、可解释、可干预。

好，我们开始慢慢拆这篇文章。先看它要解决的背景问题是什么。文章开头就说得很直接：随着 Claude 承担越来越多、越来越困难的代理工作，它的使用和成本模式看起来跟一个标准的聊天工具已经完全不同了。这里的关键词是“代理工作”，也就是 agentic work。我们简单解释一下，当你在聊天界面里跟 Claude 你一句我一句地对话，让它帮你润色一篇文章、翻译一段文档，这种是可以比较好预测成本的，因为每一轮交互你基本心里有数。但当 Claude 进入企业级的高阶用法，它可能在几分钟内自主地完成几十步操作——比如说，接到一个需求，自己去读代码库里的十几个文件，分析其中的逻辑，修改多处代码，运行测试，发现测试不通过再回过去修正，最后生成一个完整的 pull request，这中间可能要调用几次文件读写、几次搜索、几次代码执行，模型在后台持续思考、反复自我纠正。这时候，单次任务的成本可能比简单聊天高出两个数量级，而且不同任务之间的波动巨大。

传统的管理面板顶多告诉你，这周整个组织花了多少钱，总共消耗了多少 token，这就像你拿着一张模糊的信用卡账单，只知道总共花了三万块钱，但完全不知道是哪个部门在哪个项目上花的，是买设备了还是吃掉了。这种情况下，如果财务或者 IT 主管看到月底账单数字惊人，第一反应通常是慌，然后是全盘收紧预算上限，结果就是该用的地方用不了，不该用的地方可能早就浪费掉了。所以，管理员迫切需要的不是单一维度的总控，而是一整套显微镜和调节阀，可以钻下去看细节，然后有针对性地做出调整。这正是这套新功能的设计初衷。

文章提到，这次发布的更新是建立在 Anthropic 过去几个月已经提供的一系列控制手段之上的。之前已经有什么呢？比如各个层级的支出上限、访问控制和模型路由、一个带着导出功能的使用分析仪表板和 Analytics API，还有一个叫 effort controls 的东西，可以简单理解为控制模型思考努力程度的机制。这一次的新增，是把分析的丰富度和控制的粒度大大往前推了一步。我个人理解，这就好像从一个只有总开关和总电表的房子，升级到了每个房间都有独立电表、智能开关、并且还能自动预警的智能家居系统。

那我们接下来就深入看一看这几个核心的新功能到底是怎么玩的，以及对我们工程师和 AI 负责人来说意味着什么。

最直观的一个更新就是管理员的 analytics dashboard 仪表板，现在可以按群组和按用户来显示用量和成本了。别小看这个“按群组和用户”，它解决的是组织里最普遍的一个扯皮问题：到底是谁在花钱？以往我们看到一个总的成本数字，研发团队可能会说我们都是在做正经的开发任务，肯定是产品团队天天拿它生成营销文案浪费了钱，而产品团队又会反过来质疑研发。现在，仪表板可以跟企业 IT 团队已经管理着的 SCIM 群组直接打通，SCIM 全称是跨域身份管理系统，你可以简单理解成企业里那种标准化的组织架构和权限分组。一旦打通，你就不需要再在 Claude 里面重新建一遍组织结构，直接复用现有的部门分组，成本就可以按照市场部、研发部、数据平台部这样的真实组织来拆分。这意味着，你打开面板就能看到，研发一组这个月花了八百美元，其中 Claude Code 占了六百，Artifacts 创建占了五十，文件编辑占了三十。每一项活动不仅显示了次数，还直接对应到它的美元成本上，一目了然。

这种透明化有一个非常实际的好处，就是它能消解掉 AI 使用中的“公地悲剧”。在一个组织里，如果大家感觉 AI 资源是公共的、免费的、没有明确归属的，自然的倾向就是过度使用，先用爽了再说。一旦用量的归属变得明确，每个团队看到自己的名字后面跟着一个不断跳动的成本数字，那种责任感就油然而生了。你甚至可以把这些数据导入到现有的财务系统里，做内部结算，哪个部门用得多，哪个部门的成本中心就背这笔开销，这样需求就会变得更加真实和理性。所以，按群组显示成本，绝不仅仅是一个技术上的 UI 改进，它本质上是在给组织引入一套 AI 资源使用的市场机制。

你再深入一步，这次的仪表板还不仅仅是告诉你用量，它直接把你可能想评估价值的那几个结果指标也列在了成本旁边，比如 artifacts 创建了多少个，文件被编辑了多少次，用了哪些 skills 和 connectors。这里 artifacts 可以理解为 Claude 生成的那些结构化的、可复用的内容产物，比如图表、报告、网页原型；skills 是一些可插拔的专业能力模块；connectors 是连接外部数据源的连接器。把这些产出物和成本放在一起看，管理员就很容易做投入产出比的判断。假如你发现市场部用了很多 skills 来生成社交媒体文案，产生的 artifacts 数量也很多，成本却不高，那这可能就是一笔划算的生意。相反，如果某个团队大量使用昂贵的模型，生成 artifacts 却很少，那就有可能是在用大炮打蚊子，或者产出的东西质量不高被反复返工。这种视野在过去是根本没有的。

接下来，我们聊聊一个我本人特别感兴趣的点，就是 Claude Code 的专属分析面板。我相信很多听众是做软件开发的，或者至少对 AI 辅助编程非常关注。Claude Code 是 Anthropic 推出的一个专门面向软件开发者的命令行工具，它可以让 Claude 深度参与到软件工程的工作流里，直接读代码、写代码、操纵 Git、执行终端命令。那这种代理味道很重的工具，用量和成本的样子跟聊天产品完全是两个世界。管理者最想知道的是什么呢？无非就是：我的开发团队用了之后，到底有没有更高效？钱是花在了真刀真枪的代码产出上，还是花在了 AI 的各种试错和无效循环上？

这次更新里，管理控制台里加了两个针对 Claude Code 的新标签页，一个叫 usage 用量，一个叫 value 价值。用量标签里展示了什么呢？活跃的开发者数量、会话次数，以及整个组织里最常用的那些命令。而且这个数据是每天更新的。这就好比你有了一张实时更新的工程效能仪表盘，能看到今天有多少工程师正在跟 Claude 结对编程，他们整天都在让 Claude 干什么——是频繁地在运行单元测试，是提交代码，是查看代码变更历史，还是卡在某个黑盒里重复尝试。如果你是个技术经理，看到某个团队 top 命令里有一个非常高成本的操作在反复出现，你就可以马上去跟那个团队聊聊，是不是哪里用法不对，或者是不是缺少一些 prompt 的最佳实践，导致 AI 老在绕圈子。这种精细度让管理变成了可能。

而价值标签页就更巧妙了，它试图去量化那个大家都很关心但又很难量化的东西——AI 到底给我们带来了多大的生产力提升。它怎么做的呢？它会总结用量和成本的数据，然后给出几个估算指标：生产力提升幅度、每次代码提交的平均成本，以及年化价值。最有意思的一点是，每一个计算公式在标签页里都是可见的，而且输入参数是可以调整的。这是什么意思呢？Anthropic 并没有把一个黑盒的数字甩给你，说“我们认为你提升了百分之三十四”，而是把推算逻辑摊开给你看，比如它可能假设传统写一段代码需要三十分钟，Claude 协助后只需十分钟，然后乘以工程师的平均时薪来换算成节约的成本。你觉得这个假设不合理，那参数你自己可以调。这种透明且参与式的价值估算方式，我个人认为是非常高明的，因为它避免了厂商自卖自夸的嫌疑，把最终的价值判断权交回了组织自己手里。你完全可以基于自己的数据，去校验和调整这些参数，然后得出一个自己团队内部信服的投资回报率结论。这对于那些需要向 CFO 或者 CEO 申请增加 AI 预算的人来说，简直就是谈判桌上的硬通货。

好，接下来我们要聊的这个功能，可能会让很多天天被各种数据报表折磨的同学感到一阵解脱，那就是 Analytics Chat，分析聊天功能。简单来说，管理员可以用自然语言，直接在管理面板里向 Claude 提问，Claude 会返回带有图表的回答，而且这些图表可以导出和分享。你能问什么问题呢？文章举了两个例子：“哪些团队这个月的 Claude 用量翻倍了？”或者“我们哪里每个席位的价值最高？”你想想，这种问题在过去，意味着你要手动导出数据到 Excel 或者 BI 工具里，然后清洗、透视、再画图，这对于非数据岗位的技术主管来说，是一个巨大的时间黑洞。现在你可以对着 Claude 直接说出你的业务问题，它就去后面扫描你的用量和成本数据，几秒钟内给你一个可视化的答案。这意味着数据分析的门槛被直接拉到了地面。任何一个部门经理，不管会不会写 SQL，都可以自主地去探查自己团队的 AI 使用健康状况。

这个功能背后其实有一个很重要的理念变化，那就是分析的民主化。传统的组织里，数据是掌握在少数分析师手里的，任何想探究数据背后故事的人，都要排队等分析师出报告。而分析报告的产出速度，往往决定了组织对变化做出反应的速度。当 AI 的使用模式日新月异，这个月研发部刚猛推 Claude Code，下个月市场部开始大规模试用新的 skill，如果管理层要到季度总结才能看到一份静态报告，那早就错过了最佳的干预时间窗口。Analytics Chat 把你从等报告的状态，变成了即时提问的状态，你可以每天花五分钟跟它对话，像跟副驾驶聊天一样，随时掌握全局。我觉得这会慢慢形成一种新的管理习惯：管理者不再被动接收报表，而是主动向系统发问。

你可能会关心，那些我们沉淀在各种专业系统里的财务和 IT 流程怎么办？是不是又要多一个孤岛？文章紧接着就提到了 Analytics API，这个 API 允许财务和 IT 团队把 Claude 的使用和成本数据，直接拉进他们已经在用的工具里，比如 Datadog Cloud Cost Management 和 CloudZero，然后把 AI 的花费和其它的云支出、SaaS 支出并列在一起看。对于任何一个大型组织的 FinOps 体系来说，这意味着 AI 成本不再是 IT 部门里的一个黑账，而是被纳入了企业整体的云成本管理框架。你可以跟着整体的节奏来做预算、做预测、做分账。结果可以按日期范围、团队、产品线、甚至模型来过滤。而且 skills 也开始上报自己的用量和成本，新的 API 端点还能追踪插件的采用情况和 artifacts 的创建。这意味着，所有这些可插拔的能力，都变成了可度量、可计费的经济要素，企业可以像管理微服务一样管理 AI 能力的组合。

好，讲完了怎么看得清，我们接着看怎么管得住。透明度解决的是知道钱花哪儿了的问题，但知道了之后，总得有手段去调整。这就涉及到两个非常实用的机制：模型级别授权和支出预警。

模型默认值和授权这个功能，听起来很技术，但其实它解决的是一个非常现实的问题：默认选项的浪费。在企业版本的 Claude 里，存在不同能力和价格的模型，比如一个更强大的模型可能适合复杂推理和代理任务，但单次调用成本高；而一个轻快便宜的模型适合日常的文档总结和简单问答。很多用户打开 Claude，直接就在默认的对话框里开始了，他们根本不会去注意，甚至不知道可以切换模型。如果企业选择把功能最强的模型设为了默认，那就意味着全公司上下，哪怕是问个天气、写个会议通知，都可能在用最贵的引擎去处理，这个浪费放大到整个组织是非常惊人的。现在，管理员可以针对 Chat、Cowork 以及 Claude Code 这样的不同产品入口，分别设定新会话默认启动时使用哪个模型版本，让日常例行工作不再默认流向最贵的那个选择。而且，管理员还能进一步控制哪些模型对特定角色或者整个组织是可用的。比如，你可以只让高级研发工程师有权限使用最强大的模型来做架构设计，而实习生或者市场人员默认就只能用性价比更高的版本。

我做过一个简单的推演，在一个有两百人的中型技术公司里，假设每个人每天平均有十次那种本该用轻量模型就能解决的简单交互，却错误地使用了昂贵模型，单次成本差异可能就一两美分，但一个月下来，这种无意识的浪费就能达到几千美金。通过一个简单的默认模型设置，几乎不需要任何用户的配合，就能在钢丝上省下一大笔开支。这种非侵入式的治理，是我非常推荐的一种实践。它不依赖人的自觉性，不依赖于反复的培训和通知，而是通过环境的设计，让正确的做法成为阻力最小的路径。

再来看支出阈值预警。这个功能初看起来很普通，不就是到了某个额度发个通知嘛。但它设计的要点在于触发点和后续流程。当组织整体的支出达到上限的百分之七十五时，管理员会收到预警通知；到了百分之九十，再次预警。这给了管理员一段宝贵的缓冲期，不至于在突然用尽时手忙脚乱地被迫提高上限，也不会出现某个工程师正跑着一个已经运行了二十分钟的复杂代理任务，啪一下因为预算耗尽被强制中断，前功尽弃。对于用户端来说，当个人用量达到组织分配给个人的上限的百分之七十五和百分之九十五时，应用内也会收到通知。更有趣的是，你不需要退出 Claude，再跑去找管理员发邮件、发消息申请提额，而是可以直接在应用内向管理员发送一条提高限额的请求，整个流程可以在工具内闭环。这个体验细节，透露出 Anthropic 是想让成本和权限的控制，变成一种像水和电一样顺滑的基础设施体验，而不是一种带有惩罚意味的行政手段。

最后一个让管理者感到踏实的更新是 Admin API。对于那些需要同时管理很多个群组和限额的大型组织，手动去审核每一个提额请求，或者手动去识别哪些人的用量正在逼近红线，这是不可扩展的。Admin API 把成本控制的工作流搬进了脚本里，让所有这些操作都能自动化。你可以用脚本定期扫描全组织的人员用量，自动标记出那些过去一周用量增长了百分之两百以上的异常情况，然后自动发一封邮件给所属的主管确认意图；你也可以设定规则，对于某些非关键角色，当他们的用量用到百分之九十时，自动驳回提额请求或者只允许提升一个很小的额度。最终的效果是，成本控制这套肌肉记忆，可以像 CI/CD 流水线那样自动运转起来，随着组织规模的扩大而平滑扩展，不需要投入等比例的管理人力。我甚至能想象，有些团队会把 AI 的用量消耗作为 KPI 或者 OKR 里的一个健康度指标，把它挂在团队的 Slack 频道里，每天机器人自动推送。

好，我们把几个关键的新功能都拆开看了一遍。从按群组和用户的成本透视，到 Claude Code 的价值量化，到自然语言分析聊天，到模型授权和默认值，再到支出预警和自动化 API。你会发现，它们并不是彼此孤立的单点功能，而是构成了一条完整的链条：看清全景、看透局部、问出洞见、设好边界、柔性提醒、自动处置。这其实恰恰反映了一个组织在接纳 AI 进行核心业务活动时，应该具备的成熟度和治理框架。

那对我们这些工程师和 AI 负责人来说，现在马上能用起来的是什么呢？或者说，我们该以一个什么样的顺序来逐步落地这些控制手段，并且要避开哪些常见的坑？

首先，我个人强烈建议，第一步永远是从建立可见性开始，别着急设限制。很多团队一上来就把预算帽设得很低，或者粗暴地关掉高级模型，这往往会引发用户的强烈反弹，甚至导致影子 IT 的出现——既然公司提供的工具不好用，那我自费去买个人版偷偷用。正确的做法是，先花一到两个星期，利用新的管理员仪表板和 Analytics Chat，安静地观察整个组织自然的、无干预的使用状态。把那些数据拉出来，按照部门和项目维度去理解：谁在用？什么时候用？用的是什么模型？产生了哪些类型的 artifacts？谁是重度用户？哪些团队的用量跟他们的业务产出看起来相关，哪些无关？这一步其实就是在绘制你组织的 AI 使用地图。地图画好了，你才能知道哪里是高速公路，该给足资源，哪里是沼泽地，需要铺路或者警示。

在建立基线之后，就可以着手设置模型默认值和授权了。我强烈建议你把默认模型设成性价比高的那个，而不是最强的那个。然后把最强的模型当作一种特权或者说是一种需要申请的赋能工具，只开放给确实有复杂推理需求的角色和场景。这样做的好处是，没有给任何人制造障碍，他们日常百分之八十的工作仍然可以用 AI 帮他们完成，但他们要想使用高级能力时，需要主动地进行一次切换或者申请。这一个微小的摩擦，会自然地过滤掉那些本来并不需要高级能力的随意使用，而且也能让用户在使用高级模型时多一些谨慎，因为他们知道这东西很珍贵。这里要避开的一个大坑就是，不要用一刀切的方式永久剥夺某些部门使用高级模型的权力。比如市场部门在做大型活动策划时，确实可能也需要极强的创意生成能力。如果完全禁用，他们会想方设法绕过去。更好的方式是按照用量配额来管理，给每个部门分配一定的高级模型 token 额度，用完即止，或者需要额外审批。

接着是配置支出预警。我建议抓住百分之七十五和百分之九十这两个节点，并且把它们跟协作工具打通。不要仅仅依赖 Claude 自己的邮件或者应用内通知，因为管理员不会每天都盯着 Claude 控制台。你可以用 Admin API 把预警事件推送到钉钉、飞书或者企业微信的指定财务或运维群组里。这样，当某个部门用量接近上限时，相关的主管和财务 BP 能第一时间在熟悉的上下文里讨论，是马上提额，还是去跟那个部门聊聊看是不是出了什么异常。另外，针对那些时间敏感的关键代理任务，比如每晚自动跑的数据处理流水线，如果它使用了 Claude，最好单独为这条流水线分配一个更高的、独立的支出上限，避免被个人用户突然的爆发性使用提前挤爆组织总预算，导致夜间任务集体失败，第二天早上大家一睁眼发现各种报表都没出来，那就是一个灾难了。

如果你管理的组织规模比较大，那么 Admin API 几乎是必用的。我建议趁早把成本控制脚本化。比如说，你可以写一个简单的定时任务，每小时拉取一次所有用户的用量情况，计算他们的消耗速度，然后把消耗速度前百分之五的用户清单自动输出。这么做不是为了惩罚谁，而是为了及时发现有价值的使用模式，或者是用量异常的用户，然后可以主动跟他们沟通，问一下是不是在做一些很酷的新实验，是不是需要支持。这样做会传递一个非常正向的信号：我们关注成本，不是要限制创新，而是要确保资源能更主动地流向那些能创造价值的地方。

对于 Claude Code 的用户，价值标签页里的那些可调参数，我建议每个技术负责人都应该跟团队一起开个会，一起过一遍。因为生产力提升的估算，如果完全交给 AI 厂商的默认值，可能会偏向乐观。你们应该坐下来，针对自己团队实际的工程情况来设定这些参数，比如你们平时写一段中等复杂度的功能到底要花多长时间，你们愿意接受的每次提交平均成本是多少。这个过程本身，就是一次对自己软件开发效能的重新审视，最终确定下来的数字，才是真正能代表你们团队，并且能被业务方接受的数据。

还要提一个容易忽视的点，就是用户端的透明度。文章里提到，管理员可以把用量可见性扩展到单个用户，让每个人都能看到自己的成本、产品和模型分解情况，以及自己离支出限额还有多远。我强烈建议打开这个功能。当工程师看到自己的名字后面，明明白白写着这个星期 Claude Code 花掉了多少美元时，他们会自然地产生一种资源意识，会开始琢磨：我这个 prompt 是不是写得太啰嗦了？我这个自动化任务是不是可以加一个判断条件，没必要时就不调用高级模型了？这种自下而上的优化，比自上而下的硬性命令，要持久和有效得多。这就像给每个家庭都装了电表之后，大家自然而然地就会在出门时关灯一样。

最后，我们还是要回到一个更本质的视角来审视这次更新。二〇二六年的今天，AI 不再是一个新鲜玩具，它开始真正地参与社会价值和生产价值的创造。当它从一个偶尔对话的伙伴，变成深度嵌入业务流程的数字化员工时，成本就不再只是一个 IT 采购的问题，它本质上变成了一个管理问题，一个组织能力的问题。一个公司如何花费它的 AI 预算，直接反射了这个公司的 AI 成熟度。胡乱花钱的，说明还在探索期，缺乏系统性的思考；把钱卡得死死不用，说明没有找到真正的价值落点；而能够精细化、动态化、自动化地管理 AI 花费的，说明它已经到了能用好 AI 来获得持续竞争优势的阶段。

Anthropic 这次放出的整套工具，在我看来，不仅仅是在卖更多的功能点，它是在试图给行业确立一套企业级 AI 治理的参考实践。从看见、到理解、到干预、到自动化，这一套闭环，很可能会成为未来所有企业级 AI 平台的标准配置。

好，我们快速回顾一下。这篇发布告诉我们，Claude 企业版新增了更丰富的分析能力，可以按群组和用户看成本；Claude Code 有了专属的价值和用量面板；管理面板里可以直接用自然语言跟数据对话；模型可以按需授权和设默认值来省钱；有百分之七十五和九十的支出预警；还有 Admin API 把控制脚本化。这一切加起来，就是给管理员一副望远镜和一套精细的操作杆，去驾驭越来越复杂的 AI 代理工作负载，同时让投入的每一块钱都有迹可循、有据可依。

我个人很期待看到，未来半年里，有先进管理意识的组织，会开始把这些 AI 成本数据和人力资源系统、项目管理系统做更深度的关联，进而算出“每个功能点交付的人工加 AI 总成本”，那才是真正的数字化转型效率仪表盘。
