## Shownotes

Anthropic 在二〇二六年五月正式推出了 Claude Platform on AWS,让 AWS 客户第一次能用 AWS 自己的认证和账单体系,拿到和原生 Claude API 几乎同步的全套功能。这期节目带你梳理这个新产品具体包含什么,它和老牌的 Amazon Bedrock 到底有什么区别,以及企业到底该怎么在两条路之间做选择。

本期要点:
- Claude Platform on AWS 正式全面上线,新功能和原生 Claude API 同步发布,不再有滞后
- 认证走 AWS IAM,审计走 CloudTrail,账单走 AWS 发票并可抵扣已有承诺额度
- 内含 Managed Agents、Skills、MCP 连接器、代码执行、Files API 等十余项原生功能,外加 Claude Console 开发环境
- Amazon Bedrock 依然保留,数据处理方仍是 AWS,适合严格数据驻留要求的客户
- 已有 Bedrock 私有报价的企业,务必先联系客户经理确认折扣迁移,折扣不能补溯
- 多个新功能仍处于 beta 阶段,生产环境部署建议预留灰度和回滚方案

## Script

今天我们要聊的这篇文章,标题是Introducing the Claude Platform on AWS,翻译过来就是在 AWS 上推出 Claude 平台。这是 Anthropic 官方博客上发表的一篇文章,发表时间是二〇二六年五月十一日,距离我们现在录这期节目,大概过去了一个多月。这篇文章篇幅不长,官方标的阅读时长只有五分钟,但是它宣布的这件事,我觉得对所有用 AWS 又想用 Claude 的工程师来说,分量不小,值得我们花一期节目把它讲透。

简单说,这篇文章宣布的是,Claude Platform 正式登陆 AWS 了,而且是全面上线,不是小范围测试。这意味着以后你在 AWS 上用 Claude,不再只有 Amazon Bedrock 这一条路,而是多了一个新的选择,这个新选择能让你拿到跟原生 Claude API 几乎一模一样的完整功能,认证走 AWS 的身份体系,账单走 AWS 的发票,还能去抵扣你已经签的 AWS 合同额度。如果你所在的公司是那种深度绑定 AWS,所有采购走 AWS 账单,但又想用上 Claude 最新功能的团队,这篇文章里讲的东西,基本就是为你量身定做的。

要理解这件事为什么值得关注,我们先退一步,看看在这之前,AWS 上的 Claude 用户是什么处境。在这篇文章发表之前,如果你想在 AWS 生态里用 Claude,基本上只有一条路,那就是 Amazon Bedrock。Bedrock 是亚马逊自己的一个托管模型市场,里面除了 Claude,还有 Meta 的 Llama,Mistral 的模型等等很多家厂商的模型,用户通过 Bedrock 统一的接口去调用这些模型。这种模式的好处很明显,你的数据处理完全留在 AWS 的边界以内,由 AWS 来当这个数据处理方,这对很多对数据合规和地域驻留有严格要求的客户来说,是刚需,比如说金融、医疗这些受监管比较重的行业。

但是这种模式也有一个绕不开的代价,那就是功能更新的速度。Anthropic 自己的原生 Claude API 平台,新功能、新的测试特性,基本是随出随上线的。但是要把这些功能搬到 Bedrock 上,中间要经过亚马逊那边的集成、测试、适配,这个过程不可能跟 Anthropic 自己发布的节奏完全同步,客观上就会有一个时间差。对很多追新能力的团队来说,比如说想第一时间用上某个新发布的 agent 工具,或者刚出的某个 beta 功能,在 Bedrock 上往往要等上一段时间才能用到,有时候等的时间还不短。这就形成了一个矛盾,一边是想要 AWS 生态里熟悉的账单、权限、合规这一整套工具链,另一边是想要 Anthropic 最新最全的功能,这两者在过去没法同时满足,你只能选一边,或者两边各开一套账户分别用。这就是这篇文章要解决的问题。

如果你跳出 Anthropic 这一家公司,放眼整个行业去看,大模型厂商和云厂商之间的合作模式,大致可以分成几种。一种是模型厂商和某个云厂商深度捆绑,云平台几乎成了那家模型厂商默认的底层基础设施。另一种是模型厂商把自己的模型,作为众多模型里的一个选项,放进云厂商自己搭的模型市场,用户在云厂商的界面里挑模型用,这就是 Bedrock 原来的定位。而 Claude Platform on AWS 代表的是第三种模式,模型厂商自己把完整的运营能力,搬到另一家云厂商的基础设施之上,自己说了算功能怎么更新,只是借用对方的账户体系和账单渠道。这种模式相对少见,因为它要求两家公司之间有足够深的信任和商业绑定,愿意让对方的运营团队深入到自己的基础设施里去操作。

这篇文章里给出的答案,就是新增了一个叫 Claude Platform on AWS 的产品,中文我们可以叫它 AWS 上的 Claude 平台。这个东西最大的特点,用文章里的原话讲,是 first of its kind,也就是说这是 Anthropic 第一次做这种形态的产品。它的本质是什么呢,你可以理解成,Anthropic 把自己原生 Claude API 平台的完整功能集,原样搬到了 AWS 这个壳子里,但是底层的计算和数据处理,仍然是由 Anthropic 自己来运营,数据是在 AWS 的边界之外被处理的。这一点非常关键,我们后面讲到怎么选的时候还会反复回来讲这个边界问题。

打个比方,你可以把这个关系想象成,你去一家大型购物中心里的专卖店买东西。购物中心是 AWS,它提供了停车场、收银系统、会员积分这些公共设施,但是专卖店里面卖的商品、服务员怎么给你介绍产品、库存怎么管理,这些都是品牌方自己说了算,跟商场没关系。Claude Platform on AWS 就是这样一个开在 AWS 商场里的专卖店,你刷的是 AWS 的会员卡,积分记在 AWS 账户上,但是货架上的商品,也就是 Claude 的各种功能,是 Anthropic 自己一手在打理,新货上架的速度,跟你直接去 Anthropic 自己的官网旗舰店买,是完全同步的。

文章里特意强调了一句话,叫做所有新功能和新的 beta 特性,都会在 native Claude API 上线的同一天,在 AWS 上线。这句话信息量很大。过去 Bedrock 那种有时间差的模式,在这个新产品里被彻底解决了。也就是说,如果你之前因为担心功能滞后,而不敢把生产环境放在 AWS 生态里,现在这个顾虑被拿掉了。

那么 Amazon Bedrock 这条路是不是就过时了,要被淘汰了呢,文章里讲得很清楚,完全不是。Bedrock 依然存在,而且依然是那个数据处理方是 AWS,数据完全留在 AWS 边界以内的选项。文章给这两条路分别画了一个清晰的适用场景。Claude Platform on AWS,适合那些想要完整原生 Claude 平台体验的公司,说白了就是想要功能全、更新快。Amazon Bedrock,适合那些有严格的区域数据驻留要求,或者必须要求数据完全在 AWS 基础设施内处理的公司。这其实是给了客户一个真正的选择权,而不是过去那种只有一条路、爱用不用的局面。

我个人认为,这个动作背后反映的是 Anthropic 在云合作策略上的一个变化。过去大模型厂商跟云厂商的合作,基本都是厂商把模型送进云厂商自己的托管市场,功能和节奏被云厂商的集成进度卡住。现在 Anthropic 这种做法,相当于是把自己的运营能力直接铺到了 AWS 的基础设施上,自己掌控节奏,只是借用 AWS 的认证、账单这套外壳。这种合作深度,在大模型和云厂商的合作历史上,我觉得是不多见的,某种程度上说明 Anthropic 和 AWS 这层合作关系,已经走到了一个相当深的阶段。

搞清楚了这个大的定位之后,我们具体看看 Claude Platform on AWS 里面到底装了哪些东西,这部分文章列得比较详细,我们一个个过一遍,顺便讲讲每个功能在实际工作里能拿来干什么。

第一个是 Claude Managed Agents,目前还是 beta 阶段,也就是公开测试版。Managed Agents,直译是托管智能体。这里说的 agent,中文一般翻译成智能体,指的是 Claude 不只是回答你一个问题,而是可以自主拆解任务、调用工具、连续执行多个步骤,直到完成一个相对复杂的目标。Managed 这个词的意思是,这一整套智能体的运行,生命周期管理,是由 Claude 平台帮你托管的,你不需要自己搭一套基础设施去维护这些智能体的运行状态。这意味着你可以在 AWS 上,直接把这种自主智能体大规模地跑起来,文章里用的词是规模化部署智能体。如果你的团队正在做一些需要 Claude 自动跑多步流程的项目,比如说自动化的代码审查,或者跨多个系统去查资料再汇总报告,这个功能就是直接对口的。

除了代码审查这种例子,我还能想到不少场景,比如说一个电商公司想要做退换货自动处理,从客户提交申请,到核实订单状态,再到生成退款单据,中间要跨好几个内部系统,这种多步骤、跨系统的任务,正是 Managed Agents 这种规模化部署智能体最合适的用法,因为你不需要自己再单独搭一套调度和监控系统来管这些智能体的生命周期。

第二个是 advisor strategy,顾问策略,也是 beta 阶段。这个名字听起来有点抽象,从文章给的上下文看,它属于 Managed Agents 体系下的一种策略配置,大体可以理解成是给智能体一种类似顾问的工作模式,让它在执行任务的过程中,扮演一个出谋划策、给建议的角色,而不是单纯执行命令的角色。具体的实现细节文章没有展开讲,但是从命名上能感觉到,这是 Anthropic 在智能体设计模式上做的进一步探索。

第三组是 web search 和 web fetch,也就是网络搜索和网页抓取。这两个功能放在一起讲,因为它们解决的是同一类问题,Claude 本身的知识是在训练时截止的,没法知道训练完成之后发生的事情。有了网络搜索和网页抓取功能,Claude 就可以在回答你问题的时候,主动去互联网上查最新的资料,把当下真实发生的信息,补充进它的回答里。文章里的说法是用这两个功能去增强 Claude 的知识,用的是当下真实世界的数据。打个比方,这就像是给一个知识渊博但是几个月没看新闻的朋友,配了一个能随时帮他查手机的助理,他还是那个聊天聊得很好的朋友,但是现在聊起最近发生的事,也能跟上节奏了。

第四个是代码执行功能,英文是 code execution。这个功能让 Claude 可以在你调用接口的过程中,直接运行 Python 代码,生成图表,做数据分析。这个能力其实是把 Claude 从一个只会嘴上说说的角色,变成了一个能动手算一遍给你看的角色。比如说你让它分析一份销售数据,它不是凭感觉给你编一个结论,而是真的把数据丢进一段 Python 脚本里跑一遍,算出准确的数字,再生成一张趋势图给你看。这对需要做数据分析、做报表自动化的团队,是个很实用的能力。

第五个是 Files API,文件接口,处于 beta 阶段。这个功能允许你把文档上传上去,然后在多次对话之间反复引用这些文档,而不需要每次对话都重新把整篇文档贴进去。如果你的工作场景是需要让 Claude 反复查阅同一批资料,比如说一份产品手册,或者一组合同文本,这个功能能省掉很多重复传输的麻烦,也能省钱,因为不用每次都重新计算这些文档占用的 token,也就是模型处理文字时按片段计费的那个计量单位。

第六个是 Skills,技能,也在 beta 阶段。这个功能是用来教 Claude 一些最佳实践,让它在处理特定类型任务的时候,能够给出更加一致、更加符合规范的结果。你可以把 Skills 理解成,给 Claude 装了一些专门领域的操作手册,比如说你公司内部写代码评审意见有一套固定的格式和检查清单,你可以把这套规范写成一个 Skill,以后 Claude 每次帮你做代码评审,都会按照这套手册来执行,而不是每次风格都不一样。

第七个是 MCP connector,MCP 连接器,beta 阶段。这里的 MCP,全称是 Model Context Protocol,模型上下文协议,这是 Anthropic 自己推出的一套开放协议,目的是让大模型可以用统一的方式去连接外部的各种工具和数据源。有了 MCP 连接器,你可以直接把 Claude 接到任何一个远程的 MCP 服务器上,不需要自己写一套客户端代码去做这个对接。这相当于是给 Claude 配了一个万能转接头,以前你想接一个新工具,得自己写一段适配代码,现在只要那个工具支持 MCP 协议,直接插上就能用。

第八个是 prompt caching,提示缓存。这个功能是用来给重复出现的上下文内容降低成本和延迟的。简单说,如果你每次调用里都带着一大段相同的背景信息,比如一份很长的系统说明,提示缓存可以把这部分内容缓存起来,后面调用的时候不用重新处理一遍,既省钱又能让响应更快。这个功能对那些调用量很大,而且每次调用都带固定上下文的场景特别友好,比如说客服机器人,每次对话都要带着同一份产品知识库做背景,缓存之后能省下不少费用。

第九个是 citations,引用功能,用来让 Claude 的回答有据可查,基于你给的源文档来作答,并且标注出回答的哪部分对应原文档的哪个位置。这对需要严谨溯源的场景,比如说法律、医疗这些领域的应用,是个很重要的信任保障机制。

第十个是 batch processing,批处理,用来处理那种高并发、不需要实时返回结果的大批量任务。如果你要一次性处理几万条记录的分类或者摘要,批处理模式能用更省钱、更稳定的方式跑完,不需要占着实时接口一条条地排队。

除了这十个功能之外,Claude Platform on AWS 的用户还能用上 Claude Console,也就是 Anthropic 自己的开发环境,用来构建和测试基于 Claude 的应用。Console 里面能管理智能体、管理 Skills、管理运行环境,还有叫做 vaults 的东西,大概可以理解成一种用来存放密钥或者敏感配置的保险箱机制,再加上可观测性工具,可以让你监控线上运行的情况。这里说的可观测性工具,在实际工作里,通常就是能看到每一次调用花了多少 token,跑了多长时间,智能体执行到第几步卡住了,有没有报错。对于运维和 SRE 团队来说,这些数据如果还要自己从零搭一套监控系统去采集,成本不低,直接复用 Console 里现成的可观测性能力,能省下不少前期搭建的功夫。

模型方面,目前 Claude Platform on AWS 上能用的是 Claude Opus 四点七、Sonnet 四点六,还有 Haiku 四点五,这是 Anthropic 目前的模型阵容里,从最强能力到最高性价比的三档选择。文章里特别说了,以后新模型发布,会同步在 Claude Platform on AWS 上线,这也是呼应了前面讲的那个核心卖点,就是不会有时间差。

讲完了功能清单,我们再来看一个我觉得特别值得展开讲的点,就是这个产品是怎么和 AWS 原有的那套企业工具链做对接的。这一点其实是 Claude Platform on AWS 真正打动那些大企业客户的关键,因为功能再多,如果跟你公司现有的权限管理、审计合规体系接不上,对很多大公司的 IT 和安全团队来说,基本就是没法落地的。

文章里讲到,认证是走 AWS IAM 的。IAM 是 AWS 的身份与访问管理服务,公司内部通常已经用 IAM 给每个员工、每个系统账户配置了精细的权限策略,谁能访问什么资源,谁不能访问什么资源,都在 IAM 里定义好了。Claude Platform on AWS 直接接入这套体系,意味着你不需要再单独给 Claude 这个新工具搞一套新的账号体系,直接复用你现有的 IAM 账户和权限策略就行。这对企业的安全团队来说,工作量直接减少了一大块,因为他们最怕的就是每接入一个新的外部服务,都要重新审一遍权限模型,现在这一步被省掉了。

审计日志这边,走的是 CloudTrail。CloudTrail 是 AWS 用来记录账户内所有操作行为的审计服务,谁在什么时候调用了什么接口,改了什么配置,都会被记录下来,方便事后追溯和合规审查。Claude Platform on AWS 上发生的操作,也会被记进 CloudTrail,这意味着你公司现有的合规审计流程,不需要为了 Claude 这一个产品专门再搭一套独立的日志系统,所有的审计记录都汇总在同一个地方,方便统一检查。

计费方面更直接,走的是一张 AWS 账单,而且能够完全抵扣你已经跟 AWS 签的承诺额度。这里的承诺额度,英文是 commitment,很多大公司会跟 AWS 签一个未来一段时间的最低消费承诺,以换取更优惠的价格,你可以理解成公司提前买了一大笔 AWS 的代金券。过去如果你想用 Claude,但是走的是 Anthropic 自己单独的账单系统,这部分消费是没办法算进你跟 AWS 签的那个承诺额度里的,公司财务和采购部门就要单独走一套审批流程。现在 Claude Platform on AWS 的消费,可以直接抵扣这个承诺额度,等于是说你买 AWS 代金券的时候顺便就把 Claude 的用量给覆盖了,财务上少了一道单独走账的麻烦。

这三件事合在一起看,认证、审计、计费全部嵌入到 AWS 原有的体系里,我个人认为,这才是这次发布真正的护城河所在。单看功能列表,跟原生 Claude API 平台是基本对齐的,真正的差异化在于,对于那些已经把整套 IT 治理体系建立在 AWS 之上的大公司来说,引入 Claude Platform on AWS 几乎不需要新增任何治理流程,这种摩擦力的降低,在大公司的采购决策里,往往比功能本身更重要。

关于覆盖范围,文章里提到 Claude Platform on AWS 会在大多数 AWS 的商业区域提供服务,并且支持全球和美国这两种推理地理区域。这里的推理地理区域,指的是你的请求实际被处理的物理位置范围。提供全球和美国两种选项,意味着如果你的公司有数据必须留在美国境内处理的要求,你可以选择美国这个范围,如果没有这个限制,选全球范围可能会有更好的可用性或者延迟表现。这跟原生 Claude API 平台已有的地理区域机制是一致的,等于是把同一套地理区域的选择能力,原样搬到了 AWS 这边。

说了这么多功能和架构上的东西,我们现在站在听众的角度,想一想这件事对你实际工作意味着什么,你该怎么去用,又有哪些坑需要提前避开。

第一种情况,如果你所在的公司本来就是 AWS 的重度用户,所有的云资源采购、权限管理、合规审计都已经围着 AWS 这一套体系转,但是你的团队又一直想用 Claude 最新的功能,比如说前面提到的 Managed Agents,或者 Skills,过去因为只能走 Bedrock,总觉得功能慢半拍,那么现在你应该认真评估一下切换到或者新增 Claude Platform on AWS 这条路。具体怎么做,文章里给的建议是,先去访问 Claude Platform on AWS 的页面,或者去看文档,了解清楚具体的接入方式。我建议你们团队可以先拿一个非核心的项目,比如说一个内部工具,先在新平台上跑一遍,把认证、计费这套流程跑通,确认没有意外的坑,再考虑往核心业务上迁移。

如果我是你们团队负责这次评估的人,我会按这样的顺序来推进。第一步,先把现在用 Claude 的所有场景列一张清单,标注清楚哪些场景对最新功能的依赖程度高,哪些场景只是用最基础的对话能力,完全不依赖新功能。第二步,挑一个对最新功能依赖度高,但是业务重要性中等的场景,作为试点,先在 Claude Platform on AWS 上跑通认证和计费的完整链路。第三步,试点跑顺了之后,再去看试点场景里的费用,跟之前用 Bedrock 或者原生 API 时的费用做对比,确认抵扣承诺额度之后,实际成本是不是真的降下来了。第四步,把试点期间踩到的坑,整理成一份内部文档,再决定要不要往更核心的业务场景推广。这套流程虽然看起来麻烦,但是比直接把核心业务一次性搬过去,风险要小得多。

第二种情况,如果你所在的公司,因为行业监管的原因,比如说你们是金融机构,或者医疗相关的企业,要求数据必须严格留在某个地域内,由 AWS 自己来处理,不能让数据离开 AWS 的基础设施边界,那么这种场景下,你应该继续用 Amazon Bedrock,而不是切换到 Claude Platform on AWS。因为后者的数据处理方是 Anthropic,数据是在 AWS 边界之外被处理的,这跟严格的数据驻留要求是冲突的。这一点我建议你们在做选型决策的时候,一定要先跟公司的合规和法务部门确认清楚你们具体的监管要求是什么,再去判断该走哪条路,千万不要先把架构定下来,之后才发现合规上过不去,那样返工的成本会很高。

第三种情况,如果你的公司之前已经在 Bedrock 上跟 AWS 或者 Anthropic 谈过一个私有报价,英文叫 private offer,这种私有报价通常是为大客户量身定制的折扣方案。文章里特别提醒了一句,如果你有现成的 Bedrock 私有报价,在开始用 Claude Platform on AWS 之前,一定要先联系你的 Anthropic 或者 AWS 客户经理,确认你的折扣能不能正确地应用到新平台上。这里有个很容易踩的坑,文章里写得很明确,折扣是没办法补溯的,也就是说,如果你在新的 Claude Platform 私有报价生效之前,就已经产生了消费,这部分消费是拿不到折扣的。所以这里的操作顺序非常重要,一定是先把新报价谈妥、生效,然后才开始产生消费,顺序反了,钱就白花了。这个提醒看起来是个细节,但是对于年消费规模比较大的企业客户,这个顺序错了可能就是几十万甚至上百万的差异,所以我建议负责采购和法务对接的同事,一定要把这条提醒转给对应的负责人,别漏看。

第四种情况,如果你是一个独立开发者,或者小团队,本来就没有什么复杂的 AWS 治理诉求,我觉得这次发布跟你关系不大,你大概率还是直接用 Anthropic 原生的 Claude API 平台最省心,毕竟你不需要 AWS 账单和承诺额度抹账这些企业级的好处,直接用原生平台,接入流程更简单,不需要多绕一层 AWS 的认证体系。

还有一个我觉得容易被技术团队忽略,但是财务和采购团队会非常关心的角度,就是计费方式切换之后,对预算编制的影响。过去如果你的 Claude 用量走的是 Anthropic 单独的账单,财务部门做预算的时候,这部分支出是单独一条线,跟 AWS 的整体云预算是分开核算的。现在如果切换到 Claude Platform on AWS,这部分支出并入了 AWS 的总账单,还能抵扣承诺额度,这意味着你们公司原来给 AWS 云资源做预算规划的那套流程和模型,需要重新纳入 Claude 这部分的用量预测。我建议技术团队在做这个切换决策的时候,提前拉财务或者 FinOps 的同事一起评审一下,免得切换之后,某个月账单突然变了样,大家都摸不着头脑。

另外有一个细节,我觉得值得单独提一下,就是这次发布里提到的功能,有不少都还在 beta 阶段,包括 Managed Agents、advisor strategy、Files API、Skills,还有 MCP connector。beta 阶段意味着这些功能的接口、行为,未来都还有可能调整,如果你要在生产环境里依赖这些功能,建议多留一些灰度测试和回滚预案,不要一上来就把核心业务流程,完全绑死在某个还在 beta 阶段的功能上。这是我从过去看下来的一个经验,Anthropic 给功能打 beta 标签是相当谨慎的,一旦从 beta 转正,接口大概率是稳定的,但是在 beta 期间,该有的防御性设计还是不能省。

我们简单总结一下这篇文章。Anthropic 在二〇二六年五月十一日宣布,Claude Platform on AWS 正式全面上线,这是 AWS 客户第一次能够通过 AWS 自己的认证、账单和承诺额度抹账机制,拿到跟原生 Claude API 几乎完全同步的全套功能,包括 Managed Agents、顾问策略、网络搜索抓取、代码执行、Files API、Skills、MCP 连接器、提示缓存、引用、批处理,以及 Claude Console 这个开发环境。与此同时,Amazon Bedrock 这条路并没有被取代,它依然是那个数据完全留在 AWS 边界以内、由 AWS 当数据处理方的选项,适合对数据驻留有严格要求的客户。两条路各有定位,企业可以按照自己的合规需求和功能诉求去选择。

我个人的总体判断是,这次发布更多是一次渠道和治理层面的扩展,而不是模型能力本身的突破。但是恰恰是这种看起来不那么炫酷的基础设施层面的动作,往往才是决定一个产品能不能真正进入大企业生产环境的关键。模型再强,如果企业的采购、合规、审计这条线走不通,最后落地的速度还是会被卡住。这次 Anthropic 选择把完整的原生功能开放到 AWS 上,同时保留 Bedrock 这条更严格的合规路线,我觉得是一个挺务实的两条腿走路的策略。如果你所在的公司本身就是 AWS 的深度用户,这期节目里讲的这些功能和接入方式,值得你拿着文档去仔细对照一下,看看现在切过去能给你的团队省下多少功夫。

从更长的时间线去看,我觉得这次发布也释放了一个信号,以后头部大模型厂商和云厂商之间的合作形态,可能会越来越多样,不再是单一的托管市场模式一种打法。这对我们这些做技术选型的人来说,意味着以后比较云平台和模型平台的时候,要看的维度也会变得更复杂,不能只看模型能力强不强,还要看运营主体是谁,数据边界划在哪里,账单和合规又是怎么嵌进你现有体系的。
