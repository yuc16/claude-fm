## Shownotes

本期解读 Anthropic 刚发布的 Claude Code 使用指南，主题是如何选择 Claude 模型和 effort level，也就是努力档位。文章把很多工程师常见的困惑讲清楚了：模型大小决定能力边界，努力档位决定 Claude 愿意为一个任务走多远。

本期要点：
- 模型选择本质上是在选择一组固定权重，也就是能力范围和每个输出 token 的成本
- effort 不只是“多思考一会儿”，还会影响读多少文件、跑不跑测试、查证到什么程度
- 任务失败时先判断是“Claude 不知道”还是“Claude 没认真查”
- 常规任务用小模型更划算，复杂、模糊、多步任务才值得上大模型
- 默认努力档位通常是最稳的起点，真正要调的是长期工作偏好，而不是每个任务都手动拧旋钮

## Script

今天我们要聊的文章，英文标题是 Choosing a Claude model and effort level in Claude Code，中文我会翻译成，在 Claude Code 里如何选择 Claude 模型和努力档位。这篇文章发表于二〇二六年七月七日，也就是距离今天二〇二六年七月十二日只有五天，是一篇非常新的 Anthropic 官方技术博客。它的作者是 Claude Code 团队的技术成员 Lydia Hallie，文章不长，官方标注五分钟读完，但它其实回答了一个现在很多工程师每天都会碰到的问题，Claude Code 里有两个看起来都能让结果变好的旋钮，一个叫模型，一个叫 effort level，也就是努力档位。那到底什么时候该换更大的模型，什么时候该把努力档位调高，什么时候其实应该先改 prompt，补上下文，或者让 Claude 读对文件呢。

这期值得听，是因为这不是一个单纯的产品说明，而是在帮我们建立一个判断框架。过去我们用 AI 编程工具，很容易用一种朴素直觉来操作，答案不好，那就换最贵的模型，或者让它想久一点。可是 Anthropic 这篇文章说得很明确，模型和努力档位不是同一个东西。模型决定的是 Claude 的能力范围，背后是一组已经训练好的固定权重。努力档位决定的是 Claude 为了完成你这个请求愿意做多少工作，比如读多少文件，调用多少工具，跑不跑测试，中途要不要继续查下去，还是早点回来问你。对每天在仓库里和 AI 结对编程的人来说，这个区别非常关键，因为它直接关系到质量、速度和成本。

先看背景。二〇二六年这个时间点，AI 编程工具已经不只是聊天框里问一句答一句了。Claude Code 这类工具更像一个在你项目里工作的 coding agent，中文可以叫编程智能体。它能看文件，改文件，调用 Read、Edit 这样的工具，跑测试，整理计划，再把结果汇报给你。也就是说，它不只是生成一段代码，而是在一个真实软件工程环境里完成一串动作。对于中国的软件工程师来说，这个变化很熟悉。你让它修一个单测，它可能要先读测试文件，再读业务代码，再跑测试，再修改，再重新验证。你让它做一次重构，它可能要跨十几个文件，检查类型错误，确保接口没有断。这个时候，问题就变成了，不同模型之间的能力差异，和同一个模型在不同努力档位下的行为差异，到底应该怎么理解。

文章开头直接点出两个常见假设。第一个假设是，更大的模型，比如 Claude Fable 五，会比 Claude Sonnet 更聪明。Anthropic 说，这个假设基本是对的。按照行业标准 benchmark，也就是基准测试，最大的模型确实能力更强。第二个假设是，更高的努力档位意味着 Claude 在回答前想得更久。这个说法就不完整了。努力档位当然包括思考时间，但它远不止思考时间。它还会影响 Claude 读多少文件，做多少验证，以及在一个多步骤任务里推进到什么程度才回来找你确认。高努力档位下，Claude 更可能自己先多读几个文件，跑一遍测试，多检查一次，而不是很快停下来问你，你能不能提供更多上下文。低努力档位下，它会更倾向于节省 token，遇到信息不够时早一点把问题抛回给你。

这里我们先把模型选择讲透。文章说，当你在 Claude Code 里按下回车，你的输入并不是孤零零地发给模型。Claude Code 会把你的消息、系统提示词、工具定义、你的 CLAUDE.md、对话历史，以及当前放进上下文的文件，组装成一个请求，发送到 API。这里的 CLAUDE.md 可以理解成项目里的说明文档，告诉 Claude 这个仓库的约定、命令、测试方式、代码风格等等。对很多团队来说，它有点像给 AI 同事看的 onboarding 文档。

但是，这些内容到模型那里，并不是模型直接看见一大段自然语言然后像人一样阅读。服务器上首先发生的是 tokenization，中文可以叫词元化。文本会被切成一个个 token，也就是模型处理的基本片段，每个片段会映射成固定词表里的一个整数。文章举了一个简单例子，const 可能映射成一九七八，await 可能映射成四二九三。从这一步之后，你的 prompt 在模型眼里就是一串整数数组。我们平时说模型读懂了你的需求，其实在底层，就是模型根据这些数字序列做计算。

模型的工作是什么呢。文章用一个非常经典但很重要的说法，它的任务是预测下一个 token。给它一串 token，它会对词表里所有可能的下一个 token 计算概率，然后从概率较高的候选里选出一个。比如前面已经出现了 const x 等于 await，一个训练得好的编程模型会把 fetch 的概率放得很高，把 banana 这种词的概率放得很低。这个例子听起来简单，但对理解后面的模型选择非常重要。模型不是查数据库，也不是先写完整答案再吐出来，而是一次预测一个 token，再把新 token 接到序列后面，再预测下一个，如此循环。

那是什么把输入 token 变成这些概率的呢。答案是 weights，中文叫权重，也叫 parameters，参数。文章说，这些权重是数十亿个数字，组织在巨大的矩阵里。为了预测一个 token，模型会让输入通过这些矩阵，做一长串矩阵乘法，最后得到每个 token 的概率。模型知道的一切，或者说它在训练中学到的一切，都存在这些权重里。它对 TypeScript 的理解，对流行框架的熟悉，对 Go 语言惯用写法的掌握，对各种通用编程知识的记忆，都是在训练阶段被编码进这些权重里的。

这里有一个特别容易被误解的点。文章强调，每个模型的权重在训练期间确定，等你发请求的时候，它们已经是只读的。你的 prompt、CLAUDE.md、上下文文件，都不会改变模型权重。如果你听过 inference 这个词，中文通常叫推理，在这里的意思就是训练结束之后使用模型，此时权重是固定的。你给 Claude 塞文档，塞代码，塞报错日志，它可以在这一次请求里利用这些信息，但这叫 steering，中文可以理解为引导，而不是 teaching，教学。你没有把新知识写进模型权重里，只是在这次推理里把它往某个方向引导。

这个区别对工程实践非常重要。比如你使用一个非常新的内部 SDK，模型训练时根本没有见过。你把 SDK 文档贴进上下文，Claude 可能能写出正确调用，因为它在当前请求里看见了文档。但它并没有真正学会这个 SDK。下一次换一个对话，如果你不再提供文档，它仍然可能不知道。再比如，一个库在模型训练之后才发布，模型权重里不会有这个库的真实 API。你问它怎么调用，它可能生成一个看起来很像的 API，但那个 API 实际不存在。文章把这叫幻觉，hallucination。它不是一次失败的查表，而是权重根据训练中见过的模式，生成了一串看起来合理的 token。这个解释能帮我们少一点神秘化，也少一点误判。

所以，改变模型到底改变了什么。文章给出的答案很直接，改变模型，就是换一组冻结的权重来处理你的请求。你从 Sonnet 换到 Opus，或者从 Opus 换到 Fable，本质上是在选择不同能力范围、不同成本结构的一组权重。模型设置决定哪组权重处理请求，也决定每个输出 token 的价格。但它不直接决定生成多少 token。相同 prompt 下，Claude 可能生成很短的回答，也可能展开很多推理、工具调用和总结。这个总工作量，就是 effort level 要影响的东西。

接下来讲努力档位。文章说，Claude Code 在执行任务时生成的 token，大致会落在几类里。第一类是 thinking，也就是你在界面上看到的推理过程，或者动作之间的思考。第二类是 tool calls，工具调用，也就是结构化地说我要调用 Read，参数是什么，或者我要调用 Edit，改哪个文件。Claude Code 会解析这些结构化输出，然后实际执行工具。第三类是 text to you，也就是写给你的计划、进度更新和最后总结。关键是，这些本质上都是同一个生成循环产出的普通输出 token，按照同样的费率计费。thinking token 也是这样生成出来的，并且在这一轮任务后续仍然留在上下文里。当 Claude 继续写代码时，它前面自己的推理，就像它读过的文件一样，也会成为后续输入的一部分。

努力档位怎么改变这些行为呢。文章说，effort level 会作为请求的一部分发送给模型，和你的 prompt 一起进入模型。模型在训练中已经学会了在不同努力档位下应该怎样表现，这种行为模式同样被编码进固定权重里。所以努力档位不是外部有个机械计时器，说多给你三十秒思考。它更像是告诉模型，这次任务你要达到怎样的 thoroughness，也就是彻底程度，和怎样的 certainty，也就是把握程度，才可以认为任务完成。高努力档位下，模型通常更愿意多生成 token，做更多步骤，追求更高置信度的答案。

这里可以用一个通勤路上好理解的类比。模型选择像是你请了不同级别的工程师。小模型像一个熟练但经验有限的同事，大模型像一个见过很多复杂系统的资深架构师。努力档位则更像你对这个同事说，这个活你快速看一下就好，还是请你仔细查一遍，必要时把测试跑全。资深架构师低努力，也可能很快给你一个方向；普通工程师高努力，也可能把文件翻得很细。但如果问题本身需要很强的抽象判断，比如跨服务一致性、隐蔽并发 bug、复杂重构的边界，普通工程师再努力也可能到达能力上限。反过来，如果只是批量改字段名，请最强架构师认真研究半小时，就有点浪费。

文章还补充了一个很细的点，高努力档位并不意味着 Claude 会死板地把所有计划步骤都走完。Claude 经常会先制定计划，努力档位会影响计划的深度和广度。但计划不是冻结不变的。随着 Claude 调用工具拿到结果，它会更新自己对进展和正确性的判断。文章举了一个三条假设的调试计划。如果第一条假设一查就找到了 bug，那么继续调查第二条和第三条可能就没必要了。Claude 通常会明确说，第一次检查已经找到了问题，所以剩下的检查不需要了，然后跳到修复和验证。你在 Claude Code 里看到任务列表中途被调整，其实就是这种行为。高努力档位会让 Claude 更倾向于双重检查、验证更多假设，但 Anthropic 也强调，它一般不会为了消耗而人为膨胀简单任务。团队在训练模型时会特别关注 overthinking，也就是过度思考，因为这会降低实际效果。

那我们到底怎么选努力档位。文章的建议很克制，大多数任务应该使用模型的默认努力档位。默认值是一个平衡点，代表 Claude 会根据大多数人愿意为这类任务花费的 token 来伸缩工作量。也就是说，它不是一刀切地少想或多想，而是在常见使用场景里比较合理。努力档位更像一个手动覆盖选项，当你基于自己的领域或者工作类型，对彻底性或速度有强偏好时，再有意识地调整它。文章特别说，应该把它看成一种 general preference，也就是一般偏好，而不是每个任务都临时决定一次。

文章里还有一个和新版本相关的观察。Anthropic 提到，在 Claude Opus 四点八发布之后，他们测试发现，使用 Opus 四点八的默认努力设置，相比 Opus 四点七的默认努力设置，在相同任务上通常能用差不多数量的 token 产生更好的结果。这个信息透露了一个产品层面的思路，默认档位不是静态的，它会跟着模型能力和训练策略一起演进。对用户来说，最稳妥的起点不是上来就把所有旋钮拉满，而是先相信默认值，再根据具体失败类型调整。

接下来是整篇文章最有实战价值的部分。当 Claude 做错了，你应该调什么。Anthropic 说，你的第一反应不应该是立刻拧旋钮，而应该先检查上下文。你的 prompt 是不是太模糊。Claude 有没有连到正确工具。有没有配备正确 skills，中文可以理解成技能或能力配置。它有没有拿到相关文件，CLAUDE.md 写得清不清楚，任务边界是不是过大。如果一个任务本来不应该需要更高努力档位，而你却通过加 effort 才勉强让它完成，那么真正的问题往往在上游，也就是上下文、项目说明或任务拆分方式。

假设上下文已经清楚，Claude 还是做错了，文章建议问自己一个问题，它是没有足够努力，还是没有足够知道。英文里就是 did it not try hard enough，or did it not know enough。这个二分法非常好用。没有足够知道，对应的是模型能力问题。没有足够努力，对应的是努力档位问题。我们可以拿工程例子来展开。

如果 Claude 已经读了相关代码，也理解了你的目标，看起来认真尝试了，但最终方案仍然在关键抽象上错了，或者对一个微妙 bug 的因果判断不对，那更可能是模型太小。比如你在一个大型 TypeScript monorepo 里查一个只在特定缓存刷新顺序下出现的 bug。Claude 已经看了 service 层、repository 层、测试日志，也跑了测试，但它一直把问题归因到表面上的空值判断，没意识到真正原因是异步事件顺序和旧状态复用。这个时候，你把努力档位再加高，可能只会让它在错误方向上读更多文件。更合理的是换更强模型，让它有更好的抽象和推理能力。

再比如架构决策。你问是否应该把一个同步调用链改成事件驱动，涉及可靠性、可观测性、幂等、回滚、数据一致性。小模型可能能列出很多常见优缺点，但一旦要结合你项目里真实约束做取舍，它可能会给出看似漂亮但不落地的建议。文章说，大模型更擅长处理复杂或模糊的问题。这里的模糊，不是你 prompt 写得乱，而是问题本身有不确定性，需要在多个目标之间权衡。大模型在这种情况下更有价值。

什么情况下该用小模型呢。文章也很明确，常规任务用小模型。比如你能精确描述的编辑，机械性修改，或者关于已经在上下文里的代码提问。你让 Claude 把一个函数参数从 userId 改成 accountId，并更新同文件里的调用。你让它按现有风格补一个简单单测。你让它解释某个已打开文件里的工具函数做什么。这样的任务没有必要为不需要的能力付费。文章说，如果你在大模型上工作了一段时间，任务一直都很常规，降到小模型通常能提升速度，降低成本，而且不影响输出质量。

努力档位该什么时候调高呢。文章给的信号也很具体。如果 Claude 做错，是因为它跳过了一个文件，没有运行测试，或者改了一半重构就停了，那就是没足够努力。尤其当你选择的努力档位低于模型默认值时，这个问题更相关。比如你让它修复失败测试，它只看了报错栈顶的文件，没有去看 shared helper，改完也没跑测试，就说应该好了。或者你让它做跨文件重构，它改了核心类型，却忘了更新两个调用点，最后编译失败。这种情况不一定需要更聪明的模型，可能只是需要它在当前模型上多走几步，多检查，多验证。

这里我个人认为，最实用的判断方法是看 Claude 的行动日志，而不是只看最后答案。它有没有打开关键文件。有没有搜索引用。有没有运行相关测试。有没有在失败后根据错误继续迭代。有没有在总结里清楚说明改了什么、验证了什么、没验证什么。如果它根本没走这些步骤，先考虑努力档位或 prompt 里的验证要求。如果它全都做了，还是在核心判断上错，那再考虑换模型。这个判断方式能帮团队节省大量成本，因为很多时候我们以为需要最强模型，其实只是需要把任务说清楚，让它跑测试，或者让它不要跳过某个目录。

文章最后还讲了模型、努力档位和 token 消耗之间的关系。这个部分很容易被误解，因为大家常常以为大模型一定更贵，小模型一定更省。文章说，取决于任务。在常规工作上，相同努力档位下，两个模型通常都能做对。大模型可能会多做一些验证步骤，而且每个 token 价格更高，所以对常规任务来说，用小模型确实能省钱，而且不损失质量。这个结论很直接，也符合日常经验。

但在困难、多步骤任务上，情况就不同了。小模型可能需要在能力边界附近反复摸索，烧掉很多轮迭代。它可能不断读文件、改错方向、跑失败测试、再修，再失败。大模型虽然每个 token 更贵，但可能用更少步骤达到同样质量，甚至总任务成本反而更低。更重要的是，文章强调，有些任务是小模型即使开到最高努力档位也完成不了的。努力档位可以让它走得更远，但不能改变它的能力上限。这个点在 Fable 上最明显。文章说，在长程、多步骤工作上，Fable 拉开差距最明显。在 Anthropic 的测试里，它完成了一些 Opus 和 Sonnet 在任何努力档位下都达不到的任务。当然，Fable 每个 token 也最贵，所以应该留给真正需要它的工作。

文章用了一个曲线的说法。努力档位决定 Claude 愿意沿着曲线走多远，但这不意味着 Claude 一定需要走那么远才能完成任务。换句话说，高努力档位是提高上限和意愿，不是强制燃烧 token。一个简单任务可能很快完成，不会因为档位高就必然变成一篇论文。可是如果任务复杂，高努力档位会让 Claude 更有可能继续查证，继续验证，直到它认为结果足够可靠。

还有一个细节，努力档位会影响 token 消耗，但它不是硬限制。系统里真正硬的上限是 max_tokens，也就是最大输出 token 数。如果达到这个限制，响应会在中途被截断。文章说，这个工具比较钝，主要和 API 开发者有关。更柔和的控制，比如给任务预算，或者在 prompt 里要求 Claude 保持简洁，通常更有帮助。因为这些是模型训练中学会遵循的指导，它接近限制时会尝试收束任务，而不是撞上一堵墙突然断掉。对实际使用 Claude Code 的工程师来说，这意味着你不应该把 max_tokens 当成精细的成本控制旋钮。更好的方式是明确任务边界，比如这次只定位问题，不改代码，或者只修改测试相关文件，或者先给我计划和风险，不要直接大改。

我们把这些观点落到工作流里。假设你每天用 Claude Code 处理一个中型后端项目。我个人会建议先建立三种默认模式。第一种是日常模式，适合小修小改、解释代码、补测试、机械重命名。这里用较小模型和默认努力档位就够了。你要做的是把文件放进上下文，说明目标和约束，让 Claude 不要擅自扩大范围。第二种是调查模式，适合定位 bug、分析失败测试、理解陌生模块。这里可以仍然从默认努力开始，但 prompt 里明确要求它先读相关文件，提出假设，运行或说明要运行哪些测试。如果它跳步骤，再提高努力档位。第三种是复杂模式，适合架构设计、跨模块重构、长链路性能问题、安全边界问题。这类任务一开始就值得考虑更强模型，因为失败成本高，而且模糊度大。

再具体一点。如果你要让 Claude 做一次数据库字段迁移，不要只说帮我迁移这个字段。更好的说法是，先让它阅读 migration 目录、模型定义、读写路径和相关测试，然后给出影响范围。等它列出计划后，你再让它分步修改。这里如果它只改了 ORM model，没看查询层，也没看测试，你应该提高努力档位或明确要求继续检查引用。如果它看全了，但迁移方案忽略了线上兼容窗口，比如旧服务版本和新字段并存的问题，那可能是模型能力或任务抽象问题，换更大模型更合理。

如果你做前端项目，让 Claude 把一个页面组件从本地状态迁到统一 store。常规部分，比如改 import、替换 props、更新类型，小模型能胜任。但如果这个页面和缓存、乐观更新、路由跳转、错误回滚都有关，问题就不只是改代码，而是理解用户交互生命周期。这个时候，大模型的价值不是写得更花，而是更可能发现隐藏耦合。努力档位则决定它会不会继续追踪引用，会不会跑类型检查，会不会在改完后回头确认没有遗漏状态来源。

团队层面也可以从这篇文章得到一些管理启发。我个人认为，企业内部不应该简单规定所有人只能用最便宜模型，或者所有关键项目都默认最贵模型。更合理的是按任务类型建立模型和努力档位的使用准则。常规批量修改优先小模型。复杂设计评审或不熟悉领域优先大模型。失败原因如果是没读文件、没验证，就调努力档位。如果是看过也想错，就换模型。同时，把 CLAUDE.md 写好，让 Claude 知道项目命令、测试方式、目录结构、代码风格和不要触碰的区域。文章虽然没有展开团队治理，但它对上下文的强调，已经说明了这类项目文档的重要性。

这里还有一个很现实的坑，很多人会把高努力档位当成补救模糊需求的万能药。比如你说，帮我优化一下这块代码。这个请求太宽了，优化什么，性能、可读性、内存、复杂度，还是安全性，都不清楚。Claude 如果在高努力档位下工作，可能会读很多文件，做很多推断，最后给你一个你并不想要的大改。正确做法是先缩小任务，说这次只关注这个函数在大输入下的耗时，保持外部接口不变，先分析瓶颈并给出最小修改方案。这样即使用默认努力档位，效果也可能更好。努力档位不是需求澄清器，它只是让 Claude 在你给定的方向上更愿意走远。

另一个坑是把模型当成长期记忆。前面讲过，把文档放进上下文是引导，不是教学。你今天告诉 Claude 一个内部 API 的正确用法，它这次能用，不代表下次不用再说。对于内部框架和新库，最好把关键说明写进 CLAUDE.md 或者相关文档，让 Claude Code 每次都能拿到。尤其是那些容易被模型根据外部世界经验猜错的东西，比如公司内部封装的 HTTP client，定制的错误处理约定，特殊的测试命令。不要指望模型权重里有这些知识，它没有。

还有一个实践建议，是把大任务切成有检查点的小任务。文章说高努力档位会影响 Claude 在多步骤任务中走多远才回来找你。但作为使用者，你也可以主动设计节奏。比如先让它调查并给计划，不要改文件。你确认后，再让它执行第一部分。执行完要求它跑测试并总结。这样做的好处是，即使你使用较小模型或默认努力档位，也能降低跑偏风险。对于复杂任务，再强的模型也可能因为上下文太大、目标太散而走偏。人和 AI 之间的协作边界，仍然需要工程师设计。

对 API 开发者来说，文章里关于 token 的解释也有价值。你如果在自己的工具里接入 Claude，不要只盯着 max_tokens。max_tokens 是硬截断，像给输出设置一把闸刀。真正影响体验的，是你如何在 prompt 里表达预算、阶段、完成条件，以及你如何向模型提供工具。比如你可以让模型在接近预算时先给出当前结论和未完成项，而不是继续展开。你也可以把任务拆成多轮，每轮有明确产物。文章强调，柔性控制比硬截断更有帮助，这对构建 AI 开发平台的人很重要。

我们再回到最核心的判断句。模型选择，是选择一组固定权重，也就是总体能力范围。努力档位，是选择 Claude 为了这一轮请求愿意做多少工作。上下文，可以引导模型，但不能改变权重。大模型适合难题、模糊题、陌生领域和架构判断。小模型适合 routine work，也就是常规、机械、边界清楚的工作。高努力档位适合需要更彻底阅读、验证和多步骤推进的任务。默认档位适合大多数时候作为起点。

我个人很喜欢这篇文章的地方，是它没有把问题包装成玄学。它把 Claude Code 的两个旋钮拆回到语言模型的基本机制。模型就是权重，权重是训练后冻结的。努力档位就是行为偏好，会让模型生成更多用于思考、工具调用和汇报的 token。这样一来，我们就能用工程方式调试 AI 协作本身。结果不好时，不再只是抱怨模型不行，而是问，上下文够不够，工具对不对，任务边界清不清，是能力不够，还是努力不够。

最后用两三句话收个尾。Anthropic 这篇文章的核心建议是，先用默认值，不要过早调旋钮；出错时先检查上下文，再判断是换更强模型，还是提高努力档位。对我们这些每天写代码、审代码、修 bug 的人来说，真正的收益不是知道哪个模型最强，而是学会把不同任务放到正确的成本和能力档位上。我的点评是，未来会用 AI 编程的人，不只是会写 prompt 的人，而是能像调度团队成员一样，调度模型能力、上下文和验证流程的人。
