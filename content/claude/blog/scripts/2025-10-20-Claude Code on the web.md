## Shownotes

本期我们聊聊 Anthropic 在二〇二五年十月发布的网页版 Claude Code。这次更新把原本只能在终端里跑的编程助手搬到了浏览器和手机上,让你可以同时给多个仓库派发任务,在云端跑测试、改代码、提 PR,彻底告别一个一个排队等终端的日子。我们会拆解它的并行任务机制、安全沙箱设计,以及移动端预览体验,还会聊聊普通工程师在日常工作里到底能怎么用上它。

- 网页版 Claude Code 的核心卖点:连接 GitHub 仓库后,可以同时派发多个独立任务,各自跑在隔离环境里
- 适合处理 bug 积压、常规修复、后端改动这类"定义清晰"的任务,而不是开放式的复杂设计工作
- 安全机制:每个任务跑在带网络和文件系统限制的沙箱里,Git 操作通过代理服务统一管控
- 支持自定义网络白名单,比如只放行 npm 源,既能装包跑测试,又不会乱连外网
- iOS 端同步上线早期预览,意味着"通勤路上派任务,到公司看 PR"正在变成现实
- 云端会话和本地终端共享同一个用量限额,不是白嫖出来的额外算力

大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是 Claude Code on the web,翻译过来就是网页版 Claude Code。这篇文章发表于二〇二五年十月二十日,出自 Anthropic 官方博客。算一下日子,距离今天二〇二六年六月二十一日已经过去八个月了,这个时间跨度已经超过半年,所以我要提前打个招呼,文章里说的"研究预览"阶段,很可能在过去这几个月里已经有了不少变化,比如功能范围、可用人群、甚至产品名字,都可能跟今天你打开浏览器看到的不完全一样。我们今天主要是带大家回到这篇文章发布的那个时间点,理解 Anthropic 当时想解决什么问题,这个思路本身放到今天依然非常值得拆解。

先说说背景。在这篇文章出来之前,Claude Code 这款工具的使用方式基本上是绑死在终端里的。你打开命令行,敲一句话描述需求,Claude 就在你的本地机器上读代码、写代码、跑测试,整个过程你是盯着一个终端窗口在看。这种模式有它的好处,贴近开发者最熟悉的工作环境,权限也完全在你自己的机器上,但问题也很明显。第一,你只能一次盯一个任务,因为终端是单线程的交互方式,你的注意力被绑在一个会话上。第二,你必须坐在电脑前,而且通常是那台装好了开发环境的电脑前,出门在外、用手机临时看一眼进度基本不可能。第三,如果你想同时处理好几个仓库里的几个不同问题,比如一个仓库要修 bug,另一个仓库要写测试,你就得开好几个终端窗口来回切,管理成本很高。

这个痛点其实不是 Claude Code 独有的,而是所有"本地跑的 AI 编程助手"共同面对的天花板。你可以把它想象成以前的客服中心,所有客服都坐在同一个机房里,你想同时处理十个客户的问题,就得十个人同时在线,效率提升靠的是堆人头。而云端化的思路,本质上是把"执行环境"从你的笔记本电脑挪到 Anthropic 管理的云基础设施上,这样一来,你就不再是那个唯一的调度员,你可以像甩任务给团队一样,把多个编程任务同时甩给云端的 Claude,自己去做别的事情,等结果回来就行。这就是这篇文章想要解决的核心问题:让委托编程任务这件事,从"必须本地盯着"变成"可以异步、可以并行、可以随时随地查看"。

好,我们现在来拆解这篇文章里最重要的几个点。第一个重点,也是文章一开头就强调的,是并行跑任务的能力。原文用的词是可以同时跑多个编程会话,跨不同的仓库,从一个统一的界面里管理。这意味着什么呢?举个例子,假设你是一个全栈工程师,手上同时维护着三个项目,一个是公司内部的后台管理系统,一个是对外的移动端 API,还有一个是你自己业余时间维护的开源小工具。在过去,你可能得先盯着后台系统改完一个 bug,确认没问题了,再切换到 API 项目继续干活,效率受限于你的注意力是单线程的。而现在,你可以在网页版界面里,同时给这三个项目各发一条指令,比如让后台系统修一个权限校验的 bug,让 API 项目补一组单元测试,让开源工具更新一下文档里过时的示例代码。这三个任务会分别在各自独立的云端环境里跑起来,你可以像看任务列表一样,实时看每个任务的进度条往前走。

这里有个细节值得多说两句,就是"实时进度跟踪"和"主动引导"这两个能力。文章原文提到,你不仅能看到 Claude 在跑的过程,还可以主动介入,调整它的方向。这个设计其实回应了很多人对自动化编程工具的一个顾虑,就是怕它"跑偏了"。你想象一下,如果你把任务扔出去之后完全是黑箱,等十分钟后回来一看,Claude 把你没让它动的文件也改了,那种心理落差是很大的。而如果你能在它跑的过程中随时插一句话,比如说先别动这个文件,或者这个逻辑分支你理解错了,应该是这样的,那这种"边跑边纠偏"的体验,跟你带一个初级工程师结对编程其实很像,你不需要从头到尾手把手教,但关键节点你得能插得上话。这种设计哲学,我觉得是这次更新里比表面上的"云端跑"更值得关注的一点,因为它解决的是信任问题,而信任问题往往比技术问题更难解决。

第二个重点,是文章提到的"自动建 PR,并附带清晰的改动摘要"。这个听起来简单,但其实是整个工作流闭环的关键一环。我们都知道,代码改完只是一半工作,另一半是怎么让团队里的人理解你改了什么、为什么改、有没有风险。如果 Claude 在云端把代码改完,但留给你一个杂乱无章的提交记录,你还得自己花半小时去整理 PR 描述,那这个"省时间"的承诺就打了折扣。而文章里强调的自动生成 PR 和摘要,意味着 Claude 在完成代码改动的同时,会把这次改动的来龙去脉写清楚,方便你或者你的同事直接走代码评审流程。这一点我觉得特别契合中国工程师团队的实际工作场景,因为我们很多团队的代码评审是有严格的 PR 模板要求的,改动说明、影响范围、测试情况这些都要写。如果 Claude 能把这部分自动化,确实能省掉不少"写文档式"的体力活。

第三个重点,是文章里专门花了一段篇幅讲的"安全优先的云端执行"。这一点我觉得是整篇文章里技术含量最高、也最值得工程师朋友认真理解的部分。原文提到几个关键设计。第一,每个任务都跑在一个隔离的沙箱环境里,这个沙箱对网络访问和文件系统访问都有限制。你可以把这个沙箱想象成一个一次性的集装箱,每次任务开始时新建一个,任务结束后这个集装箱就被销毁,里面发生的事情不会泄露到外面,也不会影响到其他任务的集装箱。第二,Git 相关的操作,比如拉取代码、推送改动,都是通过一个安全代理服务来完成的,这个代理服务确保 Claude 只能访问你明确授权过的仓库。这个设计其实是在防一个很现实的风险,就是万一 Claude 在执行过程中出现意外行为,它的"活动半径"被严格限制在你授权的范围之内,不会越权碰到你其他的私有仓库或者别的敏感资源。

第三点是关于网络白名单的设计,文章提到你可以自定义网络配置,决定 Claude Code 的沙箱可以连接哪些域名。举个具体例子,假如你的项目依赖 npm 包,你需要 Claude 在跑测试之前先把依赖装好,这就要求沙箱能访问 npm 的官方源。但你显然不希望这个沙箱可以随意访问互联网上的任意网站,万一被某种异常指令诱导去访问一些不该访问的地址,那就是安全风险。所以这种白名单机制,本质上是一种最小权限原则的体现,只放开你明确需要的那几个域名,其他一律拒绝。这个思路其实跟我们做后端服务安全加固时常用的出口流量控制是一个道理,放行清单越精简,攻击面就越小。文章末尾还提到,如果你想深入了解这套沙箱技术的实现细节,可以去看 Anthropic 的工程博客和官方文档,这说明 Anthropic 在这件事上是有专门的技术投入和文档沉淀的,不是随口一提的营销话术。

接下来我们聊聊文章里提到的另一个有意思的点,就是移动端的预览。原文说作为这次研究预览的一部分,Claude Code 同步上线到了 iOS 应用里,方便开发者在路上也能体验用 Claude 写代码。这一点听起来好像是个小功能,但我觉得背后传递的信号挺有意思的。它意味着 Anthropic 对"编程"这件事的定位正在发生微妙的变化,编程不再被严格定义为"必须坐在电脑前用键盘敲代码",而是可以被拆解成"描述需求、派发任务、审核结果"这样几个环节,其中只有最核心的代码生成环节需要强大的计算资源,而这部分完全可以放在云端完成,前面和后面的环节反而可以挪到手机这种轕便设备上。打个比较接地气的比方,这就像以前你出差路上只能用手机回个邮件,处理不了真正需要电脑才能干的活,而现在你出差路上也可以掏出手机,给项目派个修 bug 的任务,等你到了酒店打开电脑,代码已经改好了,PR 也生成好了,你只需要审一下、合并一下。当然文章自己也很诚实地说了,这是一个早期预览,体验上肯定还有打磨空间,Anthropic 也表示会根据用户反馈快速迭代。

第四个值得讲的点,是文章对适用场景给出的明确建议。原文列了三类比较适合放到云端跑的任务,第一类是回答关于项目原理和仓库结构的问题,第二类是 bug 修复以及定义清晰的常规任务,第三类是后端改动,因为后端改动 Claude Code 可以用测试驱动开发的方式去验证改动是否正确。这个划分其实挺实在的,没有把云端编程包装成"什么都能干"的万能神器,反而很清楚地告诉你,哪些场景它发挥得好。我自己理解,这三类场景有个共同特点,就是"验证标准相对明确"。你问项目结构,答案是不是对的,你看一眼代码就知道。你修一个 bug,有没有修好,跑一下复现步骤就知道。你改后端逻辑,写好测试用例,跑一下测试就知道改得对不对。反过来想,那些验证标准模糊的任务,比如让 Claude 去设计一个全新的前端交互方案,或者去决定一个产品功能该不该做,这类需要大量主观判断和审美权衡的任务,文章里没有提,也确实不太适合丢给一个异步跑在云端、你没法实时盯着的会话去处理。这也提醒我们,委托任务这件事是有边界的,边界就是"结果是否可以被客观验证"。

讲到这里,我想插一句我自己的看法。我个人认为,这次网页版 Claude Code 的发布,真正的突破点不在于"把终端搬到了浏览器",这只是表面形式的变化,真正的突破点在于它把编程助手的工作模式,从"同步对话"转变成了"异步委托"。同步对话的意思是,你说一句,它做一点,你看一眼,再说下一句,整个过程你必须全程在场。异步委托的意思是,你把任务的边界和验收标准说清楚,然后转身去做别的事,等结果出来再回来验收。这种转变其实跟软件工程团队管理里的一个经典话题很像,就是从手把手带新人,逐渐过渡到给资深工程师分配明确任务、定期同步进展。当你团队里的工程师足够资深,你不需要每天盯着他写的每一行代码,你只需要在关键节点对齐方向,验收的时候认真审查就够了。Claude Code 的云端化,某种程度上是在尝试把 AI 助手往这个"资深工程师"的协作模式上靠,当然它现在还远远没有到完全可以放手不管的程度,这也是为什么文章特别强调了实时进度跟踪和主动引导这两个能力,本质上还是给你留了随时叫停、随时纠偏的口子。

接下来我们说说,这对咱们听众朋友的实际工作意味着什么,能怎么上手用,要避开哪些坑。

第一个建议,是从"低风险、高确定性"的任务开始试水。如果你团队里有一堆积压的小 bug,那种复现步骤清楚、影响范围小、改动量不大的修复任务,这正是网页版 Claude Code 最擅长的场景。你可以把这些任务一股脑派出去,跑在不同的云端会话里,然后你专心去做需要深度思考的架构设计工作,等那些小修小补的任务跑完了,你再统一审查 PR。这种用法的风险非常低,因为哪怕某个改动不理想,你完全可以在 PR 审查阶段把它打回去,不会对生产环境造成任何影响。

第二个建议,是把"测试驱动"作为你委托任务时的标准操作。文章里特别提到后端改动适合用 Claude Code,因为可以用测试驱动开发的方式验证。这其实给了我们一个很实用的工作方法,当你描述一个任务的时候,不要只说"帮我实现某个功能",而要尽量带上"实现完之后要通过哪些测试用例"或者"验收标准是什么"。这样一来,云端跑的 Claude 不光是在写代码,也是在自己验证代码,你拿到的 PR 质量会更高,你审查的负担也会更轻。这跟我们平时写需求文档时强调验收标准的道理是一样的,标准越清晰,执行者跑偏的概率就越低,不管这个执行者是人还是 AI。

第三个建议,是要重视那个网络白名单的配置,不要图省事就放开所有权限。我理解很多人第一次用的时候,可能会图方便把网络访问设得很宽松,这样确实能减少踩坑的概率,因为有些依赖包或者外部服务你一开始没想到要加白名单,跑到一半报错,挺烦的。但从安全的角度,我建议大家还是按最小权限原则去配置,先想清楚这个任务到底需要访问哪些外部资源,比如 npm 源、内部的包管理服务地址,把这些精确地加到白名单里,而不是一刀切地全部放开。毕竟这个沙箱是要去访问你公司的代码仓库的,哪怕概率很小,一旦出现意外的网络请求,造成的损失可能远大于你省下来的那几分钟配置时间。

第四个建议,是把云端委托和本地终端结合起来用,而不是非此即彼。文章里有一句话我觉得很多人容易忽略,就是云端会话和本地终端共享同一套用量限额。这意味着如果你本来在终端里用 Claude Code 跑得很频繁,贸然又在网页端开十个并行任务,很可能会很快碰到限额上限。所以比较合理的策略是,把那些适合异步、不需要你盯着的任务挪到云端去跑,比如夜里下班前派几个 bug 修复任务出去,第二天早上来看结果;而那些需要你紧密互动、随时调整方向的复杂任务,继续放在本地终端里,跟 Claude 进行更紧密的同步对话。这样你的整体用量是被合理分配的,而不是简单粗暴地把所有任务都一股脑塞进并行队列里。

第五个建议,也是我觉得对团队管理者特别有用的一点,就是可以把网页版 Claude Code 当成一个"巡检工具"来用。原文提到的第一类适用场景是回答关于项目结构和原理的问题,这个能力其实非常适合用在新人入职、代码审计、技术债务盘点这些场景里。比如你接手了一个不熟悉的老项目,你可以直接派一个任务,让 Claude 去梳理这个仓库的模块划分、依赖关系、关键入口在哪里,跑出来的结果可以作为你快速建立认知地图的辅助材料。当然这里要提醒一句,Claude 生成的项目说明终究是辅助性的,真正的细节和历史背景,有时候还是需要你去问团队里更熟悉这块代码的同事,或者翻一翻历史的提交记录,机器生成的内容可以帮你节省梳理的时间,但不能完全替代人对项目历史和业务背景的理解。

第六个建议,关于移动端的使用,我个人的态度是保持谨慎乐观。文章自己也说了这是早期预览,体验肯定还在打磨阶段。我建议大家在手机上更多是用来做"轻量级的派发和审查",比如出门在外突然想起来一个简单的 bug 要修,你可以掏出手机派个任务出去;或者你在通勤路上想审查一下昨晚跑完的 PR,看看改动是不是符合预期,这些都是手机端比较合适的使用方式。但如果是需要你深度介入、反复调整方向的复杂任务,我还是建议放在电脑上处理,毕竟手机屏幕看代码diff,体验上终究是有局限的。

讲到这里,可能有朋友会问,这个东西到底会不会让程序员的工作被取代。我个人的看法是,这次的更新更准确的定位是"放大单个工程师能同时处理的任务数量",而不是"取代工程师做决策"。它解决的问题是并发能力的瓶颈,而不是判断力的瓶颈。你依然需要决定哪些任务值得派发、验收标准怎么定、PR 要不要合并,这些判断性的工作并没有被自动化。反而我觉得,这种工具的出现,会让"任务拆解能力"和"验收标准设计能力"变得更加重要,因为你委托任务的清晰程度,直接决定了你拿到结果的质量。这跟带团队管人是一个道理,你布置任务越含糊,下属交付的结果偏差就越大;你布置任务越清晰,交付质量往往就越稳定可控。

好,我们来给今天的内容做个总结。这篇文章核心讲的是 Anthropic 在二〇二五年十月把 Claude Code 从一个只能在终端本地跑的工具,升级成了可以在浏览器和云端并行跑多个任务的服务,同时同步预览了移动端的体验。这次升级最重要的三个支点,一个是并行委托加实时引导的工作模式,一个是隔离沙箱加网络白名单的安全设计,一个是自动生成 PR 摘要带来的工作流闭环。对我们普通工程师来说,这次更新最大的启示,不是说我们终于可以躺着写代码了,而是我们需要重新思考怎么把任务拆解得足够清晰,怎么设计可验证的验收标准,这样才能真正发挥出"委托式编程"的效率优势。距离这篇文章发布已经过去八个月了,这个领域变化很快,如果你对这个功能的最新状态感兴趣,我建议直接去看一看现在的官方文档,确认一下哪些细节已经迭代过了。好,今天的节目就到这里,我们下期再聊,再见。
