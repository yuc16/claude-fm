## Shownotes

这期节目聊的是 Anthropic 在二零二五年十一月发布的一篇文章,主题是创业公司如何用 AI agent 自动化管理复杂业务流程。文章通过会计、安防、招聘、金融风控、科研、低代码开发等多个真实案例,展示了 agent 如何帮资源有限的初创团队拿回时间、获得专家级能力,同时还能保持质量和速度的平衡。我们会拆解这些案例背后的共同逻辑,也会聊聊从小处切入再逐步扩大 agent 应用范围的具体路径。

本期要点
- agent 和传统工具的区别在于能不能自主推进一个完整流程,而不是每一步都等人下指令

## Script

大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是 Building AI agents for startups,翻译过来就是创业公司如何打造自己的 AI agent。这篇文章是 Anthropic 官方在二零二五年十一月三日发布的,也就是大概半年多以前。考虑到现在是二零二六年六月,半年时间在 AI 这个领域已经足够发生不少变化了,所以听这期节目的时候,大家可以留意一下,文章里提到的那些产品和数字,是当时那个时间点的快照,放在今天来看,具体的数字可能已经被刷新了,但是它讲的底层逻辑,我个人认为依然是成立的,甚至会越来越重要。

我们先简单说一下,这篇文章发布的时候,行业大概处于什么阶段。二零二五年下半年,大模型已经从单纯的聊天助手,逐渐演变成能够自主执行多步骤任务的 agent,也就是智能体。这个词大家应该不陌生了,但是值得强调一下,文章里反复在区分两类东西,一类是传统的 AI 工具,你给它一个指令,它给你一个结果,每一步都需要人去把信息翻译成下一步的行动;另一类才是真正意义上的 agent,它可以在拿到一个目标之后,自己规划步骤,自己调用工具,自己执行,中间只在关键节点上让人来确认一下,也就是我们常说的人类在环路里做监督,而不是事事都要人插手。这个区别听起来简单,但其实是整篇文章的核心出发点。

那么这篇文章到底想解决什么问题呢。我们先想象一下一个典型的创业公司的状态。创始人和早期团队往往要同时兼顾产品、销售、招聘、财务、客服,几乎每个人都要戴好几个帽子。事情永远做不完,人手永远不够,这是创业的常态。过去面对这种情况,唯一的解法就是招人,但是招人这件事本身就很贵也很慢,找到合适的人需要时间,而团队本来就没有时间,招错了人造成的损失,一个精简的团队又很难扛得住。文章提出的核心观点是,AI agent 正在改变这个计算方式,因为它可以在没有持续人工指挥的情况下,自主地把一整段业务流程跑起来,这对资源紧张的创业公司来说,是一种结构性的改变,而不只是效率上的小幅提升。

文章把创业公司面临的资源约束总结成三类,我们一个一个来拆。

第一类约束是时间。创始人和早期员工花在重复性的运营杂事上的每一个小时,都是没有花在战略、融资或者产品打磨上的一个小时。请注意,这里说的不是这些杂事不重要,而是说,这些事情本可以被自动化,但是因为团队资源有限,只能由本该做更高价值事情的人来兼任。文章举了几个例子。第一个是 Campfire,一家做现代会计软件的公司,他们用 Claude 来帮助会计团队加速月末结账的流程,结果是把结账周期缩短了三天,银行对账所需要的时间减少了百分之九十,通过他们的 Ember AI 聊天界面,报表生成速度提升了百分之五十。第二个例子是 Armanino,美国最大的会计事务所之一,他们在亚马逊云的 Bedrock 平台上部署了 Claude,把手动写作类任务的时间减少了百分之六十五,后续需要澄清沟通的次数减少了百分之六十。第三个例子是教育平台 ClassDojo,它服务着四千五百万用户,覆盖了美国百分之九十的小学,他们用 Claude 驱动一个叫 Sidekick 的 AI 教学助手,把老师以前每个班级要花二十到三十分钟才能完成的行政工作,压缩到几秒钟就能搞定。

这几个例子有一个共同点,就是它们处理的都是有固定套路、可重复的工作,这种工作正是 agent 最擅长大显身手的地方。当重复性任务从原来要花几个小时,变成只需要几分钟,你就能真正把精力放在推动业务往前走的事情上,而不是被行政杂务淹没。

第二类约束是专业知识。创业公司在每个职能上都需要世界级的水准,但是找到这样的人才很难。一个早期团队不可能在安全、法律、财务分析、医学研究这些专业领域,样样都招到顶尖人才,招聘周期、薪资成本都摆在那里。AI agent 提供的是一种专家级的分析和决策能力,而这种能力按照传统方式去培养,可能需要好几个月甚至好几年的时间投入。

文章里举了一个我觉得特别有说服力的例子,叫 eSentire,这是一家管理检测与响应服务商,保护着八十多个国家的关键基础设施。他们用 Claude 复制了顶级安全运营中心分析师的调查流程,做到了百分之九十五的准确率,同时把分析时间从五个小时压缩到七分钟,威胁抑制率达到了百分之九十九点三。你想一下,一个安全事件,原本需要一个经验丰富的分析师花五个小时去排查日志、关联线索、做出判断,现在七分钟就能完成,而且准确率还能保持在很高的水平,这意味着安全团队可以处理的事件量级直接被放大了几十倍。

另一个例子是 Gradient Labs,一个面向金融机构的 AI 客服平台,他们在复杂的金融客服场景下,维持了百分之八十到九十的解决率,客户满意度达到百分之九十八。文章里有一句话我觉得讲得很到位,小团队现在能交付出原本需要雇佣博士级专家、积累多年领域经验才能达到的分析质量。这其实是在说,agent 把专业知识从稀缺的人力资源,变成了一种可以被规模化复制的能力。这对创业公司意义重大,因为招到一个真正懂行的专家本身就很难,更别说招到一整个团队的专家了。

第三类约束是速度和质量之间的取舍。不管公司大小,长期以来大家都面临一个两难的局面,跑得太快质量就容易出问题,过度追求完美又会被竞争对手甩在后面。文章认为 agent 打破了这个取舍,因为它可以在大幅提升执行速度的同时,维持很高的质量标准。

举两个例子,Brex 是一家企业信用卡和费用管理公司,他们的合规率做到了百分之九十四,而行业平均水平只有百分之七十,同时自动化处理了百分之七十五的交易,每年为客户节省了五千六百五十万美元。另一个例子是 StackBlitz,他们把 Claude 整合进了自己的产品 Bolt 之后,四周时间从零做到了四百万美元的年化经常性收入,用户开发应用的成本降低了百分之九十九,而且产品质量依然达到生产级别。这两个例子放在一起看,其实就是在说,当你不需要再用质量去换速度的时候,你同时拿到了两个世界最好的东西,工作交付得更快,而且经得起检验。

讲完三大约束,文章接下来给了一长串更具体的行业落地案例,我挑几个有代表性的展开讲讲,因为这部分信息密度很高,值得细细品味。

先说招聘领域。一家叫 Micro1 的公司,通过 Claude 驱动的技术面试平台,每天能完成三千多场 AI 面试,招聘成本降低了百分之八十五,而且经过 AI 筛选之后再进入人工面试环节的候选人,通过率是原来的五倍。这背后的逻辑很直观,招聘里最耗时间的环节往往是初筛,大量简历、大量初面,如果这部分能被 agent 接管,而且筛选质量不打折,那么人力团队就可以把精力都放在最后的关键决策上。

这个平台还有个下游的应用案例,叫 Legal Soft,他们用了 Micro1 这套 Claude 驱动的面试系统之后,把面试团队从三十三人精简到十二人,但是每个月仍然能完成一万人次的候选人筛选,每年节省四十多万美元的招聘成本,同时息税折旧摊销前利润率提升了百分之三十。你可以想象一下,这相当于用三分之一的团队规模,做出了原来同等甚至更高的产出,这种杠杆效应,对一个利润率敏感的服务型公司来说是非常实际的。

再说风控领域。Inscribe 是一家做欺诈审查的公司,他们用 Claude 驱动的 AI 风险 agent,把欺诈审查的周期从三十分钟压缩到了九十秒。这里面包含了文档验证、欺诈检测、风险分析这一整套流程的自动化,而且还要在保持严格数据隐私标准的前提下完成。结果是他们给金融机构客户带来的审查产出提升了七十倍。这个案例其实挑战了一个常见的误解,很多人觉得风控这种高风险、强合规的场景,agent 是不敢碰的,但事实证明,只要设计得好,agent 完全可以在保证隐私合规的同时,大幅提升处理效率。

还有一个我个人觉得很有意思的案例,叫 BlueFlame,他们服务的是投资团队,利用 Claude 的视觉理解能力,去处理那种充满图表和曲线的金融文档。原本这种文档分析要花四个多小时,现在压缩到几分钟就能完成。这个平台让小规模的投资团队,能够做到原本需要庞大技术部门才能完成的机构级分析,平均每天能处理三十次客户查询,而且可以同时自动比较几百家公司。如果你是一个小型的私募或者投研团队,你大概能体会这种能力意味着什么,以前你可能只能挑重点公司深入研究,现在你可以让 agent 同时扫一遍几百家公司,把人力集中在真正值得深挖的标的上。

科研领域的案例也很值得说一说,叫 Biomni,是斯坦福开发的一个平台。它把科研分析的时间,从三周压缩到了三十五分钟,提速八百倍,同时在生物医学的基准测试上保持了人类专家级别的表现。这个平台能自动完成文献综述、生物信息学分析、实验方案设计。文章里提到一个细节,研究人员原本百分之八十的时间都花在重复性的工作上,现在可以同时推进多条研究线索,而且克隆实验的设计质量,被验证为相当于五年以上经验的专家水平。这个数字背后其实反映了一个更深层的问题,科研领域长期被大量机械性的文献筛选和方案设计工作拖慢节奏,如果这部分能被自动化接管,科研人员真正能做的事情,理论上是可以成倍增长的。

最后两个例子是大家可能比较熟悉的,Lovable 和 Genspark,都是低代码或者无代码开发平台。Lovable 通过让非工程师和开发者都能用自然语言对话的方式,跟 Claude 协作构建生产级的网页应用,开发速度是传统编码方式的二十倍,六个月之内做到了四千万美元的年化经常性收入,每个月活跃用户超过一百万,这些用户从 SaaS 产品到内部工具,什么都能做出来,而且完全不用写代码。Genspark 则是搭建了一个多 agent 任务自动化系统,用 Claude 作为八个专业 AI 模型的总协调者,在四十五天之内做到了三千六百万美元的年化经常性收入。他们的 AI 幻灯片功能,能把五分钟的自动化研究工作,压缩成相当于三小时人工努力的成果,这让用户可以去做以前规模和复杂度根本不可能靠人力完成的研究项目。

讲完这些案例,我想停下来,跟大家聊聊这些故事背后真正值得我们带走的东西,也就是文章接下来给出的实践建议部分,这部分我认为是整篇文章最实用的部分。

文章说,前面这些例子展示的是可能性,但是真正要走到那一步,需要从第一天起就在落地路径上做出正确的选择。如果一开始就选错了切入点,你会浪费很多时间,也会损耗团队对 agent 这件事的信任。反过来,如果从小处入手,选对了问题,你就能为后面逐步挑战更宏大的应用打下基础。

那么具体应该怎么选切入点呢。文章给出的建议是,优先选择那些本来就已经存在人工监督、而且自动化不完美也不会造成大问题的场景。这句话听起来有点绕,我们换个说法,意思就是,你不要一上来就把 agent 用在那种一旦出错就万劫不复的核心业务上,而是先找一个本来就有人在把关、出点小错也无所谓的环节,让 agent 先把大部分工作量接过去,人继续做最后的校验。

这样做的好处是,你能在没有生死攸关的压力下,让团队慢慢摸清楚 agent 到底是怎么工作的,它在哪些地方表现好,在哪些地方容易翻车,怎么去跟它磨合,怎么去调整提示词,怎么去对接你自己的数据系统。这些经验,会成为你后面去啃更难骨头的基础,因为越往后,你要处理的就是那种做对了影响更大、但是又很难做到事事都有人盯着的复杂高风险流程。

等到 agent 已经真正在处理实际工作,而且团队对它产生了信任之后,文章建议你可以往更大的方向去构建,也就是去打造跨职能通用的共享系统,而不是每个需求都单独造一个轮子。文章举了一个很形象的例子,一个能做个性化外联的 agent,可以同时承担好几种工作,给候选人发 HR 筛选问题,给潜在客户写定制化的营销邮件,跟投资人做后续跟进沟通。每一种使用场景的反馈,都能反过来提升这个 agent 写出符合语境的文案的能力。这其实是一种很聪明的复用思路,你不是在做三个孤立的小工具,而是在打磨同一个核心能力,让它在不同场景里都能派上用场,而且越用越准。

到这个阶段,你已经有 agent 在公司不同部门里处理真实的工作,你也已经打磨和验证了一些核心能力,被很多人持续在用,证明了 agent 确实能带来实质性的胜利,你的技术团队也积累了真正把 agent 跑起来的实战经验。文章说,这时候你才算真正准备好去追求更大的挑战了。

那么要成功挑战更大的目标,文章提出了三个必须从一开始就具备的能力,我觉得这三点是整篇文章里含金量最高的建议,值得展开多讲一点。

第一个能力是从一开始就用模块化的方式去构建系统。你的需求会不断演变,如果整个系统不是模块化的,每次需求变化你都要从头重新设计,这会浪费你之前积累的所有进展。模块化的系统能让你方便地替换某个组件,或者增加新的能力,而不需要把整个系统推倒重来。这个道理其实跟我们做软件工程是一样的,只不过放在 agent 系统上,影响会被放大,因为 agent 系统里往往涉及很多个步骤、很多个工具调用,如果耦合得太死,后期改动的成本会非常高。

第二个能力是让 agent 的决策过程是可见的,也就是可观测性。文章说得很直白,当出了问题,或者某个决策看起来不太对的时候,你需要能够准确地追溯到 agent 是怎么一步步得出这个结论的。最好的做法是从第一天起就把可观测性内建到系统里,如果等到 agent 系统已经变得很复杂了再去补这个能力,会非常痛苦。这一点我个人深有同感,任何自动化系统一旦缺乏可追溯性,出了问题排查起来都是噩梦,而 agent 系统因为涉及多步推理和多次工具调用,这个问题会比传统软件更严重,所以提前规划日志和追踪机制,真的是省心省力的关键。

第三个能力是在真正重要的环节保留人工审核。自主性固然强大,但是有些决策确实需要人的判断。文章建议的设计思路是,让 agent 学会识别哪些具体情况需要主动标记出来,提请人工批准,而不是走两个极端,一种是事事都要人来批,另一种是干脆全部自动放行不管。这其实是一种很务实的中间路线,既不浪费 agent 的自主能力,又不会把高风险的决策权完全交出去。

文章总结说,这三种能力,正是那些能够获得更多责任、被赋予更重要任务的 agent,跟那些只能停留在处理简单任务上的 agent 之间的分水岭。而打造这些能力需要时间和经验的积累,这也是为什么文章反复强调,一定要从小处开始。

文章最后还提炼了一个观察,那些早期就拿到明显胜利的创业公司,模式其实非常一致,他们都是挑一个高影响力的问题,这个问题恰好占用了他们最紧张的资源,可能是候选人筛选占用了创始人大量时间,可能是堆积如山的客服请求,也可能是销售线索的资质审核,拖住了销售团队接触真正有效客户的脚步。这些最早期的部署带来的价值,不只是立竿见影的收益,更重要的是,它们教会了创业公司很多只属于自己具体情境的东西,比如自己的数据要怎么跟 agent 对接,什么样的提示词在自己的业务场景里管用,工作流程需要怎么调整才能跟 agent 配合好。这些经验,就会变成后面规模化扩张的playbook,也就是行动手册。

那么这些内容,对我们这些听节目的工程师和 AI 从业者来说,到底意味着什么呢,我想结合自己的理解,聊聊几个可以直接拿来用的思路。

第一,如果你所在的公司是一家初创公司,或者你正在考虑要不要在公司内部推动 agent 落地,文章给出的那个排序逻辑是很值得借鉴的,先看时间约束,再看专业知识约束,最后看速度和质量的取舍约束,从这三个维度里找出你公司目前最痛的那一个,然后在这个维度里,挑一个本来就有人工兜底、出错代价不高的具体环节切入。千万不要一开始就挑那种一旦出错就影响客户资金安全或者法律合规的场景,哪怕那个场景看起来价值最大,因为你的团队还没有积累足够的经验去驾驭它。

第二,在做第一个落地项目的时候,哪怕规模很小,也要刻意地把模块化设计、可观测性、人工审核节点这三件事考虑进去,不要等到系统做大了再补。这一点说起来容易,做起来其实需要一点克制,因为很多时候为了赶进度,大家会先把功能跑起来,日志和追踪先放一放,但是从文章里这些案例反推回去,几乎所有做得比较成功的公司,都是把这些基础能力当成地基去打的,不是事后补救的。

第三,不要把 agent 局限在单一场景里使用,而是去思考这个 agent 背后真正打磨的核心能力是什么,这个能力是不是可以被复用到别的场景里。就像文章里说的那个个性化外联的例子,同一个写作和沟通能力,可以同时用在招聘、市场和投资人关系上,这种复用思路能让你的 agent 投资,带来更大的回报。

第四,如果你已经有了一些 agent 落地的经验,想要挑战更复杂的场景,建议先盘点一下,你现有的系统是不是足够模块化,决策过程是不是可以被追溯,关键节点是不是设置了人工审核。如果这三个问题里有任何一个答案是否,我个人建议先把这些基础打牢,再去推进更高风险的应用,否则规模一旦上去,出问题的代价会成倍放大。

最后,我想说说自己对这篇文章的整体感受。这篇文章用了非常多的真实案例去支撑论点,从会计、安防、客服、招聘、风控、科研到低代码开发,几乎覆盖了创业公司能想到的大部分职能,这种密集的案例堆叠方式,虽然是 Anthropic 自己的官方宣传材料,但是其中传递的方法论我认为是站得住脚的,尤其是那个从小处切入再逐步扩大的路径,以及那三个让 agent 能承担更大责任的能力要求,这些建议不依赖于具体的某个模型版本,放到今天甚至以后很长一段时间,应该都依然适用。当然,我们也要清楚地意识到,文章里引用的那些具体数字,比如成本降低百分之多少,营收做到多少,都是案例公司在二零二五年这个时间点上的快照,实际操作中每家公司的情况千差万别,这些数字更适合当作参考案例去理解可能性的边界,而不是直接套用的承诺。
