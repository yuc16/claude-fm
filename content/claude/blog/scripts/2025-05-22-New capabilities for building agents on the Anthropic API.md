先说一句提醒,今天是二零二六年六月二十一日,这篇文章距离现在已经过去整整一年多了。在AI这个领域,一年的时间跨度其实相当长,所以听这期节目的时候,大家要把它当成一个历史背景来理解,也就是说,我们今天要讲的这些功能,在当时是全新发布的beta测试版本,而到今天,它们很可能已经是Claude开发者工具箱里非常成熟、甚至是默认标配的能力了。这就好像我们现在回看一篇讲某个手机刚发布某项功能的新闻稿,功能本身可能已经迭代了好几代,但理解它最初是为了解决什么问题、当时的行业是什么状态,依然很有价值。

那我们先说说背景。二零二五年上半年,整个行业对大语言模型的关注点已经从"模型本身能不能聊天、能不能写代码"这种基础能力,逐渐转移到了"怎么让模型变成一个能自主完成复杂任务的智能体"。英文里这个词叫agent,中文我们一般翻译成智能体,也有人直接叫代理。智能体跟普通的对话助手最大的区别在哪里呢,普通助手是你问一句它答一句,而智能体是你给它一个目标,它会自己拆解步骤,调用各种工具,反复试错,直到把任务完成。

但是要做出一个真正好用的智能体,光有一个聪明的大模型是不够的,你还需要给它配套很多基础设施。比如说,智能体得能执行代码去做计算和验证,得能连接到外部的各种系统比如项目管理工具、文档系统,得能记住和复用之前处理过的文件,还得能在长时间运行的过程中不把成本和延迟搞得失控。在这篇文章发布之前,如果开发者想实现这些能力,基本上都得自己动手搭建。比如说,想让Claude执行代码,你得自己搭一个沙箱环境,自己处理代码执行的安全隔离;想接入外部工具,你得自己写客户端去对接每一个第三方服务的协议;想反复使用同一份文件,你得自己管理文件的存储和引用;想维持长时间的上下文,你得自己想办法压缩或者拆分对话历史。这些事情每一件单独拿出来都不算特别难,但加在一起,就成了开发者构建智能体路上一个很大的工程负担。

所以Anthropic在这篇文章里宣布的,就是在Anthropic API上一口气推出四个新能力,分别是代码执行工具、MCP连接器、文件API,以及把提示缓存的时间从五分钟延长到一小时的能力。这四个东西配合Claude Opus四和Claude Sonnet四这两个模型一起用,目标就是让开发者不用自己搭这些基础设施,直接调用现成的能力就能做出更强的智能体。

文章里举了一个例子我觉得特别能说明问题,就是一个项目管理智能体。这个智能体可以用MCP连接器去对接Asana这个项目管理工具,读取任务列表、分配工作;可以通过文件API上传相关的报告文档;可以用代码执行工具去分析项目进度和风险;同时还能借助延长版的提示缓存,在整个过程中始终维持完整的上下文,而且成本可控。你看,这四个能力其实不是孤立的功能点,而是被设计成可以组合使用的一整套工具链,这也是这篇文章最核心的一个思路,就是说我们不是在发布四个无关的小功能,而是在搭一整套智能体的基础设施。

好,我们接下来一个一个拆开讲。

首先讲代码执行工具,英文是code execution tool。这个工具的核心能力,是让Claude可以在一个沙箱环境里运行Python代码,沙箱的意思就是一个隔离的、安全的运行空间,代码在里面跑不会影响到外部系统,也不会有安全风险。有了这个工具之后,Claude就不再只是一个"告诉你应该怎么写代码"的助手,而是真正变成了一个可以自己跑代码、自己看结果、自己根据结果调整下一步操作的数据分析师。

这个区别其实挺大的。我们设想一个场景,你给Claude一份销售数据,让它帮你分析一下哪个季度增长最快。在没有代码执行工具之前,Claude可能会写一段Python代码给你,然后说你把这段代码拿去跑一下,看看结果。这中间你得自己复制代码、自己找环境运行、自己把结果再贴回去给Claude看。这一来一回,效率其实很低,而且容易出错,比如代码里有个小bug,你自己跑出来的结果是错的,Claude还以为是对的,继续往下分析,最后结论就全错了。

但有了代码执行工具之后,流程就完全不一样了。Claude自己写代码,自己在沙箱里执行,自己看到执行结果,如果发现图表画得不对,或者数据清洗有问题,它可以自己再迭代一版代码重新跑。这个过程是在一次API调用里完成的,不需要你来回搬运代码和结果。这就好像你请了一个真正的数据分析师坐在你旁边,他自己打开电脑、自己写脚本、自己看输出、自己发现问题自己改,而不是隔着一层玻璃,你帮他按回车键。

文章里列举了几个典型的应用场景,我觉得挺有代表性,我们可以展开说说。

第一个是金融建模,比如做财务预测、分析投资组合、计算各种复杂的财务指标。这种场景里,数据量可能不大,但计算逻辑很复杂,涉及很多公式和假设条件,代码执行工具可以让Claude直接把这些公式跑出来,而不是停留在口头描述上。

第二个是科学计算,比如跑模拟实验、处理实验数据、分析研究数据集。做科研的同学应该都有体会,很多分析工作其实就是写一堆脚本不断调试,代码执行工具让这个过程可以直接在对话里完成。

第三个是商业智能,也就是我们常说的BI,比如自动生成报表、分析销售数据、做业绩仪表盘。这块我觉得对企业内部的数据分析师特别有用,很多重复性的报表工作,完全可以交给Claude自动跑数,自动出图。

第四个是文档处理,比如在不同格式之间提取和转换数据,生成格式化的报告,自动化整个文档工作流。

第五个是统计分析,比如做回归分析、假设检验,在数据集上做预测建模。这些都是数据科学里比较经典的统计方法,现在可以直接通过代码执行工具来完成。

关于费用,文章里也给出了具体数字。每个组织每天有五十个小时的免费使用额度,超出之后,每个容器每小时收费零点零五美元。这个定价模式其实跟云计算里按容器、按小时计费的逻辑很像,如果你是中等规模的开发团队,五十个小时的免费额度其实已经能覆盖不少日常的分析任务了。

接下来我们说第二个能力,MCP连接器。MCP的全称是Model Context Protocol,中文一般翻译成模型上下文协议,这是Anthropic之前推出的一个开放协议,目的是让AI模型可以用一种标准化的方式去连接外部的数据源和工具。你可以把MCP理解成一种通用插头,以前每个第三方服务都有自己的协议和接口,模型要接入每一个服务,都得专门写一套对接代码,就像每个国家的电源插座形状都不一样,你出国旅行得带好几种转换插头。而MCP想做的事情,就是定义一种统一的标准接口,只要服务方按这个标准把自己的工具暴露出来,任何支持MCP的模型都可以直接接上去用,不用再为每一家单独写代码。

那这次发布的MCP连接器,具体解决了什么问题呢。在这之前,虽然MCP协议本身已经存在,但开发者想要让Claude通过API去连接一个远程的MCP服务器,还是得自己搭建一个客户端框架,去处理连接管理、工具发现、错误处理这些细节工作。这就好比说,插头标准是定好了,但你还得自己组装一个转换器才能用,挺麻烦的。

而MCP连接器做的事情,就是把这个转换器内置到Anthropic API里了。现在开发者只需要在API请求里加上一个远程MCP服务器的网址,剩下的事情API会自动帮你搞定。具体来说,当Claude收到一个配置了MCP服务器的请求之后,它会自动完成这么几件事:首先连接到指定的MCP服务器,然后获取这个服务器上有哪些可用的工具,接着会自己推理判断应该调用哪个工具、传什么参数,然后一直执行工具调用,直到得到一个足够好的结果为止,这个过程是自主的、agentic的,也就是说Claude会自己决定要不要继续调用更多工具。同时,整个过程中的身份验证和错误处理也都是自动管理的,最后把整合了外部数据的最终结果返回给你。

我觉得这个设计思路特别值得一提,就是它把原本属于工程基础设施的那部分工作,从开发者手里挪到了API层面去做。开发者要做的事情,就从"写一套完整的客户端逻辑去管理MCP连接"变成了"在请求里填一个网址"。这个简化的幅度其实是很大的。

文章里也提到了,当时已经有越来越多的远程MCP服务器可以直接用,包括像Zapier和Asana这样的服务,这意味着开发者可以很轻松地往自己的AI应用里加各种能力,而不用每次都去做一次性的、定制化的集成工作。我个人觉得这一点其实是MCP整个生态能够发展起来的关键,因为如果每接一个新工具都要重新写一遍对接代码,这个生态是很难滚动起来的,只有把对接成本降到足够低,才会有越来越多的第三方愿意把自己的服务做成MCP服务器挂出来,然后开发者才能像逛应用商店一样,挑哪个工具就接哪个工具。

第三个能力是文件API,英文是Files API。这个功能解决的问题听起来很朴素,但实际上是开发者天天都会遇到的痛点,就是文件管理的问题。

我们设想一下,如果你在做一个需要持续引用同一份文档的应用,比如说一个客服机器人需要反复查阅产品手册,或者一个分析助手需要反复读取同一份财报数据,在文件API出现之前,意味着你每次发起一次新的对话或者新的请求,都得把这份文件重新上传一遍。这不仅浪费带宽和时间,也意味着每次请求的体积都变大了,处理起来更慢,成本也更高。

文件API做的事情很直接,就是让你可以把文档上传一次,然后在之后的多次对话里反复引用这份文件,不需要重复上传。这对于那些需要处理大量文档集合的应用特别有用,比如说企业知识库、技术文档、各种数据集这类场景。你可以想象一下,一个企业内部的知识库助手,可能要服务成百上千次的查询,如果每次查询都要重新传一遍几十页的文档,那效率和成本都是不可接受的,而现在只需要传一次,后面所有的查询都直接引用这份已经上传好的文件就行。

更有意思的是,文章里提到文件API还会跟代码执行工具结合起来用。也就是说,Claude可以在执行代码的过程中,直接访问通过文件API上传的文件去做处理,处理完之后还可以把结果,比如生成的图表和图形,作为响应的一部分返回给你。这意味着开发者可以把一份数据集通过文件API上传一次,然后让Claude在多次会话里反复对这份数据做分析,而不需要每次都重新上传。我觉得这个组合特别巧妙,等于是把"存数据"和"算数据"这两件事打通了,数据只要进来一次,后面怎么用、用几次,都不用再操心传输的问题。

第四个能力,是延长版的提示缓存,英文是extended prompt caching。这个功能我觉得需要先解释一下提示缓存本身是什么。提示缓存的英文是prompt caching,它的基本原理是,如果你每次给模型发请求的时候,前面有一大段内容是重复的,比如说你给Claude设定的系统提示、背景知识、参考文档,这些内容在很多次请求里都是一样的,那模型其实没必要每次都重新处理这一整段内容,而是可以把这部分处理结果缓存起来,下次请求过来的时候直接复用,这样既能省时间也能省钱。

那这次的更新是什么呢,就是缓存能保留的时间变长了。原来标准的缓存时间是五分钟,也就是说缓存内容五分钟之内有效,超过五分钟就失效了,需要重新计算。这次新增了一个选项,开发者可以选择把缓存时间延长到一小时,代价是这个延长版缓存会比标准版贵一些,但换来的是十二倍的时间延长。文章里给出的具体数字是,用了延长版缓存之后,对于长提示词的场景,成本最多可以降低百分之九十,延迟最多可以降低百分之八十五。

这个数字其实挺惊人的,我们来理解一下为什么会有这么大的差别。设想一个智能体应用,它需要带着一大段背景知识和示例去工作,比如说一个法律文档审查助手,每次审查都需要带着几十页的法规条文作为背景。如果用标准的五分钟缓存,那只要这个智能体处理一个任务花的时间超过五分钟,缓存就失效了,下一次请求又得把那几十页法规重新处理一遍,这个开销会随着任务时间拉长而不断累积。但如果用一小时的缓存,只要整个任务在一小时内完成,这几十页法规条文就只需要处理一次,后面所有的请求都是直接复用缓存的结果,省下来的计算量是非常可观的。

这也是为什么文章里特别强调,这个能力让那种长时间运行的智能体应用变得实际可行了。之前如果一个智能体任务要跑很长时间,比如处理一个多步骤的复杂工作流,或者分析一份很长的文档,或者要跟其他系统协调配合,这种场景下成本可能会高得让人望而却步,但现在用上延长版缓存之后,这些长时间运行的应用就能以更高效率、更低成本规模化地运行起来了。

好,讲完了这四个能力,我们再回到开头那个项目管理智能体的例子,把它们串一遍,我觉得这样能更直观地理解这套组合拳是怎么打的。这个智能体首先通过MCP连接器接入Asana,不需要自己写客户端代码,直接就能读取任务列表、给团队成员分配工作。然后通过文件API,把项目相关的报告文档上传一次,后续反复引用,不用每次都重新传。接着用代码执行工具,对项目进度数据做计算分析,识别出潜在的风险点,可能还会自动生成一些图表帮助团队成员直观地看到进度情况。整个过程中,因为用了延长版的提示缓存,这个智能体可以一直维持着完整的上下文信息,知道项目背景是什么、之前做过什么决策,而且因为缓存命中率高,成本也控制得住。你看,这四个功能单独看好像都是独立的工具,但组合在一起,就拼出了一个能够真正自主运行、办成事情的智能体。

那么这些能力对我们今天的实际工作,到底意味着什么呢,我们来聊聊这部分。

首先,如果你是一个正在用Claude API做开发的工程师,我会建议你先去看一看现在Anthropic的官方文档,确认一下这几个能力现在的可用状态,因为这篇文章发布已经一年多了,这期间这些beta功能很可能已经经历了多轮迭代,定价、接口细节、支持的模型版本都可能有变化。我们这期节目讲的是这些功能最初发布时的设计思路和能力边界,具体怎么用、现在的价格是多少,一定要以最新文档为准。

其次,从架构设计的角度,我觉得这篇文章给我们的一个启发是,如果你要自己做一个智能体应用,在决定要不要自建某个基础设施之前,先看看Anthropic或者其他平台是不是已经提供了现成的能力。比如说,如果你之前自己搭了一套代码沙箱环境去支持Claude执行代码,现在完全可以评估一下直接切换到官方的代码执行工具,省掉自己维护沙箱安全性的负担。再比如,如果你之前为了对接某个第三方工具自己写了一套客户端逻辑,现在可以看看这个工具有没有出MCP服务器版本,如果有,直接通过MCP连接器接入,能省掉大量的维护工作。

第三,关于MCP这个协议本身,我觉得值得单独说一句。MCP现在已经不只是Anthropic自己在推的东西了,它已经发展成一个比较通用的行业协议,不少其他厂商和开源项目也开始支持MCP。所以如果你在做工具集成相关的工作,学习和采用MCP协议,而不是为每个工具写定制化代码,从长期来看是更有维护性的选择。这就好比早年互联网刚兴起的时候,大家逐渐统一用HTTP协议来传输网页内容,而不是每个网站都搞一套自己的传输方式,标准化之后整个生态才能跑得更快。

第四,关于成本控制,我想提醒大家注意一个细节,就是延长版缓存虽然单价更贵,但它适用的场景是有讲究的。如果你的应用场景本身请求频率很高,两次请求之间间隔很短,完全用不到一小时那么长的缓存窗口,那标准的五分钟缓存可能就够用了,没必要为了用延长版而多花钱。但如果你的智能体任务天然就是长流程的,比如说需要好几十分钟甚至接近一小时才能跑完一个完整的工作流,那延长版缓存带来的成本节省就会非常明显。所以这是一个需要结合自己业务场景做权衡的选择,不是说越贵越好,也不是说越便宜越好,关键是匹配你实际的请求节奏。

第五,我想提一下踩坑的问题。代码执行工具因为涉及到运行代码,即使是在沙箱环境里,大家在设计应用的时候也要想清楚,哪些数据是可以放进这个执行环境的,哪些敏感数据可能不适合直接交给自动化的代码执行流程去处理。沙箱隔离解决的是运行环境本身的安全问题,但数据本身的敏感性和合规性,还是需要开发者自己把关。同样,用MCP连接器接入外部系统的时候,也要注意给Claude的权限范围,因为它是自主决定调用哪些工具、传什么参数的,如果权限设置得太宽,可能会执行一些你没预期到的操作,这个在设计阶段就要把边界划清楚。

好,我们最后总结一下这篇文章。Anthropic在二零二五年五月这个时间点,通过发布代码执行工具、MCP连接器、文件API,以及延长版提示缓存这四个能力,本质上是在补齐智能体开发所需要的基础设施,让开发者不用什么都自己搭,就能做出能跑代码、能连外部系统、能管理文件、能维持长上下文的智能体。这四个功能彼此独立,但又被设计成可以组合使用的一整套工具链,这种"既能单独用,也能拼起来用"的设计思路,我觉得是这篇文章最值得记住的一点。

我个人认为,这篇文章背后反映的其实是整个行业在二零二五年中期那个阶段的一个共识,就是单纯比拼模型能力强不强,已经不是竞争的全部了,谁能把围绕模型的这套基础设施做得更顺手、更省心,谁就能让开发者更快地把好的智能体应用做出来。从今天回头看,这个判断应该说是站住了的,因为后面一年多的时间里,整个行业在智能体基础设施这个方向上确实投入了大量精力。所以即使这篇文章本身已经有点"过时"了,它代表的这个发展方向,我觉得依然是理解今天AI智能体生态的一个很好的切入点。
