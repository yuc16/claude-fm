## Script

大家好，欢迎收听 Claude FM，我是你们的主播。今天我们要聊的这篇文章，标题是 Building Agents with Skills, Equipping Agents for Specialized Work，翻译过来就是用技能打造智能体，让智能体具备专业能力。这篇文章是 Anthropic 官方博客发的，由他们团队的几位工程师共同撰写，发表时间是二零二六年一月二十二日，距离我们录这期节目，已经过去差不多五个月了。这五个月里 AI 这个领域变化依然很快，但今天要讲的这套思路，到现在依然是 Anthropic 整个智能体战略里最核心的一块拼图，所以非常值得我们花时间拆解清楚。

这篇文章其实回答了一个很多做智能体应用的人都会纠结的问题，就是我到底要不要给每个场景做一个专门的智能体。文章给出的答案是不需要，未来的方向是一个通用的智能体，加上一套叫 Skills 的东西，中文我们可以理解为技能。我们今天就把这套思路彻底讲透，并且聊聊它对我们日常做开发、做产品到底意味着什么。

先说说背景。过去这一年多，智能体这个领域发生了几件大事。第一件是 MCP，也就是模型上下文协议，迅速成为了智能体连接外部工具和数据的行业标准，从各大厂商到开发者社区都在快速采纳。第二件事是 Claude Code 的发布，它一开始被定位成一个写代码的智能体，但慢慢地大家发现它其实是个通用智能体，只是用代码作为它做事的方式。第三件事是 Claude Agent SDK 的发布，这给了开发者一套开箱即用、可以直接用于生产环境的智能体框架。

但是 Anthropic 在把这些智能体真正用到实际工作中的时候，反复撞到同一个问题。智能体已经足够聪明，工具也足够全，但它们往往缺乏处理真实工作所需要的那种专业经验。正是这个落差，催生了 Agent Skills 这个东西。Skills 本质上是一组组织好的文件，里面打包了某个领域的专业知识，包括具体的工作流程，最佳实践，还有可以直接执行的脚本。智能体可以去访问这些文件，并且应用里面的知识。用文章里的话说，这套东西能把一个能力很强的全能选手，变成一个真正懂行的专家。

我们先讲第一个核心观点，就是从专用智能体到通用智能体加技能这个转变。Anthropic 团队说，他们以前以为不同领域的智能体应该长得完全不一样。一个写代码的智能体，一个做研究的智能体，一个做金融分析的，一个做市场营销的，每一个似乎都需要自己专属的工具和框架。这其实也是这个行业早期普遍采用的思路，几乎所有公司做垂直智能体产品，第一反应都是去搭建一套独立的系统。

但随着模型本身的智能水平不断提升，智能体处理复杂任务的能力也越来越强，Anthropic 的判断变了。他们现在认为，代码与其说是一个具体的应用场景，更应该被理解为智能体完成几乎任何数字工作的一个通用接口。这句话听起来有点抽象，我们用一个具体例子说明。假设你让 Claude Code 帮你生成一份财务报告，它可以调用各种 API 去做研究，把抓回来的数据存到文件系统里，再用 Python 去做数据分析，最后把分析结果综合成一份有逻辑的报告。整个过程，从抓数据到分析到输出，全部都是通过写代码、跑代码完成的。所以最后你需要的脚手架，说白了就是一个命令行环境加一个文件系统，非常简单。这就是为什么 Claude Code 看起来是个编程工具，骨子里其实是个通用智能体，只是它做事的方式是写代码。这个转变背后的逻辑很重要，如果代码本身就是足够通用的接口，那你就不需要为每个垂直场景单独造一套智能体系统，只需要一个足够强的通用智能体内核，再配上不同领域的知识就行。

这就引出了第二个核心问题，智能不等于专业，这个落差怎么填。文章里抛出一个很形象的类比，我自己听完印象很深。它问你，如果要报税，你会找谁，是找一个数学天才，从第一性原理出发自己推导整套税法逻辑，还是找一个已经报了几千份报税单的资深税务师。绝大多数人会选后者。不是因为税务师比数学家聪明，而是因为他有正确的专业经验积累。

今天的智能体，处境其实很像那个数学天才。它推理能力很强，遇到没见过的新情况也能现场想办法解决，这点确实厉害。但它往往缺的是一个老法师式的专业人士那种经过反复实践积累出来的经验。在合适的引导下，智能体能做出让人惊艳的事情，但问题在于，它经常缺一些重要的背景信息，没法轻易把你们公司、你们团队这么多年积累下来的专业知识直接吸收进去，而且它也不会因为反复做同一类任务，就自动学会越做越好。Skills 要填的正是这个坑，思路是把专业领域的知识和具体的操作流程，打包成一种智能体可以分阶段去访问、分阶段去应用的格式。这里的关键词是分阶段，我们接下来展开讲讲，这其实是 Skills 设计里非常巧妙的一个机制。

我们具体看一下一个 Skill 的文件结构。文章举了一个例子，是 Anthropic 自己内部用的一个品牌风格指南技能，文件夹里大概有这么几个文件，一个叫 SKILL 点 md 的核心说明文件，一个叫 docs 点 md 的文档，一个专门讲怎么做幻灯片的文档，还有一个可以直接执行的 Python 脚本，用来给 PPT 套用品牌模板。

这套设计为什么故意做得这么简单，文章里说得很清楚，因为文件是一种万能的基础形式，它能跟你现在已经在用的工具无缝衔接。你可以用 Git 给它做版本管理，可以存在 Google Drive 里，也可以直接分享给团队里的其他人。正因为简单到只是文件，做 Skill 这件事就不再是工程师的专利了。产品经理、数据分析师，还有各个领域的专家，已经在用这套东西把自己的工作流程沉淀下来。这一点特别值得强调，因为它意味着技能这个东西的生产端被极大地拓宽了，以前你要给智能体加能力，得找工程师写代码接 API，现在一个不会写代码的业务专家，也能把自己脑子里的经验整理成一份 Skill 交给智能体用。

那 Skills 里能装的内容理论上可以很多，怎么避免把模型的上下文窗口塞爆呢，这就是渐进式加载机制要解决的问题。整套机制分成三层。第一层，在运行时，模型默认只能看到这个技能的元数据，也就是头信息里的名字和描述，这一层大概只占用五十个 token 左右，token 你可以理解成模型读写文字的最小计量单位。第二层，如果 Claude 根据这个简短描述判断这个技能跟当前任务有关，它才会去读取完整的 SKILL 文件，这一层大概是五百个 token。第三层，如果还需要更详细的资料，技能里可以再放一个参考文档目录，里面是更详尽的内容，这部分可能两千个 token 起步，但只有在确实需要的时候才会被加载进来。

这套三层结构带来的好处是，你理论上可以给一个智能体配上成百上千个技能，而不会一下子把它的上下文窗口塞满，因为大部分技能平时都只是用几十个 token 的元数据安静地待在那里，只有真正被用到的那个技能，才会一层层往下展开，加载更多细节。这跟人脑其实有点像，你脑子里存了很多技能和知识，但你不会同时把所有记忆都激活，只有遇到对应场景，相关的那部分知识才会被调用出来。

接下来我们讲第三个重点，为什么是代码和文件，而不是传统意义上的工具调用。文章专门拿传统的工具调用，也就是我们平时说的 function calling，和 Skills 做了对比。传统工具的问题在哪，第一，很多工具的说明写得很粗糙，模型很难准确判断什么时候该用、怎么用。第二，模型往往没法去修改或者扩展这些工具，工具是死的，模型用得别扭也没办法。第三，工具的定义经常会占用大量上下文窗口，因为不管用不用都得提前塞进去。而代码和文件正好相反，代码是自解释的，写得好的代码本身就是文档，而且模型可以随时去修改它、扩展它，文件这种载体又完全不需要一直占着上下文，需要的时候再读取就行。

文章里举了一个特别真实的例子，我觉得很有代表性。Anthropic 团队发现，Claude 在处理幻灯片任务的时候，反反复复都在写同一段脚本，去把 Anthropic 的品牌样式套到 PPT 上。既然这个脚本反复被需要，那就让 Claude 自己把它保存下来，变成自己可以复用的工具。对应的文档里，不需要把脚本逐行解释，只需要简单写清楚品牌规范，比如开场结尾页用什么背景色什么前景色，章节页又用什么颜色，然后告诉模型，需要更新 PPT 样式的时候直接调用这个脚本就行。这个例子特别能说明 Skills 的精髓，就是把重复出现的具体操作沉淀成可执行的代码，把需要被理解的背景知识沉淀成简洁的文字说明，两者配合，既省上下文，又保证执行的稳定性，毕竟脚本跑出来的结果，比每次让模型现场重新生成代码靠谱得多。

文章说现在市面上的 Skills 大致可以分成三类。第一类是基础类技能，解决几乎所有人都需要的通用能力，比如处理文档、操作表格、制作演示文稿，把文档生成和处理的最佳实践直接封装进去，Anthropic 把这部分基础技能开源放在了一个公开仓库里，任何人都可以去看。第二类是生态伙伴类技能，一些公司开始把自己的服务做成技能的形式，让 Claude 可以直接调用，文章里提到了几个名字，这些公司在用 Skills 这套标准把自己的服务接进 Claude 的能力体系里，既扩展了 Claude 在特定领域的能力，又保持了简洁性。第三类是企业私有技能，企业把自己内部的流程、合规要求、机构知识，沉淀成专属的技能。这部分对很多公司来说才是真正的护城河，因为这些技能里装的，是别人复制不走的内部经验。

除了这三种分类，文章还总结了几个正在发生的趋势。第一个趋势是，技能的复杂度正在快速攀升。早期的技能往往就是简单的文档引用，告诉模型去哪儿查资料。但现在已经出现了非常复杂的多步骤工作流，能够协调数据抓取、复杂计算，还有跨多个工具的格式化输出。文章给了三个具体的复杂度档次。简单级别，比如一个状态报告撰写器，大概一百行说明，主要做模板和格式化的工作。中等级别，比如一个财务模型构建器，大概八百行，要做数据抓取，还要用 Python 去搭建 Excel 模型。复杂级别，比如一个 RNA 测序流程的技能，代码量超过两千五百行，要协调好几个生物信息学分析工具一起跑完整套流程。从一百行到两千五百行，说明 Skills 这套格式弹性很大，简单需求和极其复杂的专业流程都能装得下。

第二个趋势是，Skills 和 MCP 这两套体系正在自然地结合起来用。一个典型场景是做竞争分析，这个技能可能需要同时协调网络搜索、通过 MCP 接入的内部数据库、聊天历史记录，还有协作文档里的内容，把这些信息揉在一起，最后产出一份综合分析报告。换句话说，MCP 负责把外部的工具和数据连进来，Skills 负责告诉模型这些工具该怎么组合着用、用在什么场景，两者是互补关系，不是竞争关系。

第三个趋势是技能创作正在加速向工程师之外的人群扩散。产品经理、数据分析师，还有各行各业的专家，现在已经可以借助一个交互式的技能创建工具，在半小时之内创建并测试出自己的第一个技能，这个工具会一步步引导你完成整个创建过程。Anthropic 也明确说，他们还在持续投入，让创建技能这件事变得更简单，会推出更好用的工具和模板，让任何人都能把自己的经验沉淀下来、分享出去。

把前面这些内容综合起来，文章给出了一个清晰的架构图景，整套智能体系统由四层组成。第一层是推理循环，这是核心的决策系统，负责判断下一步该做什么。第二层是执行环境，包括代码运行环境和文件系统。第三层是 MCP 服务，负责连接外部工具和数据源。第四层是技能库，装的是领域专业知识和具体的操作流程。这四层各自的职责非常清楚，推理循环负责思考，执行环境负责执行，MCP 负责连接，技能负责指导。文章特别强调，这种分层让整套系统变得容易理解，而且每一层都可以独立演进，不会互相牵扯。这有点像软件系统里的分层架构，业务逻辑、数据访问、外部接口分开放，谁要升级谁，不需要把整栋楼都拆了重建。

文章举了个例子说明这种架构的灵活性，假设你往这套系统里加一个前端设计技能，这个技能会立刻让 Claude 在做前端开发的时候表现完全不一样，它能提供关于排版、色彩理论、动效设计这些专业指导，而且因为渐进式加载的机制，这个技能只有在真的要做网页界面的时候才会被激活，平时不会一直占着资源。给智能体加新能力这件事，正在变得跟插一个模块一样简单，需要什么功能，挂一个对应的技能进去就行。

接下来文章讲了两个实际落地的垂直行业案例，金融服务和医疗健康，这部分很能体现 Skills 这套思路真正在商业场景里能产生什么价值。在金融服务领域，刚发布 Skills 之后不久，Anthropic 就给金融行业版的 Claude 配了一整套针对性的技能。比如有一个专门做现金流折现模型的技能，能帮你构建估值模型，还会正确处理加权平均资本成本的计算，以及敏感性分析。还有一个做同业公司对比分析的技能，可以自动生成对比表格，列出相关的估值倍数，做行业内的对标。还有专门做财报解读的技能，能处理季度财务数据，生成投资跟踪报告。还有专门做新股研究覆盖的技能，能搭建包含财务模型的完整研究报告。还有专门做并购尽职调查的技能，能用标准化框架来梳理并购分析。还有专门做客户路演材料的技能，能按照行业惯例制作面向客户的演示文稿。你可以想象一下，这相当于把投行分析师团队里那些重复性强但又需要专业判断的标准化工作流程，一个一个打包成了技能，配给了 Claude。

在医疗健康和生命科学领域，Anthropic 同样配了一套技能，面向研究人员、临床医生和医疗软件开发者。有专门针对生物信息学的技能包，覆盖两个常用工具的部署，这两个工具在管理基因组数据流程和单细胞测序分析里是常用的基础设施。有专门做临床试验方案生成的技能，能加快临床研究方案的撰写速度。有专门帮研究人员做科学问题筛选的技能，帮你判断和定义出真正有价值的研究问题。有专门做医疗数据互操作标准开发的技能，能帮开发者写出更准确的代码，让不同医疗系统之间的数据对接更快、出错更少。还有一个特别有现实意义的，专门做事前授权审核的技能，能通过交叉核对保险覆盖范围、临床指南和患者记录，大幅减少行政负担，让患者能更快拿到需要的医疗服务。这一条挺打动我的，因为医疗体系里行政审批的繁琐和拖延是个老大难问题，如果技能真的能把这部分流程效率提上去，受益的是实实在在的病人。

文章最后讲了一个非常重要的战略决定，就是 Anthropic 正在把 Agent Skills 作为一个开放标准来发布。跟 MCP 走的是同一条路，Anthropic 认为技能也应该是可以跨工具、跨平台移植的，同一个技能不管你用的是 Claude 还是其他的 AI 平台，理论上都应该能跑起来。文章提到他们已经在跟生态里的一些伙伴合作打磨这套标准，并且已经看到了早期的采纳。这个开放标准化的动作，跟 MCP 当年走的路径很像，先在自己的产品里把这套东西验证清楚，再把它变成一个行业共享的协议，吸引更多平台和团队一起共建。文章末尾还附了一些延伸资源，包括技能文档、公开仓库、技能创建手册之类，方便大家进一步学习，我们这里就不展开了。

好，核心内容讲完了，我们聊聊这对国内做 AI 应用、做智能体开发的朋友来说，到底意味着什么，能怎么上手。

如果你是个独立开发者，或者你们团队正在用 Claude 系列模型去搭建一些智能体应用，第一个能立刻用起来的动作，是去梳理一下你们团队里那些重复出现、但又依赖专业判断的工作流程。不一定是多复杂的任务，哪怕只是一份格式固定的周报怎么写，一份对外的报价单按什么规范出，这些看起来琐碎的流程，其实都是做成 Skill 的好素材。按文章说的，做一个简单的 Skill 可能就一百行左右的说明文档，半小时就能弄完一个雏形，门槛真的不高。

第二个建议是，不要把 Skills 想成只是写文档，它真正的威力在于把可执行的代码和说明文字结合起来用。如果你发现模型反复在某个任务上写差不多的代码，那这就是一个信号，提示你应该把这段代码固定下来，存成一个脚本，再配一份简短的说明文档告诉模型什么时候用它、怎么用它。这样做的好处很明显，脚本跑出来的结果是确定的、可复现的，而让模型每次现场重新生成代码，质量是会波动的，运行结果也不一定稳定。

第三个建议是，注意渐进式加载这个机制带来的设计启发。如果你在设计自己的技能库，不要把所有信息都一股脑塞进主文件里，试着把核心的、必须知道的信息放在 SKILL 文件本身，把那些只有偶尔才会用到的细节资料，挪到一个单独的参考文档目录里。这样既能保证技能描述足够清楚，让模型一眼能判断该不该用它，又不会把不必要的细节一直堆在上下文里占地方。这一点对于做企业级应用尤其重要，因为企业内部的技能库往往会越攒越多，如果每个技能都把所有细节一次性塞进核心文件，上下文窗口很快就会吃紧。

第四个建议是，多关注 Skills 和 MCP 的搭配使用。如果你们公司已经有一些 MCP 服务，接了内部系统或者第三方数据源，那么下一步可以考虑，针对某个具体的业务场景写一个 Skill，在这个 Skill 里明确告诉模型，遇到这种场景应该先用哪个 MCP 工具去抓哪部分数据，再按什么顺序处理，最后用什么格式输出。这种组合用法，相当于把单独的工具接口，编排成一套有业务逻辑的工作流，而这恰恰是很多企业落地智能体最缺的那一环，工具都接好了，但缺一个知道怎么把这些工具串起来用的大脑。

第五个建议，如果你所在的公司有合规要求、内部规范、或者多年沉淀下来的方法论，这些信息特别适合做成私有技能，而不是简单地写进一份内部文档然后指望大家记住。文章里反复强调一个观点，新来的人或者新接入的智能体，应该一上手就自动知道你们团队看重什么，做到这一点的方式，就是把这些经验编码进技能，让任何接入的智能体直接读取应用，不需要每次都重新培训一遍。

需要避的坑，我个人觉得有这么几点值得提前想清楚。第一，不要为了做技能而做技能，如果某个任务本身就很简单，模型现场处理也很可靠，那硬把它包装成一个正式的技能文件，反而增加了维护成本。技能这个机制最适合的场景，是那些重复出现、有一定专业门槛、并且值得被沉淀复用的流程。第二，技能里的脚本和文档需要跟着业务规则的变化及时更新，毕竟这些内容是静态的文件，不会像模型本身一样自动学习新知识，如果你们公司的品牌规范变了，估值方法变了，技能文件里的内容也得跟着改，不然模型会一直照着过时的规则去执行。第三，技能开源和跨平台移植目前还处于早期阶段，标准还在演进，如果你现在要往这个生态里投入大量资源，建议先关注一下官方文档和社区的最新进展，确认接口和格式没有大的变动，再决定要不要大规模铺开。

我们简单回顾一下今天这篇文章讲的核心逻辑。Anthropic 的判断是，未来不需要给每个垂直场景单独造一个智能体，真正该投入精力的方向，是打磨一个足够通用、足够聪明的智能体内核，然后通过 Skills 这种轻量级的文件化机制，把各行各业、各个团队积累下来的专业知识源源不断地喂给它。Skills 之所以好用，关键在于它的形式足够简单，简单到只是文件，又足够聪明，靠渐进式加载，保证你可以塞进成百上千个技能而不会撑爆上下文窗口。再配合 MCP 负责连接外部世界，整套架构层次清晰，谁负责思考、谁负责执行、谁负责连接、谁负责传授经验，分得明明白白。

我个人认为，这篇文章里最值得记住的一句话，是那个数学天才和资深税务师的类比。模型的智能水平这两年涨得非常快，但智能从来不等于经验，真正决定一个智能体在具体业务里好不好用的，往往不是它推理能力多强，而是它有没有吸收到那些只有干这行多年才攒得下来的诀窍。Skills 提供的正是这样一条把诀窍传递给机器的路径，而且因为它低门槛到产品经理和业务专家都能直接上手写，这意味着以后沉淀专业知识这件事，不再只是工程师的工作。考虑到 Anthropic 还打算把这套标准开放出来，跨平台通用，我觉得这件事的影响力会比看上去更大，它很可能会变成下一阶段智能体生态里大家共同遵守的一套基础协议。
