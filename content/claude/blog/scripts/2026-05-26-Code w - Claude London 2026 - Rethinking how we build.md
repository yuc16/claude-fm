我们先讲第一条线索,Boris Cherny 的主题演讲。Boris 现在是 Anthropic Claude Code 这个产品线的负责人,他开场讲的不是什么宏大的行业趋势,而是他自己中学时候的一段经历。他说他第一次感受到编程这件事有魔法,是在中学的时候,他用德州仪器的 TI 八十三型号计算器,自己写了一些小程序,直接帮他做数学作业和考试。后来他又自学了 HTML,目的特别朴素,就是为了让自己在 eBay 上卖宝可梦卡片的商品页面做得更好看,卖得更好。你听这两个例子会发现,他完全不是奔着什么宏伟的工程目标去学编程的,他就是手痒,想解决眼前一个具体又小的麻烦,写出来的东西能跑起来,这件事本身就让他觉得兴奋。

这种兴奋感,我们平时管它叫,代码跑起来的那种成就感,英文里有时候会说,calculator feeling,计算器式的快感,意思是你按下等号,答案立刻就出来了,中间没有冗余的步骤,反馈是即时的。Boris 在演讲里提出一个挺尖锐的观察,他说,不知道从什么时候开始,写程序这件事变复杂了。中间插进来了编译器,插进来了类型检查工具,插进来了五花八门的构建系统,这些东西本身都是为了让软件更可靠、更工程化而存在的,出发点没有错,但它们客观上做的一件事,是把,我有一个想法,和,这个想法真的跑起来了,这两个时刻之间的距离,一点一点地往外推。你脑子里冒出一个点子,到真正看到它运行起来,中间要经过的步骤越来越多,反馈越来越慢,那种小时候按一下等号就出答案的轻松感,慢慢就被磨没了。

我自己听到这一段的时候,脑子里冒出来的画面是,我们很多人学 programming 的路径,基本都是先装好一整套开发环境,配好版本管理工具,搞清楚项目里十几个依赖包之间会不会打架,改一行代码,要等编译,等类型检查跑完,等构建流水线跑完,才能看到结果。整个过程更像是先学会开一辆手动挡卡车的全部操作步骤,离合、换挡、看仪表盘,一样都不能漏,然后才被允许上路。而 Boris 怀念的中学计算器编程,更像是骑自行车,你跨上去,踩两下,车就动了,中间没有那么多前置步骤。他想说的是,agent 把那种骑自行车式的轻快感,重新带回到了原本需要开卡车才能完成的复杂任务上,这才是这段回忆真正想传达的东西,不是单纯的怀旧,而是在说,门槛被压低了,但你能做的事情的规模,反而被放大了。

Boris 接着说,有了 agent 之后,这个被拉长的距离,正在重新被压缩回去。你描述一个问题,程序就自己出现了。他用了一个很形象的说法,他说这就是计算器式的快感,只不过这次这个计算器,能帮你写出一整套分布式系统。我觉得这个比喻特别精准,它说的不是 agent 让编程变简单了这种空泛的话,而是说,agent 把那种最原始的,反馈即时、动手就能验证的体验给找回来了,只是把它套用到了远比中学计算器程序复杂得多的问题规模上。如果你愿意顺着这个比喻往下想一层,以前你要靠十年专业训练才能驾驭的复杂系统,现在用最朴素的手痒冲动就能去碰一碰,这中间的门槛差,才是真正被重新定义的东西。

讲完了情怀这条线,我们再来看大会上正式发布的两项平台能力,这才是这篇文章信息密度最高的部分。这两项能力都挂在 Claude Managed Agents 底下,Managed Agents,你可以理解成 Anthropic 帮你托管整个 agent 运行循环的一种服务形态,你不需要自己从零搭建一套系统去处理任务编排、上下文管理、出错之后怎么重试这些细碎又容易出 bug 的工作,这部分逻辑由 Anthropic 在背后帮你管。这次大会带来的更新是,这套托管服务现在可以伸进你自己控制的环境里执行任务,并且可以连到你自己内网里的 MCP 服务器。我们一个一个拆开讲。

第一项叫自托管沙箱,现在是公测阶段。我们先说说沙箱这个词在这里的意思,你可以把它理解成一个隔离出来的执行环境,agent 要敲命令、跑代码、读写文件,都在这个环境里面发生,跟外面的系统隔开,这样就算 agent 出了什么差错,影响范围也圈在这个环境里,不会乱跑。过去如果你用 Claude 的托管 agent 服务,这个沙箱默认是搭在 Anthropic 自己的基础设施上的。这次更新带来的变化是,你现在可以把这个沙箱挪到你自己配置的环境里,可以是你自己公司的服务器,也可以是 Cloudflare、Daytona、Modal、Vercel 这类提供云端计算服务的厂商。

这里有个设计上的细节我觉得特别值得说一说,就是 Anthropic 把整套系统拆成了两层。一层是 agent 的编排循环,负责处理任务规划、维护上下文、出错了怎么恢复重试,这一层逻辑继续留在 Anthropic 这边运行。另一层是工具真正被执行的地方,比如敲一行 bash 命令、读写一个文件,这一层挪到了你自己控制的环境里。你可以把这个分工想象成请一个能力很强的远程顾问,他在自己的办公室里帮你做规划、做决策,这部分你不用管,但真正动手操作的环节,比如进你家里修东西,他得用你家自己的门禁系统,在你装好的监控摄像头底下干活,而不是把你家钥匙复制一份带回自己办公室。这样一来,你公司原本就有的网络策略、审计日志、安全工具,自然而然就套用在了 agent 的工具执行环节上,因为这部分压根没有离开你的地盘,你的代码、你的文件、你的仓库,都没有传出你的边界。而且因为计算资源在你自己的环境里,你可以根据具体任务去调配置,比如这个任务特别吃计算资源,你可以专门给它配一个性能更强的运行环境,镜像也是你自己定义的。

文章里提到,已经有公司在用这个能力搭真正的生产系统了,包括 Amplitude,做产品数据分析的公司,Clay,做销售线索数据整合的工具,还有 Rogo,服务金融机构的 AI 公司。这几家公司业务领域跨度也不小,从产品分析到销售线索到金融,但他们的共同点是,都属于对数据边界、合规要求比较敏感的行业,这恰好印证了自托管沙箱要解决的就是这类企业的真实顾虑。

我们可以设想一个更具体的场景,帮你理解这件事的分量。假设你在一家银行做工程师,业务部门想用 agent 帮忙处理一批客户数据的迁移和清洗工作,但银行的合规要求摆在那,客户数据一旦涉及出境或者离开公司自建的机房,审批流程能拖上几个月,走到最后大概率还是不批。如果没有自托管沙箱这个选项,你就只能眼看着这个本来很适合交给 agent 去做的任务,卡在合规这一关,一步都迈不出去。但有了这个选项之后,你可以把执行环境直接架在银行自己的机房里,数据从头到尾不出门,你的合规团队审查的对象,从,要不要相信一家外部云厂商,变成了,我们自己机房里多跑了一个程序,这是两种完全不同量级的审批难度。这也是为什么我前面说,这两项功能瞄准的不是模型能力,而是企业内部那道看不见的审批门槛。

第二项叫 MCP 隧道,目前是研究预览阶段,意味着还没有完全对外开放,需要申请才能用。在讲这个功能之前,我们先简单说一下 MCP 是什么。MCP 全称是 Model Context Protocol,模型上下文协议,是 Anthropic 推出的一套开放协议,目的是让像 Claude 这样的模型,能用一种统一的方式去连接各种外部工具和数据源,你可以把它理解成一种标准接口,你的公司内部系统只要照着这套协议包一层,Claude 就知道怎么跟它打交道,不需要每接一个新系统就重新写一套对接逻辑。

很多公司内部确实有这样的 MCP 服务器,但它们连的是公司内网才能访问的数据库或者系统,出于安全考虑,从来不对公网开放。这就带来一个矛盾,如果 Claude 的 agent 是跑在 Anthropic 的云端基础设施上,它要怎么连进一个根本不对外开放的内网地址。过去如果想打通这件事,一种做法是把这个内网服务想办法暴露出一个公网入口,再做各种白名单和权限控制去兜底,但这等于是在你的城墙上专门开一个洞,洞开得再小,也是多了一个攻击面,安全团队天然就不喜欢这种方案。

MCP 隧道给的解法挺巧妙,你在自己的网络环境里部署一个轻量的网关程序,这个网关主动往外发起一条连接,连到 Anthropic 那边,整个过程只有这一条出站连接,你完全不需要在防火墙上为了这件事开一条入站规则,也不需要对外暴露任何一个公网端口,所有经过这条连接的流量都是端到端加密的。这个设计思路,你可以类比成,以前是你要在小区大门上专门给快递员开一道侧门,告诉保安见到这个人就放行,现在变成快递员到了大堂自己打电话给你,你确认是本人之后,东西从大堂这一侧递进来,大门本身始终是关着的,没有为任何人单独开过口子。这个差别看起来细微,但对安全团队来说,意义完全不一样,前者是新增了一个永久存在的入口,后者是按需建立的、单向发起的连接,审计和控制都更干净。

文章里提到,MCP 隧道目前支持接入 Managed Agents,也就是我们前面讲的托管 agent 服务,也支持接入 Messages API,这是 Claude 最基础的调用接口。管理方式上,是由企业里的组织管理员,在 Claude Console 这个后台管理界面里去配置和管理的,Claude Console 你可以理解成 Anthropic 给企业客户提供的一个管理控制台。这个设计也透露出一个信号,就是这件事不是面向单个开发者自助开通的功能,而是被当作一项需要企业内部管理员统一把控的基础设施能力,这其实也符合它要解决的问题本身的性质,毕竟打通内网访问这种事,理应经过统一审批,不应该是哪个开发者自己就能悄悄开一条通道。

这里也可以套用一个具体场景。假设你们公司有一套内部的客户工单系统,平时只有公司内网能访问,你一直想让 Claude 帮客服团队自动分析工单、归类优先级,但这套系统从建好那天起就没考虑过对外网开放,要专门为了接 agent 给它开一个公网入口,你们的安全团队大概率不会同意,这个需求可能就一直被搁置着。MCP 隧道出现之后,你不需要去改造这套老系统本身的网络配置,只需要在内网里部署那个轻量网关,由它主动发起连接,工单系统对外的姿态完全没变,依然只对内网可见。对于很多还在用着不少历史遗留系统的公司来说,这种不需要改造老系统就能接上 agent 的方式,可能比功能本身更有吸引力,因为改造老系统往往才是最耗时间、最容易踩坑的部分。

把自托管沙箱和 MCP 隧道这两项能力放在一起看,你会发现它们其实是同一个思路的两个不同侧面。自托管沙箱解决的是,agent 动手做事的那个执行环境,放在哪里。MCP 隧道解决的是,agent 能伸手碰到的那些服务,要怎么连得上又不破坏安全边界。两者合在一起,等于把原本要交给 Anthropic 云端去做的两件事,执行环境和网络连接,都收回到了企业自己已经建好的安全边界里面。

我个人认为,这恰恰说明了一件事,就是企业愿不愿意大规模采用 agent,卡点已经不在,模型够不够聪明,这件事基本已经被验证过了,现在卡在,这套系统能不能装进我现有的治理框架里,不需要我为了用它而重新设计一套安全体系,也不需要我把信任向外扩张到一个我管不到的地方。这两项功能本质上都是在告诉企业,你不需要打破自己的边界来迁就 agent,Anthropic 把 agent 的能力伸进你的边界里来配合你。这是一种挺务实的产品思路,我觉得对于正在观望 agent 要不要进生产环境的团队来说,是个值得关注的信号。

我个人还想多说一句,这个思路让我联想到软件行业过去二十年走过的一段老路。当年云计算刚兴起的时候,很多大企业一开始也是不信任的,觉得把数据和计算放到别人的机房里,风险太大,后来云厂商想出的办法,不是逼着这些企业一步迈进纯公有云,而是先推出私有云、混合云这类过渡形态,让企业能把敏感的那部分留在自己手里,先用云的能力,再慢慢建立信任,等信任建立起来了,有些企业才逐步把更多东西迁移上去,但也有不少企业到今天依然保留着相当一部分本地部署,这不是退步,而是一种适合自己风险偏好的稳定选择。我觉得自托管沙箱和 MCP 隧道,某种程度上就是 agent 这个新事物在重走这条路,先把信任门槛降到企业能接受的程度,而不是要求所有人一步到位地信任一个完全托管在别处的黑箱。这只是我自己的类比,文章里没有提到云计算这段历史,但我觉得放在这个脉络里理解,会更容易看懂这两项功能为什么这么设计。

说到这里,我们来想一想,这篇文章对我们实际工作有什么用,能不能真的用得上。

第一点,如果你是个人开发者,或者你团队里用 Claude Code 写代码已经是日常操作了,Boris 这段计算器式快感的故事,我觉得更适合当成一种提醒,而不是单纯的情怀。它提醒你,遇到一个手痒想解决的小问题,不用端着,不用觉得这个想法太小不值得动手,先去试试能不能用一句话描述清楚,让 agent 帮你直接跑出一个能用的东西。门槛已经被压低到这个程度了,与其在脑子里反复琢磨这个方案是不是足够优雅,不如先让它跑起来,看看效果,这种轻量级的试错,现在的成本比以前低很多。

第二点,如果你所在的公司因为合规、数据驻留这类要求,一直没敢把 agent 真正用到生产环境里,自托管沙箱这个公测功能,值得专门找时间去看一看文档,跑一遍官方给的 cookbook,也就是带示例代码的实操教程。你不需要从零搭一套隔离环境和审计系统,你现有的网络策略、安全工具,直接套用过去就能生效,这中间的额外工作量,比你想象的要小。如果你所在的是金融、医疗,或者其他对数据边界特别敏感的行业,这条尤其值得认真评估,因为前面提到的 Amplitude、Clay、Rogo,走的就是这条路。

具体怎么落地,我建议按一个由小到大的节奏推进,不要一上来就想着把核心业务整个迁过去。第一步,先挑一个内部、低风险的小任务练手,比如帮内部文档做整理归类这种,不涉及核心客户数据,先验证整套链路能不能跑通,沙箱配置、运行环境镜像、网络策略这些细节都在这一步摸清楚。第二步,任务跑顺了之后,主动把安全团队拉进来一起看,而不是等他们发现了再来找你,毕竟这是公测功能,审计和合规层面的细节越早对齐越好。第三步,确认没问题之后,再逐步把范围扩大到更核心、更敏感的业务场景。这种小步快跑的节奏,比一次性大范围铺开,出问题的代价要小得多,也更容易说服公司内部那些天然比较谨慎的同事。

第三点,如果你们公司内部有一些数据库或者内部系统,你一直想让 Claude 能访问,但出于安全考虑没敢对外开放,MCP 隧道这个方向值得记下来,持续关注它从研究预览转向正式开放的进度。因为它目前还在邀请制阶段,而且是从组织管理员这个层级去配置的,所以如果你是个人开发者,这件事大概不是你自己点一下按钮就能开通的,更实际的做法是把这个信息带给你们公司负责平台或者安全的同事,让他们去申请权限,评估要不要接进来。

第四点,我想提醒一句,公测和研究预览,这两个状态都意味着还没有到完全成熟的生产级阶段。如果你打算把这两项功能用在涉及敏感数据的真实业务上,建议还是按照你们公司原本的安全评审流程走一遍,不要因为是 Anthropic 官方发布的新功能,就跳过自己内部该有的审查步骤,毕竟标着公测和预览的功能,接口和行为细节后面还可能调整。

第五点,工作坊里提到的思考预算和努力程度调节,这是一个相对独立但同样实用的技巧,如果你平时已经在用 Claude 系列模型处理偏复杂的推理任务,值得花点时间去了解一下怎么针对不同任务去调这两个参数,这能帮你在响应速度和回答质量之间找到更适合自己场景的平衡点,不是非要每次都让模型用最大力气去想,简单任务完全可以调低一些,省时间也省成本。

第六点,文章末尾提到 Code with Claude 大会下一站是东京,时间是六月五号到六号。掐着今天的日期算,二〇二六年六月十九日,东京站其实已经开完了,如果你对东京站的内容感兴趣,不用等什么直播预告,直接去找回放就行,大概率第一天的主题演讲和分会场视频已经放出来了。同时,伦敦这一站如果你错过了直播,文章里也提到主题演讲和分会场的录像已经上线,感兴趣的话可以直接去补一下 Boris 那段演讲的完整版,我这里讲的只是其中一个片段,完整演讲里应该还有更多细节和现场互动。

我们这期就把这篇文章讲到这里,简单总结一下。这篇文章表面上是一篇大会报道,但它其实串起来讲了两件事,一件是观念层面的,就是 agent 正在把,想法变成能跑的程序,这件事重新拉回到一种轻松、即时反馈的体验,跟我们小时候按计算器解题的那种快感是一脉相承的。另一件是工程和治理层面的,就是 Anthropic 通过自托管沙箱和 MCP 隧道这两项新能力,主动把 agent 的执行环境和网络连接,收回到企业自己已经建好的安全边界里,而不是要求企业把信任往外扩张。

我个人认为,这两件事放在一起看才完整,光有情怀讲故事,企业落地不了,光有基础设施讲安全,又显得冷冰冰没有说服力。这篇文章把两者放在同一场大会里讲,某种程度上也是在说,Anthropic 自己也清楚,要让更多人重新找回写代码的乐趣,光靠模型变聪明是不够的,还得让企业敢放心地把这件事交出去。如果你最近正好在评估要不要把 agent 真正用到生产系统里,这两项新功能值得你专门找时间去看一看文档。我们这期就先聊到这里,下期再见。
