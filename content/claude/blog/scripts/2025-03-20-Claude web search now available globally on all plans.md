先说说背景。在这篇文章发布之前,大语言模型最大的一个先天缺陷,大家应该都很熟悉,就是知识截止日期的问题。你问 Claude 或者其他任何一个大模型,昨天发生了什么新闻,它是答不上来的,因为它的训练数据是在某个时间点之前收集的,之后发生的事情,它完全不知道。这个问题在早期大家觉得无所谓,因为大模型主要被用来做写作、做代码、做总结这类不需要实时信息的任务。但是随着大模型越来越多地被当作生产力工具,被用来做调研、做决策支持、做信息检索,这个知识截止日期的问题就变得越来越致命。

你想象一下,一个金融分析师想问 Claude,某家公司这个季度的财报情况怎么样,模型如果没有联网能力,它能给你的答案最多是基于训练数据里的历史信息做一些推测,完全没有办法告诉你这家公司昨天刚刚发布的财报数据。这种场景下,模型不仅没用,反而可能给出过时甚至错误的信息,这是比没有答案更危险的事情,因为用户可能不知道这个信息已经过期了,会拿着错误的数据去做决策。

在这之前,行业里已经有一些办法来缓解这个问题,最常见的做法叫做检索增强生成,英文叫 Retrieval Augmented Generation,简称 RAG。简单说就是,在模型回答之前,先用传统的搜索或者数据库查询的方式,把相关的最新资料抓取过来,塞进模型的上下文里,让模型基于这些新鲜资料来回答。但是这套方案有一个明显的局限,就是它通常需要开发者自己搭建检索系统,自己接入搜索引擎的 API,自己写逻辑判断什么时候该检索、检索什么关键词、怎么把检索结果塞进提示词里。对于普通用户来说,这一套东西他们根本碰不到,只有开发者通过编程的方式调用模型接口,才能享受到这种增强。普通人用网页版或者客户端跟 Claude 聊天的时候,是完全没有这个能力的。

所以二〇二五年三月这篇文章发布的时候,行业里其实已经有竞争对手做了类似的事情,把搜索能力直接整合进了对话产品里,让普通用户在网页聊天界面就能享受到联网搜索带来的实时信息。Anthropic 这次发布,可以理解成是 Claude 在这场"对话式搜索"竞赛里补齐的一块重要拼图。这不是什么颠覆性的技术突破,坦白说,联网搜索这个功能听起来甚至有点朴素,但它解决的是一个非常实际、非常高频的痛点,所以从产品价值的角度来说,这个发布反而是非常扎实的。

好,背景讲完了,我们来拆解一下这篇文章里最核心的几个点。

第一个核心点,我想讲清楚联网搜索具体是怎么工作的,以及它跟模型原本的知识库是什么关系。这篇文章里说,当 Claude 把网上搜到的信息整合进回答的时候,它会提供直接的引用,方便你核实信息来源。这一句话看起来简单,但背后体现了 Anthropic 在设计这个功能时候的一个核心思路,就是不让 Claude 变成一个黑箱式的搜索引擎,而是让它保持透明,把信息来源亮出来。

我们可以做一个类比。假设你去问一个特别博学的朋友一个问题,这个朋友脑子里装满了各种知识,但是有些知识是好几年前学的,可能已经过时了。如果这个朋友足够靠谱,他在回答你的时候,如果发现自己记忆里的东西可能不准了,他会主动说,我去查一下最新的资料,然后查完之后告诉你,我刚才查到的是某个网站上的某篇文章说的,你可以自己去看看原文核实一下。这就是 Claude 联网搜索想要模拟的那种交互体验。它不是简单粗暴地把搜索引擎的结果列表甩给你,让你自己去一个一个点开链接筛选,而是它先帮你做了筛选、整合、归纳的工作,用对话式的语言把信息组织好讲给你听,同时还保留了可追溯性,告诉你这句话是从哪儿来的。

这个设计思路其实回应了一个大模型领域长期存在的痛点,叫做幻觉问题,英文叫 hallucination,就是模型有时候会自信满满地编造一些根本不存在的事实。如果联网搜索功能只是把搜到的内容糊里糊涂地融进回答里,不给引用,那用户根本没法判断这句话是模型自己编的,还是真的有可靠来源支撑的。有了引用之后,这个问题至少在可验证性上得到了很大改善,你可以顺着引用去原始网页核实,这对于需要严谨性的场景,比如说金融决策、学术研究,价值是非常大的。

第二个核心点,我想讲讲文章里提到的触发机制,叫做"在适用的时候 Claude 会搜索网络来支持它的回答"。这句话翻译过来听起来平平无奇,但其实隐含了一个挺有意思的工程问题,就是模型怎么判断什么时候该去搜索,什么时候不需要搜索。

我们想一想,如果你问 Claude 一个数学题,比如说一加一等于几,这种问题完全不需要联网搜索,模型自己的知识储备绰绰有余,如果每次都去触发一次网络搜索,不仅浪费时间,拖慢响应速度,还可能因为搜索引擎返回一些不相关甚至误导性的内容,反而把简单问题搞复杂了。但是如果你问的是,这个月某个城市的房价走势怎么样,这种问题模型的训练数据里大概率是没有最新信息的,这时候就应该主动去搜索。

所以这背后其实是一个判断逻辑的设计,模型需要具备一种自我评估的能力,判断当前这个问题,是不是依赖于实时信息,是不是它自己内部知识库能够圆满回答的。这种判断能力,某种程度上反映了模型对自己知识边界的认知能力,这个在 AI 圈子里有时候被称为校准能力,英文叫 calibration,就是模型知道自己知道什么,也知道自己不知道什么。一个校准得好的模型,应该是在自己有把握的领域大胆给出答案,在自己没有把握、或者明显需要最新信息的领域,主动去寻求外部帮助,而不是硬撑着凭印象瞎答。

这个设计思路对我们普通用户的好处是,你不需要自己去判断,这个问题该不该让 Claude 去联网查。你只需要把开关打开,剩下的判断交给模型自己,它觉得有必要就会去查,你只需要在最后的回答里看到引用,就知道这部分内容是有联网支撑的,没有引用的部分,大概是模型基于自己已有知识回答的。

第三个核心点,我想讲讲这次发布的推广节奏,因为这里面其实藏着不少产品策略上的考量。文章原文里说,联网搜索目前以功能预览的形式,向美国所有付费用户开放,免费版用户和更多国家的支持即将到来。然后我们看到后续的更新里说,二〇二五年五月二十七号,这个功能已经在所有计划里全球可用了。

这个节奏很值得说一说。先给付费用户开放,再逐步扩展到免费用户,先在美国试运行,再扩展到全球,这是一个非常经典的渐进式发布策略,英文有时候叫 staged rollout 或者 phased rollout。为什么要这么做,我觉得背后至少有几层考虑。

第一层考虑是成本控制。联网搜索这个功能,跟模型纯粹基于自己内部知识回答相比,多了一道调用外部搜索引擎、抓取网页内容、再把这些内容塞进模型上下文进行处理的环节,这个环节是有真实的计算成本和带宽成本的。如果一上线就向全球所有用户,包括免费用户,全量开放,服务器的压力和成本支出可能是难以预估的,所以先在一个相对可控的用户群体里,比如说美国的付费用户,做小范围测试,观察真实的使用量和成本曲线,这是一个非常稳妥的工程决策。

第二层考虑是质量打磨。联网搜索这个功能,涉及到怎么挑选搜索关键词,怎么从搜索结果里筛选出真正相关、真正可靠的网页,怎么把抓取到的网页内容压缩整合进模型的上下文窗口里,这一整套链路的工程实现,需要在真实用户的使用反馈里不断打磨。先小范围放出去收集反馈,再逐步扩大覆盖面,这样可以在问题影响范围还比较小的时候就发现并修复,而不是等到全量上线之后才暴露大规模的问题。

第三层考虑,我个人认为,也跟监管和地域差异有关系。不同国家对于网络内容、数据隐私、信息来源的合规要求是不一样的,先在美国市场跑通,再逐步评估其他国家的合规风险,扩展到更多地区,这是一个相对谨慎、风险可控的做法。当然这一点文章里没有直接说,是我自己根据行业惯例做的推测,大家可以自行判断。

第四个核心点,我想专门讲讲文章里举的那四个使用场景的例子,因为我觉得这四个例子选得挺有代表性,能帮我们理解联网搜索这个功能真正的价值落点在哪里。

第一个例子是销售团队。文章里说,销售人员可以通过分析行业趋势,了解客户的关键举措和痛点,从而转变客户account 规划的方式,提升成交率。我们可以想象一个场景,一个销售人员要去拜访一个新客户之前,他想快速了解一下这个客户公司最近在做什么大动作,有没有新的战略调整,有没有最近发生的行业新闻可能影响到这次拜访的话术。如果没有联网搜索,他可能需要自己打开浏览器,搜索这家公司的名字,翻看好几个新闻网站,自己整理出一份要点。现在他可以直接问 Claude,帮我整理一下这家公司最近的动态和行业里的关键趋势,Claude 联网查完之后,直接给他一份整理好的、带引用来源的摘要。这个时间成本的节省是非常显著的。

第二个例子是金融分析师。文章里提到,分析师可以评估当前的市场数据、财报、行业趋势,从而做出更好的投资决策,完善财务模型的假设。这个场景跟我前面讲背景的时候提到的那个例子是一致的,财报数据这种信息天生就是高度时效性的,一份三个月前的财报分析,在新的财报出来之后,基本上就失去了大部分价值。联网搜索让金融分析师能够直接在对话里获取最新的财报数据和市场动态,而不需要自己去专门的金融数据终端里手动查询,再粘贴回对话框里给 Claude 分析。这是一个把信息获取和信息分析两个步骤打通在一起的效率提升。

第三个例子是科研人员。文章说,科研人员可以通过搜索网络上的一级资料来源,捕捉新兴趋势,识别当前文献里的空白,从而写出更有说服力的资助申请书和文献综述。这个场景我觉得特别值得展开讲一讲,因为文献综述这件事,本质上是一个信息检索加信息整合的任务,而这恰恰是联网搜索加大语言模型这个组合最擅长的事情。传统做文献综述,科研人员需要在各种学术数据库里搜索关键词,一篇一篇地翻阅论文摘要,判断哪些是相关的,然后再手动整理出一个综述框架。这个过程非常耗时,而且高度依赖科研人员自己对这个领域的熟悉程度。如果借助带联网能力的 Claude,科研人员可以先让 Claude 帮忙搜索某个细分领域最近的研究进展,Claude 会给出带引用的整理,科研人员再基于这个整理去深入阅读真正核心的几篇论文,这相当于把一个大海捞针式的初筛工作,交给了 AI 来做。

第四个例子是消费者比价。文章说,购物者可以在多个信息来源之间比较产品功能、价格和评价,从而做出更明智的购买决策。这个场景我们大家都很熟悉,买一个东西之前,习惯性地打开好几个购物网站,看看价格,翻翻评价,有时候还要去专门的评测网站看看测评文章。这个过程其实挺琢磨人的,因为信息散落在很多不同的网站上,你需要自己整合。有了联网搜索之后,你可以直接问 Claude,帮我对比一下某几款产品的优缺点,价格区间,以及用户口碑怎么样,Claude 会去搜索多个来源,然后给你一份整合好的对比报告。

这四个例子其实有一个共同的特点,就是它们都不是单纯的信息查询,而是信息查询加信息整合加信息分析。这正是大语言模型加联网搜索这个组合的核心价值所在,它不是简单地替代搜索引擎,而是在搜索引擎之上,叠加了一层理解、归纳、对比、总结的能力。

好,讲完了这几个核心点,我们花点时间聊聊,这个功能对我们这些听众,尤其是做软件工程和 AI 相关工作的朋友来说,到底意味着什么,在实践当中我们应该怎么用,有什么坑需要避开。

第一个层面,我想讲讲普通使用场景下,怎么把联网搜索用得更顺手。如果你是个人用户,在用 Claude 网页版或者客户端聊天,你首先要确认一下,联网搜索这个开关有没有打开,通常是在个人资料设置里面可以找到。打开之后,你不需要特别去说请你联网搜索这种指令,Claude 会根据问题本身的性质,自动判断要不要去查。但是如果你发现某次回答里 Claude 给的信息明显过时了,或者你怀疑它没有真的去联网查证,你完全可以直接在对话里追问一句,你刚才回答的时候有没有查最新的资料,或者直接说,请帮我查一下这方面最新的情况。这种明确的指令往往能够提高触发联网搜索的概率,得到更准确的回答。

第二个层面,我想讲讲带引用这个特性,在实际使用中应该怎么对待。我个人的建议是,凡是涉及到重要决策的场景,比如说要不要投资一家公司,要不要采纳某个数据去写报告,一定要养成习惯,顺着 Claude 给出的引用链接去原始来源核实一遍。这不是因为 Claude 不靠谱,而是因为任何信息检索系统,包括我们人类自己用搜索引擎查资料,都有可能受到信息源本身质量参差不齐的影响。网络上的信息鱼龙混杂,有权威媒体的报道,也有自媒体的猜测性文章,甚至还有过时多年但仍然在被搜索引擎索引的旧文章。Claude 联网搜索这个功能,本质上是帮你做了第一轮的筛选和整理,但它不能完全替代你自己做最后一道把关。这一点,我觉得是用好这个功能最重要的心态。

第三个层面,我想讲讲对于开发者和做产品的朋友来说,这个功能背后的设计思路有什么值得借鉴的地方。如果你自己在做一个基于大模型的产品或者 agent,也就是能够自主执行多步骤任务的智能体,你大概率也会面临一个类似的设计决策,就是要不要给你的 agent 接入外部工具,比如说搜索引擎接口、数据库查询接口,以及更关键的,什么时候触发这些外部工具的调用。

Claude 这次的设计给了我们一个很好的参考样本,核心思路有这么几条。第一,工具调用应该是模型自主判断的,而不是用户每次都要手动指定,这样能大大降低用户的使用门槛,让普通人也能享受到工具增强的好处,而不只是开发者。第二,工具调用的结果应该带有可追溯的来源信息,这样不管模型给出的答案是对是错,用户都有办法去核实,这对建立用户信任非常关键。第三,渐进式发布,先在小范围验证成本和效果,再逐步扩大覆盖面,这是控制风险的稳妥做法,值得我们自己做产品发布的时候参考。

第四个层面,我想稍微拓展一下,聊聊从二〇二五年三月到现在二〇二六年六月,联网搜索这类能力在整个行业里发生了什么演变,帮大家把这篇文章放在一个更大的时间坐标系里理解。在这篇文章发布的那个阶段,联网搜索算是一个相对独立的、附加在对话能力之上的功能模块。但是后来这一年多的时间里,行业里出现了一个更大的趋势,就是这种工具调用能力,逐渐从单一的搜索功能,演变成更通用的智能体框架,模型不仅可以联网搜索,还可以调用各种各样的工具,执行文件操作,调用代码执行环境,甚至可以连接到企业内部的各种系统里去完成复杂的多步骤任务。

如果你把联网搜索看作是这个大趋势里的第一块拼图,那么今天我们看到的种种更复杂的智能体应用,其实都是在这个基础逻辑上不断叠加和扩展的结果。模型自己判断需不需要用工具,用工具之后给出可追溯的依据,这套设计哲学一直延续到了后面更复杂的智能体产品设计里。所以回头看这篇文章,它的意义不仅仅是给 Claude 加了一个联网搜索功能,更重要的是它体现了 Anthropic 在那个阶段对于如何把外部能力安全、透明地整合进对话式 AI 产品里的一种设计思路,这种思路后续被沿用和发展,变成了我们今天看到的更丰富的智能体生态。

第五个层面,我想给大家一些更具体的、可以马上上手尝试的实践建议。如果你是产品经理或者业务人员,平时需要做市场调研、竞品分析,我建议你可以把 Claude 当作一个初筛助手,先让它帮你联网搜索整理出一个大致的轮廓,包括几个关键的信息来源,然后你再针对最关键的几个来源做深入阅读和验证,这样能大幅压缩你做调研报告的时间。如果你是开发者,正在尝试设计自己的智能体产品,我建议你仔细琢磨一下,你的产品里有哪些场景是高频需要实时信息的,哪些场景模型自身知识已经足够,然后针对这两类场景设计不同的处理逻辑,而不是一刀切地全部接入搜索或者全部不接入搜索,这样既能控制成本,又能保证用户体验。如果你是普通用户,平时用 Claude 聊天比较多,我建议你养成一个习惯,在涉及到价格、日期、最新进展这类明显带有时效性的关键词时,主动留意一下回答里有没有引用标记,如果没有,不妨追问一句,请帮我确认一下这是不是最新的信息。

好,我们差不多该收尾了。今天我们聊的这篇文章,从内容上看是一个相对简短直接的产品发布通告,标题是 Claude can now search the web,讲的是二〇二五年三月,Claude 三点七 Sonnet 拿到了联网搜索的能力,并且很快在二〇二五年五月底扩展到了全球所有付费和免费用户。这个功能解决的核心问题,是大语言模型知识截止日期带来的信息滞后问题,它的核心设计亮点,是用对话式的方式整合搜索结果,并且配上可追溯的引用,既提升了用户获取信息的效率,又保留了让用户自行核实的可能性,这是一个在效率和可信度之间做了很好平衡的设计。

我个人认为,这篇文章看起来不算技术含量特别高的发布,但是它代表的产品理念,对我们今天理解智能体类产品的设计逻辑,依然很有参考价值。无论你是在用 Claude 做日常工作,还是在自己动手做基于大模型的产品,这种让模型自主判断何时调用外部能力、调用之后保持可追溯性的设计思路,都是值得借鉴的一条经验。好,这一期就到这里,我们下期再见。
