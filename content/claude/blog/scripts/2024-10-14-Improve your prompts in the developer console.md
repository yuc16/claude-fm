## Script

大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是 Improve your prompts in the developer console,翻译过来就是,在开发者控制台里改进你的提示词。这篇文章发表于二〇二四年十月十四日,也就是二〇二四年的十月十四号。

先说一句时效性提醒。今天是二〇二六年六月二十一日,距离这篇文章发布已经过去了一年八个月。在 AI 这个行业里,一年八个月几乎相当于一个完整的技术代际。这篇文章发布的时候,Anthropic 主推的模型还是 Claude 三点五 Sonnet,文章里提到的实验用的甚至是更早一代的 Claude 三 Haiku。所以你听这期节目的时候要有个心理预期,具体的产品界面、具体的模型版本号,很可能跟你现在打开控制台看到的不完全一样了。但是,这篇文章里讲的提示工程方法论,思维链推理怎么用、示例怎么组织、怎么用结构化的方式去评估和迭代提示词,这些思路是经过时间检验的,放到今天依然成立,甚至可以说是后来很多提示工程实践的基础范式。我们今天就把这篇文章当成一个时间胶囊,既讲清楚它当时解决了什么问题,也顺带聊聊它对今天的我们还有什么用。

我们先说说背景。二〇二四年下半年,大语言模型已经从一个新奇的玩具变成了真正被拿去做生产系统的工具。这时候,几乎所有做 AI 应用的团队都会撞上同一个瓶颈,那就是提示词,也就是英文里说的 prompt,写得好不好,直接决定了模型表现的上限。这话听起来好像有点夸张,但你仔细想想,模型本身的能力是固定的,你能调动出来多少能力,很大程度上取决于你怎么问它、怎么给它上下文、怎么告诉它你想要的格式。一个写得潦草的提示词,哪怕用的是当时最强的模型,效果也可能很差,而一个精心打磨过的提示词,即便用的是相对便宜、相对小的模型,也能做出让人惊喜的效果。

但问题是,写好提示词这件事本身是有门槛的。业内已经总结出了一些公认有效的技巧,比如让模型先思考再回答,这个叫思维链推理,英文是 chain of thought,还有给模型提供清晰的示例,告诉它输入是什么样、期望的输出是什么样。这些技巧听起来不难,但真正落地的时候,很多开发者其实做得不到位。一方面是因为这些技巧需要花时间去打磨,你得反复试错,调整措辞,调整结构。另一方面,这些最佳实践在不同的模型厂商之间还存在差异,你用 GPT 系列模型总结出来的提示词套路,直接搬到 Claude 上,效果未必一样好,因为不同模型对指令的敏感度、对格式的偏好是不一样的。这就导致一个很现实的痛点,很多团队想把自己原来用在别的模型上的提示词迁移到 Claude 上,却发现迁移过程很痛苦,需要靠人工一点点摸索哪里需要改。

正是在这样的背景下,Anthropic 在开发者控制台里推出了这个提示词改进器功能,英文叫 prompt improver。它的核心思路其实很巧妙,就是让 Claude 自己来改进给 Claude 用的提示词。听起来有点像是自己给自己做体检,但这恰恰是大语言模型一个挺有意思的应用场景,模型不仅可以执行任务,还可以反思和优化任务的描述方式。

接下来我们就来拆解一下,这个提示词改进器具体是怎么干活的。文章里讲到它用了五种手法,我们一个一个说。

第一种叫思维链推理的添加。简单说,就是改进器会在你的提示词里专门加一段,引导 Claude 在正式回答之前,先系统地把问题想一遍。你可以把这个理解成,原来你让一个同事直接给你一个结论,现在你换了个问法,让他先把分析过程写出来,再给结论。为什么这样做效果会更好,这里有个挺直观的类比,就像我们做数学题,直接报答案容易出错,但如果你把解题步骤一步步写出来,出错的概率就会大大降低,因为每一步都有机会被检查,逻辑链条也更连贯。对大语言模型来说道理是类似的,它在生成文字的时候本质上是一个逐字预测的过程,如果你不给它思考的空间,直接逼着它吐出最终答案,它就更容易在没有充分铺垫的情况下出现逻辑跳跃或者事实错误。但如果先让它把相关信息梳理一遍、把推理步骤写出来,相当于给了它一个脚手架,最终答案的质量自然会提升。

第二种手法叫示例标准化。这个相对技术一点,意思是改进器会把你提示词里原来零散的示例,统一转换成一种一致的 XML 格式。XML 大家可以理解成一种用尖括号标签把内容包起来的结构化写法,类似于给每一段内容贴上清楚的标签,告诉模型这部分是输入,这部分是输出,这部分是说明。这样做的好处是,模型在解析提示词的时候,能够更清楚地分辨出哪部分是示例、示例里的输入输出边界在哪里,不会因为格式混乱而产生误解。你可以想象一下,如果你给一个新员工培训材料,材料里案例和案例之间没有任何分隔,文字全部混在一起,新员工很可能看花了眼,搞不清楚哪句话属于哪个案例。统一格式之后,边界清晰了,理解自然就准确了。

第三种叫示例增强。这一步建立在第一步思维链推理的基础上,改进器会给原来的示例补充上与新提示词结构相匹配的推理过程。也就是说,光给示例的输入输出还不够,改进器还会顺手给每个示例补上一段思考过程,让示例不仅展示结果,还展示了通向结果的思路。这其实是把第一种和第二种手法结合了起来,既保证示例格式统一,又让每个示例本身也带有推理示范,相当于一个更立体的教学材料。

第四种叫重写,英文是 rewriting。这一步比较好理解,就是改进器会重新组织提示词的结构,让逻辑更清晰,同时顺手纠正一些小的语法和拼写问题。这一步听起来朴素,但其实很重要,因为很多提示词写得啰嗦、逻辑跳来跳去,或者句子结构本身就有歧义,这些问题人类读者可能凭经验能容忍,但模型对措辞的敏感度其实比我们想象的要高,一个措辞含糊的句子,真的可能导致模型理解出现偏差。

第五种叫预填充,英文是 prefill。这个概念稍微需要解释一下。在跟 Claude 对话的接口里,你其实是可以预先填一部分 Assistant 角色要说的话,然后让模型接着往下生成。这就像是你在帮模型起个头,告诉它接下来该用什么样的语气、什么样的格式往下写。比如你想让输出严格是 JSON 格式,你可以预填一个左花括号,模型大概率就会顺着这个开头继续生成合法的 JSON,而不会先寒暄几句再给你内容。提示词改进器会自动帮你加上合适的预填充内容,用来引导 Claude 的行为方向,并且强制约束输出格式。

这五种手法组合起来,基本上覆盖了提示工程里最常见、最有效的那几条经验法则。而且重点是,这个过程是自动化的,你不需要自己一条条去套用这些技巧,系统会帮你分析现有提示词,然后生成一个升级版本。生成新提示词之后,你还可以继续给 Claude 反馈,告诉它哪里还不满意,系统会基于你的反馈继续迭代,这个过程可以反复进行,直到你满意为止。

那么效果到底怎么样,文章里给了两个具体的测试数据,我觉得值得展开讲讲,因为这是少有的把提示工程效果量化出来的案例。第一个测试是多标签分类任务。具体场景是,研究人员拿了五百篇维基百科文章和对应的标题,测试 Claude 三 Haiku 能不能把一句从文章里随机抽出来的句子,准确匹配到对应的文章标题上。这其实是个挺考验语义理解和检索匹配能力的任务,因为你光看一句话,很多时候上下文信息是不完整的,模型得靠语义关联去判断这句话最可能出自哪篇文章。结果显示,经过提示词改进器处理之后的提示词,相比原始提示词,准确率提升了百分之三十。这个提升幅度其实相当可观,要知道在很多机器学习任务里,百分之三十的相对提升已经算是非常显著的进步了,而这里仅仅靠改写提示词就做到了,模型本身完全没变,这恰好说明了提示词质量对最终效果的影响有多大。

第二个测试是摘要任务。研究人员拿了十篇维基百科文章,要求 Claude 按照特定的字数范围去写摘要。这是个看起来简单但实际上很多模型容易翻车的任务,因为字数控制需要模型对自己生成的内容长度有比较精确的感知,很多时候模型写着写着就超字数了,或者写得太短不够。经过提示词改进器处理之后,字数遵从率做到了百分之百,也就是说十篇全部命中了要求的字数范围。这其实从另一个角度说明了思维链推理和清晰示例的价值,如果提示词里明确告诉模型先估算大概要写多少句话、每句话大概多长,再去组织语言,那模型对篇幅的把控就会精确很多,而这恰恰是提示词改进器自动帮你加上的内容。

讲完提示词改进器,文章接下来讲了第二个重要功能,就是示例管理。这个功能解决的是另一个很现实的痛点。文章里提到一句话,加示例是提升模型响应质量最有效的手段之一,尤其是当你需要 Claude 严格按照某种特定格式输出内容的时候。这句话我觉得特别值得反复琢磨。很多开发者写提示词的时候,精力全花在写指令上,恨不得把每条规则都写得滴水不漏,但其实从实际效果来看,给几个清晰的输入输出示例,往往比堆砌十条文字说明更管用。这背后的道理也挺直观,文字说明终究是抽象的,而示例是具体的,模型从具体案例里学习模式,比从抽象规则里推导模式,要容易得多,也更不容易出现理解偏差。这跟人类学习也是类似的道理,你教一个新人怎么写周报,光讲一堆原则他可能还是不知道怎么下笔,但你给他看两份写得好的周报样本,他一下就明白了。

在这次更新之前,如果你想在提示词里加示例,可能得自己手动在一大段文字里插入示例文本,格式全靠自己维护,稍不注意就乱了。这次更新之后,开发者可以直接在 Workbench,也就是控制台里的工作台界面,用结构化的方式管理示例,清楚地分开输入和输出,新增示例或者编辑已有示例都变得更方便。而且如果你的提示词里压根没有示例,Claude 还能帮你自动生成,系统会自动构造一些合成的示例输入,并草拟对应的输出,帮你把这个流程跑起来,省去你从零构思示例的时间。

文章总结了加示例能带来的三个方面的提升,准确性,因为示例能减少模型对指令的误读,一致性,因为示例能确保输出格式符合预期,还有性能,也就是示例能提升模型处理复杂任务的能力。这三点其实互相关联,本质上都是因为示例给模型提供了更具体、更不容易产生歧义的参照。

接下来文章讲到第三个功能,叫提示词评估器,英文是 prompt evaluator。这个工具的作用是让你能在各种场景下测试自己的提示词表现。这次更新里,评估页面新增了一个可选的列,叫理想输出,英文是 ideal output。这一列的作用是,你可以提前写好你期望模型输出的理想答案,然后系统会用五分制的评分标准,帮你对照理想输出去给模型实际生成的内容打分。这其实解决了提示工程里一个长期存在的难题,就是怎么客观衡量一个提示词改得好不好。如果没有量化的评分标准,你只能凭感觉判断这次输出是不是比上次好,这种主观判断很容易出现偏差,尤其是当你测试几十上百个案例的时候,人工逐一比对效率很低,也容易疲劳出错。有了理想输出和五分制评分,你就可以把提示词迭代变成一个有反馈闭环的过程,改一版,跑一遍评估,看分数有没有提升,这跟我们做软件工程时写单元测试、跑回归测试的思路其实是相通的,本质上都是给变更建立一个可验证的标准。

文章里还提到,在测试完新提示词之后,你可以继续在提示词改进器里给 Claude 反馈,告诉它哪里还有问题,然后系统会继续修改提示词和示例,这个过程可以反复迭代,直到满意为止。而且这个改进器不仅能响应反馈式的修改要求,还能处理一些任意性的改写需求,文章里举的例子是,你可以直接让 Claude 把提示词和示例里原来 XML 格式的输出,改成 JSON 格式的输出。这说明这个工具的定位不只是一个被动的优化器,它更像是一个能跟你对话、根据你的需求灵活调整提示词结构的助手。

文章最后引用了一个客户案例,是一家叫 Kapa.ai 的公司。这家公司做的事情是把企业的技术知识库,转化成可以直接投入生产环境使用的 AI 助手,听起来有点像是给企业知识库装上一个智能问答的外壳。这家公司的联合创始人 Finn Bauer 说,Anthropic 的提示词改进器让他们迁移到 Claude 三点五 Sonnet 的过程更顺畅,也让他们更快地把系统推上生产环境。这个案例其实印证了我们前面讲的迁移痛点,Kapa.ai 原本的提示词大概是为别的模型写的,迁移到 Claude 上需要重新打磨,而提示词改进器帮他们大大缩短了这个打磨周期。

文章最后说,提示词改进器、示例管理功能,以及理想输出功能,这三项当时都已经向 Anthropic 控制台的所有用户开放了。

好,讲完了文章本身,我们来聊聊这套思路对我们今天的实际工作意味着什么。虽然这篇文章发布的时间点已经比较早,产品细节肯定也迭代了很多轮,但我觉得这背后的方法论框架,放到今天依然非常有实践价值,我们可以一条一条对照着想。

第一点,如果你现在手头正在维护一个用 Claude 的提示词,而且这个提示词写得比较粗糙,没有思维链、没有结构化示例,那么不管你现在用的是控制台里现成的改进工具,还是自己手动按照这五种手法去改,思路都是一样的,先看看有没有让模型先思考再回答,再看看示例够不够清晰、格式够不够统一,再看看有没有用预填充去约束输出格式。这套检查清单本身就是一个很实用的自查框架,你完全可以把它当成一份提示词体检表,定期拿出来对照自己手头的提示词扫一遍。

第二点,关于示例的价值,我觉得这一点值得在团队内部反复强调。很多工程师写提示词的习惯,是遇到问题就加一条新规则,结果规则越堆越多,提示词变得又长又乱,模型反而更难抓住重点。如果你发现自己的提示词已经堆了十几条规则,效果还是不稳定,这时候不妨停下来想想,是不是该换个思路,用两三个精心挑选的示例替换掉那一堆文字规则。示例的好处是它天然带着上下文,模型不需要做抽象推理,直接照着葫芦画瓢就行,这通常比纯文字指令更稳定也更省心。

第三点,关于怎么验证提示词改得好不好。如果你的团队现在还是靠人工肉眼比对模型输出来判断提示词改进效果,我建议借鉴文章里理想输出加评分这个思路,哪怕你不用控制台里现成的工具,自己用代码写个简单的评估脚本也可以,核心是给每个测试案例准备一个参考答案,然后用某种打分标准,可以是简单的相似度匹配,也可以是再用一个模型当裁判去打分,把这个过程跑成一个可重复执行的评估流程。这样你每次改提示词,都能拿到一个客观的分数变化,而不是凭感觉说这次好像感觉好一点。长期来看,这种工程化的评估习惯,对提示词质量的稳定提升帮助非常大。

第四点,关于迁移场景。如果你的团队现在正好在考虑把某个原本跑在别的模型上的应用迁移到 Claude 上,这篇文章其实给了一个很实用的提示,那就是不要假设原来的提示词可以直接照搬过去用。不同模型对指令的敏感度、对格式的偏好确实存在差异,迁移的时候花一点时间重新打磨提示词,通常能换来明显的效果提升。哪怕你手上没有现成的自动化改进工具,光是按照思维链推理、结构化示例这两条原则去人工重写一遍,往往也能看到立竿见影的改善。

第五点,我想多说几句关于预填充这个技巧,因为我觉得它在实际工程里经常被低估。很多团队在需要模型严格输出某种格式,比如 JSON 或者特定的标签结构的时候,习惯靠在提示词里反复强调请务必输出 JSON 格式、不要输出多余文字这类指令。但实际经验告诉我们,光靠文字强调,模型有时候还是会忍不住先寒暄两句,或者在 JSON 前后加一些解释性文字。这时候预填充这个技巧就特别管用,你直接在 Assistant 消息开头预填一个左花括号或者一个固定的开头标签,模型大概率会顺着这个开头继续往下写,完全不会再去寒暄。这个技巧背后的原理,其实跟语言模型逐字生成的工作机制有关,模型在生成下一个字的时候,会受到前面已经生成内容的强烈影响,所以你给它一个强约束的开头,后面内容的走向也会被这个开头牢牢限定住。这是一个成本很低、效果很直接的小技巧,我建议大家如果还没用过,可以找个有格式要求的场景去试一下。

第六点,关于反馈式迭代这个流程。文章里提到的那种反复给反馈、让 Claude 继续修改提示词的模式,其实给了我们一个挺有启发的工作方式,就是把提示词的打磨当成一个跟模型协作的过程,而不是一次性写完就定型的东西。具体到实践上,你可以把每一轮的反馈和修改都记录下来,形成一个版本演进的轨迹,这样不仅方便回溯,也方便团队里其他人理解某个提示词为什么长成现在这个样子,避免后来的人不明就里地把已经验证过的结构随意改动。
