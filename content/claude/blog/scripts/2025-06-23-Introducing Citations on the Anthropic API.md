算一算时间,这篇文章距离我们现在,也就是二零二六年六月二十一日,已经过去整整将近一年了。这在 AI 这个行业里,真的算是挺久远的事了。所以我要先给大家提个醒,文章里提到的模型是 Claude 三点五 Sonnet 和 Claude 三点五 Haiku,这在当时是比较新的模型,但放到今天,我们已经用上了 Claude 四点几系列,甚至更新的版本了。所以听这期节目的时候,大家要把它当成一个历史背景来听,理解当时 Anthropic 是怎么思考这个问题的,而不是去纠结具体哪个模型版本现在还能不能用。不过我要强调的是,Citations 这个功能本身的设计思路,到今天依然是有效的,而且我相信经过这一年的迭代,它应该已经变得更成熟、更好用了。这也是为什么我觉得这篇文章值得拿出来讲一讲,因为它揭示的问题和解法,放到今天来看依然有参考价值。

我们先来说说背景,也就是这篇文章想解决什么问题。如果你平时有用大模型做一些严肃场景的应用,比如说法律咨询、财务分析、医疗问答,你肯定遇到过这样的烦恼。模型给你一个回答,听起来头头是道,逻辑通顺,语气也特别自信,但你心里总会犯嘌咕,它说的这句话,到底是从哪儿来的,有没有可能是编的。这就是大模型一个老生常谈的问题,我们一般管它叫幻觉,英文是 hallucination,也就是模型一本正经地说出一些看起来合理但实际上不存在或者不准确的内容。

在 Citations 这个功能出来之前,如果你想让 Claude 在回答里注明引用来源,你只能靠写提示词去硬性要求它。比如你在提示词里写,请在每句话后面注明引用自第几段第几句,请保证引用的内容准确无误。这种做法说起来简单,但实际操作起来特别麻烦,而且效果还不稳定。文章里原文用的词是 inconsistent performance,就是表现不稳定,有时候模型确实老老实实标注了来源,但有时候它会偷懒,或者干脆把引用编出来,告诉你这句话来自第三段,但你去翻第三段发现根本没这句话。开发者为了让这套提示词工程靠谱一点,往往要花大量时间反复测试、反复调整,投入的精力跟收益完全不成比例。

这其实反映了一个更大的行业背景。二零二五年那个时间点,大模型已经从单纯的聊天助手,逐渐演变成可以处理严肃业务的生产力工具,尤其是在法律、金融这些对准确性要求极高的行业。这时候大家发现,光是模型聊得好不够,还得让模型说的每一句话都能被验证、被追溯。这就好比你请了一个特别能说的顾问,他分析得头头是道,但你要做重大决策的时候,你肯定还想知道,他这个结论是看了哪份报告、哪一页、哪一句话得出来的。这种可验证性,在专业场景里是刚需,不是锦上添花。

好,理解了这个背景,我们来看看 Anthropic 给出的解法,也就是 Citations 这个功能到底是怎么设计的。

第一个核心点,我想讲讲它的基本工作机制。简单来说,当你启用 Citations 功能之后,你可以把源文档,比如说 PDF 文件或者纯文本文件,直接作为上下文传给 Claude。这里有个挺关键的设计,API 会自动把这些文档按句子切分成一个个小块,英文叫 chunk,我们可以理解成把一篇长文章切成一个个小段落或者小句子,每一块都有自己的编号和位置信息。然后这些切好的句子块,连同用户提供的其他上下文,一起喂给模型,再加上用户的问题。

模型在生成回答的时候,会分析用户的问题,然后基于这些切好的句子块生成答案,并且对答案里的每一个论断,都会附带一个精确的引用,告诉你这句话是基于哪个具体的句子块得出的。换句话说,模型不再是凭空说一句话然后让你自己去信,而是说一句话的同时,直接把这句话对应的原文出处也给你标出来了。如果你不想用系统自动的切块方式,你也可以自己提供切好的文档块,这样灵活性就更高了,你可以按照自己业务的需求去定义什么算一个引用单元。

这个设计我觉得特别巧妙的地方在于,它把引用这件事,从一个靠提示词去恳求模型完成的软任务,变成了一个由系统架构来保证的硬能力。这就好比你以前是靠口头嘱咐员工每次汇报都要附上数据来源,员工可能记得也可能忘记,现在变成系统自动给每条汇报数据打上来源标签,根本不用员工自己操心,这个可靠性自然就上去了。

第二个核心点,我想讲讲效果到底怎么样。文章里给了一个挺硬核的数据,Anthropic 内部的评测显示,Claude 自带的引用能力,比大多数开发者自己手写的提示词方案,召回准确率最多能提升百分之十五。这个召回准确率,通俗讲就是模型在该引用的时候,真正准确引用对了来源的比例。提升百分之十五听起来不算特别夸张的数字,但是在生产环境里,这种提升往往意味着大量人工核查工作的减少,意味着用户对系统信任度的显著提升。

更有意思的是文章里提到的两个真实客户案例。第一个是 Thomson Reuters,也就是路透社旗下的汤森路透,他们有一款产品叫 CoCounsel,是专门给律师和税务专业人士用的 AI 平台。这个产品的负责人 Jake Heller 提到,他们之前其实自己手动搭建过一套引用系统,但是搭建和维护成本特别高,体验也不够好。换用 Anthropic 的 Citations 功能之后,引用和溯源的工作变得容易实现、容易维护、容易部署给最终用户。这对律师这个职业来说太重要了,你想想,一个律师如果给客户出一份法律意见书,里面引用的法条或者案例如果对不上,那是要出大事的,职业声誉都会受损。所以法律行业对引用准确性的要求,几乎是零容错的。

第二个案例是一家叫 Endex 的公司,他们做的是给金融机构用的自主代理,英文是 autonomous agent,你可以理解成一个能自己规划步骤、自己执行多轮任务的智能助手,专门用来做复杂的金融研究。他们公司 CEO Tarun Amasa 给出的数据更加直观,引用功能上线之后,他们系统里来源幻觉和格式问题的比例,从百分之十直接降到了百分之零,同时每条回答里的引用数量还提升了百分之二十。他还特别提到一点,就是这个功能上线之后,他们再也不需要在提示词工程上费那么大劲去专门处理引用这件事了,省下的精力可以投入到更核心的业务逻辑优化上。这个案例特别能说明问题,因为金融研究本身就是一个多阶段、多文档交叉验证的复杂任务,引用准确率从百分之九十提升到百分之百,这个跃迁对一个严肃的金融工具来说,几乎是从可用到好用的质变。

第三个核心点,我想讲讲计费方式,这个细节其实很多人会忽略,但我觉得对实际使用很重要。Citations 功能用的是标准的按 token 计费模式,token 你可以理解成模型处理文本的最小单位,大概对应一两个汉字或者一个英文单词左右。启用引用功能之后,因为要把源文档处理成上下文喂给模型,可能会用到更多的输入 token,这部分是要正常付费的。但是有一个挺贴心的设计,就是引用本身返回的那段原文摘录,这部分输出的 token 是不收费的。

这个设计我觉得体现了 Anthropic 在产品定价上的一种思路,就是不应该因为你想要更可信、更可验证的回答,就要为这种可信性额外掏钱。如果你想一想,一个系统返回的引用片段,本质上是已经存在于你提供的源文档里的内容,模型只是把它原文摘录出来贴在回答旁边,而不是凭空生成新内容,所以从这个角度看,不为这部分内容收费,逻辑上也是说得通的。

接下来我想讲讲第四个点,也是我觉得稍微容易被忽略,但其实很关键的一个细节,就是 Citations 支持的文档格式。文章里提到,它能处理 PDF 文档和纯文本文件。这意味着开发者不需要先把 PDF 转成纯文本再喂给模型,系统直接就能处理 PDF。这一点对很多实际业务场景特别友好,因为现实世界里,大量的专业文档,比如合同、财报、研究报告,默认格式就是 PDF,如果每次都要先做格式转换这道额外的预处理,开发流程会变得更繁琐,出错概率也会增加。Anthropic 直接原生支持 PDF,某种程度上降低了开发者接入这个功能的门槛。

而且文章里特别强调了一点,这种方式不需要额外的文件存储系统支持,而是直接跟 Messages API 无缝集成。Messages API 是 Anthropic 提供给开发者调用 Claude 的标准接口。这意味着你不需要去搭建一套专门的文档管理和索引系统,只需要把文档作为请求的一部分传进去就行。这对中小团队或者个人开发者来说,意味着接入成本大幅降低,你不需要先去搞一整套检索增强生成的基础设施,英文叫 RAG,也就是 retrieval augmented generation,先把这套复杂的检索系统搭好,才能用上引用功能。

讲到这里,我想稍微往回拉一下,聊聊这篇文章背后反映的一个更宏观的产品哲学。Anthropic 一直强调 Claude 是 trustworthy 和 steerable by design,也就是说,可信和可控这两个特性,不是后期补丁加上去的,而是从模型训练阶段就内嵌进去的设计目标。Citations 这个功能,其实就是这个哲学的一次具体落地,它把可信这件事,从一句空泛的口号,变成了一个开发者真正能在 API 层面调用、能在产品里直接落地的具体能力。

那么这篇文章对我们今天的实际工作意味着什么呢,我们来聊聊实践应用这部分。

首先,如果你现在正在做或者打算做一个需要处理大量文档的应用,比如说企业内部知识库问答,或者客服系统,或者面向专业人士的研究助手,你应该优先考虑用模型原生提供的引用能力,而不是自己手写提示词去实现引用功能。这篇文章和案例已经很清楚地说明了,自己手写的方案在准确率上往往比不过系统原生支持的方案,而且维护成本还更高。这就好比你要建一栋房子,地基这种事情应该交给专业的建筑公司去打,而不是自己东拼西凑,看起来好像也能用,但风险隐患不知道埋了多少。

第二点,我觉得特别值得拿出来单独讲讲的是,这种引用功能特别适合那些容错率极低的场景。如果你做的产品是面向法律、医疗、金融这些专业领域,用户对错误的容忍度极低,一旦出现幻觉或者引用错误,可能造成的后果不仅是用户体验差,甚至可能引发实际的法律或者财务风险。在这些场景里,投入精力去启用并且测试引用功能,投资回报率会非常高。反过来说,如果你做的是一个轻量级的娱乐类聊天应用,用户对偶尔的不准确容忍度比较高,那么引用功能可能就不是你最优先要解决的问题,你可以把精力放在别的体验优化上。

第三点,我想提一下计费这块的实际影响。因为引用文字本身不计入输出 token 费用,这意味着如果你的应用场景里,模型经常需要大段引用原文,那么启用 Citations 功能在成本上可能反而比你自己实现一套引用逻辑更省钱。这一点在做成本预算的时候值得拿出来精确计算一下,尤其是当你的应用调用量比较大的时候,这种成本差异会被放大。

第四点,我想从一个开发者的角度,设想一下如果你来设计自己的应用,你应该怎么用好这个功能。假设你现在要做一个面向公司内部的合规文档问答系统,员工可以提问,系统需要根据公司的合规手册给出回答。在这种场景下,你完全可以把整本合规手册作为源文档传给模型,启用引用功能之后,每当员工问一个问题,系统给出的回答会清楚地标注每一条规定具体来自手册的哪一章哪一节。这样一来,员工不仅得到了答案,还能直接点击或者查看对应的原文,自己再做一次确认,这对合规这种事情来说,简直是降低风险的利器。如果是我自己来设计这样一个系统,我个人认为,这种内置的可追溯性,比单纯追求回答的流畅度更重要,因为合规场景里,用户要的不是一个听起来权威的答案,而是一个能拿去给监管部门交代的答案。

第五点,我想提个醒,虽然 Citations 听起来很美好,但它解决的是溯源问题,不是百分之百消除幻觉的问题。文章里用的措辞是 minimize hallucination risk,也就是降低幻觉风险,而不是彻底消除。这意味着即便你用了引用功能,模型依然有可能在理解或者总结源文档的过程中出现偏差,只是这种偏差现在可以被你或者用户更容易地发现和纠正,因为你有原文可以对照检查。所以我建议大家在实际部署的时候,还是要保留一些人工抽查的机制,尤其是在最初上线的阶段,不要因为有了引用功能就完全放松警惕,觉得万事大吉了。

第六点,我想从时间线的角度,跟大家分享一下我个人的一个观察。我们现在站在二零二六年年中回头看这篇发表于二零二五年六月的文章,会发现这种引用溯源的能力,后来其实成为了整个行业里,严肃 AI 应用的一个标配要求。也就是说,Anthropic 当时在这篇文章里提出的这套设计思路,某种程度上预见了后续行业发展的方向。今天如果你去看市面上比较成熟的企业级 AI 产品,基本上都会有类似的引用溯源功能,这已经不是一个加分项,而是一个准入门槛。所以我个人认为,这篇文章除了介绍一个具体功能之外,更值得我们关注的是它背后体现的产品判断,就是 Anthropic 很早就意识到,可验证性会成为企业级 AI 应用的核心竞争力之一,而不是一个边缘需求。
