各位 Claude FM 的听众朋友们，大家好，我是你们的主播。今天我们来聊一条虽然不长、但对企业 AI 落地影响非常大的消息。这篇文章的标题是《Claude in Microsoft Foundry is now generally available》，翻译过来就是“Claude 正式登陆 Microsoft Foundry，全面可用”。它发表于二〇二六年六月二十九日，距离今天二〇二六年七月五日才刚过去不到一周。可以说，这条新闻还带着热度，也正好踩在了整个行业从“模型炫技”到“模型落地”的节奏转换点上。

我们知道，进入二零二六年以后，各大云厂商基本上都已经完成了第一阶段的大模型能力展示，不管是 OpenAI 的 GPT 系列、Anthropic 的 Claude 系列，还是其他各家模型，大家的能力已经足够强，够能打。下一个战场就是怎么让这些模型真正跑在企业内部的环境里，既保持安全性、又降低集成门槛。而 Anthropic 和微软这次的合作升级，本质上就是把 Claude 变成 Azure 上的一等公民，像你平时用虚拟机、用数据库一样，在你自己熟悉的 Azure 环境下直接调用顶尖的 AI 模型。听起来很简单，但背后的意义其实很深。

那具体是怎么回事呢？我们一点点来拆解。

首先要理解这次“正式全面可用”到底意味着什么。在这之前，Claude 其实已经通过一个叫 Foundry Preview 的预览版本可以在微软的平台上使用了，不过那时候的形态还比较像是一个外部 API 被桥接到了 Azure 里面。而这次宣告 GA，也就是 General Availability，重点是增加了一种全新的部署模式，叫“hosted on Azure”，也就是“托管在 Azure 上”。它和原来的预览版本，也就是现在改名为“hosted on Anthropic”的那种方式，是完全不同的两个产品思路。后面我们会详细比较这两种方式，但现在你可以先记住一个关键点：hosted on Azure 意味着 Claude 推理跑在你的 Azure 环境中，用的是你现有的 Azure 认证体系、网络策略、治理控制，而且你还可以选择把数据推理限定在美国数据区内。Anthropic 来负责操作推理计算，扮演的是数据处理者的角色。这其实是一种典型的“配合你的合规要求，把 AI 能力带进你的地盘”的做法。

我们可以做个类比。这就好比之前你想用 Claude，需要专门跑到 Anthropic 的门口，刷他们发的门禁卡，在他们的房子里完成工作，然后把结果拿回来。这种方式适合个人开发者、创业公司，因为它快、简单。但现在很多大企业，尤其是金融、医疗、政府相关的机构，他们根本不让你把数据带出门。他们要求所有的计算、推理都得在自己租用的、管控严格的“办公楼”里完成，而且进出都有统一的门禁和监控。这个时候 hosted on Azure 就相当于 Anthropic 在 Azure 这个你已经租好的楼里，专门开辟了一块区域，由他们专业团队来运营 AI 推理服务，但认证依然用你的工牌，数据依然留在大楼里，账单也统一算在你的大楼物业费里。这样一来，安全、合规、采购流程全打通了。

所以说，这次 GA 最核心的价值不是模型本身变强了，而是 Claude 作为一款顶尖模型，终于以一种“企业原生”的方式嵌入了 Azure 的骨骼和血液里。对于已经在 Azure 上投了重注的公司来说，这简直就是一个迟到但重磅的礼物。

接下来我们具体看看现在你能用上什么。文章里提到，首批上线 Microsoft Foundry 的模型是 Claude Opus 四点八和 Claude Haiku 四点五。熟悉 Anthropic 模型的听众一定不陌生，Opus 是他们的旗舰模型，强在复杂推理、长文本理解和非常细致的指令遵循；而 Haiku 则是那个既快又便宜、特别适合做大批量并发任务的选手。这两个模型通过 Messages API 暴露出来，并且支持两大核心能力：prompt caching 和 extended thinking。我们一个个说。

Prompt caching，也就是提示缓存，这个东西在真正生产环境中简直就是成本控制的神器。如果你做过大模型应用，你一定遇到过这样的场景：每次请求都要把一个巨大的系统提示、一份很长的背景文档、或者一套固定的上下文重新发给模型。无论这份上下文有多长，每次都要吃 token，重复计费。而 prompt caching 允许你把那些反复使用的长文本缓存住，后续请求只要带上缓存的引用，就可以大幅减少输入 token 的开销。举个具体的例子，假设你在做一个企业内部的合同审核助手，你需要把整套公司法务规范、合同模板和常见条款库作为系统提示。这个提示本身可能就有几万 token。如果没有缓存，你审一份合同，光把这些规范重发一遍就要烧掉几万个 token 的成本；有了缓存，你第一次发过去以后，这些规范就被缓存了，后面审再多的合同，只需要花差异部分的 token。对于每天处理上千份合同的系统来说，节省的费用非常可观。

而且在 Azure 环境中用 prompt caching 还有一个额外的好处。因为推理是跑在 Azure 的数据中心里的，网络延迟非常低。如果你之前的应用是跨云调用，可能因为公网传输会损失一些性能，现在完全在 Azure 内部闭环，缓存的命中速度也会更快，整体体验更流畅。

再看 extended thinking，扩展推理。这也不是什么新概念了，但实战中用好它，能解锁 Claude 处理复杂问题的上限。简单来说，extended thinking 就是允许模型在内部进行更长的“思考”过程，它会在给出最终答案之前，生成并利用大量的中间推理步骤。这就好比让你去解一道复杂的奥数题，直接写答案你可能会出错，但给你一张草稿纸，让你把推导过程写下来再总结，正确率会急剧提升。Claude 支持你设置一个推理预算，比如“请花最多四千个 token 来思考”，模型就会在这个预算内做深度的分析、比对、推演，然后给出结论。对于代码重构、架构设计、多步逻辑推理这些任务来说，这简直是必备技能。

你可能会问，那这样会不会很慢、很贵？是的，扩展推理会消耗更多的推理时间和输出 token，所以它并不适合所有场景。但关键在于，它给了你一种“用空间换质量”的自由度。在 Azure 环境里，你甚至可以根据请求的复杂度，动态地在代码里决定要不要开启 extended thinking，比如普通对话用常规模式，用户一旦提出“请帮我设计一个分布式系统的容错方案”，就自动触发扩展推理。这种灵活性在集成了 Azure 的治理体系之后，还意味着你可以通过 Azure Policy 来限定哪些应用、哪些部门可以使用这个高级功能，控制成本的同时也保证安全。

讲完了模型和功能，我们来深入聊一聊这次发布中对工程师和架构师来说最激动人心的部分——Azure 原生集成。文章里用了几个关键词：Azure-native，existing Azure identity, networking, and governance controls。这短短的几行字背后，是一整套企业级能力的打通。我把它拆成认证、网络、治理、计费四个维度来看。

第一个，认证。在 Microsoft Foundry 里使用 Claude，你不再需要单独去 Anthropic 的控制台申请 API 密钥、然后把这个密钥小心翼翼地存到 Azure Key Vault 里还得担心它泄露。现在你可以直接用 Azure Active Directory 的身份来认证。也就是说，你的应用程序可以通过托管标识、服务主体这些标准的方式来调用 Claude，完全零密钥。对于安全团队来说，这意味着所有访问都可以被 Azure AD 的日志记录下来，可以接入条件访问策略，可以实现多因素认证，甚至可以与你们公司的身份治理系统联动。当有员工离职，他的账户被禁用后，他对 Claude 的调用权限也自动消失，不需要额外去回收一个外部 API 令牌。这在大企业里简直是把安全运维人员从一片钥匙链里解放了出来。

第二个，网络。在 Azure 上部署 Claude，你可以把它放进你自己的虚拟网络里，用 Private Link 这样的技术来确保推理数据永远不经过公共互联网。举个例子，如果你的应用后端跑在 Azure 的某个 VNet 里面，同时有些数据是高度敏感的，比如保险公司的客户健康信息，你就可以配置网络策略，让 Claude 的推理端点只允许从你特定的子网发起请求，外部完全看不到。这不仅提高了安全性，也极大降低了数据被意外暴露的风险。同时，因为推理就在同一个区域内，甚至可能就在同一个可用区，网络延迟也被压到了毫秒级。对于需要实时交互的 agent 应用来说，这直接决定了用户体验是丝滑还是卡顿。

第三个，治理。这是很多技术负责人特别看重的部分。Azure Policy 里可以定义无数种规则，比如“Claude 模型只能在美国东部和西部两个区域部署”，“不允许使用 extended thinking 除非有经理批准”，“所有对 Claude 的调用必须经过指定的 API 管理层”等等。这些策略可以在资源组、订阅甚至管理组级别强制实施。这样一来，企业就不需要依靠各个开发团队的自觉性，而是在平台层面就锁死了安全边界。之前很多公司想用大模型但迟迟不敢上，就是担心治理失控。现在 Claude 直接接入 Azure 的治理框架，等于给这些忧虑画上了句号。

第四个，计费。这也是很接地气的一点。Claude 的使用费用会直接出现在你的 Azure 账单上，跟你用的计算、存储、网络服务在同一张发票里。文章特别提到，对于符合条件并且有 Microsoft 企业协议的大客户，Claude 的使用量可以消耗 Microsoft Azure 承诺用量，也就是我们常说的 MACC。这是什么概念呢？很多大企业跟微软签了三到五年的 EA，预付了一大笔钱作为消费承诺，如果在合同期内用不完，这笔钱其实就浪费了。现在把 Claude 的推理费也纳入消耗范畴，等于给这些企业打开了一个巨大的财务优化空间。你可以把一部分原来准备花在别人家的 AI 预算，转变为消耗你早已承诺给微软的额度。CFO 看到这条消息可能会比 CTO 更开心。而且统一的账单和成本分析工具，让你能精确地看到每个部门、每个项目在 AI 推理上的花费，便于做运营决策。

好，现在我们再来聊一聊前面提到的“两种运行方式”的选择难题。Anthropic 在这篇文章里很清晰地把选择权交给了客户：一种叫 hosted on Azure，就是刚才我们一直在讲的，跑在你的 Azure 环境里，强调治理、数据驻留、与 Azure 全家桶的无缝整合；另一种叫 hosted on Anthropic，这是原来 Foundry Preview 的延续，推理实际跑在 Anthropic 自己运营的服务器上，但你还是通过微软 Foundry 的界面和 API 来访问。

这两种方式到底该怎么选？这个决策背后其实是企业当前所处的不同阶段和核心诉求的体现。

如果你是那种一切都要在自己掌控范围内的企业，比如有严格的数据常驻要求，交易数据绝对不能离开美国国境；或者你的安全政策规定所有的 AI 推理都必须经过自建的 API 网关；再或者你希望用 Azure Policy 把整个 AI 的使用行为管得死死的，那么毫无疑问，hosted on Azure 就是为你准备的。它的最大亮点是数据平面完全在你的 Azure 租户内，Anthropic 只是作为操作者来运维推理服务，你拥有最终的环境主权。

而如果你现在急需用上 Claude 全家桶里那些最新、最前沿的功能，比如一个在其他托管方式中已经上线、但还没同步到 Azure 环境里的新模型版本，或者是需要一些特定 API 参数配置的高级特性，那你可能会先选 hosted on Anthropic。它相对更灵活，模型和功能上线的速度也会快一点点，但它牺牲的是那种“开箱即用”的企业内网控制和统一账单里完美的 Azure 原生体验。好消息是，文章里明确强调，Anthropic 的目标是在未来让两种方式达到功能和模型上的完全对等。也就是说，现在的选择更像是一个时间差策略，而不是永远的阵营之分。

如果你是企业里的架构师，我个人觉得你可以在设计系统的时候，就把抽象层做好。你完全可以先用 hosted on Anthropic 的模式跑通业务逻辑，因为上得快，可以做到快速验证。与此同时，你的基础架构团队慢慢把 Azure 网络策略、Private Link、身份认证这些东西配起来，等 hosted on Azure 那边对应的模型版本 ready 了，你只需要改一个 endpoint 和认证方式，业务代码基本不用动，就能平滑迁移到更受管控的环境里。这种渐进式的策略非常符合大多数中大型企业的落地节奏。

再往下走，我们不妨把视角拉高一点，看看这次发布到底处在 AI 发展的哪个阶段，对咱们正在做以及将要做的项目意味着什么。

现在是二零二六年七月，距离大模型在开发圈里的第一波爆发已经过去了四年多。如果说二零二三年、二零二四年大家还在争论“模型会不会取代程序员”，那么到了二零二六年，讨论已经非常务实了——不是说取不取代，而是怎么把模型变成像数据库、消息队列一样可靠的基建。Claude 在 Microsoft Foundry 的全面可用，正好切中了这个“AI 基建化”的命题。

你想象一下，一个典型的 Azure 企业客户，可能已经在用 Azure OpenAI 服务部署 GPT 模型来构建内部的聊天应用、知识库问答系统。他们已经有了成熟的 CI/CD 流程、监控报警、成本归集。现在 Claude 的加入，并不是替换什么，而是多了一种战略选择。不同模型有不同的气质和能力，Opus 可能在法律文书、代码架构设计上表现突出，GPT 可能在创意生成、多模态混合上有独特优势。对于开发者来说，在同一个 Azure 控制面上，可以用同一套模板、同一个 IaC 脚本，就能把不同家族的模型当成不同的“技能包”分配给不同的业务场景。比如，常规的客户意图识别路由给 Haiku，复杂投诉处理路由给 Opus 加 extended thinking，而市场文案生成可以继续用 GPT。这种多模型的混合编排，才是企业真正落地的常态。

而且别忘了，Claude 在 Foundry 里是 Azure 原生的资源，这意味着你可以用 Bicep 或者 Terraform 来定义它的部署，把基础设施即代码的理念延续到 AI 模型的管理上。你的 CI 管道可以在部署应用的同时，自动创建 Claude 资源、配置好缓存和推理设置，然后跑一轮集成测试。这种自动化程度，在两年前还是很难想象的。

接下来，我必须得花点时间专门聊聊阅读这篇文章时很容易忽略、但实际上极为关键的一个点，就是数据处理器角色和数据驻留。文章里非常清晰地写了：“Anthropic operates the inference and is the data processor.” 这句话在法律和合规的语境下，含金量极高。

在很多数据保护法规里，企业作为数据控制者，如果你把数据交给外部服务商，你必须明确那个服务商到底是数据处理者还是数据控制者。如果 Anthropic 是数据控制者，意味着它可能有更多利用数据的权利，但这就会把你拖进复杂的联合控制协议里。现在 Anthropic 明确表示自己是数据处理者，也就是说，它按照你的指示来处理数据，不会把你的数据用于模型训练，不会拿去做分析，唯一的任务就是帮你跑推理。而且前面说了，推理可以跑在你的 Azure 环境下，数据甚至可以不经过 Anthropic 的服务器，这种架构下，你对数据的控制力达到了前所未有的高度。

美国数据区的选择也很有意思。过去用 Anthropic 的直连 API，你其实很难精准控制推理发生在哪个物理位置，可能因为负载调度绕到了地球另一边。现在，你可以把推理指定在美国数据区内，这对于很多持有美国公民数据的机构来说，直接满足了一类刚需合规。再结合 Azure 已有的全球区域能力，虽然文章只提到了美国数据区，但不难想象，未来很可能会有更多区域选择开放出来。

对于我们做技术选型的人来说，这意味着在给企业写评估报告的时候，Claude 在 Microsoft Foundry 的方案在安全审计、法务审核那里会拿到更高的分数，更容易通过，从而缩短整个采购和落地周期。这个优势有时候比模型能力多几个百分点更重要。

好了，理论讲得差不多了，我们进入最接地气的部分：如果你现在就想去试试，该怎么上手，又会踩哪些坑？

第一步当然是打开 Microsoft Foundry 的界面。你只要有一个 Azure 订阅，在 Foundry 的项目中就可以直接创建 Claude 资源。创建时你会看到一个很重要的选择，就是模型部署在哪里——是“托管在 Azure”还是“托管在 Anthropic”。如果你选择了托管在 Azure，接下来需要选择区域，比如美国东部或西部。一旦创建完成，你会得到一个 endpoint 和一个认证密钥，当然，更推荐的做法是用 Azure AD 的无密钥方式。对于熟悉 Azure 的你来说，这些步骤应该毫无陌生感。

拿到 endpoint 之后，你的调用方式跟调用任何兼容 Messages API 的服务基本一致。你可以用 Python 的 requests，或者用 Azure SDK 里集成的 AI 服务客户端，构建一个 JSON 请求，里面包含你的 model 名称、系统提示、用户消息列表，以及是否开启扩展推理的参数。响应回来的也是标准格式。你甚至可以把你之前在 Anthropic 直接 API 上跑的代码改几个连接参数就迁移过来，非常顺滑。

不过，在实际生产环境中，有三个坑值得特别提醒。

第一个坑是提示缓存的有效期和策略。虽然 prompt caching 能省钱，但缓存不是无限期的，它有保质期，通常几分钟到几十分钟，而且万一后台因为扩缩容或滚动更新清空了缓存，你就会在某个时间段内突然多出一大笔输入 token 费用。你一定要在应用中做好降级方案，比如用缓存命中信号来判断，如果未命中，即使多花点钱也要保证业务不断，同时在监控看板里把缓存命中率设成核心指标来观察。

第二个坑是 extended thinking 的预算设定。设置多少 token 的推理预算需要反复实验。给少了，模型思考不充分，效果提升不明显；给太多，不仅延迟增加，成本也直线上升，而且有时候会出现过拟合式的“想太多”，反而给出奇怪的答案。我的建议是，从实际任务中抽一批典型案例，做 A/B 测试，从零预算一直测到你认为理想的那个预算水平，找到性价比最高的拐点。另外，一定记得在 API 里限制最大输出 token，不要无限制让模型发挥，否则一张账单就能让你怀疑人生。

第三个坑是关于托管方式切换时的行为差异。虽然 Anthropic 承诺未来会拉平功能，但在当前的时间点，hosted on Azure 和 hosted on Anthropic 可能在参数接受范围、速率限制、以及个别模型微调行为的细节上存在差异。举个例子，你可能会发现同一条 prompt 在两边跑出来的回答有一点点不同，这通常是底层推理优化实现不一样造成的。所以在做迁移的时候，不要想当然地把所有测试用例重跑一遍，集成测试必须覆盖所有关键路径，确保切换方式不会引入回归问题。

如果你让我给出一条实践建议，那我会说：先把 hosted on Azure 当作你们的最终目标架构来设计安全、网络和合规，这不会变。然后如果某些功能急需，先用 hosted on Anthropic 顶上，同时把应用层设计成可配置的模型提供者，用策略模式或者简单的配置开关来切换。这样你们永远保持灵活，不会被某一种托管方式卡死。

最后，再推演一下这次发布的行业影响。假设你是一个 SaaS 公司的 CTO，你们的平台服务了上千家企业，之前你一直想在你的产品里集成一个强大的 AI 助手，但是客户对数据安全的顾虑让你很难推进，因为你要么得自建模型推理集群，要么得跟客户一个个签复杂的数据处理协议。现在有了 Claude hosted on Azure，你可以直接告诉客户：“所有 AI 推理都在您自己的 Azure 租户里完成，数据绝不离开您的环境，而且您可以完全通过 Azure Policy 控制助手的访问和记录。”这直接扫清了 B2B AI 产品最大的销售障碍。它的本质是让 AI 能力从一个需要客户信任的黑盒子，变成了客户自己基础设施的一部分。这是信任模型上的一次质变。
