## Shownotes

## Script

今天我们要聊一篇来自 Anthropic 官方博客的文章,标题是 Auto mode for Claude Code,中文我把它翻译成 Claude Code 的自动模式。这篇文章发布于二〇二六年三月二十四日,到我们录这期节目的时候,差不多过去了三个月。在 AI 这个行业里,三个月不算长,这个功能现在应该还处在持续打磨和逐步扩大开放的过程中,所以你可以把今天讲的内容,当成一个还在演进中的功能的最新进展,而不是一个已经盖棺定论的成熟方案。

那这篇文章为什么值得花时间听呢?因为它讨论的问题,几乎是所有在用 AI 写代码、用 agent 也就是智能体去自动完成任务的人,都会遇到的一个两难。你给 AI 越多自主权,它干活越快越省心,但风险也越大;你管得越死,每一步都要你点头确认,安全是安全了,但你也没法真正解放双手。Claude Code 这次推出的自动模式,就是想在这两者之间,找一个新的平衡点。我们今天就把这个平衡点到底是怎么设计出来的,讲清楚。

我自己平时用 Claude Code 的时候,印象最深的体验就是,一旦任务稍微复杂一点,涉及到改好几个文件、跑几条命令,确认框就会一个接一个地弹出来,你的注意力其实根本没办法离开屏幕,跟你预期里那种扔给 AI 一个任务、自己去干别的事的场景完全是两回事。这种割裂感,我猜很多用 agent 类工具的人都遇到过,而这篇文章正是想从产品设计的层面,去解决这个体验上的核心矛盾。

先说一下背景。Claude Code 是 Anthropic 出的一个命令行编程助手,你可以理解成一个能读你的代码、能帮你改代码、能帮你跑测试跑命令行指令的 AI 搭档。从它诞生开始,默认的权限设计就是偏保守的,只要 Claude 要写一个文件,或者要执行一条 bash 命令,系统都会先弹出一个确认框,问你这一步要不要批准。这样做的好处很明显,你对 AI 的每一步操作都有控制权,它不可能在你不知情的情况下,把你的代码仓库改得面目全非,更不可能误删什么重要文件。

但这个设计也有一个明显的代价。如果你想让 Claude 去做一个比较大、比较长的任务,比如说一次性重构一个模块,或者批量处理几十个文件,理论上你应该可以扔给它一个任务描述,然后去喝杯咖啡,回来发现活干完了。但实际上因为每一步都要你确认,你必须一直坐在电脑前,时不时地点一下同意,这跟放手不管的理想状态差得很远。这种感觉很像有些公司里那种审批流程特别严的报销系统,每一笔哪怕只有几块钱的小额支出都要走完整套审批,流程是安全的,但效率低得让人崩溃。

这种每一步都要确认带来的疲劳感,其实有个很现实的副作用,就是人会慢慢变得麻木。这跟我们生活里遇到的很多安全提示是一个道理,比如电脑上那种系统权限弹窗,一开始你还会认真看一眼再点,后来次数多了,很多人会变成看到弹窗就反射性地点同意,根本不去看内容写的是什么。如果一个安全机制设计得让用户疲于应付,那它实际起到的保护作用,可能远不如设计者期望的那么大,这也是为什么单纯靠事事都问这种思路,长期来看未必是最优解。

正因为这种摩擦感,一部分开发者会选择一个更激进的做法,就是用一个叫做跳过所有权限确认的启动参数,直接把 Claude Code 的安全阀全部打开,让它不再问任何问题,自己想干什么就干什么。这样做确实痛快,但风险也是实打实的。文章里原话讲得很直白,跳过权限检查可能会导致危险的、破坏性的结果,不应该在隔离环境之外使用。你可以把这个模式想象成,把一个能力很强、但偶尔会犯糊涂的新人,直接给了他公司所有系统的最高权限,还顺手把监控摄像头都关了,他大概率不会出事,但万一出事,可能就是大事。

这其实是整个智能体类 AI 工具发展过程中,一个共性的难题,不只是 Claude Code 一家会遇到。你给智能体越多自主权,它处理长任务的体验就越流畅,但一旦它的判断出了偏差,造成的后果也会被这种自主权放大。这跟我们生活里很多自动化系统面临的困境是一样的,比如自动驾驶要解决的核心问题之一,就是什么时候应该让系统完全自主决策,什么时候必须把控制权交还给人。

如果你了解一点自动驾驶的分级标准,会发现业界其实早就习惯了用级别去描述自主程度,从需要人全程盯着方向盘的辅助驾驶,到几乎不需要人干预的完全自动驾驶,中间还有好几个过渡阶段,每往上走一级,系统能自己拿主意的范围就更大一点,但相应地,系统对自己判断的把握也要更稳一点,出问题时的兜底机制也要更完善一点。我个人觉得,Claude Code 现在面对的处境跟这个非常类似,默认权限模式相当于辅助驾驶,每一步都要人盯着;跳过权限检查相当于直接跳到完全自动驾驶,但底下的安全冗余还没跟上;而自动模式,更像是中间那种系统大部分时候自己开、遇到复杂路况主动把方向盘交回给人的过渡阶段。自动模式,就是 Claude Code 团队针对这个难题给出的一个新方案。

我们先来看自动模式到底是怎么运作的。它的核心设计,是在每一次 Claude 要执行工具调用之前,插入一个分类器,英文叫 classifier,你可以把它理解成一个专门做风险判断的小模型,它的工作只有一个,在 Claude 真正动手之前,先看一眼这个操作有没有危险。具体来说,它重点盯防三类高风险行为,第一种是大规模删除文件,比如说一条命令把整个目录甚至整个项目清空;第二种是敏感数据外泄,比如把密钥、密码、用户隐私数据发送到外部;第三种是执行恶意代码,比如下载并运行一些来路不明的脚本。

这个分类器干活的逻辑,很像信用卡公司的风控系统。你刷卡买一杯咖啡,系统几乎是瞬间放行,你不会感觉到任何延迟;但如果你突然在国外刷了一笔异常大的金额,系统就会把这笔交易先挂起来,可能给你发个验证短信,确认一下是不是真的是你本人在操作。自动模式里的这个分类器做的也是类似的事情,大部分时候 Claude 要做的操作都是修改几行代码、跑个单元测试这种家常便饭式的动作,分类器看一眼就放行,完全不打断你的工作流;但一旦它识别出某个操作落在了高风险的范围里,就会把这个操作拦下来,不让它执行。

我们可以把文章提到的三类高风险行为,拆开来具体感受一下。大规模删除文件,你可以想象成 Claude 想执行一条类似清空整个目录的命令,这种命令一旦敲错一个参数,可能波及到的就不只是项目代码,还有你本地其他无关的文件夹;敏感数据外泄,可以想象成 Claude 在调试的时候,不小心把一个包含数据库密码的配置文件内容,当作日志打印出来,或者顺手发到了某个外部接口里;恶意代码执行,可以想象成 Claude 为了完成任务,从网上下载了一个来源不明的脚本然后直接运行,而这个脚本里到底写了什么,谁也没仔细审查过。这三类风险的共同点是,一旦真的发生,造成的损失往往是不可逆的,所以分类器会把识别这几类行为,作为优先级最高的任务。

那操作被拦下来之后,Claude 会怎么办呢?这是我觉得这篇文章里设计得比较巧妙的一个细节。它不是简单粗暴地报错、然后什么都不做,而是会引导 Claude 换一种思路,去尝试别的实现方式完成同一个目标。比如说,如果 Claude 想用一条比较暴力的命令去清理一批文件,而这条命令被分类器判定为有大规模删除的风险,Claude 不会就此放弃任务,而是会去想,有没有更保守、更精确的方式去完成同样的清理工作,比如只删除真正需要删除的那几个文件,而不是用一个可能误伤无关文件的笼统指令。

但如果 Claude 反复尝试,反复被拦截,系统不会让它无限制地一直试下去。文章里提到,如果 Claude 坚持要做一直被拦截的动作,最终会触发一个面向用户的权限提示,把决定权交还给你。这个机制我觉得特别像公司里常见的层级审批逻辑。一个基层员工遇到问题,会先自己想办法绕过去,试几种不同的解决路径,但如果试了几次都解决不了,或者每次都撞上公司的合规红线,他不会无限制地硬闯,而是会去找主管,把问题升级上去,让有决策权的人来拍板。自动模式本质上,就是把这种先自己试、试不通再升级给人的工作方式,搬到了 AI 智能体身上。

我们可以脑补一个具体的场景帮助理解。假设你让 Claude 帮你清理一个项目里所有过期的缓存文件,Claude 第一次尝试,用了一条范围比较大的删除命令,分类器一看,这条命令涉及大规模删除,直接拦截。Claude 收到拦截信号之后,不会傻乎乎地原地重复同一条命令,而是会调整策略,比如先列出符合过期缓存这个条件的文件清单,再针对清单里的每一个文件单独删除,这样风险就小了很多,分类器可能就会放行。但如果 Claude 换了几种思路,分类器还是一直判定有风险,比如说它发现这批文件的命名规律比预想的复杂,无论怎么收窄范围,都没办法做到精确匹配,这时候系统就会停下来,把这个判断权交还给你,让你亲自看一眼,到底该删哪些、不该删哪些。整个过程里,你作为人类,只在最后这个真正需要你判断的节点上被打扰一次,而不是每一步操作都要你来点头。

讲到这里,你可能会觉得这个机制听起来已经很稳了,那是不是就可以完全放心地把 Claude Code 扔到任何环境里跑了?文章里很坦诚地给出了一个不那么讨喜、但很重要的提醒,自动模式相比跳过所有权限检查,确实降低了风险,但并没有把风险降到零,官方依然建议在隔离环境中使用它。

为什么不能做到万无一失呢?文章里提到了两种典型的失效场景。第一种是用户意图本身就模糊不清,分类器没办法准确判断这个操作到底是不是你真正想要的,如果你的任务描述本身有歧义,分类器很难替你把关。第二种是 Claude 对当前环境缺乏足够的上下文信息,不知道某个看起来普通的操作,在你这个特定环境里其实暗藏风险,比如说,删除一个看起来是临时文件的东西,但实际上这个文件是某个生产服务正在读取的配置文件,分类器如果不了解这层背景,就很难识别出风险。除此之外,文章也很诚实地承认,分类器偶尔也会误判,把一些其实完全无害的操作错当成危险操作给拦下来,这会带来一些不必要的打扰。

我自己的理解是,这种降低风险但不消除风险的表述,其实是 Anthropic 在产品沟通上一直比较谨慎的风格,他们没有把这个功能包装成绝对安全的解决方案,而是把它定位成一个比之前更安全、但依然需要你自己有安全意识的折中选项。这种坦诚我个人是比较认可的,毕竟任何基于模型判断的风控系统,本质上都是概率性的,不可能做到百分之百精确,提前把这个预期讲清楚,比事后出了问题再解释,要负责得多。

这里我想多说一句自己的看法。任何风控系统其实都要在两类错误之间找平衡,一类是把危险的东西放过去了,另一类是把无害的东西误判成危险拦下来了。这两类错误是有点像跷跷板的关系,你想把第一类错误压得越低,往往就得把判断标准设得越严格,而判断标准越严格,第二类误判就会越多,体验上就会变得越来越像默认的逐步确认模式,自动化的意义也就被削弱了。所以分类器具体把这个平衡点定在哪里,直接决定了自动模式到底是真正好用,还是只是把烦人的确认框换了个出现的频率。这也是为什么文章里特别强调,会持续改进这个体验,我猜背后很大一部分工作,就是在不断调整这个平衡点。

还有一个容易被忽略、但很实际的点,就是性能代价。因为每一次工具调用之前都要多跑一次分类器的判断,这个额外的步骤本身是需要消耗算力和时间的。文章里直接提到,自动模式可能会对工具调用的 token 消耗、成本和延迟带来一些小幅度的影响。token 你可以理解成模型处理文字时计费和计算的最小单位。换句话说,这种安全性不是完全免费的,你多换来的这一层保护,是要付出一点点速度和费用上的代价的。

这跟我们平时用的很多安全工具,其实是同样的取舍。比如你手机上装的杀毒软件或者安全扫描应用,实时监控一定会占用一点系统资源,让手机稍微费一点电、跑应用稍微慢一点点,但大部分人都会觉得这点代价是值得的,因为它换来的是关键时刻能拦住一次真正的风险。自动模式里这个分类器带来的延迟,本质上也是同一类账,你在每一次工具调用上多花一点点时间和算力,换来的是整个任务可以放心地长时间运行,不需要你时刻盯着。从我个人的判断来看,这个代价应该是值得的,毕竟你换来的是可以长时间无人值守地运行任务,这种效率提升,远远盖过了单次工具调用多出来的那一点延迟。当然如果你做的是那种对响应速度极度敏感的小任务,比如就是改一行代码这种几秒钟就能搞定的事,可能默认模式或者自动模式之间的体验差异,你都感觉不出来,这种场景下用哪种模式都无所谓。

最后讲一下这个功能目前所处的发布阶段。自动模式目前支持两个模型,Claude Sonnet 四点六,和 Claude Opus 四点六。在发布的时间点上,它先以研究预览的形式,向 Team 计划的用户开放,这是 Anthropic 面向团队客户的一个订阅层级,随后会在接下来的一段时间里,逐步扩展到 Enterprise 企业版用户,以及通过 API 调用 Claude 的开发者。

这里的研究预览这个说法值得多解释一句。一般来说,科技公司用这个词,意思是这个功能已经可以让真实用户用起来了,但还没有完全定型,团队会一边观察大家实际使用中的反馈,一边继续调整背后的判断逻辑,所以你现在看到的分类器表现,跟它半年以后的表现,很可能已经不是同一个水平。这也是为什么我们在开头就提醒大家,这篇文章讲的内容还在快速演进当中,如果你听完这期节目之后去试用,实际体验跟文章描述的可能会有出入,这是正常的,不代表文章讲错了,只是这个功能本身还在迭代。这种先小范围测试、再逐步扩大开放范围的发布方式,在 Anthropic 过去推出新功能的时候也比较常见,毕竟涉及到权限和安全这种敏感领域,谨慎一点、多收集一些真实使用反馈再扩大范围,是更稳妥的做法。

讲完了这个功能本身,我们来说说,如果你是一个日常在用 Claude Code 写代码的开发者,这个东西具体要怎么用,以及用的时候要注意什么。

先说怎么开启。如果你是命令行用户,直接在终端里运行一条命令,claude 加上 enable dash auto dash mode 这个参数,就可以把这个模式打开,然后在使用过程中,用 Shift 加 Tab 这个组合键,就可以在不同的权限模式之间切换,切到自动模式。如果你用的是桌面客户端,或者 VS Code 里的插件版本,流程会稍微不一样,你需要先在设置里找到 Claude Code 这个选项,把自动模式的开关打开,然后在具体的会话里,从权限模式的下拉菜单里手动选择自动模式。值得一提的是,在桌面客户端上,这个功能默认是关闭的,需要管理员专门去组织设置里打开才能用。

如果你所在的团队有专门的管理员在负责 Claude Code 的统一配置,这里也有一个对应的开关。管理员可以在统一管理的配置文件里,把一个叫 disable Auto Mode 的字段设置成 disable,这样就可以在命令行工具和 VS Code 插件里,把自动模式整体关掉。如果团队出于合规或者其他考虑,暂时不想让任何人用这个功能,这是一个集中管控的入口。

那这个功能到底适合用在什么场景呢?我觉得最典型的就是那种任务边界相对清晰、但操作步骤很多、需要持续跑一段时间的工作。比如说你要对一个比较老的项目做一次大规模重构,把某个过时的依赖库,从代码库里几十个文件中统一替换掉;或者你要写一个批处理脚本,对几百个数据文件做格式转换;再或者你想在晚上睡觉前,把一个明确的任务交给 Claude,让它在你不在电脑前的这几个小时里,自己跑完整个流程,第二天早上起来直接看结果。这些场景的共同点是,任务目标很明确,但中间步骤又多又繁琐,如果每一步都要你手动确认,体验会非常差,而完全跳过权限检查又让你不放心,自动模式正好填上这个中间地带。

再具体一点,我们可以列举几个比较贴近日常工作的例子。如果你是后端工程师,可能会用它来跑一次跨服务的接口签名统一升级,涉及十几个微服务仓库,每个仓库都要改几处调用方式,这种重复性强、但又不是完全机械、需要判断上下文的活,正好适合让 Claude 自己跑通大半,你只要在关键节点确认一下;如果你是做测试或者质量保障的同学,可能会用它去跑一轮大范围的测试用例补全,让 Claude 扫描整个项目里测试覆盖率比较低的模块,自动补充单元测试,这种任务步骤多、单步风险低,特别适合自动模式发挥;如果你是带团队的技术负责人,也可以考虑用它去做一次代码规范的统一治理,比如把项目里几种不同风格的日志写法,统一成一种标准格式,这种任务最大的痛点就是文件数量多,逐个确认的成本会高到让人放弃,而自动模式恰好能把这种知道要干什么、但懒得一步步盯的活解放出来。

不过这里有几个坑,我建议你提前注意。第一,即便用了自动模式,官方仍然建议你在隔离环境里跑,比如说用容器,或者用一个干净的虚拟机,或者至少是一个专门用来做实验、跟你生产环境完全隔离的代码仓库。不要在一台能直接连到生产数据库、或者保存着重要密钥的机器上,开着自动模式跑任务,这一点文章里反复强调,我觉得是因为分类器终究是基于模型的判断,它不了解你环境里那些只有你自己才知道的隐性风险,留一道物理隔离,是最后一道保险。

第二,如果你的任务描述本身就很模糊,比如你只说了一句帮我把这个项目清理一下,没有讲清楚清理的边界在哪里,那分类器很难帮你兜底,因为它判断风险的依据之一就是你的意图够不够清楚。所以我的建议是,即便开了自动模式,你给 Claude 的任务描述也应该尽量具体,把你想要的范围、不想动的东西,提前讲明白,这样不仅能减少分类器误判的概率,也能让 Claude 少走弯路。

第三,如果你发现某个操作被反复拦截,触发了权限提示弹给你确认,这其实是系统在告诉你,Claude 在尝试的这件事,可能本身就触及了某种边界,这时候不要无脑点同意,花十几秒看一下它具体想干什么,再做判断,毕竟这是系统给你留的最后一道把关的机会。

从更大的视角来看,我个人认为自动模式代表的是整个智能体编程工具,从一步一审往自主执行、异常升级这种模式演进的一个缩影。其他做智能体编程工具的团队,这几年也都在尝试类似的思路,有的给自己的版本起了不同的名字,但本质上都是在解决同一个问题,就是怎么样既能让 AI 长时间自主干活,又不至于在无人看管的时候捅出大乱子。自动模式这次的做法,我觉得比较聪明的地方在于,它没有简单粗暴地用一份固定的规则列表去判断什么能做什么不能做,而是用一个会判断上下文的分类器,加上一个会自己换思路、并且知道什么时候该把问题升级给人的智能体,两者配合。这种自适应加适时止损的组合,可能会成为未来很多自动化系统设计的一个参考范式。

如果你身边也有同事在纠结要不要直接用跳过权限检查这种激进方式去跑长任务,我觉得这篇文章给出的思路值得参考,先不要一步到位地把所有权限都交出去,而是看看有没有这种带风险判断的中间方案可以用,哪怕暂时还轮不到你所在的计划开放,至少可以把这个设计思路用到你自己团队内部的工具建设上。比如说,如果你们公司也在自己搭一套内部的 agent 自动化平台,完全可以参考这个思路,在关键操作前面加一层风险判断,而不是简单地用一个全有或全无的权限开关。

我自己还想多延伸一点,如果换成是我来设计这样一个系统,我会比较关注分类器的判断标准能不能让用户看得到、改得动。文章里目前披露的信息,更多是从用户体验的角度去讲这个功能解决了什么问题,但具体分类器的判断逻辑是固定的,还是可以根据每个团队、每个项目的具体情况去做一些定制,这一点文章里没有展开。我个人是希望未来这种风控机制能给团队留一些自定义空间,比如让团队自己定义在我们这个项目里什么目录绝对不能碰、什么操作必须经过人工确认,毕竟每家公司、每个项目的风险点都不完全一样,一个通用的分类器很难覆盖所有团队的特殊情况,这也是我觉得这类功能在落地过程中,接下来比较值得关注的一个方向。

好,我们简单总结一下这期的内容。Claude Code 这次推出的自动模式,本质上是在每一步都要人工批准,和完全跳过权限检查这两个极端之间,引入了第三条路。靠一个实时风险分类器,在操作执行之前先做一次判断,安全的放行,危险的拦截并引导 Claude 换思路,如果反复被拦截就把决定权交还给人。这个设计降低了长任务无人值守运行的风险,但并没有彻底消除风险,该有的隔离环境、清晰的任务描述,这些基本的安全习惯依然不能丢。

我个人的看法是,这种渐进式的权限管理思路,可能比一步到位地追求完全自主更现实,也更容易让普通开发者放心去尝试。毕竟信任是一点一点建立起来的,与其一次性把所有权限都交出去,不如先在一个相对安全的边界里,让 AI 证明它值得被信任。这期咱们就聊到这里,下期再见。
