## Script

今天我们来聊一篇刚刚出炉的文章，标题是"Meet the Winners of the Built with Opus 4.7 Claude Code Hackathon"，中文可以翻译成"认识 Opus 四点七 Claude Code 黑客松的获奖者们"。这篇文章发表于二〇二六年六月十五日，距今刚好四天，非常新鲜。

这是 Anthropic 对他们上一场线上黑客松的总结报道。六个项目，六位来自不同背景的参赛者，用 Claude Opus 四点七这个模型，在五天之内，从零开始各自构建了一款完整的产品。参赛者里有医生、有修电器的工匠、有智利的大学教授、有一个从来没写过任何代码的二十岁年轻人，还有两位工厂里的工程师。

我把这篇文章看了好几遍，越看越觉得有意思。它表面上是一篇"看看别人用 AI 做了什么"的成果展示，但背后藏着一个更大的命题：当 AI 工具强大到这个程度，谁才是下一批真正的产品构建者？是会写代码的工程师，还是真正理解问题的领域专家？这两者之间的边界，正在以一种让人惊讶的速度消失。

这就是我今天想和你聊的核心。让我们一个一个来看这六个项目。

先说一下背景。这场黑客松叫做"Built with Opus 4.7"，是 Anthropic 举办的线上比赛，参赛者可以用 Claude Opus 四点七来构建任何他们想做的东西。值得注意的是，就在这篇文章发布的同一周，Anthropic 还在旧金山举办了一场叫做 Claude Build Day 的线下活动，这次用的是更新的 Claude Opus 四点八模型。所以这篇文章是对上一场比赛的总结，让我们可以看到那些已经完整做出来的项目是什么样的。

Claude Opus 四点七是 Anthropic 在这一代 Opus 系列里的一次重要能力提升，特别是在视觉理解方面——它能读懂复杂的电路图、原理图和截图，具备相当强的空间推理能力，同时多步骤复杂推理也有明显提升。很多参赛者正是冲着这些新能力来的，尤其是视觉理解这一块，这在我们后面聊几个项目的时候会反复出现。

好，背景交代完了，我们来逐一看六个获奖项目，我会尽量讲透每个项目的核心设计和开发方法，因为我觉得这些细节对你自己做产品是有直接参考价值的。

第一个项目叫 Medkit，创作者是 Bedirhan Keskin，来自土耳其伊斯坦布尔，是一位医生转型的软件工程师。

Bedirhan 造 Medkit 的初衷，来自一个非常真实的痛点。他说，当你独自值班在急诊室，外面有五十个病人在等着，你突然发现面前这个病例是你在医学院从来没有练习过的情况，你怎么办？你没有办法，只能在真实病人身上边做边学。

这个场景听起来让人后背发凉，但它就是现实。医学教育有一个根本性的矛盾：想要培养出有经验的医生，需要大量的临床案例练习；但真实的病人不是教学道具，你不能拿病人来给学生练手。医学院在这个矛盾里已经挣扎了很多年，他们有模拟假人、有纸质案例讨论，但缺乏一个真正能模拟完整诊疗过程的环境。

Medkit 就是为了打破这个困境。它是一个游戏化的医学培训工具，专门给住院医生和初级医生使用。学生在一个模拟的医疗诊所里，面对虚拟的模拟病人来练习诊断和治疗。这个过程是完整的：可以问病史、开化验单、查看影像资料、做出诊断、开具处方。到最后，会有一个由 AI 驱动的评分代理，对整个诊疗过程进行评估，评估标准是公开发布的临床指南，和真实的临床执照考试完全一致。

Bedirhan 在构建 Medkit 的时候，用了 Claude 托管代理这个功能。他把整个项目拆分成四个独立的 Claude Code 会话：语音引擎、内容生成、三维游戏层，以及核心应用程序。这四个会话同时推进，每一个的上下文保持干净聚焦，彼此不互相污染。这是一种非常有意思的工程组织方式，类似于把一个大项目拆成四个专注的小队，同步推进但互不干扰。

他还有一个特别的工作习惯，他几乎全程用语音输入和 Claude 交流，不打字。他把这叫做"说话而不是打字"的工作方式。对于一个从医疗背景转型来的工程师来说，这也许是最自然的沟通方式。

在开发过程中，有一个细节让我印象很深。Bedirhan 最初的计划是自己搭建语音引擎，但是 Claude 建议他用云服务提供商，说这样可以快很多。他听了这个建议，省下了大量时间。他说他最看重 Claude 的地方，不是它作为代码生成器有多快，而是它作为思维伙伴，能帮他看到他自己可能忽略的选项。

Medkit 已经在产生实际影响了。伊斯坦布尔的三所医学院和一家制药公司，准备在接下来几周开始试点。

第二个项目叫 Wrench Board，创作者是来自法国 Reignier-Ésery 的 Alexis Chapellier。他在创办自己的修电子产品管理平台 RepairMind 之前，花了多年时间修各种电子产品。

你可能没想过修一块主板有多复杂。当一台笔记本进来，显示不开机，你面对的是一张密密麻麻的电路原理图，一个电路板的三维视图，还有一大堆可能的故障点。你要在脑子里构建出整个电气系统的样子，然后一步一步缩小范围，找到那个需要测量的具体焊点，确认测量值，更新你的推断，再去下一个地方。对于经验丰富的老技师来说，这是直觉；对于遇到没见过的故障类型的人来说，这是相当有挑战性的推理过程。

Wrench Board 就是为这些独立技师做的诊断辅助工具。用户把原理图和电路板视图上传，描述一下故障症状，AI 代理就开始工作。它会把所有输入整合成一张统一的电气图，在上面进行推理，然后告诉你要测量电路板上的哪个具体焊点，读取测量结果，更新它的假设，然后继续推进，直到找出问题所在。

Alexis 在开发这个项目的时候，对 Opus 四点七的视觉理解能力押了一个大赌注。他说他知道这个方向走对了的那一刻，是当他让模型追踪主板上的一条电源路径的时候。他看到 Wrench Board 的电路板视图一步一步点亮，箭头出现了，元件被一个一个标记出来，对应的名字也浮现出来。在那一刻，他知道这个想法是成立的。

在开发方式上，Alexis 用了 Claude Code 的多代理模式，在调试阶段同时跑五到六个代理并行运行，每个代理负责一个独立的领域。他先在 Claude Design 里做出设计和规格说明，再通过内置按钮把项目直接交接给 Claude Code 来执行。

他给其他开发者的建议是：头脑风暴要深入，而且要主动推回 AI 给的限制性结论。他说在黑客松期间，Claude 好几次告诉他某件事在剩下的时间里做不到，但实际上他时间很充裕。他说你必须懂得告诉 AI，我还是要试试。

他说这场黑客松证明了，一个从修理店走出来的自学者，可以在五天之内交付一个有野心的系统。进入这场黑客松的时候，他还在申请"维持生计的工作"。我个人认为这句话是整篇文章里最有力量的几句话之一。它说的不是技术，是机会的重新分配。

第三个项目叫 Maieutic，这个词来自古希腊语，是苏格拉底教学法里"助产术"的意思——通过问问题引导对方自己产出知识，而不是直接告诉他们答案。创作者是 Paula Vásquez-Henríquez，她是智利 Concepción 大学的计算机科学教授，同时在攻读人工智能博士学位，研究方向是学生与 AI 的互动模式。

这个项目的起点是一个让很多编程教育者都在头疼的问题。Paula 说过去两年，她发现越来越多的学生能通过测试，但完全不理解自己写的代码。

她对这个现象的分析非常到位，我来大致转述一下她的原话：学生们现在用 AI 来生产代码，但他们根本不知道这些代码在做什么。他们从来没有学会如何准确地描述一个问题，如何在写代码之前先规划出方案，或者如何批判性地阅读自己的代码、注意到代码和原本意图之间的偏差。自动补全在他们还没想完问题的时候就给出了可以跑的代码，所以那个"思考你的思考"的元认知循环——那个真正把一个人培养成程序员的过程——从来没有完成。他们毕业的时候能生成代码，但不能对代码进行推理。

这是一个很深刻的观察。问题不是说 AI 让学生变懒了，而是 AI 在错误的时机介入，直接跳过了学习过程中最关键的认知环节——那个从"不会"到"理解"必须经历的思考过程。

Maieutic 这个工具的设计，就是为了在正确的时机踩刹车。具体来说，学生在写任何代码之前，必须先用自然语言描述他们的程序应该做什么。Claude 会问针对性的澄清问题，而且编辑器会被锁定，这个锁定状态会一直维持，直到规格说明足够详细、足够具体，让一个有能力的程序员可以不用猜就能实现它。

学生然后才能开始写 Python 代码，但自动补全是关掉的。工具里有一个聊天面板，对于参考类的问题——比如"这个函数的语法是什么"——会直接回答，但是对于推理类的问题——比如"这段代码为什么不工作"这种——它只会反问，拒绝替学生思考。

最核心的功能叫做"意图差异审查"，这也是我觉得设计得最聪明的一个地方。Claude 会把学生一开始写的规格说明和最终的代码进行比较，把每一处差异分类成三种：偏移，就是代码和最初的意图不一样了；修订，就是学生在写代码的过程中合理地更新了自己的想法；以及 bug，就是代码本身有错误。然后它会用一个中立的、非指责性的问题，让学生自己解释这个差异。不是"你写错了"，而是"这里和你最初说的不太一样，你能说说是什么原因吗"。

对于老师来说，有一个实时仪表板，每个学生占一行，显示一句话的认知摘要，比如"这个学生改了三次规格说明，还没有考虑到空输入的情况"。老师可以点进去看某个学生具体是怎么和 Claude 互动的，同时系统还会分析整个班级，找出大家共同的误解，让老师可以集中解决这些共性问题。

黑客松结束后，休斯顿大学的研究者联系了 Paula，想一起合著论文。她说这一周让她意识到，从理解一个问题到造出一个工具，这中间的距离已经大幅缩短了。她的原话是：我是智利 Concepción 的一位教育工作者，不是硅谷的人，但我在一周内交付了一个完整的全栈产品，因为这些工具让我留在了我真正擅长的角色里，而它们负责处理其余的事。

在开发方式上，她给其他人的建议是先想清楚再动手。她专门花了两天时间，什么代码都不写，纯粹做设计规格说明和技术规格说明。她说那两天感觉很慢，在黑客松里有真实的压力要立刻开始产出，但正是那两天，让剩下的一整周都跑得又快又准。

第四个项目叫 Virtual Puppet Theater，中文是虚拟木偶剧场，创作者是来自丹麦的全栈开发者 Rene Hangstrup Møller。这个项目是六个获奖作品里最纯粹、最不功利的一个，它不解决什么行业痛点，就是为了好玩。

Rene 对 Opus 四点七的空间推理能力很感兴趣，于是做了一个完全在浏览器里跑的应用：把摄像头拍到的视频和语音，变成一场动态的互动木偶剧表演。一个实时动画的木偶会跟着用户的动作来动，与此同时，另一个由 AI 驱动的伴侣木偶会和用户对话、斗嘴；用户随口说出一段描述，就可以变换舞台场景，或者在三维空间里凭空生成新的道具。

技术上，这个应用基于 Bun、Vite 和 TypeScript 构建，用 MediaPipe 做手部追踪，运行在 WebAssembly 里面，用 Three.js 在浏览器里以每秒六十帧的速度渲染三维舞台。一个小型 WebSocket 服务器连接到 Claude Opus 四点七，驱动 AI 木偶的对话生成和三维道具的实时生成。语音输入用 Web Speech API，语音输出用 ElevenLabs，浏览器原生语音合成作为备用。Rene 还建立了一个基于截图的反馈循环，把屏幕截图喂给 Opus，让它来改进自己的视觉输出效果。

这个项目没有明确的商业目标，没有产品规划，Rene 说他用最小的儿子测试了这个工具，儿子和木偶互动、描述场景、被 AI 的回应逗得咯咯笑。他说看到这一幕，就是他需要的全部用户验证了。

这个项目的源代码已经在 GitHub 上用 MIT 许可证开放了，任何人都可以继续开发。

Rene 给参加黑客松的人留下了一条非常实际的建议：一定要预留整整一天来做演示视频。他说做一个三分钟的视频花的时间比你想象的长得多，他自己差点没赶上截止时间，最后是用 Claude 和 Hyperframes 来剪辑的。他说 Discord 里很多参赛者提前警告过他，他们是对的，下次他会把最后整整一天都留给做演示，而不是再加一个功能。

第五个项目叫 MaestrIA，这个名字把西班牙语的"maestro"也就是工匠大师，和英文的 AI 拼在了一起。创作者是来自智利奇洛埃岛的二十岁年轻人 Benjamin Torralbo，这是六个项目里在我心里留下印象最深的一个。

Benjamin 从小跟着父亲学习木工，他父亲 Juan Rodrigo Torralbo 是获得官方认证的"Maestro Mayor"，也就是高级工匠大师，曾经修复过联合国教科文组织认定的历史教堂。但他父亲在智利的官方体系里是隐形的，就像几十万其他技工一样：有三十年深厚的专业积累，却没有正式的认可机制，也没有平台让人找到他、信任他。另一边，普通人家需要维修，不知道问题出在哪里，不知道该花多少钱，不知道该找谁，更不知道自己有没有被坑。

MaestrIA 想同时解决这两个问题。

从用户角度，使用流程是这样的：你拍一张问题的照片，用语音或者文字描述一下，分享自己的位置。Claude 开始实时流式输出它的推理过程，在你的照片上用动态包围框一步步标注问题区域，然后给出结构化的诊断：什么地方出了问题、是什么材料、严重程度是一到五的几分、大概要多少钱、要多久。然后 AI 代理在地图上列出附近按专业筛选的工匠，同时另一个代理自动起草一条 WhatsApp 消息，让你一键就能发给工匠。

这里有一个技术细节，我觉得特别值得多讲一会儿，因为它展示了一个非常有价值的产品开发思路。MaestrIA 的核心竞争力，藏在一个 JSON 文件里。这个文件包含了十七条诊断规则、七种奇洛埃岛本地木材的特征信息、十六个当地行业术语、十九个基准价格，以及九种常见的施工错误，全部来自 Benjamin 花了好几个小时采访他父亲之后整理出来的内容。

神奇的是，仅仅是把这个文件注入到每次诊断里，完全没有改动系统提示，就让准确率从百分之七十四提升到了百分之八十一。评估标准是把 AI 的诊断和他父亲这位真人大师的判断进行比较，七个百分点，就来自于把领域知识结构化、系统化之后放进 AI 的推理上下文里。也正是因为这个，MaestrIA 能诊断出"奇洛埃岛落叶松木外墙板上的毛细潮湿"，而不只是泛泛地说"木头有损坏"。它知道本地的木材、本地的工法、本地的价格，这是通用 AI 做不到的，因为通用 AI 没有这份非常特定的本地知识。

Benjamin 在这个项目开始之前没有任何编程经验。他说他把自己的角色定义为工地总监，监督 Claude 的技术执行。在开始写任何功能之前，他先让 Claude Code 设计规格说明、分阶段行动计划和安全模型，包括输入清理、频率限制、来源验证，以及用 Zod 做统一的数据校验，然后他一个代码差异一个代码差异地审查每个功能。

他给其他人的建议是：先建评估体系，再做功能。他说他做过的最重要的一件事，是在开始之前就建立了一个九个维度的评估体系，用十二个真实案例做基准，这些案例的标准答案是他父亲记录的。是这个评估体系，而不是他的直觉，告诉他什么在起作用、什么不行。他说如果再参加一次黑客松，评估体系会是第一个提交的东西。

他说 Claude Code 让一个来自奇洛埃岛、没有任何编程经验的二十岁年轻人，能够建造出他父亲可以真正用上的软件，并且可以帮助智利另外二十八万个像他父亲一样的工匠。

第六个也是最后一个项目是 ARIA，全称是 Adaptive Runtime Intelligence，自适应运行时智能，赢得了"最佳使用 Claude 托管代理"的特别奖项。

每个工厂里都有那么一两个老师傅，只要听一听机器的声音，就知道它快要出问题了。这种判断力是几十年经验积累出来的，它非常宝贵，也非常脆弱，因为它存在于一个人的脑子里，这个人退休了或者离职了，这份能力就跟着消失了。ARIA 想做的就是把这种直觉系统化，变成一个平价、快速部署的 AI 预测维护系统。

用法很简单：维修工程师上传制造商的 PDF 手册，用日常语言回答四个校准问题，十五分钟内工厂的设备画像就建立好了。然后五个代理持续监控实时信号，如果任何代理发现故障或者预测故障即将发生，它立刻生成一份工单，分析是什么组件、什么故障模式、紧急程度如何、需要哪些零件、干预窗口有多长。

ARIA 的两位创作者都有工业现场的实际经验，他们是在黑客松的"寻找队友"频道里认识的。Idriss Benguezzou 是法国的工业软件工程师，有数据和 AI 的硕士学位，他早就在脑子里规划过这个想法和大部分架构。Adam Hnaien 是自学成才的工程系学生，有丰富的 Claude Code 和多代理工作流经验，他一看到这个想法，立刻认出这是工业维护领域真正需要的东西。

他们的开发方式非常值得参考。黑客松第二天，他们整天都在用 GitHub 项目看板做规划，把每一个里程碑、每一个 issue 和每一个验收标准都确定好，然后才开始写第一行代码。Adam 说他们想从第二个里程碑开始以百分之二百的状态投入，所以需要在最开始把后面的路全想清楚。一天的规划让他们剩下的整周都在执行，而不是即兴发挥。

两人估计 Claude Code 写了大约百分之八十的原始代码行，他们自己负责领域逻辑和关键设计决策。Idriss 亲手处理了阈值评估、知识库的数据模型和异常检测逻辑，因为他说，你不能靠提示词来知道一个维修技术员实际上关注哪些指标。Adam 亲手负责了用户体验、视觉语言设计，因为他说，你也不能靠提示词来产生审美眼光。这两句话我觉得很准确地道出了人机协作的边界：数据驱动的推理和领域直觉，是人这边的护城河。

关于 Claude 托管代理这个功能，Adam 说得非常直接：没有这个，他们就会把整个黑客松周都花在搭基础设施上——沙盒 Python 环境、安全执行、会话持久化、MCP 调度——而 Anthropic 已经把这些都托管好了，直接可以用。有了这些，他们才能把整整一周都花在产品本身上。这就是五天交付 ARIA 和五周交付 ARIA 的区别。

黑客松结果公布后，有公司主动联系了他们，表示正在做类似的问题，想进一步合作。

好，六个项目全部聊完了。现在我想从这些故事里提炼出一些对你的实际工作真正有用的东西。

第一条，也是我认为最重要的一条：AI 让领域专家变成了产品构建者，这不只是一个口号，而是这六个项目共同证明的事实。

仔细看这六个项目，真正推动核心价值的，往往不是编程技能，而是领域知识。Bedirhan 懂急诊科的真实临床痛点；Paula 深刻理解编程教育里认知发展的规律；Benjamin 懂奇洛埃岛本地木工的材料、价格和行话，而且能把这些知识结构化；Idriss 懂工厂维护工程师真正关注的指标，知道什么数字超出阈值代表什么含义。如果没有这些领域知识，即使最强的 AI 也做不出来有针对性的产品。

对我们意味着什么？如果你有深度的行业经验，现在 AI 工具已经可以帮你把这份经验转化成产品，"我不会编程"这个障碍已经大幅降低了。但反过来，如果你只会编程、没有领域洞见，单靠写代码的竞争优势也在变小，因为 AI 能帮任何有洞见的人写代码。价值越来越集中在"懂问题"这一侧，而不是"能写代码"这一侧。

第二条：先做规格说明，再开始编码。

这一条在六个项目里面至少有四个明确强调了。Alexis 先做设计规格和技术规格再开始执行；Paula 专门花两天做这件事，甚至把"先写规格说明再动手"这个原则嵌入了她的产品设计本身；Idriss 和 Adam 把整个第二天都用来做规划；Benjamin 在写任何功能之前都先让 Claude 生成规格说明。

在 AI 辅助开发的时代，越想跑快，越要先停下来想清楚。因为如果方向错了，AI 帮你跑得更快，只是更快地走错路。把前期的时间花在想清楚上，剩下的执行就变得又快又准。Paula 说那两天感觉很慢，但正是那两天让剩下的一周跑得很快，这句话值得反复想想。

第三条：先建评估体系，再添加功能。

Benjamin 说得最清楚：他在开始之前就建立了一个可审计的、九个维度的评估体系，用十二个真实案例做基准，标准答案由他父亲记录。是这个评估体系，而不是他的直觉，告诉他什么在起作用、什么不行。他说如果再参加一次黑客松，评估体系会是第一个提交的东西。Idriss 的建议也印证了这一点：在建下一个功能之前，先让 Claude 审计你已经建好的部分，看看有没有问题，这个循环的价值被严重低估了。

在 AI 产品开发里，"感觉好像跑通了"和"真的在按预期工作"之间的差距可以非常大。特别是当 AI 在中间参与推理的时候，它可能言之凿凿地给出一个错误答案。只有测量才能带来改善，感觉是靠不住的。

第四条：把 AI 当思维伙伴，不只是代码生成器。

Bedirhan 的那个例子很能说明这一点。他本来打算自己搭语音引擎，Claude 建议他用云服务提供商更快，他听了，省下了大量时间。如果他只是把 Claude 当成"我告诉你做什么、你来实现"的执行工具，就不会有这个对话，也就不会有这个节省。两种使用模式之间有本质的不同：一种是你已经想好了，AI 帮你实现；另一种是你还没想好，你和 AI 一起把这个问题想清楚。后者往往能发现你自己想不到的更好路径。

第五条，专门给打算参加黑客松的朋友：一定要预留整整一天来做演示视频，不要等到最后一晚才开始。

Rene 的教训是，一个三分钟的演示视频花的时间比你做一个功能还长。在黑客松里，你的作品再好，如果视频做得乱糟糟，评审就看不出来好在哪里。把最后一整天留给做演示，而不是再加一个功能。这一条听上去很小，但在真实比赛里可能是决定名次的事。

好，到这里今天这期节目的主要内容就聊完了。

用两句话总结一下。这篇发表于二〇二六年六月十五日的文章，背后其实展示的是同一件事：在今天，从真正理解一个问题，到构建出一款能用的解决方案，这中间的距离已经大幅缩短了。不管你是医生、修电工、教授，还是一个从没写过代码的二十岁年轻人，只要你真正理解某个问题，AI 工具可以帮你把这份理解变成可以运行的产品。

我个人认为，这六个项目里最让我触动的是 MaestrIA。不是因为它技术上最复杂，而是因为它把一位老工匠三十年积累的经验数字化，变成了可以帮助二十八万人的工具，让那些在正式体系里"隐形"的专家得到了应有的流传。这里面有一种技术以外的价值，是我希望你在这期节目结束后能带走的那个东西：你身处的问题场景，你的领域知识，你对某件事的深刻理解，可能正是你最大的竞争优势。
