今天我们解读的文章标题是 Working at the frontier: How Cognition trusts Claude Fable 5 to work through the night，中文可以译为，在前沿工作：Cognition 如何信任 Claude Fable 五彻夜工作。这篇文章发表于二〇二六年七月十日，离今天二〇二六年七月十二日只有两天，所以它不是一篇需要提醒大家注意过时的旧文章，而是非常贴近当前 AI agent，也就是能自主调用工具、规划步骤并完成任务的智能体，正在发生变化的前沿现场记录。

这篇文章为什么值得听？因为它讲的不是又一个模型发布会，也不是一个抽象的跑分榜单，而是一个非常具体的问题：你敢不敢在睡觉前，把一个真实代码库里的复杂任务交给 AI，让它在云端跑一整夜，第二天早上起来再看结果？这个问题对软件工程师特别有现实感。我们平时用 AI 写个函数、解释一段代码、补一个单测，已经不稀奇了。但让它连续工作八小时，期间自己查日志、理解上下文、判断根因、做迁移，还要尽量不引入暗坑，这就完全是另一个难度等级。

文章的主角是 Cognition，这家公司做的是 Devin，一个自主 AI 软件工程师。你可以把 Devin 理解成一个不只是聊天的 coding agent。它不是简单回答你问题，而是接任务、读代码库、调用工具、跑测试、改代码、提结果。根据文章，Cognition 的客户从高速成长的创业公司到 Fortune 五百强公司都有，所以它面对的不是玩具项目，而是真正要进生产环境的工程任务。这里的要求很苛刻，因为代码不是写出来能跑一次就行。一个很小、很隐蔽的 bug，如果悄悄进入主干，后面可能会影响数据迁移、支付链路、后台任务，甚至引发生产事故。

文章采访了 Cognition 的研究高级副总裁 Silas Alberti。他和团队几乎测试过每一代 Claude 模型，把它们放进 Devin 里面用。Alberti 的判断很直接：Claude Fable 五是第一个他愿意让它通宵运行的 Claude 模型。注意这个说法的含金量。不是说它能答对更多面试题，也不是说它在某个榜单上更高，而是说一个每天和 agent 打交道、被各种模型坑过的人，第一次觉得，睡前把任务交出去，不看着它，也许是可以的。

要理解这篇文章，先要理解一个背景。Cognition 在二〇二四年初做出 Devin 的时候，行业里的 agent 其实还很脆。今天我们回头看，那时候很多 agent demo 很惊艳，但基本机制并不稳定。所谓 agent，听起来像一个会自己干活的数字员工，但真正实现起来，背后是一连串很容易断掉的环节。它要知道目标是什么，要把目标拆成步骤，要在中途调用工具，要读工具返回的信息，要根据新信息调整计划，还要记得之前为什么这么做。任何一个环节漂了，最后都可能变成看似忙了很久，其实越走越偏。

这也是很多工程师过去两年使用 coding agent 的共同感受。刚开始你会觉得它很神奇，能一下读懂很多文件，还能提出修改建议。但你给它一个稍微长一点的任务，比如把一个老的数据库访问层迁移到新的接口，或者排查一个线上延迟尖峰，它可能前十分钟表现不错，半小时后开始忘记边界条件，一小时后开始自信地改一些不该改的地方。你看它的输出，语言很流畅，态度很坚定，但你心里没底。它到底是真的理解了，还是只是把最像答案的东西先说出来？

文章里有一句话很能概括 Cognition 的态度，Alberti 说，我们不相信任何 eval。eval 是 evaluation 的简称，就是评测或者基准测试。这个说法不是说评测没用，而是说，Cognition 不会因为一个模型在公开榜单上分数高，就自动相信它能进 Devin。因为他们见过太多模型，benchmark 做得很好，一进真实开发场景就崩。文章说，他们被这种事情烧伤过很多次。一个模型可能会在题目清晰、测试完整、上下文干净的环境里表现优异，但真实代码库不是这样。真实代码库有历史包袱，有命名不一致，有半废弃模块，有 flaky test，也就是偶发失败的测试，有注释和实现不一致，还有没人愿意碰的迁移脚本。

所以 Cognition 的做法是，把新模型交给自己最有判断力的工程师，在真实工作日里用。这里原文用了 highest taste developers，可以理解为品味很高、判断很准的开发者。他们不是只看模型有没有交差，而是看这段代码是不是自己真的愿意保留。这个标准特别重要。很多 AI 写的代码看起来完整，测试也许能过，但有经验的工程师一看就知道后面会难维护，抽象层次不对，错误处理不对，日志不够，或者把复杂性藏到了更危险的地方。Cognition 要的不是“能演示”，而是“能留下”。

文章回顾了 Claude 模型在 Devin 里的一个关键跳跃。Alberti 认为，第一个真正明显的跃升发生在二〇二四年末的 Claude 三点六 Sonnet。那是第一个能够比较可靠地串联工具、保持多步骤任务的模型。也就是说，它不仅能说，还能在多个工具调用之间维持任务线索。Cognition 把它接进 Devin 以后，内部使用量增长了三倍。这个细节很有意思，因为内部使用量往往比外部宣传更能说明问题。工程师如果觉得一个工具真有用，会自然多用；如果它老添乱，就会被静默抛弃。内部使用量三倍，说明那一代模型已经从“偶尔惊喜”进入了“开始能干活”的阶段。

但即便如此，仍然有一个天花板没有突破。文章把这个天花板叫做 horizon，我这里翻译成自主工作视野，或者任务续航长度。它指的是一个 agent 在失去人类持续监督之后，能在多长时间里不跑偏、不忘目标、不陷入表面答案。Alberti 说，在 Fable 之前，你可以把任务委托给 agent，它们可能能保持几分钟，或许一个小时。但之后 session 就会 drift，也就是会漂移。你给它五个想法同时权衡，它会开始混乱。它不是完全停下，而是看起来还在工作，可是目标已经偏了。

这个 horizon 概念对我们理解 AI 工程化特别关键。很多人比较模型，只看单轮回答质量，就像看一个候选人在白板上解一道题。但真实软件工作更像长跑，甚至像值夜班。第一公里跑得快不代表能跑马拉松。一个工程师接到迁移任务，可能先要理解现状，再找所有调用点，再确认兼容性，再写迁移计划，再分批修改，再跑测试，再处理边界，再复盘。中间会遇到日志噪声、错误信息误导、测试环境差异、业务规则隐藏在旧代码里。模型的上下文再长，如果没有稳定的目标管理和自我校正能力，也会在这些干扰里迷路。

文章举了一个数据库迁移的例子。早期某个 Opus 模型从技术上完成了任务，但过程中引入了一系列细微 bug。这个场景每个工程师应该都懂。迁移任务最怕的不是大面积编译失败，因为那种问题显眼，很快能发现。最怕的是逻辑上差一点点，比如字段默认值处理不一致，事务边界变了，某些异常被吞掉，分页条件漏了，或者只在特定老数据上出错。这类 bug 很像房子里的细小裂缝，交付当天看不出来，等压力上来才暴露。

排障场景也一样。文章说，早期模型在 incident triage，也就是事故分诊或故障初步排查时，常常停留在日志表面，不会真正挖到相关那一行。而且它们被训练得倾向于无论如何给出答案，所以会自信地抓住第一个看起来合理的东西，然后停下。这个描述非常真实。线上排障最危险的不是“不知道”，而是“过早知道”。一个经验不足的人看到第一个报错，就说根因就是它；但资深工程师会问，这个报错是因还是果？它是不是每次都有？时间线对不对？有没有上游变化？有没有并发症状？模型如果总是自信地给出第一个 plausible，也就是貌似合理的解释，团队很快就会学会忽略它。信任不是靠语气建立的，而是靠一次次判断准确建立的。

那么 Claude Fable 五到底跨过了什么门槛？文章里最具体的证据来自 Cognition 自己的 Frontier Code 基准测试。Frontier Code 是 Cognition 自建的评测，因为他们认为已有评测经常奖励那种能通过测试、但无法在真实代码库中生存的代码。Alberti 把它叫做 anti-slop standard，可以翻译成反垃圾代码标准。这里的 slop 是最近 AI 圈常见的词，指大量看似成形、实际粗糙低质的 AI 生成内容。在代码里，slop code 可能能过简单测试，但结构脆弱、语义含糊、维护成本高。

在 Frontier Code 最难的子集上，之前的 Opus 模型大约得了百分之十，而 Claude Fable 五大约得了百分之三十。注意，这个分数听起来不高，三十分也不是满分。但在最难子集上，从百分之十到百分之三十，是三倍提升。更重要的是，Cognition 团队第一反应不是欢呼，而是怀疑。他们问，是不是有 bug？这不可能是真的。这个反应很工程师。真正经常做评测的人都知道，分数突然跳升，第一件事不是写宣传稿，而是查评测有没有泄漏、有没有统计错误、有没有样本偏差、有没有某类题目被意外优化。

通常，benchmark 跳了以后，工程师还要吵好几周，讨论模型在实践里到底有没有更好。因为分数和体感经常不一致。但这一次，dogfooding，也就是内部吃自己的狗粮，和数字一致。Alberti 说，这老实讲有点震惊。也就是说，模型不只是跑分漂亮，实际放进工程师日常工作里，也能让人感到明显不一样。

这种不一样最集中体现在 horizon 上。Alberti 说，他们注意到最大的东西，就是它能自给自足多久。他举了一个很形象的例子。有些任务，他晚上准备睡觉前会说，好吧，请你继续做这个，不要停，直到我醒来。然后第二天醒来，发现它连续工作了八小时，而且确实在取得真实进展。他说，在此之前没见过这种情况。

这里我们要稍微放慢一下。八小时听起来只是时间更长，但本质上是信任模型从“工具”向“同事”的转变。一个普通工具，你必须在旁边操作，像 IDE 的自动补全，或者一个问答助手。一个同事，你可以交代目标、边界和优先级，然后去做别的事。第二天你不一定直接合并他的代码，但你期待他能带回有价值的进展，而不是一堆需要你从头清理的烂摊子。Claude Fable 五让 Cognition 看到的，就是这种从“我盯着它用”到“我可以暂时离开”的变化。

文章解释说，这个 horizon 能撑住，是因为 Claude Fable 五在混乱上下文里保持清醒。messy context，混乱上下文，是软件工程的常态。我们不要想象模型面对的是一份精心整理的题目说明。它面对的可能是浏览器里的日志页面、内部调试工具、含噪声的错误堆栈、多个服务之间的调用链、不同人写的注释、还有一些不完整的历史信息。文章说，Claude Fable 五是第一个能正确使用 Cognition 内部调试工具的模型。它会在浏览器里翻日志，在噪声中得出结论。

这个能力很容易被低估。很多模型会调用工具，但会不会用工具是另一回事。就像一个初级工程师也能打开日志平台，但他可能只会搜关键词；资深工程师会看时间窗口、看请求标识、看异常前后的状态变化、看相关服务是否同时报警。工具调用不是机械动作，而是带着假设去验证。一个好的 agent 不是“我查了一下日志”，而是“我怀疑根因在某个阶段，所以我查这个时间段的这几类日志；结果支持或不支持这个假设；下一步我要缩小范围”。文章没有展开具体日志内容，但它强调 Fable 五能在噪声中抽取相关信息，这正是排障从表演变成生产力的分界线。

数据库迁移例子里，Claude Fable 五的做法也值得注意。文章说，在那个曾经绊倒早期模型的迁移任务上，它先陈述自己要保持的 invariants，然后按这些 invariant 执行。invariant 可以翻译成不变量，指在修改过程中必须始终成立的约束。比如我举一个通用例子，不是原文里的具体细节。你迁移用户状态字段时，不变量可能是：已有状态语义不能改变，历史数据映射必须一一对应，失败时不能部分写入，新旧路径在灰度期间结果一致。真正的工程迁移，核心不是“把 A API 换成 B API”，而是守住这些不变量。模型如果能先说清楚自己要守住什么，再执行，就会比一上来改文件可靠很多。

这对我们使用 AI 写代码也有启发。很多时候我们给 prompt，也就是提示词，只说“帮我迁移这个模块”或者“修这个 bug”。如果是一个人类工程师，他会主动问边界；但模型可能直接开干。我个人认为，未来有效使用 coding agent 的关键之一，就是把任务目标转化成约束和验收标准。你不只是告诉它做什么，还要告诉它什么不能被破坏，哪些行为必须保持，哪些测试或日志必须证明它没跑偏。Fable 五在文章中表现出的进步，是它能自己陈述 invariant，但这不意味着人类就可以完全省掉这一步。越关键的系统，越要把不变量写清楚。

在事故分诊场景，文章提到另一个很重要的信号：Claude Fable 五能定位根因，并且会说自己不知道什么。Alberti 说，这才是真正重建信任的东西。这句话特别值得工程团队记住。我们往往以为 AI 的价值在于给答案，但在复杂系统里，可信的“不知道”比不可信的“知道”更值钱。一个模型如果说，我认为根因很可能是某个发布引入的配置变化，因为时间线、错误分布和日志都支持；但我还不能确认下游缓存是否参与，需要再查某个指标。这样的回答，哪怕不完整，也能帮助工程师继续推进。相反，如果它用很漂亮的话把一个猜测包装成结论，反而会浪费团队时间。

这也解释了为什么文章里反复强调 trust，信任。信任不是觉得模型聪明，而是知道它在什么时候可靠，什么时候会停下，什么时候会承认不确定。对生产系统而言，不确定性管理本身就是能力。一个好的 oncall 工程师不会为了显得专业而硬给结论，他会把已知事实、假设、证据和未知项分开。Claude Fable 五如果开始具备这种行为，就不只是“更会写代码”，而是更接近工程协作中的可托付角色。

文章把这次跃升放进一个很小的类别里。Alberti 说，这属于真正的 step change，也就是阶跃式变化，大概一年出现一次。这个判断来自一个长期使用者，而不是外部观察者。过去几年大模型进步很快，但并不是每次版本更新都改变工作方式。有些更新让回答更流畅，有些让成本更低，有些让特定题型更好。但真正的阶跃，是你原本不敢做的事情突然可以试了。对 Cognition 来说，原本不敢放心让 agent 跑一整夜，现在开始可行，这就是工作方式层面的变化。

接下来，文章转向 Cognition 对未来的判断。Cognition 创立时有一个核心赌注：agent 应该在云端一次运行几个小时。注意，是云端，不是在你本地开着一个聊天窗口等它回复。云端长期运行意味着它可以像后台工人一样，持续观察、执行、汇报。可是在公司第一年，模型能力还没到。也就是说，产品愿景在前，底层模型在后。很多 AI 应用公司过去两年都处在这种状态：你想做一个真正自主的系统，但模型只支持半自主；你想做一个长期任务代理，但它只能短程冲刺。

Alberti 说，Claude Fable 五让这个赌注的完整版本变得可行，而且其中一部分已经进入产品。文章提到，Devin 可以观察一个 Slack channel，也就是团队聊天频道，在没有被明确 tag，也就是没有被点名的情况下，自己跳进一个 issue；或者监控生产环境，对某个 spike，也就是指标突然尖峰，进行初步分诊。当它做对的时候，Alberti 说，感觉就像团队里真的有一个工程师。

这段描述很能说明 agent 产品从被动到主动的变化。今天很多 AI 工具还是被动的。你问，它答；你派任务，它做。但工程团队真正忙的时候，问题不是没人会问 AI，而是问题太多、信号太散、上下文太碎。一个 issue 在 Slack 里出现，可能只是某个人说“刚才接口是不是慢了一下”；一个生产 spike 可能只持续几分钟；一个 bug 可能在用户反馈、监控图和日志之间各露一点头。主动 agent 的价值，是它能先看到这些信号，自己去扫代码库、查最近变更、找可能根因，然后带着一个可验证的修复建议来找你。

文章最后给了一个大胆预测。Alberti 预计，一两年后，百分之九十的 agent session，也就是智能体会话，会是 proactive ones，主动型会话。它们会自己发现问题，扫描代码库，然后带着修复方案给你发消息。换句话说，未来你打开工作消息，可能不是你先问 AI“帮我看看这个 bug”，而是 AI 先说“我发现某个服务的错误率在某个发布后上升，我查了相关代码，怀疑是这个条件分支导致的，我准备了一个修复，你要不要看”。这听起来像科幻，但从文章的角度看，Claude Fable 五让 Cognition 觉得这条路开始现实了。

当然，我们也要保持冷静。文章是 Anthropic 博客，讲的是 Anthropic 模型在合作伙伴 Cognition 产品里的表现，它天然会突出成功案例。原文没有公开 Frontier Code 的具体题目构成，也没有给出完整统计区间、失败案例比例、不同任务类型的详细对比。所以我们不能把它当成一篇学术论文来读。更合适的读法是，把它当作一个前沿产品团队的工程判断：他们在真实代码工作流里感到某个门槛被跨过了。这个判断本身很有价值，但我们落地时仍然要做自己的验证。

那对正在通勤路上听节目的你，实际意味着什么？如果你是软件工程师、技术负责人、平台工程师，或者 AI 应用开发者，我建议你从几个方面吸收这篇文章，而不是只记住“Claude Fable 五很强”。

第一点，也是最重要的，建立你自己的真实评测。不要只看公开榜单，也不要只看模型厂商 demo。Cognition 的“我们不相信任何 eval”听起来激进，但背后的意思是，别相信和你工作无关的 eval。你的团队如果想引入 coding agent，应该拿自己真实的历史任务做测试。比如过去半年最典型的 bug 修复，最烦人的代码迁移，最常见的告警分诊，最容易出错的测试补齐。把这些任务整理成一组内部 benchmark。看模型不只是能不能给答案，还要看它的修改能不能被资深工程师接受，能不能通过现有 CI，能不能减少人工时间，能不能避免引入新风险。

这里有一个很实用的做法。你可以把评测分成短任务和长任务。短任务测试单点能力，比如理解一个函数、补一个边界测试、解释一个错误。长任务测试 horizon，比如让 agent 在一个隔离分支上完成一个迁移，从读代码到修改到自测到写总结。很多模型短任务都不错，差距会在长任务暴露。文章里 Claude Fable 五最突出的就是长任务续航，所以如果你只测十分钟内的补全，很可能测不出真正差异。

第二点，给 agent 提供工具，但不要以为有工具就够了。文章中 Fable 五的关键能力之一，是正确使用 Cognition 内部调试工具、浏览日志并得出结论。对我们自己的团队来说，如果想让 agent 做排障，就要让它能访问合适的观测数据。当然，这里要注意权限和安全，不能把所有生产权限一股脑打开。更现实的方式是给它只读日志、指标、trace，也就是链路追踪，和代码仓库的受控访问，让它能在沙箱或受限环境里分析。然后观察它是否能提出可验证假设，而不是只把日志重新复述一遍。

如果你的日志本身很混乱，字段不规范，错误信息没有上下文，人类排障都费劲，那 agent 也很难神奇变好。某种意义上，引入 agent 会倒逼工程基础设施升级。你需要更好的日志结构，更清晰的告警命名，更可靠的部署记录，更容易关联的请求标识。以前这些是给人看的，现在也是给 AI 看的。把系统做得可观察，就是让人和 agent 都更容易工作。

第三点，学会用 invariant 驱动任务。文章里 Fable 五在迁移任务前陈述不变量，这是非常值得借鉴的工作方式。你以后给 AI 分配复杂任务时，不妨先让它输出一份“必须保持不变的行为”和“我将如何验证”。比如你要重构鉴权模块，就让它先说哪些权限语义不能变，哪些老接口必须兼容，哪些错误码不能改，哪些审计日志必须保留。你要迁移数据库，就让它先列出数据一致性、回滚策略、灰度路径、异常处理。即便模型本身很强，这一步也能显著降低跑偏概率。

第四点，把“承认不知道”纳入评估标准。很多团队评估 AI 输出，只看答案是否正确。但在复杂任务里，你还应该看模型有没有区分事实和猜测，有没有说明证据强弱，有没有指出未知项，有没有建议下一步验证。文章里 Alberti 明确说，模型会说不知道，是重建信任的关键。你可以在内部评测里故意放一些信息不足的任务，看模型会不会瞎编。如果它总是在证据不足时给确定结论，那就不适合直接进入排障链路。

第五点，不要一上来就让 agent 触碰最危险的生产路径。文章说 Alberti 愿意让 Fable 五通宵工作，但这不等于它可以无审批自动改生产。Cognition 的场景里，Devin 是产品化的 AI 软件工程师，背后一定有他们自己的安全边界和工程流程，虽然原文没有展开。我们在自己团队落地时，建议从低风险、高价值、可回滚的任务开始。比如整理 issue、定位可疑提交、生成修复草案、补测试、做迁移影响分析、在分支上提交 patch。等它的稳定性被验证，再逐步扩大权限。千万不要因为一篇文章很振奋，就直接让 agent 在核心服务上自动合并和发布。

第六点，重新思考工程组织的工作分配。文章说 Devin 处理的是工程师总是来不及做的事情：代码库迁移、积压 bug、不断延期的功能。这个定位很现实。AI agent 未必一开始就替代最核心、最需要产品判断和系统设计的工作，但它很适合吞掉那些明确、繁琐、跨文件、需要耐心的任务。对很多团队来说，技术债不是没人知道，而是没人有整块时间处理。如果一个 agent 能在夜里持续推进百分之三十、百分之五十，哪怕第二天还需要人审查，也可能改变团队节奏。

你可以想象一个实际工作流。下午下班前，工程师把一个历史迁移任务描述清楚，补上不变量和验收标准，让 agent 在云端分支上跑。它夜里读代码、找调用点、做第一版修改、跑测试、记录失败和未解决问题。第二天早上，工程师不是从零开始，而是审查它的总结、看 diff、处理剩下的难点。即使不能完全合并，这也像多了一个初级到中级工程师帮你做前期勘探。文章里说“像团队里的真实工程师”，我个人认为，短期更准确的说法可能是“像一个能持续推进、但仍需要 code review 的工程队友”。

对 AI 从业者来说，这篇文章还有一个产品层面的启发。未来 agent 的竞争，不只是模型能力竞争，还包括运行环境、工具编排、评测体系和信任界面。Cognition 之所以能判断 Fable 五有用，是因为他们有 Devin 这个真实载体，有内部调试工具，有 Frontier Code，有高水平工程师 dogfooding。如果你只是把模型接到聊天框里，很难释放长 horizon 能力。长任务 agent 需要任务状态管理、权限控制、日志访问、分支隔离、测试反馈、进度汇报、失败恢复。模型是发动机，但车架、刹车、仪表盘也同样重要。

这也是为什么“在云端跑几小时”听起来简单，做起来很难。一个长时间运行的 agent 会遇到很多工程问题。上下文会变长，工具调用会失败，测试会超时，依赖会变，网络会抖，权限会过期。它需要知道什么时候重试，什么时候换策略，什么时候请求人类帮助。它还需要留下可审计轨迹，让人类知道它为什么这么做。否则第二天你看到一大堆改动，却不知道它的推理路径，这种结果很难被信任。

文章没有把这些工程细节全部展开，但从 Cognition 的产品方向可以看出，他们押注的是 agent session 从被动指令变成主动工作单元。这对企业软件开发流程影响很大。现在很多团队的工作流是，人类发现问题，创建 ticket，分配给人，人在代码库里查，写 patch，走 review。未来可能变成，agent 先发现异常，自动创建上下文包，提出候选 patch，人类主要负责判断、批准和处理复杂取舍。这里人类不是消失，而是从大量搜集信息和机械修改中上移到评审、设计和责任承担。

不过，我们也要警惕两个坑。第一个坑是把 agent 的进步误解成“可以减少工程纪律”。恰恰相反，越想让 agent 自主工作，越需要工程纪律。测试要更完整，CI 要更可靠，代码边界要更清楚，监控要更可读，权限要更细。没有这些，agent 工作八小时可能只是制造八小时的混乱。第二个坑是把“会主动”当成“应该随便主动”。如果一个 agent 总是在 Slack 里插话、开无意义 issue、报大量低置信度问题，工程师很快会把它静音。主动性的关键是高精度、低打扰、可解释。它必须知道什么时候该说，什么时候继续观察，什么时候把不确定性标清楚。

回到文章本身，最打动我的其实不是百分之三十的 Frontier Code 分数，也不是八小时这个数字，而是 Cognition 判断模型的方式。他们关注的是代码能不能留下，任务能不能持续，日志能不能读深，错误能不能承认，信任能不能恢复。这些都是软件工程里真正重要的东西。大模型发展到今天，单轮聪明已经不够了。我们需要的是稳定、耐心、可验证、能和工程流程咬合的聪明。

我个人认为，Claude Fable 五在这篇文章里的意义，可以概括为一句话：coding agent 正在从“短程助手”迈向“长程队友”。这个转变不会一夜之间完成，也不会自动适用于所有团队、所有代码库、所有生产环境。但如果你现在还只是把 AI 当成补全和问答工具，可能需要开始准备下一阶段了。准备自己的评测集，整理可交给 agent 的任务，改善日志和测试，把不变量写进工作流，并且设计好权限和审查边界。

最后总结一下。Anthropic 这篇文章借 Cognition 和 Devin 的经验说明，Claude Fable 五让长时间自主软件工程任务出现了一个明显跃迁，尤其体现在 horizon、混乱上下文排障、迁移任务守住不变量，以及愿意承认未知这些方面。它不是在告诉我们从此可以盲目信任 AI 写生产代码，而是在提醒我们，信任的边界正在向外扩展。谁能更早建立真实评测、工具环境和安全流程，谁就更可能把这种前沿模型能力转化成真正的工程生产力。
