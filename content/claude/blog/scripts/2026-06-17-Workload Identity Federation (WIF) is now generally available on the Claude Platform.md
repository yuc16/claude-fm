静态 API 密钥的三大痛点

在过去，如果你要让一个程序调用 Claude API，做法非常简单直接：在 Anthropic 的控制台里创建一个 API 密钥，然后把这个密钥以环境变量的方式注入到你的程序里，程序在每次调用 API 的时候带上这个密钥。这个流程你可以在十分钟之内完成，整个过程没有任何障碍。

但是，这种方式在工程上有三个根本性的问题，随着系统规模的增长，这三个问题会越来越严重。

第一个问题，是密钥泄露的风险。你可能会觉得，把密钥存在环境变量里不就行了吗？但实际工程中，密钥泄露的途径多得超乎想象。有工程师不小心把 `.env` 文件提交到了 GitHub 仓库，有人在 CI/CD 的日志里打印了带密钥的调试信息，有人在给同事演示的时候做了屏幕录制，结果密钥出现在了视频的某一帧里。还有一种情况更隐蔽：有人在 Dockerfile 里用 `ARG` 传入了密钥，结果这个值被固化进了镜像层，任何能拉到这个镜像的人都能提取出来。这些都是真实发生过的安全事故。

一旦密钥泄露，补救的过程叫做密钥轮换，英文是 key rotation。你需要立刻撤销旧密钥、生成新密钥，然后把新密钥手动更新到所有用到它的地方，包括各个服务的环境变量配置、CI/CD 系统的 Secrets、Kubernetes 的 Secret 对象、各个部署环境的配置文件。如果你的系统只有一两个地方用了这个密钥，轮换还好说。但如果是一个有几十个微服务、跑在多个云环境里的系统，每次轮换密钥都是一场运维噩梦，而且轮换期间还存在可用性的风险。

第二个问题，是权限颗粒度太粗。一个静态 API 密钥代表的是一个全局的访问权限：要么有，要么没有。你很难做到让某个特定的服务只能调用某些特定的 Claude API 接口，或者只能在特定的时间窗口里使用，或者只能消耗特定上限的 token。所有拿到这个密钥的程序，都拥有完全一样的权限。

这和安全工程里一个非常基本的原则相违背，这个原则叫做最小权限原则，英文是 principle of least privilege，意思是每个主体只应该拥有它完成工作所需的最小权限，不多也不少。用一个生活里的比喻来说：你不会因为让快递员送一次包裹，就把你家的主钥匙给他，你只会给他一次性的门禁码。静态 API 密钥更像是把主钥匙交出去，而 WIF 更像是按需颁发临时门禁码。

第三个问题，是审计困难。当你有十个不同的服务都在用同一个 API 密钥的时候，Claude 平台的日志里记录的是这个密钥发出的所有请求，但你根本分不清哪条请求是哪个服务发的。这个月 Claude API 的调用量突然多了百分之三十，是哪个服务导致的？某次安全事件里，是哪个服务在异常地调用 API？根本没法查。在需要进行合规审计的企业里，这是一个非常现实的障碍：你没有办法向审计员清晰地展示，公司内部有哪些系统在用 Claude、每个系统的使用情况是怎样的。

这三个问题加在一起，构成了静态 API 密钥模式的根本局限。WIF 就是为了从根源上解决这三个问题而设计的。

OIDC：工业级身份认证的通用语言

在理解 WIF 之前，你需要先了解一个基础概念，叫做 OIDC，全称是 OpenID Connect。你不需要记住这个缩写，你只需要理解它代表什么就行了。

你可以把 OIDC 想象成身份认证领域的一种「通用语言」。今天的云原生世界里，有很多不同的身份管理系统并行存在：AWS 有自己的 IAM，Google Cloud 有服务账号，Azure 有托管标识，GitHub 有给 Actions 工作流颁发的 token，企业内部可能还有 Okta 或者 Azure Active Directory 这样的身份提供商。这些系统本来是各自为政的，互相不认识。但 OIDC 提供了一套标准协议，让所有这些系统都能用统一的方式来表达「我是谁」，让接收方能够验证「你说的是真的吗」。

OIDC 的核心载体是一种叫做 JWT 的东西，全称是 JSON Web Token，可以翻译成「JSON 网络令牌」。JWT 是一段经过加密签名的数据结构，里面包含了一组声明，英文叫 claims。这些声明就像一张「身份证」上的字段：发行者是谁、令牌颁给谁、有效期到什么时候，以及一组自定义的身份属性。

举一个具体的例子。当你的代码在 GitHub Actions 里运行的时候，GitHub 会给这次工作流运行生成一个 OIDC token，这个 token 里包含这样一些声明：仓库的全名是什么，比如 `myorg/my-claude-app`；这次触发的工作流名字是什么；运行的分支是什么；目标的部署环境是 production 还是 staging；工作流里具体是哪个 job 在跑。这些信息组合在一起，就能唯一确定「这是我公司的哪个仓库、哪个 workflow、在哪个环境下触发的运行」。

关键是这个 token 的安全属性：它是有加密签名的，只有 GitHub 的私钥才能生成这个签名，任何人都可以用 GitHub 的公钥来验证这个 token 是不是真的，但没有人能够伪造它。而且它是短命的，通常只有几分钟到一小时的有效期，过期之后完全失效，就算你拿到了这个 token，也没什么机会滥用它。

AWS、GCP、Azure 等主流云平台也都提供类似的 OIDC token 机制，原理完全一样。WIF 正是以 OIDC 作为技术基础，把各个云平台和企业身份系统的身份，都接入到了 Claude 平台的认证体系里。

WIF 的完整工作流程

好，理解了 OIDC 之后，我们来看 WIF 到底是怎么工作的。这里有三个核心概念需要先理解：身份提供商、服务账号、联合规则。

先说服务账号，英文是 service account。这是 Anthropic 在这次更新里同时引入的一个新概念，是配合 WIF 一起推出的。以前，你调用 Claude API 用的是一个 API 密钥，这个密钥实际上代表的是你整个账号或者团队的访问权限，是一个共享的身份。现在，Claude 平台引入了服务账号这个层次，每一个工作负载，也就是每一个你部署的程序、服务或者 agent，都可以有自己独立的服务账号。这个服务账号有自己独立的权限范围，有自己独立的审计日志，有自己独立的配额和访问控制。

打个比方：以前你们整个开发团队共用一张公司信用卡，所有人的消费都记在这张卡上，月底账单来了你根本分不清哪笔是哪个部门花的。现在，公司给每个部门都发了独立的费用卡，每笔支出都清晰地归属到对应的卡号和部门名下，财务审计一目了然。服务账号就是这张「独立费用卡」。

再说联合规则，英文是 federation rule。联合规则是你在 Claude Console 里配置的一条映射关系，它回答的核心问题是：当某个外部身份出现的时候，应该把它映射到哪个服务账号上，以及需要满足什么条件才能完成这个映射。

比如，你可以配置这样一条联合规则：当请求的 OIDC token 是由 GitHub 颁发的，并且 token 里的仓库声明等于 `myorg/claude-production-service`，并且 environment 声明等于 `production` 的时候，将这个外部身份映射到 `prod-claude-worker` 这个服务账号。条件的精细程度完全由你来决定，你可以只匹配仓库名，也可以同时匹配仓库名、分支名、workflow 名、environment 名，做得非常精细。

理解了这两个概念之后，整个 WIF 的认证流程就清晰了。我来完整地走一遍。

你的工作负载需要调用 Claude API，它首先向自己所在的环境请求一个 OIDC token。如果是在 GitHub Actions 里，workflow 通过 GitHub 提供的 API 获取一个工作流身份 token；如果是在 AWS 上，Lambda 函数或者 EC2 实例通过本地的元数据服务获取它绑定的 IAM Role 的 OIDC token；如果是在 GCP 上，Cloud Run 实例通过 metadata server 获取绑定的服务账号的 token。

拿到这个 OIDC token 之后，你的程序向 Claude 平台发起一个认证交换请求，把这个 token 提交过去。Claude 平台收到请求后，先验证 token 的加密签名，确认它真的是由你之前在联合配置里注册的那个身份提供商签发的，而不是伪造的。签名验证通过之后，平台解析 token 里的 claims，然后把这些 claims 跟你配置的联合规则逐一对比，看看有没有哪条规则完全匹配。如果匹配到了某条规则，平台就根据这条规则，把这个外部身份映射到对应的服务账号，然后颁发一个短命的访问令牌，这个令牌的有效期通常只有几分钟到一个小时。最后，你的程序用这个短命令牌来调用 Claude API，完成实际的业务请求。整个过程，每一次认证交换和每一次 API 请求，都会被记录到对应服务账号的审计日志里。

你注意到了吗？在这整个流程里，有没有任何一个时刻，你需要持有一个来自 Anthropic 的、长期有效的凭证？没有。你用的是工作负载自己已经拥有的身份，这个身份是你整个基础设施体系里本来就有的，不需要额外创建和管理一套 Anthropic 的凭证。这就是 WIF 最核心的价值。

各平台的实际应用场景

讲到这里，你可能会想：我平时用的是什么环境？是否适合用 WIF？我来给你走几个最常见的场景，你可以对号入座。

先说 GitHub Actions，这是目前最普遍的 CI/CD 场景之一。很多团队已经在用 GitHub Actions 来做持续集成、自动测试，也有不少团队开始在流水线里直接调用 Claude 来做代码审查、自动生成测试用例、更新文档注释。以前你需要在 GitHub 仓库的 Settings 里进入 Secrets 管理，手动添加一个名为 `ANTHROPIC_API_KEY` 的 secret，然后在 workflow 文件里引用它。这个 secret 本身倒是加密存储的，但它是长期有效的，如果有人拿到了它的值，直到你手动撤销之前它都是可用的。

用 WIF 之后，你需要做的是：在你的 workflow YAML 文件里，给调用 Claude 的 job 加上 `permissions: id-token: write` 这个声明，让 GitHub 知道这个 job 需要获取 OIDC token。然后在 Claude Console 里配置好联合规则，声明当 token 来自 GitHub、仓库是你的仓库、environment 是你指定的环境时，映射到某个服务账号。配置完成之后，你的 workflow 就可以用 GitHub 自动颁发的身份 token 来认证 Claude 了。你不需要在 GitHub Secrets 里存任何 Anthropic 的东西，密钥泄露的风险从源头上就消除了。

再说 AWS 上的工作负载。你在 AWS 上跑了一个 Lambda 函数，或者是 EKS 上的一个 Kubernetes pod，或者是一个 EC2 实例，这些资源都可以绑定一个 AWS IAM Role。IAM Role 本身就是 AWS 原生的身份机制，它天然支持 OIDC token 的颁发。你只需要确保你的工作负载绑定了合适的 IAM Role，然后在 Claude Console 里配置对应的联合规则，就完成了。你的 Lambda 函数不需要从环境变量里读取任何 Anthropic 密钥，Kubernetes Secret 里也不需要存 Anthropic 的任何东西。

Google Cloud 的情况类似。Cloud Run 服务、Cloud Functions、GKE 上的 pod，这些资源都有 GCP 服务账号的概念，GCP 也天然支持 OIDC。配置方式和 AWS 完全一样，原理相同。

Azure 方面，Azure 的托管标识机制，也就是 Managed Identity，同样是 OIDC 兼容的。Azure 上的虚拟机、App Service、Azure Functions，只要绑定了托管标识，就可以通过 WIF 来认证 Claude。

最后说企业内部系统。如果你的公司用 Okta 或者 Azure Active Directory 来统一管理身份，你的内部服务或者 agent 可以通过这些企业级身份提供商来认证 Claude Platform，完全融入你已有的企业身份体系。这对于大型企业来说意义很大，因为它意味着 Claude 的接入可以直接纳入到现有的身份治理框架里，而不是另起炉灶。

原文里有一句话我觉得很值得引用，它说：不管你是跑 GitHub Actions 的两个人的初创公司，还是有精细凭证策略的大型企业，现在你都可以用跟你其他技术栈完全一样的方式来认证 Claude Platform。这句话说清楚了 WIF 的定位：它不是只给大企业准备的复杂方案，而是一个面向不同规模团队的通用解决方案，门槛足够低，同时能力足够强。

审计日志：不容忽视的另一面

我想专门聊一下审计日志，因为原文里只是一笔带过，但我认为这个能力在企业场景里的价值和身份认证本身一样大，值得单独展开讲。

当你为每个工作负载配置了独立的服务账号之后，Claude 平台的审计日志会做这样几件事：记录每一次认证交换的详情，包括外部 token 的来源和 claims；记录每一次 API 调用的请求参数和响应状态；把所有这些记录精确地归因到对应的服务账号。

这意味着你现在可以回答很多以前根本没法回答的问题。这个月，我的哪个服务对 Claude API 的调用量最大？哪个工作负载消耗了最多的 token，花费了多少？某个时间段里，有没有在非正常时间发起请求的异常模式？这个服务账号有没有在调用它本来不应该有权限访问的接口？如果有一天你需要进行安全事件的溯源，你有完整的审计链路可以追踪。

对于需要通过合规审查的企业来说，这个审计能力尤其关键。很多企业的信息安全部门，在批准一个新的外部 AI 服务接入的时候，必须要求提供完整的访问日志和审计记录。以前用共享 API 密钥的时候，你很难给出满意的答案。现在，每个工作负载有独立的服务账号，每条请求有清晰的归属，审计员想看什么就能看到什么。

联合规则里的细粒度权限范围，英文叫 fine-grained scopes，也是审计能力的一个组成部分。你可以给不同的服务账号授予不同的权限范围：有些服务账号只能调用对话接口，有些能调用所有接口，有些只能在特定的模型上运行。这种精细化的权限分配，配合详细的审计日志，让你的整个 AI 使用情况对企业来说是完全透明可见的。

如何开始迁移

好，讲完了原理，我们来说实操。对于已经在用 Claude API 的团队，怎么向 WIF 迁移？

原文里有一个非常重要的信息：API 密钥和 WIF 是可以同时并存的。这意味着你不需要设定一个「大迁移日」，在某一天把所有东西一次性切过去，而是可以一个工作负载一个工作负载地逐步迁移。这个设计降低了迁移的风险，让你可以先在非关键的工作负载上试水，验证配置正确之后再推进到更重要的服务。

实际的迁移步骤大概是这样的。第一步，打开 Claude Console，也就是 Anthropic 提供的管理控制台界面。根据你当前使用的身份提供商，选择对应的配置向导。控制台里有分步引导，会告诉你在你那一侧需要做什么配置，比如 GitHub Actions 需要添加哪个权限声明，AWS 需要确认 IAM Role 的哪些配置。

第二步，在控制台里创建一个服务账号，给它起一个描述性的名字，比如 `github-prod-deployer` 或者 `aws-claude-inference-worker`。名字尽量能反映这个账号对应的工作负载是什么，方便后续在审计日志里识别。

第三步，配置联合规则，把你的外部身份映射到这个服务账号。你需要提供身份提供商的 issuer URL，以及你要匹配的 token 声明条件。控制台会在你填写的时候进行验证，帮你发现配置错误。

第四步，控制台会在你完成配置之后给出一个测试命令，你可以直接在目标环境里运行这个命令，验证你的工作负载能否成功走完完整的认证流程，拿到访问令牌。这个测试步骤非常重要，在正式切换之前一定要做。

第五步，修改你的代码，把原来手动读取 API 密钥的方式，切换到用 Anthropic 官方 SDK 提供的 WIF 认证方式。原文里提到 WIF 已经集成进了 Anthropic 的第一方 SDK，以及 Claude Code 命令行工具里，所以代码层面的改动通常很小，很多情况下只需要修改认证初始化的几行代码。

第六步，在目标环境里部署更新后的代码，观察实际运行情况，确认认证正常、API 调用成功之后，你就完成了这个工作负载的迁移。接下来再逐个推进其他工作负载，最终撤销那些已经不再需要的静态 API 密钥。

对于规模比较大的组织，如果你同时管理着很多个工作负载，在控制台里一条一条地配置联合规则会很低效。Anthropic 为此提供了完整的 Admin API，让你可以通过程序来创建和更新身份提供商配置、服务账号、以及联合规则。这意味着你可以把 WIF 的配置写进 Terraform 或者你自己的运维自动化脚本里，和你的其他基础设施配置一起用代码来管理，实现真正的基础设施即代码。对于有几十上百个微服务需要集成 Claude 的大型平台团队来说，这个能力是非常关键的。

需要注意的几个坑

任何新技术都有学习曲线，我根据 WIF 这类系统的一般特点，给你提几个需要特别注意的地方。

第一个坑，声明匹配条件设置得太宽松。联合规则里的匹配条件，你需要想清楚要精细到什么程度。如果你只匹配了一个 GitHub 仓库名，那这个仓库里所有的 workflow、所有的分支、无论是 pull request 触发的还是手动触发的运行，全都会被映射到同一个服务账号。这意味着一个处于草稿状态的 feature branch 上的 workflow，和你的 production 部署 workflow，用的是同一套权限。这和最小权限原则是背道而驰的。

最佳实践是至少同时匹配仓库名和环境名，让你的 production 工作负载和 staging、testing 工作负载有不同的服务账号和不同的权限范围。如果你的工作负载有更精细的安全要求，还可以进一步匹配到具体的 workflow 名或者 job 名。

第二个坑，忘记处理 token 过期的情况。WIF 颁发的是短命令牌，有效期通常只有几十分钟。如果你的工作负载要运行很长时间，或者在一个长任务里需要多次调用 Claude API，你需要确保你的代码能在令牌过期前及时刷新。用 Anthropic 官方 SDK 的话，SDK 通常会自动处理 token 刷新，但如果你是直接构造 HTTP 请求来调用 API，就需要在代码里自己处理这个逻辑，否则长时间运行的任务会在令牌过期的时候突然失败。

第三个坑，联合规则的管理没有版本化。如果你是直接在 Claude Console 里手动点击配置联合规则，随着时间推移，服务越来越多，这些规则就会变成一种「谁都不知道当初为什么这样配」的状态，出了问题也很难排查。强烈建议把联合规则的配置通过 Admin API 写进你的 Terraform 或者其他基础设施即代码工具里，让它和你的其他基础设施配置一起走代码审查和版本控制。

第四个坑，迁移过渡期的管理。在迁移过程中，你可能同时有一部分工作负载用 WIF，另一部分还在用旧的静态 API 密钥。这个双轨并行的状态需要有明确的跟踪和计划，要知道哪些密钥还在使用、哪些已经可以撤销，否则很容易出现一批「僵尸密钥」，明明对应的工作负载早已迁移完成，但旧密钥还一直挂在那里，成了潜在的安全隐患。

第五个坑，注意身份提供商的 issuer URL 配置要准确。OIDC token 的验证依赖于 issuer URL，这个 URL 必须和你实际的身份提供商完全一致。比如 GitHub Actions 的 issuer URL 是一个固定的地址，AWS 的 issuer URL 会包含你的 AWS 账号 ID 和区域，GCP 的 issuer URL 也有特定的格式。任何一处配置错误，都会导致认证失败。Claude Console 里的配置向导会帮你填入正确的默认值，但如果你是通过 Admin API 程序化配置，一定要仔细核对这些值。

我的评论与展望

说了这么多原文内容，我个人来评论几句，接下来说的是我自己的看法，不代表 Anthropic 官方立场。

我认为 WIF 的正式发布，标志着 Anthropic 在企业化道路上迈出了非常实质性的一步。这不是一个功能上的小改进，而是一次安全基础设施层面的系统性升级。它发出的信号是：Anthropic 认真地在为企业级生产使用场景做准备，而不只是面向个人开发者和小团队。

从我自己做工程的角度来看，WIF 解决了一个一直以来困扰企业接入 AI API 的核心障碍：凭证管理的非标准化。以前，你的 AWS、GCP、GitHub，这些东西都接入了你企业统一的身份管理体系，但 Claude API 是一个孤岛，它有自己的密钥，不在你的 IAM 框架里，不在你的审计体系里，让安全团队很难接受。现在，你可以告诉安全团队：Claude API 的认证跟 AWS 是一样的方式，同一套身份体系，同一套审计日志。这句话，对于推动企业内部批准 Claude 接入，是非常有分量的论据。

我个人也觉得，这个方向是所有 AI 服务提供商都应该走的。目前，仍然有很多公司的 AI 集成处于「把密钥藏在环境变量里然后祈祷它不要泄露」的阶段。WIF 提供了一个真正符合现代安全工程标准的解决方案，值得行业跟进学习。

当然，也有一些地方我会继续观察。WIF 的配置对于不熟悉 OIDC 和云身份体系的开发者来说，还是有一定的概念门槛的。你需要理解什么是 OIDC token、什么是 claims、什么是 issuer URL，这些对于只是想快速接入 Claude API 的开发者来说，可能还是会觉得有些复杂。Claude Console 里的向导式配置是个好的起点，但文档的质量和社区的实践积累，会决定这个功能到底能多快普及到更多团队。

好了，今天这期就到这里。用一句话来总结这篇文章的核心价值：WIF 把 Claude API 的认证方式，从「你需要保管一串来自 Anthropic 的密钥」，升级为「用你已经拥有的云平台身份来说话」，对于在企业里构建生产级 AI 应用的工程师和架构师来说，这是一次真正意义上的安全基础设施升级，值得认真对待，也值得现在就开始评估迁移的时间表。我是你的主播，我们下期再见。
