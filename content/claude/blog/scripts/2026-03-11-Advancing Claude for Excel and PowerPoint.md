大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是 Advancing Claude for Excel and PowerPoint,翻译过来就是"为 Claude 的 Excel 和 PowerPoint 体验带来全面升级"。这篇文章发表于二〇二六年三月十一日,出自 Anthropic 官方博客。算下来距离今天,也就是二〇二六年六月二十日,大概过去了三个多月,时间不算长,所以这篇文章里描述的功能基本上现在都还在持续生效和迭代当中,值得我们认真聊一聊。

先说说为什么这篇文章值得花时间听。如果你是做财务分析、投行、咨询、运营,或者任何需要每天在 Excel 表格和 PowerPoint 幻灯片之间来回切换的工作,你大概都有过这种体验,就是先在表格里把数据理清楚,算出几个关键指标,然后再跑到幻灯片里,手动把这些数字搬过去,排版,配色,再写一遍说明文字。这个过程本身没有什么技术难度,但特别琐碎,而且最烦的是,如果你用一个 AI 助手分别在两个软件里帮你干活,你常常得把同样的背景信息和需求,在 Excel 里讲一遍,跳到 PowerPoint 里再讲一遍。这篇文章要解决的核心问题,就是打通这种割裂感。

我们先讲讲背景。在这次更新之前,Claude for Excel 和 Claude for PowerPoint 已经存在了一段时间,作为插件分别嵌入在这两个微软办公软件里,可以帮你写公式、整理数据、生成幻灯片内容。但问题是,这两个插件原来是相对独立的,你在 Excel 里跟 Claude 聊的上下文,Claude 并不知道你同时还打开了一个 PowerPoint 文件,反过来也是一样。这就好像你雇了两个助理,一个专门管表格,一个专门管幻灯片,他们俩各自都很能干,但互相之间不通气,你每次想让他们配合干一件事,都得自己当中间人,把信息从一个人嘴里听完,再转述给另一个人。这种割裂在简单任务里不算什么大问题,但一旦涉及稍微复杂一点的、跨多个文件的工作流,比如说你要做一份完整的交易分析,从原始数据整理到估值建模,再到把结果做成给管理层看的演示文稿,这种沟通成本就会迅速累积,变得很烦人。

行业当时的状态是什么样呢。其实从更大的视角看,二〇二六年初这个阶段,各家大模型公司都在往"办公场景里的智能体"这个方向猛推。微软自己有三六五 Copilot,谷歌有 Gemini 在 Workspace 里的整合,Anthropic 这边则选择和微软合作,把 Claude 直接做成 Excel 和 PowerPoint 的官方插件。大家都意识到,模型能力本身已经足够强了,关键的竞争点已经从"模型答得准不准"转移到了"这个模型能不能真正嵌入到用户每天的实际工作流程里,减少摩擦"。这篇文章其实就是这个大背景下的一个具体落地动作,Anthropic 想说的是,我们不仅要让 Claude 聪明,还要让 Claude 在 Excel 和 PowerPoint 这两个最常用的生产力工具里,变得像一个真正了解你整体工作的助理,而不是两个互不相干的功能模块。

好,接下来我们挑文章里最重要的几个点,逐一拆开讲。

第一个核心点,也是这篇文章的标题党重点,就是跨文件的上下文共享。具体说的是什么呢,从现在开始,Claude for Excel 和 Claude for PowerPoint 可以共享同一段对话里,所有打开文件的完整上下文。也就是说,不管你同时打开了几个 Excel 工作簿,还是同时开着一个 PowerPoint 演示文稿,Claude 在任何一个应用里采取的动作,都是基于它对整体情况的了解,而不是孤立地只看眼前这一个文件。

文章里举了一个很具体的例子,我觉得特别能说明问题。假设你是一名财务分析师,你打开了一个工作簿,里面有几家可比公司的财务数据,还结合了其他数据源。这时候你可以让 Claude 直接从这个已经打开的工作簿里,把可比公司的财务数字拉出来,在 Excel 里搭建一张交易可比公司分析表,也就是我们常说的 trading comps 表。做完这一步之后,你不需要重新解释一遍这些数据是什么意思,直接跟 Claude 说,把这张估值汇总表搬到我的推介材料里去,Claude 就可以直接跳到 PowerPoint 文件里,把刚才算出来的估值结果放进幻灯片。再往下,你甚至可以让它帮你起草一封发给董事总经理的邮件,说明这次分析的结论。整个过程里,你不需要来回切换标签页,也不需要在每一步重新跟它解释一遍这批数据集到底是什么。

这个设计听起来简单,但我觉得它解决的其实是一个很本质的问题,就是减少"重新解释"这个动作本身的成本。我们平时跟人协作的时候也是这样的道理,一个真正靠谱的同事或者助理,是那种你跟他讲过一次背景,他就会一直记着,后面的每一步配合都建立在这个共同认知上,而不是每换一个场合就要你再讲一遍。文章里专门提到一句话,说减少工具之间的来回折腾,是更快做出最终交付物的关键所在。我觉得这句话说得挺到位的,因为很多时候我们在用 AI 工具的时候,真正拖慢效率的不是模型算得慢,而是人和工具之间、工具和工具之间的沟通损耗。这次升级直接针对的就是这个损耗本身。

第二个核心点,是技能这个功能正式进驻 Excel 和 PowerPoint 插件。这里的技能,英文是 skills,你可以理解成把一整套完整的工作流程打包成一个可以一键复用的操作模板。文章里给了一个很贴切的类比,说团队里总有人会摸索出某种"最佳做法",比如怎么做一次变动分析,或者怎么用公司自己的模板来写一份给客户看的演示文稿。在以前,这种经验大概只能靠这个人口头传授,或者写一份操作手册放在某个共享文件夹里,等着别人有空的时候去看。现在有了技能这个机制,这个人只需要把这套流程保存成一个技能,以后团队里任何人,只要在 Excel 或者 PowerPoint 里调用这个技能,就能瞬间复现同样的高质量流程,不需要再重新学一遍怎么做。

这个设计我觉得很巧妙的地方在于,它把个人经验变成了组织资产。本来一个分析师摸索出来的高效做法,只存在于他自己的脑子里和操作习惯里,现在变成了一个可以被沉淀、被共享、被持续优化的东西。这跟软件工程里我们说的"把隐性知识显性化"是同一个思路,只是这次落地在了财务和办公场景里。

文章里还提到,Anthropic 这次预先准备好了一批起始技能,直接覆盖了最常见的使用场景,不需要用户自己从零摸索。我们先看 Excel 这边的起始技能都包括什么。第一个是审计模型,也就是检查一个财务模型里有没有公式错误,资产负债表是不是平衡,这个对做财务建模的人来说是个特别头疼但又必须做的工作,因为一个隐藏的公式错误可能会让整个估值结果都跑偏。第二个是搭建并填充各类模型模板,文章里特别提到了三种,一种是杠杆收购模型,也就是 LBO,一种是现金流折现模型,也就是 DCF,还有一种是三表联动模型,也就是把利润表、资产负债表和现金流量表整合在一起的模型。这三种模型基本上是投行和私募股权分析师每天都要打交道的标配工具。第三个起始技能是跑可比公司分析,前面我们提到的那个例子里用到的就是这个能力。第四个是清理混乱的表格数据,可以是某个特定区域,也可以是整张工作表,甚至是整个文件,这个对处理来源不规范、格式乱七八糟的原始数据特别有用。

再看 PowerPoint 这边的起始技能,这些技能更偏向分析完成之后的呈现工作。第一个是搭建竞争格局分析的演示文稿,包括市场定位图和对每个竞争对手的深度剖析。第二个是更新一份已经存在的演示文稿,把新的信息或者额外的数据补进去,而不是从头重做一份。第三个是审阅一份投行风格的演示文稿,具体检查的内容包括数字前后是否一致,叙事和数据是不是对得上,以及语言表达是否需要打磨。这第三项我觉得特别有意思,因为很多时候一份对外的演示文稿里最容易出问题的地方,恰恰就是不同页面之间数字对不上,或者文字描述和图表数据脱节,这种问题人工检查起来特别累,容易看走眼,用一个专门的技能来做一致性核查,其实是一个很实际的应用场景。

文章里还提到一个细节,我觉得对正在用 Claude 桌面版或者网页版的听众特别有用,那就是你已经在 Claude 里设置好的技能,无论是个人的还是整个组织共享的,直接就能在 Excel 和 PowerPoint 插件里使用,不需要额外的安装步骤,这个机制跟 MCP 连接器的工作方式是一样的。也就是说,如果你的团队已经在 Claude 里沉淀了一些自己的工作流技能,这次升级不需要你重新搭建一套,直接就能继承过来用。另外,这些针对 Excel 和 PowerPoint 的起始技能,也打包在一个叫做财务分析插件的东西里,这个插件会自动安装到两个插件里,以后插件里新增的技能,也会自动同步过来,不需要你手动维护。

第三个核心点,是指令这个功能。前面我们讲的技能,处理的是一整套具体的工作流程,而指令处理的是更细粒度、但是持续生效的偏好设置。文章里举的例子包括,始终使用公司规定的数字格式来呈现 Excel 里的数据,或者要求 PowerPoint 里的每条要点都控制在一行之内,再或者是自动标记出那些引用了硬编码假设数字的单元格。这些规则你只需要设置一次,Claude 之后每次操作都会自动遵守,不需要你每次都重新提一遍。文章里还提到一个很贴心的细节,就是 Claude 本身也可以帮你写和编辑这些指令,也就是说,如果你不太确定怎么把一个模糊的偏好,转化成 Claude 能理解的明确指令,你可以直接让 Claude 帮你起草。

我觉得技能和指令这两个功能放在一起看,其实是一种很合理的分层设计。技能解决的是"怎么把一个复杂的、多步骤的任务变成一键操作"这个问题,指令解决的是"怎么让 Claude 长期记住我的一些固定偏好,不用我每次都重复"这个问题。这两者结合起来,基本上覆盖了日常办公里绝大部分重复性的沟通成本。

接下来我们再讲讲第四个值得关注的点,就是企业部署和合规这块的进展。文章里提到,Claude for Excel 和 PowerPoint 现在可以通过三大主流云平台来接入,分别是亚马逊的 Bedrock,谷歌云的 Vertex AI,还有微软的 Foundry。对于很多大型企业,尤其是金融机构来说,数据合规和安全审查是一个绕不开的门槛,很多公司已经在这几个云平台上搭建了自己的合规体系和数据治理流程,如果一个新工具想要进入这些企业,最现实的路径往往不是让企业重新搭一套合规框架,而是直接接入企业已经在用的现有基础设施。文章里用的说法是,这样可以让 Claude 在客户原本的合规体系所在的地方,直接和客户见面,这句话我觉得讲得挺直白的,本质上就是降低企业采用的门槛。除了走这几个云平台,企业也可以选择把流量通过现有的大模型网关路由过去,接到跑在这些云平台上的 Claude 模型。

还有一个挺值得一提的点,文章里说 Claude 现在也原生为微软三六五 Copilot 里的 Agent Mode 提供动力,这样一来,使用微软三六五 Copilot 的客户,可以在 Excel 里让 Claude 和 Copilot 并肩工作,一起构建、编辑、分析工作簿。这个细节其实透露出 Anthropic 和微软之间合作的深度,不只是做了一个独立的插件,还把自己的模型能力嵌入到了微软自己的 Agent Mode 产品里面去,某种程度上,这意味着即使你的公司主要用的是微软自家的 Copilot 生态,你也有机会用上 Claude 的能力,而不需要单独再装一个插件。

文章最后还提了一句,说这次升级里,跨应用的上下文共享和技能功能,目前在 Mac 和 Windows 平台上,所有付费计划的用户都可以以测试版的形式使用。另外补充说明一下,文章正文之外还有一条更新备注,说 Claude for Word 的测试版现在也在团队版和企业版计划里上线了,这条更新是在二〇二六年四月十号追加的,说明这套生态还在持续往 Word 这个方向扩展。

好,讲完了这几个核心点,我们再说说这些东西对大家实际工作意味着什么,具体能怎么上手。

如果你本身的工作里经常需要同时操作 Excel 和 PowerPoint,比如说你是做财务分析、投行、咨询、市场研究、运营汇报这类工作的,我觉得第一件值得尝试的事情,就是去体验一下这次的跨文件上下文功能。具体做法是,你可以打开一个有数据的 Excel 工作簿,再打开一个正在准备的 PowerPoint 演示文稿,在同一个 Claude 对话里,先让它从表格里提取或者计算出某些结果,然后直接让它把这些结果搬到幻灯片里去,看看这个跳转衔接得有多顺畸。如果你之前已经习惯了分别在两个软件里单独跟 Claude 对话,这次可以试着换一种用法,直接把整个分析到呈现的流程,在一个连续对话里走完,感受一下省了哪些重复解释的步骤。

第二件值得尝试的事情,是去看看官方预置的那些起始技能,具体看看哪些跟你自己的日常工作高度重合。如果你正好做的是金融建模相关的工作,比如说经常要搭建 DCF 模型或者跑可比公司分析,这些起始技能基本上是直接可以拿来用的,不需要你自己再去摸索怎么写提示词。如果你做的不是金融分析,但是你的团队里有某个固定的、反复出现的工作流程,比如每周要整理一份固定格式的运营报表,或者每个月要更新一份固定模板的客户汇报材料,我建议你可以试着把这个流程沉淀成一个自定义技能,这样团队里其他人以后遇到同样的任务,就不需要从头摸索怎么跟 Claude 描述需求了。

第三件事情,是认真考虑一下指令功能,把那些你每次都要重复提醒 Claude 的小要求,直接写成永久生效的指令。比如说你们公司对数字的小数位数有统一规定,或者幻灯片里的字体大小和排版有固定标准,这些细节如果每次都靠你临时在对话里提一句,既容易遗漏,也很啰嗦。把它写成指令之后,相当于一次性把这些规矩固化下来,以后就不用操心了。

需要提醒大家注意的一个坑,是这次升级里提到的功能目前还是以测试版的形式提供,文章里用的词是 beta,虽然功能已经面向所有付费计划用户开放,但测试版这个状态意味着可能还存在一些不稳定的地方,或者某些边缘场景下的表现还没有完全打磨好。所以如果你打算把这套工作流用在特别重要、容不得出错的正式交付物上,比如说要直接发给客户或者管理层的最终版本,我建议还是要养成人工复核的习惯,尤其是涉及到具体数字和公式逻辑的部分,毕竟财务分析这种场景,一个数字算错可能造成的后果是很实际的。另外,如果你所在的公司对数据合规有比较严格的要求,在正式启用这些功能之前,最好先确认一下公司是怎么接入 Claude 的,是直接用 Claude 账号,还是通过亚马逊 Bedrock、谷歌云 Vertex AI、或者微软 Foundry 这几条合规路径接入的,这会影响到你的数据具体是怎么流转和存储的。

好,我们最后来做一个总结。这篇文章讲的核心事情其实可以归结成两句话,第一句是,Claude 在 Excel 和 PowerPoint 之间打通了上下文,让你不再需要在两个软件里反复解释同一份数据和需求。第二句是,技能这个机制把团队里最佳实践,从个人经验变成了可以一键复用的组织资产,再配合指令功能,把那些琐碎但持续存在的偏好设置也自动化掉了。

我个人觉得这次更新最值得关注的地方,不是某一项具体的技术突破,而是这种"把 AI 助手嵌入到现有工作流缝隙里"的产品思路。很多时候我们评价一个 AI 工具好不好用,关注的都是它单次回答得准不准、聪明不聪明,但这篇文章其实在提醒我们,真正决定一个工具能不能被大规模采用的,往往是它有没有解决那些看起来很小、但天天发生的摩擦点,比如说要不要重复解释背景信息,要不要每次都手动调整格式。这次 Excel 和 PowerPoint 之间的上下文打通,以及技能和指令这两个机制,本质上都是在啃这些细碎但高频的摩擦点。我觉得这个方向,对所有做生产力工具的团队来说,都是一个值得参考的思路,真正的护城河,有时候恰恰就藏在这些不起眼的衔接细节里。
