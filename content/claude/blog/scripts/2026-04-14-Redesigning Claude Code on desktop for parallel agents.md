## Shownotes

Anthropic 在二零二六年四月对 Claude Code 桌面应用进行了一次从底层开始的全面重设计。这次改版的核心逻辑只有一句话：开发者的角色正在从"写代码的人"转变为"调度 AI agent 的人"，而原来的工具跟不上这个变化了。本期节目逐层拆解这次改版的七项核心变化，并给出并行 agent 工作方式的实践建议。

好，欢迎来到 Claude FM，我是你的主播。今天我们来聊一篇 Anthropic 官方发布的文章，英文标题叫 Redesigning Claude Code on desktop for parallel agents，我把它翻译成中文是"为并行 Agent 时代重新设计 Claude Code 桌面应用"。这篇文章发表于二零二六年四月十四日，距离我们录制这期节目大概过去了两个多月。

两个月在 AI 领域里已经算一段不短的时间，但这篇文章讲的内容我觉得非常值得专门拿出来聊，因为它不只是在介绍一个软件更新，它背后折射出的是整个 AI 辅助编程的范式正在发生一次根本性的转变。如果你是每天在用 Claude Code 的工程师，或者你在关注 agent 编程这个方向，今天这期节目你一定要听完。

我们先来谈谈背景，这样你才能理解这次改版究竟在解决什么问题。

你回想一下，一年多以前你是怎么用 AI 编程工具的。大部分人的模式是这样的：遇到一个问题，打开 ChatGPT 或者 Claude，把代码粘进去，问"这段逻辑怎么优化"或者"帮我找一下这里的 bug"，然后等它给出答案，把答案复制回来，改一改，测试一下，再继续下一个问题。这是一种非常线性的工作方式，人在中间充当了一个传话筒的角色，AI 只是一个更聪明的搜索引擎。

但是 agent 这个概念出来之后，事情开始变得很不一样了。agent，也就是智能体，它不只是回答一个问题，它可以自主地执行一系列连续操作：读取文件、分析代码、做出修改、运行测试、检查测试结果、提交代码、创建 pull request，也就是代码审查合并请求。你只需要告诉它目标，它自己去规划和执行整个流程。

Claude Code 就是 Anthropic 做的这样一个 agent 编程工具。你可以在终端里，也就是命令行界面里启动它，告诉它"帮我把认证模块重构一下，跑完测试，没问题就提个 PR"，然后它就真的会去做，一步一步执行，直到任务完成或者遇到需要你决策的关键节点。

现在问题来了。当 agent 能够自主执行任务之后，你作为开发者的角色就变了。你不再是一个低头干活的程序员，你开始变成一个管理者、一个调度员。你可以同时跑好几个 agent：一个在帮你重构一个旧模块，另一个在帮你给某个函数写单元测试，第三个在另一个代码仓库里修一个线上 bug。你的工作变成了：分配任务、检查进度、在它偏离方向的时候把它拉回来、最后审查结果再决定要不要发布。

这就是 Anthropic 在文章开头说的那段话，我来翻译一下：对于很多开发者来说，agent 工作的形态已经改变了。你不再是输入一个 prompt，然后等待。你会同时启动好几件事：一个代码仓库里的重构，另一个里的 bug 修复，再加上一个专门写测试的任务。你要在结果返回的时候挨个检查，发现跑偏了就纠正，最后审查代码变更，再决定是否发布。prompt 就是你给 agent 的指令文本。

这就是 Anthropic 重新设计桌面应用的出发点。之前的 Claude Code 主要是一个终端工具，用命令行启动，一次管一个任务，界面很简陋，功能也单一。但现在用户的实际工作场景是多任务并行的，原来的界面根本撑不住，就像你用一个记事本去管理一个十人团队的项目，格格不入。所以他们做了一次从底层开始的全面重设计。

好，我们来逐层拆解这次改版到底做了哪些核心变化，我挑出最重要的几点讲深讲透。

第一个核心变化，是全新的侧边栏和多会话管理系统。

什么叫会话？你每次启动 Claude Code 开始一个任务，就是一个会话，英文叫 session。之前你如果同时跑三个任务，就等于同时开了三个终端窗口，三个命令行，在它们之间来回切换，完全不知道哪个任务跑到哪一步了，也没有一个地方能让你一眼看到所有任务的状态。这个体验非常割裂，用一个夸张的比喻，就像你要同时管理三个项目，但每个项目的文件都散落在不同的房间里，你只能在房间之间来回跑。

新版本加入了一个侧边栏，把所有正在运行和最近运行过的会话集中展示在同一个地方。你可以一眼看到：我现在有五个任务在跑，这两个已经完成了，那个还在进行中，这个似乎出错了需要我介入。

更聪明的是，这个侧边栏可以按多个维度过滤。你可以按状态过滤，比如只看正在运行的任务；可以按项目过滤，比如只看属于某一个特定代码仓库的会话；可以按运行环境过滤，也就是区分本地运行还是云端运行的任务。你还可以把侧边栏按项目分组，这样如果你同时在维护三个不同的产品线，每个产品线下面有好几个正在跑的任务，层级结构一目了然，不会混在一起。

还有一个细节我觉得设计得特别好：当一个会话对应的 pull request 被合并或者关闭之后，这个会话会自动归档，从活跃列表里消失，让侧边栏始终只展示那些还在进行中的任务。这种自动清理的机制非常符合实际工程场景——想象一下，如果你一天要跑几十个 agent 任务，侧边栏如果不自动清理，很快就会乱成一团，根本找不到自己需要关注的那个。

这个设计其实和那些项目管理工具里的看板视图非常像，比如 Jira 或者 Linear。在那些工具里，你可以看到每一张任务卡片的状态，完成的就归档掉，进行中的留在看板上。现在 Claude Code 把这个概念搬到了 agent 任务管理上。我个人觉得这是一个非常自然的演进，因为你管理的不再只是代码，而是工作本身。

第二个核心变化，是 side chat，也就是旁路对话功能。

这个功能解决的是一个很实际的工程问题。你正在跑一个很复杂的 agent 任务，比如让它重构一个几千行的核心模块。跑到中途，你突然想问一个技术问题，或者你想临时问 Claude 某两种方案哪个更适合当前场景，但是你不想把这段对话混进主任务的上下文里。

为什么不能混进去？这涉及 agent 的工作原理。对于 agent 来说，上下文就像是它的工作记忆，它每次做决策都会参考上下文里的所有信息。你往里面塞什么，它就会根据那个信息来调整行为。如果你在任务中途插入一段无关的技术问答，很可能会让 agent 的后续行为变得不稳定，甚至朝着错误的方向走。这不是理论上的担忧，在实际使用中是真实存在的问题。

Side chat 就是给你开了一个安全的旁路。你按下 Command 加分号，或者在 Windows 和 Linux 上按 Ctrl 加分号这个快捷键，就会弹出一个侧边的对话窗口。这个窗口会从主任务的上下文里读取信息，所以它知道主任务在做什么，可以给出有针对性的回答，但是这段对话的内容不会写回主任务的上下文，不会影响主任务的走向。

举个具体例子。你让 Claude Code 帮你重构一个用户认证模块，它已经跑了一半了。这时候你突然想问：对于这个业务场景，用 JWT 还是传统的 session 更合适？你用 side chat 问，Claude 会结合主任务的背景，考虑到你现在的代码结构，给你一个有针对性的分析，但这段问答不会变成主任务执行上下文的一部分，不会干扰它接下来的重构决策。完美分离。

我个人觉得这个功能在实际使用中会非常高频。当你同时跑多个任务的时候，你脑子里需要维持一个全局视图，不断地有新问题涌现，不断地需要做决策。Side chat 就是你安全地探索这些想法的地方，不用担心把正在运行的任务搞乱。

第三个核心变化，是一套完整的集成工具箱。

原来你用 Claude Code 的时候，它在终端里跑，但是你看代码要开代码编辑器，你运行测试要另开一个终端，你看代码变更要开 diff 工具，你预览 HTML 页面要开浏览器。你不停地在五六个应用之间切换，上下文切换的心智成本非常高。每次切换都会打断你的思路，你需要重新定位自己在哪里、刚才在做什么。

新版本的桌面应用把这些最常用的工具全部集成进来了。

首先是集成终端。你可以在 Claude Code 的窗口里直接运行命令，不用再跑去另一个终端窗口。比如 agent 帮你写完了一段新功能，你可以立刻在旁边的终端里跑 npm test，或者 pytest，看测试通不通过，整个流程都在一个窗口里流畅完成，没有任何跳转。

然后是内置的文件编辑器。这个功能允许你在 Claude Code 里直接打开文件、做手动修改然后保存。这非常适合一种常见场景：agent 生成了一段代码，大体上是对的，但有一两个小地方你想自己改一下。以前你得关掉 Claude Code 或者切到 VS Code 改完再回来，现在直接在里面改就好了，改完立刻继续让 agent 接着跑。

接下来是升级后的 diff 查看器，也就是代码变更对比工具。文章里特别强调，新的 diff 查看器是专门针对大规模变更做了性能优化的。这个优化非常重要，因为 agent 一次性修改几十个文件是非常普遍的事情，原来的 diff 查看器在这种大体量变更下会非常卡顿，新版本解决了这个性能瓶颈，让你在审查 agent 的大规模改动时不会因为界面卡顿而崩溃。

还有扩展了的预览面板。以前预览面板只能展示跑在本地 app server 上的网页应用，也就是你可以预览一个本地开发服务器的输出。现在加入了对 HTML 文件和 PDF 文件的直接预览支持。这对在做文档生成、报告生成、或者前端界面开发的同学会很有价值，agent 生成了一个 HTML 报告，你直接在 Claude Code 里预览，不需要开浏览器。

你想一想，这些工具集成在一起意味着什么。它意味着 Claude Code 正在从一个命令行工具演变成一个真正意义上的完整开发工作站。管理任务、查看代码、运行测试、预览结果，全部在一个地方完成。理论上你可以整个工作日都待在这一个窗口里，极大地减少工具切换带来的摩擦。

第四个核心变化，是完全可以拖拽重排的布局系统。

这个功能的逻辑简单但很强大：所有的面板，包括终端、预览窗口、diff 查看器和对话界面，都可以通过拖拽来重新排列，形成任意的网格布局。你可以把终端拖到左上角，把预览窗口放右边，把对话框缩小放底部，随便你怎么组合。

为什么这个重要？因为不同的任务类型，你需要的工作区布局是不一样的。如果你在做前端开发，你可能希望左边是 Claude 的对话，右边上方是网页预览，右边下方是终端，看着页面实时变化的感觉非常直观。如果你在做大规模代码审查，你可能希望上面铺满 diff 查看器，下面是对话区，这样你看代码变更的时候视野更开阔。如果你在调试一个非常复杂的系统，你可能需要同时看多个文件，那就把文件编辑器拉大，把对话窗口缩小。

这种灵活性背后体现的是 Anthropic 对用户工作多样性的认可。不同的工程师有不同的工作习惯，同一个工程师在不同任务场景下也有截然不同的需求，一个固定的布局永远无法满足所有人。我把这个功能类比成双屏工作站的出现：双屏出现之前，所有工具挤在一个屏幕上，应用之间来回切换；双屏出现之后，你可以把不同工具分布在不同位置，效率大幅提升。可拖拽的面板布局做的是同一件事，只不过是在单个窗口内部实现了这种空间上的灵活性。

第五个核心变化，是三种视图模式和完善的键盘快捷键体系。

文章里介绍了三种视图模式：Verbose、Normal 和 Summary。Verbose 是详细模式，你可以看到 Claude 调用每一个工具的完整过程，它读了什么文件、执行了什么命令、得到了什么输出，每一步都完全透明。这个模式适合那些想深度了解 agent 在底层做了什么的开发者，或者当你在调试某个行为异常的时候需要逐步排查原因。Normal 是普通模式，展示主要的操作节点和结果，过滤掉很多中间细节，平时日常使用最合适。Summary 是摘要模式，只展示任务的最终结论，完全不展示中间过程，适合那种你只想知道任务完没完成、结果是什么，完全不关心具体细节的场景。

这三种模式的设计非常体贴，因为用户对 agent 的信任程度和使用熟练度是不一样的。刚开始用 agent 的工程师，通常希望看到每一步在做什么，这样才有安全感，才能及时发现和纠正问题。用久了、建立了信任之后，你知道它大概会怎么做，不需要看那么多细节，Summary 模式效率更高，让你专注在结果上。

键盘快捷键方面，新版本补全了很多常用操作的快捷键，包括会话切换、新建会话、界面导航等等。按下 Command 加斜杠，或者在其他系统上按 Ctrl 加斜杠，可以弹出全部快捷键列表。对于喜欢键盘流操作的开发者来说这是个好消息，你不需要在 agent 跑着的时候频繁拿鼠标点来点去。另外还加入了一个使用量显示按钮，可以同时看到当前对话消耗了多少上下文窗口，以及本次会话总共用了多少 token，也就是处理的文本单元数量。这对于需要控制成本的团队场景会很有用。

第六个变化，是 SSH 支持扩展至 Mac 以及更完善的云端运行能力。

SSH 是一种远程连接服务器的安全协议。Claude Code 可以在本地机器上运行，也可以通过 SSH 连接到远程机器上运行。以前 SSH 支持只有 Linux 系统，现在 Mac 系统也支持了。也就是说，不管你的远程服务器或者远程开发机器跑的是 Linux 还是 Mac，你都可以把 Claude Code 的会话指向那台远程机器，让 agent 在上面直接操作。

这对很多公司场景很有价值。比如你的代码必须在特定的服务器环境里运行，或者你需要访问公司内网的数据库，或者你在用云端的 GPU 机器跑计算密集型的任务，这些场景都可以通过 SSH 来连接，然后让 Claude Code 在那个远程环境里工作，不受本地机器的环境限制。

第七个变化，是桌面应用和命令行插件的完全对等支持。

Claude Code 支持插件系统，允许用户或者企业组织安装各种扩展功能。之前这些插件只在命令行版本里工作，桌面应用里用不了。现在两边做到了完全对等：无论是你自己在本地安装的插件，还是公司统一管理的企业级插件，在桌面应用里的行为和在命令行里完全一致。

这个对等支持对企业用户特别重要。很多企业会在 Claude Code 的插件里集成自己的内部工具，比如公司特定的代码风格检查规则、安全扫描工具、或者和内部部署系统的接口。如果桌面应用不支持这些插件，开发者就只能用命令行版本，而命令行版本缺少我们前面讲的那些图形化管理功能。现在两边彻底打通，企业用户可以同时享受到图形界面的效率优势和插件生态的扩展能力，不需要在两种模式之间做取舍。

好，聊完这七个核心变化，我们来说说这次改版对你实际工作意味着什么，以及怎么用、要避什么坑。

如果你现在还在用单任务、线性的方式使用 Claude Code，也就是一次只跑一个任务，跑完了再跑下一个，这次改版对你的直接影响可能没那么大。但我强烈建议你认真考虑一下转向并行工作方式，因为这是效率提升幅度最大的地方。

具体怎么开始？你可以尝试这样的工作节奏：早上来了，先把今天要处理的几个任务想清楚，在脑子里做一个简单的拆分，然后把相对独立的任务分别启动成不同的 agent 会话。比如让一个 agent 去修那个昨天复现了的 bug，让另一个 agent 去补充那个模块缺少的测试覆盖，让第三个 agent 去读一读最近新加的那段代码，生成一份文档说明。然后你去开你的晨会，或者去处理其他不需要深度专注的工作。回来的时候三个任务都有进展了，你挨个扫一眼侧边栏，哪个有问题就纠正，哪个完成了就审查 diff 提 PR，然后继续推进。这种工作方式的效率，比串行地一个一个等待会高出不止一倍。

这种工作方式的核心是你得学会"委托"。很多工程师，包括我认识的不少非常资深的工程师，对 agent 还有一种控制欲，总想自己掌控每一行代码，看到 agent 改了一行它觉得不太完美的代码就想立刻介入纠正。但并行 agent 工作的前提是你愿意真正地把任务委托出去，只在关键节点介入审查，而不是每一步都盯着。如果你每个 agent 都要盯着每一步，那你同时跑三个的效率还不如老老实实一个一个跑。

Side chat 功能对学习委托很有帮助。你可以一边让 agent 跑，一边用 side chat 问 Claude 这个任务的进展情况、有没有什么潜在风险、下一步应该关注什么。这样你始终保持对大局的掌控，但不需要一直盯着每一行操作。

关于要避的坑，我有几点提醒。

第一，不要因为可以并行就无限制地堆任务。每个 agent 任务都有计算成本，任务太多会导致费用急剧上升，而且你同时管理太多任务，自己也容易搞混任务之间的关系，反而不如少跑几个管好来得有效率。找到你自己的最优并发数，通常三到五个是比较合理的起点。

第二，定期检查会话状态，不要完全放任。虽然 agent 可以自主跑，但它偶尔会走偏，或者在某个地方卡住需要你介入。侧边栏的状态过滤功能就是给你快速发现这些问题用的，隔一段时间扫一眼，及时介入处理。不要等所有任务都跑完了再去看，到时候问题可能已经积累很深了。

第三，用好三种视图模式。对于新启动的、不熟悉类型的任务，建议先用 Verbose 模式，把 agent 的每一步操作都看清楚，建立对这种任务类型的信任和理解。熟悉了之后，再切到 Normal 或者 Summary 模式，效率自然会提高很多。

第四，在任何代码合并之前一定要审查 diff。agent 的代码质量整体上是不错的，但不等于每次都完美无缺，尤其是在涉及业务逻辑核心或者安全敏感区域的时候，人工审查这一步是绝对不能省的。新版本的高性能 diff 查看器就是为了让这个审查过程更顺畅，好好利用它。

最后，来做一个整体总结。

这次 Claude Code 桌面应用的全面重设计，本质上是 Anthropic 对 agent 编程时代到来的一次正式回应。它从工具层面正式承认了一个现实：现代开发者的角色正在从"执行者"转向"调度者"。你的价值不再只是写代码，而是知道该让 agent 去做什么、在什么时候介入纠正、怎么验证最终结果的质量。侧边栏的多会话管理、side chat 的旁路对话、集成的工具箱、可拖拽的布局系统，每一个功能都在真实地解决用户在并行使用 agent 时遇到的具体痛点，方向是对的，执行也很扎实。

我个人认为，工具做好了只是第一步。更重要的是我们这些使用者能不能真正改变自己的工作方式，学会和 agent 协作，而不只是把它当成一个更聪明的搜索引擎来用。从"我来写，你来帮我"到"我来调度，你来执行"，这个思维方式的转变，可能比任何一个功能更新都更有长远价值。
