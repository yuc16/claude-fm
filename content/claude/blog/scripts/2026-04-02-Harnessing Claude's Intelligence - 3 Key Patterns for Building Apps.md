## Shownotes

这一期我们聊聊 Anthropic 官方博客上的一篇文章,讲的是构建 Claude 应用时容易被忽略的一个底层问题,就是你给模型搭的外部框架里藏着大量关于它做不到什么的过时假设。文章给出三个判断框架,帮你识别哪些补丁该删、哪些决策权该交还给模型。我们会结合 SWE-bench、BrowseComp 等多组实测数据,以及一个口袋妖怪游戏智能体的生动对比案例,把抽象的设计原则讲透。

本期要点:
- 通用工具优先:bash 加文本编辑器这两个最朴素的工具,反而是 Claude 用得最熟练、随模型升级持续增值的组合
- 让 Claude 自己编排行动:给它代码执行工具,把工具结果的过滤筛选交给模型自己处理,BrowseComp 上准确率从百分之四十五点三冲到百分之六十一点六
- 让 Claude 自己管理上下文:技能包的渐进式披露、上下文编辑、子代理派生,三种手段背后是同一个逻辑
- 让 Claude 自己决定记什么:压缩与记忆文件夹,新旧模型在口袋妖怪游戏里写笔记的方式判若两人
- 外壳依然有用:缓存断点设计能省下九成token成本,专用工具在安全和用户体验边界上仍不可替代
- 最关键的习惯:定期回头问自己,这个补丁还需要吗

## Script

今天我们要聊的这篇文章,英文标题是 Harnessing Claude's Intelligence,副标题是 Three Key Patterns for Building Apps,我把它翻译成驯服 Claude 的智能,打造应用的三个核心模式。这篇文章发表在 Claude 官方博客上,发表时间是二〇二六年四月二日,作者是 Lance Martin,他是 Anthropic Claude 平台团队的技术成员。算到今天,二〇二六年六月十九日,这篇文章发表已经差不多两个半月了。两个半月看起来不长,但在 Claude 这个模型迭代速度上,已经横跨了好几个版本的更新,文章里提到的 Opus 四点五、Opus 四点六、Sonnet 四点五,都是这段时间里密集发布的型号,所以这篇文章本身讨论的,正是模型一边在快速变聪明,应用开发者的工具和假设要怎么跟上这个节奏的问题。

为什么值得专门讲这一篇。因为它不是讲怎么写更花哨的提示词,也不是讲又出了什么新功能,而是在讲一个更容易被忽略的问题,就是你给 Claude 搭的那套外部框架,我们后面统一管它叫 agent 外壳,这个外壳里藏着大量假设,假设的是 Claude 做不到某些事情,所以才需要外壳来兜底。但 Claude 一直在变强,你三个月前写的兜底代码,可能这个月就已经变成了累赘。这篇文章给出的不是具体代码方案,而是三个判断框架,帮你定期回头检查,这个外壳是不是还有必要存在。

先说说这篇文章想解决的问题,以及它出现的行业背景。文章一开头引用了 Anthropic 联合创始人 Chris Olah 的一句话,他说像 Claude 这样的生成式 AI 系统,与其说是被造出来的,不如说是被养出来的。研究员设定好生长的条件和环境,但具体会长出什么样的结构、具备什么样的能力,并不是完全可以预测的。这句话听起来有点哲学,但它点出了一个很实际的工程问题。

如果你是在养一棵植物,你不会给它焊死一个铁架子,因为你不知道它会往哪个方向长,长多高,你给的支撑结构应该是可以随着它生长不断调整的。但现实中很多团队在给 Claude 搭建应用的时候,做的恰恰是焊铁架子这件事。大家会观察当前这一代模型的弱点,然后在外壳里写死各种补丁,比如这个模型不会自己规划任务,我们就写一个任务规划模块;这个模型记不住太长的上下文,我们就写一个外部的检索系统帮它记忆;这个模型不知道什么时候该停下来,我们就写超时和强制中断逻辑。这些补丁在写下来的那一刻都是合理的,但模型在持续进化,半年之后,你可能会发现新模型已经具备了你曾经要靠外部补丁才能实现的能力,而你的补丁还在那里运行,甚至在妨碍模型发挥它新长出来的能力。

这就是这篇文章想解决的核心张力。一边是模型能力的持续生长,一边是开发者为了让产品稳定可控而搭建的固定外壳。这两者的节奏是不一致的,模型可能几个月就有质的飞跃,但应用代码、产品文档、团队的肌肉记忆往往半年甚至一年都不会更新。文章想提醒所有正在用 Claude 搭应用的人,哪怕是这篇文章里讲的经验,也是有保鲜期的,值得你三个月后再回来重新审视一遍。基于这个背景,文章给出了三个具体的思考模式,我们一个一个来看。

第一个模式,叫做用 Claude 已经熟悉的工具去搭应用。这句话听起来简单,但背后有一个挺反直觉的发现。

文章举了一个例子。二〇二四年年底,Claude 三点五 Sonnet 在 SWE-bench Verified 上拿到了百分之四十九的分数,这个分数在当时是业界最好的成绩。SWE-bench Verified 是一个专门测试大模型能不能像真实程序员一样,去修复 GitHub 上真实存在的软件缺陷的基准测试,难度很高,因为它要求模型理解一整个代码仓库的上下文,而不是回答一道孤立的编程题。而拿到这个最好成绩的时候,Claude 用到的工具只有两个,一个是 bash,也就是命令行终端,一个是文本编辑器工具,用来查看、创建和编辑文件。没有什么花里胡哨的专用工具,就是两个最朴素、最通用的工具。

这件事说明了什么。Bash 这个工具,其实根本不是为了给 AI 智能体设计的,它是几十年前为人类工程师设计的命令行交互方式。但正因为它历史悠久、用途通用,互联网上关于怎么用 bash 的语料浩如烟海,Claude 在训练过程中早就把它学得非常透彻。所以即便 bash 不是专门为智能体打造的,它仍然是 Claude 真正理解、用得最熟练的工具。而且随着模型一代一代升级,Claude 用 bash 解决问题的能力还在持续变强,这是一个会随着模型成长而持续增值的工具,不需要你重新设计。

Claude Code,也就是 Anthropic 自家的编程智能体产品,用的就是这同一套底层工具,bash 加文本编辑器。再往上看,你会发现很多后来发布的、看起来更高级的能力,其实骨子里也是从这两个朴素工具上组合搭建出来的。比如 Agent Skills,也就是技能包机制,让 Claude 可以调用一套预先写好的操作流程;比如程序化工具调用,让 Claude 能用代码的方式去表达和组织工具调用;再比如记忆工具,让 Claude 能把信息写到文件里长期保存。这三个看起来很不一样的高级能力,拆开了看,底子都是 bash 和文本编辑器的不同排列组合。

这给我们一个启发,设计新工具之前先问一句,这件事能不能用 Claude 已经很熟练的通用工具拼出来。专用工具看起来更精确,但意味着 Claude 要重新学习一套陌生的调用方式;通用工具上 Claude 是带着海量训练经验进场的,这个优势还会随着模型变强继续放大。

讲完第一个大模式,文章接着说,agent 外壳里编码了大量关于 Claude 做不到什么的假设,而随着 Claude 越来越强,这些假设都应该被重新拿出来测试一遍。文章给了三个具体场景,分别是让 Claude 自己安排行动顺序、让 Claude 自己管理上下文、让 Claude 自己决定要长期保留什么内容。我们一个个拆开讲。

第一个,让 Claude 自己来编排它的行动。

这里有个很常见的默认做法,就是不管 Claude 调用了什么工具,工具返回的结果都要原样塞回 Claude 的上下文窗口里,让它看到之后再决定下一步干什么。上下文窗口你可以理解成 Claude 这一轮对话里能看到的全部信息范围。这个做法在很多场景下没问题,但有一种情况会很浪费,就是当工具返回的结果特别庞大,而 Claude 真正关心的只是其中很小一部分的时候。

文章举的例子是读一张很大的表格,但 Claude 只想推理其中一列的数据。如果按照默认做法,整张表格的每一行每一列都要原样塞进上下文窗口,Claude 要为它根本不需要的那些行付出计算和金钱的代价,响应也会变慢。有的团队会想,那我们在工具设计的时候提前写好过滤逻辑,硬编码只返回需要的那一列,这样确实能省一点,但这只是处理了表面问题,没有解决根本问题。根本问题是,要不要把整张表格塞进上下文,这本身是一个调度决策,而这个调度决策,Claude 自己比写死的过滤代码更清楚该怎么做,因为只有 Claude 知道这一轮它具体在意哪一列、需要哪种粒度的数据。

解决办法是给 Claude 一个代码执行工具,可以是 bash,也可以是某种编程语言的交互式执行环境。有了这个工具,Claude 不再是被动地等工具结果塞进上下文,而是可以自己写代码,在代码里完成工具调用,以及工具调用之间的过滤、筛选、传递逻辑,这些中间步骤完全在代码执行环境里跑完,根本不经过上下文窗口,只有代码最终执行完之后的输出,才会进到 Claude 看得到的那部分内容里。打个比方,这就像是你让一个新助理去仓库里找十份文件里的某个数字,你不需要让他把十份文件全部扫描一遍念给你听,你可以让他自己先翻一遍,找到那个数字之后再告诉你结果,中间翻文件的过程不需要你陪着看。

这个转变意味着,原来是外壳在替 Claude 做调度决策,现在调度决策本身交还给了模型。而且因为写代码本身就是一种通用的编排方式,所以一个编程能力强的模型,天然也会是一个通用能力强的智能体,这两件事是同一种能力在不同场景下的体现。文章给了一个具体的实验数据,在 BrowseComp 这个评测网页浏览智能体能力的基准测试上,给 Opus 四点六这个模型加上自己过滤工具输出结果的能力之后,准确率从百分之四十五点三提升到了百分之六十一点六,提升幅度相当可观,而且这是一个不需要写专门浏览逻辑、纯粹靠把调度权交还给模型就拿到的提升。

第二个,让 Claude 自己管理它的上下文。

任务相关的具体指令,会影响 Claude 怎么使用 bash 和文本编辑器这类通用工具。很多团队的默认做法是,把系统提示词写得又长又细,把各种任务专属的指令都提前塞进去。这个做法的问题在于,它没办法随着任务数量增长而扩展。你的系统提示词里塞的每一个 token,都会占用 Claude 有限的注意力预算,而且很多指令其实是为了应付极少数情况才加进去的,绝大部分时候根本用不上,提前把它们都加载进上下文,纯粹是浪费。

解决办法之一是给 Claude 访问技能包的能力。每个技能包前面会有一段简短的描述,这段描述会被提前加载进上下文窗口,让 Claude 大致知道这个技能包是干什么的。但技能包真正完整的内容,只有在 Claude 判断这次任务确实需要用到它的时候,才会调用读取文件的工具,把完整内容展开读进来。这种做法专业说法叫渐进式披露,意思是先给一个目录索引,真正要用的时候再展开细节,而不是把整本书都摆在桌面上。

如果技能包是让 Claude 自由地去组装它需要的上下文,那上下文编辑做的是相反的事情,它提供一种方式,让 Claude 能主动把已经过时或者不再相关的内容从上下文里清理出去,比如很久之前的工具调用结果,或者已经用不上的思考过程片段。你可以把这个想象成整理桌面,工作进行到一半的时候,桌上堆的草稿纸越来越多,有些已经是几步之前用过就废弃的,清理掉它们能让你后面的工作更专注。

还有一种工具是子代理。子代理指的是 Claude 在处理某个具体子任务的时候,主动选择开一个全新的、干净的上下文窗口去单独处理它,处理完之后只把结果带回主线程,而不是把子任务里产生的所有中间细节都堆在主线程的上下文里。文章里提到,Claude 在判断什么时候该开一个子代理这件事上,正在变得越来越准。还是用 Opus 四点六做的实验,在 BrowseComp 这个测试上,允许它派生子代理之后,效果比单一智能体能做到的最好结果还要再高百分之二点八。

第三个,让 Claude 自己决定要长期保留什么。

长时间运行的智能体任务,迟早会撞到单个上下文窗口的容量上限,这是物理限制,绕不开。很多团队的默认假设是,要解决这个问题就得在模型外面搭一套独立的检索系统,把历史信息存进数据库,需要的时候再检索回来。但 Anthropic 这边做的更多工作,是反过来,给 Claude 简单的手段,让它自己决定要记住什么、怎么记。

第一种手段叫压缩,也就是让 Claude 自己把过去的上下文总结成一段精炼的摘要,以此维持长任务里的连贯性。文章给了一组很有意思的对比数据,同样是在 BrowseComp 这个智能体搜索任务上,不管给多大的压缩预算,Sonnet 四点五的成绩都稳定卡在百分之四十三,完全没有提升,但换成 Opus 四点五,成绩涨到了百分之六十八,再换成 Opus 四点六,同样的设置下涨到了百分之八十四。这组数字说明,压缩这个能力本身不是越用越好,它高度依赖模型本身有没有学会判断,在浩繁的历史信息里,什么才是真正值得留下的。早期模型可能机械地把所有内容都往摘要里塞,新模型则学会了取舍。

第二种手段是记忆文件夹,让 Claude 把信息写到文件里,需要的时候再读出来。在 BrowseComp Plus 这个测试上,给 Sonnet 四点五配上记忆文件夹之后,准确率从百分之六十点四提升到了百分之六十七点二。

文章里举的口袋妖怪游戏的例子特别能说明问题,这是一个长时间连续运行的游戏智能体任务。Sonnet 三点五在玩这个游戏的时候,把记忆当成了一份逐字记录的剧本,会把游戏里非玩家角色说过的每一句话都记下来,而不是去判断哪些信息真正对接下来的游戏有用。跑到第一万四千步的时候,它攒了三十一个文件,里面甚至有两份内容几乎重复的关于毛毛虫系宝可梦的笔记,而游戏进度还停留在第二个小镇,几乎可以说是在原地打转。它写下的内容是这样的,大意是说卡特皮和独角虫都是毛毛虫系的宝可梦,卡特皮没有毒性,独角虫有毒,这个信息对未来的战斗很重要,如果我们的宝可梦中毒了,应该尽快去宝可梦中心治疗。你能感觉到,这段记录读起来更像是百科全书式的背景知识,而不是真正指导下一步行动的战术笔记。

而 Opus 四点六跑到同样的第一万四千步的时候,情况完全不一样,它只攒了十个文件,但这十个文件被分门别类地整理进了不同的目录里,游戏里已经拿到了三个道馆徽章,还专门维护了一个心得文件,里面记录的是从自己过去的失败里提炼出来的教训。文章里摘了几条,大意是说,贝芽木的睡眠加缠绕组合技要在睡粉落地之前用咬住技能速攻击杀,千万别让它先布置好场面;第一世代的背包道具上限是二十件,进副本之前要先扔掉不需要的招式学习器。这些笔记每一条都是面向未来决策的,是从失败里反推出来的可执行经验,而不是单纯记流水账。

这两个例子放在一起对比特别直观,同样是给模型一个写文件的能力,旧模型用它的方式接近于无差别地堆积信息,新模型用它的方式则是主动筛选、提炼、组织,这正好印证了前面那句话,记忆系统好不好用,很大程度上不取决于你给它搭了多复杂的外部基础设施,而取决于模型本身有没有学会判断什么值得记下来。

讲完了前两个模式,文章转向第三个角度,就是 agent 外壳本身的价值。外壳不是一无是处的,它的核心作用是在用户体验、成本和安全这三件事上,给 Claude 的行动提供必要的结构和约束。这部分文章讲了两个具体的实践要点。

第一个要点,是要在设计上让上下文尽量命中缓存。这里要先说一个技术背景,Claude 的 Messages 接口本质上是无状态的,意思是 Claude 自己看不到之前每一轮对话的历史,所以每一次调用,agent 外壳都需要把新的内容,连同之前所有的操作记录、工具描述、系统指令,重新打包一遍,完整地发送给 Claude。

但发送的内容里,有一部分是反复出现、几乎不变的,比如系统提示词、工具的定义说明,这部分内容如果每次都重新计算,是很浪费的。所以 Claude 的接口支持按照预先设定好的断点位置做提示词缓存,断点之前的内容会被写入缓存,下一次调用的时候系统会检查新请求里断点之前的部分跟之前缓存的内容是不是完全一致,如果一致就直接复用缓存结果,不用重新跑一遍。这件事划算到什么程度,文章里给了具体数字,命中缓存的 token,价格只是普通输入 token 的百分之十。这意味着如果你的外壳在设计提示词结构的时候,把会变化的内容和几乎不变的内容分开摆放,把不变的部分尽量靠前、变化的部分往后放,让缓存断点尽可能往后推,你能在不影响效果的前提下,把成本压低一大块,同时因为不用重新计算,响应速度也会更快。这是一个纯粹工程层面的优化,跟模型聪明不聪明没关系,但做对了能直接体现在你的账单和延迟上。

第二个要点,是该用专用工具的场合,就大方地用专用工具,主要是为了用户体验、可观测性和安全边界这三类需求。这里要澄清一下,前面讲的都是怎么把决策权交还给模型,但这不代表外壳什么都不用管了。

Claude 本身并不天然知道一个具体应用的安全边界在哪里,也不知道这个产品的用户体验设计要求是什么。Claude 能做的只是发出工具调用,真正怎么处理这个调用,是外壳的事。如果你只给 Claude 一个 bash 工具,它确实能获得很大的操作自由度,但站在外壳的角度,它收到的永远只是一串命令字符串,不管 Claude 这次到底想干什么,呈现给外壳的形状都是一样的,这就很难做精细化的拦截和审计。而如果你把某个具体动作单独包装成一个专用工具,外壳就能拿到这个动作专属的、带类型的参数,可以针对这个具体动作去拦截它、设门槛去审批它、把它渲染成用户能看懂的界面,或者把它记录进日志方便复盘。

那什么样的动作值得被包装成专用工具呢,文章给了一个挺实用的判断标准,叫可逆性。一个动作如果做错了很难撤销,比如调用外部 API 发起一次真实的转账或者下单,这种高风险、难撤销的动作,就值得专门做成工具,并且在执行前加上用户确认的环节。再比如写文件这类工具,可以内置一个新鲜度检查,确保 Claude 当前准备要修改的这个文件,内容没有在它上次读取之后被别人悄悄改动过,避免它凭着一份过时的认知去覆盖掉别人刚做的修改。专用工具还有一个好处,是当某个动作需要直接呈现给用户看的时候特别有用,比如把一个问题渲染成一个弹窗,给用户列出几个选项,或者直接暂停整个智能体的执行循环,等用户给出反馈之后再继续往下走。最后,专用工具对可观测性也很有价值,因为一旦某个动作被定义成带类型的工具调用,外壳就能拿到结构化的参数,可以方便地记录日志、做调用链路追踪,甚至原样重放复现某一次具体的执行过程。

不过文章特别提醒,要不要把某个动作升级成专用工具,这个决定不是一锤子买卖,而应该持续地被重新评估。它举了 Claude Code 自动模式的例子,这个功能在文章发表的时候还处于研究阶段,做法是让另一个 Claude 实例去读 Claude 要执行的那串命令字符串,判断这条命令安全不安全。这种做法相当于用模型自己当裁判,去代替原本需要写死的专用工具去做安全把关,这能在一定程度上减少专用工具的必要性。但文章也强调,这种用模型互相检查的做法,只适合那些用户本身就信任 Claude 大方向、能接受一定灵活度的任务场景,对于真正高风险、一旦出错代价很大的动作,专用工具仍然有它不可替代的位置,不会被这种模式完全取代。

文章在收尾之前,讲了一个很生动的真实案例,我觉得特别值得拿出来单独说一说。Anthropic 自己在搭一个处理长时间任务的智能体的时候,发现 Sonnet 四点五有一个挺有意思的毛病,每当它感觉上下文窗口快要用满了,它会提前草草收尾,把任务匆匆结束掉,即便实际上任务还没真正做完。团队把这个现象叫做上下文焦虑,顾名思义,就是模型表现得像是在担心自己快没地方写东西了,所以提前慌着结束。为了应对这个问题,团队专门在外壳里加了一套重置机制,在合适的时机清空上下文窗口,让模型不必带着焦虑硬撑到底。

但等团队换上 Opus 四点五之后,这个所谓的上下文焦虑现象直接消失了,新模型根本不会出现提前草草收尾的行为。这时候问题来了,之前为了应对焦虑专门写的那套上下文重置逻辑,还需要留在外壳里吗,答案是不需要了,它已经从一个解决方案变成了纯粹的死重量,挂在那里不仅没用,反而可能在某些场景下打断模型原本可以顺畅完成的长任务,变成了限制模型表现的瓶颈。

这个例子特别典型地展示了文章反复强调的那个道理,你今天为了弥补模型某个短板而写的代码,可能过几个月就变成了拖后腿的累赘,而你很容易因为它曾经有用,就一直留着不去碰它。

说到这里,我们来想一想,这篇文章对我们日常具体做 Claude 应用开发,到底有什么用,能不能落地。

第一点,设计工具集的时候先别急着发明专用接口,先问能不能用 bash 加文本编辑器拼出来。如果已经在用代码执行工具,试着把更多中间步骤交给 Claude 自己用代码编排,而不是每一步结果都强行塞回上下文,比如给它一个 Python 或者 Bash 执行环境,让数据筛选和多步骤串联都在里面跑完,只把最终结果带回主对话。如果你的产品经常要处理大段日志、大表格、长文档,这一条尤其值得直接去试。

第二点,如果系统提示词已经写得越来越长、越来越难维护,这就是该拆技能包的信号。技能包只在真正用得上的时候才把完整内容加载进上下文,平时只占一点描述性开销。如果你的应用里有多个相对独立又不常同时触发的业务场景,比如客服里的退款、物流查询、账户安全处理,特别适合各自拆成技能包,而不是一股脑写进同一份系统提示词。

第三点,智能体要跑长任务时,不要急着自己从零搭外部检索和数据库系统,先试试官方提供的压缩能力和记忆文件夹够不够用。这类能力的效果跟模型版本关系很大,旧模型可能确实需要更多外部辅助,但换上新模型之后记得回头重新测一遍,看看之前搭的外部记忆系统是不是已经能简化甚至去掉。

第四点,缓存命中是性价比最高、最容易落地的一条,不需要等模型升级就能马上做。把固定不变的系统指令、工具说明放在前面,把每次都会变化的内容放在后面,减少在固定部分里夹杂动态内容,让缓存断点尽量往后推,命中率更高,账单和响应速度都会因此受益,建议尽早检查自己现在的实现。

第五点,也是最容易被忽视的一点,文章最后那句问一问自己能不能停止做某件事,我觉得特别值得反复提醒自己。建议每次模型有重大版本更新的时候,回头审查外壳里那些曾经为了弥补模型短板而写的补丁代码,逐条问自己,这个补丁解决的问题新模型是不是已经自己解决了,如果是,大胆删掉它,不要因为它曾经立过功就一直占着位置。

第六点,关于专用工具和裁判模式,我的建议是按风险高低分层处理。涉及真实金钱往来、对外部系统写操作这类很难挽回的动作,坚持用专用工具加用户确认,不要因为追求灵活就放松这层把关;但对本身可逆、出错成本低的探索性操作,可以考虑像 Claude Code 自动模式那样用模型自己做轻量级安全判断,减少不必要的硬性拦截。

整篇文章串起来看,讲的其实是同一件事的三个侧面。先是说 Claude 已经具备了远超你想象的通用能力,很多时候你不需要给它造新轮子;然后是说模型一直在进化,你今天觉得它做不到的事情,过几个月可能就已经做到了,所以要不断地把决策权往模型那边交;最后是说外壳依然有它存在的价值,但这个价值要聚焦在用户体验、成本控制和安全边界这几件真正需要人为兜底的事情上,而不是替模型去做它自己已经能做好的判断。

我个人认为这篇文章最有价值的地方,不是任何具体的技术细节,而是它给出的那个持续追问的习惯,定期回头问一句这个补丁还需要吗。这件事说起来容易,做起来很难,因为删代码总是比加代码需要更大的心理建设,谁都不想成为那个删掉兜底逻辑之后导致线上出问题的人。但如果完全不做,你的 agent 外壳会越堆越重,最后变成专门伺候上一代模型缺陷的庞大遗产系统,反而成为新模型发挥实力的障碍。我们这期就先聊到这里,如果你最近也在搭 Claude 相关的应用,不妨找个时间,回头审查一遍那些曾经合理、现在未必还合理的假设。
