大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是 What is Model Context Protocol，Connect AI to your world，翻译过来就是,什么是模型上下文协议,把 AI 连接到你的世界。这篇文章发布于二零二五年十月三十一日,也就是去年的万圣节那天,出自 Claude 官方博客。

先说一句提醒。今天是二零二六年六月二十日,距离这篇文章发布已经过去了差不多七个半月,超过半年了。在 AI 这个行业里,七个半月不算短。所以我们今天讲的内容,更多是帮你理解 MCP 这个协议本身的设计思路和价值主张,而协议生态的具体规模,比如有多少个 Connectors,有多少社区服务器,这些数字现在很可能已经远远超过文章里提到的水平了。如果你想知道最新的生态进展,建议去 Claude 官方文档或者 MCP 的开源仓库看最新状态,我们今天主要是把这个协议讲透,讲明白它为什么会被设计成这样。

好,我们先从背景说起。这篇文章一开场就抛出一个很多人都有共鸣的场景。你在用 Claude 这样的 AI 助手的时候,经常会遇到一个尴尬的情况,就是 AI 本身很聪明,能回答问题,能写代码,能帮你分析问题,但它没办法直接碰到你真正在用的那些工具和数据。你想让它帮你改一份 Google Drive 里的文档,你得自己把内容复制过来粘贴到对话框里。你想让它根据 Slack 里的一段讨论帮你写回复,你还是得手动把那段对话拷过去。你想让它根据你 IDE 里的代码做点什么,同样得复制粘贴。这个过程慢,而且很容易漏掉关键的上下文信息,比如一段很长的讨论你可能只复制了一部分,AI 看到的就是不完整的画面。

这个问题在技术上有个很经典的名字,叫做 M 乘 N 问题。什么意思呢,假设你有 M 个不同的 AI 应用,比如 Claude、其他厂商的助手、各种 agent 框架,同时你又有 N 个不同的工具和数据源,比如 Notion、Slack、Figma、数据库、各种企业内部系统。如果每一个 AI 应用都要单独为每一个工具写一套定制的集成代码,那么理论上你需要写 M 乘以 N 套集成。这个数字会随着 AI 应用和工具数量的增长而急剧膨胀,变成一个谁都搞不定的天文数字。文章里提到,MCP 这个协议的诞生,正是源于这个具体的痛点。

接下来我们讲讲 MCP 的诞生故事,这一段我觉得挺有意思的。MCP 是在 Anthropic 内部诞生的,两位主要的创造者叫 David Soria Parra 和 Justin Spahr-Summers。David 当时的困扰特别具体,他经常需要在 Claude Desktop 和自己的 IDE 之间来回倒腾代码,反复复制粘贴,这种重复劳动让他意识到,这其实就是前面说的那个 M 乘 N 问题的一个缩影。于是他把这个想法提给了 Justin,两个人一起设计出了这套协议。

值得一提的是,他们在设计 MCP 的时候,参考了一个已经存在多年、被证明很成功的协议,叫做 Language Server Protocol,中文可以叫语言服务器协议。如果你是程序员,你可能已经在不知不觉中享受过这个协议的好处了。在这个协议出现之前,每一个代码编辑器,比如 Vim、Emacs、Visual Studio,如果想要支持某一种编程语言的智能补全、跳转定义、错误提示这些功能,都需要单独为每种语言写一套适配代码。后来 Language Server Protocol 出现了,它定义了一种标准化的通信方式,让任何编辑器只要实现这个协议,就能立刻支持任何遵循这个协议的编程语言服务。这极大地减少了重复劳动。MCP 的设计哲学和它一脉相承,只不过解决的不再是编辑器和编程语言之间的连接问题,而是 AI 模型和外部工具数据之间的连接问题。

这套协议在二零二四年十一月开源发布,并且得到了 Anthropic 公司层面的支持,确保整个 AI 生态都能从中受益,而不是变成 Anthropic 一家独享的私有技术。这个开源的决定我觉得是非常关键的一步,因为协议这种东西,价值恰恰在于被广泛采用,如果只是一家公司自己用,那它本质上还是退化成了普通的私有接口,失去了协议本身的意义。

好,讲完了背景故事,我们来拆解一下 MCP 到底是怎么工作的,这是整篇文章的核心部分。

文章用了一个特别形象的类比,把 MCP 比作 AI 世界里的 USB 三代接口,也就是 USB-C。我们都用过老式的电子设备,苹果手机用 Lightning 接口,安卓手机用 micro 接口,相机又是另一种专有接口,每一种设备都有自己专属的连接方式,你出门得带一堆不同的充电线。后来 USB-C 出现了,越来越多的设备开始统一采用这个接口标准,你现在出门可能只需要带一根线,就能给手机、笔记本电脑、平板甚至某些相机充电或者传输数据。这就是标准化接口带来的好处,它不是让某一种设备变得更强,而是让所有设备之间的连接变得简单。

MCP 想做的就是这件事,只不过它连接的不是物理设备,而是大语言模型和外部系统。在 MCP 出现之前,每一个应用,每一个数据库,想要跟大语言模型对接,都得写一套量身定制的代码。Google Drive 要单独做一套集成,Slack 要做另一套,Figma 又是完全不同的一套。现在有了 MCP,这些工具只需要遵循同一个标准化的协议格式,就能和 Claude,或者其他任何兼容这个协议的 AI 应用对接起来。

那么具体在架构上,MCP 是怎么设计的呢。文章里说,MCP 采用的是一种双边的设计思路。一边是 AI 助手和聊天机器人,比如 Claude,它们会去实现一个叫做 MCP Client 的东西,也就是 MCP 客户端。客户端的作用是让 AI 应用具备连接外部系统的能力。另一边是各种各样的应用和服务,比如 Notion、Canva、Figma,它们会去实现一个叫做 MCP Server 的东西,也就是 MCP 服务器。服务器的作用是把这个应用自己的工具和数据,用标准化的方式暴露出来,供任何兼容 MCP 的 AI 客户端去调用。

这个设计思路其实非常巧妙,因为它把一个原本是 M 乘 N 的组合爆炸问题,转化成了一个 M 加 N 的线性问题。打个比方,如果你是开发一个 AI agent 产品的团队,你不需要为市面上每一个工具单独写适配代码,你只需要把 MCP 客户端这个标准实现好,就能立刻接入成百上千个已经存在的 MCP 服务器。反过来,如果你是 Notion 或者 Figma 这样的工具厂商,你也不需要去适配每一个 AI 助手,你只需要把自己的 MCP 服务器做好,做这一次集成,就能同时被 Claude,也能被其他任何兼容 MCP 的 AI 应用调用。这种从乘法关系变成加法关系的转变,正是协议类技术的核心价值所在,我们在前面提到的 Language Server Protocol 身上,已经看到过完全一样的逻辑。

而且因为 MCP 是完全开源的,任何人,不管是大公司还是个人开发者,都可以去构建自己的 MCP 服务器,或者自己的 MCP 客户端,不需要经过谁的许可,这也是为什么文章里说,这套协议催生了一个非常活跃的社区生态,有大量社区开发者贡献的服务器存在。

讲到这里,我们可以总结一下 MCP 给整个生态带来的几个具体好处,文章里列了四点,我觉得每一点都值得展开说说。

第一点,AI 助手一下子能接入成千上万个工具。一旦某个 AI 应用实现了 MCP 客户端,它就能立刻连接到所有兼容 MCP 的应用,从专业的编程工具到企业级的工作流平台,都不需要再为每一个单独开发定制集成。你可以想象一下,这就像你买了一台新手机,只要它支持 USB-C,你家里原来那一堆 USB-C 配件,充电器、耳机、外接硬盘,立刻都能用上,完全不需要重新适配。

第二点,工具和应用一次性连接所有的 AI 助手。像 Notion、Figma、Asana 这样的公司,只需要构建一个 MCP 服务器,就能同时兼容所有实现了 MCP 客户端的 AI 助手。开发者只需要做一次集成工作,就能覆盖所有的 AI 连接场景,而不是看到一个新的 AI 助手冒出来,就得重新做一遍适配。

第三点,任何人都可以构建和分享。因为这是一个开放标准,开发者或者公司发布的 MCP 服务器,天然就和任何 MCP 客户端兼容。这种开放性催生了一个非常活跃的生态,大量社区自建的服务器加速了 AI 助手能接入的工具数量的增长。

第四点,这一点我觉得特别值得琢磨,文章说 MCP 让软件天生就能被 AI 访问。传统软件的设计出发点是给人用的,有图形界面,有按钮,有菜单,这些都是为人类的视觉和操作习惯设计的。而 MCP 提供了一套平行的接口,这套接口是专门为 AI 交互设计的,这意味着应用程序可以变得真正意义上的原生支持 AI,而不是事后硬塞一个聊天框进去敷衍了事。这其实代表了一种软件设计理念的转变,以前我们做产品,默认的使用者是人,未来很多产品在设计阶段,可能就要同时考虑人类用户和 AI 代理这两种使用者,这是一个挺值得软件工程师琢磨的方向。

文章接着分了三类受众来说明 MCP 对他们各自意味着什么,我觉得这部分对我们听众来说尤其实用,我们可以分别站在这三个角色的视角想一想。

对开发者来说,你只需要遵循一个标准,就能把外部产品连接到你的 AI 应用和 agent 里。这简化了构建集成的过程,扩大了你能连接的产品数量,也提升了整个生态里连接的质量和安全性。如果你正在构建一个会连接很多应用的 agent,或者反过来,正在构建一个会被很多 agent 连接的应用,MCP 都能给你提供一个现成的兼容工具生态,省去大量重复的集成劳动。

对企业来说,MCP 能帮助你更有效地推动内部 AI 工具和应用的落地采用,因为它简化了把企业内部系统连接到 AI 的流程。这能让 AI 在你的组织内部变得更加紧密相连,扩展它对员工的能力和实用性。我自己读到这一段的时候会联想到,很多大公司内部其实有大量的私有系统,工单系统、内部知识库、审批流程,这些系统过去想接入 AI 助手,往往得专门找一个团队做定制开发,周期长,成本高。如果这些内部系统都按照 MCP 标准做了服务器封装,理论上员工就能直接用 Claude 这样的助手去操作内部系统,这对企业 IT 部门来说,既是效率提升,也是一种更容易统一管控的安全模式,因为所有的连接都走同一个标准化协议,而不是五花八门各自为政的私有接口。

对普通终端用户来说,MCP 提供的是你最喜欢的 AI 助手和你工作中常用工具之间的无缝连接。它让自动化任务变得更容易,也让你不用再在不同标签页之间来回复制粘贴。简单说一句话,MCP 让 AI 对你的世界拥有了更强的访问和连接能力。

讲完了原理和价值,我们来看看文章里举的几个具体例子,这部分我觉得特别能帮我们理解 MCP 落地之后实际是什么样子。在 Claude 产品里,这些已经接入的 MCP 服务器,被统一称为 Connectors,中文大概可以理解为连接器。

第一个例子是 Canva 连接器。通过这个连接器,Claude 可以直接在 Canva 这个设计工具的画布里生成新的设计。背后的原理是,Claude 通过 MCP 连接到 Canva 提供的工具接口,直接在画布上完成设计生成的操作,而不是生成一段描述让你自己去 Canva 里手动制作。

第二个例子是 Notion 和 Linear 这两个连接器配合使用的场景。Claude 可以访问你 Notion 里的页面内容,然后用这些内容去更新 Linear 里的工单。这个例子我觉得特别能说明 MCP 的实际价值,因为它解决的正是前面提到的那种碎片化的非结构化信息,变成结构化任务的转换过程。你平时可能在 Notion 里随手记了一些产品想法或者会议笔记,这些内容是松散的、非结构化的文字,而 Linear 是一个需要结构化数据的项目管理系统,字段、状态、优先级都得清楚。过去你得自己读一遍 Notion 笔记,提炼要点,手动在 Linear 里建工单填字段。现在 MCP 让 Claude 可以同时连着两边,直接把笔记里的信息梳理好,变成规整的工单,这个跨系统的信息搬运和结构化过程,被自动化掉了。

第三个例子是 Figma 连接器,它让 Claude 可以访问 Figma 里的设计稿。这一点对做 Claude Code 的开发者特别有意义,因为这意味着 Claude Code 可以基于 Figma 里实际创建的设计稿,直接生成网站、应用或者用户界面的可工作原型。如果你自己想一下这个流程,在没有 MCP 的情况下,设计师做完稿子,通常需要导出截图或者标注文档,工程师再对着这些静态图自己手写代码去还原界面细节,间距、颜色、字体大小,很多东西全靠肉眼比对,经常对不齐。而通过 MCP,Claude Code 能够直接读取设计稿里的结构化数据,理论上可以更准确地还原设计意图,减少这种人工对齐造成的误差和反复修改的拉锯。

文章里还列了一份当时已经接入的 Connectors 清单,包括 Notion 用于工作区文档管理,Linear 用于问题跟踪,Stripe 用于支付数据,Canva 和 Figma 用于设计辅助,Hubspot 用于自动化客户关系管理任务,Sentry 用于错误追踪,以及还有很多其他没有一一列出的连接器。文章特别提到,每一个连接器只需要花几秒钟配置,就能成为 Claude 工作上下文的一部分。考虑到这篇文章发布到现在已经过去七个半月了,我估计现在 Connectors 清单只会比这更长,种类更丰富,如果你对具体有哪些感兴趣,直接去 Claude 里翻一下连接器列表是最准的办法,这是文章里的内容不会自动更新,但产品本身一直在迭代。

那么说到这里,我们该聊聊,作为听众,你现在能怎么实际用上这些东西。文章里给出了两条路径,分别对应不同的需求。

第一条路径,如果你是普通用户,或者你只是想体验一下 MCP 带来的便利,最简单的方式就是直接在 Claude 里浏览现成的连接器。这些连接器是预先构建好的,能给 Claude 立刻接入这些工具、数据库和应用的能力,给你带来一整套新的功能。操作上也很简单,打开 Claude,浏览可用的连接器列表,点一下添加就行,几乎没有技术门槛。

第二条路径,如果你是开发者,想要自己动手构建一个 MCP 服务器,或者想让自己的产品支持 MCP,因为 MCP 是完全开源的,任何人都可以基于这个标准去对接 AI。文章提到官方的 Model Context Protocol 文档里有详细的步骤,教你怎么动手构建。如果你已经有一个想用的工具,但是还没有现成的 MCP 服务器,自己搭建一个虽然需要一些工作量,但如果你懂 TypeScript 或者 Python,难度并不算太高。官方提供的 Model Context Protocol quickstart,也就是快速入门指南,里面有可以直接拿来改的现成示例代码,可以作为你的起点。

我自己在这里想多说几句个人的看法,这些是我自己的推演和观察,不是原文里写的内容。我认为对于国内的工程师团队来说,MCP 这种协议特别值得关注的一个场景,是企业内部系统的 AI 化改造。很多公司内部有大量历史悠久的系统,比如内部工单系统、审批流程、知识库、监控告警平台,这些系统过去想要和 AI 助手打通,往往意味着专门拉一个项目组,花几个月时间做定制对接,而且换一个 AI 助手厂商,这套对接很可能就要推倒重做。如果你的团队提前把内部系统按 MCP 标准封装成服务器,理论上无论未来用 Claude,还是用其他兼容 MCP 的助手,这套对接工作都可以复用,这对企业的长期 IT 投入是有实际节省价值的。

不过我也想提一个我自己觉得值得留意的坑,这一点文章里没有展开讲,但我觉得在企业落地的时候一定要重视,那就是权限和安全的边界问题。当你把内部系统通过 MCP 服务器暴露给 AI 客户端调用的时候,本质上是在给 AI 一把能操作真实系统的钥匙,这意味着权限控制、操作审计、敏感数据的访问范围,都需要在服务器这一端提前设计清楚,而不能简单地假设 AI 调用就一定是安全无害的。比如说,如果你的 MCP 服务器对接的是支付系统或者客户数据库,你肯定不希望 AI 在没有明确授权的情况下就能执行删除或者转账这类高风险操作。这一类安全设计,我觉得在企业大规模采用 MCP 之前,是必须想清楚的前置工作,这也是为什么文章里提到企业能获得的好处之一,是更安全、由 IT 统一管控的 AI 连接能力,这背后其实暗含着,标准化协议本身确实更利于做统一的安全审查和权限管控,相比过去那种各自为政、五花八门的私有接口,要规范得多。

好,我们来给今天的内容做个总结。这篇文章核心讲的是 Model Context Protocol,模型上下文协议,它本质上是给 AI 模型和外部工具数据之间的连接,提供了一个统一的标准接口,就像 USB-C 统一了各种设备的物理接口一样。它诞生于 Anthropic 内部工程师对重复劳动的真实痛点,设计思路借鉴了已经被验证成功的 Language Server Protocol,并且在二零二四年十一月选择了开源,这个开放的决定让它快速形成了一个活跃的生态。它采用客户端和服务器两端的架构,把原本组合爆炸式的 M 乘 N 集成问题,转化成了线性增长的 M 加 N 问题,这对开发者、企业和终端用户都带来了实际的好处。在 Claude 产品里,这套协议体现为 Connectors,已经接入了 Notion、Linear、Figma、Canva、Stripe 等大量常用工具,让 AI 真正具备了跨越多个应用边界完成复杂任务的能力。

我个人的点评是,MCP 这类协议的价值,往往不是在它刚发布的时候就能被充分认识到的,而是要等到生态规模真正起来之后才会显现,就像 Language Server Protocol 当年也是被广泛采用之后才成为编辑器开发的事实标准。考虑到这篇文章发布已经过去了七个半月,我建议你如果对这个协议感兴趣,不要止步于这篇文章里的信息,去看看现在 MCP 生态发展到什么程度了,毕竟这是一个变化很快的领域。
