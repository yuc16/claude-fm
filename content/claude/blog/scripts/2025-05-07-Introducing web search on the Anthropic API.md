大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是 Introducing web search on the Anthropic API,翻译过来就是,在 Anthropic 的 API 上推出联网搜索功能。这篇文章发表于二〇二五年五月七日,距离我们录这期节目,已经过去了一年多的时间。

先给大家一个时间上的提醒。这篇文章发布的时候,正是 Claude 三点七 Sonnet 刚刚问世不久的阶段,整个行业里大模型联网搜索这件事,还属于相对早期、各家都在摸索的状态。到我们录这期节目的二〇二六年六月,联网搜索几乎已经成了主流大模型产品的标配功能,无论是 Claude、还是其他家的产品,基本上都已经把这个能力做成了用户感知不到、但默默在后台运转的基础设施。所以你听这期节目的时候,要把自己代入回二〇二五年五月那个时间点,去理解当时这个功能为什么重要、解决了什么核心痛点。它本身的具体参数、定价可能已经有所调整,但是它背后的设计思路,放到今天依然非常值得拆解。

那我们先说说背景。在联网搜索这个功能出来之前,大模型有一个大家都心知肚明的硬伤,就是知识的时效性问题。模型的知识是靠训练数据喂出来的,而训练数据总有一个截止时间,这个截止时间业内叫 knowledge cutoff,知识截止日期。一旦超过这个日期,模型对世界的认知基本上就静止了。你问它今天的股价,它给不出来,因为它根本不知道今天是哪一天。你问它某个开源框架最新版本有什么新特性,它给你的答案可能是几个月甚至一年前的状态。这对很多场景来说是致命的,比如说金融分析,你要看的是当下的股价和市场动向,法律研究,你要看的是最近的判例和监管政策变化,开发者工具场景,你要看的是某个库刚刚发布的新版本文档。这些场景统统要求实时性,而大模型本身天生缺这一块。

那在这之前,开发者要解决这个问题,基本上只有一条路,就是自己搭一套检索增强生成的系统,业内叫 RAG,也就是 Retrieval Augmented Generation。简单说就是自己写代码去调搜索引擎的接口,把搜索结果抓回来,再想办法塞进给大模型的提示词里,让模型基于这些搜索结果去回答问题。这个事听起来不难,但实际做起来工程量相当大。你要处理搜索 API 的调用、结果的清洗、和大模型的回合式协调、还要自己判断到底要不要搜、搜一次够不够、要不要根据第一次的结果再搜第二次。这些全部都要自己写逻辑去管理,而且每家公司、每个团队做出来的效果千差万别,质量参差不齐。

所以 Anthropic 这篇文章要解决的核心问题就是,把这一整套联网搜索的脏活累活,直接做成 Claude 的一个内置工具,开发者只需要在调用 API 的时候,打开一个开关,剩下的事情,包括什么时候该搜、怎么搜、搜几轮、怎么把结果整合进答案里,全部交给 Claude 自己去处理。这就是这篇文章的核心,我们接下来一点一点拆开讲。

第一个核心点,我想讲清楚的是,这个功能的工作机制到底是什么样的。原文里说得很清楚,当 Claude 接收到一个请求,它会先用自己的推理能力去判断,这个问题是不是需要联网搜索才能给出更准确的答案。这一点非常关键,它不是说你每次提问它都无脑去搜一次,而是有一个判断的过程。打个比方,你可以把这想象成一个特别靠谱的助理。你问他一个常识性问题,比如说什么是光合作用,他凭脑子里的知识就能答得很好,根本不需要跑去查资料。但如果你问他今天纳斯达克指数收盘是多少,他会立刻意识到,这个问题我脑子里没有答案,必须去查一下才靠谱。Claude 在这里扮演的就是这个有判断力的助理的角色,它会自己决定什么时候该伸手去摸搜索引擎这个工具。

判断完之后,如果确实需要搜索,Claude 会生成一个有针对性的搜索查询词。这一步其实也挺有意思,因为它不是把你的原始问题原封不动地丢给搜索引擎,而是会先在脑子里转一圈,把你的问题改写成一个更适合搜索引擎检索的关键词组合。这就好比你去图书馆找资料,你不会跟图书管理员说一整段长篇大论的话,你会先把自己的需求提炼成几个关键词,这样查起来效率更高。Claude 在后台做的事情其实跟这个类似,文章里专门提到,Claude 在背后可能会不断优化和精炼它的查询词,目的就是为了拿到更准确的搜索结果。

拿到搜索结果之后,Claude 会分析这些结果里的关键信息,然后再生成一个综合性的回答,并且在回答里附上引用来源,指向原始的资料出处。这个流程走下来,其实就是一个完整的检索增强生成的闸路,只不过这个闸路完全被封装在 Claude 内部,开发者一行检索代码都不用写。

第二个值得深挖的点,是这个功能里非常聪明的一个设计,叫渐进式多轮搜索,原文用的词是 progressive searches。这是什么意思呢,就是说 Claude 不是只搜一次就完事了,它可以像一个真正做研究的人那样,先搜一轮,看看结果怎么样,然后根据第一轮搜到的信息,再去决定第二轮要搜什么。这种行为方式,文章里称之为代理式的操作,英文是 agentically,也就是说 Claude 在这个过程里表现得更像一个自主行动的智能体,而不是一个被动等指令的工具。

我用一个具体的场景给大家还原一下这个过程。假设你问 Claude,某家上市公司最近发布的财报对它的股价有什么影响。Claude 第一轮搜索可能先去查这家公司最近的财报数据是什么,搜到之后它发现,哦,原来这次财报里有一个核心指标超预期了。基于这个新信息,它可能会发起第二轮搜索,专门去查市场分析师对这个超预期指标的反应和评论。再根据第二轮搜到的内容,它甚至可能发起第三轮搜索,去查这家公司同行业竞争对手最近的股价走势作为对比。这种一轮接一轮、层层递进的搜索方式,跟人类做研究时候的思路其实非常像,我们做调研的时候也是这样,先有个大概方向,搜到一点线索之后,再顺着线索往深处挖。

那么这种多轮搜索会不会无限制地搜下去,把成本搞得很高呢,这就引出了文章里提到的一个非常重要的控制参数,叫 max_uses,直译过来就是最大使用次数。开发者可以通过设置这个参数,来控制 Claude 在一次对话里最多可以进行多少次搜索调用。这其实是一个很务实的设计,因为联网搜索是要收费的,如果完全不设限制,理论上 Claude 可能会无止境地搜下去,既浪费时间也浪费成本。通过这个参数,开发者就可以在搜索的深度和成本之间找一个平衡点。比如说你做一个简单的问答机器人,可能把这个值设得很低,一两次就够了,但如果你是在做一个深度研究类的应用,希望 Claude 能够多方查证、交叉验证信息,你可能就会把这个值设得高一些,让它有更大的探索空间。

我自己设想了一下,如果是我来设计一个使用场景,我会怎么用这个参数。比如说我在做一个面向投资者的早间简报工具,每天早上要总结隔夜的市场动态,这种场景下时效性和效率都很重要,我可能会把 max_uses 设成比较保守的数字,比如三到五次,确保 Claude 能拿到核心信息但不会因为反复搜索而拖慢响应速度。但如果我是在做一个法律合规审查的工具,需要交叉比对多个判例和法规文件,准确性的优先级要远高于响应速度,这种场景下我会愿意把这个参数放得更宽松一些,让 Claude 有更充分的空间去做交叉验证。

第三个我想重点讲的,是引用和溯源的机制,这一点我个人认为是整篇文章里被低估、但实际上特别重要的一个设计。原文说得很明确,每一个基于网络搜索得出的回答,都会附带引用,指向具体的来源材料,这样用户就可以直接核实信息的真实性。

这个设计背后的逻辑是什么呢,我们都知道大模型有一个广为人知的问题,叫幻觉,英文是 hallucination,就是模型有时候会很自信地编造出看起来很真实、但实际上是错误甚至完全虚构的信息。如果联网搜索拿回来的信息,Claude 再加工一遍之后直接甩给你一段总结性的文字,你是没有办法判断这段总结到底是忠实地反映了搜索结果,还是模型自己又加了点想象力进去。但如果每一条信息背后都挂着引用,指向具体是哪个网页、哪篇文章说的,用户就可以点进去自己核实,这就大大提升了整个系统的可信度和可问责性。

这一点对哪些场景特别关键呢,文章里专门点名了金融服务和法律研究这两个领域。你想啊,如果你是一个做投资决策支持工具的开发者,你的用户是基金经理或者分析师,他们要根据 AI 给出的信息做出实际的投资决策,如果这个信息是编出来的,后果可能是真金白银的损失。法律场景更是如此,律师如果引用了一个不存在的判例去打官司,这在很多司法管辖区是要被处分的。所以在这些高风险、高准确性要求的领域,引用溯源机制不是一个锦上添花的功能,而是一个必须有的安全阀。

我顺着这个思路往下想,如果换我来用这个功能做产品,我会怎么把引用机制用到极致。我觉得至少应该在用户界面层面,把每一条引用做成可以直接点击跳转的链接,而不是简单地在文字末尾甩一个角标。另外,我还会考虑做一个简单的可信度分级,比如说官方网站、权威媒体来源的引用,和一些来源不明的小论坛或者自媒体内容的引用,在界面上用不同的视觉权重区分开来,提醒用户这两类信息的可信程度是不一样的。这些都是原文没有展开讲,但是顺着引用机制这个核心设计很自然能往下延伸的应用层思考。

接下来我们说说企业级的管理和控制能力,这一块文章虽然篇幅不长,但我觉得对企业级开发者来说非常实用。Anthropic 提供了三层控制手段。第一层叫域名允许列表,就是管理员可以指定 Claude 只能从哪些域名搜索和获取信息,确保搜索结果只来自经过批准的来源。第二层叫域名屏蔽列表,反过来,管理员可以指定 Claude 禁止访问哪些域名,这些域名可能包含敏感信息,或者是竞争对手的内容,或者是公司认为不适合的内容。第三层是组织级别的整体管理,管理员可以在整个组织层面,统一允许或者禁止使用联网搜索这个功能。

这三层控制设计,其实反映了企业在引入 AI 联网能力时候普遍会有的顾虑。我们站在一个企业 IT 或者合规负责人的角度想一想,如果你给公司里所有员工开放了一个可以无限制联网搜索的 AI 工具,会有什么风险。第一个风险是信息安全,员工可能不小心让 AI 去搜索一些你不希望员工接触到的内容,比如说竞争对手的内部资料,或者一些敏感的政治话题网站。第二个风险是信息质量,如果 AI 搜索的来源鱼龙混杂,什么营销软文、不靠谱的小道消息网站都搜进来,得出的结论质量就没法保证。第三个风险是合规风险,某些行业比如金融、医疗,对信息来源是有严格的合规要求的,你不能随便从一个不知名的博客上抓信息来支撑一个投资建议。

域名允许列表和屏蔽列表这两个机制,恰好就是用来对冲这几类风险的。举个具体例子,假如你是一家证券公司,要给内部分析师配一个联网搜索的 Claude 助手,你完全可以把允许列表设置成只包括交易所官网、证监会官网、几家你信任的财经媒体,这样无论分析师问什么问题,Claude 给出来的信息源头都是经过你审核的权威渠道,从根本上降低了信息以次充好的风险。反过来,如果你是一家科技公司,你可能不太需要严格的允许列表,但你会想加一个屏蔽列表,把竞争对手的官网、或者一些可能涉及商业机密泄露风险的论坛给屏蔽掉,防止员工不小心把不该看的内容搜进对话里。

第四个我想讲的点,是 Claude Code 里的联网搜索能力。文章里专门提到,联网搜索功能也在 Claude Code 里上线了,这给开发工作流加入了实时的网络信息。具体来说,开启联网搜索之后,Claude Code 可以访问最新的 API 文档、技术文章,以及关于开发工具和各种库的其他信息。文章里特别强调了几个场景,第一是处理新出现或者快速迭代的框架,第二是排查一些比较冷门、奇怪的报错,第三是实现一些需要特定版本 API 参考的功能。

这一点我觉得对我们听众,尤其是软件工程师朋友们来说,特别有共鸣。大家应该都遇到过这种情况,你在用一个开源库,这个库这两个月刚发布了一个大版本更新,API 接口变了,但是你手头的大模型脑子里的知识还停留在旧版本,你问它怎么用新接口,它给你的代码示例完全是过时的写法,跑起来各种报错。这种问题在过去,只能靠你自己去翻官方文档、去 GitHub 上看最新的 issue 和 release note 来解决。有了联网搜索能力之后,Claude Code 理论上可以自己去搜这些最新的文档和发布记录,直接给你提供跟最新版本匹配的代码建议。

我设想一下这个能力在实际工作里具体怎么用。假设你正在用某个比较新的前端框架做项目,你让 Claude Code 帮你实现一个功能,如果这个框架最近刚刚发布了一个新的 API 来简化某个常见操作,而 Claude 原本的知识库里还没有这个新 API 的信息,在开启联网搜索之后,Claude Code 完全可以主动去搜这个框架的官方更新日志和文档,发现这个新 API 的存在,然后用更现代、更简洁的写法帮你实现功能,而不是用一个已经过时甚至可能在新版本里被标记为废弃的老写法。这对于追前沿技术栈的团队来说,价值是非常直接的。

不过这里我要补充一点我个人的提醒。联网搜索虽然强大,但它本质上依赖的是搜索引擎检索回来的网页内容质量。如果你在处理的是一个特别新、网络上讨论还很少的库或者框架,搜索引擎本身能检索到的高质量资料就有限,这时候联网搜索带来的提升可能没有你想象中那么大。另外,排查报错这件事,虽然联网搜索能帮你找到别人遇到过类似报错的讨论帖,但网络上的解决方案质量参差不齐,有时候你搜到的是一个已经过时或者根本不适用于你的具体场景的解决办法,这时候还是需要结合你自己的判断去甄别,不能完全依赖搜索结果照搬照抄。

接下来我们聊聊定价。文章里给出的信息是,联网搜索功能当时在 Claude 三点七 Sonnet、升级版的 Claude 三点五 Sonnet,以及 Claude 三点五 Haiku 上开放,价格是每一千次搜索十美元,再加上标准的 token 费用。这个定价模式我觉得值得稍微展开讲一讲,因为它直接关系到大家在实际项目里怎么去做成本预算和场景取舍。

这个计价方式本质上是两层叠加的,第一层是搜索本身的费用,按搜索次数计,每一千次十美元,平均下来一次搜索大概一分钱左右。第二层是标准的模型 token 费用,因为搜索回来的网页内容也是要作为上下文塞进模型里去处理和总结的,这部分内容本身也会消耗输入 token,生成的回答会消耗输出 token,这都是按照模型本身的标准价格另算的。

这个定价结构提示我们,如果你的应用场景需要频繁触发联网搜索,而且每次搜索拉回来的网页内容篇幅都很长,那你的实际成本可能不只是搜索次数那一块,token 消耗这一块累积起来也不可小看。所以我建议如果大家要把这个功能用到生产环境里,一定要先做一轮小流量的测试,实际跑一跑典型的用户问题,看看平均下来每次对话大概会触发几次搜索、每次搜索消耗的 token 量大概是多少,这样才能比较准确地预估整体成本,而不是只看搜索次数这个表面的单价。

另外结合前面讲到的 max_uses 这个参数,我觉得这两者要联动起来考虑。如果你把 max_uses 设得很高,理论上 Claude 可以做很多轮深入的渐进式搜索,信息会更全面更准确,但是对应的成本也会线性往上涨。这其实就是一个典型的成本和质量之间的权衡,没有放之四海而皆准的最优解,完全取决于你的具体业务场景对准确性的要求有多高,以及你的预算空间有多大。

说到这里,我想跟大家聊聊,如果你是一个开发者,拿到这个功能之后,具体应该怎么上手,以及有哪些坑需要提前避开。

第一步其实非常简单,文章里说了,只需要在调用 Messages API 的时候,启用这个网络搜索工具就可以了,不需要你自己去搭任何检索的基础设施,这个门槛比起以前自建 RAG 系统,确实是大幅降低了。但我想提醒大家的是,降低门槛不等于不需要做工程设计。你仍然需要根据自己的业务场景去合理配置 max_uses 这个参数,需要考虑是不是要用域名允许列表或者屏蔽列表来约束搜索范围,确保结果的质量和合规性。如果你是做一个对外的产品,还需要认真设计前端怎么展示引用来源,这部分用户体验的打磨,工具本身是不会替你做的。

第二个要避的坑,是不要把联网搜索当成万能药,什么问题都丢给它去搜。文章里其实也暗示了这一点,Claude 是会自己判断要不要搜索的,如果一个问题靠它本身的知识就能回答好,联网搜索反而是多余的开销,既增加了延迟,也增加了成本。所以作为开发者,你在设计提示词和系统逻辑的时候,可以适当引导,让 Claude 把搜索这个能力真正用在那些确实需要实时信息的问题上,而不是滥用。

第三个坑,是要对搜索结果的质量保持一份清醒的怀疑态度。即便有引用机制能让用户去核实来源,但这并不代表搜索回来的内容本身就一定权威可靠。互联网上信息质量参差不齐是常态,尤其是涉及金融、法律、医疗这类高风险领域的时候,我建议一定要配合域名允许列表,把信息来源锁定在你信任的权威渠道上,而不是放任 Claude 去搜索整个开放网络,然后指望它自己有足够的判断力去甄别真假。模型的判断力确实在进步,但在关键决策场景里,人工设置的护栏永远比单纯依赖模型自身的判断更可靠。

第四个我想提的,是关于今天我们站在二〇二六年六月这个时间点回看这篇文章的一个心态调整。这篇文章发布的时候,联网搜索还是一个相对新颖、值得专门写一篇博客来介绍的功能。但是放到今天,这种能力基本已经是大模型产品的标准配置了,用户甚至可能感知不到背后在不在用搜索,因为很多产品已经把这个判断和切换做得非常无缝。所以如果你现在在评估要不要用类似的功能,我觉得更应该关注的已经不是这个功能存不存在,而是它的执行细节做得怎么样,比如说搜索的准确率、多轮搜索的智能程度、引用的精确度,以及企业级控制功能是不是足够细粒度。这篇文章里提到的这些设计思路,放到今天依然是评估一个联网搜索功能好不好用的核心标准。

好,那我们来给今天这期节目做一个总结。这篇文章核心讲的是 Anthropic 在二〇二五年五月给 Claude 的 API 加上了联网搜索这个工具,让开发者不再需要自己搭建检索增强生成的基础设施,Claude 自己就能判断什么时候该搜、能做渐进式的多轮搜索去做更深入的研究、每个答案都带着可验证的引用来源,并且还给企业提供了域名允许和屏蔽列表这样的精细化管理工具。同时这个能力也延伸到了 Claude Code 里,让开发者在写代码的过程中也能用上最新的文档和信息。

我个人认为,这篇文章最值得我们记住的,不是某个具体的参数或者价格数字,这些细节随着产品迭代肯定会变,而是它体现出的一种产品设计哲学,就是把复杂的工程能力封装成一个简单的开关,同时把控制权和透明度留给开发者和企业管理员。这种把强大能力和精细控制结合在一起的思路,我觉得在未来很长一段时间里,都会是 AI 基础设施类产品设计的一个重要范式。好,今天的节目就到这里,我们下期再聊,再见。
