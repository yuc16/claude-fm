大家好,欢迎收听 Claude FM,我是你们的主播。

今天我们要聊的这篇文章,英文标题是 How and when to use subagents in Claude Code,翻译过来就是,在 Claude Code 里,什么时候用、怎么用子代理。这篇文章发表于二零二六年四月七日,出自 Claude 官方博客。距离今天,也就是二零二六年六月二十号,过去了差不多两个半月。这个时间跨度不算长,所以内容基本上还是新鲜的,你现在用 Claude Code 应该已经能完整体验文章里说的这些功能。可以把它当成一份还在保质期内的实战手册来听。

先说说为什么这篇文章值得花时间听一遍。如果你平时已经在用 Claude Code 写代码,大概会有这样的体验,一开始对话很顺畅,但是session拉长之后,响应速度会变慢。这背后的原因,文章一开头就点出来了,长会话会积累重量。你读过的每一个文件,探索过的每一个支线话题,半途而废的每一个想法,都会留在上下文窗口里,拖慢响应速度,也推高token的花费。这是几乎所有重度用户都会遇到的问题,而子代理,正是Anthropic给出的解法之一。

我们先把背景铺一下。文章举了一个具体场景,假设你在一个很大的TypeScript monorepo里开发一个新功能。主线任务是实现这个功能,但开发过程里总会冒出一堆旁支琐事。比如说,你得搞清楚现有的某个服务是怎么处理鉴权的,你得找到项目里负责日期格式化的公共工具函数在哪,你还得看看设计系统里是不是已经有一个组件,跟你现在要做的东西很接近。这些事情,单独拎出来看,都不需要整个项目的完整上下文,但如果你把它们全都塞进主对话里去做,主对话的上下文就会越堆越重,噪音也越来越大。文章这时候抛出一个问题,如果这些事情可以并行处理呢。这个问题的答案,就是子代理。

那子代理到底是什么。文章给出的定义是,子代理是一个独立的Claude实例,拥有自己的上下文窗口。它接到一个任务,自己去做,做完之后,只把结果带回主对话。文章用了一个我觉得特别形象的类比,子代理就像是Claude Code session里的浏览器标签页。你打开一个新标签页去追一个支线话题,不会打断你正在看的主页面,事情办完了,你把这个标签页关掉,只留下你需要的信息,主线浏览体验完全不受影响。

每一个子代理启动的时候都是从零开始的,它不会背负主对话积累下来的历史包袱,也不会继承你在主对话里已经加载过的那些skills。这意味着它的上下文是干净的,这一点后面讲到无偏见审查的时候还会再提到,因为这恰恰是它最大的价值之一。

另外子代理还可以并行运行多个,而且每一个子代理可以拥有不一样的权限。文章举的例子是,一个负责调研的子代理可能只给它只读权限,它只能看代码不能改代码,而一个负责具体实现的子代理,可以给它完整的编辑权限。这种权限分层其实挺重要的,你可以把危险动作锁定在你明确授权的那个子代理身上,调研类的子代理天然就更安全,因为它根本没有改东西的能力。

Claude Code内置了几种现成的子代理类型。第一种叫general purpose,通用型,用来处理复杂的多步骤任务。第二种叫plan,计划型,它会先去研究代码库,然后再给你呈现一套实现策略,适合那种你还没想清楚怎么做、需要先有人帮你摸清楚地形的场景。第三种叫explore,探索型,它专门针对快速的、只读的代码搜索做了优化,如果你只是想找一个函数定义在哪、一个模式在项目里出现过几次,explore类型的子代理速度会很快。

Claude Code经常会自己主动派生子代理来处理任务,不需要你每次都手动指挥。但文章想强调的是,你也完全可以显式地指挥这个行为,甚至预先定义好可复用的专家角色,让Claude以后自动委派给它们。知道什么时候该主动伸手去用子代理,才是真正让这个功能发挥价值的关键。

接下来文章给出了五类明显能从子代理委派里受益的工作场景,我们一个一个拆开讲。

第一类场景,是当你需要靠探索代码库来理解某个东西怎么运作,作为修改它的前提条件的时候。信号很明确,就是收集上下文需要读几十个文件。让子代理去做这件探索工作,它读完之后只把一份综合摘要带回来,而不是把原始内容一股脑倒进主对话。好处是主对话保持干净,拿到的是经过提炼的发现。打个比方,这就像让助理去图书馆查资料,你要的是一份读书报告,不是让他把十本书原封不动搬回办公室。

第二类场景,是当你需要修复多个文件里的错误,或者在多个组件里更新同一种模式,而且这些子任务彼此之间没有依赖关系的时候。这里信号是子任务互相独立,好处是并行的子代理通常能更快完成整个任务。三个子代理同时跑,通常比一个接一个地做要快得多。这个道理很直观,就像刷三面墙的油漆,三个人各刷一面同时开工,肯定比一个人刷完一面再刷下一面快。前提是这三面墙之间互不影响,你才能这么分工。

第三类场景,是当你的目标是对一个实现做一次不带偏见的审查的时候。信号是,你需要的验证不应该被对话历史所影响。好处是能拿到更干净、更客观的反馈,因为全新启动的子代理不会继承主对话里积累下来的假设、上下文或者盲点。文章这里提到,斜杠clear这个命令也能重置上下文,给你全新视角,但代价是历史彻底丢掉。子代理的好处是,你能拿到同样新鲜的视角,同时主对话完整保留。

第四类场景,是在最终敲定改动之前,需要有人提供第二意见。信号是提交代码之前需要一次独立验证。好处是能抓住那些因为太熟悉代码而被自己忽略掉的问题。比如你刚写完一个支付流程的实现,自己心里清楚这段代码怎么演变过来、否决过哪些方案,但正因为太清楚,反而可能对隐藏的边界情况视而不见,或者下意识觉得实现已经覆盖了所有测试用例,实际上只是恰好迎合了测试,没有真正解决问题。独立子代理不知道这些来龙去脉,反而能从外部视角发现被忽略的东西。

第五类场景,是当一个任务本身分成明显不同的阶段的时候,比如先设计,再实现,再测试。信号是,有清晰交接点的连续阶段。好处是,每个子代理可以专注在自己负责的那一个阶段上,不会被别的阶段的上下文干扰。设计阶段的子代理不需要知道实现细节,实现阶段的子代理也不需要被设计过程中那些被否掉的备选方案分心。

文章还给了一个判断的小窍门,我觉得特别实用,可以直接当成你日常的判断标准。如果一个任务需要探索十个以上的文件,或者涉及三个以上互相独立的工作模块,这就是一个很强的信号,提示你应该主动去用子代理。这个数字门槛挺好记的,十个文件,三个独立模块,超过这个量级,基本上就该考虑分工了。

讲完了什么时候该用,我们再来看具体怎么用。文章把调用子代理的方式分成了五个层级,从最简单的随口一说,到最自动化的事件触发,自动化程度是逐级升高的。文章建议你按照工作流的实际需要来选起点,随着你发现的模式越来越清晰,可以逐步往更自动化的方向叠加。

第一个层级,也是最灵活、最容易上手的方式,就是直接在对话里用自然语言让Claude去用子代理。这个方式在所有的Claude Code界面里都能用,无论你是在终端里,还是在VS Code、JetBrains这些IDE插件里,还是在网页版或者桌面客户端,统统适用。

文章给了几个可以直接套用的自然语言模式。比如说,用一个子代理去探索这个代码库里鉴权是怎么运作的。或者,找一个独立的agent去复查这段代码有没有安全问题。或者,并行地研究这件事,同时检查API路由、数据库模型,还有前端组件。或者,派几个子代理去修复这几个不同包里的TypeScript错误。

文章特别强调,表达要明确具体。要指定范围,要在任务彼此独立的时候明确要求并行执行,还要描述清楚你期望得到的输出是什么样的。文章给了一个我觉得写得很标准的prompt结构示范,大意是,用子代理并行探索这个代码库,第一步找出所有的API端点并总结它们的用途,第二步识别数据库的模式和关系,第三步梳理鉴权的完整流程,最后要求返回每一项的摘要,而不是完整的文件内容。这个prompt之所以有效,是因为它清楚地定义了三个互相独立的任务,明确要求并行执行,而且具体说明了输出格式。Claude能准确理解你的意图,然后派生出合适的子代理去做。

文章还给了几条让对话式调用更有效的小技巧。第一条,任务范围要划清楚。说探索一下支付是怎么运作的,比说探索一下所有东西,效果好得多。第二条,要明确要求并行。直接说这几件事可以并行,或者同时处理这三件事。第三条,说清楚你想要回来的是什么。是摘要,是具体发现,还是建议,把输出格式说出来,能帮Claude把结果交付得更准确。第四条,在你需要无偏见分析的时候,主动要求全新的上下文。比方说,你可以直接说,用一个没看过我们之前讨论的子代理,这样就能确保是一次干净的评估。

这里还有一个使用上的小窍门值得一提。如果一个子代理跑的时间比较久,你可以用控制键加B这个快捷键,把它送到后台去跑。这样主对话可以继续往下推进,等子代理跑完了,结果会自动浮现出来。如果你想看现在后台都有哪些任务在跑,可以用斜杠tasks这个命令查看。这个设计其实挺贴心的,意味着你不需要干等着一个慢任务跑完才能继续手头的工作,可以先去做别的事情,系统会在合适的时候把结果递给你。

第二个层级,是自定义子代理。如果你发现自己反复在请求同一种类型的子代理,比如说总是要一个安全审查员,总是要一个测试撰写者,或者总是要一个文档校对员,这种重复出现的需求,就值得把它定义成一个固定的自定义子代理。定义好之后,Claude会在任务匹配它的描述时自动委派过去,不需要你每次都重新提示一遍。

自定义子代理是以markdown文件的形式存在的,可以放在项目级别的目录里,这样能跟团队共享,也可以放在用户级别的目录里,这样在你所有项目里都能用到。每一个自定义子代理都有自己独立的系统提示词,自己的工具权限,还可以选择性地指定自己使用的模型。最简单的创建方式是用斜杠agents这个命令,它会带你走一遍交互式的设置流程,甚至可以根据你的描述直接生成第一版草稿。当然你也可以手写这个文件。

文章举了一个安全审查员的例子。描述写的是,审查代码改动中的安全漏洞、注入风险、鉴权问题和敏感数据暴露,要求在涉及鉴权、支付或用户数据的提交之前主动使用它,工具权限只给只读类的。系统提示词要求它检查SQL注入、跨站脚本、命令注入、鉴权授权漏洞、日志和响应里的敏感数据泄漏、不安全的依赖或配置,最后返回按优先级排序的问题清单,带上文件和行号以及修复建议,态度要严格,什么都没发现就明确说没发现,不要编造问题。

配置好之后,Claude会自动把匹配的工作路由给这个子代理,你也可以直接按名字调用它,比如说,让security reviewer这个子代理去看一下当前暂存的改动。

文章里有一个我觉得非常关键的提示,他说description这个字段,是Claude用来判断什么时候该委派的依据。所以写description的时候,重点要放在触发条件上,而不是泛泛地描述这个子代理有什么能力。文章举的对比例子是,写在提交前审查代码里的安全问题,这种描述的路由效果,会比单纯写一个安全专家这种描述好得多。前者讲清楚了什么时候用,后者只讲了这是个什么角色。这个区别其实挺微妙的,但很关键,因为Claude做委派决策的时候,靠的就是这段描述文字去匹配场景,如果你写得太抽象,匹配就会变得不准。

第三个层级,是CLAUDE.md文件。自定义子代理定义的是有哪些专家可以用,而CLAUDE.md定义的是什么时候该用这些专家,也就是规则本身。Claude会在每次对话开始的时候读取这个文件,所以只要规则写在里面,行为就会在所有会话、所有团队成员之间保持一致,不需要每个人都记住要主动去要求。

文章举的例子是一条代码审查的标准,内容大意是,只要有人要求审查代码,就必须始终使用一个只有读取权限的子代理,这个子代理应该永远检查安全漏洞、性能问题,还要检查是否符合项目架构文档里规定的模式,最后返回结果的时候,要按优先级排好序,并且带上文件和行号。把这条规则写进CLAUDE.md之后,以后每一次代码审查请求,都会自动套用这套标准,不需要每次都重新讲一遍要求。

第四个层级,是skill,也就是技能。对于那些会反复运行的、复杂的多步骤工作流,skill提供了一个可复用的接口。你在项目里的skills目录下定义一次,以后既可以用斜杠加技能名字直接调用,也可以让Claude在任务匹配技能描述的时候自动加载。

skill跟CLAUDE.md的区别在于作用范围。CLAUDE.md是永远加载的,塑造的是每一次交互。skill是按需加载的,要么是你显式调用,要么是Claude把当前任务和skill的描述字段匹配上了才会加载。这就让skill成了那种应该随时可用、但不应该套用到每一次对话上的工作流的合适归宿。

文章举的例子是一个叫deep review的技能,用来做全面的代码审查。它的工作方式是,对暂存的改动跑三个并行的子代理审查。第一个做安全审查,检查漏洞、注入风险、鉴权问题、敏感数据暴露。第二个做性能审查,检查N加一查询问题、不必要的循环、内存泄漏、阻塞操作。第三个做风格审查,检查是否符合项目风格指南里记录的规范。最后把三方的发现综合成一份单一的摘要,按优先级排好问题,每一条都带上文件、行号和建议的修复方式。这样设计之后,你输入斜杠deep review,就能在暂存改动上触发这套三路并行分析。而且因为描述里提到了在提交或者发起PR之前审查暂存改动这个场景,Claude在合适的上下文出现时,也会主动联想到去用这个技能。

这里还有一个小细节值得一提,skill本质上是一个目录,而不是单个文件。除了主文件之外,它还可以带模板,让Claude去填充;可以带示例输出,展示期望的格式;还可以带脚本,作为工作流的一部分被Claude执行。这跟之前那种老式的命令格式不一样,老式格式是单个扁平文件,所有东西都得硬塞进提示词本身里面去,没有办法附带额外的资源。

第五个层级,是hooks,也就是钩子。钩子是用户定义的shell命令、HTTP端点,或者LLM提示词,它们会在Claude Code生命周期的特定节点上自动执行。钩子可以基于事件来自动化子代理的工作流,触发在特定动作上,不需要任何人手动去调用。

文章举的例子是一个Stop钩子,作用是在测试没通过之前阻止Claude结束这一轮发言。Claude发言完毕,Stop事件触发,脚本跑一下测试套件,失败就返回结构化结果告诉Claude Code这一轮要被阻止,并附上原因。Claude Code读到阻止信号后,把原因当成新指令反馈回对话,让Claude继续干活。脚本里还有个小机制防止无限循环,如果这一轮已经因为同一个钩子被阻止过,就直接放行,避免死循环。

文章最后总结说,钩子代表的是子代理编排里最自动化的方式。对话式调用,或者CLAUDE.md里写好的规则,是更适合作为起点的方式。钩子是工作流足够成熟之后,才应该上的那一层。

讲完了五个调用层级,文章给了四个具体的应用模式,正好对应前面讲的那五个该用的信号,我们快速过一遍。

第一个模式,是在给不熟悉的代码添加新功能之前,先把调研工作委派给子代理。比如说在实现用户通知功能之前,先让子代理研究一下,现在邮件是怎么发出去的,已经存在哪些通知相关的模式,新的通知逻辑应该放在代码架构里的哪个位置,总结回来之后再一起规划实现。这样拿到的是一份提炼过的总结,而不是二十个文件的原始内容。

第二个模式,是当同一种模式需要在多个文件里更新的时候,用并行的子代理同时处理。比如说要在用户、订单、商品这三个接口文件里统一更新错误处理逻辑,都遵循鉴权模块里已经确立的范式,让三个子代理同时开工,完成时间大概跟做一个文件差不多,而且互不干扰。

第三个模式,是在实现了比较复杂的东西之后,用一个没被实现过程影响过的子代理来做验证。比如说实现完支付流程之后,让一个全新的、只读权限的子代理来复查,明确告诉它不要看之前的讨论,重点检查安全漏洞、没处理到的边界情况和错误处理疏漏,态度要严格。这个子代理不知道当时权衡过哪些方案、否决过哪些思路,这种外部视角恰恰能揪出主对话因为太熟悉而忽略掉的问题。

第四个模式,是针对多阶段任务,把子代理串联起来,阶段之间有明确交接。比如把一个功能拆成流水线,第一个子代理设计API契约写入文档文件,第二个基于这份规格实现后端接口,第三个针对实现写集成测试,每个阶段先完成下一个才开始,用输出文件作为交接机制。设计阶段不被实现细节分心,实现阶段从干净的规格说明开始,测试阶段独立评估最终结果。

讲到这里,你可能已经觉得子代理听起来挺好用的,几乎想把所有事情都交给它处理。但文章接下来用了相当篇幅,专门讲清楚子代理也是有开销的,什么时候不该用它,这一部分我觉得跟前面五个该用的信号同等重要,甚至更重要,因为滥用往往比不用造成的浪费更大。

文章指出,每一个子代理都要启动自己的上下文,消耗自己的token,而且会在你和实际工作之间多加一层间接性。只有当上下文隔离、并行,或者全新视角真正能带来价值的时候,这份开销才值得付出。

文章列了五种场景,建议留在主对话里处理更简单。

第一种,是连续且互相依赖的工作。如果第二步需要第一步的完整输出,第三步又同时需要前两步的结果,这种链条式的任务,用一个会话从头处理到尾,通常比让多个子代理通过文件传递状态、像接力赛一样轮流交接更干净。子代理之间没法直接对话,所有的交接都得靠主对话去搬运或者靠落盘的文件去传递,这种方式对强依赖的连续任务来说,反而会增加摩擦。

第二种,是对同一个文件做编辑。两个子代理同时改同一个文件,几乎注定会产生冲突。这种紧密耦合的改动,应该留在同一个上下文窗口里完成,不要拆给不同的子代理。

第三种,是规模很小的任务。对于一次快速修复,或者一个很聚焦的问题,委派出去的开销会超过它带来的收益。这种情况下,直接在主对话里问,或者直接动手改,效率更高。

第四种,是专家子代理定义得太多。给每一件事都定义一个自定义子代理,听起来很诱人,但是一旦选项太多,Claude在做自动委派判断的时候,反而会变得不可靠。文章提到,大多数团队最后会安定在几个范围划分得很清楚的子代理上,而不是搞出一长串名目繁多的专家清单。

第五种,是需要子代理之间互相协调的工作。这一点是整篇文章里特别重要、也容易被忽略的一条边界。子代理只会向主对话汇报结果,彼此之间没法直接交流。如果任务恰好需要几个子代理互相沟通、确认对方做的事情,那就不是子代理该解决的形状了,文章建议换成agent teams,也就是代理团队。代理团队让多个子代理在彼此独立的会话之间协调,而不是局限在一个会话内部,代价是更重、更贵。

文章在收尾的地方说,子代理只有在被刻意使用的时候,才能发挥出它全部的价值。Claude自动触发子代理当然有帮助,但真正决定结果好坏的,是你有没有主动意识到什么时候该让它去调研、什么时候该并行、什么时候该要一次全新的视角,而不是把决定完全交给系统随缘发挥。最后的建议是,先从对话式提示词起步,留意哪些请求反复出现,等模式清晰之后再沉淀成更自动化的配置,目标是让委派变得毫不费力,注意力始终留在真正重要的工作上。

听到这里,我们说说这篇文章对日常工作的实际意义。如果你还没认真用过子代理,建议从最轻量的对话式调用开始试。下一次碰到需要先搞清楚现状的旁支任务,试着说一句,用一个子代理去查一下这个东西现在是怎么实现的,你会很快感觉到主对话变得清爽很多,因为探索过程中产生的中间噪音根本没机会进到主上下文里。

另一个实用习惯,是需要并行处理几个独立任务的时候,主动说出来,明确告诉Claude这几个任务可以同时进行。我个人认为这是最容易被忽略、但收益又最直接的一条建议。很多时候Claude有能力把任务拆开并行处理,但如果你不明确说出来,它未必会主动这么做,你得先意识到这几个任务之间没有依赖,系统不会替你做这个判断。

如果团队已经形成某些固定的审查习惯,比如每次提交前都要做安全审查,这种重复出现的规则,建议尽早沉淀成自定义子代理,配合CLAUDE.md一起用,这样能让整个团队的审查标准保持统一,不会因为某个人忘了多说一句要求,力度就打了折扣。

但也要提醒一下,千万别一上来就想把所有东西都做成自定义子代理或钩子。过多的专家子代理会让自动委派变得不可靠,这一点特别值得警惕,描述之间互相重叠模糊,Claude路由的时候反而会犹豫不决。比较稳妥的做法是先观察一段时间,挑出真正高频出现的那几个核心场景,做成少而精的子代理,而不是贪多求全。

还有一点值得拎出来说,就是子代理和代理团队的边界。如果你发现自己开始设想,要让一个子代理把进度同步给另一个子代理,或者互相确认对方的产出,这就是个信号,提示你已经走出了子代理原本要解决的问题形状。子代理彼此隔离、只向主对话汇报,这是它保持轻量、便宜的原因之一,一旦需要协作沟通,就该转向代理团队这个更重的工具。

最后,我想用几句话总结一下今天讲的这篇文章,再说说我自己的看法。

这篇文章的核心,是把子代理从一个听起来很酷炫的新特性,落地成一套可以直接照搬的判断方法。什么时候该用,看五个信号,大量文件调研、并行无依赖、需要无偏见视角、需要二次意见、多阶段任务。什么时候不该用,看五种反例,强依赖链条、同文件编辑、小任务、专家子代理太多、需要互相协作。怎么用,从对话式调用起步,逐步往自定义子代理、CLAUDE.md规则、skill、钩子叠加,自动化程度依次升高。

我个人认为,这篇文章最值得借鉴的,不是子代理这个功能本身,而是它背后那种从手动到自动逐步演进的设计思路。先用最简单的对话式提示词验证一个模式是否真的有效,等模式反复出现、确实值得固化,才沉淀成配置和规则,最后才用钩子做成完全自动触发。这种循序渐进的节奏,放在很多工具设计甚至团队流程设计上,都值得参考,不要一上来就想把所有东西都自动化,先靠对话验证,再靠配置固化。
