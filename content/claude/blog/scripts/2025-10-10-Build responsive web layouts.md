那这篇文章出现的背景是什么呢。二〇二五年下半年,正好是 Claude Code 这个命令行工具从早期尝鲜阶段走向被大量工程师日常使用的一个阶段。Anthropic 在那段时间发了不少面向具体开发场景的实战文章,告诉大家除了写后端逻辑、读代码库,Claude 在前端这种看起来很"手感导向"的领域里也能派上用场。响应式布局正是前端里最让人头疼,又最考验经验的一个环节,所以这篇文章选这个题材,其实挺有代表性的。

好,我们正式开始。先说说响应式布局这个问题到底难在哪。

我们平时做网页,肯定都遇到过这种场景。你在自己的笔记本电脑上,把一个三栏的卡片布局做得漂漂亮亮,字体大小合适,间距舒服,看起来完美无缺。结果同事拿平板打开一看,卡片挤在一起,文字溢出了容器,导航栏的几个按钮还互相重叠了。这种问题往往不是在你写代码的时候发现的,而是在测试阶段才暴露出来,运气不好的话,甚至是上线之后用户反馈才发现的。

文章里说,这个挑战的核心不只是让布局在不同屏幕尺寸下都能用,更难的是,你要预判 flexbox、grid 这些布局技术,跟媒体查询组合在一起,在各种各样的设备尺寸下会怎么互相影响。这种预判能力,说白了就是经验,而经验是需要好几年时间才能攒出来的。

那大部分开发者现在是怎么做的呢。基本上是一个不断试错、不断迭代的过程。先写好基础样式,然后在几个常见的断点上加媒体查询,接着手动在各种设备上测试,发现哪里坏了就修哪里,改完再测一遍,循环往复。这个流程是能跑通的,但它特别耗时间,而且即便你测得再仔细,总还是会有一些边界情况漏掉。

文章接下来拆解了这个传统流程里的四个具体环节,我觉得这部分讲得特别细,值得我们一个一个过一遍。

第一个环节是手写媒体查询。大家应该都写过类似的代码,先给一个容器设置一个百分之百宽度,加点内边距,然后用 at media min width 七百六十八像素这种语法,在平板尺寸下把内边距调大一点,再用一千零二十四像素这个断点,在桌面端给容器加个最大宽度并且居中。这种写法的好处是,你对布局在每个具体宽度下的表现,有非常精确的控制权。

但问题也出在这里。选断点这件事,听起来好像有标准答案,比如七百六十八是平板,一千零二十四是桌面,但文章里特别提到,真正合适的断点应该是由你的具体内容决定的,而不是某个设备分类表格里抄来的数字。举个例子,一个导航菜单可能需要在九百二十像素这个位置加断点,不是因为哪个设备规格手册上写了这个数字,而是因为恰好在九百二十像素的时候,你的菜单项开始换行了。这种断点是要靠肉眼观察、反复调试才能找出来的,没有捷径。而且随着项目越做越大,样式表里到处都是被覆盖、被重写的规则,维护起来会越来越吃力,你经常会搞不清楚到底是哪条规则在某个宽度下生效了。

第二个环节是设备测试。写完媒体查询之后,开发者一般会打开浏览器的开发者工具,切到响应式设计模式,把预设的几个设备尺寸都点一遍看看效果。Chrome 自带的这个工具确实方便,很多团队也会专门留一批真实的手机和平板,挨个测一遍在 iOS Safari、安卓 Chrome,以及不同屏幕密度下的渲染效果。

但文章里点出了一个挺关键的事实,浏览器里的模拟器只是一个近似值,它没法完全还原真实设备的行为。我自己也深有体会,一个触摸按钮在桌面端的开发者工具里看着挺大,放到真手机上摸起来却感觉很挤,因为模拟器很难还原真实的触摸精度感受。还有更隐蔽的问题,比如说手机屏幕上的刘海、动态变化的工具栏、还有软键盘弹出来之后,视口高度会发生各种意想不到的变化,这些情况模拟器经常是完全模拟不出来的。文章举的例子特别生动,一个固定定位的页脚,在 Chrome 的设备模拟器里看着好好的,可是真到 iOS Safari 上,地址栏展开的时候,这个页脚可能就把重要内容给盖住了。

设备实测确实能抓到很多真实世界的渲染问题,但这个过程太耗时间了。每改一次布局,就要在六到八台设备上重新过一遍,这种摩擦力会拖慢整个开发节奏,而且即便你测了再多设备,用户手里实际拿着的设备和浏览器组合是成百上千种,你永远测不全。

第三个环节是框架工具类。像 Bootstrap 和 Tailwind 这种现代 CSS 框架,提供了一套响应式的类名系统,把媒体查询的底层逻辑给抽象掉了。你只需要写 col md six 或者 lg flex row 这样的类名,框架就会在预设的断点上自动套用对应的样式。比如说一个 grid 容器,默认是单列,加一个 lg grid cols three,就能在大屏幕下变成三列。

这种方式确实能加快开发速度,因为断点的约定是统一的,而且很多响应式模式已经被框架提前测试过了。但代价也很明显,你会被锁定在框架预设的断点体系里,这套体系未必跟你项目的具体设计需求对得上。文章举的例子是,框架的平板断点可能是七百六十八像素触发,但你的内容实际上是在八百五十像素的时候才需要调整。想要自定义这些断点,通常得改动整个项目级别的配置文件,这一改,会影响到代码库里所有用到响应式工具类的地方,牵一发动全身。

另外框架还有一个副作用,就是它把底层的 CSS 给藏起来了,这会让调试变得更麻烦。当一个 Tailwind 布局在某个具体宽度上出问题的时候,你得先把这些工具类名翻译回它们对应的原始 CSS 属性,再去搞清楚这些属性之间是怎么互相作用的,这中间多了一层转译成本。

第四个环节,就是布局出问题之后的调试过程。这时候你会打开开发者工具,切到响应式设计模式,然后手动拖动视口宽度,一点一点找到布局到底是在哪个像素值上开始崩的。然后再去检查元素、查看计算后的样式,追溯父容器的约束条件是怎么影响子元素的。

文章里说,浏览器开发者工具对响应式调试来说是必不可少的,但要找到布局问题的根本原因,你得对 CSS 的优先级规则、盒模型,以及 flexbox 或 grid 在不同约束条件下的行为方式有很深的理解。一个在八百四十三像素这个具体宽度溢出的布局,问题往往不是某一条 CSS 规则单独造成的,而是父容器宽度、子元素内边距,还有字体大小计算这几个因素叠加在一起,只有组合到一起才会浮现出来的问题。而且这个调试过程是一轮一轮来的,每修复一处,你都要重新在多个视口宽度下测一遍,确认这个修复没有在响应式范围里的别的地方引发新问题。

听到这里,大家应该能感受到,传统的响应式开发流程,本质上是一个高度依赖经验积累、又特别消耗时间精力的体力活。那文章接下来要讲的,就是怎么把 Claude 整合进这个工作流里,来打破这种纯靠试错的局面。

文章里讲到,Claude 在这个场景下分两种使用方式,分别对应两种不同的需求。

第一种是 Claude.ai,也就是我们平时打开浏览器就能用的网页版界面。你直接在对话框里描述布局需求,然后就能拿到完整的响应式 HTML 和 CSS 代码,里面会包含正确的视口配置和媒体查询。这个方式不需要安装任何东西,随便打开个浏览器就能用,特别适合用来学习响应式设计模式,或者快速生成一些样板代码。

第二种是 Claude Code,这是一个命令行工具,它能分析你整个代码库里现有的样式表,识别出哪些地方存在跟视口相关的问题,然后直接在你的文件里把修复方案实现出来。这个方式需要用 npm 安装,还需要配置 API 访问权限,它是专门为在大型代码库里重构响应式布局这种场景设计的。

我觉得这个分工挺清晰的,一个是面向快速验证和学习的轻量级工具,一个是面向系统性重构的重型工具。我们分别来看看具体怎么用。

先说 Claude.ai 这一侧。文章强调了一个挺实用的思路,就是在你真正动手写几个小时的媒体查询、或者大动干戈重构布局之前,先用 Claude.ai 验证一下某种设计模式在不同设备上到底适不适用。这种即时反馈,能让你在开发过程中就把响应式相关的决策给确定下来,而不是等到测试阶段、甚至发布之后才发现布局是坏的。

具体怎么操作呢,你就直接用自然语言描述你的布局需求,Claude.ai 会返回带有正确视口元标签、移动优先 CSS,以及语义化 HTML 结构的可用代码。这个初版生成的代码,本身就是一个响应式技巧的范例,你可以根据自己项目的具体需求再去调整。

文章举了几个常见的请求例子,比如说创建一个三栏卡片布局,在移动端自动变成单栏,或者构建一个导航栏,在七百六十八像素以下自动折叠成汉堡菜单,再或者生成一个带响应式侧边栏的仪表盘,侧边栏在平板上自动隐藏。这些请求拿到的代码,都是可以直接拿来当学习范例、再往自己项目里改的。

文章还给了一个更具体的示例。假设你提的需求是,做一个产品落地页,要有固定头部、一个英雄区、三个特性卡片,还有一个页脚,卡片在桌面端横向排成一行,在移动端自动堆叠。Claude.ai 返回的代码里,头部用 position fixed 固定在顶部,特性卡片那部分默认是单列网格,在七百六十八像素以上变成两列,在一千零二十四像素以上变成三列,而且代码里还会带上解释断点选择理由的注释。这一点我觉得特别贴心,等于是边给你代码边给你讲课,你不光拿到了能用的东西,还顺便理解了为什么要这么写。

除了直接生成代码,文章还提到一个我觉得很有价值的用法,就是问 Claude.ai 为什么。比如你可以问,这个导航栏为什么用带 flex wrap 的 flexbox,而不是用 CSS Grid。Claude.ai 会把不同布局方法之间的权衡给你拆开讲清楚,包括浏览器兼容性的考量、布局灵活性,以及在什么场景下该选哪种方法。这种交互方式的价值在哪呢,它不是单纯让你复制粘贴一段代码,而是帮你把底层原理给搞明白,这样下次遇到类似场景,你自己就能判断该用哪种方案了,这其实是在帮你积累那种"需要好几年经验才能攒出来"的判断力,只是把积累的过程压缩了。

接下来我们说说 Claude Code 这一侧,这部分我觉得是文章里最有实战价值的内容。

当响应式问题不是孤立的一两个组件,而是横跨多个样式表、影响到整个应用的时候,Claude Code 就承担起一个系统性重构伙伴的角色。跟网页版工具不一样,Claude Code 会去探查你项目的文件结构,在多个文件里识别出跟视口相关的问题,然后提出在保持现有架构不变的前提下的修复方案。

安装方式文章里给出的是用 npm 全局安装一个叫 anthropic claude code 的包,装好之后在项目目录下输入 claude 就能启动。这里我要提醒一下,具体的安装命令和包名,大家实际操作前最好去核实一下最新的官方文档,因为工具版本一直在更新,我们这期节目录制的时候,文章本身已经是八个月前的内容了。

文章里给了一个非常具体的场景演示,我觉得特别值得展开讲讲,因为它完整还原了从发现问题到解决问题的整个链路。

假设你有一个产品仪表盘,在平板视口下整个布局都坏了,导航栏跟内容区重叠,卡片没法正常重新排列,侧边栏把主内容区域直接挤出了屏幕外。这是一个很典型也很让人头大的真实场景,我相信很多做前端的同学都遇到过类似的崩溃现场。

Claude Code 处理这个问题的第一步,是扫描项目里的 layout css 文件,识别出那些固定宽度的样式和会在小屏幕上导致溢出的布局模式。文章里说,它能找出三个具体的问题,而且每个问题都带着精确的代码行号和修复建议。这一点跟我们刚才说的传统调试流程对比一下,人工调试要靠拖动视口宽度、一点点试出问题出现的临界点,而 Claude Code 是直接扫描代码定位问题,效率上是完全不同的量级。

第二步,它会去更新 CSS,把固定宽度替换成响应式的替代方案,比如说用 max width 代替固定宽度,用 flex basis 代替固定比例,用带 auto fit 的 grid template columns 来实现自适应列数,同时还会针对具体的断点添加相应的媒体查询调整。

第三步,也是我觉得特别值得点出来的一步,Claude Code 会在三百二十像素和五百一十二像素这两个具体的视口宽度下测试更新后的样式,确认没有出现横向溢出的问题,并且布局在每个断点上都能正常响应。这意味着它不只是改完代码就算了,还会主动验证修复是否真的生效,这正好补上了人工调试流程里那种"改完一处还要重新全部测一遍"的繁琐环节。

第四步,文章里说 Claude Code 还会生成一套完整的 Playwright 测试套件,针对真实设备尺寸做验证,包括 iPhone SE、iPhone 十二、iPad、iPad Pro 还有桌面端,目的是为了防止未来再出现类似的回归问题。这一点我个人觉得特别关键,因为它把这次修复的成果,变成了一份可以长期复用的回归测试资产,以后哪怕代码改动了,这些测试也能帮你及时发现新的响应式问题,而不是等用户反馈了才知道。

好,讲完了这两个工具具体怎么用,我们来聊聊这对大家实际工作意味着什么,也就是这一期我特别想跟大家展开聊的实践应用部分。

如果你是个人开发者,或者正在做一个新项目的早期阶段,我建议的工作方式是,先用 Claude.ai 来做快速原型。你不需要任何配置和安装,打开浏览器就能描述你的布局需求,比如说让它给你做一个三栏卡片网格,在移动端自动堆叠成单列。拿到生成的代码之后,你可以直接在浏览器里跑起来看效果,如果有不满意的地方,接着用自然语言继续追问和调整,比如问它能不能把断点改到某个具体宽度,或者把卡片之间的间距调整一下。这种交互方式特别适合用来探索设计方案,因为传统流程里你想验证一个布局想法,得自己写完整套 CSS,现在你可以先让 Claude.ai 给你一个可运行的版本,看看实际效果再决定要不要往下做。

而且我建议大家在用 Claude.ai 生成代码的同时,多问几个为什么。这不只是为了拿到能跑的代码,更是为了把背后的设计逻辑学到自己脑子里。比如说,你可以追问,这个断点为什么选在这个数值,如果换成另一种布局方式会有什么区别,这种方式跟另一种方式相比浏览器兼容性怎么样。这种刻意的提问习惯,长期下来,会让你自己对响应式设计的判断力越来越强,而不是变成一个只会复制粘贴 AI 生成代码的人。

如果你是在维护一个已经存在好几年的大型项目,样式表散落在好几十个文件里,各种历史遗留的写死宽度、互相覆盖的媒体查询,这种情况下我觉得 Claude Code 的价值会体现得特别明显。你可以直接把项目某个具体的响应式问题描述给它,比如说仪表盘在平板上的布局错乱,然后让它去扫描整个项目的样式表,定位问题所在的具体文件和行号。这种系统性排查的效率,跟人工一个文件一个文件去翻、一个断点一个断点去试相比,完全不是一个量级。

我个人觉得有一点特别值得注意,就是文章里提到的那个自动生成回归测试的环节。很多团队在修复响应式 bug 的时候,往往只关注眼前这个问题解决了没有,很少会顺手把这次修复变成一份可以长期使用的自动化测试。如果 Claude Code 真的能在修完之后,自动生成针对多个真实设备尺寸的 Playwright 测试,这其实是把"一次性修复"升级成了"长期防御",对于团队的工程质量来说是个挺大的加分项。我建议大家在用 Claude Code 修复响应式问题的时候,可以养成一个习惯,修完之后明确要求它顺带生成对应的回归测试,而不是等它自己想起来。

当然这里也要提一个要避的坑。无论是 Claude.ai 还是 Claude Code 生成的响应式代码,它们给出的断点和判断,本质上还是基于通用经验和你描述的需求推出来的,并不是凭空知道你的产品具体在哪个像素宽度下视觉上会出问题。文章里反复强调的一个观点是,真正合适的断点是由内容决定的,不是由设备型号表决定的。所以即便用了这些工具,你仍然需要拿真实的内容和真实的视觉效果去验证生成出来的断点是不是真的贴合你的场景,工具能帮你跳过最枯燥、最重复的那部分体力劳动,但最终拍板这个断点合不合适,还是需要你自己用眼睛去看一看效果。

另外,我也想提醒一句,浏览器模拟器和真机测试之间的鸿沟,这个问题工具本身解决不了。无论 Claude Code 生成的测试覆盖了多少种设备尺寸,这些测试本质上还是在模拟环境里跑的,真实设备上那些动态工具栏、软键盘弹出之类的边缘情况,有时候还是得靠你拿着真手机点一点才能彻底放心。我个人的建议是,把 AI 工具定位成大幅缩短第一轮排查和修复时间的助手,而不是彻底替代真机验证这一步,这二者应该是配合关系,不是替代关系。

好,我们来给这一期做个总结。这篇文章的核心观点是,响应式布局开发长期以来是一个高度依赖经验、又特别耗时的过程,从手写媒体查询、断点选择,到设备实测、框架工具类,再到出问题之后的调试,每一个环节都有它自己的痛点和门槛。Claude 在这里提供了两种不同侵入度的解决方式,Claude.ai 适合用来快速生成原型、验证设计想法、顺便学习响应式技巧背后的原理,而 Claude Code 则适合用在已经有一定规模的代码库里,做系统性的问题排查、修复,以及顺手生成长期可用的回归测试。

我个人的点评是,这篇文章虽然篇幅不长,但它把响应式开发这个老问题拆解得很扎实,没有夸大其词地说 AI 能完全取代经验判断,而是诚实地把它定位成一个能大幅压缩试错周期的工具。我觉得这种克制的表达方式反而更可信。对于我们平时写代码的人来说,真正值得带走的,不是说从今天起再也不用懂 CSS 了,而是说我们可以把原本花在反复试错和手动测试上的大量时间,省下来去做更有判断力的设计决策,这才是这类工具真正该发挥作用的地方。
