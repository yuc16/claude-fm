大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是 Discover tools that work with Claude,中文意思大概是发现能和 Claude 配合使用的工具。这篇文章发表于二零二五年七月十四号,来自 Anthropic 官方博客。

先说一句,这篇文章距离今天已经差不多快一年了,二零二六年六月二十一号的我们回头看七月十四号发的这篇东西,确实需要带着一点时间滤镜去听。这是个值得提的时间点,因为二零二五年的年中,正是大模型行业从纯聊天工具往智能体方向迅速转型的一个分水岭。那个阶段,Anthropic 刚刚把他们自己提出的 MCP 协议,也就是模型上下文协议,推广开来没多久,各家都在抢着把自己的产品接进各种大模型的生态里。所以这篇文章其实不是在讲什么颠覆性的新技术,而是在讲一个生态层面的基础设施进展,就是 Anthropic 把原本需要工程师手动配置的连接器,做成了一个可以浏览、可以一键点击安装的目录。这件事放在当时看,意义还挺大的,因为它把 MCP 这种偏技术向的能力,变成了普通用户也能用的产品功能。如果你现在已经在用 Claude 的项目功能、用各种 connectors,这篇文章里讲的很多东西可能你已经觉得是“理所当然”了,但回到一年前的语境里,这其实是一次从极客玩具到大众产品的关键跨越。

我们先讲讲这篇文章到底想解决什么问题。Anthropic 在文章一开头就讲了一个特别真实的痛点,他们说,大部分人和 AI 的对话都是从头开始解释一切的,你的项目细节、你的截止日期、你用的工具,每次都要重新讲一遍,就像每次都要从零开始一样。这个场景我相信每一个用过 ChatGPT 或者 Claude 做日常工作的人都深有体会。你今天问它帮你写一份周报,它给你一个通用模板,看起来不错,但里面的内容全是空的,你还得自己一条一条填进去。你明天问它帮你分析一下这个季度的产品规划,它又是从零开始猜你公司是做什么的、你的团队结构是什么样、你现在的进度到哪了。这种“每次都要重新介绍背景”的摩擦,其实是过去这一两年里,AI 助手从“好用的玩具”走向“真正提升生产力的工具”路上最大的一个绊脚石。模型本身的智商已经足够高了,缺的是它对你具体工作场景的了解。

这就引出了文章里举的那个特别直观的例子。你可以让 Claude 帮你写最新功能的发布说明,它会给你一个挺通用的模板,你照着填。但如果你把 Claude 连接到了 Linear 这个项目管理工具,你可以直接说,帮我根据 Linear 里最新一个 sprint 的内容写发布说明,Claude 会真的去你的 Linear 里把那些工单拉出来,读懂里面记录的真实改动,然后生成一份专业的、可以直接发布的发布说明。这个区别听起来好像就是从“通用模板”到“真实数据”这一步,但这一步恰恰是从“看起来很聪明的聊天机器人”到“真正懂你工作的助手”之间最关键的鸿沟。我们做软件工程的人都知道,信息的准确性和上下文的完整性,往往比模型推理能力本身更决定最终产出的质量。一个模型再聪明,如果它手里拿到的素材是错的或者是空的,它也只能瞎编,业内管这个叫幻觉。而连接器要解决的,本质上就是给模型喂对的、真实的、实时的数据,从根上减少幻觉发生的空间。

好,那这个工具目录具体是什么呢。Anthropic 在文章里说,他们推出了一个新的工具目录,你可以浏览,然后一键连接到 Claude。同时他们还特别强调了几个由合作伙伴打造的新连接器,远程服务方面有 Notion、Canva、Stripe,本地桌面应用方面有 Figma、Socket,还有 Prisma。这里有一个特别值得拆开讲的细节,就是远程连接器和本地桌面扩展,这是两种不同的形态,背后的技术实现和适用场景都不太一样。

我们先说远程连接器。Notion、Canva、Stripe 这种,都是跑在云端的服务,你平时是通过浏览器或者它们自己的客户端去访问的。当 Claude 要连接这些服务的时候,走的是一种叫做远程 MCP 服务器的方式,你点一下连接按钮,跳转到对应服务做一次授权登录,类似你平时用谷歌账号登录第三方应用那种体验,授权完成之后,Claude 就可以通过网络请求去读取或者操作你在这些服务里的数据。这种模式的好处是部署简单,你不需要在自己电脑上装任何东西,Anthropic 这边负责维护这条连接通道,你只需要做一次授权。

而本地桌面应用,比如 Figma、Socket、Prisma,情况就不一样了。这些工具有相当一部分功能是跑在你本机上的,比如 Figma 的设计文件、Prisma 管理的数据库结构,这些往往涉及到比较敏感或者比较庞大的本地资源,不适合简单粗暴地都通过云端转发。所以这类工具用的是桌面扩展的形式,你需要通过 Claude 桌面客户端去安装一个扩展,这个扩展跑在你本机上,直接和本地运行的这些应用对话,数据不需要经过云端的额外环节。这种架构上的差异,其实也反映了 Anthropic 对不同安全边界场景的考量。云端服务你本来就是通过网络访问的,走远程连接逻辑顺理成章;但本地应用如果都强制走云端转发,既增加延迟,也可能引入不必要的隐私和安全风险,所以做成本地扩展,让数据尽量留在本机闭环里跑,这是一个比较负责任的工程决策。

接下来我想多花点时间讲讲这背后真正的技术底座,也就是 MCP,模型上下文协议。可能有些听众已经很熟悉了,但还是值得简单回顾一下,因为这是理解整篇文章的关键。MCP 是 Anthropic 在二零二四年底提出的一个开放协议,目的是统一大模型和外部工具、外部数据源之间对接的方式。在 MCP 出现之前,如果你想让一个大模型去读取你公司的某个内部系统,或者操作某个第三方服务,通常需要为每一个模型、每一个工具单独写一套对接代码,这就好像每个国家用不同的电源插座标准,你出国旅游就得带一堆转换头,特别麻烦。MCP 想做的事情,就是定义一套通用的“插座标准”,工具开发者只需要按照这套标准做一次 MCP 服务器,任何支持 MCP 的大模型客户端,理论上都可以直接接上去用,不需要为每个模型单独适配。

这篇文章里讲的这个工具目录,可以理解成是 MCP 协议从“技术规范”走向“消费级产品”的一次重要落地。在这之前,如果你想用 MCP 连接某个服务,可能得自己去读文档、配置环境变量、跑命令行,这对工程师来说还行,但对一个普通的产品经理、设计师、市场人员来说门槛实在太高了。而工具目录把这一切简化成了点击一下连接按钮,授权一下,完事。这就好像智能手机刚出来的时候,你装一个应用得自己去网上下安装包、手动安装、配置权限,后来应用商店出现了,一切变成点一下下载按钮就行。工具目录干的就是这件事,把 MCP 连接器的安装体验,做成了类似应用商店的一键体验。

文章里举的那几个具体场景,我觉得值得一个一个拆开来讲讲,因为每一个背后都对应着不同岗位、不同工作流的真实痛点。

第一个是用 Notion 做产品迭代加速,文章原话是把 Claude 里的讨论变成有条理的 Notion 路线图。这个场景我相信很多做产品的朋友会特别有共鸣。你和团队在 Claude 里头脑风暴了一个小时,讨论出一堆想法,这些想法分散在对话记录里,你如果想变成一份正式的产品路线图,通常得自己手动整理、复制粘贴到 Notion 里,排版、分类、加标签,一套流程下来半小时起步。如果 Claude 直接连了你的 Notion,理论上你可以让它把刚才讨论的内容,直接整理成结构化的页面写进 Notion 数据库里,省掉中间这一大段体力活。

第二个是 Canva,文章说的是把创意简报变成 Canva 上的社交媒体帖子。这个对做营销、做内容运营的人特别有用,你跟 Claude 描述一下这次campaign 的主题、风格、目标受众,Claude 理解之后,直接在 Canva 里生成对应的社媒图文素材草稿,而不是只给你一段文字描述让你自己再去 Canva 里慢慢做图。这其实是把“文案构思”和“视觉呈现”这两个原本分离的步骤打通了。

第三个场景是设计到代码的提升,把 Figma 文件变成可以直接上生产环境的代码。这个对前端工程师来说太熟悉那种痛苦了,设计师在 Figma 里画好稿子,工程师要一比一还原,量尺寸、抠颜色值、对齐间距,这个还原过程枯燥又容易出错。如果 Claude 能直接读取 Figma 文件里的设计稿信息,理解组件结构、样式细节,直接转换成可用的代码,这能省掉这个环节里相当一部分重复劳动。当然这里要说清楚,这不是说设计稿到代码就能全自动一步到位,生产级别的代码往往还涉及到业务逻辑、状态管理、组件复用规范这些更复杂的东西,Claude 给出的更多是一个高质量的起点,后面还是需要工程师介入打磨,但这个起点比从空白文件开始写要省力得多。

第四个场景是 Stripe,管理支付。文章说可以访问 Stripe 客户数据和支付信息。这个场景适合做客服、做财务对账、做运营分析的人,你可以直接问 Claude 某个客户最近的订阅状态、某笔交易有没有成功,而不需要自己打开 Stripe 后台一条条去查。

这四个场景放在一起看,其实有一条很清晰的主线,就是 Anthropic 想表达的核心观点,Claude 不再只是一个你打开网页输入框、问完问题就关掉的工具,而是变成一个能直接嵌入到你日常工作流程里、能动手帮你操作具体软件的协作者。这个定位的转变,其实呼应了整个行业在二零二五年那段时间正在发生的一个大趋势,就是从对话式 AI 走向智能体式 AI,英文叫 agentic AI。对话式 AI 的核心是回答问题,智能体式 AI 的核心是完成任务,而要完成任务,前提就是它得有能力接触到真实世界里的工具和数据,这正是工具目录和连接器要解决的基础设施问题。

讲到这里我想稍微跳出文章本身,讲讲我个人对这件事的理解。我个人认为,这篇文章表面上看起来只是一个产品功能的发布通知,篇幅也很短,但它背后反映的是 Anthropic 在那个阶段对自己产品定位的一次清晰表态,就是他们不打算把 Claude 做成一个孤立的聊天框,而是要把它做成一个生态的中枢,各种工具围着它转。这种打法其实挺聪明的,因为单纯比拼模型能力,几家头部厂商之间的差距会越来越小,但谁先把生态铺得广、把连接体验做得顺滑,谁就能在用户的日常工作习惯里扎根更深,这种粘性比单次问答质量的优势要持久得多。

好,讲完了背景和核心内容,我们来聊聊这件事对我们普通听众,也就是软件工程师和 AI 从业者,在实际工作中到底意味着什么,我们今天应该怎么去用,又有哪些坑需要提前避开。

首先第一点,如果你现在还没试过给 Claude 连接任何工具,我建议你可以先从一个最贴近自己日常工作的场景入手去试一下,而不是一上来就把所有能连的都连上。比如你平时大量使用 Notion 记录工作笔记和项目文档,那就先把 Notion 连上,试着让 Claude 帮你梳理一下某个项目的进展,看看它整理出来的内容是不是真的贴合你想要的格式和颗粒度。先验证好一个场景的实际效果,再决定要不要扩展到更多工具,这样可以避免一次性接入太多连接器之后,反而分不清到底哪个工具真正提升了效率,哪个只是凑数。

第二点,关于远程连接器和本地桌面扩展的选择,这里有一个我觉得值得提醒大家注意的地方,就是涉及到敏感数据或者本地资源比较大的场景,优先看看有没有本地桌面扩展可用,而不是默认都走远程连接。比如你的设计文件、数据库结构这些东西,如果有本地化的连接方式,数据流转路径更短,出问题的环节也更少。当然这不代表远程连接器不安全,Anthropic 在做这套连接基础设施的时候,授权流程本身是经过设计的,但作为使用方,多一份对数据流向的清楚认知总是好事,尤其是涉及公司内部数据、客户隐私数据的场景,授权前最好先确认一下这个连接器到底能读取哪些范围的数据,能不能做权限收窄,而不是无脑全部授权。

第三点,我想强调一下文章里那个 Figma 转代码的例子背后的实践经验。如果你是前端工程师,真要拿 Claude 连接 Figma 去生成代码,别指望它一步到位给出可以直接上线的成品,更现实的预期是把它当成一个起草工具,它能帮你把布局、间距、颜色这些重复性的还原工作快速搞定,省掉大量手动量尺寸的时间,但组件的状态管理、和后端接口的对接、可访问性的细节,这些还是需要工程师人工把关。把预期定准了,才不会因为它没做到完美还原就觉得这功能不靠谱,实际上它解决的是那一大块最耗时间但价值最低的体力劳动部分。

第四点,这篇文章发表已经快一年了,这一年里整个 MCP 生态发展得非常快,工具目录里收录的连接器数量,以及各家厂商支持 MCP 的程度,很可能已经和文章发表时有了不小的变化。所以如果你今天想实际去用,建议直接去 Claude 官网的目录页面看一眼当前实际收录了哪些工具,而不是完全按照这篇文章里列的那几个名字去找,因为生态这种东西更新速度比文章本身快得多,文章只是给你提供了一个理解这套机制的切入点。

第五点,从 AI 从业者的角度,如果你自己也在做产品,或者在给团队内部工具接入大模型能力,这篇文章其实给了一个很好的产品设计参考样本,就是把底层复杂的协议能力,用一个非常轻量的“浏览加一键连接”的交互包装起来,这个思路是可以借鉴的。很多团队在做内部 AI 工具的时候,容易把技术细节直接暴露给用户,比如让业务同学自己去配置 API 密钥、自己去写连接配置文件,这种门槛会直接拦住一大批本来可以用起来的人。Anthropic 这次做工具目录,核心价值不是技术突破,而是交互设计的简化,这一点对我们做产品的人来说,其实是个挺值得学习的细节。

最后再说一个小提醒,关于权限和成本的问题。一旦你给 Claude 连接了多个工具,它能做的事情就多了,相应的,你也需要更留意它每一次操作的范围。比如连了 Stripe 之后,Claude 理论上能查到客户支付信息,这种连接尽量限定在你确实需要的工作流程里用,用完之后该断开的可以断开,不必所有连接器一直常驻挂着,既减少不必要的权限暴露面,也方便你后续管理到底哪些工具真正在被使用。

好,我们来总结一下今天这期节目。这篇文章讲的核心事情,就是 Anthropic 在二零二五年七月推出了一个可以一键连接的工具目录,让 Claude 能够接入 Notion、Canva、Figma、Stripe、Linear 这些你日常工作里本来就在用的软件,背后依托的技术基础是他们自己提出的 MCP 协议,而这次发布真正的价值,不在于技术本身有多新,而在于把原本只有工程师能玩转的连接能力,变成了普通用户点一下按钮就能用上的产品体验。放在今天二零二六年年中的时间点回看,这篇文章可以算是 AI 助手从对话工具走向真正嵌入工作流的智能体协作者这条路上,一个很有代表性的中间站。我个人认为,这种生态层面的基础设施建设,虽然不像模型能力的飞跃那样容易制造话题热度,但往往才是真正决定一款 AI 产品能不能长期留在用户日常工作习惯里的关键力量。好了,这就是今天的全部内容,我们下期再见。
