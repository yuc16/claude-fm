## Shownotes

Claude Code 升级到一百万 token 上下文之后,会话该怎么管反而成了新问题。这篇 Anthropic 官方文章把"一件事做完之后该怎么办"拆成了继续、rewind、clear、compact、subagent 五个选项,并给出了每种选择该在什么时候用、为什么用的判断逻辑。本期我们逐条拆解这套框架，并完整推演一个真实工作日里这五个工具是怎么串起来用的。

- 上下文窗口是模型生成回复时能看到的全部内容，一百万 token 也挡不住"上下文腐烂"带来的性能下滑
- 任务完成后的五个选项：继续对话、rewind 倒带、clear 清空、compact 压缩、交给 subagent，各自适合不同场景
- 默认规则是新任务对应新会话，但"半相关"任务要不要开新会话，需要权衡重新读文件的成本
- rewind 比嘴上纠错更干净：跳回失败尝试发生之前的节点，带着教训重新提问，还能让 Claude 自己总结交接信息
- 自动 compact 容易在长调试之后丢掉下一步要用的信息，因为压缩恰好发生在模型最不聪明的时刻，建议主动出手
- subagent 的判断标准很简单：你还需要工具输出本身，还是只需要一个结论

## Script

大家好，欢迎收听 Claude FM。今天我们要聊的这篇文章，发表于二零二六年四月十五日，来自 Anthropic 官方旗下 Claude 产品博客，英文标题是 Using Claude Code, Session Management and 1M Context，我把它翻译成中文，大致是如何用好 Claude Code，会话管理与一百万上下文。作者是 Thariq Shihipar，他是 Anthropic 内部专门负责 Claude Code 的技术团队成员。距离今天二零二六年六月二十日，差不多过去两个多月，时间不算长，所以文章里讲的这些机制，基本就是你现在打开 Claude Code 还在用的那一整套东西，不存在过时的问题。

我之所以专门挑这篇文章来讲，是因为它解决的是一个特别具体、特别日常的问题。你每天用 Claude Code，到底是怎么管理会话的？你是只开一个会话一路聊到底，还是同时开着两个终端窗口各跑各的任务？你是不是每问一句就习惯性开一个新会话？你清楚什么时候该用 compact，什么时候该用 rewind，什么时候该把活儿交给一个 subagent 去干吗？你有没有遇到过这种情况，聊了很久之后，Claude 给出的答案突然开始变得不靠谱，你也说不清到底是哪里出了问题？这篇文章其实就是在系统地回答这些问题，而且给出了一套相当清晰的判断逻辑，值得每一个每天在用 Claude Code 写代码的人花二十分钟好好听一遍。

先说说这篇文章是怎么来的。Anthropic 在 Claude Code 里上线了一个新的斜杠命令，叫 usage，专门用来帮你看清楚自己的使用情况。这个功能不是凭空想出来的，而是来自他们跟大量客户的访谈。在那些访谈里，有一个现象反复出现，反复到值得专门写一篇文章来讲：不同用户管理会话的方式，差异大到超出了团队的预期，尤其是在 Claude Code 刚刚把上下文窗口升级到一百万 token 之后，这种差异变得更加明显。上下文一旦变大，能做的事情多了，可能踩的坑也跟着多了起来。有些人习惯一个会话从早聊到晚，几个小时不挪窝；有些人每发一条消息就习惯性开一个新会话；还有人压根没用过 rewind 或者 subagent 这些功能，遇到问题全靠手动跟 Claude 来回拉扯，硬聊到对方理解为止。Anthropic 发现，这里头藏着大量可以优化的空间，而几乎所有的优化，最终都指向同一件事，就是你怎么管理自己的上下文窗口。

那什么是上下文窗口呢。简单说，上下文窗口就是模型在生成下一句回复的时候，能够同时看到的全部内容。它包括系统提示词，也就是规定 Claude 该怎么表现的那套基础设定；包括你们目前为止的整段对话；包括每一次工具调用和它返回的结果，比如 Claude 执行了一条命令，命令到底输出了什么；还包括它读取过的每一个文件的全部内容。Claude Code 现在的上下文窗口大小是一百万 token。token 你可以粗略理解成模型处理文字时的最小单位，大概几个字符算一个，一百万听起来是个天文数字，但实际用起来，一旦项目稍微复杂一点，文件读得多一点，工具调用频繁一点，消耗起来比你想象得快得多。

但这里有个反直觉的地方，文章管它叫 context rot，我把它翻译成上下文腐烂，你也可以理解成上下文衰减。意思是说，模型的表现会随着上下文变长而逐渐下降，原因不是它记不住，而是它的注意力被摊得越来越薄。你可以把这想象成一块橡皮泥，刚开始很有弹性，能精准地包裹住每一个细节，可你不断往里面加东西，它就被拉得越来越薄、越来越大，最后捏到哪个地方都软软的，使不上劲。换一个更生活化的类比，就像你开了一个特别长的微信群聊，从中午聊到半夜，话题来回换了七八轮，群里突然有人冒出一句，刚才说的那个方案最后定了吗，这时候哪怕所有信息都还挂在群里，没被删掉，你也得翻好久才能找到答案，而且很可能还翻错了楼层。模型也是一样，上下文里堆的东西越多，越久之前、跟当前任务不相关的内容，反而会变成噪音，干扰它对眼前这件事的判断。

这里我们可以多想一层。如果这套机制完全交给系统自动处理，不给你任何手动选择，会怎么样？其实也不是不可以，自动压缩本来就一直存在。但 Anthropic 选择把 rewind、clear、compact、subagent 这几个手段都摆在台面上让你自己挑，背后的判断是，只有你自己最清楚接下来要往哪个方向走，系统能猜到大概，却猜不准细节。这跟很多自动化工具的设计哲学很像，自动挡的车开起来更轻松，但赛车手在弯道前还是会自己提前减速换挡，因为只有他知道下一个弯有多急，系统看到的永远只是路况，看不到你脑子里那条还没说出口的计划。

上下文窗口还有一个特点，它是硬性的上限，不是可以商量的软上限。当你的对话快要把这一百万 token 用满的时候，Claude Code 会自动把你们目前为止做的事情，总结成一段更短的描述，然后让模型在一个全新的上下文窗口里，接着这段总结继续往下做。这个过程，文章里叫它 compaction，我们可以叫它压缩，或者紧凑化。这件事是自动触发的，但你完全可以自己手动去触发它，这个我们后面会细讲。

文章里给出一个特别实用的场景，假设你刚让 Claude 做完一件事，它完成了，这时候你的上下文里已经积累了不少东西，包括工具调用、工具输出、你给它的各种指示。接下来要做什么，其实有五个选项可以选，很多人可能压根没意识到自己有这么多选择。第一个选项是继续，也就是在同一个会话里接着发下一条消息，这是最自然、最常见的做法。第二个选项是 rewind，倒带，双击 Esc 键，或者直接敲命令，跳回到之前某条消息，从那个点重新开始。第三个选项是 clear，清空，把整个会话清空，从零开始。第四个选项是 compact，压缩，让模型把目前的会话总结一遍，然后在这段总结的基础上继续往下做。第五个选项是交给 subagent，子代理，把接下来这一块工作整个委托给一个拥有自己独立、干净上下文窗口的代理，等它干完了，只把最终的结论拉回到当前会话里。继续聊天当然是最顺手的做法，什么都不用想，敲个字接着说就行，可后面这四个选项的存在，都是为了同一个目的，帮你管理好自己的上下文，不让它失控地一路膨胀下去。

那到底什么时候该开一个全新的会话，什么时候又该守着一个长会话不放呢。文章给出的经验法则非常干脆，开始一个新任务，就开一个新会话。逻辑也很直接，升级到一百万 token 上下文之后，你确实可以指望 Claude 做更长、更复杂的任务，比如从零开始搭一个完整的全栈应用，整个过程一个会话扛到底都没问题。可任务越长，上下文积累得越多，前面说的上下文腐烂就越容易找上门。所以即便容量变大了，也不代表你应该把所有事情都一股脑塞进同一个会话，新任务对应新会话，应该是你的默认动作。

当然，现实里很多任务并不是非黑即白的。文章举的例子是，你刚让 Claude 实现完一个功能，紧接着想让它给这个功能写文档。这算不算一个新任务呢？一部分上下文确实还有用，比如它刚才改过哪些文件、用了什么思路，这些信息能帮文档写得更贴合实际；但也不是所有上下文都还有必要留着，比如中间反复调试、反复试错的那些记录，对写文档其实帮助不大。这时候你当然可以选择直接开一个全新会话，但代价是 Claude 得把你刚刚写的那些文件重新通读一遍，才知道自己在写什么文档，这意味着更慢，也意味着更贵，因为重新读文件本身也要消耗 token。所以遇到这种半相关的任务，往往不是简单粗暴地开新会话，而是要在剩下的几个选项里，挑一个更合适的工具来用。

我们先细说 rewind，倒带。操作很简单，双击 Esc 键，或者直接敲 rewind 这个命令，就能跳回到之前任何一条消息，从那个点重新往下说。跳回去之后，那个点之后的所有消息，都会从当前上下文里被丢掉，就好像它们从来没发生过一样。这个功能听起来朴素，但文章特别强调，rewind 往往是纠错时更好的选择，比单纯继续聊天要高效得多。我们来看一个典型场景，Claude 读了五个文件，尝试了一种实现方案，结果没跑通。这时候你的第一反应可能是直接打字说，这个方法不行，换成另一个思路试试。但这样做有个问题，那次失败的尝试，包括它读过的内容、走过的弯路，全部还留在上下文里，模型接下来要在这堆失败现场的基础上继续干活，等于背着一堆没用的包袱往前走。更好的做法是，倒回到它刚读完那五个文件、还没开始动手尝试的那个时间点，然后带着你刚刚学到的教训重新提问，比如直接告诉它，别用那个方案了，某个模块根本没有暴露那个接口，直接换一条路走。这样一来，新的上下文里只留下干净的文件信息，加上你提炼出来的关键教训，没有那一整段失败过程占着地方。这就像写代码的时候用版本控制做回退，而不是在一个已经写废了的分支上拼命打补丁，干净清爽得多。如果你做过代码评审，这个思路你应该很熟悉，跟一个写错方向的分支死磕，不如直接从主干切一条新分支重新来，rewind 本质上就是把这套版本管理的直觉，搬到了对话层面。

文章里还提到一个很有意思的用法，叫从这里开始总结，你也可以直接用 rewind 这个命令来触发它。它做的事情是，让 Claude 把它在这段被丢弃的过程里学到的东西，提炼成一条交接消息。这条交接消息的感觉，有点像是未来的自己，写给过去的自己的一张便利贴，我已经试过某种做法了，结果没成功，原因是什么，你接下来直接跳过这条路，从这里继续往下走。这种交接机制特别适合那种试错成本很高、但教训值得保留的场景，你不需要把整个试错过程都背在身上，只需要把为什么不行这件事干净地传递下去就足够了。

接下来说说 compact 和 clear 这两个选项。一旦会话变长了，这两个都是用来甩掉多余上下文的办法，乍一看感觉很像，其实表现完全不同，用错了场景效果会差很多。compact，压缩，是让模型自己把到目前为止的整段对话总结一遍，然后用这段总结替换掉原来完整的历史记录。这个过程是有损的，意味着细节肯定会丢一些，但好处是你自己什么都不用写，模型可能还会比你想得更全面，主动把重要的发现、关键的文件路径都保留下来。你还可以给它一些方向性的指示来引导这次压缩，比如直接说，压缩的时候重点保留某个重构相关的内容，把之前测试调试那部分的细节丢掉。clear，清空，则完全不同，是你自己手写下来你认为重要的东西，比如我们正在重构某个中间件，约束条件是什么，真正关键的文件是哪两个，已经排除掉了哪种方案，然后从一个彻底干净的状态重新开始。clear 显然更费功夫，你得自己提炼、自己写下来，但好处是留下来的上下文，每一个字都是你主动决定要留的，不会有模型自己判断失误、把不该丢的东西丢掉的风险。

那什么时候 compact 会出问题呢？文章给了一个特别真实的场景。如果你经常跑很长的会话，可能已经隐约感觉到，有些时候自动压缩完之后，体验明显变差。这种糟糕的压缩，通常发生在模型没办法预判你接下来要往哪个方向走的时候。举个具体例子，假设你和 Claude 刚刚经历了一段很长的调试过程，排查一个很棘手的报错，这时候自动压缩触发了，把整段调试过程总结成一段简短的描述。然后你下一句话是，现在把我们之前在某个文件里看到的另一个警告也顺手修一下。问题来了，因为刚才整段会话的焦点全都在调试那个报错上，那个顺带提到的警告，很可能在压缩总结的时候，被判断成不重要的细节，直接被丢掉了。于是 Claude 压根不知道你说的另一个警告是什么，你们俩等于平白多了一轮鸡同鸭讲，你还得重新把上下文补回去，反而更费时间。

文章特别点出一个让这件事更棘手的细节，恰恰是因为上下文腐烂的存在，模型在执行压缩总结这个动作的那一刻，正好处在它整段会话里最不聪明的状态，因为这时候上下文已经堆得最满、注意力被摊得最薄。换句话说，你最需要它做出精准判断、决定哪些信息该保留的那一刻，反而是它判断力最弱的一刻，这就是为什么自动压缩容易出岔子。文章给出的应对思路是，现在上下文窗口扩大到一百万 token 了，你手里其实有更多余地，完全可以不等系统自动触发，提前主动地去执行压缩，并且亲自告诉它接下来要往哪个方向走、重点保留什么内容。这样一来，模型是在状态还比较好、还没有被拖到最疲惫的那个点上去做总结，准确率自然会高很多。

最后说说 subagent，子代理。什么时候适合用它？文章给的判断很清楚，当你提前就能判断出，接下来这一块工作会产生大量你以后根本不会再用到的中间过程，这种情况下交给 subagent 效果最好。具体机制是这样的，当 Claude 通过它的 Agent 工具派出一个 subagent 的时候，这个 subagent 会获得一个完全属于自己的、全新的上下文窗口，跟主会话完全隔离。它可以在自己的小世界里，该读多少文件就读多少文件，该试多少次就试多少次，过程里产生的所有中间内容，统统留在它自己的上下文里，等它把活儿干完，只把一份整理好的最终结论，传回到主会话。这就像你在公司里接到一个调研任务，与其自己一头扎进去翻几十份资料，不如找个实习生，让他把那几十份资料全部啃完，最后只给你交一页纸的结论，你不需要知道他具体翻了哪些页码、走了多少弯路，你只要那一页结论就够了。

Anthropic 内部用来判断要不要派 subagent 的标准非常简洁，他们管它叫一个心理测试，这个工具的输出，我之后还会再需要原文本身吗，还是说我只需要它的结论。如果答案是只需要结论，那这块工作就值得交给 subagent。虽然 Claude Code 本身已经会在合适的时候自动调用 subagent，但文章建议你也可以主动、明确地要求它这么做。文章给了三个具体的说法示例。第一种，你可以说，派一个 subagent，根据下面这份规格文件，去验证一下这次工作的结果对不对。第二种，你可以说，派一个 subagent 去通读另外那个代码库，总结一下它的身份验证流程是怎么实现的，然后你照着同样的思路在我们这边实现一遍。第三种，你可以说，派一个 subagent，根据我这次的代码改动，去把这个功能对应的文档写出来。这三个例子背后的共同点很明显，都是那种过程很啰嗦，但结论很精炼的活儿，验证、调研、写文档，都属于这一类。这里也可以多推演一步，如果你管理的是一个由好几个微服务组成的系统，经常需要参考另一个仓库的实现方式，你完全可以把通读这件事固定成一个习惯性动作，每次涉及跨仓库参考，先派一个 subagent 去读、去总结，自己永远只接收结论，而不是亲自去翻那个不熟悉的代码库，这样主会话里积累的，始终是跟你当前要写的代码直接相关的信息，不会被一堆别人家代码库的细节占满。

文章最后提到，他们整理了一张表格，把常见的使用场景、对应该用哪个工具、为什么要这么用，全部归纳在一起，方便你下次遇到类似情况的时候，能很快做出判断，不用每次都重新想一遍这套逻辑。这也再次说明，这五个选项之间的取舍，并不是玄学，而是有一套可以被总结、可以被复用的判断规则的。

说完了原文的内容，我们来聊聊这套东西对你日常写代码到底意味着什么，能直接拿去用的东西其实不少。

第一条，养成一个习惯，每次要开始一件性质不同的新任务，就主动开一个新会话，不要图省事，把好几件不相关的事情堆在同一个会话里聊。如果你发现自己已经在同一个会话里，从修一个 bug 聊到了设计一个全新的功能，这其实就是个信号，提示你该开一个新会话了。哪怕你舍不得之前积累的上下文，也要想清楚，留着这堆跟新任务无关的东西，付出的代价往往比重新读几个文件更大。

第二条，下次 Claude 尝试了一个方案没成功，先别急着在原地接着打字说不对换个思路。停一下，想想能不能用 rewind 倒回到它刚读完相关文件、还没动手尝试的那个点，再带着你刚刚学到的教训重新提问。这样做的好处是，新的上下文里没有那一整段失败尝试占地方，模型不用拖着包袱往前走，效率往往会高出一截。如果你觉得那段失败的过程里有值得保留的教训，记得用从这里开始总结这个功能，让 Claude 自己把教训提炼成一条交接消息，而不是你自己手动去总结，省下来的精力可以花在判断下一步该怎么走上面。

第三条，不要永远被动地等系统自动触发压缩。如果你已经感觉到这个会话已经聊得够长，而且你清楚接下来要往什么方向走，主动敲一下 compact 命令，并且把接下来想干什么写清楚，比如压缩的时候重点保留某个模块的改动，把之前调试时走过的弯路丢掉。这样压缩出来的结果，会比系统在你完全没准备的情况下自动帮你压缩的版本靠谱得多。尤其是如果你刚结束一段很长的调试，下一步要做的事情跟刚才的调试关系不大，这种时候千万别让自动压缩默默发生，等它发生了你才反应过来要修的东西已经被总结没了，那就只能自己再补一遍上下文，反而更费时间，也更容易让人心烦。

第四条，如果你手头这个项目，有那种特别核心、特别不容易出错、必须长期保留的关键信息，比如一项重大的架构决策，或者一条已经被排除掉、不要再被重新提起的方案，这种时候比起依赖压缩帮你总结，更稳妥的做法是用 clear，自己花两分钟把这些关键信息写下来，再开始干活。多花的这两分钟，换来的是你能完全确定，这些关键信息不会在某次自动总结里被模型判断成不重要而丢掉，这笔账长期来看其实很划算。

第五条，平时分配任务的时候，多问自己一句，这一步产生的东西，我后面还会再翻出来看吗，还是我只要一个结论。如果只要结论，那就大大方方地明确告诉 Claude，去派一个 subagent 来干这件事。常见的几类活儿特别适合这么干，验证别人或者你自己刚写完的代码是否符合某份规格文档，去翻一遍另一个代码库，总结别人某个模块是怎么实现的作为参考，根据这次的代码改动去补一份文档。把这几类零碎但啰嗦的活儿都甩给 subagent，能让你的主会话始终保持干净，专注在真正核心的那条主线任务上，不至于被一堆你以后根本不会再看的中间过程占满。

我们不妨做一次完整的推演，假设你今天接到的任务是，给一个老项目的支付模块加一个新的退款接口。你会怎么走完这一天？早上你打开 Claude Code，开一个新会话，这是新任务对应新会话的体现。你让它先去读懂现有的支付模块，它读了七八个文件，梳理出一版实现方案，你看了一眼觉得方向不对，这时候你没有继续在原地争论，而是用 rewind 跳回到它刚读完文件、还没开始写方案的那个点，告诉它这个项目的退款必须走异步队列，不能同步阻塞，带着这条约束重新让它出方案。下午，这一版方案你认可了，进入实现阶段，中间夹杂了一段长达四十分钟的调试，定位一个签名校验的老问题。调试结束之后，你判断接下来要写的单元测试和这段调试关系不大，于是主动敲一次 compact，并且告诉它压缩时重点保留退款接口的实现细节和你们刚才达成的设计结论，调试过程的细节可以丢掉。最后，你想确认这个新接口有没有破坏老的支付流程，这部分验证工作过程会很啰嗦，要跑一堆历史用例，你确实不需要看它具体怎么跑的，只要一个通过或者不通过的结论，于是你派一个 subagent 去专门做这件事，自己回到主会话继续下一个任务。这一整天下来，你几乎把五个选项都用上了一遍，而且每一次选择都有明确的理由，这才是文章真正想传达的东西，不是死记硬背这五个工具叫什么，而是养成那种遇到岔路口就下意识问自己一句接下来要干什么、上下文里哪些东西还有用的习惯。

如果你是团队里带新人的角色，我个人认为这套判断逻辑特别适合直接搬去做培训材料，因为它把原本很玄学、只能靠老手凭感觉判断的东西，变成了几条可以直接照着执行的规则，新人按这几条来，至少能避开大部分因为上下文管理不当导致的低效返工。

最后再提两个容易踩的坑。一个是别忽视上下文腐烂这个现象本身的存在，如果你发现 Claude 最近的回答质量明显下降，先别怀疑模型变笨了，很可能只是这个会话已经堆得太满，该考虑 rewind、compact 或者直接开新会话了。另一个是别迷信上下文越大越好这种想法，一百万 token 的窗口给了你更大的容错空间，但并不意味着你应该放任不管，把所有东西一股脑全塞进同一个会话，主动管理上下文，永远是比被动依赖窗口大小更可靠的做法，工具再大，也得有人会用才值钱。

整篇文章讲的核心思路，其实可以浓缩成一句话，上下文窗口是一种有限的资源，怎么花这个资源，比这个资源本身有多大更重要。继续、rewind、clear、compact、subagent，这五个选项分别对应着不同的取舍，到底用哪个，关键看你接下来要做的事情，跟眼前已经攒下来的上下文之间，到底是什么关系。我个人认为，这篇文章最有价值的地方，不是介绍了什么新功能，而是把很多人原本只是凭感觉去做的事情，变成了一套可以拿来复盘、也可以教给团队里其他人的判断标准。这跟人类团队管理知识的方式其实很像，什么时候该开会做总结，什么时候该把决策写成文档存档，什么时候该交给一个专人去单独调研、回来只汇报结论，道理是一模一样的。好，今天这期就聊到这里，我们下期再见。
