## Shownotes

本期我们聊聊 Anthropic 在二〇二六年四月发布的 Claude Code 新功能 Routines,也就是"例行任务"。这个功能让你把一段提示词、一个代码仓库和一组连接器打包成一个自动化流程,设置成按时间表运行,或者由 API 调用、GitHub 事件触发,全程跑在云端,不需要笔记本一直开着。我们会拆解三种触发方式的工作原理,过一遍官方文章里提到的七种实际应用场景,也聊聊使用门槛、每日额度限制,以及我对这件事在 agentic coding 发展脉络里所处位置的看法。

本期要点：
- Routine 是什么：配置一次提示词、代码仓库和连接器,之后按时间表、API 调用或 GitHub 事件反复运行,跑在 Claude Code 的云端基础设施上,不依赖本地设备
- 三种触发方式：定时调度像设闹钟,API 调用像装门铃,GitHub 事件订阅像追更报纸,一个 PR 对应一个会持续更新的会话
- 七种典型场景：backlog 管理、文档漂移检测、部署验证、告警分类、用户反馈处理、多语言 SDK 库迁移、团队定制化代码审查
- 使用门槛和额度：Pro、Max、Team、Enterprise 用户都能用,但每日可跑的 routine 数量从五个到二十五个不等,超额可以加购
- 我的看法：这是 Claude Code 从交互式助手向常驻自动化基建演进的一步,但也带来了权限管理和成本控制上的新课题

大家好,欢迎收听 Claude FM,这里是写代码的人也想听一听的科技播客。今天我们要聊的这篇文章,标题是 Introducing routines in Claude Code,我把它翻译成在 Claude Code 里引入例行任务。这篇文章发表于二〇二六年四月十四日,出自 Anthropic 官方博客。到我们录这期节目的时候,差不多已经过去两个多月了,但这个功能现在依然在持续演进,内容本身也还没有过时,所以今天我们就花点时间,把它从头到尾拆开讲清楚。文章开头特别说明,这次发布属于 research preview,也就是研究预览版,意思是这是一个刚刚对外开放、还在打磨阶段的早期功能,接口和细节后面很可能还会调整,这一点大家在了解的时候要心里有数。我个人认为,这次更新挺有意思,它标志着 Claude Code 正在从一个你坐在电脑前敲命令、跟它聊天写代码的工具,慢慢变成一个可以常驻在云端、二十四小时帮你值班的自动化系统。这个转变不是一夜之间发生的,但 Routines 这个功能,算是把这条路铺得更明显了。

我们先说说背景。在 Routines 出现之前,开发者已经在大量使用 Claude Code 来自动化软件开发流程里的各种环节了,比如修 bug、写测试、审查代码。但问题是,如果你想让这些自动化常态化地运行,比如每天晚上自动跑一遍,或者每次有新的 PR 提交就自动触发一次,你就得自己去搭建和维护一整套基础设施。你需要自己写 cron 定时任务,自己管理服务器或者云函数,还要自己配置各种额外的工具,比如 MCP server,也就是给 Claude Code 接入外部系统的连接协议,通过它,Claude 才能去读取 Linear 里的工单,或者往 Slack 里发消息。这些事情听起来不难,但真正做起来挺琐碎的,你得保证服务器一直在线,得处理认证和权限,得监控任务有没有跑失败,半夜任务挂了还得有人起来重启。说白了,这就像你想雇一个值班助理,但雇之前你还要先给他盖一间办公室,装好水电网络,这个门槛对很多团队来说是不低的,尤其是那些没有专门运维人手的小团队,可能因为搭基础设施这件事本身太麻烦,干脆就放弃了自动化的想法。Routines 想解决的,正是这个基础设施自己搭的痛点,它把这部分工作直接收编进了 Claude Code 的产品里。

那 Routine 到底是什么呢。按照文章的说法,一个 Routine 就是一份你只配置一次的 Claude Code 自动化,它包含三个核心要素,一个提示词,告诉 Claude 具体要做什么,一个代码仓库,告诉它在哪个项目上操作,还有一组连接器,也就是它能调用的外部工具和服务,比如 Slack、Linear、GitHub 这些都属于连接器的范畴。配置好这三件事之后,你可以让它按照一个时间表运行,也可以通过一次 API 调用触发,还可以让它在某个事件发生的时候自动响应。最关键的一点是,这一切都跑在 Claude Code 的云端基础设施上,这意味着你不需要让自己的笔记本电脑一直开着,也不需要自己去维护服务器。你配置好之后就可以合上电脑去睡觉,这个任务依然会在云端按时执行,这跟以前那种必须开着终端窗口才能跑自动化脚本的体验完全不一样。

如果换成你自己来设计这样一套系统,你大概会先纠结要不要自己起一台常驻服务器来跑定时任务,然后再纠结认证怎么做,日志怎么存,失败了怎么重试,光是这些基础设施的决策就能讨论好几次会议。Anthropic 直接把这些都做成了产品里默认就有的能力,你不需要纠结这些问题,只需要把注意力放在提示词写得够不够清楚,以及连接器选得对不对就行了。

接下来我们具体看看三种触发方式,这是这篇文章里最值得展开讲的部分,我们一个一个来拆。

第一种是定时调度,这个最好理解,就跟你手机上设个闹钟一样。你告诉 Claude Code,每天凌晨两点,去 Linear 里面拉一个优先级最高的 bug,尝试修复它,然后开一个草稿状态的 PR。到了凌晨两点,它就会自动执行这个流程,不需要任何人盯着。文章里提到,如果你之前在命令行里用过 schedule 这个命令来安排任务,那么现在这些任务已经自动升级成了 scheduled routine,也就是定时型的例行任务,不需要你重新配置。我觉得这个设计挺贴心的,老用户的习惯被很自然地承接下来了,不存在迁移成本。如果你是一个团队的技术负责人,这种定时调度最直接的用法,可能就是把团队里那些大家都知道该做但谁都不想花时间做的杂活,一股脑地丢给夜里的 Claude 去处理,等第二天早上团队开站会的时候,这些杂活已经有了初步结果。

第二种是 API 调用触发,这个可以类比成你家门口装了个门铃。每一个 Routine 在创建的时候,都会自动拿到属于它自己的一个网络接口地址,以及一个用于身份验证的令牌。你只需要往这个地址发一条 POST 请求,带上消息内容,Claude Code 就会启动这个 Routine,并且返回一个会话链接给你,你可以打开这个链接看它具体在做什么。这种触发方式的好处是,它可以被接入到任何能发起网络请求的系统里,比如你公司的告警系统、部署流水线的钩子,或者任何内部工具。文章给的例子是,读取告警的内容,找到这个告警对应的服务负责人,然后在一个叫 oncall 的值班频道里发一条带有初步解决建议的分析摘要。你能想象一下这个场景吗,半夜系统出问题,告警打过来,人还没醒,Claude 已经把问题归因到了具体的服务,甚至给出了第一步该怎么处理的建议,这对值班工程师来说,意味着从被告警惊醒到真正动手修复之间的那段摸黑摸索的时间,被大幅压缩了。

第三种是事件触发,具体来说是订阅 GitHub 仓库里发生的事件。这个可以类比成你订阅了一份报纸,只要有新内容发布,它就自动送到你手上。你可以设置一个过滤条件,比如只要有 PR 修改了某个叫 auth provider 的认证模块,Claude 就会自动生成一份变更摘要,发到一个叫 auth changes 的频道里去,提醒团队这块敏感代码被改动了。这里有个细节我觉得很关键,Claude 针对每一个符合条件的 PR,会单独开一个会话,而且这个会话不是跑完一次就结束了,后续这个 PR 上如果有新的评论,或者持续集成跑出了失败结果,这些更新都会被继续喂给同一个会话,让 Claude 能够持续跟进这个 PR 的整个生命周期,而不是只看一次快照就完事。这种设计其实更贴近真实的代码协作场景,因为一个 PR 从提出到合并往往要经历好几轮的讨论和修改,Claude 如果只在 PR 刚打开的瞬间看一眼,很多后续的上下文就丢掉了。文章最后还特别提了一句,他们计划未来让基于事件触发的 Routine,能够支持更多种类的事件来源,不只是局限于现在的 GitHub 仓库事件。这说明现在我们看到的事件触发,只是一个起点,后面应该会陆续接入更多系统,这块值得大家持续关注。

把这三种方式放在一起看,你会发现 Anthropic 其实是把自动化触发的三种最常见的范式都覆盖到了,时间驱动、主动调用驱动,以及事件驱动。这三种方式分别对应了开发者平时最常打交道的几类场景,定时巡检、外部系统集成,以及代码协作流程。如果你来设计这样一个产品,大概率也会往这三个方向去拆,因为几乎所有的自动化需求,归根结底都可以归到这三类里面去,你很难再找出第四种触发逻辑出来。

讲完了原理,我们来看看文章里举的几个实际应用场景,一共是七个,这部分我觉得对大家的工作启发会更直接一些。

第一个场景是 backlog 管理,也就是需求和缺陷积压列表的管理。具体做法是每天晚上自动跑一遍,把新进来的 issue 做分类,打标签,指派负责人,然后把当天的处理摘要发到 Slack 里。你可以想象一下,产品经理早上一上班,打开 Slack 就看到昨晚提的十几个 issue 已经被自动归类好了,哪些是紧急 bug,哪些是功能请求,分别该找谁跟进,一目了然。这种夜间自动分类的方式,本质上是把原本需要专人花半小时甚至一小时去做的整理工作,压缩成了团队成员第二天早上花几分钟确认一下就好的事情。

第二个场景是文档漂移检测,意思是代码改了,但文档没跟上,两者渐行渐远。具体做法是每周扫描一遍已经合并的 PR,找出哪些改动涉及到了对外的接口或者 API,然后检查现有文档里有没有引用这些被改动的地方,如果有,就自动开一个 PR 去更新文档。如果你所在的团队也维护过对外的 API 文档,大概都遇到过用户照着文档操作却发现接口已经变了的尴尬情况,这种信任成本的流失其实是很难量化但又确实存在的,靠人工去对齐基本上不现实,因为这是个枯燥又琐碎的核对工作,交给一个每周自动巡检的 Routine,刚好适合。

第三个场景是部署验证。你的持续部署流水线在每次发布之后,自动调用对应的 Routine,Claude 针对刚刚上线的新版本跑一遍冒烟测试,也就是最基本的功能可用性检查,同时扫一遍错误日志,看有没有出现新的异常或者回归问题,最后在发布频道里给出一个能不能继续发布的结论。这相当于给团队配了一个永远不会偷懒、不会因为是周五晚上就想着早点下班的发布质检员。很多团队都吃过这个亏,一个改动在本地测得好好的,上线之后才发现某个边缘场景出了问题,如果有个 Routine 在每次发布后第一时间把基本的检查跑一遍,至少能把最明显的回归问题挡在用户感知到之前。

第四个场景是告警分类,这跟前面 API 触发的例子有点像,但这里更具体一点。文章举的例子是把 Datadog 这种监控告警系统直接指向 Routine 的接口地址,告警一来,Claude 自动拉取相关的调用链路追踪信息,把它和最近的几次部署关联起来分析,等到值班工程师真正打开告警页面的时候,一份草稿状态的修复方案已经摆在那里了。这个场景如果放在实际团队里,影响是很直接的,大家都知道故障处理里有个关键指标叫平均修复时间,很多时候这个时间里最长的一段不是修复本身,而是定位问题、搞清楚是哪次部署引入的这个问题。如果这部分工作能提前由 Claude 在后台跑完,留给人的就只是确认和拍板,这个指标是能实实在在往下降的。

第五个场景是用户反馈处理。比如你的产品里有个反馈小组件,或者内部有个问题看板,用户提交了一条反馈或者报告了一个问题,这条信息会被推送过去触发 Routine,Claude 直接针对相关的代码仓库开一个会话,把这个问题的上下文整理好,然后直接开始起草修改方案。这个场景把用户报了个问题和工程师着手去改这两件事之间的等待时间几乎压缩到零,对于那种用户量大、反馈频繁的产品来说,这种响应速度上的提升其实是能直接影响用户体验口碑的。

第六个场景叫 library port,可以理解成库的跨语言迁移。文章举的例子是,如果你维护着一个 Python 版本的 SDK,同时还维护着一个功能对等的 Go 语言版本,每次有 PR 合并到 Python 版本的仓库,就自动触发一个 Routine,把这个改动对应着搬到 Go 版本的仓库里,并且开一个匹配的 PR。这个场景我觉得特别能体现 Routines 的价值,因为维护多语言 SDK 一直是个公认的苦活,功能要在好几个语言版本之间保持同步,纯靠人工很容易漏掉某个版本忘了更新。我猜测 Anthropic 自己内部维护多语言 SDK 的团队,可能就是这么用上了这个功能,这也算是吃自己的狗粮了。

第七个场景是定制化代码审查。在 PR 刚打开的时候,就按照团队自己定的检查清单,把安全性和性能这两块过一遍,在真正的人工审查者介入之前,先把行内评论打好。这个场景跟很多团队已经在用的自动化 lint 工具有点像,但区别在于,它不是死板的规则匹配,而是基于团队自己定义的检查标准,用 Claude 的理解能力去做更灵活的审查,相当于给每个 PR 先配一个不知疲倦的初审员。

把这七个场景放在一起看一下,你会发现它们有个共同的特点,都是那种重要但不紧急,而且高度重复的工作。这类工作恰恰是团队里最容易被拖延、被忽略,但长期攒下来又会变成大问题的那部分。Routines 瞄准的正是这个区间,它不是要去抢工程师写核心业务逻辑的活,而是把这些边角料性质的、需要持续盯着才能不出岔子的工作,交给一个不会累、不会忘、不会找借口拖延的系统去做。

说完这七个场景,我们再聊聊怎么真正用上这个功能,以及它的门槛和限制。目前 Routines 对 Pro、Max、Team 和 Enterprise 这四档订阅用户都开放,但前提是你得先开启 Claude Code 在网页端的功能。具体操作上,你可以直接打开 Claude 的网页版去创建你的第一个 Routine,也可以在命令行里输入 schedule 命令来配置。从计费的角度说,Routines 消耗的额度跟你平时跟 Claude 交互式对话消耗的额度是同一个池子,按订阅的用量上限来扣减。除此之外,官方还设置了每天能跑多少个 Routine 的上限,Pro 用户每天最多能跑五个,Max 用户每天最多能跑十五个,Team 和 Enterprise 用户每天最多能跑二十五个,如果团队用量超过了这些限制,可以额外购买用量来继续运行。我觉得这个限制是个挺合理的设计,毕竟 Routine 一旦铺开,很容易出现设置之后就忘了它在跑的情况,设个上限至少能防止额度被悄无声息地消耗掉。

讲到这里,我想说说我自己对这件事的看法,这部分纯粹是我个人的解读,大家可以保留自己的判断。我觉得 Routines 这个功能,本质上是在回应整个行业这两年一直在讨论的一个方向,就是 agent,也就是智能体,要从你问一句它答一句的交互模式,进化成它在背后持续工作,你只在需要的时候介入的模式。这次更新之所以重要,不是因为定时任务或者事件触发这些技术本身有多新,这些能力开发者自己用脚本也能搭出来,甚至用一些现成的自动化平台也能拼出类似的流程,关键是 Anthropic 把这套基础设施收进了 Claude Code 本身,而且把 Claude 真正理解代码、理解上下文的能力也一起打包进去了,这跟单纯靠规则触发的自动化不是一个量级。这意味着普通团队不需要再自己运维一套服务来承载这些自动化,门槛被实实在在地拉低了。但与此同时,我也觉得这里面藏着一些需要谨慎对待的地方。一旦你让 Claude 在没有人盯着的情况下,自动触达 Slack、GitHub、监控系统这些地方,权限管理就变得格外重要,你授权给一个 Routine 的连接器范围,直接决定了它能造成多大的影响,如果配置不当,比如不小心给了过宽的写权限,出问题的时候排查起来也会更麻烦,因为这不是一次有人在场的交互,而是一个在后台自动跑的流程。另外每日额度的限制也提醒我们,这类自动化不是没有成本的,跑得越多、越频繁,消耗的用量也越多,团队在设计 Routine 的时候,最好先想清楚哪些场景的投入产出比最高,而不是一上来就把所有能自动化的事情都丢进去跑。所以我的建议是,如果你打算上手试试 Routines,先从风险比较低的场景开始,比如文档检测或者告警摘要这种只读或者半自动的场景,等你摸清楚它的行为边界之后,再逐步放开到会自动开 PR、自动修复代码这种有实际写权限的场景。

好,我们最后总结一下。这篇文章介绍的 Routines,核心是让 Claude Code 从一个你主动唤起才工作的助手,变成一个可以配置一次、长期在云端值班的自动化系统,它支持定时调度、API 调用和 GitHub 事件这三种触发方式,文章里给出的七个场景,从 backlog 管理到多语言 SDK 同步,基本覆盖了软件团队日常运维和协作里最琐碎也最容易被忽略的那部分工作。如果你自己也在团队里负责一部分自动化或者运维相关的工作,不妨想一想,你们手头有没有那种重复了很多次、规则相对清楚,但一直没有人愿意花时间去自动化的任务,这种任务可能就是最适合拿 Routines 先试一试水的地方。我个人认为,这是 agentic coding 往前走的一个挺扎实的小步,值得正在用 Claude Code 的团队找一两个低风险的场景先跑起来试试。今天的节目就到这里,我们下期再聊,再见。
