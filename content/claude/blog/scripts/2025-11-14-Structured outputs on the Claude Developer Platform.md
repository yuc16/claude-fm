大家好,欢迎收听 Claude FM,今天我们要聊的这篇文章,标题是 Structured outputs on the Claude Developer Platform,翻译过来就是,克劳德开发者平台上的结构化输出功能。这篇文章发表于二〇二五年十一月十四日,出自 Anthropic 官方博客。

按今天的日期来算,也就是二〇二六年六月二十日,这篇文章已经过去差不多七个月了。七个月在 AI 这个行业里,已经不算一个短的时间跨度了,所以这里要提醒大家一句,文章里描述的"公测"状态,实际上早就升级过好几轮了。我们后面会专门讲到,这个功能从去年十一月的公开测试版,一路走到了今年二月初的正式可用,中间还加上了对更复杂模式的支持,也加上了对 Haiku 四点五这个更轻量模型的覆盖。所以大家听完这期节目,记住的不应该只是"去年十一月发生了什么",而应该是这个功能从诞生到成熟的完整脉络,这样你今天去用的时候,才不会被一篇半年前的文章带偏。

那我们先说说背景,这篇文章到底想解决什么问题。如果你写过调用大模型 API 的代码,尤其是那种需要模型返回结构化数据的场景,你一定遇到过这样的烦恼。你给模型一个提示词,告诉它,请按照这样的 JSON 格式给我返回结果,字段叫什么,类型是什么,你写得清清楚楚。模型大多数时候确实会照做,可总有那么几次,它会给你返回一段看起来差不多但又不完全对的东西。可能是某个字段名字打错了一个字母,可能是数字被包在了字符串的引号里,也可能是它在 JSON 前面加了一句客套话,比如说,好的,这是您要的结果,然后才贴出真正的 JSON。对人来说这无所谓,你一眼就能看懂,但对程序来说,这种偏差往往就是一个解析异常,直接让你的下游代码崩溃。

这个问题在大模型刚开始大规模商用的时候,几乎是所有开发者的共同困扰。大家想了很多土办法来应对,比如说在提示词里反复强调,只返回 JSON,不要有任何多余的文字,或者写一堆兜底的正则表达式去清洗模型的输出,把多余的标点符号、多余的客套话都去掉,再尝试解析。还有更激进的做法,是在代码里写一个重试循环,如果解析失败就重新调用一次模型,指望它这次能规矩一点。这些办法都管用,但都不优雅,本质上是在用工程手段去弥补模型输出的不确定性,代码里堆满了各种 try except,各种 if 判断,看起来像是给一个本该精确的系统打了无数个补丁。

而且这个问题在简单场景下还好,一旦你的应用变得复杂,比如说你在做一个多智能体系统,好几个 Agent 之间要互相传递结构化的消息,只要中间有一环输出格式不对,整条链路就可能卡死或者出现脏数据,排查起来非常痛苦。再比如说你在做电商搜索,用户输入一句自然语言,模型要把它翻译成一个包含价格区间、品类、品牌、排序方式等等好多字段的搜索请求,这种结构稍微复杂一点,出错的概率就会明显上升。这就是这篇文章开篇就点出来的痛点,对于生产环境里的应用和智能体来说,数据格式上的一个小错误,就可能引发连锁式的失败。

好,背景说清楚了,我们来看这篇文章给出的核心方案,也就是结构化输出这个功能本身。

第一个要讲透的点,是这个功能到底承诺了什么。文章里用的词是保证,英文是 guarantee,这个词分量很重。它的意思是,只要你提供了一个模式定义,Claude 返回的内容就一定会严格符合这个模式,不是大概率符合,也不是经过提示词优化后大概率符合,而是结构上的硬保证。这个差别很关键。我们可以打个比方,以前的做法相当于你雇了一个非常靠谱但偶尔会走神的工人,你跟他说好了零件的尺寸标准,他大部分时候都能做对,但偶尔会因为分心做出一个尺寸偏差的零件,你需要在质检环节把这些次品挑出来,返工或者扔掉。结构化输出相当于给这个工人配了一个模具,不管他状态好不好,只要他往模具里灌的是合格的原料,出来的零件形状就一定是对的,因为是模具本身在物理上限制了结果的形状,不是靠工人自觉。这种从概率层面的"大概率正确"到机制层面的"结构性正确"的跃迁,是这个功能最核心的价值。

第二个要讲的点,是它具体怎么用。文章里提到,结构化输出有两种使用方式,一种是直接基于 JSON 模式,另一种是基于工具定义。我们分别说一下。

第一种方式,你在调用 API 的时候,直接在请求里提供一份模式定义,这份定义描述了你期望的输出结构,比如说有哪些字段,每个字段是什么类型,是字符串还是数字还是布尔值,有没有嵌套的对象或者数组,哪些字段是必填的。模型在生成回复的时候,会被这份模式约束住,出来的结果一定能够直接被解析成符合这份定义的数据结构。这种方式特别适合那些你明确知道自己要什么形状的数据的场景,比如说从一段文本里抽取出人名、地点、时间这几个字段,你提前定义好这几个字段的模式,扔给模型,出来的结果直接就能往数据库里写,不需要任何中间清洗的步骤。

第二种方式,是通过工具定义来实现。如果你用过 Claude 的工具调用功能,应该知道你可以给模型一组工具的描述,每个工具有自己的名字、用途说明,以及需要传入的参数结构。模型在判断需要调用某个工具的时候,会生成一份符合该工具参数定义的调用请求。结构化输出在这里的作用,就是确保模型生成的工具调用参数,百分之百符合你定义的工具参数模式,不会出现参数名写错、参数类型不对、缺少必填参数这些问题。这个对于做智能体系统的开发者来说特别重要,因为智能体系统里模型经常要调用各种外部工具,查天气、查数据库、发邮件,这些工具背后往往对接的是真实的业务系统,一旦调用参数出错,轻则报错,重则可能引发一些不可逆的操作。有了结构化输出的保证,你就不需要再在每个工具的执行代码前面加一层参数校验和兜底逻辑了。

第三个要讲透的点,是文章特别强调的一句话,这个功能在保证结构正确的同时,不会对模型本身的表现产生影响。这句话听起来简单,但其实背后是有技术含量的。你可能会想,如果我强行约束模型的输出格式,会不会让模型在内容质量上打折扣,比如说它本来想表达一个更细致的判断,但因为模式里只允许填一个简单的枚举值,它的真实意图被压缩了,导致回答变得更生硬或者更不准确。这是做约束生成这件事一直要面对的权衡,你既要保证格式,又不能让格式的紧箍咒束缚住模型的判断力。Anthropic 在文章里明确说,结构化输出不会带来模型表现上的损耗,这其实是在告诉开发者,你可以放心地把这个功能用在对准确性要求很高的场景里,不用担心为了拿到规整的格式而牺牲掉答案本身的质量。这也是为什么后续的更新里,他们会专门强调"支持更复杂的模式",因为越复杂的模式,对这种"既要保证格式又不能伤害内容"的能力要求就越高。

第四个点,我们来说说文章里点名的三个典型应用场景,这三个场景其实也代表了结构化输出这个功能最值得投入的方向。

第一个场景是数据抽取。这个很好理解,很多公司用大模型来处理非结构化的文本或者图片,比如说扫描合同提取关键条款,处理客户反馈提取情感和诉求,处理财报截图提取财务数据。这些场景的下游往往是一套已经存在很久的系统,数据库表结构、字段名字、字段类型都是早就定好的,模型的输出必须严丝合缝地对上,任何一个字段类型不对都可能导致写入失败。结构化输出在这里相当于给模型和老系统之间架了一座桥,模型这边按照你定义的模式输出,老系统那边按照原来的方式接收,中间不需要任何人工兜底的转换逻辑。

第二个场景是多智能体架构。这个是这两年特别火的方向,简单说就是不再用一个模型单打独斗,而是设计多个专门化的 Agent,各自负责一部分任务,然后互相协作。比如一个负责检索资料,一个负责整理摘要,一个负责生成最终报告。这种架构里,Agent 之间传递的往往不是自然语言,而是结构化的中间结果,比如检索 Agent 要告诉摘要 Agent,这是我找到的三篇文档,每篇的标题、来源、相关度分别是多少。如果这个传递环节格式不稳定,整个流水线就容易在某个环节卡住,而且这种故障往往很难排查,因为出错的不是某一次调用,而是某种格式漂移在多次调用里偶尔发生,你很难复现。结构化输出能让这种 Agent 间通信变得像调用一个内部接口一样可靠,这对于把多智能体系统真正推向生产环境,是一个很关键的基础设施补丁。

第三个场景是复杂搜索工具。文章举的例子是,需要填入多个搜索字段并且符合特定模式的场景。我们可以想象一个电商或者招聘网站的智能搜索助手,用户说一句,我想找一份在上海、薪资在两万到三万之间、远程办公为主的产品经理岗位,模型需要把这句话翻译成一个结构化的查询条件,里面包含城市、薪资下限、薪资上限、工作方式、职位类别等等好几个字段,而且每个字段可能还有自己的取值范围或者格式要求,比如薪资必须是数字,工作方式必须是预先定义好的几个枚举值之一。这种场景字段一多,出错概率就会指数级上升,结构化输出能从根本上消除这类错误。

文章里还引用了 OpenRouter 这家公司首席运营官的一段话,他说结构化输出已经成为智能体技术栈里非常有价值的一部分,因为智能体不断地在消费和生产结构化数据,Anthropic 的这个功能填补了开发者面对的一个真实空白,智能体工作流可以稳定地、每一次都可靠地运行,团队可以把精力放在客户身上,而不是去调试工具调用的问题。这段引用其实也从侧面说明了一个事实,就是这个功能不是 Anthropic 自己关起门来想出来的需求,而是整个智能体生态里大家都在用脚投票确认的一个真实痛点,OpenRouter 作为一个聚合了几百万开发者、对接所有主流模型的平台,他们的反馈是很有代表性的。

接下来我们说说这个功能从发布到现在的演进时间线,这部分内容文章里是用更新说明的形式补充进去的,我觉得很值得专门讲一下,因为它能帮我们判断现在这个功能到底处在什么样的成熟度。

最早是二〇二五年十一月十四日,结构化输出以公开测试版的形式发布,支持的模型是 Sonnet 四点五和 Opus 四点一,这是当时 Anthropic 比较新的两款模型。公测阶段意味着功能能用,但还在收集反馈、打磨细节,可能还有一些边界情况没有覆盖到。接着是二〇二五年十二月四日,文章更新说现在也支持 Haiku 四点五了,这个支持不仅限于 Claude 开发者平台原生调用,还覆盖到了微软的 Foundry 平台。这一步很重要,因为 Haiku 系列一直是 Claude 家族里偏轻量、偏高性价比的选择,很多对成本敏感、调用量很大的场景,比如说大批量的数据抽取任务,往往会优先选 Haiku 这类小模型,把结构化输出覆盖到 Haiku,意味着这个功能真正下沉到了高并发、低成本的生产场景里,而不只是停留在用最强模型做实验的阶段。

再往后是二〇二六年二月四日,也就是今年年初,文章更新说结构化输出现在正式可用了,英文是 generally available,也就是我们常说的 GA。GA 这个状态在软件行业里意味着什么,大家应该都清楚,它代表这个功能已经经过了公测阶段的检验,稳定性和覆盖面都达到了可以放心用在生产环境的程度,厂商也愿意为它的稳定性背书了。而且这次 GA 不只是把"公测"的帽子摘掉,还顺带带来了一个实质性的升级,支持更复杂的模式了。这点很关键,因为公测阶段往往会先支持比较简单的、扁平的模式结构,复杂的嵌套结构、复杂的校验规则,通常是在功能成熟之后才补上的,毕竟越复杂的模式,对"保证输出一定符合"这件事的技术难度就越高。同时这次 GA 还覆盖到了 Amazon Bedrock 平台,也就是说不只是 Anthropic 自己的开发者平台,亚马逊云上跑 Claude 模型的开发者,现在也能用上这个功能了,并且是三款模型同时覆盖,Sonnet 四点五、Opus 四点五,还有 Haiku 四点五。

我们简单梳理一下这条时间线背后的逻辑,你会发现这是一个很典型的功能成熟路径,先在最新最强的两款旗舰模型上小范围公测,验证思路是否可行,然后扩展到轻量级模型,覆盖更多生态平台,最后正式转正,同时补上更复杂场景的支持,并且铺到云厂商的托管平台上。如果你是在做技术选型决策的工程师,看到这样一条时间线,基本可以判断,这个功能现在已经不是一个尝鲜性质的实验特性了,而是一个可以放心写进生产代码里的基础能力。

好,讲完了核心内容,我们来聊聊这件事对大家实际工作意味着什么,也就是实践应用这部分,这也是我想多花一些篇幅展开讲的地方,因为知道一个功能存在和真正把它用好,中间还是有不少细节要注意的。

首先,如果你现在的项目里,还在用提示词工程加正则表达式清洗加重试循环这套组合拳来应对模型输出格式不稳定的问题,我的建议是,认真评估一下把这部分逻辑切换到结构化输出的可行性。这不是说提示词工程从此就没用了,提示词依然决定了模型理解任务的方式、生成内容的质量,但格式正确性这件事,完全可以交给结构化输出这种结构性的保证机制去解决,不需要再靠提示词里反复念叨"请务必只返回 JSON"这种软约束。这两者其实是互补的,提示词管内容,结构化输出管形状,分工清楚之后,你的代码会干净很多。

具体到代码层面,我个人认为最大的改变,是你可以删掉一大批防御性代码了。以前为了应对模型偶尔输出格式不对的情况,很多团队会写一层专门的解析容错逻辑,捕获解析异常,做字符串清洗,做重试,甚至要写日志去记录到底是哪种异常格式出现了,方便后续排查和优化提示词。这些代码本身不创造业务价值,纯粹是为了应对不确定性而存在的,是一种典型的技术债。当格式正确性变成了平台层面的硬保证之后,这些代码理论上就可以大幅简化甚至删除,你的解析逻辑可以直接假设输出一定是合法的,直接反序列化就行。这不仅减少了代码量,还间接减少了潜在的 bug,因为容错逻辑写复杂了反而容易引入新的问题,比如说清洗规则把本来合法的内容也清洗掉了。

第二个值得注意的实操点,是模式设计本身。结构化输出能保证模型严格遵守你给的模式,但模式设计得好不好,依然取决于你自己。如果你的模式定义得过于死板,比如说所有字段都设成必填,但实际业务里某些字段确实可能拿不到值,那模型在没有信息的情况下,可能会被迫填入一些没有意义的占位值,这反而会污染你的数据。所以拿到这个能力之后,第一件事不是马上把所有接口都套上一个一刀切的模式,而是花点时间想清楚,哪些字段是真正必须有的,哪些字段允许为空,哪些字段适合用枚举值约束取值范围,这其实是在把以前隐藏在提示词描述里的业务规则,显性化地写进模式定义里,这个过程本身对梳理业务逻辑也是有好处的。

第三个点,是关于工具调用场景的实操建议。如果你在做智能体应用,工具定义这条路径尤其值得关注。以前我们设计工具参数的时候,经常会很谨慎,生怕参数定义得太复杂模型理解不了,或者太多嵌套结构模型容易出错,所以很多团队会刻意把工具参数设计得很扁平、很简单,牺牲一些表达力来换取稳定性。现在有了结构化输出的保证,这种顾虑可以适当放开一些,你可以设计更贴近真实业务需求的参数结构,因为保证机制会替你兜住格式正确性这一层。当然这不意味着可以无限堆复杂度,模式越复杂,模型理解任务本身的难度也会上升,这是认知层面的挑战,不是格式层面能解决的,这两者要分开看。

第四个点,我想提一下成本和模型选择的搭配问题。前面我们提到结构化输出现在已经覆盖到了 Haiku 系列,这意味着很多对成本敏感的批量任务,比如说每天要处理几十万条客服工单提取关键信息,完全可以考虑用 Haiku 加结构化输出的组合,而不需要为了格式稳定性去选用更贵的旗舰模型。这是一个挺实际的成本优化思路,以前可能有团队会觉得,小模型偶尔输出格式不稳,所以宁可多花钱用大模型来减少出错率,现在这个权衡的前提变了,格式稳定性已经不再是靠模型能力大小来保证的事情了,而是平台层面的硬约束,所以选模型的时候,可以更纯粹地基于内容质量和成本去考虑,不用再把格式稳定性也算进模型选择的变量里。

第五个点,是关于平台覆盖的问题,如果你的部署环境是在 Amazon Bedrock 或者微软的 Foundry 上,而不是直接走 Anthropic 自己的开发者平台,记得去确认一下你所在的云平台对结构化输出的支持版本和范围是否和官方原生平台完全一致,因为不同托管平台的功能上线节奏可能会有一些时间差,这点在文章的更新记录里其实也能看出来,Haiku 四点五在微软 Foundry 上的支持,和原生支持是一起公布的,但更复杂模式的支持是不是所有平台都同步跟上了,这个细节最好去查一下官方最新的文档,而不是只参考这篇半年前的文章。

第六个点,也是我觉得容易被忽略的一点,是不要把结构化输出当成万能药。它解决的是格式层面的正确性问题,但它不能保证内容层面的正确性,也就是说模型给的字段格式是对的,但字段里填的内容是不是符合事实、是不是符合业务逻辑,这依然需要你自己的校验逻辑去把关。比如说你让模型抽取一份合同里的金额字段,结构化输出能保证这个字段一定是个合法的数字,但不能保证这个数字一定是合同里真实写的那个金额,万一模型理解错了合同条款,抽取出来的数字格式完全合法,但内容是错的。所以格式校验和内容校验是两件不同的事,千万不要因为用了结构化输出,就把原来内容层面的审核和抽样检查也一并省掉了。

好,我们最后来总结一下整篇文章。这篇文章核心讲的是,Anthropic 在二〇二五年十一月给 Claude 开发者平台带来了结构化输出这个功能,它能保证模型的回复严格符合开发者预先定义的 JSON 模式或者工具定义,从根本上消除了以前那种偶尔出现的格式解析错误和工具调用失败问题。这个功能有两种使用方式,直接基于 JSON 模式,或者基于工具调用定义,适用的场景集中在数据抽取、多智能体协作和复杂搜索系统这三大类,官方特别强调这个保证不会牺牲模型本身的表现质量。从发布至今,这个功能已经从最初支持两款旗舰模型的公测版本,演进成了覆盖三款模型、支持更复杂模式、并且铺到了亚马逊和微软云平台的正式可用功能。

我个人认为,这个功能的意义不在于它有多少炫酷的新花样,而在于它把一个原本属于工程师自己用胶水代码缝合起来的脆弱环节,变成了平台层面的基础设施保证。这种从应用层兜底到平台层承诺的转变,往往是一个技术方向真正走向成熟的标志。我们这一两年看到大模型应用开发逐渐从拼凑提示词、堆砌补丁,转向更工程化、更可预测的方向,结构化输出就是这个大趋势里很具体的一块拼图。如果你正在用 Claude 做生产级的应用,尤其是数据抽取或者智能体系统,我建议你真的去试一下这个功能,亲自感受一下少写多少容错代码,这种体感往往比读十篇文章都更有说服力。
