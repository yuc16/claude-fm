## Script

今天这期节目，我们聊的是 Anthropic 官方博客上的一篇技术文章。文章的英文标题是「New in Claude Managed Agents: dreaming, outcomes, and multiagent orchestration」，我把它翻译成中文，叫做「Claude 托管智能体的三项新能力：梦境记忆、结果导向与多智能体编排」。这篇文章发表于二零二六年五月六日，距今大约六周的时间，属于相当新鲜的一手资讯。

先说说为什么今天选了这篇文章。它的篇幅不长，但信息密度非常高。Anthropic 一口气宣布了 Claude 托管智能体平台上的三个重要新特性，而这三个特性解决的，恰好是智能体在真实落地过程中最核心的三个工程难题——记忆的持续性、输出质量的稳定性，以及大规模复杂任务的协同能力。不管你现在是一线做 agent 开发的工程师，还是在评估要不要把 AI 智能体引入自己产品的产品经理，或者只是想了解 AI agent 这个方向正在往哪里走，今天这期节目都值得你认真听一下。

我们先来聊聊背景，来理解这次更新要解决什么问题。

过去一年多，整个 AI 行业对智能体的热情可谓有增无减。从各种 agent 框架的井喷，到大厂纷纷推出自己的 agent 平台，这个领域的发展速度非常快。但在真实落地的过程中，工程师们踩的坑也越来越清晰，有几个问题反复出现。

第一个问题是记忆的断层。智能体做完一个任务，换一个 session 就像失忆了一样，从零开始。上次它学到了这个客户喜欢简洁的报告格式，这次又开始啰嗦；上次它踩了一个坑，这次还会掉进去。这种「每次都是新人」的状态，让智能体很难在长期运营中积累经验、持续改进。

第二个问题是输出质量的不稳定。你不知道智能体什么时候会给你一个还过得去的答案，什么时候会给你一堆错误。更麻烦的是，很多任务的质量标准本身就比较主观，比如文案有没有符合品牌调性，报告有没有达到内部要求，这类东西很难用简单的规则来检查，只能靠人来审核，效率很低。

第三个问题是规模化的上限。遇到真正复杂的任务，一个智能体的上下文窗口、处理速度和能力范围都是有限的，根本应付不来。但怎么让多个智能体协同工作，既能并行提升效率，又能保证协作的质量和可追溯性，这本身就是一个工程难题。

Anthropic 这次推出的三个特性，dreaming、outcomes 和多智能体编排，正好分别对应这三个问题。让我们一个一个来拆解。

我们先聊「dreaming」，也就是「梦境」这个功能。这个名字起得很有意思，我很欣赏这个命名背后的隐喻。你知道人类的记忆是在睡眠过程中被整理和强化的吗？大脑会在你睡着的时候，把白天碰到的零散信息进行归类、压缩，和长期记忆融合，把重要的东西巩固下来，把不重要的东西逐渐淡化。这个过程叫做记忆巩固。dreaming 这个特性的运作机制，跟人脑的这个过程非常相似。

具体来说，dreaming 是一个定时在后台运行的进程，它会回顾你的智能体过去的所有对话记录，从里面提取出规律性的模式，然后对记忆库进行整理和更新，让你的智能体在下一次工作的时候能够表现得更好。

你可能会问，这跟 Claude 托管智能体平台原来就有的 memory 功能有什么区别？对，平台原本就有记忆功能，智能体在工作的过程中可以把学到的东西写入记忆库，供下次使用。但这种「工作中记忆」有几个固有的局限。

第一，它只能捕捉到智能体在当前 session 里自己注意到的东西。有一些规律，只有把很多次 session 的数据放在一起看才能发现，单次 session 里根本感知不到。比如说，智能体在处理某类文档时反复犯同一种错误，但每次 session 里只犯了一次，不够显眼，智能体不会特别去记它。但如果把二十次、三十次 session 放在一起看，这个规律就会非常突出。

第二，随着时间推移，记忆库里的内容会越来越杂，信噪比越来越低。早期记录的一些东西可能已经过时了，但还留在记忆库里占位置，甚至会干扰智能体的判断。记忆库如果不定期整理，它的质量会随着时间推移下降，而不是提升。

dreaming 解决的就是这两个问题。它在 session 和 session 之间运行，能做到几件单个智能体在工作中做不到的事。它能发现跨 session 的规律，能提炼出不同智能体之间的共享学习——这在多智能体场景下特别有价值，一个团队里不同智能体踩过的坑、摸索出的最佳实践，都可以通过 dreaming 汇聚到一起，让所有智能体都受益。同时，dreaming 会主动整理记忆库，把过时的、冗余的、低价值的记忆清理掉，让记忆库随着使用时间越来越精炼，而不是越来越臃肿。

这里有一个设计细节很贴心：你可以自己决定 dreaming 的自动化程度。它可以在检测到有价值的规律之后，自动更新记忆库，让改变立刻生效；也可以把改动提交给你审核，你确认没问题之后才落地。这个选项对于那些对智能体行为变化比较敏感的场景来说非常重要，特别是法律、医疗、金融这类高风险行业，你肯定不希望智能体在没有人类确认的情况下自己悄悄改变行为方式，哪怕这种改变是「改进」。

说到这里，我们来看看 Harvey 的实际案例。Harvey 是一家专注于法律 AI 的公司，他们用 Claude 托管智能体来协助处理复杂的法律工作，包括长篇法律文书的起草和文件创建。这类任务有一个特点：细节繁多，规范性强，对准确性的要求极高，而且每个客户、每种案件类型都有各自的特殊要求和处理惯例。在引入 dreaming 之后，他们的智能体能够在 session 之间保留学到的东西，包括特定文件格式的处理技巧和特定法律工具的使用经验。结果呢，他们测试下来，任务完成率提升了大约六倍。

六倍这个数字非常惊人，我第一次看到的时候也愣了一下。你想想，从一倍到六倍，意味着以前十件任务里可能只有一两件能完成到满意的水准，现在有六七件能达到同等标准。这背后的含义是：智能体不需要在每次任务里重新摸索已经摸索过的东西，它一上来就知道应该怎么处理，把精力集中在真正需要思考的部分。

好，我们再来聊第二个特性：outcomes，也就是结果评估系统。

outcomes 的核心思路可以用一句话来概括：告诉智能体什么叫做「好的结果」，然后让它自己对照标准来检查和修正自己的输出，直到达标为止。

具体的工作流程是这样的：你作为开发者，写一份评分标准，描述成功的输出应该具备哪些特征。智能体完成任务之后，会有一个独立的评估器在自己独立的上下文窗口里，对智能体的输出进行打分。这里有一个关键的设计决策：评估器和智能体是完全分开运行的，评估器不会看到智能体的推理过程，它只看最终的输出结果。这样做是为了避免评估器被智能体的中间推理过程带偏，让评估结果更客观、更公正。

如果评估器认为输出不达标，它会明确指出哪里有问题，智能体就根据这些反馈重新做，再提交，评估器再打分，循环往复，直到输出通过评分标准为止。

你可能会想，这不就是让智能体「重试」吗？其实没那么简单，区别非常大。普通的重试是盲目的，智能体不知道哪里出了问题，只是再做一遍，结果很可能差不多，甚至会犯同样的错误。而 outcomes 的机制是：评估器会明确指出具体问题在哪里，比如「这份报告的数据分析部分缺少对异常值的解释」，或者「这段代码没有处理空列表输入的边界情况」。智能体知道了具体问题所在，才能有针对性地改进，而不是漫无目的地重做。

更有意思的是，outcomes 不仅能处理有明确标准的客观任务，还能处理相对主观的质量评估。比如文案有没有符合品牌的语气和调性？设计有没有遵循视觉规范？这类东西用代码很难检查，但如果你把标准描述得足够清晰、足够具体，outcomes 同样能够评估，而且评估的结论是可以解释的，智能体知道哪里不符合标准，知道要往哪个方向调整。

关于效果，Anthropic 的内部测试数据显示，outcomes 能把任务成功率提升最多十个百分点，而且提升最明显的是难度最高的那些问题——这符合直觉，越难的任务，一次就做对的概率越低，自我纠错机制能发挥的空间越大。在文件生成方面，docx 格式的任务成功率提升了八点四个百分点，pptx 格式的任务成功率提升了十点一个百分点。

我个人认为，outcomes 这个特性的意义不只是把数字提升了几个点，更重要的是它改变了智能体和人类之间的协作方式。以前，智能体做完任务，人类来检查，不满意就让它重做——这个反馈循环以人类为中心，每一次迭代都需要人类参与。而 outcomes 让智能体能够自我检查、自我纠正，人类不需要逐次介入，只需要在最前期把质量标准定清楚，后续智能体就会自动朝着这个标准收敛。这是从「人类审核每一次输出」到「人类定义标准、机器自主执行」的范式转变，对于那些输出量大、质量要求高的场景，这个差别是巨大的。

文章里还提到了一个很实用的附加能力：现在你可以定义一个 outcome，让智能体开始跑，然后通过 webhook 在任务完成时收到通知。这意味着智能体可以在后台独立工作，不需要你一直盯着它，等它处理完了主动通知你来验收。对于那些耗时较长的任务来说，这个能力非常实用。

我们来看 Wisedocs 的案例来感受一下实际效果。Wisedocs 构建了一个文档质量检查智能体，专门用来对文档审查结果进行评分，评分标准是他们内部的质量规范。引入 outcomes 之后，审查速度提升了百分之五十，同时还能保持与团队标准的对齐。速度快了一倍，质量没掉——这个结果很能说明问题。

好，我们来聊第三个特性，也是三个里面工程深度最高的：多智能体编排，也就是 multiagent orchestration。

当一个任务的规模超出了单个智能体能够处理的范围，怎么办？多智能体编排给出的答案是：让一个主智能体负责统筹规划，把大任务拆分成若干个子任务，分发给各自专门的子智能体来并行处理，最后再由主智能体汇总结果。

这个模式听起来跟人类团队的分工协作非常相似——有一个项目经理，负责拆解需求、分配工作、把控整体进度；有若干专家，各自负责自己最擅长的那部分。文章里举了一个例子：主智能体负责调查一个系统问题，同时派出多个子智能体，分别去检索部署历史、分析错误日志、查看性能指标和用户支持工单。这些子智能体并行工作，大幅缩短了整个调查的时间，而主智能体在收到各方报告后综合做出判断。

这里有几个工程细节值得特别关注。

第一个细节是，每个子智能体可以有各自独立的模型、提示词和工具配置。这意味着你可以根据不同子任务的特点来选择最合适的模型，不需要所有智能体都用同一个模型。这一点很重要，因为不同的模型在能力、速度和成本之间的取舍是不一样的。

Spiral by Every 的做法就很有代表性。Spiral 是 Every 推出的写作辅助产品，他们的写作智能体架构是这样的：主智能体跑在 Haiku 上，负责接收用户的写作请求、在必要时问一些快速的澄清问题，然后把真正的写作任务分发给跑在 Opus 上的子智能体来完成。Haiku 速度快、成本低，非常适合处理轻量的调度工作；Opus 能力最强，适合高质量的创作任务。当用户一次性要求多份草稿的时候，多个 Opus 子智能体会并行运行，同时生成。这种根据任务复杂度来选择模型的策略，在输出质量和运行成本之间找到了一个非常好的平衡点。

第二个细节是共享文件系统。子智能体之间可以共享一个文件系统，并且各自向主智能体的上下文贡献内容。这意味着各个子智能体不是完全孤立的，它们的工作成果可以被其他智能体看到和利用，整个多智能体系统是一个有机的整体，而不是一堆孤立运行的个体。

第三个细节是事件的持久化。每个智能体都记得自己做过什么，而且这些记录是持久保存的。这一点在处理复杂的多智能体工作流时非常关键。想象一下，如果一个子智能体在任务中途失败了，你需要知道它在哪里失败的、已经完成了哪些步骤，这样才能恢复或者有针对性地重试。有了持久化的事件记录，这种错误恢复就变得可行了。

第四个细节是可观察性。文章里提到，你可以在 Claude Console 里追踪整个多智能体工作流的每一步执行过程：哪个智能体做了什么，按什么顺序做的，为什么这么做。这种全链路的可追踪性对于调试复杂的多智能体系统来说非常关键。你想象一下，如果你同时有五个子智能体在并行工作，任务最后出了问题，如果没有可视化的追踪工具，你根本不知道是哪个环节出了错，调试就是大海捞针。有了这种可观察性，调试就从大海捞针变成了按图索骥。

来看 Netflix 的案例。Netflix 平台团队构建了一个日志分析智能体，用来处理来自数百个构建任务、不同数据源的日志。他们面对的核心挑战是：一次代码变更可能影响数千个应用，出了问题，你不可能逐条去看每一条日志，你需要的是快速找到跨多个构建任务反复出现的问题模式，那才是真正需要行动的信号。多智能体编排让这个分析智能体能够把大量日志拆分成若干批次，派出多个子智能体并行处理，然后再汇总找出真正值得关注的规律。

从架构角度看，这本质上是一个经典的 map-reduce 模式：多个子智能体做 map，把大量日志分片处理；主智能体做 reduce，把各个子智能体的发现汇总成可操作的洞察。只不过 map 和 reduce 的执行单元从传统的计算节点变成了 AI 智能体。

我们再多说说 Spiral by Every 这个案例，因为它是唯一一个同时用上了这次更新的三个特性的例子，非常有代表性。

Spiral 的写作智能体里，Every 的编辑原则和用户的个人写作风格都储存在记忆库里。每当用户提交一个写作请求，Haiku 上的主智能体先接收请求，在需要的时候问一些简短的跟进问题，然后把写作任务委托给 Opus 上的子智能体来执行。如果用户要多份草稿，多个子智能体会并行生成。每份草稿生成完之后，都要经过 outcomes 的评分——评分标准是 Every 的编辑原则，以及从记忆库里调取的这位用户的个人声音和风格偏好。只有达到评分标准的草稿才会被返回给用户。

你看，这个架构把三个特性串成了一个完整的闭环：记忆提供个性化的上下文，让智能体了解这个用户的写作风格；多智能体编排提供并行生成的能力，让多份草稿可以同时生成而不是排队等待；outcomes 保证每份返回给用户的草稿都达到质量底线。三个特性各司其职，共同支撑起一个高质量的写作辅助体验。

好，聊完了这三个特性和相应的案例，我们来谈谈对实际工作的影响。如果你是一个在做 agent 开发的工程师，这些特性意味着什么，你应该怎么用，要避什么坑？

先说 dreaming。这个特性目前还在 research preview 阶段，需要申请才能获得访问权限，不是对所有开发者开放的。但即便你现在还用不上，也可以提前想好几件事：你的智能体在哪些类型的任务上会反复犯同样的错？有没有一些你希望智能体能够记住的最佳实践？你的用户有没有特定的偏好，是否希望智能体随着时间推移越来越了解他们的需求？如果有这些场景需求，dreaming 很可能能给你带来显著的收益。

不过这里有一个潜在的风险需要注意：dreaming 会主动修改智能体的记忆库，如果记忆的更新出了问题——比如把一个错误的规律固化下来了——可能会在后续所有的 session 里传播这个错误。这就是为什么 Anthropic 提供了「人工审核后再落地」的选项。对于关键业务场景，我建议你在一开始一定要开启人工审核模式，对 dreaming 的行为模式有了足够的了解和信任之后，再评估是否让它完全自动运行。不要因为「自动化」听起来很方便就跳过这个步骤。

再说 outcomes。这个特性现在是 public beta，大家都可以用。评分标准的设计是这里最关键的环节，也是最容易出问题的地方。一个好的评分标准必须是具体的、可操作的，而不是模糊的形容词堆砌。「输出质量高」是一个非常差的评分标准，因为它根本没有告诉智能体什么叫做「质量高」，评估器也无从判断。好的评分标准应该是「报告的每一个关键结论都必须有具体的数据支撑，且数据来源需要注明」，或者「文案中不得出现品牌禁用词列表中的任何一个词」这样具体的描述。越具体，评估器的评分就越稳定，智能体的改进方向就越明确。

另外，outcomes 的自我评分和重做循环会增加延迟和成本，因为每次评估和重做都需要额外的推理时间和 token 消耗。如果你的任务对响应速度要求很高，需要毫秒级的实时返回，这可能不是最合适的方案。但对于那些可以异步执行、对质量要求更高于对速度要求的任务，outcomes 的价值就非常显著了。配合 webhook 通知，你可以让任务在后台跑，不影响主流程，完成了自动通知你来处理结果。

关于多智能体编排，现在也是 public beta 阶段，可以直接使用。这里有一个很常见的误区需要提醒大家：并不是所有任务都适合拆给多个智能体来做。如果你的任务本身是高度串行的，每一步都强依赖前一步的结果，中间没有可以并行处理的空间，那么多智能体的优势就完全发挥不出来，反而会增加编排的复杂度和通信开销。

多智能体最适合的场景是：任务可以被清晰地拆分成若干个相互独立的子任务，这些子任务之间没有强依赖关系，可以并行执行，最后的汇总工作比较自然。日志分析、并行搜索和信息收集、多视角评估、批量文件处理，都是非常好的应用场景。

关于可观察性，我要重复强调一遍：多智能体系统的调试比单智能体复杂得多，一定要充分利用 Claude Console 提供的全链路追踪功能。特别是在开发和测试阶段，每一步的执行记录都是你排查问题、优化流程的重要依据。不要等到生产出了问题才去翻日志，要在开发阶段就养成看执行追踪的习惯。

还有成本的问题。多智能体编排意味着同时运行多个模型推理任务，成本会随着子智能体数量线性甚至超线性地增长。在做架构设计的时候，一定要把成本纳入考量。前面提到的 Spiral 的做法——用 Haiku 做调度、Opus 做核心任务——是一个值得借鉴的思路：根据不同子任务的复杂度选择能力匹配的模型，不要一刀切地给所有智能体都用最强的模型，这样可以在保证整体质量的前提下显著降低成本。

聊到这里，我想从更宏观的角度谈一谈我个人对这次更新的整体判断。

这三个特性放在一起，我个人认为代表了 Anthropic 在 agent 基础设施上的一个清晰的产品思路：让 agent 更像一个真正的「同事」，而不只是一个「工具」。工具是被动的，你每次告诉它怎么做它才去做，它不会主动学习，不会主动检查自己的工作，不会和其他工具协商分工；同事是主动的，它会从过去的经验中学习，会主动检查自己的工作质量，会和其他同事协作来完成超出单人能力范围的任务。dreaming 对应的是「学习成长」，outcomes 对应的是「自我负责」，multiagent 对应的是「团队协作」——这三个维度恰好是一个优秀同事区别于普通工具的核心能力。

从更宏观的行业角度来看，dreaming、outcomes 和多智能体编排解决的，其实都是同一个核心命题：如何让 AI agent 在尽量少的人类干预下，持续产出高质量的工作。这是 agent 从「好看的 demo」走向「真正可以用于生产的系统」必须跨过的门槛。从这些特性的设计思路来看，Anthropic 显然对这个目标想得比较清楚，而且已经有了一些经过真实客户验证的结果。

Harvey 的完成率提升了六倍，Wisedocs 的速度快了百分之五十，outcomes 带来了最多十个百分点的成功率提升——这些数字不是实验室里漂亮的基准测试，而是来自真实业务场景的实测数据，说服力更强。

当然，有一点我也要提醒大家：这些特性目前还处于 beta 或者 research preview 阶段，意味着 API 接口可能还会调整，一些边界情况可能还没有被充分处理，文档也在持续完善中。如果你要在生产环境里用，要做好一定的心理准备，密切关注官方文档的更新，遇到问题及时反馈。

好，今天这期节目就聊到这里。我们来做一个最后的总结。

Anthropic 的 Claude 托管智能体平台推出了三个重要的新特性。dreaming 让智能体能够像人类大脑在睡眠中整理记忆一样，在 session 之间自我学习和改进，跨 session 提炼规律、清理低价值记忆、汇聚多智能体的共同学习，Harvey 法律 AI 用后完成率提升了约六倍。outcomes 给智能体提供了一套自我评分机制，让它能够对照你写的评分标准检查和修正自己的输出，直到达到质量要求为止，Wisedocs 用后速度快了百分之五十、质量没有下降。多智能体编排让主智能体能够统筹规划、把复杂任务拆分给多个专属子智能体并行处理，每个子智能体可以有自己最合适的模型配置，Netflix 用它处理数百个构建日志的并行分析，Spiral 把它和另外两个特性结合起来，构建了一个高质量的写作辅助系统。

三个特性合在一起，指向同一个目标：让 AI agent 更像一个有学习能力、有质量意识、有协作能力的「同事」，而不只是被动执行命令的工具。如果你在做 agent 开发，这是一个值得认真研究和上手实验的更新方向。
