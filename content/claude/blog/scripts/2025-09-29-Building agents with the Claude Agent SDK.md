## Script

大家好,欢迎收听 Claude FM。今天我们要聊的这篇文章,标题是 Building agents with the Claude Agent SDK,中文可以翻译成"用 Claude Agent SDK 构建智能体"。这篇文章发表于二零二五年九月二十九日,作者是 Anthropic 团队的 Thariq Shihipar,还有好几位同事参与了编辑和补充。

先说一下时效性的问题。今天是二零二六年六月二十一日,这篇文章发表到现在已经差不多九个月了。在 AI 这个行业里,九个月已经不算短了,所以我们先简单回顾一下当时的背景,帮大家校准一下时间坐标。二零二五年九月前后,Claude Code 已经从一个内部的编程效率工具,慢慢演变成了 Anthropic 内部几乎所有重要智能体应用的底层引擎,这是这篇文章最关键的一个转折点。也就是说,这篇文章本质上是在宣布一次重新定位和改名:原来的 Claude Code SDK,正式改名为 Claude Agent SDK,因为它早就不只是用来写代码了。如果你最近几个月已经在用 Claude Agent SDK 做开发,那么文章里讲的很多东西可能你已经很熟悉,但是这篇文章的价值在于,它把整个设计哲学和最佳实践,系统地讲清楚了,这是值得我们今天专门拿出来精读一遍的原因。

我们先说说这篇文章要解决的问题,以及当时整个行业处在什么状态。在这之前一年左右,也就是二零二四年,Anthropic 团队发表过一篇很有名的文章,叫做"构建有效智能体的经验"。那篇文章总结了和客户一起打磨智能体的一些心得,但是当时整个行业对智能体的理解,还停留在"给模型一些工具,让它自己规划任务"的阶段。后来 Claude Code 发布了,这是一款主打编程效率的智能体产品。但是有意思的事情发生了:Anthropic 内部的同事们发现,Claude Code 不仅能写代码,还被拿去做深度研究、做视频创作、做笔记整理,五花八门的非编程任务都能干。换句话说,大家发现驱动 Claude Code 的那套底层机制,其实是一种通用的智能体引擎,根本不局限于写代码这一个场景。于是,这篇文章就应运而生,核心目的就是把这套底层机制,从"编程工具的内核"提升到"通用智能体框架"的高度,正式介绍给所有开发者。

那么这套机制最核心的设计哲学是什么呢?文章里讲得很直白,就一句话:Claude 需要的工具,和程序员每天用的工具是一样的。一个程序员要完成工作,需要在代码库里找到合适的文件,需要写代码、改代码,需要跑一下代码看看有没有报错,需要调试,需要反复迭代直到代码跑通为止。Anthropic 团队发现,只要给 Claude 访问用户电脑终端的能力,它就拥有了像程序员一样工作所需要的一切。这听起来好像是个很朴素的发现,但它背后的含义非常深远。因为一旦 Claude 拥有了终端、文件系统、运行命令这些能力,它就不再局限于写代码这件事,它可以读 CSV 表格文件,可以搜索网页,可以生成可视化图表,可以解读各种业务指标,总而言之,它可以做几乎所有需要在电脑上完成的数字工作。所以 Claude Agent SDK 的核心设计原则,用文章里的话讲,就是给你的智能体一台电脑,让它像人一样工作。

这个设计哲学听起来有点像什么呢?我打个比方,你可以把它想象成,你雇了一个新员工,但你不是把任务清单写在便利贴上让他凭空想象怎么完成,而是直接给他一台带着所有软件、所有权限的电脑,告诉他公司的文件夹结构是什么样的,然后让他自己摸索着去完成任务。这种思路和早期那种"对话式智能体"是完全不同的范式。对话式智能体更像是你打电话给客服,你说一句,他答一句,信息全靠你口头传达。而给智能体一台电脑的思路,则是让它自己动手,自己去翻文件、自己去验证、自己去发现问题,这种自主性是质的飞跃。

基于这个哲学,文章里举了几个典型的应用场景,我觉得很值得展开讲讲,因为这能帮我们想象自己手头的业务能怎么用这套思路。第一个是金融类智能体,可以做财务分析、做尽调材料整理这类工作。第二个是个人助理智能体,可以帮你订机票、管理日程、安排会议、整理简报,而且因为它能接入你的内部数据源,还能跨多个应用程序追踪上下文,这意味着它不是孤立地完成一个任务,而是能记住你昨天说过什么、今天的安排是什么。第三个是客服智能体,可以处理那些模糊性很高的客户请求,比如收集和审核用户数据、连接外部 API、给用户回消息,在必要的时候升级给人工客服处理。第四个是深度研究智能体,可以在海量文档集合里做综合性的调研,搜索文件系统、分析整合多个来源的信息、跨文件做交叉引用,最后生成详细的研究报告。这几个场景背后有个共性,就是它们都需要智能体长时间、多步骤地处理大量信息,而不是一问一答就能搞定的事情。

讲完了背景和设计哲学,我们来看这篇文章最核心的内容,也就是智能体的工作循环。文章提出,在 Claude Code 里,Claude 经常按照一个特定的反馈循环来工作:收集上下文,采取行动,验证工作,然后循环往复。这个三段式循环,其实可以套用到任何你想构建的智能体上面。文章用一个邮件智能体的例子,贯穿全文来讲解这个循环具体怎么落地,我们也跟着这个例子,把每个环节拆开讲清楚。

先说收集上下文这一环。文章强调一个观点很有意思,叫做"文件系统即上下文工程"。什么意思呢?就是说,你电脑里的文件和文件夹结构,本身就代表了一堆可能被拉进模型上下文窗口里的信息。当 Claude 遇到比较大的文件,比如日志文件,或者用户上传的文件,它会自己判断该怎么把这些内容读进上下文,用的方法就是调用一些经典的命令行工具,比如 grep 用来搜索关键字,tail 用来看文件末尾的内容。这意味着,你设计文件夹结构和命名方式的时候,其实就是在做上下文工程。拿邮件智能体打比方,如果你把过去的对话记录存放在一个叫"Conversations"的文件夹里,那么当用户问起以前聊过的事情,智能体就可以去这个文件夹里搜索,找到相关的上下文。这种检索方式,文章把它叫做 agentic search,也就是智能体主动地、像人一样去翻文件夹查找信息,而不是依赖提前建好的索引。

这里文章还专门对比了另一种检索方式,叫语义搜索。语义搜索一般速度更快,但是准确度较低,维护起来更麻烦,而且不够透明。它的原理是把相关内容切成一个个小块,把这些小块转换成向量做嵌入,然后通过查询向量来搜索相关概念。考虑到语义搜索的这些局限性,文章给出的建议是,优先从 agentic search 开始做,只有当你确实需要更快的检索速度,或者需要支持更复杂多变的查询方式时,才考虑加上语义搜索。这个建议我觉得对很多团队挺有指导意义的,因为现在很多团队一上来就搭建复杂的向量数据库和 RAG 检索管线,但其实很多场景下,让模型自己用类似 grep 的方式去翻文件,反而又快又准,维护成本还更低。

收集上下文的第二个重要机制,是子智能体,英文叫 subagents。Claude Agent SDK 默认就支持子智能体功能。子智能体有两个核心价值。第一是可以做并行处理,你可以同时唤起多个子智能体,让它们分头去做不同的任务,这样效率会高很多。第二是有助于管理上下文,因为每个子智能体都有自己独立、隔离的上下文窗口,它只会把跟主任务相关的精炼信息,传回给负责协调的主智能体,而不是把它自己看到的全部内容都塞回去。这就特别适合那种需要在海量信息里筛选有用内容的任务,因为大部分信息其实是没用的,如果全部塞进主智能体的上下文窗口,只会让它的注意力被稀释、效率变低。

还是拿邮件智能体打比方,假设我们给它配置了一个"搜索子智能体"的能力。这个邮件智能体就可以同时派出好几个搜索子智能体,让每一个子智能体针对你的邮件历史,跑一个不同的查询条件,然后只把真正相关的摘录内容返回给主智能体,而不是把整封整封的邮件原文都塞回来。你可以想象一下,如果你让一个人去图书馆,一次性查十个不同主题的资料,他肯定得分十次跑,每次只带回来最有用的几页笔记,而不是把十个书架的书全部搬回办公室。子智能体干的就是这个事情。

收集上下文还有第三个机制,叫做自动压缩,英文是 compact。当智能体运行的时间很长的时候,上下文的维护就变得非常关键,因为上下文窗口是有限的。Claude Agent SDK 的压缩功能,会在接近上下文上限的时候,自动把之前的消息总结压缩一下,这样你的智能体就不会因为上下文爆满而跑不动了。这个功能其实就是建立在 Claude Code 里那个压缩斜杠命令的基础上的。这三个机制,文件系统检索、子智能体、自动压缩,合在一起,构成了智能体收集和管理上下文的完整工具箱。

收集完上下文,下一步就是采取行动了,这一环节文章讲了四个具体的机制。

第一个是工具,英文叫 tools。工具是智能体执行动作最主要的building block,因为工具在 Claude 的上下文窗口里是非常突出、非常显眼的存在,这意味着当 Claude 决定怎么完成一个任务的时候,工具是它首先会考虑的主要行动选项。所以文章特别提醒,你设计工具的时候要非常谨慎,要想清楚怎么样能让上下文使用得更高效。换句话说,工具应该设计成你最想让智能体频繁执行的核心动作。拿邮件智能体来说,你可能会定义一个叫 fetchInbox 的工具,用来拉取收件箱,再定义一个叫 searchEmails 的工具,用来搜索邮件,这些就是这个智能体最高频、最核心的动作。

第二个是 Bash,也就是命令行终端。Bash 的价值在于,它是一个通用型工具,可以让智能体灵活地用电脑去完成各种各样的工作。还是拿邮件智能体打比方,假设用户的重要信息存在邮件附件里,比如一个 PDF 文件,Claude 完全可以自己写代码,把这个 PDF 下载下来,转换成文本,然后在里面搜索有用的信息。这种灵活性是预先定义死的工具做不到的,因为你不可能提前预判用户会遇到什么千奇百怪的文件格式和场景,但是给它一个通用的命令行权限,它就能现场写代码去解决这些长尾问题。

第三个是代码生成。文章里说,Claude Agent SDK 在代码生成这件事上表现得特别出色,这是有道理的,因为代码具有精确、可组合、可以无限复用的特点,这让它特别适合作为那些需要复杂可靠操作的智能体的输出形式。文章建议大家在设计智能体的时候,可以主动思考一下,哪些任务如果表达成代码会更有优势,往往这么一想,就能解锁很多新的能力。文章举了一个很实际的例子,就是 Claude.AI 里的文件创建功能,完全是依赖代码生成实现的:Claude 会写 Python 脚本去创建 Excel 表格、PowerPoint 演示文稿、Word 文档,这样才能保证格式的一致性,以及实现那些用其他方式很难做到的复杂功能。放到邮件智能体的例子里,如果你想让用户能够给收到的邮件设置自动处理规则,最自然的做法就是写一段代码,绑定到收件事件上去执行,这比靠自然语言描述规则、再让模型每次去解读规则要稳定得多。

第四个是 MCP,也就是 Model Context Protocol,模型上下文协议。MCP 提供的是标准化的外部服务接入方式,它会自动处理身份验证和 API 调用这些麻烦事。这意味着你可以把智能体接入 Slack、GitHub、Google Drive、Asana 这些服务,而不需要自己写一套定制的集成代码,也不需要自己折腾 OAuth 授权流程。放到邮件智能体的场景里,假设你想让它去搜索 Slack 消息来了解团队的背景情况,或者去检查 Asana 上的任务,看看是不是已经有人在处理某个客户的请求,有了 MCP 服务器,这些集成基本是即插即用的,你的智能体只需要调用类似 search_slack_messages 或者 get_asana_tasks 这样的工具,剩下的事情 MCP 都帮你搞定了。而且因为 MCP 生态正在快速成长,新的预制集成会不断涌现,这意味着你能够很快给智能体加上新的能力,不用自己重复造轮子,可以把精力都放在打磨智能体的行为逻辑上。

讲完了行动这一环,我们来看整个循环的最后一步,也是我觉得这篇文章里特别值得展开讲的部分,那就是验证。文章的观点很明确:能够检查并改进自己输出结果的智能体,从根本上来说更可靠,因为它们能在错误累积之前先抓出来,在偏离方向的时候能自我纠正,并且会随着迭代越做越好。这里的关键,在于要给 Claude 一些具体可行的方式去评估自己的工作成果,文章里讲了三种行之有效的方法。

第一种叫规则反馈,也就是基于明确定义的规则来评估输出,然后告诉模型哪条规则没满足、为什么没满足。文章特别提到代码 lint 检查就是规则反馈的一个典范,而且反馈的信息越详细越好。文章举了一个挺有意思的细节,说生成 TypeScript 然后做 lint 检查,通常比直接生成纯 JavaScript 效果更好,因为 TypeScript 能提供更多层次的反馈信息,比如类型检查这一层,JavaScript 是没有的。再拿邮件智能体打比方,如果智能体要生成一封邮件,你可能希望它检查一下收件人的邮箱地址格式是不是合法,如果不合法就报错;还可以检查一下,这个用户以前是不是给这个收件人发过邮件,如果是第一次发,就给出一个提醒性的警告。这种规则化的检查,把模糊的"写得好不好"问题,转化成了可以程序化判断的具体条件,模型就有了明确的改进方向。

第二种叫视觉反馈。如果智能体在做的是视觉相关的任务,比如生成界面、做 UI 测试,那么截图或者渲染图就特别有用。文章举的例子是,如果你要发一封带 HTML 格式的邮件,你可以把生成的邮件截图,再把这张截图喂回给模型,让它做视觉上的核对和反复打磨。具体可以从几个维度去检查:布局方面,元素位置摆放是否正确,间距是否合适;样式方面,颜色、字体、格式有没有按预期呈现;内容层级方面,信息呈现的顺序是不是合理,重点有没有突出;还有响应式方面,看看页面在不同情况下是不是显得错乱或者拥挤,当然单张截图能反映的视口信息是有限的。文章还提到,借助像 Playwright 这样的 MCP 服务器,你可以把这套视觉反馈循环自动化起来,自动截图、自动测试不同视口尺寸、甚至测试交互元素,而且这一切都可以嵌入在智能体的工作流程里完成,不需要人工介入。

第三种叫模型当裁判,也就是用另外一个语言模型,基于一些模糊的规则去给智能体的输出打分、做评判。文章对这个方法的态度比较审慎,明确指出这种方法通常不算特别稳健,而且会带来不小的延迟成本,但是对于那些任何性能提升都值回成本的应用场景,这种方法还是有它的用处的。拿邮件智能体打比方,你可以设计一个独立的子智能体,专门去评判草稿邮件的语气,看看它跟用户之前发过的消息风格是不是匹配协调。

讲完了整个循环,文章最后给了一段我觉得对实际工作特别有指导价值的内容,就是怎么判断你的智能体是不是真的"装备齐全"。文章建议,经过几轮智能体循环之后,你应该认真测试你的智能体,尤其要仔细看那些失败的案例,把自己放到智能体的位置上去想:它手头的工具,真的够用吗?文章给出了几个具体的自检问题,我觉得很值得每个做智能体开发的人记下来。第一个问题是,如果你的智能体理解错了任务,有没有可能是因为它缺少关键信息,这时候你可以考虑调整搜索类 API 的结构,让它更容易找到需要的东西。第二个问题是,如果智能体反复在同一个任务上失败,你能不能在工具调用里加一条正式的规则,专门用来识别和修复这类失败。第三个问题是,如果智能体自己修不好错误,你能不能给它更实用、更有创造性的工具,让它换一种思路去解决问题。第四个问题是,如果你不断给智能体加新功能,导致它的表现忽高忽低,那你应该基于真实客户使用情况,构建一套有代表性的测试集,做程序化的评估,也就是我们常说的 evals。

现在我们说说这篇文章对我们实际工作的意义,也就是普通开发者、普通工程师,今天能怎么把这些思路用起来。我觉得这篇文章最大的价值,不是介绍某个具体接口怎么调用,而是给了我们一套思考框架。如果你现在正在做一个智能体项目,不管是客服机器人,还是数据分析助手,还是内部知识库问答系统,你都可以拿这个三段式循环去对照检查一下自己的设计:你的智能体收集上下文的方式够不够好,它的行动手段是不是够灵活,它有没有办法验证自己做对了没有。很多团队在做智能体的时候,容易陷入一个误区,就是疯狂地堆砌提示词,试图靠一段超长的指令让模型表现得更聪明,但这篇文章告诉我们,真正决定智能体效果的,往往不是提示词写得多精巧,而是它有没有合适的工具、合适的文件结构、合适的验证机制。

如果你想现在就上手实践,我建议你可以先从一个具体的小场景切入,而不是一开始就想做一个全能型的通用智能体。比如说,如果你的团队有大量的客户工单需要处理,你可以先给智能体设计一个简单的文件夹结构,把历史工单按照客户或者类别归档好,让智能体用 agentic search 的方式去检索,而不是一上来就搭建复杂的向量数据库。等你发现检索速度确实跟不上,或者查询的语义变化太多样化了,再考虑叠加语义搜索这一层。工具的设计上,先想清楚这个智能体最核心、最高频要执行的两三个动作是什么,把它们做成专属工具,剩下那些长尾的、不可预测的需求,交给 Bash 通用终端去灵活应对,这样既保证了核心流程的稳定可控,又留出了应对意外情况的弹性空间。

在验证环节,我个人认为这是最容易被开发者忽略,但其实投入产出比最高的一环。很多人写智能体的时候,只关注怎么让它把任务做出来,却没有给它设计任何检查自己工作的手段。如果你的智能体输出是结构化的内容,比如代码、配置文件、json 数据,那么规则反馈几乎是零成本就能上的,你只需要把已有的 lint 工具、格式校验工具接进智能体的循环里,让它看到具体哪里错了、为什么错了,而不是简单地告诉它"重新试一次"。如果你的智能体输出涉及视觉效果,比如生成网页、生成海报、生成图表,那么花点时间接入截图和渲染反馈,通常能带来立竿见影的质量提升,这个投入我觉得是非常值得的。至于模型当裁判这种方式,我建议大家谨慎使用,先把前两种更稳健的反馈机制用足了,再考虑加上这一层,否则容易引入额外的延迟和不确定性,反而拖慢了整个智能体的响应速度。

另外要提醒大家注意的一个坑,是子智能体和上下文管理这块。如果你的智能体需要处理的信息量特别大,比如要翻阅成百上千封邮件,或者要扫描一个很大的代码库,千万不要试图把所有信息一股脑塞进主智能体的上下文窗口,这样做的结果往往是模型注意力被严重稀释,反而抓不住重点。更好的做法是参考文章里讲的子智能体模式,让多个子智能体并行去处理不同的子任务,只把提炼出来的关键信息汇报回主智能体。这种设计思路,其实跟我们做软件架构时讲的关注点分离是一脉相承的,只不过放到智能体语境下,变成了上下文窗口的分离和信息密度的提纯。

最后我想稍微展开聊聊 MCP 这块给普通开发者的启发。我个人觉得,MCP 协议的出现,某种程度上正在重塑我们对"集成开发"这件事的理解。过去如果你想让一个系统接入 Slack 或者 GitHub,你得自己处理认证、处理 API 版本差异、处理各种边界情况,现在如果已经有现成的 MCP 服务器,这些工作基本可以省掉。所以我建议,如果你正准备给智能体接入某个外部服务,先去看看有没有现成的 MCP 实现,而不是急着自己从零写一套集成代码,这样能省下大量重复劳动的时间,把精力集中在打磨智能体本身的判断力和行为逻辑上。

好,我们来简单总结一下今天这期节目。这篇文章的核心信息其实很清晰:Claude Agent SDK 的设计哲学,是给智能体一台电脑,让它像人一样去工作,而不是仅仅给它一段聊天式的提示词。整个智能体的运转,可以归纳成一个三段式循环,收集上下文、采取行动、验证工作,然后不断循环往复。在收集上下文这一环,文件系统检索、子智能体、自动压缩这三个机制各司其职;在采取行动这一环,工具、Bash、代码生成、MCP 协议提供了从精确到灵活的全谱系手段;在验证这一环,规则反馈、视觉反馈、模型裁判这三种方式,给了智能体自我纠错的能力。我个人认为,这篇文章最大的价值,不在于介绍了多少新功能,而在于把"智能体应该怎么思考、怎么设计"这件事,讲成了一套可以直接拿来套用的方法论,这对于今天还在摸索智能体落地路径的工程师来说,是相当有参考价值的一份指南。好,今天的节目就到这里,我们下期再聊,再见。
