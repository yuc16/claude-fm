Connector 是什么？它是基于一种叫做 Model Context Protocol 的协议构建的工具服务。Model Context Protocol，简称 MCP，是 Anthropic 大约在一年多前发布的一个开放标准协议。我知道"开放标准协议"这几个字听起来有点抽象，我们用一个具体的类比来帮助理解。

你想想我们日常生活中的 USB 标准。在 USB 出现之前，每种外设都有自己专属的接口，键盘用一种插头，鼠标用另一种，打印机又是另外一套，换一台电脑有时候所有外设都要重新配。USB 出现之后，这个混乱的局面被统一了：不管是什么设备、什么品牌，只要符合 USB 标准，插进去就能用。MCP 在 AI 世界里扮演的，就是类似 USB 的角色。

AI 大模型本身，默认只能访问训练数据和你输入的对话内容。但现实工作中，你需要 AI 处理的任务，往往涉及到大量的外部系统。你的代码仓库、你的数据库、公司的 CRM 系统、内部文档库、各种第三方 API。MCP 提供了一套标准的通信协议，让任何开发者都能把自己的服务包装成一个标准的 MCP 服务器，也就是一个 Connector，然后任何支持 MCP 的 AI 助手都能直接调用这个 Connector 里的工具。

举几个具体的例子会更直观。你可以做一个 GitHub Connector，让 Claude 帮你搜索 issue、查看 pull request、写代码评审意见，整个过程不需要你打开浏览器切换到 GitHub；你可以做一个连接公司内部知识库的 Connector，让 Claude 在回答员工问题时，自动检索企业文档，而不是依赖模型自己的训练数据；你可以做一个连接数据分析平台的 Connector，让 Claude 直接查询业务数据、生成报告。这些场景，有了 Connector 才能实现，而 MCP 保证了这些 Connector 可以标准化地接入 Claude。

今天，Claude 的官方目录里已经有超过三百个第三方 Connector，每天有数百万用户在使用。这个数字，是我读到这篇文章时最震惊的地方之一。要知道，一年多前这个生态几乎从零起步，现在已经有三百多个工具在跑，日活用户数以百万计，这个增速放在任何一个工具生态里都非常罕见。

但快速增长带来了一个严重的问题，而这个问题，恰好就是今天这篇文章要正面解决的：开发者对自己工具的运行状况，几乎是完全盲目的。

我们来想象一个典型的场景。假设你是一个开发者，花了两个月的时间，做了一个把公司 ERP 系统接入 Claude 的 Connector。你把它提交到目录，它上线了，用户可以发现并使用它。然后呢？你就像是把一艘船放进了大海，然后站在岸边，什么都看不见。

你不知道有多少人真正在用这艘船。你不知道船跑得顺不顺畅。你不知道哪个仓门在漏水，是查询功能报错了，还是写入功能超时了？你不知道在 Claude Code 里用的人多，还是在网页版里用的人多。偶尔会有用户给你发反馈说"我的工具用不了"，但你不知道是他一个人遇到问题，还是百分之八十的用户都在遇到同样的问题。

这种"盲飞"的状态，对于任何严肃的产品开发者来说，都是一场噩梦。而这，就是今天这篇文章要解决的核心问题：给 Connector 开发者提供"眼睛"。

好，背景建立好了，我们来正式进入文章的核心内容。

Anthropic 推出的这套可观测性功能，主要是一个面板，里面提供了三大块核心数据。我们逐一来看，每一块我都会花一些时间讲透，因为每一块背后都有一些值得细想的东西。

第一块，叫做用户采纳追踪。说白了，就是让你看到自己的 Connector 到底有多少人在用。具体来说，你能看到活跃用户数、工具调用总次数，以及你的 Connector 在目录里的排名，而且这些数据都有时间趋势，你能看到它们随时间变化的走势。

你可能会想，这不就是一个基本的统计功能嘛，有什么好说的？别小看它。对于开发者来说，工具调用总次数这个指标，特别能反映用户的真实依赖程度。一个用户可能"发现"了你的 Connector，点了连接，但从来没有真正用过里面的工具。活跃用户数告诉你有多少人"连接了"，而工具调用总次数告诉你有多少人"真正在用"。这两个数字的比值，能帮你评估你的 Connector 的真实价值交付能力。如果连接率高但调用率低，说明用户找到你了，但用了一次之后觉得没什么用，这是一个产品问题，需要去思考为什么。

目录排名这个指标也非常有意思。Claude 的目录是用户发现 Connector 的主要渠道，排名靠前的工具自然获得更多曝光。如果你能看到自己的排名随时间的变化，你就能做因果推断：什么时候你做了哪些改动，对采纳率产生了正向还是负向的影响？这是一个非常宝贵的产品优化反馈闭环，让你能从数据中学习，而不只是凭感觉做决策。

更重要的是，有了时间趋势数据，你可以做一些聪明的分析。比如，你发现某个月用户量有一个明显的跳升，回头看那段时间发生了什么，是 Claude 的某个新版本发布带来了新用户，还是你在某个渠道做了推广，还是你做了一个功能更新？把时间线对上，你就能知道什么事情真正带来了增长，从而决定把精力放在哪里。

第二块功能，是错误诊断和延迟分析。这一块，我个人认为是整个可观测性面板里最有直接价值的部分，也是最能帮你省事、省时间的地方。

文章里提到，面板会展示几个关键指标：健康评分、整体错误率，以及延迟数据。但最关键的一个细节，是这些数据可以精确到每一个工具的维度。意思是，如果你的 Connector 里有五个工具，面板会分别告诉你每个工具的错误率是多少、延迟是多少，帮你精准定位哪个工具是问题的来源。

为什么这个细粒度那么重要？让我给你举一个具体的例子，你就能感受到差别。假设你做的是一个 Jira Connector，里面包含这样几个工具：搜索 issue、创建 issue、更新 issue 状态、获取项目列表、添加评论。现在有用户反馈说"Jira Connector 有时候不好使"。

在没有工具级别监控的时候，你能怎么办？你只能去看整体的错误率，如果整体错误率只有百分之三，感觉还好，但具体哪里出问题了，你不知道。你可能需要把五个工具的代码全部翻一遍，或者在每个地方加很多日志，然后等用户再遇到问题，看看日志里有什么线索。这个排查过程可能要花好几天。

现在有了工具级别的错误拆分，情况就完全不同了。你打开面板，一眼看到："更新 issue 状态"这个工具的错误率是百分之二十七，其他四个工具全部正常。好，你马上就知道问题出在哪里了。你去看"更新 issue 状态"的实现代码，发现是因为 Jira 最近更新了 API，某个字段的格式发生了变化，你的代码还在用旧格式，所以在某些情况下会报错。你改几行代码，部署，问题解决，可能一个小时都不用。

这就是工具级别可观测性的威力。它把排查问题的时间，从"大海捞针"变成了"精准定位"。省下来的时间，可以拿去做更有价值的事情。

延迟的监控同样非常重要，而且对于 AI 场景来说，甚至比传统 API 更加关键。原因在于，用户在和 Claude 对话的时候，期待的是流畅的对话体验。如果 Claude 需要调用你的工具，而你的工具响应需要六秒、八秒，用户会明显感受到那个令人不安的停顿，体验会很差。更糟糕的是，在某些情况下，如果工具调用超时，整个 Claude 的响应可能会中断或者降级，这对用户体验的打击是非常大的。

有了延迟监控，你可以精确知道哪些工具是"慢工具"，然后去做针对性的优化。是你的数据库查询没有建索引？是你在做不必要的串行请求，其实可以改成并行？还是依赖的第三方 API 本身就慢，需要引入缓存层？这些问题，现在都有数据支撑了，你不需要靠猜。

第三块功能，是按产品线拆分的用量分析。文章里提到，你可以看到自己的 Connector 在 Claude 网页版、Claude Code，以及 Cowork 这几个不同的 Claude 产品上，分别有多少工具调用量。

你可能对 Cowork 这个名字不太熟悉。Cowork 是 Anthropic 最近推出的一个面向团队协作场景的产品，专注于多人协同和工作流自动化。它能和 Claude 以及 Claude Code 并列出现在这篇文章里，说明它已经是一个相当成熟的产品线了。

为什么要按产品线拆分？因为不同产品的用户，行为模式和需求是完全不同的。Claude Code 的主要用户是开发者，他们调用 Connector 的目的通常和代码工作流紧密相关，比如查代码库、跑测试、做代码评审、处理 pull request。他们对工具的需求，往往是高频的、精准的，而且需要快速响应，因为开发者在等待的时候很难保持专注状态。

Claude 网页版的用户群体就多元得多了。可能是产品经理在分析用户反馈，可能是市场人员在生成内容，可能是运营人员在处理业务数据，也可能是普通用户在做各种探索性的任务。他们对工具的使用方式，可能更偏向于一次性的、场景多变的，对延迟的容忍度也相对高一些。

Cowork 的用户，则主要是团队在协作完成某个比较长线的任务，可能有更长的对话链路，更复杂的多步骤工具调用序列，对工具的可靠性和一致性要求反而更高，因为一旦中间某个环节出错，整个协作流程可能都要重来。

如果你能看到你的 Connector 在不同产品线上的用量分布，你就能做出更有针对性的优化决策。比如，你发现你的 Connector 八成的调用都来自 Claude Code，而网页版几乎没有人用。这时候你应该做什么？你应该重点优化开发者工作流的体验，深度打磨那些对写代码最有帮助的工具，而不是花时间去设计一个适合非技术用户的友好界面。相反，如果你发现网页版的用量远大于 Claude Code，那你的 Connector 在普通用户群体里更有市场，你应该思考如何让工具对非技术用户更易用，降低使用门槛。

这种数据驱动的产品决策，在没有分产品线数据之前，只能靠猜。现在可以靠数据。

好，我们讲完了三块可观测性功能，现在来说第二个重要更新：应用内目录提交。

文章里提到，开发者现在可以直接在 Claude 的界面里，把自己的 MCP 服务器提交到官方目录。在这个功能推出之前，提交流程大概率是需要在 Claude 产品之外，通过某种表单或者邮件流程来完成的，整个体验是割裂的。

想象一下这种割裂感：你在 Claude Code 里开发，在本地跑 MCP 服务器做测试，一切都在 Claude 的生态里进行。但是要提交到目录，你需要跳出来，去另外一个地方填表，等审核，收到通知再回来。这种上下文切换，虽然看起来是个小问题，但对开发者效率的打击是累积性的。每次中断，都需要重新把思路切换回来。

现在这件事可以在 Claude 里直接完成。开发在哪里，提交就在哪里，整个从开发到上线的流程变得紧密闭合了。这是一个看似不大的改进，但会让整个开发体验顺畅很多。对于那些在迭代 Connector 的开发者来说，每省下一次不必要的上下文切换，都是实实在在的效率提升。

好，功能介绍完了，我们来聊权限和访问方式，这个比较实际，你需要记一下。

可观测性面板目前处于公开测试阶段，也就是 public beta，所有符合条件的用户现在就可以访问。要找到它，你需要在 Claude 里进入组织设置，也就是 Organization settings，然后找到目录，也就是 Directory 这个选项。在那里就能看到你已发布的 Connector 的数据面板。

权限要求上，你至少需要是 Team 计划或者 Enterprise 计划的管理员或者所有者角色，也就是 Admin 或者 Owner，才能访问这个面板。个人计划是看不到这些的，这个定位是合理的，因为 Connector 开发主要发生在团队和企业场景里。

对于企业计划的用户，有一个额外的灵活性设计：所有者可以创建一个自定义角色，给这个角色赋予目录管理权限，然后把这个角色分配给特定的成员。这样，做 Connector 开发的工程师就可以访问可观测性面板，即使他们本身不是管理员或者所有者。

这个权限委托机制在大型企业里很有价值。一个几百人、几千人的公司，做 Connector 开发的可能是某个技术团队的几个工程师，他们不应该也不需要拥有整个组织的管理员权限。但他们需要能看到自己工具的运行数据，以便做优化决策。通过精细化的角色委托，可以在不扩大权限范围的前提下，把这个能力准确地给到需要的人。我个人认为这是一个很细心的设计，说明 Anthropic 对企业场景的权限管理有比较深的理解。

好，接下来是我认为今天这期节目最有落地价值的部分：对于你作为一个软件工程师，这些功能意味着什么，你应该怎么做，要避开什么坑？

我们分两类人来讲。第一类是已经有 Connector 在目录里的开发者，第二类是正在考虑进入 MCP 生态的工程师。

先说第一类，已经发布了 Connector 的开发者。

你现在应该马上做的第一件事，是登录 Claude，进到组织设置的目录页面，打开你的可观测性面板，把数据看一遍。即使你现在的用户量不大，甚至还在非常早期的阶段，建立这个基线数据也非常重要。因为后续所有的优化，都需要一个参照点。没有基线，你做了改动之后，不知道有没有效果，也不知道效果有多大。

你要重点关注的，首先是错误率，而且要看工具级别的拆分。如果你发现某个工具的错误率超过百分之五，应该立刻去排查。很多时候，开发者上线之后抱着"没有收到投诉就是没问题"的心态，但这是一个非常危险的假设。在 AI 工具使用场景里，用户遇到工具调用失败，大概率的反应是"Claude 不好使"，而不是"这个 Connector 有 bug"。他们可能悄悄换用别的工具，或者对 Claude 的整体评价打折扣，而不会主动给你发反馈。可观测性面板，是你不依赖用户主动反馈就能发现问题的核心手段。你要主动去看数据，而不是等着用户来告诉你哪里坏了。

其次，你要认真留意延迟指标。在 AI 对话场景里，一般来说，你的工具最好能在两秒内返回结果，超过五秒就会明显影响对话的流畅感，超过十秒在很多情况下用户已经开始感到烦躁了。如果你发现有工具延迟偏高，先做一轮快速的优化排查：数据库查询有没有缺失索引？有没有可以并行的请求在串行执行？有没有可以缓存的数据在每次都重新查询？这些通常是最容易找到也最容易解决的性能问题。

第三，看产品线分布，思考你的 Connector 主要在哪个产品场景下被使用，然后反问自己：这和你最初设计这个工具时的预期是一致的吗？如果你发现用户在以你完全没有预想到的方式使用你的工具，这可能不是问题，而是一个发现新需求、迭代产品方向的机会。

现在说第二类人，正在考虑进入 MCP 生态的工程师。

这篇文章对你最重要的信号，是 Anthropic 在认真投入这个生态的基础设施建设。一个平台，只有在认为某个生态足够重要的时候，才会花力气给开发者提供监控工具、优化提交流程。这种投入本身，就是一种承诺：这条路是被认真对待的。对于开发者来说，这是一个积极的信号：你投入在这里的工作，不太可能因为平台放弃这个方向而打水漂。

如果你在考虑做一个 Connector，我根据今天文章里直接或者间接透露的信息，给你五条建议。

第一条建议，工具的粒度要合理。不要把所有功能都塞进一个大工具里。可观测性面板能做到按工具粒度拆分数据，前提是你的工具设计本身足够精细。如果你把查询、修改、删除全部做成一个叫做"数据操作"的工具，你的监控数据就会非常模糊，很难从中得出有用的结论，排查问题也会困难得多。合理的做法是把不同语义的操作拆分成独立的工具，每个工具做好一件事。

第二条建议，错误处理要规范。MCP 协议有标准的错误返回格式，你要确保你的工具在出错时，返回的是协议能正确解析的错误码和错误信息，而不是直接把异常堆栈抛出去。规范的错误返回，一方面让用户得到更友好的提示，另一方面也让你的监控数据更有意义，因为系统能够正确分类和统计你的错误类型。如果错误信息是一堆没有结构的异常堆栈，监控系统也没办法帮你做有意义的分析。

第三条建议，认真写工具描述。Claude 在决定什么时候调用哪个工具时，很大程度上依赖于你提供的工具描述。描述写得清晰、准确、语义丰富，Claude 就能在合适的时机调用你的工具；描述写得模糊，Claude 可能会在不该调用的时候调用，或者在该调用的时候没想到要调用。这会直接影响你的工具调用次数，进而影响用户体验和你在目录里的排名。工具描述不是代码，但同样值得你认真对待，花时间打磨。

第四条建议，权限设计最小化。你的 Connector 向用户申请的权限，只应该包含完成任务所必需的最小范围。这不只是安全要求，也是产品策略。权限范围越小的 Connector，企业用户的接受门槛越低，因为安全审查会更容易通过。如果你的工具只需要读取某个特定的数据表，就不要申请整个数据库的读写权限。这个原则在面向企业用户的 Connector 设计里尤其重要。

第五条建议，在设计阶段就考虑不同产品线的差异。今天的文章提醒了我们，你的 Connector 会在 Claude 网页版、Claude Code、Cowork 等不同的产品里被使用，不同产品的用户群体和使用场景差异很大。如果可能的话，在做工具设计的时候，先想清楚你的核心目标用户是哪个产品里的用户，然后针对那个场景做深度优化。当然，一个好的 Connector 能在多个场景里都好用是最理想的，但如果资源有限，先把一个场景做深，再扩展到其他场景，往往是更聪明的策略。

好，最后我想说一点更宏观一点的观察，这是我个人的判断，跟文章原文无关，供你参考。

可观测性功能的推出，代表着 Claude 的 Connector 生态在经历一次思维层面的升级。过去，做 AI 集成往往是一种"部署即完成"的思路——你把东西做好，接入进去，用户能用就行了，后续就不管了。但随着生态的成熟，这种思路已经行不通了。

你看传统的 SaaS 产品，背后都有一套完整的运营体系：数据分析、A/B 测试、用户行为跟踪、错误监控、性能分析。这些不是锦上添花的东西，而是产品持续迭代的基础设施，是你能够持续改进产品的前提。现在 AI 工具生态也开始走上这条路，Anthropic 给 Connector 开发者提供可观测性工具，本质上是在推动开发者把 Connector 当成一个严肃的产品来运营，而不只是一段"能跑就行"的代码。

我个人认为，这个方向是完全正确的，也是必然的。随着越来越多的企业把 Claude Connector 集成进自己的核心工作流，开发者需要对这些集成的稳定性、可靠性和质量负责。可观测性工具，是这种责任感的技术基础。没有可见性，就谈不上可控性，更谈不上持续优化。而没有持续优化，在一个竞争日趋激烈的工具目录里，你的 Connector 很快就会被淡忘。
