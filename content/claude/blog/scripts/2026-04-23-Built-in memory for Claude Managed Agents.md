在说这个新功能之前，我们先来建立一下背景，聊聊 AI Agent 在记忆这件事上长期以来面临的是什么困境，因为理解了问题，才能真正理解这个设计好在哪里。

任何跟 AI Agent 打过交道的工程师，大概都有过这样的体验。你跟 Agent 开了一个会话，花了很长时间跟它对话，帮它理解你的需求，纠正它的错误，告诉它你们团队的特定约定，告诉它这个项目里有一些特殊的逻辑不能用通用方式处理。它终于开始工作得顺手了，你心想：好，现在它懂我了。然后会话结束，你明天再开一个新会话，一切都回到了原点，它什么都不记得了。

这个问题在业内有个口语化的描述，叫「金鱼记忆」问题。金鱼的记忆据说只有七秒，当然这是以讹传讹的说法，但用来形容无状态 AI Agent 倒是挺贴切。每次对话都是全新的开始，没有历史，没有积累，没有成长。

这个局限对于一次性的任务来说还好。你让 Agent 帮你翻译一段文字、帮你整理一个表格、帮你写一封邮件，做完这件事就结束了，不需要记忆。但如果你要用 Agent 做的是持续性的工作——比如定期检查某个系统的状态、持续优化某个数据流水线、长期跟进某个项目的开发进度、反复审查同一个团队提交的代码——那金鱼记忆就是一个根本性的障碍。你没办法让它积累经验，没办法让它越用越好，每次都得重新建立上下文。

过去开发者是怎么绕过这个问题的？大体上有两条路。

第一条路是手动维护 system prompt。你把重要的上下文、历史经验、团队约定，全部写进 system prompt 里，每次启动 Agent 的时候都塞进去。这个方法可行，但有两个明显的代价。第一个代价是需要人工判断和整理：你得自己决定哪些内容值得放进 system prompt，这需要人主动去做，不可能完全自动化。第二个代价是扩展性问题：随着时间推移，需要记录的内容越来越多，而 context window 的空间是有限的，system prompt 太长之后模型的表现会打折扣，而且 token 成本也会线性增长。

第二条路是自己搭检索系统。把历史对话存下来，用向量数据库做语义检索，每次启动 Agent 的时候先检索一遍相关历史片段，把最相关的内容作为上下文喂给它。这个方案技术上更完整，理论上可以解决扩展性问题，但工程成本很高。你得选向量数据库，要维护 embedding 流水线，要写检索逻辑，要处理各种边界情况，要做效果评测，要调优检索策略。对于很多团队来说，这是一个搭起来需要几周时间、维护起来持续要人力的基础设施投入。而且坦白说，如果你的核心业务不是 AI 基础设施，这个投入很难说是值得的。

这两条路要么重，要么贵。大量的工程团队就一直在将就，能不用 Agent 记忆就不用，或者把它当成一个「有就好、没有也行」的边缘功能。

Anthropic 这次推出的功能，就是在说：你们不用自己搭了，平台帮你搞定，而且我们把它做成了 production-ready 的，企业直接用。

那我们来看 Managed Agents 是什么。简单来说，Managed Agents 是 Claude 平台上的一个托管 Agent 运行环境。你定义 Agent 的行为逻辑，平台处理生产环境的基础设施——并发管理、容错、工具调用的接入、日志记录等等。在这次发布之前，Managed Agents 已经支持让 Agent 执行代码、操作文件系统、调用外部工具，做很多复杂的自动化任务了。现在，加上了跨会话的记忆能力。

好，背景铺完了，我们来看这个记忆功能本身，从最核心的设计决策开始说。

最值得深聊的第一个点，是记忆存储在哪里，以及为什么这样选。文章里给出的答案是：记忆挂载在文件系统上，以文件的形式存储。

这个选择乍一看不起眼，但仔细想想非常有意思。我们来做个对比。在 AI 记忆系统这个领域，现在最流行的方案是向量数据库。原理大家应该都听过：你把历史对话、笔记、知识片段用嵌入模型转换成高维向量，存入数据库；需要检索的时候，把当前的查询同样转换成向量，找距离最近的几条历史记录拿出来。Pinecone、Weaviate、Chroma、Qdrant，市面上这类工具一抓一大把，LangChain 和 LlamaIndex 里都有现成的集成。

那 Anthropic 为什么没有走这条看起来很成熟的路？

文章里有一句话给了答案：「记忆是建立在 Agent 已经在用的工具之上的。」（英文原文是 Memory builds on the tools they already use。）

换句话说，设计核心思路是：与其给 Agent 引入一套全新的存取机制，不如充分利用它已经熟悉的能力。Claude 在做 agentic 任务的时候，本来就要使用 bash 命令、执行代码、读写文件。文件系统对它来说不是陌生的存储介质，而是日常工作的自然延伸。把记忆挂载到文件系统上，意味着 Agent 保存一段记忆，就像它平时执行一个 bash 脚本、往某个路径写一个文本文件一样自然；读取记忆，就像读一个配置文件或者日志文件。不需要学新的工具，不需要适配新的接口，认知负担最小。

你可以类比一下人类是怎么记忆的。你作为一个工程师，平时记录东西用什么？你可能用 Notion，可能用某个 markdown 文件，可能用代码注释，可能用 git commit message，用的都是你每天工作流里本来就在用的工具。而不是说，每当你想记一件事，你得打开一个专门的「记忆数据库客户端」，把这件事编码成向量，提交到数据库里，再检索的时候用语义搜索来查。这套流程存在，但不自然，不流畅。

文件系统给了记忆结构化的空间，这也是一个很重要的优势。你可以用不同的文件来组织不同类型的知识：一个 json 文件存用户偏好，一个 markdown 文件记录项目里的架构约定，一个 txt 文件记录过去发现的常见错误模式，一个目录专门放每周积累的业务知识。Agent 可以自己设计这套组织结构，就像一个有条理的程序员用不同目录组织自己的代码库一样，灵活且可读。这种灵活性是固定 schema 的关系型数据库不具备的，也比单纯的 key-value 存储更有表达力。

文章里还提到了一个重要的实测发现，我认为这是整篇文章里最值得划重点的句子之一：他们的最新模型在使用文件系统记忆的时候，存储的内容更全面、更有条理，而且更能辨别「这件事到底值不值得记下来」。

这个发现非常关键，因为记忆系统的价值从来不只是「能存」，更重要的是「懂得筛选」。一个什么都记的 Agent，记忆很快就会变成一堆噪音。你记住了用户随口说的一句话，记住了某个偶发的特殊情况，记住了一个一次性的临时决定，然后这些东西在下一次对话里干扰了 Agent 的判断——有时候这比没有记忆还糟糕，因为错误的记忆比空白的状态更难发现、更难修正。

真正有价值的记忆系统，必须具备取舍的能力。文件系统这个载体，加上最新 Claude 模型的判断力，让这件事变成了可能。Agent 的记忆行为更像一个聪明的人在记笔记，而不是一台录音机在录所有声音。

当然，任何设计选择都有其适用范围，文件系统也不例外。向量数据库擅长的大规模语义检索，在文件系统里天然不具备。如果你的场景是要从几十万条历史记录里、在毫秒级别内找出语义最相似的几条，文件系统不是最优解。所以这个设计有其明确的适用范围：它服务的是长期运行、积累式学习、注重可解释性的 Agent，而不是需要大规模语义检索的知识库问答系统。对于大多数企业里的 agentic 工作流来说，前者才是主要需求。

好，现在我们来看第二个核心点：企业级功能设计。

如果只是给单个 Agent 加上记忆，那其实很多开发者自己就能搭，并不新鲜。让这个功能真正具有生产价值的，是它围绕企业场景做的一系列设计。

先来说权限分级。文章给了一个具体的例子：一个公司可以建立一个 org-wide 的记忆存储，这个存储对所有 Agent 是只读的；同时每个用户可以有自己的记忆存储，这个存储支持读写操作。多个 Agent 可以同时使用具有不同访问权限的存储。

这个两层权限架构非常实用，我们来理解一下它的实际含义。org-wide 的只读层适合放什么？公司的业务知识、行业术语定义、合规要求、内部规范、通用最佳实践、跨团队共享的经验积累——这些内容相对稳定，所有 Agent 都可能需要参考，但不应该被某个 Agent 随意修改。你不会希望一个处理客服工单的 Agent，因为用户说了一句奇怪的话，就把公司的产品描述给改掉。

用户级的可写层适合放什么？用户特定的工作偏好、历史对话中建立的特定约定、针对某个具体任务积累的专项知识、这个用户团队特有的项目记录——这些内容是私有的、动态的，适合由 Agent 在工作中自主更新。

两层叠加在一起，Agent 工作的时候同时参考全局知识和个人知识，就像一个员工既遵守公司规范又有自己的工作方法，边界清晰，权限不混淆。

你可能会想到，这个结构和 Git 仓库的权限管理有点像——有全局的公共仓库，有用户自己的 fork，有明确的谁能写什么的规则。原理类似，只是应用场景换成了 AI Agent 的记忆管理。

其次是并发访问控制。文章提到，多个 Agent 可以同时访问同一个记忆存储而不会相互覆盖。在企业里部署 Agent 的时候，这几乎是一个必须具备的基础能力。你不可能让所有 Agent 排队串行地使用记忆，那样效率太低，完全抵消了 Agent 并发的优势。但如果多个 Agent 并发写入同一个存储而没有协调机制，就会出现经典的竞态条件——两个 Agent 同时读到一个旧值，各自计算之后写入自己的更新，其中一个的更新就覆盖了另一个，数据出现了不一致。Anthropic 在底层处理了这个并发协调问题，作为开发者你不需要自己写分布式锁或者事务逻辑。

然后是审计日志和回滚能力，这个在企业场景里简直是必需品。文章说，所有的记忆变更都有详细的日志，记录了是哪个 Agent、哪次会话、记录了什么内容、什么时候记录的。你可以把记忆回滚到某个早期版本，也可以从历史里精确删除特定的内容条目。

这个能力的重要性怎么强调都不为过。在真实的生产环境里，Agent 不可能永远只学到正确的东西。它可能在某次对话里被用户误导，学到了错误的信息；可能因为输入数据质量问题，形成了错误的经验；也可能因为业务规则发生了变化，过去正确的记忆现在已经不再适用，变成了负担。

如果没有审计日志，你对这些问题几乎是完全盲目的。你知道 Agent 的行为变差了，但不知道是什么时候开始的，不知道是因为记住了哪些错误内容。有了审计日志，整个学习历史都是可查的，排查起来就像调试代码一样有迹可循：找到那次出问题的会话，定位到具体的记忆条目，然后修正或删除它。

还有一个值得注意的细节：记忆的更新会在 Claude Console 里以会话事件的形式实时显示出来。这意味着开发者不需要专门去翻日志文件才能知道 Agent 在学什么，在管理后台里可以直接追踪 Agent 的学习行为，整个过程是可见的、可理解的。在 AI 系统越来越复杂的今天，这种可观测性对于建立对 Agent 的信任非常重要。你能看到它在学什么，你才敢把更重要的工作交给它。

好，我们来看文章里提到的四个早期用户案例，这是整篇文章最有说服力的部分。四个案例代表了不同的应用模式，对你思考自己的场景很有参考价值。

Netflix 的用法是我认为最通用的一类。他们的 Agent 现在可以跨会话携带上下文，包括那些需要多个回合的对话才能挖掘出来的洞察，以及人类在对话中途做的实时修正。

我们来理解一下「需要多个回合才能挖掘出来的洞察」是什么意思。有些重要的发现，不是在第一轮对话里就能出现的。Agent 问了一个问题，用户给了一个回答，Agent 根据这个回答调整了策略，又问了一个问题，用户补充了更多细节，然后才浮现出一个真正有价值的结论。这个结论如果只记录「最终答案」，可能丢失了很多推导过程；但如果要把整个多轮对话都塞进下一次会话的上下文，上下文会变得很长很贵。记忆系统可以让 Agent 自己提炼出这个洞察的核心，只保存最有价值的部分。

以前 Netflix 是怎么处理这件事的？文章直接说了：「手动更新 prompt 和 skill」。这句话背后是多少工程师的心酸。Agent 在一次会话里学到了有价值的东西，有人得看到这件事，把它整理成 system prompt 的语言，手动加进配置里，下次重新部署，才能让 Agent 记住这件事。整个流程依赖人工介入，慢，而且容易遗漏。现在这个过程自动化了。

Rakuten 的案例是文章里我觉得数字最震撼的一个。他们的任务型长期运行 Agent，通过从每次会话里学习并避免重复过去的错误，实现了一次错误率降低百分之九十七，同时整个过程在工作空间范围内、有可观测的边界。

让我们好好理解这个数字。假设 Agent 在执行某类任务时，一百次里原来会有三次犯某种特定错误，降低了百分之九十七之后就只剩下不到一次了。如果这个数字在真实业务场景里是可以验证的，那意义是非常巨大的——因为很多 Agent 的工作是要接入真实的系统、执行真实的操作，一次错误的代价可能很高，比如说误触了某个生产数据库的写操作、发出了一封内容有误的自动邮件、在审批流程里做了错误的判断。错误率的大幅下降，直接对应的是业务风险的大幅降低。

但 Rakuten 在描述这个成果时还特别强调了「workspace-scoped, observable boundaries」，也就是工作空间范围内的、可观测的边界。他们在意的不只是错误率数字，还在意这件事是在可控的范围内发生的，学习过程是透明的、可追溯的。这个强调很能说明真实业务部署时的心态——降低错误率固然重要，但如果 Agent 的学习过程是一个不透明的黑盒，那在生产环境里你是不敢真正信任它的，你的老板也不会放心，你的合规团队更不会放心。可控和可观测，有时候比性能数字更重要。

Wisedocs 的案例聚焦在文档验证流水线上。他们用跨会话的记忆来识别并记住文档里反复出现的问题类型，把验证速度提高了百分之三十。

这个案例的有趣之处在于：记忆在这里扮演的角色不是「记住用户说了什么」，而是「积累领域经验」。一个做了几百份文档验证的专业人员，和一个刚入职的新人，在遇到同一类文档格式错误的时候，反应速度是不一样的。有经验的人一眼就能看出来，因为他见过太多次类似的情况了。而 Agent 每次会话都从零开始，就像永远是第一天上班的新人。Wisedocs 的做法，是让 Agent 把每次验证中发现的问题模式记录下来，下次遇到类似情况，直接命中记忆，不需要重新推理。这就是从「新人」到「老手」的跨越。

最后是 Ando，他们在做企业内部的即时通讯平台，让 Agent 学习每个组织特定的沟通方式，而不是自己去搭一套记忆基础设施。

这个案例代表的是一类非常典型的需求：「我需要 AI 适应我的工作方式，而不是我去适应 AI 的默认行为。」以前要满足这个需求，你得自己设计存储方案、自己写检索逻辑、自己做各种集成，这是一个相当重的工程投入。Ando 选择把这个复杂度交给平台来处理，专注于上层的产品体验本身。这是托管服务最经典的价值主张：把基础设施的复杂度下沉，让应用开发者专注于他们真正在乎的事情。

好，说完了技术原理和案例，我们来聊最实际的部分：对你来说，这意味着什么，你现在可以怎么用，要避开哪些坑。

首先说什么样的场景最适合接入这个功能。我梳理了三类最典型的场景。

第一类，是定期执行、需要积累经验的自动化任务。比如每天运行的代码质量检查 Agent、每周跑的竞品动态监控 Agent、每次有新数据进来就跑一遍的数据清洗 Agent。这类任务的特点是：同一个 Agent 会在相似的工作环境里反复执行，每次执行都可能遇到一些新的边界情况，或者发现一些可以改进的规律。有了记忆，这些发现可以积累下来，Agent 的表现会随着执行次数增加而逐步提升，就像一个员工在岗位上越来越熟练一样。

第二类，是需要适配特定团队或者特定用户习惯的助手型 Agent。比如一个代码审查 Agent，需要记住这个团队偏好的编码风格、他们项目里特有的架构约定、以前审查中达成的具体共识。这类知识如果全部靠 system prompt 来维护，工作量大、难以更新、容易失控；放进动态记忆里，让 Agent 自己在工作中维护，更自然，也更可持续。

第三类，是做复杂、多步骤、长周期任务的 Agent。比如负责跟进一个持续数周的项目迁移、需要反复与外部系统交互并根据结果不断调整策略的 Agent。这类 Agent 的每次执行都是上一次的延续，没有跨会话的记忆，就像每次换了一个不了解项目背景的新人来接手，效率极低。

相对来说，对于完全一次性、彼此独立的任务，记忆的价值就小很多。你让 Agent 帮你把某份合同翻译一下，做完这件事就结束了，下次做的是完全不同的文档，这时候积累经验的价值有限。

那在使用中要注意哪些坑？我总结了四个。

第一个，记忆质量需要主动管理，不要完全放手。文章说模型会自动判断什么值得记，但这个判断在公测阶段不可能完美。特别是在项目早期的几周里，你最好定期去 Console 里看看 Agent 都在记什么，有没有记下了不准确的内容，有没有把一个临时的特殊处理方式当成了通用规则记进去。审计日志是你的好朋友，把它当成调试工具来用，而不是只在出了问题之后才去翻。

第二个，权限架构要在一开始就设计清楚，而不是用着用着再改。特别是在多 Agent、多用户共享同一个系统的场景里，哪些内容应该放在 org-wide 的只读层，哪些应该放在用户级的可写层，这个边界要在部署之前就定义好。因为一旦 Agent 开始积累记忆，后期改变权限架构的成本不低，你还得处理已有记忆的迁移和清理问题。

第三个，记忆和 system prompt 的边界要想清楚，不要把所有「背景知识」都迁移进记忆。稳定的、核心的、不应该被 Agent 自行修改的规则，还是应该放在 system prompt 里——因为放进记忆里意味着 Agent 理论上有可能通过后续会话来覆盖这些内容，如果某次对话里出现了和这条规则冲突的信息，Agent 可能会更新记忆，悄悄绕过了你的初始设定。而那些会随着 Agent 经验积累而应该演化的知识，才是记忆的真正用武之地。

第四个，如果你的业务有合规或者数据治理方面的要求，这个功能实际上是一个值得主动利用的加分项。记忆的可审计性、精确的内容删除能力、完整的版本历史、可回滚的机制——这些都是在受监管行业里做 AI 部署时，监管方和合规团队经常会问到的问题。现在这些能力是开箱即用的，你在和法务、合规、信息安全团队沟通 Agent 部署方案的时候，可以用具体的功能来回答他们的问题，而不是说「这块我们以后再处理」。

我个人还想补充一个文章里没有直接提到、但我认为很值得关注的方向：Agent 之间的间接知识共享。文章里说多个 Agent 可以共享同一个记忆存储，存储可以有不同的访问权限。这意味着在设计上，不同的 Agent 之间可以形成一种间接的协作学习。Agent A 在执行某类任务时发现了一个有价值的模式，把它写进共享存储；Agent B 在执行相关任务的时候读取了这个存储，也从中受益，避免了同样的错误。这有点像一个团队的知识库，只不过是 AI Agent 版的。这个能力如果在大规模部署的场景里用好了，可以让整个 Agent 系统的知识网络有很有趣的涌现效果。当然这也意味着需要仔细设计谁能写、写什么，一个 Agent 的错误经验如果污染了共享存储，可能影响一批 Agent 的行为，这是需要用审计和回滚来防范的风险。

好，我们来做个总结收尾。

这篇发表于二〇二六年四月的文章，宣布的功能在本质上是在做一件事：让 AI Agent 从「会忘事的工具」变成「会学习的员工」。这个变化看起来简单，但对于真正要用 Agent 做复杂、持续性工作的团队来说，影响是深远的。Rakuten 百分之九十七的一次错误率下降，Wisedocs 百分之三十的速度提升，这些早期数字已经足够说明这个方向的价值。

文件系统作为记忆存储介质，我个人认为是一个「对的选择未必是最酷的选择」的好例子。Anthropic 没有追求技术上的华丽，没有去堆砌向量检索和 RAG 管道，而是选择了和现有 Agent 能力最自然融合的方案，让最新的模型本身来负责判断什么值得记、怎么组织。企业级的权限分层、并发控制、审计日志、可回滚，则是把这个功能从「有趣的技术实验」推进到「可以用在真实生产环境」的关键设计。

我个人对这个方向是持乐观态度的。AI Agent 想要在企业里真正发挥价值，就需要积累，需要成长，需要比昨天的自己更好。内置记忆是这个方向上一块重要的拼图。现在是公测阶段，如果你的工作里有 Agent 相关的场景，这是一个值得提前上手、提前积累经验的功能。早期上手的工程师，往往也是最先踩完坑、最先积累到实战经验的那一批人。
