## Shownotes

本期要点:
- 顾问策略反转了常见的大模型做指挥官、小模型做苦力的子智能体套路,改成小模型自己扛全程,只在卡住的关键节点临时连线 Opus 要个主意

## Script

大家好,欢迎收听 Claude FM。今天我们要聊的这篇文章,标题是 The advisor strategy, give Sonnet an intelligence boost with Opus,翻译过来就是顾问策略,用 Opus 给 Sonnet 来一次智能升级。这篇文章发表于二零二六年四月九日,出自 Anthropic 官方博客。到我们录这期节目的今天,二零二六年六月二十日,已经过去了两个多月,不算太久,里面讲的功能和数据基本还是最新的参考。只是因为这个功能发布的时候还是公开测试版,如果你听到这期节目的时候时间又往后挪了不少,建议去官方文档确认一下最新的可用性和计费规则有没有变化。

这篇文章值得听的原因很简单。如果你平常在写智能体,也就是那种能够自主调用工具、连续行动去完成任务的 AI 程序,你大概都纠结过一个问题,就是该用哪个模型。便宜的模型省钱,但是关键时刻不够聪明,容易卡壳或者瞎搞;聪明的模型确实靠谱,但是大规模跑起来账单很吓人。Anthropic 这篇文章给的答案,不是让你二选一,而是教你怎么把两者拼起来用,而且现在直接做成了接口里一个现成的工具,理论上代码改一行就能用上。

我们先说说背景。现在做智能体产品的人,手里基本有三档模型可以选。Opus 是最聪明的旗舰档,推理能力强,但调用成本也最高;Sonnet 是中间档,聪明程度和成本比较均衡,是目前很多生产环境里的主力;Haiku 是最轻量最便宜的档位,速度快,但是遇到复杂判断容易露怯。如果你的智能体需要跑成千上万次任务,比如批量处理客服工单,或者批量做网页调研,每次调用都用 Opus,账单很可能让老板找你谈话;但是全用 Haiku 或者 Sonnet,又可能在一些关键决策上判断失误,导致整个任务跑偏,返工的代价反而更高。

过去业界比较流行的一种解法,是所谓的子智能体编排模式。具体做法是用一个比较聪明、比较贵的大模型当指挥官,先把一个复杂任务拆解成一个个小任务,分配给一群便宜的小模型并行执行,跑完了再汇总结果。这个思路听起来合理,实际做起来很麻烦。你得自己写任务拆解的逻辑,管理一个工人模型池,处理子任务之间的依赖关系和结果汇总,工程复杂度一下子就上去了。而且大模型在最开始拆任务的时候,如果拆得不准,后面全盘都会跟着跑偏,出了问题还很难定位到底是哪一层出错。

Anthropic 在文章里说,他们观察到不少开发者其实已经自发地收敛到了另一种做法,他们管这个叫顾问策略。思路反过来了,不是大模型指挥小模型,而是让便宜的小模型自己从头跑到尾,正常调用工具、读取结果、一步步推进任务,只有当它遇到一个自己拿不准、解决不了的决策点的时候,才临时去问一下更聪明的 Opus,听完建议接着自己干。这个模式不需要事先拆解任务,不需要维护一堆工人模型,也不需要写复杂的编排逻辑,智能体架构基本不用变,只是多了一个可以随时求助的电话。值得一提的是,这不是 Anthropic 凭空设计出来的方案,而是先有了一批开发者在实践里摸索出这个用法,官方看到这个趋势之后,才把它正式产品化、做成了一个标准工具,这种顺着社区已验证的用法去做产品,往往比凭空设计出来的功能更靠谱。

这个机制具体怎么运作,我们拆开说。顾问策略里有两个角色,一个叫执行者,一个叫顾问。执行者通常是 Sonnet 或者 Haiku,它是真正干活的那个,负责调用各种工具,比如搜索网页、读取文件、跑代码,然后根据结果一步步往前推进任务,从头到尾都掌握着主导权。顾问通常是 Opus,它不直接接触用户,也不会自己去调用任何工具,它唯一的工作就是在执行者卡住的时候,看一眼执行者已经收集到的上下文信息,然后给出一个建议。这个建议可能是一份接下来该怎么做的计划,可能是一次纠错,告诉执行者刚才的判断错在哪里,也可能干脆是一个停止信号,告诉执行者这条路走不通,别再试了。给完建议之后,顾问就退场,执行者拿着这个建议继续自己跑完剩下的流程。这里有个细节挺重要,顾问看到的不是一个全新的、从零开始描述的问题,而是直接共享执行者已经攒下的对话记录和工具调用结果,相当于顾问翻一眼执行者的笔记和素材,就能给出针对性的建议,不需要执行者重新把前因后果讲一遍,这样省下来的不只是钱,还有时间。

这个设计你可以类比成什么呢。我觉得有点像呼叫中心里的场景。大部分时候接电话的是普通客服,他们处理绝大多数常规问题完全没问题,语气好、效率高、成本也低。但是偶尔遇到一个特别棘手的客户投诉,普通客服判断不了该怎么处理,这时候他会简单把情况描述给后台的资深专家,资深专家听完给一句建议,比如该走哪个流程、该不该退款、该怎么安抚,然后挂电话,自己不会跳出来直接跟客户对话。普通客服听完建议,回去接着跟客户聊,把事情解决掉。整个通话从头到尾,客户感知到的始终是同一个客服,资深专家只是在后台打了个简短的电话。这跟顾问策略里 Opus 不产生面向用户的输出、不调用工具、只给执行者出主意,是同一个道理。这种分工带来一个很自然的好处,就是只有在真正需要高阶推理的那一瞬间,你才付出 Opus 级别的成本,剩下绝大多数的常规操作,都还是按 Sonnet 或者 Haiku 的价格在走,这跟把整个任务都交给 Opus 跑一遍,价格上完全不是一个量级。

如果拿这个跟前面说的子智能体编排模式对比,你会发现顾问策略是把权力关系倒过来了。编排模式里,大模型是项目经理,负责拆任务、分配任务、最后汇总,小模型只是按指令干活的工人,自己没有决策权,一旦项目经理一开始拆错了,底下一群工人模型跑出来的东西也救不回来。顾问策略里,主导权一直在执行者手上,执行者才是项目的负责人,Opus 顾问更像是被请进会议室临时答疑的外部专家,问完问题就走,不参与项目管理,也不替你拍板每一步该怎么走。文章特别强调,这种模式不需要任务拆解,不需要维护工人模型池,也不需要额外的编排逻辑,原来的智能体代码架构基本不用大改,只是多接入了一个可以按需调用的工具而已。如果是我自己来设计一套类似系统,我会觉得这种思路最大的吸引力,恰恰是工程复杂度的下降,你不需要在系统里再造一个调度器,出了问题排查起来也更直观,因为始终只有一条主线在跑,顾问只是这条主线上偶尔插入的一句旁白。

说完了思路,我们看看 Anthropic 这次到底做了什么产品。他们把这套顾问策略直接做成了 Claude Platform 上的一个原生工具,叫 advisor 工具,目前是公开测试版。集成方式很简单,你照常用 Messages 这套接口发请求,执行模型该填什么就填什么,比如 Sonnet 四点六,然后在工具列表里多加一项工具声明,类型填成顾问类型,模型字段指定成 Opus 四点六,再给一个叫最大调用次数的参数,设定这次请求里顾问最多能被调用几次,比如三次,其他原来已经在用的工具,不管是网页搜索还是代码执行,原样保留不动。这样配置完,执行模型就知道在遇到难题的时候,可以主动去调用这个顾问工具,而且整个过程,从执行者卡壳、调用顾问、顾问看上下文给建议、执行者拿着建议继续跑,全部都发生在同一次接口请求里面,不需要自己写代码去做模型之间的切换、传递上下文,也不需要额外的网络往返。

计费方式设计得也挺巧妙。顾问产生的文本,按顾问模型也就是 Opus 的价格来算;执行者产生的文本,按执行者模型,比如 Sonnet 或者 Haiku 的价格来算。关键是,顾问每次只需要给一段简短的判断或者计划,官方说典型情况下大概是四百到七百个文本单位,这个量级很小;而真正占大头的、动辄成千上万个单位的完整输出,始终是执行者在用它自己更便宜的价格生成的。我们可以粗略推算一下,假设一个完整任务执行者要输出几万个单位的内容,顾问只插一句几百个单位的话,即便顾问那几百个单位按 Opus 的高价计费,摆在几万个单位的总量里,占比依然很小。所以整体算下来,总账单还是远远低于把整个任务从头到尾都交给 Opus 跑一遍。这就跟你花钱请一个咨询顾问按次收费给意见,跟花钱雇这个顾问坐在工位上把活全干完,完全不是一个价位。官方也内置了成本控制,前面说的最大调用次数参数可以直接限制一次请求里顾问最多被叫几次,顾问产生的消耗也会在返回结果的用量统计里单独列出来,方便按档位去追踪开支,不会跟执行者的消耗混在一起算不清楚。

光说原理不如看数据。Anthropic 在文章里给了几组测试结果。第一组是 SWE-bench Multilingual,这是一个多语言的软件工程基准测试,专门考验模型修复真实代码仓库里的问题的能力,覆盖九种编程语言,这一点其实挺重要,因为现实世界的代码仓库不会只有英文和单一语言,多语言场景更贴近真实工作。结果是,Sonnet 加上 Opus 顾问之后,得分比单独跑 Sonnet 提升了二点七个百分点,同时每个任务的平均成本反而下降了百分之十一点九,而且这个结果是跑了五轮、每轮三百个问题取平均得出来的,不是某一次运气好的孤例。也就是说,这不是那种为了多一点效果就要多花很多钱的取舍,而是又好又便宜,两头都占了。

第二组是 BrowseComp 和 Terminal-Bench 二点零,前者是考验模型上网搜索、整理信息能力的基准,一共有一千二百多个真实问题各跑一次;后者是考验模型在终端命令行环境里完成任务的基准,覆盖八十九个任务,每个任务跑五次取平均,而且测试环境给了三倍的资源配置,保证不是因为资源不够导致跑崩。结果同样是,Sonnet 加 Opus 顾问的组合,分数比单独跑 Sonnet 更高,但是每个任务花的钱反而比单独跑 Sonnet 更少。

第三组结果我个人觉得最有意思,是把执行者换成更便宜的 Haiku。在 BrowseComp 这个网页调研基准上,单独跑 Haiku 的得分是百分之十九点七,但是给它配上 Opus 顾问之后,得分直接冲到百分之四十一点二,翻了一倍还多。当然跟单独跑 Sonnet 比,Haiku 加顾问的分数还是要低百分之二十九左右,毕竟 Haiku 本身的底子比 Sonnet 弱一些,顾问能救场但不能完全补齐基础能力的差距。可是看成本,Haiku 加顾问的每任务花费,只是单独跑 Sonnet 的百分之十五,换句话说便宜了百分之八十五。你可以想象一下这样一个场景,你要批量处理几十万条网页调研请求,比如帮一个市场团队去抓取并总结一大批竞品信息,精度要求没到顶配,但又不能太离谱,这种又便宜又比纯 Haiku 聪明不少的组合,性价比是相当突出的,适合那种任务量极大、单次任务要求没那么极致的场景。

还有一点值得一提,顾问工具不是一个独立的、需要单独接入的新系统,它就是 Messages 请求工具列表中普普通通的一项,跟网页搜索工具、代码执行工具放在一起,平等对待。也就是说原来的智能体已经有的能力完全不受影响,执行者在同一个循环里,既可以正常去搜网页、跑代码,也可以在需要的时候捎带问一句 Opus。举个例子,一个做竞品调研的智能体,可能先用网页搜索抓取几十篇资料,再用代码执行去做一些数据整理,等到要在几条互相矛盾的信息里做判断、决定到底该相信哪个来源的时候,正好可以调用顾问来拍板,整个流程不需要为了用上这个功能重新设计工具调用逻辑。

接下来说说,这件事对我们日常做开发到底意味着什么,能怎么用,要避开哪些坑。

第一个能直接落地的场景,是写代码相关的智能体。如果你在做一个自动修复缺陷、写测试、做代码评审之类的智能体,大部分日常的代码改动其实套路化程度很高,Sonnet 完全能搞定,但是偶尔会遇到那种架构层面的取舍,比如两种重构方案各有优劣、测试结果出现互相矛盾的信号、或者一个很隐蔽的并发问题不知道该怎么处理,这种时候让 Sonnet 当执行者,配一个 Opus 顾问,既能保住日常效率和成本,又能在真正烧脑的节点上借到更强的推理能力。

第二个场景,是任务量特别大、但是单次任务要求没那么极致的场景,比如批量网页调研、内容审核、客服工单分类、海量数据清洗里需要一点判断力的环节。如果你本来在用 Haiku 但是经常因为判断力不够翻车,不妨试试 Haiku 加 Opus 顾问的组合,前面的数据已经说明,这个组合能把分数顶上去一大截,同时成本依然远低于直接换成 Sonnet,更别提换成 Opus。

具体接入,官方给了三步。第一步,在请求里加一个测试版功能的请求头,告诉接口你要用这个公开测试功能。第二步,在工具列表里加上顾问工具的声明,指定好顾问模型和最大调用次数。第三步,根据自己的业务场景调整系统提示词,这一步其实是最容易被忽略,但是恰恰最关键的一步。

为什么系统提示词这么关键,因为执行者要不要去问顾问,这个判断本身就是个需要引导的行为。如果系统提示词里完全没有告诉执行者什么情况算是卡住了、什么情况值得求助,它可能会过度依赖顾问,一遇到一点小麻烦就去问一句,这样顾问调用次数上去了,前面说的成本优势就被吃掉一部分;反过来,如果完全没有引导,执行者也可能死扛到底不知道该求助,那顾问形同虚设,白白多接入一个工具却没起作用。所以比较实用的做法,是在系统提示词里明确列出几类该求助的情形,比如出现两种方案都说得通但是下错决策代价很高的时候、出现工具返回结果互相矛盾解读不一致的时候、或者面对一个超出常规套路的新情况完全没有把握的时候,把这些场景具体写出来,执行者才知道什么时候该按下求助的按钮,这样才能把这个机制的效果真正发挥出来。

另外一个要注意的坑,是别把这个策略当成万能解药。如果你的任务从头到尾都需要持续的高阶推理,不是偶尔几个决策点卡壳,而是每一步都需要深度思考,顾问策略能帮上的就有限了,这种情况可能还是直接用 Opus 独立跑性价比更高,因为顾问只在被动调用的瞬间起作用,没办法替执行者把整个任务的思考深度都垫高。顾问策略最适合的场景,是那种任务整体比较常规、但是夹杂着少数几个高难度决策点的情况,这正是文章里测试场景的共同特点,无论是修代码、做网页调研还是跑终端命令,大部分步骤都是常规操作,只有少数节点需要真正烧脑。

官方还给了一个挺实用的建议,就是不要直接全量切换,而是先拿自己手头已有的评测集,跑三组对照,单独跑 Sonnet、Sonnet 加 Opus 顾问、单独跑 Opus,三组的分数和成本都摆出来比一比,再决定要不要在生产环境里换上这个组合。毕竟不同业务场景的决策密度差别很大,文章给的那几个基准测试结果,终究是别人的场景跑出来的数据,落到自己的业务里,效果需要自己去验证。另外因为这个功能现在还是公开测试版,而且发布到现在已经过去两个多月,如果打算认真接入生产环境,建议去看一眼官方最新的文档,确认这个功能是不是已经正式发布、计费规则有没有调整、支持的模型版本有没有更新。

我们简单总结一下。这篇文章核心讲的是顾问策略,本质是把一个原来要么全用贵模型、要么全用便宜模型的二选一问题,变成了一个动态分工问题,让便宜模型负责大部分常规工作,只在关键决策点临时借用更聪明的模型出主意,Opus 顾问不抢主导权,不直接面对用户,只负责在被问到的时候给一句靠谱的判断。官方测试数据显示,这种组合能在好几个基准测试上做到分数提升、成本反而下降,Haiku 配顾问的组合更是把性价比拉到了一个很夸张的水平。现在 Anthropic 已经把它做成了接口里现成的顾问工具,理论上接入成本就是一行配置加一段系统提示词的调整。

我个人认为,这篇文章背后更值得关注的,其实不是这个具体工具本身,而是它反映出的一个设计思路的转变,就是做智能体不一定非要在模型选择上做单一决定,完全可以学着像管理团队一样给任务分层,常规工作交给便宜模型,真正的难题留给贵模型,按需调用而不是按角色固定分配。这个思路特别值得我们在设计自己的智能体系统的时候借鉴,哪怕暂时用不上 Anthropic 这个现成的顾问工具,这种分层求助的架构思想本身也可以用别的方式自己实现出来。当然我也要提醒一句,任何新工具刚出来的时候,数据都是在官方设计的测试场景里跑出来的,实际落地效果还是要靠自己的评测说话,不要看到漂亮的基准数据就直接无脑切换到生产环境里去。
