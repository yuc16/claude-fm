欢迎收听 Claude FM,我是你的主播。今天我们要聊的这篇文章,标题是How Anthropic teams use Claude Code,翻译过来就是,Anthropic 公司内部团队是怎么使用 Claude Code 的。这篇文章发表于二零二五年七月二十四日,发表方是 Anthropic 官方博客。

先说一句提醒。今天是二零二六年六月二十一日,这篇文章距离现在已经快一年了。在 AI 这个领域,一年的时间跨度不算短,模型和工具都在飞速迭代。所以听这期节目的时候,大家要清楚这是一篇带有时间戳的快照式记录,它反映的是去年夏天,Claude Code 刚刚在 Anthropic 内部全面铺开使用的那个阶段的真实状态。当时 Claude Code 作为一个命令行里的智能编程助手,正处在从小范围试用走向公司全员日常工具的关键节点。文章里提到的很多具体效率数字,比如排查问题快了三倍,研究时间减少八成,这些都是基于当时的模型能力和当时团队的工作流程得出的。如果你现在去问,可能效率提升的幅度会更高,因为模型本身在这一年里也升级了好几代。但这篇文章真正有价值的地方,不是那些具体数字,而是它揗示出来的一种工作方式的转变,这个转变到今天依然成立,甚至更明显了。

那么我们先说说背景。在这篇文章发表之前,行业里对 Claude Code 这类智能编程工具的认知,大体上还停留在它是程序员的效率工具这个层面上。你可以理解成,大家觉得这就是一个升级版的代码自动补全,或者是一个能帮你写写测试、改改 bug 的助手。但 Anthropic 作为这个工具的开发方,他们有一个特别好的条件,就是可以直接观察自己几百号员工,跨越各种岗位,是怎么真实使用这个工具的。这篇文章其实就是一次内部调研的总结,他们采访了公司里不同团队的员工,想搞清楚一个问题,这个工具到底被用在了哪些他们没有预料到的地方。

调研的结果挺有意思,文章一开头就点出了一个反差。很多用法是符合预期的,比如调试代码、熟悉陌生的代码库、管理工作流程,这些都是程序员日常会干的事。但还有一些用法完全出乎意料,比如说法务团队用它搭建了电话分流系统,市场团队几秒钟内生成几百条广告文案的变体,数据科学家在完全不会写 JavaScript 的情况下,做出了复杂的可视化界面。这些案例放在一起看,作者得出了一个核心判断,他们说,智能编程不只是在加速传统的软件开发,它正在消解技术岗位和非技术岗位之间的边界。换句话说,任何能把问题描述清楚的人,都有可能变成一个能动手解决问题的人。

我觉得这个判断是整篇文章最值得我们花时间琢磨的地方。我们接下来就分几个层面,把文章里提到的具体场景拆开讲讲。

第一个层面,是关于熟悉陌生代码库这件事。我们都知道,一个新人入职,要花多长时间才能对一个庞大的代码库建立起足够的心智模型,这是软件行业里一个长期存在的痛点。文章里举了 Anthropic 基础设施团队的例子,新来的数据科学家会把整个代码库喂给 Claude Code,让它去读里面的说明文档,这些文档在 Anthropic 内部叫做 CLAUDE 点 md 文件,你可以理解成是专门写给 AI 助手看的项目说明书。Claude 读完之后,能识别出哪些文件是相关的,还能解释数据管道之间的依赖关系,告诉你哪些上游数据源最终汇入到了哪个仪表盘。这其实替代了传统的数据目录工具,过去你可能需要专门的元数据管理系统,现在直接让 AI 读代码、读文档,就能给你一个清晰的依赖关系图。

产品工程团队的用法也很典型,他们把 Claude Code 当成处理任何编程任务的第一站。遇到要修 bug,要加功能,要做分析,第一件事就是先问 Claude,应该去看哪些文件。这听起来简单,但实际上省掉了一个特别耗时的环节,就是人工去翻代码、找上下文这个过程。你可以想象一下,如果你刚接手一个项目,光是搞清楚改一个功能要动哪几个文件,可能就要花上半天时间去翻目录结构、看调用关系。现在这个环节直接外包给了 AI,人力就可以省下来去做真正的设计和判断。

第二个层面,是写测试和代码审查这两件事。这两件事在程序员的世界里,大家普遍承认很重要,但又普遍不太愿意花时间认真做,因为它枯燥、重复、还容易被压缩在项目排期里被砍掉。文章里提到产品设计团队用 Claude Code 给新功能写完整的测试,而且他们把这个流程接入到了 GitHub Actions 里,也就是代码仓库的自动化流水线,Claude 会自动去处理拉取请求里的评论,自动修格式问题,自动重构测试用例。这意味着代码审查这个环节,有一部分工作量被自动接管了。

安全工程团队的故事更能说明问题。他们原来的工作流程是这样的,先写设计文档,然后写出一版比较粗糙的代码,接着重构,重构到最后因为时间不够,测试这一步就被放弃了。这是很多团队心里都明白、但又没办法的现实,叫做先把功能跑起来,测试以后再补,但以后往往就不会再补了。现在他们的流程变成了,先让 Claude 写伪代码,然后引导它做测试驱动开发,也就是先写测试再写实现的那种开发方式,自己只是定期检查进度。这个转变背后的逻辑是,测试驱动开发本身是被广泛认可的好实践,但它执行起来很费心力,因为你要不断地在测试和实现之间切换思维。把这个切换的负担交给 AI 去扛,人只负责把关方向,这样测试覆盖率反而提高了,代码的可靠性也跟着提高。

还有一个细节,推理团队会遇到需要用 Rust 这种不熟悉的语言写测试的情况,他们的做法是,把想测试的逻辑用自己熟悉的方式描述清楚,然后让 Claude 用目标语言的原生写法去实现。这其实是把语言能力这个门槛给绕过去了,你不需要先精通 Rust 才能给 Rust 代码写测试,你只需要懂你要测什么。

第三个层面,我觉得是整篇文章里最贴近大家日常痛点的,就是生产事故排查。线上出问题的时候,时间压力特别大,而且往往是在一个你不那么熟悉的系统里去定位问题,这种压力下人很容易出错,或者效率特别低。文章里说,安全工程团队在处理事故的时候,会把堆栈跟踪信息和文档一起喂给 Claude Code,让它帮忙顺着代码的调用链路去追踪问题。原来靠人工扫描代码定位问题,大概要花十到十五分钟,现在这个过程能快三倍。

产品工程团队也提到,有了 Claude Code 之后,他们处理不熟悉代码库里的 bug 的信心明显增强了。以前遇到不熟悉的模块出问题,可能要去找别的团队帮忙,现在他们直接问 Claude,这是我看到的现象,你能帮我修一下吗,然后自己审查 Claude 给出的方案。这个变化其实挺微妙的,它不是说 AI 替代了团队协作,而是把原来需要跨团队沟通才能解决的小问题,变成了一个人就能搞定的事情,省下来的时间和沟通成本是看不见但很真实的。

文章里有一个我觉得特别生动的案例,数据基础设施团队遇到 Kubernetes 集群,也就是常说的容器编排系统,突然不再调度新的工作负载了,这是个挺紧急的状况。他们把仪表盘的截图直接喂给 Claude,Claude 通过截图,一个菜单一个菜单地引导他们在 Google Cloud 的控制台里排查,最后定位到问题是 pod 的 IP 地址池用尽了。然后 Claude 直接给出了具体的命令,告诉他们怎么创建新的 IP 池,怎么把它加到集群里。整个过程帮他们在系统故障期间省下了二十分钟。我们都知道,在系统故障期间,每一分钟都是真金白银,二十分钟可能就是几万次请求超时和几个客户投诉的差别。这个案例也说明,Claude Code 处理图片输入、结合实际控制台界面去给指导,这个能力在排障场景里特别实用,它不只是读代码,还能读截图、读界面状态。

第四个层面,是关于功能开发和原型搭建。传统意义上,做一个新功能或者一个新应用,门槛是比较高的,你需要懂技术,还要投入相当长的时间。文章里提到产品设计团队的做法挺值得说一说,他们把 Figma 的设计文件直接喂给 Claude Code,然后设置一个自主循环,让 Claude 自己写代码、自己跑测试、自己迭代。设计师给 Claude 一个比较抽象的问题,让它自主工作一段时间,然后再去审查它给出的方案,做最后的微调。文章里还提了一个挺有意思的细节,他们让 Claude 给自己写了 Vim 风格的键位绑定,基本上没有人工参与审查。这听起来有点像是,工具开始反过来改造自己,这种自我迭代的场景,在过去是很难想象普通设计师能参与进来的。

更值得注意的是,产品设计团队发现了一个意外的用法,他们用 Claude Code 去提前梳理错误状态、逻辑流程和系统状态,在设计阶段就能发现各种边界情况,而不是等到开发阶段才暴露出来。这个事情背后的道理其实很朴素,设计和开发之间一直存在一个信息不对称,设计师画图的时候很容易忽略掉一些边界情况,比如网络断开了怎么显示,数据为空了怎么处理,这些细节往往要等程序员动手写代码的时候才会被发现,然后再返工去改设计。现在因为设计师自己就能借助 Claude Code 把这些逻辑流程在脱离实际开发的情况下提前跑一遍,这相当于把发现问题的时间点往前挪了一大截,从开发阶段挪到了设计阶段,省下了后面大量的返工时间。

数据科学家用 Claude Code 构建可视化应用这个案例也很说明问题。文章说,这些数据科学家并不熟悉 TypeScript,但他们用 Claude Code 在沙盒环境里,一次性生成了完整的 React 应用,用来可视化强化学习模型的表现。整个过程他们自己并不真正理解生成出来的代码,如果第一次生成的结果不够好,他们就稍微调整一下提示词再试一次。这个场景特别能体现文章开头说的那个核心判断,技术和非技术之间的边界正在模糊。一个不会写前端代码的人,只要能描述清楚自己想要的可视化效果是什么样子,就能拿到一个能用的应用。这在过去是不可想象的,过去你想做一个像样的可视化界面,至少得找一个前端工程师帮忙,现在这个依赖被打破了。

第五个层面,是知识沉淀和文档整理。任何一个稍微有点规模的公司,技术文档都会面临同一个老问题,信息散落在维基百科式的内部知识库里,散落在代码注释里,还散落在老员工的脑子里,新人或者跨团队的人想找到一个答案,常常要靠运气和人脉。文章提到推理团队里没有机器学习背景的成员,依赖 Claude 来解释那些模型相关的专业函数,原来靠搜索引擎要花一个小时去查的东西,现在十到二十分钟就能搞清楚,效率提升了大概百分之八十。这个数字我觉得挺真实,因为搜索引擎给你的是零散的信息碎片,你还要自己拼凑、自己判断哪些是过时的、哪些是适用的,而直接问一个理解你具体代码上下文的助手,它可以直接给你一个针对性的解释,省掉了拼凑这一步。

安全工程团队的做法是让 Claude 整合多个文档来源,生成 markdown 格式的运行手册和故障排查指南。这些被压缩浓缩过的文档,后续被当作上下文喂给 Claude,用来调试真实的生产问题。这其实是一个挺聪明的二次利用,先用 AI 把零散的知识库整理成精炼的手册,再把这个精炼的手册喂给 AI 去解决实际问题,比直接在完整的知识库里搜索要高效得多。你可以把这个理解成一种知识蒸馏,把厚厚的一本百科全书,提炼成一页纸的速查表,而且这个提炼过程也是自动化的。

第六个层面,我觉得是整篇文章里最出乎意料的部分,就是非工程团队自己动手搭建自动化工具。增长营销团队搭了一个智能工作流,可以处理包含几百条广告的 CSV 表格文件,自动识别出表现不好的广告,然后在严格的字符数限制内生成新的文案变体。他们用了两个分工明确的子智能体来配合完成这件事,整个系统几分钟内就能生成几百条新广告,这件事原本要花上几个小时。他们还做了一个 Figma 插件,能识别设计稿里的版面,程序化地生成多达一百种广告变体,做法是替换标题和描述文案,把原来需要人工反复复制粘贴几个小时的工作,压缩到每批广告处理只需要半秒钟。

法务团队的案例我觉得特别有代表性,他们做了一个电话分流系统的原型,帮助公司内部的人找到合适的律师对接。这件事本身的技术含量谈不上多高,但关键在于,这是一个完全没有开发背景的部门,自己动手解决了一个原本需要排队等开发资源才能解决的问题。这说明这套工具的影响力已经不局限在工程部门内部,而是渗透到了公司运营的方方面面。

讲完这些案例,我们来聊聊这对我们普通听众,尤其是做技术或者做产品的朋友,到底意味着什么,我们能从中学到什么、上手怎么用、又要避开哪些坑。

第一点,如果你还把这类智能编程工具单纯当成代码补全或者写 bug 修复脚本的工具,那你可能低估了它的使用边界。文章里反复出现的一个模式是,把完整的上下文喂给它,无论是整个代码库,还是设计稿、截图、文档,然后让它在这个上下文里去推理、去给方案。这意味着你日常工作里,任何一个能被清楚描述、并且有素材可以喂进去的问题,都值得你试一试用这类工具来处理,而不只是写代码的时候才想起它。

第二点,关于测试驱动开发这件事,我觉得安全工程团队的转变特别值得借鉴。很多团队心里都清楚测试驱动开发是好实践,但执行起来太累人,所以现实中常常被牺牲掉。如果你也面临类似的情况,可以试试把这个流程外包给智能编程工具,让它先帮你写伪代码,再带着你做测试驱动开发,自己只负责定期检查方向对不对。这个分工方式,本质上是把执行层面的认知负荷转移出去,把人的精力留给判断和把关。

第三点,在生产事故排查这件事上,文章给出的经验是,事故发生时不要犹豫,直接把堆栈信息、日志、截图都喂给工具,让它帮你顺着调用链路去推理,这能明显提速。但这里我要提醒一句,这是我个人的看法,生产事故排查涉及到系统的实际操作权限,AI 给出的命令和建议,在执行之前一定要有人确认,尤其是涉及到删除数据、修改集群配置这类不可逆操作的时候,不能因为追求速度就跳过人工复核这一步。文章里那个 IP 地址池用尽的案例,虽然结果是好的,但我们要清楚,这类操作如果给错了命令,代价也可能是放大故障而不是修复故障。

第四点,如果你是非技术岗位的同事,看完这篇文章应该有一种被解放的感觉。无论你是做设计、做市场、还是做法务,只要你能把自己想要的东西描述清楚,你就有可能直接动手做一个原型出来,而不需要永远排在开发资源的队伍最后面。但这里也要泼一点冷水,文章里这些非技术团队做出来的东西,大多还停留在原型和内部工具的层级,距离生产级别、需要长期维护和承担安全责任的系统,中间还有不小的距离。我个人建议是,把这类工具用在快速验证想法、做内部小工具这个场景上非常合适,但凡是涉及到对外暴露、涉及到用户数据、涉及到长期稳定运行的系统,还是应该让专业的工程团队介入做把关。

第五点,文章最后总结的那句话我觉得是整篇文章的灵魂,他们说最成功的团队,是把这个工具当成思考伙伴,而不是单纯的代码生成器。这句话背后的意思是,你跟它互动的方式,应该是探索可能性、快速做原型、把发现的东西在技术和非技术同事之间互相分享,而不是简单地丢一个需求过去等着收货。这种协作的心态,我觉得才是这篇文章真正想传达给读者的东西,工具本身会不断升级,但这种把 AI 当作一起琢磨问题的伙伴、而不是单纯下指令的执行者的心态,才是能持续受益的关键。

好了,我们简单总结一下今天这期节目。这篇文章用一系列来自 Anthropic 内部不同团队的真实案例,告诉我们 Claude Code 这类智能编程工具的使用边界,远远超出了写代码这件事本身。它帮新人快速读懂陌生代码库,帮工程团队把测试和代码审查这两件苦差事自动化,帮大家在生产事故里更快定位问题,也帮非技术团队绕过开发资源直接动手解决自己的问题。核心的转变是,技术能力的门槛在被拉低,描述问题的能力变得和写代码的能力一样重要。

我个人的看法是,这篇文章虽然已经过去快一年,里面的具体效率数字可能已经被新一代模型超越了,但它揭示的这种工作方式的转变,到今天依然在持续发生,甚至随着工具能力的提升还在加速。对我们每个人来说,真正值得思考的问题不是这个工具能不能用,而是我们有没有调整好自己的工作习惯,去把它当作一个可以并肩思考的伙伴。
