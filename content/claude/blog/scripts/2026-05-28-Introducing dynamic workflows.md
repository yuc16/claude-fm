欢迎收听 Claude FM，我是你的主播。今天我们来聊一篇非常值得花时间深入拆解的文章。

这篇文章的英文标题是 Introducing dynamic workflows in Claude Code，我把它翻译成"在 Claude Code 中引入动态工作流"。文章发表于二〇二六年五月二十八日，距今大约三周，是一篇非常新的内容，讲的是 Anthropic 刚刚正式向公众开放的一项功能——动态工作流，也就是 dynamic workflows。

为什么这篇文章值得专门花二十多分钟来讲？因为它代表的不只是一个功能更新，而是 AI 编程助手这个方向上一个非常重要的范式转变。文章里有一个案例让我第一次读到的时候觉得非常震撼——有人用动态工作流，把一个超过七十五万行代码的项目从 Zig 语言移植到了 Rust 语言，只花了十一天，测试通过率达到了百分之九十九点八。这不是实验室里的演示，而是已经实际发生的事情。今天我们就来仔细看看，动态工作流到底是什么，它是怎么做到这些事情的，以及对我们日常的软件工程工作意味着什么。

在正式讲动态工作流之前，我们先来聊聊背景，理解 Anthropic 为什么要在这个时间点推出这个东西，以及它要解决的是什么问题。

在过去一两年里，AI 编程助手这个赛道经历了一次明显的进化。最早的时候，GitHub Copilot 那一代工具，本质上是一个非常聪明的自动补全，你写代码，它给你建议下一行，或者建议一个完整的函数体。那个阶段的 AI，工作范围基本上在函数级别。

后来出现了 Cursor、Claude Code 这一类工具，它们开始能够跨文件修改代码、写测试、修复 bug，操作范围从一个函数扩展到了一个模块，甚至是一个小型项目。我们把这一阶段理解为"单个 agent 处理一个完整任务"的阶段，它的上限基本上就是一个聪明的工程师，在一个工作时段里能独立处理的工作量。

但是，当任务规模再往上走，比如你需要在一个有几千个文件的遗留代码库里做全局的安全审计，或者你要把整个后端服务从一个框架迁移到另一个框架，再或者你需要对一个核心算法的改动做全面的影响评估，单个 AI agent 就开始力不从心了。

这里面的核心矛盾是什么？有两个维度。

第一是上下文窗口的物理限制。一个 agent 在一个会话里，能直接看到的内容是有上限的。当你的任务涉及的文件数量、代码量超过这个上限，agent 要么只能分批处理，要么就只能看到局部信息来做全局判断，这两种情况都会影响结果质量。

第二是串行处理的效率瓶颈。即使上下文窗口足够大，单个 agent 处理任务本质上是串行的——它做完这个再做那个。但很多大型工程任务，从逻辑上来说是完全可以并行的。比如扫描代码库里的安全漏洞，不同模块之间完全可以同时进行分析，没有必要等一个模块分析完才开始下一个。

更深层的问题是，有些工程任务不只是"量大"，它还需要"独立交叉验证"。如果你在做一个安全审计，你当然希望不只有一个 agent 在扫描，而是有多个 agent 各自独立工作，然后交叉比对彼此的结论，这样发现的问题更可信，漏掉的风险也更低。靠单个 agent，这种机制根本做不到。

Anthropic 对这些问题的回答，就是今天要讲的动态工作流。

那么动态工作流到底是什么？我们来逐层拆解。

最直观的理解：动态工作流是一种机制，让 Claude 能够自己动态写一个编排脚本，然后用这个脚本启动几十个、甚至几百个并行运行的子 agent，每个子 agent 各自完成一部分任务，结果在汇总之前经过验证，最后你拿到的是一个协调统一的答案。

这里有两个关键词需要展开——"动态"和"并行"。

先说"动态"。动态的意思，是说 Claude 在启动工作流之前，会先理解你的任务，然后动态规划怎么拆解这个任务，拆成哪些子任务，每个子任务由什么样的 agent 来执行，各个 agent 之间的依赖关系是什么，哪些可以同时跑，哪些必须有先后顺序。这不是预先写好的固定流程，而是 Claude 根据你的具体问题临时生成的编排方案。

这就像一个有经验的工程经理接到一个大项目时会做的事情。他不会直接上手写第一行代码，而是先做项目分解，把工作拆成若干个独立的模块，评估每个模块的工作量和依赖关系，然后按照最优的方式把工作分配给团队成员，让能并行的任务同时推进。动态工作流里的 Claude，做的就是这个角色的工作。

再说"并行"。多个子 agent 同时在跑，而不是一个做完了再做下一个。这在处理大型代码库的时候尤其重要。想象一下，如果你要扫描一个有一千个文件的代码库来找安全漏洞，串行处理每个文件需要的时间和并行启动一百个 agent 各自处理十个文件需要的时间，差别是将近一百倍。当然，实际的加速比没有这么理想，因为每个 agent 启动有开销，结果汇总也有时间，但数量级上的提升是实实在在的。

文章里提到了三个典型的使用场景，我来帮你翻译一下并说说我的理解。

第一类是全代码库级别的扫描和审计，包括 bug 搜索、性能分析和安全审计。Claude 会在整个服务或仓库里并行搜索，然后对每一个发现都进行独立验证，确保最终报告里出现的都是真实的问题，而不是误报。文章里特别提到，同样的工作模式也适用于安全加固——比如对整个代码库做鉴权检查、输入校验，以及不安全编程模式的全面扫描。

第二类是大型迁移和现代化改造。框架升级、API 废弃替换、编程语言移植，这类改动往往涉及数百甚至数千个文件，工作量非常大，而且各个文件之间的改动相对独立，非常适合并行处理。文章说动态工作流可以端到端地处理这类任务，我们等一下会通过具体案例来感受一下这个"端到端"到底意味着什么。

第三类是对正确性要求非常高的关键工作。当一个错误答案的代价很高的时候，工作流会让 Claude 对同一个问题发起独立的多次尝试，还会安排专门的"对抗性 agent"来尝试推翻已有结论，在你看到最终结果之前就把潜在的错误揪出来。

现在我们来看文章里最令人印象深刻的那个真实案例——Bun 的代码移植。

Bun 是一个 JavaScript 运行时，如果你做前端或者全栈开发，可能听说过这个项目。它是 Node.js 的一个高性能替代品，以启动速度快、执行效率高著称。Bun 的创始人叫 Jarred Sumner，他用动态工作流做了一件在正常情况下需要一个团队花相当长时间才能完成的事情：把 Bun 的核心代码从 Zig 语言移植到了 Rust 语言。

具体数字是：大约七十五万行 Rust 代码，从第一个提交到合并只花了十一天，而且现有测试套件的通过率是百分之九十九点八。

我们来感受一下这件事的难度。七十五万行代码是什么量级？一个中等规模的商业软件项目，可能有几十万到几百万行代码，七十五万行处于这个范围的中间位置。但量大还不是最难的，更难的是从 Zig 移植到 Rust 这件事本身。这两门语言虽然都属于系统编程语言，都主打高性能和内存安全，但它们的设计哲学和语法差异相当大。

其中最有挑战性的是 Rust 的生命周期系统，也就是 lifetime 机制。在 Rust 里，每一个引用都有一个生命周期，编译器需要你明确标注或者能够推断出这些生命周期，才能保证内存安全。对于一个从其他语言移植过来的代码库，你需要为每一个数据结构的每一个字段想清楚它的生命周期语义，然后用 Rust 的语法正确表达出来。这是 Rust 的一大学习门槛，也是移植工作里最需要深度思考的部分。

那这十一天里，动态工作流是按照什么步骤推进这个移植的呢？文章里描述了几个阶段，我来逐一拆解。

第一个阶段是生命周期映射。一个专门的工作流负责分析原有 Zig 代码库里每一个结构体的每一个字段，然后为它们在 Rust 中确定正确的 lifetime 标注。这是移植工作的基础，也是最考验对两种语言理解深度的部分。动态工作流在这个阶段调动了大量并行 agent 来处理不同的结构体，把本来需要人工逐个审查的过程大幅压缩。

第二个阶段是代码移植本身。在 lifetime 映射完成之后，工作流开始为每一个 Zig 源文件生成对应的 Rust 源文件，要求每个文件在行为上与原始的 Zig 文件完全一致。文章说，这里用了大量并行 agent 同时处理几百个文件，而且每个文件还配了两个专门的审查员 agent 来做代码审查。你可以把这想象成一个大型工厂里的流水线——不只有负责生产的工位，还有内嵌的质检环节。

第三个阶段是修复循环。代码移植完成之后，工作流驱动构建系统和测试套件持续运行，不断发现问题、修复问题，直到构建和测试都干净通过为止。这是一个迭代收敛的过程，有点像 CI/CD 流水线里的自动化测试，但这里的"修复者"本身也是 AI agent。整个过程不需要人工介入，工作流自己驱动自己，一直跑到达到收敛条件。

第四个阶段是性能优化。在代码移植落地之后，一个过夜运行的工作流专门去处理不必要的数据拷贝问题，并且为每一个优化改动单独开了一个 PR，等待人工做最终审查。这个阶段展示了工作流的另一种用法——在基础工作完成之后，用一个独立的工作流来做质量提升。

这个案例让我印象最深刻的地方，不只是那些漂亮的数字，而是它展示的一种工程协作的新范式。人类在这个过程里的角色发生了变化：不再是主要的执行者，而是问题定义者、关键节点的决策者，以及最终结果的审查者。大量的探索性工作、执行性工作、验证性工作，都由 AI agent 的集群来承担。

当然，文章也说了，这个 Bun 的 Rust 移植版本目前还没有上生产环境，Jarred 后续还会专门写文章介绍这个过程的细节。所以我们要把这个案例理解为一个早期探索性的结果，它展示了技术上的可能性，但距离成熟的生产应用模式，可能还有一段路要走。

好，案例看完了，我们来看动态工作流在技术层面是怎么运作的，文章里有一段对工作原理的描述，让我来帮你拆解一下。

当一个工作流启动的时候，Claude 会基于你的 prompt 动态做规划，把任务拆成子任务，然后把这些子任务扇形展开，分配给并行运行的多个子 agent 来执行。子 agent 的结果在被整合进最终答案之前会先经过验证，你最终拿到的是一个统一协调过的输出。

文章特别强调了一个设计细节：agent 们从独立的角度处理问题，然后还有其他 agent 专门来挑战和反驳它们的结论，这个验证过程持续迭代，直到答案收敛为止。这就是"对抗性验证"的机制。

我个人认为，这个设计思路非常有意思，因为它把软件工程中的质量保障实践搬进了 agent 的协作框架里。我们在工程实践里有很多提升质量的手段，比如代码审查——不是写代码的人自己检查，而是另一个工程师来看；比如测试驱动开发——写代码之前先想好测试条件；比如安全领域的红队蓝队对抗——一个团队攻击，一个团队防守。动态工作流里的对抗性 agent，本质上是把这些质量保障机制自动化了，内嵌到了工作流本身里面。理论上，这可以显著提升输出结果的可靠性，因为潜在的错误在到达用户之前就已经被质疑和验证过了。

另一个重要的设计是进度持久化。文章说，工作进度会在运行过程中持续保存，所以如果一个任务被中断，它可以从断点继续，而不是从头重跑。对于那些可能运行几个小时甚至几天的工作流来说，这个特性非常关键。你可以把它理解为大型计算任务里的检查点机制——类似于机器学习训练里的 checkpoint，或者视频渲染里的断点续渲，防止意外中断导致所有已完成的工作前功尽弃。

还有一个设计细节值得注意：协调工作发生在对话之外。这意味着不管任务规模多大，整体计划都能保持在轨道上，不会受到单个会话的上下文限制。这解决了我们前面提到的上下文窗口瓶颈问题——工作流架构本身就超越了单个 agent 的上下文边界。

关于平台支持，文章说动态工作流现在在 Claude Code 的命令行工具、桌面端，以及 VS Code 插件里都已经普遍可用，支持的计划包括 Pro、Max、Team 和 Enterprise。除了直接使用 Claude Code，也可以通过 Claude API 来调用，还支持 Amazon Bedrock、Vertex AI 和 Microsoft Foundry 这几个主流的云平台，这对于企业客户来说是一个很重要的信号，因为很多大公司的 AI 基础设施已经建立在这些云平台上了。

好，背景和原理讲完了，我们来聊实践部分。如果你是一个使用 Claude Code 的软件工程师，动态工作流对你的日常工作意味着什么，应该怎么上手，又有哪些坑需要避开？

先说怎么启动。文章建议，在使用动态工作流的时候，最好先开启自动模式，也就是 auto mode，这样 Claude 可以自己判断什么情况下需要用工作流，什么情况下普通的单 agent 就够了。

触发工作流有两种方式。第一种是直接告诉 Claude 创建一个工作流，比如你可以说"帮我创建一个工作流来扫描整个代码库里的 SQL 注入漏洞"，Claude 会理解这个指令并启动相应的工作流。第二种方式是开启一个叫做 ultracode 的 Claude Code 专属设置，这个设置可以在努力等级菜单里找到。开启 ultracode 之后，会把努力等级设置为最高，同时让 Claude 自己判断什么时候需要用工作流来处理你给的任务。

在计划和权限方面，Max、Team 和 Enterprise 计划的用户，动态工作流默认是开启的，可以直接使用。Pro 计划的用户需要在斜杠 config 命令里手动开启。通过 API 使用 Claude Code 的情况下，也是默认开启的。

那在实际工作中，哪些任务特别适合用动态工作流来处理呢？

我觉得可以从三个维度来判断。

第一个维度是任务的广度。如果你的任务需要覆盖大量文件，比如涉及几十个、几百个乃至更多文件的改动，就非常适合用工作流。单个 agent 处理几十个文件也能应对，但当文件数量达到几百个，工作流的并行优势就开始显现了。

第二个维度是任务对并行化的天然友好程度。有些任务本来就可以完全并行——比如"检查所有 REST API 接口是否缺少鉴权"，你可以把代码库按模块切分，多个 agent 同时扫，完全不需要等彼此的结果。反过来，有些任务是高度串行依赖的，比如"先分析 A 模块的接口设计，根据分析结果决定 B 模块怎么改"，这种情况下并行化的收益就没那么大。在考虑用不用工作流之前，先想想你的任务有没有天然可以并行的结构。

第三个维度是对结果可信度的要求。如果你的工作结果需要非常高的可信度，比如安全审计报告、关键系统的核心逻辑重构，那么工作流里内置的对抗性验证机制就特别有价值。你可以在 prompt 里明确说你需要多个独立视角来验证结论，这是工作流相比单个 agent 的一个显著优势。

那有哪些坑需要事先知道呢？

最重要的一点，也是文章里特别强调的：动态工作流会消耗明显更多的 token，比普通的 Claude Code 会话多很多。这是因为你同时启动了几十个、甚至几百个子 agent，每个都在消耗 token。如果你用的是按量计费的 API 方案，成本会有显著提升。文章建议从一个范围比较明确的任务开始，先感受一下在你具体工作场景下的实际消耗，不要一上来就拿最大的任务来跑。

好消息是，第一次启动工作流的时候，Claude Code 会先展示即将要执行的工作流计划，请你确认才开始运行。这个确认步骤是一个很好的机会，让你了解工作流的规模和预估的资源需求，在点击确认之前可以评估一下是否合适。

第二个要注意的是任务描述的清晰度。动态工作流给了 Claude 非常大的自主权来规划和执行，但这也意味着如果你的任务描述含糊，Claude 可能会朝着你不期望的方向展开大量工作。所以在启动一个复杂工作流之前，花时间把任务描述写清楚是很值得的。你可以在 prompt 里明确说明：目标是什么，工作范围是什么（比如哪些目录、哪些文件），什么样的结果算成功，有没有特别需要排除的场景。描述越具体，工作流执行的结果越符合你的预期。

第三个坑是对输出结果的过度信任。即使有对抗性验证机制，AI agent 的输出仍然需要人工审查，尤其是当改动要进入生产环境的时候。Bun 案例里也说了，所有的 PR 都需要最终的人工审查。我个人认为，正确的心态是把动态工作流理解为一个大幅提高你工作效率的工具，而不是可以完全替代人类判断的自动化系统。它可以帮你做大量的探索性工作、生成初步结论、发现潜在问题，但关键决策、最终结果的验证，还是需要有人把关的。

对于工程团队来说，如果你们想在团队层面引入动态工作流，还有一些组织层面的事情值得提前考虑。文章提到，组织管理员可以通过托管设置来控制工作流功能的开放范围，选择性地禁用工作流。这说明 Anthropic 已经考虑到了企业客户的管控需求。对于团队而言，你可能需要制定一些内部规范，比如哪类任务适合启用工作流，工作流的输出需要经过什么样的审查流程，以及如何在团队层面管控 token 消耗预算。这些流程规范建议在推广使用之前就先想清楚，而不是等到账单出来才反应。

我个人觉得，动态工作流在技术债清理这个场景里可能特别有价值，但这是文章里没有明说的，属于我自己的推断。很多团队都有大量积压的技术债——废弃的 API 调用需要替换，过时的依赖需要升级，代码里散落着几百条 TODO 注释，测试覆盖率常年低于理想水平。这类工作量大、优先级又经常被业务需求压下去，结果就一直积压着。动态工作流在这里可能是一个很好的解法。你可以选择一个维护窗口，用一个明确定义的工作流来批量处理某类技术债，大幅降低这类工作的人工成本。当然，生成的改动还是需要人工审查，确保它们做的是正确的事情，而不只是通过了自动化测试而已。

另一个让我很感兴趣的应用场景是系统级的影响分析。当你需要改动一个核心模块，在做改动之前，你通常会花大量时间做影响评估——这个改动会波及哪些地方，有哪些下游系统依赖这个接口，有没有什么边缘情况会被破坏。这类分析需要全面覆盖代码库，正好适合工作流的并行搜索和独立验证能力。你可以让工作流先跑一遍影响分析，给你一份覆盖全局的报告，再基于这份报告来规划你的改动策略，这可能比人工做这件事要全面和快速得多。

好，我们来做今天的总结。

Anthropic 在二〇二六年五月二十八日正式对外开放了 Claude Code 的动态工作流功能。这个功能的核心是让 Claude 能够动态生成任务编排方案，启动几十到几百个并行子 agent，协同处理超大规模的工程任务。技术设计上，最值得关注的几个点是：动态任务分解给了工作流强大的适应性，并行执行解决了效率瓶颈，对抗性验证机制提升了结果的可信度，而进度持久化则让长时间运行的任务有了可靠的中断恢复能力。

文章里 Bun 的案例是目前我见过最有说服力的真实案例之一：七十五万行代码、从 Zig 到 Rust 的语言移植、十一天、百分之九十九点八的测试通过率。即便这个结果还没有进入生产环境，它展示的技术可能性已经足够令人重新思考一些工程任务的边界在哪里。

从实践角度来说，这个功能现在已经可以用了，Pro 用户需要在斜杠 config 里手动开启，其他付费计划默认可用。上手门槛不高，但需要认真对待 token 消耗的问题，建议从范围明确的任务开始探索，结果要保持人工审查的习惯。

我个人的点评是：动态工作流代表的是 AI 辅助软件工程的一个质的飞跃，从"帮你写代码"到"帮你管理工程项目"。这条路上肯定还有很多没有解决的问题，比如如何精确控制 token 消耗，如何在复杂的企业代码库环境里安全地运行大规模 agent，以及如何建立合适的人机协作审查流程。但方向是清晰的，而且文章告诉我们，这不只是在实验室里的技术演示，而是已经有人用它做出了规模化的真实结果。如果你在做软件工程工作，这个功能值得你认真花时间去了解和实验。
