大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是 Claude Code power user customization, how to configure hooks,翻译过来就是 Claude Code 高级用户定制指南,如何配置 hooks。这篇文章是 Anthropic 官方在二〇二五年十二月十一日发布的,发布到今天,也就是二〇二六年六月二十日,已经过去了大概半年多的时间。

我要先提一句时效性的问题。半年时间在 AI 这个行业里不算短,Claude Code 本身的功能可能已经有一些迭代,文章里提到的某些细节,比如具体的环境变量名字或者默认超时时间,实际使用前最好对照官方最新文档确认一遍。但是这篇文章讲的是 hooks 这个机制的设计思路和八种触发时机,这部分内容是相对稳定的架构性知识,不会因为版本更新就整体过时,所以拿来学习思路依然非常有价值。背景上,二〇二五年底正是各家 AI 编程工具从单纯的代码生成,往更深度的工作流嵌入方向发展的阶段,Claude Code 作为一个在终端里跑的智能编程助手,这时候推出 hooks 这套机制,本质上是在回应一个很现实的需求,就是开发者希望这个工具能像自己手里的编辑器一样,被精细地调教成符合自己习惯的样子,而不是反过来,自己去适应工具的默认行为。

那我们先说说,为什么需要 hooks。文章一开头举了几个特别真实的场景。比如说,每次 Claude 写完一个文件,你都要手动跑一下 Prettier 这种代码格式化工具。再比如,你每次执行 npm test 这种命令,Claude Code 都会弹出一个权限确认框,问你要不要允许执行,你点了一百次同意之后,这个弹窗看着就很烦。还有就是,每次开一个新的会话,你都要把项目背景信息重新粘贴一遍给 Claude,告诉它现在项目处于什么状态,有什么注意事项。这些都是日积月累下来的摩擦点,单个看起来都不大,但加在一起就会明显拖慢你的节奏,也会让你觉得这个工具用起来不够顺手。

hooks 这个机制,简单说,就是你可以配置一些自定义的 shell 命令,让它们在 Claude Code 会话里某个特定事件发生的时候自动触发。比如说 Claude 准备要写一个文件了,这是一个事件,你可以在这个事件发生前插入一段你自己的脚本逻辑,去检查一下这个写入操作是不是安全的。或者你提交了一个 prompt,这也是一个事件,你可以在这个时刻往 Claude 的上下文里偷偷塞一些额外信息。这套机制的核心价值,文章里总结成三类问题。

第一类问题,是消除重复的手动操作。本来需要你每次手动做的事情,比如跑格式化工具,现在可以交给一个叫 PostToolUse 的 hook 自动处理,Claude 写完文件,这个 hook 立刻就跑起来,帮你把代码格式化好。本来你要反复点击同意的那些权限确认,可以交给一个叫 PermissionRequest 的 hook 自动批准,只要满足你设定的条件就直接放行,不用你每次都伸手去点。

第二类问题,是自动强制执行项目规则。这个其实挺关键的,因为人是会犯错的,你可能某次心情不好或者太着急,就忘了检查一下要执行的命令是不是危险操作,但是 hook 不会忘,它每次都会跑,不管你当时状态怎么样。你可以用 hook 在危险命令真正执行之前就把它拦下来,比如那种类似删除整个目录的命令,或者强制推送覆盖远程分支的操作。你也可以用 hook 在文件写入之前先验证一下路径是不是合法,避免误操作覆盖了不该覆盖的文件。这些防护措施,因为是自动化的,所以是每次都生效,而不是只在你恰好想起来要检查的时候才生效。

第三类问题,是不费力地往会话里注入动态上下文。举个例子,你可以让一个叫 SessionStart 的 hook,在每次会话开始的时候,自动把你当前的 git 状态和待办事项清单喂给 Claude,这样 Claude 一开口就已经知道你现在项目进展到哪一步了。或者用一个叫 UserPromptSubmit 的 hook,在你每次提交问题的时候,自动把你这个迭代的优先事项附加进去,这样 Claude 始终对你团队当前的重点保持知情,而你不需要每次都重复说一遍。

接下来我们具体聊聊文章里讲的八种 hook 类型,这是整篇文章信息密度最高的部分,我会挑几个最重要的细讲。

先说 PreToolUse,这是最常用的一种 hook。它的触发时机,是 Claude 已经决定要用哪个工具了,但是这个工具还没有真正执行。这个时间窗口非常关键,因为你的脚本可以在这个时候介入,检查一下 Claude 打算做的这个动作,然后决定是批准、是拦截、是请求你来手动确认,甚至还可以直接修改这个动作的参数。文章里给的例子是,针对文件写入这个动作,配置一个 PreToolUse hook,让 Claude 在真正写文件之前,先把这次写入的路径拿去跑一个验证脚本,脚本觉得没问题才放行。这个机制最适合用在拦截危险的 Bash 命令上,比如那种递归强制删除的命令,或者强制推送代码这种不可逆操作。同时它也适合用来自动批准那些你已经验证过很安全、重复执行很多次的操作,这样能明显减少你被反复打扰确认的疲劳感。

第二个是 PermissionRequest,这个 hook 触发的时机,是本来 Claude Code 准备弹出一个权限确认对话框给你看的那一刻。这个 hook 相当于在弹窗出现之前,先让你的脚本来做决定,脚本可以选择允许、拒绝,或者还是按原计划弹出来问你。文章给的例子很贴近日常,就是针对 npm test 这种你跑了无数次的测试命令,直接配置自动批准,这样你就不用每次都看到那个烦人的确认框了。这个 hook 还可以反过来用,比如说阻止任何对生产环境配置文件的写入权限,或者只允许对某几个特定目录的读取操作不用弹窗确认,而别的操作还是要走正常的确认流程。这种细粒度的控制能力,其实就是把权限管理这件事,从你脑子里的记忆,变成了写进配置文件里的规则。

第三个是 PostToolUse,跟 PreToolUse 正好相反,它是在工具执行完成之后才触发。你的脚本这时候能拿到这次操作的具体结果,比如工具的输出内容是什么。文章给的最直观的例子,就是每次 Claude 写入或者编辑了一个文件之后,自动跑一遍 Prettier 这种格式化工具,把代码风格统一好。这个例子之所以被文章作者推荐成新手上路的首选,是因为这个反馈是立即可见的,你配置完之后马上就能看到文件格式变整齐了,验证成本很低,成功的获得感也很强。除了格式化,这个 hook 还能用来记录所有文件修改的审计日志,或者在代码改动之后自动跑一遍代码检查工具,把警告信息展示出来。

第四个我想多讲讲的是 SessionStart,这个 hook 在每次新会话开始,或者你恢复一个已有会话的时候触发。它的特别之处在于,你的脚本往标准输出打印的任何内容,都会自动被加进对话的上下文里,相当于 Claude 一上线就已经读到了这些信息。文章给的例子是,配置一个命令,自动跑 git status 拿到当前的代码改动状态,再读一下项目里的待办清单文件,这样 Claude 一开始工作,就已经清楚你现在的代码改动情况和接下来要做的事情。这个机制特别适合用来注入你当前的分支信息、最近的提交记录,或者跟环境相关的配置细节,本质上就是替你做了那件每次开新会话都要重复一遍的事情,就是给 Claude 交代背景。

第五个值得细说的是 Stop 这个 hook,它的触发时机,是 Claude 已经完成了一轮回复,本来应该停下来等你输入下一条指令了。但是你可以在这个时候插入一个检查逻辑,判断这个任务是不是真的做完了。文章里给的例子很有意思,它用的不是一段普通的 shell 脚本,而是配置了一个 prompt 类型的 hook,让另一个模型调用去审查任务有没有完成,如果没完成,这个脚本可以返回一个特殊的 JSON 结构,里面带一个 continue 为真的字段,这样就能强制 Claude 继续往下做,而不是停在那里等你说话。这个机制特别适合那种多步骤的工作流,比如说你要求 Claude 把一个清单上所有事项都做完才算完成,或者你要求它必须先确认代码能编译通过才能停下来。

跟 Stop 配套的还有一个 SubagentStop,这个是专门针对子代理的。如果你用过 Claude Code 里那种通过 Task 工具派生出来的子任务,你就知道有时候主任务会分裂出几个子代理去并行处理不同的工作。SubagentStop 这个 hook,触发的时机就是某个子代理完成了它自己的工作,而不是主代理完成。它的配置结构跟 Stop 几乎一样,文章举的例子也是用 prompt 类型的 hook,去评估这个子代理交出来的结果是不是达标,达标就接受,不达标就打回去要求修正。这种机制对于做复杂多代理协作的场景特别有用,相当于给每个子任务都加上了一层质量检查关卡。

第六个是 UserPromptSubmit,这个触发时机是你刚刚提交了一个 prompt,但 Claude 还没开始处理。这个 hook 输出的内容,会跟你的 prompt 一起被塞进上下文里。文章的例子是,每次你提交问题,脚本就自动读取一个记录当前迭代背景的文件内容,这样 Claude 永远都能看到你团队目前的优先事项,而不需要你每次都重复一遍。这个 hook 也可以用来在内容到达 Claude 之前做一次校验,或者根据内容把某些类型的请求直接拦掉,还可以用来附加一些动态信息,比如最近的错误日志或者测试结果。

最后一个是 PreCompact,这个稍微小众一点,但思路很巧妙。Claude Code 的对话上下文是有限的,当对话变长之后,系统会自动做一次压缩,把比较早的对话内容总结精简,这个过程里有一些细节信息难免会丢失。PreCompact 这个 hook,就是在这次压缩真正发生之前触发,给你一个机会,先把重要信息备份下来。文章的例子是,在自动压缩之前,先跑一个脚本把完整的对话记录备份成一个文件。matcher 还可以区分是系统自动触发的压缩,还是你手动触发的压缩,这样你可以针对不同情况采取不同的备份策略。

讲完了八种 hook,我们再聊聊怎么把它们真正配置起来,这部分是偏工程实践的内容,但理解清楚之后你才能真正上手。

hooks 的配置存放在 JSON 格式的设置文件里,一共分三个层级。第一层是项目级别的,放在你代码仓库里的点 claude 目录下的 settings.json 文件,这种配置是可以提交到版本控制里跟团队共享的,所以特别适合用来定义团队统一遵守的规则,比如统一的代码格式化流程。第二层是用户级别的,放在你用户主目录下的点 claude 文件夹里,这种配置会应用到你所有的项目里,适合放一些你个人习惯性的偏好。第三层是项目本地的配置,放在 settings 点 local 点 json 这个文件里,这个是你个人专属、不打算提交到仓库里共享的配置。优先级上,项目级别的设置会覆盖用户级别的设置,另外企业版还提供了一种组织统一管控的策略配置,这个适合大公司想要在所有员工的 Claude Code 上统一执行一些安全策略的场景。

这里有个小技巧值得一提,就是这个设置文件,不只能配 hooks,还能配置权限管理,你可以在项目、用户或者本地这三个层级上,精确地控制 Claude 能做什么不能做什么。比如说,你可以直接允许 Claude 读取某个目录下所有的文件,这样它就不用每次读文件都来问你要不要批准,也可以反过来,直接禁止它对某些敏感文件做任何修改。

接下来说 matcher,也就是匹配器,它是用来过滤哪些工具调用会触发你这个 hook 的关键机制。它只对 PreToolUse、PostToolUse 和 PermissionRequest 这三种 hook 生效。最简单的写法就是直接写工具的名字,比如写 Write,就只匹配写文件这个工具。如果你想同时匹配多个工具,可以用竖线符号把它们连起来,比如 Write 竖线 Edit,这样写文件和编辑文件这两个工具都会触发这个 hook。如果你想匹配所有工具,可以用一个星号,或者直接留空也行。这里有个容易踩的坑,matcher 是大小写敏感的,如果你把 Bash 写成小写的 bash,是匹配不上的,这种细节问题排查起来特别花时间,大家一定要注意。除了简单匹配,你还可以写更精细的参数模式,比如写成 Bash 加括号里面写 npm test 加星号,这样就能只针对带有特定参数的命令做匹配,如果你用的是 MCP 协议接入的外部工具,也有专门的匹配格式可以用。

再说说每个 hook 接收和返回的数据格式。所有的 hook 都会通过标准输入接收一段 JSON 数据,里面包含了这次会话的基本信息,比如会话编号、对话记录文件的路径、当前工作目录、权限模式,以及触发这次 hook 的具体事件名字。如果是跟工具调用相关的 hook,还会额外拿到工具名字和工具的输入参数,这些信息就是你的脚本用来做判断的依据。

至于 hook 执行完之后怎么把结果传回去,主要靠退出码。退出码是零,代表成功,这时候标准输出的内容,如果是 JSON 格式就会被系统解析处理,如果不是,就直接被加进上下文里。退出码是二,代表这是一个阻断性的错误,这时候标准错误输出的内容会被当成错误信息显示出来,而且这个动作会被真正阻止执行,不会继续往下跑。除了这两个之外,还有别的退出码代表非阻断性的错误,这种错误信息只在你开启详细模式的时候才会显示出来。

除了简单的退出码,hook 还可以返回更复杂的结构化 JSON,来实现更精细的控制。比如里面可以带一个叫 decision 的字段,值可以是批准、阻止、允许或者拒绝。还可以带一个 reason 字段,用来给 Claude 解释为什么做出这个决定。如果是 Stop 类型的 hook,还可以带一个 continue 字段,强制 Claude 继续工作而不是停下来。甚至还可以带一个叫 updatedInput 的字段,在工具真正执行之前,直接修改它的参数。

hooks 在运行的时候,也能访问一些环境变量,比如有一个叫 CLAUDE_PROJECT_DIR 的变量,代表项目根目录的路径,还有一个叫 CLAUDE_CODE_REMOTE 的变量,如果你是在网页环境里运行,这个值会是真。另外针对 SessionStart 这种 hook,还有一个专门的环境变量文件,可以用来把变量持久化保存下来,供后续使用。当然你平时 shell 里那些标准的环境变量也是能正常访问的。

还有几个细节文章里特别提到了。一个是 hooks 默认有六十秒的超时限制,不过这个可以针对每个 hook 单独配置。另一个是,如果同一个事件触发的时候同时匹配上了多个 hook,这些 hook 会并行执行,而不是排队一个一个跑。还有一点是,如果配置出现了完全相同的命令,系统会自动帮你去重,避免重复执行浪费资源。

讲到这里,我必须强调一下安全方面的考虑,这部分文章写得很认真,我们也要认真听。hooks 执行的是真正的 shell 命令,而且是用你自己用户的权限去执行的,这意味着如果配置不当,潜在的风险并不小。Claude Code 本身有一道安全防线,就是如果你想直接修改 hook 的配置文件,这个改动必须先在一个叫斜杠 hooks 的菜单里经过你的审核确认才能真正生效,这是为了防止恶意代码偷偷往你的配置里加东西。但是话说回来,只要你自己配置并且批准了某个 hook,它就会按照你设定的权限级别去运行,这部分责任还是落在你自己身上。文章给了几条很实用的安全建议,第一是一定要校验和清洗从标准输入拿到的数据,不要无条件信任;第二是在 shell 脚本里给变量加引号,防止命令注入这种经典漏洞;第三是脚本路径尽量用绝对路径,避免因为工作目录变化导致脚本跑错地方;第四是不要让 hook 去处理像点 env 文件或者凭证文件这种敏感信息。我个人觉得这几条建议非常值得记下来,因为很多安全事故,往往就是出在这种看起来很小、很顺手的自动化脚本里,大家在写 hook 脚本的时候,最好把它当成一段会被实际执行的生产代码来对待,而不是随手写的小玩意儿。

那如果配置出了问题,怎么排查呢,文章也专门讲了调试的方法。Claude Code 会把所有东西都记录到一个叫 transcript 的文件里,这个文件能让你看到完整的工具调用和回复过程,而且不需要你额外做任何配置。每个 hook 接收到的数据里,都带一个 transcript_path 字段,指向一个 JSONL 格式的文件,里面是完整的会话历史。你可以配置一个 SessionStart 类型的 hook,把每次会话对应的 transcript 文件路径,记录到一个日志文件里,这样以后想回看某次会话,直接能找到对应的记录文件。然后你可以用 tail 命令配合 jq 这种 JSON 处理工具,实时跟踪这个文件的内容,等于是在旁边看着 Claude 一步一步在做什么。

不过这种 transcript 只能告诉你 Claude 做了什么,不能告诉你你写的那个 hook 脚本,为什么做出了批准或者拦截的决定。所以如果你想针对某个 hook 单独调试,文章建议你自己在脚本里加日志记录。它给了一个具体的例子,写一个叫 log-wrapper 点 sh 的小包装脚本,这个脚本先把标准输入的内容读进一个变量里,用 jq 解析出工具名字和事件名字,记录下时间戳,然后再把原始输入转发给你真正想跑的那个工具命令,等那个命令跑完之后,再把它的退出码也记下来。写好这个包装脚本之后,你只需要在 hook 配置里,把原本要跑的命令,改成先调用这个包装脚本,再把真正的命令作为参数传进去就行了。这样你每次 hook 触发,都会在日志文件里留下一条记录,方便你回头排查到底是哪一步出了问题。

聊了这么多技术细节,我们回到一个更实际的问题,就是普通听众,作为日常使用 Claude Code 写代码的工程师,应该怎么上手这套机制。文章给的建议非常朴实,就是先别贪多,从一个真正能解决你日常痛点的简单 hook 开始配置,而不是一上来就想把八种 hook 全部用上。文章特别推荐用 PostToolUse 配合格式化工具作为第一个尝试的对象,原因就是这个反馈特别直接,你配置完一保存,马上写一个文件试试,看到代码自动变整齐了,这种即时的成功反馈,会让你对这套机制建立信心,也能让你熟悉整个配置流程,包括怎么写 matcher,怎么处理退出码。等你对这一个 hook 用熟练了,再根据自己实际遇到的摩擦点,逐步往外扩展,比如说接下来配一个 PermissionRequest 去自动批准你重复执行很多次的安全命令,或者配一个 SessionStart 去自动注入项目背景信息。

从我个人的角度来看,这篇文章最打动我的地方,不是某一个具体的 hook 类型多么强大,而是它背后体现出的一种产品理念,就是把一个原本只能靠你手动维护的工作习惯,转变成了写进配置文件里、每次都生效的自动化规则。这种转变的价值在于,它把可靠性从依赖人的记忆力,变成了依赖系统的执行力。你不会因为某天太累忘了跑格式化工具,也不会因为某次太着急没注意检查危险命令,因为这些事情已经不需要你去记了,它们写在配置里,每次都会自动跑。这种思路其实不只适用于 Claude Code,放大到我们平时做工程实践,凡是那种重复性强、规则明确、容易因为人为疏忽出错的环节,都值得考虑做成类似的自动化钩子。

好,这篇关于 Claude Code hooks 的文章我们就聊到这里。简单总结一下,这篇文章系统介绍了 Claude Code 里八种 hook 的触发时机和典型用法,从拦截危险操作的 PreToolUse,到自动批准重复操作的 PermissionRequest,到完成格式化收尾的 PostToolUse,再到注入上下文的 SessionStart 和 UserPromptSubmit,以及控制任务是否真正完成的 Stop 和 SubagentStop,还有保护对话历史的 PreCompact,基本覆盖了一次 Claude Code 会话从开始到结束的完整生命周期。如果你也经常在用 Claude Code 写代码,而且已经感觉到一些重复操作开始让你厌烦了,我建议你这周末就找一个最简单的痛点试着配一个 hook,体验一下这种从手动到自动的转变,我相信你会和我一样觉得这笔投入很值。今天的节目就到这里,我们下期再聊,再见。
