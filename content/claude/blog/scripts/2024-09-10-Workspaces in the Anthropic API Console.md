大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是《Workspaces in the Anthropic API Console》,翻译过来就是"Anthropic API 控制台里的工作区功能"。这篇文章发表于二〇二四年九月十日,距离今天,也就是二〇二六年六月二十一日,已经过去快两年了。

先说一下时效性的问题。两年前的九月,是个什么样的时间点呢。那时候 Claude 三点五系列刚刚发布不久,整个行业还处在"大模型能力快速提升、企业开始认真考虑把 Claude 用进生产环境"的阶段。换句话说,二〇二四年那会儿,很多公司刚刚从"玩一玩 API"过渡到"真的要把它当成生产基础设施来用"。这个阶段最大的痛点,往往不是模型能力不够,而是工程管理跟不上,你的开发、测试、生产三套环境共用一个账号,一个 API Key,谁也说不清楚这个月的账单里,到底是测试环境跑飞了,还是生产环境真的在烧钱。Workspaces 这个功能,就是在那个时间点,为了解决这类"管理混乱"的问题而推出的。

虽然这是个产品功能介绍,信息量不像论文那样密集,但我觉得它依然值得拿出来讲,原因很简单,这类基础设施型的功能,往往是被开发者忽略,却又天天在用的东西。如果你现在还在用一个 API Key 走天下,那么听完这期节目,你大概会有种"原来还能这样管理"的感觉。好,我们正式开始。

我们先讲讲背景,也就是这篇文章要解决的问题是什么。在 Workspaces 上线之前,Anthropic 的开发者控制台,基本的组织单位只有两层,最上面是你的组织,Organization,下面直接挂着一堆 API Key。如果你的团队只是一个人,或者一个很小的项目,这种结构完全没问题,你创建一个 Key,写进代码里,调用 Claude,完事。但是一旦你的团队稍微大一点,情况就开始变得棘手。

我们设想一个最常见的场景。你在一家做 SaaS 产品的公司工作,产品里集成了 Claude 来做客服助手。你们公司有开发环境、预发布环境也就是 staging、还有正式的生产环境。三个环境,理论上都需要调用 Claude 的 API。如果大家共用一个组织下面的 Key,那么会出现什么问题呢。第一,你没法区分清楚每个月的账单里,到底是开发环境的工程师在调试的时候烧的钱,还是生产环境真实用户在用的钱。第二,如果开发环境的某个测试脚本写错了,陷入死循环疯狂调用 API,它消耗的速率限额和生产环境是共享的,直接会影响到线上用户的真实体验,这是非常致命的。第三,从安全角度讲,如果开发环境的 Key 泄露了,因为它和生产环境共用同一个账号体系,影响范围可能直接波及到正式产品。

这种问题,其实在云计算领域早就有人解决过。你如果用过亚马逊的 AWS,大概知道它有一个叫 AWS Organizations 的功能,可以把不同的账号、不同的项目分门别类地管理起来,设置各自的预算和权限。Anthropic 在二〇二四年九月推出的 Workspaces,本质上就是把这套"分组管理"的思路,搬到了 Claude 的 API 控制台里。

那我们具体来看看,Workspaces 到底是什么。用文章里的话讲,Workspaces 是一种独立的环境,它能帮你组织资源、简化访问控制,并且在更细的颗粒度上设置花费限额和速率限制。说得直白一点,你可以把 Workspaces 理解成给你的 Anthropic 账号装上了一套"文件夹系统",每个文件夹,也就是每个 Workspace,都是一个相对独立的小世界,里面有自己的 API Key,有自己的预算上限,有自己的速率限制,还有自己的权限设置,谁能看、谁能改,都是按文件夹来配的。

这里我想做一个类比,帮大家建立直觉。你可以把整个 Anthropic 组织想象成一栋大楼,这栋大楼以前是开放式办公室,所有人共用一套门禁卡,共用一个总电费账户,谁开了空调、谁用了多少电,根本分不清楚。Workspaces 上线之后,这栋大楼被重新装修成了一个个独立的办公室,每个办公室有自己的门禁,自己的电表,自己的预算上限。研发部的办公室和市场部的办公室,物理上还在同一栋楼里,但管理上彼此独立,出了问题也能精准定位是哪个办公室的事。这就是 Workspaces 想要达成的效果。

接下来,我们逐个拆解文章里提到的五大能力,这也是整篇文章的核心内容,我会尽量讲透,并且结合实际场景帮大家理解。

第一个能力,是分项目设置花费限额,英文叫 spend limits。这个功能允许你针对每一个 Workspace,单独设置每月的消费上限。这个听起来简单,但它解决的是一个非常现实的财务管理问题。我们还是拿刚才那个三环境的例子来说,如果你给开发环境的 Workspace 设置一个每月五十美元的上限,给预发布环境设置两百美元的上限,给生产环境根据实际业务量设置一个比较高的上限,比如五千美元,那么即便开发环境的某个工程师手滑写了个死循环脚本,最多也只会烧掉五十美元就被挡住,不会拖累整个组织的账单。这其实就是一种"故障隔离"的思路,跟我们做系统架构的时候讲的"舱壁模式"是一个道理,一个舱室漏水,不会让整艘船沉没。

第二个能力,是资源分组,英文叫 grouping related resources。这个能力听起来比较抽象,但其实很好理解。它的意思是,你可以把 API Key、用量数据、各种配置,按照项目或者环境,归类放进同一个 Workspace 里。打个比方,如果你的公司同时在做三个产品,一个是智能客服,一个是内部知识库问答,一个是面向开发者的代码助手,那么你完全可以给这三个产品各开一个 Workspace,每个 Workspace 里装着这个产品专属的 Key、专属的配置、专属的使用记录。这样做的好处是,当老板问你"这个月我们在 Claude 上花了多少钱,主要花在哪个产品上"的时候,你不需要去翻一堆杂乱无章的日志,直接打开控制台,一个 Workspace 一个 Workspace 地看,数据天然就是分好类的。

第三个能力,是速率限制管理,英文叫 manage rate limits。这一点我觉得是技术上最容易被低估,但实际影响最大的一个功能。我们先解释一下什么是速率限制。简单说,Anthropic 为了保证整个系统的稳定性,会限制你的账号在单位时间内能发起多少次请求,或者能处理多少个 token。这个限制是有上限的,而且通常是按整个组织来算的总账。如果你的组织下面同时跑着好几个项目,大家共用这个总的速率限额,那就很容易出现"抢资源"的情况。比如说,生产环境本来应该优先保障,但是凌晨某个数据团队跑了个批量处理任务,疯狂调用 API,把整个组织的速率限额占满了,结果生产环境的真实用户请求反而被限流,响应变慢甚至直接报错。这就是典型的"内耗"。

Workspaces 允许你在每个 Workspace 内部单独设置速率限制,同时这些限制依然要在整个组织的总限额之内。这就好比说,大楼的总电闸容量是固定的,但每个办公室可以单独配一个分电闸,设定自己能用多大功率,这样即便某个办公室开足马力用电,也不会让别的办公室突然跳闸断电。对于工程团队来说,这意味着你终于可以放心地把批处理任务、离线分析任务,和实时的线上服务分开管理,互不干扰。

第四个能力,是访问控制,英文叫 streamline access control。这个功能的核心,是把用户权限的管理颗粒度,从"组织级别"细化到了"Workspace 级别"。在没有 Workspaces 之前,你给一个人组织权限,他基本上就能看到、动到整个组织下面的所有资源。这在团队变大之后是个很大的安全隐患。比如说,你请了一个外部顾问,只需要他帮你调试开发环境的某个功能,但是因为权限粒度太粗,你不得不把整个组织的访问权限都给他,生产环境的 Key、所有的账单数据,他理论上都能看到。这显然是不合理的。

Workspaces 上线之后,你可以单独给这个顾问开一个开发环境的 Workspace 权限,他只能看到、只能动这一个文件夹里的资源,生产环境对他来说完全不可见。这其实是信息安全里一个很基础但很重要的原则,叫"最小权限原则",意思是每个人拥有的权限,应该刚好够他完成工作,不多不少。Workspaces 把这个原则,真正落地到了 Anthropic 的产品里。

第五个能力,是用量监控,英文叫 monitor usage efficiently。这个功能其实是前面四个能力的自然延伸。因为你的资源已经按 Workspace 分好类了,那么控制台自然就可以按 Workspace 维度,把用量数据和成本数据拆开展示给你看。这对于做财务分析、做成本优化的同学来说,价值是非常直接的。你不再需要自己写脚本去解析账单、按项目手动分类汇总,控制台直接给你一张"分项报表"。

好,五个能力我们都讲完了。我们退一步,从整体上看看 Workspaces 这个设计背后的思路。我个人觉得,这个功能本质上不是一个"很酷的新技术",它更像是一种工程管理上的成熟标志。一个产品从"能跑起来"到"能被很多团队稳定地、可控地长期使用",中间一定要补上这种资源隔离、权限隔离、成本隔离的能力。这跟我们写代码的道理是一样的,一个脚本能跑通是一个阶段,但是把这个脚本变成一个能被多人协作维护、出了问题能快速定位的系统,又是另一个阶段。Anthropic 在二〇二四年九月推出 Workspaces,某种程度上标志着 Claude API 已经从"实验性的新奇玩具",走向了"企业级生产工具"这个阶段。

接下来,我们聊聊实践应用,也就是作为听众,你应该怎么把这个功能用起来,以及有哪些坑需要注意。

首先,如果你现在的团队,还是用一个 API Key 走天下,我建议你先别急着大动干戈地重构,而是先问自己几个问题。第一,你的团队有没有明确区分开发、测试、生产这几个环境。如果有,这几个环境的调用量、花费,你现在能不能说得清楚分别是多少。如果说不清楚,那 Workspaces 几乎是你的第一选择,你应该尽快按环境拆分出对应的 Workspace,把开发环境的 Key 和生产环境的 Key 物理上隔离开,这样无论是出于安全考虑,还是出于成本控制的考虑,都是立竿见影的改善。

第二个问题,你的公司是不是同时在做多个产品,或者多个客户项目,都用到了 Claude 的 API。如果是这种情况,我建议你按"业务线"来划分 Workspace,而不是按"环境"来划分。换句话说,如果你只有一个产品,但分了开发测试生产三个环境,那就按环境分三个 Workspace。但如果你有三个不同的产品,每个产品又各自有自己的环境,那么比较合理的做法,是先按产品分出三个大类,如果预算和团队规模允许,再在每个产品内部进一步细分环境。这里有个原则我想强调一下,就是 Workspace 的划分方式,应该跟着你"想看到的报表维度"走。你最希望在财务报表上一眼看出来的分类方式,就应该是你的 Workspace 划分方式。如果你希望按客户来计费,那就按客户分;如果你希望按团队来核算成本,那就按团队分。

第三,关于速率限制的设置,这里有个常见的坑,我提醒大家注意。文章里提到,各个 Workspace 的速率限制,是要在组织总限额之内分配的,这意味着这是一个零和博弈,你给生产环境多分一点,其他 Workspace 能用的份额就会变少。所以在设置之前,你最好先了解清楚你组织当前的总速率限额是多少,再去合理分配,而不是想着每个 Workspace 都按"理想状态"配置满,结果加起来超出了组织上限,导致配置根本生效不了,或者互相冲突。建议的做法是,先保障生产环境的速率限额,确保线上用户体验不受影响,然后再把剩余的额度,按优先级分配给测试、开发,以及批处理任务这些非实时场景。

第四个建议,是关于权限管理的。如果你的团队里有外部协作者,比如外包的工程师、咨询顾问,或者只是临时帮忙的同事,一定要养成"按 Workspace 授权"的习惯,而不是图省事直接给组织级别的权限。这个习惯一旦养成,即便未来团队规模继续扩大,你的权限体系也不会乱成一锅粥。我见过不少团队,初期人少的时候图方便,把所有人都设成管理员,等团队扩大到几十人之后才发现权限管理一团乱麻,这时候再去梳理,成本会比一开始就规范好高得多。

第五个建议,跟成本监控有关。既然 Workspaces 已经把账单按维度拆好了,那么我建议大家定期,比如每周或者每月,去控制台看一眼各个 Workspace 的花费趋势,而不是等到月底账单出来才被吓一跳。这种习惯其实跟我们做云服务成本管理时讲的 FinOps 理念是一致的,持续地、主动地去看资源消耗情况,比事后救火要有效得多。

那么,有没有团队是不太需要 Workspaces 的呢。我觉得有。如果你只是个人开发者,或者团队特别小,只有一两个人,做的也只是一个简单的小项目,那么引入 Workspaces 反而可能增加管理上的认知负担,这种情况下,一个 Key 走天下,简单直接,完全没问题。基础设施型的功能,价值是随着规模增长才显现出来的,规模小的时候,过度设计反而是一种负担,这也是我想提醒大家的,不要为了用一个新功能而用,要看自己是不是真的有这个管理复杂度的问题。

另外我想补充一点,这篇文章本身篇幅很短,信息密度不算高,它更像是一份产品发布公告,而不是深入的技术解析。所以我们今天聊的很多类比和场景推演,是我结合自己对工程团队管理的理解做的延伸,这些是我个人的看法,跟文章原文的事实部分要分开看。原文明确给出的,就是这五个能力点,以及"现在所有 Anthropic API 用户都可以在控制台里创建 Workspace"这个事实。

好,我们来做个总结。今天聊的这篇文章,讲的是二〇二四年九月,Anthropic 在 API 控制台里推出的 Workspaces 功能。它的核心价值,是把过去"一个组织一套资源"的扁平结构,变成了可以按项目、按环境、按团队灵活分组的层级结构,具体体现在五个方面,分项目设置花费限额、资源分组管理、独立的速率限制、细颗粒度的权限控制,以及按维度拆分的用量监控。这个功能两年前看可能只是个"锦上添花"的管理工具,但放到今天,如果你的团队规模已经起来了,Claude 在你的产品里跑的场景也越来越多,那这套"文件夹式"的管理体系,基本上是绕不开的标配。

我个人的评价是,这类功能往往不性感,不会像新模型发布那样引发热议,但它恰恰是决定一个产品能不能在企业里真正落地、长期稳定运行的关键基础设施。如果你现在团队里还在用最原始的方式管理 API Key 和账单,不妨今天就去控制台看一眼,花十分钟时间,把你的开发、测试、生产环境拆成对应的 Workspace,这是一个投入产出比很高的小动作。
