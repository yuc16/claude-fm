## Script

大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是 Skills for organizations, partners, the ecosystem,翻译过来就是"面向组织、合作伙伴和整个生态的 Skills"。这篇文章发表于二〇二五年十二月十八日,是 Anthropic 官方博客上的一篇产品更新公告。算到今天,也就是二〇二六年六月二十日,已经过去差不多半年了。所以在正式开始之前,我想先提醒大家一句,这篇文章里说的"现在"、"今天",指的是去年十二月的状态,半年时间在 AI 这个行业里其实是不短的,很多功能可能已经迭代了好几个版本,目录里的合作伙伴也可能更多了。但是这篇文章里讲的几个核心思路,组织级管理、技能目录、开放标准,这些方向性的东西到现在应该还是成立的,甚至可能已经变成了行业的标准做法。我们今天的重点,是把这几个思路讲透,顺便聊聊半年之后回头看,这个判断站不站得住脚。

那我们先从背景说起。Skills 这个概念是 Anthropic 在二〇二五年十月份正式推出的,简单说,它是一种教 Claude 学会"可重复工作流程"的方式。这里我先解释一下什么叫"可重复工作流程"。你想象一下,你在工作里经常要做一件事,比如说,每周整理一份周报,格式固定,要包含哪些字段、用什么语气、最后导出成什么样式,这些规则你心里很清楚,但是每次跟 AI 助手解释一遍很麻烦,而且每次解释的细节可能还不完全一样。Skills 解决的就是这个问题,你把这套规则、步骤、模板,写成一个文件夹,里面可能有说明文档,也可能有脚本、模板文件,然后 Claude 在需要的时候会自动识别出"这个任务该用这个 Skill",照着里面写好的流程去执行。这跟我们之前聊过的 MCP,也就是 Model Context Protocol,模型上下文协议,不是一个东西,但是思路上有点像。MCP 解决的是 Claude 怎么连接外部工具和数据源的问题,而 Skills 解决的是 Claude 怎么学会一套具体的、人类总结好的操作方法论。打个比较通俗的比方,MCP 像是给 Claude 配了一把万能钥匙,可以打开各种门;Skills 更像是给 Claude 发了一本操作手册,告诉它"遇到这种情况,按这几步走"。

去年十月份刚推出的时候,Skills 更多是面向个人用户和开发者的,你自己写一个 Skill,自己用,或者分享给团队里的同事手动安装。这种模式在个人或者小团队场景下没什么问题,但是放到一个几百人甚至几千人的企业里,马上就会暴露出几个问题。第一个问题是一致性,如果每个人都自己写自己的 Skill,那么同一类任务,市场部的人写的版本和财务部的人写的版本可能完全不一样,公司想要的"统一的、被审核过的工作流程"就没法保证。第二个问题是发现成本,一个新员工进了公司,他怎么知道公司里已经有哪些现成的 Skill 可以用,而不需要自己从零摸索。第三个问题更宏观一点,就是 Skills 这个东西如果只能在 Claude 里用,换一个 AI 工具就用不了,那它的价值就被锁死在一个平台里,这对整个行业的人来说都不是好事。这篇文章,本质上就是 Anthropic 针对这三个问题,分别给出的解决方案。我们一个一个来拆。

第一个核心内容,是组织级的统一管理。文章里说,Claude Team 计划和 Enterprise 计划,也就是团队版和企业版的管理员,现在可以在管理员后台,集中地给整个组织配置 Skills。这里有个关键细节,管理员配置好的 Skill,默认是对组织内所有用户生效的,也就是说不需要每个员工自己手动去开启。但同时,文章也强调了用户仍然保留了一个开关,如果某个员工觉得这个 Skill 不适合自己的工作场景,他可以单独把它关掉。我觉得这个设计挺值得说一说的,它其实是在两种治理哲学之间找了一个平衡点。一种极端是完全集中管控,所有人必须用统一的工具和流程,好处是合规和一致性有保障,坏处是灵活性差,容易引起员工的抵触情绪。另一种极端是完全放权,人人自由安装,好处是灵活,坏处是公司层面完全失去了对工作流程的可见性和控制力。Anthropic 选的这个"默认开启、个人可关闭"的模式,本质上是默认信任组织的判断,但是保留了个体的否决权。这种模式其实在企业软件里很常见,比如说很多公司的安全策略软件、办公套件的默认配置,都是这个思路,组织先把一个"大家都应该用的版本"铺好,个人有特殊需求再自己调整,而不是反过来让每个人从零开始配置。

如果我们站在一个企业 IT 负责人的角度来想这件事,这个功能其实价值挺大的。假设你是一家做跨境电商的公司的技术负责人,你的客服团队、运营团队、财务团队,每天都要跟 Claude 打交道,处理退换货话术、广告文案、报表整理这些事情。在没有组织级管理之前,你只能寄希望于每个团队自己去摸索、自己去写 Skill,质量参差不齐,而且出了问题你很难追溯是哪个版本的流程导致的。有了组织级管理之后,你可以让一个核心团队,把"标准的客服应答 Skill"、"标准的财报整理 Skill"这些东西先做好、审核好,然后通过管理员后台一次性推给所有相关的员工。这就跟企业里统一发软件镜像、统一推送安全补丁是一个道理,降低的是治理成本,提升的是一致性。

第二个核心内容,是创建和编辑 Skill 的门槛被大幅降低了。文章里提到,现在你可以直接用自然语言描述你想要的工作流程,Claude 会帮你把它构建成一个 Skill,你也可以选择直接手写指令文件。对于比较复杂的工作流程,你可以上传整个 Skill 文件夹,或者用一个叫 skill-creator 的工具来辅助创建。另外,Claude 现在还可以帮你编辑已经存在的 Skill,而且界面上新增了预览功能,可以让你在启用一个 Skill 之前,完整看到它里面到底写了什么内容,确保你清楚自己在启用什么东西。

这一点我觉得特别值得展开讲,因为它其实触及了一个很现实的矛盾。Skills 这套机制的价值,建立在"有大量高质量的、被良好维护的 Skill 存在"这个前提之上,但是写一个好的 Skill,本身也是有门槛的,你得清楚怎么组织指令、怎么写说明文档、什么时候该放脚本、什么时候该放模板文件。如果这个门槛太高,大部分人宁愿继续用最原始的方式,每次都临时跟 AI 解释一遍需求,Skills 这套体系就推不开。所以 Anthropic 在这里做的事情,本质上是用 Claude 自己的能力,去降低创建 Claude Skill 的门槛,这是一个挺有意思的自循环。你可以把它理解成,以前你想给团队写一份标准作业流程文档,得自己一字一句地写,现在你只需要把你脑子里想要的流程口头描述一遍,Claude 帮你整理成一份结构化的、机器可以执行的文档。这个降门槛的动作,对于把 Skills 从"少数极客在用的工具"变成"普通员工也能上手的能力",是非常关键的一步。

另外预览功能这个细节也值得多说两句。我个人认为,这其实是个安全和信任层面的考量。你想,如果一个 Skill 是别人写的,或者是从一个目录里下载下来的,你启用它之前,如果完全是个黑盒,你只能凭着它的名字和简介去猜它到底会怎么操作你的数据、调用哪些工具,这显然是有风险的。尤其是 Skill 这种东西,往往是带有脚本和具体执行步骤的,跟一段普通的提示词不太一样,它的"行为半径"可能更大。所以能够在启用前看到完整内容,这跟我们平时在应用商店里看一个插件需要哪些权限,其实是类似的逻辑,都是在用透明度去换取用户的信任。这一点对企业用户来说尤其重要,因为企业要考虑的不只是这个 Skill 好不好用,还要考虑它会不会触碰到敏感数据,会不会有不符合公司合规要求的操作。

第三个核心内容,是技能目录的上线。文章里说,在 claude 点 com 斜杠 connectors 这个地址,现在有一个不断扩充的、由合作伙伴构建的 Skill 集合,包括 Notion、Canva、Figma、Atlassian 这些大家比较熟悉的产品。管理员可以把这些合作伙伴的 Skill,直接在整个组织范围内统一开启,这样团队成员就能立刻用上跟他们日常工具相关的工作流程,而不需要任何自定义开发。

这个目录的意义,我觉得可以从两个角度来看。第一个角度是用户角度,以前如果你想让 Claude 跟 Notion 或者 Figma 配合,得自己摸索怎么连接、怎么写提示词才能让效果稳定,现在直接有官方认证的、合作伙伴自己打磨过的 Skill 可以用,质量和稳定性肯定比自己瞎试要好。比如说,如果你公司里设计团队天天用 Figma,产品团队天天用 Notion 记需求文档,有了对应的 Skill 之后,Claude 可能就能直接帮你把 Figma 里的设计稿信息整理成 Notion 里的任务条目,中间不需要人工搬运。这种"打通不同工具之间数据流转"的能力,以前往往要靠定制化的自动化脚本,或者一些专门的集成平台来实现,现在变成了一个可以直接启用的 Skill。

第二个角度是合作伙伴角度,对于 Notion、Canva、Figma、Atlassian 这些公司来说,把自己的产品能力包装成一个 Claude Skill,放到这个目录里,相当于多了一个分发渠道,可以触达到所有用 Claude 的企业客户。这跟应用商店生态是一个逻辑,一个手机操作系统好不好,很大程度上取决于上面有没有足够多、足够好的第三方应用。Claude 想要在企业场景里站稳脚跟,光靠自己一家公司是不够的,需要让整个生态围着它转起来,技能目录就是这个生态化战略的一个具体落地动作。

第四个核心内容,也是我觉得这篇文章里分量最重的一点,是 Anthropic 把 Agent Skills 作为一个开放标准发布出来了。文章里明确写了,他们认为 Skills 应该像 MCP 一样,在不同的工具和平台之间是可移植的,同一个 Skill,不管你用的是 Claude 还是别的 AI 平台,都应该能跑起来。文章还提到他们已经在跟生态里的一些伙伴合作,并且已经看到了这个标准的早期采用案例。

这一点为什么重要,我想稍微展开讲一讲背后的逻辑。我们回顾一下 MCP 这个协议的故事,MCP 刚推出的时候,也是 Anthropic 自己先做的,但是它没有把这个协议关起来只让 Claude 用,而是开放出来,让其他 AI 公司、其他开发者都可以基于这个协议去构建工具。结果是什么呢,MCP 慢慢变成了整个行业里 AI 模型连接外部工具的一个事实标准,不只是 Claude 在用,很多其他厂商的产品也开始支持 MCP。这种开放策略,短期看好像是把自己的护城河让出去了,但长期看,反而让 Anthropic 在这个标准的制定上拿到了话语权,而且整个生态围绕这个标准建立起来之后,Claude 作为这个标准最早、最成熟的实现者,反而获得了更大的优势。

Agent Skills 现在走的是同一条路。如果 Skills 只是 Claude 独有的东西,那么不管它做得多好,企业在投入资源去写大量 Skill 之前,都会有顾虑,因为这相当于把自己的工作流程资产绑死在一个平台上,万一以后想换别的 AI 工具,这些投入就全部作废了。但是如果 Skills 是一个开放标准,企业写的 Skill 理论上可以在不同的 AI 平台之间迁移,这个顾虑就被大大缓解了。这其实是在解决一个典型的"锁定担忧",通过把标准开放出来,反而让更多企业敢于投入资源去构建自己的 Skill 体系,而这些体系大概率还是会优先在 Claude 上跑得最好,因为 Claude 是这个标准的发明者和最早的实践者。我个人认为,这是一步很聪明的棋,跟当年 MCP 的打法一脉相承,体现了 Anthropic 在生态打法上的一贯思路,就是不追求把所有东西都攥在自己手里,而是追求成为整个生态的标准制定者和最佳实现者。

讲完了这四个核心内容,我们来聊聊这些东西对我们普通听众,具体能带来什么实际的用处,以及上手的时候要注意什么。

文章末尾其实给出了几个具体的入口,我们可以按照不同的使用场景来梳理一下。如果你是通过 Claude 的各种应用,比如网页版或者客户端在用 Claude,你可以直接去技能目录里浏览,然后在设置里找到能力这个分类下面的 Skills 选项去启用。这个路径适合个人用户,或者公司里还没有上线组织级管理的团队成员,自己看看有没有适合自己工作的现成 Skill。

如果你是用 Claude Code 在写代码,文章里说你可以从插件目录里安装 Skill,或者直接把 Skill 文件检入到你的代码仓库里。这个对开发者来说挺重要的,意味着 Skill 可以像代码一样,跟着项目一起进行版本管理,团队成员 clone 下来仓库,这些 Skill 就自动跟着项目走,不需要每个人单独再去配置一遍。我觉得这是一个很自然的设计,因为对于开发团队来说,代码仓库本来就是大家协作和共享规范的中心,把团队约定的工作流程,比如代码评审规范、测试用例生成的方式,直接放进仓库里,跟代码本身一样接受版本控制和评审,这种做法是符合工程师直觉的。

如果你是通过 Claude Developer Platform,也就是 API 来集成 Claude 能力的开发者,文章提到可以通过 slash v1 slash skills 这个接口去调用 Skill,具体细节需要查官方文档。这个对于想把 Skills 这套能力嵌入到自己产品里的团队比较有意义,比如说你自己在做一个企业内部系统,想把"标准化的报表生成流程"作为一个 Skill 接入进去,通过 API 调用,而不是让用户必须跑到 Claude 的客户端里去用。

如果你是企业管理员,文章里专门强调,管理员可以通过后台,在组织范围内统一配置 Skills,但是有一个前提条件,就是 Skills 需要开启 Code Execution,也就是代码执行能力,以及 File Creation,也就是文件创建能力,这两个开关都要打开才能用。这个细节其实挺重要的,提醒我们 Skills 不是一个完全独立于其他能力的功能,它依赖底层的代码执行和文件操作能力。这也从侧面说明了为什么 Skills 需要更谨慎的权限管理,因为它涉及到实际的代码运行和文件生成,这跟单纯的文本对话相比,风险等级是不一样的。如果你是企业的 IT 或者安全负责人,在批准开启这些底层能力之前,值得跟团队一起评估一下,哪些场景下确实需要这种能力,哪些场景下可能不需要,从而做出更精细化的权限分配,而不是一刀切地全开或者全关。

那么具体到我们每个人的日常工作,应该怎么去用这些新功能呢,我想给几个比较实际的建议。第一个建议是,如果你所在的公司用的是 Claude Team 或者 Enterprise,先去了解一下你的 IT 团队或者管理员有没有已经在后台配置好的组织级 Skill,这些往往是经过审核、跟你公司业务场景匹配的,直接用现成的,效率比自己重新摸索要高得多。第二个建议是,如果你发现公司里某个重复性的工作流程,目前还没有对应的 Skill,可以考虑自己动手写一个,现在创建的门槛已经降低了很多,用自然语言描述清楚你想要的流程,Claude 就能帮你搭建出一个初步的版本,你再根据实际效果去调整。这种"自己先用起来,验证有效之后再推荐给团队"的路径,我觉得是个挺务实的做法,可以避免一开始就追求"完美的、面向所有人的 Skill",而是从自己最痛的那个具体场景切入。

第三个建议,是在启用任何来自目录或者别人分享的 Skill 之前,养成先看一下预览内容的习惯,搞清楚这个 Skill 具体会执行什么操作,会不会涉及到你不希望被自动处理的敏感数据,尤其是涉及到代码执行和文件创建的 Skill,审慎一点总是没错的。第四个建议,是如果你本身是在做跨平台的 AI 应用开发,值得关注一下 Agent Skills 这个开放标准的进展,因为半年过去了,这个标准很可能已经有了更多的生态伙伴和更成熟的工具链支持,提前了解清楚标准的具体细节,可以帮你判断要不要把自己的工作流程投资在这个方向上。

我们也得说说这件事可能存在的一些坑或者需要警惕的地方。第一个是治理上的坑,组织级默认开启这个设计虽然方便,但是如果管理员配置的 Skill 本身质量不高,或者跟某些岗位的实际需求不匹配,反而可能造成困扰,所以企业在推行组织级 Skill 之前,最好先经过小范围试点,收集反馈,再大范围推开,不要一上来就指望一步到位。第二个是依赖性的坑,如果整个团队的工作流程都高度依赖某几个 Skill,一旦这些 Skill 出现问题,比如底层逻辑有 bug,或者跟某个外部工具的接口发生了变化导致失效,影响面可能会比较大,所以建议关键的 Skill 都应该有人负责定期维护和检查,而不是写完就丢在那里不管了。第三个是数据安全的坑,合作伙伴的 Skill 涉及到跟外部产品的数据交互,比如跟 Notion 或者 Figma 联动,企业在大范围启用之前,有必要跟自己的安全团队确认一下数据流转的边界,搞清楚哪些数据会被读取,哪些数据会被写入,这个事情不能掉以轻心。

好,我们来稍微总结一下今天讲的内容。这篇文章发表于二〇二五年十二月,讲的是 Anthropic 怎么把 Skills 这套去年十月才推出的新能力,从一个偏个人化的小工具,升级成一套真正能在企业里落地的基础设施。核心做的事情有四件,一是给企业管理员提供了组织级的统一管理能力,默认开启同时保留个人关闭的权利;二是大幅降低了创建和编辑 Skill 的门槛,用自然语言描述就能生成,而且加上了预览功能保证透明度;三是上线了一个汇聚了 Notion、Canva、Figma、Atlassian 等合作伙伴的技能目录,让企业可以直接复用现成的高质量工作流程;四是,也是我觉得最有战略意义的一点,把 Agent Skills 发布成了一个开放标准,延续了 MCP 当年开放生态的打法,试图让 Skills 成为整个行业的通用基础设施,而不是 Claude 独占的功能。

我个人对这件事的看法是,Skills 这套体系如果真的按照文章里描绘的方向发展下去,半年甚至一年之后,很可能会变成企业用 AI 助手的标配能力之一,就像现在大家已经很习惯企业软件里有统一的权限管理后台一样。当然,这篇文章已经是半年前的内容了,具体的目录里现在有多少合作伙伴,开放标准到底有多少平台真正接入了,这些都需要我们结合最新的信息去验证,但是这几个方向性的判断,我觉得到今天依然是站得住的。好,今天的内容就到这里,如果你所在的团队已经在用 Claude Team 或者 Enterprise,不妨去后台看看有没有已经配置好的组织级 Skill,自己动手试一试,我们下期节目再见。
