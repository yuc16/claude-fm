大家好，欢迎收听 Claude FM，我是你们的主播。今天我们要聊的这篇文章，标题是 Making Claude Cowork ready for enterprise，翻译成中文大概是让 Claude Cowork 为企业部署做好准备。这篇文章发表于二〇二六年四月九日，来自 Claude 官方博客。算下来到今天，二〇二六年六月二十日，差不多过去了两个多月，时间不算久，所以文章里讲的功能和产品状态，基本上还是最新的，大家可以放心地把它当成当下的参考。

为什么这篇文章值得花时间听一听呢？因为它讲的不是某个模型能力升级，而是一个更贴近日常工作的话题：当一个像 Claude Cowork 这样的智能体产品，真正要在一家几百人、几千人的公司里大规模铺开使用的时候，会遇到什么问题，Anthropic 又是怎么解决的。如果你在公司里负责 IT、负责安全合规，或者只是好奇 AI 在企业里到底怎么落地，这篇文章会给你一个非常具体的样本。

我们先说说背景。Claude Cowork 这个产品，简单理解，就是 Claude 桌面客户端上的一种能力，它让 Claude 不再只是一个聊天窗口，而是更像一个真正坐在团队里的同事。它可以连接公司里已经在用的各种工具和数据源，帮你处理具体任务，起草项目交付物，让整个团队保持信息同步。这跟大家熟悉的 Claude Code 有点像，区别在于 Claude Code 主要服务工程师，帮你写代码、改代码、跑测试，而 Claude Cowork 服务的是公司里更广泛的岗位，市场、运营、法务、财务，这些不天天写代码的同事。

在这篇文章发表之前，Claude Cowork 已经在不少企业内部小范围试用了一段时间。文章开头提到，它现在在所有付费计划上正式公开可用了，这是一个重要节点，意味着它从早期尝鲜，正式进入了规模化推广阶段。但凡是做过企业软件推广的人都知道，一个工具好不好用是一回事，能不能在几百人的组织里安全地、可控地铺开，完全是另一回事。试想你是一家公司的 IT 负责人，同事告诉你市场部已经开始用一个能自动读写公司文件、自动调用各种工具的智能体处理日常工作了，你的第一反应大概是一连串问题，它能访问哪些数据，谁有权限开通，万一误操作了怎么追溯，会不会因为用得太多导致成本失控。这些问题，正是这篇文章要回答的核心内容，所以它本质上不是在讲 Claude Cowork 多强，而是在讲 Anthropic 怎么给这个产品配上一套企业级的安全带和仪表盘，让管理员能放心把它交到更多员工手里。

文章里第一个值得深挖的点，是 Anthropic 观察到的一个使用模式，他们叫它早期信号。文章说，Claude Code 当年帮工程师完成了一次转变，让工程师从只是向 Claude 提问，变成把整个任务都交给 Claude 去做。现在在整个公司范围内，Claude Cowork 身上正在发生同样的转变，而且数据显示，绝大多数 Cowork 使用量并不是来自工程团队，而是来自工程团队之外的部门。这一点挺反直觉的，我们一般会觉得 AI 工具肯定是技术团队用得最猛，毕竟工程师对新工具接受度高。但实际情况是，运营、市场、财务、法务这些部门，才是 Claude Cowork 真正的重度用户。

更有意思的是文章接下来强调的细节，原文措辞我觉得特别精准，它说这些部门并不是把自己最核心的工作交给 Claude，而是把围绕在核心工作周围的事情交给它，比如项目进度更新、协作用的演示文稿、研究型的冲刺任务。这句话值得多展开讲讲，因为我觉得这是整篇文章最值得琢磨的洞察。什么叫核心工作，什么叫围绕核心工作的工作？拿法务部门来说，核心工作是对一份合同做出法律判断，决定哪些条款有风险，要不要签，这种判断公司目前显然还没放心完全交给 AI 做最终决策。但围绕这份核心判断，周围还有大量辅助性工作，比如把十份相似合同条款做对比表，整理历史上类似条款的处理记录，把法务这周处理了哪些案子写成简报发给管理层。这些事情技术含量没那么高，但占用时间不少，而且挺琐碎，这正是 Claude Cowork 切入的地方。市场部门也是同理，核心工作是想出一个有创意的营销活动，这个创意决策大概率还是人来主导，但围绕这个创意，团队需要做竞品调研，需要把会议记录整理成行动项清单，需要做一份汇报用的幻灯片给老板看进展，这些正是 Cowork 在帮大家分担的部分。

我觉得这个观察对理解 AI 在企业里怎么落地特别重要。很多人对 AI 取代工作有一种线性想象，觉得是先替代简单任务，再逐步替代核心决策。但 Anthropic 观察到的现象更像是另一种形态，核心决策这个内核始终留在人手里，但内核周围一层一层的辅助性、支撑性工作，正在被迅速交给智能体。这层外围工作加起来的工作量可能一点都不小，很多职场人每天大部分时间其实都耗在这层外围工作上，真正坐下来做核心判断的时间反而没那么多。所以如果 Cowork 能把这层外围工作接走，对个人来说意味着能把更多时间花在真正需要专业判断的事情上。

文章还提到一个组织演变的过程，企业早期采用 Cowork 的方式，往往是先在一个团队里试用，用顺手了，团队负责人就会想把它推广到更多团队。但一旦想往外推广，马上会冒出几个棘手问题，谁应该获得使用权限，怎么管理不同团队的花费，怎么从全公司层面看到各个团队到底在用 Cowork 做什么。这几个问题，恰好就是接下来要讲的几项新功能要解决的。

我们先看第一项新功能，叫基于角色的访问控制，英文是 role-based access control，简称 RBAC，只在 Claude 企业版上提供。具体怎么运作呢？管理员现在可以把公司用户组织成不同的组，方式有两种，一种是手动拖人进组，另一种是通过 SCIM 协议从公司的身份提供商那边自动同步。这里插一句解释，SCIM 是一种行业标准协议，作用是让企业的身份管理系统，比如 Okta 或者微软的 Azure AD，跟下游各种软件系统之间自动同步用户和分组信息。打个比方，公司 HR 系统里一旦有人入职、离职、换部门，这个变动如果配置了 SCIM 同步，就能自动反映到 Claude 的权限系统里，不需要管理员每次手动去改。分好组之后，管理员可以给每个组分配一个自定义角色，定义这个组成员能用 Claude 的哪些能力。换句话说，可以先只给某个团队开通 Cowork，比如先在产品团队试点，等团队适应了再逐步扩大，整个过程可以按自己的节奏调整，不需要一上来就全公司铺开。这个设计思路我觉得挺务实的，它实际上是把一个大决策拆成很多个小决策，每个团队、每个角色都可以单独决定开不开、开到什么程度，这样管理员的心理负担会小很多，出问题的话影响范围也是可控的。

第二项新功能是分组花费限制，原文叫 group spend limits，管理员可以在后台给每个团队设置预算上限，成本是可预测的，而且随着对各团队实际需求的了解加深，预算还可以随时调整。我们都知道 AI 模型调用是要花钱的，尤其是 Claude Cowork 这种能自主执行多步骤任务的智能体，一旦开始干活，可能连续调用很多次模型推理，消耗的 token 量比简单聊几句话要多得多。如果没有花费限制，某个团队，或者团队里某个特别热情的员工，完全可能不知不觉中把预算用超了。设置分组花费限制，本质上是给每个团队画一条财务安全线，既给团队足够的自由度去探索怎么用好这个工具，又不会让财务部门在月底账单上看到一个意外的天文数字。

接下来是这篇文章里我认为含金量最高的一部分，关于可观测性和使用分析。第一块是使用分析，Cowork 的活动数据现在会出现在两个地方，一个是管理员后台的仪表盘，另一个是分析接口，也就是 Analytics API。仪表盘这边，管理员可以按不同时间范围，追踪 Cowork 的会话数量和活跃用户数，看到的是比较宏观的趋势图，适合管理层快速了解整体采用情况。如果想看得更细，就要用到 Analytics API 了，它能提供更深层的数据，比如每个用户具体的 Cowork 活动记录，调用了哪些技能、哪些连接器，还有日活跃用户、周活跃用户、月活跃用户这几个指标，跟原本就有的 Chat 聊天和 Claude Code 的数据并列展示。这样管理员就能看清楚哪些团队真正把 Cowork 用起来了，哪些工作流跑得比较顺，下一步该把资源投在哪个方向。我觉得这一点对企业的 AI 推广策略特别重要，很多公司推广新工具往往凭感觉，凭哪个部门负责人嗓门大、反馈积极，但有了这种细粒度的使用分析之后，管理员是真的可以拿数据说话的。

第二块更重要，是扩展后的 OpenTelemetry 支持。这里先简单解释，OpenTelemetry 是一套开源的可观测性标准，被广泛用于采集系统运行时产生的指标、日志和调用链路数据，很多企业的安全和运维团队都依赖它监控自己系统里发生的事情。现在 Claude Cowork 会针对几类行为发出事件记录，工具和连接器的调用，文件被读取或修改的记录，技能被使用的记录，以及每一次 AI 主动发起的操作，到底是被人工审核通过的，还是被自动批准执行的。这些事件可以接入标准的安全信息和事件管理系统，业内简称 SIEM，比如 Splunk 和 Cribl 这两个企业里很常见的安全监控平台。另外文章还提到有一个共享的用户账户标识符，可以把这些 OpenTelemetry 事件，跟另外一个叫合规接口，也就是 Compliance API 里的记录关联起来对照查看。这项能力在团队版和企业版上都可以用。我想多花点时间讲讲这部分为什么重要，你可以想象一个智能体在公司里跑来跑去，它能读文件、改文件，能调用各种连接器操作外部系统，如果完全是个黑箱，安全团队晚上肯定睡不着觉。万一它误删了重要文件，或者把内部数据通过某个连接器发到了不该发的地方，公司怎么知道发生了什么。这就是可观测性存在的意义，把每一次工具调用、每一次文件改动、每一次审批动作都记录下来，并且能接入企业现有的安全监控体系，相当于给这个智能体装了一个完整的行车记录仪，出了事可以回放，没出事也可以做常态化审计。这其实是企业级 AI 部署里最容易被忽视、但又最不能省略的一环。

讲完可观测性，文章接下来讲了两项跟连接器有关的更新。第一个是 Zoom 的 MCP 连接器，这里提一下 MCP 是什么，它是一套让 AI 模型能够标准化连接外部工具和数据源的协议，可以理解成 AI 和各种软件系统之间的一个通用插头。今天 Zoom 正式推出了一个连接器，把会议智能直接带进了 Cowork 的使用体验里，能把 Zoom 自带的 AI 助理生成的会议摘要和行动项，连同会议转录文字和智能录制内容，一起接入到 Cowork 里。这样团队就可以把自己在 Zoom 上开会留下的内容，变成 Cowork 可以进一步加工的智能体工作流素材，如果感兴趣可以去 Claude 设置里的连接器目录把 Zoom 加进去。我觉得这个连接器挺实用的，因为开会占用了职场大量时间，但会议结束之后产生的记录、行动项往往散落各处没人系统跟进，如果 Cowork 能直接读懂这些信息，就可以帮你自动生成会议之后的跟进任务，自动核对上次会议定下的事情有没有完成。

第二个更新是单个工具的连接器控制权限，原文叫 per tool connector controls，允许管理员针对每一个 MCP 连接器，单独限制具体哪些操作可以执行，文章给的例子是可以允许读取访问但禁用写入操作，这种权限全公司统一生效，在管理员后台统一配置。这个功能听起来不起眼，但解决的是一个很关键的安全顾虑。连接器本身是双刃剑，它能让 Cowork 真正干活，比如直接改 Jira 上的工单状态、往 Slack 发消息，但同样的能力如果失控也可能造成实际破坏。有了单个工具的细粒度权限控制，企业就可以选择更保守的起步方式，比如先只允许 Cowork 读取工单信息用来生成报告，但不允许它直接修改，等团队对工具可靠性有了更多信心之后，再逐步放开写入权限。这种先只读、再逐步放权的渐进策略，我觉得是大多数谨慎企业引入自主智能体时都会采用的合理路径。

文章接下来用了三个具体企业案例，说明 Claude Cowork 在真实场景里到底被怎么用起来，我觉得这部分特别值得细讲。第一个案例是 Zapier，他们把 Cowork 连接到公司内部的数据库、Slack 和 Jira，目的是找出工程团队里的瓶颈到底卡在哪儿。最后 Cowork 交出来的成果包括一份仪表盘、一份按团队拆分的逐一分析报告，还有一份排好优先级的路线图。更有意思的是，产品团队和设计运营团队看到这个成果之后，直接把这套方法复制到了自己团队身上去用。这说明一个好的 Cowork 工作流，一旦在一个团队跑出效果，很容易被其他团队直接借鉴复用。

第二个案例是 Jamf，他们把原本一个有七个维度的复杂绩效考核流程，用 Cowork 改造成了一个四十五分钟就能走完的引导式自我评估，员工不再需要面对一份让人头大的长表格，而是跟着 Cowork 的引导一步步把工作回顾梳理清楚。做出效果之后，Jamf 又把同样思路用到了供应商评审和事故应急响应这两个完全不同的流程上。第三个案例是一家叫 Airtree 的风险投资机构，他们用 Cowork 搭建了一个董事会材料准备工作流，会自动从被投公司的网盘、Slack 上的进度更新、还有竞争对手的新闻报道里抓取信息，并且会拿这次准备的材料跟上一次的准备记录做交叉对照，看看哪些指标变化了，哪些问题还没解决。这对投资机构来说价值很直接，合伙人每个月要看几十家被投公司的进展，靠人工一家一家翻材料做对比效率很低，还容易遗漏。

这三个案例放在一起看，能发现一个共同点，Cowork 真正发挥作用的地方，往往不是某个单点任务，而是一整套需要从多个数据源汇总信息、做对比分析、生成结构化交付物的工作流。这恰好也印证了前面讲的那个核心洞察，Cowork 接的不是核心判断，而是核心判断所需要的素材准备和信息整理工作。

聊完功能和案例，我们说说这些内容对听众的实际意义。如果你是个人开发者或者普通员工，跟管理员权限关系不大，我建议重点关注两件事。第一，连接器生态正在变得越来越丰富，这次新增了 Zoom，再加上之前的 Slack、Jira、Google Drive，意味着不需要手动把各系统信息搬来搬去，可以直接让 Cowork 跨多个工具汇总信息，如果你日常工作里有大量从这个系统抄到那个系统的杂活，这正是值得尝试自动化的地方。第二，不要一开始就指望把核心判断交给它，从早期信号和三个案例都能看出来，比较成功的用法都是先从外围辅助性工作开始，比如整理材料、生成草稿、做交叉对比，等用顺手了再考虑往更深的地方拓展。

如果你是企业里负责 IT 治理、安全合规或者 AI 工具采购的人，这篇文章相当于给你提供了一份现成的部署清单。我建议按这个顺序去想问题，先想清楚谁应该有权限用 Cowork，是不是可以先在一两个愿意尝试的团队用角色和分组的方式小范围试点，而不是一刀切全公司开放。然后想清楚预算怎么控制，给每个试点团队设一个分组花费上限，避免出现意外的账单。再想清楚怎么监控，OpenTelemetry 这套东西如果你公司已经有现成的 SIEM 体系，建议尽早把 Cowork 的事件接进去，从第一天起就有完整的审计轨迹，而不是等出了问题之后才补这道防线。最后，连接器权限这件事我会特别提醒一句，宁可一开始保守一点，先只开只读权限，让 Cowork 先帮你做信息汇总和报告生成这类相对低风险的事情，等观察一段时间确认它的行为符合预期之后，再考虑逐步放开写入权限，这是一个能让你睡得安稳、又不耽误业务尝鲜的折中方案。

还有一点值得单独说一下，就是使用分析这个能力。很多企业采购新工具的时候，容易陷入凭印象做决策的状态，觉得这个工具好像挺火、那个部门反馈不错，但缺乏真正的数据支撑。Cowork 现在把这套分析能力做到了后台仪表盘和 API 两个层级，意味着完全可以建立一套常规复盘机制，比如每个月看一次各团队的活跃用户数和会话量，看看哪些团队真正用起来了，哪些团队开了权限却几乎没用，据此调整培训资源和推广重点。这种数据驱动的迭代方式，我觉得比单纯凭感觉推广要靴实得多。
