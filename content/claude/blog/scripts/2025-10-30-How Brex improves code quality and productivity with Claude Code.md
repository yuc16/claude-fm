大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是 How Brex improves code quality and productivity with Claude Code,翻译过来就是 Brex 如何用 Claude Code 提升代码质量和生产力。这篇文章发表在 Anthropic 自己的官方博客上,发表时间是二零二五年十月三十日。

先说说背景。Brex 是一家做企业信用卡和费用管理的金融科技公司,主打的概念是智能金融,英文叫 intelligent finance,意思是用智能化手段帮企业客户花钱更聪明、决策更快。这家公司本身的技术栈相当复杂,文章里提到他们用的是 Kotlin 语言加 Bazel 构建系统的大型单体仓库,这种架构在大公司里很常见,优点是代码统一管理,缺点是新人上手特别难,不同模块之间的依赖关系往往只存在于少数资深工程师的脑子里,业内管这种现象叫部落知识,英文是 tribal knowledge,意思就是知识像部落传说一样口口相传,而不是写在文档里。

那么在 Claude Code 进入 Brex 之前,这家公司面临的典型问题是什么呢。第一个问题是技术团队和非技术团队之间存在一条很深的鸿沟。比如一个内容设计师想改一行界面上的文案,流程是提工单,然后排队等工程师有空了去改,这个等待周期可能是几天甚至更久。第二个问题是,即便是工程师内部,做一些重复性的、模式化的工作,比如给数据库加一张新表、配置相应的文件和测试,也需要专门的领域知识,不是随便一个工程师都能上手。第三个问题更隐性一点,就是组织知识沉淀的问题,系统怎么运作、各个模块怎么连接,这些信息散落在不同人的脑子里,新人融入团队的成本很高。

文章采访了三位 Brex 的员工,分别是产品 AI 团队的工程负责人埃尔库勒斯·吉门内斯,数据与分析团队负责人苏米特·马瓦哈,还有高级内容设计师安迪·里德。这三个人代表了三种完全不同的角色,工程师、数据负责人、非技术的内容设计师,但他们讲的故事有一个共同的主线,就是 Claude Code 怎么改变了他们工作的方式。我们接下来就按这三条线索,逐个拆开讲。

先说工程师这条线。吉门内斯提出了一个我觉得是这篇文章里最核心的概念,他说 Claude Code 带来的是一种心智转变,英文是 mindset shift。他的原话是,以前你是坐在驾驶座上的人,现在你变成了审阅 Claude 所做改动的人,你的工作是引导方向和愿景。这句话听起来很简单,但我们仔细想想它背后的含义。传统意义上,工程师写代码的过程是,先想清楚要做什么,然后一行一行把逻辑敲出来,中间大量时间花在处理语法细节、查 API 文档、调试报错信息上面。而吉门内斯描述的新工作方式是,工程师把更多时间花在定义问题本身、设计架构、做判断和审查上,具体的代码实现交给 Claude Code 去完成,人退后一步,从司机变成了坐在副驾驶位置上掌舵方向的人,或者更准确地说,变成了坐在审稿台前的编辑。

这里有个特别值得展开讲的细节,吉门内斯说,我不觉得用 Claude Code 帮我省了时间,我把更多时间花在打磨问题本身上了。他举的例子是,同样的时间里,你可以为同一个问题尝试三种不同的实现方案,等你试到第三种方案的时候,你对问题的理解已经比一开始清楚得多,最后做出来的接口也干净得多。

我觉得这个例子特别值得展开讲讲,因为它揭示了一个反直觉的结论:效率工具带来的收益,不一定体现在做同一件事花的时间变少了,而是体现在同样的时间里,你能探索的方案空间变大了。打个类比,这就像你以前画一幅画,因为画一笔很费劲,所以你只能画一稿就定稿,稍微画错了也舍不得改。现在有了草稿工具,你可以画十遍草稿再选最满意的一张去精修。最终产出的质量提升,不是因为你画得更快了,而是因为你能试错的次数变多了,而试错次数变多,直接带来的就是对问题理解的加深。这跟我们写代码是一个道理,以前因为改一遍代码成本高,所以工程师倾向于第一版方案就是最终方案,现在改一遍的成本接近于零,工程师反而会主动多尝试几种架构设计,最后选出最优的那个。这也是为什么文章强调,这不只是产出速度变快的故事,更是产出质量变好的故事。

第二条线索是安迪·里德,这位内容设计师的故事我觉得是整篇文章里最打动人的部分。两年前他加入 Brex 做内容设计师的时候,绝对没想到自己会写代码,甚至会去做 Figma 插件开发。他原话说,过去这一个月里,如果没有 Claude Code,我真不知道自己怎么能完成一半的工作量。

里德遇到的第一个具体场景是,以前哪怕只是改一个简单的界面文案字符串,他都得提工单,然后等工程师有空帮他改,这个流程对一个内容设计师来说是相当挫败的,因为文案这种东西往往需要反复打磨措辞,但每改一次都要走一遍排队流程,效率极低。有了 Claude Code 之后,里德可以自己直接提交代码合并请求,也就是 PR,不再需要等工程师代劳。

但真正让人印象深刻的,是他后来做的一个更大的项目。Brex 内部一直有一个被搁置在待办清单里、迟迟没人动手的项目,就是要把完整的内容规范整合进设计系统的每一个组件里。这是什么意思呢,简单说,公司有一套设计系统,里面包含按钮、表单、提示框等各种界面组件,这些组件除了视觉规范之外,理应配套有文案规范,比如什么场合该用什么语气、哪些用词要避免、怎么照顾无障碍阅读需求等等。但这件事一直没人做,因为工作量太大,涉及到把内部风格指南、无障碍标准,还有外部行业最佳实践,系统性地梳理并写入每一个组件的文档里。

里德说,如果靠手动做,这个项目可能要花上几周甚至几个月。但用了 Claude Code 之后,他在几天之内就完成了。他系统性地把各种来源的内容规范提炼出来,逐一写入设计系统的组件文档。这个项目完成之后产生的影响是连锁式的,工程师和设计师在使用设计系统组件的时候,直接就能看到内置的内容规范,不再需要来回查阅各种分散的文档,这也减少了在内容上犯错的概率。

我想在这里插一句我自己的解读。这个案例特别有意思的地方在于,它展示了 Claude Code 的一种用法,跟我们平时想象的"程序员用 AI 写代码"不太一样。里德不是工程师,他做的事情其实是知识整理和内容生产,只不过最终的载体是写进代码仓库里的文档文件。这说明 Claude Code 这类工具的价值边界,已经不局限于传统意义上的软件开发,而是延伸到了任何需要跟代码库打交道、哪怕只是改改文档和文案的场景。换句话说,这类工具正在让"懂不懂写代码"这件事,变得没那么重要了,重要的是你懂不懂你要解决的问题。

第三条线索是数据团队的故事,负责人是苏米特·马瓦哈。他们团队面临的挑战是,怎么让公司里不懂 SQL 的人也能用上数据。SQL 是一种用来查询数据库的专门语言,传统上只有懂技术的分析师或者工程师才会写。马瓦哈团队做的事情,是基于 Claude Code 和 MCP 服务器搭建了一个叫 Brex Explorer 的工具,MCP 这里指的是一种让 AI 模型连接外部工具和数据源的协议。Brex Explorer 本质上是一个文本转 SQL 的接口,用户用自然语言提问题,系统自动把它翻译成 SQL 查询语句,跑出结果再总结成人话回复给用户。

文章里给了一个很具体的例子,一个销售经理可以直接打字问,在佛罗里达州杰克逊维尔,有多少客户的 Brex 账户余额超过一千万美元,并且是在过去一年内完成开户的。系统会把这句话翻译成对应的数据库查询,执行查询,然后把结果总结出来。这个例子特别能说明问题,因为这种问题如果让一个不懂 SQL 的销售经理自己去查,基本是不可能的,他得去找数据分析师帮忙写查询、跑数据、再把结果整理成报告发给他,中间的等待周期可能是几天。现在他自己打字提问,几乎是实时拿到答案。

但马瓦哈强调,真正带来杠杆效应的不是这个对外的查询工具,而是他们内部搭建的一个数据工程智能体,英文叫 AI data engineering agent。以前如果要给数据库新增一张表,需要懂相关领域知识的专业工程师来操刀,因为这涉及到一系列文件改动和测试配置,稍有不慎就会影响其他依赖这张表的系统。现在借助这个智能体,任何一个工程师,即便他不是数据领域的专家,也能完成这项任务。马瓦哈用了一个很直观的说法,他说这相当于效率提升四倍,同时参与贡献的人数还翻了一倍。文章提到,Claude Code 在他们组织内部的采用率已经达到百分之五十,马瓦哈的目标是在当月月底前做到百分之百采用率。

这三条线索放在一起看,我觉得有一个共同的底层逻辑,就是 Claude Code 把原来集中在少数专家手里的能力,分散到了更多人手中。工程师不再是唯一能改代码的人,SQL 专家不再是唯一能查数据的人,知道系统怎么运作的资深工程师也不再是唯一的知识来源。这种能力的去中心化,本质上是把组织内部的瓶颈给打开了。

接下来文章还讲了几个意外发现的用例,我觉得也值得提一提。第一个是吉门内斯在公司内部一次黑客马拉松活动中,用 Claude Code 的无头模式,英文叫 headless mode,在几个小时之内搭出了一个相当复杂的提交评审智能体,这种工作量按以前的开发周期得花上好几天。无头模式简单理解,就是 Claude Code 可以在没有交互式界面的情况下,作为后台程序被脚本调用和编排,这就给了开发者更大的自由度去把它嵌入到自动化流程里。

第二个意外用例还是里德,他后来又做了一个 Figma 插件,能够自动按照 Brex 的设计标准去审查设计稿。他的原话是,这感觉就像我有了世界上最好的实习生。这句话我听着挺有意思的,因为它把 AI 工具定位成了一个能干活、肯学习、但还需要人来把关方向的助手角色,而不是一个能完全替代人的存在。

第三个,也是我认为文章里最有分量的一个发现,是 Claude Code 在 Brex 这种复杂代码仓库里,扮演了组织知识库的角色,文章里用了一个词叫 oracle,也就是神谕、先知的意思。前面提到 Brex 用的是 Kotlin 加 Bazel 的单体仓库,系统之间怎么连接,一直是个老大难问题。现在 Claude Code 可以直接回答关于代码库功能的问题,几乎是即时响应。里德有句话说得特别生动,他说,以前开一个设计会议,大家讨论某个功能应该怎么运作,但其实没人真正搞清楚这个香肠到底是怎么做出来的,现在我的另一块屏幕上坐着这位神谕,随时可以问。

这个比喻我觉得特别精准,"香肠是怎么做出来的"这个说法在英文里常用来形容一个流程或产品背后复杂又不透明的制作过程,大家只看到结果,不知道中间经历了什么。Claude Code 在这里起到的作用,相当于把原本封闭在代码内部、只有写代码的人才懂的逻辑,翻译成了任何人都能问、都能懂的对话。这对组织知识传承的意义,我觉得甚至比单纯的代码生成更重要,因为它解决的是企业里最常见也最痛的一个问题,就是知识随着人员流动而流失。

讲完了这些案例,文章接下来总结了 Brex 摸索出来的几条最佳实践,这部分我觉得对我们普通听众来说,实操价值是最高的,值得逐条展开讲。

第一条叫结构化上下文管理。具体做法是,Brex 在单体仓库里的每一个主要目录下面,都放了一个叫 CLAUDE 点 md 的文件,里面写明这个目录对应的领域知识。比如某个目录涉及万事达卡的集成细节,或者某个目录涉及银行监管合规的要求,这些信息就写在对应目录的 CLAUDE 点 md 文件里。这样做的好处是,新加入团队的工程师,不需要依赖老员工口头传授,直接靠 Claude Code 读取这些文件,就能理解这部分业务的背景知识。这跟我们前面提到的部落知识问题,正好是对症下药,把口口相传的隐性知识,转化成了写在文件里的显性知识,而且这些文件本身又是 AI 工具可以直接读取和利用的格式。

第二条叫自动化文档校验。产品 AI 团队在持续集成和持续部署流程里,也就是我们常说的 CI CD 流程里,加了一道检查,专门用来识别代码改动有没有让现有文档过时,一旦检测到,就会提示相关人员去更新文档。吉门内斯特别强调说,拥有可信赖的、保持更新的文档,这件事本身的价值非常大。这一点其实道理很简单,但很多团队都做不到,因为文档维护通常是一件没人愿意主动做、又很容易被拖延的事情。把这件事自动化、变成流程里的强制检查项,相当于用机制保证了文档不会因为没人维护而慢慢失真,这对依赖文档做判断的 AI 工具来说尤其重要,因为如果喂给 AI 的文档本身是过时甚至错误的,那 AI 给出的回答质量也会跟着下降。

第三条叫上下文感知命令。Brex 团队给 Claude Code 配置了一些自定义命令,这些命令在执行的时候会自动加载相关的上下文信息。文章举的例子是一个叫提交合并请求的命令,执行的时候会自动先去抓取当前的 git 状态、最近的代码改动,还有相关的合并请求信息,然后才开始执行实际的任务。这条最佳实践的核心思想是,把"收集背景信息"这一步从每次都要手动重复的操作,变成命令本身自带的固定流程,减少了人在每次使用工具时需要手动准备上下文的负担。

第四条,文章把它叫做从发现开始。马瓦哈有句话我觉得说得很到位,他说,要找到你需要的东西,现在已经不再依赖于存在某个资深工程师的脑子里了,它就存在于代码库本身,人人都能访问。这句话其实是对前面三条实践的一个概括性总结,核心思想就是,组织知识应该尽量沉淀在可被检索、可被工具利用的载体里,而不是依赖个体记忆。

文章最后还提到了 Brex 团队对未来的规划。数据团队准备挑战一些以前因为工作量太大而被搁置的项目,比如分析数万亿条信用卡交易的明细数据,这种工作以前因为太繁琐,投入产出比不划算,一直没人去碰,现在 Claude Code 让这类工作变得可行了。他们还在开发针对特定角色的智能体,其中一个是专门服务于营收运营团队的。产品 AI 团队则设想未来能做到更精细的上下文管理,文档可以自动更新自己,上下文信息保持分散但又随时可被访问。

如果你是一名工程师,我觉得这篇文章最值得借鉴的一点是吉门内斯讲的那个心智转变。很多人用 AI 编程工具的时候,潜意识里还是把它当成一个更快的打字员,你告诉它写什么,它帮你写出来,你检查一下没问题就合并。但如果你能往前一步,把自己的角色定位成审稿人而不是执笔人,你会发现自己有更多的认知带宽去思考架构层面的问题,而不是纠结具体某一行代码该怎么写。具体到操作层面,我建议大家可以试着对同一个需求,让 Claude Code 给出两到三种不同的实现思路,然后自己花时间去比较这几种思路的优劣,而不是看到第一个能跑通的方案就直接采用。这跟吉门内斯讲的尝试三种方案、最后理解更清晰的体验是一致的。

如果你是非技术背景的同事,比如产品经理、内容设计师、运营,里德的故事我觉得给了一个很好的信心,就是你不需要先系统学习编程,才能开始用 Claude Code 解决你工作中的实际问题。当然这里有个前提,就是你得有一个具体、边界清晰的小任务作为起点,比如改一段文案、整理一份规范文档,而不是上来就想做一个完整的产品功能。从小任务积累信心和经验,再逐渐扩大自己能驾驭的任务范围,这是一条相对稳妥的路径。

如果你所在的团队也维护着一个比较复杂的代码仓库,我觉得 Brex 这四条最佳实践里,最值得立刻动手的是第一条,也就是给每个主要目录配一个说明性的文档文件,把领域知识写下来。这件事的成本相对低,不需要额外的工具或者流程改造,但收益是长期的,既能帮助 AI 工具更准确地理解代码库,也能直接帮助到新入职的人类同事。第二条自动化文档校验,如果你们已经有比较成熟的持续集成流程,加上这一道检查的边际成本也不高,但要注意的是,这套机制只有在团队真正愿意响应提示去更新文档的时候才有意义,如果加了检查但大家选择性忽略提示,那就只是徒增噪音。

我也想提一下这篇文章可能存在的局限性,这是我个人的判断,跟原文的观点要分开看。首先,这篇文章是 Anthropic 自己发布的客户案例,本质上带有一定的营销属性,文中提到的三四倍效率提升、百分之五十的采用率这些数字,都是受访者口头转述,没有给出具体的测量方法或者对照实验,我们听的时候可以把它当作一个方向性的参考,但不必把具体倍数当成放之四海而皆准的硬指标。其次,Brex 本身是一家金融科技公司,对代码质量和合规性的要求本来就很高,这种背景下推广 AI 编程工具,团队的审查流程和把关意识可能本来就比较扎实,所以文章里强调的"工程师从司机变成审稿人"这种转变,能不能在审查文化相对薄弱的团队里同样奏效,这点我觉得是要打个问号的。最后,考虑到这篇文章已经过去八个月,Claude Code 本身在这段时间里也在持续迭代,具体的功能边界、无头模式的能力,可能都已经跟文章发表时有所不同,如果你想动手尝试,建议去看一下当前的官方文档,而不是完全照着这篇文章里的描述去操作。

好,我们来总结一下今天这期节目。这篇 Brex 的案例文章核心讲的是一件事,Claude Code 在企业内部的价值,不只是让工程师写代码更快,而是打破了技术能力的边界,让内容设计师能独立完成原本要排队等工程师的任务,让数据团队能把查询数据库的能力开放给不懂 SQL 的业务同事,也让 Claude Code 本身变成了组织知识传承的一个新载体,缓解了过去依赖个人记忆和口头传授带来的知识断层问题。我个人觉得,这篇文章最有价值的部分,不是那些效率提升的数字,而是 Brex 摸索出来的那几条组织层面的最佳实践,把领域知识写进代码仓库、用自动化机制保证文档不过时、把上下文收集的步骤内置进自定义命令,这些做法跟具体用什么模型、什么版本关系不大,放在任何一个正在尝试推广 AI 编程工具的团队里,都值得参考。好,今天的节目就到这里,我们下期再聊,谢谢大家收听。
