## Shownotes

Claude Code 昨天上线了 Artifacts 功能，让你的 session 里的调查进展、代码分析、事故复盘，能够自动变成一个实时更新的可交互网页，在团队内部一键同步共享，再也不用截图、复制粘贴或者口头转述。这一期我们深入聊聊这个功能背后的设计逻辑、它解决了什么真实痛点，以及各个角色怎么把它用到日常工作里去。

- Artifacts 把 Claude Code session 的成果自动转化为可分享的网页，消除 AI 工作成果的手动转述环节
- 页面基于代码库、外部工具连接器和对话三层上下文构建，信息整合程度远超手工整理
- 实时更新机制：Claude Code 每次重新发布，同一链接下的页面自动刷新，并保留完整版本历史
- 权限体系完整：默认私有，只能在组织内部认证成员之间分享，不可设为公开
- 适用场景覆盖全职能角色：incident 调查、PR walkthrough、许可证审计、成本分析、架构文档、事故后复盘
- 目前是 beta 版，仅向 Claude Team 和 Enterprise 用户开放，CLI 和桌面应用均可触发

## Script

欢迎收听 Claude FM，我是你的主播。今天是二〇二六年六月十九日，如果你正在通勤路上听这期节目，你来得很巧，因为我们今天要聊的这篇文章，就在昨天，也就是二〇二六年六月十八日，才刚刚发布在 Anthropic 的官方博客上。

文章的英文标题是 "Claude Code now supports artifacts"，我把它翻译成中文是：Claude Code 现在支持 Artifacts 了。这里 artifact 这个英文词，本意是"工件"或者"制品"，在 Claude Code 这个产品里，它的意思是：把你在工作会话里产生的分析成果，自动转化成一个活的、可以实时更新的、可以在浏览器里打开和交互的网页。

这期节目，我打算用大概二十五分钟，把这个功能背后的设计逻辑、它解决了什么痛点、它的关键能力是什么，以及它对你的日常工作意味着什么，层层展开来讲清楚。文章发表于昨天，距今只有一天，所以信息是非常新鲜的，大家不用担心时效性的问题。

我们先从一个你可能非常熟悉的日常场景开始，帮你感受一下这个功能在解决什么。

你有没有遇到过这样的情况：你开着 Claude Code，让它帮你调查一个线上的 bug，或者分析一段时间的系统日志，或者梳理某个服务的代码依赖关系。Claude Code 跑了很长时间，产生了大量分析，找出了若干关键信息。然后你的 leader 发来一条消息问：能不能告诉我你那边发现了什么？

你面对这个问题，通常有几个选项。第一，把 terminal 里的输出复制粘贴到一个文档里发过去。问题是，terminal 输出往往格式很乱，充满了 AI 的中间思考过程，读起来非常费力，不适合直接作为协作材料。第二，截图。问题是截图是静态的，对方看到的内容没法交互，想找某个特定信息还得来问你。第三，直接打个电话或者开个语音，重新给对方解释一遍。这个方式最容易让对方理解，但你得把 Claude Code 告诉你的东西用人话重新组织一遍，而且语音解释不留存，下次来一个新的同事，你还得再说一遍。

更麻烦的是，调查往往没有就此结束。你跟 leader 说完之后，Claude Code 又发现了新的线索，情况又更新了。你是再截图发一次，还是再语音讲一遍，还是在 Slack 里发一段更新文字？每一次更新都是一次手动的信息传递工作，这个成本会随着调查时间的拉长而不断累积。

这就是 Artifacts 这个功能想要解决的核心问题：AI agent 在工作过程中产生的成果，怎么才能不需要人来手动转述，就能顺畅地在团队里流转和共享，而且是实时的、持续更新的。

我们现在来正式认识这个功能。

Claude Code 的 Artifacts 功能，允许你在一个 session 进行的过程中，随时生成一个实时更新的网页。这个网页会把 session 里 Claude Code 产生的重要信息，包括代码分析的结论、可视化图表、事件时间线、问题发现、推断结论等等，整理成一个结构清晰、可以在浏览器里直接打开的交互式页面。

然后，你把这个页面的链接发给你的同事，大家打开都是同一个页面。当 Claude Code 继续推进调查、有了新发现、重新发布这个 artifact 之后，页面会自动刷新，所有在看的人同时看到最新内容，不需要任何人手动通知。

这个功能听起来简单，但背后有几个设计上的考量，非常值得我们一个一个拆解。

第一个核心点，是 Claude Code 会用整个 session 的完整上下文来构建 artifact。

我说完整上下文，到底指的是什么？文章里说，Claude Code session 的上下文包括三个层次：你的代码库、你接入的各种外部工具和数据源，以及你和 Claude Code 之间的对话本身。

这三个层次，每一个都是传统方式里需要你手动去拼合的信息源。

举一个具体的例子。假设你在做一个线上事故的调查，也就是 incident investigation。这个调查会涉及哪些信息？你需要看出问题的那段代码，这在代码库里；你需要看错误率的监控曲线，这在你的监控工具里，比如 Grafana 或者 Datadog；你还需要有人把这些信息综合起来，分析出根本原因，这是 Claude Code 在对话里帮你做的推断和分析。

传统做法是，你要自己去代码库里找到那段代码，去监控系统截图，然后把你的分析思路写成文字，再把三样东西拼到一起。每一步都是手动的，而且拼出来的东西往往比较散，不容易阅读，读者需要自己去建立不同信息之间的关联。

但用 Artifacts 的话，因为 Claude Code 的 session 里本来就已经接触了这三层信息，它可以直接把这些信息整合成一个连贯的页面，不需要你再去手动拼合。文章里描述，一个事故页面可以同时展示：出问题的测试代码和这个测试背后的具体函数，来自接入监控工具的错误率图表，以及 Claude Code 推断出的根本原因。三种来源的信息，在同一个页面里，以一种对读者友好的方式呈现出来。

文章里有一句话我觉得是这个功能的核心主张，我想特别念出来：你不需要去搭数据管道，也不需要去部署基础设施。你只需要说"给我建一个页面"，Claude Code 就会从已有的东西里把这个页面建出来。

这句话背后的意思是什么？之前做这类信息整合工作，往往需要专门搭建技术基础设施：配置数据接入、写 dashboard 脚本、部署展示页面。这些工作有技术门槛，也很花时间，通常不是普通的软件工程师会在日常工作里去做的事情。Artifacts 的设计是把这些工作隐藏起来，让用户只需要表达意图，其余的事情由 Claude Code 来搞定。

我认为这背后体现了一个比较重要的工程哲学：AI 工具应该能充分利用 session 里已经存在的信息，而不是要求用户在 AI 工具之外再额外搭一套数据管道。这是从"工具"走向"能力整合"的一个体现。

第二个核心点，是实时更新和版本管理。

当 Claude Code 重新发布一个 artifact 的时候，已经打开这个页面的团队成员，会立即看到更新后的内容。文章里的描述是，每次发布都是同一个链接下的一个新版本，但同时保留版本历史，你随时可以回到任意一个历史版本去查看。

这个"同一个链接"的设计非常重要，我来解释一下为什么。分享工作成果的场景往往是这样的：你把一个链接发到 Slack 频道，所有人都去看这个链接。如果每次更新都产生一个新链接，那你每次更新都要再发一次新链接，历史聊天里的旧链接就失效了，这会制造很多混乱，也会让人不知道哪个链接才是最新的。

同一个链接永远指向最新版本，同时保留历史版本可以回溯，这个设计才是合理的。这种设计我们在其他产品里也见过，比如 Notion 里的一个页面，你给别人发了链接之后，你再去改那个页面，对方刷新之后就能看到最新内容，链接本身没有变化。Artifacts 的设计与此类似，但多了一个重要的不同：更新是由 AI 的工作进展驱动的，而不是由人工编辑驱动的。Claude Code 推进了一步，就重新发布一次，页面就更新了一次，你什么都不用做。

版本历史的功能，让你可以追溯整个调查过程是怎么演进的。这在事故复盘的场景里特别有价值。一次大的事故，你的调查可能要经历好几个阶段：刚开始你以为是 A 问题，后来发现是 B 问题，最后确认根本原因是 C。有了版本历史，这个认知演变的过程被完整记录下来了，事后复盘的时候可以看到我们是在什么时间点意识到问题在哪里的，这对于改进未来的响应流程有很高的参考价值。

文章里还提到了一个 gallery 功能，也就是一个可以浏览和管理你所有 artifacts 的地方。随着时间推移，你做过的 artifacts 会越来越多，有一个集中的管理入口，比分散在各个对话历史里要方便得多。

第三个核心点，是权限体系和企业级安全设计。

这个功能在企业场景里使用，安全是必须认真对待的问题。Anthropic 的设计原则是：每个 artifact 默认是私有的，只有创建者本人能看到，你需要主动去操作才能分享给别人。

分享的范围也有明确的限制：artifact 页面只能被你所在组织里的已认证成员查看，没有办法设置成互联网上任何人都能打开的公开页面。

为什么要有这个限制？因为 artifact 里包含的内容，往往是从你的代码库、内部系统和对话里提取出来的，可能包括系统架构的细节、安全漏洞的信息、还没有对外公开的产品计划等等。如果可以随意设置成公开，信息泄露的风险是非常高的。所以 Anthropic 的选择是直接把"公开"这个选项去掉，只允许在组织内部流转。

对于企业管理员来说，还有更细粒度的管控工具。组织级别的开关，让管理员可以整体上允许或者禁止使用 Artifacts 功能；基于角色的权限范围设置，让不同角色的员工看到的内容可以有所区分；数据保留策略，让企业可以设置 artifact 的保存时限，满足合规要求；以及一个合规 API，让管理员可以看到整个组织里所有 artifacts 的使用情况，用于审计和合规。

这些企业级的管理功能说明，Anthropic 在设计这个功能的时候，考虑的不只是单个开发者的便利，而是整个组织如何在保证安全合规的前提下使用这个功能。这是一个比较成熟的企业级产品思路。

第四个核心点，我想深入展开文章里提到的各种使用场景。文章按照不同职能角色给出了一系列具体的例子，我觉得这部分非常值得一个一个讲清楚，因为这直接关系到你在日常工作中到底怎么用这个功能。

先说软件工程师最核心的场景，PR walkthrough，也就是对 pull request 的详细讲解。

你可能会想，PR 不是已经有 PR 描述了吗，为什么还需要 artifact？这里的关键差别在于信息的整合程度和可读性。一个典型的 PR 描述，往往是由开发者手写的，而写 PR 描述是一件让很多工程师头疼的事情。你知道这个改动干了什么，但把它写成让别人能迅速理解的文字，是需要额外花心思的。

更麻烦的是，reviewer 光看 PR 描述和 diff 往往不够，他们还需要去看改动周围的代码上下文，才能理解这个改动的合理性。所以 code review 的过程经常伴随着来来回回的问题和解答，一个 PR 来回拉锯好几天的情况并不少见。

如果 Claude Code 生成的 artifact 能把代码改动、改动的上下文、修改的理由、测试覆盖情况都整合在一个页面里，reviewer 打开这个页面就能快速建立起对这个 PR 的整体理解，不需要在好几个地方之间来回切换，减少来回沟通的次数。文章里给出的示例 prompt 是：给我做一个 artifact，把这个 PR 的 diff、修改背后的理由、以及我测试了什么都放进去。

再说 SRE 和 on-call 的场景，这是文章里用笔墨最多的场景，也是我认为最能体现这个功能价值的例子。

文章里描述了这样一个具体的场景：一个工程师在晨会之前发起了一个事故调查。Claude Code 分析了日志，然后发布了第一个 artifact：一个时间线、疑似问题的几个 commit、还有一张错误率的图表。她把这个 artifact 的链接通过页面顶部的分享按钮发给了团队。到晨会开始的时候，Claude Code 已经在调查推进的过程中重新发布了两次，把最新的发现更新进去了。

文章里有一句话我觉得非常值得反复回味：有了 Artifacts，团队成员和利益相关方不需要再说"能不能给我们讲一下 agent 发现了什么"，因为他们都在看同一个视图，有相同的上下文。

我来展开解释一下这句话的意义。在一个大的 incident 调查里，信息的流转是有成本的。AI agent 发现了什么，需要工程师去理解，理解之后再用语言向 leader 和其他同事解释。每经过一次人的转述，信息都会有一定的损耗：你可能简化了某些细节，你可能用了别人不熟悉的术语，你可能遗漏了某个重要的关联信息。

而且，在 incident 的紧张环境下，这种信息传递的压力会特别大。你一边要全神贯注地分析问题，一边还要时不时地在 Slack 里更新进度，这两件事情是相互干扰的。深度分析需要注意力高度集中，被频繁打断是非常损害效率的。

Artifacts 解决的是让信息传递这件事从需要人工介入，变成自动完成。你专心分析问题，Claude Code 专心把分析成果实时同步给团队。两件事情并行，互不干扰。等到晨会开始，大家都已经有了足够的上下文，不需要你再花时间解释背景，可以直接进入问题讨论，缩短响应时间。

这是一种效率的质变，不只是量变。

我们再来看几个其他角色的场景。

法律和开源合规的场景：生成一个列出所有第三方依赖和对应许可证的 artifact，并标出哪些是 copyleft 许可证。所谓 copyleft，就是那些有传染性的开源许可证，比如 GPL 系列。如果你的项目引入了一个 GPL 许可证的依赖，有可能在法律上要求你也把自己的代码开源，这对于商业软件来说是一个很现实的风险。发现这个问题的传统做法是，专门跑一个依赖扫描工具，然后手动整理报告，发给法律团队看。Claude Code 可以直接分析代码库，生成一个清晰的 artifact，法律团队打开页面就能看到所有信息，流程顺畅很多。

隐私合规的场景：把代码里涉及个人数据的所有地方，在哪里收集、在哪里存储、在哪里打日志，整理成一个数据流向图，用于隐私审查。在 GDPR 合规或者国内个人信息保护法合规的场景下，这种数据流图是必要材料。以往做这件事，往往需要安全工程师或者架构师手动梳理，耗时且容易遗漏。让 Claude Code 从代码里直接分析生成，准确性和效率都更高。

安全审查的场景：把代码审查里发现的所有安全问题，每一条都链接到具体的代码行，生成一个 artifact。这样修复人员打开 artifact 之后，可以直接点击链接跳转到需要修复的代码位置，而不是先读一个抽象的安全报告，再去自己找对应的代码，中间多了一个不必要的查找步骤。

FinOps，也就是云财务管理的场景：把 Terraform 这类基础设施即代码的文件分析出来，按服务分组，标出主要的成本驱动项，生成一个云资源成本地图。这对于做云成本优化的团队来说，有了这个 artifact 作为基础，讨论成本问题会更有针对性，不需要先花时间让大家理解当前的资源状况。

设计师和前端工程师的场景非常有意思：生成一个包含五种 UX 变体的 artifact，每个变体都是从实际的组件库里构建出来的，这样选中哪个方案都可以直接上线。这个场景的核心价值在于消除了从设计稿到代码的这个中间翻译层。传统流程里，设计师在 Figma 里出方案，工程师再把设计稿翻译成代码，这个翻译过程本身就有时间成本，而且难免有细节损失。如果 Claude Code 直接用实际的组件库生成可交互的 UI 变体，你看到的就是真实可用的代码效果，选中之后就可以直接集成，不需要再翻译。

架构师和 Staff engineer 的场景：把某个服务的内部结构，从代码里的实际 import 图里分析出来，画成一张结构图，而不是靠脑子里记的白板图。这种从真实代码生成的架构图，比人工维护的 wiki 文档更准确。人工维护的架构文档一个最大的问题是，代码改了，文档往往没有同步更新，最终变成了过期的、误导性的材料。从代码直接生成的 artifact，只要代码是最新的，这个 artifact 生成出来就是当前真实的架构，不存在过期的问题。

工程管理者的场景：生成一个本周团队合并了哪些 PR 的汇总页面，按项目分组。这对于做周报或者跟上级汇报非常实用。以前做这种汇总，你可能要手动去 GitHub 上筛选 PR、整理到文档里，现在 Claude Code 可以帮你自动完成这件事，让管理者把时间花在更有价值的思考和决策上。

讲完了各种场景，我们来到实践部分：对你这个听众来说，你应该怎么开始用这个功能，有什么需要注意的。

第一件事，确认你是否有使用权限。这个功能目前是 beta 版，只对 Claude Team 和 Enterprise 这两个订阅套餐的用户开放。如果你在公司里有这两个套餐之一，就可以直接去试了。如果你用的是个人版，暂时还不行，需要等后续扩展。

第二件事，了解触发方式。你不需要记什么特殊的命令。在 Claude Code 的 session 里，你可以直接说：帮我生成一个 artifact；或者，把这个调查结果做成一个可以分享的页面；或者更自然地说，给我一个可视化的页面展示这次的分析结果。Claude Code 会理解你的意图，然后帮你生成。文章里也说，你可以问它一些视觉化的需求，它会自然地用 artifact 来响应。

第三件事，选择合适的时机触发 artifact 的生成。不建议在 session 刚开始的时候就要求生成 artifact，那时候信息还太少，页面会很空洞。比较好的时机是，当 Claude Code 已经完成了一个重要的分析阶段，产生了足够有价值的信息之后，再让它生成 artifact。然后随着调查的推进，你可以再次触发重新发布，让页面不断更新，越来越完整。

第四件事，养成在分享之前快速检查内容的习惯。artifact 会包含从代码库和接入工具里提取的信息，这些信息可能包含你不想让某些人看到的内容，比如一个尚未修复的安全漏洞的细节、还没有公布的产品路线图信息、或者某些内部系统的敏感配置。虽然 Anthropic 限制了只有组织内部成员才能查看，但在同一个组织内部，不同的同事也可能有不同的信息敏感级别。分享之前快速扫一眼，是个好习惯。

第五件事，把 Artifacts 当作团队协作模式变革的一个切入点，而不只是一个小工具。最好的落地方式，是在团队里找一个高频、重复、信息传递成本比较高的场景，试着用 Artifacts 来改变它。比如每周的技术状态报告，比如 on-call 的事故处理流程，比如定期的代码合规审查。一旦在一个场景里跑通了，收益得到了验证，推广到其他场景就会容易得多。

你可能会担心：如果我让 Claude Code 去生成 artifact，它生成的页面质量好不好？这个问题是合理的，我没有亲自体验过，只能说文章里描述的使用案例非常丰富，看起来是真实使用过程中沉淀下来的场景。beta 版的功能会有一些不稳定的地方，生成效果也会因 prompt 的写法而有差异。建议你多试几种不同的描述方式，找到对你的场景最有效的触发语句。

我还想从更宏观的角度谈一谈，这个功能在 AI 工具发展趋势里处于什么位置。

过去几年，AI 编程工具的进化大概经历了几个阶段。先是单行代码补全，然后是聊天式问答，然后是 agent，也就是 AI 能自主执行多个步骤、自主调用工具、完成复杂任务。Claude Code 是一个完整的 agent 工具，它可以自主写代码、运行测试、调试问题、调用外部 API，完成从端到端的工程任务。

但是，agent 有一个天然的弱点，就是它的工作过程对外是不透明的。你启动了一个 Claude Code session，它在里面干了很多事，但如果你不跟它说话，它干了什么对外面的人来说是一个黑箱。而且，即使你自己清楚它干了什么，要把这些信息传递给你的团队，还是需要你手动去做，这个环节依然是人力密集的。

这个透明度的问题，是 AI agent 在企业里大规模落地的一个核心障碍。企业不只是要 AI 能干活，还要能看到 AI 干了什么，要能让不同的人按需获取 AI 工作的结果，要能把 AI 的工作输出融入到现有的协作流程里。

Artifacts 正是在尝试解决这个问题。它让 agent 的工作成果变得可见、可分享、可追踪，而且是以一种不需要人手动介入的方式。从这个角度看，Artifacts 不只是一个"方便分享"的小功能，它是 Claude Code 从个人工具走向团队工具的一个重要步骤。

我个人认为，AI agent 要想真正在企业里大规模落地，有几个必要条件必须同时满足。第一，它要能完成复杂任务，Claude Code 在这方面已经走在前列了。第二，它要能让人类监督和控制它的行为，这是 Anthropic 一直在强调的安全性和可控性方向。第三，它要能把自己的工作成果，以人类可以消费的方式，主动同步到团队的协作空间里，让 AI 的工作成果能够无缝地融入现有的工作流。Artifacts 就是在解决这第三个条件。

从这个角度看，这个功能的意义超出了它表面上的"生成一个网页"。它代表的是 AI agent 的工作成果开始真正打通到团队协作层面，这是 AI 工具从"提升个人效率"走向"提升组织效率"的关键一步。

当然，我也有一些观察和保留意见想分享。

首先，artifact 的质量高度依赖于 Claude Code 对信息的整合和展示能力。生成一个包含图表、时间线、代码链接的页面，技术上并不简单，Claude Code 需要判断什么信息对于当前场景最重要，然后以合理的方式组织呈现出来。这个能力还在 beta 阶段，不一定每次都能达到你期望的效果。

其次，对于一些高度定制化的需求，比如你们公司有非常特定的信息展示规范，或者需要跟内部的某个特定系统做深度对接，Artifacts 可能还不够灵活。这是一个通用工具，设计上覆盖广度，不是为每个团队的特定需求深度定制的。

第三，这个功能目前只对 Team 和 Enterprise 用户开放，门槛对于小团队或者个人开发者来说偏高。如果你有能力体验，我建议去试试；如果暂时没有条件，可以持续关注后续的开放计划。

第四，我个人认为这类功能的长期价值，会在积累了足够多的使用案例之后才真正显现出来。当你的团队里有了大量的 artifacts，在不同的项目、不同的事故调查里沉淀下来，这些 artifacts 本身就会成为一种有价值的知识资产，新人可以通过回顾历史 artifacts 来快速了解系统和团队的工作方式。这个价值是随时间递增的。

好了，我们来做今天的总结。

Claude Code now supports artifacts，这篇文章发表于二〇二六年六月十八日，就在昨天。核心内容是：Claude Code 推出了 Artifacts 功能，让你可以在一个 session 里随时生成一个实时更新的网页，把 session 里的工作成果，包括代码分析、图表、事故时间线、调查结论，整合成一个结构清晰、可以在浏览器里打开的页面，然后分享给你的团队。

这个功能有几个关键特点：它会利用代码库、外部工具接入和对话三层上下文来构建页面，信息整合程度远超人工整理；它实时更新，同一个链接永远指向最新版本，同时保留历史版本；它默认私有，只能在组织内部分享，不能设为公开；它有完整的企业级权限管理工具，满足合规需求。适用场景非常广泛，从 incident 调查到 PR 评审，从依赖许可证审计到云资源成本分析，从 UX 方案展示到架构文档生成，几乎各个职能角色都可以找到对应的用法。

我个人的评价是：Artifacts 是 Claude Code 从个人工具走向团队工具的一个重要里程碑。它没有给 AI 本身增加新的分析能力，但它解决了 AI 能力如何在团队里传播和共享的问题，让 AI agent 的工作成果第一次能够以自动化的方式无缝融入团队的协作流程。这是一个务实的、方向正确的功能，对于正在推动团队引入 AI 工具的工程师和技术管理者来说，非常值得关注。
