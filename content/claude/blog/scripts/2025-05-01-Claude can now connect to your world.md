## Script

大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要讲的这篇文章,标题是 Claude can now connect to your world,我把它翻译成"Claude 现在可以连接你的整个工作世界"。这篇文章发表于二零二五年五月一日,来自 Anthropic 的官方博客。

掐指一算,这篇文章距离我们今天录制的日期,二零二六年六月二十一日,已经过去了差不多十四个月。说实话,在 AI 这个行业里,十四个月已经是一段相当长的时间了,差不多可以算是一代产品的生命周期。所以听这期节目的时候,大家要带着一个历史的眼光,把它当成一份重要的考古资料来看,理解 Claude 这条产品线是怎么一步步走到今天的,而不是把里面的功能状态当成当下的最新情况。我猜到了我们录制这期节目的时间点,这套体系应该已经经历了好几轮迭代,集成的应用数量、Research 的能力、甚至 MCP 协议本身的成熟度,都可能发生了不小的变化。但正因为这样,回头看这篇奠基性的文章,反而更有意思,因为它记录了一个关键的转折点。

我们先说说这篇文章要解决的问题,以及它发布时的行业背景。在二零二五年五月之前,大语言模型最大的痛点之一,就是它知道很多通用知识,但是不知道你公司内部发生了什么。你问它你们团队这个项目进度怎么样了,它完全没办法回答,因为它压根连不上你的 Jira,连不上你的 Confluence,连不上你的客服系统。这种"知识丰富但信息孤岛"的状态,几乎是当时所有大模型产品的共同困境。用户必须自己手动把上下文复制粘贴进对话框,效率很低,而且很容易漏掉关键信息。

为了解决这个问题,Anthropic 在二零二四年十一月,也就是这篇文章发布的半年前,推出了一个叫 Model Context Protocol 的东西,简称 MCP,中文我们可以理解为"模型上下文协议"。它本质上是一套开放的通信标准,目的是让 AI 应用能够以一种统一的、标准化的方式,去连接外部的工具和数据源。你可以把它想象成 AI 世界里的 USB 接口。在 USB 出现之前,每个外部设备都有自己专属的接口和驱动,鼠标一种插口,打印机一种插口,扫描仪又是另一种插口,非常混乱。USB 出现之后,所有设备统一标准,插上就能用。MCP 想做的就是这件事,它让任何一个工具,不管是项目管理软件,还是客服系统,还是数据库,都可以按照同一套协议,把自己的能力暴露给 AI 模型,而不需要每家都单独开发一套专属的对接方案。

不过,MCP 刚发布的时候,有一个很大的限制,就是它只能在 Claude Desktop 这个桌面客户端里使用,而且必须依赖本地服务器,也就是 local server。这就意味着,如果你想用 MCP 连接一个工具,你得自己在自己的电脑上跑一个服务进程,配置起来对非技术用户来说门槛不低,而且只能在你装了 Claude Desktop 的那台机器上生效,网页版完全用不了。这就是这篇文章要解决的核心问题:怎么把 MCP 从一个小范围的、桌面专属的极客玩具,变成一个人人都能用、随处都能用的基础设施。

好,这就引出了文章里最重要的第一个点,也就是 Integrations 这个新功能。它的核心突破,是把 MCP 的能力从本地服务器延伸到了远程服务器。什么叫远程服务器呢,简单说,就是这个 MCP 服务现在可以跑在云端,由开发者或者服务商统一部署和维护,用户不需要在自己电脑上装任何东西,只要在 Claude 里点一下连接、走一下授权登录的流程,就可以用了。而且因为它跑在云端,所以不管你是用网页版,还是用桌面版,这个连接都是生效的,不再局限于某一台设备。这一下,就把 MCP 从"少数极客的本地实验"变成了"任何人都能点点鼠标完成的标准操作",这是一个很关键的从零到一的转变,有点像是把一个需要自己组装的电器,变成了插上电就能用的成品。

文章里提到,首发的时候,Anthropic 已经联合了十个比较有代表性的服务商,做出了现成的集成,包括 Atlassian 旗下的 Jira 和 Confluence,自动化工具 Zapier,基础设施服务商 Cloudflare,客服平台 Intercom,项目管理工具 Asana,支付公司 Square,错误监控工具 Sentry,支付巨头 PayPal,研发管理工具 Linear,以及金融数据服务 Plaid。后续还会有 Stripe、GitLab、Box 这些公司加入进来。这个名单其实挺有意思,你会发现它覆盖了几个典型的工作场景,项目管理、客服支持、支付金融、基础设施运维,基本上把一个公司日常运转最核心的几条线都覆盖到了。

我们拿几个例子来具体说说,这样大家能更直观地感受到这个功能到底改变了什么。

先说 Zapier。Zapier 本身是一个自动化中枢,它已经连接了几千个不同的应用,通过预先设置好的工作流,把不同软件之间的数据自动串联起来。比如说你在 HubSpot 里有一条新的销售线索,Zapier 可以自动把它同步到你的邮件系统,再自动创建一条待办任务。现在 Claude 连上了 Zapier 的集成之后,你只需要用自然语言跟 Claude 说一句话,比如说帮我把这周的销售数据拉出来,再结合我日历上的会议安排,准备一份会议简报,Claude 就可以直接调用 Zapier 背后连接的成百上千个应用,自动完成这一整套操作。这个意义在哪里呢,以前你想做自动化,得自己去 Zapier 后台一步一步搭工作流,现在你只需要用聊天的方式描述你想要什么,Claude 帮你把这件事翻译成具体的自动化动作。这其实是把"配置自动化"这件事,变成了"对话式表达需求",门槛一下子降低了很多。

再说 Jira 和 Confluence 这个组合,这个对我们做软件工程的人来说应该感受最深。Jira 是大家用来管理任务和缺陷的工具,Confluence 是用来写文档和知识库的工具。以前你要是想让 AI 帮你梳理一下某个项目的进度,你得自己一条一条把 Jira 里的任务复制出来,再贴给模型。现在连上集成之后,Claude 可以直接读取你 Jira 里的工作项状态,直接帮你批量创建新的任务,也可以直接帮你批量生成或者总结多篇 Confluence 文档。文章里提到的一个场景是,Claude 可以帮你一边规划新产品,一边把规划过程中产生的任务自动拆解成 Jira 里的工单,同时把产品文档同步写进 Confluence。这种"一边讨论一边落地"的体验,对项目经理和技术负责人来说,效率提升是非常直观的。

第三个例子是 Intercom,这个案例我觉得特别值得展开讲一下,因为它展示了 MCP 协议双向连接的特性。Intercom 是一个客服平台,它自己也有一个叫 Fin 的 AI 客服代理。文章提到,Fin 现在本身也变成了一个 MCP 客户端,这意味着它不仅可以被 Claude 调用,它自己也可以反过来去调用别的 MCP 服务。举个具体场景,如果有用户在 Intercom 里反馈说软件出了个 bug,Fin 可以直接自动在 Linear 里建一个工单去跟踪这个 bug。而你作为开发者,可以直接跟 Claude 对话,让它去分析 Intercom 里历史上的客服对话记录和用户属性数据,找出问题出现的模式,帮你定位 bug 的根因。这就把"用户反馈,到问题归档,再到问题修复"这一整条链路,压缩在了一次对话里完成,而不需要你在 Intercom、Linear、还有各种调试工具之间来回切换标签页。这种把多个系统的上下文打通、在一个对话窗口里完成全链路操作的体验,我个人认为是这次更新里最体现 MCP 价值的一个场景,因为它不是简单的查数据,而是真正实现了跨系统的协同行动。

讲完 Integrations,我们再讲文章里的第二个重点,就是 Research 功能的升级,中文我们可以理解成"深度调研"功能。在这次升级之前,Claude 已经有一个基础版的 Research 能力,支持搜索网页和 Google Workspace,也就是谷歌的文档、邮件、日历这些办公套件。这次升级之后,Claude 多了一个进阶模式,你只要在界面上把 Research 这个按钮打开,Claude 就会把你的问题拆解成多个子问题,然后针对每一个子问题分别去深入调研,横跨成百上千个内部和外部的信息来源,最后再把所有调研结果汇总成一份完整的报告。

这里有个时间维度的细节我觉得值得专门说一下。文章里讲,大多数报告会在五到十五分钟之内完成,但是对于比较复杂的调研任务,Claude 可能会花到四十五分钟才给出结果。乍一听,四十五分钟好像挺长的,但是你换个角度想,这其实是在模拟一个真实的研究员或者分析师的工作方式,深入调研本来就不是秒回的事情。而且文章特别强调了一句话,说这种调研工作,如果让人自己手动来做,本来需要花费数小时的时间。所以四十五分钟看起来长,实际上是把原本几个小时的人工劳动压缩到了四十五分钟以内,这个对比才是关键。

另外一个很重要的细节是,因为这次同时上线了 Integrations,Research 的信息来源也跟着扩展了。原来只能搜网页和 Google Workspace,现在只要你连接了任何一个 Integration,Claude 的深度调研就可以把这个应用里的数据也纳入搜索范围。这意味着如果你把公司的 Jira、Confluence 都接上了,Claude 做调研的时候就不再只是一个泛泛而谈的网络搜索引擎,而是一个真正了解你公司内部信息的分析师。

还有一点我觉得对工程师和做严谨内容工作的人特别重要,就是引用来源这件事。文章明确提到,当 Claude 在报告里引用某个来源的信息时,它会提供清晰的引用,直接链接回原始材料。这一点看起来不起眼,但其实是建立信任的关键机制。我们都知道大模型有时候会一本正经地编造内容,这种现象业内一般叫幻觉。如果一份调研报告没有任何引用来源,你很难判断里面的内容是真实信息还是模型自己编的。有了清晰的引用链接,你可以随时点进去核实,这就把"我说了什么"和"我从哪里看到的"两件事分开了,大大提升了报告的可验证性。

文章最后还提到一个看起来不大,但其实影响面很广的更新,就是网页搜索功能向全球所有付费用户开放。这个听起来简单,但是网页搜索这件事,是很多复杂功能的地基。没有网页搜索能力,模型只能依赖训练数据里截止某个时间点的知识,完全没办法回答"今天发生了什么"这类问题。网页搜索全球开放,意味着不管你身处哪个国家,只要是付费用户,都能让 Claude 去查最新的实时信息,这是后续做深度调研、做时效性分析的基础能力。

讲到这里,我们也提一下文章末尾那个小小的更新说明。文章原文是在二零二五年五月一日发布的,但是后面有一条更新备注,时间是二零二五年六月三日,内容是 Integrations 和 Research 这两个功能,从原来只对 Max、Team、Enterprise 这几个比较贵的方案开放,扩展到了 Pro 方案,也就是说覆盖到了更广泛的个人付费用户。同时网页搜索这个时候已经是全球所有 Claude 方案都能用了,不再局限于付费用户。这个细节其实也反映了一个挺典型的产品节奏,大公司新功能往往先在高端方案里小范围验证,跑顺了之后再逐步下沉到更大众的用户群体。

好,讲完了文章的核心内容,我们来聊聊这些东西对我们这些做软件工程和 AI 相关工作的听众,到底意味着什么,我们能怎么上手用,又有哪些坑需要提前避开。

第一个实际的启发是,如果你所在的团队已经在用 Jira、Confluence、Linear、Asana 这些工具,而且你已经在用 Claude 做日常的技术沟通和文档撰写,那么把这些工具连接起来,几乎是一个立竿见影的效率提升点。你可以试着把团队的项目管理工具接进去,然后让 Claude 帮你做每周例会前的进度汇总,或者帮你把零散的讨论自动整理成结构化的工单。这种场景不需要你写一行代码,只是把已有工具用对话的方式串起来。

第二个启发,是关于开发者自己动手的部分。文章提到开发者只需要大概三十分钟,借助官方文档,或者借助像 Cloudflare 这样提供现成解决方案的服务商,就可以搭建出一个自己的 MCP 服务器,这个服务器会内置 OAuth 身份验证、传输层处理,还有集成部署能力。如果你的团队有一些内部专属系统,比如自研的工单系统,或者内部的知识库,理论上你完全可以参考这套思路,自己写一个 MCP 服务器,把这个内部系统也接入 Claude 里。这对很多中大型企业来说,其实是一个值得投入的方向,因为一旦做成,等于给全公司的所有人提供了一个统一的、可以跟内部系统对话的入口。

第三个要提醒大家注意的,是安全和权限的问题。一旦你把 Claude 接到了你的 Jira、你的支付系统、你的客服系统,这意味着 Claude 现在有能力读取,甚至在某些场景下修改这些系统里的真实数据。这就要求我们在授权的时候,一定要仔细想清楚这个集成到底被授予了哪些权限范围,有没有可能误操作,比如说不小心让模型在没有确认的情况下,直接改动了生产环境里的工单状态,或者在客服系统里发出了不该发的回复。我个人认为,在企业场景里大规模铺开这类集成之前,一定要先有清晰的权限边界设计,哪些操作允许 AI 自动完成,哪些操作必须经过人工确认,这一点比技术对接本身更重要。

第四个值得思考的点,是关于深度调研功能的使用边界。四十五分钟的深度调研听起来很强大,但是它本质上还是依赖模型的检索和总结能力,如果你拿它去做特别专业、特别需要领域权威背书的研究,比如说法律意见或者医疗判断,引用来源固然重要,但是你仍然需要有专业背景的人去复核,而不能完全把决策权交给一份自动生成的报告。我觉得比较合理的定位,是把它当成一个能大幅压缩信息收集和初步整理时间的助手,真正的判断和决策,还是要留给人。

最后,我想多说一句关于时效性的思考。我们这期节目讨论的是一篇一年多以前的文章,在 AI 这个领域,一年多的时间已经足够发生很多变化了。MCP 协议在这一年多里,据我所知已经被越来越多的厂商采纳,逐渐变成了一个跨厂商的行业标准,而不仅仅是 Anthropic 自家的东西。所以听这期节目的朋友,如果你打算现在就去尝试这些功能,我建议你直接去 Claude 的官方帮助文档查一下最新的集成列表和使用方式,因为这篇文章里提到的十个首发伙伴,到今天可能早就不是全部了,实际可用的集成数量,大概率已经比文章里写的多了不少。

好了,我们来给这篇文章做个总结。这篇文章的核心,其实就是两件事,一是 Claude 通过 Integrations,把 MCP 协议从一个本地化的极客工具,变成了一个云端化、人人可用的连接基础设施,让 AI 真正能够伸手触碰到你工作中用的每一个具体软件。二是 Research 功能的升级,让 Claude 不仅能聊天,还能像一个真正的分析师一样,花时间去做深入的、跨系统的调研,并且给出有据可查的报告。我个人认为,这篇文章标志的不是某一个具体功能的发布,而是 AI 助手这个产品形态本身的一次范式转移,它正在从一个被动回答问题的聊天框,变成一个主动连接你整个工作环境、能够采取行动的协作者。这个方向,我觉得在未来很长一段时间里都会是行业的主旋律。
