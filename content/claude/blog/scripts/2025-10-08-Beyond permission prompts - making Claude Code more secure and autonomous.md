## Script

大家好,欢迎收听 Claude FM,今天我们要聊的这篇文章,标题是 Beyond permission prompts: making Claude Code more secure and autonomous,我把它翻译过来大概是这个意思,超越权限弹窗,让 Claude Code 更安全也更自主。这篇文章发表于二〇二五年十月八日,距离我们今天录这期节目,差不多过去了八个多月。这里我要先提个醒,八个多月在 AI 这个行业里,已经算是一段不短的时间了,所以我们今天聊的这些功能,很可能在你现在用的 Claude Code 里已经是标配,甚至已经迭代了好几个版本。但是这篇文章里讲的设计思路,文件系统隔离也好,网络隔离也好,这些底层的安全模型,到今天依然是理解 Claude Code 安全机制的一个很好的切入点,所以我们还是值得花时间把它讲透。

先说说这篇文章出来的背景。二〇二五年那会儿,Claude Code 已经在很多工程师的日常工作流里站稳了脚跟,大家用它写代码,调试程序,跑测试,甚至直接在终端里让它帮你改好几个文件。但凡是用过这种 agent 工具的人都知道,有一个特别现实的问题,就是权限弹窗太多了。你让 Claude 帮你跑个命令,它会先问你,这个命令我能跑吗。你让它改个文件,它又问,这个文件我能改吗。一开始你可能还很认真地看一下,确认一下,但是用上一两周之后,你就会发现自己越来越不耐烦,看到弹窗就直接点同意,根本不细看内容。这种现象在安全圈子里有个说法,叫审批疲劳,就是审批的动作太频繁了,人反而会失去警觉性,最后审批这个机制本身就形同虚设了。这其实是个很讽刺的局面,本来设计权限确认是为了安全,结果因为太频繁,反而让用户对真正危险的操作也麻木地点了同意。

那这个问题在 Claude Code 这种代码助手身上,风险有多大呢。我们设想一下,Claude Code 能做的事情包括读你的代码库,改多个文件,执行终端命令去验证自己写的代码对不对。这权限是相当大的。如果出现一种叫提示注入的攻击,简单说就是有人把恶意指令藏在某个文件,某个网页,或者某段你让 Claude 去读取的内容里,然后 Claude 在执行任务的过程中不小心读到了这段被植入的指令,把它当成了真正的任务指令去执行,那后果可能就很严重了。比如说,Claude 可能被诱导去删除你本来没打算删除的文件,或者偷偷把你电脑上的敏感信息,像是 SSH 密钥这种东西,发送到攻击者控制的服务器上。这不是危言耸听,这是所有给 AI agent 赋予系统访问权限之后,都要面对的一个真实风险。所以 Anthropic 在这篇文章里想解决的问题就很明确,怎么样在不牺牲效率的前提下,把这种风险真正地管住。

那么文章给出的答案是什么呢,答案就是沙箱,英文是 sandbox。这个词我们玩游戏或者做软件测试的人应该都很熟悉,沙箱的意思就是划出一个隔离的空间,在这个空间里面怎么折腾都不会影响到外面的世界。文章里有一句话我觉得特别精炼,它说传统的权限确认模式,是逐次审批,而沙箱模式,是预先划定边界。这两者的差别,我打个比方,逐次审批就像是你雇了一个新员工,这个员工每做一个动作,哪怕是从文件柜里拿一张纸,都要跑过来问你能不能拿,你点头了他才敢拿。这种管理方式听起来很安全,但实际上效率极低,而且因为问得太频繁,你作为管理者很快就会进入自动点头模式,根本不会真的去想这张纸该不该拿。沙箱模式则不一样,它更像是你提前跟这个新员工说清楚,你可以自由进出三号会议室和你自己的工位,文件柜里属于你这个项目的那几个抽屉你随便用,但是财务室的门是锁着的,你进不去,外部的网络只能连这几个指定的合作方,别的连不上。在这个边界划定好之后,员工在边界内做什么事情,都不用再来一趟一趟地问你了,他可以自主地、连续地推进工作,效率自然就上来了。而你作为管理者,也不用一直被打断,你只需要在最初把边界划清楚,后面就可以放心地让他自己干活。

这篇文章里讲,真正有效的沙箱,必须同时具备两种边界,一种是文件系统隔离,另一种是网络隔离。这两个为什么要绑在一起讲,我觉得这是这篇文章里最值得深挖的一个点。我们先说文件系统隔离,它的意思是 Claude 只能访问或者修改你提前指定的那些目录,目录之外的东西它碰不到。这个很好理解,就是把活动范围限定在你的项目文件夹里,系统文件、其他项目的文件、你的个人文档,这些都在边界之外,Claude 没有资格去动。这能直接防住前面提到的那种风险,就是被提示注入之后,Claude 跑去修改或者删除一些本来不该碰的敏感系统文件。

但问题是,如果你只做了文件系统隔离,而没有做网络隔离,会出现什么情况呢。文章里举的例子非常关键,它说如果没有网络隔离,一个被攻陷的 agent 完全可以把你的敏感文件,比如 SSH 密钥,直接通过网络发送出去,泄露给攻击者。也就是说,即便你把文件系统锁得再死,只要网络这条路是开放的,攻击者依然可以让 Claude 把它读取到的任何东西,打包发到外部服务器上,这等于绕过了文件系统隔离的意义。反过来,文章也讲了另一面,如果你只做了网络隔离,没有做文件系统隔离,那一个被攻陷的 agent 也可以轻易地从沙箱里逃出去,然后获得网络访问能力。这句话我理解的意思是,如果 Claude 在文件层面有比较大的自由度,它有可能修改一些配置文件,或者篡改一些可执行脚本,从而绕开网络层面的限制,相当于是从文件这个口子,反向打开了网络这个口子。所以这两道防线必须同时存在,缺一个,另一个的安全效果都会大打折扣。这就跟现实生活中我们做安全防护一样,你家装了防盗门,但是窗户没锁,那防盗门基本上是摆设。必须门窗都锁好,才能算是真正把家保护起来。Anthropic 在设计 Claude Code 的沙箱时,就是同时抓住了这两道门。

讲完了原理,我们来看文章里介绍的两个具体落地的功能,这是这篇文章的重头戏。第一个叫 sandboxed bash tool,中文我们可以叫沙箱化的命令行工具。这个功能当时是以研究预览的形式发布的,核心思路是让你能够明确定义,Claude 这次任务可以访问哪些目录,可以连接哪些网络主机,而且这一切不需要你自己去搭建和管理一个完整的容器环境。这点其实挺重要的,因为如果是要手动搭建容器,对很多工程师来说门槛和负担都不小,而 Anthropic 把这个能力做成了一个开箱即用的运行时,任何进程,任何 agent,任何 MCP 服务器,都可以拿来直接套用这套沙箱机制,而且当时这部分代码是开源发布的,作为一个研究预览项目放在了 GitHub上。这意味着不光 Claude Code 自己能用,其他团队如果想给自己的 agent 加上类似的安全边界,也可以直接拿这套技术去改造和集成,而不用从零造轮子。

具体到 Claude Code 这个产品里,这套沙箱运行时是用来包裹住 bash 工具的,也就是 Claude 平时用来执行终端命令的那个工具。一旦命令运行在这个沙箱里,它就只能在你设定好的边界内活动,这样一来,这些命令默认就变得更安全,同时因为边界已经提前划好了,所以需要用户手动确认的次数会大幅减少,Claude 就可以更自主地连续推进任务。如果 Claude 在执行过程中,真的尝试去访问边界之外的东西,系统会立刻通知你,这时候你可以自己判断,要不要临时放行这一次访问。这个设计我觉得很巧妙,它没有把权限确认这个机制完全废掉,而是把它从一个高频骚扰的东西,变成了一个低频但是更有意义的信号。换句话说,以前是有事没事都来问你,现在是只有真正触碰到边界的时候才来问你,这样你看到弹窗的时候,会更认真地去对待,因为你知道这次弹窗是真的有点不一样,不是日常的例行公事。

那这套沙箱具体是怎么在技术上实现的呢,文章提到,它是建立在操作系统级别的底层机制之上的,在 Linux 系统上用的是 bubblewrap,在 macOS 系统上用的是 seatbelt,这两个都是各自操作系统提供的、用来做进程隔离和权限限制的原生技术。用操作系统级别的机制来强制执行这些限制,有一个很大的好处,就是它不光能管住 Claude Code 自己直接发起的操作,还能管住任何由这条命令派生出来的子脚本,子程序,子进程。这个细节很关键,因为很多攻击的手法,恰恰是利用了一层一层调用下去之后,外层的限制可能管不到内层的子进程。如果安全限制只是停留在应用层,做得不彻底的话,攻击者完全可以通过多绕一层、多套一层脚本的方式把限制绕过去。但如果限制是在操作系统这一层强制执行的,那不管你套了多少层调用,操作系统始终知道这个进程树最终是从哪个沙箱里发出来的,该限制的地方一个都跑不掉。

具体到文件系统隔离这块,这个沙箱的设计是允许 Claude 对当前工作目录有读写权限,但是阻止它修改工作目录之外的任何文件。这个设计其实很贴近日常开发场景,你在做一个项目的时候,Claude 当然需要能自由地读写这个项目里的代码文件,这是它干活的基本前提,但是你的其他项目,你的系统配置,你的个人文件,这些都不应该被波及,工作目录这个边界划得相当合理。

网络隔离这部分的实现方式也挺有意思,文章里讲,这个沙箱只允许通过一个 unix domain socket 连接到一个运行在沙箱外部的代理服务器来访问互联网。也就是说,Claude 在沙箱里面,没有直接连外网的能力,它所有的网络请求,都必须经过这个外部代理服务器中转。这个代理服务器会对域名做限制,只有被允许的域名才能连通,而且如果 Claude 试图访问一个新的、之前没出现过的域名,代理服务器会触发用户确认的流程,让你来决定是不是要放行这个新域名。如果你对安全性的要求更高,文章里还提到,你甚至可以自定义这个代理,让它对出站流量执行任意你想要的规则,比如说更细粒度地控制哪些数据可以发出去,哪些不能。而且这两个维度,文件路径和网络域名,都是可以配置的,你可以根据自己的项目需要,灵活地决定哪些路径允许访问,哪些域名允许连接。

这套机制带来的安全效果,文章里有一句话讲得很到位,它说沙箱保证的是,即便提示注入真的成功了,它的影响也会被完全隔离住,不会波及到用户整体的安全。换句话说,以前我们担心的是,只要提示注入这一关被攻破了,后面 Claude 想干什么就能干什么,这是一种全有或全无的风险模型。而现在,即便提示注入这一关被攻破了,Claude 依然被困在沙箱里,它没办法偷走你的 SSH 密钥,也没办法联系外部的攻击者服务器。这相当于把风险的上限,从无限大,降低到了一个可以预先评估和控制的范围。这篇文章里给出的具体使用方式也很简单,你只需要在命令行里运行 claude 加上 dash dash sandbox 这个参数,就可以启用这个沙箱模式。

讲完了第一个功能,我们再说说第二个,叫 Claude Code on the web,可以理解成网页版的 Claude Code,或者说云端版的 Claude Code。这个功能解决的是另外一类场景,前面讲的 sandboxed bash tool,主要还是你在自己的电脑上,本地运行 Claude Code 的时候用的沙箱机制。但是如果你想让 Claude Code 在云端跑,比如说你想让它在后台长时间地处理一个任务,或者你想从手机、从浏览器这种轻量的客户端,就能远程操作 Claude Code,这时候就需要一个云端的隔离环境。Claude Code on the web 的设计思路是,每一个会话都运行在一个隔离的云端沙箱里,在这个沙箱里 Claude 对它自己的服务器有完整的访问权限,可以自由地干活,但同时这个沙箱被设计成,确保像 git 凭证或者签名密钥这种敏感凭证信息,永远不会进入到这个沙箱环境里面。这句话翻译过来就是,即便这个沙箱被攻破了,攻击者拿到的也只是一个干净的、没有任何敏感密钥的运行环境,真正有价值的凭证信息,根本就不在这个房间里,自然也偷不走。

那 Claude 在云端总要跟代码仓库打交道吧,要拉代码,要提交代码,这些操作需要身份认证,凭证又不在沙箱里,这是怎么做到的呢。文章里给出了具体的实现方式,Claude Code on the web 用了一个自定义的代理服务,专门用来透明地处理所有跟 git 相关的交互。在沙箱内部,git 客户端是用一个专门构建的、范围受限的凭证,去向这个代理服务发起身份认证的,这个凭证本身的权限是被严格限定的。然后这个代理服务会去校验这个凭证,同时也会校验这次 git 交互的具体内容,比如说确认这次推送的目标分支,是不是你提前配置好允许推送的那个分支。校验通过之后,代理服务才会附上真正用来跟 GitHub 通信的认证令牌,把请求发送出去。这个设计的精妙之处在于,沙箱里面的 Claude,自始至终都没有接触过那个真正能够操作你 GitHub 仓库的高权限令牌,它手里拿到的,只是一个范围很窄、而且经过代理服务层层校验的临时凭证。这相当于在 Claude 和你真正的代码仓库之间,加了一层有判断力的中间人,这个中间人知道什么该放行,什么不该放行,而 Claude 自己根本摸不到那把真正的钥匙。

好,讲完了这两个核心功能,我们来聊聊这些东西对我们实际工作意味着什么,这部分我想多展开讲一讲,因为我觉得这才是这篇文章真正落到我们日常工程实践里的地方。

首先一个直接的影响是,如果你是天天用 Claude Code 写代码的工程师,沙箱模式能明显改善你的使用体验。我们前面讲过审批疲劳这个问题,我相信很多用过类似 agent 工具的朋友都深有体会,一开始用的时候你会很谨慎地看每一个权限请求,但是用着用着,你就会变成机械式地点同意,这其实是一种很危险的状态,因为真正危险的那一次请求,很可能就藏在你机械点同意的众多请求里面,被你一并放过去了。开启沙箱模式之后,你日常的大部分操作,读写当前项目目录,执行项目相关的命令,这些都不会再触发弹窗,因为它们本来就在你划定的安全边界之内。只有当 Claude 真的尝试去碰边界之外的东西,你才会收到提示。这意味着你收到的每一次提示,信息密度都更高了,你会更认真地去看,这其实是把人类注意力这种稀缺资源,用在了真正值得关注的地方,而不是浪费在大量重复的、本来就该被允许的日常操作上。

第二个值得说的点是,如果你是负责团队工程效率或者安全策略的人,比如说你是技术负责人,或者是平台团队的同学,沙箱这个机制给了你一个新的工具,去管理团队里 AI agent 的使用边界。你可以提前针对不同的项目,配置好哪些目录是可以让 Claude 自由读写的,哪些网络域名是业务上确实需要访问的,把这些配置沉淀下来,变成团队的标准配置模板。这样一来,不管是哪个工程师在用 Claude Code 处理这个项目,他享受到的都是同一套被验证过的安全边界,而不是每个人凭自己的感觉去点权限弹窗,造成团队内部安全水位参差不齐的情况。这对于稍微大一点的工程团队来说,其实是个挺实际的好处,因为安全这件事,最怕的就是靠个体的警觉性去兜底,人是会累、会疲劳、会犯错的,而提前配置好的边界,不会因为某个人今天状态不好就松懈下来。

第三个我想讲的是给做 agent 开发或者 MCP 服务相关工作的朋友。文章里特别提到,这套沙箱运行时是开源的,可以用来包裹任意的进程、agent,还有 MCP 服务器。这意味着,假如你自己在构建一套基于大模型的自动化工具,不管是代码助手,还是数据处理 agent,还是别的什么自动化流程,你都可以考虑把这一套文件系统隔离加网络隔离的思路,移植到你自己的系统里,而不需要从头去研究怎么在 Linux 和 macOS 上分别去调 bubblewrap 和 seatbelt 这些底层接口。这对中小团队来说意义比较大,因为安全基础设施这种东西,从头自己写,成本是很高的,而且很容易写得有漏洞,直接复用一套已经被验证过、而且是大公司在生产环境里真实用过的方案,性价比会高很多。当然这里我要提个醒,具体这个开源项目现在叫什么名字,在哪个仓库,后续有没有改名或者迁移,这篇文章发表的时间点距今已经过去八个多月了,如果你真的打算去找这个项目落地使用,建议直接去查最新的官方文档,确认一下现在的名称和使用方式,不要完全照着这篇老文章里的描述去找,因为这种细节是最容易随着版本迭代发生变化的。

第四个点,我想讲讲云端协作场景下,Claude Code on the web 这种设计,对于那些需要跟外部协作者或者跟代码仓库做交互的团队意味着什么。很多团队在引入 AI agent 写代码的时候,最担心的问题之一就是凭证泄露,你给 agent 配置了能推送代码的权限,万一这个 agent 出了问题,或者被恶意注入了指令,它是不是有可能拿着你的高权限令牌去干一些你不希望的事情,比如说推送到一个不该推送的分支,或者把仓库的访问权限信息泄露出去。Claude Code on the web 这种把凭证彻底排除在沙箱之外,通过代理服务做范围校验和令牌注入的设计思路,其实给我们提供了一个很好的参考模板,就是不要把高权限的凭证直接交给 agent 本身去掌管,而是让 agent 拿着一个范围很窄的临时凭证,真正的高权限操作,交给一个有规则、可审计的中间层去完成。这种思路不光适用于 git 操作,其实也适用于任何你想让 AI agent 去操作的外部系统,数据库也好,云服务的控制台也好,本质上都可以套用类似的代理加范围限定凭证的设计。

第五个我想提一下实际操作中需要注意的坑。沙箱机制虽然好,但它不是万能的,它解决的是把风险关在一个边界里,而不是消灭风险本身。也就是说,如果你把整个项目目录设置成可写,而你的项目目录里恰好放了一些本来不该放在那里的敏感信息,比如说你不小心把一个密钥文件放在了项目根目录下,没有加到 gitignore 里,那这个沙箱是保护不了你的,因为按照你的配置,这个目录本身就是在安全边界之内的,Claude 是有权限去读写它的。所以沙箱这个机制提醒我们的,其实是另外一件更基础的事情,就是平时管理项目的时候,本来就应该把敏感信息和代码分开存放,该用环境变量的用环境变量,该用专门的密钥管理服务的用密钥管理服务,不要把什么都堆在项目目录里。沙箱是给你的项目划了一个边界,但是边界内部的卫生情况,还是要靠你自己去维护。另外网络隔离这部分,如果你为了方便,把很多域名都加到了允许列表里,这个允许列表本身的维护也是有成本的,你加得越宽松,等于你给沙箱开的口子越大,这跟门锁本身没坏,但是钥匙发得太多是一个道理,所以建议大家在配置允许访问的域名列表时,尽量按照实际需要来加,而不是图省事一股脑放开。

最后我想从整体上做个总结。这篇文章其实传递了一个挺重要的理念转变,就是面对越来越自主的 AI agent,安全这件事不能再靠人不停地盯着、不停地点头来保证了,因为人的注意力天然是有限的,而且会疲劳。更可持续的做法,是把安全这件事,从行为层面的逐次审批,挪到架构层面的边界设计上去。你把边界划清楚了,Claude 在边界之内可以非常自由、非常高效地去干活,而你不需要时时刻刻盯着每一个动作,只有真正触碰边界的事情,才需要你来做判断。我个人觉得这个思路特别值得我们这些做工程的人去借鉴,不光是用在 Claude Code 这个产品上,我们自己在设计任何涉及自动化、涉及权限分配的系统时,都可以想一想,我们现在是在靠人不停地确认来保证安全,还是已经把边界提前设计好了。如果是前者,那随着系统规模和使用频率的增长,审批疲劳这个问题迟早会找上门来,而后者才是一个真正能够长期撑得住的方案。这篇文章发表到现在已经过去了八个多月,AI agent 这个领域发展得很快,具体的产品功能和细节,可能已经跟文章里描述的不完全一样了,但是文件系统隔离加网络隔离这种双重边界的设计思路,我觉得在可预见的未来,依然会是构建安全 agent 系统的一个基本范式。好,这期就先聊到这里,我们下期再见。
