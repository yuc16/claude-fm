## Script

今天这期节目，我们来聊一篇来自 Anthropic 官方博客的文章，标题叫做"The evolution of agentic surfaces: building with Claude Managed Agents"，我把它翻译成"智能体构建面的演化：基于 Claude Managed Agents 进行开发"。这篇文章发表于二零二六年六月十日，距离我们录这期节目只有九天，可以说是最新鲜的一手资料。

为什么我觉得这篇文章值得专门做一期节目？它不是在发布某个新模型，也不是在公布什么 benchmark 分数，而是在系统性地回答一个困扰几乎所有 AI 工程团队的核心问题：把 AI agent 推进生产环境，为什么这么难？Anthropic 正在做什么来改变这个局面？如果你现在正在考虑把 agent 引入你们的系统，这篇文章里有很多值得认真思考的工程经验。

在我们开始之前，先介绍一下这篇文章的作者背景，这很重要。文章由 Gagan Bhat 和 Isabella He 写成，他们都是 Anthropic Applied AI 团队的技术人员。Applied AI 团队是 Anthropic 内部一支非常特别的队伍，他们同时工作在产品、研究和客户三个维度的交叉点上：要了解最新的模型能力，要懂得真实生产环境的工程挑战，还要持续跟各行各业的客户深度合作。所以这篇文章里的观察，不是凭空想象出来的理论推演，而是建立在大量真实工程案例上的总结。这也是我觉得它特别值得读的原因。

好，我们先来铺垫背景。

在 AI 应用开发领域，有一个现象被反复提及，只要你认真尝试过把 AI 能力引入工程系统，你一定对这种感受很熟悉：搭一个 demo 很快，快到令人兴奋。一两天，你就能有一个在演示环境里表现出色的东西，给人留下深刻印象。但是，要把它真正推进生产环境，往往要花三个月、六个月，甚至更长时间。

更让人沮丧的是，在这段时间里，大量的工程精力不是花在让 AI 更聪明上，而是花在基础设施上：进程怎么管理、会话状态怎么存储、权限怎么控制、凭证怎么保护、出了问题怎么排查、流量上来了怎么扩展。这些工作做完了，你的产品并没有变得更有竞争力，你只是让系统能正常地跑起来、跑稳。这种"做了很多，但没做有价值的事"的感觉，是很多工程师在 AI 应用开发早期阶段都体会过的。

这种落差在 agent 场景下尤其明显。普通的 LLM 调用相对简单：你发一个请求，等一个响应，整个生命周期在几秒内完成，要么成功要么失败，失败了重试就好。但 agent 不一样，它要执行多步骤任务，自主调用外部工具，操作真实的文件系统或者代码库，在多个工具之间协调。这个过程可能持续几分钟、几十分钟甚至几个小时。任何一个环节出了问题，你都需要能够追溯和调试，弄清楚 agent 在哪个决策点走偏了、为什么走偏。

Anthropic 的 Applied AI 团队在跟客户合作的过程中，反复看到同一个规律：对大多数团队来说，基础设施才是原型和生产之间真正的分水岭。他们不缺 AI 能力，Claude 本身足够强；他们缺的是把这个能力可靠地运行在生产环境里所需要的工程基础。团队在安全性、状态管理、权限控制、harness 调优上烧掉了大量开发周期，但这些工作并没有让他们的产品更有差异化竞争力，只是为了让系统能够正常跑起来。

正是为了解决这个问题，Anthropic 推出了 Claude Managed Agents。文章把它定义为"一套可组合的 API，用于构建和部署生产级 agent"，它把一个为性能调优的 agent harness 和生产级的基础设施打包在一起，让团队能够在几天而不是几个月内从原型走到上线。

要真正理解 Claude Managed Agents 是什么、为什么这样设计，我们需要先看一下 Anthropic 的整个 API 体系是如何一步步演化到今天这个形态的。这段历史很重要，因为它能帮你理解 Managed Agents 要解决的问题到底从哪里来。

故事从二零二三年说起。

那一年，Anthropic 向开发者开放 API，设计非常朴素——文本进、文本出。你发一个 prompt，Claude 返回一个 completion，所有的逻辑、所有的 harness、所有的基础设施都由你自己来搭。这种设计给了开发者最大的灵活性，你可以用 Claude 做任何事情。对于单轮任务来说，这完全够用：给一篇文章做摘要、给一条支持工单分类、把一段文本改写成另一种风格，这些任务都能在单次请求内完成，清晰简洁。

但随着时间推移，人们想交给 Claude 的任务越来越超出了单轮的边界。他们希望 Claude 不只是回答一个问题，而是把一件事从头做到尾：先查一些信息、根据查到的结果采取行动、观察发生了什么变化、然后决定下一步怎么走，如此往复。而且他们希望 Claude 能在他们已有的系统里工作——在代码库里修改文件、在内部 wiki 里查资料、在工单系统里更新状态——而不只是在一个孤立的对话框里说话。

用原始 API 来实现这个需求，就意味着你要自己搭一个 agent 循环：问模型该做什么、执行工具、把结果喂回去、再问模型、再执行，不断循环，直到任务完成。这个循环的逻辑本身不复杂，难的是维护它。因为模型在持续更新，Claude 的能力、行为和最佳实践都在变化，你的 harness 需要随之调整。而且这种调整不是一劳永逸的，它会随着每次模型更新产生新的需求。

后来，Anthropic 发布了 Claude Code，一个可以让 Claude 直接与你的代码库交互的智能编码工具，内部有一套专门设计的 harness：agent 循环、工具执行、子 agent 管理、上下文管理，以及让它成为一个有效 agent 所需的各种能力。开发者们看到了这套 harness 的效果，自然就想能不能用同样的机制来构建自己领域里的 agent。

于是 Anthropic 发布了 Claude Agent SDK，让开发者可以基于驱动 Claude Code 的同一套机制来构建自己的 agent，而不用从零维护一个自研的循环。对很多团队来说，这是一个真正的转折点：harness 已经针对 Claude 进行过调优，有成套的基础设施原语，而且会随着 Claude Code 的迭代持续改进。

但即便有了 Agent SDK，把 agent 推进真正的生产环境还是充满挑战。文章列举了一系列具体的问题，我来一一说说，因为每一个都代表着真实的工程痛点：

托管和扩展的问题，agent 的进程要跑在哪里，对于可能运行几个小时的长时间任务，进程能否保持存活，当使用量增长了怎么横向扩展；会话管理的问题，agent 的历史和运行进度存在哪里，运行中断了能否恢复继续，你能回去查看之前 session 里发生了什么吗；文件系统管理的问题，做真实工作意味着要产生成果，比如编辑代码、写文件，agent 在哪里操作这些文件，session 之间这些文件怎么管理；执行隔离的问题，agent 写的代码要在某个地方运行，如果代码有问题影响范围有多大，在生产里你愿意信任哪个边界来执行未知代码；凭证的问题，agent 需要访问你的系统，它如何获取这个权限而不把敏感信息暴露给它生成的代码；可观察性的问题，当 agent 自主工作了一个小时、做了些出乎预料的事情，你能重建它走过的每一步吗。

这些问题在 Agent SDK 里有部分答案，但随着越来越多的团队把 agent 从本地开发推进生产，加上模型能力在持续提升，agent 运行更长、执行更多代码、触碰更多系统，扩展性、安全性和隔离性方面的挑战变得越来越突出。Claude Managed Agents 就是为系统性解决这些生产级挑战而设计的。

好，背景和演化脉络说到这里，我们来深入看 Managed Agents 的架构设计。

文章用了一个非常精准的比喻：大脑和双手的分离。

先来说说"大脑和双手没有分离"是什么情况。在传统的 agent 架构里，harness，也就是负责调用 Claude、管理 agent 逻辑的"大脑"，和代码执行环境，也就是实际运行工具和代码的"双手"，通常放在同一个容器里。这种单体架构在开发阶段非常方便：所有东西都在一起，调试容易，修改方便。

但放进生产环境，这个设计就带来了几个严重的问题，而且这几个问题彼此缠绕，很难单独解决其中任何一个。

第一个问题是启动延迟。每次启动一个新的 agent session，都要先等容器启动完毕，Claude 才能开始工作。容器启动需要时间：拉取镜像、分配计算资源、初始化运行环境。对于那些根本不需要执行任何代码的 session，比如只是需要 Claude 分析一段文字然后给出结论的轻量任务，这个启动时间完全是浪费。用户感受到的就是：我提了一个请求，等了好几秒才看到第一个字出来，这种等待很容易让人失去耐心。

第二个问题是安全风险。当 harness 和代码执行环境在同一个容器里，凭证也在同一个容器里。数据库密码、API key、服务令牌，这些通常作为环境变量存在，而 Claude 生成的代码就在旁边运行。如果发生了 prompt injection 攻击，攻击者通过精心设计的输入，诱导模型去读取自己所处环境的变量，就可能把凭证泄露出去。你可以在同一个容器内设置很多防护措施，但这是一种被动防御的思路，从架构上就把凭证和代码放在一起，本身就是一种根本性的风险。

第三个问题是容错性差。运行 agent 的进程崩溃了，session 的状态可能跟着丢失。你要么接受"从头来过"的后果，要么自己实现一套复杂的检查点和恢复机制，而这套机制本身又需要设计、实现和维护。

Claude Managed Agents 通过把大脑和双手彻底分开，从架构层面一次性消解了这些问题的根源。

调用 Claude 的 harness 运行在一个地方，执行代码的沙盒运行在另一个完全隔离的地方，两者之间的连接是一个"session"——一个只能追加、不能修改的事件日志，记录每一次模型调用、每一次工具调用和每一次返回结果，而且这个日志存在所有运行进程之外，在服务端持久化。

这个架构的意义是：Claude 可以在沙盒容器启动之前就开始推理，对于不需要执行任何工具的 session，容器根本不需要启动。凭证存在一个与沙盒完全隔离的地方，沙盒里的代码看不到它们。整个 session 的事件日志在任何时间点都完整存在，进程死了 session 还在，想恢复就从最后一个事件往后接着跑。

基于这个架构，文章详细介绍了四个核心优势。

第一个核心优势是凭证安全。

有了大脑和双手的分离，凭证可以完全放置在沙盒之外。文章介绍了一个叫做 Vault 的功能，可以理解为"凭证保险柜"。MCP 服务器、命令行工具、GitHub 仓库等各类工具的访问令牌，都存在这个独立的保险柜里，通过信封加密的方式保护。当 agent 需要调用某个工具的时候，一个代理程序会向 Vault 发起一个经过签名的请求，Vault 验证签名、解密凭证、完成调用。整个过程里，Claude 生成的代码在沙盒里执行，沙盒里根本没有任何凭证可以接触。

文章说得很清楚，这不是"保护容器内的凭证不被读取"，而是"凭证根本就不在容器里"。这是两个完全不同层次的安全保证。前者是防御性的，依赖于防护措施的完整性；后者是结构性的，消除了整个攻击面的存在。即使 prompt injection 真的发生了，代码在沙盒里执行，沙盒里什么凭证都没有，攻击者没有可以窃取的东西。

另外，Managed Agents 的 Vault 还解决了一个多租户场景下很实际的工程问题：追踪哪个凭证是代表哪个最终用户来行动的。在一个 SaaS 应用里，可能有成千上万的用户，每个用户有不同的权限范围，agent 在帮不同用户工作时需要使用不同的凭证，管理这种追踪关系本来相当繁琐，Vault 把这个问题一并处理了。

第二个核心优势是低延迟。

在 Managed Agents 里，Claude 可以在沙盒容器还没启动的时候就开始推理。对于不需要执行任何工具的 session，容器甚至完全不需要启动，Claude 直接在 harness 里完成所有工作，节省了整个容器启动环节的时间。当 agent 确实需要执行代码时，容器在 Claude 思考的同时已经在后台并行启动了，到 agent 真正需要用到容器的时候，容器通常已经准备就绪。

根据 Anthropic 的内部测试数据，这个架构把首个 token 的响应时间，在中位数情况下缩短了大约百分之六十，在最慢的百分之五的情况下缩短了超过百分之九十。

我想在最慢情况下缩短百分之九十这个数字上多停一停。很多工程团队优化延迟时，主要关注中位数，中位数变好了就觉得完成了。但在实际的用户体验里，最慢情况往往比平均值更重要。那些等待时间最长的用户，感受最差的那一批人，他们的体验在这里缩短了九成以上。这不只是数字变好看，这是把一部分用户的体验从"令人沮丧地等待"提升到"完全可以接受"的量级差异。尤其对于企业级场景，偶发的极慢请求往往比稳定的平均延迟更影响用户信任感。

第三个核心优势是持久会话与可观察性。

文章里有一个重要的思维框架转变：Managed Agents 不是以"请求-响应"的方式来思考 agent 的工作，而是以事件流的方式来思考。一个 session 不是一个请求，而是一个持续的事件流：每一个模型调用、每一个工具调用和每一个返回结果，都按序追加到一个事件日志里，而这个日志不在任何运行中的进程里，它存在进程之外，持久化在服务端。

让我用一个具体场景来解释这有多重要。

假设你有一个 agent 在帮你进行一项复杂的代码重构任务，涉及三十个文件，预计需要四十分钟。做到第二十分钟、处理完第十五个文件的时候，运行这个 agent 的服务器发生了意外重启。在传统的无状态架构下，这意味着你要从头来过，损失二十分钟的工作，还要重新跑一遍，而且无法完全确认它是否会走同样的路径、做出同样的决定。

在 Managed Agents 里，整个 session 的事件历史都持久化在服务端，沙盒的文件系统状态也在崩溃前做了检查点。你只需要恢复这个 session，agent 会从第十五个文件之后的地方继续，就好像什么都没发生一样。你不需要自己设计检查点系统，不需要管理状态的存储，Managed Agents 把这一切都处理了。

同样的持久化还带来了强大的可观察性。当你的 agent 工作了四十分钟，最后做了一些出乎预料的事情，你可以打开 session 的事件日志，像看慢动作回放一样，逐步重播每一个模型调用和工具调用，看它在每个决策点上收到了什么输入、做了什么判断、调用了什么工具、得到了什么结果。你不需要在代码里提前埋大量日志，整个 session 本身就是完整的运行记录。文章提到 Claude Developer Console 提供了一个原生的可视化时间线视图，让你可以在图形界面里审查 agent 的完整运行轨迹，深度检查任何一条执行路径。

文章还介绍了两个建立在 session 持久化之上的特色功能，值得多说几句。

一个是 Memory，也就是跨 session 的记忆。agent 可以在不同的 session 之间保持对特定信息的记忆，而不是每次都从零开始建立上下文。

另一个叫 Dreaming，这个命名非常有意思，翻译过来可以叫"后台反思"。它是一个定时运行的后台进程，会定期回顾你的 agent session 日志和记忆库，从中识别规律，提炼和整理记忆，让 agent 随着时间推移越来越好。比如用户有哪些反复出现的偏好、agent 在哪类任务上总是犯类似的错误，这些模式都会被 Dreaming 流程识别出来，然后存进记忆，让下一次运行的 agent 受益于这些历史经验。

我个人对 Dreaming 这个设计非常感兴趣，因为它触碰了 AI agent 一个很根本的问题：agent 怎么变得越来越好？目前大多数 agent 系统都是无状态的，每次运行都是全新的开始，不从过去的运行里学习，也没有地方积累经验。Dreaming 的思路是：每一次 session 都是学习材料，session 与 session 之间的间隙，是反思和成长的机会。

你可以想象一个经验丰富的工程师。他不只是在工作时间发挥技能，下班后他也会反思今天哪些地方做得好、哪些地方踩坑了，第二天再来工作时，处理类似问题会更稳更快。Dreaming 就是在给 agent 模拟这个"下班后反思"的过程，只不过是在 session 之间的间隙里自动完成的。

第四个核心优势是部署灵活性。

默认情况下，所有的编排逻辑和代码执行都在 Anthropic 的云容器里进行，这条路最省力，适合快速上线，你只需要定义 agent 配置，其余的 Anthropic 全部处理。

但对于有数据安全合规要求的企业，这条路走不通。有一类很常见的约束是：数据和代码不能离开公司的私有环境。对于这类场景，Managed Agents 提供了自托管沙盒选项。在这个模式下，orchestration 层，也就是调用 Claude 的"大脑"部分，还是跑在 Anthropic 的云上，但代码执行层，也就是"双手"部分，可以部署在你自己的私有云或本地基础设施里，agent 的文件操作、代码运行和所有网络访问都不会离开你的安全边界。

文章还介绍了 MCP tunnels，也就是 MCP 隧道。MCP 是 Model Context Protocol 的缩写，是一种让 AI 模型调用外部工具和服务的标准协议。很多企业的内部工具和数据源都以 MCP server 的形式存在，但这些 server 运行在私有网络里，不暴露到公网。MCP 隧道解决的正是这个问题：它让 Anthropic 的 orchestration 层可以安全地连接和调用你私有网络里的 MCP server，而不需要把这些服务器暴露到公网上。

这两个功能的组合提供了非常精细的控制粒度：自托管沙盒控制"代码执行在哪里发生"，MCP 隧道控制"Anthropic 如何访问你私有网络里的服务"。你可以精确选择什么东西留在你的安全边界内、什么东西可以与外部通信。

在进入客户案例之前，我想专门把文章里一个非常生动的例子拿出来讲，因为它深刻揭示了"harness 随模型演化"这件事有多重要，也直接解释了 Managed Agents 的另一个核心价值——让 Anthropic 来承担这个持续维护的成本。

在 Claude Sonnet 四点五版本上，Anthropic 的团队发现了一个他们叫做"context anxiety"的现象，中文可以叫"上下文焦虑"。表现是这样的：当 agent 在运行中接近上下文长度上限时，它会开始出现一种可观察的行为变化，急着把手头的工作草草收尾，匆忙结束，而不是利用剩余的空间继续把任务做完。就好像一个考试还剩最后十分钟的学生，开始匆忙写结论，尽管很多题目其实还没做完。

为了解决这个问题，Anthropic 在 harness 里加入了 context reset 机制，让 agent 在接近上下文上限时能够自动重置和整理上下文，帮助它保持连贯。在 Claude Sonnet 四点五上，这个修复效果很好。

但到了 Claude Opus 四点五，情况变了。新版模型更聪明、更稳定，"上下文焦虑"这个行为根本就不再出现了，模型已经能够自己管理好这个问题。于是，当初精心设计的 context reset 机制，在新模型上变成了纯粹的额外开销，不再有任何意义，只是在拖慢系统，增加不必要的复杂性。

这个故事说明了什么？harness 和模型是深度耦合的，而这种耦合会随着时间变成一种负担。一个为旧模型调优的优化，可能对新模型造成干扰。如果你的团队不持续追踪模型更新、不断重新评估 harness 里的各种假设，你可能最终是在用一个过时的架构限制一个更聪明的模型。

对大多数产品团队来说，做这件事并不是他们的核心能力，也不应该是。这就是 Managed Agents 的核心价值主张：让 Anthropic 来维护 harness 随模型的演化，让你的团队专注在真正能带来竞争优势的地方——领域知识、上下文设计和用户体验。

好，现在我们来看真实客户是怎么用 Managed Agents 的，这部分是文章里最有参考价值的内容，它把抽象的技术特性落地成了具体的商业场景。

第一个案例是 Notion。

Notion 把 Managed Agents 用在他们的 Custom Agents 功能上。用户可以直接在 Notion 的任务面板里把工作分配给 Claude，就像分配给一个同事一样。Claude 接到任务之后，会自动读取相关的文档、会议记录、关联数据，完成工作，生成的代码、幻灯片、网站等成果会送回 Notion 工作区，等待用户审核和使用。

文章特别提到了规模：可以同时运行几十个任务并发进行。还有一个早期原型的测试数据：把原来大约十二小时的工作压缩到了大约二十分钟。当然这是特定任务下的测试数字，不同场景会有差异，但这个数量级的效率跃升是非常有说服力的。能同时并发几十个任务，意味着基础设施的横向扩展能力不是靠 Notion 自己来解决的，而是由 Managed Agents 的平台来承接的。

第二个案例是 Rakuten。

这家日本大型电子商务企业用 Managed Agents 在产品、销售、市场和财务四个不同业务线，各自部署了专业化的 agent，每个 agent 从开始开发到上线服务生产流量，大约只需要一周时间。一周上线一个生产级 agent，这个速度如果要自己搭所有的基础设施，是不可能实现的。这个数字背后的含义是：工程团队的时间花在了定义 agent 的专业能力和业务逻辑上，而不是花在搭基础设施上。

第三个案例是 Sentry，我个人觉得这个案例是技术上最有参考价值的。

Sentry 是一个错误监控平台，有一个叫 Seer 的调试 agent，可以分析错误报告、定位问题根因。在此基础上，他们用 Claude agent 配合 Seer 做了一个更完整的修复流程：Seer 分析问题找到根因，Claude 根据这个分析写出修复补丁，然后自动开一个 PR 等待工程师审核。从问题被检测到、到 PR 开出，整个流程是全自动的。更值得注意的是：整个系统由一名工程师在几周内完成了构建，而不是需要一个团队花几个月来搭。一名工程师几周，这不只是速度的提升，这是团队规模要求的根本性变化。

另外还有 Asana 构建了 AI Teammates 功能，agent 可以在项目里自主接手任务完成工作；Atlassian 把开发者 agent 嵌入了 Jira 的工作流，让软件开发的日常流程变得更自动化。

这几个案例有几个共同的主题：上线速度从几个月压缩到几天或几周；所需团队规模从一个团队降到一名工程师；并发规模从单个任务到几十个同时运行；覆盖的场景跨越了技术开发、业务流程、内容生产各个领域。

好，技术和案例都聊完了，最后我们来聊一聊对于实际在工作里构建 AI 系统的工程师来说，这篇文章意味着什么，你能直接上手用什么、有哪些坑要提前想清楚。

首先是判断你的项目是否适合用 Managed Agents。

文章给了一个很有用的框架：对于需要极度自定义的 agent 场景，自己维护 harness 是合理的，你对 agent 的每个行为有完整的控制权。但对于任务模式相对固定、不需要极度定制化的工作负载，用托管服务往往更高效。

几个实用的判断问题可以帮你想清楚：你的 agent 任务需要运行多长时间，如果超过几分钟的长时间任务，session 持久化就变得非常关键，自己可靠地实现这个其实相当有挑战；你的 agent 需要接触到敏感凭证吗，如果是，架构层面的凭证隔离比在同一容器内做防护要安全得多；你的团队有精力持续跟踪模型更新、重新评估 harness 假设吗，如果没有，这个维护成本可能远超你的预期；你需要同时运行多少个并发 session，如果是几十个以上，扩展性问题的工程复杂度会相当高。

第二是几个需要提前想清楚的坑。

坑一：不要低估 harness 维护的长期成本。前面那个 context anxiety 的案例很有说服力——你今天针对某个模型行为精心设计的优化，可能在下一版模型里变成负担。而且你往往不是立刻就能发现这个问题，可能要等到性能分析或用户反馈时才注意到。如果选择自己维护 harness，把这个持续维护的成本计入你的工程预算。

坑二：凭证安全在生产环境是高压线，不是可以先欠着再补的债。在原型阶段把 API key 放环境变量里让 agent 直接读，这是图方便。到了生产，尤其是当 agent 在处理来自外部用户的输入时，prompt injection 的风险不是理论上的，而是真实存在的攻击面，从架构上就要把凭证和代码执行隔离开。

坑三：可观察性要从一开始就当成必要条件，不要当成事后补的东西。一个 agent 在生产里自主工作了半个小时，然后做了一些完全出乎预料的事情，如果没有完整的事件日志，调试过程基本上是在盲猜。这种情况在生产里一旦出现，代价非常高，而且很难解释给上级或者客户听。

坑四：如果打算用 Dreaming 这类从历史 session 里学习的功能，要注意数据质量的问题。如果你早期的 session 数据里充满了失败案例、调试记录、不代表正确期望行为的运行结果，Dreaming 从这些数据里提炼出的记忆可能会产生误导，让 agent 把错误的模式固化成"学到的经验"。就像机器学习需要管理训练数据质量一样，你需要有意识地管理 agent session 的数据质量，才能让这类学习机制发挥正面效果。

如果你想开始尝试 Claude Managed Agents，文章提到有两个入口：一个是 Claude Developer Console 开发者控制台，有从模板或者自然语言描述快速生成 agent 的引导流程，另一个是在 Claude Code 里，通过特定命令启动一个交互式向导，一步步帮你配置第一个 Managed Agent。

好，最后来做一个收尾。

这篇文章本质上是一篇产品发布文章，但它包含了很多有价值的工程思考和真实的生产经验，值得仔细读。

从 API 到 Agent SDK 再到 Managed Agents，Anthropic 走的这条路有一个清晰的逻辑：先给你基础工具，再给你调优好的框架，最后给你托管的生产平台。每一步都在降低把 agent 推进生产的门槛。这种渐进式的产品演化，反映的是对工程现实的尊重，没有跳跃到"所有东西我来帮你搞定"，而是随着行业对 agent 系统理解的加深，逐步接管越来越多的基础设施层。

我个人认为，这个方向是对的。AI 应用早期，大家都要自己造轮子，因为没有现成的可用，而且我们还在摸索什么是 agent 系统的最佳实践。但随着越来越多的团队在真实生产中积累了经验，这些经验值得被系统化、标准化，打包成可复用的基础设施，让后来的团队不需要每次都从零开始重蹈覆辙。

当然，这里有一个值得持续关注的问题，文章没有正面回答，但我觉得任何认真考虑在生产里用这套系统的团队都应该提前想清楚：当 agent 基础设施由平台提供商来维护，harness 的更新可能改变 agent 的行为，而这个变化不是你主动触发的。你的 agent 从一个版本到下一个版本可能会有细微的行为差异，不是因为你改了代码，而是 Anthropic 更新了 harness。这种依赖关系的取舍，值得在架构层面事先想清楚，建立自己的评估和监控机制。
