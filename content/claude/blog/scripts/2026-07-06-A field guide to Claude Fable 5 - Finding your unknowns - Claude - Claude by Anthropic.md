## Script

今天我们要聊的文章是 Anthropic 发布在 Claude 官方博客上的 A field guide to Claude Fable Five Finding your unknowns，中文我会翻成，Claude Fable 五实战指南，找到你的未知。这篇文章发表于二零二六年七月六日，也就是距离今天二零二六年七月十二日只有六天，所以它非常新，基本就是 Anthropic 最近对 Claude Code 和新一代编程模型使用方式的一次经验总结。

这篇文章值得听，不是因为它发布了一个炫酷的新 benchmark，也不是因为它展示了某个惊人的模型参数，而是因为它讲了一个很多工程师现在已经隐约感受到、但还没有完全说清楚的问题。那就是，当模型越来越强，尤其是像 Claude Fable 这样可以做更长链路、更复杂工程任务的模型出现以后，真正限制产出的，往往不再是模型能不能写代码，而是你能不能把任务里的未知说清楚。

作者 Thariq Shihipar 是 Anthropic 的 technical staff 成员。他在文章开头用了一个非常经典的比喻，地图和疆域。地图，就是你给 Claude 的东西，包括 prompt，也就是提示词，包括技能说明、上下文、需求描述、各种文档。疆域，就是任务真正发生的地方，比如你的代码库、真实用户、现有架构、历史包袱、性能约束、团队规范，以及那些只有运行起来才会暴露的问题。

地图和疆域之间的差，就是文章里反复强调的 unknowns，也就是未知。我们平时会说，给 AI 写一个好 prompt 很重要，但这篇文章把问题讲得更精确。不是 prompt 长就好，也不是你把每一步都规定死就好，而是你要知道，Claude 在执行任务时会遇到哪些它没法从地图里推断出来的东西。一旦遇到未知，模型就必须猜。它会根据通用最佳实践、上下文里的线索、训练中学到的模式，猜测你想要什么。猜对了，你会觉得它很神。猜错了，你会觉得它怎么这么自作主张。

作者说，Claude Fable 是他遇到的第一个模型，让他明显感觉到工作质量的瓶颈在于自己澄清未知的能力。这个判断很有意思。早期我们用 AI 写代码，瓶颈经常是模型能力不够。你说得再清楚，它也可能忘记需求、改坏文件、理解不了架构。但到了更强的 agent，也就是能自主执行多步任务的智能体阶段，模型确实能查代码、改代码、跑测试、迭代修复。于是问题变成，如果它跑偏了，很可能不是因为它懒，而是因为你没有告诉它哪些地方不能猜，哪些地方可以自由发挥，哪些地方应该先停下来问你。

文章还有一个关键提醒，提前规划当然重要，但光规划不够。未知可能藏在实现深处。你以为只是加一个按钮，结果到了代码里发现状态管理完全不支持。你以为只是接一个新的认证提供商，结果发现老系统里有一堆历史兼容逻辑。甚至有些未知会告诉你，原来的问题定义本身就是错的，你不该用这个方案解决，而应该换一条路。所以作者把和 Fable 协作描述成一个迭代过程，在实现前、实现中、实现后，不断发现自己的未知。

我们先把文章最核心的概念讲清楚。作者把未知分成四类。第一类叫 known knowns，已知的已知。也就是你已经知道，而且你会明确写进 prompt 的内容。比如，我要做一个登录页，要支持邮箱和密码，要符合现有设计系统，要用项目里的表单组件。这些是你说得出来，也愿意写下来的东西。

第二类叫 known unknowns，已知的未知。也就是你知道自己还没想明白的地方。比如，你知道这个新功能需要缓存，但你还不确定缓存策略。你知道要加权限控制，但还没决定角色粒度。你知道这个页面可能要适配移动端，但还没确定交互细节。对于这一类未知，比较好的做法是直接告诉 Claude，我这里还没有决定，你可以先帮我分析方案，或者实现时遇到这里先问我。

第三类很微妙，叫 unknown knowns，未知的已知。听起来有点绕，它指的是那些你自己其实知道，但因为太习以为常，所以不会写下来的东西。比如你们团队的某个页面永远不用弹窗，因为产品风格不喜欢。比如你一眼就能看出一个按钮间距不对，但你很难提前把所有视觉偏好写成规则。再比如你知道这个业务里免费用户不能看到某些字段，但因为项目里到处都是这种约定，你以为它显而易见。对模型来说，这些未必显而易见。它看不到你脑子里的直觉。

第四类是最危险也最有价值的一类，unknown unknowns，未知的未知。也就是你根本没意识到自己不知道。你不知道应该问什么问题，不知道好的结果长什么样，不知道这个领域有哪些坑，不知道行业里成熟方案是怎么做的。比如你第一次让 Claude 帮你做视频调色，你甚至不知道 color grading，也就是调色，到底包含哪些维度。你只会说，画面有点灰，帮我变好看。模型可能给你几个版本，但你也不知道怎么判断。

作者说，最厉害的 agentic coders，也就是擅长和编程智能体协作的工程师，通常未知很少。他提到看 Boris 或 Jarred 这样的人写 prompt，会发现他们非常清楚自己想要什么。他们既熟悉代码库，也熟悉模型的行为模式。这里我个人认为很值得中国工程团队注意。我们经常把使用 AI 编程理解成偷懒工具，好像让它多写一点，自己少想一点。但高手恰恰相反，他们不是不思考，而是把思考前置，把模糊点找出来，再把适合模型发挥的空间交给模型。

不过，作者也强调，这不是天赋，而是一项可以训练的技能。减少未知、规划未知、让模型帮助你发现未知，这就是 agentic coding 的核心能力。这里面的变化是，Claude 不只是执行者，也可以是你的思考搭档。它可以极快地搜索代码库，读文档，甚至查互联网。它对很多通用主题的平均知识量超过单个工程师。它也能比人更快地从失败里迭代。关键是，你要告诉它你的起点在哪里。

比如你可以说，我对这个代码库很熟，但对这个认证模块不熟。或者说，我懂后端，但前端设计审美很差。再比如，我知道用户流失发生在 onboarding，也就是新手引导之后，但我不知道应该从产品、数据还是代码层面切入。这样的上下文不是客套话，而是帮助 Claude 判断应该怎么和你合作。如果你不说，它可能默认你是专家，直接给你一堆术语和实现。如果你其实是新手，你会漏掉重要问题。如果它默认你是新手，又会浪费大量时间解释你早就知道的东西。

接下来文章进入了非常实用的部分，作者按实现前、实现中、实现后三个阶段，分享了自己使用的一组模式。我们不要把它当成固定流程，而要理解成一套发现未知的工具箱。不同任务不一定每一步都用，但复杂任务越多，越应该有意识地挑几种。

开工前第一个模式叫 blind spot pass，盲区扫描。这个说法很直白，就是让 Claude 帮你找你没意识到的未知。适合什么场景呢？比如你要在一个陌生代码区域加功能，或者你要做自己不熟的工作，比如设计、视频、数据分析、安全策略。你可能不知道该问什么问题，也不知道历史上做过什么，也不知道有哪些坑。

作者建议可以直接在 prompt 里使用 blind spot pass 和 unknown unknowns 这样的词。比如，你可以说，我正在添加一个新的 auth provider，也就是认证提供商，但我完全不了解这个代码库里的认证模块。请你做一次盲区扫描，帮我找出相关的未知的未知，并且帮助我更好地给你写后续 prompt。你看，这个 prompt 的目的不是立刻让 Claude 写代码，而是先让它帮你校准地图。

换成我们日常工作里的例子，如果你接手一个老系统，要加企业微信登录，你可能本能地会说，帮我实现企业微信登录。但更好的开局是，先让 Claude 阅读现有登录、会话、用户绑定、权限和配置相关的代码，告诉你这个系统里接入新登录方式可能涉及哪些模块，哪些历史约定需要注意，哪些地方可能有安全风险，哪些问题需要你先和产品或后端同事确认。这个过程花十分钟，可能能避免后面两小时的返工。

第二个模式是 brainstorms and prototypes，头脑风暴和原型。它主要解决未知的已知。也就是你不知道怎么说，但你看到了就能判断。产品和设计里特别常见。你可能说不出什么样的 dashboard，也就是仪表盘，好看，但你一看四个方案，就知道第二个更像你想要的。你可能说不出编辑器工具栏该怎么布局，但让 Claude 做一个单文件的 HTML 原型，用假数据摆出来，你马上能指出，按钮太密，分组不清，或者主要操作应该放左边。

作者强调，早期识别这些未知的已知非常有价值，因为如果等到实现阶段才发现，成本会高很多。一个小小的功能或规格变化，可能导致完全不同的代码实现。尤其对 agent 来说，它已经改了一大堆文件之后，你再说，其实我想要另一种交互，回滚和重做会复杂得多。与其让它先接后端、加状态、写测试，不如先让它做一个假页面、假数据、假流程，让你用眼睛和直觉反应。

这里我个人认为，可以把原型看成和 AI 协作里的低成本沙盘。真正的代码库像城市道路，牵一发而动全身。原型像桌上的积木，你可以随便推倒重来。比如你要给内部运营后台加一个异常订单处理页，不要一上来就让 Claude 改真实路由、接真实 API、改权限。先让它用一个 HTML 文件或者一个临时组件做三种布局，一个强调搜索筛选，一个强调批量操作，一个强调异常原因分析。你看完之后再决定方向，后面的实现会稳定很多。

第三个模式是 interviews，访谈。做完头脑风暴之后，你大概率仍然有未知。这个时候可以让 Claude 反过来采访你。文章里的示例 prompt 很好，意思是，请你一次问我一个问题，围绕任何不明确的地方，优先问那些我的回答会改变架构的问题。这个优先级很关键。不是所有问题都同样重要。按钮文案当然可以问，但如果某个答案会影响数据模型、接口设计、权限边界，那就应该先问。

想象一下你要做一个团队协作功能。Claude 可以问，你们是否需要实时同步。如果答案是需要，就可能引入 WebSocket 或订阅机制。如果不需要，普通轮询就够。它可以问，权限是继承项目权限，还是单独按文档配置。如果答案不同，数据模型会变。它可以问，被删除成员的历史操作是否保留。如果答案不同，审计日志和删除策略都会变。这些问题如果在实现前问清楚，后面就少很多架构级返工。

第四个模式是 references，参考资料。作者说，有时候你根本描述不出自己想要什么，可能因为你没有语言，也可能因为事情太复杂，写出来太费劲。这时最好的办法是给参考。参考可以是图、文档、截图，但作者特别强调，最好的参考是 source code，也就是源代码。

这一点对工程师很重要。我们经常给 AI 一张截图，说照着这个做。截图能表达视觉外观，但表达不了组件结构、状态变化、边界情况、命名习惯、数据流。源码则包含非常丰富的细节。比如你有一个 Rust 的限流库，里面实现了你想要的退避语义，你可以让 Fable 读那个文件夹，然后在 TypeScript API 客户端里重新实现同样的行为。即便语言不同，模型也能从源码里理解语义、接口、错误处理和边界条件。

在公司项目里，这个方法特别适合做一致性。比如你要在新服务里实现一个权限判断，老服务已经有成熟逻辑。不要只口头描述，直接指给 Claude 看老服务的相关目录，说这里的语义是我想要的，请你读懂后在新服务里实现同等行为，但要符合新项目的框架约定。这样比你自己总结一大段规则更可靠，也能减少漏掉隐含约定的风险。

第五个模式是 implementation plan，实施计划。当作者觉得准备实现了，他通常会让 Claude 先写一个实施计划，供自己审查。重点不是让 Claude 罗列所有机械步骤，而是让它把最可能变化、最需要人类决策的部分放在前面，比如数据模型、类型接口、用户体验流程。那些纯机械的重构可以放后面，因为作者相信模型能处理。

这个思想也很实用。我们很多时候让模型写计划，它会生成一串看似完整的任务清单，比如修改文件 A，新增文件 B，运行测试，更新文档。听起来很勤奋，但对你真正有帮助的，是暴露决策点。比如它应该告诉你，我计划新增一个字段表示状态，但这里有两个选择，一个是复用现有枚举，一个是新建表。或者它应该告诉你，这个 UX 流程里我假设用户可以撤销操作，如果不需要撤销，实现会简单很多。计划的价值在于让你提前干预会影响方向的选择。

然后到了实现中。作者说，计划满意之后，他会开一个新的 session，也就是新的会话，把前面产出的 artifact，也就是原型、规格文档、计划之类的材料传进去。这样 Claude 获得一个干净的上下文窗口，但又保留了规划阶段整理出的关键信息。这个做法背后有一个现实问题，长对话里上下文可能变得嘈杂，模型可能被早期探索里的废方案影响。开新会话相当于重新整理行李，只带上确定要用的东西。

不过作者也很诚实地说，无论你规划得多好，未知的未知总会藏着。agent 在工作中可能发现某个边界情况，导致它必须偏离原计划。比如计划里说新增一个字段就行，结果代码里发现这个对象被很多缓存层序列化，直接加字段会破坏兼容。或者计划里说改某个共享组件，结果发现它被二十个页面复用，改动风险太大。这个时候模型需要选择一条保守路线继续前进。

为了让这些偏离不变成黑箱，作者会要求 Claude Code 维护一个临时的 implementation notes 文档，也就是实现笔记。要求是，如果遇到迫使你偏离计划的边界情况，请选择保守方案，把原因记录在 Deviations，也就是偏离项下面，然后继续。这是非常工程化的做法。它不是要求模型每一步都来打扰你，而是让模型自主处理，同时把关键决策留下审计线索。

我个人认为，实现笔记可能是团队引入 AI 编程时最容易被低估的习惯。很多人用 Claude 或其他工具写完代码，只看 diff，也就是代码差异。但 diff 告诉你改了什么，不一定告诉你为什么这么改。尤其是模型中途遇到问题、绕路、权衡时，如果没有记录，reviewer 很难知道它的真实思路。实现笔记相当于飞行记录仪。出了问题可以复盘，没出问题也可以帮助下一次迭代。

接着是实现后。文章提出两个很有意思的模式，一个是 pitches and explainers，提案和解释文档。很多工程师会觉得，代码写完、测试通过就结束了。但在真实组织里，交付一个东西常常还需要获得认同和批准。你要让产品、设计、安全、资深工程师或团队负责人理解你做了什么，为什么这么做，风险在哪里，哪些未知已经处理过。

作者说，把 pitch 和 explainer 做成最终文档的一部分，有两个好处。如果评审者一开始和你有同样的未知，这份文档能加速理解。如果评审者是专家，他们关心的是你有没有考虑到他们会预想到的未知和失败点，这份文档能加速批准。比如你把原型、规格、实现笔记打包成一份可以发到 Slack 的文档，开头放 demo 动图，让别人先看到效果，再看决策和风险。这里虽然文章提的是 Slack，但对我们来说，飞书、企业微信、钉钉里的评审消息也一样适用。

另一个模式是 quizzes，小测验。这个很新鲜，也很值得试。作者说，一次长工作会话后，Claude 可能完成了比你意识到更多的事情。只读代码差异，只能给你一个浅层理解，因为很多行为依赖现有代码路径。于是他会让 Claude 先给自己一份带上下文、直觉解释、具体改动的报告，然后在底部附一个关于这次改动的小测验。他只有在自己满分通过测验后才合并。

这听起来有点夸张，但背后的原则非常严肃。AI 帮你写代码，不代表你可以不理解代码。尤其在生产系统里，你合并的代码最终还是你的责任。小测验是一种主动检查理解的方式。比如 Claude 可以问，这次改动在用户没有权限时会返回什么。或者问，为什么没有在客户端缓存某个结果。又或者问，如果这个 API 超时，页面会怎么表现。你答不上来，说明你还没有拥有这次改动。

这里我想补一句个人观察。过去我们 review 人写的代码，通常默认作者理解自己的改动，只是别人来找问题。但 AI 生成代码打破了这个默认。人类作者可能并没有完整经历每一个设计细节。所以小测验其实是在恢复责任链。它让你从复制粘贴的使用者，重新变成理解系统的工程师。对团队来说，这比单纯要求每个 AI 生成的 PR 都多一个 reviewer 更基础。

文章后半部分用 Fable 发布视频的制作过程，把这些方法串起来。作者说，Fable 的发布视频是端到端用 Claude Code 剪出来的。而视频编辑对他来说是新领域，他并不是专家。所以他先从自己知道的东西开始。他知道 Claude 可以用代码编辑视频，也可以做转录，但他不确定准确度够不够。

于是他让 Claude 解释 Whisper 这类语音转文字是怎么工作的，以及用 ffmpeg 是否能够准确剪掉嗯、啊这样的口头停顿，或者比较长的静默。这里就体现了盲区扫描和学习未知。作者没有一上来就说，给我剪一个完美视频，而是先让 Claude 帮他理解这个领域的基础能力和限制。

接着，他想让 Claude 创建一个和自己说话的词语同步的 UI，也就是画面元素跟语音转录的词同步出现。但他不确定这件事是否可行，所以他让 Claude 用 Remotion 和转录文本先做一个原型视频，验证思路是否成立。这就是用原型降低成本。如果原型发现节奏不对或者技术不可行，就不会浪费大量时间在完整视频制作上。

最后，视频画面看起来有点 muted，也就是偏灰、偏不鲜活。作者知道这和 color grading 调色有关，但他并不真正懂调色。起初他想让 Claude 做几个版本给他挑，但后来发现问题更深，他不知道调色里的好到底长什么样。于是他改成让 Claude 教自己调色，帮助他发现未知。这个转变很关键。很多时候我们以为缺的是选项，其实缺的是判断标准。Claude 给你十个方案，如果你没有审美或领域知识，十个方案也只是十个盲选按钮。

把这个例子映射回软件开发，其实非常常见。比如你让 Claude 优化数据库查询，它可以给你几个版本，但如果你不知道你的系统瓶颈在 CPU、IO、锁竞争还是网络往返，你可能选错。你让 Claude 改前端性能，它可以做懒加载、memo、虚拟列表，但如果你不知道当前卡顿来自渲染、数据量还是接口延迟，也可能越改越复杂。所以有时候最好的 prompt 不是帮我做，而是先教我如何判断什么算好。

文章最后回到地图和疆域。模型越强，你用正确方法能完成的事情越多。当一个长周期任务返回了错误结果，很可能不是因为你需要一个更长的 prompt，而是你需要花更多时间定义未知，或者创建一个允许你和 Claude 在未知中调整的实施计划。每一个解释文档、头脑风暴、访谈、原型和参考实现，都是在修复成本变高之前，低成本发现你不知道的事情。

对我们这些通勤路上听节目的工程师来说，怎么把这篇文章用到明天的工作里？我建议你可以从一个非常简单的习惯开始。下次你打开 Claude Code，不要第一句就说，帮我实现某某功能。先说，我要做这件事，我对哪些部分熟，哪些部分不熟，请你先做一次盲区扫描，找出可能影响方案的未知。这个开场会显著改变后续协作质量。

如果你做的是业务功能，第二个习惯是先做低成本原型。尤其是涉及页面、流程、交互、文案、运营后台、数据看板的时候，先让 Claude 用假数据做几个方向。不要急着接真实后端，不要急着把状态塞进真实 store，不要急着改路由。你先看，先反应，先把脑子里的未知的已知说出来。比如你看到后说，这种卡片式不适合我们，应该更像表格。或者你说，最重要的信息不是订单号，而是异常原因。模型得到这些反馈后，再进入正式实现。

如果你做的是架构或后端，第三个习惯是让 Claude 采访你，而且明确要求优先问会改变架构的问题。你可以直接说，一次问一个问题，不要一次丢十个给我。因为一次一个问题更像真正的技术讨论，你也更容易认真回答。它问你数据保留策略，你想清楚再答。它问你是否需要幂等，你想清楚再答。它问你错误重试语义，你想清楚再答。这个过程会逼你把架构假设从脑子里拿出来。

第四个习惯是多给参考源码。不要低估源码作为 prompt 的价值。你们公司可能已经有类似功能、类似组件、类似策略。直接让 Claude 读，不要让它从零发明。比如你可以说，新模块的权限语义请参考这个旧模块，但实现风格要符合当前服务。或者说，这个页面的视觉密度请参考现有报表页，不要照搬截图，要读组件结构。参考源码能让模型更接近你们组织的真实疆域，而不是停留在互联网上的平均最佳实践。

第五个习惯是在实现时要求保留实现笔记。尤其是超过半小时的任务，或者会改多个文件的任务，一定要让 Claude 记录偏离计划的地方。你可以约定，如果发现边界情况，先选保守方案，写下原因，继续推进。这样你 review 时不是在一堆 diff 里猜发生了什么，而是先读它的决策日志，再看代码。对团队协作来说，这种日志也方便交接和复盘。

第六个习惯是合并前让 Claude 考你。这个方法可能一开始有点尴尬，但非常有效。你可以让它生成一份变更说明，然后给你五到八个问题，覆盖行为、边界情况、回滚方式、测试覆盖和风险点。你答错的地方，说明还需要回去读代码或让它解释。这个习惯尤其适合 AI 参与比例比较高的改动。因为你不能只知道它通过了测试，还要知道它为什么应该通过测试。

当然，这套方法也有坑。第一个坑是过度具体。作者说，如果你太具体，Claude 会严格照做，即使中途其实应该转向。很多工程师为了防止模型乱来，会写非常死的步骤。但复杂任务里，真实代码库可能会推翻你的步骤。更好的方式是，把不可违反的约束说死，把可以调整的地方说清楚，并要求它遇到需要转向的情况记录原因。

第二个坑是过度模糊。如果你只说，按最佳实践来，Claude 很可能采用行业平均方案，但行业平均方案未必适合你的任务。比如最佳实践可能建议引入一个更完整的状态管理抽象，但你的小功能只需要本地状态。最佳实践可能建议做通用化组件，但你们团队当前最需要的是快速验证。你不说项目目标，模型就会用默认价值观替你做选择。

第三个坑是把 Claude 当成全知专家。文章虽然强调 Claude 能帮助你发现未知，但并不是说它永远正确。它搜索代码很快，知识面很广，迭代速度也快，但它仍然可能误读约定，可能过度自信，可能漏掉组织内部没有写下来的规则。所以人类的角色不是消失，而是从逐行写代码，转向定义问题、识别未知、审查决策、承担责任。

第四个坑是把所有流程都仪式化。不是每个小改动都需要盲区扫描、四个原型、访谈、计划、实现笔记、解释文档和小测验。修一个明显的 typo，当然直接修。关键是你要判断任务的不确定性和改动成本。如果任务跨领域、跨模块、涉及用户体验、涉及数据模型、涉及安全或长期维护，那就值得多花前置时间。文章的精神不是增加流程，而是在便宜的时候发现问题。

我个人认为，这篇文章真正重要的地方，是它把 AI 编程从“提示词技巧”推向了“工程协作方法”。以前我们问，怎么写 prompt 才能让模型听话。现在更好的问题是，怎么让模型帮助我理解我自己还没理解的任务。Claude Fable 的能力越强，它越像一个执行力很强的同事。执行力强的同事最怕什么？最怕方向模糊但又不被允许提问。你要么把他管得太死，浪费他的判断力；要么给他太大自由，让他在关键处猜错。

如果用一句话总结原文，就是，下一次项目开始时，先不要急着让 Claude 写代码，而是让 Claude 帮你找到未知。已知的已知写进 prompt，已知的未知拿出来讨论，未知的已知通过原型和参考逼出来，未知的未知通过盲区扫描、访谈和解释来发现。这样，地图会越来越接近疆域，模型的能力也会更真实地转化成你的生产力。

最后我的点评是，这篇文章短，但方法密度很高。它没有神化 Claude，也没有把人类工程师贬低成按按钮的人。相反，它要求我们变成更好的问题定义者、更好的评审者、更好的协作者。对于正在把 AI 编程工具引入日常开发的团队来说，这可能比任何一个单独的提示词模板都更值得实践。
