## Script

今天我们聊的文章是 Agent identity in Claude Tag: a new access model for autonomous, team-wide AI，我把它翻译成《Claude Tag 中的 agent 身份：面向自主、全团队 AI 的新访问模型》。这篇文章发表于二〇二六年六月二十四日，距离今天二〇二六年六月二十八日只有四天，非常新。它不是一篇模型能力展示，也不是一次产品发布会式的宣传，而是在讲一个企业 AI 真正落地时绕不开的问题：当 AI 不再只是你的私人助手，而是进入团队频道、连接公司系统、自己跑任务的时候，它到底应该用谁的权限。

这件事值得听，是因为很多团队现在都在经历同一个转折。过去我们用 AI，最常见的模式是一个人打开一个聊天窗口，问问题、贴代码、让它总结文档。这个时候权限问题相对简单，因为它基本是在帮“我”做事。可一旦 AI 进入 Slack 频道，旁边有三个工程师、一个产品经理、一个数据同学，大家都可以叫它查资料、改代码、跑查询，问题马上变复杂了。它是在代表谁？谁的 GitHub 权限生效？谁能为它的操作负责？如果一个没有仓库权限的人在频道里问它仓库内容，它该不该回答？这篇文章就是 Anthropic 给 Claude Tag 设计的一套答案。

先把背景说清楚。Claude Tag 可以理解成 Claude 进入团队协作空间的一种形态。你不是单独打开 Claude 聊天，而是在团队已经工作的地方，比如 Slack 频道里，直接提到 Claude，让它参与讨论、读取上下文、调用工具、给出建议，甚至去 GitHub 开 pull request，或者查询数据仓库。Anthropic 在文中把传统个人聊天称为 single player AI experience，也就是单人模式的 AI 体验；把 Claude Tag 这种放进共享频道里的形态称为 multiplayer AI experience，也就是多人模式的 AI 体验。这个命名很形象。单人游戏里，所有东西都围绕一个玩家展开；多人游戏里，世界是共享的，规则就不能再只看一个人的背包和装备。

在单人模式下，访问控制可以很直觉。比如你把 Google Drive、GitHub、日历连接给 Claude，它使用的是你的账号权限。它能读什么文档，取决于你能读什么；它能创建什么 issue，取决于你在项目里的权限；它能安排什么会议，取决于你的日历授权。这有点像你请了一个助理坐在你电脑前，你把钥匙给他，他按你的身份办事。只要你自己管理好授权边界，这个模式就能跑起来。很多个人生产力工具其实都是这么做的。

但 Anthropic 说，这个模型不适合 Claude Tag，原因主要有两个。第一个原因是 agent 的自主性在变强。这里的 agent，中文可以叫智能体，指的是不只是回答一句话，而是能够分解任务、调用工具、等待事件、持续执行的一类 AI 系统。文章里提到，一个 AI agent 能够可靠独立完成的任务长度，大约每四个月翻一倍。这个说法很重要，它说明 Anthropic 看的不是今天某个聊天窗口里十分钟的交互，而是未来越来越长、越来越自动化的工作流。agent 可能会把任务排到以后执行，也可能在人下线很久以后响应某个事件，还可能根据用户设定的 routine，也就是例行规则，在满足某些条件时自动行动。

你可以想象一个例子。某个工程团队在频道里对 Claude 说，如果 nightly build 失败，请自动看最近的提交、定位可能的 culprit，也就是可疑变更，整理一个分析，并在必要时开一个修复分支。这个任务可能在晚上发生，最初发出指令的人已经睡了。第二天早上，Claude 已经看了日志、查了 GitHub、读了相关文档，还在频道里留下了总结。这个时候如果它一直用某个具体用户的个人权限，就会出现很多尴尬问题。那个人离职怎么办？那个人夜里被撤销权限怎么办？这个动作在审计里算谁做的？如果它访问了那个人私有空间里的资料，其他频道成员是不是间接看到了不该看的东西？

第二个原因是多人协作。文章给了一个很典型的场景：一个频道里有三个工程师和一个产品经理正在一起 debug。Claude 在这个频道里，大家都能引导它。那权限到底跟谁走？如果跟第一个提问的人走，后面其他人追问时是不是要切换？如果跟权限最高的人走，就可能把敏感信息暴露给权限低的人；如果跟权限最低的人走，Claude 可能什么也做不了，整个协作价值大幅下降。更麻烦的是，团队频道本身就是一个工作空间，里面的讨论往往属于团队而不是某个个人。把一个团队级智能体绑在某个人身上，本质上是错位的。

所以 Anthropic 提出了 agent identity，也就是智能体身份。它的核心思想是，Claude 在这些工具里不代表某一个人，而是拥有自己的账号、自己的服务身份、自己的访问边界。文章举了几个例子：在 Slack 里，它以 Claude app 的身份发帖；在 GitHub 里，它以 Claude GitHub App 的身份打开 pull request；查询数据仓库时，它使用管理员预先配置的 service account，也就是服务账号。这样一来，Claude 的每一次动作都可以被明确归到 Claude 这个智能体身份上，而不是混在某个员工的个人账号里。

这个变化听起来像是权限配置的小细节，但我个人认为，它其实是企业 AI 从玩具走向基础设施的关键一步。因为企业系统里，身份就是边界，身份就是责任，身份就是审计入口。一个人有没有权限，不只是技术问题，还牵涉组织结构、合规要求、数据分级和事故追责。过去很多 AI 工具为了让体验顺滑，会默认借用个人凭证。但到了团队级、长期运行、跨系统操作的 agent，这种做法迟早会遇到天花板。agent identity 的意思是，AI 也需要一张“工牌”。这张工牌不是人的工牌，也不是偷偷挂在人名下的临时证件，而是组织正式发给它的身份。

文章特别强调，因为没有使用个人用户凭证，共享频道就不会变成通往某个人私人文档的侧门。这个点非常实用。很多人一开始会觉得，让 Claude 读取更多上下文当然好，但最怕的是它不小心把 A 的私有文档带到 B 所在的频道里。比如一个高管在自己的 Drive 里有并购计划，一个工程师频道里有人让 Claude 总结公司未来路线，如果 Claude 正好借用了高管账号，那就可能发生严重泄露。agent identity 通过让频道使用独立配置的凭证，避免把个人私有权限带进公共协作空间。

那这个身份具体怎么配置呢？文章说，管理员会在 workspace，也就是工作区层面定义一个 identity。这个 identity 包含 Claude 在各处默认持有的一组连接和能力，所有频道默认继承它。然后在有需要的时候，管理员可以在 channel，也就是频道层面进行覆盖。比如工程频道可以额外访问 GitHub 和数据仓库，销售或客户成功相关的 CRM 连接可以只放在某个私有频道里。这个设计有点像云上权限管理里的默认策略加局部覆盖，也有点像公司门禁系统。每个员工进大楼都有基础权限，但研发实验室、财务室、法务档案室都有额外门禁。Claude 的身份也是这样，不是全公司一把通用钥匙，而是在不同空间里拿不同门卡。

除了凭证本身，管理员还能定义几类东西。文章提到 repository access，也就是仓库访问，规定 Claude 能读写哪些代码仓库。还提到 connectors，也就是连接器，包括 Claude 做事需要使用的工具和 API key。同一个服务，在组织内可以用不同权限级别的 API key 连接。比如普通频道里，Claude 对数据仓库只有只读权限；数据团队的私有频道里，Claude 可能有写权限。文章还提到 skills and plugins，也就是技能和插件，它们可以是一些指令、脚本和资源文件夹，Claude 会动态加载，用来提升特定任务表现。最后还有 standing instructions，也就是常驻指令，为每个频道提供定制上下文和行为规范。

如果你是工程负责人，可以把这里想成四层配置。第一层是它能进哪些系统，比如 Slack、GitHub、Drive、数据仓库。第二层是进了系统能看哪些对象，比如哪些 repo、哪些表、哪些目录。第三层是它能做什么动作，比如只读、开 PR、写入表、触发脚本。第四层是它应该怎样做事，比如这个频道的代码风格、发布流程、需要先问谁、哪些话题不要碰。文章没有用四层这个说法，这是我个人为了便于理解做的整理，但它能帮助我们把 agent identity 从一个抽象概念落到日常配置上。

文章里有一句非常关键的话：agent identity 把问题从“这个用户能做什么”，替换成“这个 agent 在这个 compartment 里能做什么”。这里的 compartment 可以理解成隔间、舱室、权限空间。传统访问控制列表，也就是 Access Control List，通常围绕用户展开。谁能读某个仓库，谁能写某张表，谁能看某个文档，一条条都是人的权限。但在 Claude Tag 里，一个频道成员即使自己没有直接访问某个 repo 的权限，只要频道配置允许 Claude 访问这个 repo，他就可以请求 Claude 去读取它。Anthropic 也承认，这一点“不寻常”，但他们认为这是面向自主、多人智能体所必需的一步。

这个地方可能会让很多做安全和平台工程的听众皱眉。一个没有 repo 权限的人，为什么能让 Claude 读 repo？这不是权限绕过吗？我觉得要分两层看。按传统用户权限模型看，它确实打破了“用户直接权限决定一切”的直觉。但按团队工作流看，频道本身可能就是一个授权空间。比如某个 incident 频道，参与的人未必平时都有生产日志权限，但在事故处理期间，团队需要一个受控工具把相关信息汇总出来。关键不是让每个人获得所有底层权限，而是让 agent 在频道边界内，用被审计、被限定的身份执行工作。也就是说，权限不再只附着在人身上，也附着在协作场景上。

当然，这个模型要成立，边界必须非常硬。文章说，Claude Tag 会为每个私有频道创建一个独立身份，而一个 workspace 里的公开频道共享 workspace 级身份。法律频道里的 Claude 身份不能访问没有授予给该频道的代码；工程频道里的 Claude 身份也不能读取没有授予给它的法律文件。更重要的是，memory，也就是记忆，和 access，也就是访问权限，都要遵守这些边界。Claude 在私有频道里学到的东西，不会出现在更广泛的 workspace 里。这个点对企业很重要，因为 AI 系统的风险不只是当下读到了什么，还包括它把什么长期记住了、未来又在哪里复用。

这里可以用“房间里的白板”来类比。每个私有频道像一个会议室，Claude 在里面有一块白板，记录这个房间的上下文和习惯。它还拿着这个房间对应的门卡，只能打开被授权的柜子。你不能因为它在法务会议室听过某个合同讨论，就让它在全员大厅里随口引用。也不能因为工程频道给它代码仓库权限，就让销售频道顺手通过它查代码细节。agent identity 的难点就在于，把记忆边界和工具边界一起做成强隔离。只隔离工具不隔离记忆，会泄露上下文；只隔离记忆不隔离工具，会出现越权访问。

文章还提到一个默认行为：身份属于频道，所以频道里的任何人默认都可以 tag Claude，也就是提到 Claude 让它工作。管理员可以把每个频道的 profile，也就是配置档案，收敛到最小权限成员的水平。对 Enterprise 企业版计划，还可以使用 RBAC，也就是基于角色的访问控制，进一步决定哪些成员可以调用 Claude。这样频道就同时管理两件事：这个 agent 能碰什么，以及谁能让这个 agent 动手。这个双重控制很关键，因为有些频道虽然有敏感工具连接，但并不是每个成员都应该能触发所有 AI 行为。

举个现实一点的例子。数据团队有一个私有频道，Claude 在里面可以访问某些数据仓库表，并且具备写入某个临时分析 schema 的权限。频道里可能有数据工程师、分析师、产品经理。你可能希望所有人都能让 Claude 总结数据口径，但只有数据工程师能让它执行写操作或生成会落库的任务。原文没有展开到动作级 RBAC，只说 Enterprise 可以限制哪些用户能调用 Claude Tag。我个人认为，未来企业很可能会要求更细的“谁可以触发哪类动作”，因为只控制能不能叫 Claude，粒度还是偏粗。不过文章后面提到的未来方向，正是在往更细粒度走。

Anthropic 也分享了他们在内部运行 Claude Tag 的观察。原文说，Claude 的价值会随着工具和上下文访问而复利增长。每连接一个系统，其他系统也变得更有用，因为 Claude 可以跨系统组合上下文。它可以从 Slack 线程里抽取问题背景，从 Drive 文档里读取设计方案，从工单系统里找到相关 ticket，再从数据仓库里跑一个查询，最终给出单一工具无法提供的回答。这是很多团队对 AI 的真实期待：不是再多一个搜索框，而是把散落在组织里的上下文拼起来。

这个“复利”非常好理解。一个只接 Slack 的 Claude，可以总结讨论；一个只接 GitHub 的 Claude，可以看代码；一个只接数据仓库的 Claude，可以回答指标问题。但如果它同时知道 Slack 里大家在争论什么、GitHub 里最近改了什么、文档里设计初衷是什么、数据里结果如何，它就能回答更高层的问题。比如“为什么上周转化率下降，和新版 onboarding 改动有没有关系”。这个问题需要产品讨论、代码变更、埋点定义、数据查询一起看。单个系统都只能给碎片，agent 的价值在于把碎片合成因果链。当然，这也意味着权限边界更重要，因为越能组合，越可能组合出敏感信息。

原文给出的实践建议是，最能从 Claude 获得价值的团队，往往从一开始就给它相对慷慨的访问，然后根据组织管理员偏好再逐步收缩。Anthropic 建议先在几个频道里建立 baseline profile，也就是基线配置，查看 audit trail，也就是审计轨迹，然后在工作确实需要的时候，一次一个明确授权地扩展访问。这里的语气值得注意。它不是说一上来给全公司所有权限，也不是说默认什么都不给。它强调的是在足够有用和足够安全之间找一个可观察、可回滚、可审计的起点。

如果你在公司里要试这类东西，我个人会建议从低风险但高上下文价值的频道开始。比如工程团队的开发协作频道，可以先给 Claude 只读访问少数非敏感仓库、公开技术文档、issue tracker，再加上频道的 standing instructions，让它知道代码评审规范、发布流程和常见组件名。先不要急着给生产数据库写权限，也不要急着接所有 Google Drive。跑一两周后，看它实际调用了哪些工具、哪些回答有帮助、哪些请求被挡住、哪些权限其实没用。然后再决定是否给某个私有频道增加更强连接。这样做的好处是，团队能用真实工作负载来校准权限，而不是在会议室里凭想象设计一套完美策略。

文章还说，如果组织需要更细粒度控制，管理员可以在特定频道禁用 Claude Tag，也可以应用 RBAC，把 Claude Tag 的访问限制给特定用户。这一点在大公司尤其常见。不是所有频道都适合有 AI。比如人力资源投诉频道、并购谈判频道、董事会材料频道，哪怕技术上能隔离，组织上也可能选择先禁用。安全治理很多时候不是“能不能做”，而是“要不要做”。一个成熟的平台，必须允许管理员说不，允许某些空间完全没有 agent，而不是默认把 AI 撒进每一个角落。

还有一个很容易被忽略但非常重要的区别：直接消息，也就是 DM，和共享频道不一样。文章说，在 Claude Tag 里，DM 使用用户个人的 claude.ai 账号运行，也就是使用这个用户自己的连接器、凭证和名字。换句话说，私聊 Claude 时，它更像你的个人助手；在共享频道里，它更像频道里的团队智能体。Anthropic 认为，DM 适合处理那些永远不应该放进频道的任务和工具，比如邮件草稿，或者只有你自己拥有许可证的软件。这个区分很合理，因为个人工作和团队工作本来就应该有不同身份边界。

你可以这样判断：如果任务结果本来就应该被频道里的成员共享，适合用频道里的 Claude；如果任务涉及你的个人通信、个人授权、个人账号里的私有内容，就应该放在 DM。比如让 Claude 帮整个工程频道总结一个 incident 的 root cause，也就是根因分析，这是频道任务。让 Claude 帮你写一封给老板的绩效沟通邮件，这是个人任务。让 Claude 根据团队代码和工单生成修复建议，是频道任务。让 Claude 用你个人购买的某个 SaaS 软件做分析，是个人任务。这个边界如果团队一开始不讲清楚，后面很容易出现“我以为它用的是我的权限”或者“我以为频道里大家都能看”的误会。

从实现角度，文章给了一段安全细节。管理员把一个连接加入频道 profile 时，这个 credential，也就是凭证，会被独立存储，并映射到该频道的身份。请求发生时，凭证会在 network boundary，也就是网络边界处被注入。任何发往管理员未允许 host，也就是主机的出站流量，都会被直接阻断。审计方面，使用 agent credentials，也就是智能体凭证所做的每一个 routine、每一次 memory write，也就是记忆写入，以及每一次 network call，也就是网络调用，都会被记录下来。因为 Claude 使用自己的服务账号行动，这些动作也会出现在各个连接系统自己的日志里。

这段话信息量很大。凭证在请求时于网络边界注入，意味着模型本身不应该随便“看到”或持有原始密钥。出站流量按允许列表控制，意味着即使某个指令诱导 Claude 去访问奇怪的外部地址，也会被网络层挡住。每个 routine、记忆写入、网络调用都进审计，意味着出了问题可以追溯“什么时候、哪个身份、在什么频道、调用了什么”。同时，连接系统自己的日志里也会看到 Claude GitHub App 或服务账号的动作，这让企业可以把 AI 操作纳入既有的 SIEM、安全审计和合规流程。原文没有展开这些集成，但对做平台和安全的听众来说，这是落地时必须问的地方。

这里也有一个实践坑要提醒。很多团队引入 AI 工具时，只看模型效果，不看日志可用性。等到某个回答泄露了敏感上下文，或者某次自动操作产生副作用，才发现日志里只有“某个用户调用了 AI”，看不到它到底读了哪些文档、调用了哪些 API、为什么写入了某段记忆。agent identity 的好处之一，就是让 AI 操作成为一等公民，可以被单独撤销、单独审计、单独限权。文章说，围绕 distinct Claude identities，也就是不同 Claude 身份来工作时，撤销某个身份就会终止 Claude 在所有使用该身份的地方的访问。这比在几十个用户账号里逐条追查 agent 行为省力得多。

当然，agent identity 并不是银弹。它解决的是“AI 用谁的身份工作”这个基础问题，但策略怎么配，仍然考验组织。比如频道身份应该按最小权限成员配置，还是按团队任务需要配置？公开频道共享 workspace 级身份，会不会过宽？私有频道身份太多，会不会管理复杂？哪些 memory 可以写，哪些只能临时使用？这些问题原文没有逐一给标准答案，它给的是一个框架：身份跟频道走，访问按配置授予，行为被审计，必要时用 RBAC 限制调用者。我个人认为，对大多数公司来说，最难的不是技术接入，而是把数据分类、团队边界和 AI 使用场景梳理清楚。

文章最后讲了未来计划。Anthropic 计划加强 Claude Tag 的安全能力，包括 just-in-time credential grants，也就是即时凭证授权。它的意思是，当某个敏感动作发生时，用户可以在当下批准一次，而不是永久扩大 agent 的权限。比如 Claude 平时没有生产写权限，但在一次事故处理中，它提出需要执行某个受控修复动作，相关负责人可以临时批准这一次。动作完成后，权限不长期保留。这个模式在云权限和零信任安全里很常见，现在被引入 agent 场景，非常自然。

另一个未来方向是 identity-aware overlay，可以理解成身份感知的叠加层，面向有更复杂 clearance structure，也就是许可级别结构的组织。它会在 agent 的 scope，也就是范围之上，再增加用户级检查。也就是说，Claude 只有在频道 profile 允许，并且发起请求的用户自身权限也允许时，才会行动。这实际上是在调和前面那个“不寻常”的点。当前模型强调“agent 在频道里的权限”，未来叠加层会让“请求者自己的权限”也参与判断。最终结果可能是双钥匙机制：频道给了一把钥匙，用户也要有对应资格，两者同时满足，Claude 才能开门。

对软件工程师和 AI 从业者来说，这篇文章的启发不只适用于 Claude Tag。只要你在做企业 agent，都会遇到同样问题。你可能在做一个内部研发助手，一个数据分析 agent，一个客服自动化系统，或者一个能帮运维处理告警的 bot。只要它进入多人空间、连接多个系统、执行长任务，就不要再简单地说“用发起人的 token 就行”。你需要为 agent 设计身份、权限边界、凭证注入、出站控制、审计日志、记忆隔离和撤销机制。否则，模型越能干，风险越不受控。

如果让我把这篇文章落成一份团队检查清单，我会问几个问题。你的 agent 有没有独立服务账号？它在不同频道或项目里的身份是否隔离？它能访问哪些仓库、文档、数据表，是不是能从配置里一眼看出？它的凭证是否和个人账号分离，能不能一键撤销？它的每次工具调用、网络访问和记忆写入有没有审计？频道成员能不能触发它，是否需要 RBAC？个人私聊和团队频道是否使用不同授权模型？这些问题不是为了把 AI 卡死，而是为了让 AI 能被放心地放进真实工作流。

我也建议产品和平台团队在设计体验时，不要把权限提示藏得太深。频道里的人应该清楚知道，这个 Claude 当前代表的是频道身份，不是我的个人身份。它能访问什么，不能访问什么，最好能以比较直观的方式展示。否则用户会把聊天体验里的“自然语言便利”误解成“权限天然正确”。未来的企业 AI 产品，很可能需要像云控制台一样，把 agent identity、profile、connector、audit trail 这些概念产品化，让普通管理员也能理解，而不是只有安全工程师能配置。

收个尾。Anthropic 这篇文章的核心很简单：从 single player AI 到 multiplayer AI，访问模型必须变化。Claude Tag 让 Claude 进入团队频道，agent identity 则给它一张独立、可审计、可撤销、可按频道限定的工牌。它把问题从“这个人能做什么”转成“这个智能体在这个协作空间里能做什么”，这是一个不太符合传统直觉、但非常符合未来 agent 工作方式的转变。

我个人认为，这篇文章真正重要的地方，不在于 Claude Tag 本身，而在于它把企业 AI 的权限问题讲得很具体。未来一年，很多组织会发现，模型能力不是唯一瓶颈，身份和访问控制才是 agent 能否进入生产环境的门槛。谁先把这套基础设施想清楚，谁就更有机会让 AI 从聊天窗口走进团队的日常工作。
