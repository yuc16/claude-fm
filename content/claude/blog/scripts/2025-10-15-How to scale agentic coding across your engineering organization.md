大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,英文标题是 How to Scale Agentic Coding Across Your Engineering Organization,我把它翻译成中文,大概意思是,如何把智能体编程能力推广到整个工程团队。这篇文章发表于二零二五年十月十五日,来自 Anthropic 官方博客。

先说一句时效性提醒。今天是二零二六年六月二十一日,距离这篇文章发表已经过去八个月了。在 AI 圈子里,八个月不算短,但这篇文章讲的核心是组织管理和团队协作层面的方法论,不是某个具体模型能力的介绍,所以它的参考价值不会因为模型迭代而迅速过期。背景上,二零二五年下半年正是 Agentic Coding 工具,也就是能够自主规划、自主执行多步骤编程任务的 AI 编程助手,开始在企业里大规模落地的阶段。Claude Code 这样的工具已经不只是帮你补全一行代码,而是可以理解整个仓库的上下文,自己拆解任务,一步步把一个功能做完。这个时候,很多技术负责人面临的问题已经不是"这个工具好不好用",而是"怎么让几百人、几千人的工程团队都用起来,而且用得好"。这正是这篇文章要回答的问题。

我们先说说背景。文章一开头就指出,随着 Agentic Coding 工具逐渐成熟,技术领导者面对的是一个非常实际的挑战,就是怎么从孤立的小范围实验,走向整个组织范围内的采纳。这句话其实点出了一个很多团队都会遇到的尴尬处境。可能你们团队里已经有那么三五个工程师,自己摸索着用 Claude Code 干活,效率提升很明显,但是这种经验没有办法复制到整个团队,甚至隔壁组完全不知道这件事。文章认为,成功和挣扎的实施之间,差别往往在于执行方式。那些认真规划部署的团队,看到了开发速度和工程师满意度的实质提升;而那些急于求成、跳过规划环节直接全员推广的团队,反而会遇到阻力,结果参差不齐,而且很难证明这件事到底有没有价值。

文章团队跟不同行业的工程团队合作之后,发现了一些共性的规律。成功的采纳,很少取决于你用的是哪个具体工具,而更多取决于你怎么处理工作流程的变化,怎么培养团队的技能,怎么处理团队内部的动态关系,以及怎么衡量成功。这句话我觉得特别值得划重点,因为它把这件事从"工具选型问题"重新定义成了"组织变革管理问题"。如果你是技术负责人,这意味着你要解决的不是"该买哪家的产品",而是更接近于推动一次内部的工作方式变革,这跟当年推广敏捷开发、推广 DevOps 文化,其实是同一类难题。

好,接下来我们具体看看,Agentic Coding 工具到底能用在哪些场景上,文章给出了四个典型应用。

第一个是legacy系统的现代化改造,也就是老旧代码库的迁移升级。很多公司都有那种十年前写的、用老技术栈搭的核心系统,想升级到新平台,但是工作量巨大,谁都不愿意接这个烫手山芋。文章说,开发团队现在用智能体工具来帮助完成这类迁移,原本可能要花上几年时间的项目,现在可以明显加速。但是文章也很谨慎地加了一句,这类项目仍然需要仔细的人工监督和测试,来保证业务逻辑被正确地保留下来。这个提醒我觉得很重要,因为遗留系统里往往藏着大量没有写在文档里的隐性业务规则,AI 再强,也得靠人来把关这些隐性知识有没有被正确迁移。

第二个场景是新人入职的提速。新加入的工程师可以直接向智能体提问,了解代码库的架构、依赖关系、实现模式,这是对传统文档的补充,而不是替代。这样可以缩短新人真正能够独立贡献代码之前所需要的时间。我自己设想一下这个场景,如果你刚入职一家公司,面对一个几十万行代码的仓库,过去你可能要拉着老员工一行一行问,现在你可以直接问智能体"这个模块的认证流程是怎么实现的",它能给你一个有上下文的回答,这确实能省下大量来回沟通的时间成本。

第三个场景是故障响应的辅助。SRE 团队和运维团队可以构建智能体来帮助诊断和处理常见的运维问题。文章特别强调,对于复杂问题,人工监督仍然很重要,但是常规性的故障,往往可以用更少的人工介入来处理。这里的边界划分很清楚,智能体处理的是"常规"故障,复杂的、需要经验判断的故障,还是要靠人。

第四个场景是更广泛的跨职能参与。产品经理在写需求文档的时候,可以直接探索代码库里的技术约束;设计师可以直接从设计稿生成可运行的原型。文章特别说明,这并不是要替代工程师的工作,而是让跨团队的协作可以基于更充分的信息来进行。这一点我觉得对很多公司其实是个新思路,过去产品和设计提需求,经常是"拍脑袋"式的,工程师再去判断技术上行不行,现在产品和设计自己就可以先摸一摸技术的边界,沟通成本会降低不少。

文章特别强调,这四个场景只是起点,不是 Agentic Coding 应用场景的全部清单。

接下来我们进入这篇文章的核心部分,也就是怎么具体地推广落地。文章提出了一个非常清晰的路径,我把它拆成几个阶段来讲。

第一个阶段是组建试点小组。文章给出了一个具体的数字范围,二十到五十名开发者,而且强调这批人最好是已经在用 AI 辅助工具的开发者,也就是说不是从零开始挑选,而是挑那些已经有一定 AI 工具使用习惯、接受度比较高的人。这个试点小组承担三个作用。第一,验证这个技术在你们自己的代码库上到底好不好用,毕竟每家公司的代码库千差万别,通用的评测结果不代表在你这儿一定适用。第二,找出哪些工作流程是真正有价值的,而不是纸上谈兵地猜。第三,培养内部专业能力,这批人未来会成为推广更广泛采纳时的种子力量。

试点期间具体要做什么呢,文章给了一份相当具体的清单。比如说,创建自定义的 slash command,也就是快捷指令,用来处理常见任务,像数据库迁移或者功能脚手架搭建这类重复性工作。再比如,构建 CLAUDE.md 文件,这是个专门的配置文件,用来记录代码规范和项目特定的上下文信息,我们后面会专门讲这个。还有,识别值得自动化的重复性工作流程,比如生成样板代码、生成测试、更新依赖包这些。另外,设立一个专门的沟通频道,用来排查问题和分享经验。最后,还可以为第三方工具的身份验证开发一些包装脚本。

文章的意思是,试点阶段的目标不是马上出成果交差,而是要在你扩大范围之前,先把机会和挑战都摸清楚。这是一个很务实的态度,我个人认为,这恰恰是很多公司容易跳过的一步,大家往往急于看到全员推广的效果数字,反而忽略了试点阶段沉淀下来的经验和工具,才是后续规模化的真正基础。

第二个阶段,文章建议的不是分阶段、排队式的推广,而是用一次启动活动,把整个组织团结起来,统一开始体验。原文用的词是 kickoff event,我翻译成启动活动或者叫百花齐放式的活动。具体做法是,让试点小组的人分享他们摸索出来的技巧和提示词写法,同时让所有人一起动手实验。

这个形式有几个好处。首先,它能在一个低风险的环境里,直观地展示这个工具的能力。文章里有一句话我觉得特别真实,那些原本对 AI 辅助持怀疑态度的工程师,往往在亲手体验之后,看法就会改变。这其实符合很多人的共同经验,你跟一个怀疑论者讲一百遍"这个工具很厉害",不如让他自己上手敲十分钟代码,效果立竿见影。其次,这种协作的氛围,也容易激发出试点小组自己都没想到的创造性应用场景,因为人多了,角度自然就多了。

文章还特别提了一个细节,活动要保持轻松和有活力,提供食物对出席率和团队气氛都有帮助。这听起来是个小细节,但我觉得挺接地气的,毕竟推广一个新工具,首先得让大家愿意来、愿意试,氛围感这种软性的东西,很多技术团队容易忽视。

第三个阶段,随着越来越多的人开始使用这些工具,试点小组要转变角色,从最初的探路者,变成内部的顾问和支持力量。他们可以组织工作坊,制作教学内容,在别人遇到困难的时候充当资源。

文章特别比较了这种方式和外部培训项目的差别,认为内部冠军式的推广通常效果更好,因为这些人理解你具体的工作环境,能够提供来自真实项目的相关案例,他们说的是组织内部的语言,也了解组织特有的痛点。这一点我完全认同,外部培训讲师讲的是通用方法论,但是没法告诉你"我们这个遗留系统里那个奇怪的认证逻辑该怎么处理",只有自己人才知道这些细节。

接下来文章专门讲了 CLAUDE.md 文件,我觉得这是整篇文章里操作性最强的一部分内容,值得单独展开讲讲。CLAUDE.md 文件,简单说,就是一个文档,用来记录仓库的约定、环境配置和项目特定的行为规则,相当于给智能体一份"项目说明书"。文章认为,当这份文件被系统化地在团队之间共享的时候,它的价值会变得更大。

文章给了四条具体建议。第一,创建项目级别的文件,把 CLAUDE.md 文件提交进代码仓库的根目录,这样所有参与这个项目的人,都会自动继承同样的配置和上下文,不需要每个人单独配置一遍。第二,把它当成文档来对待,当架构决策发生变化,或者出现了新的代码模式的时候,要及时更新 CLAUDE.md,并且把这些更新和代码改动一起放进 pull request 里提交,也就是说,文档更新和代码更新应该是同一个动作的一部分,而不是事后补的。第三,把它纳入新人入职流程,让阅读项目的 CLAUDE.md 文件,成为开发者入职checklist里的一项,新成员不仅要理解代码库本身,还要理解怎么在这个具体的上下文里使用 Claude Code。第四,考虑分支级别的差异,如果项目在不同分支上有显著不同的模式,可以维护针对特定分支的 CLAUDE.md 内容。

文章举了一个典型项目级文件大概会涵盖哪些内容,通常包括开发环境的要求、测试和代码规范、关键的架构模式,以及当前的工作重点。文章用了一个说法,叫这样的文件创造了"活文档",能让 Claude Code 始终跟你们团队不断演化的实践保持一致。我自己理解,这个思路其实跟我们写 README 或者写架构决策记录是一脉相承的,区别在于,CLAUDE.md 这份文档不只是给人看的,也是直接喂给智能体作为工作上下文的,所以它的及时性和准确性,直接决定了智能体输出结果的质量。如果这份文档过时了,智能体可能会按照已经废弃的规范去写代码,这是个值得注意的坑。

讲完了怎么推广,我们来看看文章里关于怎么衡量效果的部分,这部分对管理者来说尤其重要,因为试点阶段最常被问到的问题就是,这个东西到底值不值这个钱,投入产出比怎么算。

文章很明确地提出,单纯看写了多少行代码,这个指标捕捉到的是活动量,不一定是价值量,所以不能只看代码行数。取而代之,文章列了六个维度。

第一个是迭代吞吐量,也就是 sprint throughput。已经建立了比较成熟 DevOps 实践的团队,可以把采纳智能体工具的时间点,和功能交付速度的变化关联起来看。

第二个是任务完成时间。衡量标准任务在引入工具前后分别花了多长时间,这个颗粒度更细的视角,能看出智能体工具在哪些具体环节带来的价值最大。

第三个是迁移速度。追踪现代化改造老系统所需要的时间,迁移得越快,释放出来的工程资源就能投入到别的优先事项上。

第四个是开发者满意度。通过调研,了解工程师把时间花在重复性工作上,还是花在创造性工作上的比例变化。文章特别提到,工作满意度对人才留存和生产力都很重要,这一点我觉得经常被纯效率指标掩盖,但其实非常关键。

第五个是入职周期。衡量新员工达到有意义的生产力所需要的时间,周期越短,培训成本越低,团队的产能恢复得也越快。

第六个是跨职能效率。追踪其他团队,比如产品和设计,需要专门的工程支持来做原型和测试的频率有没有下降。如果下降了,说明组织层面的技术能力变得更普及了。

文章还提到,Claude Code 本身内置了 Activity Metrics,也就是活动指标功能,可以追踪被采纳的代码行数、建议的采纳率、每日活跃用户和会话数、组织整体和按用户的花费情况,以及每个开发者的个人指标。这相当于工具自带了一部分的度量基础设施,省去了你自己搭建监控体系的工作。

文章在这部分的结尾说了一句话,我觉得特别实在,有时候最有说服力的衡量方式,反而是最简单的那种,就是拿出具体的例子,告诉大家某个任务现在只需要原来的一小部分时间就能完成。当你能够指出具体的、有意义的效率提升案例时,价值就变得不言自明了。这其实跟我们做产品演示是一个道理,一个具体生动的案例,往往比一整页的统计图表更有说服力。

接下来文章讲了几个在推广过程中会反复出现的典型问题,以及怎么应对,这部分我觉得是最贴近日常工作的内容,值得我们仔细讲讲。

第一个常见问题是,新用户经常给智能体工具下达过于宽泛的任务,而且没有提供足够的上下文,结果得到的是让人沮丧的结果。文章给出的解决方案是测试驱动开发,也就是 TDD,先写测试来定义清楚什么叫成功,包括必须实现的功能、边界情况、错误处理这些。然后再一步一步地增量实现功能,每次只写刚好能让一个测试通过的代码量。文章举了个例子,如果你在做用户认证功能,可以先从基础的登录验证开始,然后加上密码哈希处理,再加上会话管理,一步一步往上叠加,而不是一口气把整个认证系统都丢给智能体去做。每完成一步,就跑一遍测试,审查一下改动,再继续往下走。智能体可以帮你分析测试结果,但是要等到当前功能确实跑通了,才扩大范围去做下一步。

这个方法听起来好像增加了步骤,但实际上是把一个模糊的大任务,拆解成了一连串清晰、可验证的小任务,这正好弥补了智能体在"理解你脑子里没说出来的意图"这件事上的天然短板。如果你直接说"帮我做一个用户认证系统",智能体只能按照它自己对"标准认证系统"的理解去猜,猜错的概率自然就高。但如果你先给出测试用例,等于是把你脑子里的验收标准,直接翻译成了机器可以理解的语言。

第二个常见问题是反馈太模糊。像"这个不工作"或者"这个按钮太大了"这种描述,根本没有给 AI 提供足够的信息去真正解决问题。文章建议要具体,要把完整的错误信息一并提供,包括完整的报错文字、调用栈,以及触发问题的具体操作,直接把终端输出或者浏览器控制台的报错复制粘贴进对话里。同时要说明你的环境,操作系统、语言版本、框架细节,以及相关的依赖项,因为 AI 需要这些上下文才能给出准确的解决方案。对于界面问题,要截图,并且精确描述哪里出了问题,文章给了个对比的例子,"登录按钮在移动端屏幕上超出容器边框二十像素"比"这个按钮看起来怪怪的"要好得多。同时还要清楚说明期望的行为和实际的行为分别是什么,比如"期望接口返回二百状态码并带上用户数据,实际返回的是四百零一,提示无效令牌"。最后,把相关的文件内容,也就是和问题相关的具体代码、配置或者数据,一并提供进去。

第三个常见问题,是沟通方式本身没有调整过来。跟 AI 工具清晰沟通,是需要练习的一项技能,很多开发者期望它能立刻读懂自己的心思,结果一旦输出和预期不一致,就感到沮丧。文章建议你可以这样想,如果是把同样的请求讲给一个同事听,他能理解吗,如果不能,那就先预判他会问哪些问题,把这些信息提前给到。

具体的方法上,文章给了几条建议。结构上要先讲高层目标,再补充实现细节,比如先说"构建一个用户管理的REST接口",然后再具体说明各个端点和要求,这样比把所有信息混在一起说效果更好。用具体的技术语言而不是模糊的字眼,比如"把数据库查询优化到响应时间从两秒降到五百毫秒以内",这比单纯说"让它更快一点"清楚得多。给出具体的成功样例,比如"参照这个已有的接口模式",或者"使用这份代码风格指南",这种具体参照物比抽象的要求更有指向性。把复杂的工作拆分成连续的提示,比如先"创建数据库表结构",再"实现商品目录接口",再"加上购物车功能",每条指令只专注一个明确的目标。从简单开始逐步迭代,比如先做一个"基础的用户登录表单",再加上输入验证,再实现密码强度要求,这种循序渐进通常比一开始就把所有要求都列出来效果更好。对输出给出具体的反馈,比如"错误处理太笼统了,要为邮箱格式和密码长度加上具体的校验",这比单纯说"把校验修一下"指向性强得多。还有,在基于之前的工作继续推进时,要明确引用前面的成果,比如"用刚才写好的那个认证中间件,现在加上基于角色的权限控制"。

这一连串建议,我个人感觉,本质上讲的是同一个原则,就是把和智能体协作这件事,当成管理一个聪明但完全不了解你公司背景的新同事来对待。你给的信息越具体、越结构化、越分步骤,它给你的产出质量就越高。这跟管理初级工程师的逻辑其实很相似,只是因为智能体反应快、不会累,很多人会下意识地用更随意的方式跟它沟通,结果反而踩了更多的坑。

文章在结尾部分做了一个总结,我把它的核心意思梳理一下。Agentic Coding 把软件开发这件事,从工程师事事亲自动手写每一行代码,转变成了工程师去引导和监督实现过程。看到好结果的组织,共同点是把精力放在打基础上,而不是急着把工具一下子推给所有人。具体路径就是,先从一个聚焦的试点小组开始,培养内部的专业能力,搭建好支撑成功落地的基础设施,然后再通过黑客松和内部冠军这样的形式,有计划地逐步扩大范围。

文章最后强调,从试点走到全员生产环境,这条路需要耐心和系统性的规划。那些愿意在这件事上投入打基础的组织,往往能看到实实在在的回报,包括更快的开发速度、更高的工程师满意度,以及有能力去挑战那些过去因为成本太高而搁置的困难项目。

那我们来说说,这篇文章对我们普通听众,尤其是身处技术团队或者负责推动团队效率提升的同学,到底有什么实际的启发。

第一点,如果你正打算在自己团队推广 Agentic Coding,不要一上来就追求全员覆盖,先找一小批愿意尝试、本身就对 AI 工具有兴趣的人,给他们时间去摸索,把摸索出来的经验,比如好用的快捷指令、好用的提示词模板,沉淀成可以复用的资产,这比直接发一个全员通知"大家都开始用起来"要扎实得多。

第二点,CLAUDE.md这一类项目说明文件,是个性价比很高的实践,几乎没有什么技术门槛,核心就是把项目的约定、规范、架构要点写清楚,并且像维护代码一样持续维护它。如果你们团队还没有这类文件,我个人建议可以从这周就开始建,哪怕先写得简单一点,后面再慢慢补充完善。

第三点,在衡量效果的时候,不要只盯着代码行数这种容易获取但意义有限的数字,试着结合任务完成时间、入职周期这些更贴近业务价值的指标,如果一时半会没法量化,至少先收集几个具体生动的案例,这些案例在说服老板或者说服团队其他人的时候,往往比一堆图表更有效。

第四点,如果你自己在用智能体工具的时候经常觉得它"不太懂我",可以回头检查一下,自己是不是给的任务太宽泛、反馈太模糊、沟通结构太混乱。文章里讲的测试驱动、具体反馈、分步推进这几条,几乎可以直接当成日常使用的checklist来对照检查。

最后,我想说说我自己的一点感受。这篇文章虽然发表在去年十月,但是它讲的不是某个具体功能或者某个模型版本的能力,而是一整套关于怎么在组织里推动技术变革的方法论,这种方法论的保鲜期通常比模型本身要长得多。我个人认为,这篇文章最有价值的地方,恰恰不是那些听起来很时髦的"智能体能做什么"的场景介绍,而是那些朴素到有点像常识的建议,比如先小范围试点、把内部专家变成讲师、把效果讲成具体的故事而不是抽象的数字。这些做法放在任何一次技术变革里都适用,不只是这一次的 Agentic Coding。
