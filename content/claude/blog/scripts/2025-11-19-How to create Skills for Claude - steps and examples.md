## Shownotes

这一期我们聊聊 Anthropic 官方写的一篇关于如何给 Claude 创建 Skills（技能）的实操指南。Skills 本质上是一份叫 SKILL.md 的说明文件，它能把 Claude 从一个万能助手，变成你某个具体业务场景里的专家。我们会拆解一个好技能需要哪三个核心要素，怎么测试技能是否靠谱，以及官方给出的几个真实案例里藏着哪些设计巧思。

## Script

大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是 How to create Skills for Claude，key steps and examples，我把它翻译过来就是,如何为 Claude 创建技能,关键步骤与示例。这篇文章是 Anthropic 官方在二零二五年十一月十九日发布的。

算一下时间,今天是二零二六年六月二十日,距离这篇文章发布已经过去了大概七个月。七个月,在 AI 这个行业里,真的不算短。所以我想先提个醒,这篇文章里提到的一些功能细节,比方说 Claude 点 ai 里 Skills 功能目前是用户个人独享、组织级管理还没上线这种描述,到我们听这期节目的时候,很可能已经有了新的进展。所以大家可以把这篇文章当作理解 Skills 这个功能设计思路的一个基础读物,具体的产品状态,还是建议去 Claude 的官方文档里再确认一下最新情况。

那么我们先说说背景。这篇文章发布的时间点,正好是 Claude 的 Skills 功能刚刚成熟、开始被广泛推广使用的阶段。在这之前,大家用 Claude 或者类似的大模型 agent,遇到一个反复出现的复杂任务,通常只有两个选择。第一个选择是,每次都重新写一遍很长的 prompt,把背景、格式要求、注意事项都说一遍,这样很累,而且容易漏掉细节,效果也不稳定。第二个选择是,干脆自己搭一个定制化的 agent 系统,写代码、接 API、做工程化部署,这个投入就更大了,对很多非工程背景的业务人员来说,门槛太高。

Skills 这个功能,就是想在这两个极端之间,找到一个甜蜜点。它本质上是一份叫做 SKILL 点 md 的说明文件,你把它教给 Claude,告诉它在遇到某一类具体场景的时候应该怎么做。这就好像你请了一个非常聪明、什么都懂一点的新员工,但是你们公司有一些特殊的流程,比如财务报表必须用哪种格式,合同审核必须走哪几步,这些是这位新员工不可能凭空知道的。Skills 就是你写给这位新员工的一本岗位操作手册,他遇到对应的场景,就会主动翻出这本手册来照着做。

文章里有一句话我觉得说得特别到位,它说 Skills 的力量在于,它能够把组织里的隐性知识沉淀下来,标准化输出结果,并且处理那些原本需要反复解释,或者要专门投入资源去搭建定制 agent 才能完成的复杂多步骤工作流。换句话说,Skills 解决的核心问题,就是知识沉淀和复用的问题。

好,理解了背景,我们来看文章里讲的最核心的几个部分。我把它拆成五个层次来讲:第一,什么是好技能的起点;第二,三个核心组件,尤其是描述这一项;第三,怎么把技能部署到不同平台;第四,怎么系统性地测试一个技能;第五,几条贯穿全文的设计原则,以及文章给出的几个真实范例里藏着的巧思。

我们先说第一层,起点。文章一开始就强调,在你动手写任何东西之前,先把问题想清楚。一个强的技能,要解决的是具体的、有可衡量结果的需求。文章举了个例子,如果你写的描述是帮我处理一些财务方面的事情,这个描述就太模糊了。但如果你写的是,从 PDF 文件里提取财务数据,并且格式化成 CSV 表格,这个描述就清晰多了,因为它明确说明了输入格式是什么、要执行什么操作、期望得到什么样的输出。

这其实跟我们写产品需求文档,或者写工程任务卡片是一个道理。你给团队提需求,如果只说帮我优化一下体验,大家是没法干活的,必须要把输入是什么、动作是什么、产出标准是什么这三件事说清楚。文章建议你在动笔之前问自己四个问题:这个技能具体要完成什么任务?什么场景应该触发它?成功的标准是什么?有哪些边缘情况或者限制条件?这四个问题想透了,你才有资格开始写。

第二层,也是我认为整篇文章里最关键的一部分,就是一个技能的三大组件:名字、描述、指令。文章特别强调了一句话,我觉得值得反复划重点,那就是,名字和描述,是 SKILL 点 md 文件里唯一会影响触发的两个部分。也就是说,Claude 决定要不要去调用某个技能、要不要去翻这本操作手册,完全取决于这两项写得好不好,跟你后面写了多详细的执行步骤,在触发这个环节上是没有关系的。

名字这一项相对简单,要求是简单直白,用小写字母加连字符,比如说 pdf-editor,或者 brand-guidelines,名字要短、要清楚,一眼就能看出这个技能大概是干什么的。

描述这一项,文章说它是最关键的组件,因为它决定了技能什么时候会被激活。写描述的时候,你要站在 Claude 的角度去思考,重点描述触发条件、能力范围,以及使用场景。一个强的描述需要同时具备几个要素:具体的能力点,清晰的触发信号,相关的上下文,以及明确的边界。

文章给了一个对比特别直观。弱的描述是这样写的,这个技能可以帮助处理 PDF 和文档相关的事情。听起来好像没错,但其实信息量几乎为零,Claude 看到这句话,很难判断什么时候该用它、什么时候不该用。强的描述是这样写的,这是一个全面的 PDF 处理工具箱,可以提取文本和表格,创建新的 PDF,合并或拆分文档,处理表单。当 Claude 需要填写 PDF 表单,或者需要以程序化的方式批量处理、生成、分析 PDF 文档时使用,适用于文档工作流和批量操作场景,不适用于简单的 PDF 查看或者基础格式转换。

你听这两段描述的差别,强版本里给了 Claude 好几个不同维度的信息,具体的动词,比如提取、创建、合并;具体的使用场景,比如填表单、批量操作;还有明确的边界,告诉 Claude 什么时候不要用这个技能。这种边界条件其实特别重要,很多人写技能描述的时候只想着我什么时候要用它,却忘了告诉系统什么时候不要用它,结果就是技能被过度触发,在不该出现的场景里也跳出来添乱。

第三个组件是指令,也就是具体的执行步骤。指令应该写得结构化、可扫描、可执行,用 markdown 的标题、用列表呈现选项、用代码块展示示例。文章建议的结构层次是,概览、前提条件、执行步骤、示例、错误处理,以及限制说明。复杂的工作流,要拆解成一个个有明确输入输出的阶段。同时,指令里要包含具体的正确用法示例,也要明确说明这个技能不能做什么,这样可以防止误用,也能管理好使用者的预期。文章还提到一点很实用,SKILL 点 md 文件本身可以引用额外的参考文件和资源,给 Claude 提供更细致的指导,这个我们后面讲限制原则的时候还会展开。

第四层,我们说说怎么把写好的技能真正用起来,部署到不同的 Claude 使用场景里。文章列了三种主要的入口。第一种是 Claude 点 ai 这个网页应用,你在设置里添加自定义技能,不过需要提一句,这需要 Pro、Max、Team 或者 Enterprise 套餐,并且要开启代码执行功能,而且在网页端上传的技能是个人独享的,不会自动在整个组织里共享,管理员也没法统一管理,这是当时的限制,大家用的时候可以留意一下现在是否已经更新。第二种入口是 Claude Code,也就是命令行编程工具,你在项目或者插件根目录下建一个叫 skills 的文件夹,里面放技能文件夹,每个文件夹里放一份 SKILL 点 md,插件装好之后 Claude 会自动发现并使用。第三种入口是 Claude 开发者平台,通过 Skills 这个 API 接口,用 POST 请求把文件上传上去,文章里给了一个具体的 curl 命令示例,包括需要带哪些请求头。这三种入口其实对应着三种不同的用户画像,普通用户走网页端,工程师走命令行,而要做规模化集成的开发者走 API,这个设计思路还是挺清晰的,照顾到了不同技术水平的人群。

第五层,我们重点说说测试,这一部分我觉得是整篇文章里最有实操价值的部分,因为它给出了一个非常具体的测试矩阵,大家如果真的要动手写技能,这部分值得反复回看。

文章说,在正式上线一个技能之前,一定要用真实的场景去测试它,系统性的测试能够暴露出指令里的漏洞、描述里的歧义,以及只有在实际使用中才会冒出来的边缘情况。

测试矩阵建议覆盖三类场景。第一类是正常操作,用这个技能应该完美处理的典型请求去测试。比如你做了一个财务分析的技能,那就试着说分析一下微软最新的财报,或者给这份十 K 文件做一个数据包,这些基础测试用来确认你的指令确实能按预期工作。

第二类是边缘情况,用不完整或者不寻常的输入去测试。比如数据缺失的时候会怎样,文件格式不符合预期的时候会怎样,用户给的指令很模糊的时候会怎样。一个好的技能应该能优雅地处理这些情况,要么给出降级但仍然有用的结果,要么明确告诉用户还需要补充什么信息才能继续。

第三类是范围之外的请求,用一些看起来相关、但实际上不应该触发这个技能的任务去测试。文章举的例子很形象,如果你做了一个 NDA,也就是保密协议审核的技能,那就试着请求审核一下这份雇佣合同,或者分析一下这份租赁协议。一个边界清晰的技能,这时候应该保持沉默,把任务让给其他技能或者 Claude 的通用能力去处理。

除了这三类基础测试,文章还建议做更深一层的验证,分成触发测试和功能测试两组。触发测试要回答的问题是,这个技能在该激活的时候有没有激活,在不该激活的时候有没有按兵不动。这里有个细节我觉得特别值得注意,文章建议你既要用明确指名道姓的请求去测试,比如说请用财务数据包这个技能来分析这家公司,也要用自然语言的请求去测试,比如说帮我理解一下这家公司的财务状况。这两种说法背后是同一个意图,但表达方式天差地别,一个真正鲁棒的技能,应该在两种说法下都能正确触发。同时还要用相似但实质不同的请求去检验边界在哪里。功能测试这一组,关注的是输出的一致性,也就是相似的输入,跑多次会不会得到差不多的结果;还有可用性,一个完全不懂这个领域的人能不能顺利用起来;以及文档准确性,你写的示例是不是真的对应着实际的行为。

讲到这里我想插一句我自己的看法。我个人认为这套测试矩阵的价值,其实不只局限在 Skills 这个具体功能上,它本质上是一套很通用的提示工程评估框架,放到任何需要校验 AI 行为边界的场景里都适用,比如你在设计一个客服机器人的话术规则,或者在调试一个内部知识库问答系统的检索策略,都可以套用这个正常场景、边缘场景、越界场景三分法去做系统性的压力测试,这比凭感觉调一调好多了。

接下来文章总结了几条贯穿全文的设计原则,我挑几个讲讲。

第一条原则是,不要凭空臆想着去写技能。只在你真的反复遇到某个任务的时候才去做这个技能。文章给了一个判断标准,问自己两个问题,这件事我至少已经做过五次了吗,我未来至少还会再做十次吗,如果两个答案都是肯定的,那做一个技能就是值得的投入。这个标准其实挺朴素的,但很实用,它提醒我们技能不是用来炫技的,而是用来解决真实的重复性劳动的。

第二条原则是,要明确告诉 Claude 什么样的输出才算好的输出。如果你在做财务报告的技能,就要在指令里写清楚必须包含哪些章节、格式标准是什么、需要做哪些校验、质量门槛在哪里,把这些标准写进指令,Claude 才能拿着这把尺子去自我检查。

第三条原则,文章提到 Anthropic 自己提供了一个叫 skill-creator 的技能,专门用来引导你创建结构良好的新技能,它会主动问你一些澄清性的问题,给你的描述提出改进建议,帮你把指令格式调整得更规范。这个工具在 GitHub 的 Skills 仓库里,也可以直接在 Claude 点 ai 上用,文章特别建议,如果你是刚开始写第一批技能,这个工具会特别有价值,相当于有个向导陪着你走完整个流程。

接下来我想重点讲讲文章里关于上下文窗口管理的那部分,这个我觉得是很容易被忽略,但其实非常重要的设计细节。

文章提醒,写技能的时候要避免把不必要的内容塞满上下文窗口。每写一段信息进去之前,你要想清楚,这段信息是不是每次都需要被加载,还是只在特定条件下才需要加载。这听起来有点像我们做软件架构设计时讲的按需加载,懒加载的思路,只不过这里加载的对象不是代码模块,而是知识内容。

文章提出了一个叫菜单式方法的设计模式。如果你的技能覆盖了多个不同的流程或者选项,那么 SKILL 点 md 这个主文件里应该只描述有哪些可选项,然后用相对路径去引用分别存放在不同文件里的具体内容。Claude 看到任务之后,只会去读取跟当前任务相关的那一个文件,其他文件在这次对话里完全不会被打开,自然也不会占用上下文空间。文章特别强调了一点,这些分开存放的文件之间不需要是互相排斥的关系,核心原则就是把内容拆成合理的小块,让 Claude 根据手头的任务自己去挑选需要的那一块。

这个设计思路在文章给出的范例里体现得特别明显,我们来看看那个处理 Word 文档的 docx 技能范例。这个技能的主文件里有一棵清晰的决策树,告诉 Claude 如果是读取分析内容该走哪条路,如果是创建新文档该走哪条路,如果是编辑已有文档,又要分自己的文档和别人的文档,普通文档和法律、学术、商业、政府类的正式文档,分别对应不同的处理流程。其中编辑别人的文档,或者编辑正式文档,文章要求必须走一个叫redlining,也就是修订模式的工作流,这个工作流里详细讲了怎么用 pandoc 把文档转成带修订痕迹的 markdown,怎么把改动拆解成一批一批,每批三到十个改动,怎么只标记真正变化的文字而不是整句话重新写一遍,这样修订记录才专业、才方便审阅。这个范例里还提到一个细节,主文件要求 Claude 必须完整读完两份引用文件,一份大概五百行,一份大概六百行,而且明确写着不许设置任何范围限制,要从头到尾完整读完。这其实就是渐进式加载思路的具体体现,主文件保持精简,只在真正需要深入某个具体操作的时候,才把详细的参考文件完整加载进来。

第二个范例是品牌视觉规范这个技能,叫 brand-guidelines。这个技能写得很直接,把 Anthropic 官方的品牌色号,深色、浅色、中灰、浅灰,还有三个强调色橙色、蓝色、绿色,具体的十六进制色值都写在指令里,还规定了标题用什么字体、正文用什么字体、字号多大的标题要用什么样式。这个范例之所以被文章称作强范例,理由很朴素,它提供的是 Claude 本来根本不可能知道的精确信息,比如这家公司内部的具体色号和字体名称,这种信息是没办法靠通用知识推断出来的,必须靠技能文件明确地教给它。

第三个范例是前端设计技能,叫 frontend-design。这个范例我觉得很有意思,它要解决的问题是,大模型生成的网页设计,经常会陷入一种千篇一律的所谓 AI 风,比如说总是用 Inter 这种字体,总是用紫色渐变配白色背景这种配色,布局也总是大同小异。这个技能的指令里明确列出了一份黑名单,禁止使用过度泛滥的字体家族,禁止使用俗套的紫色渐变这类配色方案,要求 Claude 在设计之前先做一个旗帜鲜明的美学方向选择,极简、极繁、复古未来、有机自然、奢华精致,选一个方向然后坚决地执行下去,而不是各种风格混着来,变成四不像。这个范例给我的启发是,技能不光可以用来教 Claude 怎么做一件正确的事,也可以用来告诉它哪些套路是要主动避开的,这种负向约束在写技能指令的时候,价值其实和正向指导是一样大的。

讲完了这几个核心范例,我们再说回团队协作这个层面。文章提到,不管你的团队规模大小,都建议建立一个共享的技能文档仓库。对于小团队,可以简单一些,用一个模板格式记录名字、描述、指令和版本信息就够了。对于中大型团队,文章建议建立一套更正式的治理流程,给每个业务领域指定技能负责人,比如财务、法务、市场各自有人专门维护对应的技能;维护一个中心化的知识库或者共享文件夹作为技能图书馆;给每个技能配上使用示例和常见问题排查指南;对技能做版本管理,变更要记录在更新日志里;并且定期,比如按季度,回顾一下哪些技能该更新、哪些已经过时该退役了。文章里还特别提醒了一句,目前在 Claude 点 ai 网页端,Skills 还是个人独享的,组织级的统一管理和共享功能据说正在路上,所以哪怕暂时没有这个功能,也建议大家先把共享文档仓库这套治理习惯建立起来,等官方功能上线的时候可以无缝接上。

好,讲到这里,这篇文章的核心内容基本上讲完了。我们来想一想,这些东西对于正在听这期节目的工程师和 AI 从业者,实际工作里能怎么用。

如果你所在的团队里,有那种每周都要重复处理的任务,比方说每次新人入职都要写一遍同样格式的周报模板说明,或者每次客户合同审核都要把同样几条法律风险点检查一遍,或者数据分析师每次接到新需求都要解释一遍公司内部报表的格式规范,这些场景就是 Skills 最适合发挥作用的地方。你可以先按照文章给的那四个问题,问清楚这个技能要解决什么、什么场景触发、成功标准是什么、边缘情况有哪些,然后照着名字、描述、指令这三件套去写一份 SKILL 点 md,描述这一项一定要花最多的心思去打磨,因为它直接决定了触发的准确度。写完之后,别急着上线,按照文章给的三类测试矩阵,正常场景、边缘场景、越界场景,各设计几个测试用例跑一遍,看看触发是不是符合预期,输出是不是稳定。

需要避开的坑,文章里其实也点得很清楚。第一个坑是描述写得太笼统,导致技能该触发的时候不触发,或者不该触发的时候乱触发,这两种情况其实都源于同一个问题,就是描述里缺少具体的能力点、触发场景和边界条件。第二个坑是把所有内容一股脑塞进一个文件,不考虑渐进式加载,结果上下文窗口被无关信息占满,该用上的时候反而被挤占了空间,这时候就该考虑用菜单式方法,把内容拆分成多个按需加载的文件。第三个坑是凭空臆想着去写技能,而不是从真实的重复性任务出发,这样写出来的技能往往不接地气,实际用起来才发现指令里漏了一堆现实场景里才会出现的边界情况。

好,我们来做个总结。这篇文章用一种很工程化、很可操作的方式,把如何写好一个 Claude 技能这件事拆解成了清晰的步骤,从需求定义,到名字描述指令三件套,到跨平台部署,到系统性测试,再到上下文管理和团队治理,逻辑链条相当完整。我个人认为这篇文章最有价值的部分,不是那几个具体的范例,而是它反复强调的那种产品思维,把技能当成可以测试、可以迭代、需要明确成功标准的产品来对待,而不是写一段提示词就完事了。如果你最近正好在琢磨怎么把团队里那些反复出现的知识沉淀下来,这篇文章值得你完整读一遍原文,跟着它给的检查清单,动手写一个属于自己团队的第一个技能试试看。
