大家好,欢迎收听 Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是《Generate better prompts in the developer console》,翻译过来就是《在开发者控制台里生成更好的提示词》。这篇文章发表于二零二四年五月二十号,出自 Anthropic 官方博客。

先说一句,这篇文章距离今天已经过去两年多了,在 AI 这个行业里,两年多的时间跨度其实相当于好几个技术世代。所以听这期节目的时候,大家心里要先有个时间坐标:二零二四年五月,那个时候 Claude 三系列刚刚发布没多久,提示词工程这个概念在开发者社群里正处于从"玄学"向"工程学科"过渡的关键阶段。那时候大家普遍的痛点是什么呢,就是写一个好用的 prompt,门槛比想象中高很多。同一个任务,不同的人写出来的提示词,效果可能天差地别,而且很难判断到底是模型能力不够,还是你的提示词本身没写到位。今天我们站在两千零二十六年回头看这篇文章,会发现里面提到的很多基础技巧,其实已经内化成了整个行业的常识,但它的诞生背景和解决思路,依然值得我们仔细拆一遍,因为这能帮我们理解,提示词工程从手艺活儿走向标准化的第一步,到底是怎么走出来的。

我们先说说背景。在这篇文章发布之前,提示词工程基本上是一门靠经验积累的手艺。你去网上搜"如何写好 prompt",会看到五花八门的教程,有人说要多给例子,有人说要分步骤思考,有人说要把任务讲得越细越好。这些建议单独看都没错,但问题在于,新手往往不知道该怎么把这些零散的技巧组合起来,变成一份真正能跑通、能稳定输出的提示词。更现实的问题是,即便是有经验的提示词工程师,每次面对一个新任务,也要花不少时间从零开始构思结构、调整措辞、反复试错。这就好比你会做饭,但每次做一道新菜都要重新设计菜谱、重新试调料比例,即便你经验丰富,这个过程依然耗时间。Anthropic 在这个背景下推出了提示词生成器,它的核心思路非常直接:既然 Claude 本身已经具备很强的语言理解和推理能力,那为什么不让 Claude 自己来帮你写提示词呢。你只需要用人话描述清楚你想让模型做什么、希望输出什么样的格式,Claude 就会调用一系列提示词工程的最佳实践,自动生成一份结构完整、逻辑清晰的提示词模板。这个功能上线之后,直接嵌入在 Anthropic 的开发者控制台里,意味着开发者不需要跳出工作流程,在写代码、调试应用的过程中就能顺手生成一份高质量的初始提示词。

接下来我们把文章里提到的几个核心技巧,一个一个拆开讲清楚。

第一个技巧,叫角色设定,英文是 role setting。这个技巧的意思是,在提示词的开头,明确告诉 Claude 它现在要扮演什么角色,具备什么样的专业身份。文章里举了一个内容审核的例子,提示词的开头是这样写的:你将扮演一名内容审核员,根据提供的内容审核政策,把聊天记录分类为通过或者拒绝。这句话看起来很简单,但它起到的作用其实不小。你可以把这个想象成你去请一个顾问帮你看材料,如果你只说帮我看看这份文件,对方可能不知道该用什么标准去看;但如果你说请你以一名资深审核专家的身份,按照这份政策审核这份材料,对方立刻就知道自己应该调用哪一套知识体系、用什么语气和标准去做判断。对 Claude 来说也是类似的道理,角色设定相当于给模型一个明确的"人格滤镜",让它在后续推理的时候,默认调用与这个角色相关的知识和判断标准,而不是泛泛地、笼统地回答问题。这个技巧听起来简单,但很多新手写提示词的时候恰恰会漏掉这一步,直接就甩任务描述,模型当然也能做,但效果往往不如先给它一个清晰的身份定位。

第二个技巧,是思维链推理,英文叫 chain of thought reasoning。这个概念现在大家应该都很熟悉了,但放在二零二四年的时间点上,这还是一个相对前沿、需要专门强调的技巧。思维链的核心思想是,给模型一段"草稿空间",让它在正式给出答案之前,先把自己的思考过程写出来。文章里举了一个产品推荐的例子,提示词里写道,在一个叫 scratchpad 的标签里,先头脑风暴出三个不同的产品推荐,基于这位顾客过去的交易记录,并且针对每一个推荐都简要说明理由,解释为什么觉得这个推荐适合这位顾客。你可以把这个过程类比成考试的时候老师要求你写出解题步骤,而不是只写最终答案。为什么这样做效果会更好呢,这其实跟语言模型生成文字的方式有关。模型是一边生成一边往前推进的,如果你直接要求它给出最终结论,它没有机会"回头检查"自己的逻辑,只能一次性把答案吐出来;但如果你先让它把思路写在草稿区里,它就相当于把一个复杂问题拆解成了多个小步骤,每一步都建立在前一步的基础上,这样最终给出的答案往往更连贯、更经得起推敲。尤其是面对那种需要多方面权衡的任务,比如内容审核需要同时考虑多条政策条款,或者产品推荐需要综合考虑用户的多次购买记录,思维链能明显提升输出质量。当然,这里有个小细节值得提一下,草稿区的内容在很多场景下是不需要展示给最终用户的,你可以在拿到模型输出之后,通过简单的文本处理把草稿部分截掉,只展示最终结论,这样既保留了思维链带来的质量提升,又不会让用户看到啰嗦的中间过程。

第三个技巧,是用 XML 标签给变量和长文本划定边界,这个技巧在文章里被反复提到,我个人觉得它其实是三个技巧里最容易被低估、但实操价值最高的一个。简单说,就是当你的提示词里有需要替换成具体内容的"变量",比如用户上传的代码、用户的提问、需要翻译的文本,这些内容如果比较长或者比较复杂,最好用类似 code 这样的标签把它包起来,清楚地告诉模型这一段是变量内容,跟前后的指令文字是分开的。文章里举了一个代码翻译的例子,提示词是这样写的,你的任务是把一段代码从另一种编程语言翻译成 Python,这是需要翻译的代码,然后用 code 标签把 CODE 这个变量包起来,再往后说这段代码是用某种语言写的,这里的语言变量就直接写在句子里,不需要额外用标签包起来。这个细节其实很有意思,它说明 Claude 在设计模板的时候,是有意识地区分了变量的"复杂度"。像编程语言名称这种简短、单一、不会产生歧义的变量,直接写在句子里就足够清晰;但像一整段代码这种内容多、格式复杂、容易和周围指令文字混在一起的变量,就必须用标签明确隔开,否则模型在解析提示词的时候,可能会搞不清楚哪部分是你给它的指令,哪部分是需要它处理的具体内容。这就好比你写一份合同,简单的日期、金额可以直接写在正文里,但附件里的复杂条款列表,你一定会单独列出来,用清晰的标题和分隔线隔开,道理是一样的,结构清晰直接决定了对方理解的准确度。

除了这三个核心技巧,文章里还提到了一个细节,就是 Claude 在生成模板的时候,有时候会主动写一些示例输入和示例输出,用来给自己的回答定调子。这个其实也是提示词工程里一个经典手法,叫少样本示例,英文是 few-shot examples。意思是,你不光告诉模型要做什么,还顺手给它看几个"标准答案"长什么样,模型看到示例之后,会更准确地把握你想要的输出格式、语气、详细程度。文章特别强调,这些示例是可以编辑的,如果你觉得 Claude 生成的示例不够贴合你的实际需求,你完全可以自己动手改一改,把示例换成更贴近你业务场景的内容。

那么这个提示词生成器本身,又是怎么实现的呢。文章里专门讲了一段,说这个生成器的背后,其实也是一份很长的提示词,而且这份提示词本身就大量运用了我们刚才说的那些技巧,这一点我觉得挺有意思,算是一种自我指涉,用提示词工程的方法,去构建一个能生成提示词的提示词。具体来说,这份背后的长提示词里包含了大量示例,展示了从一句任务描述到一份完整提示词模板之间的转化过程,这其实就是前面说的少样本示例技巧在更高层级上的应用。同时,这份长提示词还会引导 Claude 先规划模板的结构,然后再动手写,这其实就是思维链技巧的应用,先想清楚框架,再填充细节。另外,这份提示词还有一个文章里用了一个挺形象的词,叫"脊梁",就是用 XML 标签标记每个部分的开头和结尾,让整篇提示词的结构一目了然。所以你可以看到,这个生成器与其说是某种全新的黑科技,不如说是把提示词工程里已经被验证有效的几个技巧,系统性地组合起来,然后用这套组合本身去生成新的提示词,形成一个闭环。

文章还提到一个具体的格式细节,就是生成出来的模板里,变量会用一种叫 handlebars 的标记方式来表示,简单说就是用两层花括号把变量名包起来,比如双花括号 CODE 双花括号这样的写法。这个设计的好处在于,变量的边界非常清晰,你拿到模板之后,只需要在代码层面做一次简单的字符串替换,把这些花括号占位符换成实际的内容就行,不需要再额外解析和判断。这对于把提示词集成到实际应用代码里,是个挺友好的设计。

讲到这里,我们可以聊聊这篇文章里提到的一个真实案例,就是一家叫 ZoomInfo 的公司。ZoomInfo 是一家做营销和销售情报的平台,他们用 Claude 来给客户生成可执行的建议。文章引用了他们一位首席数据科学家的话,他说这个新的提示词生成功能,让他们能更快地达到生产级别的输出质量,这个功能向他展示了一些他之前没有用过的提升性能的技巧,大大减少了调试应用所花的时间。具体的数字也很直接,他们用这个功能搭建了一个全新的 RAG 应用,也就是检索增强生成应用,只用了几天时间就做出了最小可行产品,而且把提示词调优所需要的时间减少了百分之八十。这个数字其实挺值得琢磨的,百分之八十的时间节省,意味着原本可能需要两三周反复试错才能调好的提示词,现在用生成器先打个底,再做局部微调,可能几天之内就能搞定。这对于任何一个需要快速验证产品想法的团队来说,都是实打实的效率提升。

好,讲完了文章本身的内容,我们来聊聊这些东西对今天的我们,到底有什么实际意义。

首先要说清楚一点,这个具体的"提示词生成器"按钮,作为二零二四年五月控制台里的一个功能入口,它的具体界面和操作方式,这两年里很可能已经迭代过好几轮了,如果你现在打开 Anthropic 的开发者控制台,看到的交互细节未必和文章里描述的完全一样。但是,这篇文章真正值得我们带走的,不是这个按钮本身,而是它背后总结出来的那套提示词工程方法论。换句话说,即便这个具体功能换了样子,甚至被整合进了更大的工作流里,角色设定、思维链、变量标签化这几个原则,依然是写好提示词的底层逻辑,这一点在今天用 Claude 三点五、甚至更新的模型版本,依然成立。

那具体到我们日常的工作里,这些技巧可以怎么用呢。我给大家举几个场景。

第一个场景,假设你在做一个客服机器人,需要让 Claude 根据用户的提问和历史工单,判断这个问题应该转给哪个部门处理。如果你只是简单地把任务甩给模型,说帮我判断这个问题该转给哪个部门,效果可能不稳定。但如果你应用角色设定的技巧,先告诉 Claude 你将扮演一名经验丰富的客服分单专员,熟悉公司各部门的职责划分,然后再给出具体的分类规则和待判断的工单内容,模型给出的判断往往会更准确、更符合业务预期。这一步几乎不需要额外成本,就是多写一两句话,但带来的提升是实打实的。

第二个场景,如果你的任务比较复杂,需要模型做多步判断,比如审核一段广告文案是否违反多条平台规则,这时候思维链就特别有用。你可以在提示词里明确要求,先在一个标签里逐条核对每一项规则,写下判断理由,再给出最终结论。这样做的好处是,你不仅能拿到最终结果,还能拿到模型的推理过程,这对于后续排查为什么某个案例判断错了,是非常有价值的调试线索。如果你完全跳过这一步,直接要结论,一旦出现误判,你很难知道模型到底是哪个环节出了问题。

第三个场景,关于变量标签化,这一点在我们做 RAG 应用,也就是检索增强生成应用的时候特别重要。比如你检索回来一段很长的参考资料,要塞进提示词里,再加上用户的原始问题,这时候千万不要把参考资料和问题混在一段话里,一定要用清晰的标签把检索到的资料、用户的问题、输出格式要求,分别隔开。这样不仅模型理解得更准确,你自己在调试的时候,也能很清楚地看出每个部分输入了什么内容,排查问题效率会高很多。

还有一个我想多说两句的点,就是文章里提到的少样本示例,可以编辑这件事。我觉得这其实暗示了一种很健康的工作方式,就是不要指望一次生成就完美,而是把生成器产出的模板当作一个高质量的起点,然后根据自己实际遇到的边界案例,不断往里面补充示例、调整措辞。这跟我们今天讨论 agent 开发、讨论提示词版本管理的思路是完全一致的,提示词从来不是写一次就定型的东西,而是需要持续迭代、持续根据真实数据反馈来打磨的活文档。

如果让我个人评价一下,我会说,这篇文章在当时最大的价值,不是介绍了什么颠覆性的新技术,而是把一堆原本散落在各种博客、论坛、最佳实践清单里的提示词技巧,第一次以一种产品化的方式集成进了开发流程里。这其实反映了整个行业在那个阶段的一个普遍诉求,就是从单纯依靠个人经验的手艺,逐渐走向有工具、有标准、可复制的工程方法。放在今天看,提示词工程这件事的形态又往前走了一大步,我们现在更多在讨论怎么给 agent 设计工具调用流程,怎么管理长上下文里的信息密度,怎么做系统性的评测来衡量提示词改动带来的实际效果变化,但角色设定、分步思考、结构化输入这几个基本功,依然是所有更复杂技巧的地基。我个人觉得,如果你是刚入行不久的开发者,把这篇文章当作提示词工程的入门教材来读,依然是一个性价比很高的选择,因为这几条原则到今天也没有过时。
