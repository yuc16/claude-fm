先说说为什么这篇文章值得你花二十多分钟听完。如果你平时在用 Claude Code 写代码、做代码审查、或者跑一些比较长的自动化任务,那么从 Opus 四点六升级到 Opus 四点七的这个过程,其实不是无脳的版本号加一,它背后藏着几个会直接影响你体验的变化。比如说,同样的提示词,在新模型上可能会消耗更多 token,响应节奏也变了,默认行为也不一样了。如果你什么都不调整,直接把老的用法照搬过来,很可能会觉得新模型变慢了、变贵了,甚至觉得它不听话了。但如果你了解了这篇文章里讲的几个调整方法,你会发现 Opus 四点七其实是目前为止 Anthropic 在编码、企业工作流和长时间自主任务上最强的一个通用模型。所以这篇文章本质上是一份升级指南,告诉你怎么把这把更锋利的刀用得更顺手。

我们先铺垫一下背景。在 Opus 四点七发布之前,行业里对大模型编程助手的一个普遍痛点是什么呢,是模型在长任务里容易跑偏,容易在多轮对话之后忘记早期上下文,或者需要用户不停地介入确认、纠偏。很多团队在用 AI 写代码的时候,实际上还是把它当成一个需要手把手带的初级程序员,你给一行指令,它做一点,你看一眼,再给下一行指令。这种用法的问题在于,人始终是瓶颈,模型的能力再强,也被你的反应速度和你愿意盯着屏幕的时间给限制住了。

Anthropic 在这篇文章里给出的核心论断,其实是一个用人方式上的转变。他们说,Opus 四点七处理模糊任务的能力比四点六强很多,发现 bug 和做代码审查的能力也提升明显,跨会话保持上下文的可靠性更高,面对不那么明确的任务时,需要的指导也更少。换句话说,这个模型已经具备了独立完成一段相对完整工作的能力,而不只是执行单步指令的能力。基于这一点,文章给出的核心建议是,与其把 Claude 当成一个需要你逐行指导的结对编程伙伴,你应该把它当成一个你正在委托任务的资深工程师。这句话我觉得是整篇文章里最重要的一句定调,后面所有具体的技巧,本质上都是在为这个角色转变服务。

接下来我们拆解几个最关键的技术点。第一个,是关于 token 用量和模型行为为什么会变化。文章里提到,这次升级带来了两个会直接影响 token 消耗的改动。第一个是更新了 tokenizer,也就是模型用来把文字切分成 token 的那套编码规则发生了变化,这个变化本身就会让同样一段文字对应的 token 数量和过去不完全一样。第二个,也是更值得关注的一点,Opus 四点七在更高的效力等级上,有更强的倾向去多思考,尤其是在一段比较长的会话里,越往后面的轮次,模型倾向于想得更多。这两个因素叠加起来,就导致很多人直接替换模型之后,会发现账单或者等待时间比预期要高。

这里有个细节特别值得展开讲一下,就是模型在自主异步任务和交互同步任务里的行为是不一样的。什么叫自主异步任务呢,简单说就是你一次性把任务说清楚,然后让模型自己跑完,中间不需要你来回插话,这种场景下往往只有一个用户回合。而交互同步任务,指的是你跟模型来回好几轮对话,边讨论边调整方向。文章里特别强调,在交互式场景里,模型在每一次用户发言之后,会进行更多的推理。这样做的好处是,它在长会话里的连贯性更好,指令遵循更准确,代码质量也更稳定。但代价就是,会消耗更多 token。

我觉得这一点用一个类比会更好理解。你可以把它想象成,你在指导一个刚入职但能力很强的工程师做一个项目。如果你每隔五分钟就走过去问他进展怎么样,提一个小建议,改一下方向,他每次被你打断之后,都需要重新在脑子里捋一遍上下文,确认你刚才说的话跟之前的目标是否一致,有没有冲突,这个重新捋思路的过程本身就要花时间和精力。而如果你一开始就把需求、约束条件、验收标准一次性讲清楚,然后让他自己埋头把整个模块做完,中间不去打扰他,他的整体效率反而更高,因为他不需要反复在不同上下文之间切换。Opus 四点七的这个特性,其实就是把这个道理在工程层面给放大体现出来了。轮次越多,模型为了保持连贯性付出的推理成本就越高,这是一个此消彼长的关系,而不是单纯的越聊越快。

理解了这个机制之后,文章给出了第二个核心内容,也就是一套具体的使用建议,我们一条一条来看。

第一条建议,是在第一轮对话里就把任务说清楚。文章原话强调,一个写得好的任务描述,应该包含意图,也就是你到底想达成什么目标,约束条件,比如有哪些技术限制或者业务规则不能违反,验收标准,也就是怎么判断这个任务做完了、做对了,以及相关的文件位置,告诉模型应该去看哪些文件。如果你把这些信息一次性给全,Opus 四点七就有足够的上下文来交付一个高质量的结果。反过来,如果你的提示词本身是模糊的,然后指望着在接下来的好多轮对话里慢慢把细节补全,这种做法不仅会拉低 token 使用效率,有时候还会拉低整体的输出质量。这一点其实跟我们刚才讲的轮次成本是直接对应的,你每多问一句,模型就要多做一次上下文整合的推理,而这个推理成本本来是可以靠你提前把话说清楚来省下来的。

第二条建议,是尽量减少必须的用户交互次数。文章里讲得很直白,每一次用户发言都会带来额外的推理开销,所以建议你把想问的问题打包在一起问,一次性把模型需要的上下文都给到,让它能够持续不断地往前推进,而不是走两步停一下等你确认。这个建议听起来简单,但其实是反直觉的,因为很多人用惯了那种边走边看、随时纠偏的工作方式,觉得这样更有控制感。但对于 Opus 四点七来说,这种打断式的工作方式恰恰是效率最低的用法。

第三条建议,是在合适的场景下使用 auto mode,也就是自动模式。文章说,对于那些你信任模型可以安全执行、不需要频繁确认的任务,auto mode 能够大幅缩短任务的整体周转时间。它尤其适合那种你已经把上下文一次性给全了的长时间运行任务。在这篇文章发表的时候,auto mode 刚刚针对 Claude Code Max 用户开放了研究预览,你可以用 Shift 加 Tab 这个组合键来切换开关。我自己理解这个功能,其实就是把刚才那个第一条第二条建议给做成了一个产品层面的开关,你把活儿交代清楚,然后真正地放手让它去跑,而不是表面上说放手,实际上还是隔几分钟看一眼。

第四条建议,是给已完成的任务设置通知。文章建议你可以让 Claude 在完成任务之后播放一个提示音,而且它本身就有能力帮你创建基于 hook 机制的通知。这个建议看起来很小,但其实跟前面几条建议是一套组合拳。如果你已经决定把一个长任务完整地交给模型去跑,而不去中途打断它,那你自然就需要一个机制,在它做完之后能第一时间通知到你,而不是你时不时跑去看一眼终端。

讲完了使用习惯层面的建议,我们再来看第三个核心内容,也是这篇文章里信息量最大的部分,关于效力等级的调整。

什么是效力等级,你可以理解成模型在多努力地思考这个问题上,给你提供的一个可调档位。文章里提到,Opus 四点七在 Claude Code 里的默认效力等级现在是 xhigh,这是一个全新的等级,位置在 high 和 max 之间,目的是让用户在面对困难问题时,能够更精细地控制推理深度和响应延迟之间的取舍。官方明确建议,大多数 agentic 编码工作,也就是那种需要模型自主规划、自主执行多个步骤的编码任务,都应该用 xhigh 这个档位,尤其是在一些对智能水平要求比较敏感的任务上,比如设计 API 接口和数据库 schema,迁移老旧代码,以及审查大型代码库。

文章里把效力等级从低到高排了一遍,分别是 medium 和 low,这两个是比较低的档位,适合简单直接、不太需要深度推理的任务,然后是 high,再往上就是新增的 xhigh,作为默认推荐档位,最高的是 max,留给那些真正棘手、值得不计代价投入推理资源的问题。文章特别提醒,如果你是从老模型升级过来的老用户,不要直接把你过去用习惯的那个效力等级原样搬过来,而是应该重新试验一下,因为同样名字的档位,在新模型上的实际表现已经不一样了。你甚至可以在同一个任务执行过程中,在不同的效力等级之间来回切换,来更精细地管理 token 用量和推理深度。

还有一个细节我觉得值得专门提一下,如果你是老的 Claude Code 用户,但是你自己从来没有手动设置过效力等级,系统会自动把你升级到 xhigh,当然你依然可以手动调整。这说明 Anthropic 是真心相信 xhigh 是大多数编码任务的最优默认选择,所以才会主动把存量用户也迁移过去,而不是只对新用户生效。

这里我想用一个类比帮大家理解效力等级到底意味着什么。你可以把它想象成,你雇了一个工程师,让他帮你处理不同难度的任务。如果只是改一个变量名,或者修一个一眼就能看出来的小拼写错误,你肯定不希望他坐下来闭目沉思十分钟,你希望他扫一眼就直接改完。但如果你让他设计一套新系统的数据库表结构,或者把一个用了八年的老系统迁移到新框架上,这种任务你反而希望他能多想一会儿,把边界情况、兼容性问题、未来扩展性都考虑进去,哪怕多花一点时间,也比草草了事强。效力等级,本质上就是把这种你脑子里天然会做的轻重判断,显式地变成了一个你可以直接控制的参数。而 xhigh 之所以被设为默认,是因为 Anthropic 判断,在典型的编码场景里,稍微多想一点带来的质量提升,通常是值得这一点额外延迟和 token 成本的。

第四个核心内容,是关于思考机制本身的一次比较大的架构调整,叫做自适应思考。这一点我觉得很多老用户容易踩坑,所以值得专门讲清楚。

在过去,Anthropic 的模型支持一种叫 Extended Thinking 的机制,你可以给模型设置一个固定的思考预算,告诉它最多可以用多少 token 来做内部推理,然后再给出最终答案。但文章里明确写了,固定思考预算的 Extended Thinking 在 Opus 四点七里不再支持了。取而代之的,是自适应思考。这个机制的核心思路是,思考变成了在每一步都可选的动作,模型自己决定在这一步要不要多想,想多久。它可以对简单的问题快速给出回应,在某一步明显不需要思考的时候直接跳过思考,把思考的 token 预算,投资在那些真正可能从深度推理中受益的地方。文章说,放到一整段自主运行的任务里来看,这种机制累积起来,既能带来更快的响应速度,也能带来更好的整体使用体验。

文章还特别提到一点,这一代的自适应思考相比之前有了明显改善,具体表现是模型变得不那么容易过度思考了。过度思考是什么意思呢,就是模型对一个本来很简单的问题,也要绕很大一圈、做很多不必要的推理才给出答案,这种行为既浪费 token,也拖慢响应。Opus 四点七在这方面做了优化,减少了这种不必要的绕圈子。

那如果你想自己控制模型的思考强度,该怎么办呢。文章给出了两个非常具体的提示词模板。如果你希望模型多想一点,可以直接在提示词里说,请仔细地、一步一步地思考之后再回答,这个问题比看起来更难。这句话相当于明确告诉模型,这不是一个可以随便应付的简单问题,值得投入更多推理。反过来,如果你希望模型少想一点、更快给出答案,可以说,优先快速响应而不是深入思考,如果不确定就直接给出答案。文章也很坦诚地指出,这样做确实能省 token,但在比较难的步骤上,你可能会牺牲一些准确性。我觉得这两句提示词模板特别实用,建议大家直接记下来,在日常使用中按需替换。

接下来是第五个核心内容,也是这篇文章里我认为最容易被忽略,但实际影响很大的一部分,就是默认行为的几个变化。如果你之前已经针对 Opus 四点六精心调过提示词或者整套工作流,这部分内容会直接关系到你的设置是不是还适用。

第一个变化,是回答长度现在会根据任务复杂度来校准。Opus 四点七不像四点六那样默认就比较啰嗦,简单的查询类问题,你会得到更短的回答,而开放式的分析类问题,你会得到更长、更详尽的回答。文章建议,如果你的场景对回答长度或者风格有特定要求,一定要在提示词里明确说出来。这里有个挺有意思的细节,文章特别提到,用正面的例子去描述你想要的风格,效果比用负面的禁止性指令要好。也就是说,与其告诉模型不要这样写,不如直接给它看一个你觉得理想的样例,告诉它照着这个调子来。

第二个变化,是模型现在调用工具的频率变低了,转而更多地依靠内部推理。文章说,这种调整在很多情况下反而能带来更好的结果,因为不是每个问题都需要去翻文件、去搜索才能回答,有时候模型凭借推理就能想清楚。但如果你的场景恰恰需要模型更积极地去搜索、去读文件,比如说你希望它在改代码之前先把整个相关模块的上下文都读一遍,那你就需要在提示词里明确地描述,什么时候应该用工具,为什么要用这个工具。

第三个变化,是模型默认生成的子代理数量变少了。这里先解释一下子代理是什么,在 Claude Code 里,主模型有时候会派生出一个或者多个子任务执行体,分别去处理某一部分工作,这就是子代理,有点像主管把任务拆给几个小组分别去做。Opus 四点七在要不要派生子代理这件事上变得更谨慎了,更倾向于自己一个人把活干完,而不是动不动就分发任务。文章提醒,如果你的场景确实能从并行的子代理里受益,比如说你需要在很多个文件之间做同样的修改,或者要并行处理一堆相互独立的小任务,那你就应该在提示词里把这个并行需求明确写出来。文章甚至给了一个示例提示词的写法,大意是,对于你自己一次就能在单个回答里直接完成的工作,比如重构一个你已经能看到完整代码的函数,不要专门派生子代理去做,但当任务需要在多个文件之间铺开,或者需要同时读取多个文件的时候,应该在同一轮里同时派生多个子代理。这个示例非常具体,我建议大家如果平时确实依赖子代理并行处理任务,可以直接把这段话原样搬到自己的系统提示词里去用。

文章在收尾部分还提到一个判断,Opus 四点七在长时间运行的任务上表现优于之前的模型,这让它特别适合那些过去因为需要人工监督而成为瓶颈的场景,比如说涉及多个文件的复杂改动,目标不明确的调试任务,跨越整个服务的代码审查,以及多步骤的自主代理工作。文章给出的最后一条建议很简洁,就是把效力等级保持在 xhigh,然后看看仅凭第一轮对话,模型能把任务推进到什么程度。

好,讲完了文章本身的内容,我们来聊聊这些信息对我们日常工作到底意味着什么,具体应该怎么落地。

第一个可以马上去做的调整,是改变你写提示词的习惯。如果你之前习惯于先给一个大致方向,然后边看模型的输出边补充细节,现在更建议你换一种节奏,在开始之前先花两三分钟,把任务的目标、限制条件、验收标准、相关文件路径都想清楚、写完整,一次性发给模型。这样做的好处不仅是省 token,更重要的是,你会发现模型给出的第一版结果质量明显更高,因为它一开始就拿到了完整的全局信息,而不是带着残缺的理解往前跑,跑到一半才被纠偏。

第二个可以落地的调整,是重新审视你手头那些需要反复确认的工作流。如果你发现自己经常在和 Claude Code 的对话里,每隔几步就要确认一下,要不要这样做,可以试着把这些确认点提前合并到最初的任务说明里。比如说,你可以提前声明,遇到某种情况按某种规则处理,遇到不确定的情况优先选择哪种方案,这样模型在执行过程中就不需要停下来等你点头,整个任务可以更连贯地跑完。如果你的工作环境支持,也可以试试 auto mode,前提是你对这个任务的风险有清楚的判断,知道哪些操作是模型可以放心去做的。

第三个建议,是把效力等级当成一个真正会用的工具,而不是设置完就不再管的开关。比如说,你在做日常的小修小补,改个文案、调个样式,这种任务完全可以用 medium 或者 low,没必要让模型多想。但当你要设计一个新模块的接口规范,或者准备把一段老代码迁移到新框架,这种时候不妨主动确认一下效力等级是不是在 xhigh,甚至在遇到特别棘手的边界情况时,临时切到 max。这种按任务难度动态调档的习惯,比一刀切地用同一个档位处理所有任务,效果会好很多。

第四个建议,跟自适应思考有关。如果你发现模型在一些明显很简单的任务上,响应速度比预期慢,可以考虑在提示词里加一句类似优先快速响应这样的话,把节奏调快。反过来,如果你正在处理一个你直觉上觉得有坑、容易出错的任务,比如一个看起来简单但实际上涉及很多边界情况的 bug,那不妨主动提示模型,这个问题比看起来更难,请仔细思考,把推理空间留出来。这两句话几乎不需要额外学习成本,随手就能用,但能实实在在地帮你在速度和准确性之间找到一个更合适的平衡点。

第五个建议,是关于子代理这件事。如果你的团队经常用 Claude Code 处理那种需要扫描很多文件、做批量修改的任务,比如说统一改一个 API 调用方式,改动遍布几十个文件,这种场景下,建议你在系统提示词或者项目的配置文件里,明确写清楚什么时候应该并行派生子代理。如果不这样明确说,你可能会发现模型现在更倾向于自己一个人慢慢做完,而不是像以前那样动不动就拆给好几个子任务并行处理,这在批量任务上会拉长整体耗时。

最后还有一个比较容易被忽略,但我觉得挺重要的坑要提一下,就是不要假设你过去针对老模型调出来的那一整套提示词和参数设置,可以原封不动地搬到 Opus 四点七上继续用。文章里反复强调这一点,无论是效力等级,还是关于回答长度、工具调用频率、子代理数量的这些默认行为变化,背后的设计理念都已经发生了调整。如果你是那种已经为老模型精心调过提示词的重度用户,升级之后,值得专门留出一点时间,重新跑一遍你的核心工作流,看看哪些地方需要补一句话、调一个参数,而不是简单地以为新模型只是老模型的加强版,所有设置都能照搬。

好,我们来给今天这期节目做个总结。这篇文章的核心思路,其实就是一句话,Opus 四点七已经强到可以让你换一种合作方式了,从手把手带着走的结对编程,转向一次性交代清楚、然后放手托管的委托式协作。围绕这个核心思路,文章给出了几个具体的支撑点,效力等级新增了 xhigh 这个默认推荐档位,自适应思考取代了固定预算的 Extended Thinking,让模型自己判断什么时候该多想,以及一系列默认行为的微调,包括回答长度、工具调用频率和子代理数量,这些调整背后都对应着同一个理念,模型变得更自主、更值得信任,所以也需要你用更完整、更一次性的方式去给它提供任务上下文。

我个人的看法是,这篇文章其实反映了整个 AI 编程工具这两年的一个大趋势,就是从微观操控走向宏观委托。早期我们用这些工具,更像是在用一个很聪明的自动补全,你还是要一句一句盯着。但现在的趋势很明显,工具厂商们都在鼓励用户往后退一步,把更大块的任务整体交出去。这对我们这些每天要写代码、要做代码审查的人来说,意味着我们的角色定位也要跟着调整,从执行细节的人,变成定义清楚目标和边界条件的人。这件事说起来容易,但其实对很多人来说是个不小的思维转变,需要刻意练习。好,这就是今天这期节目的全部内容,我们下期再见。
