## Script

大家好,欢迎收听Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是Claude can now use tools,翻译过来就是"Claude现在可以使用工具了"。这篇文章发表于二〇二四年五月三十日,也就是二〇二四年五月三十号。

先说一句,这篇文章距离我们今天,也就是二〇二六年六月二十一号,已经过去整整两年多了。在AI这个行业里,两年多基本上等于好几个时代的跨度。所以我要提前给大家打个预防针,这篇文章里提到的很多具体功能、具体限制,现在很可能早就被新版本覆盖甚至彻底改变了。我们今天讲这篇文章,更多是把它当成一个历史节点来看,理解当时这件事为什么重要,以及它给后面整个行业埋下了什么样的种子。

那我们先把时间拨回二〇二四年五月。那个时候,Claude三代模型刚刚发布没多久,业界对大模型的认知还停留在一个阶段,就是模型很会聊天,很会写东西,也能做一些推理,但是它本质上是个封闭的系统。你问它一个问题,它只能根据训练时学到的知识来回答,它没法去查一下你公司数据库里的实时数据,没法帮你真的发一封邮件,也没法帮你查一下今天的天气或者股价。这就好比你请了一个特别博学的顾问,但是这个顾问被关在一个没有电话也没有网络的房间里,他只能凭脑子里的知识跟你聊,没法真正帮你办事。

而这篇文章讲的工具调用,英文是tool use,本质上就是给这个顾问配了一部电话,让他可以打电话给外部的服务,查资料,办事情,然后再把结果整合到他给你的回答里。这件事听起来简单,但它其实是大模型从一个纯粹的对话引擎,变成一个真正能采取行动的智能体的关键一步,也就是后来大家说的agent这个概念的地基之一。

我们具体讲讲工具调用是怎么运作的。简单说,开发者会先给Claude定义一套工具,你可以理解成给它一份说明书,告诉它有哪些工具可以用,每个工具需要传入什么参数,能返回什么结果。比如说你可以定义一个叫查天气的工具,告诉Claude这个工具需要传入城市名字,会返回当天的天气情况。然后用户用很自然的语言提问,比如说今天北京天气怎么样,适合穿什么衣服。Claude看到这句话之后,会判断这个问题需要用到查天气这个工具,于是它会生成一个结构化的调用请求,里面写明要调用哪个工具,传入什么参数。这个请求被开发者的程序接收到之后,去真正执行这个查询,拿到结果,再把结果喂回给Claude,Claude再用这个结果生成一段自然语言的回答,告诉用户北京今天多少度,建议穿什么。

这个流程里最关键的设计,是Claude本身并不会真的去执行那个动作,它只是负责判断该用哪个工具,以及该传什么参数,真正的执行权还是在开发者手里。这个设计其实非常重要,我们可以打个比方,这就好比一个高级助理,他不会自己跑去银行转账,他只会跟你说,你需要给某某账户转一笔钱,金额是多少,然后真正按下转账按钮的动作,是你自己或者你的系统去完成的。这种设计既保留了灵活性,又把风险控制权交还给了开发者,这一点放在今天来看依然是非常稳妥的工程思路。

这篇文章里举了几个典型的应用场景,我觉得很值得展开讲讲,因为它们其实代表了工具调用最核心的几类用法。

第一类是从无结构的文本里提取结构化数据。文章里举的例子是从发票里提取名字、日期、金额,这样可以减少人工录入数据的工作量。这个场景其实在企业里非常普遍,你想想看,一个财务部门每天要处理大量的发票、合同、报销单,这些文件格式五花八门,有的是扫描件,有的是邮件附件,人工一条条录入既慢又容易出错。用工具调用的思路,你可以让Claude先理解这份文档的内容,然后调用一个写入数据库的工具,把提取出来的结构化信息直接存进系统里。这其实就是把模型的语言理解能力和后端系统的执行能力打通了。

第二类是把自然语言请求转换成结构化的API调用。文章里举的例子是让团队可以用简单的指令自助完成常见操作,比如说取消订阅这种请求。这个场景的价值在哪里呢,我们设想一下,一个客服系统里,用户说我要取消订阅,这句话背后其实对应的是后端系统里一个具体的API接口调用,可能需要传入用户ID,订阅类型,取消原因等等参数。如果没有大模型,你得做一套规则系统去解析用户的各种说法,什么我不想用了,我要退订,帮我停掉这个服务,各种表达方式都要覆盖到。但有了Claude的工具调用能力之后,模型可以直接理解用户的自然语言意图,然后准确地把它翻译成那个取消订阅接口需要的参数格式,这就大大简化了系统设计的复杂度。

第三类是通过搜索数据库或者调用网络API来回答问题,文章里举的例子是客服聊天机器人可以给用户提供即时准确的回答。这个场景大家应该最熟悉,因为现在满大街都是带AI客服的网站。它的核心逻辑是,模型本身的知识是有边界的,也是有时效性的,但是公司内部的数据库里存的是实时的订单状态、库存信息、用户的历史记录,这些信息模型自己是不知道的。通过工具调用,Claude可以在回答用户之前,先去查一下数据库,拿到最新的真实数据,再结合自己的语言能力把这个数据组织成一段自然流畅的回答。这样既保证了回答的准确性,又保留了对话的自然感。

第四类是通过软件API自动完成一些简单任务,节省数据录入或者文件管理上的时间,减少错误。第五类则更进一步,是协调多个速度更快的Claude子智能体去处理更细粒度的任务,文章里举的例子是根据与会者的日程自动找出最优的会议时间。这个例子其实已经很有agent味道了,你可以想象一个场景,你要安排一个五个人的会议,每个人的日程都不一样,传统做法是你一个个去问,大家来回发邮件协调,非常低效。如果有一个主智能体去协调多个子智能体,每个子智能体负责查询一个人的日程,然后主智能体汇总信息,找出大家都空闲的时间段,这就是典型的多智能体协作完成复杂任务的雏形。我们今天看到的很多多agent系统,其实思路上跟这个例子是一脉相承的。

讲完这几个应用场景,我想花点时间讲讲文章里提到的几个技术细节,因为这些细节看似很小,但其实反映了当时Anthropic团队对产品体验的思考深度。

第一个细节是工具调用支持流式输出,也就是streaming。这个设计是为了减少用户等待的时间,让交互体验更流畅。我们想象一下,如果你在用一个客服聊天机器人,你问了一个问题,然后屏幕上一片空白,过了五六秒才突然弹出一整段回答,这种体验是很糟糕的,用户会觉得这个东西卡住了,是不是出问题了。而流式输出就是让回答像打字一样一个字一个字地蹦出来,哪怕整体耗时是一样的,但用户的心理感受会完全不同,他会觉得这个系统是在实时响应他,而不是在那边发呆。把流式输出和工具调用结合起来,意味着即便Claude中间要停下来调用一个外部工具,这个过程也可以做得比较顺滑,不会让用户感觉到明显的卡顿。

第二个细节是强制工具使用,英文是forced tool use。这个功能允许开发者明确指定Claude必须使用哪个工具,而不是把选择权完全交给模型自己判断。这个设计听起来好像有点反直觉,我们不是希望模型越聪明越自主越好吗,为什么还要去强制限定它的选择。但其实这正是工程落地里非常实际的一个需求。你想,如果你在做一个特定场景的应用,比如说一个专门用来查询订单状态的客服界面,你其实不需要模型去发挥创造力决定用什么工具,你需要的是它百分之百地调用查询订单这个工具,把参数填对就行。给开发者这种强制指定的能力,其实是在保证场景的确定性和可控性,这对于企业级应用来说是非常重要的一个考量。我们可以这样理解,自由发挥和精确执行,在不同场景下哪个更重要是不一样的,Anthropic在这里给了开发者选择的空间,而不是把所有决定权都锁死在模型自己手里。

第三个细节是工具调用同时支持图片输入。这意味着Claude不仅可以根据文字描述去判断该调用哪个工具,还可以根据图片内容来做判断。比如说你给它一张商品的照片,它可以识别出这是什么商品,然后调用一个查询库存的工具,告诉你这个商品还有没有货。这种多模态加工具调用的组合,在当时是相当前沿的能力。

第四个细节是Opus模型在输出里加入了thinking标签,用来展示Claude的推理过程,方便开发者调试。这个设计我觉得特别值得展开讲讲,因为它触及了一个非常本质的问题,就是模型的可解释性。我们都知道大模型经常被叫做黑箱,你问它一个问题,它给你一个答案,但是你不知道它是怎么一步步想出这个答案的。如果模型在执行复杂任务的过程中出了错,比如说调用了错误的工具,或者传错了参数,开发者往往很难定位到底是哪一步出了问题。而thinking标签的作用,就是让模型把自己的思考过程显性地写出来,比如说它先判断用户想干什么,再判断需要用哪个工具,再判断这个工具需要什么参数,这些中间步骤都被记录下来,开发者就可以打开这个标签看看模型的思路链条,从而更快地定位问题出在哪里。这个设计其实跟后来业界普遍流行的链式思考,也就是chain of thought的思路是高度一致的,只不过在这篇文章发表的时候,它是作为工具调用功能里的一个调试辅助手段被提出来的。

文章里也很坦诚地提到了一个限制,就是当时的Claude三代模型还不支持并行工具调用,也就是说Claude在一次回答里只能依次调用工具,不能同时发起多个工具调用请求。这个限制如果放在今天来看,可能会让人觉得有点局限,因为现在很多场景下,你确实希望模型能同时查好几个数据源,然后把结果汇总起来,这样效率会更高。但是我们要理解,这是二〇二四年中期的技术水平,模型的工具调用能力还在早期发展阶段,先把单次调用做稳做准,再去支持并行调用,这是一个很合理的技术演进路径。我个人觉得,这个细节恰恰说明了这篇文章是一个阶段性的产品发布,而不是一个终极方案,后续版本肯定会在这个基础上持续迭代。

接下来我想讲讲文章里提到的三个客户案例,因为这些案例其实比单纯讲技术原理更能说明工具调用这个功能落地之后产生的真实价值。

第一个案例是StudyFetch,这是一个AI原生的学习平台,他们用Claude的工具调用能力来驱动一个叫Spark点E的个性化AI辅导老师。他们把跟踪学生学习进度、导航课程材料和讲座内容、创建交互式用户界面这些功能都做成工具,接入到Claude里。StudyFetch的联合创始人兼技术官在文章里说,工具调用之后,他们的AI辅导老师能够像一个真正的智能体一样,展示交互式界面,在对话上下文里跟踪学生的学习进度,还能在讲座和材料之间自由导航。而且他们提到,接入工具调用之后,正向的用户反馈提升了百分之四十二。这个数字其实挺有说服力的,它说明工具调用不只是一个技术上炫酷的功能,而是真的能转化成用户体验上的实际提升。你可以想象一下,一个学生在用这个AI辅导老师学习的时候,他不再是单纯跟一个聊天框对话,而是这个AI能主动帮他打开对应的课件,标记他学过的章节,甚至根据他的学习进度调整接下来该学什么内容,这种体验的丰富度跟纯文字对话相比是完全不同量级的。

第二个案例是Intuned,这是一个浏览器自动化平台,他们用Claude来驱动云平台里的数据提取功能。Intuned的联合创始人提到,他们用的是当时刚发布的Claude三代Haiku模型,这是Claude三代家族里速度最快、成本最低的一个版本。他说在接入这个模型并跑了他们自己的基准测试之后,发现这个模型在质量、速度和价格这三者的组合上是无可匹敌的,而且Haiku帮助他们把客户的数据提取任务规模化到了一个全新的水平。这个案例其实点出了一个很重要的商业逻辑,就是很多实际的工具调用场景,并不需要用最强最贵的模型去做,反而是又快又便宜的轻量级模型,配合工具调用的能力,就能高效地完成大批量的重复性任务,比如说从网页里抓取结构化数据。这对今天我们做应用选型也是有参考价值的,不是所有场景都要无脑上最贵的模型,合适的模型加上合适的工具设计,往往是更划算的组合。

第三个案例是Hebbia,他们在给金融和法律服务的领先企业打造AI知识工作者。Hebbia的产品经理提到,他们用Claude三代Haiku来生成实时建议,自动化撰写提示词,以及从长文档里提取关键的元数据信息。她说Claude三代Haiku的工具调用功能,解锁了他们平台生成可靠建议和提示的能力,而且速度很快。这个案例其实代表了金融法律这种专业服务行业里一个典型的痛点,就是从业人员经常要处理大量冗长的文档,比如合同、招股书、监管文件,人工去翻找关键信息非常耗时。如果AI能够通过工具调用,直接从这些长文档里精准抓取出关键的元数据,比如说合同的生效日期、违约条款、关键金额,这对专业人士的工作效率提升是非常实际的。

好,讲完了原理和案例,我们来聊聊这件事对今天的开发者和从业者到底意味着什么,我们应该怎么去理解和应用这套思路。

首先要明确的一点是,工具调用这个概念在二〇二四年五月被Anthropic作为一个正式可用的能力推出,这个时间点放在整个行业的发展脉络里,其实正好是agent热潮真正爆发的前夜。我们现在回头看,二〇二四年下半年到二〇二五年,整个行业里关于智能体的讨论越来越密集,各种agent框架层出不穷,后来还出现了像MCP这样的标准化协议,专门用来规范模型和外部工具、外部数据源之间的连接方式。如果你去看MCP这类协议解决的问题,其实跟这篇文章里讲的工具调用是同一个母题,只不过MCP是想把这种连接方式标准化,让不同的模型、不同的工具之间可以用统一的协议互相通信,而不需要每个开发者都去重新发明一套对接方式。所以我们可以说,这篇二〇二四年五月的文章,其实是后面那一整条技术演进路线的起点之一,理解了工具调用最初的设计思路,再去看后面的协议和框架,会觉得很多设计选择都是顺理成章的延续。

其次,我觉得文章里那个关于强制工具使用的设计细节,放在今天依然是很有参考价值的工程思路。现在很多团队在做agent应用的时候,容易陷入一个误区,就是觉得给模型越多自主权,系统就越智能越好。但实际工程落地的时候,你会发现很多场景其实需要的是确定性和可控性,而不是无限的自主发挥。比如说在金融、医疗这类对准确性要求极高的场景里,你可能宁愿牺牲一部分灵活性,去明确限定模型在某个步骤必须调用某个工具,执行某个固定的流程,这样可以大幅降低出错的概率。所以如果你今天在设计一个agent系统,我建议你不要默认追求最大化的自主性,而是要根据具体场景去判断,哪些环节适合让模型自由判断,哪些环节需要你用规则去约束它的行为边界。

第三点,关于模型选型,Intuned那个案例给我们的启发是,不是所有工具调用场景都需要用最强的模型。如果你的任务是结构化程度比较高、重复性比较强的数据提取或者简单决策,完全可以考虑用更轻量、更便宜、速度更快的模型去配合工具调用,这样既能保证任务质量,又能大幅降低成本,尤其是在你需要规模化处理大量请求的时候,这种成本差异会被放大很多倍。当然了,今天的模型生态比二〇二四年丰富太多了,Claude也已经经历了好几代的迭代,具体该选哪个版本的模型,需要结合当下最新的模型能力和价格去做判断,但这个分层选型的思路本身,我觉得到今天依然是成立的。

第四点,我想提一下thinking标签这个设计带来的启发。很多团队在做agent调试的时候,经常会遇到模型行为不可预测、出错难以定位的问题。这篇文章告诉我们,一个很朴素但很有效的解决办法,就是让模型把推理过程显性化,变成你可以读、可以检查的文本,而不是一个完全不透明的黑箱输出。如果你今天在搭建复杂的agent流程,我建议你一定要设计好日志和推理过程的记录机制,这样当系统出问题的时候,你才有线索去排查到底是哪一步判断错了。

最后我想提醒大家一点要避的坑,就是不要把工具调用这个能力想象得过于神奇。文章里反复强调的一个设计原则是,模型本身并不直接执行动作,它只是负责判断和生成调用请求,真正的执行和风险控制,始终是在开发者手里的程序代码里完成的。这意味着,如果你今天要做一个涉及真实世界操作的agent应用,比如说会真正转账、真正发邮件、真正删除数据的那种,你必须在执行层面设计好充分的校验和保护机制,绝对不能假设模型的判断永远正确,然后就让它直接去执行高风险的动作。这一点我觉得在今天做agent安全设计的时候,依然是最基本但也最容易被忽视的原则。

好,我们来总结一下今天这期节目。这篇发表于二〇二四年五月三十号的文章,标题是Claude can now use tools,讲的是Anthropic把工具调用这个功能在Claude三代全系列模型上正式开放,覆盖了Anthropic自己的接口,也覆盖了亚马逊云和谷歌云的平台。文章核心讲的是,工具调用让Claude从一个只能聊天的语言模型,变成了一个可以连接外部世界、真正帮你办事的智能体雏形,文章用提取数据、转换请求、回答问题、自动化任务、协调多智能体这五类场景,具体说明了这个能力能干什么,又用流式输出、强制工具选择、多模态支持、推理过程可视化这几个设计细节,体现了当时团队对开发者体验的细致考量。三个真实客户案例,StudyFetch、Intuned和Hebbia,分别从教育、浏览器自动化和金融法律服务三个不同的行业角度,印证了这个功能在实际产品里能带来真实的效率和体验提升。

我个人的看法是,这篇文章虽然现在看起来内容不算多,篇幅也很短,但它在整个行业的发展脉络里其实是一个很关键的节点,它标志着大模型开始真正意义上具备了和外部世界交互的标准化能力,后面整个agent生态的爆发,包括各种工具协议和框架的出现,都是在这个基础之上往前走的。所以如果你是今天刚开始做agent开发的工程师,我觉得回头看一眼这种早期的设计文档,理解一下最初这些功能为什么被这样设计出来,对你理解今天更复杂的系统会很有帮助。技术在往前跑,但很多最基本的设计原则,比如说执行权和判断权要分离,自主性和可控性要平衡,推理过程要可见可调试,这些原则到今天依然没有过时。
