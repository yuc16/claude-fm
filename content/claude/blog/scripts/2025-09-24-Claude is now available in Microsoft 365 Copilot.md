大家好,欢迎收听Claude FM,我是你们的主播。今天我们要聊的这篇文章,标题是Claude is now available in Microsoft 365 Copilot,翻译过来就是,Claude现在可以在Microsoft 365 Copilot里用了。这篇文章发表于二零二五年九月二十四日,来自Anthropic官方博客。

我看了一下今天的日期,是二零二六年六月二十一日,距离这篇文章发布已经过去了大约九个月。所以这是一篇有一定时效性的新闻稿,我们听的时候要带着一点历史视角。九个月前,正是大模型企业级应用竞争最激烈的一个阶段。那个时候,OpenAI和微软的合作关系已经持续了好几年,微软的Copilot系列产品几乎是和GPT系列模型深度绑定的代名词。但就在那个节点,微软做了一个很有意思的动作,主动把竞争对手Anthropic的Claude模型,引入到自己最核心的生产力产品矩阵里。这件事本身释放的信号,可能比技术细节更值得我们琢磨。所以今天这期节目,我们不仅要讲清楚文章里写了什么,还要花点时间聊聊这背后反映的行业趋势,以及九个月后的今天回头看,这个趋势是不是还在延续。

我们先说说背景。在这篇文章发布之前,微软的三百六十五Copilot,也就是Microsoft 365 Copilot,已经是企业级AI助手里装机量最大的产品之一了。它嵌入在Word、Excel、PowerPoint、Outlook、Teams这些大家每天都要打开的办公软件里,本质上是把生成式AI的能力,直接焊接到了全球数以亿计的白领员工的日常工作流里。这个产品的底层模型,长期以来主要依赖OpenAI的GPT系列。这很自然,因为微软是OpenAI最大的投资方和云基础设施提供商,两家公司的利益高度绑定。

但是企业市场有它自己的逻辑。如果你是一个大型企业的IT负责人,你会发现,不同的大模型在不同任务上的表现是有差异的。有些模型擅长写代码,有些擅长长文档的推理和总结,有些在处理复杂的多步骤研究任务时表现更稳。当你的企业里有几万名员工,每天要处理的任务五花八门,从写一封邮件,到做一份几十页的市场调研报告,再到搭建一个自动化的客服流程,你会很自然地产生一个需求,就是我能不能根据具体场景,灵活选择最适合的模型,而不是被绑定在单一供应商的单一模型上。

这种需求在过去一两年里,在企业客户那边的声音越来越大。所以微软在这个时间点做出这样一个决定,某种程度上是在回应市场的真实诉求。文章里提到,这次合作不是泛泛地说"Claude现在可以用了",而是非常具体地落在了两个产品功能上,一个叫Researcher智能体,一个叫Copilot Studio。我们接下来就重点拆解这两个东西分别意味着什么。

先说Researcher。这是微软在Microsoft 365 Copilot里推出的一个推理智能体,中文可以理解成深度研究助手。它的设计目标,是用来处理那种复杂的、需要多步骤推理的研究类任务。打个比方,如果你需要做一份详细的市场进入策略,或者分析某个新兴的产品趋势,又或者要写一份全面的季度业务报告,这些任务都不是一句话就能让AI秒回答的,它需要模型先去查资料,再做交叉验证,再组织逻辑,再输出结构化的结论,中间可能要反复迭代好几轮。这种工作流,业内现在统称为agentic research,也就是智能体式的研究,核心特点是模型不再是被动回答一个问题,而是主动规划步骤、调用工具、检索信息,然后自己判断下一步该做什么。

Researcher这个智能体的厉害之处,在于它推理的范围不只是公开的网络信息,还包括你企业内部的、受信任的第三方数据,甚至是你自己工作环境里的全部内容,比如你的邮件、你的聊天记录、你参加过的会议记录,以及你存在云端的各种文件。换句话说,它不是一个孤立的搜索引擎,而是一个能够理解你整个工作语境的助手。它的目标是帮你交付一种"按需获得的高水平专业能力",这句话翻译得有点抽象,我用一个例子来说明。假设你是一个产品经理,需要写一份关于某个细分市场未来一年发展趋势的报告,如果靠你自己,可能要花两三天时间,翻遍各种行业报告,再结合公司内部过去几个季度的销售数据和客户反馈邮件,才能拼出一份像样的分析。而有了Researcher这样的智能体,理论上你可以让它自己去检索外部行业报告,同时去翻你公司内部存档的邮件和会议记录,把这些信息融合起来,生成一份初稿,你再在这个基础上做修改和润色。这个工作量级的压缩,是agentic research类产品最核心的价值主张。

那么这次更新带来的变化是什么呢,就是这个Researcher智能体,现在可以选择用Claude Opus 4.1来驱动,而不是只能用微软原本默认配置的模型。Opus 4.1是Anthropic当时最强的旗舉模型之一,在复杂推理、长文本理解和多步骤任务规划上有比较突出的表现。微软选择把Claude Opus 4.1接入到Researcher这个最需要深度推理能力的功能里,这个选择本身是有逻辑的,因为深度研究类任务,恰恰是检验一个模型推理能力上限的最佳场景。如果一个模型在简单问答上表现得都差不多,但在需要五步以上推理链条、需要跨多个信息源做交叉验证的任务上,模型之间的差距就会被放大。微软愿意把Claude放进这个对推理能力要求最高的功能模块,某种程度上也是对Claude推理能力的一种背书。

我们再说说第二个功能,叫Copilot Studio。如果Researcher是一个面向最终用户的、开箱即用的研究助手,那么Copilot Studio就是一个面向开发者和企业IT团队的智能体构建平台。你可以把它理解成一个低代码或者无代码的智能体搭建工具,企业里的技术人员,甚至一些懂业务流程但不太会写代码的业务人员,都可以用它来创建和定制属于自己企业的专属智能体。

在这次更新之前,Copilot Studio里能选的底层模型选项是有限的,这次更新之后,Claude Sonnet 4和Claude Opus 4.1正式成为Copilot Studio里可选的模型之一。这意味着企业在搭建自己的内部智能体的时候,可以根据具体任务的性质来挑选最适合的模型。文章里提到几个关键词,一个是深度推理,一个是工作流自动化,一个是灵活的智能体任务。这几个词其实对应了企业智能体落地的几种典型场景。

我给大家举几个具体的应用场景,方便大家更直观地理解这个能力意味着什么。比如说,一个法务部门想搭建一个智能体,用来自动审阅合同条款,识别其中的风险点,这种任务对模型的长文本理解能力和逻辑严谨性要求很高,可能更适合用Claude这类擅长长文本推理的模型来驱动。又比如,一个客服部门想搭建一个自动回复客户邮件的智能体,这种任务可能更看重响应速度和成本效益,企业可能会选择另一个更轻量的模型。再比如,一个人力资源部门想做一个智能体,自动总结员工满意度调研的开放式问答内容,提炼出几个核心主题,这种归纳总结类的任务,也是可以根据实际测试效果去匹配最合适的模型。

文章里特别提到一个细节,叫做多智能体系统和prompt工具,这两个词背后其实代表了智能体应用从单一对话模式,走向编排化、协作化的趋势。所谓多智能体系统,通俗讲就是不再依赖一个全能型的模型来处理所有任务,而是把一个复杂的业务流程拆解成几个子任务,每个子任务由一个专门的智能体来负责,这些智能体之间互相传递信息、互相协作,最终完成整个流程。这种架构的好处是,你可以针对每个子任务挑选最适合的模型,而不是用一个模型去硬扛所有类型的工作。Copilot Studio开放Claude作为可选模型之后,企业在设计这种多智能体系统的时候,就多了一种选择,可以把某些特别需要深度推理或者长文本处理能力的子任务,单独分配给Claude来处理。

这里我想多展开讲一下,为什么"模型可选"这个事情,对企业级用户来说是一件比表面上听起来更重要的事。我们打个比较生活化的比喻。你可以把不同的大模型想象成不同领域的专业顾问。有的顾问特别擅长写商业计划书,逻辑清晰,文笔老练。有的顾问特别擅长处理数据,擅长从一堆杂乱的数字里抓出规律。有的顾问反应特别快,适合处理大量重复性的简单咨询。如果你公司只能签一个顾问公司的独家合同,只能用他们家派来的那一个顾问处理所有事情,哪怕这个顾问总体水平不错,但碰到他不擅长的领域,效果一定会打折扣。而如果你有权利根据具体项目,从市场上挑选最合适的顾问,你整体的产出质量和效率会明显提升。企业级AI市场现在正在经历的,就是从前者向后者过渡的过程。

这也解释了为什么微软愿意在自己最核心的产品里引入竞争对手的模型。因为对微软来说,如果死守单一模型供应商,反而可能丧失企业大客户,因为这些大客户本身就有多模型并用的现实需求,如果微软不提供这种灵活性,客户可能会转向其他更灵活的平台,或者自己搭建独立于微软体系之外的AI工作流。所以从商业逻辑上讲,微软选择开放模型选项,本质上是为了留住和吸引那些对AI能力有更精细化需求的企业客户。而对Anthropic来说,能够进入微软三百六十五Copilot这样一个拥有海量企业用户基础的平台,是一次极佳的分发渠道扩张,可以让更多企业用户在不需要单独签约、单独部署的情况下,就能接触到Claude的能力。这是一个双方都能从中获益的合作。

接下来我想讲一个文章里提到的、容易被忽略但其实很关键的细节,就是关于数据托管和服务条款的问题。文章原文里写得很清楚,Anthropic的模型是托管在微软管理的环境之外的,并且适用的是Anthropic自己的服务条款和条件,不是微软的服务条款。这句话听起来很技术性,但它背后涉及的是企业客户在使用这个功能之前必须要搞清楚的合规问题。

我们展开讲一下这意味着什么。通常情况下,如果一个企业使用微软三百六十五Copilot里默认集成的OpenAI模型,那么这个数据处理的链路,从合规角度讲,是完全在微软自己的责任边界和服务协议覆盖范围之内的。但当你选择切换到Claude模型驱动的Researcher或者Copilot Studio功能时,你的请求数据实际上是被发送到Anthropic托管的基础设施上去处理的,这就意味着这部分数据的处理,适用的是Anthropic的服务条款,而不是微软的。对于一般的个人用户来说,这种区别可能没什么实际影响,但对于那些有严格数据合规要求的行业,比如金融、医疗、政府机构,这种细节是企业IT和法务部门在决定是否启用这个功能之前,必须仔细审阅的。

这也解释了为什么文章里特别强调,这次功能的开放,是需要企业管理员主动在Microsoft 365管理中心里手动启用对Anthropic模型的访问权限的,而不是默认对所有用户开放的。这是一种典型的渐进式、可控的功能放开策略。微软没有把这个选项默认打开,而是设计成一个opt-in,也就是需要主动选择加入的机制,这给了企业IT团队足够的时间去做风险评估、做合规审查,确认这个功能符合自己组织的数据治理政策之后,再决定是否对内部员工开放。

另外文章里还提到一个词,叫Frontier Program,中文大概可以理解成前沿项目计划。这是微软用来分阶段、限定范围地把最新功能开放给特定客户群体的一种机制。我们可以理解成一种早期体验计划或者邀请制的内测项目。文章说,Claude在Researcher里的能力,是通过这个Frontier Program,逐步开放给那些已经持有Microsoft 365 Copilot许可证、并且主动选择加入的客户。同样的,如果企业想在Copilot Studio里试用Claude来构建自己的智能体,也需要通过这个Frontier Program申请加入。这种分阶段开放的方式,其实在大型科技公司推新功能的时候是很常见的,目的是控制风险,先在小范围内收集真实的使用反馈,根据反馈再决定是否扩大开放范围,或者做相应的调整。

好,讲完了这次更新的具体内容和背后的机制,我们来聊聊这件事对我们普通听众,尤其是软件工程师和AI从业者来说,到底意味着什么,我们在日常工作里能怎么去理解和应用这个趋势。

第一点,我觉得最值得大家关注的,是企业级AI采购逻辑正在从"绑定单一供应商"转向"按场景挑选模型"。如果你所在的公司正在或者将要采购企业级的AI办公套件,不管是微软的Copilot,还是其他厂商的类似产品,你都应该多问一句,这个产品支持多模型选择吗,不同的模型在哪些具体任务上表现更好。这其实是一个值得企业内部专门去做评估的课题。我自己的建议是,如果你是企业内部负责AI工具选型的工程师或者架构师,你不应该满足于"能用就行"这种心态,而是应该针对你公司最核心的几个高频任务场景,比如客户邮件处理、内部知识检索、合同审查、报告生成,分别用不同的模型跑一遍小规模的对比测试,看看实际效果差异有多大,再决定具体的模型分配策略。这种精细化的模型选型工作,在未来一两年里,很可能会成为企业AI落地团队的标准动作之一。

第二点,我想讲讲Researcher这种深度研究智能体的工作模式,对个人工程师有什么借鉴意义。即便你所在的公司暂时还没有接入这类企业级产品,你完全可以借鉴它的设计思路,在自己日常使用Claude或者其他模型的时候,主动采用类似的工作流。比如说,当你要做一个比较复杂的技术调研,不要指望一句话提示词就能拿到满意的答案,而是可以借鉴Researcher的思路,先让模型帮你规划调研步骤,再分步骤去检索信息,验证信息,最后再做汇总。这种多步骤、可迭代的使用方式,往往比一次性提一个大而模糊的问题,效果好得多。这也是为什么我个人觉得,理解agentic的工作模式,对所有AI从业者来说都是一项值得投入时间去打磨的技能,而不仅仅是了解一下概念就算了。

第三点,关于Copilot Studio这种低代码智能体构建平台,我觉得它代表的趋势是,智能体的搭建门槛正在持续下降,越来越多不是专业程序员的业务人员,也将有能力自己动手搭建符合自己部门需求的专属智能体。这对软件工程师来说,既是一种挑战,也是一种机会。挑战在于,过去很多简单的内部工具开发需求,可能会被业务部门用这种低代码平台自己解决掉,不再需要专门排期找工程团队开发。机会在于,工程师可以把更多精力投入到那些真正需要复杂工程能力的、低代码平台无法覆盖的高价值系统建设上去,同时也可以承担起为业务部门提供智能体架构咨询、模型选型建议、安全合规把关这类更偏向顾问性质的角色。我个人认为,未来懂业务又懂模型特性的复合型人才,会越来越值钱。

第四点,我想提醒大家注意的是这次合作里那个数据托管和服务条款的细节,这其实给所有想要在企业环境里引入第三方模型的团队提了一个醒。不管你是在用微软的产品,还是在自己公司内部搭建agent系统,只要涉及到调用外部模型供应商的API,你都应该清楚地知道,你的数据实际流向了哪里,处理这部分数据适用的是谁的服务条款和隐私政策。这不是危言耸听,而是企业级AI落地里非常实际的一个合规盲点。很多团队在做技术选型的时候,容易只关注模型能力强不强、速度快不快、价格贵不贵,反而忽略了数据治理这个维度。我建议大家在做相关决策之前,一定要让法务或者合规团队提前介入,把这些条款仔细过一遍,而不是等出了问题才回头补救。

第五点,从更宏观的角度看,这次合作也反映出整个大模型行业,正在从"赢家通吃"的叙事,慢慢走向"多元共存"的格局。九个月前这篇文章发布的时候,行业里还有很多声音在讨论,某一家公司是不是会一统天下,垂直整合到底有多重要。但从这次微软和Anthropic的合作来看,至少在企业级市场,客户的实际需求,以及供应商之间为了争夺这些客户而做出的策略调整,正在共同把市场推向一个更开放、更互联的方向。对于我们这些做技术的人来说,这其实是个好消息,因为这意味着我们有更多的选择空间,不会被锁死在某一家公司的技术栈和定价策略里。

好,我们简单总结一下今天讲的这篇文章。Anthropic在二零二五年九月二十四日发布了这条消息,宣布Claude Sonnet 4和Claude Opus 4.1正式进入微软的Microsoft 365 Copilot生态,具体落地在两个功能上,一个是微软的Researcher深度研究智能体,现在可以选择用Claude Opus 4.1驱动,帮助用户处理复杂的多步骤研究任务,推理范围覆盖网络信息、企业内部数据,甚至个人的邮件和会议记录。另一个是Copilot Studio,一个面向企业开发者的低代码智能体构建平台,现在Claude Sonnet 4和Claude Opus 4.1成为可选的底层模型,方便企业根据具体业务场景,灵活搭配最合适的模型来驱动自己的专属智能体。这次开放是通过微软的Frontier Program分阶段进行的,需要企业管理员主动在管理后台开启权限,并且要注意,使用Claude驱动的功能,数据处理适用的是Anthropic自己的服务条款,这是企业在决定是否启用之前需要仔细评估的合规细节。
