今天我们要聊的文章标题是 Working at the frontier: How Cursor knew Claude Fable 5 was ready for the hardest 1% of problems，我把它翻译成中文就是，在前沿工作，Cursor 如何知道 Claude Fable 五已经准备好处理最难的百分之一问题。这篇文章发表于二〇二六年七月十七日，今天是二〇二六年七月十九日，所以这是非常新的内容，基本就是 Anthropic 在 Claude Fable 五发布后，借 Cursor 这个真实开发工具团队的视角，来解释一个问题，怎样判断一个模型不是只会刷榜，而是真的能扛住专业软件工程里最难、最模糊、最耗脑子的任务。

这期值得听，是因为它讲的不是一个抽象的模型能力宣传，也不是单纯说某个模型在某个 benchmark 上高了几个百分点。它真正有意思的地方在于，Cursor 作为一个 AI 编码 agent，也就是可以帮助开发者读代码、改代码、执行任务的智能代理，它同时支持各家主流前沿模型，也有自己的模型。所以 Cursor 在某种程度上是一个非常特殊的观察点。它每天能看到大量开发者把不同模型放进真实工程环境里用，看到他们什么时候满意，什么时候怀疑，什么时候在任务做到一半悄悄切到另一个模型。这个信号，比单纯的公开排行榜更接近我们写代码时的真实感受。

文章的主角是 Cursor 的工程师 Nate Schmidt。他负责的是 evals 和 model behavior，翻成中文就是模型评测和模型行为研究。简单说，他的工作不是看模型会不会答一道题，而是看模型在真实工程问题里会不会理解人的意图，能不能自己定位根因，会不会犯那些让开发者抓狂的错。比如，你给它一个报错栈，它到底是机械地改最表面的那一行，还是会追到调用链深处，把真正的问题找出来。又比如，用户自己判断错了，说某个模块坏了，模型会不会盲从，还是敢于挑战用户的假设。

这篇文章要解决的核心问题，其实是 AI 编程工具这一两年里越来越明显的一个矛盾。公开 benchmark 的分数还在涨，但开发者的体感和这些分数经常对不上。你可能也有类似体验。一个模型在榜单上很漂亮，论文题、算法题、单文件修 bug 都表现很好，可一旦放进一个有历史包袱的代码库，里面有奇怪的业务逻辑、旧的抽象、没有写清楚的约束、半截的测试、互相影响的模块，它就开始变得不稳定。它可能会很自信地改一堆代码，但改完之后你发现它没理解项目真正要做什么。

所以 Cursor 团队后来自己做了一个评测，叫 CursorBench。这个评测的关键不在于把题目设计得多么标准化，恰恰相反，它想捕捉真实工程师使用模型时那种乱糟糟、没说全、甚至带点误导的 prompt。这里 prompt 指的是你给模型的指令或上下文。文章里举了两个特别典型的例子。一个任务就是把一段 stack trace，也就是错误调用栈，直接贴进去，然后只写一个词，fix，修一下。模型需要自己推断用户想要什么，找到根因，修改代码，并且验证修复。另一个任务更刁钻，用户会告诉模型一个错误的判断，说坏的是某个模块。评测要看模型会不会跟着用户一路走进死胡同，还是会停下来检查事实，发现真正出问题的地方并不在那里。

如果你是工程师，你应该马上能理解这个评测为什么有价值。真实工作里，我们很少给 AI 一个像考试题一样完整的描述。我们经常是刚从线上报警里跳出来，看到一段日志，心里有点怀疑但不确定，然后顺手把信息扔给模型。我们说的可能是，看看这个为什么挂了，或者帮我把这个接口改一下，或者这个测试怎么突然红了。很多隐含信息都没写出来，比如最近谁改过代码，业务上哪些行为不能变，测试为什么这么设计，某个函数名虽然难看但不能随便重命名。优秀的编码 agent 必须从这些不完整线索里反推出任务目标，而不是等你把所有东西写成规格说明书。

文章说，Claude Fable 五在 CursorBench 上以 Max effort，也就是最高努力模式，拿到了百分之七十二点九的成绩，刷新了新高。这个数字本身当然重要，但更重要的是 Cursor 团队看到的过程。Schmidt 说，当他们把新模型跑过 CursorBench 时，正确答案只是入场券。他们真正评分的是，模型到底有没有理解自己被要求做什么。很多 eval 看起来是，这里有一个定义好的问题，这里有约束，请你修好。但真实用户的 prompt 并不是这样。模型必须推断用户有个问题，推断用户想表达什么，识别根因，修复，验证，然后再把结果汇报回来。

这句话很关键。它把模型能力从会不会写代码，提升到了会不会承担一个工程任务。以前我们说 AI coding，很多时候想的是自动补全、生成函数、写测试、解释代码。再往前一步，是让模型当 pair programmer，也就是结对编程搭档。你告诉它一个小目标，它给建议，你来判断。但文章里的方向更像 agentic coding，代理式编码。模型不是只回答问题，而是接手一段任务链。它需要读环境、制定方案、动手修改、跑验证、发现失败、再调整。这个过程里最难的不是某一行代码，而是始终记得整个任务为什么存在。

Cursor 团队一开始看到 Claude Fable 五在这些模糊任务上表现太好，甚至有点怀疑。Schmidt 在文章里说，当时他们觉得可能只有两种情况，要么模型真的很聪明，要么模型在作弊。这里的作弊不是说它真的有什么道德问题，而是评测里可能存在泄漏，或者模型可能用了一些不该用的捷径，导致分数虚高。所以他们去看 trace，也就是模型实际执行任务时留下的推理和操作轨迹。尤其看那些最难的任务，表面 prompt 很简单，但真正要解开需要理解整个系统。

他们看到的结果是，Claude Fable 五不断从一些以前没有模型能做出来的任务里挖出胜利。更有意思的是，它完成任务时需要的操作更少，也就是相对于完成的工作量来说，它更省 token。token 可以理解成模型处理文字和代码的基本单位，越省 token 通常意味着同样任务里上下文更紧凑，成本更低，也可能说明它少走弯路。当然，文章没有把这部分展开成一篇论文，但这个观察对工程应用很重要。我们平时用 agent 最怕的不是它慢一点，而是它开始无休止地翻文件、改来改去、跑一堆没必要的命令，最后不仅花钱，还把项目弄乱。如果一个模型能少做无效操作，更快收敛到根因，那它的实际价值会明显高很多。

这里我想打个比方。你让一个初级工程师修一个线上 bug，他可能会盯着报错那一行看半天，加几个判空，改一处试一下，不行再改另一处。每一步看起来都有道理，但整体是在局部搜索。一个资深工程师则可能先问，这个系统的核心不变量是什么，最近哪些路径变了，这个异常在业务流程里意味着什么，然后用更少的试探定位问题。Claude Fable 五在文章中的形象，更接近后者。它不是只看刚刚发生了什么，而是试图理解整个任务的结构。

文章里最有画面感的例子，是 Schmidt 自己做的一个个人测试，叫登陆月球。他之前把 Claude Opus 接进一个可编程的太空飞行模拟器，给了一个非常简单的一行 prompt，建一枚火箭并把它降落到月球上。然后他把这个过程放在第二块显示器上跑了十二到十六个小时。模型会发射火箭，进入轨道后燃料耗尽，于是下一次加很多燃料，结果火箭太重，连大气层都出不去。这个循环很像我们看到某些 agent 做复杂任务时的样子。它看到上一轮失败的直接原因是燃料不够，于是补燃料。但它没有理解，这个补救会改变整枚火箭的质量、推力、轨道策略，导致新的失败。

后来 Schmidt 用同样的空白 prompt 重新跑实验，这次用 Claude Fable 五。几分钟后，火箭升空，进入低轨，然后又回来了。表面上看，这是跟之前一样的失败，没有登月。但 Schmidt 看了 transcript，也就是执行记录，发现 Fable 五是故意不在第一次尝试登月。它决定先做一个初始任务，只进入轨道并收集遥测数据，再用这些数据指导下一次飞行。几次尝试之后，他第二块显示器上的发动机声音停了，月球上出现了着陆器。整个过程用了几个小时，而之前 Opus 跑了十二个小时以上还没有结果。

Schmidt 对这个差别的总结很精炼。他说，Opus 做的是 local reasoning，也就是局部推理，它在想刚刚发生了什么，下一秒马上要发生什么。而 Fable 做的是 global reasoning，也就是全局推理，它在想整个任务。这个说法非常值得我们停一下。局部推理不是没用，很多编程任务就是局部的。比如把一个函数改成异步，把一个类型错误修掉，给一个组件补一个 loading 状态，这些都不一定需要宏观规划。但最难的百分之一问题通常不是这样。它们需要你为了最终目标，先做一些看似没有直接完成目标的准备工作。比如先加观测点，先写回归测试，先重构边界，先理解数据流，甚至先证明某条路走不通。

这也是为什么登月实验特别像软件工程。你让模型建火箭登月，最笨的策略是每次失败后补一个局部 patch。燃料不够，加燃料。速度不够，加推力。姿态不稳，加控制。每个 patch 也许都对，但组合起来可能越来越糟。真正有效的策略是把任务分解成阶段，先实现入轨，再验证转移轨道，再设计着陆，再控制燃料余量。软件工程里的大重构也是这样。你不能一上来就把所有模块全改掉，然后祈祷测试通过。你要先搞清楚系统骨架，建立安全网，确认哪些行为不能变，再一步步迁移。文章想表达的是，Claude Fable 五开始能更自然地做这种全局任务规划。

那什么时候应该用 Claude Fable 五，而不是用更便宜、更快但没那么聪明的模型呢？Schmidt 给了一个简单规则。如果你很清楚从 A 到 B 的路径长什么样，你可能不需要 Fable。如果你在 A 点，但完全不知道 B 点在哪里，Fable 是一个很好的选择。他还说，当他想把某个东西以正确的方式构建出来时，Fable 是他第一个想到的模型。

这句话听起来像广告，但在工程决策上其实很实际。我们日常开发里有大量任务，路径很清楚。比如按照已有模式加一个接口，改一个字段名，生成一段样板代码，把一个错误消息翻译一下，或者根据已有测试补实现。这种任务用顶级强模型未必划算。因为瓶颈不在智能，而在速度、成本、上下文接入、工具执行。可另外一些任务，瓶颈就是智能。比如一个长期没人敢碰的重写，一个跨多个模块的性能问题，一个边界条件很多的并发 bug，一个旧系统迁移，一个你自己都还没想清楚的架构方案。这时候，用便宜模型绕十圈，可能还不如用强模型一次把问题空间摸清楚。

文章里说，Claude Fable 五让 Cursor 团队重新开始做一些以前搁置的项目。那些项目大家都同意，做完会更好，但没有人能证明花几周时间值得。比如某些重写，某些架构清理，某些长期技术债。Schmidt 的说法是，它降低了这类任务的 activation energy，也就是启动能量。这个物理学比喻非常妙。很多工程团队不是不知道哪里该改，而是启动成本太高。你要先读一堆代码，画依赖关系，找调用方，确认行为，写迁移计划，跟同事同步，然后才敢动第一刀。强 agent 如果能帮你扛住足够多的骨架工作，就会让这些以前被认为不划算的任务重新变得可做。

他还说，这让团队可以朝 global optimum，也就是全局最优移动，而不是停留在 local optimum，局部最优。软件工程里我们太熟悉局部最优了。一个模块明明设计得很别扭，但每次需求来了，我们都只在旁边再包一层，因为彻底重写太贵。一个接口明明命名混乱，但所有人都知道别碰，因为影响范围不清楚。一个性能瓶颈明明出在架构层，但我们先加缓存、加索引、加重试，短期都有效，长期系统越来越复杂。局部最优不是错误，它经常是现实妥协。但如果 AI agent 能降低探索和重构成本，团队就可能更频繁地问一个以前问不起的问题，我们能不能把这件事做对，而不是只把这次火扑灭。

文章还提到一个对团队协作很有启发的小细节。Cursor 是一个 lean 的团队，也就是比较精简，强调个人 owner，站会不多。现在 Schmidt 在碰共享代码之前，会让 agent 先去读同事最近的 commits，也就是提交记录，然后标出可能冲突的地方。这样两个人都不用停下手里的工作去互相确认。这个例子不炫技，但非常现实。很多开发协作的成本不是来自写代码本身，而是来自状态同步。谁改了什么，为什么改，跟我即将做的有没有冲突，某个抽象是不是已经在另一个分支变化了。以前这些信息要靠会议、聊天、代码评审、个人记忆来补齐。现在 agent 可以先做一层背景调查，把潜在冲突提醒出来。

当然，Cursor 并不是所有事情都用 Claude Fable 五。文章明确说，为了平衡成本和性能，他们会把 Fable 五和更快、更轻的模型搭配起来。常规工作交给轻模型，真正能力成为约束的问题再请 Fable 五出场。Schmidt 认为，这种组合是他们跑过的最有效配置。如果他遇到一个 really gnarly problem，也就是特别棘手的问题，文章里还用到 p九九 的说法，意思是最难的百分位九十九那类问题，他优化的目标就是 time to solution，也就是到达解决方案的时间。他认为 Fable 是解决他们最难问题的最佳模型。

这里对我们听众的实际意义很大。很多团队现在上 AI 编码工具时，容易犯两个相反的错误。一个错误是过度节省，所有任务都默认用便宜模型，结果在难问题上反复失败，工程师花大量时间 babysit，也就是像带小孩一样盯着模型、纠正它、补上下文、审计结果。你表面上省了模型调用费，实际上把昂贵的工程师注意力赔进去了。另一个错误是过度迷信，所有任务都上最强模型，成本爆炸，还不一定提升体验，因为简单任务的瓶颈可能根本不是推理能力。Cursor 的经验更像一个分层策略。普通任务用轻模型跑，复杂而模糊的任务用强模型扛，关键是判断什么时候能力是瓶颈。

如果落到个人开发者的工作流，我个人认为可以先建立一个很简单的任务分级。第一类是路径明确的任务。你知道要改哪个文件，知道参考模式，只需要模型帮你快一点。这类任务可以用普通模型，prompt 也可以短。第二类是路径半明确的任务。你知道目标，但不知道影响范围。这时候可以让 agent 先读代码、列计划、找测试，再决定是否让它动手。第三类是路径未知的任务。你只有一个症状，或者一个模糊的改进方向，不知道根因在哪里。这类任务就应该考虑上最强模型，并且给它足够权限去探索、验证和回滚。用文章里的话说，如果你在 A 点，知道 B 在哪里，没必要动用重武器。如果你连 B 在哪里都不知道，那就别让一个只能局部推理的模型带你迷路。

你可能会问，既然 Fable 五这么强，那是不是 prompt 可以更随意？文章里的例子确实说明，它能处理更含糊的真实 prompt，比如一个错误栈加一个 fix。但这不等于我们应该故意写烂 prompt。我个人认为，更好的做法是把人类意图、边界和验证方式说清楚，把探索路径留给模型。比如你可以说，线上某个接口最近延迟升高，怀疑和缓存失效有关，但这个判断不一定对，请你先找证据，再给出最小风险修复方案，并说明怎么验证。这样的 prompt 不是把解法喂给模型，而是告诉它任务的成功标准和不要盲从你的假设。文章里第二个评测任务正是看模型会不会挑战用户错误的模块判断。我们在真实使用时也应该允许模型说，你的假设可能不对。

另一个可以直接借鉴的实践，是给自己的团队做小型 eval。你不一定要做 CursorBench 那么完整，但可以收集你们项目里真实发生过的十几个难问题。不要只选算法题，也不要只选能在单文件里解决的小 bug。可以包括一次线上报错栈，一次错误定位，一次跨模块重构，一次性能瓶颈，一次用户描述不完整的产品 bug。然后把当时的上下文尽量还原，分别喂给不同模型，看它们能不能理解任务、能不能自己找根因、会不会被错误假设带偏、会不会验证修复。这个小 eval 的价值不在于得出一个全球通用排名，而在于回答一个更具体的问题，哪个模型适合你们的代码库和工作方式。

但这里要注意一个坑。真实工程评测不能只看最后有没有 patch。很多 agent 会生成看起来合理的修改，但没有跑对测试，或者没有覆盖边界，甚至偷偷绕开失败测试。Cursor 团队看 trace 的做法很值得学。你要看模型为什么这么做，它有没有读关键文件，有没有意识到风险，有没有验证自己的结论。如果一个模型十次里八次能给出看似可用的 diff，但推理路径经常是猜的，那放进生产代码库会很危险。反过来，一个模型可能某次没有完全修好，但它准确指出了根因和下一步验证方向，这在复杂问题里也很有价值。

再说成本。文章的态度不是强模型万能，而是强模型要用在能力约束的地方。这对于中国工程团队尤其现实，因为很多团队会关心预算、响应时间、私有化、合规和稳定性。一个可行的落地方式是，把 agent 工作流拆成几层。第一层是检索和阅读，让轻模型做摘要、找文件、整理最近 commit。第二层是常规修改，让中等模型按明确指令改代码。第三层是难题攻关，让最强模型做全局推理、方案设计和关键修改。第四层是审查和验证，可以再用不同模型交叉检查，或者回到人类 reviewer。这样既不会把所有压力放在最贵模型上，也不会在最难问题上省错钱。

还有一个很重要的实践点，是让 agent 先建立观测，而不是直接动刀。登月例子里，Fable 五第一次没有直接冲月球，而是先入轨收集遥测数据。这对软件工程来说，就是先加日志、先跑基准、先补测试、先确认复现路径。很多开发者使用 AI agent 时太急，一上来就让它 fix。模型也会迎合你，马上改代码。可如果问题足够复杂，最好的第一步可能不是修，而是弄清楚系统现在到底怎么坏。你可以在 prompt 里明确要求，先不要修改，先给出你需要验证的假设和最小观测计划。等证据够了，再让它改。强模型的全局推理能力，应该用来减少盲修，而不是让盲修变得更快。

对团队协作来说，文章里让 agent 阅读同事最近 commit 的做法也可以扩展。比如你准备改一个公共模块，可以让 agent 先总结这个模块过去一周的变化，列出可能依赖它的调用方，指出正在进行中的相邻改动。你做 code review 前，可以让 agent 先解释这个 PR 的意图、影响面和风险点。你准备重构前，可以让 agent 生成迁移清单，标出哪些测试能保护行为，哪些地方没有覆盖。注意，这些都不是让模型替你做最终判断，而是让它降低你进入上下文的成本。人类工程师最宝贵的是判断力，AI 最适合先把信息铺开，让你不用从零读起。

那我们也要讲清楚文章没有说什么。它没有给出 CursorBench 的完整题集，也没有详细解释百分之七十二点九这个分数背后的统计方法。它也没有说 Claude Fable 五在所有场景都超过所有模型，更没有说人类审查可以取消。它讲的是 Cursor 团队在自己的评测和真实工作流里观察到，Fable 五在最困难、最含糊、最需要长程规划的问题上改变了他们对 coding agent 能力边界的判断。我们不能把这篇文章读成绝对结论，但可以把它读成一个很有价值的行业信号。

文章最后说，Schmidt 还在寻找 Claude Fable 五的极限。下一步，他想看模型能在无人看管的情况下管理一个后端系统多久，目标是连续运行几天到几周。Cursor 内部也在用这个模型主动寻找性能瓶颈和用户痛点，而不是等用户报告之后再处理。同时，他们也在构建更复杂、更贴近现实的 eval 环境，用来衡量下一代模型。Schmidt 说，有一类问题以前大家甚至不会去想，因为看起来根本不可接近。有了 Fable，他很兴奋去推动这条边界。

这其实是整篇文章最值得我们记住的地方。AI 编码工具的进步，不只是让现有任务更快完成，也可能改变我们愿意尝试什么任务。过去因为太贵、太乱、太不确定而被放弃的重构、迁移、性能治理、系统理解，可能会因为 agent 的能力提升重新进入待办列表。当然，这不意味着技术债会自动消失，也不意味着架构判断可以外包。但它意味着工程师可以把更多精力放在定义目标、审查方案和把握风险上，而不是被大量机械的上下文恢复和局部试错耗尽。

我个人认为，这篇文章最有启发的一点，是它把模型评估从回答正确，推进到了理解使命。一个模型能不能写出一段正确代码，已经不是最前沿的问题。更前沿的问题是，它能不能在目标模糊、信息不全、用户还可能判断错的情况下，抓住真正要解决的工程问题，并且自己规划验证路径。Cursor 用 CursorBench、trace 分析和真实工作流去观察这件事，给了我们一个比公开榜单更接近实战的视角。

最后用两三句话收个尾。Claude Fable 五在这篇文章里的价值，不只是分数高，而是它让 Cursor 看到 coding agent 从局部修补走向全局任务推理的可能。对我们来说，真正该学的不是盲目追最强模型，而是把最强模型放到最难、最模糊、最需要判断的地方，并且用自己的真实任务去评估它。AI 编程下一阶段的竞争，很可能不再是谁更会写代码片段，而是谁更能理解整个工程使命。
