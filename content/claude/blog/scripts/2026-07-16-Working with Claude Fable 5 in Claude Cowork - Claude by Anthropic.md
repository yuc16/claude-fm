今天我们解读的文章是 Working with Claude Fable Five in Claude Cowork，中文我译作，在 Claude Cowork 中使用 Claude Fable 五。这篇文章发表于二〇二六年七月十六日，也就是三天前。今天是二〇二六年七月十九日，所以这是一篇非常新的文章，讲的是 Anthropic 在当前这一代智能体办公产品里，怎么让最强的通用可用模型承担更长、更复杂、更不确定的知识工作。

这期为什么值得听呢？因为这篇文章表面上是一个产品使用指南，读起来只有五分钟，但背后其实透露了一个很重要的变化。过去我们使用大模型，很多时候像是在和一个很聪明但容易忘事的实习生合作。你要把任务拆得很细，要一步一步盯着它，提醒它格式，提醒它目标，提醒它前面说过什么。可这篇文章讲的 Claude Fable 五，Anthropic 的定位已经不只是聊天助手，而更像一个能自己规划、自己调用工具、自己检查结果，并且能跑很久的同事。这个变化对软件工程师、产品经理、数据科学家、AI 应用开发者都很关键，因为它会改变我们设计工作流、写提示词，也会改变我们判断哪些任务适合交给 AI 的标准。

先说背景。Claude Fable 五是 Anthropic 当时最强的通用可用模型，文章里用的说法是 generally available model，也就是普通用户和团队可以实际使用的模型，而不是只在实验室里展示的研究模型。它的核心能力指向三件事，长时间运行，复杂任务，异步工作。这里的异步工作很重要。它不是你发一句话，它回一句话的聊天节奏，而是你给它一个目标，它自己去读文件、查资料、拆任务、并行处理、生成草稿、回过头检查，甚至一个任务可能跑几个小时，或者跨越几天。

Anthropic 举了几个典型例子。比如做深度研究，然后把研究结果写进第一版备忘录。比如在生成董事会汇报材料之前，先做尽职调查。再比如打开一个文件夹，逐份审阅多个合同，并且做 redline，也就是类似法律文档里的批注修订。这些任务共同点是什么？不是单步问答，不是写一段文案，不是查一个 API 怎么用，而是有很多步骤，前一步会影响后一步，中间还充满判断。

我们用工程师熟悉的类比来理解。以前你让模型帮你修一个 bug，比较像把某个函数贴过去，让它看一下哪里错了。现在 Claude Fable 五想处理的是，你把整个 repo、issue 列表、日志、监控告警、用户反馈和上线时间线都给它，然后说，请找出最近一周支付失败率上升的原因，判断影响范围，给出修复方案，写一份给管理层看的说明，再开几个 Jira 任务。这就不是一次代码补全了，这是一个跨数据、跨工具、跨角色的完整工作。

文章强调，要最大化 Claude Fable 五的能力，你需要改变和它协作的方式。Anthropic 说，过去那些最佳实践仍然重要，比如提示词，也就是 prompt，要写清楚；上下文要给足；技能，也就是 skills，要把重复流程沉淀下来，比如每周团队更新、销售电话准备、客户反馈分析。这些都没有过时。只是模型更强之后，它能把这些上下文、偏好和技能应用到整个长任务里，而不是只在开头几轮记得，跑着跑着就忘了。

这句话其实很关键。以前的大模型在长任务里常见问题是失焦。你开头说，这份报告要给 CFO 看，语气要简洁，重点是预算风险，不要写成技术细节。然后它生成到第三个小节，可能就开始变成面向工程团队的长篇解释。你再提醒它，它又改回来。这样的互动模式，本质上你还在做项目经理，而且是一个很累的项目经理。Claude Fable 五想要达到的状态是，你像给一个能力很强的同事 brief 一样，说明情况，约定什么叫好结果，然后把工作交出去。你不用每一步都检查，它自己会在过程中检查，你更多时间花在决定到底应该做什么，而不是盯着它怎么做。

接下来要把 Claude Cowork 讲清楚。Claude Cowork 是 Anthropic 的 agentic AI system for knowledge work。agentic 这里可以理解成智能体式的，也就是不是单纯生成文本，而是能够围绕目标规划、调用工具、读写文件、拆分任务并推进结果。知识工作则包括研究、分析、写报告、做演示、整理客户反馈、处理项目管理系统里的信息等等。

文章说，Claude Cowork 是为了创造 finished work，也就是成品工作，而不是只给你灵感或半成品。你给它一个目标，它会管理剩下的过程。当任务又大又复杂时，它会把大任务拆成多个部分同时运行，每个部分由一个 subagent，也就是子智能体来处理。子智能体可以理解为 Claude 的一个独立实例，拿到大任务中的一小块，完成之后向主流程汇报。这样 Claude 就能更快处理大量信息。

如果你做过分布式系统，这个机制很好理解。一个大查询如果顺序扫描所有数据会很慢，就拆成多个分片并行处理，最后聚合结果。但难点也在这里。并行不是免费午餐。每个子任务如果理解不一致，最后合并时就会出现口径混乱。一个子智能体认为客户取消是因为价格，另一个认为是因为 onboarding 太复杂，第三个只看到了地域因素，最后主智能体要做综合判断。Claude Fable 五的优势就在于，它更擅长在开始时设计划，在过程中根据计划检查结果，并且能让这些步骤彼此衔接。

文章举了一个财务预算的例子，很适合说明长任务里的错误传播。假设你让 Claude 根据今年实际数据建立明年预算。它要读取电子表格，提取 run rate，也就是当前运行速率或者趋势基准，然后预测每一项费用，把预测值和目标对齐，再写总结。如果它一开始误读了某个 run rate，比如把月度费用当成季度费用，后面每一条预测都会被污染。这和工程里的根因错误很像。一个基础配置错了，后面所有测试都可能看起来合理，但其实全错。文章说，Claude Fable 五会在开工前规划流程，并在运行过程中检查结果，所以有机会在任务还没结束时发现这个误读的数字，然后修正它，而不是等你最后审阅时才发现整份预算都要重做。

这里我个人认为，Anthropic 想传递的不是单纯“模型更聪明了”，而是“错误发现的位置前移了”。传统聊天式 AI 的错误常常在成品里暴露，用户看到结果后再返工。智能体式工作流如果设计得好，错误会出现在计划面板里，出现在中间检查里，出现在来源追溯里。这对于企业采用 AI 非常重要，因为企业最怕的不是 AI 不会写，而是 AI 写得很像真的，但中间依据错了。

那在 Claude Cowork 里，什么时候该选 Claude Fable 五呢？文章说，它不是默认模型。发布时默认是 Claude Sonnet 五。Sonnet 五适合日常任务，尤其是那些你自己也能很快过一遍的工作。比如改一封邮件，整理会议纪要，生成一个简单计划，写一段内部说明。Opus 则适合形状比较清楚的深度工作，也就是你知道最终结果大概长什么样，只是需要模型投入更多推理和生成能力。Claude Fable 五面向的是最复杂、最模糊、甚至过去模型可能够不着的项目。

这个分类很实用。你可以把模型选择想象成团队里分配人力。日常小活儿，不需要让公司最资深的人来做。结果样式明确但要求很高的工作，可以交给一个稳定可靠的专家。最难的是那些问题本身还没完全定义的项目，比如“我们为什么企业客户续费变慢了，给我一套可以执行的改进方案”，或者“把我们现有的开发流程重新设计成更适合 AI coding 的流程”。这类任务不只是执行，还要不断做判断。文章说，Claude Fable 五会花更多时间思考，也消耗更多使用额度，但对于耗时很长或者出错代价很高的工作，这个成本可能值得。

Anthropic 还建议，可以用 effort 设置进一步调节。effort 可以理解为努力程度或推理投入。更高 effort 时，Claude Fable 五会在开始任务前做更多规划，在运行过程中做更多检查。适合那些复杂、多步骤、你希望它从头到尾完成的项目。更低 effort 则更快，但仍然能利用 Fable 五本身的智能。适合那些需要前沿判断，但不需要深度探索的任务，比如由很多简单步骤组成的智能体运行，或者结果很容易检查的工作。文章还提到，在他们测试中，Claude Fable 五低 effort 往往可以达到甚至超过早期模型最高 effort 的表现。

这句话对工程团队很有启发。我们在落地 AI 时经常纠结成本和质量。很多人默认最强模型加最高推理就是最好，但实际不是。你要看任务的风险、可验证性、复杂度。如果是批量把一百个客服记录按既定标签分类，标签很清楚，样本也容易抽检，那不一定要最高 effort。如果是给董事会做一份关于明年市场进入策略的材料，而且要基于十几个信息源、多个内部表格、竞争动态和财务目标，那就值得把 effort 拉高。模型能力不是开关，而是资源调度问题。

文章还讲了安全机制。Claude Fable 五带有一组新的 classifiers，也就是分类器，是独立的 AI 系统，用来检测涉及网络安全、生物和化学相关的潜在滥用请求。一旦触发，响应会自动改由 Claude Opus 四点八处理，并且会告知用户。聊天从那一刻起会停留在 Opus 上，如果想回到 Claude Fable 五，需要新开一个对话。Anthropic 说他们把这些防护调得比较保守，这样才能既安全又快速地把 Mythos 级模型开放给通用使用，所以有时会误伤一些无害请求，尤其是在 Claude Cowork 里只是轻微碰到相关话题的短语。他们也说会继续减少这些 false positives，也就是误报。

这段对实际使用很重要。比如你在做安全团队的季度风险报告，里面出现了漏洞、攻击路径、渗透测试这些词，可能就会触发分类器。或者你在生物医药公司做商业分析，文档里有化合物、实验流程、靶点这些词，也可能被误判。原文没有给规避策略，也不应该教人绕过安全系统。我的建议是，在企业工作流里要把这个行为当作系统特性来设计。也就是说，如果某个任务突然切换到 Opus 四点八，不要惊慌，先确认模型提示和任务记录。如果你确实需要 Fable 五继续做普通知识工作，就新开对话，并且把任务描述聚焦在合规、非操作性的分析目标上。

接下来是文章里非常有意思的一段，关于从想法开始工作。Anthropic 说，当你启动一个任务时，你不一定一开始就知道自己到底要完成什么。这个早期阶段，Claude Fable 五可以成为强大的 thought partner，也就是思考伙伴。在 Claude Cowork 里，它不是空想，它可以基于你的 skills、工具和知识工作。

比如你看到一条与你行业或公司有关的新闻公告，可以对 Claude 说，这是公告，请结合你知道的我们组织情况，告诉我对我们有什么变化。你读完输出之后，开始形成策略，而 Claude 会在同一个 Claude Cowork 对话里继续制定计划，生产你需要的资产。这里的重点是同一个对话。因为头脑风暴时你们已经讨论过目标、限制、取舍、方向，真正进入执行时，Claude Fable 五已经带着这些上下文。

这和我们平时开会很像。如果你和同事一起讨论了一个小时，最后说，那你把方案写出来。同事不需要你重新讲一遍背景，他知道刚才大家否掉了哪些方向，知道老板关心什么，知道这个方案不能依赖新增招聘。可如果你把写方案交给另一个没参加会议的人，你就要重新 brief。Claude Cowork 的价值是让 brainstorm 到 production 的链条在同一个上下文里完成。

文章举了 Anthropic 内部一个数据科学家的例子。这个数据科学家带着一个新 analytics dashboard，也就是分析仪表盘的想法来找 Claude Cowork，当时团队还在探索这个仪表盘应该显示什么。因为 Claude Fable 五在对话期间能读取团队使用数据，它知道哪些问题通常要过几周才会被发现，于是对指标进行排序，找出哪些指标如果提前展示，就能更早发现这些问题。到对话结束时，数据科学家得到了一组值得加入的指标短名单，还有一个可点击的原型。

这个例子背后的方法论是，别只让 AI 生成你已经想好的东西，而要让 AI 利用真实材料帮你想清楚问题。很多团队做仪表盘会犯一个错误，就是先问大家想看什么指标，然后把所有人提到的指标堆上去。结果仪表盘又复杂又没用。更好的方式是回看历史问题，看看哪些异常发现得太晚，再倒推出应该监控什么。Claude 如果能访问真实使用数据，就能把“我们觉得重要”变成“过去事实证明有用”。这就是上下文的价值。

原文给了两个很直接的 prompt。一个是，让 Claude 采访你。可以说，在你开始之前，问我所有你需要知道的问题，以便把这件事做好。另一个是，让 Claude 给方向。可以说，这是我大概想要的东西，给我三个可能的方向，并且每个方向给一个快速样例。这两个提示词看起来简单，但很适合长任务。因为它们不是让模型马上交付，而是先让模型帮助定义任务。对于不确定的问题，最危险的不是答案不好，而是问题定义错了。

再往下，文章重点讲 context，也就是上下文。Claude 在 Claude Cowork 中使用的上下文包括你的 prompt，也就是提示词；你共享的文件和文件夹；以及你连接的工具，比如 Asana、Hubspot、Jira、Slack 等。Anthropic 建议，当你在 Claude Cowork 里给 Claude Fable 五布置任务时，可以想象你是在给同事 brief 一份报告。你不会只递给他一堆规则。你更可能会告诉他，这份报告给谁看，什么时候要，它要达到什么目的，然后相信他能做出好判断。

这点非常重要，因为很多人写 prompt 的方式还停留在规则堆叠。不要使用复杂词汇。不要超过两页。不要用被动语态。不要加入没有依据的结论。不要写得像广告。所有这些 constraints，也就是约束，都有用。原文也说，约束仍然有用，比如“控制在两页以内，用简单语言”。但是约束只告诉 Claude 什么不能做。上下文告诉 Claude 这件工作是为了什么，因此当遇到你没有预先写进约束的情况时，它才能自己做正确决定。

举个例子。你让 Claude 写一份事故复盘。你可以写很多规则，比如语气客观、不要甩锅、列出时间线、给出行动项。但如果你补充上下文，说这份复盘要发给一个刚刚因为事故损失严重的大客户，同时也会被内部 SRE 团队用于改进值班流程，那 Claude 就会知道它需要兼顾外部信任和内部技术深度。它可能会在客户版本里更强调影响、补救和承诺，在内部版本里更详细写监控缺口、变更流程和 ownership。这种判断很难靠一堆“不要怎样”来完成。

文章说，Claude Fable 五能处理长、多步骤任务，部分原因正是它会利用上下文。当它工作过程中遇到问题或决策点，会在你共享的上下文里找答案。它还会利用上下文检查自己的工作。所以你可以给它一些具体判断标准。比如给它一份报告早期草稿和最终版本，Claude 就可以通过比较两者差异，推断你的标准是什么。哪里被删掉了，哪里被扩展了，语气怎么变了，结构怎么调整了。这比你抽象地说“写得更高管友好”要更可操作。

对于软件团队，这个思路可以迁移到很多地方。你想让 Claude 生成设计文档，就不要只说“按我们团队风格写”。你可以给它过去一份比较差的设计文档和最终通过评审的版本，让它学习差异。你想让它帮你做代码评审总结，可以给它几个历史 PR 的 review 记录和最后合并说明。你想让它分析客户反馈，可以给它之前人工做过的分析报告。模型越强，越能从这些样例里提取隐含标准。

不过，文章也提醒，上下文多不是没有代价。Claude 在长对话里为了保持理解，每次你发送新消息，都会重新读取整个 conversation，也就是整段对话。因此，一个很长的对话可能会消耗更多使用额度。Anthropic 建议，为新任务开一个新对话。定时任务也会计入额度，所以要偶尔检查，把不再需要的关掉。

这个提醒很实际。很多人把一个项目所有事都塞进一个超长聊天，觉得这样模型记得多。但如果这个对话已经包含大量历史争论、过期假设、旧版本文件和无关闲聊，模型不仅消耗更高，还可能被噪音干扰。我的建议是，把对话当作工作台，而不是垃圾箱。一个任务一个工作台。真正长期有用的信息，应该沉淀成项目说明、共享文件、memory，或者 skill，而不是靠一条无限增长的聊天记录维持。

然后文章进入委托任务的主题。你可能习惯把任务拆成很多部分，一步一步提示 Claude。Claude Fable 五需要的中间提示少得多，所以你可以在 Claude Cowork 里委托完整工作。文章也没有说永远不要写步骤。如果过程对你很重要，当然应该写。比如某个指标必须每次用完全相同的方法计算，这时步骤就是规范。但如果你关心的是结果，那么描述目标，让 Claude 自己找路径，往往更好。因为一步一步的 prompt 可能把它限制在你想到的步骤里，而目标给它空间寻找更好的方法。

这对工程师尤其反直觉。我们习惯把复杂问题拆成确定流程，因为编程就是控制细节。但 AI 智能体不是传统程序。你如果把每一步写死，就像请了一个经验丰富的工程师，却只允许他照着你写的 shell script 做事。这样可以保证可重复，但会牺牲判断力。什么时候要流程，什么时候要目标，要看任务性质。合规报表、财务口径、指标定义，适合流程化。开放研究、根因分析、战略建议、客户洞察，适合目标化。

文章给了几种委托方式。先是委托方法。你把材料给 Claude，描述想要结果。比如，这是上季度客户反馈，请找出取消订阅上升的原因，以及我们应该改变什么。这里可能有很多合理分析路径。可以按客户规模看，可以按行业看，可以按生命周期阶段看，可以按销售承诺和产品体验差距看。正确路径取决于反馈内容。Claude Fable 五擅长的就是读完所有材料，选择方法，然后用反馈本身检查结论，再把结果交给你。你可以审查答案，不必事先指定它怎么走。

再是委托流程。skill，也就是技能，可以教 Claude 你们团队使用的流程。比如怎样生成报告，怎样格式化 deck，也就是演示文稿，怎样跑分析。你不需要告诉它使用哪些技能、按什么顺序使用。你可以说，像我们一直那样整理季度回顾。Claude Fable 五会在合适时机调用合适技能。这里的关键是把重复流程从 prompt 里拿出来，变成可复用资产。

文章还有一个小标题是委托时机，后面紧接着说，要把比过去更难的工作交给 Claude Fable 五，甚至是你原本以为 AI 做不了的工作。哪怕工作混乱、不清楚，或者可能要几个小时、几天才能完成。你可以描述它，看看模型能不能达到那个水平。选一个你负责的结果，说清楚你会如何判断它好不好，然后让 Claude 提出步骤。有些工作还可以变成 scheduled task，也就是定时任务，自动运行。Claude 可以把流程保存成 skill，并按计划通过你已经连接的工具来执行。

这段很有启发，但也需要谨慎。文章鼓励大家挑战模型上限，不要只给它碎片任务。我个人认为这确实是使用强模型的分水岭。你如果永远只让它改邮件、写摘要、翻译文本，你感受到的提升会很有限。真正的提升来自把一个完整责任交出去，比如“每周一早上，帮我查看上周所有 P 零和 P 一事故、相关 Slack 讨论、Jira 状态和客户反馈，生成一份工程运营风险简报”。但同时，你要从可验证任务开始。先选你自己熟悉领域里的工作，这样你能快速判断它有没有靠谱，而不是一上来就把陌生高风险任务全自动交给它。

Claude Fable 五能承担长任务，部分原因是它会设计划并遵循计划。在 Claude Cowork 里，你可以看到这个计划。对话旁边的面板会列出它打算做什么，然后显示它正在读取和写入哪些文件，使用哪些工具和技能。这个面板是你早期纠偏的机会。原本你可能到最终输出才发现的错误，现在会表现为计划里某一步不对。你可以用一句话纠正计划，Claude 会调整，而不必从头开始。

这个设计非常像人类团队里的项目计划和执行日志。你给同事布置任务后，不希望他一周后突然丢给你一个成品，中间完全黑箱。更好的方式是，他先给你一个 plan，说我会先看客户反馈，再按取消原因聚类，然后交叉对比账户规模，最后给出建议。你一看就能发现，如果我们这个季度最大的变化是定价策略，那它不能只看产品反馈，还要看销售和账单数据。于是你早早纠正方向，避免最后返工。

当工作完成后，Anthropic 建议像审查同事工作一样审查 Claude 的工作。打开它生成的文件，读一遍。如果觉得哪里不对，运行记录仍然在对话里。你可以回看它列出的步骤，包括读了哪些文件，用了哪些工具。文章还说可以展开它的 thinking，查看某个决策背后的 reasoning，也就是推理。或者直接问它，这个数字从哪里来？Claude 会指向来源。

这里要注意，审查不是不信任，而是专业协作的一部分。我们不会因为同事很强就完全不 review 代码，也不会因为财务同事可靠就不看预算假设。AI 也一样。最好的使用方式不是盲信，也不是事事手工重做，而是利用它留下的计划、来源和中间记录做抽样和关键点审查。比如你可以专门检查几个高影响数字，几条关键结论，几个最容易产生口径差异的地方。这样既省时间，又能控制风险。

文章建议从你知道如何验证的工作开始，比如给你的团队写下一季度计划的第一版。如果你很了解团队优先级，就能很快判断 Claude 的版本站不站得住。等结果证明可靠，再逐步委托更大的工作。这个路径很稳。对企业团队来说，可以先用 AI 做 first draft，也就是第一稿，做分析准备，做材料汇总。等团队对它在某类任务上的表现有信心，再让它跑定时流程，或者承担更多端到端工作。

最后一部分是给 Claude Fable 五提供上下文的建议。文章说，更强的模型会提高每个连接的价值。从你共享的文件夹，到团队工作的工具，连接越有用，模型越能发挥。原文列出了一些建议，其中有些小节在我们拿到的文本里没有展开，比如 connect your tools，也就是连接你的工具，以及 revisit what you set up for earlier models，也就是重新检查你为早期模型设置过的内容。不过文章整体意思很清楚，Claude Fable 五越能长期执行，越需要可靠、丰富、更新及时的上下文。

连接工具这件事，虽然原文没有展开，但从前文可以推出来。Claude Cowork 依赖 Asana、Hubspot、Jira、Slack 等工具来理解真实工作。如果你只在 prompt 里描述“客户最近不满意”，模型只能泛泛分析。如果它能读 Hubspot 里的销售记录、Slack 里的客户成功讨论、Jira 里的 bug 状态和共享文件夹里的季度总结，它就能做更接近真实业务的判断。当然，这也意味着权限治理更重要。不要为了省事把所有数据都无脑开放。企业应该按角色、项目和数据敏感级别配置连接，让 Claude 能看到完成任务所需的材料，但不要超过需要。

文章特别展开了写作声音，也就是 voice 和 tone。每个新模型都有自己的写作默认值，语气、长度、常用表达都可能不同。Anthropic 说你可能会注意到 Claude Fable 五有一些默认，比如在长会话中写得更简短，或者不太容易读懂。声音、语气和风格都可以定制。你可以通过 prompt 或项目说明告诉它，比如使用朴素、直接、清楚的语言。如果你使用 memory，也就是记忆功能，要偶尔检查 Claude 保存了哪些关于你写作偏好的信息。对于需要特定声音的文档，可以让 Claude 通过连接器或现有文件读你过去的写作，把它发现的风格保存成一个 skill，下次做类似工作时调用。Anthropic 说，他们发现 Claude Fable 五比早期模型更严格遵循长期指令，也更擅长在需要时使用保存材料。

这一段对很多团队很重要，因为“AI 味儿”常常不是能力问题，而是风格配置问题。很多模型默认写作喜欢堆形容词，喜欢说“在快速变化的商业环境中”，喜欢给每个段落加总结句。工程团队内部其实更需要短、直、可执行。你可以建立团队写作 skill，比如事故公告怎么写，设计文档怎么写，周报怎么写，客户邮件怎么写。以后不要每次 prompt 里重复“不要废话”，而是让 Claude 学会你们的标准。

至于重新检查为早期模型设置过的内容，虽然原文没有展开，我个人理解这是一个很实用的提醒。因为模型变强后，旧提示词可能反而束缚它。你以前为了防止模型跑偏，写了很多非常细的步骤、很多重复提醒、很多硬性格式。到了 Claude Fable 五，这些东西可能还有效，但也可能让它无法选择更好的方法。所以升级模型时，不应该只把旧工作流原封不动搬过去。你应该问问自己，哪些规则是业务必须，哪些只是为了弥补旧模型能力不足。如果是后者，就可以简化，让模型承担更多判断。

把这些内容落到我们听众的日常工作里，可以怎么做？如果你是软件工程师，可以从三个任务开始试。一个是技术调研。不要只问“某某框架怎么样”，而是把你的项目约束、现有架构、性能瓶颈、团队经验、上线周期都给 Claude，让它给出几条方案、取舍和迁移计划。一个是事故复盘。把日志摘要、时间线、监控截图说明、相关 issue 和聊天记录给它，让它生成第一版复盘，并标出证据来源。一个是需求拆解。把 PRD、历史 bug、客户反馈和当前 roadmap 给它，让它识别风险、拆出工程任务、补充验收标准。

如果你是 AI 应用开发者，可以重点观察它对长工作流的启发。Claude Cowork 的设计不是让一个模型一次性吞下所有任务，而是有计划，有子智能体，有工具连接，有技能，有运行记录，有检查点。你在自己产品里做 agent，也可以借鉴这个思路。不要只做一个“会调用工具的聊天框”。要考虑任务如何拆分，子任务如何汇报，主流程如何聚合，用户在哪里能看到计划，如何纠偏，如何追溯来源，如何把成功流程沉淀成技能，如何控制使用额度和权限。

如果你是团队负责人，可以从“可验证、低风险、高耗时”的工作开始。比如每周项目状态汇总，客户反馈主题分析，竞品动态简报，招聘候选人材料整理，研发度量报告第一稿。不要一开始就全自动发给客户或高管。先让 Claude 生成，团队审查，记录问题。几轮之后，你会发现哪些部分稳定可靠，哪些需要更多上下文，哪些必须保留人工判断。然后再考虑定时任务和更大范围的委托。

也要避几个坑。不要把模型选择当作炫技。不是所有任务都需要 Claude Fable 五。简单任务用 Sonnet 可能更快、更省。不要把上下文当作越多越好。上下文要相关、干净、可更新。不要把约束写成牢笼。对开放任务，给目标和判断标准，比写死每一步更有价值。不要忽视计划面板。智能体时代，真正的交互不只发生在最终答案里，也发生在执行计划和中间记录里。不要忘记额度。长对话、定时任务、多个工具读取都会消耗使用量。不要忽视安全分类器带来的模型切换，尤其是安全、生物、化学相关领域的团队。

回到原文，Anthropic 最后说，随着前沿智能继续发展，Claude Cowork 会越来越有能力，能够支持更长时间运行的工作，并解锁更多知识工作场景。学习如何 prompt，如何管理上下文，如何检查 Claude 的工作，如何委托任务，会帮助你利用 Claude Fable 五以及未来模型的能力。

我的总结是，这篇文章真正讲的是从“操作 AI”到“管理 AI 同事”的转变。Claude Fable 五的价值不只是回答更聪明，而是能在 Claude Cowork 里围绕一个复杂目标持续推进、调用工具、检查自己、形成成品。对我们来说，关键能力也要升级：少写一点机械步骤，多讲清楚目标、背景和判断标准；少盯每一句输出，多审计划、证据和关键结论。谁先学会这种协作方式，谁就更可能把下一代 AI 的能力转化成真实生产力。
