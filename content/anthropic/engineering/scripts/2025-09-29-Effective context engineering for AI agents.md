今天我们来聊一篇来自 Anthropic 工程团队的文章，英文标题是 Effective context engineering for AI agents，我把它翻译成"给 AI 智能体做有效的上下文工程"。这篇文章发表于二〇二五年九月二十九日。

现在是二〇二六年六月，距离文章发表已经过去了将近九个月。在 AI 这个领域，九个月说短不短，说长也不算特别长，但足以让很多东西发生显著变化了。所以在进入正题之前，我想先帮你定一下位：这篇文章写于什么样的历史时刻？

二〇二五年下半年，整个 AI 行业正处于一个非常关键的过渡期。大家都能感觉到，AI 不再只是一个"问一答一"的聊天工具了，越来越多的工程师开始尝试构建能够自主执行任务的智能体，也就是 agent。agent 能够连续调用工具、做多步骤推理、跑几分钟甚至几十分钟的任务。Anthropic 自家的 Claude Code 就是在那段时间里迅速成为开发者圈里最热议的产品之一，大家发现让 Claude 自主写代码、改代码、跑测试，效果出乎意料地好。

就是在这个背景下，Anthropic 工程团队写了这篇文章。它要解决的问题是：当 agent 开始在一个循环里反复推理的时候，工程师面临的核心挑战是什么？答案是：上下文管理。而他们把解决这个问题的方法论，叫做 context engineering，上下文工程。

我个人认为，这篇文章放到今天仍然非常有价值。因为它触及的底层问题——怎么管理大模型的注意力资源——这个问题并没有随着模型能力的提升而消失。恰恰相反，随着 agent 承担的任务越来越复杂、运行时间越来越长，上下文管理反而变得愈发重要。

好，我们正式进入内容。

先来聊一个很多人都在用、但未必真正理解区别的对比：prompt engineering 和 context engineering，提示词工程和上下文工程，到底有什么不同？

提示词工程大家应该都不陌生。它的核心问题是：怎么写指令，让模型给出你想要的答案？你要怎么写 system prompt（系统提示），怎么给模型一个角色定义，怎么用 few-shot 也就是少样本示例来引导模型的输出格式。这套方法论在大模型应用的早期非常有效，因为当时绝大多数的应用场景本质上都是"单次问答"——你给模型一段输入，它输出一段结果，完事儿。

但是，当我们开始构建 agent 的时候，整个问题的结构就完全变了。

一个 agent 不是执行一次推理就结束，它是在一个循环里反复执行推理。你可以想象它是一个永不停歇的工作流：推理，调用工具，拿到工具结果，再推理，再调工具，循环往复，可能持续几分钟、几十分钟甚至几个小时。

在这个过程中，模型的上下文窗口里装的东西，早就不只是你精心设计的 system prompt 了。里面还有工具调用的历史记录，工具返回的大段数据，之前步骤产生的中间结论，从外部检索回来的文档，等等。这些内容会不断积累，形成一个持续增长、持续演化的信息池。

这时候，传统的提示词工程就不够用了。你需要回答一个更大的问题：在任务执行的每一个时刻，模型的上下文窗口里应该装什么信息？哪些信息应该被保留，哪些应该被清除，哪些应该被压缩，哪些要在需要的时候才动态加载进来？

这套解答方法，就是上下文工程的全部内容。Anthropic 给它下的定义是：在推理期间，策划和维护最优 token 集合的一系列策略。这里有个关键词："最优"。不是最多，而是最优——信噪比最高的那些。

理解了这个定义，我们来聊一个更底层的问题：为什么上下文必须被精心管理，而不是随便往里塞？

这里有一个非常重要的概念，文章把它叫做 context rot，我译为"上下文腐化"。

很多人可能会有一个直觉：现在的大模型上下文窗口越来越长了，都支持几十万甚至上百万个 token 了，那我直接把所有信息都塞进去不就完了，省心省力？

这个直觉是错的。或者说，这个直觉低估了一件事：模型能"装"多少，和模型能"用好"多少，是两回事。

研究人员有一种专门的测试方法叫做"大海捞针"，英文是 needle-in-a-haystack。做法是：把一小段关键信息，藏在一大段与它无关的文本中间，然后问模型这段关键信息是什么。实验结果非常清晰地显示：随着"大海"变得越来越长，模型找到"针"的准确率会显著下降。这个现象不是某一个模型特有的问题，而是在所有大模型上都普遍存在，只是程度有所不同。

为什么会这样？要解释这个现象，我们得稍微往底层走一步，聊一聊 Transformer 架构。

现在主流的大语言模型都是基于 Transformer 架构构建的。Transformer 的核心机制叫做注意力机制，英文是 attention。它的工作方式是：上下文窗口里的每一个 token，都要跟其他所有 token 建立关系，计算彼此之间的关联程度。如果上下文里有 n 个 token，那么就需要计算 n 的平方这么多对关系。

当 n 比较小的时候，这没什么问题，每一对关系都能得到充分的计算资源。但当 n 很大的时候，n 的平方就是一个天文数字，模型的注意力会被极度摊薄，对每一对关系的精准度都在下降。

你可以把这个过程想象成一个人开会的场景。如果会议室里只有你和另外两个人，那你可以轻松关注到他们说的每一句话，听清楚所有细节。但如果会议室里坐着三百个人，所有人同时在讲话，那不管你多聪明，你的注意力都会被稀释到极限，你只能大致抓住一些主要内容，很多细节会遗漏。

文章把这个叫做"注意力预算"，英文是 attention budget。大语言模型的注意力是一种有限的资源。每往上下文窗口里加入一个新的 token，就会消耗一点注意力预算，降低一点模型对其他信息的关注精度。上下文越长，这个预算被摊薄得越厉害，模型的表现就越容易出问题。

值得注意的是，这个下降是一个梯度，不是一个悬崖。模型不会在某个 token 数量达到之后突然变笨，而是随着 token 数量的增加，它的信息召回精准度和长程推理能力都会温和地、持续地下降。

还有一个有趣的细节：模型的训练数据里，短序列的文本通常比长序列多得多。这意味着模型在短上下文上积累了更多的训练经验，而在极长上下文上，它的参数可以说是相对"薄弱"的。

这些底层现实加在一起，就决定了一件事：上下文必须被当成稀缺资源来对待，而不是无限的容器。好的上下文工程，核心目标就是找到能让模型产生期望行为的、最小的、信号密度最高的 token 集合。

好，明白了这个底层逻辑，我们来看文章里提到的具体实践方法。文章从几个主要的上下文组成部分来逐一讲解。

首先是系统提示，system prompt。

文章说，系统提示要极度清晰，用简单直接的语言，以"合适的高度"来呈现信息。什么叫"合适的高度"？文章给了一个非常形象的描述：工程实践中有两种常见的失败模式，一种是高度太低，一种是高度太高。

高度太低的意思是，有些工程师喜欢把所有的业务逻辑都硬编码进系统提示里，把每一种可能的情况都写得非常细，用大量类似 if-else 的规则来穷举所有可能性。你可以想象一个产品经理把需求文档直接当成系统提示来用：第一步做什么，如果遇到情况 A 怎么处理，如果遇到情况 B 又怎么处理，如果 A 和 B 同时出现再怎么处理……这种提示词又长又脆。一旦业务逻辑有任何变化，整个提示词就要重写，而且因为规则之间很容易产生冲突，维护起来非常痛苦。

高度太高是另一个极端：有些工程师只给模型一些非常笼统、高层次的指导，比如"你是一个有用的助手，请尽力完成用户的任务"，完全没有给模型足够的具体信号来理解什么叫"有用"、什么叫"完成任务"。这种提示词会导致模型在执行任务时拿捏不准方向，表现出各种不稳定的行为。

合适的高度在中间：足够具体，能给模型清晰的行为导向；又足够灵活，给模型留有判断空间，不需要把每个边界情况都预先规定好。

文章的建议是：从一个尽量简洁的提示词开始，用能力最强的模型来测试，观察它在哪里会失败，然后有针对性地补充指令和示例。这个"最小可行提示词"的思路，和软件工程里的"最小可行产品"原则是一脉相承的。

其次是工具，tools。

在 agent 架构里，工具是模型与外部世界交互的接口。模型通过工具来调用外部 API、查询数据库、执行代码、浏览网页，等等。文章强调，工具的设计必须非常注重效率，包括两个维度：工具返回的信息要尽量精简，同时工具的设计要引导 agent 做出高效的行为。

有一个很常见的反模式：工具集过于臃肿。如果你给 agent 提供了几十个工具，其中很多功能有重叠，那每次需要行动的时候，agent 都要先从几十个工具里选一个。这种选择本身就是认知负担，而且很容易选错。

文章里有一句话说得非常好，我直接翻译给你听：如果一个人类工程师都没有办法明确地说清楚某种情况下应该用哪个工具，那你就不应该期望 AI agent 能做得更好。

所以工具集要精简，每个工具的职责要清晰，工具之间的功能边界要明确，不能有歧义。这跟设计一套好的代码库的原则完全一样：模块高内聚、低耦合。

第三个是示例，也就是 few-shot examples，少样本示例。

在提示词里加入示例，告诉模型"这是我期望的输入输出格式"——这个技巧大家都知道，文章也肯定了它至今仍然非常有效。但文章同时指出了一个非常常见的误用：很多工程师会把大量的边界情况都堆到示例里，试图用示例穷举所有可能的规则。

文章不推荐这样做。更好的做法是精心挑选一小组多样化的、典型的示例，这些示例要能够有效展示期望的行为模式。文章里有一句话：对于大语言模型来说，一个好的示例，相当于一千个字的规则说明。质量比数量更重要。

好，讲完了静态的上下文组成方式，我们来看一个非常有意思的动态策略，文章把它叫做 just-in-time context，我翻译成"恰时上下文"或者"即时检索"。

传统的做法是在 agent 开始执行任务之前，就把所有可能相关的信息都检索出来，放进上下文窗口里。这叫预计算检索，或者基于嵌入向量的预推理检索。它的问题是：你在任务开始之前根本不知道 agent 具体会需要哪些信息，所以你倾向于多塞一点，以防万一。结果就是上下文变得很臃肿，里面有大量实际上根本用不上的信息，白白消耗了宝贵的注意力预算。

just-in-time 的思路正好相反：agent 在运行过程中，只维护一些轻量级的"索引"——比如文件路径、数据库查询语句、网页链接——然后在真正需要某个信息的时候，才通过工具调用来把它加载进来。

文章举了 Claude Code 的例子，非常生动。Claude Code 在处理大型代码库或者数据库的时候，不会把整个代码库或者数据库的内容都倒进上下文里。它会先写一个精准的查询语句，只拿它需要的那些数据；或者用类似 head 和 tail 这样的系统命令，只看文件的开头和结尾，而不把整个文件都读进来。这样，上下文里装的永远都是当下最相关的信息，而不是一大堆背景噪音。

这个思路和我们人类的认知方式其实非常接近。你不会把公司所有的文档都背下来。你脑子里有一张索引地图，知道哪些文档在哪里，需要的时候去找就行了。文件系统、收件箱、书签收藏夹，本质上都是这种外部索引系统。

just-in-time 还有一个很有趣的副产品，叫做"渐进式披露"，英文是 progressive disclosure。意思是，agent 在探索环境的过程中，会像剥洋葱一样逐层发现和积累有用的信息。一个文件的名字就能告诉你它大概是干什么用的；一个文件的大小暗示了它的复杂程度；时间戳暗示了它的相关性。这些元数据本身就是有价值的信号，帮助 agent 做出更聪明的探索决策。

当然，这种策略也有代价：运行时的动态检索比预计算检索慢，因为每次都需要实时调用工具。文章说最理想的策略往往是混合的：有些上下文可以预先加载，有些则让 agent 在运行时按需检索。Claude Code 本身就是这个混合策略的典型案例——项目里的 CLAUDE.md 文件会在启动时直接放进上下文，而具体的代码文件则通过 glob 和 grep 等命令在运行时按需查找。

对于内容相对静态的业务场景，比如法律文书分析或者金融报告处理，混合策略可能更适合，因为预计算检索能显著提升速度。而对于代码编写这种高度动态的场景，就更依赖运行时探索。

现在我们来到文章里我认为最有实操价值的一部分：对于需要长时间运行的任务，怎么让 agent 在跨越多次推理的过程中保持连贯性。文章介绍了三种技术。

第一种叫做压缩，英文是 compaction。

想象这样一个场景：你的 agent 正在做一个复杂的代码重构任务，已经跑了半个小时，上下文窗口快满了。里面积累了大量的工具调用记录、中间结果、debug 过程……怎么办？

compaction 的思路是：先把当前的对话历史做一个高保真的摘要，然后用这个摘要重新启动一个新的上下文窗口，agent 从摘要的状态继续往下跑。

听起来好像很简单，但细节非常关键。压缩不能太激进——如果把太多细节都丢掉，agent 后续可能会丢失一些当时看起来不重要、但实际上很关键的背景信息。文章说，好的 compaction 是一门艺术，关键在于判断什么值得保留、什么可以丢弃。

在 Claude Code 里，compaction 的具体实现是这样的：当消息历史快撑满上下文窗口的时候，系统会把整段历史传给模型，让它做一次精炼的总结。总结的内容要保留架构决策、尚未解决的 bug、关键的实现细节；同时丢掉那些已经用过、不再需要的冗余工具调用结果和重复信息。最终，用这段压缩后的摘要加上最近访问的五个文件，构成新的上下文窗口，让 agent 继续工作。用户感受到的是任务的连续性，而不需要关心底层的上下文管理。

文章还提到了一种最轻量、最安全的 compaction 形式：工具结果清理。当一个工具在很早之前被调用过，它的原始返回结果已经用过了，不再需要，那就可以把那段原始结果从上下文里删掉，只保留从它那里提取出来的关键信息。这是一种"外科手术式"的清理，风险极低，但能有效减少上下文的体积。

文章给工程师的调优建议是：先从最大化召回率开始——也就是确保 compaction 提示词能够捕捉到所有重要信息，不遗漏；然后再迭代，提高精准度，去掉冗余内容。先求全，再求精。

第二种技术叫做结构化笔记，英文是 structured note-taking，也叫 agentic memory，智能体记忆。

这个思路是给 agent 加一个"外置记忆本"。具体做法是：agent 在执行任务的过程中，定期把重要的信息写到上下文窗口之外的持久化存储里——比如一个专门的 NOTES.md 文件。在后续需要这些信息的时候，再把笔记加载回上下文窗口。

这种方法实现了一种低开销的持久化记忆机制。agent 不需要把所有历史都保留在上下文窗口里，只需要在关键时刻把重要信息记下来，在需要时读回来。就好像一个人在做一个超级复杂的项目，脑子里装不下所有的细节，就用一个笔记本来辅助记录，定期翻阅。

文章里有一个非常精彩的案例，我推荐你细细品味一下：Anthropic 做了一个让 Claude 玩口袋妖怪游戏的实验。口袋妖怪这类游戏，需要角色在地图上移动、捕捉和培育精灵、与其他训练师对战，整个游戏过程可能横跨成千上万个步骤，需要在极长的时间维度上保持策略的连贯性。

这远远超出了任何单个上下文窗口所能容纳的范围。但通过结构化笔记，Claude 能够维护非常精确的状态记录，比如："在过去的一千两百三十四步里，我一直在一号公路上刷级，皮卡丘已经升到了八级，目标是十级"；或者："挑战道馆时，水系招式对岩石系精灵特别有效"；还有它自己绘制的已探索区域地图，记录哪些道路通向哪里、哪些物品在哪里可以获取。

这些信息完全靠上下文窗口是无法维持的。但通过结构化笔记，Claude 在每次上下文重置之后，都能读取自己之前写的笔记，无缝接续几个小时前的探索，完成那些需要非常长远策略规划的任务。

更有趣的是，文章说这套笔记系统完全是 Claude 自己发展出来的，没有人事先规定它应该记录什么格式的笔记。它根据任务的需要，自己摸索出了一套有效的信息记录方式。这说明，好的结构化笔记能力，是模型智能的一种自然涌现。

Anthropic 在 Sonnet 四点五版本发布的时候，同步在开发者平台上推出了一个记忆工具的公测版，本质上就是一个基于文件系统的外部存储接口，帮助 agent 在会话之外积累和查询知识库。

第三种技术叫做子智能体架构，英文是 sub-agent architectures，也叫多智能体架构。

这个思路是把一个大任务分解，交给多个专门的子智能体来并行处理。主智能体负责维护高层次的任务计划，把具体的子任务分配下去；每个子智能体在自己干净的上下文窗口里，专注地深入处理一个特定问题。

子智能体可能会用掉几万个甚至更多的 token 来做深度探索，但最终只需要把一个精炼的摘要——通常是一千到两千个 token——返回给主智能体。这样，主智能体的上下文窗口里装的永远都是高浓度的关键信息，不会被子智能体的探索细节所污染。

这种模式有一个非常清晰的优势：关注点分离。每个子智能体只专注于自己那块，主智能体只负责协调和整合。这和软件工程里的微服务架构、或者人类团队里的分工协作，背后的道理是一样的。

文章提到，Anthropic 在构建自己内部的多智能体研究系统时，这个模式相比单智能体方案取得了显著的性能提升。在需要大量并行探索的复杂研究任务上，多智能体架构的优势会特别明显。

三种技术各有适合的场景。compaction 最适合需要大量来回交互、对话式的任务，能够保持连续的对话流；结构化笔记适合有明确里程碑的迭代式开发工作，比如长期的代码项目；多智能体架构最适合需要大量并行探索的复杂研究和分析任务，能从分工协作中获益最大。

现在，我们来聊聊这些内容对你日常工作的实际意义。

如果你现在正在构建一个基于大模型的应用或者 agent，文章里有几个建议值得你记下来。

第一，从最简单的提示词开始。不要试图在第一天就把所有情况都考虑进去。先用最小可行的提示词测试，在最好的模型上先跑起来，然后根据观察到的具体失败模式，有针对性地补充指令和示例。每一条加进去的内容都要有它存在的理由。

第二，对你的工具集保持克制。宁可少几个工具，也要保证每个工具的职责清晰无歧义。如果你的工具集里有两个工具，你自己都搞不清楚什么时候该用哪个，那就先把它们合并成一个，或者把它们的边界定义得更清楚。

第三，如果你在做 RAG，也就是检索增强生成的系统，考虑引入 just-in-time 检索的思路。不要把所有可能相关的文档都预先塞进上下文，而是给 agent 提供检索工具，让它在需要的时候自己去查。这样既能保持上下文的精简，又能让 agent 按需获取最相关的信息。

第四，如果你的任务需要运行很长时间，要提前设计好上下文管理策略。是用 compaction、结构化笔记，还是多智能体架构，取决于你任务的具体特性。这些策略不是互斥的，可以组合使用。

有一个坑特别值得提醒：不要因为模型支持很大的上下文窗口，就觉得可以随意往里塞内容。上下文越长，模型召回信息的精准度越低，同时推理速度也越慢，成本也越高。这个代价是真实存在的，不要因为"装得下"就忽视它。

最后，我来做一个整体的总结和点评。

这篇文章提出的核心洞见非常清晰：在 agent 时代，工程师的核心竞争力不再仅仅是"怎么写好 prompt"，而是"怎么在每一次推理的时刻，给模型提供最有价值的信息组合"。上下文是大模型有限的注意力资源，要像管理内存、管理数据库索引一样，认真地工程化地去管理它。

我个人最欣赏这篇文章的地方是，它把一个原本很模糊的直觉——"要给模型提供好的上下文"——转化成了一个有结构、可操作的工程框架。从系统提示的写法，到工具的设计原则，到动态检索策略，再到长时间任务的三种管理技术，它给出了一条从浅到深的完整路径，不是空谈理念，是真正可以落地的方法论。

文章的最后一句话也很值得回味，它说：随着模型能力的提升，对上下文进行精心工程化管理——把它当成有限的、珍贵的资源来对待——这件事不会变得不重要，反而会始终是构建可靠、高效的 agent 的核心所在。

这句话我很认同。工具越强大，用好它的责任就越大。
