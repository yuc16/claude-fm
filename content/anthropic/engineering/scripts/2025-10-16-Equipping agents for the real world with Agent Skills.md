但具体的工具接口、配置格式、平台支持细节，可能已经有相当多的迭代。所以我今天讲的重点，会放在这篇文章的设计理念和工程思想上，这些内容的生命周期远比某个具体的 API 参数要长。

好，我们先来搭建一下背景。

二零二五年下半年，是 AI agent——也就是 AI 智能体——大规模进入实用阶段的关键时期。如果说二零二三年和二零二四年，整个行业还在讨论大语言模型的基础能力够不够强、推理能力是不是真的靠谱，那到了二零二五年，这些问题基本有了答案，焦点也随之转移了。行业开始面对一个更加实际的工程问题：我们手上有了一个通用的、聪明的 AI 智能体，现在怎么让它在具体的行业场景里真正好用起来？

这是个一点都不简单的问题。通用智能体的优势是灵活，能举一反三，适应各种场景。但它有一个天然的局限：它不了解你们公司的内部系统是怎么组织的，不知道你们行业特有的术语和规范，也不熟悉你们团队特定的工作流程。它是一个极其聪明的通才，但不是你那个特定领域的专家。

要把这个通才变成真正好用的专业工具，你就需要想办法把领域知识传递给它。工程师们为此探索出了几种解法，每种都有各自的优缺点。

第一种方法是在系统提示词里写很长的专业说明，把所有的背景知识都塞进去。这最直接，立竿见影。但问题是，每次对话都要把这些内容全量加载到 context 里，不管当前任务用不用得到，都得占着 token。而且，一旦系统提示词过长，模型的表现往往会变得不稳定，它可能会遗忘提示词后段的内容，或者被早期的指令过度约束。

第二种方法是针对特定领域专门训练一个微调模型。这个方法的效果理论上最好，因为知识直接烧进了权重里。但问题是成本极高，开发周期长，而且每次基础模型升级，微调模型就得重新走一遍流程。更麻烦的是，微调的过程需要大量标注好的训练数据，在很多垂直领域根本凑不齐。

第三种方法是用工程化的架构来动态注入知识，比如 RAG，也就是检索增强生成，在每次对话时从知识库里查询相关内容然后注入进来。这个方法确实解决了 token 效率问题，但实现起来需要大量的定制开发，一个项目做完下一个又要从头来，难以形成可复用的模式。

这些方法都没有错，但都不够优雅，也不够可扩展。它们有一个共同的问题：每次你想让 agent 具备一个新的领域能力，你要么消耗大量 token，要么花大量工程时间，要么两者都有。Anthropic 的工程师们在这个问题上思考了很久，最终提出了 Agent Skills，也就是"智能体技能"这个解决方案。

那 Agent Skills 到底是什么呢？文章里有一句比喻我觉得说得非常好：做一个 skill，就像给新入职的员工写一份专项的入职手册。

我来把这个比喻展开，因为它很值得细品。假设你公司来了一个极其聪明的新员工，学历顶尖，学习能力超强，一般的技术问题他都能很快找到答案。但他第一天上班，有些事情再聪明也无法直接知道：你们公司的代码仓库是怎么组织的，内部的服务叫什么名字，那个看起来很奇怪的缩写代表什么，季度结算的流程是先做哪一步再做哪一步，哪些外部系统调用需要特别的认证方式，哪些操作在生产环境里是绝对不能碰的。

这些东西不是靠聪明就能猜出来的。它们是你们公司特有的隐性知识，是多年积累下来的约定和规范。以前这种知识的传递方式，要么靠老员工言传身教，要么散落在各种内部文档里，新人找起来很费劲，而且每来一个新人，这个传授过程就要重演一遍。

Agent Skills 的思路就是：把这些专业知识系统化地整理成一份结构化的手册。这个手册是可以复用的，可以版本控制的，可以在团队之间分享的，可以随着业务变化持续更新的。任何时候给 Claude 安装这个 skill，它就能快速获得这些领域知识，不需要重复的手动传授，也不需要每次都在系统提示词里重新把这些背景铺一遍。

从技术格式上来说，一个 skill 就是一个普通的文件夹，里面有一个核心文件叫做 SKILL.md。这个文件的开头必须有一段 YAML 格式的元数据，至少要包含两个字段：name，也就是这个 skill 的名称；还有 description，也就是这个 skill 的功能描述。就这么简单，这就是一个最小化的 skill。

你可能会问，就这么简单？对，就这么简单。但这个看似平淡的格式背后，撑着一个非常聪明的设计理念，文章把它叫做 progressive disclosure，我把它翻译成"渐进式信息披露"。这是整篇文章里我认为最值得深挖的核心设计思想，理解了它，你就真正理解了 Agent Skills 的灵魂所在。

渐进式信息披露的思路是：不要一次性把所有信息都扔给 agent，而是按需给信息，用多少拿多少。文章把这个过程分成了三个层次。

第一个层次，就是那段 YAML 元数据里的 name 和 description。当 Claude 启动的时候，它会把所有已经安装的 skill 的 name 和 description 预先加载到自己的系统提示词里。注意，仅仅是这两个字段，不是 skill 的全部内容。这就好像你给了 Claude 一份目录页，告诉它"你手边有这些专业工具可以调用，每个工具大概能做什么事"，但每个工具的具体操作说明书先不给他看。

这一层的关键作用是：让 Claude 知道边界，知道什么时候该考虑用哪个 skill，而不需要承担加载所有 skill 完整内容的 token 成本。想象一下，如果你安装了十五个 skill，而且每次对话都强制把这十五个 skill 的全部内容都塞进 context，那 context window 还没开始处理用户的实际任务，就已经被各种背景文档占去大半了。第一层的元数据设计，巧妙地避免了这个问题。

第二个层次，是 SKILL.md 文件的主体内容。当 Claude 根据 name 和 description 判断，当前用户的任务确实与某个 skill 相关的时候，它会主动去读取这个 skill 的完整 SKILL.md 文件，获取详细的操作说明和工作流程。这个时候 Claude 才真正"翻开手册"，了解具体该怎么做。

这个触发动作是由 Claude 自主决策的，不是系统强制加载的。Claude 会分析用户消息的意图，结合 skill 的 description，判断当前任务是否需要这个 skill 的帮助，然后决定要不要去读取完整内容。这个决策能力，正是大语言模型在语义理解上的优势所在。

第三个层次，是 skill 文件夹里包含的额外子文件。对于比较复杂的 skill，主体的 SKILL.md 里会包含核心的工作流程，同时引用其他文件来处理特定的子场景。这些额外的文件，Claude 只会在真正需要的时候才去读取，不会默认全部加载。

文章里给出了一个非常具体的例子来说明这三层结构，就是 PDF skill。这是 Anthropic 用来给 Claude 增加 PDF 文件操作能力的一个 skill，特别是处理填写 PDF 表单这个需求。

在第一层，PDF skill 的 description 会告诉 Claude 大概是"处理 PDF 文件、提取内容、填写表单"之类的内容，足够让 Claude 在用户提出 PDF 相关需求时想到要调用这个 skill。

在第二层，完整的 SKILL.md 里包含 PDF 操作的核心说明：支持哪些类型的操作、基本的工作流程是什么、有哪些常见的注意事项。同时，这个 SKILL.md 里还引用了两个额外的文件：一个叫 reference.md，是 PDF 格式规范的参考资料；另一个叫 forms.md，是专门关于如何填写 PDF 表单字段的详细说明。

这就是第三层的按需加载。当用户的任务是"帮我把这个 PDF 里的文字提取出来"，Claude 读完 SKILL.md 就够了，完全没有必要再去读 forms.md，因为这个任务和表单填写毫无关系。但如果用户的任务是"帮我把这份 PDF 申请表填好"，Claude 就会意识到需要读取 forms.md，获取表单填写的详细操作说明，然后再开始工作。

这个拆分设计的价值在于：它精确地匹配了信息的使用频率和任务相关性。表单填写是一个相对特殊的子任务，不是每次 PDF 操作都用得到。把这部分说明单独放在子文件里，可以确保大多数 PDF 操作任务不会因为加载一堆用不上的表单说明而浪费 token，让 Claude 的 context 始终保持干净和高效。

文章里有一个类比我觉得非常贴切：就像一本组织良好的手册，先有目录，再有各个章节，最后还有详细的附录。你查资料的时候，先看目录定位大方向，再翻到对应的章节找具体内容，真有特别细的技术问题才去查附录。你不会一打开手册就从第一页往后顺着读，那效率太低了。

这个设计有一个非常重要的推论：由于 agent 可以随时按需读取文件，不需要一次性把所有内容塞进 context，所以理论上打包在一个 skill 里的信息量是没有上限的。你可以把一个 skill 做得非常详尽，包含几十个子文档，覆盖几百种操作场景，只要文件组织结构合理，Claude 都能在需要的时候找到对应的内容，而不会因为信息量太大而撑爆 context。这对于需要大量专业知识的企业级应用场景来说，是一个非常重要的能力。

好，我们来看 Agent Skills 的第二个重要能力：在 skill 里打包可执行代码。

大语言模型非常擅长语言理解、推理和生成，但有一类任务它天生就比不上传统程序代码：确定性操作。文章里举了一个非常经典的例子：让模型对一个列表排序。如果你让 Claude 直接通过 token 生成来输出排好序的列表，它需要一个 token 一个 token 地生成，既慢又贵，而且在列表很长的时候还可能出错；但如果你让它运行一段排序算法的代码，计算机处理这件事是纳秒级别的，而且结果每次都完全一致。

确定性，是很多工程应用里必须满足的基本要求。在生产系统里，你不能接受同样的输入偶尔给出不一样的结果。比如，解析特定格式的文件结构、对数据做精确的数值计算、按照固定规则格式化输出内容、调用需要严格签名验证的 API、生成需要精确对齐字段的结构化数据……这些任务，代码比语言模型更可靠，更高效，更可重复。

在 PDF skill 的例子里，就打包了一段预先写好的 Python 脚本，这段脚本的功能是：读取一个 PDF 文件，然后把里面所有的表单字段都提取出来，包括每个字段的名称、字段类型，以及当前已填写的内容。Claude 在需要的时候可以直接执行这个脚本，不需要把 PDF 的全部原始内容或者脚本代码本身都加载进 context，就能得到一份结构化的字段清单，然后决定下一步怎么操作。

你可以把这理解成：Claude 是一个经验丰富的工程师，手边有一套专用工具箱。有些问题，他用大脑思考推理就能搞定；但有些问题，他会直接从工具箱里拿出对应的工具来做，更快、更准、更可靠。Skill 里打包的代码，就是 Claude 工具箱里这些专用工具。

对于工程师来说，这意味着什么呢？它意味着，你在多年实践中积累的那些专用脚本、自动化工具、数据处理程序、格式转换工具，可以直接打包进 skill，让 Claude 在合适的场景下自动调用它们。你不需要重新教 Claude 怎么实现这些功能的算法细节，你只需要把现成的代码放进 skill，再配上清晰的说明，告诉 Claude 什么情况下应该运行这段代码就够了。这是一种很自然的人机协作分工：创意性的、语义性的工作由 Claude 主导，确定性的、程序性的操作由代码承担。

接下来，我们来聊构建 skill 的最佳实践。这是文章里我认为最有实操价值的部分，文章给出了四条核心建议，我来一条一条展开讲，因为每一条背后都有清晰的工程逻辑。

第一条建议是：从评估开始，不要从想象开始。

很多工程师的直觉是：我知道 Claude 在某个领域不够好，所以我来写一个 skill 来增强它。这个出发点没有错，但问题在于，你以为 Claude 需要什么，和它实际上需要什么，往往差距很大。

文章的建议是先拿一批真实的任务让 agent 跑起来，观察它在哪些地方卡住了，哪些地方需要用户反复补充信息，哪些地方频繁出错，哪些地方走出了意想不到的弯路。然后，针对这些具体的痛点，逐步构建 skill 来填补缺口。

这个建议的背后是一个重要的认知：context 的需求极难靠提前想象来准确预测。只有通过实际的运行观察，你才能发现那些出乎意料的信息盲区。那些在你脑子里显而易见的背景知识，Claude 可能压根不知道；而你以为 Claude 需要大量补充说明的技术原理，它实际上可能已经掌握得很好。

举一个具体场景：你以为 Claude 需要更多的技术文档来解释你们系统的架构，所以你在 skill 里写了很多关于系统设计的说明。但实际跑起来你发现，Claude 真正卡壳的地方，是你们公司内部特有的命名约定——比如某个字段在不同系统里叫不同的名字，或者某个操作流程里有一个从来没被文档化过的隐式前提条件。如果你没有先跑一遍任务来观察，这个误差可能永远不会被发现，skill 写了半天，解决的不是真正的问题。

第二条建议是：为扩展性设计结构。

随着你的 skill 越做越完善，SKILL.md 会越来越长。当它开始变得臃肿、难以一眼看清结构的时候，这是一个信号：应该把内容拆分了。

拆分的原则很明确：如果某些内容只在特定的子场景下才用得到，就把它单独放到子文件里，通过 SKILL.md 里的引用来组织它们。如果某两类任务通常是互斥的，在同一次任务里几乎不会同时用到，就给它们各自建立独立的文件，而不是把两种场景的说明都堆在 SKILL.md 里。

还有一点文章特别提到：代码在 skill 里可以有两种完全不同的用途，一种是供 Claude 直接执行的工具脚本，另一种是供 Claude 阅读参考的示例代码，用来理解某种操作的逻辑或格式。这两种用途要在 skill 的说明里写得清清楚楚，不然 Claude 可能搞混，把应该读的代码去执行了，或者把应该执行的脚本只当文档阅读了一遍就算了。

第三条建议是：从 Claude 的视角来思考。

这一点是最容易被忽略，但往往也是最重要的一条。skill 的 name 和 description，是 Claude 在每次对话开始时，决定"要不要调用这个 skill"的主要依据。如果这两个字段写得含糊，或者和另一个 skill 的描述高度相似，Claude 的触发判断就会出现偏差。

比如你安装了两个 skill，一个叫"数据处理"，另一个叫"数据分析"，两个 skill 的 description 也写得差不多。Claude 在面对一个数据相关的任务时，可能在该用分析 skill 的时候触发了处理 skill，或者反过来，或者两个都触发了，造成混乱。更糟糕的情况是，它根本不知道该用哪个，干脆两个都不用，完全靠自己推理，结果当然比有 skill 辅助时差很多。

所以，在真实任务里密切观察 Claude 是怎么使用你的 skill 的，是开发 skill 过程中必不可少的一步。要看它有没有在不该触发的时候触发了，有没有在该触发的时候没有触发，有没有读取了一些它实际用不上的子文件，有没有在某些子文档上过度依赖。根据这些观察来不断调整 name、description，以及文件的组织方式。

第四条建议是：和 Claude 一起迭代，而不是自己闷头写。

这是文章里我最喜欢的一条建议，因为它非常有实践智慧，也和前面那几条建议形成了一个完整的方法论闭环。

具体的做法是：在你用 Claude 处理一个实际任务的过程中，如果 Claude 走出了一条非常顺畅的解题路径，你可以在任务结束后直接让它把这个成功的经验整理成可以复用的 skill 内容；如果它在某个环节走偏了，就让它做自我反思，分析哪个步骤出了问题，应该在 skill 里增加什么样的说明来避免类似的偏差在未来重复发生。

这种迭代方式有一个很大的优势：你不需要在写 skill 之前就想清楚所有的内容应该是什么。你可以通过实际的任务协作，逐渐发现 Claude 真正需要哪些信息和工具，然后把这些内容沉淀到 skill 里。这比凭空想象要可靠得多，而且能发现很多你自己都没意识到的隐性知识。

让我给你们描述一个具体的应用场景。假设你们团队里有一位非常资深的后端工程师，他在处理某一类线上故障排查上有一套自己的方法，效率极高，准确率也高。但这套方法从来没有被系统地文档化，都存在他的脑子里，靠着多年的经验积累出来的。你可以让他和 Claude 一起处理几个这类故障排查任务，在过程中让 Claude 把他用到的排查步骤、判断逻辑、常见陷阱、关键指标都记录下来，然后整理成一个故障排查的 skill。以后不管是 Claude 自己处理类似问题，还是你们团队的新工程师借助这个 skill 来学习，都能享受到这位资深工程师多年的经验积累。这是一种非常有价值的组织知识管理方式，把过去只能靠"带徒弟"传承的隐性知识，转化成了可以被系统使用的显性知识资产。

好了，讲完了最佳实践，我们来聊安全问题。这是任何新技术都绕不开的话题，在 skill 这里尤其不能跳过，因为它涉及到可执行代码，潜在的安全风险比普通的提示词工程要高一个量级。

文章特别指出：skill 给 Claude 提供了指令和可执行代码，这让它变得非常强大，但也带来了相应的安全风险。一个被恶意构建的 skill，可能会指示 Claude 去做一些有害的事情，比如悄悄地把你系统里的敏感数据发送到外部服务器，或者在你的本地环境里执行一些你从来没有授权过的操作，或者引入带有恶意逻辑的代码依赖。

这个风险的本质，和软件供应链安全是同一类问题，只是载体不同。你安装了一个第三方的 npm 包，如果这个包里藏着恶意代码，你的系统就可能被攻击。skill 本质上也是一种"可以影响 agent 行为的外来内容"，如果它来自一个不可信的来源，你就有被利用的风险，而且风险不仅限于代码执行，还包括指令级别的操控，比如让 Claude 改变它处理信息的方式，或者绕过你原本设置的安全限制。

文章的安全建议是：优先只安装来自可信来源的 skill，比如官方发布的 skill、你自己构建的 skill，或者是你充分信任的团队成员构建的 skill。如果必须使用一个来源不那么明确的第三方 skill，在安装之前要做认真的代码审查。

审查时要特别注意几个维度。第一是代码里引入的外部依赖，仔细看有没有引入可疑的第三方库，这些库的行为是否透明可验证。第二是 skill 文件夹里打包的资源文件，比如图片、数据文件、脚本，逐一检查它们的实际内容，而不是只看文件名就信任了。第三是有没有任何指令让 Claude 去连接外部的、你没有显式授权的网络地址。第四是有没有指令让 Claude 把你的本地文件内容、对话记录或者系统信息发送到外部目标。

我个人认为，随着 skill 生态的持续成熟，评估一个 skill 是否安全，会成为每个使用者都需要具备的基本素养，就像我们现在评估 npm 包、Docker 镜像或者 pip 包的安全性一样。目前可能还没有成熟的自动化工具来做 skill 安全扫描，但这个需求会越来越强烈，相关的工具和规范也一定会随之出现。

好，我们来聊聊这篇文章提到的未来方向，以及 Agent Skills 对我们今天实际工作的价值。

文章发表时，Agent Skills 在 Claude.ai 的网页端、Claude Code 命令行工具、Claude Agent SDK，以及 Claude Developer Platform 上都已经可以使用了。两个月后，Anthropic 把它作为开放标准正式发布，这意味着它不只是 Anthropic 自己的产品功能，而是一个任何平台都可以采用的通用规范。如果这个标准得到广泛支持，你在一个平台上构建的 skill，理论上可以在其他支持这个标准的平台上直接使用，这对 skill 的复用价值是一个很大的提升。

文章提到了两个非常有前景的未来发展方向。

第一个是与 MCP，也就是 Model Context Protocol，模型上下文协议，的深度融合。MCP 是 Anthropic 推出的一个开放标准，用来让 AI 模型连接各种外部工具和数据源，比如数据库连接、文件系统访问、各种 SaaS 服务的 API。文章预期，skill 和 MCP 会形成一种很自然的互补关系：MCP 解决"如何连接外部工具"的问题，而 skill 解决"如何教会 agent 用这些工具来完成复杂的多步骤工作流程"的问题。

这两者的结合，可以类比成程序员工具箱和操作手册的关系：MCP 提供了工具，skill 提供了用工具的方法。只有工具没有手册，agent 可能不知道应该按什么顺序调用哪些工具；只有手册没有工具，agent 想做的事情做不了。两者配合，才能让 agent 在复杂的工作场景里真正有效运转。

第二个展望更加大胆：让 agent 自己来创建、编辑和评估 skill。也就是说，agent 在完成一个任务的过程中，如果发现自己走出了一条特别高效的解题路径，它可以主动把这套方法沉淀成一个新的 skill，供自己以后使用，也可以分享给其他的 agent 或者用户。

这是一个在概念上非常有想象力的方向。它描述的是一种 agent 自我成长的机制：不只是完成任务，而是在完成任务的过程中不断提升自己的能力。当然，这里面有很多工程上的挑战需要解决，比如如何确保自动生成的 skill 质量足够稳定、如何防止 agent 把错误的经验也固化进 skill、如何做 skill 的自动审查和版本管理……这些问题都不简单。但这个方向本身，代表了 AI agent 从"执行工具"走向"学习主体"的一种演化，值得我们持续关注。

好，最后来讲讲这篇文章对我们日常工作最直接的意义。

如果你现在正在做基于 Claude 的 AI 应用开发，最直接的价值是：你可以把你自己或者团队积累的专业知识，系统化地整理成 skill 包，让你的 agent 快速具备这些领域能力。而且因为 skill 是一个普通的文件夹，没有什么特殊的技术门槛，任何工程师都可以上手构建，不需要专门的 AI 工程背景。

比如你们公司是做电商数据分析的，可以把内部的数据字段定义、指标计算规则、常用分析维度、业务术语表，整理成一个数据分析 skill；以后每次部署一个新的 agent 来处理数据分析任务，直接安装这个 skill，它就能在你们特定的数据环境里有效工作。这个过程比每次都在系统提示词里重复写这些背景，要高效得多，也更容易维护和更新。

如果你在一个比较大的组织里工作，skill 还有一个组织层面的价值值得重视：它可以把不同部门、不同专家的知识，变成可以共享、可以版本控制、可以持续演化的数字资产。就像代码仓库管理代码一样，skill 库可以成为组织的知识资产库，让经验不再只存在某几个人的脑子里，而是成为整个组织可以调用的能力。

如果你是个人开发者，最简单的起点是：观察你在日常使用 Claude 的过程中，有哪些背景信息是你每次都要手动提供的，然后把这些内容整理成一个最简单的 SKILL.md 文件，放进一个文件夹里，安装到你的 agent 上试试看。不需要很复杂，从一个能用的最小版本开始，然后根据实际使用中的观察慢慢迭代。

在此也提醒一下几个常见的坑。第一个坑是把 SKILL.md 写得太长太全，什么都往里塞，结果破坏了渐进式信息披露的设计初衷，让 Claude 每次都要把一大堆可能用不上的内容全部读进来。正确的做法是保持核心文件精简，把细节放在有针对性的子文件里。第二个坑是 name 和 description 写得不够精准，导致 Claude 的触发判断不稳定，一定要在真实任务里多测试几次，确认触发行为符合预期。第三个坑是对打包在 skill 里的代码缺乏安全边界意识，在部署到生产环境之前，要认真考虑这些脚本在不同权限和环境下运行的影响范围。

好，我们来做一个收尾。

这篇文章的核心贡献，是提出了一种轻量级、可组合、可复用的方式来给通用 agent 增加领域专业能力。Agent Skills 的设计简洁而实用，特别是渐进式信息披露这个机制，非常优雅地解决了"灵活性"和"效率"之间的矛盾，用最小的格式复杂度，实现了理论上无上限的信息承载能力。

我个人认为，这篇文章指向的方向，代表了 AI 工程实践里一个重要的认知转变：从"用提示词工程来定制 agent"，走向"用知识工程来扩展 agent"。背后的核心思想是：模型的通用智能是一个基础能力层，而领域知识是一个独立的维度，把这两者解耦，然后按需组合，比把一切都揉在一起要更容易扩展，也更容易维护，更容易在组织内部传承和共享。

这篇文章发表至今已经近八个月，产品细节上可能有不少更新，但它的设计理念和工程方法论，在相当长的时间里都会有参考价值。如果今天这期节目让你对 Agent Skills 产生了兴趣，建议去查一下 Anthropic 的最新文档，看看当前版本有什么新的进展和变化。
