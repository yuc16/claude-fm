好，我们开始今天的节目。

今天要聊的这篇文章，英文标题是"An update on recent Claude Code quality reports"，我把它翻译成"关于近期 Claude Code 质量问题的更新报告"。这篇文章发表于二零二六年四月二十三日，作者是 Anthropic 的工程团队。

距今大约一个半月，这件事还是很新鲜的。为什么值得认真聊？这不是那种发布新技术、新突破的文章。它是一份工程事故的公开复盘——一个 AI 公司在面向所有用户承认自己搞砸了什么，为什么搞砸，怎么修复的，以及下次怎么避免。我认为这种文章反而比那些光鲜亮丽的发布公告更有价值，因为它展示的是一个真实的工程团队在管理一个复杂 AI 系统时，会踩哪些坑，会做出哪些错误的权衡，又是如何发现和纠正的。对于我们这些每天使用 AI 工具、或者正在构建 AI 产品的工程师来说，理解这些工具的失效模式，比仅仅会用它们要重要得多。

在讲具体发生了什么之前，我们先交代一下背景。

Claude Code 是 Anthropic 发布的一款命令行工具，允许开发者在终端里直接和 Claude 对话，用来写代码、调试程序、执行复杂的工程任务。你可以把它想象成一个住在你终端里的 AI 工程师：你告诉他目标，他自己想办法调用各种工具，读文件、写代码、跑测试、分析报错，多步骤地推进任务。这类能够自主执行多步骤任务的 AI 系统，通常被称为 agent，就是智能体的意思。

Claude Code 的定位一直是走高质量、高推理能力的路线。核心卖点就是 Claude 模型在复杂代码理解和生成任务上比竞品更强。所以当用户开始大规模反映 Claude Code 感觉变笨了，这对 Anthropic 来说是非常严重的信号——这直接打在他们最核心的价值主张上。

从今年三月开始，陆陆续续有用户在反馈说 Claude Code 的回答质量下降了。有人说模型感觉变差了，有人说 Claude 开始忘事，有人说代码质量不如以前，还有人说使用额度消耗得比预期快很多，但说不清是不是自己用多了。这种担忧在 AI 社区其实很普遍——大家都害怕厂商为了省成本会悄悄替换更弱的模型，或者在后台做了某种影响性能的优化，而用户几乎没有办法从外部验证这件事。

Anthropic 的第一反应是：我们没有故意降级模型，API 也就是应用编程接口那一层完全没有受到影响。但经过仔细调查，他们确实发现了三个相互独立的问题，每个问题影响了不同场景下的不同用户，合在一起就造成了那种广泛的、模糊的整体质量下降感。所有三个问题都在四月二十日这个版本，也就是二点一点一一六版本里被解决了。

这三个问题，我们一个一个来拆解。

第一个问题，关于推理努力级别的权衡失误。

先解释一个概念：推理努力级别，英文叫 reasoning effort，是 Anthropic API 里的一个参数，用来控制模型在给出回答之前"想多久"。想象你在解一道数学题，你可以扫一眼给个直觉答案，也可以认认真真列式一步步推导。Claude 的推理努力级别就是控制它更接近哪种方式。等级越高，思考越深入，回答质量越好，但速度越慢，消耗的计算资源也越多——在 AI 语境里这个计算资源就体现为 token 消耗。token 可以理解成 AI 处理文字信息的计量单位，模型"思考"本身也是要消耗 token 的。

Claude Code 里有几个档位：medium 是中等，high 是高，xhigh 是超高。当时 Anthropic 刚发布了 Opus 四点六这个模型，Claude Code 里的默认推理努力级别设置的是 high。

问题来了：在 high 模式下，有些用户遇到了极端的长尾延迟。所谓长尾延迟，就是大多数时候响应速度还可以，但偶尔会出现超级漫长的等待，长到界面看起来像卡死了一样。这种情况虽然发生概率不高，但一旦发生用户体验就非常糟糕，而且会消耗大量使用额度。

二零二六年三月四日，Anthropic 做了一个决定：把默认推理努力级别从 high 降到 medium。他们的逻辑是：内部评估测试显示，medium 在大多数任务上只比 high 差一点点，但延迟显著更低，不会出现那种极端的长尾延迟，同时还能帮用户把有限的使用额度用得更久。听起来好像是一个合理的工程权衡，而且他们也在产品界面里加了弹窗说明，告诉用户默认设置发生了变化，可以手动调整。

但用户很快就给出了否定的反馈：Claude Code 感觉变笨了。Anthropic 也做了几次补救措施，比如在界面上加提醒、增加内联的努力级别选择器、把"ultrathink"这个手动触发超高推理的命令重新带回来。但绝大多数用户都保持了 medium 这个默认值没有改动，因为默认就是默认，大多数人不会主动去翻设置页面改一个他们可能都没意识到已经变了的选项。

四月七日，Anthropic 承认了这个权衡做错了，撤销了这个决定。现在的新默认是：Opus 四点七用 xhigh，也就是超高，其他所有模型用 high。

这件事背后有一个很值得思考的工程决策问题：你的评估测试说 medium 和 high 差不多，但用户说有差距，你该信谁？这不是一个简单的问题，因为评估测试的数据是量化的，用户反馈是主观的，而且用户有时候对"差别"的感知会受到预期的影响。但这件事的结局说明，当你的评估测试和真实用户感受出现矛盾时，用户感受很可能在揭示评估测试的盲区。

这个盲区在哪里呢？可能是因为内部的评估题目集，英文叫 benchmark，和真实用户遇到的任务之间存在分布差异。benchmark 里的题目相对标准化，而用户实际使用时遇到的是各种各样的复杂、边缘、非典型场景。在这些场景里，high 和 medium 的差距就被放大了。这是一个在 AI 产品里几乎无处不在的问题：benchmark 上的表现和真实用例上的表现之间存在 gap，而 benchmark 无论设计得多么精心，都不可能完全覆盖真实世界的分布。

还有一个从行为心理学角度来看的解释：用户对 AI 助手能力的感知有一种不对称性。速度慢一些是可以忍受的，你可以等，可以理解。但感觉"变笨了"在心理上是不可接受的，哪怕客观测量上差距并不大。主观感受的差距可以比客观测量的差距大很多倍。这个心理机制对我们设计任何面向用户的 AI 产品都有重要启示：在用户感知和测量指标出现分歧的时候，要非常谨慎地检查是不是测量工具本身有问题。

第二个问题，思维历史清除的 bug。这是三个问题里技术上最有意思的一个，也是影响最隐蔽、最难复现的一个。

我们先来解释几个背景概念。Claude 有一种能力叫扩展思考，英文叫 extended thinking，就是在给出最终回答之前，模型会先进行一段内部的推理过程，就像你解一道复杂的题目会先在草稿纸上推算一样。这些推理内容不是直接呈现给用户的，而是 Claude 自己内部的"打草稿"过程。

重要的是，这些推理内容会被保存在对话历史里。这样在后续的对话中，Claude 能看到自己之前为什么做了某个决定、为什么调用了某个工具、为什么选择了某个方向。你可以把它想象成一个工程师在做复杂任务时随手记的工作笔记。每次拿起这个任务，先翻翻之前的笔记，才知道上次做到哪一步、为什么这样设计、当时考虑了哪些约束。失去了这些笔记，即使手头的代码还在，工程师也很难无缝接续。Claude 失去推理历史也是类似的处境。

Claude Code 还用到了一种叫提示词缓存的技术，英文叫 prompt caching。原理是：在连续的多次 API 调用之间，如果输入内容基本一样，就把这部分内容存起来复用，不需要每次都重新传送。这样可以显著降低 API 调用的成本和延迟。

但有一个自然的问题：如果用户离开了很久，比如超过一个小时，之前的缓存就过期了。用户回来续上对话，系统需要把所有对话历史，包括大量积累的推理内容，全部重新发给 API。这些未被缓存的内容会产生相当大的 token 消耗，让恢复会话变得既慢又贵。

所以三月二十六日，Anthropic 决定做一个效率优化：如果一个会话空闲超过一个小时，在用户重新开始对话时，清除掉历史里老旧的推理内容，只保留最近的一条。反正缓存已经失效了，清掉那些旧推理记录可以大幅减少需要重新传送的内容量，降低恢复会话的成本。清除这一次之后，就恢复正常，后续的推理内容继续正常积累。

这个设计逻辑本身没有问题，是合理的工程优化。但实现上出了 bug。本来只应该"清除一次"的操作，变成了"每一轮对话都清除"。一旦一个会话越过了那个空闲一小时的阈值，从那个时刻开始，后续的每一轮对话都会执行清除操作，把之前积累的所有推理内容全部丢掉，只留下最新的一条。

效果是什么？每次 Claude 回复完，推理历史就被清空了。下一轮她只能看到自己最近想了什么，之前所有"为什么"都消失了。随着对话继续进行，Claude 在执行任务，但她越来越不记得自己为什么要这样执行这个任务。

还有一个更糟糕的叠加效应。在 Claude Code 里，Claude 经常在执行任务过程中调用各种工具：读文件、写代码、跑命令、分析输出。如果你在 Claude 正在执行工具调用的过程中发了一条消息，这会触发一个新的对话轮次。而在这个新轮次里，就连当前这一轮刚产生的、还很新鲜的推理内容也被清掉了。所以 Claude 在任务执行到一半的时候，突然失去了刚才决定要这样做的记忆，她只知道手头在做什么，不知道为什么在做。

用户感受到的就是：Claude 开始忘事了，会重复已经完成的操作，会做出莫名其妙的工具调用选择，会在同一个地方一直转圈。很多用户还反映使用额度消耗得比预期快很多，这也是这个 bug 的副作用：因为每次清除推理历史之后，缓存就失效了，每个请求都变成了缓存未命中，需要重新传送大量 token，额度当然消耗得更快。

这个 bug 是四月十日才被修复的，距离它被引入过了整整两周。为什么这么久才发现？文章里给出了几个原因，每一个都很有代表性。

第一，这个 bug 只在特定条件下触发——会话必须先空闲超过一小时。内部测试基本上是连续运行的，不会遇到这个阈值，所以测试从来没有覆盖到这个场景。

第二，公司内部同时有两个不相关的实验在并行运行。其中一个改变了内部构建版本里思考内容的显示方式，恰好把这个 bug 的外在表现遮蔽掉了。工程师们用内部版本测试，触发不了这个 bug；外部用户用公开构建版本，踩到了这个 bug。内部测试环境和外部真实使用环境之间的差异，让这个问题长期没有被内部的测试流程捕捉到。

第三，这个问题涉及三个不同层面的交叉：Claude Code 自己的上下文管理逻辑、Anthropic 的 API 层、以及扩展思考功能。代码改动通过了多轮人工代码审查、单元测试、端到端测试、自动化验证，还有内部员工的日常使用，全都没有发现问题。这说明多层次的测试和审查不是万能的，特别是当 bug 藏在不同系统层级的交叉点上时，每一层都觉得自己那一侧是对的。

文章里还提到了一个特别有意思的后续：他们事后用 Opus 四点七的代码审查功能来重新审查当时那个出问题的代码提交。当提供了足够多的相关代码库作为上下文之后，Opus 四点七找到了那个 bug；而同样的任务，Opus 四点六没有找到。

这说明什么？说明更新的、更强的模型在代码审查这个具体任务上有实质性的提升，而且这种提升不是抽象的，是在真实 bug 发现这件事上可以量化的。也说明给 AI 代码审查工具提供足够多的相关代码上下文，对找到深层次的跨文件 bug 至关重要。基于这个发现，Anthropic 计划把多仓库上下文能力正式集成进代码审查工具，这个改进后的版本也会提供给用户使用。

第三个问题，减少冗长度的系统提示词变化。

背景是：Anthropic 的最新模型 Opus 四点七有一个非常显著的特点，它非常话多。Anthropic 在这个模型的发布说明里就提到了这一点，说它在处理困难问题时倾向于产生更长的输出，这种冗长和它更强的推理能力有内在联系——话多可能就是想得深的外在表现。

但话多也带来了实际问题：每次输出的 token 更多，用户的使用额度消耗更快，用户阅读起来也更费力。对于要在任务过程中频繁调用工具的 agent 来说，工具调用之间的废话如果太多，体验也很差。

在发布 Opus 四点七之前的几周，Anthropic 就开始针对这个新模型优化 Claude Code 的各种配置。他们用了几种手段来控制冗长度：模型训练层面的调整、改进界面体验，以及调整系统提示词。

系统提示词，英文叫 system prompt，是在每次对话开始之前发给模型的一段指令，用来设定模型的行为规范。你可以把它想象成给新员工的工作守则，在他开始干活之前，先告诉他我们这里的规矩是什么，什么可以做，什么要避免，回答要什么风格。

在系统提示词里，他们加了这样一条指令，原文的意思是：工具调用之间的文字不超过二十五个词，最终回复不超过一百个词，除非任务需要更多细节。

这条指令的初衷是：让 Claude 在用工具执行任务的过程中，工具调用之间的说明文字保持简洁，不要说太多废话，把精力放在做事上，而不是解释自己在做什么。

经过多周内部测试，跑了一套评估没有发现显著的质量下降，所以他们很有信心地在四月十六日把这条指令随 Opus 四点七一起上线了。

然而，在更广泛的用户使用场景里，问题很快就出现了。这条指令和系统提示词里其他已有的指令组合在一起，产生了意想不到的副作用：代码质量下降了。

Anthropic 做了更广泛的消融测试，就是把系统提示词里的每一行逐个去掉，观察去掉哪一行会改变评估测试的结果，从而理解每一行的实际影响。结果发现，就是这条长度限制指令，对 Opus 四点六和 Opus 四点七都造成了三个百分点的代码质量下降。

三个百分点，听起来不大，但在 AI 编程助手的竞争环境里，这是相当显著的差距，足以让用户明显感觉到代码质量的变化。四月二十日，这条指令被撤销了。

这个案例揭示了系统提示词工程里的一个经典陷阱：单独测试某条指令的时候没有问题，但它和其他指令组合在一起就可能出事。任何一条系统提示词里的指令都不是孤立存在的，它们之间会互相影响，甚至互相强化或者互相抵消，这种交互效应往往不是靠直觉能预测的。而且这次的问题还有另一层：他们用来测试的评估集，没有足够覆盖那些会因为这条指令而受到负面影响的场景。如果评估集的分布本身就有盲区，那测试通过并不代表没问题。

为什么三个问题合在一起这么难以发现？

现在你能理解为什么这件事那么难处理了。三月四日，推理努力级别从 high 降到了 medium，影响了所有使用默认设置的 Claude Code 用户。三月二十六日，思维历史 bug 被引入，影响的是那些会话空闲超过一小时的用户。四月十六日，提示词变化上线，影响的是特定任务场景里的代码生成质量。

这三个变化影响的是不同时间段、不同使用场景下的不同用户子集，合在一起呈现出的是一种宽泛的、不一致的整体下降感。有人感受到"变笨了"，有人感受到"容易忘事"，有人感受到"不知道为什么出错了"，有人感受到"额度消耗异常快"。没有一个一致的故障现象，没有一个简单可复现的场景，有的只是一种广泛弥漫的、很难说清楚的"不对劲感"。

Anthropic 在三月初就开始调查用户反馈了，但最初很难把这些报告和正常的用户反馈噪音区分开来。用户满意度本来就有自然波动，新模型上线的时候用户期望也会变化，总会有人觉得这个版本不如上一个版本。没有内部测试能稳定复现这些问题，调查就变得极其困难。

这里面有一个很深刻的工程管理教训：当系统足够复杂，多个相互独立的小变化可以产生一个看起来像单一系统性故障的合成效应。这要求你不仅要追踪每个单独的变化，还需要有能力理解这些变化之间的交互效应。在大型 AI 系统里这尤其困难，因为 AI 模型的输出本身就有随机性，把"这是 bug 导致的"和"这是正常的模型输出方差"区分开来，本来就是一件很难的事。

Anthropic 准备怎么改进？

文章最后一部分是 Anthropic 宣布他们计划做哪些改变来防止类似的事情再发生。

第一个改变是扩大内部员工使用公开版本的比例。Anthropic 内部有自己的测试构建版本用来开发新功能，但这导致内部员工用的不是用户真实使用的版本，会错过只在公开版本里出现的 bug。他们要确保更大比例的内部员工使用和外部用户完全相同的公开构建，这样内部员工才能遇到和用户一样的问题，而不是活在一个被隔离保护的测试环境里。

第二个改变是升级代码审查工具，并提供给用户使用。基于前面提到的发现，当 AI 代码审查工具能访问更多相关代码库作为上下文时，发现深层 bug 的能力有实质性提升。他们要把这个多仓库上下文能力正式集成进内部代码审查流程，同时也把改进后的版本提供给 Claude Code 用户使用。

第三个改变是对系统提示词的改动增加更严格的管控流程。每次系统提示词变化都要跑一套更全面的、针对每个模型单独的评估测试；继续做消融测试来追踪每一行指令的具体影响；用新开发的工具让提示词变化更容易被审查和溯源；在 CLAUDE.md 这个配置文档里增加规范，确保针对特定模型的改动只对那个模型生效。任何可能和智能水平产生权衡的改动，都要增加浸泡期、更广泛的评估集和渐进式发布，这样能在影响扩散到所有用户之前更早发现问题。

第四个改变是提升外部沟通的透明度。他们在 X 平台开了一个叫 ClaudeDevs 的账号，专门用来深入解释产品决策和背后的思考逻辑；在 GitHub 上也会同步发布集中的讨论串。这篇事后复盘文章本身就是这种透明度努力的直接体现。

作为对用户的一种补偿，他们重置了所有订阅者的使用额度，把这段时间里被 bug 额外消耗的额度还给用户。

这些改进对你的实际工作意味着什么？

好，说了那么多背后发生了什么，现在来聊聊这件事对正在使用或者打算使用 Claude Code 的工程师来说有什么实际意义。

第一点，理解推理努力级别对任务质量的影响，以及主动管理它。如果你在用 Claude Code 做复杂的任务——重构大型模块、调试难以复现的问题、设计系统架构——你需要知道自己当前用的是什么推理努力级别。现在的新默认是更高的设置，Opus 四点七用 xhigh，其他模型用 high，这对大多数用户来说是好消息。但如果你在做一些比较简单的、重复性的任务，主动切换到 medium 级别可以让有限的使用额度用得更久。关键是要有意识地做这个选择，而不是依赖默认设置然后对结果感到困惑。

第二点，理解会话长度和上下文管理对 Claude 表现的影响。那个思维历史 bug 已经修复了，但这件事揭示了一个更基本的事实：在 AI agent 里，上下文不是无限的，不同的上下文管理策略会显著影响模型的行为。如果你的 Claude Code 会话已经进行了非常长的时间，或者你中途离开了很久再回来，Claude 对之前决策脉络的"记忆"可能没有你想象的那么完整。对于需要跨越较长时间的复杂工程任务，定期开新的会话并在开头提供关键的背景信息，可能比依赖一个积累了大量历史的超长旧会话更可靠。

第三点，系统提示词工程里的组合效应警示，这对那些在自己的 AI 应用里使用系统提示词的工程师特别有用。这个案例清楚地说明：单独测试一条指令没有问题，不等于它和其他指令组合起来没有问题。如果你在调整系统提示词来控制模型行为，做消融测试是必要的，要用足够广泛的评估集，还要特别注意那些影响输出风格和长度的指令——这类指令很容易在特定任务类型上产生意想不到的质量影响，而且影响可能只在某些任务分布上才显现。

第四点，如何有效地向 AI 工具提供反馈。文章最后特别感谢了那些用斜杠 feedback 命令提交反馈、或者在网上发出具体可复现例子的用户，明确说是这些人最终让他们能够识别和修复这些问题。这提醒我们：泛泛的"感觉变差了"对工程团队的帮助非常有限。如果你遇到了 AI 工具表现异常的情况，尽量记录下具体信息：用的是什么模型、在什么任务场景下出的问题、具体的输入是什么、得到了什么不符合预期的输出。这种具体的、有复现步骤的反馈，才是真正能帮助工程团队定位问题的信息。

第五点，从这件事里提炼一个更大的视角：AI 工具的质量不是静态的。它会因为模型更新、配置调整、系统提示词变化而改变，而且这些变化可能不会提前通知你。建立一套对你自己核心使用场景的定期观察习惯，比如定期用同一批代表性的任务来测试 AI 工具的表现，能帮你更早察觉到质量变化，而不是等到"感觉不对了"才开始追溯。

好，我们来做个总结。

这篇文章本质上是 Anthropic 的一份公开工程事故报告。在今年三月到四月间，三个相互独立的改动影响了 Claude Code 用户的使用体验：推理努力级别被错误地降低了默认值、思维历史管理出现了每轮清除的 bug、以及一条控制输出长度的系统提示词指令和其他指令叠加后导致代码质量下降。三个问题分别在四月七日、十日和二十日被修复。

我个人认为，这篇文章最有价值的地方不是"他们出了什么问题"，而是"他们如何描述自己出了什么问题"。Anthropic 没有含糊其辞，没有用模糊的措辞来掩盖细节，而是把每个问题的技术层面讲得相当清楚：是什么改动、为什么要做这个改动、改出了什么问题、为什么难以发现、怎么修复的，还有接下来准备怎么做才能避免类似的事情再发生。这种程度的工程透明度在任何行业都不容易做到，在竞争激烈、信息不对称的 AI 行业尤其如此。

对我们这些使用 AI 工具的工程师来说，这件事最重要的启示或许是：任何足够复杂的系统，都会以你预料不到的方式失败，AI 系统也不例外。保持批判性的观察，记录异常，用具体可复现的方式反馈问题，不仅有助于产品本身的改进，也能让我们自己对这些工具有更清醒、更务实的认识。
