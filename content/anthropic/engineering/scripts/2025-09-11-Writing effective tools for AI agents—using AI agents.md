## Script

好，听众朋友们，欢迎收听 Claude FM。我是你的主播。今天这期节目，我们要聊的是 Anthropic 工程博客上的一篇深度文章，英文标题是 Writing effective tools for AI agents—using AI agents，直译过来就是"用 AI 智能体来为 AI 智能体写出更有效的工具"。这篇文章发表于二〇二五年九月十一日。

今天是二〇二六年六月十三日，距这篇文章发表已经过了将近九个月。在 AI 这个领域，九个月可以发生很多事情——各大模型又推了好几波新版本，agent 应用从热门概念逐渐走向了越来越多的生产实践，MCP 这个模型上下文协议也从新鲜事物变成了相当多工程师的日常工具。所以我要先给大家一个时效性提醒：文章里提到的一些具体数字、版本号、平台特性可能已经有所变化。但是，文章所讨论的核心设计原则——也就是怎样为 AI agent 设计出真正好用的工具——这些原则是相当基础的，具有持久的参考价值。

这篇文章写于一个很特殊的节点。二〇二五年下半年，正是业界开始认真思考"agent 工程"是一门独立学问的时候。大家都在 MCP 上面构建各种工具，但是跑起来之后发现效果参差不齐。有的工具 agent 用起来很顺手，有的工具 agent 反复出错，有的工具 agent 压根不知道该在什么时候用。这篇文章就是 Anthropic 内部团队，在大量实验和内部工具开发的基础上总结出来的经验。文章的主要作者是 Ken Aizawa，同时有来自研究、MCP、产品工程、应用 AI 等多个团队的同事参与贡献。

好，我们进入正题。为什么为 agent 写工具，比为人类写 API 要难得多？

你可能会想，这件事有什么难的，不就是把 API 封装一下，然后告诉模型说"你有这些工具可以用"吗？

文章的回答是：难就难在 AI agent 是一个根本不同于传统软件的系统。

文章用了一个非常清晰的区分：传统软件是确定性的，而 AI agent 是非确定性的。

什么是确定性系统？你写一个函数，叫做 getWeather，传入"纽约"，每次都会以完全相同的方式去获取纽约的天气数据——相同的代码路径，相同的返回格式，没有任何意外。这是两个确定性系统之间的契约，非常可靠，非常可预测。

但 AI agent 不是这样工作的。同样的问题，比如用户问"今天出门要带伞吗"，agent 可能调用天气工具，也可能从训练数据里直接回答，还可能先反问用户"你在哪个城市"。不同的对话历史、不同的工具组合，都可能让 agent 走向完全不同的路径。偶尔 agent 还会出现幻觉，或者完全没搞清楚某个工具该怎么用，传了错误的参数，甚至根本没用该用的工具。

所以，工具其实是一种新型软件——它是确定性系统和非确定性 agent 之间的契约。写这种新型软件，需要完全不同的思维方式。这是文章最核心的出发点。

我们先来看文章的第一个大主题：如何构建、测试和迭代你的工具。

文章给出了一个三步迭代循环：快速搭原型、系统做评估、与 agent 一起优化。

先说第一步，快速搭原型。

文章建议先不要想太多，快速立起一个工具原型来。如果你在用 Claude Code 来帮你写工具，有一个很实用的小技巧：把你依赖的软件库、API 文档或者 SDK 文档一起喂给 Claude，特别是那些 llms.txt 格式的文档。这种格式专门为语言模型优化过，把文档压缩成高密度、易于理解的形式，模型能更好地利用这些信息来生成代码。

工具写好之后，把它包装成一个本地 MCP 服务器，或者打包成桌面扩展，连接到 Claude Code 或者 Claude Desktop 里测试。工具也可以直接通过 Anthropic API 调用来做程序化的测试，这在自动化评估流程里非常有用。

文章特别强调一点：你自己要亲手用这些工具，不要只看 agent 怎么用。要自己动手感受一下，才能找出那些不顺手的地方。同时也要收集真实用户的使用反馈，建立对实际场景的直觉。这一步看似基础，但在真正迭代之前对工具质量有个感性认识，是非常必要的。

第二步，跑系统评估。

这是整篇文章花笔墨最多的地方，也是最有操作价值的部分。

评估的第一件事是生成评估任务。文章说可以和 agent 协作来快速生成几十个任务。但这里有一个关键的质量要求：任务必须贴近真实世界，要基于真实的数据源和实际的业务场景，不要做那种过于简化的沙盒环境，因为那样根本测不出工具在真实复杂场景下的表现。

文章专门给出了好任务和差任务的对比例子，我觉得这个对比非常有教学价值，来详细说说。

一个好任务的例子是：安排一个下周和 Jane 的会议，讨论我们最新的 Acme 项目，附上上次项目计划会议的笔记，并预订一个会议室。你看，这个任务需要什么？需要查日历找可用时间，找到正确的会议参与者，检索上次的会议文档，还要预订会议室——可能需要几十次工具调用，中间涉及多个不同的服务，这才是一个真正有挑战性的测试。

再看差任务的例子：安排一个和 jane 邮箱的下周会议。信息都已经给死了，直接调用日历工具就能完成，完全测不出 agent 在复杂情况下的表现，也看不出工具的问题。

还有一个好任务例子也很典型：某个客户反映自己的一笔购买被扣了三次款，找出所有相关的日志记录，并判断是否有其他客户受到了同样问题的影响。这种任务需要多步检索、跨数据源关联，还需要一定的推理，非常贴近真实的运维场景。

每个评估任务都要配上可验证的期望结果。验证方式可以简单到精确的字符串匹配，也可以复杂到用 Claude 来充当裁判——也就是所谓的 LLM-as-judge 方法。不过文章特别提醒：验证器要宽松适度，不要因为格式差异、标点不同或者等价的表达方式，错误地拒绝了正确答案。

评估跑起来之后，要收集哪些指标？文章建议不只看总体准确率，还要看每个工具调用的耗时、总的工具调用次数、总的 token 消耗、工具调用出错的次数和类型。这些数据能帮你发现很多隐藏的问题。

举个例子，如果某类操作出现了大量重复的工具调用，可能是分页大小或者 token 限制参数需要调整，也可能是工具设计得太细粒度了，需要合并。如果经常出现参数无效的报错，可能是工具描述写得不够清楚，或者缺少参数使用示例。

文章里还提到了一个真实的线上案例：当他们推出 Claude 的网页搜索工具时，发现 Claude 会莫名其妙地在搜索查询里加上"2025"这个年份，导致搜索结果有时间偏差，性能下降。他们是通过改进工具描述来纠正这个行为的——不是改模型，而是改描述。这个案例我们稍后还会回来讲。

文章还建议在评估 agent 的系统提示里加上一条指令：让 agent 在做工具调用之前先输出推理过程，也可以直接开启 Claude 的 interleaved thinking 功能。这样做有两个好处：一方面能触发模型的链式思维，让推理能力发挥得更充分；另一方面你能通过阅读 agent 的思考记录，看清楚它在哪里卡壳，在哪里误解了工具的用途，从而找到精准的改进方向。

第三步，与 agent 一起优化。

这是文章最有创意的部分。文章说，你可以把所有评估 agent 的对话记录拼接起来，直接扔给 Claude Code 来分析。Claude 非常擅长批量分析这些记录，指出工具描述之间的矛盾、工具实现上的效率问题，还能批量重构工具，确保改动之后工具实现和描述保持内部一致性。

这里有一个很重要的经验提醒，我觉得值得特别说出来：agent 反馈里没说的，往往比说了的更重要。LLM 不一定会把它真正遇到的问题明确说出来，所以要仔细读原始的对话记录，不只是 agent 的总结，还要看每一个工具调用和工具返回的原始内容，要从字里行间读出 agent 的困惑所在。

文章还强调要保留一个留出的测试集，避免过拟合。他们发现，即使是经验丰富的研究人员手写的工具，或者 Claude 自己生成的工具，通过这套系统化的评估迭代，都还能额外挖出性能提升的空间。这个结论挺让人震惊的，它说明这套方法是真的有效，不只是理论上好听。

好，方法论说完了，我们进入文章的第二大主题：写高质量工具的核心原则。文章总结了五条，我们逐一来讲。

第一条原则：选择正确的工具去实现，同样重要的是决定哪些工具不去实现。

文章说，一个最常见的错误，是把现有的软件功能或者 API 端点逐一包装成工具，不管这些工具对 agent 到底适不适合用。

为什么会有这个问题？文章给了一个非常精妙的解释：AI agent 和传统软件有完全不同的"可供性"——借用认知心理学的说法，就是感知到的可以采取什么行动的方式。AI agent 的上下文空间是有限且宝贵的，而计算机内存相对来说便宜且充裕。这两种系统的资源约束完全不同，所以适合它们的工具设计自然也不同。

文章用了一个通讯录搜索的例子，我觉得特别生动。传统软件处理通讯录搜索，可以把全部联系人加载进内存，逐一检查，因为这种遍历在硬件层面是高效的。但如果给 AI agent 提供一个返回全部联系人列表的工具，让 agent 一个 token 一个 token 地把整个列表读完，才能找到目标联系人，这就是在浪费 agent 最宝贵的资源——上下文空间。

这就好比说，你要在一本厚厚的通讯录里找人，你不会从第一页翻到最后一页逐行阅读对吧？那种暴力搜索的方式，对人和 AI 来说都不是自然的路径，对人是效率低下，对 AI 是在浪费有限资源。更好的方式，是直接翻到字母对应的那一页。

所以，更好的工具设计是实现一个 search contacts，而不是 list contacts。让工具自己去处理搜索逻辑，只把真正相关的结果返回给 agent。

文章给了几组工具合并的具体例子，我逐一说一下，这些例子对实际工程非常有指导价值。

第一组：与其分别实现"列出用户"、"列出事件"、"创建事件"三个工具，不如实现一个"安排活动"工具，让它在底层自动处理查找可用时间和创建事件这两个步骤。agent 只需要描述意图，工具自己去完成后续的协调工作。

第二组：与其实现一个"读取日志"工具，不如实现一个"搜索日志"工具，它只返回与查询相关的日志行和少量上下文，而不是把几十万行的日志全部塞给 agent。

第三组：与其实现"按 ID 获取客户"、"列出交易记录"、"列出备注"三个分散的工具，不如实现一个"获取客户上下文"工具，把客户最近最相关的所有信息一次性汇总返回。

这背后有一个清晰的设计哲学：工具应该让 agent 能像人类一样来分解和解决任务，同时减少因为中间输出而消耗的上下文。工具太多，或者工具之间有功能重叠，会分散 agent 的注意力，让它难以找到高效的解题路径。文章说，这种精心的、有选择性的工具规划，能带来非常显著的性能提升。

我个人认为，做到这一点需要你对业务场景有深刻的理解。你要真正搞清楚：用户最典型的工作流是什么？哪些步骤几乎每次都是连在一起做的？把这些模式识别出来，然后把它们封装进一个工具，才能让 agent 真正顺手。

第二条原则：用命名空间来划清工具之间的边界。

当你的 agent 可能同时连接几十个 MCP 服务器、几百个不同工具时，命名就变得非常关键了。工具如果功能有重叠，或者目的不够清晰，agent 就会感到困惑，不知道该用哪个。

命名空间，就是给相关工具加上统一的前缀，可以有效划清这些边界。文章举例：按服务命名，比如 asana search、jira search；或者更细粒度地按资源命名，比如 asana projects search、asana users search。这样 agent 在面对几百个工具时，一看名字就能快速缩小范围，选对该用的那一组。

文章还提到了一个挺有意思的细节：前缀式命名还是后缀式命名，这个看似微小的选择，在他们的评估里对最终结果有相当显著的影响，而且不同的语言模型对这两种方式的反应还不太一样。所以文章建议根据你自己的评估来做选择，不要盲目照搬别人的方案。

这条原则背后有更深的逻辑：通过精心命名，你在做两件事。一方面减少了 agent 上下文里需要同时处理的工具数量；另一方面，把原本要 agent 自己推理"该用哪个工具"的认知负担，提前固化在了命名设计里。这两点加在一起，降低了 agent 整体犯错的概率。

第三条原则：让工具返回有意义的上下文，而不是技术性的原始数据。

这一点我觉得非常实用，也是很多人在实际构建工具时容易踩的坑。

文章说，工具响应应该返回高信噪比的信息，要优先考虑对 agent 有直接用处的字段，而不是追求数据的完整性和灵活性。要避免返回那些低级的技术性标识符，比如随机的 UUID、带尺寸前缀的图片 URL、MIME 类型这些东西。相对地，像 name、image url、file type 这样的字段，更有可能直接帮助 agent 理解数据的含义，做出正确的后续决策。

还有一个非常有意思的发现：agent 处理自然语言名称和有语义的标识符，远远好于处理随机的字母数字 UUID。文章发现，仅仅是把随机的 UUID 替换成更有语义含义的标识符，甚至只是换成从零开始的数字 ID，就能显著提升 Claude 在检索任务上的精度，减少幻觉的发生。这背后的道理其实很直觉：UUID 是给机器看的，对语言模型来说是一串没有意义的噪音；而语义化的名称或者有规律的数字 ID，模型能建立更好的推理链条。

当然，有些场景下 agent 确实需要同时用到自然语言和技术性标识符。比如先用人名搜索用户，然后拿到 ID 再发消息。为了处理这种情况，文章建议在工具里暴露一个响应格式的枚举参数，让 agent 自己控制返回"详细"还是"简洁"的格式。文章给了具体的数字：详细格式返回两百零六个 token，简洁格式只需要七十二个 token，差了将近三倍。在长对话和复杂任务里，这种弹性设计能节省大量上下文。

而且，不只是内容，连工具响应的结构形式——是 XML、JSON 还是 Markdown——也会对性能有影响。文章的结论是：没有放之四海皆准的最优格式，因为语言模型是用下一个 token 预测来训练的，对不同格式的熟悉程度取决于训练数据的分布。最优的结构需要通过你自己的评估来确定。

第四条原则：优化 token 效率。

上下文的质量很重要，上下文的数量同样不可忽视。

文章建议，任何可能消耗大量上下文的工具响应，都应该实现分页、范围选择、过滤或者截断机制，并且给这些参数设置合理的默认值。文章提到，Claude Code 默认把工具响应限制在两万五千个 token 以内。随着模型的有效上下文长度在持续增长，具体的数字会变，但设计上下文高效的工具这件事不会过时。

如果你选择截断响应，一定要给 agent 清楚的引导信息，告诉它接下来应该怎么做，而不是让它面对一条截断提示一头雾水。你可以在截断提示里明确告诉 agent 更好的策略：比如在做知识检索的时候，做多次小范围的精准搜索，比一次大范围的宽泛搜索要好，不只返回的信息更相关，消耗的 token 也更少。

工具报错的处理同样值得精心设计，这一点很多人会忽略。文章做了一个很直观的对比：一个没用的错误响应，只返回一个错误代码或者一大串堆栈追踪，agent 看完还是不知道该怎么办，只能盲目重试；而一个好的错误响应，应该清楚地告诉 agent 具体哪个参数有问题，期望的格式是什么，或者该怎么修改才能成功。这个设计思路和写给人类开发者的 API 文档是完全相通的——好的错误信息要具体、要可操作，而不是一堆让人一头雾水的技术细节。

第五条原则：对工具描述做提示工程。

这是文章重点推荐的、效果最显著的改进方向，也是五条原则里最容易被低估的一条。

工具的描述和参数说明，会被加载进 agent 的上下文，所以它们能整体引导 agent 的工具调用行为。文章建议写工具描述的时候，想象你在向一个刚入职的新同事解释这个工具——你需要把那些你习以为常的背景知识都说清楚：专门的查询格式是什么，业务术语如何定义，底层资源之间的关系是什么。要避免歧义，清楚描述期望的输入和输出。

特别是参数命名，要做到绝对清晰：不要用一个叫 user 的参数，要用 user id；不要用 date，要用 start date 和 end date。这些看起来是小事，但参数名的歧义会直接导致 agent 的调用错误，而且很难从对话记录里定位到根因。

文章给了一个非常有说服力的案例：Claude Sonnet 三点五在 SWE-bench Verified 这个代码工程能力评估上达到了当时的最优性能。靠的是什么？靠的就是对工具描述做了精确的微调——显著降低了错误率，提升了任务完成率。这个结果直接证明了工具描述的质量对 agent 性能的影响是实质性的，绝不是可有可无的细节。

还有一个细节值得回头说一说——我前面提到的那个网页搜索工具的案例。发现 Claude 会在搜索查询里加上年份之后，他们的解法不是重新训练模型，而是改工具描述，加了几句关于不要附加年份的说明。问题就解决了。这个案例说明，工具描述不只是被动的文档，它是主动引导 agent 行为的机制，是你和 agent 之间协调行为的主要手段。

文章最后还提到，如果你在给 Claude 写工具，要了解工具是怎么被动态加载进 Claude 的系统提示的；如果在写 MCP 服务器的工具，工具注解可以帮助声明哪些工具需要开放式的网络访问，哪些会做破坏性的修改。这些信息有助于 agent 在调用工具之前做更好的风险判断。

好，五条原则讲完了，我们来把这些内容串起来，谈谈对实际工作意味着什么。

如果你现在正在做 agent 开发，或者在维护一个 MCP 服务器，这篇文章能给你的最大启发是什么？

首先，要抵制住"把 API 一一映射成工具"的本能冲动。在动手写工具之前，先想清楚 agent 最需要解决的是哪几个核心工作流。从工作流出发来设计工具，而不是从已有的 API 端点出发。这两种出发点，会导致完全不同的工具设计，最终 agent 的体验也会有巨大差距。

其次，尽早建立评估体系，而且要建得足够真实、足够复杂。很多人会把建评估当成项目后期的事，但文章的经验告诉我们，评估应该在你刚有原型的时候就开始。评估任务要贴近真实场景，强评估任务甚至可能需要几十次工具调用。而且评估不是一次性的活动，它是贯穿整个开发周期的持续测量工具。

第三，善用 agent 来帮你优化工具。把评估对话记录喂给 Claude Code，让它帮你分析和重构——这个"用 AI 改进 AI 工具"的元循环，文章证明是真实有效的，效率远高于人工逐条审查。而且文章也明确说了，他们自己的很多工具优化建议，正是通过这个过程发现的。

第四，认真对待工具描述，像写核心业务代码一样认真，甚至更认真。工具描述是你和 agent 之间最直接的沟通界面，描述越精准，agent 的行为就越可预测，越少出现意外的错误。

要避开的坑：不要为了覆盖所有情况而返回大量 agent 用不到的字段；不要在工具响应里塞满 UUID 这样的技术标识符；不要把错误信息写成只有开发者才看得懂的堆栈追踪；也不要把工具拆得太细，让 agent 完成一件事需要调用十几个工具，中间还要在工具之间传递技术 ID。

我个人认为，这篇文章最有价值的地方，不是某一个具体的技术技巧，而是它提供的整体思维框架：为 agent 设计的工具，要像为人类设计工具一样，注重认知上的可用性，而不只是功能上的完备性。一个好的工具应该是"对 agent 来说符合直觉的"。文章还观察到了一个有意思的规律——对 agent 最符合直觉的工具，往往对人类来说也最容易理解。这并不是巧合，而是说明好的工具设计有其内在的普遍原则，跨越了人类和 AI 的边界。

文章的结尾也提到：随着 MCP 协议的持续更新和底层模型能力的不断提升，agent 和工具交互的具体机制会持续演化。但只要我们用这种系统化、评估驱动的方式来不断迭代，工具的质量就能跟上 agent 能力的成长节奏。

好，今天这期节目就到这里了。这篇来自 Anthropic 工程团队的文章，展示了一套完整的、评估驱动的工具开发方法论，以及五条核心设计原则：选对工具、用好命名空间、返回有意义的上下文、优化 token 效率、认真做工具描述的提示工程。如果你正在做 agent 工程，强烈建议把文章原文找来读一遍，特别是工具评估和提示工程那两部分，有很多实操细节今天没有全部展开。我们下期再见。
