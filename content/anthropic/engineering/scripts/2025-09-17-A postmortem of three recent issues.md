今天我们来读一篇 Anthropic 工程博客发布的文章，英文标题是 A Postmortem of Three Recent Issues，我把它翻译成"三个近期问题的事后复盘"。这篇文章发表于二〇二五年九月十七日。

我需要在开头说明一下时效性：今天是二〇二六年六月十三日，距离这篇文章发表已经过了将近九个月。对于一篇讲基础设施事故的技术文章来说，九个月算不上很久，里面描述的工程问题和修复思路依然非常有参考价值。但你需要知道当时的背景：这篇文章发表的时候，Claude 四系列刚刚推出不久，Anthropic 正处于产品大规模铺开的关键节点，用户量在快速增长，而与此同时整个 AI 行业竞争非常激烈。在这个背景下，部分用户开始反映 Claude 的回答质量变差了，这件事的舆论影响不小。

所以 Anthropic 选择这个时间点发布一篇技术详解，背后有两层用意：一是向用户交代清楚到底发生了什么，二是表明他们的工程文化——愿意公开承担责任、解释失误、并说明改进计划。这在 AI 公司里算是比较少见的做法，值得我们认真看。

好，我们进入正文。

这次事故的时间线大致是这样的：从二〇二五年八月初开始，一些用户陆续反映 Claude 的回复质量下滑了。但在最初阶段，这些反馈很难和正常的"用户对 AI 的期待差异"区分开来——每个用户对好回答的标准不一样，AI 回答本身也有随机性，所以工程团队起初把这些报告当作噪声处理。

到了八月底，报告的频率和持续性让工程团队意识到这不是普通噪声，他们正式开启了调查。调查结果出乎意料：他们发现了三个独立的基础设施 Bug，这三个 Bug 恰好在相近的时间窗口内同时活跃，相互叠加，让排查工作变得极为复杂。

在讲三个具体 Bug 之前，我想先强调 Anthropic 在文章里说的一句话，因为这句话当时对很多用户来说很重要。他们明确写道：我们从不会因为需求量大、时间段，或者服务器负载而主动降低模型质量。这次用户报告的所有问题，都是基础设施 Bug 造成的，而不是 Anthropic 悄悄省成本。

这句话之所以要特别强调，是因为当时网络上确实流传着"AI 公司会在高峰期偷偷给你用差一点的模型"这种说法。Anthropic 的声明是明确否认这一点。当然，你可以选择相信或者不相信，但至少他们把话说清楚了，并且给出了技术层面的完整解释来支撑这个说法。

现在我们来看三个 Bug，逐一拆解。

第一个 Bug，我把它叫做"上下文窗口路由错误"。

这个 Bug 在八月五日被引入。要理解它，你需要先了解一点背景：Anthropic 在这段时间里，正在为即将推出的支持一百万 token 超长上下文窗口的 Sonnet 4 版本做准备。他们有一批专门针对超长上下文优化过的服务器，这些服务器的配置和普通服务器不同。

问题在于，一部分普通的短上下文请求——也就是日常对话、代码补全这类没有超长文本的请求——被错误路由到了这批为超长上下文配置的服务器上。这就像是你去快餐店点了一个汉堡，厨房按照宴席备餐流程来处理你的订单，最后给你端出来的东西不对劲。

这个 Bug 在八月五日引入时，影响范围只有约百分之零点八的 Sonnet 4 请求，相对有限。但关键的转折点来自八月二十九日：那天工程团队做了一次常规的负载均衡调整——这是很正常的运维操作，目的是平衡各服务器的流量。但这次调整无意中大幅增加了被错误路由到超长上下文服务器的短请求数量。

到了八月三十一日最糟糕的那个小时，受影响的 Sonnet 4 请求比例达到了百分之十六，也就是大约六分之一的请求出了问题。

这里有个让情况更复杂的细节：Anthropic 的路由是"粘性"的，英文叫 sticky routing。这个机制的意思是，一旦你的某个请求被路由到了某台服务器，后续的跟进请求大概率也会被路由到同一台服务器。这个设计本来是为了效率和一致性，但在这次事故里，它产生了一个副作用：有些用户运气不好，第一个请求就落到了错误的服务器，之后的整个对话都在质量下降的环境里进行；而另一些用户的路由一直是正常的，完全感知不到任何问题。

这就造成了一个非常令人困惑的社区反馈图景：有人说 Claude 变差了，有人说没有，有人说时好时坏，大家说的好像不是同一个产品。从工程团队的角度看，这种矛盾的信号是最难处理的，因为你很难从中找到规律性的根因。

在 Claude Code 用户群体中，这个影响尤其值得关注。在受影响时间段内，大约有百分之三十的 Claude Code 用户，至少有一条消息被路由到了错误的服务器类型，收到了质量下降的回复。

这个 Bug 的修复方案相对直接：修正路由逻辑，确保短上下文请求和长上下文请求分别被送到正确的服务器池。修复在九月四日部署，到九月十六日全面覆盖了第一方平台和 Google Cloud 的 Vertex AI，Amazon Bedrock 在九月十八日完成。

第二个 Bug，我把它叫做"TPU 服务器的 token 污染"。

这个问题在八月二十五日随一次部署引入。受影响的是运行在 TPU 芯片上的 Claude 服务器，具体涉及到 Opus 四点一和 Opus 四这两个模型，以及 Sonnet 四。

问题的表现很直观，甚至有些荒诞：如果你用英文问 Claude 一个问题，它可能会在回答中间突然冒出泰文字符或者中文字符。文章里给了一个具体例子，说用户可能会在英文回答中间看到泰文的"สวัสดี"，这是泰文的"你好"。代码生成任务里也会出现明显的语法错误。

这是一个运行时性能优化引发的配置错误。某种优化操作偶尔会给本不应该出现在当前上下文中的 token 赋予过高的概率——比如在英文对话里，泰文字符的概率本应接近零，但这个 Bug 会让它异常升高。

这个 Bug 只影响第一方 API，不影响 Amazon Bedrock 和 Google Cloud Vertex AI 上的用户。在九月二日，工程团队定位了问题并回滚了那次变更。修复之后，他们还在部署流程里加入了检测异常字符输出的专项测试，避免类似问题再次出现。

从用户体验的角度来说，这个 Bug 的表现最"显眼"——突然出现的泰文字符很容易被察觉和截图上报，这反而让它在三个 Bug 里相对好定位。

第三个 Bug 是最复杂的，也是这篇文章技术含量最高的部分。我们来慢慢讲。

这个 Bug 的名字可以叫"XLA 编译器的精度混用问题"。要理解它，需要先搭建一些基础概念。

当 Claude 生成文字的时候，背后的过程大致是这样的：模型每次预测下一个 token，会给词表里所有可能的 token 算出一个概率分布。然后从这个分布里随机采样，选出下一个词。为了避免生成完全无意义的输出，Claude 使用了一种叫做 top-p 采样的机制——只考虑那些累积概率达到某个阈值的 token，比如零点九九或者零点九九九。这个机制叫做核采样，它的作用是把那些极低概率的奇怪词排除在候选之外。

Claude 的模型规模很大，大到单个芯片装不下，需要把计算分散到数十个 TPU 芯片上并行处理。这意味着计算概率分布的操作本身就是分布式的，需要在多个芯片间协调数据。排序这些概率值，在分布式环境下是一个相对复杂的操作。

好，有了这个背景，我们来看 Bug 是怎么一步步发展的。

故事要从二〇二四年十二月说起——没错，这是一个隐藏了将近九个月的深层问题。当时，Anthropic 的工程团队发现了一个 TPU 实现上的问题：在温度参数设为零的时候，采样逻辑偶尔会把概率最高的那个 token 给丢掉。他们针对这个特定场景部署了一个临时修复方案，相当于打了个补丁。

到了二〇二五年八月二十五日，工程团队部署了一版对采样代码的全面重写，目的是修复精度问题，并改进处理概率值在 top-p 阈值边界附近的逻辑。在这次重写里，他们认为已经从根本上解决了问题，所以移除了十二月那个临时补丁。

但移除那个补丁，暴露了一个更深的 Bug。

这里涉及到一个叫做浮点精度的技术细节。Claude 的模型在计算下一个 token 的概率时，用的是 bf16 格式——这是一种十六位的浮点数表示方式，精度相对较低但速度快、内存省。然而，TPU 芯片的向量处理器是以三十二位浮点数为原生格式的，精度更高。

TPU 的编译器叫做 XLA，它有一个默认开启的优化选项，名字叫 xla_allow_excess_precision，字面意思是"允许超额精度"。这个优化的逻辑是：如果用更高精度计算能提升运行效率，编译器会自动把部分操作从十六位升级到三十二位。

表面上看，提高精度是好事，为什么会出问题？问题在于"不一致"。在确定哪个 token 概率最高的过程中，有些操作在十六位精度下算，有些操作被编译器优化到三十二位精度下算。同一个概率值在两种精度下表示不同，导致这两部分操作在"谁是概率最高的 token"这个问题上得出了不同的答案。

结果就是：概率最高的那个 token，有时候会从候选列表里消失，不参与采样。这对生成质量的影响是显著的，因为被丢掉的恰恰是模型"最应该"选的词。

更让工程团队头疼的是这个 Bug 的表现极度不稳定。同样的提示词，一次能正常运行，下一次就出问题。不稳定性的来源很诡异：Bug 的触发条件取决于当前 batch 的大小，甚至取决于在它之前或之后运行了什么其他操作。而且一旦工程师开启调试工具来排查问题，Bug 的行为就会改变。你可以想象这对调试工作来说有多绝望——为了看清楚 Bug，你开了灯，但开灯这个动作本身就让 Bug 躲起来了。

这种现象在底层系统调试里有个专有名词，叫做 Heisenbug——来自量子力学里海森堡的不确定性原理，意思是观测行为本身影响了被观测对象的状态。

十二月那个临时补丁之所以能掩盖这个问题，是因为它恰好绕过了触发这个 Bug 的代码路径。一旦补丁被移除，Bug 就浮出水面了。

最终的解决方案有两条路并行推进。一方面，Anthropic 与 XLA:TPU 团队合作，从编译器层面修复这个精度不一致的根本问题。另一方面，他们在自己的代码里把之前用的"近似 top-k"操作换成了"精确 top-k"，并把一些额外的操作统一到三十二位精度。

"近似 top-k"和"精确 top-k"是什么意思呢？在寻找概率最高的若干个 token 时，精确算法能保证结果完全正确，但计算成本更高；近似算法通过接受一些低概率 token 的误差来换取速度，理论上不应该影响质量，但这个 Bug 恰恰让它错误地把最高概率 token 也丢掉了。Anthropic 发现，精确 top-k 在现在的硬件条件下性能损耗已经不那么严重了，所以他们决定切换到精确版本，用少量的效率代价换取正确性。

顺便说一下，文章里特别提示：这次切换到精确 top-k 之后，在概率值恰好处于 top-p 阈值边界附近的 token 可能会有轻微的行为变化。如果你在用 Claude API 并且精调过 top-p 参数，可能需要重新测试一下效果是否符合预期。

讲完三个 Bug 的技术细节，我们来看这篇文章里我认为最有深度的部分：为什么这些问题被检测到这么晚？

Anthropic 自己做了一个相当坦诚的自我剖析，指出了两个核心问题。

第一个问题是评测系统的灵敏度不足。他们的标准做法是用 benchmark 测试集来衡量模型质量，再结合安全评测和性能指标。但这次的问题是，Claude 在面对孤立的错误时往往能自我恢复——就是说，即使单个 token 生成出了问题，后续的上下文可能会把它"拉"回来，让最终结果看起来还过得去。这让 benchmark 上的分数没有明显下降，但真实的用户体验已经变差了。

这是一个很典型的"评测与现实脱节"的问题。你把精力都用在优化 benchmark，但 benchmark 的设计本身就不够贴近真实使用场景。这在整个 AI 行业里都是一个普遍挑战，不只是 Anthropic 面临的。

第二个问题更有意思，也更有张力：隐私保护机制限制了排查能力。Anthropic 有严格的内部隐私和安全规定，限制工程师访问用户与 Claude 的真实对话，尤其是当这些对话没有被用户主动标记为问题反馈的时候。这个限制的出发点是正确的——保护用户隐私是应该的。但副作用是，工程师无法直接审查那些有问题的真实对话来重现和诊断 Bug。

这是一个典型的安全性与可观测性之间的权衡。要调试问题，你需要看数据；但看数据本身可能侵犯隐私。这个两难困境在系统设计层面没有完美答案，只能找到一个可接受的平衡点。

还有一个问题是这三个 Bug 同时活跃，但症状各不相同，发生在不同平台、不同比例上，造成了一堆看起来互相矛盾的用户报告。比如有人说 Claude 突然夹杂奇怪字符，有人说 Claude 在他的场景里答非所问，还有人说一切正常。这些信号在工程师看来像是随机噪声，而不是指向某个系统性问题。直到负载均衡调整在八月二十九日把受影响流量放大，才形成了足够强的信号让团队意识到这不是普通的随机波动。

那么 Anthropic 打算怎么改？文章提出了三个方向。

第一，开发更灵敏的评测工具，能更可靠地区分正常实现和有问题的实现，而不是依赖那些容易被蒙混过关的 benchmark。

第二，不只是定期跑评测，而是在真实生产系统上持续运行评测。这样的话，上面说的那个路由错误，如果在生产流量上持续跑着质量监控，应该很快就能检测到，而不需要等用户反馈积累到阈值。

第三，开发能在不侵犯用户隐私的前提下、更好地利用社区反馈来调试问题的工具和基础设施。这是最难的一个，因为需要在隐私保护和可观测性之间设计出新的机制。

说到这里，我想聊聊对工程团队和 AI 从业者来说，这篇文章意味着什么。

如果你在用 Claude API 做产品，这篇文章有几个直接的实践意义。第一，如果你的应用对响应质量有严格要求，光靠调用 API 然后看最终输出是不够的，你需要建立自己的输出质量监控——不一定要很复杂，但要能快速检测到系统性的质量下滑。第二，如果你调过 top-p 参数，现在值得重新验证一次，因为底层的 top-k 实现已经从近似版本换成了精确版本，边界行为有轻微变化。第三，Anthropic 明确鼓励用户在发现质量问题时通过 bug 命令或者点踩按钮上报，这类具体反馈在这次事故里起到了关键作用，相当于人肉监控补充了系统监控的盲区。

如果你是在做 AI 基础设施或者大模型部署相关的工作，这篇文章里的两个技术细节值得特别关注。第一个是精度混用问题：在分布式推理环境下，如果不同计算节点或不同操作对同一个数值使用了不同精度表示，会导致结果不一致。这不是 Anthropic 独有的问题，任何在 TPU 或者 GPU 上做多芯片分布式推理的团队都可能遇到。第二个是近似算法的隐性假设：用近似算法换性能，前提是近似误差只影响低概率的 tail，不影响 top 候选。一旦 Bug 破坏了这个假设，近似算法会产生比精确算法更严重的错误。这个教训适用范围很广。

最后，说一说这篇文章我个人觉得最值得肯定的地方。Anthropic 选择把这么多技术细节公开出来，包括他们自己评测体系的缺陷、隐私机制带来的排查困难、以及一个隐藏了九个月的深层编译器 Bug——这是需要一定勇气的。在 AI 公司普遍把基础设施当作黑盒的行业背景下，这种程度的透明度是不多见的。

当然，你也可以换个角度看：这篇文章发表在事故发生之后，部分原因可能是社区已经注意到了问题，公开解释是更好的公关策略。但不管动机是什么，技术细节本身是真实可信的，对整个行业都有参考价值。

我个人认为，最值得所有做 AI 系统的工程师记住的一句话，是文章里说的那个原则：模型质量是不可妥协的，哪怕要为此接受轻微的性能损耗。在他们决定从近似 top-k 切换到精确 top-k 的那一刻，工程团队说了一句话：模型质量是不可谈判的，我们接受这个小小的效率代价。这句话背后的工程文化，我觉得值得学习。

好，今天这期节目就到这里。这篇文章叫 A Postmortem of Three Recent Issues，发表于二〇二五年九月十七日，作者是 Anthropic 的 Sam McAllister，背后有一整个工程团队的贡献。如果你想看原文的所有技术注脚，包括 XLA:TPU 编译器的细节，推荐找到原文仔细读一遍，细节很扎实。我们下期再见。
