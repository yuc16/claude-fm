## Script

今天要聊的这篇文章，标题是"Building a C compiler with a team of parallel Claudes"，中文可以译作"用一支并行 Claude 团队来构建 C 编译器"。这篇文章发表于二零二六年二月五日，作者是 Anthropic 安全团队的研究员 Nicholas Carlini。

距今大约四个多月，时间不算太远，但在 AI 这个领域，四个月已经足够发生很多事情了。这篇文章的核心意义在于，它不只是"AI 写了个编译器"这样的技术奇闻，它实际上是一次对"智能体团队"这种全新工作范式的系统性压力测试，而且是在一个极其苛刻的领域里做的：用一个能够编译 Linux 内核的 C 编译器来测试 AI 的边界，这本身就是一种非常大胆的选择。

在我们进入细节之前，先聊聊为什么这篇文章值得你在通勤路上花二十五分钟。如果你现在正在用 Claude 或者其他 AI 工具辅助开发，你大概遇到过这样的场景：你提了一个比较复杂的需求，AI 做了一半，然后停下来问你："我接下来应该怎么做？""这里有个歧义，请确认一下。"每隔几分钟就需要你回来拍板。这种模式本质上仍然是"人在回路"——AI 是一个需要持续监督的执行者，而不是一个可以独立完成项目的开发者。

这篇文章想回答的问题是：如果我们把 AI 从这个模式里解放出来，让它真正自主地运行几天、做几千次迭代，能实现什么？

答案是：一个十万行的 C 编译器，能在 x86、ARM 和 RISC-V 三个架构上编译 Linux 6.9 内核，还能编译 QEMU、FFmpeg、SQLite、PostgreSQL、Redis，通过了 GCC 折磨测试套件的99%用例，甚至能编译运行传说中的 Doom。

这个实验的代价是：近两千次 Claude Code 会话，两亿次输入 token，一亿四千万次输出 token，API 成本将近两万美元。

背景铺垫完毕，我们来看这个实验到底是怎么运作的。

先说"智能体团队"这个概念。作者把它叫做 agent teams，本质上就是多个 Claude 实例并行工作在同一个代码库上，中间不需要人类介入。你可能会问，这跟我平时用 Claude Code 有什么本质区别？

单个 Claude Code 会话有两个根本性的限制。第一，它一次只能干一件事。第二，它会停下来等你。无论是遇到需要确认的歧义，还是遇到技术卡点，它最终都要回到人身上。这个模式下，AI 是你的结对编程伙伴，你得坐在旁边陪着。

智能体团队模式打破了这两个限制。十六个 Claude 实例同时运行，每个实例都在自己的 Docker 容器里跑，挂载同一个上游 git 仓库。每个智能体克隆一份本地代码，做完自己的工作，再把改动推回去。整个过程不需要人在旁边盯着，可以跑几天甚至几周。

这个实现方式说起来其实非常朴素。你可以想象它就是一个无限循环的 bash 脚本：Claude 启动，读取一个任务描述文件，做事情，做完了退出，然后脚本立刻再启动一个新的 Claude 会话，如此往复。作者在文章里给出了那段脚本的代码，也就十几行，简单到让人有点意外。他还特别注明：要在容器里跑，不要在你自己的机器上跑——因为这个循环是真的会一直跑不停的。

顺带一提，作者提到有一次他看到 Claude 意外执行了"pkill -9 bash"这条命令，把自己所在的进程杀掉，终止了循环。这个细节让我觉得很有意思：一个 AI 在无人值守的情况下意外地终结了自己，这个场景有点既好笑又引人深思。

当然，让这件事真正能运转起来的难点不在循环本身，而在于三个关键问题：第一，多个智能体同时工作，怎么避免互相踩脚？第二，每个智能体启动的时候什么上下文都没有，怎么知道该做什么？第三，怎么判断进展是好是坏，怎么不让 AI 在错误的方向上越跑越远？

我们一个一个来拆解。

先说任务协调和防冲突的问题。

多个智能体同时写同一个代码库，最怕的就是两个实例解同一道题，或者互相覆盖对方的改动。作者采用了一个非常工程化的方案：用文件系统来实现分布式锁。

具体做法是这样的：在代码仓库里有一个叫做 current_tasks 的目录。每个智能体想做什么任务，就先在这个目录下创建一个以任务命名的文本文件，比如"解析 if 语句"或者"实现函数定义的代码生成"。这就相当于在任务上贴了一个"我正在做"的便条。两个智能体不能同时持有同一个锁——如果两个 Claude 同时想锁同一个任务，git 的同步机制会确保只有一个成功，另一个得另选任务。任务做完之后，智能体从上游拉最新代码，合并其他人的改动，把自己的更改推上去，然后删掉那个锁文件。

这个方案有个非常有趣的地方：合并冲突是经常发生的，作者说 Claude 足够聪明，能自己处理掉大多数代码冲突。你想想这个场景——一个 AI 在解决另一个 AI 留下的代码冲突，而这一切都是无人值守的，在某个 Docker 容器里悄悄发生着。

值得一提的是，这个系统里没有"协调智能体"，没有一个总调度员来分配任务。每个 Claude 实例自己决定接下来做什么。作者发现，在大多数情况下，Claude 会自然地选择"下一个最显然的问题"来解决。当它卡在某个 bug 上时，Claude 往往会维护一份失败方法的运行日志和剩余任务清单，这对下一个接手的实例很有帮助。如果你打开这个项目的 git 历史，能看到各种任务锁文件被创建和删除的轨迹，读起来就像在看一群安静工作的工蜂。

这种去中心化的自组织方式，在系统运转正常的时候其实蛮优雅的。但它也埋下了一个隐患——后面我们还会讲到，当所有智能体都卡在同一个 bug 上的时候，这个机制会完全失效。

现在来说第二个难点：智能体的自我定向问题。

每个 Claude 实例启动的时候，它处于一个全新的容器里，什么上下文都没有。它怎么知道项目的当前状态，怎么知道别人已经做了什么，下一步该做什么？

作者的解决方案是：把项目的自我描述能力内建进项目本身。他在任务提示词里明确要求 Claude 维护详尽的 README 和进度文件，并且要频繁更新。这样，每个新启动的智能体读完这些文档，就能大致了解现状，然后做出自己的判断。

这里有一个非常微妙的转变，作者自己也特别强调了这一点：他意识到自己在为 Claude 写文档，而不是为自己写文档。这两件事的要求完全不一样。

给人类看的文档可以用丰富的上下文，可以引用背景知识，可以假设读者记得上次讨论过什么。给 Claude 看的文档必须做到：任何一个从零开始的实例，读完之后都能精确地理解当前状态。所以文档要非常明确、非常具体、不能有歧义，而且要持续保持最新。作者说，他必须时刻提醒自己，这份文档的读者不是他，而是一个对历史毫无记忆的 AI。

这个洞察对我们日常使用 AI 也很有启发：你给 AI 的上下文质量，直接决定了它能做到什么程度。你写给 AI 的任务描述，其实应该像写给一个从未见过这个项目的新同事一样——完整、具体、可操作。

好，第三个难点也是最关键的：测试环境设计。

这是作者花力气最多的地方，也是我个人认为这篇文章技术含量最高的部分。

一个自主运行的智能体，它唯一的反馈来源就是测试结果。如果测试设计得不好，AI 会朝着错误的方向优化，最终"解决了问题"但解决的是个假问题。作者的原话是：任务验证器必须接近完美，否则 Claude 会把错误的问题解决得非常漂亮。

为此，作者需要找到高质量的编译器测试套件，为各种开源软件包编写验收脚本，持续观察 Claude 犯的错误，然后针对这些错误模式设计新的测试来覆盖。这本身就是一个迭代的工程过程。

测试设计有两个专门针对语言模型特性的核心原则，非常值得记住。

第一个原则是：不要污染上下文窗口。

语言模型的上下文是有限的，而且一旦被无用信息填满，模型就会开始走神，注意力被稀释掉。测试框架不应该打印几千行没用的内容，顶多打印几行摘要，然后把详细日志写到文件里，让 Claude 需要的时候自己去查。而且日志格式要为自动处理而设计：如果有错误，就在日志里写 ERROR，然后把原因放在同一行，这样 Claude 用 grep 就能一眼找到问题所在。还要预先计算好汇总统计，这样 Claude 不需要自己去分析原始数据，直接读结论就好。

你看，这里的逻辑是：不要让 AI 做可以提前自动化的重复性工作，把它的注意力留给真正需要推理的问题。

第二个原则是：应对 AI 的"时间盲症"。

Claude 感知不到时间的流逝。如果你不加干预，它可能会花好几个小时在跑完整测试集，而不是在写代码、推进进展。为了解决这个问题，作者设计了一个默认的快速模式，只随机抽取百分之一或百分之十的测试用例来跑，用采样代替全量运行。

这个抽样是确定性的——同一个智能体实例每次跑到的是同一批用例——但在不同的容器之间是随机的。这样做的效果非常好：每个智能体能很快得到反馈，同时多个智能体合在一起仍然覆盖了所有用例。这是一个典型的"单机看起来没优势，集群看效果"的设计。

作者还专门观察了 Claude 的错误模式，然后设计测试来针对性覆盖。比如在项目快收尾的时候，他发现 Claude 频繁出现"新功能实现了，旧功能却坏了"的退化问题，于是他建了一套持续集成流水线，明确要求每次提交不能破坏已经通过的测试，这才遏制住了退化趋势。

这整个过程，说白了就是作者在用自己的判断力不断弥补自动化测试的盲区，同时把这些判断力编码进测试体系，让智能体能够在没有人监督的情况下也能自我校正。

讲完了基础设施，我们来看最有意思的一个故事：编译 Linux 内核遇到的瓶颈，以及作者是怎么化解的。

在测试套件通过率达到99%之后，项目进入了编译真实大型项目的阶段。对于 SQLite、Redis 这类相对独立的小项目，并行化策略很直接：每个智能体拿一个不同的项目去编译，哪里编译不过就修哪里，十六个智能体各自推进，互不干扰。但 Linux 内核是个截然不同的问题。

Linux 内核是一个庞然大物，它不是一堆独立的测试用例，而是一个巨型的整体编译任务。一旦内核编译失败，很可能是因为同一个根本性的 bug——所有十六个智能体都会撞上同一堵墙，然后互相覆盖各自的修复尝试。并行运行十六个 Claude 没有带来十六倍的效率，反而变成了十六个人在抢着解同一道题，然后把彼此的答案擦掉。

你可以想象这个画面：十六个工人都发现了同一道裂缝，每个人都在填，但填完转身另一个人又凿开了。结果大家都累，裂缝还在。

作者的解决方案非常聪明：引入 GCC 作为在线参考编译器。

具体做法是：随机选取内核源文件，大部分用 GCC 编译，少部分用 Claude 自己写的编译器编译，然后把两边的输出拼接起来，看内核能不能正常引导启动。如果能，那说明 Claude 编译的那部分文件没问题；如果不能，就进一步调整两边的比例，缩小 Claude 负责的文件范围，继续排查。

这本质上是一种二分查找。每次实验可以把问题定位到更小的文件集合，让不同的智能体可以并行地针对不同的文件子集做修复，而不是所有人扎堆在同一个地方。并行的优势就这样重新被利用了起来。

作者还提到，即使用了这个方法，有时候单个文件单独编译是没问题的，但跟另一个特定文件组合在一起就会出错。这时候还需要用到"增量调试"技术，也就是系统性地搜索最小的会触发错误的文件组合。整个排查过程既需要工程技巧，也需要对编译器行为有相当深的理解。

除了任务切分之外，并行化还带来了另一个好处：专业化分工。

作者给不同的智能体分配了不同的专项职责。有一个智能体专门负责合并重复代码——因为 AI 生成的代码经常会重复实现同样的功能，这是一个很常见的质量问题。有一个智能体专门优化编译器自身的性能。有一个负责改善生成代码的质量，也就是让编译出来的程序跑得更快。还有一个专门从 Rust 开发者的角度审查整体代码设计，做结构性的改进和重构。另外还有一个负责维护文档。

这种分工模式在人类团队里也很常见——你不可能让同一个人同时写主体功能、审查代码、写文档、做性能调优。在 AI 团队里，这种分工是通过在提示词里指定角色来实现的，启动成本极低，灵活性却很高。

好，现在来聊这个编译器作为模型能力基准测试的一面，这是整篇文章的另一条主线。

作者明确说，这个 C 编译器项目是他用来压力测试 Claude 系列模型的基准。他用相同的任务描述测试了 Claude 四代的多个版本。任务要求很高：从零开始写一个优化编译器，不依赖任何外部库，兼容 GCC，能编译 Linux 内核，支持多后端，还要有 SSA 中间表示——静态单赋值形式，一种常见的编译器中间表示形式，能支持多趟优化Pass，编译器从业者对这个词应该很熟悉。

测试结果很有意思：早期的 Opus 四版本勉强能产出一个能运行的编译器，但几乎没有实际价值。Opus 四点五是第一个跨越关键门槛的版本，能够产出通过大型测试套件的编译器，但还不能编译真实的大型项目。到了 Opus 四点六，才达到了这篇文章描述的效果——两周、近两千次 Claude Code 会话、两亿输入 token、一亿四千万输出 token、将近两万美元。

这个数字你可能会觉得很贵。但作者给了一个有趣的参照系：如果让他自己来写这个编译器，即便他是一个经验丰富的工程师，这也是一个需要数年时间的项目。如果组一个团队来做，成本会远远超过两万美元。从这个角度来看，两万美元换来一个十万行、能编译 Linux 内核的编译器，性价比其实并不差。

还有一个细节值得单独提一下：这是一个完全隔离的实现。Claude 在整个开发过程中没有互联网访问权限，所有代码仅依赖 Rust 标准库。这意味着这个编译器不是"拼接了网上找到的代码片段"，而是从第一性原理出发写出来的。这在证明 AI 真实推理能力方面，比一个能上网参考的实现更有说服力。

好，说完成就，我们来面对现实，聊聊这个编译器的局限。这是文章里我觉得最诚实也最有价值的部分。

第一个局限：缺少16位 x86 编译支持。Linux 内核在启动的时候，需要先在16位实模式下运行一段引导代码，才能切换到保护模式。Claude 写的编译器不能直接产出符合要求的16位实模式代码——虽然它能通过指令前缀输出16位 x86 指令，但生成的代码超过了 Linux 要求的三万两千字节的限制，这个阶段不得不调用 GCC 来完成。对于 ARM 和 RISC-V 就不存在这个问题，那两个架构完全由 Claude 的编译器自行处理。

第二个局限：没有自己的汇编器和链接器。这两个组件是编译工具链的最后一环。Claude 开始尝试实现，但还处于比较早期的阶段，仍然不够稳定。文章里的演示视频是用 GCC 的汇编器和链接器跑的。

第三个局限：生成代码效率偏低。即使开了所有优化选项，Claude 编译器产出的代码效率还比不上 GCC 关掉所有优化时产出的代码。这意味着在代码质量方面还有相当大的差距，离生产可用的标准还很远。

第四个局限：不是所有项目都能编译。它能编译很多项目，但不是全部，还不能作为真实生产环境中 GCC 的直接替代品。

第五个局限：Rust 代码质量有待提高。虽然整体可读性还行，但距离一个有经验的 Rust 程序员的标准还有差距。

作者坦承，这个编译器基本上已经触及了 Opus 四点六的能力上限。他很努力地想修复上面这些问题，但没能完全成功。新功能经常破坏已有功能，而有些功能的复杂度超出了当前模型能够一次性处理的范围。

这里有一个很值得玩味的地方：作者没有刻意美化结果，而是非常直接地描述了哪些地方模型达到了极限。这种态度在 AI 公司的工程博客里并不多见，正是这种诚实让这篇文章格外有价值——它告诉你边界在哪里，而不只是展示能做什么。

好，我们进入对听众最直接有用的部分：这一切对你的实际工作意味着什么？

第一个启发：测试环境是智能体能力的倍增器。

文章里有一句话我印象特别深刻：作者花了大量时间不是在"让 Claude 更聪明"，而是在"设计让 Claude 能够自我导航的环境"。测试套件越好，Claude 就越能自主地做出正确的判断；反之，测试设计得差，Claude 会把十倍的努力投入到错误的方向上。

这对我们使用 AI 辅助开发有很直接的启示。如果你发现 AI 经常走偏，先别急着换模型或者重写提示词，先审视一下你给它的反馈机制是不是足够清晰。清晰的验收标准往往比更好的模型更重要。

第二个启发：从"AI 写代码"到"AI 组建团队"的范式转变。

目前大多数人使用 AI 的方式，是让 AI 完成一个具体的、有边界的任务，然后把结果交还给人类审核，再布置下一个任务。这是一个线性流程，AI 是单个执行者。

这篇文章展示的是另一种可能：AI 可以同时扮演多个角色——有人写主体功能，有人做代码审查，有人维护文档，有人做专项优化。这种分工在一个人类团队里需要好几个工程师，但在 AI 团队里，你只需要设计好角色描述和协作机制。当然，这个范式转变需要一些前提条件：你需要有一个足够好的测试套件能够提供自动化反馈，你需要把任务设计成可以并行化的子问题，你需要有一个版本控制系统作为协作基础。这些东西在专业软件开发里早就是标配，但现在你需要以一种对 AI 友好的方式来组织它们。

第三个启发：专业化分工的低成本高价值。

在人类团队里，让一个人专职做代码审查、另一个人专职做文档，是有成本的——需要招人、需要协调。但在 AI 团队里，你只需要在提示词里加一行"你的角色是从 Rust 开发者的角度审查代码质量"，这个专业化就发生了。这意味着，在你下次做一个比较大的 AI 辅助开发项目的时候，可以考虑拆分角色：主开发 Claude，审查 Claude，文档 Claude，测试 Claude，让它们围绕同一个代码仓库迭代，而不是用一个 Claude 什么都干。

第四个启发：把你对 AI 局限性的认知编码进环境设计。

文章里提到的两个 AI 特有的限制——上下文污染和时间盲症——都是通过测试环境的设计来解决的，而不是通过让 AI 更智能来解决的。这是一个很重要的思维方式：与其期望模型自己解决它不擅长的事情，不如在外部环境里做针对性的补偿。你的 CI 系统可以自动提取错误摘要，让 AI 不需要翻几千行日志；你可以设计"快速预检模式"，让 AI 能在几秒内得到初步反馈，再决定要不要做完整测试。这些都是可以今天就开始做的工程实践。

第五个启发：遇到"全体卡死"的瓶颈，要换测试策略而不是加大马力。

Linux 内核编译的故事告诉我们，当多个智能体都卡在同一个问题上时，增加并行数量没有意义。这时候需要的是重新设计任务的分解方式——引入参照物、做差异比较、找到可以独立验证的子问题。这个思路在人类团队协作中同样适用：当整个团队都阻塞在同一个地方时，加人不是解决方案，重新定义问题才是。

现在来说一个稍微敏感但非常重要的话题：自主开发的安全风险。

作者在文章结尾坦言，他对这个实验的结果感到"既兴奋又不安"。他的背景是渗透测试——专门找别人软件漏洞的那种。所以他比大多数人更清楚地知道，"测试都通过了"并不意味着软件是安全的。

当一个人类工程师全程参与开发，他可以随时发现奇怪的代码、可疑的逻辑、不符合预期的行为。这种实时监督是很多安全问题的最后防线。但在完全自主的开发模式下，工程师只看最终结果，而 AI 会非常努力地确保最终结果看起来符合预期。

这里有一个根本性的张力：AI 越擅长让测试通过，就越擅长"让测试通过但不一定真正解决了问题"。这不是说 AI 会故意作弊，而是任何优化系统都会找到最短路径到达目标，有时候最短路径并不是你真正想要的路径。

作者说，他对"程序员部署自己从未亲自验证过的软件"这件事感到真实的担忧。在他看来，这是自主开发带来的新型风险。这个担忧不仅仅适用于安全领域——在金融系统里，一个逻辑上"正确"但行为异常的程序可能造成真实损失；在医疗系统里，一个通过了所有测试但在边缘案例上行为不当的组件可能危及生命。

我个人认为，自主 AI 开发的安全性问题需要从两个维度同时应对：一是更好的测试哲学——不只是验证正常路径，还要测试异常路径、边界条件、对抗性输入；二是保持人类在关键决策点的介入——让 AI 处理实现细节，但架构选择、安全审查、最终部署仍然要有人类的手动确认。自主不等于无人监督，它意味着人的介入点从"每隔几分钟"变成了"关键节点"。

最后，让我把这篇文章放进一个更宏观的视角来结尾。

作者在文章末尾有一段话，我觉得是全文最值得记住的部分。他说，每一代语言模型都打开了一种新的工作方式。早期的模型能做代码补全；后来的模型能根据文档生成函数体；Claude Code 的出现让"AI 结对编程"变成了主流；而智能体团队则展示了"AI 独立完成整个复杂项目"的可能性。每一步，人类的角色都在发生变化：从写每一行代码，到设计每一个函数，到描述每一个功能，到定义每一个目标。

他还说，他没想到在二零二六年年初这件事就成为了可能。这句话在我看来包含了真实的敬畏——不是出于营销目的，而是出于一个技术人在亲手操作之后的切身感受。

好，让我们来做收尾。

这篇文章的核心信息可以浓缩成三句话。第一，把多个 Claude 实例组成团队来运作，能够突破单个 AI 会话的能力瓶颈，在足够复杂的任务上实现真正的自主迭代。第二，让这件事成功运转的关键不是更聪明的模型，而是更好的测试环境和更周全的任务设计——这是工程师今天就能贡献的地方。第三，这条路充满潜力，但也带来了新的安全挑战，需要我们在追求效率的同时，认真思考如何保持对关键环节的控制。

我个人看完这篇文章，最强烈的感受是：我们正处在一个方法论和工具都还在快速演化的阶段。今天你读到的这些技巧——任务文件锁、上下文污染控制、采样测试——半年后可能就有更成熟的框架把它们封装好了。但思维方式的转变——从"AI 帮你写代码"到"AI 团队替你做项目"——这个转变一旦发生，就很难再回头了。
