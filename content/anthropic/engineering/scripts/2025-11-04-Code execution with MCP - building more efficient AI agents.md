在正式讲内容之前，我先帮大家做一个时间定位。今天是二〇二六年六月，距离这篇文章发表已经过去了将近八个月。在 AI 这个领域，八个月可以发生很多事。所以我觉得有必要先交代一下时代背景，帮大家理解这篇文章是在什么样的阶段被写出来的。

MCP，也就是模型上下文协议，是 Anthropic 在二〇二四年十一月正式开源发布的。到这篇文章写作的二〇二五年十一月，MCP 刚满一周岁。在这一年里，社区以非常快的速度建起了数以千计的 MCP 服务器，Python、TypeScript、Java、Go 等各大编程语言都有了官方 SDK，各大 AI 公司和开发工具厂商也纷纷宣布支持这个协议。可以说，MCP 在短短一年内就成了事实上的行业标准。

这篇文章描述的，是工程师们在大规模部署 MCP 之后遇到的一个现实困境：工具越接越多，代理越来越慢，成本越来越高。文章提出了一个用代码执行模式来破解这个困境的方案。虽然今天距离文章发表已经过了七个多月，但这个问题在今天依然非常普遍，甚至随着 MCP 生态继续壮大，问题变得更加突出了。所以这篇文章读来依然非常有实操价值，甚至比当时更有参考意义。

好，背景交代完，我们进入正题。

我们先用几分钟把 MCP 的基本概念说清楚，因为理解了这个基础，后面的问题和解法才能看得透彻。

MCP 解决的核心问题是什么？用一句话来说：它提供了一个统一的"插头规格"，让 AI 代理可以方便地对接各种外部工具和数据源，而不需要为每一种对接单独写一套适配代码。

你可以用充电接口来做类比。在 USB-C 普及之前，不同品牌的手机用不同的充电接口，苹果用 Lightning，安卓用 Micro-USB，各种设备互不兼容，非常麻烦。USB-C 出来之后，只要有一根线，大部分设备都能充电。MCP 在 AI 代理和外部工具之间做的事情，跟这个很相似。

在 MCP 之前，如果你要让自己的 AI 代理能用 Google Drive，你得自己写对接 Google Drive API 的代码；要用 Salesforce，再写一套 Salesforce 的；要用 Slack、GitHub、Notion……每一对组合都得单独处理。这个工作量是乘法的，随着工具数量增加，开发和维护成本急剧上升。

有了 MCP，工具方只需要按照协议写好自己的 MCP 服务器，代理方只需要实现一次 MCP 客户端，两边就能自动对接。这是一个非常聪明的解耦设计，这也是 MCP 能在一年内迅速被整个行业采纳的根本原因。

好，MCP 的基础概念说完了。我们来看它在规模化使用时碰到的问题。

文章归纳出两个核心痛点。第一个是工具定义把上下文窗口撑爆了，第二个是中间数据反复流过模型，消耗额外的 token。我们一个一个来讲清楚。

先说第一个问题，工具定义的 token 爆炸。

大部分 MCP 客户端的实现方式是这样的：在代理开始回答用户的问题之前，先把所有已经连接的 MCP 服务器上的所有工具定义，一股脑儿全部加载进模型的上下文窗口。每一个工具的定义里通常包括工具名、功能描述、每个参数的名字、数据类型、是否必填、返回值的格式……大概每个工具要占几百个 token。

文章举了两个典型工具定义的例子，一个是 Google Drive 的文档获取工具，一个是 Salesforce 的记录更新工具。光是这两个工具的定义写下来，就需要相当篇幅的文字说明。现在你可以想象一下，如果你连接了三四十个 MCP 服务器，每个服务器有五六十个工具，那工具定义的数量就达到了几千个。

文章给出了一个非常具体的数字：光是把工具定义塞进上下文，就要消耗十五万 token。这意味着什么呢？模型在真正处理用户的请求之前，要先"读"完这十五万 token 的工具定义，这直接拉高了每次对话的延迟和成本。而且，这十五万 token 里的绝大部分，对于当前这个具体任务来说根本用不上。你让代理帮你整理一份 Google Sheet，它却要先把 Salesforce 的、Slack 的、GitHub 的、数据库的、还有七七八八其他服务的工具定义都处理一遍，然后才能开始干活。

这就好比你去餐厅点餐，服务员在你点菜之前，先把整本菜单从头念到尾，连你永远不会点的菜，什么松露鱼子酱、什么鹅肝慕斯，全部念一遍。你只是想点一碗番茄蛋花汤，但你得先等服务员把整本菜单念完。这种体验非常糟糕，而且是纯粹的浪费。MCP 大规模使用时的工具定义问题，本质上就是这样一个局面。

再来说第二个问题，中间数据反复流过模型。

文章用了一个非常具体的场景来说明：你让代理帮你把 Google Drive 里的一份销售会议记录下载下来，然后把内容附到 Salesforce 里对应的跟进记录里。这个任务听起来很平常，对吧？

但是按照传统的直接工具调用模式，实际发生的事情是这样的。代理首先调用 Google Drive 的文档获取工具，把完整的会议记录文本拿下来，这整段文本进入模型的上下文窗口。然后代理要调用 Salesforce 的记录更新工具，它需要把会议记录的内容作为参数传进去，也就是说，这段文本得再经过模型一次。

如果这是一个两小时的销售会议，全程录音转写成文字，文本量可以非常大。文章估算说，光这一个文档就可能额外消耗五万个 token，而且这五万 token 要在上下文里出现两次。

更糟糕的是，如果文档足够大，超过了模型的上下文窗口的上限，整个工作流就直接断掉了。没有任何回旋余地，工作流崩溃，任务失败。

文章还点出了一个容易被忽视的风险：当模型需要手动把大段数据从一个工具调用搬到另一个工具调用的时候，出错的概率会上升。这个道理其实很直觉，人类大段文字复制粘贴也会出错，而且很难被检查到。模型在这方面同样有局限，而且出错了可能更难发现。

好，两个问题都说清楚了。我们来看文章提出的核心解法。

文章提出的思路是：不要让代理直接调用工具，而是让代理写代码，通过代码来调用 MCP 服务器。这个转变听起来只是换了一种方式，但背后的影响其实是全面的。让我从最基础的地方来解释。

当代理的工作方式是"写代码"的时候，它可以做到按需加载工具定义。具体怎么实现？文章给出了一个 TypeScript 的实现思路：把所有 MCP 服务器上的工具，以文件树的形式呈现给代理。最顶层是一个叫 servers 的目录，里面按服务器名称分了子目录，比如 google-drive、salesforce。每个子目录里，每个工具对应一个独立的文件，比如 getDocument 对应 Google Drive 的文档获取工具，updateRecord 对应 Salesforce 的记录更新工具。每个文件里有清晰的接口定义，告诉代理这个工具接受什么输入、返回什么输出，以及一句简短的注释说明这个工具做什么。

代理的工作方式就变成了：先列出 servers 目录下有哪些服务器，根据当前任务，只读取自己需要的那几个工具文件，搞清楚接口之后，开始写代码调用这些工具。

这就像是代码里的按需加载，类似于程序启动时的懒加载或者动态 import。你不需要在程序启动的时候就把所有模块全加载进内存，只在真正用到的时候才去加载。代理面对几千个工具的时候，可以只加载当前任务需要的那十几个工具的定义，其他的根本不看。

回到那个 Google Drive 到 Salesforce 的例子，用代码方式来写，就是几行：导入 google-drive 模块，导入 salesforce 模块，调用 getDocument 拿到会议记录文本，存进一个变量，然后调用 updateRecord 把这个变量传进去。整个过程，会议记录文本在代码执行环境里流转，从 Google Drive 出来直接进入 Salesforce，全程没有经过模型。

文章给出的数字非常惊人：从十五万 token 减少到两千 token，节省了百分之九十八点七。将近百分之九十九的 token 消耗被消掉了。这不是小幅优化，这是数量级的差异。

按需加载的具体实现方式有两种。一种是刚才说的文件系统方式，代理通过列目录、读文件来按需获取工具定义。另一种是给代理提供一个叫 search tools 的搜索工具，代理用关键词搜索，比如搜索"salesforce"，只拿到跟 Salesforce 相关的工具定义。这个搜索工具还可以支持不同的详细程度：只要工具名、要工具名加描述、或者要带完整参数定义的全量信息。代理根据当前需要选择合适的详细程度，进一步控制 token 用量。

我们继续看代码执行模式带来的其他好处。

第二个好处是数据过滤可以在执行环境里完成，而不是在模型上下文里完成。

设想你需要处理一个有一万行数据的 Google Sheet。传统模式下，代理调用工具把全部一万行拿下来，全部进入模型上下文，然后在这个庞大的上下文里进行筛选。一万行数据占用了大量 token，模型要在这个臃肿的上下文里工作，既慢又贵。

用代码模式，代理可以把一万行数据全拉下来，然后用一行 filter 函数，只保留状态是"待处理"的那些订单，再打印前五条给模型看。模型上下文里出现的是五行数据，而不是一万行。绝大部分数据在代码执行环境里完成了处理，从未进入模型的上下文。

这个模式可以延伸到非常多的场景：数据聚合、多个数据源的关联查询、从复杂结构里提取特定字段……所有在代码里一行能搞定的数据处理逻辑，都不需要再让模型在上下文里一行一行地处理了。而且代码处理数据，远比模型在上下文里"手动"搬数据更可靠，出错的概率要低得多。

第三个好处是复杂逻辑可以在代码里一步到位，不需要在工具调用和模型之间反复切换。

文章举了一个等待部署完成通知的例子。代理可以在代码里写一个 while 循环：每五秒查一次 Slack 频道的消息历史，看有没有包含"deployment complete"字样的消息，有了就停下来输出结果，没有就继续等。这个轮询逻辑完全在代码层面执行，模型不需要参与每一轮的判断。

如果用传统的直接工具调用模式来做这件事，每一次检查都需要模型介入，生成下一步指令，等待工具返回结果，再由模型决定要不要继续等，每一轮都需要时间和 token。相比之下，代码执行模式的优势非常明显。

文章还提到了一个比较细节但很有意思的点：代码执行还能减少"首 token 延迟"。在传统模式下，代理要评估一个条件分支，需要等模型生成下一个 token 来表达判断结果，这本身就有等待时间。但如果代理把整个条件树写成代码，交给代码执行环境来处理，就跳过了这段等待。对于有很多条件分支的复杂流程，这个积累起来的延迟优化是相当可观的。

第四个好处，也是我个人认为在工程实践上被低估最多的一个：安全性和数据隐私保护。

代码执行模式下，中间结果默认待在执行环境里，不会自动流进模型的上下文窗口。模型只能看到你明确打印或者返回的内容。这意味着，如果你的工作流里有某些数据你不希望模型看到，这些数据可以静默地流过整个工作流，却从未进入模型的视野。

文章进一步描述了一个更高级的机制，叫做 PII 自动令牌化，这里的令牌化是数据脱敏领域的概念，指把真实数据替换成无意义的占位符。PII 是"个人可识别信息"的缩写，具体就是姓名、邮箱、手机号这类敏感数据。

文章给了这样一个例子：你要把 Google Sheet 里的客户名单，包括姓名、邮箱、手机号，导入到 Salesforce 里。代理写的代码会遍历每一行，把对应数据传给 Salesforce 的更新接口。在这个过程中，MCP 客户端可以拦截数据流，在数据进入模型上下文之前，把真实邮箱替换成 EMAIL 加数字这样的占位符，手机号替换成 PHONE 加数字，姓名替换成 NAME 加数字。

如果代理去打印这些数据，它看到的是这些占位符，而不是真实的个人信息。当这些占位符被传递给下一个 MCP 工具调用的时候，MCP 客户端自动把它们还原成真实数据。

最终结果是：真实的客户数据从 Google Sheets 流到了 Salesforce，但全程没有经过模型。模型不知道这些数据是什么，也就无法意外地把它们写进日志、用在不该用的地方，或者在错误信息里暴露出来。

这个机制还可以进一步扩展为一套确定性的安全规则，你可以定义哪种类型的数据可以从哪个数据源流向哪个目的地，哪些不行。对于处理医疗数据、金融数据、用户隐私信息的应用来说，这是在架构层面解决合规问题的有力工具，而不是靠事后审计来打补丁。

第五个好处是状态管理和技能积累。

代码执行加上文件系统访问，让代理可以把中间结果写入文件，在下次运行的时候读取并继续。这解决了代理"短暂记忆"的问题：不是每次都从头开始，可以接着上次的进度走。文章举的例子是，代理从 Salesforce 查出一千条线索数据，转成 CSV 格式写入文件，下次继续处理时直接读文件，不需要重新查询数据库。

更有意思的是技能积累这个概念。文章描述了这样一个过程：代理在完成某个任务的过程中，写出了一段好用的代码，可以把这段代码保存成一个可复用的函数，放在 skills 目录里。比如"把 Google Sheet 另存为 CSV"这个操作，代理做过一次之后，就可以把这个功能封装成一个函数，下次遇到类似需求，直接调用这个技能，不需要重新推导逻辑。

如果再给这些技能函数加上一个 SKILL.md 的说明文档，就形成了一套结构化的技能库，模型可以查阅这个文档，了解自己手头有哪些高级能力可以用，然后选择合适的技能来处理新任务。

文章把这个过程描述为代理在"进化自己需要的脚手架"，随着时间推移，代理积累的技能越来越多，处理复杂任务的能力也越来越强，而且这些能力是可以持久化的，不会因为对话结束就消失。这和人类工程师积累自己的工具库、脚本库，是同一个道理。

当然，文章也非常坦诚地指出了代码执行模式的代价，没有回避问题。

运行代理生成的代码，需要一个安全的执行环境。你需要做好沙箱隔离，确保代理的代码不能影响到宿主系统；需要设置资源限制，防止代码无限制地消耗 CPU 或内存；还需要有监控和日志，追踪代码执行的情况并能在出问题时快速定位。这些基础设施建起来需要工程投入，运维起来也比直接工具调用复杂得多。

所以文章的建议是，要把代码执行带来的好处和这些实现成本放在一起权衡，根据自己的实际规模和场景做决定。代码执行模式不是万能药，在工具数量少、场景简单的情况下，直接工具调用依然是更低成本的选择。

顺带一提，文章里提到 Cloudflare 也独立探索出了相同的路径，他们把这个模式叫做 Code Mode，核心洞察和 Anthropic 的这篇文章完全一致。我个人认为，能在多个团队独立验证的技术路径，通常是值得信任的。这说明这个方向是工程师在实践中自然发现的解法，而不是某个实验室拍脑袋想出来的概念。

好，核心内容都说完了。我们来聊聊对你实际工作的意义。

如果你现在正在做代理应用的开发，或者正在评估怎么大规模使用 MCP，我建议你先评估一下自己当前的工具规模。如果你的代理连接的 MCP 服务器不超过五个，工具总数不超过几十个，传统的直接工具调用可能已经够用，不需要引入代码执行的额外复杂性。但如果你已经在连接几十个服务器、工具数量上了几百个，或者你的工作流里有大量数据需要在工具之间传递，那这篇文章提出的方案就非常值得认真对待。

具体怎么起步？文章给的文件树方案是一个很好的切入点。你可以先把现有的 MCP 工具整理成文件树结构，让代理通过读文件的方式按需加载工具定义。这一步改动本身并不复杂，但能带来非常显著的 token 节省。如果你的场景涉及大量数据处理，再进一步把过滤和变换逻辑下沉到代码里，让模型只看处理后的结果而不是原始数据。

有几个坑要特别提醒。第一，不要低估安全执行环境的建设难度。代理生成的代码在执行之前需要经过合理的审查和限制，特别是涉及文件系统写入、网络请求、系统调用等高权限操作的时候。沙箱设计做不好，可能会带来比你预期更大的安全风险。这不是说说而已，代理生成的代码是用户意图和模型推理的混合产物，和你手写的受控代码有本质差异，需要认真对待。

第二，如果你的应用场景涉及用户的敏感数据，比如 HR 系统、客户 CRM、医疗数据、金融记录，那 PII 令牌化这个机制是真的值得深入研究和实现的。让敏感数据不经过模型，不仅是技术优化，更是合规要求的基本线。现在国内外对 AI 应用处理个人信息的监管都越来越严，在架构层面就把这个问题解决掉，比事后补丁要省事得多。

第三，技能积累这个设计在长期运行的代理系统里非常值得投入，但很多团队在快速迭代的压力下容易忽视这一块。如果你在做的是一个长期运行的、会处理大量不同任务的代理系统，认真设计一套技能沉淀和复用的机制，会让你的系统越用越强，而不是每次面对新任务都要重新发明轮子。

最后我们来做个总结。

这篇文章的核心观察是：MCP 让代理可以方便地对接很多工具，这是好事，但当工具数量一多，传统的"全部预加载、直接调用"模式就会遇到 token 爆炸和数据反复流过模型的问题。解法是换一种思路，让代理以写代码的方式来使用 MCP 服务器，通过代码执行环境来处理数据，通过文件系统来按需加载工具定义。这样可以把 token 消耗降低九成九，减少延迟，同时带来更好的数据隐私保护和状态管理能力。代价是需要投入安全的代码执行基础设施，需要根据实际规模权衡是否值得。

我个人认为，这篇文章最有价值的地方在于它改变了一个思维框架：AI 代理不应该被视为一个只能按预定工具菜单操作的机器人，而应该被视为一个会写代码的工程师。让代理用代码来表达逻辑，利用代码执行环境来处理数据，不仅效率更高，也更接近软件系统真正工作的方式。文章最后说得很好：上下文管理、工具组合、状态持久化，这些在代理时代看似新颖的问题，在软件工程里都有成熟的解法，代码执行模式正是把这些已有的模式带进了代理架构。
