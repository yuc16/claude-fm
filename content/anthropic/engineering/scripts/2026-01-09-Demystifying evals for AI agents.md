## Script

今天要聊的这篇文章是 Anthropic 工程团队发表在官方工程博客上的一篇重磅分享，英文标题是 Demystifying Evals for AI Agents，我把它翻译成"AI 智能体评测揭秘"。这篇文章发表于二〇二六年一月九日，距今约五个月时间。

为什么我用"重磅"这个词？因为 Anthropic 在这篇文章里做的事不是宣传某个新模型，而是把他们在 agent 评测这件事上踩过的坑、总结出来的方法论，系统性地整理出来对外分享。对于正在做 AI 产品的工程师来说，这是非常罕见的第一手工程经验，很多内容你在教科书里是找不到的。

你可能会问，eval 这个东西有什么好专门讲的？不就是写测试吗？但如果你真的做过 AI agent，你会知道这件事远比听起来难得多。传统软件测试面对的是确定性系统，你输入 A，它永远返回 B。但 AI agent 是概率性的，而且它跨越多个步骤执行任务，前面的每一个决策都会影响后面的走向，就像下一局棋，每一步都在改变棋盘上的局面。这种特性让评测变得非常复杂，也让大多数团队在这个问题上要么完全不做，要么做了但做得很不靠谱。

这期节目我会带你完整走一遍 Anthropic 在这篇文章里分享的内容：从最基础的术语定义，到按 agent 类型分的评测策略，再到从零开始搭建评测体系的八个步骤。听完你应该能带走一套可以立刻开始用的思路框架。

在讲具体内容之前，我想先说一件事：很多团队在 agent 开发早期其实可以完全不靠 eval 走得很远。靠手动测试、靠团队内部先用、靠直觉，是完全可以完成早期快速迭代的。而且这个阶段如果花大量时间建 eval 体系，感觉像是在拖慢速度，团队会觉得这是过度工程，没有必要。

但这篇文章说，这种状态是有一个明确的"崩溃点"的。这个崩溃点通常出现在 agent 上了生产、开始有真实用户的时候。问题的表现是这样的：用户开始反馈 agent "感觉比之前差了"，但团队没有办法核实，只能猜测。调试模式变成了等用户投诉、手动复现问题、改一个 bug、然后祈祷没有其他地方出问题。团队没法区分真正的退步和正常的随机噪音，没法在上线之前自动化地跑几百个场景来验证，也没法量化改进效果。文章把这个状态叫"飞盲"，没有仪表盘，凭感觉开飞机。

文章里举了 Claude Code 自己的例子。Claude Code 最早也是靠 Anthropic 内部员工反馈和外部用户反馈快速迭代，后来才逐步加上了 eval，先是针对简洁性和文件编辑等较窄的方向，再扩展到过度工程化这类更复杂的行为评测。这些 eval 帮他们识别问题、指导改进，也让产品团队和研究团队之间有了更清晰的沟通语言，研究员知道该优化什么指标。

另一个值得关注的点是：eval 会直接影响你采用新模型的速度。文章说，没有 eval 的团队，每次新模型出来都要花几周时间做手动测试才能决定要不要升级；而有 eval 的团队，几天之内就能跑出新模型的强弱项，调好 prompt，完成升级。这个效率差在模型迭代越来越快的今天，意味着很大的竞争优势差距。

好，现在进入正题，先把文章里的核心术语一次讲清楚，这些词后面会反复出现，我不希望你在听的时候被卡住。

任务，英文叫 task，也叫 problem 或者 test case，就是一个具体的测试用例，有明确的输入和成功标准。比如"帮用户退款并发送确认邮件"就是一个任务。

试次，英文叫 trial，就是对一个任务的一次执行。因为 AI 输出有随机性，同一个任务通常要运行多次试次，才能得到统计上可靠的结论。

评分器，英文叫 grader，是用来打分的逻辑单元。一个任务可以有多个评分器，每个评分器里可以包含多个断言，也就是多个判断条件。

执行记录，英文叫 transcript，也叫 trace 或者 trajectory，是一次试次的完整记录，包括模型的所有输出、工具调用、中间推理过程。对于 Anthropic API 来说，这就是 eval 运行结束时完整的消息数组，包含所有 API 调用和所有返回的响应内容。

执行结果，英文叫 outcome，是任务完成后环境的最终状态。这跟模型说了什么不是一回事。举个例子：一个预订机票的 agent，模型最后一句话可能是"您的航班已预订"，但真正的执行结果是看 SQL 数据库里有没有真的存在一条预订记录。模型说了什么是 transcript 里的内容，任务是否真正完成才是 outcome。这个区别非常重要，因为很多问题就藏在这个缝隙里——模型说了正确的话，但实际上什么都没做成。

还有两个 harness 的概念，harness 这个词可以理解为"框架"或者"运行脚手架"。eval harness 是运行 eval 的整套基础设施，负责提供任务指令和工具、并发运行多个任务、记录所有步骤、打分、汇总结果。agent harness 则是让模型能以 agent 身份工作的系统，负责处理输入、协调工具调用，然后返回结果。当我们说在评测"一个 agent"时，我们实际上是在评测 agent harness 加上模型一起协作的效果。文章特别说明，Claude Code 本身就是一个灵活的 agent harness，Anthropic 基于 Agent SDK 的核心原语构建了他们内部用于长时间运行的 agent harness。

最后，评测套件，英文叫 evaluation suite，是一组任务的集合，用来衡量特定的能力或行为。比如客服 agent 的评测套件里可能包括退款处理、取消订单、投诉升级这几类任务。

好，术语讲完，我们来看文章里最核心的技术内容：评分器的三大类型。

文章把评分器分成基于代码的、基于模型的、基于人工的三大类。

先说基于代码的评分器。这类方法包括：精确字符串匹配或者正则表达式匹配，二元测试——比如原来失败的测试现在是否通过了——静态分析包括代码规范检查、类型检查、安全扫描，结果验证也就是检查环境最终状态，以及工具调用验证检查 agent 调用了哪些工具、传了什么参数，还有对 transcript 的分析比如总共用了几轮对话、消耗了多少 token。代码评分器的优点很明显：快、便宜、客观、可复现，而且容易调试。缺点是太死板，如果 agent 找到了一个合法但你没有预料到的解法，可能会被误判为失败，而且对于需要主观判断的任务也没法捕捉那些细微的差别。

再说基于模型的评分器，也就是用另一个大语言模型来打分，业内通常叫做 LLM-as-judge。方法包括：基于评分标准的打分、自然语言断言、两两比较、基于参考答案的评测，还有多个模型联合评判取得共识。这类方法的优点是灵活，能够处理开放式任务，能捕捉细微差别，可以大规模运行。缺点是有不确定性，比代码评分器贵，而且需要和人工评分结果对比校准，才能确保它真的在评估你想评估的东西。

最后是人工评分器，包括领域专家评审、众包评判、抽样检查、A/B 测试。人工评分是黄金标准，质量最高，最接近真实用户的判断，也是校准模型评分器的参考基准。缺点是贵、慢，在复杂专业领域想大规模做人工评测，访问到足够多的专家本身就是一个难题。

实际使用中，这三类方法通常是组合起来用的。对于每个任务的评分，可以是加权模式——多个评分器的加权总分要超过某个阈值才算通过；也可以是二元模式——所有评分器都必须通过；或者两者的混合。

好，讲完评分器，文章接下来按 agent 类型讲了四种主要类型各自的评测策略。这部分非常实用，我来逐一拆解。

第一类是编程 agent，也就是写代码、调试 bug、运行命令的那种，工作方式类似一个人类开发者。这类 agent 的评测天然适合确定性评分器，因为软件测试本来就是对错分明的：代码能不能跑、测试用例能不能通过。

文章提到了两个被广泛使用的编程 agent 基准测试。一个是 SWE-bench Verified，它把真实 GitHub 上的 Python 项目 issue 交给 agent 解决，通过运行测试套件来评分，一个解法只有同时修好了原本失败的测试、又没有破坏已有的通过测试，才算成功。文章说过去一年里，大语言模型在这个基准上的得分从百分之四十提升到了百分之八十以上，进步非常显著。另一个是 Terminal-Bench，测的是端到端的技术任务，比如从源码编译 Linux 内核，或者从头训练一个机器学习模型，难度更高，也更贴近真实工程场景。

文章还特别强调，除了检查最终结果，还值得评估 agent 的执行过程，也就是 transcript 本身。比如用代码规范检查工具评估生成代码的风格质量，或者用模型评分器评估 agent 和用户交互的方式是否合适。结果好不代表过程好，两者都值得关注。

第二类是对话 agent，也就是在客服、销售、辅导等场景里跟用户持续互动的那种 agent。这类 agent 面临一个独特的挑战：交互本身的质量是评测对象的一部分，不只是最终有没有完成任务，而是整个对话体验好不好。

对话 agent 的评测通常需要另一个大语言模型来扮演用户，和 agent 进行模拟对话。文章说 Anthropic 内部也用这个思路做对齐审计，让一个模型扮演挑剔甚至对抗性的用户，来压力测试模型的行为边界，看模型在长时间、高压力的对话里会不会失守。

成功标准是多维度的。文章举了个具体例子：一个客服 agent 处理退款请求，要同时满足三个条件——工单状态要变为已解决，整个交互要在十轮以内完成，以及对话语气要体现出对客户的理解和关怀。三个维度都要通过，才算是真正成功的处理。文章提到 tau-Bench 和 tau2-Bench 这两个基准测试就采用了这种多维度的评测框架，涵盖零售客服、航空订票等场景，用一个模型扮演用户角色，让 agent 在真实的多轮对话里完成任务。

第三类是研究 agent，负责收集信息、综合分析、产出报告的那种。与编程 agent 不同，研究质量没有简单的对错之分。什么叫"全面"，什么叫"来源可靠"，这些标准都取决于具体的任务场景：一个市场调研、一个企业并购的尽职调查、一个科学综述，对质量的要求是完全不同的。

评测研究 agent 有几个特有的挑战。专家之间对"综合分析是否全面"可能有不同看法；参考内容会随时间变化，导致基准答案失效；输出越长越开放，犯错的空间也越大。

文章推荐的策略是组合多种评分器。Groundedness 检查验证每一个声明都有来源支撑，Coverage 检查确认关键事实有没有被涵盖，来源质量检查确认参考了权威来源而不是随手找到的第一条搜索结果，然后用 LLM 评分器检查综合是否连贯完整、有没有逻辑漏洞。而且，由于研究任务的主观性，LLM 评分器需要比其他类型更频繁地和专家人工判断进行对齐校准。

第四类是计算机操作 agent，就是通过截图、鼠标点击、键盘输入来操控图形界面软件的那种 agent，而不是通过 API 或者代码执行来与系统交互。评测这类 agent 需要在真实或沙箱环境中实际运行，然后检查是否达成了预期结果。

文章提到了 WebArena 和 OSWorld 两个基准测试。WebArena 测的是浏览器任务，通过检查 URL 和页面状态来验证 agent 是否正确导航，同时还会检查后端状态，确认操作真的生效了，而不只是显示了一个确认页面。OSWorld 扩展到了完整操作系统控制，评测脚本会检查文件系统状态、应用配置文件、数据库内容和界面元素属性。

Anthropic 在 Chrome 插件产品里做了一个值得参考的优化：他们专门开发了 eval 来判断 agent 应该用 DOM 方式还是截图方式来完成不同的任务。DOM 方式执行快但消耗大量 token，截图方式相对慢但 token 效率更高。比如让 Claude 总结维基百科一篇文章，从 DOM 里提取文本更高效；但让 Claude 在亚马逊上找一个新款笔记本电脑外壳，截图反而更合适，因为提取完整 DOM 的代价太高了。选对了才能同时保证速度和准确率，这就是为什么有时候看似细节的技术决策，也值得用 eval 去量化验证的原因。

讲完四种 agent 类型，文章引入了两个量化 agent 稳定性的指标，这是我觉得最容易被忽视、但在工程实践上非常重要的部分。

由于 AI 输出有随机性，同一个任务今天跑可能通过，明天跑可能失败，所以"这个任务通过了还是没通过"这个问题没有那么简单，你需要一种方式来刻画这种不确定性。

第一个指标是 pass@k，读作"pass at k"。它的含义是：在 k 次尝试里，至少有一次成功的概率是多少。直觉上，k 越大这个分数越高，因为你有更多机会命中目标。pass@1 的百分之五十意味着在整个 eval 套件里，模型第一次就能完成的任务占一半。在编程场景里，我们通常最关注 pass@1，因为用户希望一次就得到正确答案。但在其他场景下，比如一个工具可以生成多个方案让用户挑选，那只要有一个成功就算有价值，这时候 pass@k 就是合适的指标。

第二个指标是 pass^k，读作"pass power k"，也就是 pass 的 k 次方。它的含义是：k 次尝试里，全部都成功的概率是多少。k 越大这个分数越低，因为要求每次都成功是更高的标准。文章举了一个非常直观的数字：如果 agent 每次试次的成功率是百分之七十五，运行三次全部成功的概率就是零点七五的三次方，大概是百分之四十二。

注意，一个"感觉还不错"的百分之七十五成功率，转换到连续三次都要成功的要求上，直接变成了不足一半的概率。如果你的 agent 是面向真实用户的服务，用户期望每次都能可靠地完成任务，那这个数字可能完全不符合产品要求，而你之前看平均通过率时根本没有意识到这一点。

选哪个指标取决于你的产品定位：pass@k 适合允许多次尝试的工具，pass^k 适合要求稳定可靠的用户服务。文章说，很多团队只看平均通过率，不区分这两个维度，容易对自己的 agent 产生错误的乐观判断。

好，现在来讲文章里最干货的部分：从零到有，怎么建立一套你能真正信任的评测体系。文章给出了八个步骤，我来逐一拆解，每一步都有它背后的逻辑。

第零步，尽早开始。文章说很多团队会拖延，觉得要等到有几百个测试用例才值得搭这套东西。但实际上，从真实失败案例中提炼出来的二十到五十个测试用例，就是一个很好的起点。在 agent 开发早期，每一次系统改动往往有非常明显的效果，这种"大效果尺寸"意味着小样本就够用了。越到后期，agent 越成熟，才需要更大、更难的 eval 来检测更细微的变化。而且还有一个现实问题：越晚建 eval 越难建，因为早期阶段产品要求自然而然就能转化成测试用例，等你 agent 已经在生产环境跑了很久，你要从一个活的系统里反向工程出成功标准，难度大多了。

第一步，从你已经手动测试的东西开始。你在开发过程中每次发版前会手动检查哪些行为？用户最常做的任务是什么？如果你已经在生产，就去翻 bug 记录和用户支持队列，把用户报的失败案例直接转化成测试用例，按用户影响程度排优先级。这样你的评测套件一开始就是基于真实使用场景的，而不是你凭空想象出来的场景。每一个用户投诉其实都是一个你还没有覆盖的场景，修了这个 bug 之后，加一个用例保证它不会再复发。

第二步，写清晰无歧义的任务，配上参考答案。怎么判断一个任务写得好不好？文章给了一个简洁的标准：两个领域专家独立看这个任务，应该能对"通过还是失败"达成相同的判断。如果他们有分歧，说明任务描述还不够清晰。任务描述的模糊性会直接变成指标里的噪音，模型评分器的评分标准含糊同样会导致判断不一致。

文章还特别提醒了一个细节：如果一个任务在大量试次里通过率都是零，比如 pass@100 为零，这通常不是 agent 太差，而是任务本身写错了。他们审计 Terminal-Bench 时发现，有些任务让 agent 写一个脚本但没指定路径，而评分器却假设了特定路径，于是 agent 用自己认为正确的方式完成任务，却被误判为失败。这种问题会让你以为模型表现很差，但其实是测试设计本身有问题。

每个任务还应该配一个参考答案，也就是一个已知能通过所有评分器的正确解法，用来证明这个任务是可解的，同时验证评分器的配置是正确的。

第三步，构建平衡的问题集。不只测应该发生的情况，也要测不应该发生的情况。单边的测试会导致单边的优化。文章举了 Claude.ai 里网络搜索功能的例子：团队需要防止模型在不该搜索的时候搜索，同时又要保留它在该研究的时候积极搜索的能力。如果只测"模型应该搜索的场景"，可能最后优化出一个什么都要搜的 agent；如果只测"不该搜的场景"，可能优化出一个从不搜索的 agent。两个方向都要测，而且具体比例的平衡需要多轮迭代才能找对，文章说他们在这件事上经过了很多轮的调整。

第四步，搭建稳健的评测基础设施和稳定的环境。agent 在 eval 里的工作方式要跟生产环境基本一致。每次试次都要从干净的环境开始，避免不同试次之间共享状态——比如残留的文件、缓存的数据、被耗尽的资源。文章提到，他们内部的一些 eval 里，Claude 能通过看上一次试次遗留的 git 历史来获得不公平的信息优势，这就是环境隔离做得不够的典型案例。如果多个试次因为同一个基础设施限制失败，比如内存不足，这些试次就不是独立的，eval 结果也就失去了对 agent 性能的指示意义，你看到的失败可能是基础设施的问题而不是模型的问题。

第五步，用心设计评分器。文章有几条具体建议。第一，能用确定性评分器就用，不行再上 LLM 评分器。第二，不要过度检查 agent 走的路径，应该评分 agent 产出的结果，而不是它具体调用工具的顺序，因为 agent 经常会找到你没预料到的合法解法，太死板的路径检查会惩罚这种创造性，而这种创造性往往正是你想要的。第三，对于有多个组件的任务，要设计部分得分，而不是全有全无。一个客服 agent 正确识别了问题并验证了用户身份但没有处理退款，这比什么都没做要好，这个差距应该体现在分数里，让你知道进展到哪一步了。第四，LLM 评分器要经过人工校准，而且要给它一个逃生口，比如当信息不足时返回"未知"，避免它因为信息不够就随便给一个结论。

文章在这里分享了两个特别说明问题的案例，我觉得非常值得展开讲。

Opus 四点五在 CORE-Bench 基准测试上最初只拿了百分之四十二的分数，直到一名 Anthropic 研究员深挖之后发现了好几个问题。评分器过于死板，agent 回答九十六点一二被判错，因为期望的答案是九十六点一二四九九一这样更精确的数字；有些任务描述存在歧义；还有一些随机任务根本无法精确复现，每次跑结果都不一样。修复了这些问题并用了更宽松的 scaffold 之后，同样的模型得分从百分之四十二跳到了百分之九十五，整整五十三个百分点的差距，全来自评测系统本身的问题而不是模型的问题。

另一个案例来自 METR 的时间跨度基准测试。有一些任务要求 agent 把某个指标优化到某个目标值，但评分器的实际配置是要求超过那个目标值，而不是达到。结果，严格按照指令行事的 Claude 被扣分，而忽略指令直接最大化的模型反而得分更高。这种评测设计的缺陷会给你一个完全相反的结论，让你以为行为越不守规矩的模型表现越好。这两个案例都说明了同一件事：在你相信 eval 分数之前，你需要深入读懂这个 eval 本身。

第六步，读执行记录。文章说得非常直接：除非你读 transcript，否则你根本不知道你的评分器是不是在正确评估你想评估的东西。当一个任务失败了，transcript 能告诉你是 agent 真的犯了错，还是评分器拒绝了一个实际上有效的解法。失败案例应该让你感觉"公平"，就是看了之后清楚地知道 agent 哪里做错了，而不是感觉"评分器判错了"。

Anthropic 内部专门投资了查看 eval transcript 的工具，团队成员会定期花时间阅读它们。这不是一个可选项，而是 agent 开发的核心技能。文章说，transcript 是你验证 eval 在测量真正重要的东西的唯一方式。

第七步，监控能力评测的饱和。一个通过率达到百分之百的 eval，只能用来检测退步，已经完全不能反映改进空间了。文章把这种状态叫做 eval saturation，也就是 eval 饱和。当 eval 接近饱和时，剩下的全都是最难的任务，大幅度的能力提升只体现为很小的分数增长，容易让你以为进步不大，其实进步明显。

代码评审公司 Qodo 就遇到了这个问题：他们起初的单次编程 eval 没有捕捉到 Opus 四点五在更长更复杂任务上的改进，所以对这个新版本印象平平，觉得没什么提升。后来他们开发了一个新的 agent 级别的 eval 框架，专门覆盖更长、更复杂的 agent 任务，才看到了真正的进步幅度。这个案例说明，eval 难度需要随着模型能力的提升而持续升级，否则 eval 就变成了一个没有意义的仪式。

第八步，通过开放贡献和持续维护保持评测套件的健康。eval 套件是一个活的产物，需要明确的归属和持续维护，不能建好了就放在那里不管。文章说效果最好的做法是：建立专职的 eval 团队来负责核心基础设施，同时让领域专家和产品团队贡献大多数具体的任务用例。

更重要的是，文章建议实践"eval 驱动开发"：在 agent 能完成某项能力之前就先写好 eval，然后不断迭代直到 agent 达成目标。内部很多时候会构建那些"目前勉强能用但赌的是几个月后模型会更强"的功能，能力评测从低通过率开始，当新模型上线时，跑一遍 eval 就能立刻知道哪些赌注兑现了。

文章还说，产品经理、客户成功经理，甚至销售，只要对产品要求和用户有深刻理解，都可以用 Claude Code 提交一个 eval 任务的 PR。把 eval 的门槛降低到非工程师也能贡献，是让 eval 套件保持鲜活的关键。

在这期内容的最后，文章还专门对比了六种理解 agent 性能的方法：自动化 eval、生产监控、A/B 测试、用户反馈、手动 transcript 阅读、以及系统性的人工研究。这几种方法不是互相替代的，而是映射到不同的开发阶段。自动化 eval 特别适合上线前和 CI/CD 流程，生产监控在上线后才能发挥作用，A/B 测试需要足够的流量才有统计意义，人工研究则适合校准 LLM 评分器或者处理高度主观的任务。效果最好的团队是把这几种方法组合起来用的：自动 eval 负责快速迭代，生产监控提供真实数据，周期性的人工评审做校准。

好，我们来收个尾。这篇文章是 Anthropic 把他们在 agent 评测上积累的工程实践系统性整理出来的一次分享，从术语定义、评分器类型，到四种 agent 类型的评测策略，再到八步实践路径，覆盖非常全面，而且每个观点都有真实案例支撑，不是纸上谈兵。

我个人认为，这篇文章最深刻的价值不在于某个具体技巧，而在于它传递了一种工程文化：eval 不是上线前的最后一道检查清单，而是贯穿整个 agent 产品生命周期的核心实践。它把"感觉不对"变成了可以量化、可以追踪、可以持续优化的东西。没有 eval，你在飞盲；有了 eval，你才有了一个明确的山头可以爬，而且知道每次改动是爬上去了还是滑下来了。

对于在座的工程师，我最想留下的一句话是：开始写 eval 的最好时机是 agent 立项的第一天，第二好的时机，就是今天。
