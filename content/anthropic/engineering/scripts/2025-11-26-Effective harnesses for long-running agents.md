## Shownotes

当你让 AI agent 独立完成一个需要好几天的大型项目，它会在哪些地方翻车？Anthropic 工程团队用「克隆 claude.ai」这个真实任务做了系统性实验，发现了两个最致命的失败模式，并设计出一套两阶段框架来解决它们。本期我们逐层拆解这套框架的每个关键设计决策，以及背后的工程逻辑。

- 核心挑战：每个新上下文窗口对 agent 来说相当于失忆，需要从零恢复状态
- 失败模式一：agent 一口气想把整个项目做完，结果上下文用尽，留下半成品烂摊子
- 失败模式二：做了几个功能之后就宣告完成，大量需求根本没有碰过
- 解法一：初始化 agent 负责打地基——init.sh 脚本、进度记录文件、JSON 格式功能验收清单、初始 git 提交
- 解法二：编程 agent 每次只做一个功能，完成后做 git commit、更新进度文件，保持环境「干净收工」
- 端到端测试的重要性：用浏览器自动化工具像真实用户一样验收，而不只是跑单元测试或 curl

大家好，欢迎收听 Claude FM，我是你们的主播。今天这期节目，我们要聊的是一篇来自 Anthropic 工程博客的文章，英文标题是 Effective Harnesses for Long-Running Agents，中文我把它翻译成「如何为长时间运行的 AI 智能体打造高效的运行框架」。这篇文章由 Anthropic 工程师 Justin Young 执笔，发表于二〇二五年十一月二十六日。

在正题开始之前，我需要先和大家交代一个时间背景。今天是二〇二六年六月十三日，也就是说这篇文章距今已经过去了大约六个半月。你可能会想，六个月对于 AI 领域来说是很长的时间了，这篇文章还值得认真听吗？我想先直接回答这个问题。

二〇二五年十一月，正是 AI agent——也就是能够自主执行复杂任务的人工智能代理——从概念开始真正走向工程实践的关键节点。Claude 四点五 Opus 这个版本已经问世，各大公司都在把 agent 往生产环境里推，整个行业的主要矛盾，正在从「AI 能不能理解复杂指令」，快速转向「AI 在真实环境里有多可靠、能持续多久」。Anthropic 这篇文章，正是在那个时期，用一个具体的工程实验来回答一个很现实的问题：怎么让 AI agent 在一个大型项目上持续工作好几个小时乃至好几天，而不是跑偏、卡死、或者中途崩掉。

这个问题在今天只会更重要，不会更不重要，因为 agent 的使用已经更加普及了。这篇文章里面的工程思维和设计原则，我认为经得起时间的检验，有充分的理由在今天认真学习。

好，背景说到这里，我们进入正题。

先来想象一个场景。你让一个 AI agent 去帮你从零开始做一个完整的 Web 应用，比如克隆一个类似 claude.ai 的聊天界面，包括用户登录、对话管理、主题切换、历史记录这些完整功能。从用户的角度来看，这个请求没什么特别的，不就是让 AI 写代码嘛，现在的大模型很强了，应该搞得定吧？

但从工程实现的角度来看，这里面有一个根本性的困难，叫做「上下文窗口的限制」。不管一个大模型的上下文窗口有多长，一个生产级别的 Web 应用所涉及的代码量、设计决策、调试过程、测试验收，都足以轻松地超过任何一个上下文窗口的容量。这意味着，完成这样一个任务，agent 必须跨越多个上下文窗口来工作，分多次、分多个 session 来推进项目。

而这就带来了一个真正的难题：每一次启动一个新的 session，对 agent 来说就像失忆了一样。上一个 session 里发生了什么，做了哪些设计决策，遇到了什么 bug，代码库现在是什么状态，全都荡然无存。每次都要重新去理解项目的现状，才能继续推进工作。

Anthropic 的工程师在文章里用了一个非常直观的比喻。他们说，想象一个软件开发团队，工程师们在轮班工作。每次新的工程师来接班，他对上一个班次发生了什么一无所知——没有交接记录，没有进度说明，两眼一抹黑就开始干活。你能想象这种开发模式会产生多少混乱吗？改了一半的功能、未解决的 bug、不知道从哪里来的代码，每个接班的人都得先花大量时间去侦探现场，才能搞清楚自己处于什么状态。这就是跨上下文窗口的 AI agent 所面临的真实处境。

当然，Anthropic 的 Claude Agent SDK——也就是一套用来构建和运行 AI agent 的工程框架——已经提供了一个叫做 compaction 的机制。Compaction 的意思是「压缩」，当上下文窗口快满的时候，系统会自动把之前的对话历史进行压缩，只保留最关键的信息，腾出空间让 agent 继续工作。理论上，有了 compaction，agent 应该可以无限期地持续推进一个项目。

但实验结果告诉我们，光有 compaction 是不够的。压缩后的上下文，并不总能把所有关键信息完整、清晰地传递给下一个 session。有些关键的上下文在压缩过程中丢失了，有些指令变得模糊了，于是下一个 session 的 agent 拿到的，是一个不完整的图景。文章明确指出，即使使用了 compaction，Claude 四点五 Opus——当时最强的编程模型——在只给一个高层次提示词的情况下，比如「构建一个 claude.ai 的克隆应用」，也无法产出一个生产级别的 Web 应用。

失败的方式相当有规律，暴露出了两个非常典型的问题。

第一个问题，研究人员把它叫做「一次性完成整个项目」的倾向。模型拿到任务之后，会试图一口气把所有功能全部实现出来，而不是一步一步来。听起来很有效率对吧？但在实际执行中，这非常危险。因为上下文窗口是有限的，当 agent 企图一次性完成所有工作的时候，往往会在上下文快用完的时候，某个功能做了一半，实现了一部分但没有收尾，也没有留下任何说明或文档。

然后，等到下一个 session 启动的时候，新的 agent 实例面对的是一个残缺的代码库，但根本不知道上一个 session 做到了哪一步，哪些是故意留到下一步的、哪些是做了一半忘了收尾的。于是它只好花大量时间去猜测、探索、排查，有时候甚至不得不把上一个 session 的工作部分推倒重来。这不仅效率极低，还很容易陷入恶性循环——每个新 session 花在「搞清楚现在什么状态」上的时间越来越多，真正推进新功能的时间越来越少。

这个场景让我想到一个现实类比。假设你要装修厨房，请了个承包商。第一天他来了，旧橱柜全部拆掉，水管挪了一半，瓷砖揭了几块。但因为时间到了他就走了，什么记录都没留。第二天来了另一个承包商，一看这个现场，搞不清楚水管是故意断开的还是出了问题，不知道揭下来的瓷砖是要换掉还是要修补，只好花半天时间研究，最终还是摸不着头脑，只能从最保险的方案重新来过。效率完全被打乱了。而且这种情况下，前面做的很多工作不仅没有积累，反而成了拖累。

第二个问题，叫做「提前宣告完成」。这个比第一个更隐蔽，也更危险。在经过一两个 session 之后，部分功能已经实现了，项目有了一些可见的进展。这时候，后续的 agent 看到这些已经完成的工作，扫视一遍觉得差不多了，就开始写总结然后宣布任务完成。但实际上，可能还有七八十个功能点完全没有碰过，只是因为已经实现的部分看起来挺完整的，agent 就做出了「任务已经完成」的错误判断。

这种「假阳性」在工程实践中是相当有破坏力的。你以为 AI 帮你把项目做好了，结果一上线或者一测试，发现到处是坑，核心功能根本没有实现。这个问题用简单的提示词优化根本解决不了，需要在架构层面做出改变。

好，两个核心问题我们搞清楚了。Anthropic 的解决方案是什么？

他们的核心思路是把「完成一个复杂项目」这件事，拆分成两个子问题分别解决。第一个子问题是，在项目刚开始的时候，怎么把环境搭建好，让后续的所有工作都有章可循、有迹可查。第二个子问题是，在每一个新的 session 里，怎么快速上手、只做一件事、然后干净地结束。

为了解决这两个子问题，他们设计了两种不同角色的 agent：初始化 agent，英文叫 Initializer Agent；以及编程 agent，英文叫 Coding Agent。这两者除了初始的提示词不同之外，使用的是同一套工具和同一个底层框架，区别仅在于各自的任务定位。

我们先来仔细说说初始化 agent。

初始化 agent 只在项目第一次启动时运行一次，它的全部职责就是把「地基」打好，让后续的所有编程 agent 都能站在一个清晰的基础上工作，而不是每次都在摸索基本情况。它需要做几件核心的事情。

第一件事，写一个 init.sh 脚本。这里的 sh 是 shell 脚本的格式，这个脚本做两件事：启动开发服务器，然后自动跑一个基础的端到端测试，验证应用的核心功能是否正常工作。在克隆 claude.ai 的实验里，「基础测试」的意思是打开浏览器，发送一条消息，确认能收到 AI 回复。每一个后续的编程 agent，在开始一个新的 session 时，都会先找到这个脚本、按步骤启动服务器、跑一遍基础测试，在十几秒内就能知道「当前应用是好的还是坏的」。有了这个脚本，agent 不需要每次都重新研究怎么启动项目，节省了 token，也消除了因为环境配置错误而产生的调试时间。

第二件事，建立一个进度记录文件，文章里命名为 claude-progress.txt。这个文件是跨 session 传递工作状态的核心载体，可以把它理解成一本「工程日志」。每个编程 agent 在完成工作之前，必须把这次 session 做了什么、遇到了什么问题、下一步建议从哪里着手，都清楚地写入这个文件。下一个 session 启动时，第一件事就是读这个文件，相当于接班工程师在开始干活之前先看一遍上一个班次的交接报告。有了这份报告，接班的 agent 就不需要去猜测现状，可以直接进入工作状态。

第三件事，也是这套方案里我认为最有设计感的一个点，就是创建一个结构化的功能验收清单文件，文章里叫做 feature_list.json。这个 JSON 文件里列出了项目需要实现的所有功能，每一条功能都有详细的描述、具体的验收步骤，以及一个叫做 passes 的字段，初始值全部设为 false，也就是「未通过」。

在文章的实验里，克隆 claude.ai 这个任务总共产生了超过两百个功能条目。每一条都写得非常具体，举个例子，其中有一条是「新建对话按钮能创建一个新的会话」，验收步骤包括五点：打开主界面，点击新建对话按钮，确认新会话已创建，检查对话区域显示欢迎状态，确认新会话出现在侧边栏里。这个颗粒度，不是那种模糊的「应用需要有聊天功能」，而是非常具体、可以一步步操作验证的条目。

你可能会想到一个问题：为什么用 JSON 而不是 Markdown 来写这个功能列表？文章里提到了一个非常接地气的工程发现：经过实验对比，模型在处理 Markdown 文件的时候更容易不小心修改或者删除其中的内容，而 JSON 格式有更严格的结构约束，模型会更加谨慎地对待它，不容易发生意外改写。此外，在 prompt 里研究人员还加了专门的强硬措辞，明确说「不可以删除或者修改任何一条功能条目，因为这样做会导致功能缺失或者引入 bug」。这种用明确的语言约束来引导 AI 行为的做法，是一个值得记住的工程技巧。

编程 agent 每次修改这个文件，只能把某个条目的 passes 字段从 false 改成 true，代表这个功能已经完成并通过了验收测试。除此之外，任何其他修改都是明令禁止的。

第四件事，是创建一个初始的 git 提交。Git 是代码版本控制工具，这个初始提交记录了项目刚搭建好时的完整状态，也是整个项目历史追踪的起点。后续每次编程 agent 完成一个功能，都会做一次提交，整个项目的演化过程就会一目了然。

好，地基打好了。现在我们来讲编程 agent 是怎么工作的。

编程 agent 负责每一个具体的开发 session。每次一个新的上下文窗口开始，就相当于启动了一个新的编程 agent 实例。它的核心使命是三件事：快速恢复状态、只做一件事、然后干净收工。

文章给出了一个标准化的 session 启动流程，我把它逐步给大家过一遍。第一步，运行 pwd 命令，确认自己在哪个工作目录里。这个步骤看起来很基础，但文章特别提到，这类「定向」步骤，对于 agent 迅速找到方向是有实际帮助的。第二步，读取 claude-progress.txt 进度文件，了解上一个 session 做了什么、遇到了什么问题。第三步，读取 feature_list.json 功能列表，找到优先级最高的、passes 字段还是 false 的功能，作为本次 session 的工作目标。第四步，查看最近二十条 git log，了解代码库近期的变动情况，确认有没有最近才引入的改动可能影响接下来的工作。第五步，找到 init.sh，启动开发服务器。第六步，用浏览器自动化工具做一次基础的端到端测试，确认应用核心功能正常，发现并修复任何在上一个 session 中被遗漏的 bug，然后才开始实现新功能。

最后这一步格外重要。文章里说，如果 agent 跳过基础测试、直接去做新功能，很可能会在一个本来就已经坏掉的基础上继续叠加新代码，让问题更难发现、更难排查。先确认基础正常，再推进新功能，是一个防御性的工程习惯，虽然多花了一点点时间，但避免了后续更多的浪费。

在实际开发阶段，编程 agent 遵循「每次只做一个功能」的原则。这听起来像是废话，但文章反复强调，这个原则是解决「一口气想做完所有事」这个失败模式的关键所在。agent 每次进入 session，只从功能列表里挑一个优先级最高、还没完成的功能，专注实现它，用测试工具验收它，确认通过了所有验收步骤，然后才把 feature_list.json 里对应条目的 passes 改成 true，做一个描述性的 git 提交，更新进度文件，然后干净地结束这个 session。

文章还专门定义了什么叫「干净收工」。所谓干净的状态，就是代码没有明显的 bug，代码是有序的、有文档的，任何一个开发者来到这个代码库，都可以直接在现有基础上开始做新功能，而不需要先花时间去清理上一个 session 留下的混乱。这其实就是我们对人类工程师的标准职业要求，而现在我们把同样的要求显式地传达给了 AI agent。

git 提交在这里还有一个额外的实用价值：它是一个可靠的回退机制。当 agent 发现某次改动引入了新 bug，或者整个方向走错了，它可以用 git revert 命令回退到上一个提交的干净状态，重新来过。这个能力在人类工程师的日常工作中是理所当然的，但很多人在设计 AI agent 工作流的时候，会忘记把这个工具权限配备给 agent。结果 agent 一旦走进死胡同，就只能在一个破损的状态上继续硬撑，越改越乱。文章里说，正是因为有了这个回退能力，agent 在遇到代码问题时能够更有信心地尝试和纠错，而不是陷入焦虑性的反复修补。

现在我们来讲这套方案里一个非常关键的细节：端到端测试。

文章发现，如果没有明确的指引，agent 在完成一个功能之后，倾向于用单元测试或者用 curl 命令直接调接口的方式来验证，然后就把功能标记为完成。这两种验证方式都有其价值，但都有一个根本性的盲点：它们都不是从真实用户的角度来体验功能。很多 bug 只有在真实的前端界面交互、真实的浏览器渲染环境下才会暴露出来，而单元测试和 curl 命令根本发现不了这类问题。

举个具体的例子。一个按钮的点击事件可能在单元测试里完全正常，但在真实浏览器里，因为 CSS 样式的层叠问题，这个按钮实际上是被另一个元素遮挡住了，用户根本点不到。这种 bug，你在服务端日志里看不出来，用 curl 测接口也发现不了，只有用浏览器真正去点击这个按钮，才能发现它不可用。

Anthropic 的解决方案是为 agent 配备浏览器自动化工具，具体用的是 Puppeteer——这是一个可以用代码控制 Chrome 浏览器的工具，通过 MCP 插件的方式集成到 agent 的工具集中。有了这个工具，agent 可以像真实用户一样，打开浏览器，点击按钮，在输入框里输入文字，观察界面的变化，验证整个功能链路是否按预期运转。

文章说，引入端到端测试工具之后，效果非常显著，agent 发现并修复了很多仅从代码层面完全看不出来的 bug。这个实验结果有很强的说服力：你给 agent 配备什么样的测试工具，直接决定了它能发现什么层次的问题。测试工具的质量，是 agent 能力上限的一个重要边界。

当然，Puppeteer 这个工具也有自己的局限性。文章坦诚地指出，Claude 通过 Puppeteer 无法看到浏览器原生的弹窗，也就是那种用 JavaScript 的 alert 函数或者 confirm 函数弹出来的系统级弹窗。依赖这类弹窗交互的功能，因为测试覆盖不到，bug 的比例就明显更高。这是工具层面的限制，文章承认目前没有完美的解法，只是在观察到这个问题之后，设计时会尽量避免在功能实现中使用原生弹窗。

好，现在让我们把这些知识点联系到实际工作中，聊聊对我们作为工程师或者 AI 工具用户的实际意义。

如果你正在开发或者使用 AI agent 来完成长周期的复杂任务，这篇文章给了我们几个可以直接参考的设计原则。

第一个原则是「结构化的状态恢复机制」。不管你的任务是写代码、做研究报告、处理数据分析，还是别的什么，只要任务需要跨越多个 session，就必须在设计阶段想好：agent 每次启动时怎么知道现在处于什么状态？怎么知道上次做了什么？怎么知道下一步该干什么？一个结构化的进度记录文件，配合一套标准化的启动检查流程，是目前最务实的解法。关键是要让这套机制「可读、可结构化」，而不只是一段自然语言的总结，因为结构化的信息比自然语言更难被 agent 误读或者丢失。

第二个原则是「用验收清单来定义完成」。在给 agent 布置一个复杂任务之前，先花时间把「什么叫做完成」明确定义出来。把所有要实现的功能或者要达到的目标，拆解成具体可验证的条目，初始标记为「未完成」，然后让 agent 逐条推进、逐条验收。这个机制一方面防止了「提前宣告完成」的问题，另一方面也让你作为人类监督者，随时都能看到项目的真实进度，而不是只能相信 agent 的自述。

第三个原则是「把测试工具当作架构设计的一部分来考量」。你不能指望 agent 自己发明好的测试方法。作为设计者，你需要主动为 agent 配备合适的测试工具，并且在 prompt 里明确告诉它：用什么工具、从什么角度验证、达到什么标准才算通过。在 Web 应用场景下，这意味着把浏览器自动化工具集成进来。在你自己的业务场景里，你需要想清楚什么是「端到端的验证」，然后把对应的工具给 agent 配备好。这个决策是框架设计者的责任，不应该丢给 agent 自己决定。

第四个原则是「把版本控制作为标配工具给 agent 用」。不管是代码还是文档，只要你的任务会产生可以版本化的输出，就把 git 或者类似的版本控制工具配备给 agent，并且明确要求它在完成每个阶段性工作之后做一次提交。这不仅是为了留下历史记录，更重要的是给 agent 提供了一个可靠的回退机制。当 agent 走进死胡同的时候，它有路可退，而不是被迫在一个破损的状态上继续叠加。这个机制，从根本上提高了长周期任务的容错性。

在实际操作层面，如果你现在就想基于这套思路搭建自己的 agent 工作流，最直接的切入点是：设计你的进度文件格式、设计你的验收清单格式，然后写两套 prompt，一套用于初始化阶段，一套用于每次迭代阶段。Anthropic 的文章里提到他们也有配套的 quickstart 代码示例，感兴趣的话可以直接去工程博客找到原文链接查看。

当然，这篇文章也提出了一些它自己还没有解决的开放问题，值得我们继续关注。

第一个开放问题是关于单 agent 还是多 agent 架构的取舍。这篇文章探索的是单一通用编程 agent 的方案，但研究者自己也提到，也许把任务分配给多个专门化的 agent，效果可能会更好，比如一个专门负责测试的 agent、一个专门负责代码审查的 agent、一个专门负责清理技术债务的 agent。专门化意味着每个 agent 的 prompt 和工具集都可以针对特定子任务做优化，理论上比通用 agent 更精准。在二〇二六年的今天，这个方向已经有了更多新的实践和研究，但从工程落地的角度来说，多 agent 架构的协调成本也会更高，怎么找到最优的粒度，仍然是一个开放问题。

第二个开放问题是关于领域泛化的。这套框架是专门针对全栈 Web 应用开发场景设计的。能不能把同样的思路迁移到科学研究、金融建模、内容创作这些完全不同的领域？文章认为核心原则应该是可以泛化的，只是具体的工具选择和验收方式需要根据领域来调整。我个人认为，进度文件、结构化的任务清单、版本控制这些设计模式，确实有很强的普适性，是框架设计的通用思路。但端到端测试的具体形态，在不同领域会有很大差异，在科学实验场景里可能是重现性验证，在数据分析场景里可能是跑回归检验，需要根据具体业务来定义。

好，最后我们来做一个收尾。

这篇发表于二〇二五年十一月的文章，用一个具体的工程实验告诉了我们：要让 AI agent 在复杂项目上持续稳定地工作，仅靠大模型本身的能力是不够的，你还需要在框架层面做大量工程设计。初始化环境、追踪进度、增量推进、干净收工、端到端验收，这五个环节，每一个都是针对实际观察到的失败模式设计出来的，每一个都经过了真实实验的验证。

我个人的评价是：这篇文章最可贵的地方，不是提出了多么新颖的算法，而是展示了一种「把 AI 当工程师管理」的思维方式。当 agent 的能力越来越强、承担的任务越来越复杂的时候，我们怎么设计管理 AI 的框架，和 AI 本身能力的进步同等重要，甚至更加紧迫。这种思维，是我认为这篇文章真正值得带走的核心洞察。
