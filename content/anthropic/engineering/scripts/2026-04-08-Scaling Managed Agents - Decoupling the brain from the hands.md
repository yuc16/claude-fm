好，我们先从背景说起。

要理解这篇文章，首先要理解一个叫做 harness 的概念，中文可以翻译成"执行框架"或者"控制框架"。我个人觉得"脚手架"这个说法更形象一些。它的作用是驱动 Claude 不断地循环运转：给 Claude 提供必要的上下文信息，让 Claude 决定接下来要调用哪个工具，处理工具返回的结果，再把结果更新到 Claude 的上下文里，如此往复，直到任务完成。你可以把它想象成一个项目经理，Claude 是那个执行任务的工程师，harness 负责协调整个流程，负责在每个环节上传下达。

现在，harness 在设计的时候，必然会对 Claude 的能力做出各种假设。这些假设反映的是"当下 Claude 不擅长做什么"，然后 harness 就去填补那个空缺。

文章一开头就举了一个非常具体的例子。Anthropic 的工程师发现，Claude Sonnet 四点五在处理长任务时，会有一个特殊的行为：当它感知到自己的上下文窗口快装满的时候，会倾向于提前结束任务，不管任务有没有真正完成。这个行为在圈子里有个名字，叫做"上下文焦虑"，英文是 context anxiety。就好像一个员工，看着一份越来越长的工作清单，开始觉得今天肯定完不成了，干脆早点收摊，不再处理新需求。

为了解决这个问题，工程师就在 harness 里加了一个自动重置上下文的机制，在上下文快满的时候主动帮 Claude 做一次清理，让它能继续工作下去。听起来是个合理的补丁。

但故事没有在这里结束。当他们把同样的 harness 用在更新版本的 Claude Opus 四点五上时，发现了一个耐人寻味的情况："上下文焦虑"这个行为已经不见了。新版本的模型足够强，不再有这种问题。但是，harness 里的那段重置逻辑还在，成了一段没人需要但也没人想着去删的死代码。

这看起来只是一个小问题，但它揭示了一个更深层的工程困境。随着时间推移，这样的死代码会越来越多，harness 会越来越臃肿，越来越难以理解。更麻烦的是，你根本不知道 harness 里哪些假设已经过时了。模型能力的变化是渐进的，不会有人在某天早上发布公告说"Claude 的上下文焦虑问题已修复，请删除对应代码"。你只能在观察到行为变化之后，事后才意识到某个假设悄悄地失效了。

而且这个问题不只是死代码那么简单。一个更坏的情况是：你不知道某段代码是不是还需要，你不确定删掉它会不会出问题，于是就保留着。随着模型的持续演进，这种不确定性只会越积越多，最终让整个 harness 变成一个没人敢动、逻辑不清晰的黑盒。

Anthropic 的答案是建立一个叫做 Managed Agents 的托管服务，核心目标是设计一套接口，让它能够活过任何特定的实现，包括 Anthropic 今天自己运行的实现。

好，现在我们来看他们具体遇到了什么工程问题，又是怎么解决的。

第一个大问题，关于把所有东西塞进一个容器带来的困境。

文章里用了一个工程界非常经典的类比来解释这个问题，叫做"宠物与牲口"，英文是 pets versus cattle。宠物是那种你精心照料的、独一无二的服务器，它有自己的名字，你认识它，出了问题你得花大量时间去诊断和抢救。牲口则是标准化的、可以随时替换的服务器，哪头出了问题，直接杀掉，再从一个标准模板起一头新的，不需要治疗。

Anthropic 最初的设计，是把所有东西都放进一个容器里：Claude 的 harness、运行代码用的沙盒、还有整个会话的历史记录，全都共享同一个环境。这样做有一些好处，比如文件操作可以直接是本地的系统调用，速度快，也不需要设计任何服务间的边界。但代价是，这个容器变成了一个宠物。

一旦容器崩溃，会话记录就跟着消失了。一旦容器卡住，工程师就得花时间去"抢救"它。而且这种抢救工作极其困难，因为工程师能看到的只有一个 WebSocket 事件流，而这个事件流告诉不了你问题出在哪个环节。harness 代码里有 bug 导致卡死、网络传输中途丢包、还是容器直接下线——这三种截然不同的原因，在事件流里看起来都是一样的症状，都是沉默，都是无响应。

要想真正弄清楚发生了什么，工程师需要打开一个 shell，直接进到容器里面去检查。但这个容器里同时存放着用户数据。进去调试的行为本身就带来了安全风险——工程师进入了一个本不该随意访问的环境。结果就是，他们实际上处于一个既没有透明度又不敢轻易进去看的两难困境：不进去不知道发生了什么，进去了又破坏了安全隔离。

除此之外，还有一个联动的问题。因为 harness 假设它要操作的所有资源都在同一个容器里，当企业用户想把 Claude 连接到他们自己公司的私有云环境时，这个假设就成了路障。用户的数据在他们自己的网络里，但 Claude 的 harness 在 Anthropic 的网络里，中间有一道物理上的网络隔阂。解决方案只有两条路：要么把用户的私有网络和 Anthropic 的网络打通，在企业安全合规层面极其复杂；要么把整个 harness 搬到用户自己的环境里去运行，这又带来了部署和运维的额外负担。两条路都很重，根源都是 harness 里那个看起来很自然的假设："我旁边的就是我能操作的。"

Anthropic 的解法，是把这个整体拆开，变成三个独立的组件。

大脑，就是 Claude 和它的 harness，负责推理和决策。双手，就是代码沙盒和各种工具，负责实际执行动作。记忆，就是会话的事件日志，负责持久化地记录所有发生过的事情。

这三个组件被拆分成了三个独立的接口，彼此之间只通过定义清晰的接口通信，而不是共享同一个运行环境。

harness 不再住在容器里。它调用容器的方式，和调用任何其他工具的方式完全一样：传入一个名称和参数，得到一个字符串形式的返回结果。仅此而已。容器从此变成了牲口。如果容器崩溃了，harness 把这次失败当成一次普通的工具调用错误，把错误信息传回给 Claude，Claude 自己决定是重试还是换一个策略。重新启动一个新容器，只需要一行标准化的初始化指令，就像从模板创建一台新的虚拟机一样简单快捷。工程师不再需要去"抢救"失败的容器。

harness 本身也变成了牲口。因为所有的会话状态都持久化在外部的事件日志里，harness 进程内部没有任何需要在崩溃后保留的东西。当一个 harness 崩溃时，启动一个新的，告诉它会话 ID，它就能用 getSession 这个接口去拉取完整的历史事件日志，从上次中断的地方继续往下走，就好像什么都没发生过一样。调试也因此变得透明——工程师可以检查 harness 进程本身，而完全不需要进入存放用户数据的沙盒。

第二个重要的改变，是安全边界的重构。

原来那个把所有东西都放在同一个容器里的方案，有一个在安全上很令人不安的地方。Claude 生成的代码会在这个容器里运行，而这个容器里同时存着访问各种外部服务所需的凭证——API 密钥、OAuth 令牌，各种东西。

这意味着什么？意味着如果攻击者成功通过 prompt 注入——也就是在 Claude 处理的内容里藏入恶意指令——让 Claude 执行一段有问题的代码，那段代码只需要读取一下容器的环境变量，就能拿到所有凭证。拿到凭证之后，攻击者可以用这些凭证去开新的、不受约束的会话，让 Claude 在更广泛的范围内为攻击者工作，造成更大的危害。

你可能会想，可以对凭证做细粒度的权限控制啊，让每个密钥只能做最少的事情。这确实是一个缓解措施，但文章指出它有根本性的局限：这种做法依然是基于"Claude 用有限权限能做什么"的假设，而 Claude 正在越来越聪明，这个边界在不断移动。今天你认为的"最小权限"，在更聪明的 Claude 面前，可能暴露的攻击面比你预期的要大。

结构性的解法是：让凭证永远不会出现在 Claude 生成代码运行的沙盒里。不是用权限去限制 Claude 用凭证能做什么，而是让 Claude 的代码根本就看不到凭证。

Anthropic 用了两种具体的模式来实现这一点。

第一种，把认证绑定到资源的初始化阶段。以 Git 代码仓库为例：访问令牌在沙盒启动的时候被用来克隆仓库，同时被写入本地 git 配置的 remote 地址里。之后在沙盒运行期间，Claude 可以正常执行 git push、git pull，这些操作会正常工作，但 Claude 本身从来没有接触过那个令牌的具体内容。令牌完成了它的使命，然后就不再存在于运行时环境里了。这就像你入住酒店时刷了一下门禁卡，之后在房间里活动，门禁卡本身已经不在你手边了。

第二种，外部凭证保险库加上专用的代理层。对于通过 MCP 协议接入的自定义工具，OAuth 令牌存放在一个完全在沙盒外部的安全保险库里。当 Claude 调用 MCP 工具时，请求会经过一个专门的代理服务，这个代理持有一个和当前会话绑定的、专属的会话令牌。代理拿着这个会话令牌去保险库里取出真正的凭证，然后以 Claude 的名义向外部服务发起请求。整个过程里，harness 不知道任何凭证的内容，沙盒更不知道。

这个设计的精妙之处在于，安全性不依赖于"Claude 不被骗去执行坏事"，而依赖于架构上的隔离，让危险代码根本就接触不到危险的数据。随着 Claude 能力的提升，这种结构性隔离反而会越来越有价值，因为 Claude 能执行的操作范围在扩大，而安全边界始终保持清晰。

第三个重要的设计，是把会话日志当作一个活在上下文窗口之外的外部档案馆。

对于需要几十轮、几百轮交互才能完成的长任务，Claude 的上下文窗口迟早会装满。这是所有做长时间任务的 agent 系统都必须面对的问题。行业里常见的几种解决方案——压缩历史、生成摘要、用记忆工具写文件——都有一个共同的根本问题：它们是不可逆的决策。

你把一段历史压缩成摘要，原始信息就消失了。你决定把某些旧的工具调用结果删掉，那些信息就再也找不回来了。做这个删减决策的时机，是在当下。但你当下根本不知道任务的后续步骤会不会突然需要那段被删掉的信息。这是一个信息上的赌注，而且一旦押错，代价是任务失败或者产生幻觉，再或者是不得不重头来一遍。

Managed Agents 对这个问题的回答，是把会话日志设计成一个活在 Claude 上下文窗口之外的、持久化的、可随时查询的档案馆。

文章里把这个机制叫做会话作为上下文对象。它通过一个叫做 getEvents 的接口，让 harness 可以按位置来选取事件流的任意片段来读取。这个接口非常灵活：你可以从上次读到的地方接着往下读，也可以在某个关键时刻之前往回倒几条事件，看看事情发生前的完整经过，还可以在执行一个重要操作之前，专门把早期的相关上下文重新调出来看一遍，做充分确认后再行动。

更重要的是，从事件日志里取出来的内容，在传入 Claude 的上下文窗口之前，可以在 harness 层做任意的加工处理。可以为了提高 prompt 缓存命中率而调整内容的顺序；可以做各种上下文工程，比如把最相关的信息放在上下文的特定位置；可以过滤掉当前步骤不需要的信息，减少噪音。

文章里特别强调了一个设计原则：他们有意识地把"上下文的持久化存储"和"上下文的管理策略"这两件事拆分开。会话服务只负责一件事：保证所有事件是持久的、可随时查询的。至于具体用什么策略来选取、排序、转换这些事件，完全是 harness 的职责，不同的 harness 可以有完全不同的策略。

这个分离背后有清醒的工程判断：随着未来模型能力的变化，最优的上下文管理策略本身也会改变。今天对 Claude 最有效的上下文组织方式，明天的 Claude 未必还需要。把策略逻辑封装在 harness 里，意味着策略可以独立迭代，而底层的事件存储接口可以保持稳定。

现在来看一组硬核的性能数据，这是文章里我觉得最值得记住的一部分。

文章里提到了一个叫做首令牌时间的指标，英文缩写是 TTFT，全称 time-to-first-token。它衡量的是：从用户提交一个任务，到系统输出第一个回复字符，中间的等待时间有多长。这个指标之所以重要，是因为它是用户体感最直接的延迟——在这段时间里，用户看着一个空白的界面，什么反馈都没有，干等着。等待一秒和等待十秒，用户的感受天差地别。

在旧的耦合设计里，每一个新的会话启动，都必须先走完一整套容器启动流程：初始化容器、克隆代码仓库、启动相关进程、从服务器拉取待处理的事件……所有这些都必须完成，Claude 才能开始真正地思考第一个问题。即使这个任务根本就不需要用到任何沙盒功能，这套启动流程照样必须走完。用户等待的时间里，很大一部分都花在了这些"准备动作"上，而不是 Claude 真正在思考。

解耦之后，容器只有在 Claude 通过工具调用明确请求执行代码的时候才会被启动。对于不需要沙盒的会话，harness 从编排层拉取完事件日志，Claude 就可以立刻开始推理，不需要等任何容器准备好。更进一步，因为 harness 变成了无状态的，启动多个 harness 进程变得极其轻量。

结果是，p 五十也就是中位数延迟，下降了大约百分之六十。p 九五也就是第九十五百分位的延迟，下降幅度超过了百分之九十。

这两个数字放在一起很有意思。p 五十的改善是显著的，但 p 九五的改善更是令人印象深刻。超过百分之九十的降幅，说明之前那些特别慢的启动过程——比如容器启动遇到资源争抢、或者克隆大仓库本身需要很长时间——这类长尾延迟基本上被彻底消除了。对用户体验来说，这是质变级别的改善。

接下来说扩展性，也就是文章标题里"规模化"这个词的具体含义：多个大脑，多只双手。

先说多个大脑。因为 harness 变成了无状态的，所有的会话状态都在外部的事件日志里，harness 进程本身不保存任何东西。这让横向扩展变得极其自然：要处理更多并发的任务，只需要启动更多的 harness 进程就行。这些进程按需连接到它们需要的沙盒，用完就释放，不需要预先锁定任何资源。在旧的设计里，每一个活跃的大脑都需要一个专属的容器，大脑数量和容器数量是一比一绑定的。现在，多个大脑共享一个容器资源池，按需取用，资源利用率显著提升。

再说多只双手。每一只双手对于大脑来说，都是同一个接口：传入名称和参数，得到字符串形式的返回。这个接口背后可以是任何东西：一个 Docker 容器、一个 MCP 服务器、一个自定义的 API 接口，甚至——文章里用了一个让人会心一笑的例子——一个 Pokemon 游戏模拟器。harness 完全不关心双手背后具体是什么，只关心接口格式是否正确。

这种设计带来了一个之前没有的能力：一个大脑可以同时持有多只双手，把不同的子任务分发到不同的执行环境里并行处理。更有意思的是，大脑之间可以互相传递双手，一个 agent 可以把自己的工具能力委托给另一个 agent，形成更复杂的协作结构。

文章坦诚地说，他们最初没有用多手设计，是因为当时的模型还没有能力同时管理多个执行环境。这确实是一个认知复杂度更高的任务：你要清楚地知道现在有哪些执行环境可用、每个环境当前的状态如何、哪个子任务该发到哪个环境去、不同环境之间是否有数据依赖。这比在一个单一的命令行里操作要复杂得多。随着模型能力的提升，单容器的设计不再是因为模型需要它，反而变成了一个不必要的瓶颈，所以才到了做架构升级的时机。

文章的最后，作者们用了一个跨越几十年的类比来解释整个设计背后的核心哲学，这个类比我觉得非常值得仔细品一品。

几十年前，操作系统的设计者们面临一个根本性的问题：如何为"还没有被想到的程序"设计一个系统？他们不知道未来会有什么样的程序，但他们需要设计一个能支撑这些未知程序的底层基础。

他们的解法是把物理硬件虚拟化成足够通用的抽象。进程、文件、文件描述符，这些抽象被设计得足够简洁、足够通用，可以支撑接下来几十年里出现的各种程序，包括那些设计者们完全想象不到的程序。那个 read 函数，不管背后是一九七零年代的磁盘组还是今天的固态硬盘，接口始终如一。底层实现可以随意替换和升级，但抽象层保持稳定，程序不需要重写。

Managed Agents 试图在 AI agent 领域做同样的事。它把 agent 的各个组件虚拟化成了足够通用的抽象：会话是事件的只读追加日志，harness 是驱动 Claude 循环的控制逻辑，沙盒是执行代码和文件操作的环境。每个组件都是一个接口，接口背后的实现可以随时替换，而不会影响其他组件。

文章把 Managed Agents 叫做"元框架"，英文是 meta-harness。它不规定应该用什么具体的 harness，只规定 harness 应该遵守的接口形状。Claude Code，那个被 Anthropic 内部广泛使用的 harness，可以跑在这个元框架里。针对特定领域高度优化的专用 harness，也可以跑在这个框架里。未来还没有被发明出来的 harness，理论上也可以跑在这个框架里，只要它遵守同样的接口协议。

这个设计的核心假设只有几条：Claude 需要能操纵状态的能力，需要能执行计算的能力，需要能扩展到多个大脑和多只双手的能力。但在这些能力之外，元框架不对 Claude 将来需要什么做任何假设。这正是操作系统设计哲学的精髓：对接口形状有主张，对接口背后的实现没有主张。

好，说了这么多架构层面的内容，我们来聊聊如果你是一个实际工作中构建 AI agent 系统的工程师，这篇文章能给你带来什么。

第一件值得带走的事，是工程心态上的校准：定期主动质疑你的假设。文章开头那个上下文焦虑的例子是一个很好的提醒。今天你系统里的每一个"因为 Claude 不擅长 X，所以我们加了 Y 的补丁"，都应该定期拿出来重新审视。你可以问自己三个问题：这个假设当初是基于哪个版本的模型的？那个版本的限制现在还存在吗？如果限制消失了，这段代码还有存在的必要吗？Claude 的能力在快速演进，上个季度需要绕开的限制，这个季度可能已经不复存在了。如果你不主动检查，这些过期的假设就会慢慢把你的系统变成一团没人敢动的黑盒。

第二件事，关于状态的归属问题。你的 harness 是有状态的吗？如果是，它就是一个宠物，而宠物是运维风险。试着问问自己：harness 里的状态，有哪些是必须住在 harness 里的，有哪些可以外置到一个持久化的存储里去？如果 harness 在任何时刻崩溃，重启一个新的 harness 能无缝地从上次中断的地方接着工作吗？如果不能，原因是什么，能改变吗？Managed Agents 选择把所有会话状态完全外置，让 harness 成为纯粹无状态的进程，这个思路在你自己的系统设计里是否有借鉴的空间？

第三件事，关于安全边界的检查。如果你在构建的 agent 系统里，Claude 需要访问某些外部服务，请仔细想想：这些服务的凭证在哪里？Claude 生成的代码能读到这些凭证吗？如果能读到，你的安全边界就依赖于 Claude 不被操纵，而这是一个会随着 Claude 能力增强而持续受到压力的假设。更可靠的做法是结构性地隔离：在代码运行的沙盒里，根本就不存在凭证，而是通过初始化时绑定或者代理层来处理认证。Anthropic 用的那两种模式——初始化时克隆并注入、外部保险库加代理——都是可以直接参考的工程实践。

第四件事，如果你正在用 Anthropic 的平台做开发，Managed Agents 现在已经是一个真实可用的产品，有官方文档，可以上手试用。文章最开头就提到了这一点，你可以通过 Anthropic 的官方文档来找到入门指引。这不只是一篇架构理论文章，它背后是一个今天就可以接入的服务。

第五件事，我个人认为最值得长期回味的，是文章里隐含的那个命题：在一个能力快速演进的生态系统里，做基础设施的核心挑战不是"怎么支撑今天的需求"，而是"怎么让今天的设计决策不妨碍明天的可能性"。你花大力气优化了一个 harness，但如果这个 harness 里耦合了太多关于当前 Claude 能力的具体假设，下一代模型出来之后，这些优化可能都要重做，甚至要推倒重来。而如果你设计的是足够通用的接口，底层实现可以随着模型能力的演进不断替换和升级，这个系统才能真正长寿。

这是一个软件工程里的老道理——接口重于实现。但在 AI 时代，它有了新的紧迫性：因为"实现"变化的速度，比过去任何一个技术时代都要快得多。

好，今天就到这里。我们聊了 Anthropic 工程博客里的这篇文章，英文标题"Scaling Managed Agents: Decoupling the Brain from the Hands"，中文翻译是"规模化的托管智能体：将大脑与双手分离"，发表于二零二六年四月八日。

核心思路用一句话总结就是：把大脑、双手和记忆分离成三个独立的接口，让它们通过稳定的协议协作，而不是紧耦合在一个脆弱的整体里。这一分离，带来了更好的可调试性、更清晰的安全边界、更低的延迟，以及更强的横向扩展能力。

我个人认为，这是一篇值得仔细读的工程文章，因为它把一个平时说起来很虚的设计原则——接口重于实现——用非常具体的架构选择和真实的性能数据落了地。对于任何在认真做 AI agent 基础设施的团队，这篇文章里的思路都值得拿去对照自己的系统，认真想一想。
