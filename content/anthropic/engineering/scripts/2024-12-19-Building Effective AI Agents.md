这篇发表于二〇二四年十二月的文章，来自 Anthropic 工程师与数十个客户团队合作后的一手总结。他们发现，真正跑得好的 agent 系统，往往不靠最复杂的框架，而是靠最简单的可组合模式。距今已近一年半，这套方法论不仅没有过时，反而被越来越多踩过坑的团队所验证。

- 工作流（workflow）vs 智能体（agent）的本质区别：谁掌握控制权
- 五种核心工作流模式：提示词链式、路由、并行化、编排者-执行者、评估者-优化者
- 为什么在 agent 系统里，工具设计和提示词工程同样重要
- ACI（agent-computer interface）：被严重低估的工程工作
- 三条核心原则：简单、透明、工具文档投入到位
- 决策框架：你的项目到底需要 workflow、agent，还是一个优化好的单次调用就够了

首先聊一个让很多人头疼的基础问题：到底什么是 agent？

这个问题听起来简单，但 Anthropic 的工程师发现，不同的团队对这个词的理解差异非常大。有人认为 agent 必须是完全自主的，能在没有人类干预的情况下，连续执行数小时甚至数天的复杂任务；也有人把一个简单的多步骤流程就叫 agent，只要其中调用了大语言模型就算。这种概念上的混乱，会直接导致技术选型的混乱，你搞不清楚自己到底需要什么，也搞不清楚评估成功的标准是什么。

为了解决这个问题，文章做了一个清晰的分类。他们把所有这类系统统称为"agentic systems"，也就是"具有智能体特征的系统"，然后在内部做了一个非常关键的架构区分，把它分成两大类。

第一类叫 workflow，工作流。这类系统里，大语言模型的调用路径、工具的使用方式，都是通过预先写好的代码来编排的。整个流程是确定的，模型在里面承担特定角色，但整体走向是你事先规划好的，模型没有控制权。

第二类才是真正意义上的 agent，智能体。在这类系统里，模型本身负责动态地决定要做什么、先后顺序是什么、要用什么工具、要走哪条路径。控制权在模型手里，而不是在预先写好的代码里。

这个区分非常重要，因为两类系统在复杂度、成本、可控性上有根本的不同，适合的场景也截然不同。理解这个区分，是接下来一切判断的基础。

文章的整体逻辑，是从最简单的单次大语言模型调用出发，逐步升级，一直到完全自主的 agent，帮你搞清楚每个层次适合什么场景。而且文章从一开始就立了一个基调：找到能解决你问题的最简单方案，只有在简单方案确实不够用的时候，才增加复杂度。这看起来像废话，但在实际项目里非常难贯彻，所以才值得一再强调。

我们先从最基础的构建单元说起。

所有 agentic 系统的基础构建单元，文章叫它"增强型大语言模型"，英文是 augmented LLM。

什么叫增强？就是在基础模型上叠加三种能力：检索能力，让模型能从外部知识库或数据库里拿到相关信息，不局限于训练数据；工具调用能力，让模型能主动与外部系统交互，比如调用 API、执行代码、发送请求；以及记忆能力，让模型能在多轮对话或多个任务之间保留关键信息。

现在的模型，比如 Claude 系列，已经能够主动生成搜索查询——也就是说，它知道自己需要什么信息，然后主动去找，而不是被动等你告诉它；它能自己判断哪个工具最适合当前的情况，而不是需要你显式指定；它还能决定哪些信息值得保留下来供后续步骤使用。这些能力的组合，就是一个可以工作的 agentic 系统的最小单元。

文章特别强调两点。第一，这些能力的配置要针对你的具体场景来裁剪，不要一上来就把所有工具都打开——工具越多，模型选择的难度越大，出错的概率也越高。第二，工具的接口要简洁，文档要清晰，这样模型才能正确地理解和使用它们。关于工具设计，后面我们会有专门的深入讨论，这是整篇文章里我认为最有实践价值的部分。

好，现在进入文章最核心的干货：五种工作流模式。

这五种模式构成了一个从简到繁的渐进体系。我会用类比和例子把每种模式讲清楚，你也可以对照自己的项目，想想哪种最适合你的场景。

第一种，叫提示词链式调用，英文叫 prompt chaining。

你可以把它想成一条流水线。任务从一端进来，经过一个个工位的加工，从另一端出去。每个工位是一次大语言模型的调用，上一个工位的输出是下一个工位的输入。你可以在任意两个工位之间插入一个质检环节，也就是程序化的检查，只有结果满足要求才继续往下传，否则就报警或者重试。

什么时候适合用这个模式？当你的任务能被清楚地分解成若干个有明确顺序的步骤，而且每一步的输入和输出都比较确定。核心逻辑是：与其让一次大语言模型调用承担所有事情，不如把任务拆开，让每次调用只做好一件事。每件事做好了，整体的准确率反而会更高，代价是总体的耗时会长一点。

举两个具体的例子。第一个：内容本地化。你想把一段产品说明翻译成五种语言。流程可以是：先写好中文原稿，用程序检查字数和格式是否符合要求，通过之后再分别翻译成五种语言，最后再做一轮本地化审查。每个步骤专注做好一件事，比让一个调用同时做所有事要可靠得多。

第二个：写一篇技术文章。你可以先让模型生成文章大纲，然后用另一次调用检查这个大纲是否覆盖了所有关键点、逻辑是否清晰，检查通过之后再基于这个大纲扩展成完整的文章。这种两步走，比直接让模型"给我写一篇完整文章"通常能得到更好的结果，因为第一步的结构约束帮助第二步少走弯路。

第二种，叫路由，英文是 routing。

路由这个词，学过网络的人肯定不陌生。路由器的作用是判断数据包应该往哪个方向转发。在大语言模型系统里，路由的作用类似：先对输入做分类判断，然后把它分发给最适合处理这类输入的后续流程。

这个模式解决的核心问题是：不同类型的输入，往往需要非常不同的处理方式。如果你用同一个提示词处理所有类型的输入，要么提示词变得非常臃肿，充满了各种条件分支，要么对某些类型的输入效果很差。路由把"判断"和"处理"这两件事分开，各自做到最优。

文章给了一个非常贴近现实的例子：客服系统。用户发来的消息，有的是在问产品功能，有的是在申请退款，有的是在报告软件 bug，有的是在投诉服务态度。这四类问题，需要完全不同的处理逻辑、不同的后端系统、不同的回复风格。路由系统先判断这条消息属于哪一类，然后交给专门的处理流程，每个流程只需要做好自己的那件事。

还有一个更有意思的用法：按难度路由。简单的、常见的问题，比如"你们的退款政策是什么"，交给小模型处理，比如 Claude Haiku 四点五，速度快、成本低，效果完全够用；复杂的、少见的问题，比如需要理解多个历史订单之间关系的退款纠纷，交给大模型处理，比如 Claude Sonnet 四点五，能力更强，虽然贵一点，但用在刀刃上。这个策略能够在整体性能和运营成本之间取得很好的平衡，是一个非常实用的优化手段。

第三种，叫并行化，英文是 parallelization，里面有两个子模式。

第一个子模式叫分段并行，英文叫 sectioning，就是把一个大任务分解成几个相互独立的子任务，同时启动多个模型实例来处理，最后用程序把结果合并。

举个场景：内容安全审核。你收到一个用户请求，你希望一边生成回复，一边同时检查这个请求有没有违规内容。如果把这两件事放在同一个模型调用里，往往效果不理想，因为两件事的目标有时候会互相干扰。更好的做法是让两个实例并发跑：一个专注于生成高质量回复，另一个专注于做安全检测，互不干扰，最后再根据检测结果决定是否返回回复。这种拆分，通常能得到比合并处理更好的效果，而且总耗时和单次调用接近，因为两个实例是同时跑的。

第二个子模式叫投票，英文叫 voting，就是对同一个任务，用多个不同的提示词或者多个模型调用来各自独立处理，然后对结果进行聚合或者多数投票。

举个代码安全审查的例子。你要检查一段代码是否有安全漏洞。你可以同时启动三个审查实例，每个实例的提示词关注不同类型的漏洞：一个专门找输入验证的问题，一个关注权限控制，一个检查数据泄露风险。只要有任何一个实例标记出了问题，就整体发出警报。这种方式可以显著降低漏报率，代价是成本倍增。在安全敏感的场景里，这个代价通常是值得付的。

第四种，叫编排者-执行者模式，英文是 orchestrator-workers。

这个模式里有两个角色。编排者，也就是 orchestrator，是一个中央大脑，负责理解整体任务，然后动态地把它分解成若干子任务，分配给不同的执行者。执行者，也就是 workers，接受编排者分配来的子任务，专注完成自己那一块，把结果交回来，再由编排者汇总整合。

你可能会想，这和并行化不是很像吗？关键区别在于：并行化里的子任务是你事先定好的，在写代码的时候就固定了；而编排者-执行者模式里，有哪些子任务、每个子任务是什么内容，是编排者在运行时根据具体的输入动态决定的。

来个非常直观的例子。假设你在做一个 AI 辅助的代码重构工具。用户给了一个需求："把这个系统的认证模块升级成支持单点登录"。到底需要改哪些文件，每个文件需要做什么样的改动，这件事在你写代码的时候是不知道的，因为每个用户的代码库结构都不一样。编排者会先分析整个代码库，理解模块之间的依赖关系，然后动态制定一个执行计划，把具体的修改任务分配给各个执行者。这就是这个模式的核心价值：适合任务的具体结构在运行时才能确定的场景。

第五种，叫评估者-优化者模式，英文是 evaluator-optimizer。

这个模式的核心是一个迭代循环：一个模型负责生成输出，另一个模型负责评估这个输出，给出具体的改进意见，然后第一个模型根据意见修改，再评估，再修改，直到结果满意或者达到迭代上限。

文章里用了一个很贴切的比喻：这就像一个作家反复打磨文章的过程。你写了初稿，交给编辑，编辑提出意见，你根据意见修改，再提交，再修改，直到两个人都满意。评估者-优化者模式把这个过程自动化了。

这个模式特别适合什么场景？文章给了两个判断标准：第一，当你把具体的改进意见告诉模型，它确实能改得更好——也就是说，还有改进空间，而且模型能响应反馈；第二，评估者模型有能力提供真正有效的反馈，而不是说一些"这很好但还可以改进"这种没有信息量的话。

文学翻译是这个模式的绝佳用例。第一个模型做翻译，第二个模型扮演一个挑剔的文学评论家，指出哪个地方翻译腔太重、哪个词语没有传达原文的情感色彩、哪里的节奏和原文差太多，然后第一个模型根据这些具体意见重新翻译。多轮迭代下来，翻译质量会有非常明显的提升。

讲完五种工作流，我们来聊聊真正的 agent——让模型自主规划、自主决策的那种系统。

文章认为，agent 开始在生产环境里真正发挥作用，是因为现在的大语言模型在几个关键能力上已经足够成熟：能理解复杂的输入，能进行推理和规划，能可靠地调用工具，以及能从错误中恢复和调整。这些能力的组合，才使 agent 在真实场景里变得可用。

一个典型的 agent 工作方式是这样的：接到任务之后，先规划，然后一步一步地执行。在执行过程中，它会持续地从环境里获取反馈——工具调用的结果是什么、代码跑出来的输出是什么、搜索到了哪些信息——然后根据这些反馈来调整后续的计划。在一些关键的决策节点，它可以主动暂停下来，请求人类的确认或者补充信息。整个过程通常会设定停止条件，比如最大迭代次数，防止它在某个地方无限循环。

那问题来了，既然 agent 这么强大，为什么不一上来就用 agent？

文章给出了非常清醒的警告：agent 有两个特有的风险，你必须充分理解。

第一，错误会叠加。在一个多步骤的 agent 系统里，一个早期步骤的小错误，可能会引发后续一连串的错误，而且这种叠加效应往往是隐蔽的。你可能要等到 agent 跑了十几轮、消耗了大量时间和 token 之后，才发现整个方向早在第三步就已经跑偏了。这和工作流的情况非常不同，工作流里每个步骤都是可预测的，出了问题很容易定位。

第二，成本更高、延迟更长。agent 的本质是多轮迭代，每一轮都要消耗 token，来回的时间也在累积。如果你的任务其实用一个设计得当的工作流就能解决，用 agent 是在浪费资源。

所以，文章的建议是：在沙箱环境里进行大量测试，在不确定的地方设置人工审核节点，加入合理的停止条件。agent 的自主性是它的优势，但这种自主性在出错的时候同样会自主地跑偏，你需要设计足够的护栏。

什么场景下 agent 真正有价值？那些任务边界是开放的，你根本无法预先规划所有步骤，只能让模型在运行时动态决策的场景。文章给了两个自己实际做过的例子：解决 SWE-bench 测试集里的真实 GitHub issue，需要理解整个代码库的结构、定位问题根源、修改多个文件，整个过程的步骤数和具体操作事先无法预知；以及 Claude 的"computer use"功能，让模型通过截图来感知屏幕状态、通过键盘和鼠标操作来控制电脑完成任务，这类任务的多样性和不确定性，天然需要 agent 级别的自主决策能力。

好，现在来聊一个我认为整篇文章里最有实践价值、也最容易被忽视的部分：工具设计。

文章有一个核心论断：工具的设计和文档，应该和系统提示词本身受到同等程度的重视。换句话说，你花多少心思在优化提示词上，就应该花多少心思在设计工具上。这句话听起来显而易见，但在实际项目里，工具文档往往是被随手一写就算了的部分，而提示词才是工程师们反复打磨的对象。

为什么工具设计这么重要？你想一想，模型是通过什么来理解它能做什么、怎么去做的？是工具的名称、参数定义和文档说明。如果这些写得不清楚，模型就会用错工具、传错参数、在不该用某个工具的时候用了它，或者在该用的时候不知道有这个工具。这类问题，在实际生产里是非常常见的错误来源，但很多工程师不会去调查工具定义有没有问题，而是去调整提示词，结果越调越乱。

文章引入了一个非常有启发性的类比。我们在设计面向人的软件时，会做大量的用户研究，投入大量精力研究人机界面，英文叫 human-computer interface，简称 HCI。用户能不能方便地完成任务，和界面设计的好坏关系极大，大家都知道这件事的重要性。设计 agent 的工具，本质上是在设计模型和计算机之间的界面，文章把这个叫做 agent-computer interface，简称 ACI。你在 HCI 上投入了多少精力，在 ACI 上就应该投入对等的精力。这个类比我觉得非常到位，因为它让一件看起来像"写个注释"的事情，上升到了系统设计的高度。

具体怎么做工具设计？文章给出了几条非常实用的建议。

第一条，站在模型的角度想问题。假设你现在就是这个模型，你只能看到工具的名称、参数名、参数描述和文档说明。基于这些信息，你能准确判断什么时候该用这个工具、该传什么参数吗？如果你自己需要仔细思考半天，那这个工具定义肯定也会让模型困惑。一个写得好的工具定义，通常应该包括：这个工具能做什么、什么时候应该用它、用法示例、边界情况的处理说明，以及它和其他类似工具的区别在哪里。

第二条，认真对待参数的名称和描述。这件事听起来琐碎，但很多工程师会把它当成可以随手应付的工作。文章的建议是，把它想成你在给刚入职的新手工程师写函数文档。参数名要能直观反映它代表什么，描述要说清楚预期的格式、有效的取值范围，以及不同情况下应该传什么值。当你有好几个功能类似的工具时，这件事尤其重要，因为模型很容易把它们搞混。

第三条，实际测试模型怎么用你的工具。不要假设你的工具定义是清楚的，要用大量真实的输入来跑，看看模型会在哪里出错，针对那些错误迭代改进工具的定义。这是一个测试驱动的迭代过程，不是一次性的设计工作。

第四条，从设计上防止错误，也就是防呆设计，工程里有个专业词叫 poka-yoke，来自日本制造业的概念，意思是通过改变设计本身，让常见的错误从物理或逻辑层面就很难发生。

文章分享了一个他们自己踩过的坑，来自做 SWE-bench agent 的真实经历。他们发现，当 agent 在执行过程中切换了工作目录之后，如果工具接受相对路径，模型就会频繁出错，因为它弄混了当前目录和文件的实际位置。他们尝试了调整提示词，但效果不好。最后他们的解决方案非常干脆：修改工具定义，强制要求所有文件路径必须是绝对路径，相对路径一概不接受。改完之后，这类错误彻底消失了。

更让我印象深刻的是文章里接下来的一句话：在整个 SWE-bench agent 的开发过程中，他们花在优化工具定义上的时间，比花在优化整体提示词上的时间还要多。这说明什么？说明工具设计是真正影响 agent 能力上限的关键因素之一，而不是一个可以随便对付的细节工作。

好，我们把这篇文章的核心内容都讲完了，文章最后还总结了三条构建 agent 系统的核心原则。

第一条：保持简单。在设计层面，能简单就简单，不要为了炫技或者追求架构上的"优雅"而引入不必要的复杂度。每增加一层复杂度，都意味着更难调试、更难维护、出了问题更难定位，也意味着需要更多的 token 来解释清楚上下文。这条原则听起来简单，但在实际项目里真的很难坚持，因为复杂的架构往往看起来更专业、更有技术含量，很多工程师会在这里迷失。

第二条：保持透明。要让 agent 的规划步骤显式地展现出来，不要让它的决策过程变成一个黑盒。你需要知道它在想什么、为什么做了某个决定，这样才能在出问题的时候快速定位原因，也才能让用户和其他工程师对这个系统建立足够的信任。

第三条：投入足够精力做好工具文档和测试。这条我们刚才已经深入讲过了，不再展开。

那么，这篇文章对你的实际工作意味着什么？我来帮你做一个具体的应用转化。

如果你现在正在评估要不要引入某个 agent 框架，文章给了一个很直接的建议：先直接调用 API，不要急着用框架。大多数模式，包括我们刚才讲的那五种，用几十行代码就能实现，而且你对底层发生了什么有完整的掌控。等你真正遇到了框架能解决而手写很难解决的问题，再引入框架。如果引入了框架，要确保你理解这个框架在底层做了什么，不要把它当成黑盒来用。文章明确说，对框架底层行为的错误假设，是他们在客户项目里见到的最常见错误来源之一。

如果你现在要设计一个新的系统，我建议你用文章的逻辑作为决策框架，依次问自己三个问题。第一，一个经过充分优化的单次大语言模型调用，加上检索和上下文示例，够不够解决我的问题？很多时候，答案是够的，你根本不需要多步骤系统。第二，如果单次调用不够，用五种工作流里最简单的那个能解决吗？第三，只有当工作流明显不足时，才考虑真正的 agent。这个决策顺序，是很多团队没有遵循、后来付出代价的地方。

关于那五种工作流，我帮你做一个快速对照。你的任务有明确的步骤顺序，而且每步的输入输出很确定？提示词链式是首选。你的输入类型差异很大，需要不同的处理方式？路由来帮你。任务里有几个可以同时做的独立子任务，或者需要多角度验证同一个结论？并行化。任务的具体结构在运行时才知道，需要动态规划？编排者-执行者。你有明确的质量评判标准，而且多轮迭代能显著改进输出？评估者-优化者。

有两个坑我特别建议你绕开。第一个，过早引入复杂度。太多团队一上来就设计了三层 agent 嵌套的系统，结果调试起来极其痛苦，而他们要解决的核心问题，其实用一个两步的提示词链式调用就能完美解决。记住文章的原则：只有当复杂度能带来可量化的改进时，才值得引入。第二个，低估工具设计的重要性。你的模型能不能把任务做好，很大程度上取决于它能不能正确地理解和使用工具。写好工具定义、用真实输入来测试、根据模型的实际行为来迭代改进，这件事没有捷径，但回报是非常直接的。

好，我们来做一个总结。

这篇文章的核心思想，可以用一句话概括：在 AI agent 领域，成功不在于构建最复杂的系统，而在于为你的具体需求构建最合适的系统。从最简单的方案出发，通过充分的测量和测试来决定是否增加复杂度，并且把工具设计当成和提示词工程同等重要的工程工作来对待。

我个人认为，这篇文章在二〇二四年底那个时间点出现是非常有价值的。当整个行业都在追逐最新框架、最酷架构的时候，Anthropic 的工程师出来说：等一等，最有效的往往是最简单的那个。一年半后的今天，我认为这个判断经受住了时间的检验。随着越来越多的团队在生产环境里把各种复杂架构踩过一遍，简单、可控、可维护，已经成为越来越多工程师在设计 agentic 系统时的首要考量，而不是"用了多少个 agent"这种指标。

如果你在通勤路上听到这里，我给你留一个思考题：你现在正在做的项目里，有没有哪个部分引入了不必要的复杂度？回去之后可以对照文章里的五种工作流模式，也许能找到一个既省时省钱又更可靠的设计。
