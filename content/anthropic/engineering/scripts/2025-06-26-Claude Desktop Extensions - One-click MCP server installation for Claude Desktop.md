## Shownotes

本期深度拆解 Anthropic 的 Desktop Extensions（桌面扩展）技术方案——一种把 MCP 服务器打包成单一可安装文件的新格式（.mcpb）。从此，安装 MCP 服务器不再需要 Node.js、不再需要手动编辑 JSON 配置文件，用户双击一个文件就能完成安装。我们会讲清楚它的底层机制、manifest 设计哲学，以及它对开发者、普通用户和企业用户各自意味着什么。

**本期要点**

- MCP 服务器安装的"五大痛点"：开发工具依赖、手动配置、依赖冲突、无发现机制、手动更新
- .mcpb 文件的本质：一个包含全部依赖的 ZIP 压缩包，加上一个 manifest.json 配置清单
- manifest.json 的核心能力：用户配置声明、模板变量替换、平台差异化适配、敏感信息安全托管
- 打包工具链：mcpb init 和 mcpb pack 两条命令完成全流程，Claude Code 可辅助生成代码
- Anthropic 开源规范和工具链，期望其他 AI 桌面应用也支持 .mcpb，打造跨应用通用标准
- 企业级支持：MDM 和 Group Policy 集中管控、私有扩展目录、黑名单机制保障合规

今天这期节目，我们来聊一篇 Anthropic 工程博客上的文章，英文标题是 Claude Desktop Extensions: One-click MCP server installation for Claude Desktop，中文翻译过来是"Claude 桌面扩展：一键安装 MCP 服务器"。这篇文章发表于二零二五年六月二十六日。

在进入正题之前，我需要先说一下时效性。今天我们录制这期节目的时候，是二零二六年六月，距离这篇文章发表已经过去了将近整整一年。在 AI 工具这个领域，一年可以发生很多事。回头看这篇文章，它代表的是 AI 桌面工具从"开发者专属玩具"转向"普通用户可用的生产力工具"这个关键历史拐点上的一个重要技术决策。当时的背景是：MCP 协议已经发布了大约半年，开发者社区已经涌现出成千上万个 MCP 服务器，但如何让普通用户方便地安装和使用这些服务器，是整个生态面临的最核心障碍。这篇文章就是 Anthropic 对这个问题的正式回答。带着这个历史背景去理解它，会更有收获。

好，我们进入正题。

要理解这篇文章在讲什么，先得搞清楚它要解决的问题是什么。所以我们先花几分钟复习一下 MCP 是什么，以及它的安装困境。

MCP 全称是 Model Context Protocol，中文叫模型上下文协议，是 Anthropic 在二零二四年底发布的一个开放标准。它的核心思路说起来很直白：让 Claude 这样的大语言模型，通过一个标准化的接口，访问本地的文件系统、数据库、开发工具，甚至是你机器上正在运行的各种应用程序。

你可以把 MCP 服务器想象成一根专用的数据线。一端插进 Claude，另一端插进你的工具。有了这根线，Claude 就能真正深入到你的工作环境里，读取你的文件、查询你的数据库、操作你的代码编辑器，而不只是在聊天窗口里靠你不停地复制粘贴来提供上下文。这个概念本身非常有吸引力，开发者社区的反应也非常积极。在 GitHub 上，MCP 服务器的数量在短短几个月内就突破了数千个，覆盖了文件读写、数据库操作、代码执行、浏览器控制、各种 API 集成等方方面面的场景，从数字上看，MCP 生态的增长速度是惊人的。

但这里有一个问题，一个相当严重的问题：安装门槛高得离谱。

文章把这个问题总结成了五个具体的痛点，我一个一个来讲，因为这五个痛点是后面所有解决方案的出发点，理解它们很重要。

第一个痛点，叫做开发工具依赖。绝大多数 MCP 服务器是用 Node.js 或者 Python 写的，这意味着在你安装任何 MCP 服务器之前，你得先在电脑上装好对应的语言运行环境。对于习惯命令行的开发者来说，这可能只是十分钟的事情。但对于一个产品经理、设计师、市场人员，或者任何没有编程背景的职场人来说，"去安装 Node.js"这件事本身就是一道过不去的坎。他们可能不知道去哪里下载，不知道怎么配置环境变量，装完了也不知道怎么验证是否成功。光是这第一步，就能挡住百分之九十的潜在用户。

第二个痛点，手动编辑配置文件。Claude Desktop 使用一个 JSON 格式的配置文件来管理所有的 MCP 服务器连接信息，这个文件藏在操作系统的某个目录里。每次想添加一个新的 MCP 服务器，你都要手动找到这个文件，按照特定的 JSON 格式，在正确的位置插入新的配置项。JSON 对括号、逗号、引号的要求极其严格，哪怕是一个多余的逗号或者一个漏掉的引号，都会导致整个配置文件解析失败，然后 Claude Desktop 会默默地启动失败，而你可能完全不知道原因出在哪里。这种体验，对于第一次尝试的用户来说，足以让他们彻底放弃。

第三个痛点，依赖冲突。我来举一个具体的例子帮你理解这个问题。假设你安装了两个 MCP 服务器，服务器 A 需要 Python 三点九，服务器 B 需要 Python 三点十一，这两个版本之间不完全兼容，你在同一台机器上只能让其中一个正常运行。或者你用 npm 安装了两个 Node.js 的服务器，它们都依赖同一个第三方库，但要求的版本不一样，相互冲突。这种依赖管理的噩梦，对于专业开发者来说都够头疼的，对于普通用户来说完全是无解的困境，他们甚至不知道错误信息在说什么。

第四个痛点，没有发现机制。假设你今天想找一个能帮你操作本地日历的 MCP 服务器，你要怎么找？去 GitHub 上搜索关键词，从一堆搜索结果里挑一个，看 README 判断这个项目靠不靠谱，再看 Stars 数量和最近的更新时间，再决定要不要安装。这整个过程既费时间，又完全没有质量保证，依赖的是用户自己的经验和判断力，普通用户根本没有能力独立完成这个评估。

第五个痛点，更新复杂。好不容易把一个 MCP 服务器装好了，过了一个月，开发者推送了新版本，修复了一些 bug，加了几个新功能。你想升级，怎么办？基本上要重走一遍安装流程，再次手动处理依赖，再次编辑配置文件，再次祈祷不出问题。没有任何自动化的机制帮你做这件事。

这五个问题叠加在一起，就形成了一道无形的高墙，把 MCP 服务器丰富的生态圈在了"懂技术的开发者"这个相对小的群体里。用文章里的说法，MCP 服务器尽管功能强大，但对于非技术用户来说基本上无法触及，而这直接限制了整个生态的天花板。

Anthropic 给出的解决方案，叫做 Desktop Extensions，中文叫桌面扩展，对应的文件格式叫点 mcpb，这是 MCP Bundle 的缩写。这里我需要做一个时间线的注解：这篇文章最初发布时，文件格式叫点 dxt，但在二零二五年九月十一日，官方把扩展名统一更新成了点 mcpb。两个格式功能完全相同，只是名字改了，所有已有的点 dxt 文件仍然可以正常使用，但官方建议新项目使用点 mcpb。后面我们统一用点 mcpb 来称呼它。

那这个点 mcpb 文件，本质上是什么？

本质上，它就是一个 ZIP 格式的压缩包。你可以把它类比成手机上的应用安装包——安卓系统有 apk 文件，苹果 iOS 有 ipa 文件，macOS 的应用有时候打成 dmg 文件。这些格式的共同点是：把一个应用程序运行所需要的一切，代码、资源文件、依赖库，全部打包进一个单独的文件里。用户不需要知道里面有什么，不需要自己处理依赖关系，只要双击打开、走完安装向导，应用就装好了，可以直接用。

点 mcpb 对 MCP 服务器来说，就是要做同样的事情。一个点 mcpb 文件里包含：服务器代码、所有第三方依赖库、图标文件，以及最核心的一个 manifest.json 配置清单。用户下载这个文件，双击，点击安装，整个过程不需要打开任何终端，不需要了解任何关于 Node.js 或者 Python 的知识。

用文章里非常直白的对比来说明这个变化有多大。

在有桌面扩展之前，安装一个 MCP 服务器的流程是：首先自己安装好 Node.js，然后在终端里输入 npm install 之类的命令，接着手动找到配置文件目录，打开 JSON 文件编辑，重启 Claude Desktop，最后祈祷它能跑起来。

在有了桌面扩展之后，安装一个 MCP 服务器只需要：下载一个点 mcpb 文件，双击打开，点击安装按钮。三步，全程图形界面，不需要任何命令行操作。

这个体验差距，就是 Desktop Extensions 要弥合的核心价值。如果你觉得这个变化听起来平淡，不妨想一想 iPhone 出现之前智能手机的安装流程，再对比 App Store 把一切简化成"点搜索、点安装"之后，移动应用的用户规模增长了多少倍。同样的逻辑在这里完全适用。

好，我们现在深入看 Desktop Extensions 的技术核心，这部分有不少很有意思的设计细节。核心是 manifest.json 这个文件。

每一个 Desktop Extension 里，唯一必须包含的文件就是这个 manifest.json。它是整个扩展的元数据中心，告诉 Claude Desktop 关于这个扩展的所有信息：叫什么名字、是谁开发的、版本是多少、服务器是什么类型、入口文件在哪里、启动命令是什么、需要用户提供哪些配置、声明了哪些工具能力等等。

最简化版本的 manifest 需要包含这几样东西：MCPB 规范的版本号、扩展的机器可读名称、语义版本号、描述文字、作者信息，以及一个 server 配置对象。server 对象里指明服务器的类型是 node 还是 python 还是二进制可执行文件，入口文件的相对路径，以及启动这个服务器的具体命令和参数。

仅凭这些基本信息，Claude Desktop 就能把这个扩展跑起来。但 manifest 真正强大的地方，是它的用户配置声明机制。我来仔细说说这个，因为我觉得它是整个设计里最精心的部分。

想象你在开发一个 MCP 服务器，这个服务器需要连接一个外部服务，需要用户提供一个 API 密钥才能工作。在传统方式下，你大概会在 README 里写一段说明："请在配置文件里找到这个字段，把你的 API Key 填进去。"然后用户要自己找到那个隐藏在系统目录里的 JSON 文件，按照特定格式手动编辑，格式还不能写错。很多人在这一步就彻底放弃了。

Desktop Extensions 提供了一套更优雅的解决方案。在 manifest 里，你可以声明一个 user_config 字段，用结构化的方式描述你需要用户提供哪些配置项。每个配置项可以有自己的类型、界面标题、描述说明文字、是否必填、是否是敏感信息、默认值是什么。

当用户安装这个扩展的时候，Claude Desktop 会根据这个声明，自动生成一个友好的配置界面——就像你在 App Store 安装一个应用后，它弹出来问你"是否允许访问通讯录"一样，用户只要在界面里填写对应的值就行了，完全不需要了解任何关于 JSON 格式的知识，就像填写一个普通的网页表单。

对于标记了 sensitive 为 true 的配置项，比如 API 密钥，Claude Desktop 不会把它存成普通的文本，而是存进操作系统的安全凭据管理系统里——macOS 上是 Keychain，Windows 上是 Credential Manager。这些敏感值不会以明文形式出现在任何配置文件里，也不会被其他应用程序轻易读取。这是一个非常重要的安全提升，尤其对于需要存储 API 密钥这类高敏感信息的场景来说意义重大。

从开发者的角度来看，这个机制的优雅之处在于模板变量的替换机制。你在 server 配置里，可以用 ${user_config.api_key} 这样的占位符来引用用户填写的值。Claude Desktop 在启动服务器进程的时候，会自动把这个占位符替换成用户实际提供的值，然后通过环境变量的方式传递给服务器进程。你的服务器代码完全不用关心"这个密钥是从哪里来的"，只要从约定好的环境变量里读取就行了。这是一种很清晰的职责分离——manifest 负责声明需要什么，Claude Desktop 负责收集和传递，服务器代码负责使用。

除了 user_config 里的占位符，还有两类系统级的模板变量值得单独提一下。第一个是 ${__dirname}，它代表这个扩展被解压安装之后的实际目录路径。MCP 服务器安装之后会被放到 Claude Desktop 管理的某个内部目录里，这个路径在不同机器、不同系统用户之间是动态的，你没办法预先知道它是什么。有了 ${__dirname}，你只需要写相对的子路径，比如 ${__dirname}/server/index.js，Claude Desktop 会自动把它展开成正确的完整绝对路径。第二类是系统环境变量，比如 ${HOME} 代表当前用户的主目录，${TEMP} 或者 ${TMPDIR} 代表系统临时目录，这些变量在不同操作系统上有不同的实际值，通过模板变量来引用，比在代码里硬编码路径要健壮得多。

manifest 还支持平台差异化配置，这个功能在实践中非常有用。同一个扩展，在 Windows 和 macOS 上可能需要不同的启动命令。比如 Windows 上 Node.js 的可执行文件叫 node.exe，而不只是 node；临时目录在 macOS 上的环境变量叫 TMPDIR，在 Windows 上叫 TEMP，这些细节差异如果不处理，扩展就无法在两个平台上都正常工作。manifest 的 server 配置里有一个 platforms 字段，可以分别为 win32、darwin（也就是 macOS）和 linux 声明各自的差异化配置，哪个平台上运行就用哪套配置，其余配置保持共享。这样你只需要维护一套代码，一次打包，就能同时支持 Windows 和 macOS，不需要为每个平台单独发布一个安装包。

另外，manifest 还支持提前声明工具列表和提示词列表，把这个扩展提供哪些能力写清楚。即使 Claude Desktop 在实际运行时会通过 MCP 协议动态地从服务器获取工具定义，在 manifest 里提前声明还有一个很实用的好处：用户在安装之前就能看到这个扩展提供哪些具体工具，对功能有清晰的预期，而不是装了才知道里面有什么。

现在我们来说说打包流程，也就是开发者具体怎么把一个现有的 MCP 服务器打包成一个 Desktop Extension。

Anthropic 提供了一个叫做 mcpb 的命令行工具，通过 npm 全局安装 @anthropic-ai/mcpb 这个包就可以使用。装好之后，整个打包流程基本上就是两条命令。

第一条命令是 mcpb init。在你的 MCP 服务器项目目录里运行这条命令，它会启动一个交互式向导，问你一系列关于这个服务器的问题：服务器是什么类型的、入口文件在哪里、需要哪些用户配置、支持哪些平台。根据你的回答，它会自动生成一个完整的 manifest.json。如果你不想走完整的交互流程，可以加一个 --yes 参数，工具会用默认值直接生成一个最基础的 manifest，你再手动补充需要的字段。

第二条命令是 mcpb pack。运行这条命令，工具会先验证你的 manifest.json 格式是否正确，所有必填字段是否都已经填写，格式有没有问题，然后把整个项目目录连同所有依赖一起打包成一个点 mcpb 文件。

就这两步，生成的点 mcpb 文件就可以分发给用户安装了。开发者可以把它拖进 Claude Desktop 的设置窗口直接安装测试，确认功能正常之后，提交给 Anthropic 审核，申请上架到内置的扩展目录。

说到内置目录，这也是这次发布的重要组成部分。Claude Desktop 里新增了一个官方扩展目录，用户可以直接在应用内浏览和搜索可用的扩展，一键安装，不需要去 GitHub 自己找，不需要自己评估代码质量，因为 Anthropic 的团队会对提交的扩展进行质量和安全审查，审核通过才会上架。这套机制让发现好扩展变成一件轻松的事，也给了用户一个基本的质量和安全背书。

说到安全，这是一个绕不开的话题。当你允许用户在桌面应用里安装第三方代码、并让它以本地进程身份运行时，潜在的安全风险是相当高的。Desktop Extensions 通过多个层面来管控这个风险：内置目录的人工审核是第一道关；敏感配置存在操作系统密钥串里，不以明文暴露是第二道；用户安装前可以看到扩展声明的工具和权限，是第三道。

对于企业用户，这篇文章专门提到了企业级管控能力。Windows 的 Group Policy 和 macOS 的 MDM（也就是移动设备管理协议）都可以用来集中控制扩展的安装行为。企业 IT 可以把经过审批的扩展预先推送到员工设备上，可以把特定的扩展或者特定的发布者加入黑名单禁止安装，可以完全禁用内置的公共扩展目录，还可以部署一个仅限内部使用的私有扩展目录，员工只能从这个经过企业审核的目录里安装扩展。这些工具加在一起，给了企业 IT 相当精细的管控手段，既不会完全堵死员工使用 AI 工具提效的路，又能把安全风险控制在可接受的范围内。

现在我想聊一个我认为整篇文章里最有战略意义、但也最容易被忽略的部分：开源。

Anthropic 在这次发布里，把整个 Desktop Extensions 生态的基础设施全部开源了，包括完整的 MCPB 规范文档、打包和验证工具、参考实现代码，以及 TypeScript 类型定义和 JSON Schema。

为什么这件事的意义不能小看？

如果点 mcpb 是一个只有 Claude Desktop 支持的私有格式，那它顶多是 Anthropic 的一个产品功能，方便的是 Claude 的用户。但如果这个格式是完全开放的规范，任何其他 AI 桌面应用都可以照着这份规范实现对点 mcpb 的支持，那格局就完全不一样了。

文章里的原话是这么说的：我们希望 MCPB 格式不只是让本地 MCP 服务器在 Claude 里更便携，而是在其他 AI 桌面应用里也能通用。

对于 MCP 服务器的开发者来说，这意味着：你只需要按照 MCPB 规范打包一次，就能在所有支持这个格式的 AI 应用里分发和使用，不需要为每一个应用单独维护一套安装流程。这大大降低了维护成本，也降低了整个生态的碎片化程度。

对于其他 AI 桌面应用的开发者来说，不需要从零设计自己的扩展格式，可以直接采用已经设计好的 MCPB 规范，还有现成的工具和参考实现可以参考，节省了大量的重复发明轮子的工作。

文章特别指出，规范版本号定为零点一，这不是谦虚，而是明确的信号：这是一个邀请社区参与共同演进的起点，而不是钦定的终态。这种开放的姿态，和 MCP 协议之所以能快速成为 AI 工具集成的事实标准如出一辙。Anthropic 现在把同样的策略复制到了桌面扩展格式上。通过建立标准、开放生态来确立影响力，比闭源私有格式在长远来看更有价值。

好，我们来到实践应用这一部分。针对不同角色的听众，我来分别说说这次发布对你意味着什么，以及有哪些具体可以操作的事情。

如果你是一个已经开发了 MCP 服务器的开发者，我强烈建议你现在就着手把你的服务器打包成 Desktop Extension。这件事的成本真的非常低。你需要做的核心工作是写一个 manifest.json，然后跑两条命令，测试一下打包结果。如果你的服务器代码本身已经是功能完整可运行的，这整个打包过程可能只需要一两个小时。但收益可能是巨大的：你的服务器可以从只有少数懂命令行的开发者能安装，变成所有 Claude Desktop 用户都能一键安装，受众规模可能增长几十倍甚至更多。

在设计 user_config 这个部分的时候，我建议仔细思考你的服务器需要用户提供哪些配置，然后把每一项都用正确的类型声明出来，不要偷懒把所有东西都设计成一个字符串让用户自由填写。如果需要 API 密钥，用 string 类型加上 sensitive 设为 true，这样 Claude Desktop 会自动把它托管到操作系统的密钥串里，不会以明文暴露。如果需要用户指定一个本地目录，用 directory 类型，Claude Desktop 会自动提供一个文件夹选择对话框，用户点一下就能选好，不需要手动输入路径字符串。如果需要一个数字参数，用 number 类型并设置合理的最小值和最大值范围，让 Claude Desktop 帮你做范围校验，避免用户输入无效值。

跨平台测试是一个容易被忽略的坑。很多 MCP 服务器的作者是在 macOS 上开发的，在发布之前从没有在 Windows 上测试过。但 Claude Desktop 同时支持 Windows 和 macOS，你提交给官方目录的扩展也需要在两个平台上都能正常工作。常见的坑有这几类：文件路径的分隔符在 Windows 是反斜杠，macOS 是正斜杠；系统临时目录在 macOS 上的环境变量叫 TMPDIR，在 Windows 上叫 TEMP；Node.js 的可执行文件在 Windows 上叫 node.exe 而不是 node。这些差异都可以通过 manifest 的 platforms 字段来处理，为每个平台声明各自的差异化配置，不要把这些判断逻辑写死在服务器代码里。

文章里还提到了一个很实用的技巧，我觉得很值得一试，那就是用 Claude Code 来辅助开发 Desktop Extension。Anthropic 内部自己就是这样做的，效果据说相当好。文章提供了一段专门为这个任务设计的提示词，核心思路是让 Claude Code 先去读 MCPB 的规范文档和示例，彻底理解 manifest 格式和各种最佳实践，然后根据你简要描述的需求，生成一个完整的、可以直接使用的扩展项目，包括 manifest.json、服务器实现代码、错误处理逻辑、日志调试能力，还有基本的文档说明。如果你有一个想法但不知道怎么落地，这个辅助方式可以帮你非常快速地从零到一。你只需要描述清楚你想让这个扩展做什么，剩下的架构和代码可以交给 Claude Code 来起草，你来审核和调整。

如果你是一个 Claude Desktop 的普通用户，我的建议是：更新到最新版本，去设置里找找新增的扩展目录，看看有没有符合你实际工作场景的扩展，试着安装一两个感受一下整个流程。官方目录在刚上线的时候，扩展数量可能还比较有限，但随着开发者社区的持续参与，生态会越来越丰富。作为早期用户，现在正是在别人还没发现之前，找到那些真正能改变你工作效率的工具的好时机。

如果你在企业里负责 AI 工具的采购和推广，桌面扩展的企业级管控能力是你值得重点关注的。很多公司之所以迟迟不敢在内部大规模推广 Claude Desktop，就是因为缺乏对第三方插件的管控手段，担心员工乱装不受信任的扩展带来数据泄露风险。现在有了 MDM 和 Group Policy 的支持，配合私有扩展目录，你可以设计一套完整的扩展使用规范：经过安全审查的扩展统一推送给员工，公共目录或外部来源的扩展不允许安装，并且定期审计员工设备上的扩展安装情况。这套机制配合好了，能在保障安全合规的前提下，让员工真正用上 MCP 生态里那些有价值的工具。

有一个在实践中可能会遇到的问题值得提前了解，文章里没有明说，但你做开发的时候会碰到，就是打包体积的问题。Desktop Extensions 把所有依赖都打包进来，解决了环境依赖的问题，但代价是文件体积会比较大。如果你的 MCP 服务器引用了一个很重的机器学习库或者大型框架，打出来的点 mcpb 文件可能有好几百 MB，这会直接影响用户的下载速度和安装体验。作为开发者，要有意识地精简依赖树，能用轻量级替代方案的就不要引入重型框架，把包体积控制在一个合理的范围内。这是一个体验设计上的基本原则，但在把所有依赖都塞进一个包之后，这个问题会变得更加突出。

最后我想聊一个文章里提到的内部用例，因为它很形象地展示了 Desktop Extensions 的价值边界究竟在哪里。

Anthropic 内部有一个团队，把一个叫 PyBoy 的 GameBoy 模拟器打包成了一个 Desktop Extension，直接让 Claude 控制这个模拟器里正在运行的游戏。这个方向和他们之前做过的那个"Claude 玩宝可梦"的研究一脉相承——都在探索把大语言模型直接接入游戏环境，能产生什么有趣的涌现行为。

这个例子听起来很好玩，但我觉得它背后揭示了一件很严肃的事。一个把模拟器连给 AI 控制的疯狂想法，本来可能只是团队内部某人的一个小实验，分享给同事麻烦，让别人复现更麻烦。有了 Desktop Extensions 之后，这样的实验可以被打包成一个单独的文件，发给任何人，任何人双击安装就能在自己的机器上复现完全相同的环境，做完全相同的实验。这种"从实验到可分发"的飞跃，对于 AI 工具生态的多样性来说意义重大——它把那些本来只能留在少数人实验室里的创意，变成了人人可以触及的可安装工具，大大降低了创意在社区里传播和验证的成本。

你现在日常使用的那些本地工具——你的笔记软件、你的项目管理应用、你的本地数据库、你常用的各种命令行工具——理论上都可以通过一个 Desktop Extension 和 Claude 打通，让 Claude 真正理解并操作你的工作环境，而不是让你每次都要把信息复制粘贴到聊天框里。以前这条路对于普通用户来说太难走，需要太多技术门槛。Desktop Extensions 让这中间的距离缩短为一次双击。

好，我们来收尾。

这篇文章的核心，可以用一句话概括：Desktop Extensions 是 Anthropic 为 MCP 生态设计的"最后一公里"解决方案。MCP 协议解决了 Claude 怎么和本地工具通信的问题，Desktop Extensions 解决了普通用户怎么安装和管理这些通信桥梁的问题。两者加在一起，才构成了一个完整的本地 AI 工具生态的基础设施。这篇文章发表于二零二五年六月，那个时候整个 AI 助手生态正处于一个关键拐点——从"只有开发者能玩"转向"所有人都能用"，Desktop Extensions 是推动这个转变的重要力量之一。

我个人觉得，这次选择把 MCPB 规范完全开源，这个战略决策比任何一个具体的技术细节都更值得关注。如果这个格式真的被其他 AI 桌面应用广泛采纳，成为行业事实标准，Anthropic 就相当于在整个 AI 桌面工具生态里埋下了一个持久的影响力锚点。这种先发优势的价值，往往要在三五年之后才会变得清晰可见。我们可以持续观察。
