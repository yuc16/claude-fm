二〇二五年下半年，是 AI agent 从实验室走向真实生产环境的关键转折期。在这之前，大家谈 agent 更多停留在演示层面——模型调几个工具、走几步流程，给投资人看一个 demo。到了二〇二五年下半年，很多公司开始把 agent 系统真正部署到生产环境，用来跑真实业务。这时候暴露出来的工程问题完全不同了：不再是"模型够不够聪明"，而是"系统够不够稳定、够不够高效、成本能不能控住"。

这篇文章出现的时间点非常准，它回答的正是那批开始认真搭 agent 系统的工程师最头疼的三个问题：工具太多，说明书就把上下文撑满了，怎么办？中间处理数据太多，token 消耗爆炸，怎么办？工具调用老是出错，参数格式不对或者选错工具，怎么办？今天我们来回顾这篇文章，是因为这些问题到今天依然真实存在，这些解决思路到今天依然值得借鉴。

好，我们从头开始讲，先建立问题背景。

想象你在给一个 AI agent 配备工具，就像给一个新来的技术员工配齐工作台上的各种软件。这个 agent 要能操作 GitHub 的仓库、在 Slack 里发消息、查 Jira 的工单、看 Sentry 的报错日志、拉 Grafana 的监控面板、分析 Splunk 的日志数据，再加上 Google Drive 的文档。光这几个系统，合起来就有将近六十个工具。

在 AI agent 的世界里，每一个工具都需要有一段工具定义，也就是 tool definition，用来告诉模型这个工具能干什么、需要哪些参数、返回什么格式的数据。这些定义本质上是一段 JSON，但为了让模型在各种情况下都能正确理解和使用，往往写得很详细，少则几百字，多则几千字。

文章给了一组非常直观的数字。GitHub 这一个服务有三十五个工具，光是这些工具的说明书就占大约两万六千个 token；Slack 有十一个工具，占两万一千个 token；Sentry 有五个工具占三千个，Grafana 五个工具占三千个，Splunk 两个工具占两千个。把这五个系统的工具全部接进来，还没开始处理任何用户请求，光工具说明书就消耗了大约五万五千个 token。要是再把 Jira 加进来，仅 Jira 一家就需要一万七千个 token，你的总开销轻松突破七万。

Anthropic 自己在内部的一个项目里，在做优化之前，工具定义一共占用了十三万四千个 token。想一想，Claude 的上下文窗口哪怕有二十万个 token，被这一项就吃掉了将近七成，剩下的空间要用来承载系统提示、对话历史、用户的真实请求，就相当紧张了。

但 token 消耗还只是问题的一半。另一半是准确率。工具越多，模型在选择工具时出错的概率也在上升。文章特别点名了最常见的失败模式：工具名字相似时，模型很容易选错。比如有两个工具叫 notification-send-user 和 notification-send-channel，前者是给某个特定用户发通知，后者是给一个频道发广播，模型很容易搞混，然后你的用户就在莫名其妙地收到本该发给频道的消息，或者频道突然被一堆本来是私信的内容淹没。

除了工具选择的问题，工具调用本身也有效率瓶颈。当 agent 需要处理大量数据的时候，每次工具调用返回的结果都会进入模型的上下文，积累下来占用大量 token，而且很多中间数据其实是噪音，对最终决策没有帮助，反而可能分散模型的注意力。

这就是文章要解决的三个核心问题。Anthropic 给出了三个功能，一个一个来解决。

第一个功能叫 Tool Search Tool，我把它翻译成"工具搜索工具"。

这个名字初听有点绕，因为它本身就是一个工具，专门用来搜索其他工具的。它的核心思路是：工具不用全部提前加载到上下文里，按需取用就好。

用一个生活类比来解释。你去一个有五万本书的大图书馆。传统做法是，你进馆之前，图书管理员把所有书的完整目录打印出来，几百页，递给你，说："这是我们的所有藏书，你自己找吧。"你要先把目录看完才能知道要借什么，而这本目录本身就占了你桌子的大半个空间。工具搜索工具的做法是：门口放一台检索机，你进来什么都不用拿，想找什么直接在机器上搜，系统马上把相关的几本书推荐给你，你拿着书去读就好。大部分书始终在书架上，只有你当前需要的才被摆到面前。

在技术实现上，这个功能需要你在配置工具的时候，给每个工具加一个参数，叫 defer-loading，设置成 true，表示这个工具是延迟加载的，一开始不要放进模型的上下文。一旦这样设置，模型在初始状态下的上下文里，只有 Tool Search Tool 这一个工具本身，大约占五百个 token。当模型在处理任务中发现自己需要某类能力，比如需要和 GitHub 交互，它就调用 Tool Search Tool 搜索"github"，系统返回匹配到的工具定义，比如 github-create-pull-request 和 github-list-issues 这两个，大约占三千个 token，这时候这两个工具的定义才被动态注入到模型的上下文里。其他 Slack 的、Jira 的、Google Drive 的那几十个工具，全程都不需要占用上下文空间。

让我们对比一下数字。传统方案下，五十多个工具全部提前加载，消耗大约七万两千个 token，加上系统提示和最初的用户问题，总共大概七万七千个 token，还没开始任何实际工作。使用 Tool Search Tool 之后，总消耗降到了大约八千七百个 token，节省了整整百分之八十五的上下文空间。

准确率的提升同样非常显著。Anthropic 内部做了一批包含大量工具的 MCP 评估测试，结果是：Claude Opus 四的工具选择准确率从百分之四十九提升到了百分之七十四；Opus 四点五从百分之七十九点五提升到了百分之八十八点一。原因很直观：当上下文里只有几个相关工具的时候，模型不需要在几十个工具里做选择，名字相似工具之间互相干扰的问题也消失了，选错的概率自然大大降低。

在具体实现上，平台提供了两种内置的搜索方式。一种是基于正则表达式的搜索，适合精确匹配关键词的场景；另一种是基于 BM25 算法的搜索，BM25 是一种经典的文本相关性算法，更擅长处理语义相近的查询，比如模型说"我需要找跟消息通知相关的工具"，BM25 能比正则更准确地找到对应工具。如果这两种都满足不了需求，你也可以自己实现一个自定义搜索工具，比如基于向量嵌入的语义搜索。

有一个工程细节值得单独说一下：Tool Search Tool 不会破坏 prompt caching，也就是提示词缓存。这一点在生产环境里很重要。提示词缓存是一个降低 API 调用成本的机制，原理是把重复出现的系统提示和工具定义缓存起来，下次调用就不用重新处理。如果每次都有新的工具定义动态加入，缓存就会失效，成本就上去了。而 Tool Search Tool 的机制是把延迟加载的工具完全排除在初始提示词之外，它们只有在被搜索到之后才出现，所以系统提示和核心工具定义保持稳定，缓存命中率不受影响。这是一个很细心的设计。

当然，Tool Search Tool 也有它的适用边界，不是所有情况都需要用。它增加了一个搜索步骤，也就是说每次用工具之前，模型要先搜一下，这会带来额外的延迟。文章建议，当你的工具定义消耗超过一万个 token，或者你开始遇到工具选择准确率问题，再考虑启用。如果工具库本身很小，不超过十个工具，或者每次会话几乎所有工具都要用到，那就直接预加载，反而更省事。

文章还给了一个平衡策略：把你最常用的三到五个核心工具设置为默认加载，不需要延迟，其他工具才设置为延迟加载。这样高频操作可以直接使用，不需要先搜索，而不常用的工具再走按需发现的路径。

好，我们来聊第二个功能：Programmatic Tool Calling，缩写是 PTC，中文可以叫"程序化工具调用"。

如果说 Tool Search Tool 解决的是"工具说明书太多"的问题，PTC 解决的是另一个维度的问题：工具调用产生的中间结果太多，全都堆进上下文，不但占空间，还会干扰模型对关键信息的判断。

我再用一个类比来帮助理解。假设你要做一道复杂的菜，需要二十种食材。传统做法是：你每次去超市买回一种食材，拿给厨师看，厨师记在脑子里，告诉你去买下一种。二十种食材，来回跑二十趟。厨房里堆满了原材料，厨师脑子里要同时记住二十种东西的状态。PTC 的做法是：厨师写一段程序，描述整个备料和烹饪流程，交给一个助理去执行，助理按照程序买食材、处理食材、完成烹饪，最后只需要把做好的菜端上来。厨师不需要盯着每一个中间步骤，也不需要把所有原材料都存在自己脑子里。

用文章里的具体例子来说明。假设你要找出哪些团队成员超出了他们第三季度的差旅预算。你有三个工具：get-team-members 用来获取部门成员名单，大概返回二十个人；get-expenses 用来查某个员工某个季度的报销明细，每人大概有五十到一百条记录，包括机票、酒店、餐饮等逐条明细；get-budget-by-level 用来查某个职级对应的预算上限。

传统的工具调用方式是这样的：模型请求调用 get-team-members，获得二十人名单，名单进入上下文；然后对每个人逐一调用 get-expenses，二十个工具调用依次进行，每人五十到一百条明细，总共一千到两千条记录全部返回到模型上下文；再调用几次 get-budget-by-level 获取预算上限。然后模型要在上下文里逐一加总每个人的报销额，与预算上限对比，找出超支的人。整个过程里，那两千条报销明细中的绝大多数都是无关信息，光是读取和处理这些数据就消耗了大量 token，而且这些噪音数据也可能影响模型的注意力和判断。

PTC 的做法完全不同。模型不是一条一条地请求工具，而是写一段 Python 代码，把整个工作流程的逻辑都写进去。这段代码在一个沙盒执行环境里运行，代码里调用这些工具，数据在代码里处理，最终只把需要告诉模型的结论返回给它。

模型写的代码大概是这样的逻辑：先获取工程部门的所有成员，提取所有不同的职级，并行查询每个职级的预算上限，建立一个职级到预算的映射表；然后并行发起对所有二十名成员的报销查询；接着遍历每个成员，计算总报销额，与预算上限对比，把超支的人记录下来，最后输出结果列表。

整个运算过程中，两千条报销明细、各种中间变量全部在代码运行环境里处理，不会进入模型的上下文。模型最终只看到执行结果：比如三个人超出了预算，各自超出了多少金额。从原来可能占五十多千字节的原始报销数据，变成最终结论只有约一千字节。

Anthropic 的测试数据很有说服力。使用 PTC 之后，在复杂研究任务的测试集上，平均 token 消耗从四万三千五百八十八降到了两万七千二百九十七，减少了百分之三十七。

延迟方面的改善同样明显。传统方式下，二十个工具调用如果顺序执行，每次都要等前一个结果返回，触发一轮模型推理，再发下一个请求。PTC 里的代码可以用异步方式并行发起所有调用，二十个查询同时进行，总等待时间大幅缩短。文章里提到，当 agent 一次性编排二十个以上的工具调用时，可以省掉十九个以上的推理回合。

准确率也有提升：内部知识检索测试从百分之二十五点六提升到了百分之二十八点五；GIA 基准测试从百分之四十六点五提升到了百分之五十一点二。原因是用代码表达工作流逻辑比用自然语言推理要精确——循环、条件判断、数据聚合全都是显式的，不容易出偏差；而且模型不需要在上下文里"眼睛扫一遍"几千条原始数据，减少了认知干扰。

文章里还提到了一个真实的产品案例：Anthropic 内部的 Claude for Excel 项目，用 PTC 来读写有几千行数据的 Excel 表格，没有撑爆上下文窗口。这对很多需要处理大型表格数据的业务场景非常有启发——以前这类任务要么做不了，要么要手动分批，现在有了一个更优雅的工程解法。

你可能会问：让模型写代码来调用工具，这靠谱吗？写出来的代码会不会出错？这是一个合理的担忧。文章的回答是：Claude 本身非常擅长写代码，让它用 Python 来表达工具编排逻辑，比让它用自然语言逐步推理要精确得多。而且代码跑在沙盒环境里，出了问题也不会影响外部系统的安全。

PTC 也有它不适用的场景。如果只是做一个简单的单次工具调用，PTC 带来的额外代码执行开销就没有必要了。另外，如果你的任务要求模型审阅所有中间数据来做决策——比如不只是看总额，而是要对每一条报销记录做判断——那就不应该用 PTC 来屏蔽中间数据，让模型直接看到原始结果更合适。

好，我们来看第三个功能，也是三个里面实现最简单但效果同样显著的：Tool Use Examples，中文叫"工具使用示例"。

这个功能要解决的问题，来自 JSON Schema 的一个根本性局限。JSON Schema 非常擅长定义结构——哪个字段是字符串，哪个是整数，哪些是必填的，取值范围是什么。但它没法表达使用模式，也就是"怎么才算用得对、用得好"这类经验性规则。

文章举了一个支持工单工具的例子，这个工具有这些字段：标题、优先级（低中高紧急四个选项）、标签数组、报告人信息（包含 ID、姓名、联系方式）、到期日期、升级规则（包含等级、是否通知经理、SLA 小时数）。

JSON Schema 能告诉模型这些字段的类型，但留下了一堆问题没有回答。

比如日期字段，到底该用什么格式？"2024-11-06"这种国际标准格式？"Nov 6, 2024"这种英文习惯写法？还是带时区的完整时间戳？Schema 只说这是一个字符串，完全没有格式约束。

再比如报告人 ID，到底是纯数字"12345"，带前缀的"USR-12345"，还是完整的 UUID？从 Schema 里根本看不出来，因为三种都是合法的字符串。

还有标签的写法：大写还是小写？空格分隔还是连字符？"Bug Fix"和"bug-fix"都是合法的字符串数组元素。

嵌套结构的填写规律：什么时候该填报告人的联系方式？紧急 bug 要填，这说得过去，但功能请求要填吗？内部任务要填吗？Schema 说这些字段是可选的，但没说什么场景下"应该"填。

升级规则和优先级之间的对应关系：Critical 级别的 bug 应该设几级升级？SLA 给几小时算合理？这些业务约定在 Schema 里完全没有体现。

如果模型靠猜来填这些字段，就会产生格式不一致、参数组合不对的问题，运气不好还会直接导致 API 调用失败，或者产生逻辑上不合理的工单数据。

Tool Use Examples 的解决方案非常直接：在工具定义里加一个字段叫 input-examples，里面放几个完整的调用示例。文章给出了三个示例，覆盖了三种典型场景。

第一个示例是一个生产环境的紧急登录报错，填了所有字段：标题描述清楚，优先级是 critical，标签包含 bug、authentication、production，报告人有完整的 ID（USR-12345）、姓名、电子邮件和电话，日期格式是 2024-11-06，升级规则是二级升级、通知经理、四小时 SLA。

第二个示例是添加暗色模式的功能请求，标题写清楚了，标签是 feature-request 和 ui，报告人有 ID 和姓名，但没有联系方式，没有升级规则，没有到期日期。

第三个示例是更新 API 文档的内部任务，整个示例只有一个标题字段，其他什么都没有。

从这三个例子，模型能学到一整套使用规范：日期格式是 YYYY-MM-DD；用户 ID 是 USR- 加五位数；标签是连字符分隔的小写词；优先级越高、越紧急，需要填的字段越多；功能请求不需要升级规则；内部任务只需要标题。这些规则如果都写成文字说明，会又长又难读，但通过三个具体例子，模型能快速建立起正确的使用直觉。

Anthropic 内部测试的结果：加了 Tool Use Examples 之后，复杂参数处理的准确率从百分之七十二提升到了百分之九十，接近二十个百分点的提升，代价只是在工具定义里多了几个示例。

当然，示例会增加工具定义的 token 消耗，所以要用在最值得的地方。最适合加示例的情况是：有复杂嵌套结构的工具、可选参数很多且填写规律不明显的工具、有领域特定约定的 API，以及容易被相互混淆的功能相似工具——比如 create-ticket 和 create-incident，功能相近但使用场景和参数习惯都不同，加几个对比示例能大大减少模型选错和填错的概率。

如果工具很简单，比如就一个参数、含义一目了然，或者参数格式是 URL 或者邮件地址这类模型本来就熟悉的格式，就不需要加示例，白占 token。

好，现在我们把三个功能放在一起，谈谈实际工程中怎么用这些工具。

文章给了一个务实的策略：不要一上来就把三个功能全部堆上去，先找到你当前最大的瓶颈，针对性地解决。

如果你现在的主要问题是上下文被工具说明书撑满——接入了很多服务，每次会话一开始就有大量 token 被工具定义消耗——那就先启用 Tool Search Tool。

如果你的主要问题是中间数据太多——agent 在处理数据密集型任务时，大量中间结果堆在上下文，token 消耗爆炸——那就试试 Programmatic Tool Calling。

如果你的主要问题是工具调用频繁出错——模型老是用错参数格式，或者在功能类似的工具间选错——那就给关键工具加上 Tool Use Examples。

当然，这三个功能也可以叠加使用，而且效果互补：Tool Search Tool 确保找到正确的工具，PTC 确保高效执行，Tool Use Examples 确保调用时参数正确。对于一个复杂的生产环境 agent，三个一起用，能同时解决"找工具难、用工具贵、调工具准确率低"这三个核心问题。

在工程细节上，有几个建议值得记住。

第一，工具的名称和描述要清晰、有信息量。Tool Search Tool 是基于工具名称和描述来做匹配的，所以描述越准确，搜索效果越好。好的命名方式是"search-customer-orders"配上"按日期范围、状态或金额搜索客户订单，返回包含商品、物流和支付信息的订单详情"这样的说明；糟糕的写法是"query-db-orders"配上"执行订单查询"，既没有告诉模型能按什么条件搜索，也没有说返回什么信息。

第二，在系统提示里告诉模型你接入了哪些类别的工具。比如："你可以使用与 Slack 消息、Google Drive 文件管理、Jira 工单跟踪和 GitHub 仓库操作相关的工具，使用工具搜索来查找具体功能。"这样模型在需要某类能力的时候，能更准确地构造搜索词，找到正确的工具，而不是盲目搜索。

第三，对于 PTC，工具的返回格式要在工具描述里写清楚。因为模型要写代码来解析这些返回值，如果描述里清楚地说明了每个字段的类型、可能取值、时间格式是 ISO 8601 还是 Unix 时间戳，模型写出来的解析代码就会更准确、更健壮。

第四，给 PTC 开放的工具，优先选那些可以并行运行的、互相独立的操作，以及操作本身是幂等的、出了问题可以安全重试的工具。这类工具能从 PTC 的并行执行和错误处理能力里获益最多。

第五，Tool Use Examples 里的数据要用真实数据，不要用"string"、"value"、"foo"这种占位符。示例里的城市名要是真实城市，价格要是合理的数值，时间格式要和你的实际系统一致。每个工具配一到五个示例就够了，覆盖最简单用法、几种典型场景，以及容易出错的边界情况。

还有一个时效性上的补充。这些功能在二〇二五年十一月发布时是 beta 版，启用时需要在 API 调用里加一个 beta 头信息。到今天二〇二六年六月，距发布已近七个月，这些功能的状态很可能已经更新，建议使用时以最新的官方文档为准，避免因为 beta 接口变动导致代码不兼容。

好，我们来做最后的总结。

今天这期节目讲的是 Anthropic 在二〇二五年十一月发布的三个高级工具使用功能。Tool Search Tool 通过按需发现工具，把工具定义对上下文的占用减少了百分之八十五，同时工具选择准确率大幅提升，Opus 四从百分之四十九涨到了百分之七十四；Programmatic Tool Calling 让模型用代码编排复杂任务，平均 token 消耗减少了百分之三十七，并行执行减少了大量推理来回，准确率也有提升；Tool Use Examples 通过提供具体调用示例，把复杂参数处理的准确率从百分之七十二提升到了百分之九十。

我个人的评价是：这三个功能的设计思路都非常务实，不是为了学术新奇而设计的，而是直接针对生产环境里真实暴露出来的工程痛点。它们背后有数据支撑，有真实的内部产品验证，而且三个功能的边界很清晰，各解决一类问题，组合在一起又能形成互补。对于所有在认真搭建 agent 系统的工程师来说，这篇文章值得认真读，它回答的这些问题，迟早你都会遇到。
