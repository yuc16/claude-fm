## Script

今天要跟大家聊的这篇文章，题目叫做"Quantifying infrastructure noise in agentic coding evals"，中文我把它翻译成"量化智能体编程评测中的基础设施噪声"。文章发表于二零二六年二月五日，来自 Anthropic 工程博客，主要作者是 Gian Segato，Nicholas Carlini、Jeremy Hadfield、Mike Merrill 和 Alex Shaw 也为这项工作做出了贡献。

今天是二零二六年六月十二日，距离文章发表过去了大约四个多月。在 AI 这个行业，四个月不短，但这篇文章讨论的核心问题——AI 编程评测到底有多可信——随着模型能力和市场竞争的持续升温，只会越来越重要，一点都没有过时。

先说说背景，让没有特别关注这个话题的朋友也能快速进入状态。

二零二六年初，AI agent 真正开始从实验室产品变成很多工程团队实际在用的生产力工具。越来越多的团队开始用 AI agent 来处理真实的软件工程任务，修 bug、写新功能、做代码审查。与此同时，衡量这些 agent 能力的评测集也成了行业里最被广泛引用的参考指标。

SWE-bench 和 Terminal-Bench 是其中两个最知名的。SWE-bench 要求模型解决真实 GitHub 上报出的 issue，也就是真实的软件 bug 或者功能请求；Terminal-Bench 则更广泛地测试在终端环境里完成各种编程任务的能力。这些排行榜上的分数，不只是研究人员关注的学术指标，它们是媒体报道的依据、投资者判断赛道格局的参考，甚至是企业技术选型的重要信号。

问题在于，这些排行榜上的顶尖模型，往往只相差几个百分点。第一名七十三，第二名七十一点五，第三名七十点八。每一个小数点的差距都被当成"这家公司的模型更智能"的证明，被放大解读。

就在这个背景下，Anthropic 的工程师们做了一件相当务实的事：他们去测了测，这些精确到小数点后两位的分数，到底有多少是基础设施配置带来的噪声。

结论让人有点啊了一下。在他们的实验里，仅仅改变运行评测的基础设施配置——也就是给 AI 分配多少内存和 CPU——就能让 Terminal-Bench 二点零的分数产生高达六个百分点的差距，而且这个效应的统计显著性很强，P 值小于零点零一。六个百分点，足以在很多排行榜上把第一名打到第五名，或者把第五名推上第一名。

这就是今天我们要仔细拆解的故事。

要理解这个发现，首先需要搞清楚一个根本性的区别：传统的 benchmark 和现代的 agentic coding eval，测的其实是完全不同的东西，而这个区别正是问题的起点。

传统的语言模型 benchmark 是静态的。你给模型一个问题——一道数学题、一段代码补全、一个选择题——模型输出一个答案，你判断答案对不对。整个过程是静态的，运行环境无关紧要。这个模型在你的笔记本上跑还是在超级计算机上跑，给出的文本答案是完全一样的，因为它只是在做文本预测，不执行任何依赖计算资源的操作。

但 agentic coding eval 完全不同。你想象一下 Terminal-Bench 的工作方式：模型会被放进一个真实的 Linux 环境，拿到一个任务描述，然后它可以做任何事情。写代码，运行程序，安装软件包，看报错，修改代码，再运行，来来回回跑好多轮，直到完成任务或者时间到了为止。

这个过程跟一个真实的工程师坐在电脑前写代码，在结构上是完全一样的。你给一个工程师配备一台内存充足、CPU 强劲的高配服务器，他能完成的工作，跟你给他一台内存只有两 G 还动不动崩溃的旧笔记本能完成的工作，根本就不是一个量级。

这个道理说出来像废话，但在 AI 评测领域，这个道理长期被忽视。大家默认"把模型放进去，跑出来的分数就是模型能力的纯粹体现"。但实际上，这是在把一个端到端的系统测试，当成了纯粹的模型能力测试。运行时环境不再是一个被动的容器，它成了整个解题过程的一部分。两个用不同资源预算运行的 agent，测的根本不是同一道题。

好，背景说清楚了。我们来讲 Anthropic 工程师们具体是怎么发现这个问题的，这个发现过程本身就很有故事性。

他们在内部搭建了一套跑 Terminal-Bench 二点零评测的基础设施，用的是 Google Kubernetes Engine，也就是 GKE，一种云端容器化部署里最常用的平台之一。你可以把 Kubernetes 想象成一个大型任务调度系统：你告诉它每个任务需要多少内存和 CPU，它负责把这些任务分配到实际的服务器上执行。

搭好之后，他们跑了一批评测，把分数和 Terminal-Bench 官方排行榜一对比，发现对不上。

更让他们意外的是另一个数字：在自己的基础设施上，大约有百分之六的评测任务是因为基础设施错误而失败的，不是因为模型答错了。最常见的错误类型，是容器因为内存超出上限被强制杀掉，用技术行话叫做"OOM kill"，out-of-memory kill，内存溢出终止进程。

百分之六。在一个一百道题的评测集里，有六道题不是因为"模型没答对"而失败，而是因为"程序跑着跑着被操作系统一刀砍掉了"而失败。这些失败跟模型智不智能完全没有关系，是纯粹的基础设施问题。

那为什么会有这么高的错误率呢？答案在一个很关键的技术细节里。

Kubernetes 容器的资源管理有两个独立的参数。第一个叫 requests，也就是资源申请量，是系统保证你能拿到的最低资源量，是一个下限。第二个叫 limits，是资源限制量，是容器能使用的最高上限，超过这条线，容器会被立刻杀掉。

Terminal-Bench 的文档里给每个任务都规定了推荐的 CPU 和内存数值，但它只给了一个单一的数字，没有说这个数字该用作 requests 还是 limits。

Anthropic 的工程师按照最直觉的方式处理这个规格：把推荐值同时设为 requests 和 limits，保证量和上限完全相同，零余量。逻辑上听起来公平合理，你要多少给你多少，不多也不少。

但问题来了。真实程序的内存使用从来都不是一条平坦的直线，它是有波动的，有峰值的。一个任务在运行的大部分时间里可能只用了规格量的八成，但在某个特定的瞬间——加载一个大文件，启动一个子进程，运行一批内存密集型的测试——内存会短暂地超出规格量。

如果 limits 等于规格量，这个瞬间的超出就会触发 OOM kill，任务失败。而实际上，只要给一点点余量让这个峰值过去，任务本来可以顺利完成。

再看 Terminal-Bench 官方排行榜是怎么做的。他们用的是另一家沙盒服务商，这家服务商的实现更宽松：允许容器在短时间内超出分配的上限，只要是短暂的超用，不会立刻终止进程。这就像家里的电表有额定功率，但如果你只是瞬间开了微波炉导致短暂超标，电表不会立刻跳闸。Anthropic 最初的 Kubernetes 配置，相当于一旦超标就立刻断电，不给任何缓冲。

这就解释了为什么官方排行榜的基础设施错误率更低、分数也更高。

发现这个差异之后，工程师们意识到：这不只是"我们的配置有问题需要修一下"的小事情，背后有一个更大的问题需要回答——资源配置对评测分数的影响，到底有多深？

为了系统地回答这个问题，他们设计了一个非常干净的受控实验。所有变量锁死：同一个 Claude 模型，同一套评测脚手架，同一批 Terminal-Bench 二点零的题目。只改一个参数：资源上限是 Terminal-Bench 推荐规格的几倍。

他们测试了六种配置，从一倍——也就是严格按照推荐规格设置硬上限，申请量等于上限——一路增加，直到完全不设任何上限为止。

数据出来之后，里面藏着两层故事。我来分别讲解。

第一层故事，关于基础设施可靠性。

随着资源上限倍数增加，基础设施错误率单调递减，非常规律。从一倍时的百分之五点八，逐步下降，到三倍时降到了百分之二点一，最后在完全不设上限时降到了百分之零点五。统计显著性很强，从一倍到三倍的下降，P 值小于零点零零一，这绝对不是随机波动，是真实存在的效应。

道理也简单：上限越宽，那些因为瞬间内存波动被"误杀"的容器越少，基础设施引入的错误越少，测试越稳定，越接近"公平测量模型能力"的理想状态。

但在一倍到三倍这个区间里，有一个值得注意的细节：虽然基础设施错误率在显著下降，任务成功率的变化却在统计噪声范围之内，P 值等于零点四，没有统计显著性。

这说明什么？说明那些在一倍配置下因为资源超限而崩溃的任务，绝大多数本来就不会被模型正确解决——模型在这些任务上走的路根本就不对。容器崩了，只是比"答错了"更早地结束了这道题，并没有改变最终结果。

用一个类比来理解：考场的空调坏了，一些考生因为太热中暑交了白卷。修好空调之后，这些考生不再中暑，但他们还是不会做那些题，分数没有大幅提高。修空调是让测试环境更公平，而不是让考生变得更聪明。

这就是第一层故事：从一倍到三倍，额外的资源主要是在修复评测基础设施的可靠性问题，让测试结果更准确地反映模型能力，但不会让模型的表现变好或变差。这也是 Terminal-Bench 官方排行榜采用宽松沙盒背后的隐含逻辑——它是在追求基础设施稳定性，而不是在给 agent 开小灶。

然后是第二层故事，也是这篇文章最让人深思的核心发现。

从三倍配置开始往上，趋势发生了明显的变化，一种不成比例的分离出现了。

在三倍到完全不设上限这个区间里，基础设施错误率继续下降，但只额外降了一点六个百分点。然而，任务成功率却跳升了将近四个百分点。两个数字完全不成比例，差距扩大了。

这说明：在这个区间里，额外的资源不只是在消除基础设施噪声，而是在实实在在地帮助 AI agent 解决原本解决不了的问题。资源的增加，改变了 agent 能用的策略，因此改变了最终结果。

文章给了几个非常具体的例子。

有一个任务叫"bn-fit-modify"，要求做贝叶斯网络拟合。贝叶斯网络是概率图模型领域的经典工具，运算起来有一定复杂度。面对这个任务，不同的 AI agent 会有截然不同的默认策略。很多模型的第一反应是：安装现成的工具库！安装 pandas、networkx、scikit-learn，调用现成的函数，三步搞定。这是非常合理的工程思路，对人类工程师来说也会是同样的选择——不造轮子，用成熟的库。

在资源宽松的环境下，这个策略完全可行：安装依赖包，调用函数，任务完成。但在资源紧张的环境下，这个策略从一开始就注定失败——容器在安装这些依赖库的过程中就内存溢出了，模型一行解题代码都还没写到。

但解这道题还有另一条路：用 Python 标准库，从头实现贝叶斯网络所需的数学运算，不依赖任何第三方库。这个方法写起来复杂，需要扎实的数学基础，但内存占用极小，在任何资源配置下都能运行。

关键在于：有些模型默认走"安装工具库"的策略，有些模型默认走"从头实现"的策略。这是不同模型的不同风格，反映的是各自的训练偏好和数据分布。而资源配置，就决定了哪种策略能够成功。

文章还提到了另外两个任务。"rstan-to-pystan"要求把 Stan 统计语言的代码转换成 Python，"compile-compcert"要求编译一个 C 语言编译器。这类任务本身就需要大量内存，在资源紧张的时候成功率几乎为零，一旦有了足够的内存，成功率会显著提升。原因不是模型忽然变聪明了，而是这些任务从物理上就需要更多内存才能运行。

我们用一个生活类比来帮助理解这件事。想象你要考察厨师的烹饪能力，给所有人出同一道题：在一小时内做出一道有代表性的菜。

如果你给每个厨师全套厨具、充足的食材，测出来的是"在充足条件下谁的厨艺更好"。

如果你只给每人一口锅和三种食材，测出来的是"在极端限制下谁的应变能力更强"。

这两种测试都有意义，都值得做。但如果你把两组厨师在不同条件下的分数混在同一张排行榜上，结论就失去了意义。

更重要的是：不同厨师面对"自由发挥"的指令，会有完全不同的默认策略。有的厨师习惯用复杂精致的厨具做出精细的料理，有的厨师习惯用最简单的手法做出可靠的结果。资源条件，决定了哪种风格能够胜出。

这就是 AI agent 在 agentic coding eval 里面临的处境。

文章用了一句很准确的话来总结：紧资源配置无意中奖励了高效精简的策略，宽松资源配置奖励了能充分利用一切可用资源的策略。两种能力都是合理的测试目标，但如果你不说清楚资源配置，把两种配置下跑出来的分数放在同一个排行榜上比，这个排行榜就变成了一团无法清晰解读的数字。

讲完了 Terminal-Bench 的主实验，文章还做了两个重要验证，来看看这个发现有多普遍。

第一个是把同一个实验在不同的 Anthropic 模型上重复。结论是方向一致，幅度各有不同。每个模型在资源增加时都表现出成功率提升的趋势，说明这不是某个特定模型的特殊行为，而是一个普遍现象。不同模型幅度不同，也进一步印证了"不同模型有不同的默认策略，因此对资源边界的敏感度不同"这个观察是真实的。

第二个验证是在 SWE-bench 上做的交叉实验，用一个完全不同的评测集来看效果是否一致。他们选了两百二十七道题，每道题做了十次采样，把可用内存从基准值逐步增加到五倍。

结果是：同样的趋势出现了，但幅度更小。在五倍内存下，比一倍内存时的成功率高了大约一点五四个百分点。这个数字比 Terminal-Bench 上的六个百分点要小很多，原因也很自然：SWE-bench 的任务普遍更轻量，修复一个 GitHub issue 通常不需要安装大型依赖，不需要做内存密集型的编译，所以内存边界效应没那么明显。

但是，即使效应更小，方向是完全一致的。"资源配置影响评测分数"这个结论，不是 Terminal-Bench 独有的特性，而是 agentic coding eval 这类端到端测试的普遍规律。只要是测试 agent 在真实环境中的行为，资源配置就不是中性的背景变量。

文章还延伸讨论了一个更广泛的视角：在 agentic eval 的整个系统里，潜在的噪声来源远不止内存这一项。

时间限制是另一个维度。给 agent 更多时间，它可以尝试更多方法，有时候能解决原本解决不了的问题。在某些任务上，时间边界和内存边界的作用机制是一样的——充足才能跑通，不够就直接失败。

CPU 的速度和核心数会影响编译和测试的耗时，在有时间限制的情况下，CPU 更强的环境就是有优势的。并发度，也就是同时运行多少个评测任务，会影响每个容器实际能拿到的资源，你配置了四 G 内存，但如果同时有五十个任务在抢资源，每个任务实际拿到的可能远不到四 G。出口网络带宽会影响安装依赖包的速度，在有时间限制的情况下，慢网络也会让某些依赖重型工具的策略直接失效。

最有趣的是文章提到的一个观察：他们发现评测通过率会随一天中的时间不同而波动。最可能的原因，是 API 请求延迟随着流量变化而波动。早上用户少，API 响应快，agent 每轮的思考-行动循环就短，在有限的时间内能完成更多轮迭代；下午高峰期 API 响应慢，有些任务可能就在等待响应的时候把时间配额耗尽了。他们没有把这个效应正式量化，但这个现象本身说明的问题很深刻：你在一天中哪个时间点跑 benchmark，可能也会影响你看到的数字。

这些因素加在一起，就是文章所说的：在 agentic eval 里，"模型能力"和"基础设施行为"之间的边界，比一个精确到小数点的数字所暗示的要模糊得多。模型提供商可以通过使用专用硬件来屏蔽这些变量，但外部评测者很难做到同样的控制。

好，讲完了发现和分析，我们来聊聊这篇文章给出的具体建议，以及对我们实际工作的意义。

文章给评测框架开发者的核心建议是：把资源"申请量"和"硬上限"作为两个独立的参数分别指定，而不是只给一个单一的值。一个单一的精确值，会让申请量等于上限，没有任何余量，导致瞬间的内存波动就能触发 OOM kill，让评测结果不稳定。

合理的做法是：用推荐规格作为申请量的下限，然后设置一个更高的硬上限来容纳短暂的波动。硬上限应该是申请量的几倍，文章的建议是用实验来校准：找到一个倍数，使得这个倍数附近的分数差异在统计噪声范围之内，但基础设施错误率已经降到了合理水平。根据他们在 Terminal-Bench 上的数据，三倍是一个不错的起点：把基础设施错误率从百分之五点八降到了百分之二点一，而对成功率的影响仍然在统计噪声范围内，P 值零点四。这个权衡是合理的：基础设施噪声被大幅压制，但没有给 agent 提供过于宽裕的资源，让评测偏离原本要测试的能力。当然，具体的倍数会因评测集和任务分布而异，应该跟分数一起公开报告出来。

对于消费排行榜结果的人，文章的核心信息是：在资源配置没有标准化之前，排行榜上小于三个百分点的差距，应该保持审慎的怀疑态度，直到有人在相同的硬件和配置下复现了这个结果。文章说得很直白：几个百分点的领先，可能意味着真实的能力优势，也可能只是对方用了更大的虚拟机。

这对我们实际工作有几个很具体的启示，我来逐一说说。

如果你在公司里负责评测 AI agent 或者模型，最直接的行动点是：把基础设施配置当成实验变量来管理，而不是一个设定好就不用管的背景参数。就像你会仔细记录 prompt 格式、采样温度、模型版本一样，你也应该版本化地记录容器的内存申请量和上限、CPU 配额、时间超时、并发度这些参数。

如果今天的评测结果和上周不一样，第一件要检查的事情不是模型变没变，而是配置变没变。这听起来简单，但在实践中很多团队的评测基础设施管理是非常粗糙的，换了个集群、改了个参数，然后对着变化的数字百思不得其解。

如果你要向非技术的决策者汇报模型性能，这篇文章提醒你：要给数字加上必要的配置说明。"我们的 agent 在 Terminal-Bench 上得了七十二分"和"在三倍资源上限下，我们的 agent 在 Terminal-Bench 上得了七十二分"，是两个意义完全不同的陈述，后者才是对决策有意义的信息。

如果你在为生产环境选型 AI coding agent，这篇文章还有一个很重要的实践提示：要在你实际的生产环境配置下做评测，而不是在宽松的测试环境下。很多团队踩过这个坑：在高配测试机上跑评测，选了跑分最高的模型，部署到资源相对紧张的生产容器，结果发现生产里的表现比测试差很多。原因不是模型降级了，而是资源配置变了，导致 agent 能用的策略集合变了。

更好的做法是：用跟生产环境一致的资源配置来做选型。如果你的生产容器只有四 G 内存，就在四 G 内存的配置下测试各个候选模型，选在这个约束下表现最好的那个。在这个场景下，一个"倾向于写精简代码"的模型，很可能比"依赖重型工具、在宽松资源下跑分极高"的模型更适合你的实际部署场景。

文章的方法论本身也值得借鉴。他们的实验设计非常教科书式的干净：单一变量，六个数据点，明确的统计显著性检验，P 值都有报出来，结论清晰可以复现。当你在日常工作中遇到"这次的结果和上次不一样"的困惑，用这种系统化的受控实验方法，逐一量化每个潜在变量的影响幅度，往往比凭直觉猜测效率高得多。这套思路不只适用于评测，在任何需要排查性能差异的工程场景里都是有价值的。

最后我想给出一点个人的思考，跟原文作者的立场不一定相同，仅供参考。

我个人认为，这篇文章揭示的问题，是 AI 评测领域一个更大的系统性困境的一个侧面。整个行业花了大量精力让 benchmark 的数字更精确——更多的题目、更严格的统计方法、更好的评分标准——却相对忽视了一个根本性的问题：我们在精确测量一个多大程度上代表真实能力的东西？

基础设施噪声只是这个问题的一个维度，而且算是比较容易量化和控制的那种。还有 prompt 敏感性的问题，同样的任务换一种描述方式，分数可能差三四个百分点。还有数据污染的问题，如果模型在预训练时见过评测集的题目，高分是能力的体现还是记忆的体现？还有任务分布的问题，在 SWE-bench 上表现很好的模型，在你公司实际的代码库上表现如何？

从这个角度看，Anthropic 这篇文章的价值不只在于它解决了一个工程问题，更在于它提供了一个思维框架：把评测环境的每一个参数，都当成一个潜在的混淆变量，系统地去量化它的影响。这个框架，比文章里任何一个具体的数字都有更长久的价值。

好，今天的节目到这里。这篇来自 Anthropic 工程团队的文章，用一个非常干净的实验，量化了一个长期被行业忽视的问题：AI 编程评测中，基础设施配置本身就是一个重要变量，仅这一项就能产生高达六个百分点的分数差距。他们的核心建议很具体——把资源申请量和硬上限分开设置、用实验校准合适的倍数、在资源配置没有标准化之前对小于三个百分点的排行榜差距保持审慎。

对我们每个在这个行业里工作的工程师来说，记住一句话就够了：下次看到排行榜数字，先问一句，这是在什么配置下跑出来的。那个领先，可能是真实的能力差距，也可能只是人家的虚拟机内存更大。
