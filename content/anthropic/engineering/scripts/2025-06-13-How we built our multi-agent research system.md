今天要聊的这篇文章，英文标题是 How we built our multi-agent research system，中文可以翻译成"我们是如何构建多智能体研究系统的"。这篇文章发表于二零二五年六月十三日，作者是 Anthropic 公司工程团队的多位成员，包括 Jeremy Hadfield、Barry Zhang、Kenneth Lien、Florian Scholz、Jeremy Fox 以及 Daniel Ford，这是一篇来自一线工程师的实战报告。

在我们正式开始之前，我要先交代一个重要的时间背景。今天是二零二六年六月十三日，这篇文章正好是一年前发表的。一年在 AI 领域，是一段相当漫长的时间。我们今天回顾它，不仅仅是为了了解 Anthropic 当时是怎么做的，更是为了从这份"第一批吃螃蟹的人"的经验中，提取出那些真正经得起时间考验的工程原则。

当这篇文章发表的时候，AI agent 系统，也就是人工智能自主执行任务的系统，刚刚从实验室走向真正的生产环境。大多数公司要么刚刚开始尝试构建自己的 agent 系统，要么还在观望。Anthropic 的 Claude Research 功能是当时少数真正跑在生产环境里的多智能体系统之一。这篇文章记录的，正是从这个系统的诞生到稳定运行的整个过程中积累的第一手经验教训。放到今天来看，多智能体系统已经不再是新鲜概念，但这篇文章里的很多原则——关于 prompt 工程、关于评测策略、关于生产运维——依然是工程界的宝贵参考。如果你正在构建 AI agent 应用，或者正在考虑是否要引入多智能体架构，这期节目值得你认真听完。

我们先来理解一下这篇文章要解决的根本问题：为什么研究任务需要多智能体系统？

研究，或者说 research，是一种非常特殊的认知任务。它的特殊性在于，你在开始的时候根本不知道正确的路径是什么。这和软件开发里写一个排序算法不一样，排序算法是有确定步骤的，你可以写一个固定的流程来执行它。但研究不是这样的。

假设你要研究"当前全球半导体供应链面临的主要风险"，你会怎么开始？你可能先搜索一些宏观报道，然后发现台积电的某次公开声明很有意思，于是深挖进去，然后你发现美国的出口管制是一个关键变量，然后去查具体的政策文件，然后你发现荷兰 ASML 的光刻机出口也受到了限制，又是一条新的线索……整个过程是高度动态的，每一步的发现都在改变下一步的方向。这就是文章里说的"路径依赖"——研究过程本质上是不可预知的，你只有走到那一步才知道下一步该往哪走。

传统的解决方案，比如 RAG 系统，也就是检索增强生成，是无法应对这种动态性的。RAG 的工作方式是：给一个问题，从知识库里捞出几段最相关的文本片段，然后让大语言模型基于这些片段生成答案。整个过程是一次性的、静态的。这对于知识库里已有答案的问题效果不错，但对于需要动态搜索、多轮迭代的开放式研究问题，它无能为力。

AI agent 的出现改变了这个局面。agent，也就是智能体，指的是能够自主使用工具、在循环中持续执行操作的大语言模型。它可以搜索一遍，看看结果，根据结果调整搜索策略，再搜索一遍，如此循环，直到找到满意的答案。这种动态的、多步骤的工作方式，正是研究任务所需要的。

而多智能体系统，也就是多个 agent 协同工作，进一步把这种能力提升到了一个新层次。文章里用了一个很好的类比：个体人类在过去十万年里变得更聪明了，但人类社会在信息时代的能力提升是指数级的，这是因为我们建立了集体智慧和协作机制。同样，单个 AI agent 有它的极限，但一组 agent 协同工作，可以完成远超单个 agent 能力的任务。

那么，Anthropic 的 Research 系统具体是怎么工作的？

他们采用了一个叫做 orchestrator-worker 的架构模式，中文通常叫"编排者-工作者"模式，或者说"主导-从属"模式。整体架构分两层：第一层是主导 agent，负责分析用户的问题，制定研究策略；第二层是多个子 agent，并行地从不同角度探索研究问题。

具体流程是这样的：用户提交一个研究查询之后，主导 agent 首先分析问题，判断它的复杂度和所需的探索方向，然后创建多个子 agent，每个子 agent 负责探索问题的一个特定方面。这些子 agent 同时工作，各自使用搜索工具收集信息，然后把自己的发现汇报给主导 agent，最后由主导 agent 综合所有信息，生成最终的研究报告。

这里有一个关键的设计理念，文章里用了"压缩"这个词来描述。搜索的本质是压缩——从海量的网络信息中提炼出有价值的洞察。每个子 agent 在自己独立的上下文窗口里工作，探索不同的方向，然后把最重要的信息浓缩成一个精炼的摘要传递给主导 agent。这种设计有两个关键优势：一是并行处理大幅提升速度；二是每个子 agent 有自己独立的探索轨迹，减少路径依赖，能更全面地覆盖问题的各个维度。

说到性能数据，文章给出了一个非常直接的对比：在他们内部的研究评测中，以 Claude Opus 四作为主导 agent、Claude Sonnet 四作为子 agent 的多智能体系统，比单独使用 Claude Opus 四的单智能体系统，性能高出了百分之九十点二。将近翻倍的提升，这个数字是相当惊人的。

文章里举了一个具体的例子。任务是：找出标普五百指数信息技术板块中所有公司的全部董事会成员。这是一个典型的"广度优先"任务：你需要同时追踪几十家公司，每家公司都要独立查询，而且任何一家公司的信息都不影响另一家的查询。多智能体系统把这个任务分解给多个子 agent，每个子 agent 负责查找几家公司的信息，并行执行，很快就汇集了完整的答案。而单智能体系统只能顺序搜索，一家查完再查下一家，速度极慢，而且在有限的时间和 token 预算内往往无法找到完整的答案。

那多智能体系统的能力提升，到底是从哪里来的？文章里给出了一个非常有意思的量化分析。他们对 BrowseComp 评测数据进行了分析，这个评测专门测试浏览 agent 找到难以发现信息的能力。他们发现，影响性能的三个因素——token 用量、工具调用次数、模型选择——合在一起解释了百分之九十五的性能差异。而仅仅是 token 用量这一个因素，就解释了百分之八十的差异。

这个发现其实揭示了一个深刻的规律：多智能体系统的能力，核心来源是它能消耗更多的 token 来解决问题。每个子 agent 都有自己独立的上下文窗口，整个系统在并行地消耗大量 token 进行推理和搜索，这是任何单个 agent 在单个上下文窗口里无法达到的规模。更有意思的是，文章还提到，把模型从 Claude Sonnet 三点七升级到 Claude Sonnet 四，带来的性能提升，比把 Sonnet 三点七的 token 预算翻倍还要大。这说明更新更强的模型不只是在"同样消耗 token"的情况下表现更好，而是能更高效地利用每一个 token，是一个巨大的效率乘数。

当然，多智能体系统的能力提升是有代价的。文章里非常坦诚地给出了数据：agent 使用的 token 量通常是普通对话的四倍，而多智能体系统使用的 token 量则是普通对话的十五倍。这意味着每次多智能体研究任务的成本，可能是一次普通对话的十几倍。这个成本是真实存在的，意味着多智能体系统只有在任务的价值足够高的情况下才是经济合理的。

这也是为什么多智能体系统并不适合所有类型的任务。文章里明确指出，它最适合三类场景：第一，可以被大量并行化的任务；第二，信息量超过单个上下文窗口的任务；第三，需要同时接触多个复杂工具的任务。反过来，如果任务本身是高度顺序依赖的，或者需要所有 agent 共享同一个上下文，那多智能体目前就不是好的选择。比如文章里提到，大多数编程任务其实没有那么多真正可以并行化的子任务，而且 agent 目前在实时协调和委派方面也还不够成熟，所以并不特别适合多智能体架构。

好，理解了基本架构和性能原理之后，我们来看这篇文章最精华的部分：prompt 工程，也就是提示词工程。

文章总结了八条经过实战验证的 prompt 工程原则。我觉得这八条不只适用于多智能体系统，很多也适用于各种类型的 AI agent 开发。我们一条一条来讲。

第一条原则叫"像你的 agent 一样思考"。这句话初听有点禅意，但背后的工程实践非常具体：要优化 prompt，你必须能够准确预测一个 prompt 会让 agent 产生什么样的行为。Anthropic 的工程师的做法是，使用 Claude 的 Console 工具，把系统里实际使用的 prompt 和工具原样复现，然后一步一步地观察 agent 的操作。这种直接观察的方式，让他们发现了很多用日志和指标看不出来的问题：agent 在已经有了足够信息的情况下还在继续搜索；agent 用了过于冗长复杂的搜索词，导致搜索结果很差；agent 在有更合适工具的情况下选择了错误的工具。这种"亲身感受 agent 决策过程"的方式，是构建准确 agent 心智模型的最有效方法。只有建立了准确的心智模型，你才能直觉性地知道哪一句 prompt 修改会产生什么样的效果。

第二条原则：教会主导 agent 如何委派任务。这是多智能体系统特有的挑战。在一个多 agent 系统里，主导 agent 需要把复杂问题分解成子任务，然后把每个子任务分配给一个子 agent。如果子任务描述得不够清晰、不够具体，整个系统就会出现各种协调失败：子 agent 们重复做同样的事情，或者遗漏了重要的方向，或者完全误解了自己的任务。

文章里给出了一个很真实的失败案例。他们最开始允许主导 agent 给子 agent 下达简单的指令，比如说"研究半导体短缺"。结果子 agent 们因为得到的指令太模糊，各自按照自己的理解去做——一个子 agent 去研究二零二一年的汽车芯片危机，另外两个子 agent 都在研究当前二零二五年的供应链，完全没有实现分工，而是在重复劳动。后来他们要求主导 agent 为每个子 agent 提供：明确的研究目标、具体的输出格式要求、应该使用哪些工具和数据来源，以及清晰的任务边界，明确告诉每个子 agent 不应该覆盖哪些方面。有了这些细化之后，子 agent 之间的分工明确多了，系统的整体效率大幅提升。

第三条原则：根据查询复杂度来调整努力程度。这个原则解决的是一个很具体的问题：agent 不擅长自己判断一个任务需要多大的投入。如果不给明确的指导，agent 要么在简单问题上过度投入，召唤了很多子 agent 来处理一个用单次搜索就能回答的问题；要么在复杂问题上投入不足，没有给充足的资源来处理需要深度研究的问题。

Anthropic 的解决方案是，直接在 prompt 里写明资源分配的规则。大概是这样的层次：简单的事实查询只需要一个 agent，做三到十次工具调用；简单的对比任务可能需要两到四个子 agent，每个做十到十五次调用；复杂的深度研究则可能需要超过十个子 agent，每个都有清晰划分的责任范围。这些明确的规则，让主导 agent 能够高效地分配资源，避免了资源的浪费和不足。在他们的早期版本里，最常见的失败模式之一就是用户提了一个非常简单的问题，结果 agent 召唤了大批子 agent 出来，不仅速度慢，而且浪费了大量成本。

第四条原则：工具设计和工具选择至关重要。这一条很多人可能会低估，但文章里说得非常清楚：agent 和工具之间的接口，就像人和计算机的界面一样重要。一个工具描述写得不清楚，可能让 agent 走完全错误的路。想象一下，如果你有一个能查询 Slack 消息的工具，但工具描述写得模糊，agent 可能根本不知道什么情况下该用这个工具，结果在应该查 Slack 的时候去搜了网络，当然什么也找不到。

随着 MCP 服务器的普及，MCP 是让模型接入外部工具的一种协议，这个问题变得更加突出，因为 agent 现在可能同时面对几十个工具，这些工具的描述质量参差不齐。Anthropic 给 agent 嵌入了明确的工具选择启发规则：先检查所有可用的工具；把工具选择和用户的实际意图匹配；用网页搜索做广泛的外部探索；优先使用专业工具而不是通用工具。

更有意思的是他们做的一个工具测试 agent。这个 agent 的任务是：给它一个有缺陷的 MCP 工具，它来尝试使用这个工具，然后根据使用中遇到的问题，重写工具的描述，让未来的 agent 能避开那些坑。通过对同一个工具进行几十次测试，这个 agent 发现了工具使用中的关键细微差别和 bug，最终改写后的工具描述让后续使用该工具的 agent 完成任务的时间缩短了百分之四十。让 agent 来改进工具描述，这是一个非常有创意也很实用的工程方法，我个人认为这是整篇文章里最有启发性的一个设计。

第五条原则：先广后窄。这条搜索策略说起来很简单，但确实解决了一个 agent 的常见毛病。agent 在搜索时往往有一种倾向：使用过于冗长、过于具体的搜索词，以为越精确越好，结果搜出来的东西很少，甚至什么都搜不到。就像你在搜索引擎里输入一个几十个字的长查询词，往往比不上一个简洁的几个字的关键词组合效果好。

人类优秀的研究者是怎么做的？先用宽泛的关键词了解这个领域的大体情况，看看都有哪些信息来源，然后根据发现的信息逐步缩小焦点，深挖真正有价值的方向。Anthropic 在 prompt 里明确引导 agent 模仿这种策略：先用简短的宽泛查询词，评估可用的信息之后，再逐步缩小焦点。

第六条原则：引导思考过程。这一条涉及到 Claude 的一个特殊功能叫做 extended thinking，也就是扩展思考模式。在这个模式下，Claude 会在给出最终回答之前，先输出一段可见的推理过程，就像是给了模型一个可以控制的草稿本。在研究系统里，主导 agent 使用扩展思考来规划整体研究方案：评估哪些工具适合这个任务，判断查询的复杂度需要几个子 agent，以及定义每个子 agent 的具体职责。子 agent 也会在每次工具调用之后，用穿插的思考过程来评估刚获取的信息质量，识别还有哪些信息缺口，然后决定下一次查询的方向。他们的测试表明，扩展思考显著改善了 agent 的指令遵循能力、推理质量和整体效率。

第七条原则：并行工具调用可以改变速度和性能。这是八条原则里效果最立竿见影的一条。Anthropic 的早期 agent 是顺序执行搜索的：先调用工具 A，等结果回来，再调用工具 B，再等结果，然后才是工具 C。这种方式速度非常慢。

他们引入了两种并行化机制：第一，主导 agent 同时启动三到五个子 agent 而不是一个一个地启动；第二，子 agent 自身也并行调用三个以上的工具，而不是顺序调用。这两个改动的叠加效果惊人：对于复杂研究任务，整体时间缩短了最多百分之九十。原本需要几个小时的研究，现在几分钟就能完成。而且不只是快了，覆盖的信息量也大幅增加了，因为同样的时间内可以探索更多的方向。

第八条原则：让 agent 自己改进自己。这是最有前瞻性的一条原则。他们发现 Claude 四系列模型在 prompt 工程方面有相当强的能力：给它一个 prompt 和一个具体的失败案例，它能诊断出问题的根源，并提出有针对性的改进建议。前面说到的工具测试 agent，就是这个原则的直接应用。这意味着多智能体系统未来可能形成一种自我改进的闭环：系统在使用过程中发现问题，然后自动优化自己的工具描述和 prompt，提升后续的性能，而不是完全依赖人类工程师来优化每一个细节。

以上八条原则的背后，有一个更高层次的设计理念：好的 prompt 不应该是一堆严格的规则，而应该是一套协作框架，定义分工方式、问题解决方法和资源预算。好的 prompt 是在教 agent 如何思考，而不只是告诉它做什么。

讲完了 prompt 工程，我们来看评测部分。

评测多智能体系统，面临一个根本性的挑战：你无法规定"正确的路径"。在传统软件测试里，你可以写：给定输入 X，系统应该执行步骤 A、B、C，最终输出 Z。但 agent 系统不是这样工作的。同一个问题，一个 agent 可能搜索了三个来源就得到了答案，另一个 agent 可能搜索了十个来源才得到同样的答案，两者都是正确的，只是路径不同。你不能因为 agent 没有按照你预期的步骤走就判它失败，因为根本不存在唯一正确的步骤。

那怎么评测？文章里给出了几个非常实用的建议。

第一个建议：立刻开始评测，不要等到有了完整的测试集。这个建议听起来很简单，但在实践中很多团队会犯一个错误：觉得"只有几十个样本不够有代表性"，于是推迟评测，等着积累几百个样本再开始。Anthropic 的经验是，在 agent 开发的早期阶段，系统里有大量低悬的果实，每一次 prompt 调整的效果都非常显著。一次好的 prompt 修改可能把成功率从百分之三十提升到百分之八十。这种量级的变化，用二十个样本就能清楚地看到，根本不需要几百个样本。延迟开始评测，只会延迟你的迭代速度，而迭代速度在 AI 开发里是最关键的变量之一。

第二个建议：用大语言模型作为评测裁判，也就是 LLM-as-judge 方法。研究类任务的输出是自由格式的文本，通常没有唯一正确答案，这让传统的程序化评测方法完全失效。Anthropic 的评测裁判从几个维度来评分：事实准确性，也就是论断是否与来源吻合；引用准确性，也就是引用的来源是否真的支持相应的论断；完整性，也就是请求的各个方面是否都覆盖到了；来源质量，也就是是否使用了一手权威来源而不是质量较低的二手来源；以及工具效率，也就是是否用了合适的工具、次数是否合理。

他们尝试过用多个裁判分别评测不同的维度，但发现用单个模型、单个 prompt 一次性输出零到一的浮点分数和通过失败评级，和人类评判的一致性反而是最好的。这个发现和直觉可能有些不同，但逻辑上是合理的：多个裁判引入了更多的不一致性，而一个训练良好的模型用一个精心设计的 prompt，可以做到相当稳定的评判。

第三个建议：人工评测永远不可替代。无论自动化评测做得多好，人工测试都能发现自动化系统遗漏的问题。文章里举了一个很有趣的例子：他们的人工测试者发现，早期的 agent 系统性地偏好 SEO 优化的内容农场，而忽略了学术论文 PDF 或者个人博客这类内容质量更高但搜索排名较低的来源。这种偏见在自动化评测的指标里很难体现出来，但人工检查一眼就能看出来。他们后来在 prompt 里加入了来源质量的启发规则，专门引导 agent 识别和优先选择权威来源。

好，最后我们来看生产工程部分。这部分对于真正在做 AI 应用的工程师来说，可能是最有实际价值、但也是最少被公开讨论的内容。

文章里有一句话我很喜欢："最后一公里往往成了整段旅程的主要部分。"意思是，从一个能在开发机上跑起来的 prototype，到一个真正可靠的生产系统，中间的距离远比大多数人预想的要长得多。

在传统软件里，一个 bug 可能导致某个功能失效，影响范围通常是有限的、可预测的。但在 agent 系统里，错误的传播方式完全不同。每一步的错误都可能影响下一步的决策，然后又影响到再下一步，这种复合效应让错误的影响范围变得很难预测。一个工具调用失败，可能导致 agent 走向完全不同的探索路径，最终产生完全错误的研究结论。

文章里总结了几个关键的生产工程挑战。

第一个挑战是状态性和错误的累积。agent 系统的一个根本特性是，它是有状态的。agent 可能持续运行几十分钟甚至更长，在这段时间里维持着复杂的内部状态。如果中途出现系统故障——服务器崩溃、网络超时、工具接口报错——你不能简单地让 agent 从头重来。重新开始意味着浪费大量已经花费的计算资源，对用户来说体验也极差。

Anthropic 的解决方案是构建能够从故障点恢复的系统，使用定期检查点来保存 agent 的状态，这样出现故障时可以从最近的检查点恢复而不是从零开始。他们还有一个有意思的做法：当一个工具发生故障时，不是让整个 agent 崩溃，而是直接告诉 agent"这个工具现在不可用"，让它自己决定怎么应对。他们发现这种做法效果出奇地好——agent 往往能找到替代方案，或者调整策略，继续完成任务。这就是把 AI 的自适应能力和确定性的工程保障结合起来的思路。

第二个挑战是调试的困难。agent 系统是非确定性的，就算给它完全一样的输入，每次运行的轨迹也可能不同。这让传统的调试方式失效了。用户会报告说"agent 找不到很明显的信息"，但工程师不知道为什么：是搜索词用得不好？是选择了质量差的来源？还是某个工具调用失败了？在没有完整可观测性的情况下，这类问题几乎无从排查。

Anthropic 为此建立了完整的生产追踪系统，对每个 agent 的每次工具调用、每次决策都进行记录，这样可以在出现问题时回溯完整的执行轨迹，诊断出根本原因。他们同时监控 agent 的高层决策模式和交互结构，注意，是高层的统计模式，而不是对话的具体内容，这样可以在保护用户隐私的同时发现系统性的问题。比如他们可能注意到某类查询的平均工具调用次数突然上升了，这通常意味着 agent 在某类问题上陷入了低效的搜索循环，需要进行 prompt 调整。

第三个挑战是部署协调。agent 系统和普通的 Web 服务不一样。普通 Web 服务是无状态的，你更新代码，重新部署，流量切过来，一切正常。但 agent 系统是高度状态化的，某个时刻可能有几百个 agent 正在执行任务，每个 agent 都处于自己独特的中间状态。如果你直接切换到新版本，这些正在运行的 agent 的状态就和新版本的逻辑不匹配了，轻则行为异常，重则完全崩溃。

Anthropic 的解决方案叫做"彩虹部署"：同时运行多个版本，旧版本继续处理已有的 agent 请求直到它们完成，新版本接管新进来的请求，逐步把流量从旧版本转移到新版本。这个过程需要维护多个版本的代码同时运行，增加了工程复杂度，但保证了平滑过渡，不会打断正在执行任务的 agent。

第四个挑战是同步执行的瓶颈。目前他们的系统里，主导 agent 是同步等待子 agent 的——它发出一批子 agent，然后等着，等所有子 agent 都完成了，才能处理结果、决定下一步。这种方式简化了协调逻辑，但也带来了明显的效率损失：如果有十个子 agent，其中九个很快就完成了，但有一个特别慢，整个系统就被这一个拖慢了。而且主导 agent 在等待过程中无法介入，无法根据子 agent 的中间进展调整方向，子 agent 之间也无法相互协调。文章里提到，未来转向异步执行是一个重要方向，但会带来结果协调、状态一致性、错误传播等一系列新的工程挑战，目前还没有特别成熟的解决方案。

好，说了这么多理论和 Anthropic 的实践，我们来到最实用的环节：对于正在做 AI 应用开发的工程师来说，这篇文章能拿走什么立刻上手的东西？

首先是架构选型。如果你正在考虑用多智能体架构，先问自己几个问题：这个任务可以被自然地分解成多个并行的子任务吗？任务本身的价值足够高，能覆盖比单 agent 高十倍左右的成本吗？任务需要处理的信息量超过了单个上下文窗口的容量吗？如果三个问题的答案都是肯定的，那多智能体架构可能是合适的。如果任务是高度顺序依赖的，或者成本敏感，先考虑单 agent 方案，不要为了用多智能体而用多智能体。

其次是 prompt 工程的优先级。如果你的 agent 行为不符合预期，在改架构之前，先把工作重心放在 prompt 上。几个立竿见影的建议：给子任务提供非常具体的描述，不要让子 agent 自己理解模糊的指令；在 prompt 里明确写入资源分配规则，什么复杂度的任务用多少 agent、多少次工具调用；给 agent 清晰的工具选择启发规则；对于搜索型任务，明确引导先广后窄的策略。这些改动不需要修改任何代码，只需要修改 prompt，但效果往往非常显著。

关于工具描述，这是经常被忽视但影响巨大的一点。你的工具描述应该清晰地回答：这个工具能做什么，不能做什么，什么情况下应该优先使用它，输入输出的格式是什么，有哪些常见的陷阱需要避免。如果有条件，可以让模型自己尝试使用工具，然后根据使用情况改进工具描述，效果可能远超你手工打磨的版本。

关于评测，哪怕你现在只有五个测试案例，今天就可以开始。选最能代表真实使用场景的案例，每次改 prompt 或者改工具描述之后，跑一遍这几个案例，看看有没有改善。这种快速迭代的飞轮，比等到建立了完整的评测体系之后才开始迭代，要有效得多。

关于生产化，如果你的 agent 系统要上线到真正的生产环境，一定要在早期就认真对待几件事：用检查点机制保护 agent 的中间状态，不要让单点故障导致所有工作丢失；建立完整的执行追踪，记录每次工具调用的输入输出，出了问题才能排查；设计优雅的工具失败处理，告诉 agent 工具不可用，让它自己想办法，而不是直接让整个任务崩溃；以及设计能够支持版本平滑切换的部署机制，不能因为更新代码就破坏正在运行的 agent。

还有几个来自文章附录的补充建议我觉得也值得一提。如果你的 agent 系统涉及跨很多轮会话的状态修改，评测时把重点放在最终状态是否正确，而不是中间每一步是否按照某个固定路径执行。对于可能运行几百轮的长对话 agent，要有上下文压缩和记忆机制，让 agent 在接近上下文窗口限制的时候，能把已完成的工作摘要存储到外部记忆，然后生成一个新的干净上下文继续工作，而不是在上下文溢出时丢失所有历史信息。最后，考虑让子 agent 直接把输出写入文件系统，而不是通过主导 agent 层层传递——这对于代码、报告这类结构化输出特别有用，可以避免信息在多层传递中的损耗，也能减少主导 agent 处理大量结构化输出带来的 token 开销。

好，来到最后总结的时间了。

这篇文章是一份真实的工程战地报告。Anthropic 的工程师们把 Research 功能从概念变成了一个真正跑在生产环境里的多智能体系统，这期间踩了很多坑，也总结出了很多经验。文章的核心信息可以浓缩成这样几点：多智能体系统的能力提升是真实的，但代价也是真实的，根本驱动力是 token 的大规模并行消耗；prompt 工程依然是最重要的性能调节杠杆；评测要尽早开始，不要等完美；从 prototype 到 production 的距离，远比预想的要长，状态管理、可观测性和平滑部署是不可跳过的工程基础。

我个人的看法是，这篇文章发表一年之后来看，它的价值比发表时还要大。因为现在构建多智能体系统的团队更多了，碰到这些问题的工程师更多了，但有详细第一手经验分享的文章依然很少。如果你的团队正在认真做 AI agent 产品，我建议把这篇文章找出来，让每一个工程师都认真读一遍。
