## Script

今天我们来聊一篇 Anthropic 工程博客上的文章，英文标题是 Harness Design for Long-Running Application Development，我把它翻译成"长时运行应用开发中的 Harness 设计"。这篇文章发表于二零二六年三月二十四日，距今大约三个月。

作者是 Anthropic Labs 团队成员 Prithvi Rajasekaran。他花了好几个月时间同时打磨两件事：一件是让 Claude 生成高质量的前端设计，另一件是让它不靠任何人工干预，独立完成完整软件应用的开发。今天这篇文章，就是他在这两个方向上探索的总结。

我特别喜欢这类文章，因为它不是在讲"AI 会改变一切"的宏大叙事，也不是炫产品功能的营销材料，而是一个工程师老老实实地告诉你：我碰到了什么问题，我怎么解决的，结果如何，哪些方法没用，哪些出乎意料地好用。对咱们这些每天要设计和维护 AI 系统的工程师来说，这种第一手工程经验非常难得。

好，先来说背景。

作者之前已经和同事们做过两件事：一个是前端设计技能（frontend design skill），另一个是长时运行代码 Agent 的 Harness——也就是让 AI Agent 持续跑几个小时、自动写代码的那套系统框架。通过提示词工程和 Harness 设计，他们把 Claude 的表现提升到了高于基准线的水平，但最终两个方向都撞上了天花板，再难突破。于是他开始寻找新的方法。

文章要解决的核心问题可以归结为两个：第一，怎么让 AI Agent 在长时间运行的过程中不跑偏；第二，怎么让 AI 对自己的输出做出真正有用的评判。我们就从这两个问题出发，一层一层往里拆。

先聊第一个问题——AI Agent 长时间工作为什么容易出问题。

你可能有过这样的经历：给 Claude 或者其他大模型一个复杂的任务，让它持续工作。一开始还好，但随着任务推进，它越来越容易出错，逻辑越来越混乱，或者在快完成的时候突然开始草草收场，把东西随便包个头就交出去。这背后有一个根本性的技术原因：上下文窗口的限制。

所谓上下文窗口（context window），就是模型在处理一次任务时能"记住"的信息上限。当任务需要跑好几个小时，模型处理的代码、文件、日志、反馈不断累积，上下文窗口慢慢填满，模型就开始失去前后连贯性。更糟糕的是，作者观察到一种现象，他把它叫做"context anxiety"，也就是上下文焦虑——有些模型在感觉到自己快到上下文上限的时候，不是继续认真工作，而是开始仓促地收尾，就像一个学生考试快到时间了，把剩下的题目胡乱填一遍一样。

针对这个问题，作者区分了两种处理上下文增长的策略：压缩（compaction）和重置（reset）。这两个方案看起来差不多，但在工程上有本质区别。

压缩的思路是把对话历史里较早的部分总结一下，让同一个 Agent 在压缩后的历史基础上继续工作，保持会话连续性。这就像你在一个很长的会议记录里写摘要，把前几十页浓缩成几段话，然后接着往后记。

重置的思路则更激进：彻底清空上下文，启动一个全新的 Agent，然后通过一份精心设计的"交接文件"来传递前一个 Agent 的工作状态和下一步要做什么。这就像项目交接，前任把所有关键信息整理清楚，新接手的人从零开始读，但手里有完整的地图。

两者的核心区别在于：压缩虽然保持了连续性，但不能解决上下文焦虑——模型还是知道自己已经跑了很长时间，焦虑感依然存在。重置能提供真正的干净状态，但代价是必须把交接文件设计得足够完整，确保新 Agent 能够有效接手，否则信息丢失会造成新的问题。

作者在 Sonnet 四点五（Sonnet 4.5）上做测试时发现，这个模型的上下文焦虑非常严重，只靠压缩根本不够，必须用重置才能让长任务保持质量。而后来他们构建新的三智能体系统时用的是 Opus 四点五（Opus 4.5），这个模型基本消除了上下文焦虑的问题，可以依赖 Claude Agent SDK 的自动压缩机制，不再需要手动重置了。

这个观察本身就很有价值：不同模型在长任务上的表现差异很大，Harness 的设计要跟具体模型匹配，不能一套方案打天下。

现在说第二个问题——AI 自我评估的困境。

如果你让 AI 对自己产出的代码或者设计做评价，它通常会给出非常积极的反馈，就算实际质量很一般。这不是 AI 在撒谎，而是模型的一种内在倾向——它是在对话中训练出来的，给出正面、有帮助的回应是它的默认模式。就算你在提示词里明确说"请严格评估"，它往往也会先列一堆优点，然后温和地提一两个小建议，最终结论还是"整体很好"。

对于有客观测试标准的任务，比如代码能不能运行通过，这个问题还不算最严重，因为总有二元判断可以绕过自我评估。但对于主观性很强的任务，比如"这个界面设计好不好看"，就根本没有这种二元判断，完全靠"感觉"，而 AI 的感觉偏向正面。

作者的解法非常清晰：把做事的 Agent 和评判的 Agent 分离。你可能会问，评估器不也是大模型吗，它不也会偏向正面吗？作者也承认这一点，但他发现了一个关键的不对称性：让一个独立的评估器变得严苛，比让一个生成器自我批评，要容易得多。原因在于评估器没有"自尊"，它不需要为自己的输出辩护，你只需要通过提示词、评分标准和少量示例来校准它的判断尺度，它就能表现得相当严格。一旦有了这个外部反馈，生成器就有了具体的改进目标，迭代循环就转起来了。

这个思路让作者联想到了 GAN，也就是生成对抗网络（Generative Adversarial Networks）。GAN 是一种经典的图像生成技术：一个生成器负责生成图像，一个判别器负责判断图像真假，两者相互博弈，生成器在对抗中不断提升生成质量。作者把这个结构借过来，应用到了 AI Agent 的生成与评估上——生成器负责生成内容，评估器负责打分提意见，生成器根据反馈迭代改进。

他首先在前端设计上测试这套思路，因为设计的自我评估问题在这里最明显。不做任何干预的话，Claude 生成的前端界面倾向于"安全但无聊"——技术上没问题，但视觉上平淡，一眼就看出来是 AI 批量产出的东西，没有任何设计师会有的那种个人风格和审美冒险精神。

为了让评估器能有效地给设计打分，作者做了一件关键的事：他把主观的"好不好看"转化成了四条具体的评分标准，同时放进生成器和评估器的提示词里。

第一条是设计质量（Design Quality）——设计是否有整体感，而不是零件的堆砌。好的设计里，颜色、字体、布局、图像这些元素组合在一起，能创造出一种独特的气质和个性感。

第二条是原创性（Originality）——是否有定制化的设计决策，还是说全是现成模板、库的默认样式、或者一眼就能认出来的 AI 生成套路。比如那种"紫色渐变配白色卡片"的组合，是 AI 设计的典型标志，直接扣分。

第三条是工艺（Craft）——技术执行层面的质量：字体层级是否清晰、间距是否一致、颜色是否和谐、对比度是否合适。这更像是个"能力达标检查"，大多数合理的实现默认就能过关，但如果连基本功都做错了就会失分。

第四条是功能性（Functionality）——独立于美观性的可用性检查：用户能不能理解界面是做什么的，能不能找到主要操作，能不能在不靠猜测的情况下完成任务。

作者特别强调：他给设计质量和原创性比工艺和功能性更高的权重，因为 Claude 在工艺和功能性上本来就做得不错，但在设计质量和原创性上容易产出他称之为"AI 糊弄料"（AI slop）的东西——就是那种看起来像样但毫无特色的结果。通过在评分标准里明确惩罚这类模式，就把模型推向了更有美学冒险精神的方向。

评估器还配备了 Playwright MCP，这是一个可以直接操作浏览器的工具，让评估器不是对着静态截图打分，而是真正去导航页面、截图研究，然后才给出评分和详细的改进意见。整个循环每次运行迭代五到十五轮，有时候长达四个小时。

作者还加了一个策略性决策机制：生成器在每次评估之后，要判断是在当前方向上精细打磨，还是换一个完全不同的美学风格重新来过。这个机制带来了一些非常有意思的结果。

他举了一个例子：他让模型生成一个荷兰艺术博物馆的网站。到第九次迭代，模型生成了一个干净的深色主题落地页，视觉精致但在意料之中。然后到了第十次迭代，模型完全放弃了之前的方案，把整个网站重新设计成了一个空间体验：一个用 CSS 透视渲染的三维房间，棋盘格地板，艺术作品挂在墙上，用"门"来导航不同的画廊空间，而不是普通的滚动或点击。这种创意飞跃，是单次生成完全不可能出现的。

还有一个值得一提的细节：评分标准里的措辞本身也会产生意想不到的影响。作者加了一句"最好的设计是博物馆级别的"，发现这个表达会把生成结果推向某种特定的视觉风格。措辞不只是描述期望，它本身就在引导模型的审美走向。

受到前端设计实验的鼓励，作者把这套思路扩展到了完整的全栈应用开发上。生成器加评估器的循环，自然地对应了软件开发里的代码审查和 QA（质量保证）流程——结构是一样的，只是评判的维度从美学变成了功能正确性。

他构建了一个三智能体系统，分别是规划者（Planner）、生成器（Generator）和评估器（Evaluator）。

规划者的任务是把一到四句话的提示词，扩展成完整的产品规格说明书。作者特别叮嘱规划者：要有雄心壮志，但要聚焦在产品背景和高层技术设计上，不要规定具体的技术实现细节。这个设计决策背后有清晰的逻辑：如果规划者一开始就搞错了某个技术细节，错误会一路传导到下游实现，越滚越大。与其把路线图规定得太死，不如定清楚要交付什么，让执行的 Agent 自己去摸索路径。作者还要求规划者主动在产品规格里加入 AI 功能，让最终应用带有 Claude 集成。

生成器负责实际写代码。它按冲刺（sprint）的方式工作，每次从规格里挑一个功能来实现，技术栈是 React 加 Vite 加 FastAPI 加 SQLite（后来升级到了 PostgreSQL），还使用 Git 做版本控制。每个冲刺结束后，生成器先自我评估，然后再交给评估器把关。

评估器是整套系统的质量守门人。它同样使用 Playwright MCP，像真实用户一样点击运行中的应用，测试 UI 功能、API 接口和数据库状态，然后按照产品深度、功能性、视觉设计和代码质量这几个维度打分。每个维度都有硬性门槛，任何一条没过就判冲刺失败，生成器拿到详细反馈重新来过。

还有一个值得专门讲的设计细节：冲刺合同（sprint contract）。在每个冲刺正式开始之前，生成器和评估器要先协商一份合同，提前约定好这个冲刺"完成"的定义是什么。生成器提出它打算实现什么、用什么标准来验证完成，评估器审核这个提案，两边来回沟通直到达成共识。之后生成器才开始写代码，评估器按合同来打分。

你可能会问：为什么要这么麻烦？直接写就好了，写完再评估不行吗？原因很微妙：产品规格是故意写得比较高层次的，这样规划者才不会在早期引入实现错误。但评估器要测试功能，就需要知道具体的实现细节才能验证。冲刺合同就是这个桥梁，它在代码开始写之前就把"什么算成功"确定下来，防止生成器和评估器在完成标准上各说各话。

所有通信通过文件来进行：一个 Agent 写文件，另一个读文件然后响应，再写回去或写新文件，循环往复。这个设计让通信是异步的、持久化的、可检查的，整个系统的流程也更容易追踪和调试。

作者用这套系统做了一个对比实验，任务是"创建一个有关卡编辑器、精灵编辑器、实体行为和可玩测试模式的二维复古游戏制作工具"。

单 Agent 跑了二十分钟，花了九美元；完整 Harness 跑了六个小时，花了两百美元，贵了超过二十倍。但质量差距是立竿见影的。

单 Agent 的版本，初看上去还行，但一上手问题就出来了：布局浪费空间，固定高度的面板让大片区域空置；操作流程死板，如果你不先创建精灵和实体就想往关卡里填东西，界面不会给你任何引导；最关键的是，游戏本身根本跑不起来——实体能显示出来，但对输入没有任何响应，原来实体定义和游戏运行时之间的连线根本就没接上。作者去翻了代码，发现深层的连接逻辑彻底断了，但界面上没有任何提示。

完整 Harness 的版本就不一样了。规划者把那一句话的提示词扩展成了一个十六个功能、分布在十个冲刺里的详细规格，远超单 Agent 的尝试范围。除了基础的编辑器和玩法模式，规格里还包括精灵动画系统、行为模板、音效和音乐、AI 辅助的精灵生成器和关卡设计师，以及带可分享链接的游戏导出功能。更重要的是，游戏能玩了——实体会响应输入，物理系统基本运转（虽然跳上平台的时候会有一点穿模，但核心机制是工作的）。因为规划者读取了前端设计技能，应用还有一套一致的视觉设计语言。

从评估器的日志里可以清楚地看到它的工作方式。文章里列出了几个评估器抓到的具体问题，质量非常高，直接能用于修复。比如：矩形填充工具应该支持点击拖动来填充区域，但实际上只在拖动的起点和终点放置了图块，相关的函数存在但鼠标抬起时没有被正确触发；再比如：删除实体功能要求同时设置 selection 和 selectedEntityId 两个变量，但点击实体时只设置了 selectedEntityId，导致删除功能实际上从来没法触发，评估器甚至给出了具体的代码修复建议；还有一个 FastAPI 路由问题：用于重新排序动画帧的 PUT 路由被定义在了带有帧 ID 参数的路由之后，FastAPI 把字符串"reorder"当成了一个整数参数来解析，直接返回了四百二十二错误。这种级别的具体诊断，是靠模型真正点进去操作应用才能发现的，静态的代码审查根本看不出来。光是这些问题，就已经让整个 Harness 的费用物有所值了。

当然，调出这个质量的评估器也不是一蹴而就的。作者说了一句很直白的话：Claude 开箱即用是一个很差劲的 QA Agent。在早期的实验里，他看着评估器发现了真实的问题，然后自我说服觉得"没什么大不了"，最终批准通过。它测试得比较表面，不会主动去探测边界情况，更隐蔽的 bug 经常漏掉。调优的过程是：反复读评估器的日志，找到它的判断和自己的判断出现分歧的具体例子，然后用这些例子去更新提示词，再测试，再看日志。这个过程循环了好几轮，才让评估器达到他满意的水准。

第一版 Harness 的结果令人满意，但也很笨重、很慢、很贵。下一步自然是想办法在不降低性能的前提下把它简化。

这背后有一个作者明确说出来的原则，他引用了 Anthropic 之前写的文章"Building Effective Agents"里的一句话：找到最简单的解决方案，只在必要时增加复杂度。Harness 里的每个组件，都是对"模型自己做不到某件事"的一个假设，这些假设会随着模型能力进步而过时，值得定期重新检验。如果你不检验，就会一直维护着一堆其实已经不需要的脚手架，既增加成本，又增加系统复杂度。

恰好在作者迭代 Harness 的过程中，Opus 四点六（Opus 4.6）发布了。新模型在计划能力、长任务持续性、大代码库操作能力和代码审查调试能力上都有实质提升，长上下文检索也有明显改善。这些正好是之前 Harness 在弥补的能力短板。

作者的做法是逐个移除组件、观察影响，而不是一次大刀阔斧地重构。他首先移除了冲刺（sprint）结构。冲刺的作用是把工作分解成更小的块，帮助模型连贯处理。既然 Opus 四点六已经能原生处理这种分解，不再需要外部脚手架，那就去掉。测试结果证实了这个判断：在没有冲刺分解的情况下，生成器能连续运行超过两个小时，仍然保持连贯性。这在 Opus 四点五时代是做不到的。

他保留了规划者和评估器，因为两者显然还有价值。没有规划者，生成器会直接开始写代码而不先做规格，最终产出功能丰富程度不如有规划者的版本。评估器则变成了移除冲刺结构后在整个构建完成时做一次全面评估，而不是每个冲刺后评估一次。

作者对评估器的价值做了一个很有意思的分析：随着模型能力提升，"不需要外部检查就能可靠完成"的任务范围会扩大，在这个范围之内，评估器是多余的开销；只有在任务超出模型当前能力边界的地方，评估器才是不可或缺的。这个边界随着模型进步而向外移动，但总还是存在的。所以评估器不是一个非此即彼的选项，而是一个需要根据实际任务复杂度判断成本收益的组件。

改进后的 Harness 用来测试一个更有挑战性的任务：构建一个在浏览器里运行的数字音频工作站（DAW），也就是一个用来作曲、录音和混音的音乐制作程序。这次运行花了大约四个小时，成本约一百二十五美元。

从运行日志来看，构建阶段一共跑了三轮，第一轮构建就连续运行了两小时七分钟。评估器在第一轮结束时给出了这样的反馈：应用外观出色，AI Agent 集成扎实，后端不错；但主要问题是核心的 DAW 功能大量都只是"展示品"——时间轴上的片段不能拖动，没有合成器旋钮或鼓垫的界面，没有可视化的效果器编辑。这些不是边缘功能，而是一个 DAW 可用的核心交互，规格里明确要求了，但没实现。

到了第二轮反馈，评估器又抓到了几个漏掉的地方：音频录制只有按钮没有真正的麦克风捕获，片段的边缘拖动和分割功能没实现，效果可视化只有数字滑块没有图形。这些问题让第三轮的生成器又跑了近十一分钟来修复。

最终的应用虽然距离专业音乐制作软件还差得很远，但核心功能是齐备的：可以工作的编排视图、混音器、播放控制，都在浏览器里跑起来了。作者还通过工具调用让 Agent 帮他创作了一段短小的歌曲片段——设定速度和调性、铺旋律、建鼓轨、调混音比例、加混响，从头到尾完全通过工具驱动完成。

好，核心内容讲完了，我们来聊聊对实际工作的意义。

第一点：评估器的设计需要认真投入。

不要以为给评估器一个"你是一个严格的 QA 工程师"的提示词就够了。你需要给它提供具体的、可操作的评分标准，用真实的失败案例来校准它的判断尺度，然后认真读它的输出日志，反复迭代提示词。评估器的质量不是靠一个好模板就能搞定的，它需要多轮调优，而且越是主观性强的任务，调优成本越高。值得注意的是，作者不是一次就把评估器做好了，而是通过多轮"读日志、找分歧、更新提示词"的循环才让它达标。这个循环本身就是一个需要认真对待的工程工作。

第二点：把主观评估标准具体化这件事，用处很广。

你在很多场景都会遇到类似的问题：如何让 AI 对主观性的东西做出稳定、有用的评判。作者的做法可以直接套用：把"好不好"这个问题拆解成一组具体的可检查维度，每个维度有明确的正向和反向例子。这不需要追求完美——就算标准是不完整的，它也比一句"请严格评估"有用得多，因为它给了模型具体的抓手。另外，评分标准的措辞本身会影响输出风格，这点要在实践中反复体会。

第三点：冲刺合同（sprint contract）这个模式值得借鉴。

在任何多智能体工作流里，如果你有一个生成 Agent 和一个评估 Agent，可以考虑在它们之间加一个"确认接口"，让两者在工作开始前就对输出预期达成共识。这能有效减少"生成器觉得完成了但评估器觉得差很远"的情况，避免大量无效的迭代。这个模式在其他场景也适用——任何时候你有两个 Agent 需要对接，都可以考虑在它们之间加一个预先对齐的步骤。

第四点：关于上下文焦虑，要主动测试你用的模型。

文章里区分了压缩和重置两种策略，关键在于：如果你的模型有明显的上下文焦虑，光靠压缩是不够的，必须用重置才能保证长任务质量。但重置的代价是交接文件的设计复杂度——你需要认真思考，什么信息必须传递给下一个 Agent，什么信息可以丢弃，格式是结构化的 JSON 还是自然语言描述，新 Agent 能不能仅凭这份文件就顺利接手工作。这是一个容易低估工作量的地方。在使用新模型之前，建议先跑几个长任务测试一下它的上下文焦虑程度，再决定要不要加这层复杂度。如果新模型已经解决了上下文焦虑，那就可以直接依赖 SDK 的自动压缩，省掉整个重置机制的工程复杂度。

第五点，也是最重要的一点：Harness 要随模型更新而主动演化。

作者说了一句我觉得非常精准的话：Harness 里的每个组件都是一个假设，这些假设会随着模型进步而过时。每次有新模型发布，都值得回顾一遍：有哪些组件是我加进来是因为旧模型做不到某件事，但新模型已经能原生处理了？有哪些新的能力空间，可以让 Harness 探索之前根本做不到的任务？

这里有一个很容易踩的坑：你可能会忍不住在拿到新模型之后一次性推翻重来，想从零设计一套更好的 Harness。作者就做过这个尝试，结果是不仅无法复制原有 Harness 的性能，还很难判断哪些新的设计决策真正有效、哪些只是引入了新问题。他后来转向了更保守的逐组件移除策略——一次只动一个部分，测试影响，然后决定是否保留。这种方法工作量更大，但对结果的掌控力也更强。

这种思维方式对 AI 工程师来说非常重要，因为这个领域的土地本身在移动。今天的最佳实践，明天可能就是多余的脚手架。而明天的最佳实践，今天甚至还想象不到。

好，我们来总结。

这篇文章的核心贡献，是把"生成对抗"的思路系统地应用到了 AI Agent 的代码生成和设计生成上：把生成和评估分离成两个独立的 Agent，解决了自我评估过于宽松的问题；通过精心设计评分标准，把主观的"好不好"变成了可操作的评判维度；通过规划者把短小提示词扩展成完整规格，降低了对用户的技术门槛；通过冲刺合同提前锁定完成标准，减少生成器和评估器之间的误解。

我个人认为，这篇文章最有价值的地方，不在于某一个具体的技术选择，而在于它示范了一种面对 AI 工程问题的工作方法：找出明确的失败模式，有针对性地设计机制来解决，再通过控制实验验证哪些组件真的有效，然后随着外部条件变化重新检验假设。这种方法论，在整个 AI 工程领域都是稀缺的。

作者最后那句话也值得记住：有趣的 Harness 组合空间不会随着模型进步而缩小，它只会移动。随着模型越来越强，我们不是变得不需要工程，而是有机会做更有雄心的工程。
