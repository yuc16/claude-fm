## Script

今天我们要聊的这篇文章，英文标题是"Designing AI resistant technical evaluations"，我把它翻译成"如何设计能抵抗 AI 的技术评估题"。这篇文章发表于二〇二六年一月二十一日，作者是 Tristan Hume，Anthropic 性能优化团队的负责人。

先说为什么这篇文章值得你花二十五分钟来听。它记录了一场非常罕见的真实竞赛：一个工程师和自家 AI 模型之间的猫鼠游戏。他设计了一道精心打磨的技术评估题，新版 Claude 出来，题目被解掉，他不得不重新设计，然后再次被解掉，这个循环发生了整整三次。这不是一个抽象的关于 AI 影响招聘的讨论，而是有时间线、有具体数据、有工程细节的真实案例。听完你会对 AI 能力的进步速度，以及它对工程工作的实际冲击，有一个非常具体的感知。

让我先给你交代背景。

二〇二三年十一月，Anthropic 正在准备训练和发布 Claude Opus 三。他们刚拿到了新的 TPU 和 GPU 集群，一个大规模的 Trainium 集群也快上线了，花的钱比以前多了很多。但问题是没有足够的性能工程师来驾驭这个新规模。

你可能会问，性能工程师到底是什么？简单说，就是专门让程序跑得更快的人。但在 AI 公司，这不是一般意义上的"后端性能优化"，不是改改 SQL 查询、加个缓存这种层次。这里说的是深入到硬件加速器的最底层，让矩阵计算每一个时钟周期都不浪费。这需要你同时懂加速器架构、懂指令级并行、懂内存层次结构，是市场上极度稀缺的复合能力。

Tristan 在 Twitter 上发了条帖子：我们在招人，感兴趣的发邮件来。结果涌进来的候选人数量，远超标准面试流程能处理的规模。每个候选人要经过多轮面试、占用多个工程师几个小时，如果几十个候选人同时在排队，整个团队的正常工作就会被打乱。他需要一种更高效的筛选方式。

于是他花了两周，设计了一道外带作业题——就是那种你拿回去在规定时间内完成的题目，不是现场面试。

这种形式本身有几个对评估性能工程能力特别有价值的优点：时间窗口更长，工程师在实际工作中很少面对一小时内的编程截止时间，四小时的窗口更接近工作真实节奏；环境更真实，没有人在旁边看着、不用边做边解释；有时间理解系统和搭建调试工具，这在五十分钟的现场面试里根本没法实现；最后，可以允许候选人使用 AI 辅助工具，就像他们在工作中会做的那样。

Tristan 给这道题定了四个设计原则。我觉得这四个原则本身就值得记下来，放在你设计任何类型的技术评估时参考。

第一，代表真实工作。题目要让候选人真正感受到这份工作是什么样的，不是抽象的算法题，而是真实会遇到的那类挑战。性能优化本质上是一门实验科学：你要观察机器的行为，形成假设，验证假设，然后迭代。这个过程要在题目里体现出来。

第二，高信噪比。题目不能只靠一个关键洞察。如果关键就是那一个灵光一现的时刻，运气的权重就太大了，测出来的结果就不可靠。好的题目要有很多维度让候选人展现能力，要有足够深度，让即使最强的候选人也无法在规定时间内把所有东西都做完——这样才能看到得分的真实分布，而不是大家都做完了、根本分不出高下。

第三，不依赖特定领域知识。基础扎实的人可以在工作中学具体知识。如果要求候选人事先掌握某个非常窄的领域，就在不必要地缩小候选人池，可能因此错过最优秀的人。

第四，好玩。快速的开发循环，有深度的有趣问题，发挥创意的空间。这一点不只是为了候选人体验，人在做自己觉得有趣的事情时往往表现出真实的最高水平。

带着这四个原则，Tristan 用 Python 构建了一个模拟加速器：一台假想的机器，特性类似 TPU。这台机器有几个让性能优化变得有趣的特性，我来一一给你解释清楚。

第一个特性是手动管理的暂存内存，英文叫 scratchpad memory，暂存区。你想象一下在一个小厨房里做饭，台面空间非常有限。普通 CPU 有自动缓存，就像有个助手会预判你接下来需要什么食材，提前帮你放在台面上。但很多加速器不是这样的，你必须手动告诉它："现在把这块数据从主内存搬到暂存区，之后从暂存区读，别每次都去访问慢速的主内存。"如果你不合理安排，程序就会频繁去访问慢速内存，大量时间浪费在等待数据搬运上，而不是真正在做计算。

第二个特性是 VLIW，全称 Very Long Instruction Word，超长指令字。这台机器每个时钟周期可以同时执行多个不同类型的操作，就像一个人能同时用左手炒菜、右手切菜、脚踩开关控制火力——但这些并行操作不是自动调度的，是你写代码时就必须明确安排好的。如果写的指令之间有依赖关系，或者某个执行单元没用上，你就白白浪费了这个周期的并行能力。这要求你在写代码时像一个微型调度员，精确安排每条指令落在哪个执行单元的哪个时钟周期上。

第三个是 SIMD，单指令多数据。用一条指令同时处理一组数据，比如一次对八个数做加法，而不是一次一个地加。这在向量化计算里非常普遍，是现代硬件提升吞吐量的主要手段之一。

这台模拟机器有配套的 Perfetto 追踪工具，能实时展示每一条指令的执行情况，就像真实 Trainium 集群上的工具一样，候选人可以直观地看到自己的程序在机器上的每一个动作。

题目本身是一个并行树遍历算法。Tristan 特意没有选深度学习相关的问题，因为当时大多数性能工程师还没有深度学习背景，而基础能力强的人完全可以在工作中学具体领域知识。候选人从一个完全串行的实现开始，逐步挖掘这台模拟机器的各种并行能力：第一步是多核并行，用来热身；然后选择做 SIMD 向量化，或者 VLIW 指令打包优化；原版还藏了一个 bug，候选人需要先发现并修掉，这在考察他们构建调试工具的能力。

初版结果非常好。第一批从 Twitter 进来的候选人里，有一个人的得分明显高于所有其他人。他二月份入职，两周后就开始优化内核，还找到了一个能绕过上线阻塞编译器 bug 的方案——那个 bug 是 tensor 索引的数学运算溢出了三十二位整数。

接下来一年半，大约一千名候选人完成了这道题，它帮助 Anthropic 招到了大多数当前的性能工程师。最有价值的是，它对简历不那么光鲜的候选人特别有用：几个表现最好的工程师来自本科直接应届，但他们在外带题上展现的能力足够强，Tristan 有信心录用他们。候选人反馈非常积极，很多人做超了四小时还停不下来。最强的无限时间提交甚至包含了完整的优化微编译器，以及几个 Tristan 自己都没预料到的巧妙优化。

好，这就是这道题有多好。接下来，我们来聊它是怎么被打败的。

二〇二五年五月，Claude 三点七 Sonnet 悄悄爬到了这样一个水平：超过百分之五十的候选人，如果把所有工作完全交给 Claude Code，结果反而比他们自己做更好。注意，这还只是 Sonnet，不是当时最强的模型。Tristan 随后测试了预发布版的 Claude Opus 四，结果更直接：Opus 四在四小时限制内，拿出了比几乎所有人类候选人都更优化的解答。

这意味着什么？评估题的核心假设是：候选人之间的得分差异，反映他们真实的能力差异。但如果存在一个"最优策略"是"把一切都交给 AI 去做"，题目就失去了区分意义。你测到的不再是候选人的能力，而是他们有没有想到用 AI。

不过 Tristan 注意到，Claude Opus 四虽然打败了大多数候选人，还没有追上顶尖候选人的水平。这给了他一点空间。他做了一件事：用 Claude Opus 四去测试题目，找出模型开始真正卡壳的地方，那个地方就成为新版题目的出发点。他同时清理了代码，添加了新的机器特性来增加深度，去掉了多核并行——Claude 已经轻松解决了那部分，保留它只会拖慢开发循环，不增加有效信号。时间限制从四小时缩短到两小时，因为四小时的调度开销会让整个流程拖延好几周。

这是第二版，强调的是优化洞察，而不是调试和代码量。它工作了好几个月，直到 Claude Opus 四点五出现。

Tristan 测试预发布版 Claude Opus 四点五 的场景是这样的：他看着 Claude Code 工作了两个小时，解决方案逐渐改进。Claude 解掉了初始瓶颈，实现了所有常见的微优化，不到一小时就达到及格线。然后它停下来了，认定自己遇到了无法突破的内存带宽瓶颈。大多数人类候选人也会在同样的地方停下来做出同样的结论。但实际上存在一些利用这个问题特殊结构的巧妙技巧，可以绕过这个瓶颈。

Tristan 告诉 Claude 一个可以达到的目标循环数，Claude 思考了一会儿，找到了那个技巧。然后调试、调优，继续做更多优化。到两小时时，它的得分和最强人类候选人在两小时内的表现持平——而那个人类还大量使用了 Claude 四来辅助。用内部测试框架验证，不仅能在两小时内超过人类，给它更多时间得分还会持续提升。那道题被彻底击败了。

现在面对一个更根本的问题：下一步怎么办？

有同事建议禁止候选人使用 AI。Tristan 明确表示不想这么做，理由分两层。第一层是执行层面：你没法知道候选人有没有偷用，无法强制执行。第二层是逻辑层面：Anthropic 的工程师工作时就有 AI 辅助，如果评估中禁止用 AI，但上岗后可以用，你测的就不是候选人真实工作状态下的能力，而是他们在一个人为制造的、不自然的限制下的表现。

还有人建议把门槛提高到"明显超过 Claude Code 单独能做到的水平"。Tristan 发现这个方案有一个微妙的陷阱：Claude 工作得非常快，而人类候选人通常会花一半时间读题、理解问题，然后再开始优化。一个试图引导 Claude 工作的人类，很可能一直处于"事后才理解 Claude 刚刚做了什么"的状态。最优策略可能就变成了：坐在那里看 Claude 自动跑，偶尔点点头。这不是在评估候选人的能力，而是在评估他们的坐姿耐力。

他还指出，Anthropic 性能工程师的实际工作内容已经发生了变化。它越来越像是：艰难的调试、系统设计、性能分析、搞清楚怎么验证系统的正确性、以及让 Claude 写的代码变得更简洁优雅。这些东西很难在两小时内做客观评估，需要大量共同上下文才能看出能力差异。

他面对的是一个真实的三角困境：如果再投入时间设计新题目，要么新 Claude 版本也能解掉，要么题目难到人类两小时内根本完不成——两个结果都是失败。但他还是决定试，因为 Claude 可以帮他快速实现任何他能设计的东西，大大降低了设计新题的成本。

他选了一个基于自己在 Anthropic 真实工作中做过的优化任务：在二维 TPU 寄存器上做高效数据转置，同时避免内存 bank conflict，也就是内存访问冲突的问题。他把它简化成一个模拟机器上的问题，让 Claude 在不到一天内实现了大部分代码。然后测试 Claude Opus 四点五。

Claude 找到了一个他自己都没想到的优化——它意识到可以把整个计算做转置，而不是转置数据本身，于是把整个程序重写了。在真实工作场景里这种方法行不通，所以 Tristan 修改题目把这条路堵上。Claude 在新版上有进展，但找不到最高效的解法。看起来像是终于找到了有效的新题目。

但他还是有一点不放心，于是用 Claude Code 的 ultrathink 功能，开了更长的思考预算，重新测了一遍。

Claude 解掉了。它甚至知道解决 bank conflict 问题的具体技巧。

Tristan 回头想，明白了为什么。数据转置和 bank conflict 是很多硬件平台上工程师长期研究的经典难题，Claude 的训练数据里有大量相关内容。即使 Tristan 自己是从第一原理推导出来的，Claude 有更大的经验工具箱可以调用，它可以直接模式匹配到已知的解法。

这是整篇文章里我觉得最深刻的一个教训：不要试图用"对你来说很难"的问题来难倒 Claude。因为 Claude 读过的东西比你多得多，它的难和你的难根本不在同一个参照系里。

他需要的是一类问题：在 Claude 训练分布之外，但又不是凭空捏造的。这和他原来的设计原则之间存在矛盾——原版之所以好，恰恰是因为它像真实工作；而像真实工作意味着人类积累了大量相关知识，这些知识不可避免地进入了 AI 的训练数据。

他最终想到了 Zachtronics 游戏。如果你没听说过，Zachtronics 是一系列编程解谜游戏，比如"深圳 I/O"，英文叫 Shenzhen I/O。这些游戏的特点是：用极其受限的、非常规的指令集来编程解谜。在"深圳 I/O"里，你的程序分散在多个互相通信的芯片上，每个芯片只能存大约十条指令，只有一两个状态寄存器。想做聪明的优化，你要把状态编码进指令指针或分支标志位里，用非常规的方式思考整个计算流程。

这类问题的核心价值在于：它足够非常规，Claude 没有太多直接相关的训练数据可以调用。你没法靠模式匹配到已知答案，必须从第一原理出发推理，一步一步构建自己对这个系统的理解。而这正是 Tristan 想测试的能力。

他设计了一套新的外带题，使用一个极小的、高度受限的指令集，目标是优化解法，让它用最少的指令数完成任务。用 Claude Opus 四点五 测试：失败了。他又充实了更多题目，让同事验证：即使是比他更不熟悉这类问题的人，也能超过 Claude 的表现。

还有一个有意思的细节：他故意不提供任何可视化或调试工具。起始代码只检查你的解法是否有效，除此之外什么都没有。候选人如果需要调试，要么自己写打印语句，要么让 coding 模型在几分钟内生成一个交互式调试器。怎么在调试工具上投资，这本身就是信号的一部分——判断什么时候花时间搭工具、什么时候直接蛮力调试，是一种工程判断力。早期结果是正面的：得分和候选人过去工作的质量有很好的相关性，Tristan 最能干的一个同事，拿到了比目前所有候选人都高的分数。

他们还把原版题目作为公开挑战发布了出来，提供无限时间。文章给出了一组性能基准，用模拟机器的时钟周期数衡量，数字越小越好。我把这些数字读给你听，感受一下这场人机竞赛有多激烈：Claude Opus 四在测试框架里跑了很多小时，达到两千一百六十四个周期；Claude Opus 四点五在一个随意的 Claude Code 对话里，是一千七百九十个周期，大约和人类在两小时内的最佳表现持平；在内部测试框架里跑两小时，是一千五百七十九个；Claude Sonnet 四点五跑了更多时间后，是一千五百四十八个；Claude Opus 四点五跑了十一点五小时，是一千四百八十七个；用改进后的框架跑了很多小时，是一千三百六十三个。如果你能做到一千四百八十七个周期以下，也就是超过 Claude 发布时的最佳成绩，他们希望你把代码和简历发过来。

好，现在来谈谈这一切对我们这些听众意味着什么。

第一个值得带走的经验是关于设计技术评估题。题目的质量和题目的耐久性是两件完全不同的事情。Tristan 的原版题目在工程质量上几乎无可挑剔，但它有一个结构性脆弱点：它基于真实的性能优化挑战，真实挑战意味着人类积累了大量相关知识，这些知识不可避免地进入 AI 的训练数据。一道"像真实工作"的题目，恰恰因为它像真实工作，而变得容易被 AI 攻破。这意味着如果你负责技术招聘，需要定期用最新模型跑一遍你的评估题。如果 Claude 能在规定时间内轻松及格，题目就该更新了。这不再是一次性的设计，而是需要持续维护的系统。

第二个经验是关于是否允许在评估中使用 AI。Tristan 的立场是：不要禁止。工程师工作时就会用 AI，你应该评估他们在有 AI 辅助的真实状态下的表现。但"允许 AI"不等于随便怎么用都行——题目的设计要决定 AI 能在多大程度上完成工作，以及人类在哪些维度上仍然起决定性作用。这个问题也适用于非招聘场景：如果你在做技术评审、绩效评估或代码审查，同样需要想清楚，在有 AI 工具的前提下，你到底在评估什么。人类的判断力、系统思维、识别 AI 错误的能力、在 AI 无法帮助时从第一原理解决问题的能力——这些才是越来越稀缺、越来越有价值的东西。

第三个经验是关于如何选择评估题目的类型。Zachtronics 风格解法背后有一个可以泛化的原理：要找在 AI 训练分布之外的问题。什么样的问题在训练分布之外？通常是那些规则由你自己定义、不依赖任何已有人类经验体系的问题。你可以设计一个自己发明的、规则全新的小游戏，或者一个基于你独特业务逻辑的约束优化问题——只要规则是你制定的，AI 就没有现成的训练数据可以匹配。这个原理不只适用于招聘，同样适用于 AI 能力的基准测评和安全测试：如果你复用已有的基准，模型可能已经在训练中见过某个版本；真正测到能力边界，需要设计它没见过的情景。

第四个经验是关于职业发展方向的。文章里有一段话很值得关注：Anthropic 性能工程师的工作，越来越像艰难的调试、系统设计、性能分析、验证系统正确性，以及让 AI 写的代码变得更简洁优雅。那些能让 AI 生成初版实现的工作，已经在往 AI 身上迁移。工程师的核心价值在向更高层次移动：方向判断、系统边界、正确性验证、以及与 AI 的深度协作。如果你现在在考虑职业方向，"能写性能好的代码"会越来越成为基线，真正稀缺的能力会是：你能在什么样的问题上比 AI 更有洞见，以及你能不能以更高的抽象层次来驾驭 AI 的输出。

来做个收尾。

这篇文章的核心，是一个工程师在和自家产品赛跑的真实故事。每一次他以为设计了足够好的评估，新版 Claude 就解掉了它，迫使他重新出发。这个循环本身，是 AI 能力快速进步最生动的一个侧影。

我个人认为，Tristan 最终的选择——放弃真实性、拥抱新奇性——是一个务实但也令人有些感伤的妥协。他在文章结尾说："原版之所以有效，是因为它像真实工作。新版之所以有效，是因为它模拟的是新颖的、没有先例的工作。"这句话很简洁，背后藏着一个很深的问题：当真实工作和新颖问题之间的距离越来越大，我们的评估体系究竟在测什么？他说他不想相信人类只有在几小时以上的任务里才有优势，他还在找证明这一点的方法。我希望他能找到——那个答案，对我们所有在 AI 时代工作的工程师来说，都很重要。
