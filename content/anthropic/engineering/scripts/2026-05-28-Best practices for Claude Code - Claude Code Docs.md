## Script

大家好，欢迎收听 Claude FM。今天这期，我们要聊的是一篇来自 Anthropic 官方工程博客的实践指南，原题是"Best practices for Claude Code"，中文可以翻译成"Claude Code 最佳实践"。这篇文章发表于二〇二六年五月二十八日，距今不到三周，可以说是非常新鲜的第一手干货。

为什么觉得这篇文章值得专门花二十五分钟聊一聊？说实话，市面上讲"如何用 AI 写代码"的文章已经多到数不清了，但绝大多数都停留在演示层面——看，我给 AI 说了个需求，它帮我写出来了，很厉害对不对。而 Anthropic 这篇文章完全不在这个层次上。它从工程实践的视角出发，把 Claude Code 在真实开发场景里的种种细节、工作流设计、容易踩的坑点和具体操作策略，系统地梳理了一遍。如果你现在正在使用 Claude Code，或者在认真考虑把它引入你的日常工作流，这篇文章几乎就是你的操作手册。

我们先来说背景。Claude Code 是 Anthropic 推出的一款命令行 AI 编程工具，可以直接跑在你的终端里，帮你读代码、写文件、执行命令、调试问题、跑测试。和浏览器里那种对话式 AI 不一样，Claude Code 是真正嵌入在你开发环境里的——它能直接操作你的文件系统，能运行你项目里的构建和测试，能调用各种工具。从某种意义上说，它更像一个能独立干活的初级工程师助手，而不仅仅是个代码补全插件。

那既然它这么能干，为什么还需要最佳实践？你可能会想，难道不是把任务丢给它、坐等结果就行了吗？这就是整篇文章最核心的出发点：因为有一个非常关键的约束条件，理解并管理好这个约束，是所有最佳实践的基础。

这个约束就是 context window，也就是上下文窗口。文章第一段就开门见山地说：Claude 的上下文窗口会填满，而且填得很快，当它填满的时候，性能就会下降。

我们来解释一下 context window 是什么。你可以把它想象成 Claude 的工作记忆，或者说它的桌面面积。一次完整的对话里，所有的消息记录、它读过的每一个文件内容、每一次运行命令的输出，全都摆在这张桌面上，都占用窗口空间。大语言模型的这个窗口是有上限的。一次调试会话，或者让 Claude 去探索一个大型代码库，可能轻轻松松就消耗掉几万个 token。当窗口快满的时候，Claude 就会开始"忘记"之前的指令，犯更多错误，输出质量明显下降。

文章用了一个非常精准的说法：context window 是你最需要主动管理的资源。这个视角很重要。很多人把 AI 助手当成一个无限容量的大脑，但它实际上更像一张有限面积的工作台——你往上堆的东西越多，找东西就越难，出错的概率就越高。整篇文章的所有建议，几乎都可以归结为同一件事：合理分配你的 context 资源，确保 Claude 在处理真正重要的任务时，工作台上是干净的、高度相关的内容。

建立了这个基础认知之后，我们来一条一条拆解文章里的核心实践。

第一条，也是我觉得最重要的一条：给 Claude 提供一种能自己验证工作的机制。

你来想象一个场景：你叫 Claude 修复一个 bug，它改了一堆代码，然后告诉你"好了，修完了"。你怎么知道它真的修好了？如果没有任何自动化验证机制，Claude 停下来的唯一依据就是"看起来完成了"。而"看起来完成了"并不等于"真的完成了且没有引入新问题"。这时候你就成了这个验证循环里唯一的一个人。每一个隐藏的错误，都要等你亲自来发现，亲自来纠正。这完全抵消了让 AI 帮你干活节省下来的时间。

文章的建议是：给 Claude 一个能返回通过或者失败信号的东西，让它自己形成验证闭环。这个东西可以是测试套件，可以是构建命令的退出状态码，可以是 lint 工具，可以是把输出和预期结果做对比的脚本，甚至可以是截图和设计稿的视觉比对。只要这个东西能产生一个 Claude 可以读取和判断的信号，整个循环就能自动闭合：Claude 完成工作，运行验证，读取结果，如果不通过就继续修，直到验证通过为止。

文章里给了几个非常直观的对比例子，我来讲两个感受一下差别。

第一个是写函数的场景。如果你说"帮我写一个验证邮箱格式的函数"，Claude 会写一个看起来合理的实现，然后告诉你完成了。但如果你说"写一个 validateEmail 函数，测试用例如下：user at example.com 应该返回 true，字符串 invalid 应该返回 false，user at 点 com 应该返回 false，实现完成后把这些测试跑一遍"，这就完全不同了。Claude 不仅会写实现，还会自己运行测试来验证，如果测试挂了就继续修，直到全部通过为止。你只需要等最终结果，而不需要在中间做任何人工干预。

第二个是 UI 场景。与其说"把 dashboard 做得好看一点"——这个需求模糊到 Claude 根本无从下手——不如截一张你想要的设计稿，粘贴到 prompt 里，然后说"按照这张设计来实现，实现完成后截一张结果的截图，和原设计对比，列出差异，然后逐一修正"。这就把视觉验证嵌入进了工作流，Claude 不是在猜你的审美，而是在对着一个明确的标准做客观比对。

文章还提到了几种更系统化的验证方式。有一种叫 Stop hook，就是在 Claude 每次想要结束任务之前，自动运行你预设的验证脚本。只要脚本返回失败，Claude 就不能结束，必须继续处理。这对于无人值守的自动化任务来说特别有价值——你提交了任务去睡觉，醒来之后看到的是已经自动迭代到通过验证的结果，而不是一个"看起来完成了"但不知道对不对的输出。还有一种是让第二个 Claude 实例来独立审核第一个 Claude 的输出，两个实例各自有独立的 context，互不影响。这样就避免了"自己出题自己批改"的问题——独立审核者没有原先 Claude 的那些推理过程和思路包袱，它只看结果，评判会更客观。

第二条核心实践：先探索，再规划，最后才编码。

这个三段式工作流是文章里我觉得非常有洞见的一部分。很多人用 Claude Code 的方式是：说清楚需求，Claude 立刻开始写代码。文章指出，这种方式很容易产出"解决了错误问题的代码"——代码写得很好，测试也通过了，但它解决的不是你真正要解决的问题。听起来很荒谬，但这在实际工程里发生的频率比你想象的高得多，特别是在你自己对问题还没完全想清楚的时候。

文章推荐的流程是先进入 plan mode，也就是规划模式。在这个模式里，Claude 只读文件、只回答问题，不做任何实际修改。你可以让它先把所有相关的代码读一遍，理解现有结构、明确边界条件、搞清楚依赖关系，把你自己都还没意识到的隐藏约束找出来。这个探索阶段结束之后，再让 Claude 写出一份详细的实现方案。

方案写出来之后，有一个非常实用的细节：你可以按 Control G 把这份方案在编辑器里打开，直接修改它，然后再让 Claude 按修改后的方案来执行。这就在执行之前插入了一个人工对齐的检查点，在 Claude 真正动手之前先确认方向是对的。这个环节的成本非常低，但能避免很多"改了好几百行代码发现方向从一开始就错了"的惨痛情况。

确认方案之后，切出 plan mode，让 Claude 开始实现，结合前面说的验证机制来检查结果。

当然，文章也说得很务实：规划是有开销的，不是每个任务都值得走完整流程。如果任务范围很清晰、改动很小——比如修个拼写错误、加一行日志、重命名一个变量——直接叫 Claude 干就行了。有一个很好用的判断标准：如果你用一句话就能说清楚要改什么，就跳过规划；如果说不清楚，或者会涉及多个文件，或者你对这块代码不熟悉，就先走规划流程。规划不是负担，是在避免一种更大的负担。

第三条核心实践：提供精确的上下文。

Claude 能推断意图，但它不会读心术。你脑子里有的那些"不言而喻"的上下文，Claude 完全没有。文章里有很多很有说服力的改写前后对比，我来讲几个最典型的。

写测试的场景。"帮我给 foo.py 加测试"这个 prompt 太宽泛了——Claude 不知道你最想覆盖哪个场景，不知道你对 mock 的偏好，不知道你用什么测试框架。换成"为 foo.py 写一个测试，覆盖用户未登录的边缘场景，不要用 mock"，这就给了具体的文件名、具体的场景和具体的约束条件，Claude 的输出质量会有质的提升。

调试 bug 的场景。"修复登录 bug"这四个字对 Claude 来说几乎等于什么都没说。但换成"用户反映在 session 超时之后登录会失败，请检查 src/auth/ 下的认证流程，特别是 token 刷新的部分，先写一个能复现问题的失败测试，然后修复，修复完成后确认测试通过"，这就给了症状、可能的位置和对完成的具体定义。Claude 有了明确的终点，不会在修了一半的时候迷失方向。

理解代码历史的场景。很多时候你想知道某个 API 为什么设计成现在这个样子，这种问题你直接问 Claude，它只能猜测。但如果你说"去看一下 ExecutionFactory 的 git 提交历史，总结一下它的 API 是怎么演变成现在这样的"，Claude 就会去读真实的提交记录，给你一个有据可查的答案，而不是凭空编造的解释。

参考已有模式的场景。"添加一个日历组件"这个需求太抽象，Claude 可能会引入你不想要的第三方库，或者风格和现有代码格格不入。但如果你说"看一下首页上的 widget 实现，HotDogWidget.php 是个好例子，按照这个模式实现一个新的日历 widget，让用户能选择月份并前后翻页来选年份，不要引入代码库里没有的新库"，这同时给了参考点和约束边界，产出的代码会和你现有的代码库自然融合，而不是像一个外来的异物。

还有一条我觉得特别实用的小技巧：你可以问 Claude "你觉得这个文件有哪些可以改进的地方"。这种开放式问题往往能让 Claude 发现你自己都没意识到的问题。有时候你只关注你已经注意到的那个问题，但 Claude 通读完整个文件之后可能会发现几个更隐蔽的 issue，甚至包括一些潜在的安全隐患或者性能问题。这种用法把 Claude 从一个"执行指令的工具"变成了一个"有时候比你更全面"的审查者。

第四条核心实践：把你的工作环境配置好。

这一块文章讲得非常详细，涉及好几个不同的工具和机制，我来重点讲几个最重要的。

首先是 CLAUDE.md 这个文件。这是一个特殊的配置文件，Claude Code 在每次对话开始时都会自动读取它。你可以把常用的 bash 命令、代码风格规则、工作流约定等等写进去，让 Claude 在每次会话里都自动获得这些持久化的上下文，而不需要每次都重复交代。

文章对 CLAUDE.md 的建议非常清晰：保持简短，保持人类可读。它给了一个非常好用的自我审查标准：问自己"如果删掉这条内容，Claude 会犯错误吗"？如果不会，就删掉。这个标准执行起来会让你的 CLAUDE.md 变得非常精炼，因为臃肿的配置文件反而会让 Claude 忽略掉真正重要的指令——关键内容被大量噪音淹没了，Claude 读完之后并不能有效地遵守所有规则。你花了很多心思写进去的规范，最后成了装饰品，这比不写还糟糕。

什么适合写进 CLAUDE.md 呢？Claude 无法从代码里猜出来的 bash 命令，和语言默认规范不同的代码风格规则，测试说明和你偏好的测试框架，仓库约定比如分支命名规范和 PR 惯例，项目特有的架构决策，开发环境的特殊设置比如必须设置的环境变量，还有一些不直观的行为和常见陷阱。

什么不应该写呢？Claude 读代码就能搞清楚的东西，标准的语言规范，详细的 API 文档，经常变动的信息，以及"写干净的代码"这种显而易见的废话。

CLAUDE.md 还支持多层级配置，这一点非常有用。家目录下的 CLAUDE.md 对你所有的 Claude 会话都生效，适合放你的个人全局偏好；项目根目录下的 CLAUDE.md 可以签入 git 里和团队共享，适合放团队约定；还可以放一个 CLAUDE.local.md，用于你个人的项目特有笔记，加进 gitignore 不和团队共享，适合放一些个人的临时工作记录或者你自己的特殊习惯。如果你在一个 monorepo 里工作，父目录和子目录都可以各自有 CLAUDE.md，它们会被按层级自动读取，非常适合在公共约定和模块特有约定之间做分层管理。

其次是 hooks，也就是钩子机制。Hooks 可以在 Claude 工作流的特定节点自动触发你预设的脚本。和 CLAUDE.md 里写的"建议性"指令不同，hooks 是确定性的——它保证会在那个时间点执行，没有任何商量余地。文章里举了两个例子："每次编辑文件之后自动运行 eslint"，以及"阻止向 migrations 文件夹写入任何文件"。这两件事如果只写在 CLAUDE.md 里，是否真的被遵守要看 Claude 当时的状态；但做成 hooks，就变成了一个程序层面绕不过去的机制。而且你完全可以叫 Claude 帮你写这些 hooks——比如直接说"帮我写一个在每次文件编辑后运行 eslint 的 hook"，它会生成对应的配置。

然后是 subagents，也就是子代理。子代理运行在独立的 context 里，有自己的工具访问权限，独立于你的主会话之外。它的核心价值在于：当你需要让 Claude 去做大量文件探索或者调研工作时，把这些工作交给子代理来做，探索过程中读的那些文件所消耗的 token 就不会进入你的主会话 context。子代理做完之后会汇报一个摘要，你的主 context 只增加了这个精炼的摘要，而不是几千行的文件内容和中间过程。你可以把子代理理解成外包出去的调研任务——调研结论你要，调研过程里看的那些具体材料你不需要保留在自己的工作台上。

第五条核心实践：主动管理你的会话状态。

这部分我觉得是最容易被忽视，但在实际使用效果上影响非常大的内容。

文章的核心观点是：频繁纠正 Claude 不是一个高效的工作方式。如果你已经对同一类问题纠正了两次，Claude 还在犯同样的错，这时候应该果断用 /clear 命令清空 context，然后把你从这两次失败里学到的教训吸收进去，写一个更精准的初始 prompt 重新开始。一个清晰的全新会话，几乎总是比一个充满了失败尝试和纠错记录的长会话效果更好。这个建议听起来简单，但实际执行需要一定的自律——因为你会觉得清空之前所有的对话记录很可惜，好像之前的工作都白费了。但你需要认识到：那些失败的尝试记录对 Claude 来说不是宝贵经验，而是干扰信号，让它在噪音里越陷越深。

文章还介绍了 /compact 命令：可以让 Claude 对当前的会话历史做一次压缩摘要，释放 context 空间，同时保留最重要的内容。你可以带指令来控制压缩的侧重点，比如"/compact 重点保留所有 API 相关的改动"，这样压缩后最关键的上下文还在。你甚至可以只压缩一部分历史——选择某个检查点之前的内容做压缩，保留之后的完整记录；或者反过来，压缩最近的部分，保留早期的完整历史。这种精细的控制能力让你在保留关键信息的同时大幅减少 context 占用。

还有一个 checkpoint 机制，也就是检查点。Claude Code 会在每次做改动之前自动给当前的文件状态存一个快照。如果某次改动搞坏了，你可以双击 Escape 或者运行 /rewind 来打开回滚菜单，选择只还原代码、只还原对话，或者两者都还原。而且这个检查点是跨会话持久化的——关掉终端再重新打开，检查点还在，你照样可以回滚到之前任意一个保存点。文章里有一句话我觉得特别有力：与其小心翼翼地规划每一步操作，不如大胆地叫 Claude 先试一个风险较高的方案。如果行不通，回滚，换个方向重来。这和你用 git 分支做实验是完全一样的逻辑——大胆尝试，随时有退路，因为退路已经自动替你存好了。

接下来聊聊自动化和水平扩展，这是文章后半段的重点。

当你在单个交互会话里用熟了 Claude Code 之后，下一步就是把这种能力乘以 N。文章里的这部分，视角从"单个人和单个 Claude 的协作"拓展到了"系统性地组织多个 Claude 实例来处理更大体量的工作"。

最基础的方式是非交互模式。在命令行里加上 -p 参数带上你的 prompt，Claude 就会直接执行并输出结果，不需要开一个交互式会话。这是把 Claude 集成进 CI 流水线、pre-commit hook 或者任何其他自动化工作流的标准方式。输出格式支持纯文本、JSON 和 streaming JSON，方便你做程序化处理，也方便你把 Claude 的输出作为管道的一个环节来使用。

再往上是并行运行多个 Claude 实例。文章提到了几种具体方式：用 git worktree 让多个命令行会话在各自独立的代码分支里运行，相互之间完全隔离，编辑不会冲突；用桌面应用来可视化地管理多个本地会话，每个会话各自有独立的 worktree；或者在云端的隔离虚拟机里运行会话，不占用本地资源，特别适合长时间运行的重量级任务。

文章里还描述了一个很有意思的协作模式：会话 A 专注于实现功能，会话 B 专注于代码审查。A 实现完之后，你把代码和要求交给 B 来审查，B 给出反馈后再传回 A 来修正。这是一个简单但有效的多智能体工作流，两个 Claude 各自有干净的 context，各自专注于一件事，通过你手动传递信息来协作。这个模式的好处是 B 在看代码时没有 A 的那些"背景包袱"，它的评审是独立的，不会受到 A 在实现过程中所做的种种决策的影响。

对于大规模迁移任务——比如整个代码库的 Python 二到三的升级、几千个文件的 API 改动——文章建议先让 Claude 生成一份任务清单，然后并行分发给多个 Claude 实例来分别处理。每个实例处理一个子集，最后汇总结果。这是一个经典的 fan-out 扩展模式，适合任务量大但每个单元相对独立的工作。

最后还有一种叫做 adversarial review 的做法，也就是对抗性审查。在 Claude 长时间无人值守地完成了一大块工作之后，不要急着相信结果，而是启动一个全新的子代理，只给它看这次的 diff 和你的需求标准，让它独立评估结果是否满足要求。关键在于：这个审查者没有看到生成这份代码的所有思考过程和推理逻辑，它只能靠结果和标准来判断，所以评估是真正独立的。不过文章这里有一个非常务实的提醒：如果你告诉审查者"找出问题"，它就会找出问题，哪怕代码其实是对的。这会导致过度工程化——多余的抽象层、防御性代码、覆盖永远不可能发生的边界情况的测试。所以你要明确告诉审查者，只报告影响正确性或者影响实际需求的问题，其他的都是可选项，不要追着优化主观上的代码品味问题。

文章最后专门列了几种常见的失败模式，我们快速过一遍，因为这些坑真的很多人都踩过，而且往往踩了之后还不知道为什么这次 Claude 用起来感觉比上次差。

第一种叫"厨房水槽会话"：你让 Claude 做任务 A，然后突然问了一个完全不相关的问题，然后又回来继续任务 A。Context 里充满了不相关的内容，Claude 的表现就会变差。对策是在不相关任务之间用 /clear 及时清空。

第二种叫"反复纠正"：Claude 犯了同一类错误，你纠正，它还犯，你再纠正。Context 被各种失败的尝试和补丁污染了，Claude 在噪音里越陷越深。对策是纠正两次还不见效，就果断清空，带着教训写更好的 prompt 重新开始。

第三种叫"过度臃肿的 CLAUDE.md"：规则太多太长，Claude 读完之后真正重要的规则已经淹没在噪音里。对策是无情地删减——那些 Claude 已经能在没有指令的情况下做对的事情，直接删掉对应的行，或者转成 hooks 来用确定性的方式保证执行。

第四种叫"信任但不验证的空档"：Claude 给出了一个看起来合理的实现，但实际上没有处理 edge case。这是最危险的一种失败，因为你看代码觉得没问题，但线上运行就出问题了，而且你很难确定是什么时候引入的。对策是永远提供验证机制，没有验证就不叫完成，没有验证就不要交付。

第五种叫"无边界探索"：你叫 Claude 去"研究一下"某个问题，但没有给具体的范围，结果它读了几百个文件，把你的主会话 context 塞得满满的。对策是给调研任务明确的边界范围，或者用子代理来做探索，让 context 消耗发生在子代理的隔离空间里，不影响你的主会话。

好，我们来做个收尾。

Anthropic 这篇最佳实践文章，表面上是在讲 Claude Code 的使用技巧，但它背后的核心哲学是：把 AI 编程助手当成一个需要认真设计交互方式的协作系统来对待，而不是一个"丢需求进去，等结果出来"的魔法黑盒。Context window 是你最宝贵的资源，要主动管理而不是被动消耗；验证机制是你最可靠的安全网，要提前设计而不是事后补救；精确的上下文是你最高效的沟通方式，要养成习惯而不是觉得模糊就行；而工作流的设计，决定了你能从这个工具里真正榨出多少生产力。

我个人觉得，这篇文章最值得记住的一点是它提供的一个思维定势的转变。很多工程师觉得 AI 编程工具用起来效果参差不齐，主要原因是 AI 自身能力的局限，等下一代模型更强了自然就好了。但 Anthropic 这篇文章在说：在当前这个阶段，瓶颈很大程度上不在 AI 的能力，而在于你怎么设计交互、怎么提供验证反馈、怎么管理 context。用好了这些方法，你现在就能显著提升使用效果，而不需要等。
