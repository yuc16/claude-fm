好，先来说背景。

二零二四年九月是什么样的一个时间节点？那时候，大语言模型已经被广泛认可，GPT 四和 Claude 三都已经发布，行业的焦点已经从"模型够不够强"转向了"怎么把模型落地到具体业务场景"。企业级的 AI 应用开始大规模涌现：客服机器人、内部知识库问答系统、法律合规助手、技术文档搜索工具，这些场景一个比一个真实，一个比一个紧迫。

但几乎所有团队都碰到了同一个问题：光是有一个聪明的语言模型是不够的，你还得让它真正"知道"你的业务。模型的训练数据有截止日期，你公司内部的产品手册、历史合同、行业规范、内部流程，它根本不知道。你不可能把所有这些内容都加入模型训练，定制训练一个专属模型又太贵、周期太长。于是，RAG 就成了那个时代最热门的工程方向。

RAG 是 Retrieval-Augmented Generation 的缩写，中文叫做检索增强生成。名字听起来有点学术，但原理其实很直白：给语言模型配一个"外挂知识库"。用户问问题的时候，先去知识库里搜最相关的几段内容，然后把这几段内容连同用户的问题一起发给模型，让模型基于这些找到的信息来回答。

RAG 在那个时期成了 AI 工程里最炙手可热的方向之一。LangChain、LlamaIndex 这些 RAG 框架的下载量急速攀升，大量教程和最佳实践涌现出来。但大家很快又发现了另一个共同的痛点：RAG 系统的检索环节，经常表现得让人不满意。你明明知道答案就在知识库里，但搜索就是搜不出来，或者搜出来的不是最相关的那段，模型只能胡乱生成答案，甚至一本正经地编造。这个问题，就是这篇文章想要正面解决的。

我们先把传统 RAG 的工作原理说清楚，因为理解了它是怎么工作的，才能理解它为什么会出问题。

传统 RAG 的预处理分三个步骤。第一步，把知识库里的所有文档切成一个个小的文本片段，每个片段通常是几百个 token，大概就是一两段话的量，太长了向量计算很慢，太短了信息又太零散。第二步，用一个向量嵌入模型，也就是 embedding model，把每个文本片段转换成一个向量，你可以把向量理解为一串数字，它表示这段文字的"语义坐标"——语义相近的两段文字，它们在数学空间里的坐标距离也比较近。第三步，把这些向量存到向量数据库里，向量数据库特别擅长快速找出距离最近的那些向量。

查询的时候，用户的问题同样被转成向量，然后在数据库里找语义最相近的那些文档片段，把找到的片段塞进 prompt，发给语言模型生成答案。

这套流程在很多场景下是管用的，但它有一个天然的弱点：纯向量检索只捕捉语义上的相似性，对于需要精确匹配的查询，它经常失效。

文章里举了一个很典型的例子。假设你的知识库是一个技术支持系统，里面有几千份产品手册和故障排除文档。用户问的是"错误码 TS 九九九该怎么处理"。向量检索可能会找出很多讲错误码、讲故障排除的内容，但未必能精确命中"TS 九九九"这个具体的字符串。原因是向量模型在训练时学的是语义层面的关联，而 TS 九九九这种产品专有的代号，模型可能从来没有见过，它在语义空间里的位置是混乱的，搜起来不准。

这时候，一个比向量技术古老得多的方法登场了：BM25。

BM25 全称是 Best Matching 25，直译是"最佳匹配二十五"，是一种词法检索算法，做的是关键词层面的精确匹配。它的理论根基是一个叫做 TF-IDF 的概念，TF 是词频，就是某个词在这篇文章里出现了多少次，IDF 是逆文档频率，衡量的是这个词在整个知识库里有多稀有。一个词在当前文档里频繁出现，但在其他文档里很少出现，它对这篇文档就特别有代表性，权重就高。BM25 在 TF-IDF 基础上做了几个改进：它考虑了文档的长度，防止长文档单纯因为词多就占便宜；它还引入了一个饱和函数处理词频，防止某个词出现一百次的片段把出现五次的片段完全压制，避免高频词垄断结果。

把向量检索和 BM25 结合起来使用，就能在语义匹配和词法匹配两个维度上互补。向量检索找出语义上相关的内容，BM25 找出关键词精确匹配的内容，两个结果用一种叫做排名融合的技术合并去重，最后把综合排名最高的片段送给语言模型。这个 embedding 加 BM25 的组合，在文章发表之前已经是业界比较成熟的做法，也确实比纯向量检索效果更好。

但文章真正的核心贡献，不是这个组合，而是它发现并解决了一个更根本的问题——上下文丢失。

我们来认真想想这个问题。把文档切成小块是必要的，长文档放不进向量模型，也放不进有限的 prompt 空间。但一旦切成小块，每个小块就只包含原文档里某一小段的内容。很多时候，这段内容脱离了文档的整体背景，就变得模糊，甚至失去了被检索的价值。

文章给的例子极其经典，值得仔细品味。假设你的知识库里存的是美国上市公司向证券交易委员会提交的财务报告，用户问的是"ACME 公司二零二三年第二季度的营收增长是多少"。知识库里有一个文档片段，内容是"该公司的营收比上一季度增长了百分之三"。

这段话，从内容来看，很可能正是我们要找的答案。但问题在于，这个片段里没有公司名称，没有年份，没有季度信息。这些信息在原始的财报里可能出现在更早的章节里，切块之后就永远地丢失了，无法跟着这个片段一起走。

检索系统带着"ACME 公司"和"二零二三年第二季度"这些关键词去搜索，这个片段根本搜不到，因为片段里就没有这些词。就算搜到了，模型也无法确认这段"百分之三"说的是不是 ACME 公司、是不是二零二三年第二季度的数据。

这个问题，文章叫做"上下文丢失"，它不是偶发的边缘案例，而是任何切块式 RAG 系统都必然面临的系统性缺陷。越是信息密集、结构复杂的文档，这个问题就越严重。

Anthropic 提出的 Contextual Retrieval，也就是上下文检索，给出了一个看起来简单但非常有效的解法：在把每个片段编码成向量之前，先给这个片段加一段解释性的上下文说明，把这个片段在整篇文档里的位置和背景用简洁的语言写出来。

还是用 ACME 公司的例子。原始片段是"该公司的营收比上一季度增长了百分之三"。加上上下文说明之后，它变成了这样的形式：前面先说"这段内容来自 ACME 公司就二零二三年第二季度业绩向证券交易委员会提交的报告，该公司上一季度的营收为三亿一千四百万美元"，然后再接原始片段的内容。

现在，这个文本片段里有了公司名称，有了时间信息，有了具体的财务背景数字。不管是向量检索还是 BM25，都能从这段加了上下文的文字里找到 ACME、二零二三年、第二季度这些关键信息，检索的准确率自然就上去了。

文章把这个方法拆成两个子技术。第一个叫 Contextual Embeddings，上下文嵌入，就是在生成向量的时候，用的是加了上下文说明的版本。第二个叫 Contextual BM25，上下文 BM25，就是建立词法检索索引的时候，同样用加了上下文说明的版本。两者合用，效果最好。

那问题来了：谁来给每个片段写这段上下文说明？

显然不可能靠人工。一个大型知识库可能有几十万甚至几百万个片段，手动给每个片段写背景说明是完全不现实的。文章的解决方案是用 Claude 来自动生成。

Anthropic 设计了一个专用的 prompt，逻辑非常清晰：把整篇文档提供给 Claude，同时把当前要处理的片段也提供给它，然后让它用简洁精炼的语言，写出这个片段在整篇文档里所处的位置和上下文背景。用的是 Claude 三 Haiku——Haiku 是 Claude 三系列里体量最小、速度最快、成本最低的版本，特别适合做需要大批量调用的处理任务。每个片段生成的上下文说明通常在五十到一百个 token 左右，非常精炼，不是长篇大论。

你可能立刻会有一个疑问：处理每个片段的时候都要把整篇文档发给 Claude，如果文档很长、知识库又很大，这个成本不就爆炸了吗？

这里有一个关键技术彻底改变了成本结构：prompt caching，也就是提示词缓存。Anthropic 在二零二四年为 Claude API 推出了这个功能。核心原理是：你可以把一段经常重复使用的文本内容缓存起来，后续的 API 调用可以直接引用缓存，不需要每次都重新传输这段内容，缓存的部分 token 费用大幅打折。

对于同一篇文档里的所有片段，你只需要把文档正文缓存一次，然后处理每个不同片段时，只需要发送片段本身和处理指令，缓存的文档内容在经济核算上只算一次。

文章按照实际参数做了估算：假设每个片段是八百个 token，每篇文档是八千个 token，上下文生成的指令是五十个 token，生成的上下文说明是一百个 token，用 Claude 给整个知识库一次性生成所有上下文说明的总成本，大约是每百万文档 token 一点零二美元。这个数字对于一次性的预处理任务来说，对大多数商业应用场景是可以接受的，况且这是一次性的成本，不是每次查询都要花的。

好，说完方法，我们来看实验结果。

Anthropic 在四个不同的知识领域做了系统性测试：代码库、小说、ArXiv 学术论文、科学论文。评估指标用的是检索失败率，也就是在返回的前二十个片段里，有多少比例的相关文档没有被检索到——他们叫做"一减去 recall at 20"。他们用了多个向量嵌入模型做对比，性能最好的是 Gemini Text 零零四这个嵌入模型。

实验结果如下。基础线是传统的纯向量嵌入检索，失败率是百分之五点七。仅仅加上上下文嵌入这一项，失败率就降到了百分之三点七，降低了百分之三十五。再叠加上上下文 BM25，失败率进一步降到百分之二点九，比基础线降低了百分之四十九，也就是将近一半的失败被消除了。

这已经是很可观的提升了。但文章还在此基础上加入了另一个技术——重排序，也就是 reranking。

重排序的基本思路是这样的。在初步检索之后，你可能得到了一百五十个候选片段，质量良莠不齐，有非常相关的，也有沾边的，还有凑数的。重排序是一个精细筛选步骤：用一个专门训练过的重排序模型，对这一百五十个候选片段逐一评分，评估每个片段和用户问题的相关程度，然后只保留得分最高的前二十个，把这二十个送给语言模型来生成最终答案。

重排序模型做的事情比向量相似度搜索精细得多。向量搜索是在数学空间里比较坐标距离，计算的是一种数学意义上的接近程度。而重排序模型真正理解用户问题和候选片段之间的内容关系，能做出更准确的判断。文章使用的是 Cohere 提供的重排序模型，Voyage 也有类似的产品。

加入重排序之后，结合上下文嵌入和上下文 BM25，整体检索失败率从最初的百分之五点七进一步降到了百分之一点九，三种技术叠加，总降幅达到百分之六十七，也就是三分之二的失败被彻底消除了。

这个数字意味着什么？对于一个 RAG 系统来说，检索失败就是硬失败——模型根本没有拿到正确的信息，无论它有多聪明，都无法给出正确的答案，甚至可能基于错误的信息编造出看起来合理的回答，这比什么都不说还糟糕。把检索失败率从接近六个百分点压低到不到两个百分点，就是把一个"还凑合"的系统变成了一个真正可以依赖的系统，这对于实际业务的影响是巨大的。

文章还总结了一些细节发现值得一提。在测试的几个嵌入模型里，Gemini 和 Voyage 的效果明显优于其他选择，这提示我们在做技术选型时不能想当然地选最常见的那个。关于每次给模型提供多少片段，测试结果显示提供二十个比十个或者五个效果更好，但文章也明确指出，不是越多越好——片段数量超过某个阈值，反而会分散模型的注意力，效果下降。所有这些优化手段的效果是可以叠加的，上下文嵌入、上下文 BM25、重排序三者合用，总效果比任何单一方法都好得多。

好，技术内容我们说完了。接下来说对你实际工作的意义，这才是最重要的部分。

首先有一个前提判断必须在一切开始之前先做：你的知识库到底有多大？

文章里有一段特别重要的提醒。如果你的知识库总量不超过二十万个 token，差不多就是五百页文档的量，那你根本不需要搭 RAG 系统。最简单也最有效的做法，就是把整个知识库直接放进 prompt 里，完整地给模型看，不做任何切块和检索。结合 prompt caching，这个方案的 API 成本可以降低百分之九十，延迟可以降低一半以上。

很多工程师一上来就搭 RAG，是因为 RAG 是业界的"标准方案"，看起来专业，技术含量够高。但这往往是过度设计。如果长上下文能解决问题，就直接用，省掉所有检索管线、向量数据库、召回策略的复杂度。只有当知识库大到真的装不进上下文窗口，才真正需要 RAG 这套体系。先评估体量，再决定方案，这一步省掉了很多团队走弯路的时间。

好，假设你的知识库确实很大，必须用 RAG。怎么把这篇文章里的方案落地？

我来说一个相对具体的实施路径，六个环节。

第一个环节：搭建双路检索管线。你需要两套并行的索引——一套向量索引，做语义检索；一套 BM25 索引，做精确词法匹配。向量数据库可以选 Pinecone、Weaviate、Qdrant，或者如果你的系统已经在用 PostgreSQL，也可以用 pgvector 扩展来存储向量。BM25 部分，Elasticsearch 或者 OpenSearch 是比较成熟的选择，如果数据规模不大也可以用专门的轻量级 BM25 库。两路检索的结果用排名融合算法合并，常用的有 Reciprocal Rank Fusion，也就是倒数排名融合，实现不复杂，效果不错。

第二个环节：用 Claude 批量生成上下文说明。这是 Contextual Retrieval 的关键步骤，也是这整个方案区别于传统 RAG 的核心所在。实现的时候有一个关键点：对同一篇文档里的所有片段，要复用同一份缓存，也就是把文档内容放进 prompt cache，然后循环处理每个片段，每次只传片段本身和处理指令，这样成本才可控。Anthropic 已经开源了详细的 cookbook 代码，可以直接作为参考。处理完成后，把加了上下文说明的片段用于建立向量索引和 BM25 索引，而原始片段你也要单独保留，因为最终送给语言模型的是原始片段，加了上下文的版本只用于检索阶段，不直接暴露给模型生成答案。

第三个环节：设计你自己的上下文生成 prompt。文章里提供的是一个通用模板，对大多数场景已经够用。但如果你的业务领域有特定的术语、专业缩写或者复杂的背景知识，强烈建议做定制化。举个例子，如果你处理的是医疗领域的文档，可以在 prompt 里加入常用医学术语和缩写的解释；如果是金融行业，可以说明文档的报告类型和财务指标的定义；如果是法律领域，可以区分合同、判决书、法规这些不同的文档类型，让 Claude 在生成上下文时能够选取更准确的描述角度。通用 prompt 是起点，定制化 prompt 是上限。

第四个环节：仔细设计片段切分策略。文章专门提到，片段的大小、切分边界和片段之间的重叠量都会影响最终的检索效果，但这里没有普适的最优解，需要根据你的文档类型调整。对于结构化程度高的文档，比如技术手册、法律法规、标准规范，按自然章节或段落切分，能保持语义的完整性。对于连续叙述性的长篇文档，比如研究报告、故事叙述、访谈记录，固定大小加一定比例重叠的切法更稳健，重叠能防止关键信息正好被切在片段边界上，导致信息碎裂。重叠量一般在百分之十到百分之二十之间，具体取决于文本的密集程度。

第五个环节：考虑是否加入重排序步骤。如果你的应用对准确性要求高、对延迟有一定的容忍度，比如后台分析、报告生成、合规审查这类场景，强烈建议加入重排序。初步检索多拿一些候选片段，比如一百到一百五十个，然后用重排序模型精细筛到最终的二十个左右。如果你做的是实时对话场景，对延迟非常敏感，可以减少初步检索的候选数量，比如拿五十个而不是一百五十个，这样重排序的时间会短很多，损失的准确性相对有限，整体性价比依然比不做重排序好。

第六个环节：建立评估体系，持续测量，用数据驱动调优。文章里特别强调了这一点，我也想专门展开说一下，因为这一步在实际项目中是最容易被跳过的，但往往也是最关键的。你需要准备一批有代表性的问答测试集，覆盖你的业务场景里各种典型的查询类型——包括精确查询、模糊查询、需要跨多个片段综合信息的复杂查询、涉及专有术语的技术查询。测量检索召回率，也就是相关文档有多少比例被找到了，同时也要测量精确率，也就是找到的内容里有多少是真正相关的。每次调整任何参数，都要重新跑一遍评估，看指标的变化。不要依赖主观感受，AI 系统的行为很难凭直觉准确判断，数据才是你最可靠的指南针。

除了这六个环节，我还想说几个在实际项目里特别容易踩到的坑。

第一个坑：用同一个上下文生成 prompt 处理所有类型的文档。不同类型的文档，需要的上下文说明侧重点完全不同，放在一起用一个通用 prompt 处理，生成的上下文质量会参差不齐。如果你的知识库包含多种文档类型，花点时间针对每种类型写一个专属的 prompt，回报是值得的。

第二个坑：在没有做自己的数据对比测试的情况下，直接假设文章里的最优配置在你的场景下也是最优的。文章里测试发现 Gemini 和 Voyage 的嵌入效果最好，这在二零二四年九月是成立的，到了今天可能已经不是最新结论。而且不同领域的数据，最优嵌入模型也不一定相同。建议在项目初期就做一个小规模的对比实验，在你自己的数据上验证选型。

第三个坑：一次性把所有优化手段全堆上去，然后发现效果不理想却不知道问题在哪里。正确的做法是逐步叠加：先建立纯向量检索的基准线并测量，然后加 BM25 再测量，然后加上下文再测量，最后加重排序再测量。每一步的贡献都有据可查，出了问题知道在哪里查，调参也有方向。

第四个坑：把精力全部花在检索阶段，忽略了生成阶段的质量。文章里有一句很有用的提醒：当你把加了上下文说明的片段送给语言模型的时候，最好在 prompt 里明确区分哪部分是检索辅助的上下文说明、哪部分是原始片段的正文内容。这样模型能更好地理解信息的来源和权威程度，生成质量会更高。如果你把加了前缀说明的文本和原始片段混在一起不加区分地发给模型，模型可能会把上下文说明误认为是文档的正文内容，导致生成的答案出现混淆。

好，我们来做个总结。

这篇 Contextual Retrieval in AI Systems，发表于二零二四年九月十九日，提出了一个清晰而有效的工程方案：传统 RAG 系统把文档切块进行检索是必要的，但这个操作本身带来了上下文信息的丢失，而通过用 AI 给每个片段自动生成一段简洁的上下文说明，再结合语义向量检索和词法 BM25 检索，叠加重排序步骤，可以把检索失败率从基础线的百分之五点七压低到百分之一点九，降幅达到百分之六十七。这些技术手段不是竞争关系，而是叠加关系，合用效果最好。

我个人认为，这篇文章最值得学习的地方，不只是具体的技术方案，而是它展示的工程研究方法。作者没有笼统地说"我的方法好"，而是把一个复杂的系统拆解成若干个可以独立评估的子问题，为每个子问题设计解法，做严格的对比实验，量化每个改动的贡献，最后确认效果可以叠加。这种思维方式——把工程问题转化为可量化、可迭代的实验问题——对于任何在做 AI 应用开发的工程师来说，都是一种非常核心的能力。

我个人还觉得，这篇文章揭示了一个很有意思的产品与技术的关系：prompt caching 这个基础设施功能，不只是降低了用户的 API 成本，它实际上让 Contextual Retrieval 这个原本在经济上难以规模化的方案变得切实可行。当一个基础能力突破了某个成本阈值，一整类新的应用范式就会随之打开。这种模式在 AI 发展的历史里会反复出现，而识别这些节点，对于做产品方向判断的人来说非常重要。
