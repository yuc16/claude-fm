## Script

今天要聊的这篇文章，英文标题叫 The "think" tool: Enabling Claude to stop and think，中文可以翻译成"think 工具：让 Claude 学会停下来思考"。文章发表于二〇二五年三月二十日，作者是 Anthropic 工程团队。

在我们正式进入内容之前，有一点时效性的提示必须先说清楚。今天是二〇二六年六月十三日，这篇文章发表到现在已经过去了将近十五个月。在 AI 这个领域，十五个月是非常长的时间。事实上，Anthropic 在二〇二五年十二月十五日给这篇文章追加了一条更新说明，大意是：扩展思考能力已经大幅提升，因此在大多数场景下，他们推荐优先使用扩展思考功能，而不是专门搭建一个 think 工具。

那我们为什么还要花二十五分钟来讲这篇文章？原因有两个。第一，think 工具背后的思维方式——给模型在行动过程中专门开辟一块思考空间——这个思路本身非常有价值，理解它能帮你更好地理解整个 AI agent 的设计逻辑。第二，在特定的工程场景下，比如你的模型不支持扩展思考，或者你需要对思考过程有更细粒度的控制，think 工具依然是一个可行的工程选项。所以今天我们把它当做一个经典案例来研究，而不仅仅是一个过时的技术方案。

好，背景交代完毕，我们来聊聊这篇文章到底讲了什么。

让我先把时间拉回到二〇二五年初。那个时候，AI agent 这个方向刚刚开始爆发。大家都在用各种大语言模型来搭建能自主完成多步骤任务的系统：比如让模型帮你查数据库、调接口、根据查询结果做决策、然后再执行操作，整个流程下来可能要调用十几个工具。听起来很美好，但实际落地的时候工程师们普遍发现一个问题：模型在这种长链路任务里非常容易出错，而且出错的方式很有规律——不是模型"笨"，而是模型在处理第七步的时候忘了第二步收集到的关键信息，或者没有认真核对当前情况是否满足执行条件。

举一个具体的例子。想象你在搭建一个机票客服的 AI agent，用户说"我要取消我的航班"。这看起来很简单，但实际上要做对，模型需要：先验证用户身份，查询订单详情，检查是不是在起飞二十四小时内，看看有没有保险，确认是否已经值机，算清楚退款金额，然后向用户确认，最后执行取消。每一步都有可能影响下一步的决策，任何一个环节判断失误都会导致整个流程出错，轻则退款金额算错，重则违反了公司的退改政策。

Anthropic 的工程师在研究这个问题的时候发现，模型出错的一个核心原因是：它在调用工具、拿到结果之后，往往会立刻去决定下一步动作，而没有一个专门的空间让它停下来把已有信息梳理一遍。就好像一个人在做多步骤的计算题，每算出一个中间结果就直接往下算，从来不在草稿纸上写下来，那出错的概率自然就高了。

think 工具的核心思路就是给模型一块草稿纸。

具体来说，think 工具是一个特殊的工具定义，你把它加到模型可以调用的工具列表里，它的功能只有一个：接受一段文字输入，记录下来，不做任何外部操作，也不查询数据库，也不调接口，就是把这段文字存进日志。模型在调用了其他工具、拿到结果之后，可以选择先调用 think 工具，把自己对当前情况的分析写出来，确认信息是否完整、下一步要做什么、有没有违反什么规则，然后再决定真正的下一步动作。

这个设计非常简洁，简洁到让人觉得"就这？"。但正是这种简洁背后，藏着一个深刻的洞见：强迫模型把思考过程显式化，会让它在复杂场景下显著减少错误。

这里我要花一点时间把 think 工具和 Anthropic 的另一个功能——扩展思考，也就是 extended thinking——区分清楚，因为这是整篇文章最容易搞混的地方，也是最值得深想的地方。

扩展思考是什么？简单来说，是模型在开始生成回复之前，先进行一段内部推理。你把问题发给模型，模型在给出答案之前，会有一段"想"的过程，这个过程用户可以选择看到也可以选择不看。它更像是模型在落笔之前先打草稿，整理思路，然后给出一个更深思熟虑的答案。扩展思考在数学推导、代码编写、逻辑推理这类任务上效果很好，因为这类任务的信息在问题里基本上就已经给全了，模型需要的是更深层的推理而不是更多信息。

think 工具解决的是另一类问题。在 agent 任务里，模型在开始行动的时候往往并不具备完成任务所需的全部信息，它需要在行动过程中不断获取新信息，每一次工具调用都可能带来新的关键数据。这种情况下，在行动开始之前做的那些思考，可能在调用了几个工具之后就过时了。think 工具的价值在于允许模型在获取新信息之后，随时停下来重新分析当前状态，而不是一路猛冲到底。

打个比方：扩展思考像是一个侦探在接到案子的时候，先在脑子里把已知线索推演一遍再开始行动。think 工具像是同一个侦探，在调查过程中每次得到新证据，都会先在笔记本上写下"这个证据说明什么、推翻了哪个假设、接下来应该问谁"，然后再继续行动。两者并不冲突，但解决的是不同阶段的问题。

原文里说得很直白：think 工具更适合那些模型无法仅凭用户提问就得出答案、需要处理外部工具返回结果的场景。think 工具处理的推理没有扩展思考那么全面系统，它更聚焦于对新获得的信息的即时处理和分析。

理解了这个区别，我们再来看测试结果就会清晰很多。

Anthropic 用来评测 think 工具效果的基准叫做 tau-bench，全名是 τ-bench。这是一个专门设计来测试模型在真实客服场景中使用工具能力的综合基准。它会模拟真实的用户对话，让模型扮演客服 agent，遵循复杂的客服策略指引，使用各种工具访问和操作数据库。

tau-bench 里有两个主要的测试域：一个是航空客服域，一个是零售客服域。航空域的规则非常复杂，退改签政策涉及大量条件分支，正是前面说的那种"每一步都依赖前一步信息"的典型场景。零售域相对简单一些，规则没有那么繁琐。

tau-bench 用的主要评估指标叫做 pass 的 k 次方，写作 pass^k。它测量的是对于一个给定任务，连续进行 k 次独立试验，全部成功的概率，然后对所有任务取平均。注意，这和我们常见的 pass at k 是不一样的。pass at k 是说 k 次里只要有一次成功就算过。pass^k 要求每次都成功，测的是稳定性和一致性。对于客服这类应用来说，稳定性才是核心，因为你不可能指望模型十次里有五次答错，哪怕另五次是对的。

好，来看具体数字。

在航空域，Anthropic 测试了四种配置：基准配置，就是啥都不加；只用扩展思考；只加 think 工具；以及 think 工具加上优化后的 prompt。

结果非常惊人。基准配置在 k 等于一的时候，也就是单次成功率，是零点三三二。只加 think 工具，提升到零点四〇四。只用扩展思考，提升到零点四一二，和 think 工具差不多。但是，think 工具加上优化后的 prompt，直接跳到了零点五八四。这是相对基准线百分之七十六的提升，绝对提升幅度超过了二十五个百分点。

文章标题里说"54% 相对提升"，这个数字对应的是另一个版本的数据，大方向是一致的：优化后的组合方案远超其他单独配置。

你可能会问，从零点四跳到零点五八，这差距大吗？在这类基准测试里，这是非常显著的差距。要知道，这个任务的难度本来就很高，每一个百分点的提升背后都需要模型在多个关键决策点上全部做对。

我们再看 k 值更大的情况，也就是要求模型连续多次成功。当 k 等于五的时候，基准配置只有零点一，think 工具单独用也只有零点一，而 think 加优化 prompt 达到了零点三四。在五次全对这个严苛标准下，优化方案的成功率是基准的三点四倍。这说明优化方案带来的不只是偶尔运气好，而是系统性的稳定性提升。

那这个"优化后的 prompt"到底优化了什么？这是文章最有实操价值的部分，我想花比较多的时间来讲清楚。

优化 prompt 的核心思路是：不只是告诉模型"你可以用 think 工具"，而是明确告诉模型在什么时候用、用来做什么、怎么用。

原文里展示了两个 think 工具使用示例，作为 prompt 的一部分。第一个例子是用户要取消航班，prompt 里列出了需要核查的事项：验证用户 ID、订单编号、原因；检查取消规则，是不是在订票二十四小时内，不是的话看舱位等级和保险情况；确认没有已经值机或者飞过的行程段；最后制定行动计划，收集缺失信息，核实规则，获取确认。

第二个例子更复杂：用户要订三张票去纽约，带两件托运行李。prompt 里把整个分析框架列得非常清楚：先要知道用户 ID 来查会员等级，因为托运费用和会员等级直接相关；然后分三种情况算行李费，普通会员每人一件免费，银卡每人两件免费，金卡每人三件免费；还要核查支付规则，最多一张旅行证书、一张信用卡、三张礼品卡，而且旅行证书剩余金额不退不找零；最后列出五步行动计划。

这个 prompt 设计的精妙之处在于：它不是在告诉模型"要仔细思考"，而是在给模型示范"在这类场景下，思考应该包含哪些具体步骤、哪些具体检查项"。这就好比你培训一个新员工，光说"遇到复杂问题要多想想"没什么用，但如果你给他一张检查清单，告诉他每次接到取消订单的请求，都要先过一遍这张清单，效果就完全不同了。

这个发现对工程实践有很强的指导意义：提示工程中的示例，比通用指令有效得多。告诉模型"按步骤思考"，不如直接给它看一个具体场景里"步骤是什么样的"。

我们再看零售域的结果。在零售域，四种配置里只测了三种：基准、扩展思考、以及 think 工具不加额外 prompt。结果是 think 工具拿到了最高分，pass^1 是零点八一二，基准是零点七八三，扩展思考是零点七七。

有意思的是，在零售域，think 工具不加任何专门的 prompt，就已经超过了基准和扩展思考。原因在于零售域的规则没那么复杂，模型光是有了一块草稿纸，自然就能用上，不需要太多指引。这印证了文章里的一个结论：难度越高的场景，优化 prompt 的价值越大；简单场景光有工具就够了。

除了客服场景，Anthropic 还把 think 工具用到了 SWE-bench 的评测上，也就是让模型来修复真实的开源软件 bug。在那个场景里，think 工具的描述稍微调整了一下，更侧重于鼓励模型用它来头脑风暴解决方案，分析测试结果，考虑不同修复方式的优劣。在 n 等于三十次的抽样实验里，加了 think 工具平均提升了一点六个百分点，统计检验的 p 值小于零点零零一，效应量 d 等于一点四七，属于非常大的效应量。这个结果帮助 Claude 三点七 Sonnet 在 SWE-bench 上拿到了当时的最优成绩，分数是零点六二三。

好，我们来聊聊实践应用，也就是这篇文章对你的实际工作意味着什么。

如果你正在搭建 AI agent 系统，第一个问题是：你的任务场景适不适合用 think 工具？原文总结了三类最适合的场景。

第一类是需要仔细处理工具输出的任务。如果你的 agent 需要从前一个工具调用的结果里提取关键信息，然后基于这个信息决定下一步，而且一旦判断错误后续就很难纠正，那 think 工具在这里就有价值。比如查询了用户账户余额之后，需要判断是否满足某个条件再决定是否执行下一步操作，这个判断点就是一个很好的 think 工具使用时机。

第二类是规则和策略密集的场景。你的 agent 需要遵循大量详细的业务规则，任何一条违反都会导致问题。这类场景里，think 工具能帮助模型在执行之前把所有相关规则都过一遍，确认自己的决策不会违规。合同审核、合规检查、金融产品推荐，都是这一类。

第三类是序列决策场景。每一步的结果直接影响下一步的选项，一步走错，整个链路都可能崩溃。这种场景对模型的状态跟踪能力要求很高，think 工具能帮它在每个关键节点重新对齐。

反过来，原文也明确说了不适用的场景：如果任务只需要一次工具调用，或者多个并行的独立工具调用，那加了 think 工具基本上没什么帮助。如果任务本身规则简单，模型默认表现就已经很好了，额外的 think 只是增加了 token 消耗，收益很小。

说到 token 消耗，这是 think 工具一个实实在在的成本。每次模型调用 think 工具，都会产生额外的 token，既有输入 token，也有输出 token。在生产环境里，这意味着更高的 API 成本和更长的延迟。所以你在决定是否引入 think 工具的时候，需要权衡：任务的错误代价有多高，额外 token 成本是否可以接受。

如果你决定要用，原文给了很具体的实施建议。

第一步，先找出你的 agent 目前表现最差的场景，特别是那些策略合规失败或者长链路任务出错的情况，这些就是引入 think 工具最值得测试的地方。

第二步，加入 think 工具定义。这个工具的技术实现非常简单，你只需要在工具列表里加一个工具，描述清楚它的用途，接受一个字符串输入，返回成功即可，后端不需要做任何实际操作。难的不是代码，而是工具描述的措辞——要让模型知道什么时候该调用这个工具，以及用它来做什么。

第三步，在系统 prompt 里提供使用示例。原文明确指出，对于长而复杂的指引，把它们放在系统 prompt 里而不是工具描述里效果更好。重点不是给一段通用说明，而是给具体场景下的具体例子，就像前面说的航空客服那两个例子。你的 prompt 应该展示出在你的业务场景里，一个好的"思考过程"是什么样子的，应该检查哪些点，应该按什么顺序梳理信息。

第四步，监测实际使用情况。在你的日志系统里记录下来 think 工具被调用的频率、内容是否符合预期、是否在关键决策点前被触发。有些时候你会发现模型在没有必要的地方调用 think，浪费了 token；有些时候你会发现模型在应该停下来想的地方没有调用。根据这些观察调整你的 prompt 和工具描述，这是一个需要迭代的过程。

这里还有一个工程细节值得一提：think 工具不会对外部世界产生任何影响，也不会改变你现有工具的行为。它纯粹是一个"日志记录"工具，用来让模型把思考过程显式化。这意味着你可以非常安全地在现有系统上做 A/B 测试，加了 think 工具的配置和不加的配置，行为上唯一的区别就是模型有没有把中间思考过程记下来，其他一切照旧。

现在我要回到开头提到的那条更新说明。二〇二五年十二月，Anthropic 在这篇文章顶部加了一段提示，说明自从这篇文章发布之后，扩展思考的能力已经大幅提升，在大多数场景下他们推荐直接使用扩展思考功能，而不是专门实现一个 think 工具，因为扩展思考提供了类似的好处——给模型空间来推理复杂问题——同时集成更好，性能更强。

这个更新说明意味着什么？如果你今天才开始搭建 agent 系统，而且你用的是 Claude 的最新模型，那你应该先去看扩展思考的文档，而不是急着实现 think 工具。但如果你用的模型不支持扩展思考，或者你需要对思考过程有更细粒度的控制和记录，或者你在用其他不支持扩展思考的模型，那么 think 工具的思路依然完全适用。

更重要的是，这两个功能背后的核心洞见是一致的：给模型显式的思考空间，比让它一步到位地给出答案，在复杂任务上要可靠得多。不管你最终用哪种实现方式，理解这个设计原则，对你做 AI agent 系统的整体架构会有帮助。

我们还可以从另一个角度来理解这篇文章的价值。Anthropic 在这里其实是在把人类专家的工作方式，显式地编码进 AI 系统的设计里。优秀的人类分析师在面对复杂问题的时候，会在纸上列出关键约束条件，画出决策树，逐步核查信息。think 工具做的事情和这个完全类似：它给了模型一个机制，让模型能够模仿这种系统化的、逐步核查的工作方式，而不是靠直觉一步跳到答案。

这个设计思路的另一个含义是：你在为 agent 写 prompt 的时候，可以把你对"专家级分析过程"的理解，通过 think 工具的使用示例，教给模型。你的领域知识和模型的语言理解能力可以真正结合在一起，而不是靠模型凭空猜测什么是好的推理过程。

最后我想说说我自己的看法。我个人认为，这篇文章最有价值的不是那些具体的测试数据，而是它揭示了一个工程实践上的重要认知：模型能力的发挥，很大程度上取决于你给它提供了什么样的"工具基础设施"和"工作流程示范"。一个好的 agent 系统，不只是选一个好模型然后把问题扔给它，而是需要精心设计模型的工作流程，给它合适的工具，告诉它在什么时候用什么工具，怎么用。think 工具是这个思路下一个非常干净的例子：用最小的工程代价，换来了显著的可靠性提升。

今天这期节目的核心结论就是：如果你在搭建需要长链路工具调用、策略合规验证或者复杂序列决策的 agent 系统，给你的模型加一块草稿纸，让它在关键决策点前停下来想清楚，是目前已经经过测试验证的、低成本高收益的工程实践。具体用 think 工具还是扩展思考，取决于你用的模型和你对思考过程控制粒度的需求，但这个核心思路在二〇二六年依然成立。
