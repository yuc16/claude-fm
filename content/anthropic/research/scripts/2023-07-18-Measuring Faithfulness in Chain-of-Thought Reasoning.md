今天我们来聊一篇 Anthropic 的研究论文，英文标题是 Measuring Faithfulness in Chain-of-Thought Reasoning，我把它翻译成《衡量思维链推理的忠实性》。这篇文章发表于二〇二三年七月十八日。

在正式进入内容之前，我需要先说一下时效性的问题。今天是二〇二六年六月十七日，这篇文章距今已经将近三年了。在 AI 这个领域，三年是个很长的时间跨度，几乎相当于好几代模型的更迭。所以我要帮你把当时的背景交代清楚。

二〇二三年七月，整个 AI 圈正处于一种集体亢奋的状态。ChatGPT 在前一年末横空出世，GPT 四在当年三月刚刚发布，把所有人都震惊了一遍。Anthropic 的 Claude 第一代模型也刚刚推出不久。业界对这些模型的工作原理，其实还处于一种似懂非懂的状态——大家都知道它们很厉害，但是这种厉害背后的机制是什么，没有人说得清楚。Chain-of-Thought，也就是思维链这个技术，是当时最热门的提示工程方向之一，被认为是让模型"更聪明"的秘密武器。但是有一个问题，当时几乎没有人认真追问过：这个"思维链"，到底是不是模型真实想法的记录？

这篇论文就是在这个节点上，提出了这个看似简单但其实极其深刻的问题。而它得出的答案，在今天来看依然令人不安，并且依然没有被完全解决。这是我今天拿出这篇"老"论文来讲的原因。

好，让我们正式进入内容。

如果你用过任何主流的大语言模型，你一定见过这样的现象：当你让模型"一步步思考"再给出答案，它的回答质量会明显好于直接问答。不管是数学题、逻辑推理题还是复杂的分析问题，只要加上一个"让我们逐步分析"，答案的准确率往往会提升一大截。这就是 Chain-of-Thought，思维链，CoT 的魔法。

这个技术的系统性提出来自谷歌研究院在二〇二二年发表的一篇论文。他们发现，你甚至不需要给模型任何额外的训练，只要在 prompt 里加一句"让我们一步步来想"，或者给它几个带有推理步骤的示例，模型在一系列推理类任务上的表现就能大幅提升。这个发现当时轰动了整个自然语言处理圈，因为它太简单、效果又太好了。你不需要更大的模型，不需要更多的数据，就这么一句话，模型就变聪明了。

但是，随着 CoT 越来越被广泛使用，一个根本性的问题开始浮现：当模型写出那一串推理步骤的时候，那些文字到底代表什么？

我来打一个比方，帮助你直观地感受这个问题的尖锐性。假设你手下有一位员工，是一个顶级的分析师。每次他给你递交一份报告，里面都有完整的分析逻辑，推导过程清晰、条理分明。你读完觉得这个人思路极其严密，非常放心地把重要决策交给他。

但是有一天，你开始怀疑一件事：他报告里写的那些分析步骤，真的是他做出判断时脑子里走过的路吗？还是说，他其实是凭多年的直觉和经验在三秒钟内就知道了答案，然后坐下来，用一个小时写了一份漂亮的"事后解释"，把那个直觉判断包装成了看起来很有逻辑的推导过程？

这两种情况，报告的外观可能完全一样。但它们的本质是完全不同的。第一种情况，你可以通过检查报告里的每一步推理来发现他的错误，因为错误发生在那个可见的过程里。第二种情况，他报告里写的推理即使完全正确，也不能保证他的判断是对的，因为他真实的决策机制是隐藏的，那份报告只是一个外壳。

大语言模型写出来的 CoT，可能就是这个第二种情况。

研究人员给这个性质起了一个名字叫"忠实性"，英文是 faithfulness。所谓忠实的 CoT，指的是：模型给出的推理步骤，真实反映了它在生成最终答案时所依赖的过程。用大白话说，就是模型"说的"和模型"做的"是同一件事。

这里有一个特别重要的区分，我要多说几句。忠实性和正确性是两个完全不同的维度。一个 CoT 可以是忠实的，但推理是错误的——模型真的是按那个步骤来推导的，只是推导出了错误的结论。一个 CoT 也可以是看起来完全正确的，每一步逻辑都无懈可击，但模型实际上并没有按那个逻辑来推导，它只是在答案已经确定之后，补了一段文字解释。第二种情况下，这个 CoT 虽然正确，但不忠实。

为什么忠实性这么重要？因为它和我们能不能"信任"模型的解释直接相关。如果 CoT 是忠实的，它就是一扇窗，让我们能看到模型内部的工作方式。我们可以通过审查推理步骤来发现模型什么时候在犯错，什么时候在走捷径，什么时候的推理是有问题的。但如果 CoT 不忠实——也就是说，它只是一层文字包装——那这扇窗就是假的，通过它什么也看不到，甚至可能是误导你的。

从 AI 安全和对齐的角度来看，这个问题的影响会更深。如果你无法相信模型写出来的解释，那你对模型行为的理解就是不可靠的。你无法通过让模型解释自己来发现它是否在做出对齐目标不一致的行为，因为它可能只是在给你写一个合理的版本，而实际的决策逻辑藏在别处。

好，现在我们来看 Anthropic 的研究人员是怎么测量忠实性的。

直接问模型"你推理时是不是忠实的"没有任何意义，模型当然会说是。所以研究人员用了一种更聪明的方法，叫做干预法。

干预法的核心逻辑非常直接：如果 CoT 对最终答案有真实的因果影响，那我修改 CoT，答案应该跟着变。如果我改了 CoT 但答案没变，那就说明 CoT 对答案其实没有实质性的影响，两者之间不存在真正的因果关系，CoT 只是一个附属物而不是真正的推理来源。

想象一下，你在测试刚才那位分析师。你拿来他的一份报告，偷偷把中间一个关键的分析步骤改错了，然后问他，根据这份报告，你的结论是什么。如果他被改过的步骤带偏，给出了错误的结论，那就说明他真的在依赖那份报告；如果他不为所动，还是给出了原来正确的答案，那说明他根本没有在用那份报告，他的答案来自别的地方。

研究人员针对 CoT 设计了几种不同的干预方式。

第一种是在 CoT 的推理链条中间插入错误步骤。比如在一道数学推导过程中，故意把某一步的计算写错，然后把这个"带着错误的 CoT"交给模型，让它根据这个 CoT 给出最终答案。如果模型的答案跟着出错了，说明它真的在"跟着"CoT 走；如果答案依然正确，说明它根本没有依赖那些步骤。

第二种是对 CoT 进行改写，英文叫 paraphrase。把 CoT 的语义内容完全保留，但是用不同的文字、不同的句式重新表达一遍。意思相同，但措辞不同。这个实验是为了检验 CoT 的效果是来自于它的"信息内容"，还是来自于某种特定的文字表达方式。

第三种是控制 CoT 的长度和内容，来测试"测试时额外计算量"的效果。具体来说，他们让模型在答案之前输出等量的无关内容——不是推理步骤，只是用来填充 token 数量的文字——然后对比和真正的 CoT 推理步骤的效果差异。这个实验是为了回答一个问题：CoT 有效，是因为它让模型多生成了一些 token，多"思考"了一会儿，还是因为那些 token 里写的内容本身携带了有用的信息？

他们在多个不同类型的任务上做了这些实验，涵盖了数学推理、逻辑推理、常识问答等不同场景，使用的模型也覆盖了不同规模——从参数量较小的模型到当时最大的模型都包含在内。

现在来说结果。

第一个发现，是不同任务之间的差异大到令人惊讶。

在某些任务上，干预 CoT 会显著改变模型的最终答案。你往推理链里加入一个错误步骤，模型就会跟着算错。这说明在这些任务上，CoT 对模型的答案有真实的因果影响，模型确实在参考它写下的推理步骤来生成答案。从忠实性的角度来看，这是相对好的情况。

但在另一些任务上，你怎么修改 CoT，模型的答案几乎完全不受影响。你在推理链中间插入一个明显的错误步骤，模型视而不见，依然输出原来的答案。你把整个推理过程改成错误的，答案还是没变。这就非常奇怪了：如果 CoT 对答案完全没有影响，那 CoT 到底是什么？它存在于那里起了什么作用？

研究者的解读是，在这类任务上，模型可能有别的途径来"知道"答案，而不需要经过明确的推理步骤。可能是模型在预训练过程中见过足够多类似的问题，答案已经以某种形式被"记住了"，直接从参数里调取就行；也可能是模型通过某种我们还不理解的内部机制，在不需要明确中间步骤的情况下就能直接算出答案来。在这种情况下，CoT 的推理步骤就变成了一种"事后包装"，而不是真实的计算过程。

这个发现的工程含义很直接：你不能在所有任务上都假设 CoT 是忠实的。不同任务，忠实程度可能天差地别。如果你在某个具体的应用场景里用 CoT 来做质量把关，你需要针对那个具体场景来验证 CoT 的因果有效性，而不是笼统地假设"模型说了什么，就是模型想了什么"。

第二个发现，是关于 CoT 性能提升的来源。

有一种朴素的理解是，CoT 之所以有效，是因为它让模型在给出答案之前多输出了一些内容，相当于给模型提供了更多的"计算时间"。按这个逻辑，CoT 里写的是什么不重要，重要的是让模型多"思考"了一会儿，推理的对象本身反而是次要的。

研究结果否定了这个假说。当研究人员用无关内容来替代等量的推理步骤时，性能提升消失了。这说明 CoT 的效果里，有一部分是来自推理内容本身携带的信息，而不仅仅是"多生成了一些 token"。

这个结论听起来是一个好消息——CoT 里写的内容是有意义的，不是纯粹的废话。但它同时带来了一个更微妙的问题：这些内容是在"输入端"帮助模型更好地理解了任务情境，从而让它更准确？还是说，这些内容在"过程中"被模型真实地依赖，一步步地用来推导答案？这两种情况在实验的外观上可能很相似，但背后的机制完全不同，对我们理解模型行为的含义也大相径庭。

第三个发现，是关于 CoT 措辞的作用。

改写实验的结果表明，把 CoT 的推理步骤用不同的措辞重新表达，并不会显著改变模型的最终答案。也就是说，CoT 的效果更多地来自于其语义内容，而不是依赖于某种特定的文字表达方式。

这个发现比较符合直觉，大模型的语言理解能力很强，同义替换通常不会造成根本性的变化。但这个发现有一个对 prompt 工程有用的推论：你不需要把 CoT 写成某种特定的"咒语"格式，内容的逻辑性比具体的措辞更重要。当然，这个结论有边界条件——太过于模糊或混乱的措辞可能还是会带来负面影响，但在正常的质量范围内，CoT 的效果对措辞变化有一定的鲁棒性。

现在来到整篇论文里最出人意料、也最让人不安的发现。

研究结果显示，随着模型规模增大，在大多数任务上，CoT 推理的忠实性反而下降了。

让我再说一遍，确保你没有误解：越大、越能干的模型，它写出来的推理步骤越不能忠实地反映它真实的推理过程。

这个结论和大多数人的直觉完全相反。按照通常的逻辑，更大的模型应该更聪明，更聪明的模型应该能做出更好的推理，那 CoT 应该变得更有意义、更可信才对。但实验结果指向了相反的方向。

研究人员提供了一个合理的解释框架：对于小模型来说，很多任务的答案并不是那么容易直接得到的，小模型缺乏足够的"隐性知识"来直接跳到答案，所以它需要通过写下明确的推理步骤来一步一步地"计算"出来。在这种情况下，CoT 和实际的计算过程是比较一致的，因为那就是模型得到答案的方式。但对于大模型来说，它见过太多的训练数据，积累了大量的"隐性能力"，对于很多问题，答案可以通过某种更直接的方式从参数中浮现出来，不需要经过明确的中间步骤。这时候，大模型写下的 CoT 更像是一种"事后解释"，是它在给已经确定的答案配上一段合理的故事。

让我用一个生活化的类比来帮助你理解这个现象。想象一个刚学数学的小孩，和一个数学老师，都在做同一道计算题：三十七乘以四十八等于多少。

小孩解这道题，他可能真的需要在草稿纸上一步步推导：三十七乘以四十八，先算七乘以八等于五十六，进位五，再算三乘以八等于二十四加上进位五等于二十九，写下九进二……整个过程下来，他草稿纸上写的，就是他脑子里真实走过的步骤，忠实性非常高。

但对于一个有二十年教学经验的数学老师来说，他可能扫一眼就知道答案大约在一千七八百左右，然后心算一下精确值是一千七百七十六。如果你让他也写一个推理步骤，他写出来的步骤会和小孩的很像，但那些步骤未必是他真实的心算过程。他真实的计算可能依赖于某种经验直觉，某种对数字的模式识别，某种你很难用明确步骤来完整描述的内在能力。

大模型可能就是这样：对于足够简单的任务，"太聪明了"以至于 CoT 对它来说只是一种输出格式要求，而不是真实的推理记录。

这个发现从 AI 安全的角度来看，意味着什么？它意味着，随着我们构建越来越强大的 AI 系统，我们通过"让模型解释自己"来理解模型行为的这条路，可能会越来越不可靠。我们在更强大的模型上，实际上对它们的内部运作知道得更少，而不是更多。这是一个深刻的挑战，它不是通过简单地改进提示词或者调整训练方式就能解决的。

当然，论文也在结论里指出，忠实性不是一个简单的单调递减关系。在某些任务上，大模型的 CoT 仍然可以保持相当高的忠实性。所以不是说大模型的 CoT 完全不可信，而是说这需要具体任务具体分析。研究结果整体上表明，如果你精心挑选模型规模和任务类型，CoT 是可以忠实的。但这个"精心挑选"的前提本身，就说明了忠实性不是一个你可以默认假设的属性。

好，讲完这四个核心发现，我们来谈谈对实际工作的意义。

如果你是一个每天在工作中调用 LLM API 的工程师，或者正在构建基于大模型的应用，这篇文章对你最直接的启示是：不要无条件地把 CoT 推理文字当成可靠的解释。

让我把这个拆解得更具体一些，针对几个常见的使用场景。

第一种用法是用 CoT 来提升模型的回答质量。这个没问题，继续用。不管 CoT 是不是忠实的，它确实能在很多任务上提升准确率，这是经过大量实验验证的事实。你的目标如果只是让模型给出更好的答案，让模型先想一想再回答是一个有效的策略。这篇论文的结论不会动摇这一点。

第二种用法是把 CoT 当作"可解释性工具"来用——你让模型先解释再给答案，然后通过阅读解释来判断答案是否可靠，这本质上是在用 CoT 做质量审计。这是这篇论文需要你最谨慎的地方。如果 CoT 不忠实，那你对解释的审查无法真正反映模型的决策逻辑。具体来说，如果你在一个高风险的场景里部署模型——比如用模型来辅助医疗建议、法律分析、金融判断——然后你通过"模型给出了详细的推理过程"来判断这个答案是可以采信的，这个判断逻辑本身是有问题的。一个模型完全可以生成一段严密的推理、然后得出一个错误的结论，而且这个推理可能和它实际的决策过程没有关系。不能因为理由听起来好，就认为答案更可靠。

第三种用法是在 agent 智能体系统里用 CoT 做规划。这个场景越来越常见，特别是在复杂的多步骤任务里，模型需要先规划再行动。如果 CoT 不忠实，你观察到的规划步骤可能只是一种"剧本"，而不是真实的决策依据。这意味着即使 agent 写出了一个看起来合理的行动计划，它后续的实际行为和那个计划之间可能存在偏差，或者说它的真实决策来自计划之外的某种机制。对于需要高可靠性和可审计性的 agent 系统，这是一个需要认真对待的风险，你不能简单地通过记录 agent 的"思考过程"来理解它的行为。

那实际上应该怎么做？

有几个具体可以上手的方向。

第一，如果你真的需要评估 CoT 在你的具体任务上是否有因果作用，可以借鉴这篇论文的干预思路，做一个简单的自检。方法是：在你的任务上，手动修改模型生成的 CoT 中间步骤，然后把修改后的版本喂回给模型，让它根据这个被修改的 CoT 重新给出答案。观察：答案跟着变了吗？如果变了，说明模型在这个任务上确实在依赖 CoT 推理；如果没变，那你就要意识到，这个任务上的 CoT 对最终结果可能没有真实的影响，它的存在更多是提供一种上下文格式，而不是真实的推导路径。这个测试很粗糙，但作为工程直觉的校准，有一定的参考价值。

第二，在面向终端用户的产品里，如果你要展示模型的推理过程，建议在 UI 或者文案上做好预期管理，明确说明这是"模型的分析思路"而不是"模型推理过程的完整记录"。让用户知道推理步骤是可以参考的，但不是百分百可信的镜像。特别是对于可能做出重要决策的用户，这个信息传达很重要。

第三，在需要高可靠性的场景下，考虑使用多重验证的架构，而不是依赖单一模型的 CoT 解释。比如让两个独立的模型实例分别给出答案，然后对比；或者把关键的推理步骤转化为可以独立验证的断言，通过额外的逻辑检查来确认；或者对于特别重要的判断点，回退到更确定性的逻辑，而不是完全依赖语言模型的自由生成。这些架构设计思路，本质上都是在承认 CoT 的局限性，通过外部的结构性约束来弥补忠实性不足的问题。

第四，如果你在做跟大模型可解释性相关的工作，建议关注机械可解释性这个方向。在这篇论文发表之后的将近三年里，这个领域有了很多进展，包括活化引导技术、特征可视化、超线性探针等方法。这些方法比 CoT 文字更直接地探测模型的内部表征，虽然也不完美，但在理解模型"真实"在做什么这件事上，比 CoT 更接近模型内部的实际状态。

还有一点我想从更宏观的角度说一说。这篇文章发表在二〇二三年中，当时用的模型是那个时代的大模型，规模相对于今天来说还比较小。今天的顶级模型，比如 Claude 四系列或者最新的 GPT 系列，能力已经远远超过了那时候。如果"越强的模型越不忠实"这个趋势是成立的，那今天的模型在忠实性方面面临的挑战可能比当年更严峻，而不是更轻松。

当然，也有一种可能是，在更大规模的训练数据、更好的训练方法、以及专门针对忠实性进行优化的 RLHF 或者其他对齐技术的加持下，今天的模型在某些方面已经改善了忠实性问题。但这是一个开放的研究问题，没有定论。我个人的建议是：不要假设这个问题已经自动被解决了。在你没有针对具体任务验证之前，对 CoT 解释的信任应该是有条件的，而不是默认的。

最后，来总结一下今天这期节目。

Anthropic 在二〇二三年发表了这篇关于思维链忠实性的研究，核心发现是四个：第一，不同任务之间，模型对 CoT 的依赖程度差异很大，有些任务上 CoT 有真实的因果影响，有些任务上 CoT 几乎被完全忽略；第二，CoT 的性能提升不只是来自额外的生成计算量，内容本身携带了信息；第三，CoT 的效果对措辞变化有一定的鲁棒性，不依赖于特定的表达方式；第四，最反直觉也最值得警惕的发现是，越大越强的模型，在大多数任务上产生的 CoT 越不忠实。

这篇文章的价值不在于给出一个"CoT 可以用"或者"CoT 不能用"的简单结论。它的价值在于提醒我们，当我们把 CoT 当成一扇透视模型内心的窗口时，那扇窗子可能是磨砂的，甚至可能是一面贴着风景画的镜子。你以为你在看模型的内部，其实你看的是模型画给你看的图。

我个人认为，这是整个 AI 可解释性领域最根本的挑战之一。随着模型越来越强，我们与其说越来越了解它们，不如说在某些维度上越来越不了解它们。解决这个问题，需要的不是更好的 prompt，而是更深入的机制研究。
