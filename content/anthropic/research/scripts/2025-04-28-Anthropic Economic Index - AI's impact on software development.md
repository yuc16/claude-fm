## Script

今天聊的这篇文章，英文标题是 Anthropic Economic Index: AI's impact on software development，中文可以翻译成《Anthropic 经济指数：AI 对软件开发的影响》。这篇研究报告来自 Anthropic，它的分析数据来自二〇二五年四月六日到十三日这整整一周。我们今天录制这期节目的日期是二〇二六年六月十五日，距离这篇报告的发布已经过去了大约一年多的时间。

既然超过了半年，我必须先跟你聊一聊时效性的问题，这件事值得花一点时间说清楚。AI 领域的时间尺度非常特殊，一年相当于其他行业好几年的发展速度。这篇报告描述的是二〇二五年上半年的状态——那时候 Claude Code 刚刚正式推出没多久，AI coding agent 这个产品品类还处于最早期的商业化阶段。各种专用 AI 编程工具都还没有像今天这样普及，整个行业对"让 AI 直接干活"这件事到底能走多远，还存在很多不确定性。

所以这篇报告代表的是一个历史基准点：当 AI 编程工具刚刚从"聊天辅助"进化成"自主执行任务的 agent"的那个转折时刻，开发者实际上是怎么用它的，用来做什么，谁在用，用得有多深。这张快照的价值在于，它记录了一个趋势的起点。今天很多我们觉得"理所当然"的 AI 编程行为模式，在那个时候才刚刚开始显现。读这篇报告，能帮你理解这些模式的起源，也能让你更好地思考接下来还会怎么演变。好，背景交代完了，正式开始。

这篇报告是 Anthropic 经济指数系列研究的一部分。Anthropic 一直在做一件很有意思的事情：除了做模型、做产品，他们还在系统地研究 AI 对整个经济体系和劳动力市场的影响。这个系列之前发布过好几篇报告，有一个贯穿始终的发现——从事计算机相关工作的人，使用 Claude 的比例远远超出了他们在整体劳动力中的占比。

你可以用一个简单的比喻来理解这件事。假设全体美国劳动者里面，每一百个人里只有三四个人是程序员，那么按正常比例，Claude 的用户里程序员也应该只占百分之三四左右。但实际情况是，程序员在 Claude 用户里的比例要高得多，可能达到百分之三十甚至更高。这种"不成比例的使用"说明，软件开发是目前 AI 工具应用最深入、最广泛的职业领域之一。不只是工作中用，Anthropic 还发现，计算机科学相关的学位课程里 AI 使用率也显著偏高——因为学 CS 要大量写代码，学生自然而然就成了 AI 工具的重度用户。

正因为这种不成比例的使用，Anthropic 决定专门对软件开发这个场景做一次深入研究，看看开发者到底在怎么用 AI，用来做什么，谁在用。

他们分析了五十万次与编程相关的对话，来自两个平台。一个是 Claude.ai，就是大多数人用来跟 Claude 聊天的那个网页或者 app，属于通用聊天界面。另一个是 Claude Code，这是一个专门为软件开发设计的 AI agent，它可以直接在你的电脑上运行，读你的项目文件、写代码、跑测试、看报错、然后自己迭代修复，能够独立完成一整条开发任务链，不只是回答你的问题。分析方式是用 Anthropic 自己开发的隐私保护分析工具——它不直接读对话内容，而是把对话提炼成高层次的、匿名化的描述，比如"这次对话是关于 UI 组件开发的"，然后对这些描述进行统计分析。这个设计既保护了用户隐私，又能从海量对话里提取出有意义的规律。

好，现在进入第一个核心发现，也是我认为整篇报告里最重要的一组数字。

报告把所有对话分成两大类。第一类叫"自动化"，英文是 automation，意思是 AI 直接替你完成任务，你扮演的是一个"下指令然后验收结果"的角色。第二类叫"增强"，英文是 augmentation，意思是 AI 跟你协作，帮你提升效率，但你仍然是主导者，AI 是辅助工具。

结果出来了：在 Claude Code 上，百分之七十九的对话被归类为自动化；在 Claude.ai 上，这个比例是百分之四十九。

我需要你暂停一下，好好感受这个差距。同样是程序员在用 AI 做编程相关的工作，只是换了一个工具，AI 扮演的角色就差了整整三十个百分点。这不是小差异，这是根本性的使用模式差异。

研究者们进一步把自动化和增强各自细分成几个子类型，这里更有意思了。

自动化里面有两种主要模式。第一种叫"指令式"，英文是 Directive。这种模式里，用户告诉 AI 要做什么，AI 就直接去做，中间几乎不需要反复交互。"帮我写一个实现用户登录功能的组件"——AI 直接给你一段完整的代码，完工。在 Claude Code 上，指令式对话占了百分之四十三点八；在 Claude.ai 上是百分之二十七点五。

第二种叫"反馈回路"，英文是 Feedback Loop。这种模式里，AI 做了一部分工作，然后等用户给反馈——最典型的场景就是用户把报错信息复制粘贴回给 Claude，Claude 再根据报错继续修改，然后用户再试，再把结果反馈回来，如此循环。在 Claude Code 上，反馈回路占了百分之三十五点八；在 Claude.ai 上是百分之二十一点三，将近少了一半。

增强类别里，包括"学习型"——用户跟 AI 聊是为了学知识、理解概念——这些模式在 Claude Code 上的比例都明显低于 Claude.ai。道理很直接：用 Claude Code 的人主要目的是"完成任务"，而不是"学到什么东西"。这个定位从根本上决定了它的用户行为模式。

这组数据揭示了一个很重要的规律：不同的工具设计会塑造不同的使用行为。Claude Code 从一开始就被设计成让 AI 去"干活"的工具，所以用它的人自然而然就进入了"让 AI 做，我来审"的模式。而聊天界面的设计是对话，所以用户更倾向于用它来问问题、学东西、和 AI 一起探讨方案。同样的 AI 能力，不同的交互界面，就塑造出了完全不同的使用方式。

现在我要说那个非常值得注意的细节了。即便是在"自动化"这个大框架下，反馈回路模式依然非常普遍——在 Claude Code 上占了三分之一以上的对话。这意味着什么？这意味着在二〇二五年四月这个时间点，即便是最先进的 AI coding agent，也还没有达到"完全自主、完全不需要人看"的程度。人还在回路里。AI 做了大部分工作，但还需要人类来验证结果、告知报错、引导方向。

报告在展望部分对这一点保持了审慎的态度。它说，随着 AI 能力不断提升，这种"还需要人类介入"的模式可能会逐渐减少。更强大的 agent 需要的人工监督会越来越少。但它也没有断言这一定会发生，因为软件开发里存在大量的"隐性知识"——业务背景、团队约定、历史遗留约束——这些东西很难通过对话完整传递给 AI，可能会长期成为人类介入的理由。

我个人认为，"反馈回路"这种模式在可预见的未来不会消失，但它的形式会进化。不是从"粘贴报错信息"这种低层次操作，变成更高层次的"我来把控架构方向，AI 来执行所有实现细节"。人的角色会升级，但不会退出。

好，来看第二个核心发现：开发者在用 AI 做什么。

从编程语言的分布来看，结论非常鲜明。JavaScript 和 TypeScript 合计占了所有查询的百分之三十一，HTML 和 CSS 合计又占了百分之二十八。两者加起来，前端相关的语言一共占了将近六成的查询量。Python 排第三，占百分之十四，SQL 占百分之六。系统编程、运维、基础设施相关的语言，份额都相当有限。

这个数字分布让很多人有点意外。后端、数据库、服务器端不是技术含量更高的领域吗？为什么用 AI 最多的反而是写 HTML 和 CSS？

来拆解一下背后的逻辑。首先，写前端代码有一种特殊的痛苦：工作量巨大，但很多时候不需要深度思考。给一个卡片加圆角阴影、做响应式布局、实现表格排序功能——这些事情如果手写要耗很多时间，但逻辑其实挺机械的，有点像填格子。AI 处理这类任务有天然优势，因为它见过无数相似的代码，可以立刻给你一个可以用的版本，而且质量不错。

其次，前端的"验证成本"特别低。你让 AI 写了一段样式代码，刷新浏览器一秒钟就能看到结果对不对。如果不对，截个图或者描述一下哪里不对，让 AI 改。整个"试-验-改"的循环非常紧密，这非常适合反馈回路模式。相比之下，让 AI 写后端逻辑，你要验证对不对就复杂得多——要跑集成测试、要模拟各种数据场景、要检查边界条件——人类在中间介入的成本更高，所以更多人选择在后端任务上自己写，只在不确定的时候问一下 AI。

第三个原因，就是报告里特别提到的一个现象，叫做"vibe coding"，我把它翻译成"氛围编程"，也有人叫它"感觉编程"。

氛围编程的意思是，你不需要精确地告诉 AI 写什么代码，只需要用自然语言描述你想要的"感觉"或者"效果"，然后让 AI 去实现所有细节。"我想要一个看起来很高端的深色主题登录页，带毛玻璃背景效果，按钮有按压动画，整体感觉很现代"——就这样，AI 给你搞定 CSS 属性、动画参数、颜色配置这些细节。你不需要知道 backdrop-filter 怎么写，不需要记住 cubic-bezier 的参数，AI 替你操心这些。

氛围编程最初主要在两类人群里流行：一类是完全不会写代码的创业者，他们用 AI 来快速搭出产品原型；另一类是设计师，他们有很强的视觉感知但不熟悉代码语法。但随着 AI 能力增强，越来越多有经验的开发者也开始用这种方式工作，不是因为他们不懂代码，而是因为"描述效果让 AI 写第一稿、然后我来改"比"我自己从头写"要快得多。你把节省出来的时间放到更值得你脑力投入的地方。

报告里对此有一个推测，我觉得特别值得认真对待。它说，如果氛围编程继续向主流工作流渗透，那些专注于"做简单应用和用户界面"的开发岗位可能会比其他方向更早受到 AI 冲击。理由是这类工作的核心价值——把设计稿翻译成组件代码——正在被 AI 快速取代。这些开发者可能需要向更高层次转型，去做用户体验设计、产品逻辑把控、信息架构规划，而不是实现具体的按钮样式和交互细节。

我个人觉得这个方向判断是对的，但有一个地方需要补充。即便 AI 能生成一个组件的代码，让这个组件在真实项目里真正"好用"——符合团队的设计系统规范、通过无障碍访问标准、在各种屏幕尺寸和异常交互状态下都行为正确——这部分工作还需要有经验的工程师深度介入。AI 生成的是"第一稿"，而让第一稿成为"能上线的产品代码"，这中间的距离在复杂项目里依然不小。但这个距离在缩小，这是确定的趋势。

好，来看第三个核心发现：谁在使用这些工具。

报告尝试对用户进行分类，把对话归入不同的使用者类型：学生、个人项目开发者、创业公司从业者、企业员工、学术研究者，等等。研究团队自己非常诚实地说，这部分是最不确定的分析，因为他们只能从对话内容来推测用户的真实身份和工作背景，没有直接的用户数据。所以这部分结论要当成"有参考价值的估算"来看，而不是精确数字。

即便如此，模式还是非常清晰的。

创业公司是 Claude Code 的主要早期采用者。百分之三十三的 Claude Code 对话来自创业公司相关的工作，比这类用户在 Claude.ai 上的占比高了将近二十个百分点。相比之下，企业用户在 Claude Code 上的占比只有百分之二十三点八，甚至还略低于他们在 Claude.ai 上的份额。也就是说，企业用户使用通用聊天模式还可以，但切换到专用 agent 这种更高强度的使用方式上，他们明显落后了。

这个差距背后的原因不难理解。创业公司天然适合当早期采用者，因为他们的决策链条短。工程师觉得这个工具有用，跟技术负责人说一声，甚至自己直接就开始用了，不需要走什么审批流程。他们对工具的风险容忍度也更高——一个新工具就算不完美，能提升效率就值得用，出了问题就调整。而且创业公司在人力上永远是紧缺的，"让一个工程师的产出相当于两三个工程师"这种叙事，对人少事多的小团队吸引力是巨大的。

大公司的逻辑完全不同。他们需要考虑的问题多得多：这个工具会不会把我们的源代码发送到第三方服务器？AI 生成的代码在版权归属上有没有风险？怎么把这个工具整合进现有的代码审查规范和 CI/CD 流程？安全团队过没过审？法务有没有评估？这些问题每一个都需要专门的流程，都需要时间，而且这些顾虑是真实的、合理的，不是大公司的保守主义在作怪。结果就是，等所有流程走完、工具批下来，创业公司的早期采用者已经积累了一年的实战经验，建立起了相当的竞争优势。

报告里把这个现象类比为历史上的技术扩散模式。回忆一下云计算和 SaaS 软件的推广过程——同样是创业公司先冲，大公司几年后跟上。AI 工具可能也会走这条路，只是时间轴可能更压缩。AI 这种通用性极强的工具，如果真的能带来显著的生产力提升，早晚采用者之间的差距有可能比历史上任何一次技术切换都更大，因为它渗透的是核心的研发能力本身。

还有一个数字让我印象很深。学生、个人项目开发者、教程和学习类用户这几个群体，合计占了两个平台对话总量的将近一半。这说明 AI 编程工具不只是职场生产力工具，它在"学编程"这个场景里的渗透率也非常高。大量的人在用 Claude 来学习编程、做个人项目、完成课程作业。这批人今天是学生，五年后就是进入职场的工程师。他们从一开始就是在 AI 高度辅助的环境下学写代码的，他们的工作方式、对工具的期望、技能结构，都会跟我们这代人有根本性的不同。这会带来什么影响，现在其实还很难预测。

我想在这里补充一个报告自己也承认的重要局限。这次分析只包括了 Claude.ai 的免费和付费个人版，以及 Claude Code 的第一方 API 版本，没有包括企业版、团队版和 API 直接调用的数据。这意味着很多大公司内部的工具——基于 Claude API 构建的内部开发助手、代码审查工具、文档生成系统——都不在这五十万条数据里面。所以这篇报告对企业使用情况的描绘是有低估的，报告自己也明确说了这一点。这个局限性在读这篇报告的时候要记在心里。

现在来到报告里的"向前看"部分，这是我最想展开聊的地方，因为这里有几个问题是真正值得软件工程师认真思考的。

第一个问题：随着 AI 能力不断增强，现在这种"人类仍然在反馈回路里"的模式会一直持续下去吗？还是说人会被逐渐推出去？

这个问题没有简单的是或否。一方面，你可以说随着模型越来越强，它出错的概率越来越低，需要人类纠错的场合自然就减少了。如果 AI 第一次生成的代码就能直接跑通、直接满足需求，你根本不需要把报错粘贴回去。另一方面，也有观点认为，随着 AI 处理的任务变得更复杂——不只是写一个组件，而是设计整个系统的架构——人类的高层次判断反而变得更重要，只是介入的形式从"逐行审代码"变成了"审核架构决策和技术路线"。我个人倾向于认为，在简单、重复性高的任务上，人的介入会越来越少；在复杂、需要深度理解业务约束的任务上，人的角色会持续存在，只是形式会发生质变。

第二个问题：当 AI 能够处理更大规模的代码工程，开发者的角色会不会从"写代码"变成"管理和引导 AI 来写代码"？

这个转变已经在局部发生了，只是在不同复杂度的任务上发生的速度不一样。写一个按钮的交互效果，很多开发者已经在用"描述效果让 AI 写"这种模式了。设计一个新功能的数据接口，你可能还需要自己深度参与。搭一个全新系统的技术选型，AI 目前主要是给你候选方案，最终拍板的还是人。这个边界在不断上移。有一天，日常的开发工作可能真的主要是：给 AI 解释业务需求，设定技术优先级，审核 AI 的方案，验收结果，然后把可用版本交给产品团队——而不是每天面对 IDE 写几百行实现代码。这一天到来的速度，可能比很多人预期的要快。

第三个问题，也是报告里我觉得最有深意的推测：AI 编程能力的提升，会不会反过来加速 AI 自身的研发进度？

AI 的研究和开发本身就大量依赖软件工程——写模型训练代码、写评估脚本、写数据处理管道、写实验框架。如果 AI 能显著加速这些工作，那就意味着 AI 研发的速度本身会加快。而更快的 AI 研发会产生更强的模型，更强的模型又能进一步加速研发工作，循环往复。这是一个潜在的正向飞轮，自我强化的加速循环。到了二〇二六年的今天，我们已经能看到这个飞轮的轮廓越来越清晰。这也是为什么软件开发领域的 AI 进展，对整个 AI 行业的未来速度有如此关键的意义。

好，最后来聊聊这篇报告对你的实际工作意味着什么，有哪些可以马上去做的事情，以及要避开哪些坑。

第一件事，如果你还没有认真体验过专用的 AI coding agent，现在是时候了。不是聊天式地问问题，而是真正让 AI 去执行任务：给它一个有一定复杂度的任务，让它自己去读项目文件、写代码、跑测试。你需要亲身感受"自动化"和"增强"之间的区别，这种感受比读任何文章都真实。报告里数字上的差距——百分之七十九对百分之四十九——用文字描述的时候只是两个数字，但亲自用过之后，你会真正理解那三十个百分点意味着什么。

第二件事，对自己现在做的工作做一个诚实的评估。你每天写的代码里，有多少是"模式化的实现"，有多少是"真正需要深度理解业务的设计决策"？前者受 AI 冲击会更早、更快；后者的壁垒会相对持久。这不是让你悲观，而是让你提前知道应该把精力放在哪里积累。

第三件事，建立健康的 AI 使用习惯。AI 生成的代码，你要能看懂、能审、能为它负责。不要因为 AI 生成得快，就把一段你自己搞不清楚在干什么的代码直接推进生产环境。理解是你的责任，也是你相对于 AI 最核心的护城河。使用 AI 的正确姿势是：它提高了你的产出上限，而不是降低了你的能力门槛。

第四件事，如果你在创业公司，现在就是系统性建立 AI 工作流的最好时机。报告里说的那个早期采用者优势是真实存在的，而且这个优势还在累积。搞清楚你们团队哪些工作最适合用 AI 自动化，建立一套稳定的使用流程，积累实战经验。这种积累本身就是竞争优势，而且它会随着时间复利增长。

第五件事，如果你在大公司，不要因为公司流程还在走就完全放任自己等待。你可以在个人项目或者沙盒环境里先深度使用，把工具用熟。等公司流程走完、工具批下来的时候，你已经是内部专家了，而不是从零开始学。这种提前积累的成本很低，收益却可能很高。

最后要说的是，报告的结尾有一个很有意思的观察。它说软件开发可能是观察 AI 对职场影响的"领先指标"——也就是说，软件开发里现在发生的变化，可能预示着其他行业在几年后会经历的事情。如果这个判断是对的，那么软件工程师们处于一个独特的位置：你们既是这场深刻变化的最早亲历者，也是最有条件理解它、适应它的人。这既是挑战，也是优势。

好，我们来到了收尾的时候。

这篇《Anthropic 经济指数：AI 对软件开发的影响》是迄今为止基于真实使用数据、对 AI 编程工具使用模式记录最系统的研究之一。五十万次真实对话，不是问卷调查，不是专家预测，是真实的行为数据。它的三个核心结论值得记住：专用 coding agent 正在把更多的任务推向自动化，前端和用户界面是 AI 最先深度渗透的开发领域，创业公司正在这波变化里抢跑，传统企业还在追赶。

我个人认为这篇报告最大的价值，不在于它给了什么答案，而在于它帮我们把问题问得更精确了。不是"AI 会不会影响软件开发"这种宏观问题，而是"在哪些具体任务上、以什么速度、对哪类从业者影响最先到来"——这些才是值得每个工程师认真思考的问题。而这些问题，在二〇二六年的今天，我们已经开始看到更多的答案了。
