## Script

今天要聊的这篇文章，英文标题是 Project Fetch: Phase two，我把它翻译成"取球计划：第二阶段"。这篇文章发表于二〇二六年六月十八日，也就是昨天，来自 Anthropic 的前沿红队，作者是 Michael Ilie、C. Daniel Freeman 和 Kevin K. Troy，三位都是 Anthropic 专门负责能力评估和安全测试的研究员。

为什么我今天要把这篇刚出炉的文章拿出来讲？因为它用一个非常具体、非常接地气的实验，回答了一个我们很多人都在心里默默问着的问题：AI 模型到底进步了多少？这个问题不是通过某个 benchmark 分数来回答的，不是告诉你在考试卷上提高了几个百分点，而是通过一个真实的物理机器人任务，用秒表来计时，用双手来操作，给你一个直接的、让你感受得到的对比数字。

而这个数字，让我第一次读到的时候真的愣了一下。

要讲 Phase Two，我们必须先把 Phase One 的故事说清楚，因为整篇报告是建立在前一次实验的基础上的。时间回到二〇二四年八月，大概两年前。那时候，Anthropic 做了一个听起来有点像综艺节目的内部实验。他们把一批自家员工拉到一个仓库里，这些人不是机器人专家，是普通的工程师和研究员，然后给他们一只机器狗。

机器狗这个东西，你可能在短视频里见过。就是那种四条腿、能跑能跳、有点像科幻电影里道具的机械宠物，学术上叫四足仿生机器人，英文叫 robotic quadruped。波士顿动力公司的 Spot 就是这类产品里最有名的。这种机器人通常搭载了摄像头、激光雷达也就是 lidar 等多种传感器，能够感知周围环境，也能通过编程接口来被软件控制。

Phase One 的规则很简单：把这群员工随机分成两队。一队可以使用当时 Anthropic 最新的模型 Claude Opus 四点一，另一队什么模型都不能用，只能靠自己，靠网上搜索和自己的脑子。然后给他们布置一系列递进的任务，从简单到复杂一步一步来。

第一步，用厂商提供的遥控手柄操控机器狗走动，感受一下它的基本操控。第二步，连接机器狗上面装载的传感器，包括摄像头视频流和激光雷达的空间数据，让电脑能够读取这些信号。第三步，不用手柄了，写代码，通过软件编程接口来手动控制机器狗运动。第四步，开发一套监测系统，能够追踪机器狗在三维空间里走的路径。第五步，写一个计算机视觉程序，能够自动检测出画面里那个橙色的海滩球在哪里。最后，第六步，把以上所有模块整合起来，让机器狗能够全自主地去寻找球、追上球，把球推回到起点那片假草坪上。

结果是什么？有 Claude 帮助的队伍完成得更多，速度更快。这个结果在意料之中，毕竟 Claude 那时候在很多编程和问题解决任务上已经很强了。

但这里有一个最关键的前情提要：在正式实验开始之前，研究人员做了一个预测试，直接让 Claude Opus 四点一自己独立去完成这些任务，完全不带任何人类协助。结果非常惨。模型在最早期的"连接机器狗传感器"这一步就卡住了，根本没法推进到后面的任务。就连那支没有任何 AI 帮助的人类队伍，都比独自运作的 Claude 做得好。

所以 Phase One 的结论可以用一句话来说：Claude 能够显著提升人类的效率，但还没有能力独立完成这类需要专业知识和动手操作的任务。这是一个典型的人机协作阶段，人还是主导，模型是放大器。

而这一次，Phase Two 要回答的问题是：才过了不到一年，这个局面有没有发生变化？

Phase Two 的实验设置有一个根本性的改变：研究人员这次不再让 Claude 扮演助手角色，而是让它直接当主角。他们用的模型是 Claude Opus 四点七，在 Claude Code 这个代理编程工具里运行。Claude Code 是 Anthropic 推出的一款可以在终端里操作的 AI 编程代理，它能够读写文件、运行代码、调用系统命令，理论上可以端到端完成整个软件开发任务，不需要人类在中间手把手地操作。

实验里把模型的思考模式设置为自适应思考，计算力投入调到最大值。这意味着模型在解决问题时，会自主决定要在某个问题上思考多久、想多深，遇到复杂情况会充分推理再行动，而不是急于给出第一个想到的答案。

人类研究员在这个实验里的角色被削减到了极低限度：把笔记本电脑插到机器狗上，输入初始的提示词，然后批准模型要执行的命令，以及在模型完成一个阶段后批准它进入下一个任务。仅此而已。没有任何技术指导，没有告诉模型"这个方向走不通换一个"，没有帮它纠正错误，就是一个审批按钮的人肉执行者。

有几个任务做了相应的调整。因为 Claude 没有物理的手，所以遥控手柄操控那一步没法直接测。还有一个重要的点是，用模型编写的程序让人类研究员实际操控机器狗取球这件事，研究人员确认了程序是能运行的，但这部分的时间没有被纳入计时，因为那取决于人类操控者的技术。

对于可以测的任务，研究人员做了三次独立试验来测量每个环节的时间，并且对模型的成功程度做了定性评估。三次重复试验的设计让结果更可信，因为你可以看到稳定性，而不是单次偶然好运的数据。

好，现在来看那组让人愣住的数字。

结论是：在 Phase One 实验里，至少有一支人类队伍完成过的每一个任务上，Opus 四点七完成同样任务的速度，最低都是人类的十倍。

你没听错，最低都是十倍。

更细的对比数字是：如果只看两支人类队伍都完成了的那四个核心任务，Opus 四点七比没有 Claude 帮助的那支队伍平均快了三十七倍，比有 Claude 加持的那支队伍也快了整整十八倍。文章标题所说的"大约快二十倍"，是所有任务的综合平均。

我来帮你感受一下这个差距。假设一支人类队伍平均花了四个小时完成某个任务，Opus 四点七大概只需要六七分钟。这不是工具优化意义上的效率提升，这是数量级上的跨越。你可以想象一下，如果你的团队做了整整一天的事情，一个 AI 代理二十分钟搞定，这会对你的工作流程、项目计划、团队结构产生什么样的影响？

代码量的对比同样让人印象深刻。Opus 四点七不只是完成得快，写的代码还少得多。具体来说，它生成的代码量大约是有 Claude 协助的那支人类队伍的十分之一，但完成的效果却持平甚至更好。这个数字背后的含义是：Opus 四点七在解题的时候，找到了更直接、更精准的路径，没有走弯路。人类在摸索过程中产生了大量试探性的代码，不断推翻重来，废弃了许多失败的尝试；而 Opus 四点七能够更快地收敛到正确方向，然后一击即中。

文章里提到了一个很典型的例子：面对"如何连接机器狗传感器"这个问题，人类团队会纠结于多种不同的技术方案，拿不定主意，来回比较，浪费了大量时间。而 Opus 四点七能够迅速判断哪条路最优，直接走过去。而且模型第一次写的代码很多都能直接跑起来，不需要反复调试。这对于快速推进任务来说，是非常关键的优势。

当然，模型也不是完美的，研究人员没有回避这一点。他们观察到一个失误：在物体检测环节，Opus 四点七默认选择了一个过时的检测算法，而不是更现代的方案。不过即使选了次优的算法，模型仍然能够识别问题、想办法绕过去、最终完成任务。唯一的代价是其中一次海滩球检测试验比另外两次花了更多时间。研究人员认为这个时间偏差，就是那次错误的算法选择导致的。

整体来看，对于它能力范围内的任务，Opus 四点七表现相当稳定。三次试验的时间差距很小，说明这不是偶然的好运，而是真正的、可重复的能力。文章原文的表述是"within its capability envelope, Claude is now quite reliable"，翻译过来就是在能力包络内，Claude 现在相当可靠。

好，讲完了令人印象深刻的部分，我们来看看模型还做不好什么。这一块同样重要，因为诚实的边界划定，比夸大的结论更有价值。

最核心的局限是：让机器狗精确地把球推回到起点，这个任务 Opus 四点七现在还做不好。

我来解释一下为什么这个任务难。在 Phase One 实验里，练习了一段时间之后，人类志愿者可以用手柄来控制机器狗，轻轻地、精准地把球从球场一端推回到起始点那片假草坪上。听起来简单，但背后有一个非常复杂的实时认知过程：你需要实时看到球现在在哪里，判断球的运动方向有没有偏，分析偏差跟你上一个操作动作之间的关系是什么，然后快速决定下一步操作要怎么调整，整个过程是一个以眼睛和双手为媒介的高速闭环。

做过控制系统的同学对"闭环控制"这个概念一定不陌生。简单解释一下：开环控制就是你发出一个指令，不管结果怎样，你都不去管；闭环控制则是你行动，观察结果，根据结果调整下一步行动，如此循环，不断往目标收敛。你开车的时候一直在做的事情就是闭环控制——你看到车偏向路边了，马上微调方向盘，你的眼睛和手形成了一个实时的反馈回路，可能每秒钟要经过好几次这样的循环。人类的神经系统对这类任务经过了数百万年的进化优化，只要练习一下，大部分人都能做得很好。

Opus 四点七在这里遇到了麻烦。它确实能让机器狗移动到球的后面，摆好位置，准备推球。但实际执行推球的过程控制太差，推出去经常跑偏，没能成功把球稳定地推回起点。文章的原文措辞是"efforts to do so were poorly controlled"，控制太差。这个表现，和 Phase One 里那些刚开始尝试写自动取球程序的人类参与者差不多，都没能成功完成这个最终任务。

这里有一个值得注意的细节：一位具有更丰富机器人实际经验的研究员，他没有参与过 Phase One，用更多时间、加了额外的辅助框架和脚手架结构，最终成功完成了自动取球的编程任务。研究团队因此认为，如果给当前的 Claude 模型更多时间和更完善的辅助结构，很可能也能做到这一点。但他们强调，他们真正想看到的，不只是"理论上能做到"，而是"像完成其他任务一样快速且可靠地做到"——那才是真正意义上的能力突破。

我个人认为，精确的物理闭环控制对于语言模型来说，是一个结构性的挑战，不只是能力程度高低的问题。闭环控制需要的是高频率的感知-行动循环，你需要在很短的时间窗口里持续地读取传感器数据、做出判断、发出控制指令，然后马上看结果。而语言模型天然是一个"思考-输出"的顺序处理结构，它不是为实时高频的反馈控制设计的。要解决这个问题，可能不只需要更强的语言模型，还需要专门设计的感知-行动集成架构，更接近于机器人控制专用系统的那种设计方式。这是一个值得继续关注的开放性问题。

讲完了结果和局限，我想花一些时间在这篇文章里我认为最有深度的一个观察上。

研究人员在文章里写到，从他们在网络安全领域的研究中，他们已经看到过同样的能力进化模式，现在在机器人领域也看到了类似的轨迹。这个模式分三个阶段：第一个阶段，模型对人类有帮助，人类的能力被放大，但人类仍然是主导；第二个阶段，人类对模型有帮助，模型开始接近某些任务的能力边界，但需要人类来提供指导和纠正；第三个阶段，模型基本上可以独立完成任务了，不再需要专家级别的人类手把手介入。

这个三阶段模式，如果你把它叠加到 AI 编程助手的发展历史上，你会发现惊人的吻合。最初，GitHub Copilot 出来，能帮你自动补全几行代码，你仍然完全主导整个开发过程；后来，Claude 系列模型能够写整个函数、整个类，你开始越来越依赖它，偶尔需要纠正它走偏的方向；再后来，Claude Code 这类代理工具能够端到端完成整个功能的开发，现在很多工程师已经更多在扮演"需求描述者"和"质量审查员"的角色，而不是逐行写代码的人了。

在机器人领域，Phase One 对应的是第一阶段，模型帮助人类；Phase Two 对应的是从第二阶段过渡到第三阶段，在大多数中间任务上模型已经基本独立了，只有最难的精确控制任务还在第二阶段。

研究团队还特别点出了一个很重要的事实：这些进步不是 Anthropic 专门针对机器人能力做了什么定向优化训练的结果，而是来自更通用的大规模模型能力提升，也就是所谓的 scaling，把模型训练得更大更深更全面。也就是说，机器人任务上的能力飞跃，只是通用能力提升的一个副产品，研究人员在这个方向上并没有做专门的努力。这让这个发现更加耐人寻味——如果通用 scaling 就能带来这么大的跨越，再过一年两年，会出现什么？

文章最后有一个很有力量的比喻。作者说，AI 模型在转向更具代理性的编程工作时，开始使用了现有的软件编辑工具，比如字符串替换这类操作——这曾经让很多人觉得神奇，现在已经习以为常了。而现在，我们很可能正在进入一个类似的转折点，只不过不是在软件世界，而是在物理世界。作者的原话是"the early era of physical agentic AI"，物理代理型 AI 的早期时代。

好，现在我们来聊最务实的部分：这件事对你，一个在第一线做开发的工程师，到底意味着什么，能上手做什么，要注意避什么坑。

先说一个最直接的结论：如果你现在正在做任何涉及机器人、工业控制设备、物联网硬件，或者任何需要跟物理设备打交道的项目，你应该认真把大语言模型纳入你的工具箱，而不是把它当作一个只能写业务逻辑的软件工具。

过去，跟机器人或者自动化设备打交道，需要非常专业的技术背景。你需要了解 ROS，也就是机器人操作系统，需要懂传感器融合，需要熟悉运动学和动力学，需要知道各种通信协议，这些都是门槛很高的专业知识。一个 Web 工程师或者数据工程师走进这个领域，面对的是一堵很高的墙。而这篇文章展示的是，一个没有机器人专业背景的普通工程师，在 Claude Opus 四点七的帮助下，可以以极低的门槛完成传感器数据接入、控制程序编写、计算机视觉集成这些过去需要专家才能完成的任务。门槛变低，意味着你能做的事情变多了。

那么具体可以怎么用起来？

第一点，做快速原型不要再从零手写了。如果你需要给某个硬件设备写一个控制程序，先让 Claude Code 帮你生成一个框架，你在这个框架上迭代和调整。根据这篇文章的结果，Opus 四点七第一次写的代码很多直接就能运行，不需要大量调试，这在原型验证阶段能节省大量时间。你可以用这些节省下来的时间做更高层面的设计决策，或者更快地从原型推进到可测试的版本。

第二点，传感器数据处理是一个特别适合让 AI 来介入的环节。不管是摄像头视频流的物体检测，还是激光雷达点云数据的解析，还是各种工业传感器信号的读取和格式转换，这类任务虽然有技术门槛，但已经有大量成熟的开源库，模型能够非常快速地找到合适的工具并把它们组合成可以运行的程序，不需要你先花几周时间研究各种库的 API 文档。

第三点，调试和问题排查可以让模型帮你做。机器人系统或者嵌入式系统的调试往往很麻烦，问题可能出在硬件层面也可能出在软件层面，症状和根本原因之间的关系很不直观。你可以把错误日志、传感器原始数据、代码一起提供给模型，让它帮你分析，往往比你自己盯着看快很多。

第四点，读设备文档和整理接口文档可以让模型帮你做。机器人厂商和工业设备厂商的文档质量通常参差不齐，有时候写得非常难懂，API 设计也不直观。让模型帮你读文档、整理关键接口、生成带注释的使用示例，可以省去大量啃文档的时间，让你更快进入实际编码阶段。

说完可以做什么，我必须认真讲几个坑，这些是实际操作中非常容易踩到的。

第一个坑，精确控制场景不要过度依赖模型。这篇文章已经非常明确地展示了，在需要实时反馈调整的闭环控制任务上，当前的语言模型还不够好。如果你的应用场景需要很高的精度，或者需要很快的响应速度，大模型现在还不是合适的主控工具。对于这类任务，传统的控制算法比如 PID 控制器，或者专门训练的强化学习策略，仍然是更可靠的选择。语言模型更适合做上层的任务规划、决策逻辑，和代码生成，而不是做底层的实时控制回路。

第二个坑，安全测试不能省，而且必须是你主动做，不能等模型自己来保证。模型生成的代码可能会让机器人或者自动化设备做出你没有预期的动作。文章里提到了一个让人印象深刻的细节：在 Phase One 实验里，一支人类队伍差点被失控乱跑的机器狗撞到。如果是 AI 模型在控制机器人，这种情况同样可能发生，甚至因为 AI 的行为有时候难以事先预测，风险可能更高。在把模型生成的代码部署到真实硬件之前，你必须有充分的沙盒测试环境、速度上限设置、紧急停止机制、碰撞检测。这些工程安全保障，需要你在系统架构层面显式地加进去，不能指望模型自己来负责。

第三个坑，不要盲目信任模型的技术选型。这篇文章里那个"默认选了一个过时的物体检测算法"的例子，是一个非常典型的提醒。模型的训练数据有截止时间，可能不知道你所在技术领域最新最好的工具或方案。在你的应用场景里，最适合的库和方案可能是最近半年内才成熟的，而模型没有见过。在关键的技术选型上，你需要自己做判断，或者在提示词里明确告诉模型使用哪个具体版本、哪个具体的库。别把架构决策完全交给模型来做。

第四个坑，不要把这篇报告的结论无限泛化到你的具体场景。这篇报告测的是特定的模型、特定的任务类型、特定的硬件设备，在一个受控的实验环境里做的三次试验。你的项目可能涉及不同类型的机器人、不同的任务需求、不同的环境条件和干扰因素，模型的表现可能跟这里报告的数字差别很大。在做大规模的技术路线投入之前，先做一个小规模的验证实验，用你自己的实际硬件和任务来摸清楚模型的能力边界在哪里，再根据结果决定投入多少。

最后我想分享一个更宏观的视角，这也是我认为这篇报告对你最有价值的地方。

AI 能力扩展的方向，正在从纯软件世界向物理世界蔓延。过去两三年，我们目睹了 AI 在代码生成、文本分析、内容创作这些纯软件任务上的能力爆炸式增长。很多人可能没有意识到，这种能力正在开始触及"操控物理设备"这个更硬核的领域，而且进展速度比很多人预期的要快。

对于做嵌入式系统开发的工程师、做工业自动化的工程师、做机器人系统集成的工程师，这个信号值得高度重视。不是危机感，而是机遇感——你的工作方式在未来几年里可能会发生根本性的变化。就像今天的软件工程师不是被 AI 取代了，而是开始用 AI 来极大放大自己的产出，物理世界的工程师也会经历类似的转型。那些早早开始探索和掌握这些工具的人，会在生产力上建立起越来越难以追赶的优势。

而且有一点非常值得记住：这篇文章里的 Opus 四点七不是专门为机器人优化的模型，它的能力只是通用能力提升的副产品。这意味着，即使 Anthropic 没有专门做机器人方向的研究，随着通用模型的持续提升，你的工作领域也会持续受到冲击。这不是某个特定方向的技术进步，而是一个全面弥漫的底层能力浪潮。

如果你现在还没有认真探索 Claude Code 或者类似的代理工具在你具体工作场景里能做什么，这篇报告是一个很好的提醒，现在开始正是时候。从你最熟悉的那一块领域入手，找一个真实的有价值的小任务，试着让模型来做，认真感受一下它在哪里很强、在哪里会犯错。这种第一手的经验，比读任何评测报告都更有参考价值，因为它告诉你的是你自己场景里的边界，而不是别人的实验室数据。

好，今天这期就到这里。

这篇发表于二〇二六年六月十八日的 Project Fetch: Phase two，用一个严格计时的真实物理实验告诉我们：在不到一年的时间里，大模型在机器人控制任务上的能力，从"需要人类手把手指导"跨越到了"独立完成、速度快十到三十七倍"。它在精确的物理闭环控制上还有明显的局限，这是一个尚未跨越的门槛，但这个进化的速度本身，已经足够让每个人重新审视自己所在领域的未来。

我个人认为，这篇报告里最值得记住的不是那组令人眼花的速度数字，而是那个三阶段的能力进化规律：先是模型帮人，然后人帮模型，最终模型独立完成。这个规律在网络安全领域已经应验，在软件编程领域已经应验，现在在机器人领域开始应验。下一个应验的领域会是哪里，我觉得是值得我们每个人认真去想的问题。
