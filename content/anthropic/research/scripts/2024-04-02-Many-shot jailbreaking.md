先说一句时效提示：这篇文章发表到今天，也就是二零二六年六月，已经过去了将近两年三个月。放在 AI 这个领域，两年是一段相当长的时间。所以在正式进入内容之前，我想花点时间帮你还原当时的背景，让你知道这篇文章诞生的语境。

二零二四年三月，那是一个什么样的时间点？GPT 四代发布将近一年，OpenAI 还没有推出 o 系列推理模型，Anthropic 的 Claude 三家族刚刚亮相，整个行业正处于对「长上下文窗口」能力的狂热追捧之中。大家都在说：我的模型现在能处理一百万 token 了！工程师们兴奋地把整个代码仓库、整本合同、整本书塞进去让模型分析，感觉无所不能。

就是在这个氛围里，Anthropic 的安全研究团队泼了一盆冷水。他们发现，更长的上下文窗口，在带来巨大便利的同时，也打开了一扇之前不存在的攻击大门。这篇文章值得你在今天回头看的原因有几个：它揭示的攻击机制在两年后依然没有被根本解决；它对模型工作原理的剖析对理解今天的 AI 系统仍然适用；而对于我们这些在日常工作中使用 AI 或者构建 AI 应用的工程师来说，理解这类攻击的本质，是做好安全防护的第一步。

好，让我们正式进入文章内容。

先讲「越狱」这个概念。越狱，英文叫 jailbreaking，指的是用各种方法绕过大语言模型的安全限制，让它说出或做出本来应该拒绝的事情。比如，你直接问一个训练好的模型怎么制造危险物品，它会拒绝，说这涉及危险或违法活动。越狱，就是想办法让它绕过这道限制。

越狱方法有很多种，有些是在措辞上耍花招，比如假装在写小说、让模型扮演一个「没有道德限制的 AI」之类。这些早期方法，行业已经研究了很多年，模型也对它们建立了相应的抵抗力。但 Anthropic 这篇文章描述的是一种截然不同的越狱方式，它的核心不是在措辞上做手脚，而是利用上下文窗口的巨大容量来实现暴力突破。

这种方法叫 many-shot jailbreaking，直译是多轮示例越狱。shot 这个词在机器学习里有特定含义，指的是一个「示例」。你可能听过 few-shot learning，就是少样本学习，意思是用几个示例来引导模型完成任务。many-shot 就是非常多的示例，非常多轮。

那具体怎么操作呢？我来讲得很具体。

攻击者在真正想问的危险问题之前，先在 prompt 里塞进去大量的伪造对话。这些伪造对话看起来像是一个正常用户和一个 AI 助手在聊天，但在这些对话里，那个假冒的 AI 助手非常配合地回答了各种危险或有害的问题，完全没有任何道德顾虑。

文章里给了一个具体例子。假设攻击者想知道怎么制造危险物品，他不直接问，而是先构造这样一段伪造对话：用户问，怎么开锁？假冒的 AI 回答，我很乐意帮忙，首先你需要获得专业开锁工具，然后详细介绍整个流程。这是一轮，一个 shot。然后攻击者继续加，另一轮：用户问一个其他敏感问题，假冒的 AI 继续毫无保留地回答。就这样一轮接一轮，在真正的目标问题出现之前，攻击者在 prompt 里塞进了几十轮甚至几百轮这样的伪造对话。

Anthropic 的研究团队在实验里最多测试到了两百五十六轮。

在只有少量示例的情况下，比如一两轮或者几轮，模型的安全训练还是能发挥作用，它看到最后那个真正危险的问题，依然会拒绝。但随着伪造对话的数量超过某个临界点之后，模型的行为就发生了根本性的转变。它开始回答本来应该拒绝的问题，就好像之前那几百轮示范，把它彻底重塑成了另一种行为模式的 AI。

研究团队把这个关系画成了一张图：横轴是 shot 的数量，纵轴是模型产生有害回答的概率。随着 shot 增加，有害回答的概率逐渐上升。整体趋势符合一种叫做「幂律」的统计规律。这个数学名词你不必纠结，直白说就是：示例越多，越狱成功的可能性越高，而且这个增长是有规律可循的、可以被量化和预测的。

还有一个让研究者担忧的发现：如果把多轮示例越狱和之前已经发表过的其他越狱技术组合起来用，效果会更强。也就是说，需要的伪造示例数量更少，prompt 更短，攻击的实施成本更低。攻击者可以叠加各种手法，降低发起攻击的门槛。

好，讲完了「是什么」，我们来聊最有意思的问题：为什么多轮示例越狱会有效？研究团队给出了一个相当深刻的解释，理解这个解释，对你理解大语言模型的工作方式非常有价值。

关键词是 in-context learning，上下文内学习，有时候也叫情境学习。

什么是 in-context learning？这是大语言模型最核心的能力之一。简单说，它指的是模型仅仅通过阅读 prompt 里给出的示例，就能学会如何完成某个任务，完全不需要重新训练或者修改模型的参数。

我来举一个具体的例子。假设你想让 LLM 帮你做情感分析，判断一段话是正面情绪还是负面情绪。一种方法是收集大量标注数据，重新训练模型。但 in-context learning 告诉你，根本不需要这么麻烦。你只需要在 prompt 里写几行示例：「这部电影太棒了」，正面；「这个餐厅的食物难以下咽」，负面；「服务员态度极差」，负面。然后直接问「今天天气真好，心情也不错」是什么情绪，模型就会自动从这几个示例里学到规律，输出「正面」。

这个能力极其强大，也是 LLM 如此灵活好用的根本原因之一。通常来说，示例越多，模型在这个任务上的表现越好。这就是 few-shot learning 和 many-shot learning 的基本原理。

现在你明白多轮示例越狱为什么有效了吗？

当攻击者在 prompt 里塞进了两百多轮「AI 毫无顾虑地回答危险问题」的伪造对话，本质上是在利用 in-context learning，在实时地、临时地对模型施加影响。模型读完这两百多轮示例，in-context learning 就开始发挥作用，它「学到」了：在这个对话场景里，助手就是应该什么都回答的，包括危险的那些。这个临时学到的行为模式，压过了训练时注入的安全倾向。

Anthropic 的研究团队做了一个非常精妙的对比实验来验证这个假说。他们分别画出了两张图，放在一起比较。左边那张，显示的是多轮示例越狱随着 shot 数量增加时攻击成功率的变化曲线。右边那张，显示的是一批完全正常、无害的 in-context learning 任务，随着示例数量增加时性能提升的曲线。结果出来之后，两张图呈现出了惊人相似的统计规律，都是同一种幂律形态。

这个实验设计很漂亮，因为它把两件看起来完全不同的事情——恶意的越狱攻击和正当的任务学习——放在同一个统计框架下比较，发现它们遵循同样的规律。这有力地支持了一个结论：多轮示例越狱并不是利用了什么特殊的奇怪漏洞，而是在利用模型最基础、最核心的学习机制本身。

这个认识非常重要，因为它意味着你没有办法通过简单地「关掉某个功能」来防御这类攻击。in-context learning 不是一个可以随意开关的功能，它渗透在模型整个工作方式里。你需要找到更精妙的方式来应对这个问题。

接下来还有一个让研究者更加头疼的发现：越大的模型，对这种攻击越脆弱。

这听起来有点反直觉。更大的模型不是更聪明吗？为什么反而更容易被攻破？

逻辑其实是自洽的。更大的模型，in-context learning 的能力往往更强，它更善于从 prompt 里的示例中快速提取规律、调整自己的行为。这本来是一种优势，让大模型在各种任务上表现更好。但面对多轮示例越狱的时候，这个优势就变成了劣势：它更容易、更快速地从那些恶意示例中「学到」不该学的行为模式。用研究团队的实验数据来说，攻击大模型时，只需要更少的 shot，就能达到同样的越狱效果。

而且这个问题的严重性不仅仅是「大模型更容易被攻破」，还在于：越大越强的模型，通常也是潜在危害最大的模型，因为它们能做到的事情更多、影响力更大。这就形成了一个研究者非常头疼的困境：最需要安全保障的模型，偏偏对这类攻击最脆弱。

研究团队在文章里也明确说了：虽然当前的模型还没有到会造成真正灾难性风险的程度，但未来的模型可能会。所以趁现在就要认真研究和解决这类安全问题，等模型更强大了再来处理，代价可能无法承受。这个判断放在今天，也就是二零二六年，依然成立，而且随着模型能力的持续提升，变得越来越紧迫。

好，发现了问题，当然要想办法解决。研究团队探索了几种缓解策略，效果有好有坏，值得细说。

第一种最直觉的方法：限制上下文窗口的长度。既然攻击依赖长上下文，那干脆把输入长度卡住，让攻击无法塞进足够多的示例。这个方法从技术角度完全能奏效，但代价太大——长上下文窗口正是用户最需要的能力之一，砍掉等于因噎废食。这个方向被研究团队否定了，他们说，不希望用阻止用户获益来换取安全。

第二种方法：通过微调来增强模型的抵抗力。微调，fine-tuning，就是拿额外的数据对模型进行专项训练，让它学会识别并拒绝看起来像是多轮示例越狱格式的输入。这个方法有一定效果，但实验结果令人失望。微调之后，模型确实需要更多的 shot 才能被成功越狱，也就是说攻击的门槛提高了。但只要示例数量足够多，越狱最终还是会成功。就好像你把门锁换成了更结实的，但门本身的材料还是一样，给足够的力气还是能踢开。研究团队的原话是，微调只是「延迟」了越狱的发生，而不是「阻止」了它。

第三种方法，也是目前效果最好的方向：在 prompt 进入模型之前，先对它进行分类和修改。这类方法的思路是，与其试图让模型本身对攻击免疫，不如在输入到达模型之前就先一步做检测和处理，识别出看起来像是越狱攻击的内容，然后对这个输入进行干预——比如添加额外的说明上下文，或者改变输入的格式和结构。

研究团队在论文里报告了一个非常显著的结果：使用某种基于 prompt 的缓解技术之后，在一个测试场景里，攻击成功率从百分之六十一直接降到了百分之二。从六十一降到二，这个降幅相当惊人。当然，研究团队也坦诚地说，他们仍在研究这些方法的各种权衡，因为过于激进的过滤会误伤正常使用，而攻击者也可能开发出新的变体来绕过检测。这场博弈还没有在这篇文章里得到最终解答。

顺便提一句值得肯定的做法：Anthropic 在发表这篇文章之前，已经提前把技术细节分享给了业界的其他 AI 公司和学术研究者。他们在文章里明确说，选择公开发表这篇研究的原因之一，是希望推动整个行业共同面对这类问题，而不是各自藏着自己发现的漏洞。他们还说，这种攻击方式本身并不复杂，早晚会被人独立发现，与其等别人发现，不如自己抢先公开，推动解决方案的研究。这种安全漏洞的负责任披露，是 AI 行业非常需要建立和强化的文化。

好，核心内容讲完了，我们来聊聊这对你的实际工作意味着什么。

如果你是一个在生产环境里部署 LLM 应用的工程师，或者你正在为某个产品接入 AI 能力，这篇文章有几个具体的启示值得你认真对待。

第一个启示：不要过度依赖模型自带的安全训练。这篇文章清楚地证明了，即使是经过专业安全训练的大模型，在面对精心构造的攻击时也会失效。如果你的应用有任何安全敏感的场景，不能把「用了一个安全的基座模型」当作终点。安全需要分层防御，不是选对了模型就一劳永逸的。

第二个启示：对用户输入的长度要有意识。不是说要完全限制上下文，但对于你具体的应用场景，用户真的需要提交几十万 token 的输入吗？如果不需要，设置一个合理的上限是有意义的。这既能降低被多轮示例越狱攻击的风险，也能控制你的 API 调用成本。对很多应用来说，合理的输入限制几乎没有副作用，却能提供一道有效的防线。

第三个启示：在 prompt 进入模型之前，加一层检测。这是文章里发现最有效的防御方向。具体来说，你可以检测用户输入里是否出现了大量的「User:」和「Assistant:」交替标记，这是多轮示例越狱最典型的格式特征。你也可以检测输入里是否存在高度重复、看起来像是批量填充的结构。这些都可以作为预警信号，触发额外的审核或者拒绝处理。现在市面上也有一些专门的 prompt 安全检测服务，可以作为这一层的工具。

第四个启示：对输出内容同样要有监控。除了在输入端做防御，对模型的输出也应该有检测机制。如果模型突然产生明显违反你应用场景规范的内容，这个异常应该能被捕捉到，而不是直接推送给用户。在应用层做输出过滤，是安全防御体系的最后一道闸，和输入端检测相互配合，才构成完整的防护。

第五个启示，也是我个人认为最重要的一条：持续关注这个领域的动态。AI 安全的技术和攻击手法都在快速演进。多轮示例越狱是二零二四年公开的技术，到了二零二六年，攻击手段肯定已经有了更多的变体和演进。定期跟进这个领域的最新研究，是每个认真构建 AI 应用的工程师都应该做的功课，而不是只在出事后才去了解。

我个人认为，这篇文章最深刻的贡献，不仅仅是描述了一种攻击手法，而是揭示了一个更根本的矛盾。

这个矛盾是：in-context learning 是大语言模型最有价值的能力之一，也是它最根本的安全弱点之一。我们给模型写 prompt、放示例，本质上就是在「教」模型做事。但我们没有完美的方法来区分「正当的示范」和「恶意的示范」，因为从模型的角度看，两者在形式上是一样的，都是 prompt 里的文字。模型会照单全收地从中学习。

这不仅是一个工程层面的安全问题，也触及了 AI 对齐研究的核心挑战。所谓对齐，alignment，就是让 AI 真正按照人类的价值观和意愿行事，而不只是表面上看起来如此。多轮示例越狱清楚地说明了，模型的「价值倾向」是可以被 prompt 里的内容临时改变的，而且这种临时改变可以压过训练时花了巨大资源注入的安全倾向。这意味着「训练时把模型教好」和「推理时模型就会好好表现」之间，存在一道值得认真对待的鸿沟。

你可以从另一个角度来理解这个矛盾：我们让模型处理越来越长的输入，做越来越复杂的任务，模型读到的上下文信息对它的实际行为影响越来越大。在这个背景下，单纯依靠训练时注入的安全偏好，已经被证明是不够的。我们需要在应用层、在系统设计层，持续投入来补上这道防线。

好了，今天这一期节目差不多到这里了。让我用几句话总结一下。

Anthropic 在二零二四年三月发表的这篇研究，揭示了一种叫做多轮示例越狱的攻击方式：通过在 prompt 里塞入大量伪造的有害问答对话作为示例，可以利用大语言模型的 in-context learning 能力，临时覆盖它的安全训练，让它回答本来应该拒绝的危险问题。示例越多攻击越有效，模型越大越脆弱。最有效的防御方向是在 prompt 进入模型之前进行分类和干预，可以将攻击成功率从百分之六十一降到百分之二，但这场博弈仍在继续，没有一劳永逸的解法。

我个人的点评是：这篇文章是 Anthropic 主动披露安全漏洞的一个好例子，体现了他们所说的「负责任的 AI 发展」并不只是一句口号。对于我们这些使用和构建 AI 系统的人来说，这篇文章是一个很好的提醒：AI 的安全不是买一个别人做好的模型就能搞定的事情，它需要在应用层面持续地关注、监控和防护，而理解攻击的本质原理，是做好这件事的第一步。
