## Script

今天聊的这篇文章，英文标题是 Measuring AI agent autonomy in practice，我把它翻译成"实测 AI 智能体的自主性：真实部署中的数据观察"。文章来自 Anthropic 的研究团队，发表于二〇二六年二月十八日。

此刻距离这篇文章发表已经过去了将近四个月。四个月在 AI 领域说长不长，但已经足够发生很多重要变化了。这篇文章研究的时间窗口是二〇二五年晚些时候到二〇二六年初，那正是 AI agent，也就是 AI 智能体，从"技术极客的玩具"开始向"真实生产系统"大规模渗透的关键节点。在那个时候，Claude Code 刚刚开始快速扩张用户群，多 agent 协作系统还处于早期探索阶段，整个行业对 AI agent 在现实世界里到底是什么样的行为模式，其实知之甚少。Anthropic 这篇文章，是业内为数不多真正拿出第一方生产数据来认真分析这个问题的尝试，这就是它的价值所在。

为什么我觉得这篇文章值得花二十五分钟好好聊一聊？因为它做了一件在 AI 研究里其实相当罕见的事情：不在实验室里测 benchmark，而是拿着真实的生产数据，回答了每个在用 AI agent 或者在构建 AI agent 产品的人都想知道的问题。人们在实际工作中，到底把多大的自主权交给了 AI？他们信任它吗？他们怎么监督它？随着经验积累，这种监督方式会怎么变化？agent 在做有风险的事情吗？这种基于实证的研究，在 AI 安全领域非常稀缺，也非常有价值。

那我们先从最基础的问题说起：到底什么是 agent？

这个词在过去一年被滥用得很厉害，每家公司都说自己的产品是 agent，但定义却五花八门。Anthropic 在这篇文章里给出了一个务实、可操作的定义：agent 就是配备了工具、能够采取行动的 AI 系统。这里的工具包括运行代码、调用外部 API、向其他 agent 发送消息等等。这个定义的好处是可操作性强，你不需要去判断一个系统有没有"真正的意图"或者"自主意识"，只要它能使用工具、采取行动，它就是 agent。用这个定义，研究 agent 就等于研究工具调用，把一个抽象的哲学问题转化成了可以用数据来回答的工程问题。

但研究 agent 并不简单。文章里坦诚地说了三个挑战。第一，没有统一的行业定义，大家说的 agent 可能根本不是一回事。第二，agent 进化得太快了，去年很多最先进的 agent 还是单线程对话，今年已经有多 agent 系统能自主运行好几个小时了。第三，Anthropic 作为模型提供商，对客户的 agent 架构其实所知甚少。比如，他们没有可靠的方法把同一个用户发到 API 上的不同请求串联成一个"会话"。这就好比你是一家电话公司，你能看到每一通电话，但你不知道同一个人打的两通电话是不是在聊同一件事，不知道它们是不是同一个任务的两个步骤。

为了在这些约束下尽量把研究做扎实，Anthropic 采用了两个互补的数据来源。

第一个是公开 API 的流量。这给了他们覆盖数千个不同客户的广泛视野。但因为没法把独立请求串成完整会话，他们的分析单位是"单次工具调用"。这就像你研究一条河流，与其试图追踪每一条鱼游了多远，不如在某个截面统计每分钟有多少鱼经过、每条鱼的状态如何。你失去了连贯性，但获得了覆盖广度。

第二个是 Claude Code，Anthropic 自家的编程 agent。因为是自家产品，他们可以把跨会话的请求全部串联起来，完整地看到一个 agent 从接到任务到完成任务的整个过程。这就好比你不仅能在截面数鱼，还能给每条鱼贴上标签、跟踪它完整的洄游旅程。代价是 Claude Code 只是一个产品，而且用户绝大多数在做软件工程相关的事，代表性有局限。

好，背景说完了，我们来看文章里最重要的几个发现。

第一个，也是数据上最令人震惊的：Claude Code 自主运行的时长，在三个月内几乎翻了一倍。

他们测量的是"一个回合"的时长，也就是从 Claude 开始工作，到它停下来之间经过了多少时间，不管停下来是因为任务完成了、它自己主动来问问题了，还是被用户打断了。这个指标虽然不完美，但能直接反映用户允许 Claude 在无人干预状态下连续工作多长时间。

先说中位数。大多数回合其实很短，中位数大概是四十五秒，过去几个月基本稳定，波动不大。这是预期之内的结果：大量新用户持续涌入，新手会更谨慎，会更频繁地确认和干预，所以整体平均值被压低了。

真正有意思的是尾部数据。从二〇二五年十月到二〇二六年一月，百分之九十九点九分位的回合时长，从不到二十五分钟涨到了超过四十五分钟。

这个数字意味着什么？它意味着，在那些最雄心勃勃的使用场景里，用户已经愿意让 Claude 连续工作四十五分钟以上，中间不去打断它。你可以想象一下这是什么感觉：你早上来了，给 Claude 交代一项任务，然后你去开会，四十五分钟后你回到桌子前，发现它已经替你推进了一大截工作。这不是普通的问答助手，这是真正意义上的自主工作了。

更有意思的一点是，这个上涨趋势在几次模型发布节点上是平滑的，没有出现陡然的阶梯式跳升。如果自主时长纯粹是因为模型变聪明了，那每次新模型上线应该有一个明显的台阶。但数据显示的是缓慢而稳定的持续增长，这说明背后有更复杂的因素在起作用：用户在日积月累地建立信任，把越来越有挑战性的任务交给 Claude；同时产品本身也在每天持续优化。这是一个人机协作关系成熟化的过程，而不仅仅是模型能力提升的结果。

文章里还提到了 Anthropic 内部的一组数据。从二〇二五年八月到十二月，他们自己的员工在用 Claude Code 处理最复杂任务时，成功率翻了一倍，同时每次会话的平均人工干预次数从五点四次降到了三点三次。用更少的干预，完成更复杂的任务，取得更好的结果。这个趋势非常清晰。

文章还引用了外部评测机构 METR 的数字作为对比。METR 测出 Claude Opus 四点五可以以百分之五十的成功率完成"需要人类花将近五个小时才能做完"的任务。而 Claude Code 实际的百分之九十九点九分位回合时长只有大约四十二分钟。这两个数字差距悬殊，但你不能直接比较，因为 METR 测的是任务的"人类难度"，不是模型实际跑了多久；而且 METR 是在理想的无干预环境里测的，现实中 Claude 会暂停问问题，用户会打断，机器速度也受各种因素影响。但把这两个数字放在一起看，传达出一个很重要的信号：模型在实践中被授予的自主空间，可能还远远低于它实际能承担的上限。文章把这叫做"部署过剩"，英文是 deployment overhang，意思是模型的能力已经到位了，但现实部署给的空间还没跟上。这既是机会，也是一种风险提醒。

好，第一个发现说完了。第二个发现是关于人类如何随着经验积累改变他们的监督策略。

Claude Code 默认是"每一步都需要用户审批"的模式。但用户可以开启"全自动审批"，也就是让 Claude 自己做决定，不用每次都来请示。研究发现，随着用户使用 Claude Code 的经验增加，开启全自动审批的比例在稳步上升。新用户，也就是用了不到五十次会话的人，大约百分之二十的会话会用全自动审批。到了七百五十次会话的老用户，这个比例超过了百分之四十。这个趋势本身很直觉，用得越多，越信任，越愿意放手。

但更有意思的是另一个数据：随着经验增加，用户打断 Claude 的频率也在同步上升。新用户大概在百分之五的回合里会打断 Claude，有经验的用户大约是百分之九的回合里会打断它。

这不是矛盾吗？一边是越来越多地放手，一边是越来越频繁地打断？

其实这两件事完全不矛盾，它们反映的是监督策略的根本转变。新用户的策略是"逐步审批"——我每一个动作都要点确认，所以 Claude 基本没机会走偏，我也不需要事后打断。有经验的用户的策略是"放手运行，异常干预"——我让 Claude 跑着，我同时干自己的事，当我发现有什么不对劲，或者 Claude 走偏了，我才去打断。这是从"逐步控制"到"例外管理"的跨越。表面上看，有经验的用户既给了更多自主权，又有更多打断，其实他们在用一种更成熟的协作模式：把注意力集中在真正需要判断的关键节点上，而不是每一个微小步骤。

文章里还提到了公开 API 上的一个类似现象。对于复杂度最低的任务，比如修改一行代码，百分之八十七的工具调用有某种形式的人工参与；而对于高复杂度任务，比如自主寻找安全漏洞或者编写编译器，只有百分之六十七有人工参与。表面上看，越复杂的任务人工参与反而越少，这很反直觉。但结合 Claude Code 的数据就说得通了：复杂任务往往来自有经验的老用户，而老用户更倾向于放手让 agent 自主运行，在关键时刻才干预。

这个发现有一个核心启示：有效的监督不等于批准每一个动作。文章明确说，有经验的用户并不是在放弃监督，而是在用更成熟的方式监督，持续关注大局，在关键时刻干预。对我们设计 agent 系统，这有非常直接的指导意义，我们待会儿会详细讲。

接下来是第三个发现，在安全角度我觉得意义最重大。

Claude Code 主动暂停询问的频率，比人类主动打断它的频率更高。

在最复杂的任务上，Claude 主动提问的频率是最简单任务的两倍以上，而且比用户打断它的频率更高。这说明 Claude 对自己的不确定性有一定的感知。当它遇到模糊的需求、需要做重大决策之前，或者意识到前方可能有坑，它会倾向于停下来确认，而不是闷着头往前冲。文章把这叫做 agent 主动限制自身自主性，不是人类在拉闸，是 Claude 自己踩刹车。

研究团队还统计了 Claude 停下来的原因分布。最常见的是向用户提供几种方案让他们选择，占百分之三十五。其次是收集诊断信息或测试结果，占百分之二十一。然后是澄清模糊或不完整的需求，占百分之十三。还有请求缺失的凭证或访问权限，占百分之十二。最后是在采取某个重要行动前寻求确认，占百分之十一。

用户打断 Claude 的原因呢？最常见的是提供缺失的技术背景或纠错，占百分之三十二。其次是 Claude 速度太慢、卡住了或者操作过于繁琐，占百分之十七。还有些人是因为已经获得了足够帮助、可以自己继续了，占百分之七。另外还有人是想亲自做下一步，比如手动测试或者部署。

这个发现在 AI 安全讨论里很有意义。我们通常说的 AI 监督，往往是指"人类在盯着 AI 看，发现问题就叫停"。但实际上，AI 自身的不确定性识别也是安全链条的一个重要环节。一个能主动说"我不确定这样做是否符合你的预期，我们先确认一下"的模型，比一个需要人类时刻盯着、出错之后才被发现的模型，在实际部署里要安全得多。Anthropic 把这作为明确的训练目标，并在文章里呼吁其他模型开发者也这样做。从这个角度来说，agent 的安全性不只是靠外部约束堆出来的，也需要模型内部的自我感知和主动刹车机制。

好，我们来看第四个也是最后一个大发现：风险分布和应用领域。

为了衡量 agent 行为的风险和自主程度，Anthropic 用 Claude 对公开 API 上的每个工具调用进行评分，风险从一到十分，一分是出了问题也没什么后果，十分是可能造成重大伤害；自主性同样从一到十分，一分是完全按人类明确指令执行，十分是完全独立决策、几乎不需要人类参与。然后他们把类似的行为聚合成簇，对每个簇算平均分。

总体来看，结果是相对乐观的。百分之八十的工具调用来自看起来有某种安全措施的 agent，比如受限的权限或者需要人工审批的流程。百分之七十三有人类以某种形式参与其中。只有百分之零点八的行为是不可逆的，比如向客户发送邮件或者执行不可撤销的操作。

但乐观的同时，有些数据点值得认真对待。在风险最高的那些工具调用集群里，出现了什么内容？出现了 API 密钥窃取后门的实现指令，出现了对病人医疗记录的访问，出现了实验室危险化学品的处置操作，出现了自主执行加密货币交易的指令。文章也说，这些里面很大比例可能是安全评估和红队演练，不是真正的恶意生产行为。但问题在于，他们没法准确区分，这恰恰说明了从外部监控视角来看的局限性：我们连什么是真实生产操作、什么是测试行为都搞不清楚。

更值得关注的是应用领域的分布。目前软件工程相关的工具调用占到整个公开 API agent 活动的将近百分之五十。这个比例是压倒性的。排在后面的是商业智能、客户服务、销售、金融、电商，但每一项都只有几个百分点。

为什么软件工程会遥遥领先？文章给了一个很好的解释：软件工程的输出特别容易被验证。你写了代码，你可以运行它，看会不会报错，看结果对不对。这种即时反馈、易于校验的特性，让软件工程师特别容易建立起对 agent 的信任。你可以随时给 agent 设定约束，检查它的输出，发现错误立刻纠正。失败成本相对可控，试错空间相对宽松。

相比之下，法律、医疗、金融这些领域，验证 agent 的输出需要和产出它本身差不多的专业知识。你没法轻易判断 agent 给的法律建议是不是正确的，你也没法在没有专业背景的情况下快速审核一份医疗诊断建议。这就大大拖慢了信任建立的速度。但这些领域已经在开始实验了，随着 agent 能力持续提升，高风险行业的渗透只是时间问题。一旦 agent 大规模进入这些领域，整个部署生态的风险画像将发生根本性的变化。

好，四个核心发现聊完了，现在说说这些对我们实际工作意味着什么。对那些正在构建 agent 系统的工程师和产品经理，文章给出了几个很具体的行动方向。

第一，投资部署后的监控基础设施。这是文章里反复强调的一个点。预部署评测，也就是在实验室里测模型能力，是必要的，但不够。很多关键的使用模式只有在真实部署后才会暴露出来，靠实验室测试根本发现不了。Anthropic 能拿出这篇文章里的数据，是因为他们提前建了基础设施来收集这些信息，这不是理所当然发生的，是一个有意为之的工程决策。

如果你在做 agent 产品，你需要能够回答这些问题：你的用户在什么时候打断 agent？打断的原因是什么？哪些任务失败了？用户的自主授权比例随时间怎么变化？哪些类型的操作引发了最多的人工干预？这些问题，pre-deployment 测试根本没法回答，你必须在线上采集数据才能知道答案。文章里提到了 OpenTelemetry 集成，这是 Claude Code 正在推进的事情，把 agent 的运行过程结构化地记录下来，形成可分析的轨迹数据。对于你自己的系统，至少要记录每次工具调用的时间戳、来源、结果，以及用户是否进行了干预。有了这些数据，你才能分析哪里出了问题，哪里还有优化空间，也才能在出现安全问题时有迹可循。

第二，在模型训练和 system prompt 设计里，把"遇到不确定时主动暂停确认"这个行为显式地强化进去。文章里说 Claude Code 主动暂停的频率比用户打断它更高，这不是自然而然发生的，这是有意训练的结果。一个能说"我不确定这样做是否符合你的预期，我们先确认一下"的 agent，比一个埋头执行、出错之后才被发现的 agent，在实际部署里要安全得多。

你可能会担心这样做会让 agent 显得"不够能干"，总是来问问题，用户会觉得烦。但文章的数据告诉你，在复杂任务上 Claude 的暂停大多是用户认为有必要的，而且最常见的暂停原因是提供几个方案让用户选择，这反而是在给用户赋权，是在提升决策质量，而不是在给用户添麻烦。关键是要让暂停有意义、有针对性，在真正需要人类判断的节点停下来，而不是每一步都来确认。

第三，设计支持"监控和干预"的交互界面，而不只是"审批"。这是我觉得最被忽视、但实际上最重要的一点。

很多人在设计 agent 产品的时候，会自然地想到"审批流"——在每一步操作之前弹出确认框，让用户点批准或者拒绝。这看起来很安全，但数据告诉我们，随着用户经验增加，他们会觉得这种逐步审批既繁琐又低效，然后直接开启全自动，绕过了所有审批。你为安全而设计的机制，反而被经验用户主动规避了。

更成熟的监督模式应该是：让用户能随时清楚地看到 agent 在做什么，但不强制审批每一步；同时提供简单直接的干预入口，让他们在任何时候都能轻松说"停，换个方向"。这就像开车用自动驾驶，你不需要批准每一次方向盘的微调，但你随时可以接管方向盘，而且你始终知道车在做什么。关键是可见性加上可接管性，而不是强制性的每步审批。Claude Code 在这方面的一个具体实践是实时引导功能，让用户可以在 Claude 运行过程中随时输入新的方向，不必等它完成当前任务才能提供反馈。如果你在设计自己的 agent 产品，值得认真考虑类似的交互模式。

第四，从今天开始就为高风险领域做准备。软件工程占将近百分之五十这件事，本质上是个历史偶然，软件工程师是最早大规模使用这些工具的人群，他们的习惯和工具链塑造了当前的生态格局。但文章的数据显示，医疗、金融、网络安全这些高风险领域的实验性部署已经在增长。一旦这些领域的 agent 用量起来，整个生态系统的风险画像会完全不同。今天的"大多数都是低风险"的结论，明年可能就不成立了。所以现在就把监控体系、安全边界和应急机制建好，比到时候追着问题跑要明智得多。特别是如果你所在的团队正在探索把 agent 引入医疗、金融或者合规相关场景，这个时机点非常重要。

第五，不要指望法规来规定具体的操作流程。文章里有一段话我觉得写得非常清醒：不要立法要求所有 agent 必须获得用户对每个操作的批准。这种一刀切的规定会在低价值场景制造大量摩擦，而对那些真正高风险的边界场景却没有实质帮助，因为真正懂行的人会绕过它，真正危险的操作也不会因为多了一个确认框就变得安全。更有意义的监管目标应该是：确保人类有能力有效地监控和干预，而不是规定具体的干预形式。随着 agent 技术和相关测量方法的成熟，监管的重心应该放在"人有没有足够的能力和工具来介入"，而不是"人有没有在每一步点了一个确认按钮"。

我个人认为，这篇文章最大的价值不在于任何单一的数据结论，而在于它建立了一个框架，一个用实证方法研究 agent 在真实世界中行为的框架。AI 安全领域长期以来都是靠推测和原则在讨论问题，缺乏实测数据的支撑。这篇文章说：我们可以测量，我们必须测量，而且我们已经开始测量了。如果更多的模型提供商和产品团队都建立起类似的测量基础设施，AI 安全的讨论就会从"我们认为应该怎样"逐渐转向"我们看到的实际上是怎样"，这是一个质的跃升。

当然，这篇文章也坦诚地说明了局限性。数据来源单一，只有 Anthropic 自己的数据，其他模型提供商上的 agent 可能有完全不同的行为模式。Claude Code 的用户群体高度集中在软件工程领域，不能代表所有 agent 场景。Claude 对风险和自主性的评分是 Claude 自己来打的，这里面存在自我评估偏差的风险。数据窗口是二〇二五年底到二〇二六年初，AI 领域变化这么快，一些具体结论可能已经在演变之中。这些局限性不是在否定这篇文章，而是在提醒我们：这是一个开始，而不是终点。Anthropic 自己也说，他们会继续迭代方法、持续发布更新的研究结果。

最后做个收尾。这篇文章告诉我们，AI agent 的自主性正在稳步提升，但这个过程是模型、用户和产品三者共同塑造的结果。Claude 自己会踩刹车，用户在积累信任并调整监督策略，产品设计在塑造整个交互范式。当前大多数 agent 行为还集中在低风险的软件工程场景，但高风险领域的早期实验已经出现，未来的风险边界一定会向外扩展。面对这个趋势，无论是模型开发者、产品团队还是政策制定者，最重要的事情是投资到一件事上：理解 agent 在真实世界里的实际行为，而不只是在实验室里的理论能力。监控基础设施不是一个锦上添花的功能，它是确保 AI 部署安全的地基。
