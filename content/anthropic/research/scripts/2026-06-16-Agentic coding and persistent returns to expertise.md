## Script

今天要聊一篇昨天刚刚发出来的研究，英文标题是 Agentic coding and persistent returns to expertise，我把它翻译成《智能体编程与专业知识的持续回报》。这篇文章发表于二零二六年六月十六日，作者是 Anthropic 研究团队的 Zoe Hitzig、Maxim Massenkoff、Eva Lyubich 等几位研究员。

为什么我觉得这篇值得专门拿出来讲？因为它在一个大家都在争论却缺乏数据的问题上，给出了目前我见过的最大规模的实证证据——那个问题就是：在 AI 编程工具越来越强大的今天，究竟是什么决定了你能不能用好它？是你的编程水平？是你的学历背景？还是别的什么？这篇研究用四十万次真实会话的数据给出了一个相当清晰的答案。

在正式进入内容之前，先说说研究的规模和方法，这很重要，因为它决定了这份数据的可信程度。他们分析了大约四十万次 Claude Code 的交互式会话，来自大约二十三万五千名用户，时间跨度是二零二五年十月到二零二六年四月，整整七个月。这不是实验室里设计好的测试，是真实用户在日常工作里使用的完整记录。

他们使用的是一套隐私保护分析工具，研究者不会直接读取个人会话内容，所有的分类和标注都通过模型自动完成，然后只汇报聚合统计结果。所有分类任务用的是 Claude Sonnet 四点六。这套方法的局限性他们自己也坦承了，我后面会提到，但总体上这是一份扎实的大规模研究。

好，先来看第一个核心发现：人和 Claude 之间的分工到底是什么样的。

你可能以为用 Claude Code 写代码，是一种来来回回的协作，用户写一段，Claude 补全一段，有点像更厉害的 Copilot。但实际的分工模式比这个清晰得多，也更像一种经典的组织架构。

研究者做了一件很有意思的事，他们把一次会话里所有有意义的决策分成两类：规划决策，和执行决策。规划决策是"做什么"——要实现哪个功能，选哪个技术路线，什么叫做完成了，要不要改变方向。执行决策是"怎么做"——改哪个文件，写什么代码，用什么变量名，运行哪条命令。

然后他们问：这两类决策里，分别有多少比例是用户做的，多少是 Claude 做的？

答案非常干净：平均来说，用户做了大约百分之七十的规划决策，但只做了大约百分之二十的执行决策。

翻过来说就是：Claude 在执行层面几乎全权负责，但在方向层面用户仍然是主导的。

这个分工跟我们日常工作里的一种角色关系非常像——一个懂业务的产品经理或者架构师，对着一个执行力很强的工程师说"我们要做什么、为什么做、做成什么样叫成功"，然后工程师去决定具体的实现方式。用户是那个懂业务的角色，Claude 是那个执行的角色。

这个分工在数量上也体现得很明显。一次典型的会话有大约四个来回，用户每发一条 prompt，Claude 平均会自主执行大约十个动作——读文件、写代码、跑命令，一套组合拳下来，平均输出大约两千四百个词的内容。而且动作数量的长尾非常惊人：大约百分之二的会话，Claude 每条 prompt 平均执行超过一百个动作；每两百七十个会话里会有一个平均超过两百个动作；每两千三百个里甚至有一个超过五百个动作。五百个动作是什么概念？那基本上是 Claude 在接到一条指令之后，连续自主工作了很长一段时间，期间几乎不需要人介入。

而且执行层面的自主度和规划层面的控制权之间有很强的关联。当用户控制了超过百分之八十的执行决策，Claude 每轮大约执行八个动作。但当 Claude 接管了超过百分之八十的规划决策，它每轮会执行大约十六个动作。换句话说，Claude 拿到更大的自主权时，它会做更多的事，这很符合直觉。

好，第二个核心发现，也是这篇研究最反直觉的地方：什么决定了你能用好 Claude Code。

很多人的第一直觉是：编程水平越好，用 Claude Code 就越有效。这个直觉是错的，或者至少是不完整的。真正重要的是你在当前任务所在领域的专业知识深度。

研究者从会话记录里评估每个用户在这个任务上的专业水平，从一到五：新手、初学者、中级、高级、专家。他们不看用户的职位头衔，也不看他们会不会写代码。他们看的是三个信号：第一，用户给出指令时表述的精确程度；第二，用户让 Claude 验证什么，验证的内容是否体现了深层的领域理解；第三，在会话过程中，是用户更多地在纠正 Claude，还是 Claude 在纠正用户。

这里有一个研究里明确给出的关键定义，我觉得非常重要，值得仔细听一下：专业知识是任务特定的，不是人特定的。

一个有十年经验的高级工程师，第一次写 Rust，在这个任务上就是新手。一个从来没用过 Python 的会计，如果她能清楚地告诉 Claude 这个 Python 脚本必须执行哪些具体的对账规则，而且能在 Claude 的实现里发现它在月末关账时处理了一个错误的边界情况，那这个会计在这个任务上就是专家。

这个定义把"专业知识"从抽象的职业标签里剥离出来，锚定到了具体任务上。这很重要，因为它说明：你对自己所在领域的理解，就是你用 Claude Code 的核心资产，和你会不会写代码关系不大。

专业知识对 Claude 产出的影响有多大？数字非常惊人。新手级别的会话，每条 prompt 平均让 Claude 执行大约五个动作，产出大约六百个词。而专家级别的会话呢？每条 prompt 引发超过十二个动作，产出大约三千两百个词——动作数是新手的两点四倍，输出文字是新手的五倍多。

而且这个规律在所有类型的任务里都成立，不管是写新代码、调 bug、做数据分析还是写文档，专家用户就是能让 Claude 每次干更多有价值的工作。

你可以想象两个用户用 Claude Code 做一个数据处理任务。第一个用户只是大概知道自己要什么，说"帮我做一个销售数据分析"。Claude 只能做最通用的东西，做完之后还有大量的来回确认，最终可能做了很多不符合实际需求的工作。第二个用户非常清楚自己所在公司的业务规则，她说"帮我写一个脚本，从这个格式的 CSV 导入数据，按照我们公司的财年规则重算每个销售代表的提成，特别注意要处理那种一笔订单被拆分成多个发货单的情况，最终输出要和我们 ERP 系统的导出格式一致，这样我可以直接导入"。Claude 一下子拿到了所有重要的约束条件，可以一次性完成大量正确的工作。这两者之间的差距，就是专业知识在起作用。

第三个核心发现：职业背景的重要性，比很多人想象的低很多。

研究者把用户按职业分类，对应美国劳工统计局的职业分类体系，大类有软件和数学类职业、商业和金融、艺术和设计、管理、自然和社会科学、法律、医疗等等。然后他们比较不同职业用户在产生代码的会话里的成功率。

成功率分两个层次来看。"验证成功"是最严格的标准，要求分类器判断成功，而且有可核实的证据，比如有 git commit 和这次工作对应，或者测试通过了，或者用户明确表示满意。宽松一点的是"至少部分成功"。

软件相关职业的用户验证成功率大约是百分之三十四，其他所有职业是百分之二十九。差距五个百分点。换成宽松标准：软件从业者百分之八十九，其他职业百分之八十八——差距只剩一个百分点。

数据集里前十大职业，每一个职业的验证成功率都落在软件工程师的七个百分点范围内。最有意思的是，管理层职业的成功率甚至略高于软件工程师。研究者的解释是，管理层可能有更好的能力清晰地指派任务、表达预期、确认结果，这些技能在指挥 AI agent 上有直接的迁移价值。他们甚至开玩笑说，也许"表现得像个经理"本身就能带来更高的成功率。

这个发现在七个月的观察期里保持稳定：软件从业者和其他职业之间的差距既没有变大，也没有变小，但两个群体的成功率都在上升。随着 Claude 越来越强，各种职业背景的人都在受益。

第四个核心发现，也是实践上最有用的一个：专业知识对成功率的影响，尤其是在遇到问题的时候。

整体来看：新手级别的会话验证成功率是百分之十五，部分成功率是百分之七十七。中级到专家级别的验证成功率在百分之二十八到三十三之间，部分成功率在百分之九十一到九十二之间。

这已经很有差距了，但更有说服力的数字来自"遇到麻烦"的会话里。研究者定义了一类情况叫"遇到麻烦"，就是会话里有明确的失败证据：报错了、测试没过、用户在反复重试同一件事、或者用户表达了不满。

在这类遇到麻烦的会话里，新手最终达到验证成功的只有百分之四，而专家级别达到了百分之十五。更说明问题的是"放弃率"：在遇到麻烦的会话里，新手最终什么都没做成、一行代码都没产出就放弃的比例是百分之十九，而中级及以上用户的放弃率只有百分之五到七。

专业知识的核心价值之一，就是当 Claude 跑偏或者卡住了的时候，你有能力识别问题出在哪里，然后把它拉回正轨。新手看到 Claude 输出了一些看起来很多很复杂的东西，不知道对不对，不知道问题在哪，最后只好放弃。专家能说"这里不对，因为这个业务规则你没有考虑到，重新来"。

还有一个很重要的非线性特征，研究里特别强调了：大部分的收益来自从新手到中级的跨越，从中级再往上到专家的额外增益相对有限。也就是说，你不需要成为某个领域的顶级权威，只要有扎实的工作级别理解，成功率就已经和领域专家差距不大了。

第五个值得关注的发现：任务类型和价值在七个月里的演变趋势。

先看任务类型。二零二五年十月的时候，调 bug 占了所有会话的百分之三十三，是最大的单一类型。到二零二六年四月，这个比例降到了百分之十九，下降了将近一半。调 bug 让出的空间被两类东西填补：运维操作类从百分之十四增长到百分之二十一，写作和数据分析从大约百分之十增长到接近百分之二十。

我个人认为这个变化有两层含义。第一层是 Claude 本身能力的提升——模型越来越能一次性把代码写对，所以专门回头调 bug 的需求减少了。第二层是用户用法的演变——大家开始把 Claude Code 用于更广泛的场景，不只是编程，而是用于整个技术工作的生命周期：部署、监控、数据分析，甚至写文档和报告。

然后是任务价值的变化。研究者用一个很聪明的方法估算每次会话的经济价值：把每次会话的工作内容和自由职业平台上的真实职位发布做比较，看"这个工作如果挂到平台上值多少钱"。他们自己也说这个绝对值不应该被认真对待，但用来做相对比较是有意义的。

按这个方法，从二零二五年十月到二零二六年四月，平均会话的估算价值上涨了大约百分之二十七。分类来看更加明显：新建项目相关的任务价值上涨了百分之四十三，运维操作上涨了百分之三十四，调 bug 上涨了百分之三十二。不只是任务数量变多了，而是每个任务本身也变得更复杂、更有价值了。

好，研究内容聊得差不多了，我们来谈谈这些发现对实际工作的意义，这部分我会结合自己的一些看法，说明哪里是原文的数据结论，哪里是我个人的延伸解读。

第一个启示：提升领域理解，而不只是学提示词技巧。

很多人在刚开始用 AI 工具的时候，把大量精力放在研究怎么写更好的 prompt，怎么让 AI 更听话、输出更精准。这当然有价值，但这篇研究告诉我们，更根本的杠杆是你对自己工作领域的理解深度。提示词技巧是术，领域理解是道。你对问题理解得越透彻，你给出的约束条件越精确，你提出的验收标准越清晰，Claude 就能在每次交互里做更多正确的工作。

这对不同背景的人意味着不同的东西。如果你是产品经理，你对用户需求和业务规则的理解就是你最大的资产。如果你是数据分析师，你对数据来源、指标定义和分析目的的清晰度决定了 Claude 能帮你多少。如果你是工程师，你对架构约束、性能要求和代码库上下文的掌握，就是你区别于行外人的核心优势。

第二个启示：把"会话 debug"当成一项需要培养的能力。

研究发现，专家和新手的最大差距不在于一帆风顺的时候，而在于遇到问题的时候。这里我个人认为有一种能力需要专门培养，就是诊断会话本身出了什么问题，而不只是诊断代码里有什么 bug。

当 Claude 给你的结果不对，你需要能回答：是 Claude 误解了你的需求？是你的需求表达本身有歧义？还是 Claude 在技术实现上走了一条错误的路？这三种情况需要完全不同的纠偏方式。第一种需要你重新描述需求。第二种需要你自己先想清楚到底要什么。第三种需要你有足够的技术判断力告诉 Claude 哪里错了。

这种诊断能力，本质上就是你对这个领域有多懂的直接体现。你越懂，你就越能快速识别问题出在哪一层，越能用最少的来回把会话拉回正轨。

第三个启示：重新审视你在会话里的分工方式。

研究告诉我们，平均来说用户做七成规划决策、两成执行决策，这是现状。但这个比例不是命中注定的，你可以有意识地调整。

我个人的建议是：把规划层面的核心判断坚决留在自己手里，同时更大胆地把执行细节交给 Claude。什么叫核心规划判断？比如架构选型、关键业务逻辑的边界条件定义、什么样的输出算成功。这些东西如果 Claude 判断错了，后面的工作全部返工。什么叫可以交给 Claude 的执行细节？具体文件怎么组织、变量怎么命名、用什么具体的 API 调用——只要方向对，这些 Claude 做得比大多数人更细致。

换个角度想：如果你现在在会话里花了大量时间讨论怎么实现一个功能，那可能是在做一件低价值的事情。你更应该把精力放在"实现什么"和"什么叫做对了"这两个问题上，然后把"怎么实现"完全交给 Claude。

第四个启示：不需要等到成为专家才开始。

这篇研究里有一个令人鼓舞的数据：大部分的收益来自从新手到中级的跨越，从中级到专家的额外增益相对有限。这意味着，对于任何一个领域，你不需要做十年才能有效地使用 Claude Code。只要你有扎实的工作级别业务理解，成功率就已经和领域专家差距不大了。

这个洞察对想要跨界使用 Claude Code 的人特别有价值。比如一个对前端一知半解的后端工程师，如果他对自己在做的那个业务系统非常熟悉，他在 Claude Code 上的成功率可能会比一个根本不懂这个业务的前端专家还要高。所谓"够用"的领域知识，在 AI 工具面前就已经能产生相当强的回报了。

第五个启示：在高价值任务上建立可核实的验收标准。

随着 Claude 越来越能处理复杂、高价值的任务，一个容易掉入的坑是：Claude 做完了，看起来很顺利，但你不知道对不对。研究里"验证成功"的定义——需要有 git commit、测试通过、或者明确确认这样的可核实证据——这个思维框架值得借鉴到日常工作里。

在重要任务开始之前，先想清楚：什么样的证据能证明这件事做对了？是测试通过？是输出满足特定格式？是某个具体的业务规则被正确执行了？把这个验收标准提前想好，然后告诉 Claude，让它在完成之后验证这些条件。这样你才能真正知道结果是否可用，而不是依赖 Claude 说"任务完成了"来判断。

最后来说说这篇研究的局限性，研究者自己说得很坦诚，我觉得有必要复述一下。

他们无法观察真实世界的结果：会话里写的代码到底有没有被用上，有没有产生实际价值，他们不知道。他们的所有分类都依赖模型读取会话记录，这里有固有的误差，他们在附录里也做了一些验证，但分类器的准确性始终是个挑战。另外，这份报告排除了非交互式使用，也就是通过命令行自动化调用的那些 Claude Code 使用，而那部分的量也相当大。

这些局限性不削弱这篇研究的价值，但提醒我们不要把这里的数字当成精确的真理，而是当成一个方向性的信号。

好，我们来做个收尾。

这篇研究的核心信息非常清晰：在 AI 编程时代，代码只是一种媒介，真正决定你能用好 AI 工具的，是你对自己工作领域的理解深度。编程背景的重要性正在下降，领域专业知识的回报正在上升，而且这个趋势在七个月的数据里稳定存在。

我个人觉得最令人深思的一个结论，是"中级就够了"这个发现。AI 工具不是在奖励那些已经无所不知的顶尖专家，它是在放大每一个对自己领域有扎实工作级别理解的人的能力。这种放大效应如果真的扩散到各行各业，可能是这波 AI 浪潮里最深刻的社会变化之一——不是替代人，而是让更多有领域理解的人能够做以前需要专门技能才能做的事情。
