## Script

这篇文章的英文标题是 Building Effective AI Agents，我把它翻译成「构建有效的 AI Agent」。它发表于二〇二四年十二月十九日，作者是 Anthropic 的两位工程师 Erik S. 和 Barry Zhang。

今天是二〇二六年六月十六日，这篇文章距今已经过了将近一年半。在 AI 这个领域，一年半意味着什么？意味着 Claude 三点五早已成为历史，Claude 四系列已经全面铺开；意味着各种 Agent 框架从野蛮生长进入了市场淘汰阶段；意味着很多当年信誓旦旦说要「全自动 AI 办公」的团队，悄悄缩减了 Agent 的权限，或者把系统推倒重来。

所以你可能会问：这篇发表于一年半前的文章，还值得花二十五分钟听吗？我的回答是：非常值得，甚至比刚发表时更值得。原因是，这篇文章讲的不是某个具体的模型或者某个特定的框架，而是 Agent 系统背后的工程方法论。方法论的生命周期比技术细节长得多。事实上，文章里的很多判断，在一年半后的今天来看，反而更显得有预见性。很多团队走过弯路、踩过坑之后，才发现 Anthropic 在这篇文章里早就说过那些坑在哪里、该怎么绕开。

让我先交代一下这篇文章的背景。二〇二四年年底，AI Agent 这个概念非常火。你打开任何一个技术社区，都能看到各种 Agent 框架的教程：LangChain、AutoGen、CrewAI、Dify……大家都在问，怎么让 AI 自主完成复杂任务？怎么给它接上各种工具？怎么让多个模型互相协作？这种热度催生了一批非常复杂的系统：多层嵌套的 Agent 调用链，动辄几十个工具，号称全自动。

在这种浪潮下，Anthropic 的工程师们在同期和几十个企业团队一起合作，亲眼目睹了各种真实案例。他们的结论有点反直觉：做得最成功的团队，用的不是最复杂的框架，而是最简单、最可组合的模式。这篇文章就是要把这些模式总结出来，告诉你一个 Agent 系统应该怎么设计，以及更重要的——什么时候不该用 Agent。

好，我们正式进入正文。

第一个要搞清楚的核心概念，是 Agent 和工作流的区别。这两个词在中文技术圈里经常被混用，但 Anthropic 给出了一个非常清晰的区分，我觉得值得仔细讲一讲。

工作流，英文是 Workflow，是指通过预定义的代码路径来编排大语言模型和工具的系统。什么叫预定义的代码路径？就是说，整个系统的执行逻辑，是由人类工程师事先写好的。什么时候调用哪个模型，输出结果传给谁，出了什么情况走哪条分支，这些都在代码里写死了。模型在这里面扮演的是一个处理节点，而不是决策者。

Agent 则完全不一样。Agent 是指由大语言模型动态地指导自己的处理流程和工具使用方式的系统。也就是说，模型自己决定下一步做什么，用哪个工具，这一步要不要继续还是停下来等待人类反馈。执行路径不是预先写好的，而是模型在运行过程中实时生成的。

你可以用一个很具体的类比来理解这个区别。工作流就像一条精密设计的工厂流水线：零件从哪个工位来、在这里做什么加工、然后传到哪个工位，所有这些都是设计师在建线时就定好的，流水线严格按照设计轨道运转，不会偏离。而 Agent 更像一个有经验的工程师：你告诉他「帮我把这个 bug 修掉」，他自己决定先读哪些文件，要不要跑一下测试，要不要去查文档，遇到了意外情况自己想办法，每一步都是基于当前状况做出的实时判断，没有固定剧本。

Anthropic 把这两类都称为 Agentic Systems，也就是 Agent 性系统。但它们的适用场景和风险特征非常不一样，后面我会详细说。

在介绍具体模式之前，文章提出了一个基础概念，叫做增强型大语言模型，英文是 Augmented LLM。这是所有 Agent 系统的最小构建单元，理解它是理解后续所有内容的前提。

普通的大语言模型，你给它一个问题，它输出一个答案，仅此而已。增强型大语言模型，是在这个基础上加了三类能力：第一是检索能力，模型能主动生成查询去访问外部知识库，而不只是依赖训练时的知识；第二是工具使用能力，模型能调用外部的 API、执行代码、与外部系统交互；第三是记忆能力，模型能记住对话上下文，或者把信息持久化存储供后续调用。现在的 Claude 系列模型原生支持这三种能力，它可以自己决定什么时候该搜索、该用哪个工具、该记录什么信息。

文章还特别提到了 Model Context Protocol，简称 MCP，这是 Anthropic 在二〇二四年年底刚刚发布的一个协议标准，让开发者可以用统一的接口给模型接入各种第三方工具。到今天，MCP 已经成为 AI 工具集成领域的事实标准，生态非常庞大。这也是这篇文章里一个很有预见性的判断。

好，我们现在来逐一拆解文章介绍的五种核心工作流模式。

第一种模式叫提示词链式调用，英文是 Prompt Chaining。顾名思义，就是把一个复杂任务拆解成一系列有顺序的步骤，每一步 LLM 的输出作为下一步 LLM 的输入，一步一步往后传。在每个步骤之间，你可以插入一个检查门，英文叫 gate，也就是一个程序化的质量控制节点，如果检查不通过就停下来报错或者重试，通过了才继续往下走。

什么时候用这个模式？当你的任务可以被清晰地分解成固定的几个子步骤，而且每个子步骤相对独立、相对容易，每步之间有明确的交接点时。

文章举了一个很直观的例子：生成营销文案。第一步，让模型写一段英文文案；第二步，程序检查这段文案的字数、关键词和品牌调性，不符合就打回重写；通过了，第三步，再让另一个模型把它翻译成中文。这就是一个标准的三步提示词链。

我来解释一下为什么这样做比让一个模型一步到位更好。你有没有注意到，当你让模型同时完成多件事的时候，它的注意力是分散的，有时候反而每件事都做得不够好？就像人类一边开车一边打电话，每件事的质量都打折扣。把任务拆开，让每次 LLM 调用只专注在一件事上，整体质量往往更高。这就是用延迟换准确率的核心逻辑。当然，代价是总时间变长了，这个权衡你需要根据具体场景来判断。

第二种模式叫路由，英文是 Routing。路由的含义是：先判断输入属于哪个类别，然后把它分发给对应的专门处理流程。

为什么要这样做？因为不同类型的输入往往需要非常不同的处理策略，如果用一个通用的提示词模板来处理所有输入，你可能会发现，为了照顾 A 类型优化的提示词，在 B 类型上效果变差了，反之亦然。把它们拆开，每类输入有自己专属的提示词、工具和处理逻辑，互不干扰，整体效果更好。

文章给了两个非常典型的例子。第一个是客户服务场景：用户的问题可能是普通咨询、退款请求或者技术故障报告，这三种问题分别需要不同的处理权限和工具。你不可能用同一套流程同时处理退款申请和技术支持问题。路由就是先让 LLM 判断这个问题属于哪类，然后转给对应的子流程。

第二个例子更贴近工程师的日常工作，也更有意思：把简单的、常见的问题路由给便宜的小模型，比如 Claude Haiku 四点五；把复杂的、罕见的问题路由给能力更强的大模型，比如 Claude Sonnet 四点五。这样可以在保证质量的前提下大幅控制成本。想想看，如果你的系统一天处理十万条用户查询，其中七成是很简单的问题，那这七成如果都交给小模型处理，费用节省是非常可观的。这个思路现在很成熟，但在二〇二四年年底，Anthropic 能在官方文章里把这个讲得这么清晰，是很有指导价值的。

第三种模式叫并行化，英文是 Parallelization。这个模式是让多个 LLM 同时处理任务，然后把结果汇总起来。文章把并行化分成两种变体，它们的出发点不太一样，值得分开讲清楚。

第一种变体叫分区，英文是 Sectioning。就是把一个大任务切成互相独立的几块，让多个 LLM 实例同时处理不同的块，最后把结果拼起来。这样做的目标是提速，把串行变并行。

文章给出的一个典型用例是安全过滤场景：一个模型实例负责生成对用户问题的回复，同时另一个模型实例对用户的问题进行安全审查，看看有没有不合适的内容或者越权请求。这两件事并行发生，最后只有通过安全审查的回复才会展示给用户。文章特别指出，把这两件事分给两个 LLM 实例分别处理，往往比让同一个 LLM 调用既要生成回复又要做安全判断效果更好。原因还是注意力分散的问题：让一个模型同时兼顾两件性质不同的事，它对每件事的处理质量都会受影响。

第二种变体叫投票，英文是 Voting。就是让多个 LLM 实例独立地处理同一个任务，然后通过某种投票机制得出最终结论。这不是为了提速，而是为了提高可靠性和置信度。

文章举的例子是代码安全审查：你有一段代码，需要判断有没有安全漏洞。你让三个配置不同的 LLM 同时审查这段代码，任何一个发现了问题就标记出来。这样做的逻辑是什么？是因为大语言模型在某些问题上存在随机性，一个 LLM 可能错过某个漏洞，但三个独立的 LLM 同时审查，错过同一个漏洞的概率就大幅降低了。同时，你还可以设置不同的投票阈值：如果只要一个 LLM 发现问题就报警，那误报率会比较高；如果要三个都发现才报警，漏报率会更高。你可以根据业务需求来调节这个阈值，在误报和漏报之间找到平衡。

我个人觉得，投票这个模式特别有意思，因为它引入了多样性来对抗单点判断的不确定性。这和统计学里的集成学习，英文叫 ensemble，思路是相通的：与其让一个专家做判断，不如让多个独立的专家各自判断，然后综合结论，往往更可靠。

第四种模式叫编排者与工作者，英文是 Orchestrator-Workers。这个模式里有一个中央的 LLM 扮演编排者的角色，它负责分析当前任务，动态地把任务拆解成子任务，分配给多个工作者 LLM 去执行，然后收集工作者的结果，综合成最终答案。

听起来和并行化的分区很像，对吧？文章专门解释了关键区别：在分区模式里，子任务怎么划分是提前写好的，代码里写死了「把任务分成三块，分别处理」；而在编排者-工作者模式里，编排者根据当前具体任务的内容来动态决定怎么拆分、拆成几份、每份做什么。也就是说，子任务的划分方式本身也是模型在运行时决定的。

什么时候适合用这个模式？当任务的复杂度是不可预测的，你没办法提前知道需要多少个子步骤。文章举的例子是大型代码库的修改任务：你给 Agent 一个功能需求，要修改代码库里的若干个文件，但具体要改哪些文件、每个文件改什么，取决于代码库的当前状态和需求的具体内容，没法提前规定。编排者先分析任务，再决定拆成几个工作单元，再分配给工作者执行，这才是合适的处理方式。

这个模式今天已经在很多 coding agent 产品里普遍应用了，包括我们正在用的 Claude Code，内部就有类似的结构在工作。

第五种模式叫评估者与优化者，英文是 Evaluator-Optimizer。这是一个循环结构：一个 LLM 生成答案，另一个 LLM 评估这个答案并提供具体的改进反馈，生成者根据反馈来修改答案，修改后的版本再送回给评估者，如此循环迭代，直到答案达到质量标准，或者达到预设的最大迭代次数。

文章给出了两个判断这个模式是否适用的标准：第一，当你给模型提供具体的反馈时，它的输出能明显改进；第二，模型能够提供这种有质量的、可操作的反馈。如果两个条件都满足，这个模式就很可能奏效。

文章举的一个例子特别有启发性：文学翻译。翻译是个非常微妙的任务，第一遍机器翻译往往能做到字面准确，但未必能捕捉到原文的语气、节奏、文化内涵，以及作者独特的语言风格。如果这时候让另一个 LLM 充当编辑，给出「这里的表述太正式，失去了原文的随意感」或者「这个意象在目标语言里没有对应的文化联想，需要意译而不是直译」这样具体的反馈，然后翻译者根据反馈修改，经过几轮迭代，译文质量会显著提升，比单次翻译好得多。

这个模式让我想到人类自己的写作和修改过程。一个认真的作家写完初稿，交给编辑，编辑给出意见，作家修改，再给编辑看，如此往复。现在 LLM 可以同时扮演作家和编辑两个角色，速度比人快得多，代价只是多了几次 LLM 调用和更多的时间。关键是，评估者提供的反馈必须具体、可操作，泛泛的「继续改进」是没用的。

说完了五种工作流模式，我们来讲真正意义上的自主 Agent。

文章描述了 Agent 的典型工作方式：首先接收用户的指令，或者和用户讨论清楚任务目标；然后自主地规划和执行，可能要运行很多轮，每一轮都要从真实环境中获取反馈，比如工具调用的结果、代码实际运行的输出；遇到不确定的关键节点，可以暂停下来询问人类；完成任务后停止，或者达到最大迭代次数时强制停止。

文章特别强调了一个概念，叫从环境中获取真实反馈，英文叫 ground truth。Agent 的每一步决策，都应该建立在真实世界的反馈基础上，而不是建立在自己对结果的推测上。举个例子：一个写代码的 Agent，每次修改代码之后，应该实际运行测试套件，看看测试有没有通过，再决定下一步。如果它只是自己推断「这段代码应该能跑」，而不去真正运行，就失去了和真实世界的连接，错误会悄悄积累。

自主 Agent 的风险也非常值得重视。文章说得很直接：Agent 的自主性越强，错误复合的可能性就越高。如果第一步判断出了偏差，Agent 可能带着这个偏差继续走十几步，每一步都在错误的方向上越走越远，而且你很难知道它在哪里开始偏的。文章给出的建议是：在沙箱环境里做大量测试，配置完善的安全护栏，设置明确的停止条件，比如最多执行多少步，以及在执行不可逆操作之前必须请求人类确认。

什么时候该用真正的自主 Agent？文章的答案是：当任务是开放式的，你没办法预先知道需要多少步才能完成，而且很难硬编码一个固定的执行路径时。如果你知道任务总是这几个步骤，用工作流就好，不需要 Agent 的自主性，那只是徒增风险和成本。

接下来我要讲文章里我认为最有价值的一部分：Agent 设计的三个核心原则，以及工具工程这个概念。

第一个原则是简洁性。这是整篇文章反复强调的基调，就是从最简单的方案开始，只有当复杂度能明显带来更好的结果时，才去增加复杂度。很多团队的失败，就在于一开始就想搭一个全自动的复杂 Agent 系统，结果花了几个月调参、调框架，最后效果还不如一个精心设计的单次 LLM 调用，同时还多了一堆难以维护的代码和难以复现的 bug。

文章还提到了框架选择的问题。市面上有很多 Agent 框架，Anthropic 在文章里也列举了一些，包括他们自己的 Claude Agent SDK，还有 AWS 的 Strands Agents SDK，以及 Rivet、Vellum 这类图形化工具。这些框架确实能帮你快速上手，但文章的建议是：先用原生的 LLM API 自己实现，搞清楚底层在发生什么，再考虑要不要引入框架。很多工程师在使用框架的时候，对框架在做什么有错误的假设，而这些错误的假设是实际生产中很多问题的根源。

第二个原则是透明性。Agent 应该显式地展示它的规划步骤，让人能够跟踪它的推理过程，知道它做了什么决定、为什么做这个决定。这对于 debug 非常重要，对于建立用户和产品团队对 Agent 的信任也至关重要。一个黑盒 Agent，就算表现不错，你也会一直不放心，因为你不知道它是怎么工作的，也就不知道它在什么情况下会出错。

第三个原则是精心设计的 Agent 与计算机接口，文章把它叫做 ACI，全称是 Agent-Computer Interface，也就是 Agent 和计算机之间的接口层，具体来说就是工具的设计。

我们都听说过 HCI，也就是人机交互界面。设计一个好的 HCI，需要做大量的用户研究，投入很多时间和精力来理解用户的心智模型和使用习惯。文章的核心论点是：给 Agent 设计工具接口，也需要同样量级的投入。工具设计得好不好，直接决定了 Agent 能不能用好这个工具，进而决定整个系统的表现上限。

文章给出了一套非常具体的工具设计建议。第一，把自己代入模型的视角，想象你只能看到这个工具的名称、描述和参数定义，你能明白该怎么用它吗？如果你自己读了之后还需要思考半天，那模型一样会感到困惑。一个好的工具定义通常包括明确的用途说明、使用示例、边界情况的处理，以及和其他相似工具的区别说明。

第二，参数名要有意义，要能自解释。文章把这类比成给初级开发者写文档。你写的参数名，要让一个刚入职一个月的工程师看完就明白是什么意思，而不是满屏的 a、b、c 或者 flag 一、flag 二。当你有很多功能相似的工具时，参数名的清晰度尤其重要，因为模型要在这些工具之间做选择，含糊的名字会导致它选错工具。

第三，一定要实际测试模型怎么用你的工具。在 Anthropic 的工作台里运行大量不同的输入，观察模型在哪里会出错，然后根据观察结果修改工具定义，再测试，如此循环迭代。不能只是写完工具定义就觉得没问题了，必须跑起来看实际行为。

第四条建议是文章里一个我特别喜欢的概念，叫做 Poka-yoke，这是一个日语词，意思是防呆设计，就是通过设计本身来防止错误的发生，而不是依赖使用者的注意力。

文章举了一个真实的例子，来自他们在 SWE-bench 基准测试上的 coding agent 开发经历。SWE-bench 是一个让 Agent 自动修复真实 GitHub 代码仓库里的 bug 的评测基准。他们在开发过程中发现，Agent 在工作一段时间后，有时候会从代码仓库的根目录切换到某个子目录，切换之后，Agent 在使用文件操作工具时，经常用相对路径来指定文件，而相对路径是相对于当前工作目录的，目录一变路径就指向了错误的位置，导致操作失败。

他们想到的解决方案非常简单：修改工具的设计，强制要求所有文件路径参数必须传入绝对路径，如果传入相对路径，工具直接拒绝并报错。做了这个改动之后，这类路径错误就完全消失了，Agent 的行为变得稳定可靠。

这个例子揭示了一个非常重要的工程直觉：很多你以为是模型能力不足的问题，其实是工具设计问题。模型不是真的不理解路径，它只是在某些状态下有时候会选择用相对路径，而这个坏习惯通过工具设计就能彻底消除，不需要去改模型本身的行为。

文章还讨论了工具输出格式的设计。同样的文件修改操作，你可以让模型输出 diff 格式，也可以让它重写整个文件；同样的代码输出，可以放在 markdown 代码块里，也可以放在 JSON 的某个字段里。这些看起来只是输出样式的不同，但对模型来说，不同格式的难度差异很大。

写 diff 格式有个特殊的要求：在 chunk header 里，模型要在写新代码之前就说清楚这个改动涉及几行，这需要预判，会引入额外的认知负担。把代码放进 JSON 字段里，需要把代码里的换行符和引号都转义，不仅格式繁琐，还容易出错。文章的建议是：选择模型在训练数据中见过最多的自然格式，避免给模型施加额外的格式负担，让模型能把注意力集中在解决真正的问题上。

好，我们来聊实践应用。文章最后介绍了 Anthropic 认为特别有价值的两个 Agent 落地场景，这两个场景也给我们提供了很多实际工作中可以参考的模式。

第一个场景是客户服务。文章的判断是，客服是 Agent 的天然适配场景，理由很充分。首先，客服对话本身就是多轮的、动态的，很难用固定流程覆盖所有情况；其次，客服需要访问外部系统，比如查询客户信息、查订单历史、查知识库文章，这些都可以通过工具集成来实现；第三，客服可能需要执行一些真实操作，比如发起退款、创建工单、更新账户信息，这些操作可以做成工具让 Agent 调用；第四，客服的成功标准相对清晰，用户的问题被解决了就算成功，可以直接度量，这意味着你能知道 Agent 做得好不好，能驱动迭代优化。

文章还提到了一个很有意思的商业信号：一些公司已经开始采用纯按结果付费的定价模式来销售客服 Agent，也就是只有 Agent 真正解决了用户的问题，才向企业客户收费。这种商业模式本身就是一种信心的表达，说明在某些客服场景下，Agent 的效果已经稳定到可以拿来做效果承诺了。

第二个场景是软件开发。文章指出，代码领域是大语言模型落地效果最显著的方向之一，能力已经从早期的代码补全演进到了现在的自主解决问题。Agent 在这里特别有效，有几个结构性原因：代码的对错有客观标准，可以用自动化测试来验证，不需要人工逐行核查；Agent 可以根据测试失败的具体信息来迭代修改代码，形成一个闭环的反馈循环；代码问题通常有清晰的问题陈述和明确的成功标准，比很多自然语言任务边界清楚得多。

Anthropic 自己的 Agent 在 SWE-bench Verified 基准测试上已经能够根据 pull request 的描述，自动处理真实的 GitHub issue，不需要人工参与整个修复过程。当然，文章也强调，自动化测试能验证功能上的正确性，但人工审查仍然必不可少——你需要确认代码改动是否符合更大的系统架构设计意图，是否和其他模块的演进方向一致，这些是自动测试很难覆盖的。

那么对于我们自己的实际工作来说，这篇文章到底意味着什么，有哪些可以直接拿来用的东西？

如果你现在正在考虑在项目里引入 Agent 系统，我建议把文章的方法论浓缩成几条具体的工作准则来执行。

第一条：每次考虑用 Agent 之前，先问自己：一个精心设计的单次 LLM 调用，配上合适的检索和工具，能不能解决这个问题？很多时候答案是可以的，那就不要引入 Agent。Agent 听起来更高级，但复杂度是有实际代价的，体现在延迟、成本、调试难度和系统的不可预测性上。

第二条：如果确实需要多步骤处理，先评估能不能用工作流。工作流里的执行路径是预先确定的，可以精确控制每一步做什么，出了问题也知道从哪里排查。如果你的任务每次需要的步骤都不一样，才考虑真正的自主 Agent。

第三条：如果要用框架，先自己手写一个简单版本来理解底层逻辑。用几十行代码直接调用 API，把核心功能跑通，再决定要不要引入框架。对框架底层原理的错误假设，是很多团队调 bug 最浪费时间的地方。

第四条：把工具设计当成和提示词工程同等重要的事情来对待。每一个工具，都要认真写清楚描述，参数名要能自解释，要写使用示例，要处理边界情况。写完之后，跑真实的输入测试，观察模型在哪里会选错工具或者用错参数，然后修改工具定义，再测试，循环迭代。

第五条：设计好停止条件和人工介入节点。设置最大迭代次数，防止 Agent 无限循环；在执行不可逆操作之前，比如发送邮件、删除数据、进行交易，强制要求人工确认，不要让 Agent 直接执行。

第六条：先在沙箱环境里充分测试，再上生产。Agent 的错误可以复合，而且往往在压力测试时才会暴露一些边缘情况。生产环境里出了问题，影响面和修复成本都比你想象的大。

最后我来做个总结。这篇文章最珍贵的地方，不只是那五种工作流模式的分类框架，而是它背后的工程哲学：在 AI 这个领域，复杂性不等于能力，简单的方案往往更可靠、更可维护、更值得信赖。这个观点在二〇二四年年底需要一点勇气来说出口，因为那时候整个行业都在追捧越来越复杂的 Agent 系统。但一年半过去了，我个人认为市场已经在逐渐验证这个判断：真正在生产环境里跑得好、用户信赖的 AI 系统，往往都是那些设计克制、可观测、容易调试的系统。

用文章的原话来作为今天的结束语：成功不在于构建最复杂的系统，而在于为你的需求构建正确的系统。先从简单的提示词开始，用全面的评估来优化，只有当更简单的方案不足以解决问题时，才引入多步骤的 Agent 系统。
