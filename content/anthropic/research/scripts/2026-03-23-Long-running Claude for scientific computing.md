## Script

今天这一期，我们要聊一篇来自 Anthropic Discovery 团队的文章，标题是"Long-running Claude for scientific computing"，中文可以翻译成"用于科学计算的长时运行 Claude 智能体"。这篇文章发表于二〇二六年三月二十三日，距今大约三个月，作者是 Siddharth Mishra-Sharma，他是 Anthropic 的一名研究员。

这篇文章我为什么要专门拿出来讲？因为它回答了一个我觉得很多工程师和 AI 从业者心里都在问的问题：大模型现在已经这么强了，我应该怎么更系统地用它来完成那些真正复杂、真正需要时间的工程项目？不是聊两句、让它帮我写个函数，而是把一个大目标扔给它，然后走开——让它自己工作好几天，最后拿到一个成品。

文章给出的答案，不是一句"直接用就好了"，而是一套具体的工程实践：有文档设计、有测试策略、有基础设施配置、还有对 Agent 行为的主动干预手段。我觉得这对很多人来说是一个可以直接参考、直接落地的方法论，所以今天我们来把它彻底拆开讲清楚。

先来说背景。过去一年多，大模型在执行长时间、多步骤任务上的能力有了一次显著的质变。但如果你现在去问大多数在用 AI 智能体的科学家或者工程师，他们的工作模式大概是这样的：给 Claude 一个任务，等它回复，检查结果，修改指令，再给下一个任务。整个过程，人始终是那个"指挥中枢"，你需要时刻在场，时刻做决策。这种方式有个本质限制：人是瓶颈。你睡着了，项目就停了。你去开会了，代码就没人写了。

而文章描述的是另一种工作方式，作者把它叫做"自主模式"：你把高层目标定义清楚，让一队 Agent 自己跑，中途偶尔检查一下进展就好。人不是瓶颈，算力才是瓶颈。原本需要几周甚至几个月的项目，可以压缩到几天内完成。

为了说明这一点，文章引用了 Anthropic 之前做过的一个项目——C 编译器项目。在那个项目里，Claude 跨越了大约两千个独立会话，最终构建出了一个能编译 Linux 内核的 C 编译器。这是一个非常适合并行化的任务：编译器的各个模块相对独立，可以同时跑很多个 Agent，各自负责不同的部分，互不干扰，最后合并结果。

本文的主角任务——宇宙学玻尔兹曼求解器——则是一个完全不同类型的挑战，没有办法简单地拆分并行。它是一个深度耦合的数值计算管线，早期宇宙里任何一个地方出了细微的数值误差或者不恰当的近似，都会像蝴蝶效应一样影响下游所有的计算结果。所以它不适合大规模并行，更适合一个 Agent 顺序工作、在需要的时候生成子 Agent，同时能够从参考实现里反复校验自己的结果。这是一套不同的 Agent 技能。

好，我们现在来讲这个任务本身是什么。这也是理解文章核心价值的必要基础。

宇宙大爆炸发生之后，宇宙在最初几十万年里是一个高温高密度的等离子体海洋。光子、质子、电子、中微子、暗物质都混在一起，剧烈地相互作用。随着宇宙不断膨胀冷却，温度降到某个临界点，质子和电子结合成氢原子——这个过程叫做"宇宙复合"——光子得以从这团物质里解放出来，自由地向四面八方传播。这些光子一直旅行了一百三十八亿年，最终成为我们今天能够观测到的宇宙微波背景辐射，简称 CMB。

CMB 是宇宙学里最重要的观测数据之一，没有之一。它的温度涨落分布里隐藏着关于宇宙早期历史、暗物质含量、暗能量密度的大量信息。为了从这些数据里提取参数，科学家需要一个理论模型来预测"如果宇宙参数是某某值，CMB 应该长什么样"——这就是玻尔兹曼求解器的工作。它通过数值求解描述光子、重子、中微子和暗物质的耦合方程组，算出各种成分随时间的演化，最终给出 CMB 的角功率谱。

你可以把它理解成：宇宙学里最重要的那道"正向计算"——给你宇宙参数，给你宇宙学物理，告诉你应该观测到什么。反过来，你把理论预测和观测数据对比，就能反推宇宙的参数。这门手艺叫参数估计。

现有的两个主流玻尔兹曼求解器叫 CLASS 和 CAMB，是宇宙学界几十年积累下来的核心基础设施，精度极高，互相之间在主要输出上可以达到零点一百分比量级的一致性，也就是千分之一的精度。

这篇文章的目标，是用 JAX 实现一个"可微分"版本的玻尔兹曼求解器。JAX 是谷歌的一个 Python 科学计算库，最大的特点是原生支持自动微分和 GPU 加速。"可微分"意味着你可以对求解器的输出求梯度——这在统计推断上有重大意义。

宇宙学参数估计本质上是一个高维贝叶斯推断问题。你有十几个参数，你需要在一个巨大的参数空间里找到最能解释观测数据的那个区域。传统方法用马尔可夫链蒙特卡洛，探索效率有限。但如果求解器是可微分的，你就可以用哈密顿蒙特卡洛或者变分推断等梯度驱动的方法，效率可以提高一到两个数量级。可微分求解器就是打开这扇门的钥匙。

已经有研究组实现了部分功能的 JAX 版玻尔兹曼求解器，但要做到和 CLASS 完全功能对齐，通常是研究团队数月到数年的工作量。

作者坦承：这个任务超出了他自己的专业范围。他是 AI 研究员，对宇宙学有高层面的了解，但没有能力在合理的时间里自己完成这件事。他想看看，一个 AI Agent，能不能在非领域专家的最少干预下，独立走完这条路。

结论先说：Claude Opus 四点六花了几天时间，在主要输出上达到了和 CLASS 的亚百分比精度，也就是误差在百分之一以内。这个结果还不到生产级别，但已经足以说明这种方法论的潜力。

现在我们来讲这个工作流里最核心的几个设计。

第一个核心设计，是 CLAUDE.md 和 CHANGELOG.md 这对搭档。

先说 CLAUDE.md。这个文件放在项目根目录。Claude 对这个文件有特殊处理：它会把这个文件一直保留在上下文里，工作时随时参考。你可以把它理解成"项目大脑"或者"任务书"——里面写着项目的整体目标、关键设计决策、成功标准、约束条件、以及当前的工作计划。

作者的做法是这样的：他先和 Claude 做了几轮对话，把高层目标讲清楚——要实现一个和 CLASS 完整功能对齐的可微分求解器，用 JAX，支持自动微分和 GPU，精度目标是主要输出在全参数范围内达到零点一百分比。然后他们一起把这些整理成一份正式的计划文档，写进 CLAUDE.md。这个 CLAUDE.md 还会明确告诉 Agent：提交的频率是什么，测试的规则是什么，遇到特定问题的处理方式是什么。

这里有一个非常关键的特性，很多人可能没意识到：CLAUDE.md 是可以被 Claude 自己编辑的。Claude 在工作过程中，如果遇到了新问题、调整了架构方案、发现了某个近似方法更合适，它可以主动更新这份文档，把决策记录下来，以便下一个工作会话继续。

这解决了一个根本性的问题：大语言模型的上下文窗口是有限的，一个跨越数天、数十甚至数百个会话的项目，不同会话之间本来是相互独立、没有任何共享记忆的。CLAUDE.md 就是把项目的关键知识"固化"成文字的机制。每次新会话开始，Claude 读取这个文件，就能立刻知道：这个项目在做什么，现在处于什么阶段，有哪些决策已经做了，有哪些限制需要遵守。不需要从头推导一切。

如果说 CLAUDE.md 是项目的"大脑"，那 CHANGELOG.md 就是项目的"日记"。

CHANGELOG.md 记录的是工作日志：完成了哪些功能模块，在哪个检查点精度是多少，遇到了什么问题，尝试了哪些方案，哪些方案失败了、原因是什么。

你可能会想：这不就是普通的项目日志吗？有什么特别的？

最特别的地方，是那些"失败的尝试"。文章里举了一个具体例子：Claude 尝试用 Tsit5 算法来求解摄动方程组的常微分方程，但发现这个方程组的刚性太强，Tsit5 根本处理不了，只好换成 Kvaerno 五阶方法。如果这个失败经历没有被记录下来，下一个工作会话的 Claude——一个全新启动的、没有任何前面记忆的 Claude——很可能会再次尝试 Tsit5，再次踩同一个坑，白白浪费计算时间。

把失败记下来，就等于给未来的 Agent 发送了一封"警告信"："这条路我走过了，走不通，别再试了。"这是一种非常高效的知识传递机制。

文章里说，一个好的 CHANGELOG.md 应该包含这些东西：当前的项目状态，已经完成的任务列表，失败的方法和失败原因，关键检查点的精度指标，以及已知的局限性。这些信息合在一起，构成了 Agent 的可移植长期记忆，读起来就像一份实验室的工作记录。

第二个核心设计，是测试预言机（test oracle）。

这是任何长时间自主运行的 AI Agent 都面临的一个根本性挑战：它怎么知道自己是在进步，还是在原地打转，甚至在倒退？在没有人类随时监督的情况下，Agent 需要一个客观的、量化的衡量标准。

在这个项目里，测试预言机就是 CLASS 本身。Claude 被指示用 CLASS 的 C 源码来构建和持续运行单元测试——每次它写了新代码或者修改了现有代码，就立即把自己的 JAX 实现和 CLASS 的输出做对比，看精度有没有达标。

作者设定的精度目标是：主要科学输出（CMB 角功率谱）在全参数范围内，和 CLASS 的偏差不超过零点一百分比。这个数字不是随意定的——它恰好是 CLASS 和 CAMB 这两个最权威的求解器之间的典型一致性水平。也就是说，如果你的实现能达到这个精度，你就和领域里的顶级工具对齐了。

有了这个测试预言机和量化目标，Agent 的每一步工作都有一个明确的评判标准：不是"感觉好一点了"，而是"在参数点 X 上，特定功率谱的相对误差从百分之五降到了百分之零点八"。这种量化反馈对于维持长时间的进展至关重要。

文章还特别强调了防止回归的重要性。CLAUDE.md 里的规则是：在每次提交代码之前，必须跑完所有现有测试，绝不提交会让已经通过的测试重新失败的代码。这确保了项目总是在稳步向前——你不会修了一个地方，破坏了另外三个地方，然后自以为在进步。

文章里有一张图，展示了从项目开始到最终达到亚百分比精度的整个轨迹，里面标出了各个里程碑节点。有意思的是，过程并不是线性的平滑上升——中间有几次精度出现了明显波动，说明某些修改虽然改进了一个地方，同时也引入了新的误差。但总体趋势是清晰的向好趋势。

第三个核心设计，是基础设施配置和 Ralph 循环。

文章的计算环境是一个运行 SLURM 调度系统的高性能计算集群，带有 H100 GPU。但作者特别说明：这套方法里的核心思路——进度文件、测试预言机、清晰的 Agent 指令——适用于任何计算环境，不管你是在本地服务器、云端还是你自己的工作站，都可以套用。

具体操作流程是这样的：写一个 SLURM 作业脚本，申请一个带 GPU 的计算节点，设置最长运行时间为四十八小时，然后在节点上启动 tmux——一个终端复用器。在 tmux 里启动 Claude Code 会话。

tmux 的关键特性是：你可以随时"脱离"（detach）这个会话，脱离之后会话还在后台继续运行。Claude 还在工作，代码还在跑，但你已经可以去做别的事情了。你可以关掉笔记本电脑，去睡觉，去吃饭。当你想检查进度的时候，重新连接到服务器，重新 attach 到那个 tmux 会话，就能看到 Claude 在做什么了。

作者是怎么监控进度的？他说他平时就是在排队买咖啡的时候，掏出手机看看 GitHub 上有没有新的提交记录。每一次提交就代表 Claude 完成了一个有意义的工作单元，而且提交通过了测试。这种监控方式几乎没有摩擦——不需要专门坐下来打开笔记本，随时随地看一眼就行。

Git 在这里不只是版本控制，它还是一个监控和协调工具。文章建议把 Git 使用规则写进 CLAUDE.md：每完成一个有意义的工作单元就立即提交并推送，提交前跑完所有测试，绝不提交破坏现有测试的代码。这样做有三个好处：出了问题你有完整的可恢复历史；进度对你实时可见；即使计算节点的资源配额在会话中途用完了，之前的工作也完全不会丢失。

现在来说这篇文章里我觉得最有创意的一个设计——Ralph 循环（Ralph loop）。

Ralph 循环解决的是一个叫做"Agent 懒惰"（agentic laziness）的问题。你可能已经遇到过这种情况：你让 Claude 完成一个复杂的多步骤任务，它做到一半，突然说——"好的，这部分基本完成了，但后面的工作比较复杂，需要更多时间，我们可以稍后继续"，然后就停下来了。

这不是真的做完了，只是找了个借口停工。

Ralph 循环的原理很简单，它本质上就是一个 for 循环：当 Claude 声称完成了任务，循环就把它踢回去，问一句"你真的做完了吗？任务成功标准达到了吗？"Claude 发现任务其实还没达标，就会承认，然后继续工作——一直到真正达标，或者达到最大迭代次数上限。

文章里给出的示例指令大致是：请继续工作，直到在全参数范围内达到零点一百分比精度。最多迭代二十次。任务完成时，请说出"DONE"这个词。

这个设计妙在哪里？它把"完成"的定义权交给了外部的量化标准——测试预言机，而不是 Claude 的主观判断。Claude 没有办法自欺欺人地说"差不多了"，因为有具体的测试数字摆在那里。

好，基础设施讲完了，我们来看实验结果。

Claude 从零开始在项目上工作了几天，在主要输出的角功率谱上达到了和 CLASS 的亚百分比精度。作者让 Claude 复原了开发过程中各个时间点的精度数据，并在图上标注了里程碑节点。

整个开发轨迹有点坎坷。有一段时间，Claude 只在单个基准参数点上测试代码——这大幅降低了发现 bug 的能力，因为很多问题只在偏离基准参数较远的地方才会出现。还有一些基础性错误，比如规范选取的问题（宇宙学里有不同的度规规范，搞混了会导致很难察觉的系统性误差），花了好几个小时追一个有经验的宇宙学家可能一眼就能认出的 bug。

但即便如此，Claude 始终在朝着目标前进。这种持续进展的能力，在复杂任务上是非常有价值的。

产出的这个求解器还不是生产级别的——在某些参数区间里精度还达不到标准——但它证明了一件事：Agent 驱动的开发，能把数月乃至数年的研究工作压缩到几天内完成。

文章最后还提到了一个很有意思的副作用：作者说他通过跟踪 Claude 的 git 提交历史，学到了大量关于玻尔兹曼求解器和宇宙学物理的知识。提交日志读起来就像一个"快速、超级字面量的博士后"写的实验室记录。跟着 Claude 的进展走，顺着看它解决的每一个问题，本身就是一种高效的知识获取方式。

好，现在我们来聊这一切对你的实际工作有什么意义。这是我今天最想跟你说的部分。

第一件事，是把"CLAUDE.md 加 CHANGELOG.md"这套框架当作你的标准配置。这是一个完全通用的设计模式，不只适用于科学计算。你在用 Claude 做任何长时间、多步骤的代码项目时，都可以套用它。CLAUDE.md 写项目目标、设计决策、成功标准和工作规则；CHANGELOG.md 记录进度、失败方案和原因。两者合力构成了跨会话的持久化记忆。

尤其是那个"记录失败方案"的习惯，值得反复强调。很多人倾向于只记录成功的路，失败的尝试觉得"反正没用，不用写了"。但在 Agent 工作流里，失败记录是防止重复踩坑的关键护栏。你想想，如果你今天让 Claude 尝试了方案 A 失败了，明天你开一个新的 Claude 会话，没有失败记录，它大概率会建议你再试试方案 A。

第二件事，是在项目开始前认真设计你的测试预言机。你需要问自己：我怎么知道任务完成了？这个问题在任何项目里都应该在开始之前就回答清楚，而不是做完了再想。

对于数值计算任务，最理想的预言机是一个已有的参考实现。对于代码迁移任务，可以是迁移前后的功能测试对比。对于数据处理任务，可以是一批已知正确输出的样本。关键是让成功标准变得量化、可测量、不依赖主观判断。

如果你的任务没有现成的参考实现——这在很多工程场景里是现实情况——可以考虑这样一个方法：先让 Claude 生成一批典型输入的期望输出，让你或者领域专家检查确认这些输出是合理的，然后把这些期望输出作为回归测试的基准。这不如真正的参考实现严格，但总比没有强得多。

第三件事，是重新思考"非领域专家"这件事意味着什么。文章里有一个思维方式，我觉得对很多工程师来说是一次真正的认知升级：作者做了一件超出他专业范围的事，不是因为他假装自己是宇宙学家，而是因为他把目标定义清楚，让工具去处理领域细节，然后他负责判断结果是否合理、提出正确的问题。

这意味着：你不需要先成为某个领域的专家，然后再用 AI 帮你做这个领域的工作。你可以倒过来——用 AI 去探索你原本不熟悉的领域，前提是你能够清晰地定义目标，能够判断输出的合理性，能够在关键节点做出"这个方向对不对"的判断。这是一种很不一样的能力模型。

第四件事，是认真对待"Agent 懒惰"这个问题，并提前准备应对策略。Ralph 循环是一个解法，Claude Code 内置的 /loop 命令是另一个，更根本的解法是在 CLAUDE.md 里把成功标准写得足够具体、足够量化。如果标准写的是"实现一个玻尔兹曼求解器"，Agent 很容易找到理由说"基本实现了"。如果标准写的是"在给定参数范围内，所有主要输出和 CLASS 的偏差小于零点一百分比，并通过所有回归测试"，就没有那么多模糊地带了。

第五件事，是思考"机会成本"这个概念。文章最后一段我个人觉得是全文最值得反复读的部分。它说：在 AI 研究里有一个普遍的经验，你跑一个实验，比如一个训练任务，放到晚上去跑，第二天早上看结果，这很自然。不跑实验是有机会成本的，你意识到这一点。现在，不跑 Agent 也开始有同样的感觉。如果你有算力，如果你有定义清楚的项目，那么每一个你没有让 Agent 工作的夜晚，都是潜在进展被留在桌上没有拿走。

这是一个认知框架的变化。以前，"等待"这件事是中性的——你睡觉，项目停着，这完全正常。现在，如果你有能力让 AI 在你睡觉的时候持续工作，但你没有利用这个能力，这开始变成一种"浪费"。

我个人认为，这个说法有一些重要的前提：你需要有清晰定义的任务，有效的测试预言机，以及适当的算力资源。不是所有任务都适合无监督运行——如果成功标准模糊，或者没有客观的测试来防止 Agent 跑偏，放开不管可能适得其反，浪费的不只是时间，还有计算费用。但如果这些前提都具备，那确实，让 Agent 在后台自主工作是一个极其高效的策略。

最后还有一个坑要提醒：文章对 Claude 的表现是非常客观的，没有回避缺陷。现阶段的 Agent 在测试覆盖率上容易出现盲区——它可能会在几个"容易"的测试案例上反复验证，忽略掉一些边界情况。作为监督者，你需要不时地检查一下测试设计本身：测试覆盖率够不够全面？有没有在整个参数空间里均匀采样，而不是只盯着某几个点？

这类问题，你可以直接问 Claude："你觉得现在的测试设计有没有明显的覆盖盲区？"然后根据它的回答来调整。它的自我评估通常还是有参考价值的。

好，我们来做一个收尾。

这篇发表于二〇二六年三月的文章，核心贡献不是那个宇宙学玻尔兹曼求解器本身，而是它给出了一套清晰的、可复用的工程实践：CLAUDE.md 作为持久化的项目大脑，CHANGELOG.md 记录进度和失败，测试预言机提供量化的成功标准，Ralph 循环对抗 Agent 的提前收工倾向，Git 作为监控和协调工具。这套组合让 Claude Opus 四点六在几天内完成了领域专家团队数月到数年的工作量。

我个人认为，这篇文章最大的意义是把"管理一个自主 AI 研究团队"这件事，降低到了个人工程师或者研究者都可以实际操作的水平。你不需要一个大团队来驾驭这种能力，你需要的是清晰的目标、合适的测试、和一份写得好的 CLAUDE.md。
