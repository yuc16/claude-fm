## Shownotes

Anthropic 研究者 Laura Luebbert 团队发布了一项关于科学 AI agent 的重要研究：用 VirBench 基准测试发现，当前最强模型在无工具辅助的病毒序列检索任务中，准确率仅有 16.9% 到 91.3%，且同一问题三次问得到三个差异极大的答案。当他们为 agent 配备 gget virus 这个确定性检索工具后，所有模型准确率均突破 90%，最高达 99.7%。这揭示了一个对所有 AI 工程师都适用的核心洞见：在高准确性要求的领域，好的工具层可以对冲模型能力的差距，让模型的选择变得不那么重要。

- 生物数据基础设施像为人类设计的老城窄巷，AI agent 再强也难以可靠导航——瓶颈不在推理，在确定性执行层的缺失
- VirBench 测试：Claude Sonnet 4 对同一埃博拉序列检索题三次分别返回 106、15、5 条，正确答案是 266 条
- 序列数据不完整导致进化树推算的最近共同祖先时间出现百年偏差：1922 年 vs. 正确的 2014 年
- gget virus 确定性检索层加入后，GPT-5.5 准确率达 99.7%，模型间性能差距大幅缩小
- 核心结论：加上确定性检索层，模型的选择变得不那么重要了——便宜模型加正确工具胜过昂贵模型裸奔
- 对 AI 工程师的启示：高准确性任务中，先在模型和数据源之间加入确定性工具层，而不是换更强的模型

## Script

欢迎收听 Claude FM，我是主播。

今天我们来聊一篇非常值得深读的 Anthropic 研究文章，英文标题是 Paving the way for agents in biology，中文翻译过来是"为生物学领域的智能体铺路"。这篇文章发表于二〇二六年六月八日，到今天才过了五天，是非常新鲜的内容，算是第一时间把它端上来给你听。

文章的主要作者是 Laura Luebbert，她和来自布罗德研究所、NCBI、柏林夏里特医学院以及多所大学的合作者共同完成了这项工作。文章的署名方式很有意思——Laura 写了这篇文章，但她特别注明这是基于 Ferdous Nasri、Sarah Gurev、Patrick Varilly 等多位研究者的共同成果，这种集体贡献的呈现方式在学术界并不多见。

我为什么要给这篇文章专门做一期节目？因为它触及了 AI agent 应用里一个非常关键、却很少被系统讲清楚的核心矛盾——我们花了大量精力让语言模型变得更聪明，但真正制约 agent 能不能用、好不好用的，往往不是模型有多聪明，而是它工作的那个数据环境有多混乱、多不友好。

这个道理说出来好像显而易见，但这篇文章做了一件非常有价值的事：它把这个模糊的直觉，变成了一个用严格基准测试量化过、有真实疫情案例支撑、有工程解决方案完整验证的具体问题。对于任何在做 agent 应用的工程师来说，这篇文章的思路都直接适用，不管你做的是生物学还是法律、金融、还是任何其他垂直领域。

好，我们先从一个你可能没想到的地方开始讲起：意大利的山顶小镇。

文章用了一个非常妙的比喻。作者说，把 AI 智能体放进生物学的数据基础设施里，就像开着一辆现代汽车穿越一座在汽车发明之前就规划好的老城区。这座城市可能非常精美，当年的规划者也很有心思，但满是狭窄曲折的小巷，根本不适合现代车辆通行。

你可能会想：这不就是路窄车太大的问题吗，换辆小一点的车不就行了？

但作者要说的不是这个。她说，无论你的车多强大，如果街道太窄、弯道太急、路线完全依赖本地知识，你就根本开不动。问题不在车，在路。这是她给整篇文章定下的核心比喻，后面我们会反复回到这里。

这不是在打比方，这是生物数据的真实现状。格式五花八门，数据库散落各处，检索脚本各写各的，没有统一标准。每个数据库有自己的标识符系统、自己的筛选逻辑、自己对接口的理解，而且很多关键操作逻辑只存在于浏览器界面里，根本没有对应的编程接口可以调用。

为了让这个问题更具体，我们来对比一下编程领域的情况。为什么软件领域的 coding agent，也就是写代码的智能体，发展得那么快？代码生成、自动修 bug、解决 GitHub issue——这些已经有非常成熟的工具和工作流了。

原因在于，软件本身天然就提供了结构化的数字工作流和可靠的接口。你让 agent 修一个 GitHub issue，它生成的代码跑一遍测试，对错一眼就看出来了。这是一个清晰的、可验证的、可重复的反馈信号。

但生物学呢？数据检索的基础设施脆弱、异构，高度依赖人工操作，缺乏可靠的编程接口。更要命的是，生物学里很少有像"代码编译通过"这样既简单又有意义的验证信号。结果可能看起来完全合理，但其实早就错了。而且你不知道它错了，要等到下游的结论跑偏之后才会发现——那时候可能已经是一篇论文发出去了，或者一个公共卫生建议发布了。

这里有一个非常重要的认知：agent 的瓶颈，不只是推理能力不够，还有确定性执行层的缺失。一个科学家可以清楚地表达意图——比如，帮我找出所有已知的人类激酶里含有这个特定结构域的蛋白，把它们的三维结构全部拿来——但 agent 往往缺乏一个可靠的、可重复的方式来访问包含这些信息的数据库。

文章在这里提到了一个你可能认识的人：Andrej Karpathy。他是前特斯拉 AI 总监，深度学习领域的重要人物，也是把 AI 概念普及给大众的核心贡献者之一。几个月前，他做了一个关于 AI 时代软件的演讲，抱怨了一件让他又好气又好笑的事。

他用 AI 把一个小 web 应用的代码写得非常顺畅，但当他想要真正把这个应用上线——需要接入用户认证、接支付系统、做部署——他发现自己在各种浏览器控制台里点来点去，愣是耗掉了整整一个星期。Karpathy 说："代码是最简单的部分！大部分工作都在浏览器里点点点。"文档不停地告诉他去这个 URL、点那个下拉菜单。他的结论是：没有人应该这样工作。我们必须为智能体而设计。

Laura 把这个故事拿来做呼应，说得非常到位：Karpathy 在软件世界里刚刚体验到的那种痛，生物学研究人员其实已经忍受了很多年了。在一个充满异构信息、隐性约定、只能靠人类在浏览器里手动操作的环境里，想让智能系统可靠地工作，有多难受。

其实在 AI agent 诞生之前，生物信息学界就已经在尝试解决这个问题了，涌现出了一批工具：Biopython、BioPerl、BioJulia、Entrez Direct、BioMart、gget，等等，这些都是把生物数据从浏览器界面搬到可以直接计算的地方的努力。但问题是，生物数据不住在一个有统一接口的单一数据库里，而是散落在一张极其复杂的路网上，每条路都有自己的标识符系统、格式标准、筛选逻辑，有的可以编程访问，有的根本不行。

病毒学的数据访问尤其是个硬骨头。研究人员在做疫苗设计、开发诊断工具、构建蛋白质模型训练数据的时候，往往需要先从一个叫 NCBI Virus 的数据库里拿病毒序列。NCBI Virus 是美国国立生物技术信息中心维护的病毒序列数据库，整合了 GenBank、RefSeq，以及背后由美国、欧洲、日本联合维护的国际数据共享生态 INSDC，背后还有一个叫 Pathoplexus 的系统在做实时数据同步。整个系统提供一个可以在网上搜索的界面，研究人员需要在这个界面上手动设置各种复杂的筛选条件。

在很多病毒学实验室里，NCBI Virus 的数据集策划说明，是以长长的筛选条件清单形式在研究者之间口耳相传的。一个老研究员会告诉新来的博士生：你要先进这个界面，把这几个选项都勾上，然后在这里填这个值，这个字段要注意用这个约定，不能用另一种写法——全靠人传人，没有任何自动化。这正是 Karpathy 抱怨的那种浏览器点击工作流，只不过在生物学里这个问题已经存在几十年了。

为什么要把这件事自动化这么难？作者做了一个很有说明性的解释。如果你想要所有二〇二五年发布的、含有表面糖蛋白的 SARS-CoV-2 序列，一个经验丰富的病毒学家在浏览器里几次点击就能搞定。但用编程方式实现，可能需要同时调用 NCBI 的多个 API——REST 接口、Datasets 接口、E-utilities 接口——还要分页获取结果，还要对不同来源的标识符进行协调，还要下载几百个 GB 的数据之后再在本地进行二次过滤。这根本不是一个简单的数据库查询，是一个需要大量领域知识和工程技巧才能正确实现的任务。

接下来，文章给了一个非常震撼的真实背景，让这个问题从抽象的工程痛点变得具体而紧迫。

二〇二六年五月，刚果民主共和国爆发了由 Bundibugyo 病毒引发的埃博拉疫情。五月十四日，金沙萨的 INRB 研究所检测了十三份血液样本，第二天就确认八份呈阳性，随即宣布疫情爆发。到五月二十九日，世卫组织已经报告了超过一千例确诊和疑似病例，两百多人死亡。与此同时，研究人员也生成了首批近乎完整的疫情基因组，确认这是一次新的病毒溢出事件，不是之前毒株的继续传播。

面对这批新基因组，公共卫生官员必须快速回答三个紧迫的问题。第一，这次疫情的病毒和历史上的埃博拉病毒有多大差异？第二，现有的诊断工具——检测试剂——还能识别它吗？第三，现有的治疗药物还能保护患者吗？

要回答这三个问题，研究人员必须把新基因组和历史上积累的埃博拉基因组进行比对，而这些历史数据就存在 NCBI Virus 和 Pathoplexus 里。但关键的第一步——从数据库里拿到正确、完整的历史数据——根本没法轻松自动化。研究人员还是得在网页界面里手动点，手动复现复杂的筛选条件，然后祈祷得到的数据集是完整且正确的。在疫情每分每秒都在发展的时候，这是一种巨大的阻碍。

好，现在我们进入文章最核心的实验部分。

Laura 和她的团队开发了一个叫做 VirBench 的基准测试。VirBench 包含一百二十道真实场景下的病毒序列检索题，横跨四十种病原体，每道题都有经过人工核验的正确答案。这些题目来源于真实的病毒学工作场景：病毒监测、诊断工具设计、蛋白质模型训练数据构建，覆盖了研究者在实验室里真实会遇到的检索需求。

我来给你读一道具体的题，感受一下难度。题目是：请从 NCBI 检索 TaxID 为三〇五二四六二 的 Orthoebolavirus zairense，也就是扎伊尔埃博拉病毒，的病毒序列，条件是：宿主为人类，采样地点在非洲，采样时间在二〇一四年一月一日到二〇一四年六月二十日之间，序列最短长度为一万五千两百个碱基，最多有一千九百个不明碱基，也就是碱基序列里标注为 N 的位置，并排除实验室传代的样本。

你可能会觉得：这不就是设几个筛选条件吗？听上去挺直接的。

但现实是，真正去编程实现这些条件，你需要协调 NCBI 的三套不同 API，分批次分页获取大量数据，对不同来源的标识符做协调处理。更关键的是，某些条件在 API 里根本没有直接的对应参数——比如"排除实验室传代样本"这个条件，在 API 里怎么实现？你必须先全量拉取记录，然后根据记录里的元数据字段去做本地判断，而这些元数据字段的填写方式本身就不统一，在不同病毒不同时期的规范里可能有不同的约定。这不是一个简单的查询，是需要大量专业知识和工程技巧才能正确实现的任务。

他们测试了当时几个主流的科学研究智能体系统：Claude Sonnet 四和 Claude Opus 四点七，开源版本的 Biomni，Edison Analysis，以及 GPT 五点二 pro 和 GPT 五点五。总共做了三百六十次运行。

结果怎么样？平均准确率从最低的百分之十六点九到最高的百分之九十一点三，差距相当大，而且越新的前沿模型表现越好——这个趋势是真实存在的。

但这里有一个关键问题：对于科学数据检索任务，百分之九十一的准确率依然不够用，门槛实际上是百分之百。你漏掉或者拿错一条序列，可能让你认为某个诊断工具能覆盖当前流行毒株，但实际上不能；或者让你推算出疫情开始的时间比真实时间早了好几个星期，或者晚了好几个星期，从而影响溯源分析和应对策略。在公共卫生决策里，这种误差的代价可以是人命。

更让人头疼的是变异性问题。同一个模型，面对同一道题，问三次，可能给出三个截然不同的答案。这种不稳定性彻底破坏了科学工作流所需要的两个核心属性：准确性和可重复性。

文章给了一个具体的数字，让你感受一下有多离谱：Claude Sonnet 四面对那道扎伊尔埃博拉病毒序列检索题，三次运行分别返回了一百零六条序列、十五条序列、五条序列，而正确答案是两百六十六条。用完全相同的 prompt，三次跑出来的数字差这么多。

这种不稳定性对下游分析的影响有多严重？文章做了两个非常有说服力的演示。

第一个是系统进化树分析。研究人员用这些不同的序列集构建了系统进化树。你可以把系统进化树理解成一张病毒家族的族谱，它显示了疫情中不同病毒样本之间的进化关系，可以追踪传播路径，也可以推算共同祖先的时间。

进化树有一个非常重要的量叫做最近共同祖先时间，英文缩写是 TMRCA。直白来说，这就是疫情的"起源时间点"——推算这批病毒样本最早从哪个共同祖先分支出来的时间。这个时间直接影响你对疫情起源时间和地点的判断，也影响你判断病毒在被发现之前已经悄悄传播了多久。对于疫情响应和后续追踪来说，这是一个非常关键的数字。

用人工精心策划的 NCBI Virus 序列集构建的进化树，得到了二〇一四年一月的 TMRCA，和已有的文献报告完全吻合，对应的是二〇一四年那次埃博拉疫情的起源时间，百分之九十五的置信区间在一月二十七日到三月十四日之间。

然后看看用 Sonnet 四三次检索结果构建的树。

其中两次的序列集明显不完整。最荒谬的一次，推算出来的 TMRCA 是一九二二年——一百年前！这意味着进化树算法在数据严重不完整的情况下，反推出了一个完全不符合生物学事实的共同祖先。如果有人用这个结果去发表论文或者汇报疫情，结论是这个病毒的祖先起源于一百年前，这在科学上是完全错误的，在公共卫生应对上也毫无意义。另一次序列集乍看上去还算合理，但实际上没有拿到来自几内亚的序列，把 TMRCA 推到了二〇一四年四月，把推算的疫情起始时间错后了将近三个月。一个早三个月、一个晚三个月，说的可能是两次完全不同的疫情传播事件。

第二个演示是药物靶点分析。研究人员检索了埃博拉病毒的糖蛋白序列，来研究两种抗体疗法的靶点区域：maftivimab 和 MBP 一三四。这两种药物是世卫组织在本次疫情中优先推荐的治疗候选方案，都是针对扎伊尔埃博拉病毒开发的抗体药物。

研究者想知道：在历史序列里，病毒在这些抗体靶向的区域上发生过哪些突变？这类分析可以帮助判断，随着病毒持续进化，这些药物是否还能保护患者。如果靶点区域的突变率很高，说明药物可能面临逃逸风险；如果基本保守，说明药物的持久性更好。这是一个直接影响治疗决策的分析。

用 Sonnet 四三次运行的结果做分析，第一次接近人工查询的结果，第二次漏掉了大部分突变位点，第三次又给出了一组完全不同的位点。三次运行，三种对药物靶点可变性的不同印象，可能导致对药物有效性做出截然不同的结论。如果你把其中某一次的结果拿去做决策——比如告诉临床医生这种药物的靶点区域基本稳定——但实际上是因为你漏了大部分突变位点，那这个结论是错的，代价可能是患者的生命。

这两个案例共同说明了文章的核心观点：在生物学里，看起来无关紧要的数据检索选择，可以直接改变生物学结论。更危险的是，错误的答案往往看起来挺合理，你不会一眼就看出问题所在。而且序列检索只是一个更长工作流的第一步，错误会被放大并传递到所有下游分析里，最后以一个听起来很有说服力的结论出现，但根基已经腐烂了。

那这些失败的根源具体在哪里？作者做了详细的失败模式分析。

记录数量特别多的病毒，比如流感 A、HIV-1、SARS-CoV-2，智能体在检索大量结果时经常会中途停止分页，或者在后续过滤步骤中出错，导致最终数据集被严重截断或污染。还有一个难点是元数据字段的含义高度依赖上下文和约定——比如"部分基因组"和"完整基因组"的区分，在不同病毒、不同时期的规范里含义不一样，机器很难正确推断处理方式。而且随着查询条件越来越复杂，尤其是超过三四个同时作用的筛选条件，性能就会快速下降。

智能体通常是理解了任务的，问题在于，它们缺乏一个机器可执行的、可验证的、可重复的方式来真正完成这项任务。答案可能看起来合理，实际上早就偏了。

这就引出了文章提出的解决方案：gget virus。

Laura 和她的团队与 NCBI 的研究人员合作，在 gget 工具包里开发了 gget virus 这个专门模块。gget 是 Laura 之前开发的一个生物信息学工具包，gget virus 是专门针对病毒序列检索的新功能。

这个工具的核心思想非常清晰：提供一个确定性的检索层。所谓"确定性"，就是给定同样的输入，无论运行多少次，永远得到同样的输出。这不依赖语言模型的随机性，不依赖模型临时推断 API 文档的含义，而是通过精确的工程实现来保证正确性。这是一个工程问题，不是一个模型问题。

但实现起来比想象中难得多。NCBI Virus 实际上是一个覆盖多个底层资源的门户，背后是美国、欧洲和日本联合维护的国际序列数据库。回答一个看起来简单的查询，往往需要从多个地方拼凑信息。

gget virus 需要协调 NCBI 的三套不同 API——REST 接口、Datasets 接口、E-utilities 接口——来精确复现网页界面的行为。它会判断哪些过滤条件可以通过现有的 API 直接实现，哪些必须先把数据拿回来再在本地进行检查，因为网页界面暴露的某些筛选语义并没有对应的编程端点。对于 SARS-CoV-2 或流感 A 这样数据量极大的情况，它做好了完整的分批获取机制，确保结果完整而不是被随机截断。

当某个过滤条件需要用到存储在另一个数据库里的信息时——比如 GenBank 记录中关于某个序列是否包含特定病毒蛋白的注释——gget virus 会去检索那些额外的记录，用它们来应用过滤条件，并在最终输出里保留这些信息。最后，它返回人类和机器都能读取的标准化输出，并附上详细的操作日志，清楚地说明最终结果是怎么一步一步得出来的。这意味着结果不只是一个数字，而是一个可以审计和复现的过程。

效果如何？给所有被测试的智能体加上 gget virus 工具之后，所有模型的准确率都超过了百分之九十，其中 GPT 五点五的准确率达到了百分之九十九点七。运行间的变异性基本被消除了。而且——这一点非常重要——不同模型之间的性能差距大幅缩小了。

文章里有一句话非常值得记下来："加上一个确定性检索层，使得模型的选择变得不那么重要了。"

我们来仔细品一品这句话。它说明的是：问题的根源本来就不在模型够不够聪明，而在于执行层够不够可靠。当你补上了那个缺失的确定性工具层，不同模型之间的差距自然就消失了。更实际的含义是：一个相对便宜的模型，搭配正确的工具，可以实现和最昂贵的前沿模型差不多甚至更好的结果，而且稳定性更好，可审计性更高，使用成本更低，使用门槛也更低，能让更多研究者用得上——而不是只有有条件用最贵模型的团队才能得到可靠的结论。

文章把 gget virus 的作用比喻成：在原来只有步行小街的老城里，挖通了一条现代高速公路隧道，配上完善的匝道、顺畅的交叉口和明确标注里程碑的出口编号。同样的目的地，但换了一条机器可以驾驶的路。

在这里，文章提出了一个更大的视角：context engine，上下文引擎。gget virus 是其中一个例子。作者认为，我们需要系统性地为生物数据建立这类可靠的、智能体可访问的基础设施。我们希望模型在生成假设、设计实验、推断机制的时候有创造力，但支撑这种创造力的底层——基因标识符、数据模式、检索逻辑、坐标系统、元数据约定、数据访问路径——这一层必须无聊地可靠。用一个简洁的说法：创意归模型，可靠性归工具层。这两层的要求和设计哲学是不同的，不能混在一起。

目前已经有一些类似的努力，比如 ToolUniverse、Edison Scientific 的 Robin、Biomni，以及其他生物医学智能体系统，都在把智能体和生物数据源可靠地连接起来。这是一个正在形成的新方向。

文章还探讨了一个有趣的反面情况：如果模型能力提升得足够快，以至于它们能自己搞定这些混乱的接口——自己协调标识符、自己处理分页、自己从失败中恢复——那 gget virus 这类工具还有存在的意义吗？

作者的回答很务实：即使模型在技术上可以做到，也不意味着每次都应该让它们重新发明轮子。能勉强啃下混乱工作流的模型，可能太贵、太慢、太难审计、太难在监管场景下信任，不适合日常的科学工作。而且即使模型的能力最终让今天的工具变得过时，这个教训对于数据基础设施的设计者依然成立：在设计数据库和 API 的时候，需要把智能体作为用户考虑进去，需要为规模化使用而设计。

好，到了我最喜欢的部分：这对你意味着什么，实际工作里能上手做什么，要避什么坑。

第一个核心认知转变：agent 应用的能力上限，不只取决于你用了什么模型，更取决于你这个 agent 所在的执行环境有多机器友好。如果你在给某个垂直领域构建 agent 应用——法律、医疗、金融、制造、科学研究——你应该先问一个诚实的问题：我这个领域的核心数据查询，是为人类设计的还是为机器设计的？我的数据接口有没有和网页界面一致的编程端点？元数据字段有没有清晰的文档？标识符有没有跨数据源保持一致？过滤逻辑能不能完全通过编程实现？

如果你诚实地回答这些问题，往往会发现你的环境里充满了隐性约定和人工知识，和 NCBI Virus 没有本质区别。这时候的正确反应不是换一个更强的模型，而是先把基础设施梳理清楚。

第二个实践教训是关于高准确性任务的处理方式。语言模型天生有随机性，同一个 prompt 问三次得到三个不同答案，在创意写作里是优点，在结构化数据检索里是灾难。如果你的 agent 工作流里有任何一个步骤是高准确性要求的，你需要在模型和数据源之间加一个确定性层，把这部分逻辑从模型的推断中剥离出来，固化成一个可以精确测试、精确执行的工具函数。这个工具函数的行为必须像数学运算一样：给定同样的输入，永远给出同样的输出。实现这个工具可能需要大量工程工作，但这是一次性的投入，而且带来的是可以信赖的确定性。

第三个教训是关于如何评估你的 agent 系统。VirBench 给我的一个重要启发是：一个好的评估方案不应该只问"模型答对了没有"，还要问"同一个问题问三次，答案一不一致"。可重复性是可靠性的前提条件，而不是可有可无的加分项。如果你在为自己的 agent 系统设计评估方案，我强烈建议把多次运行的一致性纳入评估维度，而不只是看平均准确率。一个准确率很高但变异性也很高的系统，在生产环境里其实并不可靠。

第四个教训，也是我觉得最有普遍意义的：生物学的问题不是特例。凡是智能体被插入一个为人类使用而设计的环境，这种摩擦就会出现。Karpathy 在 web 部署里遇到的问题，科学研究者在数据库里遇到的问题，本质上是同一件事。人类使用的界面里充满了隐性约定、上下文依赖和无法通过编程简单获取的操作逻辑。你的 CRM 系统是这样，你的 ERP 系统是这样，你公司内部的知识库很可能也是这样。

如果你正在某个垂直领域做 agent 应用，我建议你做一个类似 VirBench 的小实验：把你领域里最核心的三到五个数据查询或者操作任务，用你的 agent 各跑三次，比较答案的一致性。如果变异很大，你下一步要做的不是换一个更强的模型，而是给这些任务写一个确定性的工具，把正确答案的逻辑从模型的随机推断中剥离出来。这是性价比最高的改进路径，而且往往能带来比换模型大得多的提升。

最后一点，是给数据平台或者 API 设计者的。如果你参与设计某个数据库、API 或者知识库，现在就可以开始问：我的 agent 用户会怎么使用这个接口？我的 API 暴露的过滤语义和网页界面一致吗？我的元数据字段有清晰的文档和统一的约定吗？我的标识符在跨数据源时是稳定一致的吗？如果这些问题的答案是否定的，你的系统就是下一个 NCBI Virus——未来每一个需要在上面构建 agent 工作流的人都会撞上同样的墙，然后花大量时间写各种补丁和胶水代码。提前把这些问题解决掉，比后来一个个打补丁要高效得多。

好，最后我们来做一个简短的总结。

这篇文章的英文标题是 Paving the way for agents in biology，发表于二〇二六年六月八日。

它的核心贡献是三件事：第一，用 VirBench 这个严格的基准测试，量化地展示了当前最强的科学研究智能体在病毒序列检索上有多不可靠，准确率从百分之十七到百分之九十一不等，而且同一问题三次问答案各不相同。第二，通过埃博拉病毒的案例研究，生动地说明了这种不可靠在真实科学工作流中会导致什么后果——推算出疫情起始时间偏差近百年，对药物靶点的变异分析得出完全不同的结论。第三，开发并验证了 gget virus，一个确定性检索层，把所有模型的准确率提升到九十五百分以上，大幅消除了运行间的变异性，同时缩小了不同模型之间的性能差距。

我个人认为，这篇文章最重要的贡献不是 gget virus 这个工具本身，而是它所体现的设计哲学：确定性归工具，创造力归模型。把这两层明确分开，然后分别用不同的标准去设计和评估，这对于任何想把 AI agent 用于严肃场景的团队来说，都是一个值得反复咀嚼的原则。不要等下一个更强的模型来帮你解决数据基础设施的问题，先把你能控制的那部分做得可靠、确定、可审计——这是现阶段性价比最高的做法。
