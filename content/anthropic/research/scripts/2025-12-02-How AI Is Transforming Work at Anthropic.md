各位听众好，欢迎收听 Claude FM，我是你们的主播。今天这期节目，我们来深读一篇非常值得关注的内部研究报告。

这篇文章的英文原标题是 "How AI Is Transforming Work at Anthropic"，直译成中文就是"AI 正在如何改变 Anthropic 的工作方式"。它发表于二零二五年十二月二日，作者是 Anthropic 自己的研究团队，主导者是研究员 Saffron Huang 和她的几位同事。

在进入正题之前，我必须先说一个重要的时间背景。今天是二零二六年六月十五日，这篇文章距今已经将近七个月了。七个月，在 AI 领域不是一段短时间。就在这七个月里，新一代更强大的模型相继推出，工具的能力又上了台阶，整个行业的状态也在持续演变。这篇研究是基于二零二五年八月的调研数据，当时 Anthropic 最强的模型还是 Claude Sonnet 四和 Claude Opus 四，而这两款模型在今天已经不再是最前沿的选择了。所以你在听这期节目的时候，要把时间坐标记在心里：这是 AI 大规模进入工作场所最关键的一段早期过渡期，是一个充满混沌、充满可能性的历史窗口。它记录的，是这个转变发生时人们真实的感受和反应。

那我们为什么还值得认真花二十五分钟来聊这篇研究呢？理由有三个。

第一，Anthropic 的工程师是全世界最早、最深度使用 AI 编程工具的群体之一。他们今天经历的，很可能就是整个行业工程师群体在未来两到三年内会普遍经历的事情。所谓"看一看最早的采用者，就能看到别人的明天"。第二，这不是一篇泛泛的预测文章，而是基于一百三十二份问卷、五十三次深度访谈和整整二十万条 Claude Code 使用记录的实证研究。它有数据、有方法、有局限性说明，可信度比一般的观点文章高出一个量级。第三，也是最重要的一点：这篇研究没有只说好话。它诚实地面对了一些让人不舒服的问题——AI 会不会让工程师变钝？职场的人际关系会不会因此退化？工程师最终会不会把自己的工作给"自动化"掉了？这种诚实本身就很难得。

好，进入正题。

首先来说背景。Anthropic 此前一直在研究 AI 对整体劳动力市场的影响，研究对象是各种各样的职业群体。但有一天，研究团队想到了一个问题：与其研究别人，不如先研究我们自己。Anthropic 的工程师天然就是 AI 工具最重度的使用者，他们既有最早的访问权限，又在相对稳定的行业里工作，还是推动 AI 转型的亲历者。把镜头转向内部，或许能提供一些关于 AI 如何重塑工作的最早期、最具体的信号。

于是在二零二五年八月，研究团队展开了这次内部调研。他们面向全公司的工程师和研究员发放问卷，最终收到了一百三十二份有效回复，覆盖了公司里研究部门和产品部门的多个团队。同时，他们对其中五十三位愿意进一步交流的员工进行了深度访谈，挖掘数字背后真实的感受和思考。最后，他们还用一个保护隐私的分析工具，从公司内部的 Claude Code 使用日志里分析了二十万条交互记录，比较了二零二五年二月和二零二五年八月两个时间点的变化。

三种方法组合起来，既有主观感受，也有客观行为数据，互相印证，可信度远比单一的问卷或者采访要高。

好，我们来看核心发现。我挑最重要的几个点，一个一个展开讲透。

第一个发现：生产力数字的戏剧性增长。

问卷结果显示，一年前，也就是大约二零二四年八月前后，Anthropic 的工程师在工作中使用 Claude 的比例平均是百分之二十八，他们对 Claude 带来的生产力提升的自我评估是百分之二十左右。

一年之后，这两个数字都发生了惊人的跳跃：使用 Claude 的工作比例上升到了百分之五十九，接近翻倍；生产力提升的自我评估则达到了百分之五十，是一年前的两倍半。研究团队还找到了一个客观指标来印证这些主观感受：Anthropic 全面引入 Claude Code 之后，每位工程师每天成功合并入主干的代码提交数量——行话叫 pull request，你可以理解成"成功被团队接受的代码改动件数"——增加了百分之六十七。这是一个相对客观的产出量指标。

在问卷里还有一个有意思的规律：研究团队问工程师，在用 Claude 做某类任务时，和不用 Claude 相比，时间花费和输出量各有什么变化。几乎所有任务类别里都出现了同样的模式：时间在净减少，但输出量在净增加，而且增幅更大。

举个具体的例子来理解这个模式。假设你每周花十小时在调试上，用上 Claude 之后可能降到了八小时；但这八小时里，你处理的 bug 数量或者排查的问题复杂度，可能是以前的两到三倍。时间省了一点，但产出量大幅提升了。

不过，当研究团队仔细看原始数据，他们发现时间节省并不是均匀的，而是呈现出两端极化的模式——有人大幅节省时间，也有人反而花了更多时间。为什么反而更多？工程师们解释说，有时候他们需要花大量时间去调试和清理 Claude 写出来的代码，或者需要额外的认知投入去理解那些不是自己写的代码逻辑。有一个场景被好几个人提到，就是在 Claude 的帮助下一路往前写，结果不知不觉走进了一个复杂的代码死角，然后需要花更多时间找出路。所以，并不是"用了 AI 就一定更快"，使用方式非常关键。

还有一个特别值得关注的发现：工程师估计，在他们用 Claude 完成的工作里，有百分之二十七的任务是"如果没有 AI，这件事根本不会做"的。这些任务包括什么？扩展实验规模、建一些"有了最好、没有也不影响"的辅助工具，比如交互式数据看板；还有一些以前因为人工成本太高而被放弃的探索性工作。这个数字告诉我们，AI 不仅仅是在把原来的工作做得更快，它还在开辟原本根本不存在的新工作领域。有位研究员分享了一个很生动的比喻，他说："大家常常把强大的 AI 模型想象成一辆更快的车。但如果你有了一百万匹马，那就完全是另一种玩法了——你可以同时探索一百万条不同的路径，这在以前想都不敢想。"

第二个发现：工程师们是怎么决定"什么给 Claude 做、什么自己做"的。

这是访谈里最有实操价值的部分。经过对五十三位工程师的深度对话，研究团队梳理出了一套相当清晰的委派判断框架，工程师们倾向于把满足以下条件的任务委派给 Claude。

第一个维度是容易验证。如果检查 Claude 的输出所需的时间和精力，远远小于自己从头做的成本，那这件事就值得委派。反过来，如果判断 Claude 做得对不对本身就要花很大力气，可能就得不偿失了。有工程师直接说："对我来说，当验证工作的代价远小于创作工作的代价时，Claude 就令人叹为观止。"

第二个维度是任务的独立性。如果一个子任务和整个系统其他部分的耦合很少、边界清晰，就可以让 Claude 来做。如果这件事和其他代码、业务逻辑、组织上下文高度纠缠，Claude 在没有充分背景的情况下，很难做好。

第三个维度是代码质量要求。如果是一次性用的调试代码、探索性的研究脚本，出了问题影响也有限，那直接交给 Claude。如果是生产环境里的核心模块，质量要求严苛，就需要更谨慎地审核。

第四个维度，我觉得最有人情味：无聊程度。有工程师说，"我对一个任务越兴奋、越想亲自做，就越不会用 Claude；我对一件事越觉得无聊、越有抵触情绪，就越容易打开 Claude 来帮忙。"这个逻辑挺合理的，人天然愿意保留自己有热情的工作，把苦差事外包出去。

还有一个"冷启动问题"被多位工程师提到。Claude 对你的代码库一无所知，你需要花时间传递背景信息、把上下文讲清楚。如果一件事本来你自己十分钟能搞定，花时间给 Claude 建立上下文可能就已经超过十分钟了，那还不如直接自己做。

关于信任的建立，有一位工程师用了一个特别有共鸣的比喻。他说他用 Claude 的方式，就像他使用谷歌地图导航经历的变化。一开始，他只在完全陌生的路线上用导航，就好比只让 Claude 写他不熟悉的 SQL 语句；后来，开始在大致熟悉但最后一公里不确定的地方用；再后来，连每天上班的固定路线也打开导航，因为它可能知道当天的拥堵情况。他说："我现在用 Claude 就是这种状态——完全信任它，按它的建议走。"

但与此同时，也有工程师提醒，这种信任需要经验来支撑。一位安全团队的工程师分享了一个例子：Claude 曾经提出过一个"非常聪明但实际上很危险"的方案，那种只有经验丰富的人才能识别出问题的方案，就像一位很有才华但资历不足的初级工程师可能会建议的事情。这个例子说明，有效使用 Claude，本身就需要使用者具备足够的判断力和技术深度。

第三个发现：技能版图在扩张，但深度可能在流失。

这是整篇研究里让我感触最深的部分，因为它揭示了一个真实的矛盾。

好的一面是清晰可见的。工程师们在 Claude 的帮助下，跨越了原来的专业边界，开始做以前做不了或者不敢做的事情。有位专注后端开发的工程师，用 Claude 帮他构建了一套复杂的前端界面，效果超出他自己的预期，设计团队看到之后都惊讶地问"这是你做的吗"，他笑着说"不是我做的，是 Claude 做的，我只是给出了提示词"——这里的提示词就是 prompt，就是你和 Claude 说的那些话。在 Claude Code 的使用数据里，这种全栈化的趋势也很明显：安全团队用它来分析不熟悉的代码库，安全对齐团队用它来构建数据可视化，非技术背景的员工用它来调试网络故障。大家都在借助 Claude 突破自己以前的能力边界。

这带来了什么好处？工程师说，反馈循环变快了。以前，如果你需要跨团队协作，要开会、要排期、要等反馈，一件事可能拖几周。现在，你可以用 Claude 快速做出原型，在同一个工作会议里就得到实时反馈，几周的流程可能压缩成几个小时。研究团队还发现，有一位资深工程师说，AI 工具让初级工程师变得更有生产力，也更敢去承接以前觉得超出自己能力范围的项目。

但另一面的担忧也是真实的。有工程师说，他们开始担心"深度技能的萎缩"，担心那种在手工做事过程中自然积累起来的、无法被显式教授的系统性理解，正在悄悄流失。

有一位工程师说出了一个很有洞察力的观察：手工调试一个复杂问题时，你会顺带阅读很多当时看起来跟这个问题"不直接相关"的文档和代码，但这些额外的阅读，其实是在帮你构建对整个系统的心智模型。而 Claude 可以直接把你带到问题所在，省掉了这段旅途，也省掉了旅途中的风景。

另一位工程师说，他以前会系统地探索一个新工具的所有配置项，了解它能做什么、不能做什么。现在他直接问 AI，得到"怎么用"的答案。结果他对工具的理解停留在了"会用"的层面，而不是"深懂"的层面。他坦言，以前在和同事交流时，他能随口说出很多知识，现在他需要临时去问 AI。

还有一个更深层的担忧，我把它称为"监督悖论"。有效使用 Claude，需要对它的输出进行持续的监督和验证；但这种监督能力，恰恰依赖于你自己的技术深度。如果 AI 的过度使用导致你的技术判断力逐渐下降，你对 Claude 输出的质量控制能力也会随之下降，两者互相强化，形成一个令人担忧的正反馈循环。有一位工程师说得很坦率："我担心的不是我自己的技能本身，而是技能萎缩之后，我是否还有能力安全地使用 AI——这才是核心问题所在。"

为了对抗这个趋势，一些工程师专门保留了"不用 AI 的练习时间"。有人说，"就算我知道 Claude 能轻松解决某个问题，我有时候也会故意不问它，保持自己的能力。"

关于软件工程的意义感，受访工程师们的体验截然不同。有人感到真实的失落，"我编程二十五年了，这是我职业身份的核心，现在感觉一个时代结束了。"也有人释然，"我以为我是真的喜欢写代码，但现在我发现我真正喜欢的是写代码带给我的东西——问题被解决的成就感，产品变得更好的满足感。用 Claude 一样能有这种感觉。"还有工程师做了一个有意思的历史类比：早期程序员需要手动管理内存、写汇编语言，后来出现了高级编程语言，自动处理了这些低层细节。也许现在，自然语言就是新的编程语言，工程师需要学会的是如何在更高的抽象层次上思考，而不是执着于低层实现的细节。

第四个发现：职场社交关系正在悄悄地改变。

访谈里有一个细节，我认为特别值得关注。有工程师说，他现在每天问的问题比以前多多了，但其中有百分之八十到九十是直接去问 Claude 的，而不是去问同事。Claude 成了第一个"咨询窗口"，把大部分常规的、有明确答案的问题拦截下来；只有真正需要组织上下文、人情判断、或者超过 AI 能力范围的问题，才会流向同事。有人评价说，这把他对同事依赖度降低了百分之八十，"但剩下那百分之二十至关重要，那是我一定要去找人聊的。"

对这个变化，不同的人有不同的感受。有人觉得挺好，"不需要再担心打扰同事了，有问题直接问 Claude，随时随地，不用等。"有人则感到有点落寞，说"我很喜欢和人协作，但现在我意识到我需要别人的时候越来越少了，这让我有点难过。"还有人直接抵制这种变化，说"我不喜欢'你去问过 Claude 了吗'成为大家的默认回答，我非常珍视和人面对面工作的体验。"

更深层的变化在于师徒关系的弱化。Claude 现在可以给初级工程师提供大量的技术指导，很多以前需要去请教高级工程师的问题，现在直接问 Claude 就能解决。有位资深工程师说："年轻同事找我问问题的频率明显低了。虽然他们因此学得更快、问题解决得更有效，但我有一丝遗憾。"

那些在提问和回答之间自然产生的东西——关系的建立、文化的传承、非正式的职业指导、跨代际的知识流动——这些东西是很难被量化的，也是很难被 AI 完全替代的。但当这些互动减少时，我们很难立刻察觉，因为表面上的工作效率并没有下降，甚至还在上升。

至于职业的长期走向，受访者的情绪相当分裂。有人说"短期我很乐观，但长期我觉得 AI 最终会做所有的事情，把我们都变得无关紧要"；有人更加务实，"我每天来上班，感觉像是在为自己失业做贡献。"但也有更乐观的声音，"我担心初级工程师，但我也知道初级工程师往往是最渴望拥抱新技术的群体，我对这个行业的整体走向还是保持乐观的。"还有许多人的答案则是坦诚的"不知道"，说"三年后我的工作会是什么样子，真的很难说。"

第五个发现：Claude Code 的客观使用数据。

除了问卷和访谈，研究团队还系统地分析了二十万条内部 Claude Code 使用记录，比较了二零二五年二月和二零二五年八月两个时间点的差异，发现了几个相当清晰的客观趋势。

首先，任务复杂度在显著提升。研究团队对每条记录用一到五分的量表评估难度，一分对应基础的小改动，五分对应需要数周乃至数月人工专家工作量的高难度任务。从二月到八月，平均复杂度从三点二上升到了三点八，工程师们在把越来越复杂的任务委派给 Claude。

其次，Claude 的连续自主操作能力大幅提升。研究团队追踪了 Claude 在每次对话里，不需要人类干预就能连续执行多少个工具调用——工具调用包括修改文件、运行命令、读取代码等具体行动。从二月的平均九点八次，增长到了八月的平均二十一点二次，涨幅超过百分之一百一十六。换句话说，以前 Claude 做十步就需要人来指一下方向，现在它能一口气做二十步再来找你确认。

与此同时，每次对话中人类发言的轮次减少了三分之一，从平均六点二轮降到了四点一轮。人类需要更少的干预来完成同样的任务了。

任务类型的分布也发生了显著变化：实现新功能的任务从百分之十四上升到了将近百分之三十七，接近翻倍；代码设计和规划类任务从百分之一上升到了将近百分之十。工程师们正在把更高层次的工作也逐渐纳入 AI 协作的范围。

此外，研究团队发现百分之八点六的 Claude Code 任务属于"修复小毛病"——那些平时因为优先级太低而一直被搁置的小改进，比如重构可读性差的代码、建立小型辅助脚本、创建终端快捷键。这些小事单独看都不重要，但积少成多，可能会对整体工程效率和日常工作体验产生相当可观的正向影响。

好，研究发现说完了。我们来聊聊对你的实际工作意味着什么，以及有哪些地方需要特别注意。

第一条实践建议：建立你自己的委派框架，要有意识地做选择。不要随机决定什么用 AI 做、什么自己做。Anthropic 工程师总结出来的那几个维度——容易验证、低风险、任务独立性强、本来就打算丢掉的代码——是非常好的起点。在你接到一个任务的时候，先在脑子里过一遍这几个问题，然后再决定是否委派。有意识地选择，效果远好于无意识地随手用。

第二条：重视"冷启动成本"，提前建立 AI 友好的上下文。如果你的代码库很复杂、有大量的内部约定和历史沿革，让 AI 理解这些背景需要投入相当大的精力。一个很好的做法是在项目里建立系统性的上下文文档，把代码库的核心结构、关键设计决策、常见问题和约定整理出来，让每次开启新的 AI 对话都有一个好的起点。这个投入是一次性的，但节省的时间是持续的。

第三条，也是最重要的一条：主动保护你的深度技能，特别是那些你用来判断 AI 好坏的能力。"监督悖论"这个问题是真实存在的——你越依赖 AI，你的技术判断力可能越退化，而技术判断力退化之后，你反而更难有效地监督 AI，这是一个危险的循环。我建议你定期安排"不用 AI 的时间"，保持亲手解决问题的能力。特别是在你的核心专业领域，不要让这里的深度空洞化。要记住，你的判断力就是 AI 最重要的安全网。

第四条：重新审视你的职业定位。从"写代码的人"到"管理 AI 系统的人"，这个转变在 Anthropic 的工程师里已经开始发生，他们有人说自己工作的七成以上已经变成了代码审查和修订，而不是全新的代码创作。你可以问问自己：在你的工作里，有哪些能力是 AI 很难替代的——对业务的深度理解、在团队里的协调能力、对用户真实需求的敏感直觉、识别 AI 输出中的微妙错误？把这些东西识别出来，然后有意识地去发展和巩固它们。

第五条：维护人与人之间的真实连接，不要让它悄悄地侵蚀掉。当越来越多的日常问题可以被 AI 回答时，和同事之间真正有价值的互动可能会变得更稀少，但也因此更宝贵。主动去创造那些需要深度理解、创意碰撞和信任建立的合作机会。特别是如果你是资深工程师，不要因为初级同事来问问题的频率降低就放任这个趋势——那些非正式的指导和交流，正在做着 AI 无法完整替代的事情。

最后，说说我个人的看法。

我个人认为，这篇研究最有价值的地方，不是那些生产力数字，而是它展示了一家公司愿意诚实地面对变化的复杂性。它不只说"AI 让我们更高效了"，也说"AI 让我们失去了一些东西"；不只说"工程师们更快乐了"，也说"有人感到失落、孤独、对未来充满不确定"。这种诚实本身，就是一种难得的示范。

我们正处于一个没有历史先例的时期。AI 带来的生产力提升是真实的，但代价和挑战也同样真实。这不是一个有简单答案的时代。没有人知道两三年后工程师的工作究竟是什么样子，甚至连 Anthropic 自己都说，他们对未来充满不确定。研究团队最后说的那句话，我觉得是整篇文章最好的结尾："没有人知道会发生什么，最重要的是保持高度的适应能力。"这句话，送给你，也送给我自己。
