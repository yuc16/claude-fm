## Script

今天要聊的这篇文章，原标题叫 Confidential Inference via Trusted Virtual Machines，中文翻译是"通过可信虚拟机实现机密推理"。这篇文章发表于二零二三年十一月三日，由 Anthropic 和一家叫 Pattern Labs 的合作机构联合发布。

我要特别提醒你注意一下时效性。今天是二零二六年六月十五日，这篇文章距今已经过去了将近两年半。在 AI 领域，两年半是一个相当漫长的周期。那个时候，Claude 三还没有发布，GPT-四系列刚刚站稳脚跟，行业里大家还在争论大模型到底能不能做复杂推理，还在为"幻觉"问题焦虑。可信执行环境和机密计算，在 AI 圈子里还是一个很小众的话题，大多数人连这个词都没听过。

但正因为如此，这篇文章放到今天来读，反而更有价值。它的问题意识走在了当时的前面，而这些问题，随着 AI 深入企业核心业务，已经从"超前研究"变成了"紧迫需求"。文章里描述的很多技术方向，在这两年多里也确实在持续演进。所以带着"历史视角"来听，你既能看懂这套技术的底层逻辑，也能感受到行业是怎么一步步走到今天的。

先来想一个场景，帮你理解这篇文章在解决什么问题。

假设你是一家公司的工程师，正在用 Claude 处理一份涉密的客户合同。你把合同内容粘进对话框，Claude 帮你提炼关键条款。在这整个过程里，你的数据到底在多少个环节处于"明文状态"？也就是说，在哪些地方，理论上有人或者有程序能直接读到你的内容？

很多人的第一反应是：不是加密传输吗？浏览器到服务器有 HTTPS，应该是安全的啊。

这个反应是对的，但只对了一半。HTTPS 保护的是数据"在路上"的那一段，也就是从你的浏览器到 Anthropic 服务器这个传输过程。但数据到了服务器里面，必须被解密才能处理，对吧？你没办法对密文做推理，就像你没办法读一本被涂黑的书一样。所以在服务器内部，总有那么几个时刻，你的数据是以明文形式存在的——任何能访问这台服务器的人，从系统管理员到潜伏的攻击者，在这些时刻理论上都能看到你的内容。

这个"明文窗口"，才是数据安全真正的脆弱地带。

Anthropic 这篇文章要回答的问题就是：我们能不能把这个明文窗口缩到最小？能不能不光把它缩小，还用密码学证明给用户看，而不是让用户单纯相信我们的承诺？

这是一个非常有意思的转变。从"信任我们"到"验证我们"。

文章里明确说了，推动这个研究有两个驱动力。第一个是模型权重安全。Claude 这样的前沿模型，它的参数权重是 Anthropic 最核心的资产，是凝结了海量算力和数据的竞争力所在。如果这些权重被窃取，任何人都可以部署一个"山寨版 Claude"，后果不堪设想。文章里特别提到了兰德公司发布的一份关于保护 AI 模型权重的报告，那个时候行业已经开始认真对待这个威胁了。第二个驱动力是用户数据安全，这就是我们刚才讲的那个场景——你发给 Claude 的内容，必须有更强的保护。

为了解决这两个问题，Anthropic 引入了一套叫做"机密计算"的技术体系，英文叫 Confidential Computing。这不是 Anthropic 自己发明的新名词，而是行业里已经发展了好几年的成熟方向。英特尔的 SGX 技术、AMD 的 SEV 技术，都是这个领域的具体实现。机密计算的核心思想是：在处理器层面，创造一个叫做"可信执行环境"的隔离区域。在这个区域里运行的代码、处理的数据，即使是操作系统、虚拟机管理程序，甚至是数据中心的物理管理员，也无法从外部访问。这个隔离是由硬件来保证的，不是软件层面的隔离，所以理论上更可靠、更难被绕过。

你可以把它类比成一个特殊的保险箱。普通保险箱能锁住里面的东西，但你没法知道保险箱本身有没有被人动过手脚。这个特殊保险箱不一样：它不但能锁住里面的东西，还能主动向外界证明"我确实是那个你认识的保险箱，里面运行的是你亲自审核过的代码，而且我没有被动过"。这个"向外主动证明自己"的能力，叫做"远程证明"，英文叫 Remote Attestation，这是整个机密推理的核心密码学机制。

好，现在让我们深入到 Anthropic 的具体架构设计里来看看。

当你的请求发出去，到达 Anthropic 这边，需要经过两个关键的服务组件。第一个叫 API 服务器，它负责接收你的提示词，把文字转成模型能理解的 token，也就是数字单元，处理各种请求逻辑，就像一个调度员。第二个叫推理服务器，这才是真正运行 Claude "大脑"的地方，它在专用的硬件加速器上——也就是 GPU 或者 TPU 上——跑模型的计算，生成回答。

文章的聚焦点在推理服务器。原因有两个：一方面，推理服务器是模型权重真正存放和使用的地方，是模型安全的核心；另一方面，推理服务器的硬件加速器在当时还没有完整的机密计算支持，工程挑战更大，更值得专门讲清楚。

这里就有一个很实际的工程约束问题。不是所有的 GPU 都原生支持机密计算，在二零二三年末的时候尤其如此。那怎么办？不能因为硬件没跟上就放弃安全目标。Anthropic 设计了一个很聪明的折中方案：引入一个极其精简的"可信加载器"，英文叫 loader，作为可信环境和普通计算之间的中间层。

这个架构是这样的：整个推理服务器被刻意分成两个部分。大部分的代码和逻辑运行在"不可信"区域——这里可以频繁更新，可以代码量很大，架构很复杂，但有一个铁律：它永远接触不到明文数据。另一部分是那个小小的、高度精简的可信加载器，它运行在一个单独的、被 hypervisor（虚拟机管理程序）从硬件层面隔离的虚拟机里。

Hypervisor 是什么？就是你在云服务器上常见的那个虚拟化层，相当于给不同的虚拟机划出了互不干扰的地盘，即使跑在同一台物理机上，虚拟机之间也互相看不到对方的内存。这个隔离在安全上意味着：就算不可信区域的推理服务器主体被攻破了，攻击者也进不了旁边那个运行着可信加载器的隔离虚拟机。

这个可信加载器只做几件非常简单的事情：接收来自推理服务器的加密数据，在可信环境内部解密，然后把明文送到 GPU 做计算；拿到 GPU 的计算结果之后，立刻重新加密，再传回给不可信的推理服务器主体，最终返回给你。

就这几步，没有别的。文章展示的架构向外提供的接口，甚至是把自己伪装成一个"虚拟加速器"——对推理服务器来说，它就像在对着一块普通 GPU 发请求，根本感知不到中间有个可信加载器在帮它处理加密和解密。

这种设计哲学在安全领域有个术语，叫做"攻击面最小化"。你暴露在风险中的代码越少，可能出现漏洞的地方就越少，被攻击的概率也就越低。一个两万行的系统，有两万行值得审计；一个两百行的系统，你把每一行都检查十遍也花不了多长时间。加载器就是有意把这个安全敏感的核心代码控制在最小规模。

还有一个细节我觉得特别值得提：这个可信加载器只接受"被安全 CI 服务器签名"的程序。CI 是持续集成的缩写，就是工程里常见的自动化构建和测试系统。这里说的意思是，任何想要在这个可信加载器里运行的代码，必须先经过多名工程师的代码审查流程，然后由一个专门的、受严格访问控制的 CI 服务器用密钥对它签名，加载器启动的时候会验证这个签名。没有正确签名的代码，直接拒绝运行。

这就像给这个保险箱设了一道非常严格的内容准入机制：不是 Anthropic 公司的人进不来，而是连 Anthropic 自己没有通过完整审批流程的代码，也进不来。这防止的不光是外部攻击，也防止内部的疏失或者恶意行为。

那么问题来了：我们怎么知道这个可信加载器真的在按照描述的方式运行？怎么知道它真的在隔离的环境里，调试功能真的被关掉了，运行的代码真的是经过审查的那个版本？光靠 Anthropic 自己说，还不够。这就到了文章最关键的密码学部分了。

这里要介绍一个叫做 TPM 的硬件组件，全称是 Trusted Platform Module，中文叫可信平台模块。TPM 是一个独立的安全芯片，很多电脑和服务器里都有，它是整个证明机制的"信任根"。

TPM 的核心能力是：忠实地记录和报告系统从启动到运行的完整过程。你可以把 TPM 想象成一个无法被篡改的公证员。系统每次启动，TPM 都会对启动过程的每个环节做密码学意义上的"拍照"——用哈希函数，把当前的状态变成一个固定长度的指纹。固件加载完毕，拍一张指纹。操作系统内核启动，再拍一张指纹。应用程序开始运行，再拍一张。每一张新指纹都把前一张指纹包含进去，形成一个连续的链条，最终生成一个总的哈希值，代表了这台机器从开机到现在，到底运行了什么。

然后，TPM 用它内置的私钥对这个总哈希值签名，生成一份"证明书"。这个私钥从出厂就烧死在芯片里，外部无法提取、无法伪造。

有了这份证明书，外部的验证者就可以检查：这台服务器运行的代码哈希值，是不是我们审核并签名过的那个版本？调试功能是不是确实被禁用了？隔离配置是不是我们期望的那个状态？如果所有检查都通过，一个叫做"密钥服务器"的组件，才会把解密密钥释放给这台加载器，让它能够解密用户数据。

注意这个顺序非常关键：先证明自己是谁、运行着什么，再拿钥匙。不是先给钥匙、再相信你不会乱用。这就是"密码学保证"的含义——不是"请信任我们"，而是"这是数学证明，你可以自己验算"。

文章里把这个完整的链条描述得非常清晰。用户的请求在到达 Anthropic 服务器之前就被加密了；到了 API 服务器，解密、处理完成之后，重新加密才能往下传；推理服务器拿到的是加密状态的请求，只有当它送进那个通过了证明的可信加载器，才会在严格隔离的环境里被解密，做计算；计算完成之后，结果立刻被重新加密，再传回给你。模型权重的路径更简单：加密存储，进了加载器再解密，永远不从那里出去。

整个过程里，明文数据只在那个极小的、经过硬件隔离的可信环境里短暂存在，每一步都有密码学机制在守护着边界。

文章还提到了一个更进一步的设想，我认为这是整个架构里最有意思的部分，值得多说几句。

现在描述的系统，它的密钥服务器是由 Anthropic 自己运营的。这意味着整个信任链的最后，还是落在"你得信任 Anthropic 这家公司"。如果密钥服务器本身被攻破，或者被 Anthropic 的内部人员滥用，整个机制就失效了。这是一个现实的风险，文章坦诚地承认了这一点。

文章里提出的方向是：引入独立的第三方来参与验证可信代码，并且管理独立的密钥服务器。什么意思呢？就是不再由 Anthropic 一家说了算，而是让安全审计机构、监管机构，或者客户自己信任的第三方，也加入到这个证明和验证的链条里。任何一份用户数据的解密，都需要多个独立方共同"点头同意"。

这种思路在密码学里叫做"多方信任"，类似于金融体系里的多重签名机制，或者说就像需要多把钥匙同时插进去才能开启的核弹发射装置——没有任何单一的点可以独自绕过安全控制。对于那些对数据合规要求极高的行业——医疗、金融、法律——这个特性从合规角度来说几乎是必须的。

文章在未来方向部分还提到了几个很有意思的想法，虽然写得比较保守，但我认为值得展开讲。

第一个是出站带宽限制。文章说，随着模型能力越来越强，可能需要在持有明文模型权重的服务器上，额外加一层网络出站带宽限制。这个听起来有点奇怪，为什么限制带宽和安全有关？背后的逻辑是防范"侧信道攻击"。即使你的数据在处理过程中是加密的，一个拥有服务器访问权的攻击者，如果能观察到网络出站流量的时序和流量模式，理论上可以从这些"旁路信息"里推断出部分数据内容。通过限制带宽、增加随机噪声，可以压缩这个泄露渠道。这是密码学工程里一个很精妙但很容易被忽视的点。

第二个是安全分类器签名。文章设想，未来可能要求推理在执行之前，必须有一个专门的安全分类模型先审查这次请求，只有它"点头"了，才能拿到签名，加载器才会执行推理。这相当于在密码学保护的基础上，还加了一道内容安全的门禁，把安全检查从"事后审计"变成"事前准入"。

第三个方向，文章专门向芯片设计师喊话，希望硬件厂商能把机密计算的支持原生内置到加速器芯片里。如果 GPU 本身就有硬件可信执行环境，就不需要那个折中的加载器设计了，整个可信边界可以收缩得更小，系统也更简洁。从二零二六年的视角来看，这个方向确实在推进——英伟达的 H 百系列芯片已经有了一定的机密计算能力，但完整的生态建设还在继续完善当中，行业仍然在爬那个技术曲线。

好，现在我们来聊这些技术对你的实际工作意味着什么，能上手用什么、怎么用、要注意什么。

第一类听众是在企业里做 AI 应用的工程师和架构师。这篇文章告诉你，在选择 AI 服务商的时候，除了看模型能力和价格，应该把数据安全架构作为独立的评估维度。你现在可以问服务商一些具体的问题：你们的推理基础设施有没有可信执行环境支持？有没有远程证明能力，我能不能自己验证你的环境状态？密钥管理是单点还是多方？有没有独立审计报告？这些问题在两三年前还是比较超前的，很多销售可能都答不上来。但随着合规压力增大、数据安全事件增多，这些问题正在成为企业采购清单里的标准项，尤其是在金融、医疗、政府这些对数据敏感度高的领域。

第二类是正在考虑把高敏感数据送进大模型的产品决策者。你现在可能的策略有三条：第一，对数据做脱敏处理，去掉能识别身份的信息再送进去；第二，本地私有化部署，把模型跑在自己控制的基础设施上；第三条路，机密推理，也就是这篇文章描述的方向，在云端服务器上用密码学保证数据的处理安全性。

这三条路各有优缺点。脱敏简单但会损失信息质量；私有化部署给你最大的控制权，但成本高，而且你得自己管硬件和模型更新；机密推理理论上能两全，但工程复杂度是真实存在的，并不是一个可以随手打开的开关。

我个人认为，对于大多数中小企业，短期内最实用的还是脱敏加上合同约束。机密推理更适合那些数据监管要求极严格、预算充足、有专业安全团队的大型机构。

第三类是关注 AI 安全合规的朋友。这篇文章给了一个非常有价值的思考框架：安全保证应该从"声明性合规"升级到"可验证性合规"。什么意思？过去的合规方式是：服务商说"我们承诺不泄露你的数据"，然后给你一份隐私政策让你签字。可验证性合规是：服务商说"我们的系统在密码学上无法泄露，这里有证明，你可以自己验算"。这个转变的意义在于，你不需要再相信服务商的自我声明，你可以用数学来核实。

这个方向在监管层面也在演进。欧盟 AI 法案、国内的各类 AI 安全标准，都在越来越强调"可证明的"、"可审计的"安全性，而不光是声明性的安全承诺。了解机密推理这个技术方向，能帮你在和监管、法务、客户谈数据安全问题的时候，拿出更实质性的技术依据。

当然，机密推理不是银弹，有几个现实局限必须说清楚，不然你会高估它的保护边界。

第一个是硬件依赖的问题。在文章发表的时候，GPU 的机密计算支持还不完整，所以设计里仍然有一部分依赖于数据中心的物理安全和虚拟机管理程序软件的安全性。这两块不在密码学保证的范围内。文章坦诚地说了，他们还在继续弥合这个缺口，在此之前，依赖计算供应商维护物理层面的安全。

第二个是性能开销。加密解密、证明验证，这些操作都有实际的计算成本。文章没有详细给出性能数字，但在实际部署中，这个额外开销在延迟敏感的场景里不可忽视。特别是对那些追求极低响应时间的应用，这是需要认真做 benchmark 评估的。

第三个是密钥服务器本身的安全。整个系统的信任最终落在密钥服务器上。密钥服务器本身如果被攻破，加载器拿不到密钥，系统就停摆了；如果密钥服务器被欺骗，把密钥给了一个伪装成可信环境的恶意节点，整个保护就失效了。文章提到的多方密钥管理能缓解这个风险，但也引入了新的协调复杂度。没有完美的设计，只有权衡。

第四点是我个人认为很多人容易误解的边界问题。这套机密推理机制，保护的是数据在推理过程中的处理安全——也就是"当你的数据被 Claude 处理的那一刻，有没有人能偷看"。它不解决的是推理完成之后的数据生命周期问题：日志怎么存、缓存怎么清、Anthropic 会不会把你的对话用来训练下一代模型。这是两个完全不同维度的隐私问题，机密推理只管前者。如果你的关切是后者，你需要看的是服务商的数据保留政策和模型训练数据协议，而不是这套技术架构。

来做个总结。Anthropic 在二零二三年十一月发布的这篇关于机密推理的研究报告，核心思路是：把用户数据的明文暴露窗口缩到最小，用可信执行环境隔离敏感的解密操作，用 TPM 和远程证明链在密码学层面保证运行环境的可信性，最终把 AI 服务的隐私保护从"公司声明"升级为"数学可验证的保证"。架构上的关键创新是那个精简的可信加载器设计，用极小的可信计算基，在不完全支持机密计算的硬件环境下，实现了可接受的安全隔离。

我个人对这个方向是比较乐观的。机密计算技术本身已经发展了很多年，有扎实的理论和实践积累，把它引入 AI 推理场景是非常自然的延伸，不是为了安全而造新概念。而随着 AI 深入企业核心流程，这种可验证的数据保护能力只会越来越重要。从今天二零二六年的视角来看，这篇两年半前的研究方向，正在逐渐从"前沿探索"变成"行业标准"。这不是一篇可以立刻给你一个工具去用的技术文章，但它给了你一套理解和评估 AI 数据安全的框架，这个框架的价值，在接下来几年里只会越来越大。
