## Script

今天这期节目，我们要聊一篇让我看完之后坐不住的文章。文章的英文标题是"LLM-discovered 0 days"，我把它翻译成"大模型发现的零日漏洞"。这篇文章发表于二零二六年二月五日，作者是 Anthropic 的 Frontier Red Team，也就是前沿红队，核心作者包括 Nicholas Carlini、Keane Lucas 等六位研究员。

文章在 Anthropic 发布 Claude Opus 四点六的同一天上线，而且这件事不是巧合——整篇文章就是在用真实数据说明这个新模型在安全领域做到了什么。

为什么我说看完坐不住？因为它描述的不是一个"将来有一天 AI 会很厉害"的预测，而是一件已经发生的事情：AI 模型现在就可以在真实的、经过多年充分测试的开源代码库里，独立发现人类安全研究员没有发现的高危漏洞，而且已经找到并验证了超过五百个。这个数字，不是估算，不是概念验证，是真实报告、真实修复中的漏洞。

你可能会说，AI 找漏洞这件事我早就听说过了。但等你听完这篇文章的细节，你会发现，这一次有什么东西真的不一样了。

我们先来把背景说清楚。在网络安全领域，有一个词叫做"零日漏洞"，英文是 zero-day，也写成 0-day。它的意思是，这个漏洞在被攻击者利用之前，软件的开发方完全不知道它的存在，留给他们修复的时间是零天。零日漏洞之所以危险，核心在于时间差：攻击者可以从容利用它，防御方却完全没有准备。

在过去很多年里，发现零日漏洞主要靠两种方式。

第一种叫做模糊测试，英文是 fuzzing。简单说，就是向程序喂大量随机的、畸形的输入数据，看哪个输入会让程序崩溃或者产生异常行为。这个方法听起来很暴力，但非常有效。谷歌有一个项目叫 OSS-Fuzz，专门对开源软件做大规模的模糊测试，已经帮助发现了几千个漏洞。很多大型项目每天都有成百上千个 CPU 核心在持续运行模糊测试，累积了数以百万小时计的测试时间。

第二种是人工代码审计。有经验的安全研究员一行行读代码，用积累多年的直觉判断哪里可能有问题。这种方法能发现非常复杂、非常隐蔽的漏洞，但代价巨大：速度慢、成本高，能做这件事的人在全世界也就是几千人的规模。

这两种方法各有局限。模糊测试不理解代码逻辑，只能靠概率碰运气，对那些需要满足复杂前置条件才能触发的漏洞几乎无能为力。人工审计能理解逻辑，但人的精力是有限的，一个几百万行代码的项目，人工审计永远不可能覆盖所有角落。

Anthropic 的研究团队认为，AI 在这里有机会填补两者的空白——既能理解代码语义，又能大规模地处理代码。他们去年秋天就提出，AI 在网络安全领域正在迎来一个拐点，进展可能会非常快。而这篇文章，就是他们拿出的第一份有实质分量的证据。

我们来看看他们具体是怎么做的。

首先是实验设置。他们把 Claude 放进一个虚拟机里，也就是一台模拟的计算机环境，里面装了最新版本的开源项目代码，以及标准的工具：常用的命令行工具、调试器、模糊测试工具。

关键点在这里：他们没有给 Claude 任何专门的指令，没有写自定义的脚手架，没有告诉它应该怎么找漏洞。用文章的话说，他们只是在直接测试 Claude 的"开箱即用"能力，依靠的唯一前提是：现代大语言模型已经是通用能力足够强的 agent，可以自己推理出怎样最好地使用手边的工具。

这个设计很重要，因为它排除了一种常见的质疑。你可以说：如果你给 AI 写一套专门针对漏洞挖掘的框架，精心设计每一步的提示词，那结果好不算稀奇。但他们没有这样做，他们就是丢给 Claude 一个代码库和一堆标准工具，让它自己想办法。

在这个设置下，他们重点关注内存破坏类漏洞。原因是这类漏洞可以相对容易地被客观验证：程序真实地崩溃了，或者用地址消毒器这类工具可以捕捉到真实的内存错误。崩溃是客观事实，不依赖模型的判断，这就把"模型是否在幻觉"这个问题很大程度上排除掉了。

找到可能的崩溃之后，他们还要经过多轮筛选：让 Claude 对找到的崩溃做批评、去重和优先级排序，然后由 Anthropic 自己的安全研究员逐一手工验证每一个漏洞，并且手写修复补丁。随着发现的漏洞数量增多，他们还引入了外部的人工安全研究员来帮助验证。整个流程的核心原则是：宁可漏掉一些真实漏洞，也绝不提交任何误报。

结果是：超过五百个经过人工验证的高危漏洞，已经在向相关开源项目报告，初批补丁已经开始被合并进入代码库。

文章里重点描述了三个案例，我认为这三个案例是整篇文章最值得细读的部分。它们展示的不只是"AI 找到了漏洞"这个结果，而是"AI 是怎么找到漏洞的"这个过程。而这个过程，才是真正令人震惊的地方。

第一个案例：GhostScript。

GhostScript 是一个用来处理 PostScript 文档和 PDF 文件的工具，被嵌入到大量的服务器环境、打印系统和文档处理软件里，是一个历史非常悠久的项目，经过了多年大量的安全审计和模糊测试。这不是一个没有人关注过的边角项目，而是一个处于持续安全关注下的知名代码库。

Claude 最开始的尝试和人类安全研究员一样：先试模糊测试，没有找到有价值的东西；改成手动分析代码，同样没有突破。这两种常规方法都失效之后，Claude 换了一个完全不同的思路——它开始读这个项目的 Git 提交历史。

这个操作本身就非常有意思。Git 提交历史是开源项目特有的一份资产，它记录了代码的每一次变动以及变动的理由。人类安全研究员有时候也用这个技巧，叫做"patch diffing"，通过分析安全补丁的代码差异来推断补丁之前的代码是如何被利用的。

Claude 在 GhostScript 的提交历史里找到了一个和"MM 混合值的栈边界检查"相关的提交。它的推理原文是英文，我来翻译一下：这个提交在添加栈边界检查，这说明在检查被加上之前，代码是存在漏洞的。如果想触发这个漏洞，我需要测试添加修复之前版本的代码。

接着，Claude 把这个逻辑往前推进了一步：既然这里被修了，有没有其他地方调用了同一个函数，但是没有加上对应的检查？它去找这个函数的所有其他调用者，果然在另一个文件里发现了一处相同的调用，却没有同样的边界检查保护，成功构造出了一个让 GhostScript 崩溃的证明文件。

你看，Claude 做的这件事，和模糊测试的逻辑完全不同。模糊测试是在盲目地投喂随机输入，期待某个输入碰巧触发问题。Claude 做的是理解代码变更的语义：这次提交在干什么，为什么要这样改，改的是否完整，逻辑上有没有遗漏的角落。这种能力，只有具备语言理解能力的系统才能做到，传统工具没有办法复现。

更深刻的一点是：这个漏洞在 Git 历史里其实留下了"线索"——那个修复补丁本身就在暗示之前存在问题。但没有人去追问"这次修复是否彻底"，以及"有没有类似的代码路径被遗漏了"。Claude 不需要这个问题被明确提出，它自己把这条推理链走完了。

第二个案例：OpenSC。

OpenSC 是一个处理智能卡数据的命令行工具。同样，Claude 先试了模糊测试，没有突破；然后试了手动分析，同样没有明显发现。之后，Claude 换了方向，开始在代码库里寻找那些在历史上频繁出现安全问题的函数调用模式。

它找到了一段使用 strcat 函数的代码。strcat 是 C 语言里一个非常古老的字符串拼接函数，长期以来被安全研究员视为危险函数，原因是它在拼接字符串的时候不检查目标缓冲区是否有足够的空间。如果拼接后的字符串超出了缓冲区的边界，就会发生缓冲区溢出，这是最经典的高危漏洞类型之一。

文章里给出了 OpenSC 里的那段具体代码，我来解释一下它的逻辑。代码先声明了一个四千零九十六字节的文件名缓冲区，然后在填充路径之前，通过计算剩余空间来做保护：把缓冲区大小减去目标路径的长度再减二，作为允许写入的最大长度。听起来有保护，但接下来，代码依次用 strcat 追加了一个斜杠和一段文件路径。

问题出在这里：保护逻辑只考虑了一个变量，但最后写入缓冲区的内容是两部分的叠加。在特定的攻击条件下，攻击者可以构造一个足够长的路径，使得两部分内容加起来超出了预先留出的空间。

文章里特别提到了一个统计数据，我觉得非常有说服力：这段代码在现有模糊测试工具的覆盖统计里，被研究的频率非常低，原因是要触发到这个代码路径，需要满足很多前置条件——你得先完成智能卡的初始化流程，得到一个有效的卡片上下文，然后再走到文件名处理的这段逻辑。传统的模糊测试对所有代码路径投入大致均等的资源，而真正容易被触发的代码路径早就被研究透了，深层路径反而长期被忽视。Claude 的不同之处在于，它能推理哪些代码片段从语义上来说更可能有问题，然后把精力集中在那里，而不是均匀地撒网。

第三个案例：CGIF。

CGIF 是一个处理 GIF 图片格式的库。这个案例在三个里面我认为是最精彩的，不是因为 Claude 怎么找到这个漏洞，而是因为 Claude 怎么证明这个漏洞是真实存在的——构造概念验证的过程本身，展示了一种让人意想不到的深度推理。

CGIF 里有一个隐性假设：经过压缩的数据，大小肯定会比压缩前的原始数据更小。这个假设在几乎所有情况下都是成立的，所以库的作者为压缩输出分配缓冲区时，用的是基于原始数据大小估算出来的值。这看起来是一个合理的工程选择。

GIF 格式使用一种叫做 LZW 的压缩算法，我来简单解释一下它的工作原理，因为这对理解漏洞至关重要。

LZW 会维护一个动态构建的字典，把在数据流里经常重复出现的字符串组合记录下来，然后在压缩的时候，把这些组合替换成字典里更短的编码。你可以把它想象成一个不断学习的速记本：第一次看到"深度学习"这四个字，完整写出来；第二次再看到，就可以只写一个编号代替，因为字典里已经有了对应条目。编码比原文短，所以最终输出的文件变小了，这就是压缩。

对于了解大语言模型的听众，文章里特别类比了一下：LZW 算法和 BPE 分词，也就是大语言模型里常用的词表构建方法，在精神上是非常相似的，都是把高频出现的单元合并成更短的编码。

但 LZW 的字典大小有上限，默认是四千零九十六个条目。当字典被填满的时候，LZW 不能继续添加新条目，于是它会往输出流里插入一个特殊的"清除"标记，然后重置字典，重新开始构建。这个清除标记本身是额外的数据，它的大小算在输出里。

Claude 想明白了一件关键的事：如果能构造一个特殊的输入，让 LZW 算法的字典频繁地填满，然后频繁地清除，那么这些清除标记累积起来，会让输出的"压缩"数据实际上比原始数据更长，从而超出 CGIF 预先分配的缓冲区大小，触发缓冲区溢出。

Claude 的推理过程非常具体，文章里引用了它的原始分析，我来翻译一下：要触发溢出，需要图像像素数量很少，但让 LZW 写入大量的编码。能产生最多编码的输入模式是：完全没有压缩发生，每个像素单独变成一个编码，同时频繁发生字典重置。用四色调色板，字典填到四千零九十六个条目之后重置，这样每次重置都产生额外的清除标记，最终使输出数据量超过输入数据量。

文章特别指出，即使你对 CGIF 做到百分之百的行覆盖和分支覆盖，这个漏洞依然可能不会被传统的模糊测试发现，因为触发它需要一个非常特定的输入序列：不是随机的，不是概率上容易命中的，而是需要理解 LZW 算法的内部状态才能设计出来的那种输入。模糊测试没有这种理解，它只能靠运气，而这个输入序列运气很难碰到。

这三个案例放在一起，我们可以清楚地看到一个模式：Claude 不只是在"更快地"做人类安全研究员以前做的事情，它在用一种从根本上不同的方式思考问题。它读 Git 历史，它理解算法原理，它推断哪段代码在语义上更值得关注，它把逻辑链一步步走到底。这些都是需要"理解"而不只是"搜索"的能力。

接下来我们说说这篇文章的第二个主要部分：防护措施。

Anthropic 和 Claude Opus 四点六同步推出了一个新的检测机制，他们称之为"探针"，英文是 probes。探针的工作原理是：在模型生成响应的过程中，实时测量模型内部的激活状态，也就是模型运行时的内部数值分布，来检测模型当前是否在执行某种特定的有害行为。

你可以把这个机制想象成给 AI 模型做实时的脑电图。不是等模型生成完一段回答之后再从外部审查它说了什么，而是在模型"思考"的过程中，从内部观察它在做什么。他们针对网络安全领域，开发了专门的网络安全探针，用于追踪和理解 Claude 被滥用于攻击行为的可能性。

在执法层面，他们更新了网络安全相关的处理流程，并且扩大了应对滥用的处置范围。最值得注意的一点是，他们可能会实施实时干预，包括直接屏蔽被检测为恶意的网络流量。

文章也直接承认，这样做会给合法的安全研究和防御性工作带来摩擦。他们表示愿意和安全研究社区合作，解决由此产生的具体问题。这个表态我个人认为是诚实的——没有任何防护机制能做到零误伤，承认这一点并且说明应对计划，比假装问题不存在要好。

文章结尾提出了两个面向整个行业的警告，这两个问题我觉得比技术细节更值得我们认真对待。

第一个警告：语言模型已经具备了发现真实漏洞的能力，而且可能很快就会在速度和规模上超过最顶尖的人类安全研究员。这不是假设，是从现有数据往前外推的合理判断。五百个经过人工验证的高危漏洞，来自于一个没有做任何专门优化的通用模型，针对的是已经被大量测试过的代码库。这个能力如果继续按照最近的进步速度发展，会去到哪里？

第二个警告：现有的漏洞披露规范面临严峻挑战。行业里有一个被广泛遵循的标准，叫做"负责任披露"，通常给软件厂商九十天时间来修复漏洞。这个九十天的窗口，是在人类研究员一次发现几个漏洞的背景下设计的。但如果 AI 一次向同一个开源项目报告几十个漏洞呢？如果每周都有新的一批涌进来呢？很多开源项目的维护团队本来就捉襟见肘，他们有没有能力消化这个速度？九十天的窗口在这种新节奏下还合适吗？这些问题现在没有答案，但必须有人开始认真讨论了。

好，现在我们来到我最喜欢的部分：这篇文章对我们实际工作的意义是什么，能上手用什么，怎么用，要避开哪些坑。

第一点，对于写代码的工程师来说，这篇文章是一个非常清晰的信号：AI 辅助安全审计，已经从"有一天会有用"变成了"现在就可以做"。如果你负责维护一个 C 或者 C++ 的项目，特别是那种对外接受不受信任输入的项目，比如文件解析库、网络协议栈、压缩工具、格式转换库，你现在就可以把 Claude 拉进来做有针对性的代码审查，让它重点关注那些历史上频繁出现问题的模式：内存分配和释放的对称性，字符串操作有没有边界检查，整数运算有没有溢出的可能，资源的生命周期管理是否正确。

第二点，这篇文章最重要的方法论启示是：要在有工具的环境里用 AI，而不是只在聊天窗口里问 AI。我经常听到有工程师说，用 AI 做代码安全审查效果不好，误报太多，或者说的都是废话。我觉得这里面有一个很根本的差异被忽视了：Claude 在有工具调用能力、有真实运行环境的情况下，和在纯文本对话界面里的表现，是完全不同的两个量级。文章里描述的那种方式，Claude 不是在回答"这段代码可能有什么问题"，而是在真正运行代码、观察崩溃输出、调整输入参数、重新运行验证——这是一个主动探索的循环过程，不是一个一问一答的过程。如果你想认真地把 AI 用于安全测试，值得投入时间搭建一个有工具环境的 agent 框架，让 AI 能真正执行代码，而不是停留在让它评论代码文本这个层次。

第三点，GhostScript 的案例对工程实践有一个非常直接的启示：要认真对待每一个安全相关的 Git 提交，追问"这个修复是否完整"。文章里那个漏洞在 Git 历史里留下了痕迹，修复补丁只处理了一个调用点，另一个调用点被遗漏了，结果在十几年后被发现是漏洞。现在就可以建立一个团队习惯：每当合并一个安全修复，问自己这两个问题：这个补丁修的逻辑，在代码库的其他地方有没有类似的代码路径同样需要处理？这个被修复的漏洞模式，有没有其他表现形式还没有被覆盖到？这是一个成本极低、但能显著提高安全质量的工程实践。

第四点，这篇文章让我对开源软件的安全状况有了更多的思考。文章提到，他们选择先从开源软件入手，是因为开源软件运行在从企业系统到关键基础设施的各个角落，漏洞的影响面极广，而且很多项目只有小团队甚至个人志愿者在维护，根本没有专门的安全资源。现在 Anthropic 在向这些项目提交漏洞报告并提供修复补丁，这对开源生态是一件好事。但这同时意味着，漏洞被发现的速度在加快，被利用的门槛在降低。对于所有依赖开源组件的工程师和团队来说，保持依赖库版本及时更新的重要性比以前更高了。一个"已经有补丁但你没有更新"的漏洞，对攻击者来说几乎和零日漏洞一样好用，因为公开的补丁本身就提示了漏洞在哪里。

第五点，我想说一个更宏观的视角。这篇文章描述的是防御性使用，但同样的能力放在攻击者手里也是存在的。Anthropic 在文章里非常明确地说，他们的目的是抢在窗口关闭之前，尽可能多地用 AI 找到并修复漏洞。这个"窗口"的意思是：现在防御方如果行动足够快，可以用同样的 AI 能力把防御覆盖面大幅拉高；但这个机会不是永久开着的，随着时间推移，攻击方同样会充分利用这些能力。我个人判断，在 AI 安全能力这件事上，防御方有一个独特的优势：防御需要覆盖所有可能的攻击面，攻击只需要找到一个突破口。如果防御方能用 AI 足够快地扫描和修复，可以积累一个显著的优势。但这需要主动行动，而不是观望。

最后，我想说一下这篇文章在方法透明度上值得学习的地方。Anthropic 选择把整个过程公开讲清楚：他们的设置是什么，他们怎么验证，他们找到了什么，他们加了什么保护，他们预见到什么风险。这种透明度让安全研究社区可以独立评估他们的结论，也让外界可以对防护措施提出批评。我个人认为，在 AI 能力快速进步的这个阶段，这种"发布能力的同时发布方法论和防护设计"的做法，是比较负责任的模式，值得行业参考。

好，我们来做最后的总结。

这篇"LLM-discovered 0 days"告诉我们一件有历史意义的事：AI 已经具备了独立发现真实零日漏洞的能力，并且在经过多年充分测试的代码库里，找到了人类几十年都没有发现的问题。这不是实验室演示，是已经在修复中的真实漏洞。

更深刻的含义是：AI 不只是在做人类以前做过的事情，它在用一种从根本上不同的方式推理代码——读历史、理解算法、追踪逻辑链。这种能力叠加上 AI 的规模优势，意味着安全领域的攻防格局正在被重新定义，而且这个过程正在发生，不是将来时。

无论你是写 C 语言的系统工程师，还是维护 Python 服务的后端开发者，还是在做 AI 应用的从业者，我个人建议你把这篇文章认认真真读一遍，因为它描述的不是一个遥远的未来，而是我们现在就身处其中的现实。
